## 题目一：HBase 的整体架构是怎样的？请描述核心组件及其职责。

### 参考答案

HBase 是一个基于 Google BigTable 思想设计的分布式列族存储系统，其架构采用 **Master-Slave** 模式，核心组件包括以下几部分：

### 1. Zookeeper（协调服务）
- 负责 HBase 的分布式协调，管理 Regionserver 的活状态。
- 实现 Master 高可用：当 Master 宕机时，通过 ZK 选举新的 Master。
- 存放 `META` 表位置信息、集群配置等元数据。

### 2. HMaster（主节点）
- 管理所有 Region 的分配与调度。
- 负责表（Table）的创建、删除、修改（DDL 操作）。
- 监控 Regionserver 健康状态，自动进行负载均衡和故障恢复。
- 在 HBase 2.x 中，HMaster 角色已弱化，很多管理功能下放到 Regionserver。

### 3. Regionserver（数据节点）
- 负责数据的读写操作，是实际存储数据的地方。
- 一个 Regionserver 包含多个 Region，每个 Region 对应一张表的一段行键范围。
- 承担数据 flush、compact、split 等存储策略。

### 4. Region（数据分区）
- 是 HBase 中数据存储和负载均衡的基本单元。
- 每个 Region 由起始行键（StartKey）和结束行键（EndKey）标识。
- 当 Region 过大时，会自动拆分为两个子 Region。

### 5. Store（列族存储）
- 每个 Region 由一个或多个 Store 组成，每个 Store 对应一个列族（Column Family）。
- Store 由 MemStore 和 StoreFile（HFile）组成。

### 6. Write-Ahead Log（WAL）
- 所有写操作先写入 WAL，再写入 MemStore，确保数据持久化和故障恢复。

### 数据读写流程简述
- **写**：Client → ZK 获取 META 位置 → 找到目标 Regionserver → 写入 WAL → 写入 MemStore → 返回成功。
- **读**：Client → ZK → META → 定位 Region → 先查 MemStore，未命中则查 StoreFile。

> 📎 来源：https://hbase.apache.org/book.html#arch.overview

## 题目二：HBase 的 RowKey 设计原则有哪些？如何设计一个高效的 RowKey？

### 参考答案

RowKey 是 HBase 中用于唯一标识一行记录的主键，其设计直接影响查询性能和读写吞吐量。核心原则如下：

### 1. 长度原则
- RowKey 长度建议控制在 **8~16 字节**（最大 64KB）。
- 原因：HBase 持久化到 HDFS 时，数据按 HFile 组织，过长的 RowKey 会浪费存储空间、增加 MemStore 内存压力。

### 2. 散列原则
- 避免单调递增或递减的 RowKey（如时间戳 `20260708120000`），否则会导致所有数据集中写入同一个 Region（热点 Region）。
- **解决方案**：对单调部分做 MD5 散列、反转时间戳、加盐（Salt）等。

### 3. 唯一性原则
- RowKey 必须唯一定位一行数据，否则同键数据会相互覆盖。

### 4. 查询模式匹配原则
- RowKey 设计应匹配业务常见查询场景：
  - **等值查询**：将目标字段放在 RowKey 前缀（如 `userId_orderId`）。
  - **范围查询**：将时间维度放在 RowKey 尾部，支持字典序范围扫描（如 `deviceId_timeReverse`）。
  - **多条件组合**：可使用组合键，并善用 Scan 的 startRow/stopRow。

### 5. 避免热点策略

| 策略 | 说明 | 示例 |
|------|------|------|
| 加盐（Salt） | 在 RowKey 前拼接 Hash 前缀，打散数据 | `MD5(userId).substring(0,1)_userId` |
| 随机散列 | 对时间戳取反，让新数据分布在不同 Region | `deviceId_reverseTimestamp` |
| 哈希 | 对业务键做 Hash，兼顾均匀分布 | `SHA(orderId).substring(0,6)_orderId` |

### 6. 典型场景示例

**场景：电商订单系统，按用户查询订单**
```
# 差的设计（热点问题）
RowKey: orderId（单调递增）

# 好的设计
RowKey: MD5(userId).substring(0,2) + "_" + userId + "_" + orderId
```
这样不同用户的数据自然散列到不同 Region，避免写入热点。

> 📎 来源：https://hbase.apache.org/book.html#rowkey.design

## 题目三：HBase 的存储结构是怎样的？请解释 MemStore 和 HFile 的工作原理。

### 参考答案

### 1. HBase 存储层次结构

```
Table
 └── Region
      └── Store（每个 Column Family 一个）
           ├── MemStore（写缓存）
           └── StoreFile（HFile） × N
                └── HFile（HDFS 上的数据文件）
```

### 2. MemStore（写缓存）
- **作用**：接收所有写请求，先写入内存中的 MemStore（按 RowKey 排序的跳表结构），而不是直接落盘。
- **优势**：利用内存排序避免随机写，大幅提升写入性能。
- **Flush 触发条件**（满足任一即触发）：
  - MemStore 大小超过 `hbase.hregion.memstore.flush.size`（默认 128MB）。
  - Region 中所有 MemStore 总大小超过 `hbase.regionserver.global.memstore.size`（默认堆内存 40%）。
  - WAL 文件数量超过 `hbase.regionserver.max.logs`（默认 32）。
- **Flush 过程**：满足条件时，将 MemStore 内存数据按列族写出为 HFile，然后清空 MemStore。

### 3. HFile（HBase 数据文件）
- HBase 底层数据文件格式，构建在 HDFS 之上。
- **数据结构**：HFile 是有序键值对存储，包含以下主要区域：
  - **Data Block**：存储实际的 KeyValue 数据，按 B+ Tree 索引加速范围扫描。
  - **Meta Block**：存储元信息，如布隆过滤器（Bloom Filter）数据。
  - **File Info**：文件级别的元数据（最后修改时间、键值对数量等）。
  - **Index Block**：数据块索引，支持快速定位 KeyValue。

### 4. HFile 的索引机制
- **B+ Tree 树索引**：每个 HFile 有一棵 B+ Tree 索引，索引节点指向数据块位置。
- **布隆过滤器（Bloom Filter）**：快速判断一个 Key 是否存在于 HFile 中，避免无效读取。
  - 每个 HFile 存储一个 BloomFilter 位数组。
  - 写入时将 Key 加入 BloomFilter，读取时先检查，O(1) 判断是否存在。

### 5. Minor Compaction 与 Major Compaction

| 类型 | 说明 | 触发条件 |
|------|------|----------|
| Minor Compaction | 合并少量小 HFile 为大 HFile，删除重复版本和删除标记 | HFile 数量超过 `hbase.hstore.compaction.min`（默认 3） |
| Major Compaction | 将一个 Store 下所有 HFile 合并为一个，删除所有过期数据和 tombstone | 可手动或配置自动触发（默认每 7 天） |

### 6. 数据写入完整流程

```
1. 写入请求 → Regionserver
2. 先写 WAL（Write-Ahead Log）到 HDFS（保证持久化）
3. 数据写入 MemStore（内存有序跳表）
4. 满足 Flush 条件 → 生成新的 HFile 写入 HDFS
5. 后台定期 Compaction 合并 HFile，减少文件数量
```

> 📎 来源：https://hbase.apache.org/book.html#perf.compactions

## 题目四：什么是 HBase 协处理器（Coprocessor）？它有哪两种类型？各有什么用途？

### 参考答案

### 1. 协处理器概述

协处理器是 HBase 提供的扩展机制，允许用户将自定义代码**部署到 Regionserver 节点**，在数据存储层直接执行自定义逻辑，而无需在客户端做额外处理。

**核心价值**：减少网络传输，提升性能，实现服务端计算下推。

### 2. 两种协处理器类型

#### 类型一：Observer（观察者）

**原理**：类似数据库触发器，在特定事件前后插入自定义逻辑。

**使用场景**：
- **PreGet**：读取数据前执行，可用于权限校验、结果过滤。
- **PostGet**：读取数据后执行，可添加列级访问审计。
- **PrePut / PostPut**：写入数据前后执行，可用于数据验证、触发关联更新。
- **PreDelete / PostDelete**：删除数据前后执行，清理关联数据。
- **PreOpen**：Region 打开时初始化资源。

**示例（PrePut 拦截器）**：
```java
@Override
public void prePut(ObserverContext<RegionCoprocessorEnvironment> c,
                   Put put, WALEdit edit) throws IOException {
    // 在写入前校验数据，例如拒绝不合规的列值
    if (containsInvalidData(put)) {
        throw new DoNotRetryIOException("Invalid data detected");
    }
}
```

#### 类型二：Endpoint（端点）

**原理**：类似数据库存储过程，允许在服务端执行复杂的多行、多列族聚合计算，并将结果返回客户端。

**使用场景**：
- 在服务器端执行 SUM、AVG、COUNT 等聚合运算，减少数据传输。
- 跨列族、跨列的复杂计算。
- 自定义过滤器逻辑。

**示例（自定义聚合 Endpoint）**：
```java
// 定义协议接口
public interface ColumnAggregationProtocol extends CoprocessorProtocol {
    long sum(byte[] family, byte[] qualifier) throws IOException;
}

// 服务端实现
public class ColumnAggregationEndpoint implements RegionObserver, ColumnAggregationProtocol {
    @Override
    public long sum(byte[] family, byte[] qualifier) throws IOException {
        // 在 Region 内部直接遍历 HFile 计算聚合值
        // 返回聚合结果给客户端
    }
}
```

**客户端调用**：
```java
HTableInterface table = ...;
byte[] startRow = ...;
byte[] endRow = ...;
CoprocessorRpcChannel channel = table.coprocessorProxy(ColumnAggregationProtocol.class, startRow);
ColumnAggregationProtocol service = (ColumnAggregationProtocol) channel;
long result = service.sum(Bytes.toBytes("cf"), Bytes.toBytes("amount"));
```

### 3. 加载方式

| 方式 | 说明 |
|------|------|
| 静态加载（表级） | 通过 HBase 配置 `hbase.coprocessor.region.classes` 指定，实现类必须在 HBase classpath 下 |
| 动态加载（表级） | 通过 HBase Shell 或 API 在表创建时指定：`alter 'tableName', METHOD => 'table_att', 'coprocessor' => '...' ` |
| Observer 动态加载 | 适合临时拦截逻辑；Endpoint 动态加载有限制 |

### 4. 使用注意事项
- 协处理器运行在 Regionserver 上，代码异常可能导致 Region 不可用，需做好异常处理。
- Observer 不能跨 Region 调用；Endpoint 可以并行调用多个 Region 后合并结果。
- 高并发场景下注意资源占用，避免内存泄漏。

## 题目五：HBase 如何实现高可用（HA）？如果一个 Regionserver 宕机，数据会丢失吗？

### 参考答案

### 1. HBase 高可用机制

#### 1.1 WAL（Write-Ahead Log）保证数据不丢失

- 所有写操作先写入 WAL（预写日志），再写入 MemStore。
- 即使 Regionserver 宕机，WAL 仍保存在 HDFS 上，通过 WAL 可以重放未持久化的数据。
- **宕机恢复流程**：
  1. Regionserver 宕机，Zookeeper 检测到心跳消失。
  2. Master 将该 Regionserver 上的所有 Region 分配给其他活着的 Regionserver。
  3. 其他 Regionserver 读取宕机服务器的 WAL 日志，将未 flush 的数据重放到新的 Region 中。
  4. 数据恢复后对外提供服务。

#### 1.2 数据副本（基于 HDFS）

- HBase 数据最终存储在 HDFS 上，HDFS 默认 3 副本机制。
- 即使某个节点损坏，HDFS 会自动复制缺失的副本，保证数据不丢失。
- 副本策略：
  - 机架感知：3 个副本分布在不同机架，提升容灾能力。
  - Region 副本：HBase 2.x 支持同一 Region 的多副本部署（Replica），支持强一致性读。

#### 1.3 Master 高可用（基于 Zookeeper）

- HBase 支持配置多个 Master（`hbase.master`) 通过 Zookeeper 竞争锁实现主备切换。
- 主 Master 宕机后，备用 Master 自动接管，无需人工干预。

#### 1.4 META 表高可用

- `hbase:meta` 表记录所有 Region 的位置信息，存储在 Zookeeper 中。
- META 表本身也是 HBase 表，同样有 Region 副本和 WAL 保护。

### 2. 数据可靠性分析

| 场景 | 数据是否丢失 | 原因 |
|------|-------------|------|
| 单个 Regionserver 宕机 | **不丢失** | WAL + HDFS 副本保证 |
| 单个 HDFS DataNode 宕机 | **不丢失** | HDFS 自动补副本 |
| 整个机架故障 | **可能丢失**（默认 3 副本同机架时） | 配置机架感知可缓解 |
| 磁盘损坏 | **不丢失**（仅 1 块磁盘） | HDFS 副本在其他节点 |
| MemStore 未 Flush 时宕机 | **不丢失** | WAL 重放恢复 |

### 3. 提升 HBase 可用性的最佳实践

```xml
<!-- hbase-site.xml 配置示例 -->
<property>
    <name>hbase.regionserver.failfast.enabled</name>
    <value>true</value>  <!-- 快速failover，减少恢复时间 -->
</property>
<property>
    <name>hbase.meta.replica.count</name>
    <value>3</value>     <!-- META 表多副本 -->
</property>
<property>
    <name>hbase.regionserver.region.replica.wait.for.primary.flush</name>
    <value>false</value>
</property>
```

### 4. 读高可用：Region Replica（2.x）

- HBase 2.x 支持将一个 Region 部署为多个副本（主副本可写，备副本只读）。
- 读请求可以打到备副本，分担主副本压力，提升读取吞吐量。
- 主副本宕机时，备副本自动升级，不影响读写。
