## 题目1：Java基础 | ThreadLocal内存泄漏与弱引用机制

**题目：** ThreadLocal是如何导致内存泄漏的？为什么JDK采用弱引用而不是强引用？实际开发中如何避免？

**【答案】**

### ThreadLocal结构回顾

每个Thread对象内部有一个`ThreadLocalMap`，本质是一个自定义的HashMap。它的key是ThreadLocal对象（即ThreadLocal的弱引用），value是我们存入的值。

```java
// JDK中ThreadLocalMap的Entry定义
static class Entry extends WeakReference<ThreadLocal<?>> {
    Object value;
    Entry(ThreadLocal<?> k, Object v) {
        super(k); // key是弱引用
        value = v;
    }
}
```

### 内存泄漏原因分析

**弱引用机制：** JDK使用弱引用（WeakReference）来引用ThreadLocal key。当ThreadLocal对象不再被其他强引用持有时（如方法执行完毕，局部变量ThreadLocal=null），GC运行时自动回收这个ThreadLocal对象，Entry中的key被清理为null。

**但是！value是强引用：** Entry的value仍然持有对应ThreadLocal value的强引用。只要Thread还在（线程池场景下线程被复用），整个ThreadLocalMap就存在，Entry的value就无法被回收——这就形成了**内存泄漏**。

具体泄漏路径：
1. 线程执行方法，创建ThreadLocal并set()值，value进入ThreadLocalMap
2. 方法结束，ThreadLocal局部变量被GC回收（因为key是弱引用）
3. 线程被线程池复用，生命周期很长
4. ThreadLocalMap中的Entry.key=null，但value依然存在且无法访问（泄漏）

### 为什么要用弱引用？

如果用强引用：当ThreadLocal对象不再使用时，ThreadLocalMap中对应Entry的key仍然引用着ThreadLocal对象，导致ThreadLocal永远无法被GC回收——即每次set()都会产生内存泄漏，且随时间累积。

用弱引用至少保证了：**只要ThreadLocal对象不再被外部引用，key会自动被GC清理，只剩value可能泄漏**，缩小了泄漏范围。

### 如何避免内存泄漏？

**最佳实践：在finally或try-finally中手动remove()**

```java
ThreadLocal<User> userThreadLocal = new ThreadLocal<>();
try {
    User user = getCurrentUser();
    userThreadLocal.set(user);
    // 业务逻辑
} finally {
    userThreadLocal.remove(); // 关键！
}
```

**ThreadLocal使用原则：**
- 尽量使用局部变量
- 在Web请求/线程任务结束时务必remove()
- 避免将大对象放入ThreadLocal（加剧泄漏影响）
- 线程池场景下，线程复用周期长，泄漏危害更大

## 题目2：Java进阶 | AQS同步队列与Condition条件队列的关系

**题目：** AbstractQueuedSynchronizer（AQS）中的同步队列和Condition条件队列有什么区别？await()和signal()为什么必须在独占锁持锁状态下调用？

**【答案】**

### 两类队列的本质区别

AQS内部维护了两个核心队列：

**1. 同步队列（Sync Queue）**
- 用于管理所有等待获取**独占锁/共享锁**的线程
- CLH队列的变体，通过prev/next指针串联
- 队列头节点是正在获得锁的线程（或已获得锁的线程）
- 当锁释放时，队列头节点的下一个waiter被唤醒竞争锁

**2. 条件队列（Condition Queue）**
- 通过`ConditionObject`实现，每个Condition对象内部有一个独立队列
- 调用`await()`时：当前线程从同步队列移出，加入该Condition的条件队列，释放锁
- 调用`signal()`时：将条件队列头节点线程移出，重新放入同步队列等待获取锁
- 一个锁可以有多个Condition（一个互斥锁对应多个条件变量）

### 关键数据结构对比

| | 同步队列 | 条件队列 |
|---|---|---|
| 所在位置 | AQS内部（CLH变体） | ConditionObject内部 |
| 节点唤醒 | 锁释放时自动唤醒队首 | signal时移动到同步队列 |
| 队列数量 | 每个AQS只有1个 | 每个Condition有1个，可创建多个 |
| 线程状态 | waitStatus表示等待状态 | waitStatus=CONDITION(-2) |

### 为什么await/signal必须在持锁状态调用？

**这是设计层面的强制约束，不是技术限制。**

`await()`的语义是："等待某个条件为真，在条件为真之前不持有锁"。实现逻辑是：
1. 当前线程必须已持有独占锁（否则无法调用release()释放锁）
2. 将线程加入条件队列（原子操作，需要对AQS状态写操作）
3. 完全释放锁（state=0），让其他线程可以进入同步队列
4. 当被signal重新入同步队列时，需要再次持有锁才能继续

如果`await()`在不持锁状态下调用，意味着线程没有持有锁却尝试释放锁——这在AQS的设计语义上是矛盾的。

## 题目3：Spring | 为什么Spring三级缓存能解决循环依赖

**题目：** Spring是如何通过三级缓存解决Bean的循环依赖问题？三级缓存分别存储什么？为什么二级缓存不足以解决所有循环依赖？

**【答案】**

### Spring三级缓存（DefaultSingletonBeanRegistry）

```
一级缓存: singletonObjects         → 完整Bean实例（可直接使用）
二级缓存: earlySingletonObjects    → 早期暴露的Bean（已实例化但未完成属性注入/初始化）
三级缓存: singletonFactories       → Lambda表达式 ObjectFactory<?>（工厂，用于生成early引用）
```

### 三级缓存解决循环依赖的过程

以A→B→A为例（setter注入场景）：

**Step1：创建A，触发属性注入b**
1. `getSingleton(A)` → 一二三级缓存都没找到
2. 调用`createBean(A)`，创建A的ObjectFactory放入三级缓存
3. A属性注入时，需要B → 调用`getBean(B)`

**Step2：创建B，触发属性注入a**
4. `getSingleton(B)` → 缓存都没找到
5. 调用`createBean(B)`，创建B的ObjectFactory放入三级缓存
6. B属性注入时，需要A → 调用`getBean(A)`

**Step3：再次获取A，B触发循环依赖破解**
7. `getSingleton(A)` → 三级缓存找到了A的ObjectFactory
8. 调用`objectFactory.getObject()` → **从三级缓存升级到二级缓存**，得到A的早期引用
9. 将这个早期A引用注入给B，B创建完成
10. B被放入**一级缓存**（完整Bean）

**Step4：B返回给A，A完成创建**
11. A获得B的引用（来自一级缓存的完整B），A创建完成
12. A从三级缓存中删除ObjectFactory，升级到一级缓存

### 为什么二级缓存不够？

**场景：代理对象的循环依赖**

如果A是代理Bean，B依赖A，A依赖B：
- 创建A时，ObjectFactory中存储的是**能产生代理对象的工厂**（三级缓存）
- B从三级缓存拿到A的早期引用，这个引用需要是**代理对象**，而不是原始对象
- 代理对象的生成发生在`BeanPostProcessor`阶段，在属性填充之后

如果只有二级缓存：
- 当A被"早期暴露"时，放入二级缓存的是普通对象（实例化完成但无代理）
- 后续A完成属性注入，BeanPostProcessor生成代理，覆盖了二级缓存中的普通对象
- B持有的A引用还是普通对象，不是代理对象 → **代理失效**

**三级缓存的设计：** 三级缓存存储的是工厂（`ObjectFactory`），每次从三级缓存获取对象时，都通过工厂产生，这确保了**每次获取的都是最新的对象**（可能是代理对象）。

### 构造器注入循环依赖为什么不能解决？

三级缓存解决的是**setter注入和@Bean方法注入**的循环依赖。构造器注入要求：**对象在构造时必须完全初始化完成**，形成死锁，无法通过缓存解决。

## 题目4：Spring Boot | Spring MVC请求完整生命周期与拦截器链

**题目：** 描述一个HTTP请求从发起到Controller方法执行，再到响应返回，Spring MVC完整经过哪些组件？拦截器链与过滤器链的区别是什么？

**【答案】**

### 完整请求生命周期

```
请求 → FilterChain（Servlet Filter）
    → DispatcherServlet（Servlet）
        → HandlerMapping（找到对应Controller方法）
        → HandlerInterceptor.preHandle（拦截器前置处理）
        → HandlerAdapter调用Controller
            → @Controller业务逻辑
        → HandlerInterceptor.postHandle（拦截器后置处理）
        → View解析/JSON序列化
        → 异常处理（@ExceptionHandler）
        → HandlerInterceptor.afterCompletion（最终清理）
    → FilterChain.doFilter继续
响应
```

### 各组件详解

**1. Filter（过滤器）**
- Servlet规范层面的组件，与Spring无关
- 可处理所有请求（包括静态资源）
- 在DispatcherServlet之前执行，可做字符编码、登录校验、CORS等

**2. DispatcherServlet**
- Spring MVC的前端控制器（Front Controller）
- 接收所有匹配URL的请求，是整个请求处理的中央调度器

**3. HandlerMapping**
- 根据请求URL找到对应的Controller方法
- 典型实现：`RequestMappingHandlerMapping`（处理@RequestMapping）

**4. HandlerInterceptor（拦截器）**
- Spring MVC提供的拦截器接口，三个方法：
  - `preHandle`：Controller方法执行前调用，返回true才会继续
  - `postHandle`：Controller方法执行后、视图渲染前调用
  - `afterCompletion`：整个请求完成后调用（无论是否异常）

### 过滤器链 vs 拦截器链核心区别

| | Filter | HandlerInterceptor |
|---|---|---|
| 归属 | Servlet规范（javax.servlet） | Spring MVC（org.springframework.web.servlet） |
| 执行顺序 | 在DispatcherServlet之前 | 在DispatcherServlet内部 |
| 能拦截的范围 | 一切请求（Servlet级别） | 只拦截进入Controller的请求 |

## 题目5：Spring Cloud | Ribbon客户端负载均衡机制

**题目：** Ribbon是如何实现客户端负载均衡的？它与Nacos/Discovery的配合流程是怎样的？有哪些负载均衡策略？

**【答案】**

### Ribbon是什么

Ribbon是Netflix开源的**客户端负载均衡器**（Client Side Load Balancer），负载均衡器在客户端自己维护所有服务实例列表，自己选择用哪个。

### 工作流程

```
消费者微服务启动
    → RibbonClientConfiguration配置
        → 注册ServerList（服务实例列表）
            → 从Eureka/Nacos拉取或心跳更新服务实例
消费者发起HTTP请求
    → RestTemplate被@LoadBalanced拦截
        → LoadBalancerInterceptor拦截请求
            → 解析URL中的服务名（如 http://user-service/user/1）
            → 调用Ribbon的ILoadBalancer
                → 使用IRule选择服务实例
                → 发起真正的HTTP调用
```

### 关键组件

| 组件 | 职责 |
|---|---|
| `ServerList` | 存储可用服务实例列表，支持动态刷新 |
| `IRule` | 负载均衡策略，选择具体哪个实例 |
| `IPing` | 心跳检测，判断服务实例是否存活 |
| `ILoadBalancer` | 负载均衡器核心，协调上述组件 |

### 负载均衡策略（IRule）

**1. RoundRobinRule（默认）** — 轮询，每次按顺序选择下一个

**2. RandomRule** — 随机，随机选择一个实例

**3. RetryRule** — 重试，先轮询，失败则重试一定次数内再选

**4. WeightedResponseTimeRule** — 加权，根据响应时间分配权重，响应快的概率高

**5. AvailabilityFilteringRule** — 过滤策略：连续失败3次的实例进入"黑名单"，跳过

**6. ZoneAvoidanceRule（默认区域感知）** — 优先选择同Zone实例，避免跨Zone网络延迟

## 题目6：消息队列 | RocketMQ事务消息的执行流程与回查机制

**题目：** RocketMQ事务消息是如何保证本地事务与消息发送原子性的？请描述两阶段提交流程，以及事务消息回查（TransactionCheck）的触发条件和处理逻辑。

**【答案】**

### 两阶段提交流程

**第一阶段：发送半事务消息（Half Message）**

Producer向RocketMQ发送一条"半事务消息"（Half Message）：
- MQ收到消息后**不会投递给消费者**（状态=Prepared）
- 同时向Producer回调"发送成功"

```java
transactionMQProducer.sendMessageInTransaction(msg, new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        // 执行本地业务（库存扣减、订单创建等）
        try {
            orderService.createOrder(msg.getKeys(), content);
            return LocalTransactionState.COMMIT_MESSAGE; // 成功，提交
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE; // 失败，回滚
        }
    }

    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 回调检查：查本地库确认事务状态
        return queryOrderStatus(msg.getKeys());
    }
});
```

**第二阶段：提交/回滚**

- 如果本地事务执行成功 → Producer回调`COMMIT_MESSAGE` → 半事务消息转为正常消息，投递给消费者
- 如果本地事务失败 → Producer回调`ROLLBACK_MESSAGE` → 半事务消息被丢弃

### 事务回查机制

**为什么需要回查？**

Producer在回调`executeLocalTransaction`后，如果Broker未收到提交/回滚指令（网络抖动、Producer崩溃），Broker需要主动回查确认事务状态。

**触发条件：**
- 发送半事务消息后，Broker等待超时（默认6秒）未收到`commit/rollback`
- Broker向Producer发送"检查事务状态"请求
- Producer的`TransactionListener.checkLocalTransaction()`被调用

### 事务消息与普通消息的核心区别

| | 普通消息 | 事务消息 |
|---|---|---|
| 发送方式 | `producer.send()` | `producer.sendMessageInTransaction()` |
| 消息状态 | 发送即可消费 | 先Prepared，提交后才可消费 |
| 消费时机 | 发送即投递 | 事务提交后投递 |

## 题目7：消息队列 | Kafka消费者组Rebalance时机与影响优化

**题目：** 哪些场景会触发Kafka Consumer Group的Rebalance？Rebalance带来的问题是什么？实际项目中如何优化？

**【答案】**

### 触发Rebalance的典型场景

**1. 组成员变化**
- 新消费者加入组
- 已有消费者离开（主动unsubscribe或崩溃）

**2. 订阅主题变化**
- 通过`consumer.subscribe()`动态修改了订阅的topic列表

**3. 分区数变化**
- Kafka管理员为topic增加了新分区

**4. 消费者会话超时**
- 消费者在`session.timeout.ms`内未发送心跳，被Broker判定为"死亡"

### Rebalance带来的问题

**1. 消费暂停（Stop The World）**
- Rebalance期间，所有消费者停止处理消息
- 消息积压（Lag上升）

**2. 重复消费**
- Rebalance发生在消费者拉取消息之后、处理完成之前
- 消费者可能重新拉取同一条消息 → **重复消费**
- 需要消费端幂等处理

### 优化策略

**策略1：设置合理的session.timeout.ms**
```properties
session.timeout.ms=30000   # 不要太小，避免误判
heartbeat.interval.ms=10000 # 一般为 session.timeout.ms 的 1/3
max.poll.interval.ms=300000 # 处理逻辑复杂时调大
```

**策略2：使用静态成员（Static Membership）**
```properties
group.instance.id=consumer-1  # 消费者重启后ID不变，不会触发完整Rebalance
```

**策略3：使用cooperative.sticky分区分配策略**
```properties
partition.assignment.strategy=cooperative.sticky  # Kafka 2.4+，增量再均衡
```
只迁移必要的分区，消费者的处理可以继续进行。

## 题目8：AI Agent | LangChain中Memory组件的核心机制与对话上下文管理

**题目：** LangChain的Memory组件如何管理多轮对话的上下文？ConversationBufferMemory、ConversationSummaryMemory和VectorStoreRetrieverMemory分别适用于什么场景？

**【答案】**

### Memory在Agent中的作用

在大模型应用（Agent）中，每次请求只能携带有限的上下文。Memory组件负责：
- **存储**：将历史对话/消息持久化存储
- **管理**：提供读写接口，供Agent在每轮对话时获取历史上下文
- **压缩**：当对话历史过长时，做摘要/压缩，避免超出LLM的context窗口

### 主要Memory类型

**1. ConversationBufferMemory**

最基础的内存，直接存储原始消息列表：

```python
from langchain.memory import ConversationBufferMemory

memory = ConversationBufferMemory(
    return_messages=True,
    memory_key="chat_history"
)
memory.save_context({"input": "今天天气如何"}, {"output": "晴天"})
```

**适用场景：** 对话轮次少（<20轮）、上下文总长度在LLM context范围内。不做摘要，保留完整信息。

**2. ConversationSummaryMemory**

当对话很长时，自动对历史做摘要：

```python
from langchain.memory import ConversationSummaryMemory

memory = ConversationSummaryMemory(
    llm=OpenAI(model="gpt-3.5-turbo"),
    memory_key="chat_history"
)
```

每次添加对话后，LLM会自动将历史"压缩"为一段摘要。**适用场景：** 中长对话（20-100轮），需要在有限context窗口内保持信息完整性。

**3. VectorStoreRetrieverMemory**

将每条记忆向量化，存入向量数据库，检索时取最相关的记忆：

```python
from langchain.memory import VectorStoreRetrieverMemory

memory = VectorStoreRetrieverMemory(
    retriever=vectorstore.as_retriever(search_kwargs={"k": 3}),
    memory_key="relevant_memories"
)
```

检索时根据当前输入query，找出最相关的历史记忆。**适用场景：** 对话跨度大、历史记忆需要"选择性回忆"的场景。

### 在Agent中集成Memory

```python
from langchain.agents import OpenAIConversationalAgent

agent = OpenAIConversationalAgent(
    llm=llm,
    prompt=prompt,
    memory=memory  # 将memory注入Agent
)
```

### 实际开发注意事项

- **Token限制：** memory的总大小必须在LLM context窗口之内
- **摘要频率：** 不要等对话太长才摘要——中途摘要丢失的信息更少
- **检索精度：** VectorStoreRetrieverMemory的效果依赖embedding质量，需选择合适的embedding模型

## 题目9：算法 | LRU缓存淘汰算法

**题目：** 请实现一个 Least Recently Used（LRU）缓存，要求 `get(key)` 和 `put(key, value)` 的时间复杂度均为 O(1)。

**【答案】**

### 思路分析

LRU要求：
- `get(key)`：访问某个key，将其标记为"最近使用"
- `put(key, value)`：插入或更新key，达到容量上限时淘汰"最久未使用"的key

O(1)要求我们需要同时支持：
- **快速查找**（HashMap）
- **快速排序/淘汰最近未使用**（链表：按访问顺序排列，尾部是最久未使用）

解决方案：**HashMap + 双向链表**

- HashMap：key → 链表节点的映射，O(1)查找
- 双向链表：节点按访问顺序排列，头部是最近使用，尾部是最久未使用
- get时将节点移到头部
- put时新节点插入头部，容量超限时删除尾部节点

### Java实现

```java
import java.util.*;

public class LRUCache {
    private static class Node {
        int key, value;
        Node prev, next;
        Node(int k, int v) { key = k; value = v; }
    }

    private final int capacity;
    private final Map<Integer, Node> cache;
    private final Node head, tail;

    public LRUCache(int capacity) {
        this.capacity = capacity;
        this.cache = new HashMap<>(capacity);
        this.head = new Node(0, 0);
        this.tail = new Node(0, 0);
        head.next = tail;
        tail.prev = head;
    }

    public int get(int key) {
        Node node = cache.get(key);
        if (node == null) return -1;
        moveToHead(node);
        return node.value;
    }

    public void put(int key, int value) {
        Node node = cache.get(key);
        if (node != null) {
            node.value = value;
            moveToHead(node);
        } else {
            Node newNode = new Node(key, value);
            cache.put(key, newNode);
            addToHead(newNode);
            if (cache.size() > capacity) {
                Node removed = removeTail();
                cache.remove(removed.key);
            }
        }
    }

    private void moveToHead(Node node) {
        removeNode(node);
        addToHead(node);
    }

    private void addToHead(Node node) {
        node.prev = head;
        node.next = head.next;
        head.next.prev = node;
        head.next = node;
    }

    private void removeNode(Node node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }

    private Node removeTail() {
        Node node = tail.prev;
        removeNode(node);
        return node;
    }
}
```

### 复杂度分析

| 操作 | 时间复杂度 |
|---|---|
| get | O(1) |
| put | O(1) |
| 空间 | O(capacity) |

## 题目10：算法 | 字符串通配符匹配（正则*与?）

**题目：** 实现一个正则表达式匹配算法，支持 `*`（匹配任意字符序列）和 `?`（匹配单个任意字符）。不使用语言内置的正则库，要求 `isMatch("aa","*")` 返回 true。

**【答案】**

### 思路分析

**动态规划：**
- `dp[i][j]` = p的前j个字符是否能匹配s的前i个字符
- 初始化：`dp[0][0] = true`（空正则匹配空字符串）
- `dp[0][j]`：p从0到j，只有当p全为`*`时才为true（如`***`匹配空串）

### DP状态转移

```java
public boolean isMatch(String s, String p) {
    int m = s.length(), n = p.length();
    boolean[][] dp = new boolean[m + 1][n + 1];
    
    dp[0][0] = true;
    for (int j = 1; j <= n; j++) {
        dp[0][j] = dp[0][j - 1] && p.charAt(j - 1) == '*';
    }
    
    for (int i = 1; i <= m; i++) {
        for (int j = 1; j <= n; j++) {
            char sc = s.charAt(i - 1);
            char pc = p.charAt(j - 1);
            
            if (pc == '*') {
                // * 可以匹配空串(dp[i][j-1]) 或 任意字符(dp[i-1][j])
                dp[i][j] = dp[i][j - 1] || dp[i - 1][j];
            } else if (pc == '?' || sc == pc) {
                dp[i][j] = dp[i - 1][j - 1];
            } else {
                dp[i][j] = false;
            }
        }
    }
    
    return dp[m][n];
}
```

### 状态转移图解

以 `s="aa"`, `p="*"` 为例：

```
        ""   *   
     ┌─────┬─────┐
  "" │ true│ true│   dp[0][0]=true, dp[0][1]=dp[0][0]&&p[0]=='*'
     ├─────┼─────┤
  a  │false│ true│   dp[1][1]: *匹配空或'a' → true
     ├─────┼─────┤
  a  │false│ true│   dp[2][1]: *匹配"a"或"aa" → true
     └─────┴─────┘
```

### 复杂度分析

- **时间复杂度：** O(m×n)，其中m为s长度，n为p长度
- **空间复杂度：** O(m×n)，可用滚动数组优化至O(n)
