## 题目 1：Spark 的运行架构是怎样的？Driver、Executor、Task 的关系是什么？

### 参考答案

Spark 采用 **Master-Slave** 架构，主要包含以下组件：

| 组件 | 职责 |
|------|------|
| **Driver（驱动器）** | 运行 `main()` 方法，创建 `SparkContext`，负责任务调度、将 Job 划分为 Stage 和 Task、分发 Task 到 Executor |
| **Executor（执行器）** | 存在于 Worker 节点上，负责执行分发的 Task，将数据保存在内存或磁盘，每个 Executor 有独立的 JVM 进程 |
| **Worker** | 集群节点管理者，负责启动和管理 Executor 进程 |
| **Master** | 集群资源管理者（Standalone 模式下），负责任务调度和资源分配 |

### 运行流程

1. **提交应用**：用户通过 `spark-submit` 提交应用，Driver 启动并创建 `SparkContext`
2. **申请资源**：Driver 向 Cluster Manager 申请 Executor 资源
3. **分发代码**：Cluster Manager 通知 Worker 节点启动 Executor
4. **发送 Task**：Driver 发送 Task 到对应 Executor，Executor 执行并将结果返回 Driver
5. **完成清理**：应用结束后，Driver 通知 Cluster Manager 注销 Executor

### Driver-Executor-Task 关系

- 一个 Spark Application 只有一个 Driver
- 每个 Executor 是一个独立的 JVM 进程，包含多个 **Task 线程**
- Task 是 Spark 的最小执行单元，每个 Task 处理一个数据分区
- 一个 Task 在一个 Executor 上执行，不会跨节点

> **来源参考**：[Spark 核心概念：Driver 与 Executor 详解](https://spark.apache.org/docs/latest/cluster-overview.html)

## 题目 2：RDD 的特性是什么？为什么 RDD 要设计成不可变的？

### 参考答案

### RDD（Resilient Distributed Dataset）五大特性

1. **`RDD` 是由一系列分区（Partition）组成的**  
   RDD 的数据被划分为多个分区，每个分区可以在集群的不同节点上并行计算。

2. **`RDD` 对每一步操作都有计算函数（Function）**  
   每个 RDD 都记录了父 RDD 的依赖关系（Lineage），用于容错恢复。

3. **`RDD` 依赖于父 RDD（Dependencies）**  
   RDD 之间的依赖关系分为窄依赖（Narrow Dependency，一对一或范围依赖）和宽依赖（Wide Dependency，一对多，常发生 Shuffle）。

4. **`RDD` 有分区器（Partitioner）**  
   对于 Key-Value 类型的 RDD，可指定分区策略（如 `HashPartitioner`、`RangePartitioner`）。

5. **`RDD` 优先选择数据就近的节点计算（Locality）**  
   Spark 会尽量将 Task 分发到数据所在节点，减少网络传输。

### 为什么 RDD 设计成不可变的？

| 原因 | 说明 |
|------|------|
| **简化并行计算** | 不可变性天然适合数据并行，因为无需担心数据竞争（Data Race）和锁机制 |
| **天然支持容错（Lineage）** | 每个 RDD 记录了完整的转换链，当某个节点数据丢失时，只需从父 RDD 重新计算该分区，而不需要复制整个数据集 |
| **支持函数式编程** | 不可变数据结构更容易实现惰性求值（Lazy Evaluation），便于 Spark 优化执行计划 |
| **避免副作用** | 分布式环境下，修改共享可变状态是危险的，不可变性从根本上避免了并发修改问题 |

> **来源参考**：[RDD 论文《Resilient Distributed Datasets: A Fault-Tolerant Abstraction for In-Memory Cluster Computing》](https://www.usenix.org/system/files/conference/nsdi12/nsdi12-final138.pdf)

## 题目 3：DataFrame 和 DataSet 的区别是什么？与 RDD 相比有什么优势？

### 参考答案

### DataFrame vs DataSet 对比

| 特性 | DataFrame | DataSet |
|------|-----------|---------|
| **数据结构** | 类似二维表（Row 类型的 RDD） | 强类型的 JVM 对象（RDD + Schema） |
| **语言支持** | Python/Java/Scala/R/SQL | 仅 Scala/Java（需要编译时类型） |
| **类型安全** | 无编译时类型检查 | 有编译时类型检查 |
| **序列化** | 无类型，运行时检查 | 编码器（Encoder）支持堆外序列化 |
| **操作方式** | DSL / SQL | Lambda 表达式 / DSL |
| **空值处理** | 支持 | 不支持（`null` 会抛异常） |

### DataFrame = DataSet[Row]

在 Scala 中，`DataFrame` 实际上是 `DataSet[Row]` 的别名，二者底层实现相同。

### DataFrame/DataSet 相比 RDD 的优势

1. **Spark SQL 优化器（Catalyst）**  
   DataFrame/DataSet 享有 **Query Optimization**。Catalyst 优化器会：
   - 谓词下推（Predicate Pushdown）
   - 列剪枝（Column Pruning）
   - 常量折叠（Constant Folding）
   - 广播 join（Broadcast Join）自动优化

2. **堆外内存管理（Off-Heap）**  
   DataFrame 使用 Tungsten 引擎，支持 **堆外序列化**，减少 GC 开销，提升内存效率。

3. **统一 API 与 SQL 支持**  
   同一种 API 既支持 DataFrame 操作，也支持直接写 SQL，门槛更低。

4. **更丰富的算子**  
   内置 `select`、`groupBy`、`join`、`窗口函数` 等 DSL，操作更简洁。

```scala
// RDD 版本（无优化）
rdd.filter(_.age > 30).map(_.name)

// DataFrame 版本（享有 Catalyst 优化）
df.filter($"age" > 30).select($"name")
```

> **来源参考**：[Spark SQL, DataFrames and Datasets Guide](https://spark.apache.org/docs/latest/sql-programming-guide.html)

## 题目 4：Spark Shuffle 的工作原理是什么？为什么 Shuffle 是性能瓶颈？

### 参考答案

### Shuffle 的触发时机

当 **一个 RDD 的分区需要依赖父 RDD 的多个分区数据** 时（即宽依赖），会触发 Shuffle。典型算子包括：`groupByKey`、`reduceByKey`、`join`、`repartition`、`sortByKey` 等。

### Shuffle 写阶段流程（Shuffle Write）

1. **Map 端缓冲区**：每个 Task 将数据写入内存缓冲区（默认 32KB/buffer）
2. **分区排序**：按目标分区号和 Key 进行排序（使用 TimSort）
3. **溢写磁盘**：缓冲区满时溢写到磁盘文件（spill file）
4. **合并文件**：一个 Task 产生多个 spill file，Task 完成后合并为最终的 **shuffle file**（每个分区一个 block）
5. **索引文件**：同时生成索引文件（`.index`），记录每个分区在文件中的偏移量

### Shuffle 读阶段流程（Shuffle Read）

1. **拉取数据**：每个 Task 根据分区号向不同的 Map 节点拉取数据
2. **聚合处理**：对拉取的数据按 Key 进行聚合（combine）或分组
3. **缓存数据**：聚合后的结果缓存在 Task 的内存或磁盘

### 为什么 Shuffle 是性能瓶颈？

| 原因 | 说明 |
|------|------|
| **磁盘 I/O** | Shuffle 产生大量磁盘读写，是最耗时的操作之一 |
| **网络 I/O** | 跨节点数据传输量大，带宽成为瓶颈 |
| **内存压力** | Map 端缓冲区、Aggregation 缓存都会消耗大量内存 |
| **磁盘空间** | Shuffle 文件会写入磁盘（默认 `spark.local.directory`），占用磁盘空间 |
| **启动开销** | Shuffle 触发 Job 划分，每个 Stage 之间需要 Barrier，同步开销大 |
| **OOM 风险** | 数据倾斜时，部分 Task 处理数据量远超其他 Task，易导致 OOM |

### 优化建议

- **减少 Shuffle 次数**：尽量用 `map-side` 预聚合（如 `reduceByKey` 优于 `groupByKey`）
- **广播变量**：小表广播可避免 Shuffle（`broadcast join`）
- **合理分区数**：避免过多或过少分区（一般 2-4 倍 CPU 核数）
- **数据倾斜处理**：加盐（Salting）、双重聚合、打散空键

> **来源参考**：[Shuffle 机制深度解析 - Apache Spark 官方博客](https://spark.apache.org/docs/latest/rdd-programming-guide.html#shuffle-operations)

## 题目 5：Spark 的内存管理模型是怎样的？堆内内存和堆外内存有什么区别？

### 参考答案

### Spark 内存管理模型（ Unified Memory Model ）

Spark 2.x 引入 **统一内存管理**，将内存划分为两大区域：

```
+------------------------------------------+
|            统一内存（Unified）             |
|  = 执行内存（Execution）+ 存储内存（Storage）|
|                                          |
|  动态共享：一方不足时可向另一方借用          |
|  空闲时相互归还                           |
+------------------------------------------+
|           预留内存（Reserved） 300MB        |
+------------------------------------------+
|            用户内存（User Memory）         |
|  用于 RDD 隐式转换、用户数据结构等          |
+------------------------------------------+
```

### 内存区域详解

| 区域 | 占比 | 用途 |
|------|------|------|
| **系统预留（Reserved）** | 300MB | 保障 Spark 内部核心组件的基本运行 |
| **统一内存（Unified）** | 约 60%（`spark.memory.fraction=0.6`） | Execution + Storage 动态共享 |
| **用户内存（User Memory）** | 约 40%（`spark.memory.storageFraction=0.5`） | RDD 隐式转换、UDF、用户数据结构 |

### 堆内 vs 堆外内存

| 特性 | 堆内内存（On-Heap） | 堆外内存（Off-Heap） |
|------|---------------------|---------------------|
| **管理方式** | JVM GC 自动管理 | 手动通过 `sun.misc.Unsafe` 管理 |
| **GC 影响** | 存在 GC 停顿（Stop-The-World） | 无 GC 压力 |
| **大小限制** | 受 JVM 堆大小限制（`--driver-memory`） | 可突破 JVM 堆限制，受系统物理内存限制 |
| **序列化** | JVM 对象序列化（Java/Kryo） | 堆外序列化（Tungsten Encoded） |
| **速度** | 较快（直接操作对象） | 稍慢（需 JNI 调用） |
| **开启方式** | 默认 | `spark.memory.offHeap.enabled=true` + `spark.memory.offHeap.size` |

### Tungsten 堆外内存管理

Spark 引入 **Tungsten 项目**，使用 `sun.misc.Unsafe` 直接操作堆外内存：

- **二进制格式**：数据以二进制形式存储，避免 Java 对象头开销
- **缓存友好的内存布局**：数据按列存储，提升 CPU Cache 命中率
- **序列化效率高**：Kryo/Serializer 序列化后直接存入堆外内存

### 内存分配公式

```
堆内内存 = JVM 堆大小
  ├─ Reserved Memory = 300MB（固定）
  ├─ User Memory = (JVM Heap - 300MB) × 40%
  └─ Unified Memory = (JVM Heap - 300MB) × 60%
        ├─ Storage Memory（初始 = Unified × 50%）
        └─ Execution Memory（初始 = Unified × 50%）
```
