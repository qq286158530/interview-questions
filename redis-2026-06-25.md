## 题目一：Redis 和 Memcached 有什么区别？为什么 Redis 比 Memcached 更适合做缓存？

### 参考答案

**共同点：**
- 都是基于内存的数据库，一般都用来当做缓存使用
- 都有过期策略
- 两者的性能都非常高

**核心区别：**

| 区别 | Redis | Memcached |
|------|-------|-----------|
| 数据类型 | 支持 String、Hash、List、Set、ZSet 等丰富数据类型 | 只支持最简单的 key-value 类型 |
| 持久化 | 支持 RDB、AOF、混合持久化，重启后可恢复 | 无持久化功能，数据全在内存，重启即丢失 |
| 集群 | 原生支持主从复制、哨兵、切片集群 | 无原生集群，需客户端自行实现分片 |
| 高级特性 | 支持发布订阅、Lua 脚本、事务等 | 不支持 |

**为什么 Redis 适合做缓存？**
- **高性能**：Redis 读写操作直接操作内存，QPS 可达 10W+，是 MySQL 的 10 倍
- **高并发**：直接访问 Redis 能承受的并发远大于直接访问 MySQL，可以承接大量请求而不压垮数据库
- **丰富的数据类型**：满足各种业务场景需求，如用 ZSet 做排行榜、Set 做交集计算等

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？混合持久化是怎么工作的？

### 参考答案

**Redis 有三种持久化方式：**

1. **AOF 日志（Append Only File）**
   - 每执行一条写操作命令，就将命令以追加方式写入一个文件
   - Redis 重启时读取 AOF 文件，按顺序执行命令来恢复数据
   - **写回策略**：`always`（同步写回，每次执行完写命令后立即写盘）、`everysec`（每秒写回，默认值）、`no`（由操作系统决定何时写回）

2. **RDB 快照（Redis Database）**
   - 将某一时刻的内存数据以二进制方式写入磁盘
   - 通过 `bgsave` 命令由子进程生成 RDB 文件，主线程不被阻塞
   - 触发机制（满足任一即触发 bgsave）：900秒内≥1次修改、300秒内≥10次修改、60秒内≥10000次修改

3. **混合持久化（Redis 4.0+）**
   - 结合 RDB 和 AOF 的优点
   - AOF 重写时，先以 RDB 格式写入全量数据，再将增量命令以 AOF 格式追加
   - 重启恢复时先快速加载 RDB 部分，再执行后续 AOF 增量命令

**RDB vs AOF 对比：**

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件体积 | 小（二进制） | 大（文本命令） |
| 恢复速度 | 快（直接加载） | 慢（逐条执行命令） |
| 数据完整性 | 可能丢失较多数据 | 取决于写回策略，可做到几乎不丢 |
| 快照频率 | 低（全量快照） | 高（记录每次操作） |

## 题目三：Redis 为什么是单线程的？单线程为什么还能这么快？

### 参考答案

**Redis 确实是单线程的**，指的是"接收客户端请求→解析请求→读写数据→发送响应"这个过程由主线程完成。

但 Redis 并不只是单线程，**后台还有 BIO 线程处理耗时任务**：
- `bio_close_file`：异步关闭文件
- `bio_aof_fsync`：异步 AOF 刷盘
- `bio_lazy_free`：异步释放内存（大 key 删除用 `unlink` 代替 `del`）

**单线程为什么这么快？**

1. **内存瓶颈而非 CPU**：Redis 的大部分操作都在内存中完成，采用高效数据结构（SDS、跳表、压缩列表等），瓶颈是内存和网络带宽，不是 CPU

2. **无锁竞争开销**：单线程不存在多线程切换和锁竞争，不会有死锁问题，也不必担心并发读写的线程安全问题

3. **I/O 多路复用**：使用 epoll/select 机制，一个线程处理多个 Socket 请求，内核同时监听多个连接，有请求到达时才交给 Redis 线程处理，实现高并发

**Redis 6.0 引入了多线程**：为了提升网络 I/O 的并行度，对网络读写采用了多线程处理（默认只针对写响应数据），但**命令执行仍是单线程**，性能提升至少一倍。

## 题目四：Redis 的过期删除策略和内存淘汰机制有什么区别？

### 参考答案

**过期删除策略（针对已设置 TTL 的 key）：**

| 策略 | 说明 |
|------|------|
| **定时删除** | 给每个过期 key 创建定时器，.timer 到期立刻删除。对 CPU 不友好，CPU 紧张时影响性能 |
| **惰性删除** | 访问 key 时才检查是否过期，过期则删除。对 CPU 友好，但可能堆积大量过期 key 占用内存 |
| **定期删除** | 每隔一段时间随机检查部分 key，删除其中过期的。平衡了前两种策略，Redis 默认使用 |

> Redis 使用的是**惰性删除 + 定期删除**组合策略。

**内存淘汰机制（当内存达到 maxmemory 时）：**

| 策略 | 说明 |
|------|------|
| `no-eviction` | 不淘汰，返回错误（默认） |
| `volatile-lru` | 从已设置过期时间的 key 中淘汰最近最少使用的 |
| `volatile-ttl` | 从已设置过期时间的 key 中淘汰最早过期的 |
| `volatile-random` | 从已设置过期时间的 key 中随机淘汰 |
| `volatile-lfu` | 从已设置过期时间的 key 中淘汰最少使用的 |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用的 |
| `allkeys-lfu` | 从所有 key 中淘汰最少使用的 |
| `allkeys-random` | 从所有 key 中随机淘汰 |

**实际使用建议**：
- 缓存场景推荐 `allkeys-lru`，自动保留热点数据
- 仅当存在明确冷数据时才用 `volatile-*` 策略

## 题目五：Redis 主从复制、哨兵模式和集群模式有什么区别？如何实现高可用？

### 参考答案

**主从复制（Replication）**
- 一主多从架构，主节点可读可写，从节点只读
- 主节点执行写操作后，将命令异步同步给从节点
- **优点**：读写分离提升吞吐量，数据备份
- **缺点**：主节点故障时需要手动切换，无法自动故障转移

**哨兵模式（Sentinel）**
- 在主从复制基础上增加 Sentinel 节点（通常 3 个）监控主从节点
- Sentinel 自动完成故障检测和自动 failover：当主节点不可用时，选举从节点升级为新主节点，并通知应用端更新连接地址
- 客户端通过 Sentinel 获取当前主节点地址，实现透明故障切换

**切片集群（Cluster）**
- Redis Cluster 采用 Hash Slot（16384 个槽）实现数据分片
- 每个主节点负责一部分槽，支持横向扩展
- 每个主节点可配置从节点，实现自动故障转移
- **数据分布**：通过 `CRC16(key) % 16384` 确定 key 属于哪个槽

**三种模式对比：**

| 特性 | 主从复制 | 哨兵模式 | Cluster |
|------|---------|---------|---------|
| 自动故障转移 | ❌ | ✅ | ✅ |
| 数据分片 | ❌ | ❌ | ✅ |
| 横向扩展 | ❌ | ❌ | ✅ |
| 复杂度 | 低 | 中 | 高 |
| 节点数量 | ≥2 | ≥3 | ≥6 |

**高可用方案选择**：
- 小型项目：主从复制 + 哨兵
- 大型项目：Redis Cluster，支持百万级 QPS 和 PB 级数据存储
