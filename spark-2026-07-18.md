## 题目一：Spark 任务调度流程是怎样的？

### 参考答案

Spark 任务调度分为几个阶段，整体流程如下：

**1. Application 提交**
用户通过 `spark-submit` 提交 Application，Driver 启动并初始化 `SparkContext`。

**2. 构建 DAG**
Spark 将用户代码转换为一个有向无环图（DAG），RDD 之间的依赖关系构成 DAG。

**3. DAG 划分 Stage**
DAGScheduler 根据 RDD 依赖关系划分 Stage：
- **窄依赖**（Narrow Dependency）：一个父 Partition 至多被一个子 Partition 使用（如 map、filter），可以 pipeline 执行，不需要 Shuffle，划入同一 Stage
- **宽依赖**（Wide Dependency）：一个父 Partition 被多个子 Partition 使用（如 groupByKey、join），需要 Shuffle，触发 Stage 边界划分

**4. 创建 TaskSet**
DAGScheduler 将每个 Stage 生成一批 Task（一个 Partition 对应一个 Task），封装为 TaskSet 提交给 TaskScheduler。

**5. TaskScheduler 提交 Task**
TaskScheduler 通过 Cluster Manager（YARN/Kubernetes/Standalone）将 Task 调度到 Executor 上执行。

**6. Task 在 Executor 执行**
Executor 接收到 Task 后，启动线程执行：
- Task 运行时读取数据（从 HDFS/缓存/上一个 Stage 输出）
- 执行计算逻辑
- 将结果写入 Shuffle 或最终输出

**整体流程图：**
```
Application → SparkContext → DAG → DAGScheduler → Stage划分 → TaskSet
→ TaskScheduler → Cluster Manager → Executor执行
```

## 题目二：RDD 和 DataFrame、DataSet 的区别是什么？

### 参考答案

**1. RDD（Resilient Distributed Dataset）**

- **特性**：弹性分布式数据集，是 Spark 最底层的数据抽象
- **类型安全**：强类型，支持编译时类型检查
- **操作方式**：函数式 API（map、filter、reduce 等），需要开发者自己写逻辑
- **优化**：未经过 SQL 优化器，每个 RDD 操作独立优化
- **Schema**：没有内置结构，需要手动解析
- **适用场景**：需要细粒度控制、类型安全、低层操作

**2. DataFrame**

- 类似于关系型数据库的表，有明确的 Schema（列名+列类型）
- 底层逻辑计划经过 **Catalyst 优化器** 优化
- 支持 Spark SQL 操作，API 更简洁
- Row 对象存储数据（弱类型）
- **适用场景**：结构化/半结构化数据，SQL 查询，ETL

**3. DataSet**

- 兼具 DataFrame 的优化和 RDD 的类型安全
- 是 DataFrame 的升级版，JVM 对象存储
- 支持 Encoders 进行序列化/反序列化
- **适用场景**：需要类型安全又需要 SQL 优化

**三者的关系：**

```
RDD[Person]  ←→  DataSet[Person]  ←→  DataFrame（= DataSet[Row]）
     ↑                    ↑                ↑
  强类型              强类型             弱类型
  无优化              Catalyst优化        Catalyst优化
```

**注意**：在 Spark 2.x 之后，DataFrame 和 DataSet 已统一，DataFrame 只是 DataSet[Row] 的别名。

## 题目三：Spark Shuffle 的过程是什么？有哪些优化策略？

### 参考答案

**什么是 Shuffle**

Shuffle 是 Spark 中将数据按 Key 进行重新分布的过程，俗称"洗牌"。当一个 Stage 的输出需要被下一个 Stage 使用时，必须经过 Shuffle。

**Shuffle 写磁盘过程（Shuffle Write）**

1. Map Task 计算完数据后，将结果写入 **MapOutputTracker** 记录的本地磁盘文件（bucket）
2. Spark 使用 **SortShuffleWriter**：
   - 先将数据写入一个 buffer（默认 32KB）
   - 按 partitionId 分组排序
   - 每个 partition 生成一个数据文件（`.data`）和一个索引文件（`.index`）
3. 数据文件按 partition 划分，相邻数据在同一个文件段中

**Shuffle Read 拉取过程（Shuffle Read）**

1. Reduce Task 启动，向 **MapOutputTracker** 请求 Map 端输出位置
2. 通过 **BlockStoreShuffleFetcher** 从多个 Map 端拉取属于本 partition 的数据
3. 拉取的数据先放入 buffer，进行 **merge、sort、aggregation**
4. 执行用户传入的 reduce 函数

**Spark Shuffle 演进**

| 版本 | Shuffle 实现 |
|------|-------------|
| 1.x | Hash Shuffle（每个 Mapper 为每个 Reducer 产生一个文件，文件数 = M×R） |
| 2.x+ | SortShuffle（每个 Mapper 只产生 2 个文件，文件数 = 2M） |

**SortShuffle 优化策略**

1. **Bypass SortShuffle**：当 shuffle partition 数 < `spark.shuffle.sort.bypassMergeThreshold`（默认200）且没有 aggregation 时，绕过排序，接近 HashShuffle 性能
2. ** tungsten-sort**：使用 `sun.misc.Unsafe` 直接操作堆外内存，减少 GC 开销
3. **Push-based Shuffle**（Spark 3.0+）：允许 Map 端将数据推送到 Reducer 端，减少 Reducer 端合并开销

**常见调优参数**

```scala
spark.shuffle.file.buffer   // Map 端写缓冲区，默认 32KB
spark.reducer.maxSizeInFlight  // Reducer 端拉取缓冲区，默认 48MB
spark.shuffle.sort.bypassMergeThreshold  // Bypass 阈值
spark.sql.shuffle.partitions  // Shuffle 分区数，默认 200
```

## 题目四：Spark 的内存管理机制是怎样的？

### 参考答案

Spark 有两套内存管理机制：**静态内存管理**（1.5 之前）和**统一内存管理**（1.6 之后，默认）。

**统一内存管理（Unified Memory Management）**

统一内存 = Execution 内存 + Storage 内存，二者可以动态借用。

**内存划分（以堆内内存为例）：**

```
┌──────────────────────────────────────────────┐
│                   堆内内存                      │
│  ┌────────────┬─────────────┬─────────────┐  │
│  │  Reserved  │   Unified   │   User      │  │
│  │   300MB    │   Memory    │   Memory    │  │
│  └────────────┴─────────────┴─────────────┘  │
└──────────────────────────────────────────────┘
```

**核心区域说明：**

1. **Reserved Memory**（保留内存）：300MB，系统预留
2. **Unified Memory**（统一内存）：
   - **Execution Memory**：执行内存，用于 Shuffle sort、hash join、 aggregation 等算子的临时数据
   - **Storage Memory**：存储内存，用于缓存 RDD/Broadcast/广播变量等
   - 两者可以相互借用，空闲时共享，有需求时按规则回收
3. **User Memory**（用户内存）：用户代码（数据结构、自定义变量等）

**Execution 与 Storage 动态占用规则：**

- 当 Execution 内存不足，可以 **抢占** Storage 内存（但需等待 Storage 清理缓存）
- 当 Storage 内存不足，可以 **抢占** Execution 内存（但 Execution 不会主动释放，需等待任务完成）
- 两者都有最低保障：`spark.memory.fraction`（默认 0.6）控制统一内存占总内存比例，`spark.memory.storageFraction`（默认 0.5）控制各自的最低保障

**堆外内存（Off-heap）**

- 通过 `spark.memory.offHeap.enabled` 开启
- 使用 `spark.memory.offHeap.size` 设置大小
- 减少 GC 开销，适合大内存场景

**常见内存问题排查**

- OOM：增加 `spark.executor.memory`，降低 `spark.memory.fraction`
- 数据倾斜导致 OOM：增加 Shuffle 分区数，启用广播 join
- GC 频繁：`spark.memory.fraction` 调小，或开启堆外内存

## 题目五：广播变量（Broadcast）的作用和原理是什么？如何使用？

### 参考答案

**为什么需要广播变量**

在分布式计算中，如果每个 Task 都持有一份相同的数据副本，会造成：
- 大量网络 IO（每个 Task 都要传输数据）
- 内存浪费（相同数据重复存储 N 份）

广播变量将**只读的大表/配置数据**在每个 Executor 节点只存储一份，所有在该节点运行的 Task 共享此副本，大幅减少网络传输和内存消耗。

**原理**

```
Driver 端：                     Executor 端：
broadcast.collect()            Task1 / Task2 / Task3
    ↓                               ↑共享同一个变量副本
BlockManager                      BlockManager
(以 TorrentProtocol 传输)         (本地读取)
```

1. Driver 端调用 `sc.broadcast(value)` 创建广播变量
2. 广播变量被分成多个 **block**（默认 4MB）
3. Driver 通过 **BlockManager** 将 block 推送到各 Executor 的 **BlockManager**
4. Executor 上的 Task 从本地 BlockManager 读取，不再跨节点拉取
5. 广播变量在 Executor 内共享，Task 结束后不释放（除非取消广播）

**使用示例**

```scala
// 在 Driver 端创建广播变量
val broadcastList = sc.broadcast(List("北京", "上海", "深圳"))

// 在 Task 端使用（使用 .value 获取值）
def filterCity(city: String): Boolean = {
  broadcastList.value.contains(city)
}

rdd.filter(record => filterCity(record.city)).collect()
```

**注意事项**

1. **只读**：广播后不要修改，会导致数据不一致
2. **适合小表广播**：数据过大（> 20MB）反而占用过多 executor 内存
3. **选择合适的广播格式**：使用 Kryo 序列化减少传输体积
4. **及时取消广播**：不再使用时调用 `broadcast.unpersist()` 释放本地缓存

**相关调优参数**

```scala
spark.sql.broadcastTimeout        // 广播超时，默认 5 分钟
spark.broadcast.blockSize         // 广播块大小，默认 4MB
spark.serializer                  // 推荐使用 KryoSerializer
```

**广播变量 vs 累加器（Accumulator）**

| 特性 | 广播变量 | 累加器 |
|------|---------|--------|
| 用途 | 分发只读数据 | 汇总分布式计算结果 |
| 操作方向 | Driver → Executor | Executor → Driver |
| 能否修改 | 只读 | 只写（累加） |
| 是否需要同步 | 否 | 是 |
