## 题目一：Spark 架构与核心组件

**Spark 的核心架构是怎样的？Driver、Executor、Cluster Manager 分别承担什么角色？**

### 参考答案

Spark 采用 **主从架构**，核心组件如下：

| 组件 | 角色 | 说明 |
|------|------|------|
| **Driver（驱动器）** | 主导执行 | 运行 `main()` 方法，创建 `SparkContext`，将 Job 划分为多个 Stage，构建 DAG 有向无环图，向 Executor 发送 Task |
| **Executor（执行器）** | 真正干活 | 驻留在 Worker 节点上，以线程池方式并发执行 Task，负责缓存数据、存储结果 |
| **Cluster Manager** | 资源管理 | 负责集群资源分配和调度，支持 Standalone、YARN、Kubernetes 等多种模式 |

**作业执行流程：**
1. 用户提交 Spark Application，Driver 创建 `SparkContext`
2. `SparkContext` 向 Cluster Manager 申请 Executor 资源
3. Cluster Manager 在 Worker 节点上启动 Executor 进程
4. Driver 解析代码，构建 DAG 图，切分为多个 Stage
5. Scheduler 将 Stage 包装成 TaskSet，分发到各个 Executor 执行
6. Executor 以线程池方式并行执行 Task，计算结果写入存储
7. Application 结束，回收资源

> **来源：** Spark 官方文档 Architecture 章节 · https://spark.apache.org/docs/latest/cluster-overview.html

## 题目二：RDD 原理与特性

**RDD（弹性分布式数据集）有哪些核心特性？为什么 RDD 是"只读"的？**

### 参考答案

RDD（Resilient Distributed Dataset）具有五大核心特性：

| 特性 | 含义 |
|------|------|
| **分区列表（Partitions）** | 数据集被划分为多个分区，每个分区是并行的基本单位 |
| **计算函数（Computations）** | 每个分区都有 compute 函数，定义如何计算该分区的数据 |
| **依赖关系（Dependencies）** | RDD 之间存在依赖（窄依赖/宽依赖），用于失效恢复 |
| **分区器（Partitioner）** | 可选，决定 KV 类型 RDD 的分区策略（如 HashPartitioner） |
| **位置偏好（Preferred Locations）** | 可选，提示数据分片的最优位置（实现数据本地性） |

**为什么 RDD 是只读的？**
- **简化 lineage（血统）**：只读特性使得每个 RDD 的转换操作都能被完整记录，形成 DAG，倒数第二层RDD 依赖失效RDD 的父 RDD 直接重新计算即可恢复，无需日志Checkpoint
- **保证确定性**：相同的输入经过相同转换永远得到相同输出，便于分布式重算
- **实现简洁**：避免了并发修改同一分片数据的锁开销问题

> **来源：** Matei Zaharia, et al. "Resilient Distributed Datasets: A Fault-Tolerant Abstraction for In-Memory Cluster Computing"（Spark 原始论文）· https://www.usenix.org/system/files/conference/nsdi12/nsdi12-final138.pdf

## 题目三：Shuffle 机制与优化

**Spark 中的 Shuffle 是什么？宽依赖和窄依赖有什么区别？Shuffle 有哪些常见优化策略？**

### 参考答案

**宽依赖 vs 窄依赖：**

| 类型 | 定义 | 示例 | 能否 Pipeline |
|------|------|------|--------------|
| **窄依赖（Narrow Dependency）** | 每个父分区至多被一个子分区使用 | `map`、`filter`、`union` | ✅ 可以同一 stage 内 pipeline 执行 |
| **宽依赖（Shuffle Dependency）** | 一个父分区可能被多个子分区使用 | `groupByKey`、`join`、`sortBy` | ❌ 必须等父分区全部完成，引入 Shuffle |

**Shuffle 过程：**
1. **Shuffle Write**：Shuffle 阶段（Map 阶段）将数据按分区写入本地磁盘文件，生成 index file + data file
2. **Shuffle Read**：下一阶段（Reduce 阶段）从各节点的磁盘拉取属于自己分区的数据，进行聚合/排序

**常见 Shuffle 优化策略：**

| 优化项 | 说明 |
|--------|------|
| **Shuffle 合并（Consolidation）** | 同一个 Executor 内的 Shuffle 文件合并为少量文件，减少文件数 |
| **Map 端聚合（Map-side Combine）** | 在 Map 端先做一次 combine，减少写入数据量（如 `reduceByKey`） |
| **广播变量（Broadcast Join）** | 小表广播到 Executor 内存，避免大表 Shuffle（适用于小表 join 大表） |
| **合理设置分区数** | 分区过多增加小文件，分区过少数据倾斜；建议 `核数 * 2~4` |
| **Tungsten Sort-Based Shuffle** | 使用堆外内存和 Unsafe 优化序列化，WSSort，避免 JVM GC 开销 |

> **来源：** Spark 官方文档 Shuffle Behavior · https://spark.apache.org/docs/latest/rdd-programming-guide.html#shuffle-operations

## 题目四：广播变量与内存管理

**Spark 的广播变量是什么？它解决了什么问题？Spark 的内存模型是怎样的？**

### 参考答案

**广播变量（Broadcast Variable）：**

广播变量将 **只读数据** 一次性发送到各节点 Executor 的内存中，供该节点上所有 Task 共享访问，避免每个 Task 都持有一份副本。

```python
# 广播变量使用示例
broadcast_var = sc.broadcast({"banned_words": ["xxx", "yyy"]})

# 在 Executor 中所有 Task 共享这一份数据
rdd.map(lambda x: x in broadcast_var.value)
```

**解决的问题：**
- 节省网络传输：相同数据只传输一次（而非每个 Task 传输一次）
- 节省内存：各 Task 共享同一份数据，而非各自复制

**Spark 内存模型（JVM Heap 内存）：**

Spark 堆内内存分为两大区域：

| 区域 | 占比（默认） | 用途 |
|------|-------------|------|
| **Execution（执行内存）** | 约 60% | Shuffle 排序、Hash 聚合、RDD 缓存等计算相关 |
| **Storage（存储内存）** | 约 40% | RDD 缓存、广播变量、Task 中间结果等 |

另外还有 **堆外内存（Off-Heap）**，通过 `spark.memory.offHeap.enabled=true` 开启，使用 JDK DirectByteBuffer 分配，不受 JVM GC 管理，适合大内存场景。

**内存淘汰策略：** 当 Execution 空间不足时会溢出到磁盘（ExternalAppendOnlyMap/Spill）；Storage 空间不足时会通过 `LRU` 淘汰旧缓存。

> **来源：** Spark 官方文档 Memory Management · https://spark.apache.org/docs/latest/tuning.html#memory-management-overview

## 题目五：DataFrame 与 Dataset 区别及优化

**DataFrame 和 Dataset 有什么区别？Spark SQL 的查询优化流程是怎样的？**

### 参考答案

**DataFrame vs Dataset：**

| 对比项 | DataFrame | Dataset |
|--------|-----------|---------|
| **语言** | 纯 Spark 内部优化（逻辑计划与 Catalyst 解耦） | 支持强类型 Scala/Java，需编译时检查 |
| **API** | 无需泛型，DSL 类似 SQL | 支持泛型，LINQ-style 链式调用 |
| **序列化** | 使用 Tungsten 堆外二进制编码，无对象开销 | 编码为 JVM 对象（Serializable） |
| **编译时检查** | ❌ 无（运行时才发现错误） | ✅ 有（编译时检查类型安全） |
| **向下兼容** | Dataset[Row] = DataFrame | Dataset[T] 含泛型 T |

**本质：** DataFrame 是 **Dataset[Row]** 的别名，二者在 Spark 内部共享同一套存储和执行引擎。

**Spark SQL 优化流程（Catalyst 优化器）：**

```
SQL / DSL
    ↓
Unresolved Logical Plan（未解析逻辑计划）
    ↓ (Analyzer + Catalog)
Analyzed Logical Plan（解析后逻辑计划）
    ↓ (Optimizer Rules: 谓词下推/列裁剪/常量折叠/...)
Optimized Logical Plan（优化后逻辑计划）
    ↓ (Spark Planner)
Physical Plan（物理计划）
    ↓ (Cost Model 选择最优)
Executed Physical Plan → RDD
```

**常见优化规则：**

| 优化规则 | 作用 |
|---------|------|
| **谓词下推（Predicate Pushdown）** | 将 Filter 下推到数据源层，提前过滤减少读取量 |
| **列裁剪（Column Pruning）** | 只读取 SELECT 的列，减少 I/O |
| **常量折叠（Constant Folding）** | 编译期计算常量表达式，无需运行时求值 |
| **短路查询（Short Circuit）** | LIMIT 1 时找到一条即停止扫描 |
| **广播 Join（Broadcast Hash Join）** | 小表放入内存，避免 Shuffle |

> **来源：** Spark 官方文档 Spark SQL, DataFrames and Datasets Guide · https://spark.apache.org/docs/latest/sql-programming-guide.html
