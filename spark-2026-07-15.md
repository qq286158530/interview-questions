## 题目 1：Spark Shuffle 机制 — Shuffle 为什么要排序？Hash Shuffle 有哪些问题？

**问题：** 请详细描述 Spark Shuffle 的执行过程，为什么 Shuffle 操作需要排序？Hash Shuffle 存在哪些性能问题？如何优化？

**答案：**

### Shuffle 执行过程

Spark Shuffle 是 **洗牌** 操作，用于将数据按照 Key 进行重新分区的过程，发生在 **Stage 与 Stage 之间的边界处**。以 `groupByKey` 为典型代表：

1. **Map 阶段**：每个 Task 将数据写入**本地磁盘文件**，并按照 Key 计算目标分区 ID（Hash 或 Range）
2. **Map 输出合并**：同一 Executor 上的多个 Map Task 输出合并为少量文件（减少文件数）
3. **Fetch 阶段**：每个 Reduce Task 通过网络从各个 Map Task 所在节点**拉取**属于自己的数据分片
4. **Reduce 阶段**：数据到达后进行 Sort/Merge，输出最终结果

### Shuffle 排序的必要性

- **Hash Shuffle（不排序）**：数据无序，相同 Key 的数据可能分散在不同文件块，Reduce 端需要多次磁盘 IO 才能聚合同一 Key 的所有数据
- **Sort-Based Shuffle（排序）**：数据按 Key 有序排列，相同 Key 的数据连续存储，Reduce 端只需顺序读取即可高效聚合
- 排序还能配合 **Tungsten 排序** 实现直接操作二进制数据，跳过 JVM 对象序列化开销

### Hash Shuffle 的问题

| 问题 | 说明 |
|------|------|
| **文件过多** | 每个 Map Task 为每个 Reduce Task 产生一个文件，M × R 个文件，容易撑爆文件系统 |
| **小文件 IO 开销** | 大量小文件导致随机磁盘 IO，远慢于顺序读写 |
| **内存压力** | Mapper 端需要缓存大量文件描述符，内存消耗大 |

### 优化策略

1. **开启 Map Output 合并**（`spark.shuffle.consolidateFiles=true`）：同一 Executor 内多个 Map Task 共用输出文件，文件数从 M×R 降为 **Executor数 × R**
2. **使用 Sort-Based Shuffle**（Spark 1.2+ 默认）：`spark.shuffle.manager=sort`，文件数降至 2M（数据文件+索引文件）
3. **调整缓冲区**：`spark.shuffle.file.buffer`（默认 32KB）调大可减少磁盘写入次数
4. **合理设置分区数**：避免过多小分区导致大量 Shuffle 文件

- Apache Spark 官方文档 - Shuffle Operations：https://spark.apache.org/docs/latest/rdd-programming-guide.html#shuffle-operations
- 美团技术团队《Spark 性能优化指南》：https://tech.meituan.com/sparkshuffleoptimization

## 题目 2：RDD 血缘机制与容错 — RDD 如何实现 lineage 恢复？

**问题：** RDD 的 lineage（血缘关系）是什么？为什么说 RDD 是"懒执行"的？当某个分区数据丢失时，Spark 如何通过 lineage 恢复数据？

**答案：**

### 什么是 RDD Lineage

RDD 的 lineage（血统/血缘）是一个 RDD 演变历史的 DAG（有向无图），记录了**当前 RDD 是如何从源头 RDD 经过哪些 Transformation 转换而来**。

```scala
val lines = sc.textFile("hdfs://...")           // RDD[String]
val words = lines.flatMap(_.split(" "))          // RDD[String]
val pairs = words.map((_, 1))                   // RDD[(String, Int)]
val counts = pairs.reduceByKey(_ + _)           // RDD[(String, Int)]
```

上述 DAG 为：`textFile → flatMap → map → reduceByKey`

### 懒执行（Lazy Evaluation）

Spark 的 Transformation（转换操作）都是**懒执行（Lazy）**：
- 调用 `map`、`filter`、`flatMap` 等 API 时，**不会立即执行**，只是记录血缘关系
- 只有遇到 **Action（动作操作）** 时，如 `collect`、`count`、`saveAsTextFile`，才会触发完整的 Job 执行
- 懒执行的好处：** Pipeline 优化**（将多个 Transformation 合并成一个 Stage，减少数据流动）

### 容错机制 — Lineage 恢复

Spark **不存储数据副本**，而是利用 lineage 实现容错：

1. **窄依赖（Narrow Dependency）**：子 RDD 每个分区只依赖父 RDD 固定数量的分区（如 `map`、`filter`）。一个分区丢失，只需重算对应的父分区，代价 O(N)
2. **宽依赖（Wide Dependency）**：子 RDD 每个分区依赖父 RDD 所有分区（如 `groupByKey`、`join`）。一个分区丢失，需要重算父 RDD 的所有分区，代价 O(N²)

**恢复过程：**
- 当 Task 失败，DAGScheduler 根据 lineage 找到该分区的父依赖
- 从源头 RDD 开始**重新计算**丢失的分区（不需要全量重算）
- 对于宽依赖场景，Spark 会**缓存中间计算结果**（checkpoint）以减少恢复成本

**为什么不用 Checkpoint？**  
Lineage 如果过长（比如多次宽依赖链），重算代价会非常大，此时可以用 `rdd.checkpoint()` 将 RDD 物化到磁盘，切断 lineage。

**来源：**  
- Apache Spark 官方文档 - RDD Programming Guide：https://spark.apache.org/docs/latest/rdd-programming-guide.html
- 《Learning Spark, 2nd Edition》- Holden Karau 等

## 题目 3：DataFrame vs Dataset vs RDD — 三者的区别与适用场景

**问题：** 请比较 Spark 中 DataFrame、Dataset 和 RDD 的异同，什么时候该用 DataFrame？什么时候该用 RDD？

**答案：**

### 三者对比

| 特性 | RDD | DataFrame | Dataset |
|------|-----|-----------|---------|
| **数据结构** | 无schema，纯分布式集合 | 有 schema（列名+类型） | 有 schema，类型安全 |
| **语言支持** | Scala/Java/Python/R | Scala/Java/Python/R | Scala/Java |
| **编译时类型检查** | ✅ 强类型 | ❌ 运行时检查 | ✅ 强类型（静态语言） |
| **查询优化** | 无（Catalyst 不可用） | ✅ Tungsten + Catalyst | ✅ Tungsten + Catalyst |
| **序列化** | Java/Kryo（手写） | **OFF-HEAP Tungsten 二进制** | 同 DataFrame |
| **GC 压力** | 高（JVM 对象） | 低（堆外内存） | 低 |
| **创建方式** | `parallelize`、`textFile` | `read.parquet`、`toDF` | `toDS` |

### 内部原理

**DataFrame 优化（Catalyst Optimizer）：**
```
User Code → Logical Plan → Logical Optimization → Physical Plan → Codegen → Executed Code
```
- **谓词下推（Predicate Pushdown）**：将 Filter 下推到数据源，减少读取数据量
- **列剪裁（Column Pruning）**：只读取需要的列，减少 IO
- **常量替换**：在编译期完成常量表达式计算

### 何时用 DataFrame / Dataset

- **ETL、数据分析、BI**：用 DataFrame，API 简洁，优化器自动优化执行计划
- **需要强类型 + 优化**：用 Dataset（typed DataFrame），既有类型安全又有优化
- **非结构化数据处理**：用 RDD（无法用 DataFrame 表示时）

### 代码示例

```scala
// RDD — 需要手动序列化
case class Person(name: String, age: Int)
val people = sc.parallelize(Seq(Person("Alice", 30)))
people.map(p => (p.name, p.age)).reduceByKey(_ + _)

// DataFrame — 自动优化
val df = spark.read.parquet("people.parquet")
df.filter($"age" > 25).groupBy($"name").count().show()

// Dataset — 类型安全 + 优化
import spark.implicits._
val ds = df.as[Person]
ds.filter(_.age > 25).groupBy(_.name).count().show()
```

### 总结

> **DataFrame = Dataset[Row]**（Row 是一种通用无类型容器）  
> **DataFrame 和 Dataset 的性能远超 RDD**，因为 Tungsten 二进制存储 + Catalyst 优化  
> **优先使用 DataFrame/Dataset**，除非需要操作非结构化数据或进行低-level 控制

**来源：**  
- Apache Spark 官方文档 - DataFrames, Datasets, and RDDs：https://spark.apache.org/docs/latest/sql-programming-guide.html
- Databricks 博客《Deep Dive into Catalyst Optimizer》：https://databricks.com/blog/2015/04/13/deep-dive-into-spark-sqls-catalyst-optimizer.html

## 题目 4：Spark 广播变量 — 什么是广播？广播变量如何提升 Shuffle 性能？

**问题：** 什么是广播变量？Spark 为什么默认不支持跨 Task 共享变量？在什么场景下广播变量能显著提升性能？

**答案：**

### Spark 默认变量行为

默认情况下，当 Spark 在集群上并行执行函数时，会将**函数中用到的每个变量**复制一份到**每个 Task**（发送到每个节点）。这带来两个问题：

1. **传输成本高**：大对象被反复传输，浪费网络带宽
2. **存储成本高**：每个 Task 都持有一份副本，内存浪费

```scala
// 每个 Task 都有一份 broadcastVar 的副本 —— 效率低下！
val broadcastVar = sc.parallelize(Seq(1, 2, 3))
data.map(x => x * broadcastVar.value.sum)  // broadcastVar 被复制 N 份
```

### 广播变量的原理

**广播变量**（Broadcast Variables）允许将一个**只读变量**缓存到**每个节点的内存**中，而非每个 Task：

1. Driver 端：将变量序列化（Kryo）后分发到各个 Worker 节点
2. Executor 端：变量存储在 **Executor JVM 的 Heap 内存**（或堆外），所有 Task 共享同一份引用
3. Task 访问时：**本地读取**，无需网络传输

```
未广播: Task1 ──copy──→ [Var]   Task2 ──copy──→ [Var]   (N 份拷贝)
已广播: Executor ──[Var]── ← Task1, Task2, Task3        (1 份本地读取)
```

### 性能提升场景

| 场景 | 提升效果 |
|------|---------|
| **小表 JOIN 大表**（维度表 join） | 避免大表 Shuffle，只 Shuffle 小表 |
| **配置表 / 字典查询** | 每条记录都在本地查表，无网络开销 |
| **机器学习特征向量** | 特征向量只传输一次，所有迭代复用 |
| **UDF 依赖的静态数据** | 避免每个 Task 重新加载 |

### 代码示例

```scala
// 广播 lookup 表
val lookupTable = Map("A" -> "Apple", "B" -> "Banana", "C" -> "Cherry")
val broadcastLookup = sc.broadcast(lookupTable)

// Task 读取广播变量（本地访问）
val result = data.map(row => {
  val key = row.getString(0)
  broadcastLookup.value.getOrElse(key, "Unknown")
})
```

### 注意事项

1. **只适合小表/小数据集**：广播变量存在 Executor 堆内，数据过大会导致 OOM
2. **`spark.sql.autoBroadcastJoinThreshold`**：默认 10MB，Spark 自动对小表做广播 JOIN
3. **广播噎噎**：大表广播导致接收节点 OOM，需通过 `spark.sql.broadcastTimeout` 调整超时

- Apache Spark 官方文档 - Broadcast Variables：https://spark.apache.org/docs/latest/rdd-programming-guide.html#broadcast-variables
- Databricks 博客《Apache Spark 广播变量最佳实践》：https://databricks.com/blog/2016/05/27/apache-spark-brings-hybrid-shuffle.html

## 题目 5：Spark 内存管理 — 堆内内存 vs 堆外内存，Storage 内存与 Execution 内存如何划分？

**问题：** Spark Executor 的内存是如何划分的？堆内（On-Heap）和堆外（Off-Heap）内存有什么区别？当 Storage 内存和 Execution 内存发生竞争时会发生什么？

**答案：**

### Executor 内存结构（Spark 2.x+ Unified Memory）

Spark Executor 的 JVM Heap 被统一内存管理器（Unified Memory Manager）划分为以下区域：

```
┌─────────────────────────────────────────────────────┐
│                   Executor Memory                   │
│  ┌──────────────────┬──────────────────────────────┐ │
│  │  Memory (60%)   │         Disk (40%)          │ │
│  │ ┌─────────────┐ │ ┌──────────────────────────┐ │ │
│  │ │  Storage    │←→│  Execution (shuffle etc) │ │ │
│  │ │  Memory     │ │ └──────────────────────────┘ │ │
│  │ │  (可借用)   │ │ ┌──────────────────────────┐ │ │
│  │ └─────────────┘ │ │     Other (系统预留)     │ │ │
│  │                 │ └──────────────────────────┘ │ │
│  └──────────────────┴──────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

| 区域 | 用途 | 配置参数 |
|------|------|---------|
| **Storage Memory** | RDD 缓存（`persist`、`cache`）、广播变量、Unroll | `spark.memory.fraction`(默认0.6) × 堆内存 |
| **Execution Memory** | Shuffle 排序/聚合的缓冲区、Hash 聚合表、代码缓存 | 同上 |
| **User Memory** | 用户数据结构、Scala 对象元数据 | `(1 - spark.memory.fraction)` × 堆内存 ≈ 40% |
| **Reserved Memory** | 系统预留（300MB） | 固定 |

### 堆内 vs 堆外内存

| 特性 | On-Heap（堆内） | Off-Heap（堆外） |
|------|----------------|-----------------|
| 存储位置 | JVM Heap | `sun.misc.unsafe` 直接分配的堆外内存 |
| GC 影响 | 受 GC 管理，有停顿 | 不受 GC 管理，无停顿 |
| 序列化 | Java 对象（Kryo 可选） | Tungsten 二进制（完全堆外） |
| 配置 | `spark.executor.memory` | `spark.memory.offHeap.enabled=true` + `spark.memory.offHeap.size` |
| 适用场景 | 默认场景 | 超大数据集、避免 GC 开销的高性能场景 |

### Storage vs Execution 内存竞争

Spark 采用**统一内存管理**（Unified Memory Management，Spark 1.6+）：

- **Storage 和 Execution 内存可以相互借用**（`spark.memory.storageFraction`，默认 0.5）
- Storage 空闲时，可借给 Execution；Execution 空闲时，可借给 Storage
- **但 Storage 持有者被 Execution 驱逐时会落盘**（`blocks stored on disk`）
- Execution 占用时，Storage 只能持有无法驱逐；反之 Storage 持有时，Execution 只能等待释放

### 常见 OOM 原因与排查

1. **大量 Shuffle 数据无法落盘** → Execution 内存不足：调高 `spark.executor.memory` 或 `spark.shuffle.file.buffer`
2. **缓存 RDD 后计算内存不够** → Storage/Execution 比例失衡：调低 `spark.memory.fraction`
3. **广播变量过大** → 单机 OOM：确保广播变量 < `spark.sql.autoBroadcastJoinThreshold`，或改用 Shuffle Hash Join

```scala
// 设置堆外内存（适合超大数据集场景）
val conf = new SparkConf()
  .set("spark.memory.offHeap.enabled", "true")
  .set("spark.memory.offHeap.size", "8g")
```
