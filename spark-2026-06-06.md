## 题目一：Spark 架构核心组件及其作用

### 问题
Spark 的整体架构是怎样的？请描述核心组件及其职责。

### 答案

Spark 采用 **主从架构**，核心组件如下：

| 组件 | 角色 | 职责 |
|------|------|------|
| **Driver** | 主节点 | 负责执行用户代码，构建 `DAG`，将 `DAG` 拆分成 `Stage`（任务集），向 `Executor` 分配任务 |
| **Executor** | 从节点 | 启动时向 Driver 注册，以 **线程池** 方式执行 Task，一个 Executor 可并行运行多个 Task |
| **Cluster Manager** | 集群管理器 | 负责集群资源管理（YARN / Mesos / Standalone），在 Spark On YARN 模式下分为 `Cluster`模式和 `Client` 模式 |
| **SparkContext** | 编程入口 | Driver 的核心对象，用户通过它创建 RDD、提交 Job、预约算子 |

**工作流程：**
1. 用户提交 JAR/Python 脚本 → Driver 端创建 `SparkContext`
2. SparkContext 根据代码构建 **DAG（有向无环图）**
3. DAG 被调度器拆分为多个 **Stage**，Stage包含一组可以并行执行的 **Task**
4. Task 通过 **TaskScheduler** 分发到各 Executor 的线程池中执行
5. 结果汇总回 Driver

> https://spark.apache.org/docs/latest/architecture.html

## 题目二：RDD 的五大属性与 lineage（血统）

### 问题
RDD（弹性分布式数据集）的五大属性是什么？请解释 lineage 的作用与容错机制。

### 答案

**RDD 的五大属性：**

1. **`getDependencies()`** — 依赖关系：RDD 记录其父 RDD 的依赖（宽依赖或窄依赖），用于构建 DAG
2. **`compute()`** — 计算函数：RDD 记录将父分区转换为分区的 `Iterator[T]` 计算逻辑
3. **`partitions()`** — 分区列表：RDD 的数据被划分为多个分区，每个分区可在不同节点并行处理
4. **`partitioner()`** — 分区器（可选）：决定 RDD 数据的分区策略（如 `HashPartitioner`、`RangePartitioner`）
5. **`preferredLocations()`** — 优先位置（可选）：记录每个分区的最优计算位置，利于数据局部性优化

**Lineage（血统）机制：**

RDD 通过 **lazy模式** 记录数据的**转换 lineage（血缘关系）**，而非真实存储数据。每个 RDD 都知道自己是如何从源头 RDD 一步步通过算子转换来的。

**容错原理：**
- 当某个分区数据丢失时，Spark 根据 lineage重新计算该分区
- 从源头 RDD 开始，逐一应用沿途的 `compute()` 函数即可恢复，无需全量复制数据
- **窄依赖** 支持在原地重新计算（管道式），效率高；**宽依赖** 需要等父 RDD 所有分区就绪

> **来源参考：** Matei Zaharia, et al. — *Resilient Distributed Datasets: A Fault-Tolerant Abstraction for In-Memory Cluster Computing* (Spark 原始论文)  
> https://www.usenix.org/system/files/conference/nsdi12/nsdi12-final138.pdf

## 题目三：Shuffle 机制与优化策略

### 问题
什么是 Shuffle？为什么 Shuffle 是 Spark 性能瓶颈？有哪些优化方法？

### 答案

**什么是 Shuffle：**
Shuffle 是 **将数据按照某种规则重新分组** 的过程。当 RDD 依赖链中遇到 `groupByKey`、`reduceByKey`、`join`、`cogroup` 等需要**跨节点聚合**的算子时，数据必须在不同节点之间进行传输和重新分区。

典型流程：`Map Task (写本地缓冲)` → `溢写磁盘` → `Fetch 远程分区` → `Reduce Task 聚合`

**为什么是性能瓶颈：**
1. **磁盘 I/O**：大量数据先写磁盘再读磁盘，涉及序列化和反序列化
2. **网络 I/O**：跨节点数据传输消耗带宽
3. **内存压力**：Map 端缓冲区、Shuffle 聚合内存均占用 JVM Heap
4. **磁盘小文件**：产生大量中间文件（HDFS 不友好）

**优化策略：**

| 优化手段 | 说明 |
|---------|------|
| **Map端预聚合** | 用 `reduceByKey` 替代 `groupByKey`，在 Map 端先做局部合并再 Shuffle，减少数据量 |
| **合理设置分区数** | 分区过多导致小任务多、开销大；分区过少导致并行度不够；推荐 CPU核数的 2-4 倍 |
| **使用 Kryo 序列化** | Kryo 比 Java 序列化体积小、速度快，配置 `spark.serializer=org.apache.spark.serializer.KryoSerializer` |
| **开启 Shuffle 合并** | `spark.shuffle.consolidateFiles=true`，多个 Reduce 任务合并同一 executor 内的中间文件，减少小文件数 |
| **广播大表** | 小表 `broadcast` 广播到每个节点，替代 Shuffle Join，避免大表跨节点传输 |
| **存储级别优化** | 对需要重复使用的 RDD 开启 `persist(MEMORY_AND_DISK)`，避免重复计算 |

> **来源参考：** Spark 官方文档 — *Shuffle Operations*  
> https://spark.apache.org/docs/latest/rdd-programming-guide.html#shuffle-operations

## 题目四：Spark 内存管理模型（ Unified Memory ）

### 问题
Spark 2.x 后的统一内存管理模型是怎样的？请说明各内存区域的作用与配置。

### 答案

Spark 2.x 引入 **Unified Memory（统一内存）** 管理，将堆内存分为两大区域：**存储内存（Storage Memory）** 和 **执行内存（Execution Memory）**，二者可以动态共享。

**内存区域划分：**

```
┌──────────────────────────────────────────────────────┐
│                    堆内存 (Heap)                        │
│  ┌──────────────┬──────────────────┬────────────────┐  │
│  │  预留区 │     统一内存区      │    用户内存 │  │
│  │ (300MB)      │  Storage / Exec │  (RDD metadata)│  │
│  └──────────────┴──────────────────┴────────────────┘  │
│                      │ │
│         ┌────────────┴────────────┐                   │
│         ▼ ▼                    │
│  ┌──────────────┐         ┌──────────────┐             │
│  │ Storage Memory│         │Execution Memory│          │
│  │ (存储广播变量 │         │(Shuffle 排序 │          │
│  │  cached RDD) │         │  join 聚合等) │           │
│  └──────────────┘         └──────────────┘ │
└──────────────────────────────────────────────────────┘
```

**各区域作用：**

| 区域 | 用途 | 配置参数 |
|------|------|---------|
| **预留内存 (Reserved Memory)** | 300MB，系统预留 | 固定，不可配置 |
| **执行内存 (Execution Memory)** | Shuffle 排序、Hash Join聚合、Map 端缓存等 | `spark.memory.fraction ≈ 0.6`，`spark.memory.storageFraction ≈ 0.5` |
| **存储内存 (Storage Memory)** | 广播变量、RDD Cache（persist）、内部元数据 | 与 Execution 共享，动态伸缩 |
| **用户内存 (User Memory)** | 用户代码（RDD 转换时的元数据、用户自定义数据结构） | `1 - spark.memory.fraction ≈ 0.4` |

**动态协商规则：**
- 一方内存不足且另一方空闲时，可**借用**对方内存
- 一旦 Storage 被 Execution 借走，Storage 需要清理已有缓存才能回收（懒清理）
- `spark.memory.storageFraction`决定 Storage 的**最小保留量**

**常见 OOM 原因：**
- `spark.memory.fraction` 设置过高 → 用户内存不足
- 广播大表导致 Storage 爆满 → 及时 `unpersist`
- 单一 Executor 上的分区数据量过大 → 调大 `spark.executor.memory` 或增加分区数

> **来源参考：** Spark 官方文档 — *Memory Management*  
> https://spark.apache.org/docs/latest/tuning.html#memory-management

## 题目五：广播变量（Broadcast Variable）的原理与使用场景

### 问题
广播变量是什么？它的工作原理是怎样的？使用广播变量有什么注意事项？

### 答案

**什么是广播变量：**
广播变量是 Spark 提供的**只读共享变量**，将一份小数据集**高效复制到每个节点**的 Executor 内存中，所有 Task 共享访问，避免每个 Task 各自持有一份副本造成重复内存消耗。

**工作原理：**

```
Driver 端
  ↓广播变量（序列化为 Torrent）
  ↓ 广播块（Block）通过 BitTorrent 协议分发
Executor 节点
  ↓ 接收并缓存到 Executor 内存（Storage Memory）
  ↓ 所有 Task 共享读取
```

- Driver 将广播变量按 `broadcastMode`（默认 `Serializable`) 序列化后切分为多个 Block
- 采用 **P2P BitTorrent 分发**：已收到 Block 的 Executor 同时作为种子向其他节点分发
- 存储在 **Storage Memory**，使用 `BroadcastManager` 管理
- Task 通过 `broadcast.value` 访问（内部调用 `BroadcastManager` 读取）

**使用场景：**

| 场景 | 说明 |
|------|------|
| **大表 join 小表** | 将小表广播，避免 Shuffle 大表（`df1.join(broadcast(df2))`）|
| **跨 Stage 共享配置** | 维度表、字典表、配置参数等只读数据 |
| **广播大文件路径** | 将 HDFS 文件列表广播给所有节点 |

**注意事项：**

1. **广播变量不宜过大**：默认广播块大小 4MB，超出则报错；超过 `spark.broadcast.blockSize` 需调大
2. **只读不可修改**：广播变量是只读的，在 Task 中修改不会同步到 Driver
3. **及时释放**：用完调用 `broadcast.unpersist()`释放 Storage 内存，否则一直占用直到 Executor 销毁
4. **序列化方式**：默认使用 Kryo 序列化，比 Java Serializable 更紧凑，可通过 `spark.sql.broadcastTimeout` 控制超时
5. **广播阈值**：Spark SQL 中 `autoBroadcastJoinThreshold`（默认 10MB）自动对小表启用广播

```scala
// Scala 示例
import org.apache.spark.sql.functions.broadcast

// 方式一：broadcast 函数
val result = bigTable.join(broadcast(smallTable), "key")

// 方式二：SQL
spark.sql("SELECT /*+ BROADCAST(s) */ * FROM bigTable t JOIN smallTable s ON t.key = s.key")
```

> https://spark.apache.org/docs/latest/rdd-programming-guide.html#broadcasting-variables
