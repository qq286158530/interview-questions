## 题目一：Flink 的架构是怎样的？为什么说它是真正的大数据流处理引擎？

### 参考答案

Flink 采用**流优先（Streaming-first）**架构，核心是**流处理引擎**，批处理只是流处理的特例。

**整体架构分为三层：**

| 层级 | 说明 |
|------|------|
| **API 层** | DataStream API（流）、DataSet API（批），现已统一为 DataStream API |
| **运行时（Runtime）层** | 真正的流处理引擎，基于 **Flink Runtime** 执行 DAG |
| **物理层** | 支持 Standalone、YARN、Kubernetes 等多种部署模式 |

**核心组件：**

1. **JobManager** — 协调整个作业：调度 Task、协调 Checkpoint、管理算子状态
2. **TaskManager** — 实际执行 Task 的工作节点，每个 TM 管理多个 **TaskSlot**
3. **Slot** — 资源分配单位，决定一个 TM 能并行执行多少任务

**为什么说 Flink 是真正的流处理引擎？**
- **事件时间处理**：支持 WaterMark 机制，处理乱序事件
- **精确一次（Exactly-Once）**：通过 Checkpoint + 两阶段提交保证
- **低延迟**：基于流式计算，无需等待数据到齐
- **背压机制**：TaskManager 之间自动背压，无需额外组件

## 题目二：什么是 Watermark？它是如何解决乱序和迟到数据的？

### 参考答案

**Watermark（水位线）** 是 Flink 用于处理**乱序事件**和**迟到事件**的机制，本质是一个**时间戳**，表示"所有时间戳 ≤ Watermark 的数据已经全部到达"。

**核心思想：**

```
Watermark(t) = max(当前已到达事件时间戳) - 最大延迟时间
```

当 Watermark 超过窗口结束时间时，窗口触发计算。

**两种生成策略：**

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **有序流** | `WatermarkStrategy.forMonotonousTimestamps()`，时间戳单调递增 | 数据有序、无乱序 |
| **乱序流** | `WatermarkStrategy.forBoundedOutOfOrderness(Duration)`，设置最大延迟 | 数据有乱序，需等待 |

**示例代码：**

```java
WatermarkStrategy<Event> wmStrategy = WatermarkStrategy
    .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner((event, timestamp) -> event.getTimestamp());

DataStream<Event> stream = env
    .fromSource(source, wmStrategy, "Kafka Source");
```

**迟到数据处理策略：**

- **窗口允许迟到**：`window().allowedLateness(Duration.ofMinutes(5))`
- **侧输出迟到数据**：`sideOutputLateData(OutputTag)`
- **默认丢弃**：超过 Watermark 的数据直接丢弃（不再触发窗口）

**乱序处理流程示例：**

```
事件时间线: 1s → 3s → 2s(乱序) → 5s → 4s(乱序) → Watermark=4s → 触发窗口
```

## 题目三：Flink 的 Checkpoint 和 Savepoint 有什么区别？如何保证 Exactly-Once？

### 参考答案

**Checkpoint** 和 **Savepoint** 都是 Flink 的状态持久化机制，但用途不同：

| 维度 | Checkpoint | Savepoint |
|------|-----------|-----------|
| **触发方式** | Flink 自动周期性触发 | 手动触发（用户命令） |
| **用途** | 故障恢复，保证容错 | 计划性停止/升级/迁移 |
| **清理策略** | Job 重启后自动清理旧 Checkpoint | 需手动删除 |
| **存储位置** | 由 Flink 自动管理 | 用户指定目录 |
| **依赖组件** | 需开启 Checkpointing | 需配置 Savepoint 目录 |

**Exactly-Once 语义实现原理：**

Flink 通过 **Checkpoint + 两阶段提交（Two-Phase Commit, 2PC）** 实现端到端精确一次：

```
1. CheckpointCoordinator 周期性向所有 Source 发送 Barrier
2. Source 插入 Barrier 并做快照（Checkpoint Snapshots）
3. Barrier 随数据流向下游传递，各算子完成快照
4. 所有算子快照完成后，向 CheckpointCoordinator 确认
5. 两阶段提交：
   - Prepare 阶段：预提交事务
   - Commit 阶段：确认提交事务
```

**关键配置：**

```java
env.enableCheckpointing(10000);                // 每 10s 做一次 Checkpoint
env.getCheckpointConfig().setCheckpointPolicy(CheckpointingMode.EXACTLY_ONCE);
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(5000); // 间隔 5s
env.getCheckpointConfig().setTolerableCheckpointFailureNumber(3);
```

**端到端 Exactly-Only 需要：**
- Source 支持回溯（如 Kafka 消费者_OFFSET 提交）
- 算子状态做 Checkpoint
- Sink 支持事务（如 Kafka Producer 的事务、JDBC 事务）

## 题目四：Flink 的窗口函数有哪几种？滚动窗口、滑动窗口、会话窗口的区别是什么？

### 参考答案

Flink 支持两大类窗口：**基于时间**和**基于计数**，以及三种窗口类型：

### 窗口类型对比

| 窗口类型 | 英文 | 特点 | 示例 |
|---------|------|------|------|
| **滚动窗口** | Tumbling Window | 窗口长度固定，不重叠 | 每 5 分钟统计一次 |
| **滑动窗口** | Sliding Window | 窗口长度固定，可重叠 | 每 5 分钟统计最近 10 分钟 |
| **会话窗口** | Session Window | 动态窗口，间隙触发 | 按用户活跃会话统计 |

### 1. 滚动窗口（Tumbling Window）

```java
// 每 5 分钟一个窗口，不重叠
DataStream<Tuple2<String, Integer>> stream = ...
    .keyBy(0)
    .window(TumblingProcessingTimeWindows.of(Time.minutes(5)))
    .sum(1);
```

### 2. 滑动窗口（Sliding Window）

```java
// 窗口长度 10 分钟，滑动步长 5 分钟
.keyBy(0)
.window(SlidingProcessingTimeWindows.of(Time.minutes(10), Time.minutes(5)))
.sum(1);
```

### 3. 会话窗口（Session Window）

```java
// 间隙超过 30 秒则开启新窗口
.keyBy(0)
.window(ProcessingTimeSessionWindows.withGap(Time.seconds(30)))
.sum(1);
```

### 全窗口函数 vs 增量聚合函数

| 类型 | 算子 | 说明 |
|------|------|------|
| **增量聚合** | `reduce()` / `fold()` / `sum()` | 每条数据触发，更新简单状态，内存友好 |
| **全窗口** | `apply()` / `process()` | 等窗口关闭后一次性处理，可访问窗口所有数据 |

```java
// 全窗口示例：计算窗口内 Top 3
.windowAll(TumblingProcessingTimeWindows.of(Time.minutes(5)))
.process(new ProcessAllWindowFunction<Tuple2<String, Integer>, String, TimeWindow>() {
    @Override
    public void process(Context ctx, Iterable<Tuple2<String, Integer>> values,
                        Collector<String> out) {
        // 一次性拿到窗口内所有数据排序
    }
});
```

## 题目五：Flink 的状态管理机制是怎样的？Keyed State 和 Operator State 有何区别？

### 参考答案

Flink 是有状态的流处理引擎，状态分为两类：

### 状态类型对比

| 维度 | Keyed State | Operator State |
|------|------------|----------------|
| **作用域** | 按 Key 分组，每个 Key 独立状态 | 作用于算子实例，所有 Key 共享 |
| **存储结构** | `Map`、`Value`、`List`、`Aggregating`、`Reducing` | `ListState`、`UnionListState`、`BroadcastState` |
| **使用方式** | 必须在 `keyBy()` 后使用 | 可在任何算子使用 |
| **重分布策略** | 按 Key 自动路由到对应分区 | `LIST_BROADCAST`（广播）、`UNION`（联合）、`OFFSET`（按索引） |

### Keyed State 示例

```java
DataStream<String> stream = ...
    .keyBy(r -> r.getKey())          // 先 keyBy
    .map(new StatefulMapper());

public class StatefulMapper extends RichMapFunction<Record, Output> {
    private ValueState< Long> countState;   // 单值状态

    @Override
    public void open(Configuration parameters) {
        countState = getRuntimeContext().getState(
            new ValueStateDescriptor<>("count", Long.class)
        );
    }

    @Override
    public Output map(Record r) {
        Long count = countState.value() == null ? 0L : countState.value();
        countState.update(count + 1);
        return new Output(r, count);
    }
}
```

### Operator State 示例（Kafka Source 算子状态）

```java
public class MySourceFunction extends SourceFunction<Record>
        implements CheckpointedFunction {

    private ListState<Long> offsetState;

    @Override
    public void initializeState(FunctionInitializationContext ctx) {
        offsetState = ctx.getOperatorStateStore().getListState(
            new ListStateDescriptor<>("offset", Long.class)
        );
    }

    @Override
    public void snapshotState(FunctionSnapshotContext ctx) {
        offsetState.clear();
        offsetState.add(currentOffset);
    }
}
```

### 状态后端（State Backend）

| 后端 | 配置 | 适用场景 |
|------|------|----------|
| **HashMapStateBackend** | 默认 | 小状态、低延迟，数据存 JVM 堆 |
| **EmbeddedRocksDBStateBackend** | `EmbeddedRocksDBStateBackend()` | 大状态、海量数据，数据存 RocksDB |

```java
// 配置 RocksDB 状态后端
env.setStateBackend(new EmbeddedRocksDBStateBackend());
env.getCheckpointConfig().setCheckpointStorage("hdfs://namenode:9000/flink-checkpoints");
```

### 状态过期与清理

```java
// 状态 TTL（Time-To-Live）
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.days(1))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)
    .build();

ValueStateDescriptor<Long> descriptor = new ValueStateDescriptor<>(
    "count", Long.class);
descriptor.enableTimeToLive(ttlConfig);
```
