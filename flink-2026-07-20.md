## 题目一：Flink 的架构是怎样的？为什么说它是真正的流计算框架？

### 参考答案

Flink 采用 **流优先（Stream-first）** 的架构设计，整个系统核心是一条**事件驱动的流处理流水线**。

### 核心架构组件

```
┌─────────────────────────────────────────────────────┐
│                   Flink 架构                         │
├─────────────────────────────────────────────────────┤
│  JobManager（Master）                                │
│  ├── Dispatcher —— 接收提交，启 JobMaster           │
│  ├── ResourceManager —— 管理 TaskManager 资源       │
│  └── JobMaster —— 协调 Job 执行，做执行图调度       │
├─────────────────────────────────────────────────────┤
│  TaskManager（Worker）                               │
│  ├── Slot —— 资源分配单元，默认 1核1G               │
│  └── Task —— 具体执行算子的线程                     │
└─────────────────────────────────────────────────────┘
```

### 与其他框架的关键区别

| 特性 | Flink | Spark Streaming | Storm |
|------|-------|-----------------|-------|
| 计算模型 | **原生流**（DataStream） | 微批次（Mini-batch） | 原生流 |
| 延迟 | **低延迟（毫秒级）** | 亚秒级~秒级 | 最低 |
| 背压机制 | 本地流控 + 网络栈反馈 | 限速拉取 | 无内置 |
| 事件时间支持 | ✅ 完整 | ✅ 有限 | ❌ |
| 状态管理 | ✅ 强状态（rocksdb） | ✅ 弱状态 | ❌ |

### 核心亮点

1. **分布式快照（Chandy-Lamport）**：Flink 用 Checkpoint 基于 Chandy-Lamport 算法实现分布式一致性快照，不阻塞流处理
2. **流图 vs 执行图**：DataStream API 编译出 StreamGraph → 优化成 JobGraph → 提交给 JobMaster 调度到 TaskManager
3. **资源模型**：TaskManager 提供 Slot，JobManager 动态分配 Slot 给 Task，支持动态资源调度

> 📚 **来源**：[Flink 官方文档 - 架构](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/flink-architecture/)

## 题目二：什么是 Watermark？Flink 如何处理乱序事件？

### 参考答案

### Watermark（水位线）的本质

Watermark 是 Flink **事件时间（Event Time）** 处理的核心机制，它是一条**时间戳递增的标记线**，表示"所有早于该时间戳的数据已经全部到达"。

```
事件时间轴：
──────────────────────────────────────────────────────→
      W(10:00)  W(10:05)  W(10:10)
         ↓         ↓         ↓
   [已到达]   [已到达]   [等待中]
   不再等待    不再等待    可能迟到
```

### 为什么需要 Watermark？

现实场景中数据会因为网络延迟、机器故障等原因**乱序到达**。如果用事件时间做窗口计算，必须等窗口内的数据到齐才能触发计算。Watermark 就是来解决"**窗口何时可以关闭**"这个问题的。

### Watermark 的生成方式

#### 1. Punctuated（标点式）—— 有数据就有 Watermark
```java
DataStream<Tuple2<Long, String>> stream = env.addSource(new SourceFunction<>() {
    @Override
    public void run(SourceContext<Tuple2<Long, String>> ctx) {
        // 每来一条数据，发一个 Watermark
        ctx.emitWatermark(new Watermark(event.getTimestamp()));
    }
});
```

#### 2. Periodic（周期性）—— 按时间间隔生成
```java
WatermarkStrategy
    .forBoundedOutOfOrderness(Duration.ofSeconds(10)) // 最大乱序等待10秒
    .withTimestampAssigner((event, ts) -> event.getTimestamp());
```

### 窗口触发时机

```java
WindowedStream<Event, String, TimeWindow> windowed = stream
    .keyBy(Event::getKey)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .allowedLateness(Time.minutes(1)); // 允许额外迟到1分钟
```

**触发条件**：`Watermark Time >= 窗口结束时间 + allowedLateness`

### 处理乱序的完整策略

```
1. 设定 Watermark 延迟（maxOutOfOrderness）
2. 设置 allowedLateness（额外等待时间）
3. 使用 sideOutputLateData 收集迟到数据

OutputTag<Event> lateTag = new OutputTag<Event>("late-data"){};
windowed
    .sideOutputLateData(lateTag)
    .process(new WindowFunction<>() {
        public void process(...) {
            // 主逻辑：只处理准时+允许迟到范围内数据
        }
    });

// 收集迟到太久的数据
DataStream<Event> lateStream = result.getSideOutput(lateTag);
```

### 常见面试追问

> **Q：Watermark 传递规则是什么？**
> A：多输入算子（如 union、join）取 min（最慢链路）；单输入算子透传。下游算子收到上游 watermark 后，只有当新 watermark 大于当前值才更新。

> **Q：Watermark 断了会怎样？**
> A：下游窗口永远不会触发，导致数据卡住。因此要保证 Watermark 严格单调递增。

> 📚 **来源**：[Flink 官方文档 - Event Time & Watermarks](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/event-time/)

## 题目三：Flink Checkpoint 的原理是什么？如何实现精确一次语义？

### 参考答案

### Checkpoint 的核心思想

Flink Checkpoint 基于 **Chandy-Lamport 分布式快照算法**，实现在**不停止流处理的情况下**对整个状态做一致性快照。核心思想是"**屏障（Barrier）**"机制。

### Barrier 机制详解

```
Source  ──barrier──> 算子A  ──barrier──>  算子B
   │                      │                      │
   ↓                      ↓                      ↓
checkpoint sync      checkpoint sync         checkpoint sync
```

1. **JobManager 定期**（可配置 interval）向所有 Source 算子发送 `TriggerCheckpoint` 消息
2. Source 算子**插入 Checkpoint Barrier** 到数据流中（不阻塞数据），并保存自己的状态到状态后端，然后往下游发送 Barrier
3. 中间算子收到所有输入的 Barrier 后（**Barrier 对齐**），做状态快照，然后继续往下游传递
4. 所有算子完成快照后，通知 JobManager，Checkpoint 完成

### 状态后端选择

```java
// 内存状态后端（开发测试用）
env.setStateBackend(new EmbeddedMemoryStateBackend());

// RocksDB 状态后端（生产推荐，支持增量 checkpoint）
env.setStateBackend(new EmbeddedRocksDBStateBackend());

// 设置增量 checkpoint
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(Duration.ofSeconds(10));
env.getCheckpointConfig().setCheckpointTimeout(Duration.ofMinutes(10));
env.getCheckpointConfig().enableUnalignedCheckpoints(); // 开启不对齐 checkpoint
```

### 精确一次（Exactly-Once）语义配置

```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

// 开启 Checkpoint，间隔 3 秒
env.enableCheckpointing(3000);

// Checkpoint 存储方式：分布式文件系统
env.getCheckpointConfig().setCheckpointStorage("hdfs:///flink/checkpoints");
// 或本地测试
env.getCheckpointConfig().setCheckpointStorage("file:///tmp/flink-checkpoints");

// 精确一次语义
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);

// Barbier 对齐超时，避免堵住
env.getCheckpointConfig().setCheckpointTimeout(Duration.ofMinutes(10));
env.getCheckpointConfig().setAlignedCheckpointTimeout(Duration.ofSeconds(1));

// 允许 checkpoint 失败次数
env.getCheckpointConfig().setTolerableCheckpointFailureNumber(3);
```

### 三种语义对比

| 语义 | 描述 | 适用场景 |
|------|------|---------|
| **At-Least-Once** | 数据至少被处理一次，可能重复 | 日志统计、监控 |
| **Exactly-Once** | 数据恰好处理一次（端到端） | 金融交易、订单处理 |
| **End-to-End Exactly-Once** |  Source + Flink + Sink 全链路精确一次 | 需要 Source/Sink 配合 |

### 端到端精确一次的实现

```java
// Kafka Source 精确一次
KafkaSource<String> source = KafkaSource.<String>builder()
    .setBootstrapServers("localhost:9092")
    .setTopics("input-topic")
    .setGroupId("my-group")
    .setStartingOffsets(OffsetsInitializer.committedOffsets())
    .build();

DataStream<String> stream = env.fromSource(source,
    WatermarkStrategy.noWatermarks(), "Kafka Source");

// Kafka Sink 精确一次（事务写入）
KafkaSink<String> sink = KafkaSink.<String>builder()
    .setBootstrapServers("localhost:9092")
    .setRecordSerializer(KafkaRecordSerializationSchema.builder()
        .setTopic("output-topic")
        .setValueSerializationSchema(new SimpleStringSchema())
        .build())
    .setDeliveryGuarantee(DeliveryGuarantee.EXACTLY_ONCE)
    .setTransactionalIdPrefix("flink-")
    .setKafkaProducerConfig(props)
    .build();
```

## 题目四：Flink 窗口函数有哪些？如何选择？滚动窗口、滑动窗口、会话窗口的区别？

### 参考答案

### Flink 窗口分类

```
窗口类型
├── Time Window（时间窗口）
│   ├── 滚动窗口（Tumbling）—— 不重叠，首尾相接
│   ├── 滑动窗口（Sliding）—— 可重叠，有重叠区域
│   └── 会话窗口（Session）—— 动态触发，间隙分隔
├── Count Window（计数窗口）
│   ├── 滚动计数窗口
│   └── 滑动计数窗口
└── Global Window（全局窗口）—— 所有数据进入同一窗口（需自定义触发器）
```

### 滚动窗口 vs 滑动窗口 vs 会话窗口

#### 1. 滚动窗口（Tumbling Window）
```java
// 每5分钟一个窗口，互不重叠
DataStream<Tuple2<String, Long>> windowed = stream
    .keyBy(r -> r.f0)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .sum(1);

// 计数滚动窗口：每100条数据一个窗口
.keyBy(...)
.window(TumblingProcessingTimeWindows.of(Integer.valueOf(100)));
```

```
滚动窗口（size=5，slide=5）：
[0-5) [5-10) [10-15) [15-20)
 |-----|-----|-----|-----|
   0     5     10    15
```

#### 2. 滑动窗口（Sliding Window）
```java
// 窗口大小10分钟，滑动步长5分钟（每5分钟输出一次）
.keyBy(...)
.window(SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(5)))
.reduce((a, b) -> a);


// 计数滑动窗口
.window(SlidingProcessingTimeWindows.of(100, 20));
```

```
滑动窗口（size=10，slide=5）：
|-----|-----|-----|-----|
0     5     10    15    20
  |-----|-----|-----|-----|
```

#### 3. 会话窗口（Session Window）
```java
// 会话间隔超过3分钟则开启新窗口
.keyBy(...)
.window(SessionWindows.withGap(Time.minutes(3)))
.reduce((a, b) -> a);

// 动态间隙（基于事件本身）
.keyBy(...)
.window(SessionWindows.withDynamicGap(ev -> ev.getGapTime()));
```

```
会话窗口（gap=3）：
[数据A]--[数据B]--[gap>3]--[数据C]--[数据D]
  窗口1                窗口2
```

### 窗口函数详解

#### ProcessWindowFunction（全窗口函数）
```java
.windowAll(TumblingEventTimeWindows.of(Time.minutes(5)))
.process(new ProcessAllWindowFunction<Event, Result, TimeWindow>() {
    @Override
    public void process(Context ctx, Iterable<Event> elements,
                        Collector<Result> out) {
        // 可以访问窗口元信息
        long count = 0;
        for (Event e : elements) {
            count++;
        }
        out.collect(new Result(ctx.window().getStart(), count));
    }
});
```

#### ReduceFunction + WindowFunction（增量聚合）
```java
// 效率更高，边来边聚合，窗口触发时做少量计算
.keyBy(...)
.window(TumblingEventTimeWindows.of(Time.minutes(5)))
.reduce(
    (v1, v2) -> new Tuple2<>(v1.f0, v1.f1 + v2.f1), // 增量聚合
    new ProcessWindowFunction<...>() { ... }         // 全量处理（可获取窗口上下文）
);
```

#### AggregateFunction（更灵活的增量聚合）
```java
.window(TumblingEventTimeWindows.of(Time.minutes(5)))
.aggregate(new AverageAggregate());

public static class AverageAggregate
    implements AggregateFunction<Tuple2<String, Long>,
                                 Tuple2<Long, Long>, Double> {

    @Override
    public Tuple2<Long, Long> createAccumulator() {
        return new Tuple2<>(0L, 0L);
    }

    @Override
    public Tuple2<Long, Long> add(Tuple2<String, Long> value,
                                   Tuple2<Long, Long> acc) {
        return new Tuple2<>(acc.f0 + value.f1, acc.f1 + 1);
    }

    @Override
    public Double getResult(Tuple2<Long, Long> acc) {
        return (double) acc.f0 / acc.f1;
    }

    @Override
    public Tuple2<Long, Long> merge(Tuple2<Long, Long> a,
                                     Tuple2<Long, Long> b) {
        return new Tuple2<>(a.f0 + b.f0, a.f1 + b.f1);
    }
}
```

### 窗口选择建议

| 场景 | 推荐窗口 |
|------|---------|
| 固定周期统计（如每5分钟订单量） | 滚动窗口 |
| 需要平滑输出（如移动平均） | 滑动窗口 |
| 用户行为分析（操作中断→新会话） | 会话窗口 |
| 数据量达到阈值触发 | 计数窗口 |

## 题目五：Flink 状态管理机制是怎样的？如何做性能优化？

### 参考答案

### Flink 状态分类

```
Flink State
├── Keyed State（按键分区的状态）
│   ├── ValueState<T>         —— 单值状态
│   ├── ListState<T>          —— 列表状态
│   ├── MapState<K, V>        —— 键值对状态
│   ├── ReducingState<T>      —— 聚合后状态
│   └── AggregatingState<I, O>—— 聚合输出状态
│
└── Operator State（算子状态，全域共享）
    ├── ListState<T>          —— 列表状态（算子扩缩容时均分）
    ├── UnionListState<T>     —— 联合列表状态（扩缩容时全量广播）
    └── BroadcastState<K, V>  —— 广播状态（所有并行实例一致）
```

### 状态编程示例

```java
public class CounterFunction
    extends KeyedProcessFunction<String, Event, Long> {

    // 声明状态
    private ValueState<Long> countState;
    private ListState<Event> eventListState;
    private MapState<String, Integer> categoryCount;

    @Override
    public void open(Configuration parameters) {
        // 初始化状态
        countState = getRuntimeContext()
            .getState(new ValueStateDescriptor<>("count", Long.class));
        eventListState = getRuntimeContext()
            .getListState(new ListStateDescriptor<>("events", Event.class));
        categoryCount = getRuntimeContext()
            .getMapState(new MapStateDescriptor<>("category", String.class, Integer.class));
    }

    @Override
    public void processElement(Event event, Context ctx, Collector<Long> out) {
        // 使用状态
        Long currentCount = countState.value();
        if (currentCount == null) currentCount = 0L;
        currentCount++;
        countState.update(currentCount);

        // 更新 MapState
        Integer catCount = categoryCount.getOrDefault(event.category, 0);
        categoryCount.put(event.category, catCount + 1);

        // 定时器：每天凌晨清空计数
        long dayStart = (ctx.timestamp() / 86400000) * 86400000 + 86400000;
        ctx.timerService().registerEventTimeTimer(dayStart);

        out.collect(currentCount);
    }

    @Override
    public void onTimer(long timestamp, OnTimerContext ctx, Collector<Long> out) {
        // 定时器触发，清空状态
        countState.clear();
        eventListState.clear();
        categoryCount.clear();
    }
}
```

### 状态后端对比

| 状态后端 | 存储介质 | 容量 | 读写性能 | 适用场景 |
|---------|---------|------|---------|---------|
| **HashMapStateBackend** | JVM Heap | 受限于内存 | 最高 | 小状态、低延迟场景 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘） | 可达 TB 级 | 较低（序列化开销） | 大状态、增量 Checkpoint |

```java
// 推荐：生产环境用 RocksDB
env.setStateBackend(new EmbeddedRocksDBStateBackend());

// 开启增量 Checkpoint（只写入变更的 SST 文件）
env.getCheckpointConfig().enableIncrementalCheckpointing();

// 状态大小估算：状态条目数 × 单条大小 × 并行度
```

### 状态性能优化技巧

#### 1. 合理设计状态 key（避免数据倾斜）
```java
// ❌ 不好：热点 key 导致某些 TaskManager 负载过重
.keyBy(event -> event.userId)  // 超级用户数据量巨大

// ✅ 好：加随机后缀打散
.keyBy(event -> event.userId + "#" + ThreadLocalRandom.current().nextInt(10))
```

#### 2. 状态过期清理（TTL）
```java
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.days(1))
    .setUpdateType(StateTtlConfig.UpdateType.OnCreateAndWrite) // 创建或写入时更新
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired) // 不返回过期数据
    .cleanupInRocksdbCompactFilter(1000) // RocksDB compaction 时清理
    .build();

ValueStateDescriptor<MyState> stateDesc =
    new ValueStateDescriptor<>("myState", TypeInformation.of(MyState.class));
stateDesc.enableTimeToLive(ttlConfig);
```

#### 3. 开启 Checkpoint 压缩
```java
env.getCheckpointConfig().setCheckpointCompression(true); // LZ4 压缩
```

#### 4. 避免大状态对象
```java
// ❌ 大对象作为状态
ValueStateDescriptor<MyHeavyObject> desc = ...

// ✅ 只存必要字段，用精简结构
ValueStateDescriptor<MyStateLite> desc = ...
```

### 状态扩容（Rescale）时的数据分配

```
Operator ListState（扩缩容均分）:
  Task0: [A, B]  Task1: [C, D]  → 新 Task0: [A] Task1: [B] Task2: [C] Task3: [D]

UnionListState（全量广播）:
  Task0: [A, B]  Task1: [C, D]  → 每个新 Task 收到 [A, B, C, D]，自己选择
```
