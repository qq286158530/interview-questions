## 题目一：Redis 有哪些主要的数据类型？适用场景分别是什么？

### 参考答案

Redis 支持五种基本数据类型，每种类型都有其独特的适用场景：

**1. String（字符串）**
- 最基础的数据类型，最大存储 512MB。
- 适用场景：缓存简单变量（如用户 Token、session 信息）、计数器（incr/incrby）、分布式锁（setnx + expire）。
- 示例命令：`SET key value`、`GET key`、`INCR counter`

**2. List（列表）**
- 双向链表实现，支持从两端插入/删除，插入/删除时间复杂度 O(1)。
- 适用场景：消息队列（lpush + brpop）、最新消息列表（lpush + ltrim）、任务队列。
- 示例命令：`LPUSH list item`、`RPOP list`、`LRANGE list 0 -1`

**3. Hash（哈希）**
- 键值对集合，适合存储对象。
- 适用场景：存储对象属性（如用户信息 `{name: "张三", age: 25}`）、购物车数据。
- 示例命令：`HSET user:1001 name "张三"`、`HGET user:1001 name`、`HGETALL user:1001`

**4. Set（无序集合）**
- 无序且唯一（去重）的字符串集合，支持交、并、差运算。
- 适用场景：标签系统、好友关系、共同关注/粉丝、共同好友功能。
- 示例命令：`SADD tags python`、`SINTER tag1 tag2`（交集）、`SRANDMEMBER set 3`（随机取元素）

**5. Sorted Set / ZSet（有序集合）**
- 每个元素关联一个分数，按分数排序，支持范围查询。
- 适用场景：排行榜（积分排序）、延时队列（按时间戳排序）、权重队列。
- 示例命令：`ZADD leaderboard 100 "user1"`、`ZREVRANGE leaderboard 0 9 WITHSCORES`

**补充：三种高级数据类型**
- **Bitmap**：位图，适合签到、统计活跃用户。
- **HyperLogLog**：基数统计，适合 UV 统计。
- **Geospatial**：地理位置，适合附近的人、打车定位。

**来源：**
- https://www.redis.io/docs/data-types/
- https://cloud.tencent.com/developer/article/1685201

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 参考答案

Redis 支持两种持久化机制：

**RDB（Redis Database）—— 快照方式**

- 定时将内存中的所有数据快照写入磁盘，生成 `dump.rdb` 文件。
- 触发方式：
  - `BGSAVE` 后台异步执行（主进程 fork 子进程执行，不阻塞）
  - `SAVE` 同步执行（阻塞，通常不使用）
  - 定时自动触发（通过 `save` 配置项，如 `save 900 1` 表示 900 秒内 ≥1 次写操作则触发）
  - `SHUTDOWN` 时自动触发
- 优点：恢复大数据集速度快（fork 子进程对性能影响小）。
- 缺点：可能丢失最近一次快照后的数据（数据完整性不如 AOF）。

**AOF（Append Only File）—— 日志追加方式**

- 将每个写操作命令追加到日志文件末尾，Redis 重启时重放命令恢复数据。
- 配置：`appendonly yes` 开启。
- 同步策略（`appendfsync`）：
  - `always`：每条命令都同步写入磁盘，可靠性最高，性能最差。
  - `everysec`（默认）：每秒同步一次，最多丢失 1 秒数据。
  - `no`：由操作系统决定何时同步，性能最好，可靠性最低。
- 优点：数据安全性更高（特别是 everysec 模式下）。
- 缺点：AOF 文件通常比 RDB 大，恢复速度慢。

**RDB vs AOF 对比表**

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件大小 | 小（紧凑二进制） | 大（文本命令） |
| 恢复速度 | 快 | 慢 |
| 数据安全性 | 可能丢失数据 | 丢失少 |
| 性能影响 | fork 消耗资源 | 持续写入开销 |
| 场景 | 容忍数据丢失 | 要求高可靠性 |

**如何选择（最佳实践）：**

1. **同时开启 RDB + AOF**（推荐）：RDB 做定期全量备份，AOF 做实时增量，保证数据安全且恢复速度可接受。
2. **只开 AOF**：`appendfsync everysec`，足够应付大多数场景。
3. **只开 RDB**：适合数据量大的场景，且对丢失容忍度高（如缓存层）。

**来源：**
- https://www.redis.io/docs/management/persistence/
- https://developer.redis.com/learn/how-to-persist

## 题目三：Redis 的过期 key 删除策略是怎样的？

### 参考答案

Redis 采用两种过期 key 删除策略，协同工作：

**1. 惰性删除（Lazy Expiration）**

- 原理：key 被访问时（GET 等命令），才检查是否过期，若过期则删除。
- 优点：对 CPU 友好，不会在 key 没被访问时做无用功。
- 缺点：如果过期 key 长期不被访问，会一直占用内存（内存泄漏风险）。
- 实现：每次访问 key 时调用 `expireIfNeeded()` 函数检查。

**2. 定期删除（Active Expiration）**

- 原理：Redis 每隔一段时间（默认每 100ms，随机抽取带过期时间的 key 进行检查）主动扫描并删除已过期的 key。
- 优点：定期清理内存，防止大量过期 key 堆积。
- 实现细节：
  - 每 100ms 随机抽取 `hz`（默认 10）个设置了过期时间的 key。
  - 删除其中所有已过期的 key。
  - 若过期 key 比例超过 25%，继续抽取删除（直到比例降到 25% 以下或超时）。
- 配置参数：`hz`（每秒检查频率）和 `dynamic-hz`（根据负载动态调整）。

**补充：内存淘汰策略（当内存达到上限时）**

当 Redis 内存达到 `maxmemory` 上限，会触发内存淘汰策略删除 key：

| 策略 | 说明 |
|------|------|
| `noeviction` | 不删除，返回错误（默认） |
| `volatile-lru` | 在带过期时间的 key 中，使用 LRU 算法淘汰 |
| `allkeys-lru` | 在所有 key 中，使用 LRU 算法淘汰 |
| `volatile-lfu` | 在带过期时间的 key 中，使用 LFU 算法淘汰 |
| `allkeys-lfu` | 在所有 key 中，使用 LFU 算法淘汰 |
| `volatile-random` | 在带过期时间的 key 中随机淘汰 |
| `allkeys-random` | 在所有 key 中随机淘汰 |
| `volatile-ttl` | 淘汰最快过期的 key |

**最佳实践建议：**
- 不要依赖过期key自动删除做精确到期处理，定时任务配合 RabbitMQ / Redisson 更可靠。
- 生产环境务必配置 `maxmemory` + 合适的淘汰策略，防止 OOM。
- 合理设置 `hz` 参数，在 CPU 和内存回收效率间做平衡。

**来源：**
- https://www.redis.io/docs/reference/expires/
- https://redis.io/docs/manual/eviction/

## 题目四：Redis 主从复制原理是什么？哨兵（Sentinel）和集群（Cluster）有什么区别？

### 参考答案

**主从复制（Replication）**

Redis 主从复制采用 **异步复制** 机制：

- **首次同步（全量同步）**：
  1. 从库向主库发送 `PSYNC ? -1`（或 `SYNC`，旧版本命令）。
  2. 主库执行 `BGSAVE`，生成 RDB 快照，同时将后续写命令存入缓冲区（replication buffer）。
  3. 主库将 RDB 文件发送给从库，从库加载到内存。
  4. 主库将缓冲区中的增量命令发送给从库执行。
- **增量同步（命令传播）**：
  - 主库每执行一个写命令，异步发送给所有从库（`主库 → 从库` 单向传播）。
  - 从库通过心跳（`replconf ack <offset>`）向主库报告已同步的偏移量。
- **关键参数**：
  - `repl-diskless-sync`：无磁盘复制，通过网络直接传输 RDB。
  - `replication buffer`：主库用于缓存增量命令的缓冲区，若从库落后太多可能溢出导致全量同步。

**哨兵模式（Sentinel）**

哨兵是 Redis 的**高可用（HA）解决方案**：

- 监控主从库健康状态（`PING` 机制）。
- 自动故障转移（从从库中选举新主库，其他从库指向新主库）。
- 提供服务发现（客户端通过哨兵获取当前主库地址）。
- 哨兵本身也是分布式（多个哨兵节点，quorum 机制避免单点故障）。

**集群模式（Cluster）**

Redis Cluster 是 Redis 的**分布式解决方案**：

- **数据分片**：16384 个 slot，平均分配到各主节点（每个节点管理一部分 slot）。
- **去中心化**：节点间通过 Gossip 协议通信，无需独立代理。
- **高可用**：每个主节点可配置从节点（replica），主节点宕机后从节点自动补位。
- **客户端路由**：客户端可请求任意节点，节点通过 MOVED 错误告知正确节点。
- **不支持跨 slot 操作**（如 `MGET` 跨多个 key 需要用 `Hash Tag` 保证 key 在同一 slot）。

**对比表**

| 维度 | 主从复制 | 哨兵模式 | 集群模式 |
|------|---------|---------|---------|
| 数据副本 | ✅ 支持 | ✅ 支持 | ✅ 支持（分片副本） |
| 自动故障转移 | ❌ 需手动 | ✅ 自动 | ✅ 自动 |
| 数据分片 | ❌ 无 | ❌ 无 | ✅ 16384 slot |
| 写可靠性 | 单点写 | 单点写 | 多节点写（部分） |
| 节点发现 | 手动配置 | 哨兵通知 | Gossip 协议 |
| 适用规模 | 小规模 | 中等规模 | 大规模分布式 |

**来源：**
- https://www.redis.io/docs/manual/replication/
- https://redis.io/docs/management/sentinel/
- https://redis.io/docs/management/cluster/

## 题目五：Redis 如何实现分布式锁？有什么注意事项？

### 参考答案

**Redis 分布式锁基本实现**

使用 `SET key value NX PX <milliseconds>` 原子性加锁：

```bash
# 加锁（value 用唯一标识，如 UUID，防止误删他人的锁）
SET lock_key unique_id NX PX 30000

# 解锁（Lua 脚本保证原子性，先比较 value 再删除）
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

**核心要点：**
- `NX`：key 不存在时才设置，保证互斥。
- `PX`：设置过期时间，防止锁无法释放（进程崩溃等）。
- **解锁必须验证 value**（客户端持有锁的凭证），防止误删其他客户端的锁。

---

**进阶方案：Redlock（红锁）**

单节点 Redis 锁存在单点故障风险，Redis 官方提出 Redlock 算法：

1. 客户端获取当前时间戳 `T1`。
2. 依次向 N 个 Redis 实例（独立节点）用相同 key 和 value 执行 `SET NX PX`。
3. 客户端计算 `T2 = now() - T1`，如果 **超过半数节点** 获取成功，且总耗时 < 锁 TTL，则认为加锁成功。
4. 失败时向所有节点发送 `DEL` 释放锁。

**注意事项（避坑指南）**

| 问题 | 说明 | 解决方案 |
|------|------|---------|
| 锁误删 | 解锁时未验证 value，删了别人的锁 | 用 Lua 脚本原子性比较 + 删除 |
| 锁过期但任务未完成 | 锁 TTL 设短了，任务还在跑但锁没了 | 延长锁 TTL / 用看门狗（watchdog）续期 |
| 单点故障 | 主库宕机，锁丢失 | Redlock 多节点 / 开启 AOF + `appendfsync always` |
| 脑裂问题 | 主从切换时多个客户端同时加锁成功 | Redlock quorum 机制 |
| 锁重入 | 同一进程需要重入 | value 中存计数器，重入时 +1，释放时 -1 |
| 性能 | Redlock 需要多节点通信 | 权衡可靠性和性能 |

**Redisson 框架推荐**

生产环境推荐使用 Redisson，它内置了：
- 看门狗自动续期（锁自动延期）
- 可重入锁、公平锁、读写锁等变体
- 所有操作原子化封装，开箱即用

```java
RLock lock = redisson.getLock("myLock");
lock.lock(); // 自动续期
try {
    // 业务逻辑
} finally {
    lock.unlock();
}
```
