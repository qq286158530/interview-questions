## 题目一：Redis 和 Memcached 有什么区别？

### 参考答案

**共同点：**
- 都是基于内存的数据库，常用作缓存
- 都有过期策略
- 性能都很高

**核心区别：**

| 区别点 | Redis | Memcached |
|--------|-------|-----------|
| 数据类型 | 支持丰富数据类型（String/Hash/List/Set/ZSet等） | 仅支持简单的 key-value |
| 持久化 | 支持 RDB 和 AOF 两种持久化方式 | 无持久化功能，数据全存内存 |
| 集群 | 原生支持主从、哨兵、切片集群 | 无原生集群，需客户端实现分片 |
| 高级特性 | 支持发布订阅、Lua脚本、事务 | 不支持 |
| 存储方式 | 部分数据可持久化到磁盘 | 数据仅存内存 |

**为什么选Redis作为MySQL缓存？**
- **高性能**：Redis QPS 可达 10W+，是MySQL的10倍，第一次访问MySQL后缓存到Redis，后续直接读缓存
- **高并发**：直接访问Redis承受的请求远大于MySQL，分担数据库压力

## 题目二：Redis 单线程模型是怎样的？为什么单线程还这么快？

### 参考答案

**Redis单线程指的是：** 主线程负责接收客户端请求→解析请求→数据读写→发送响应整个流程，由一个线程完成。

**但Redis并非完全单线程：** 2.6版本起会启动后台线程处理关闭文件、AOF刷盘；4.0+还新增lazyfree线程异步释放内存。

**单线程工作流程（Redis 6.0之前）：**
1. 调用 `epoll_create()` 创建 epoll 对象
2. 绑定端口并监听 socket
3. 主线程进入事件循环：
   - 先处理发送队列，通过 `write()` 发送数据
   - 调用 `epoll_wait()` 等待事件
   - 连接事件→调用 `accept()` 获取连接→注册读事件
   - 读事件→`read()` 获取数据→解析命令→执行命令→加入发送队列
   - 写事件→`write()` 发送数据

**单线程为什么这么快？**

1. **内存为王**：大部分操作在内存中完成，采用高效数据结构，瓶颈是内存/带宽而非CPU
2. **无竞争开销**：单线程避免了多线程切换、锁竞争、死锁等问题
3. **IO多路复用**：一个线程处理多个IO流（select/epoll机制），监听多个Socket连接请求

## 题目三：Redis 有哪几种持久化方式？各有何优缺点？

### 参考答案

**RDB（Redis Database）：** 定时生成数据快照，保存为 `dump.rdb` 文件

**AOF（Append Only File）：** 记录所有写操作命令到日志文件

**对比：**

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件体积 | 小（紧凑二进制） | 大（纯文本命令） |
| 恢复速度 | 快（直接还原内存） | 慢（逐条执行命令） |
| 数据安全性 | 可能丢失最近数据 | 可配置（always/everysec/no） |
| 性能影响 | fork子进程，CPU开销大 | 写操作和记录日志可异步 |

**AOF三种写回策略：**
- `Always`：每次写命令后同步写回，最多丢失一条数据，但影响性能
- `Everysec`：每秒异步写回，兼顾性能与数据安全
- `No`：由操作系统决定何时写回，性能最好但可能丢失较多数据

**推荐策略：** RDB+AOF混合使用，用RDB做快照，AOF记录增量操作。

## 题目四：Redis 哨兵机制是如何实现主从故障转移的？

### 参考答案

**为什么需要哨兵？**
主从架构中主节点挂了需要人工介入切换，哨兵可自动完成主从切换。

**哨兵的核心职责：** 监控、选主、通知

**故障判断流程：**
1. 哨兵每1秒向主从节点发送 `PING` 命令
2. 未在 `down-after-milliseconds` 时间内响应→标记为「主观下线」
3. 多个哨兵协商（至少quorum个哨兵赞成）→判定「客观下线」

**选主过程（从所有从节点中选新主）：**
1. 过滤网络状态不好的从节点（断连次数>10次的过滤）
2. 三轮考察：
   - 第一轮：优先级 `slave-priority` 高的胜出
   - 第二轮：复制进度最靠前的（`slave_repl_offset` 最接近 `master_repl_offset`）
   - 第三轮：ID号最小的胜出

**故障转移四步骤：**
1. 挑选最优从节点，执行 `SLAVEOF no one` 升级为主节点
2. 其他从节点指向新主节点（发送 `SLAVEOF` 命令）
3. 通过发布订阅通知客户端新主节点信息
4. 旧主节点重新上线后设为新主节点的从节点

## 题目五：Redis 的数据类型以及底层实现是什么？

### 参考答案

**五种核心数据类型及底层结构：**

| 数据类型 | 底层实现（元素少时） | 底层实现（元素多时） |
|----------|---------------------|---------------------|
| String | SDS（简单动态字符串） | SDS |
| List | quicklist（3.2+）/压缩列表 | quicklist/双向链表 |
| Hash | 压缩列表 / 哈希表 | 哈希表 |
| Set | 整数集合 / 哈希表 | 哈希表 |
| ZSet | 压缩列表 / 跳表 | 跳表 |

**String类型（SDS）优势：**
- O(1)获取长度（len属性）
- 二进制安全（不是以空字符判断结束）
- 动态扩容不会缓冲区溢出

**List类型：** Redis 3.2后统一使用quicklist（双向链表+压缩列表的组合）

**ZSet（有序集合）：** 跳表实现，查询复杂度O(logN)，比平衡树更节省内存

**特殊数据类型：**
- BitMap（2.2+）：二值状态统计（签到、用户在线状态）
- HyperLogLog（2.8+）：海量数据基数统计（UV计数）
- GEO（3.2+）：地理位置存储（附近的人、滴滴叫车）
- Stream（5.0+）：消息队列，支持消费组
