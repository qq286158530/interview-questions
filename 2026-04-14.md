## 题目1：Java 中 AQS（AbstractQueuedSynchronizer）的核心原理是什么？ReentrantLock 是如何基于 AQS 实现的？

【答案】

**AQS 核心原理**

AQS 是 Java 并发包（JUC）的核心基础组件，采用 **CLH 锁队列**（Craig, Landin, Hagersten）变体实现。AQS 通过一个 `volatile` 的 `state` 变量表示同步状态，并维护一个 FIFO 的双向等待队列。

核心结构：
- **state**：代表资源状态，子类通过 `getState()`、`setState()`、`compareAndSetState()` 访问
- **FIFO 队列**：存储等待获取资源的线程节点，是一个双向链表
- **Node**：每个等待线程封装为一个 Node，包含 `waitStatus`（CANCELLED/SIGNAL/CONDITION/PROPAGATE）

AQS 支持两种模式：
1. **独占模式（Exclusive）**：如 ReentrantLock，同一时刻只有一个线程持有锁
2. **共享模式（Shared）**：如 Semaphore、CountDownLatch，多个线程可同时获取

**获取锁流程（独占模式）：**
1. `tryAcquire()` 尝试CAS修改 state 成功 → 获取锁成功
2. 失败则创建 Node 加入等待队列尾部（自旋 CAS 入队）
3. 进入 `acquireQueued()` —— 节点在队列中自旋：
   - 如果当前节点是 head 的直接后继，尝试 `tryAcquire()`
   - 若失败且需要阻塞，通过 `LockSupport.park()` 挂起线程
4. 前驱节点释放锁时，会唤醒后继节点

**ReentrantLock 如何基于 AQS 实现：**
- `Sync` 抽象类继承 AQS，重写 `tryAcquire()` 和 `tryRelease()`
- `NonfairSync` / `FairSync` 继承 Sync，分别实现非公平/公平获取
- **可重入**：state 记录持有次数，同一线程重复 acquire 时累加计数器，release 时减计数至0才真正释放
- **公平锁**：在 `tryAcquire()` 中增加 `hasQueuedPredecessors()` 检查，避免插队

```java
// 非公平锁 tryAcquire 核心逻辑
if (compareAndSetState(0, 1)) {
    setExclusiveOwnerThread(Thread.currentThread());
    return true;
} else if (getExclusiveOwnerThread() == Thread.currentThread()) {
    setState(getState() + 1);
    return true;
}
```

**ConditionObject**（条件队列）：每个 Condition 实例关联一个单向条件队列，支持 `await()` / `signal()`，实现 Object.wait() 的功能但更灵活。

## 题目2：JDK 1.8 HashMap 在高并发场景下可能导致死循环，请详细解释原因，并说明替代方案有哪些？

【答案】

**死循环原因分析**

JDK 1.7 HashMap 在并发 resize 时可能产生死循环，JDK 1.8 已修复此问题，但并发安全问题依然存在。

**JDK 1.7 死循环根源——头插法**

JDK 1.7 HashMap 使用**头插法**进行链表迁移：
```java
// JDK 1.7 transfer() 核心代码
void transfer(Entry[] newTable, boolean rehash) {
    Entry[] src = table;
    for (int j = 0; j < src.length; j++) {
        Entry<K,V> e = src[j];
        if (e != null) {
            src[j] = null;
            do {
                Entry<K,V> next = e.next;
                int i = indexFor(e.hash, newCapacity);
                e.next = newTable[i];  // 头插到新表
                newTable[i] = e;
                e = next;
            } while (e != null);
        }
    }
}
```

**死循环场景（两个线程同时 resize）：**

假设链表：A → B（头插法 reverse 后变成 B → A）

1. Thread1 执行到 `e = A, next = B` 时被挂起
2. Thread2 完整执行了 transfer，链表变为 B → A
3. Thread1 恢复继续：e = A, next = B；将 A 头插到新表；e = B, next = null（注意：B.next 已被 Thread2 改为指向 A！）将 B 头插到新表，但 B.next = A
4. 结果：新表变成 A → B → A，形成环形链表

当 get() 查找不存在的 key 时遍历到环形链表，会触发死循环。

**JDK 1.8 的修复：**
JDK 1.8 使用**尾插法**替代头插法，保证同一链表内节点顺序不变，不会产生环形链表。但 **JDK 1.8 依然不是线程安全的**，并发 put 可能导致数据覆盖（丢失更新）。

**并发场景下的替代方案：**

| 方案 | 适用场景 | 特点 |
|------|---------|------|
| `ConcurrentHashMap` | 高并发读写的通用场景 | 分段锁（JDK 1.7）/ CAS + synchronized（JDK 1.8） |
| `Collections.synchronizedMap()` | 简单改造现有代码 | 全局锁，性能差 |
| `Hashtable` | 遗留代码兼容 | 全局锁，不推荐 |
| `ConcurrentLinkedQueue` | 仅需队列场景 | 无锁设计 |

## 题目3：什么情况下 JVM 会触发 Full GC？CMS 和 G1 收集器在处理 Full GC 时有什么不同？

【答案】

**Full GC 触发条件**

Full GC 是老年代或全局区域的垃圾收集，停顿时间通常较长，是性能调优的重点。

**触发 Full GC 的典型场景：**

1. **老年代空间不足**：Eden + Survivor 区对象晋升到老年代时，老年代可用空间不足
2. **显式调用 `System.gc()`**：未明确禁用时不一定会触发 Full GC（`-XX:+DisableExplicitGC` 可禁用）
3. **永久代（元空间）满了**：JDK 8 以前 PermGen 满时触发 Full GC；JDK 8+ 元空间不足也会触发
4. **CMS 并发模式失败（Concurrent Mode Failure）**：CMS 在老年代回收期间，新生代对象晋升但空间不足
5. **空间分配担保失败**：Minor GC 前，老年代最大可用连续空间 < 历代晋升对象平均大小，触发 Full GC
6. **大对象直接进入老年代**：对象大小超过 `PretenureSizeThreshold` 阈值直接分配在老年代

**CMS vs G1 处理 Full GC 的差异：**

| 维度 | CMS | G1 |
|------|-----|-----|
| **收集类型** | 老年代收集器（搭配 ParNew 新生代） | 整堆收集器（新生代 + 老年代） |
| **Full GC 触发** | `Concurrent Mode Failure` 或 `promotion failed` | 回收速度跟不上分配速度时 |
| **Full GC 算法** | 单线程标记-清除-压缩（`mark-sweep-compact`），STW时间长 | Serial GC 单线程压缩（默认），但停顿仍长 |
| **并发阶段** | 4阶段：初始标记→并发标记→重新标记→并发清除 | 4阶段：初始标记→并发标记→最终标记→筛选回收 |
| **碎片问题** | 内存碎片化，需要 Serial Old 作为备选整理 | 分 region 收集，可设置停顿时间目标（`MaxGCPauseMillis`） |
| **G1 的优势** | — | 优先回收价值最高Region（Garbage First），减少 Full GC 频率 |
| **Mixed GC** | 无 | 老年代达到阈值时触发 Mixed GC（混合回收） |

G1 设计目标就是尽量减少 Full GC。它通过 `IHOP` 参数控制：当老年代空间占比超过 IHOP 时启动并发标记，在 Mixed GC 中回收大部分老年代 Region，尽量避免退化到 Full GC。

## 题目4：Spring AOP 的实现原理是什么？JDK 动态代理和 CGLIB 有什么区别？什么情况下会失效？

【答案】

**Spring AOP 核心实现原理**

Spring AOP 基于 AOP 联盟（AspectJ）的标准接口设计，通过**代理模式**在运行时生成代理对象来实现的。

**两种代理机制：**

**1. JDK 动态代理**
- 要求目标类实现至少一个接口
- 基于 `java.lang.reflect.Proxy` 和 `InvocationHandler` 接口
- 代理类是与目标类**平级的类**（实现相同接口）
- 运行时生成字节码，性能较好

**2. CGLIB 动态代理**
- 目标类可以没有接口
- 基于 ASM 字节码操作库，继承目标类生成子类
- 覆盖非 final 方法（`final` 方法无法被覆盖）
- 性能略低于 JDK 动态代理（JDK 1.8+ 已优化差距很小）

Spring 优先策略：目标类有接口 → JDK代理；无接口 → CGLIB。可强制配置 `proxy-target-class="true"` 始终使用 CGLIB。

**AOP 失效的场景（重点！）：**

1. **自我调用（this.xxx()）** —— 最常见！
   - 通过 `this` 调用同类方法，不会走代理对象的拦截器链
   - 解决：注入自身代理对象（`AopContext.currentProxy()`）

2. **方法内部调用** —— 同上，`this` 调用不走代理

3. **静态方法**：Spring AOP 不支持（代理无法拦截 static/final/private）

4. **代理对象被强转后丢失**：通过 `ApplicationContext.getBean()` 获取的是代理对象，直接转型为具体实现类会 `ClassCastException`

## 题目5：Spring Boot 的 `@Conditional` 注解族有什么作用？实现一个自定义 Starter 的核心步骤是什么？

【答案】

**@Conditional 注解族详解**

`@Conditional` 是 Spring 4.0 引入的注解，用于根据特定条件决定是否注册 Bean。它的出现解决了 Spring Boot **自动配置**的核心问题——根据 classpath、环境变量、配置文件等条件，按需加载组件。

**核心注解族：**

| 注解 | 作用条件 |
|------|---------|
| `@ConditionalOnClass` | classpath 下存在指定类时生效 |
| `@ConditionalOnMissingClass` | classpath 下不存在指定类时生效 |
| `@ConditionalOnBean` | 容器中存在指定 Bean 时生效 |
| `@ConditionalOnMissingBean` | 容器中不存在指定 Bean 时生效 |
| `@ConditionalOnProperty` | 配置属性满足条件时生效 |
| `@ConditionalOnResource` | 存在指定资源文件时生效 |
| `@ConditionalOnExpression` | SpEL 表达式为 true 时生效 |
| `@ConditionalOnWebApplication` | 是 Web 应用时生效 |

---

**自定义 Spring Boot Starter 步骤**

标准 Starter 命名约定：`spring-boot-starter-xxx`

**核心步骤：**

1. **定义配置属性类**
```java
@ConfigurationProperties(prefix = "xxx")
public class XxxProperties {
    private boolean enabled = true;
    private String host = "localhost";
    private int port = 8080;
    // getters/setters
}
```

2. **编写自动配置类**
```java
@Configuration
@ConditionalOnClass(XxxService.class)
@ConditionalOnProperty(prefix = "xxx", name = "enabled", havingValue = "true", matchIfMissing = false)
@EnableConfigurationProperties(XxxProperties.class)
public class XxxAutoConfiguration {
    @Bean
    @ConditionalOnMissingBean
    public XxxService xxxService(XxxProperties properties) {
        return new XxxService(properties);
    }
}
```

3. **注册自动配置（Spring Boot 2.7+）**
在 `resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 中：
```
com.example.autoconfigure.XxxAutoConfiguration
```

4. **使用方只需引入依赖 + yaml 配置即可自动生效**

## 题目6：Spring Cloud Gateway 是如何工作的？它相比 Zuul 有哪些性能优势？请描述一个请求的完整路由流程。

【答案】

**Spring Cloud Gateway 工作原理**

Spring Cloud Gateway 基于 **Spring WebFlux**（响应式编程模型）和 **Netty** 构建，是非阻塞式的异步网关，性能远优于基于 Servlet 的 Zuul。

**核心概念：**

1. **Route（路由）**：网关的基本构件，包含 `id`、`uri`（目标地址）、`predicates`（谓词匹配条件）、`filters`（过滤器链）
2. **Predicate（谓词）**：匹配 HTTP 请求的条件，如 `Path=/user/**`、`Method=GET`
3. **Filter（过滤器）**：在请求前后执行逻辑，如鉴权、限流、请求头修改

**请求完整路由流程：**

```
客户端请求
    ↓
Gateway Handler（DispatchHandler）
    ↓ 匹配 RoutePredicateHandlerMapping
Route Predicate（/api/user/** → 匹配路由规则）
    ↓
Gateway WebHandler（处理过滤链）
    ↓
Pre Filter 链（顺序执行，如：Token验证 → 限流 → 请求头处理）
    ↓
Proxy Exchange（Netty 路由）
    ↓
目标服务（http://user-service/api/user/xxx）
    ↓
Post Filter 链（逆序执行，如：响应头处理 → 日志记录）
    ↓
客户端响应
```

**Spring Cloud Gateway vs Zuul 性能对比：**

| 维度 | Zuul 1.x | Zuul 2.x | Spring Cloud Gateway |
|------|---------|---------|---------------------|
| 线程模型 | 同步阻塞（Tomcat线程池） | 异步非阻塞 | 异步非阻塞（Netty + WebFlux） |
| 连接方式 | BIO | NIO | NIO（Netty） |
| Filter | 同步 Filter | 异步 Filter | 异步 Filter（WebFlux） |
| 性能 | 低 | 中 | 高 |
| 编程模型 | 传统同步 | 异步（Callable/DeferredResult） | 响应式（Flux/Mono） |

## 题目7：Nacos 的服务注册与发现机制是怎样的？它是如何实现配置热更新的？AP 和 CP 模式如何切换？

【答案】

**Nacos 服务注册与发现机制**

Nacos（Naming and Configuration Service）是阿里开源的动态服务发现、配置管理和服务管理平台。

**服务注册流程：**

1. 应用启动时，Nacos Client 向 Nacos Server 发送注册请求（POST `/nacos/v1/ns/instance`）
2. 注册信息包含：`namespaceId`、`group`、`serviceName`、`clusterName`、`ip`、`port`、`weight`
3. Nacos Server 将实例信息存储到**双层Map**：
   - 外层 Key：`serviceName` → 内层 Map
   - 内层 Key：`clusterName` → `instances` 列表
4. 注册后，Nacos Client 与 Server 保持**心跳长连接**，默认每5秒发送一次心跳
5. Nacos Server 在**15秒**内未收到心跳，将实例标记为不健康；**30秒**未收到，删除实例

**配置热更新实现：**

1. 客户端启动时，通过 `ConfigService` 订阅某个 `dataId`（配置文件）
2. Nacos Server 在配置变更后，向订阅的客户端发送长轮询或 WebSocket 通知
3. 客户端收到变更通知后，拉取最新配置
4. Spring Boot 集成 Nacos Config 后，通过 `@RefreshScope` 实现 Bean 的自动刷新

**AP 与 CP 模式切换：**

| 模式 | 使用协议 | 适用场景 | 切换方式 |
|------|---------|---------|---------|
| AP | Distro（最终一致） | 注册/发现（服务临时实例） | 默认 |
| CP | Raft（强一致） | 配置管理、持久化服务 | `curl -X PUT '$NAOCS_SERVER:8848/nacos/v1/ns/operator/switches?entry=serverMode=CP'` |

## 题目8：Kafka 的分区（Partition）策略有哪些？消费者组（Consumer Group）如何保证消息不丢失和不重复消费？

【答案】

**Kafka 分区策略**

**生产者分区策略（Producer Partitioner）：**

1. **指定 Partition**：直接指定发送到特定 Partition
2. **Key 分区**：默认使用 `hash(key) % partitionCount`，相同 Key 的消息一定发到同一 Partition（保证有序性）
3. **轮询策略（Round Robin）**：无 Key 时按 Round Robin 发送到各 Partition，负载均衡
4. **自定义分区器**：实现 `Partitioner` 接口，通过 `headers`、`body` 等信息决定分区

**消费者组（Consumer Group）机制：**

- 一个 Consumer Group 包含多个 Consumer 实例
- Group 订阅相同 Topic，每个 Partition 只分配给 Group 内**一个** Consumer 消费
- 不同 Group 之间互不影响，各自独立消费全部消息

**保证消息不丢失（三端保证）：**

| 端 | 保证措施 |
|----|---------|
| **Producer** | `acks=all`（ISR所有副本确认）+ `retries=MAX` + `enable.idempotence=true` |
| **Broker** | `replication.factor=3` + `min.insync.replicas=2` + `unclean.leader.election=false` |
| **Consumer** | 手动提交 offset + 业务处理成功后再提交（**先处理后提交**） |

**防止重复消费（幂等性消费）：**

1. **业务唯一键**：消费前先查询 DB 唯一键是否已处理（Redis SET NX / 数据库唯一索引）
2. **幂等表**：记录已处理的 `msgId`，消费前检查
3. **分布式锁**：消费前加分布式锁（Redis SETNX），处理完释放
4. **Kafka 事务（Exactly-Once）**：`producer.initTransactions()` + `consumer.poll()` + 业务DB在同一个事务

## 题目9：RabbitMQ 的消息确认机制（Confirm 和 ACK）是如何工作的？如何设计一个可靠的消息消费流程？

【答案】

**RabbitMQ 消息确认机制**

RabbitMQ 有两套独立的确认机制：**生产者确认（Publisher Confirms）** 和 **消费者确认（Consumer Acknowledgements）**。

**1. 生产者确认（Publisher Confirms）**

| 模式 | 方式 | 可靠性 | 性能影响 |
|------|------|-------|---------|
| `CORRELATED`（推荐） | 异步回调，RabbitMQ 返回 `basic.ack`/`basic.nack` | 高 | 低（异步） |
| `WAIT_FOR_CONFIRMS` | 同步等待，发送后阻塞等待确认 | 高 | 中（同步等待） |
| 无确认 | Fire-and-Forget，发送即完事 | 低 | 最高 |

**2. 消费者确认（Consumer ACK）**

RabbitMQ 默认使用**自动 ACK**（Consumer 收到消息即确认）。可靠消费需要**手动 ACK**：

```java
DeliverCallback deliverCallback = (consumerTag, delivery) -> {
    try {
        processMessage(delivery.getBody());
        channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
    } catch (Exception e) {
        channel.basicNack(delivery.getEnvelope().getDeliveryTag(), false, true); // requeue=true
    }
};
```

- `basicAck`：确认已处理，RabbitMQ 删除该消息
- `basicNack`：requeue=true 放回队列重试；requeue=false 进入死信队列

**可靠消息消费流程设计：**

```
1. 消费前：检查 Redis 幂等key（SETNX msgId），若已存在则跳过
2. 接收消息：开启手动 ACK
3. 业务处理：try-catch 包裹，失败时记录到重试表
4. 业务成功：ACK + 写幂等key（TTL=7天）
5. 业务失败（不可重试）：NACK + requeue=false → 死信队列 → 人工处理
```

**死信队列（DLX）设计：**
- 当消息被拒绝且 requeue=false，或超时未 ACK 时，进入死信队列
- 配置：`x-dead-letter-exchange` + `x-dead-letter-routing-key`

https://www.rabbitmq.com/docs.confirms

## 题目10：RocketMQ 的事务消息是如何实现的？半消息机制（Half Message）的作用是什么？

【答案】

**RocketMQ 事务消息原理**

RocketMQ 事务消息借鉴了 **2PC（两阶段提交）** 的思想，通过"半消息"机制保证本地事务与消息发送的一致性。

**核心流程：**

```
Producer                    Broker                    Consumer
   |                          |                          |
   |  1.发送Half Message       |                          |
   |  ──────────────────────→ |                          |
   |  2.返回 Half OK           |                          |
   |  ←────────────────────── |                          |
   |                          |                          |
   |  3.执行本地事务           |                          |
   |  （在业务DB中操作）        |                          |
   |                          |                          |
   |  4.提交事务状态            |                          |
   |  （COMMIT/ROLLBACK）      |                          |
   |  ──────────────────────→ |                          |
   |                          |                          |
   |                          |  5.若COMMIT：投递真实消息 |
   |                          |  6.若ROLLBACK：删除半消息  |
```

**详细步骤：**

1. **发送半消息（Half Message）**
   - `producer.sendMessageInTransaction()` 先发送一条特殊的"半消息"
   - 半消息**对 Consumer 不可见**，内容与正常消息完全一样，只是状态为"pending"
   - 发送成功后才执行本地事务

2. **执行本地事务**
   - 在 `TransactionListener.execute()` 中执行业务逻辑
   - 返回 `COMMIT_MESSAGE` 表示成功，`ROLLBACK_MESSAGE` 表示失败
   - 返回 `UNKNOWN` 表示状态未知

3. **提交事务状态**
   - **COMMIT**：半消息标记为已提交，对 Consumer 可见
   - **ROLLBACK**：直接丢弃半消息
   - **UNKNOWN**：进入**事务状态回查**

4. **事务回查（Transaction Check）**
   - 如果 Broker 在超时时间内未收到事务状态，触发 `checkLocalTransaction()`
   - Broker 回调 Producer 的 `checkListener`，要求确认事务最终状态

**半消息机制的作用：**
- **隔离性**：半消息对 Consumer 不可见，避免本地事务未提交时消费者就开始处理
- **可恢复**：即使本地事务失败，Broker 也不会投递消息（消息被回滚删除）
- **回查保障**：即使事务状态丢失，Broker 主动回查确保最终一致

## 题目11：什么是 MCP（Model Context Protocol）？它与 Function Calling 有什么区别？AI Agent 如何通过 MCP 扩展工具能力？

【答案】

**MCP（Model Context Protocol）概念**

MCP 是 **Anthropic 于 2024 年末开源**的 AI Agent 工具扩展协议，旨在为 AI 模型提供**标准化的工具调用和上下文获取能力**。它解决了 AI Agent 接入外部工具时的碎片化问题。

**核心架构：**

```
┌─────────────────────────────────────────┐
│          AI Model / Agent               │
│   System Prompt + Tools Schema          │
└──────────────┬──────────────────────────┘
               │  MCP Protocol（JSON-RPC）
┌──────────────▼──────────────────────────┐
│          MCP Host（运行在 Agent 侧）    │
│   MCP Client ←→ MCP Server              │
└──────────────┬──────────────────────────┘
               │  stdio / HTTP / WebSocket
┌──────────────▼──────────────────────────┐
│   MCP Server（每个工具/数据源一个）     │
│   - File System Server                  │
│   - GitHub Server                       │
│   - Database Server                     │
└─────────────────────────────────────────┘
```

**MCP 核心能力：**

1. **Tools（工具调用）**：AI Agent 可调用外部工具，类似 Function Calling
2. **Resources（资源访问）**：AI 可读取外部数据（如文件内容、数据库记录）
3. **Prompts（提示模板）**：预定义的 Prompt 模板，供 Agent 复用

**MCP vs Function Calling 对比：**

| 维度 | MCP | Function Calling（OpenAI） |
|------|-----|---------------------------|
| **标准化程度** | 通用协议，任何模型/平台可用 | OpenAI 专有，各厂商实现不一 |
| **架构** | 客户端-服务器模式，支持多工具聚合 | 单函数调用，需为每个函数定义 schema |
| **状态管理** | 支持服务器推送（streaming） | 纯请求-响应 |
| **工具发现** | 支持动态发现可用工具 | 需预先注册函数列表 |
| **生态** | 快速扩展（GitHub、Filesystem、Slack等已有实现） | 依赖各平台接入 |

**AI Agent 通过 MCP 扩展工具的流程：**

1. **MCP Server 启动**：运行 `npx @anthropic/mcp-server-github` 等，暴露工具接口
2. **MCP Host 配置**：Agent 配置连接这些 Server，获取工具清单（JSON schema）
3. **LLM 推理**：模型根据上下文判断需要调用哪些工具
4. **工具执行**：MCP Host 通过 MCP 协议调用 Server，返回结果给模型
5. **结果整合**：模型整合工具返回结果，继续推理或生成最终回复

## 题目12：AI Agent 的规划能力（Planning）是如何实现的？Chain of Thought、Tree of Thought 和 ReAct 有什么区别？

【答案】

**AI Agent 规划能力概述**

AI Agent 的规划能力是指在复杂任务中，模型能够**分解目标、制定步骤、执行并根据反馈调整计划**。这是"自主 Agent"与"简单问答"的核心区别。

**主流规划方法对比：**

**1. Chain of Thought（CoT，思维链）**
- **原理**：在 Prompt 中要求模型"Think Step by Step"，让模型逐步推理
- **特点**：一次性生成完整推理链，无执行反馈
- **适用**：数学题、逻辑推理等单步推导任务
- **缺陷**：推理链错误时无法自我修正

**2. Tree of Thought（ToT，思维树）**
- **原理**：在每个推理节点，模型生成多个可能的分支，并评估每个分支的可行性
- **特点**：探索式搜索（类似 BFS/DFS），可回溯
- **适用**：写作创意、复杂规划、博弈类问题
- **缺陷**：token 消耗大，响应延迟高

**3. ReAct（Reasoning + Acting）**
- **原理**：模型在推理（Thought）和行动（Action）之间交替：
  ```
  Thought: 分析当前状态，决定下一步行动
  Action: 执行工具调用（如搜索、计算）
  Observation: 观察执行结果
  → 循环直到任务完成
  ```
- **特点**：与外部环境/工具交互，根据观察结果动态调整推理
- **适用**：需要外部知识的复杂任务（搜索、数据库查询、代码执行）
- **优势**：推理过程可追溯，能根据反馈自我修正

**对比总结：**

| 维度 | CoT | ToT | ReAct |
|------|-----|-----|-------|
| 推理方式 | 线性推理 | 树状探索 | 循环推理+行动 |
| 环境交互 | 无 | 无 | 有（通过工具） |
| 自修正能力 | 无 | 部分（有回溯） | 强（每步反馈） |
| token 消耗 | 低 | 高 | 中 |
| 适用场景 | 纯推理任务 | 创意/规划任务 | 复杂工具任务 |

**ReAct 的典型实现（以 LangChain Agent 为例）：**

```python
# 伪代码示例
while not done:
    thought = llm.think(history)      # 模型分析当前状态
    if "search" in thought:
        action = extract_search(thought)
        result = search_api(action)    # 执行搜索
        history += f"Observation: {result}"
    elif "finish" in thought:
        done = True
```

ReAct 通过在 Thought 中明确"为什么这样做"，让推理过程可解释、可纠错，是当前 AI Agent 实现复杂任务的主流方案。

## 题目13：如何用 O(n) 时间复杂度和 O(1) 空间复杂度找到两个链表的第一个公共节点？

【答案】

**双指针法（最优解）**

**核心思路**：设两个链表长度分别为 `m` 和 `n`。让 `p1` 从 A 链表头出发，走完 A 后从 B 头继续走；`p2` 从 B 链表头出发，走完 B 后从 A 头继续走。两者最终会在公共节点相遇。

**数学证明**：
- `p1` 走过的路径长度 = `m + (n - c)`，其中 `c` 是公共部分长度
- `p2` 走过的路径长度 = `n + (m - c)`
- 两者走的总路程相同，且第一次相遇点即为公共节点

**代码实现：**

```java
public ListNode getIntersectionNode(ListNode headA, ListNode headB) {
    if (headA == null || headB == null) return null;

    ListNode pA = headA;
    ListNode pB = headB;

    while (pA != pB) {
        // A走完换B，B走完换A
        pA = (pA == null) ? headB : pA.next;
        pB = (pB == null) ? headA : pB.next;
    }

    return pA; // 相遇点或null
}
```

**图解：**

```
A:    a1 → a2 → a3
                ↘
                  c1 → c2 → c3
                ↗
B:    b1 → b2 → b3 → b4

pA 路径: a1→a2→a3→c1→c2→c3→b1→b2→b3→b4→c1(c2(c3)相遇)
pB 路径: b1→b2→b3→b4→c1→c2→c3→a1→a2→a3→c1(c2(c3)相遇)
```

**复杂度分析：**

| 复杂度 | 分析 |
|--------|------|
| **时间** | 每个指针最多走 m+n 步，O(m+n) |
| **空间** | 只用两个指针，O(1) |

## 题目14：什么是单调栈（Monotonic Stack）？如何用单调栈解决 Next Greater Element 问题？

【答案】

**单调栈核心概念**

单调栈是一种**特殊的数据结构**，栈内元素保持**单调递增**或**单调递减**的顺序。它能在 O(n) 时间复杂度内解决"找每个元素右边/左边第一个比它大的元素"等问题。

**核心思想**：
- 遍历数组，维护一个递增栈（存放下标）
- 当遇到比栈顶更大的元素时，栈顶元素的"Next Greater Element"就找到了
- 被"解除封印"的元素出栈，继续处理

**Next Greater Element 问题：**

给定数组 `nums = [2, 1, 2, 4, 3]`，返回每个元素右边第一个比它大的元素，没有则返回 -1。

**代码实现：**

```java
public int[] nextGreaterElement(int[] nums) {
    int n = nums.length;
    int[] result = new int[n];
    Arrays.fill(result, -1);  // 初始化为-1
    Stack<Integer> stack = new Stack<>();  // 递增栈，存放下标

    for (int i = 0; i < n; i++) {
        // 当前元素比栈顶大，说明栈顶的 Next Greater Element 找到了
        while (!stack.isEmpty() && nums[i] > nums[stack.peek()]) {
            int idx = stack.pop();
            result[idx] = nums[i];
        }
        stack.push(i);
    }
    return result;
}
```

**执行过程：**

```
nums = [2, 1, 2, 4, 3]

i=0, nums[0]=2: stack=[0]
i=1, nums[1]=1: 1<2，不出栈，stack=[0,1]
i=2, nums[2]=2: 2=2，不出栈（严格小于才出栈），stack=[0,1,2]
i=3, nums[3]=4: 4>2→result[2]=4出栈; 4>1→result[1]=4出栈; 4>2→result[0]=4出栈; stack=[3]
i=4, nums[4]=3: 3<4，不出栈，stack=[3,4]

result = [4, 4, 4, -1, -1]
```

**单调栈的变形：**

| 问题类型 | 栈类型 | 出栈时机 |
|---------|--------|---------|
| Next Greater Element | 递增栈 | 当前 > 栈顶 |
| Next Smaller Element | 递减栈 | 当前 < 栈顶 |
| Previous Greater Element | 递增栈 | 当前 > 栈顶（从左往右扫） |

**典型应用场景：**
- 柱状图最大矩形（LeetCode 84）
- 去除重复字母（LeetCode 316）
- 每日温度（LeetCode 739）
- 最短无序连续子数组（LeetCode 581）

## 题目15：什么是字典树（Trie）？它的时间和空间复杂度是多少？请实现一个简单的 Trie 类。

【答案】

**字典树（Trie）核心概念**

字典树（发音 /try/），也称前缀树，是一种树形数据结构，用于高效存储和检索字符串集合。核心思想是**利用字符串的公共前缀来减少查询时间**。

**结构特点：**
- 每个节点代表一个字符
- 从根节点到某一节点的路径，拼接起来就是一个字符串
- 每个节点有多个子节点（对应可能的下一个字符）

**时间/空间复杂度：**

| 操作 | 时间复杂度 | 说明 |
|------|-----------|------|
| 插入 | O(m) | m = 字符串长度 |
| 查找 | O(m) | m = 字符串长度 |
| 前缀查询 | O(m) | m = 前缀长度 |
| 空间 | O(ALPHABET_SIZE × m × n) | 取决于字符集大小和字符串数量 |

**Java 实现：**

```java
class Trie {
    private class Node {
        Node[] children = new Node[26];  // 26个字母
        boolean isEnd = false;            // 是否是单词结尾
    }

    private Node root = new Node();

    // 插入
    public void insert(String word) {
        Node cur = root;
        for (char c : word.toCharArray()) {
            int idx = c - 'a';
            if (cur.children[idx] == null) {
                cur.children[idx] = new Node();
            }
            cur = cur.children[idx];
        }
        cur.isEnd = true;
    }

    // 查找完整单词
    public boolean search(String word) {
        Node cur = root;
        for (char c : word.toCharArray()) {
            int idx = c - 'a';
            if (cur.children[idx] == null) return false;
            cur = cur.children[idx];
        }
        return cur.isEnd;
    }

    // 前缀匹配（以某字符串开头）
    public boolean startsWith(String prefix) {
        Node cur = root;
        for (char c : prefix.toCharArray()) {
            int idx = c - 'a';
            if (cur.children[idx] == null) return false;
            cur = cur.children[idx];
        }
        return true;
    }
}
```

**Trie 的实际应用：**
- **搜索引擎自动补全**：输入前缀，枚举所有以此前缀开头的词
- **IP 路由（最长前缀匹配）**：网络路由表
- **敏感词过滤**：构建敏感词 Trie 树，O(n) 时间扫描文本
- **字符串排序**：按字典序遍历 Trie 即可得到排序结果

**优化方案：**
- 对于大字符集（Unicode），用 HashMap 替代数组存储子节点，节省空间
- 使用 **压缩前缀树（Compressed Trie）** 合并只有单个子节点的链，减少节点数
