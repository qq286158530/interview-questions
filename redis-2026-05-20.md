## 题目1：Redis数据类型与底层结构

**Redis支持哪些数据类型？String类型的底层实现是什么？为什么Redis的String类型既可以存储整数也可以存储字符串？SDS（Simple Dynamic String）相比C语言字符串有哪些优势？**

### 参考答案

Redis 支持五种基本数据类型：**String（字符串）、List（列表）、Hash（哈希）、Set（集合）、Zset（有序集合）**，以及后来新增的 **Stream（流）**。

**String 类型的底层实现**是 **SDS（Simple Dynamic String）**，位于 `src/sds.h/sds.c` 中，并非直接使用 C 原生 char*。Redis 根据字符串长度不同，使用 `sdshdr5/8/16/32/64` 五种头部结构体：

| 类型 | 长度上限 | 用途 |
|------|---------|------|
| sdshdr5 | 32B | 用于长度<=31的短字符串 |
| sdshdr8 | 256B | 存储8位长 |
| sdshdr16 | 64KB | 存储16位长 |
| sdshdr32 | 4GB | 存储32位长 |
| sdshdr64 | 36PB | 存储64位长 |

SDS 相比 C 字符串的核心优势：

1. **O(1) 获取字符串长度**：SDS 在头部存储 `len` 字段，`strlen` 时间复杂度为 O(1)，而 C 字符串需要遍历到 `\0`，为 O(n)。
2. **杜绝缓冲区溢出**：C 字符串拼接/追加时不检查空间，可能溢出；SDS 通过 `free` 字段记录剩余空间，追加前检查并自动扩容。
3. **减少内存重分配次数**：SDS 采用 **空间预分配（pre-allocation）** 和 **惰性空间释放（lazy free）** 策略。追加时若 `free` 足够则直接使用，避免每次修改都调用 `realloc`。
4. **二进制安全**：C 字符串以 `\0` 标识结尾，不适合存储图片等二进制数据；SDS 以 `len` 字段而非 `\0` 判断结尾，可以存储任意二进制数据。
5. **兼容部分 C 字符串函数**：SDS 末尾保留 `\0`，可以直接使用部分 C 字符串函数（如 `printf` `%s`）。

String 类型之所以既能存整数也能存字符串，是因为 Redis 在存储整数时会对值进行判断：如果可以解析为 long long 类型的整数，则使用 `OBJ_ENCODING_INT` 编码，以整数形式直接存储（共享 0~9999 常用整数对象），节省内存；否则使用 SDS 存储。

**来源：**
- Redis 官方文档 - Data Types：https://redis.io/docs/data-types/
- Redis 源码 src/sds.h：https://github.com/redis/redis/blob/unstable/src/sds.h

## 题目2：Redis持久化机制

**Redis有哪些持久化方式？RDB和AOF各自的实现原理、优缺点是什么？如果同时启用了RDB和AOF，Redis重启时使用哪个文件恢复数据？为什么？RDB的bgsave和AOF的rewrite分别是什么时候触发的？**

### 参考答案

Redis 有两种主要持久化方式：

### RDB（Redis Database）

**实现原理：** 通过 `bgsave` 命令 fork 一个子进程，遍历整个内存数据集，将数据序列化后写入 RDB 文件（二进制格式）。父进程继续处理请求。

**触发方式：**
- 手动：`bgsave` / `save`（同步，阻塞）
- 自动：根据 `save m n` 配置（如 `save 900 1` 表示 900 秒内≥1次写操作则触发）
- 关机：`shutdown` 且启用了 RDB
- 主从复制：主节点执行 `bgsave` 后发送 RDB 给从节点

**优点：**
- 单文件，紧凑，备份/恢复极快（恢复整个数据集只加载一个文件）
- 适合做灾难恢复、冷备份（如异地存储）
- fork 子进程，不阻塞主进程（写时复制）

**缺点：**
- 实时性差，两次快照之间数据可能丢失
- fork 过程在数据量极大时会导致短暂阻塞
- 不同 Redis 版本间 RDB 文件格式不完全兼容

### AOF（Append Only File）

**实现原理：** 每执行一条写命令，就以 Redis 协议文本格式追加到 AOF 文件末尾。AOF 文件记录了完整的写操作序列，恢复时重放这些命令即可重建数据。

**rewrite 机制（bgrewriteaof）：** AOF 文件随时间会不断膨胀。rewrite 策略是将当前内存数据用最少的命令重写生成新的 AOF 文件，再替换旧文件。

**触发方式：**
- 手动：`bgrewriteaof`
- 自动：根据 `auto-aof-rewrite-percentage` 和 `auto-aof-rewrite-min-size` 配置（如 100% + 64MB）
- rewrite 期间，父进程继续将新命令追加到旧 AOF 文件，同时将增量缓冲到 AOF rewrite buffer

**优点：**
- 实时性好（可选 `appendfsync` 策略）
- AOF 文件是追加日志，损坏时可修复（redis-check-aof --fix）
- 兼容性好（旧版本协议可顺序重放）

**缺点：**
- 文件体积通常比 RDB 大，恢复速度慢
- 高并发场景下可能产生性能瓶颈（取决于 `appendfsync` 策略）

### appendfsync 策略

| 策略 | 说明 | 安全性 | 性能 |
|------|------|--------|------|
| `always` | 每次写命令同步落盘 | 最高 | 最差 |
| `everysec`（默认） | 每秒同步一次 | 较好 | 较好 |
| `no` | 由操作系统决定 | 最差 | 最好 |

### 混合持久化（Redis 4.0+）

开启 `aof-use-rdb-preamble yes` 后，rewrite 时新 AOF 文件前半部分为 RDB 格式（二进制），后半部分为 AOF 格式（文本）。恢复时优先加载 RDB 部分加速，同时保留增量 AOF 保证数据完整性。

### 重启恢复优先级

**Redis 重启时优先使用 AOF 恢复数据**，因为 AOF 记录更完整（默认情况下），只有当 AOF 关闭时才会使用 RDB。

如果同时开启了 RDB 和 AOF，Redis 使用 **AOF** 文件恢复数据，因为 AOF 的数据更完整。若 AOF 文件损坏或不存在，才会 fallback 到 RDB。

**来源：**
- Redis 官方文档 - Persistence：https://redis.io/docs/management/persistence/
- Redis 官方文档 - Append Only File：https://redis.io/docs/management/optimization/aof/

## 题目3：Redis缓存三大问题

**什么是缓存穿透、缓存击穿、缓存雪崩？各自产生的原因是什么？请分别给出至少2种解决思路（代码或配置层面均可）。**

### 参考答案

### 1. 缓存穿透

**定义：** 查询一个**根本不存在于缓存、也不存在于数据库**的数据。每次请求都穿透到数据库，数据库压力倍增。

**原因：**
- 恶意攻击：攻击者故意用大量不存在的 key 发送请求
- 业务疏忽：查询条件未做校验，错误地查询了不可能存在的数据

**解决思路：**

**① 布隆过滤器（Bloom Filter）**
- 在缓存层前加一层布隆过滤器，所有合法 key 都写入过滤器
- 查询时先检查布隆过滤器，若判定 key 不存在则直接返回，避免查库
- 优点：内存占用极低（使用 bitmap），查询效率高 O(k)
- 缺点：存在误判（false positive），即实际存在但判定为不存在（实际不会发生）

```java
// 布隆过滤器示例（Google Guava）
BloomFilter<String> filter = BloomFilter.create(
    Funnels.stringFunnel(StandardCharsets.UTF_8),
    1000000, 0.01); // 期望插入100万条，误判率1%
filter.put("product:123");
if (!filter.mightContain("product:123")) {
    return null; // 必定不存在，直接返回
}
```

**② 缓存空值（null value）**
- 即使数据库查不到，也将 `null` 或 `"NULL"` 写入缓存，并设置较短 TTL（如 60s）
- 后续相同 key 查询直接命中缓存的空值，避免打爆数据库

**③ 接口层增加校验**
- 对请求参数做基础合法性校验（如 id <= 0 直接拒绝）

---

### 2. 缓存击穿

**定义：** 某个**热点 key 突然失效**，导致大量并发请求同时击穿缓存打到数据库，造成数据库瞬时压力过大。

**原因：** 热点 key 过期的瞬间，大量请求涌入

**解决思路：**

**① 互斥锁（分布式锁）**
- 只允许一个线程重新加载缓存，其他线程等待锁释放后从缓存获取
- 实现：Redis 的 `SETNX` / `SET key value NX EX timeout`

```java
String value = redis.get(key);
if (value == null) {
    String lockKey = "lock:" + key;
    if (redis.set(lockKey, "1", SetParams.nx().ex(10))) {
        // 获取锁成功，从数据库加载数据
        value = db.query(key);
        redis.setex(key, 3600, value);
        redis.del(lockKey);
    } else {
        // 未获取到锁，短暂等待后重试
        Thread.sleep(50);
        return redis.get(key);
    }
}
```

**② 逻辑过期（不设置TTL，用版本号/时间戳判断）**
- key 永不过期，value 中存储 `data + expireTime`
- 查询时判断 `expireTime` 是否已过期
- 若过期则用互斥锁异步重建缓存

```java
CacheEntry entry = redis.get(key); // entry = {data, expireTime}
if (System.currentTimeMillis() > entry.expireTime) {
    // 逻辑过期，异步重建缓存
    if (redis.setnx("rebuild:" + key, "1")) {
        cacheRebuildExecutor.submit(() -> rebuild(key));
    }
}
return entry.data;
```

---

### 3. 缓存雪崩

**定义：** 大量缓存 key 在**同一时间段集中失效**，导致大量请求直接打到数据库，与缓存击穿类似但规模更大。

**原因：**
- 缓存服务宕机
- 大量 key 设置了相同的 TTL，在同一时刻集中过期

**解决思路：**

**① 缓存服务高可用**
- 使用 Redis Cluster/Sentinel 集群，避免单机故障
- 使用 Redis 主从 + 自动故障转移

**② 过期时间随机化**
- 给 key 的 TTL 加上随机偏移量：`TTL = baseTTL + random(0, baseTTL * 0.1)`

```java
int baseTtl = 3600;
int actualTtl = baseTtl + new Random().nextInt(300); // 3600~3900s
redis.setex(key, actualTtl, value);
```

**③ 多级缓存**
- 本地缓存（如 Caffeine/Guava Cache）+ Redis + 数据库
- 即使 Redis 不可用，本地缓存仍可支撑一部分请求

**④ 熔断/限流**
- 使用 Sentinel/Hystrix 对数据库访问做熔断和限流保护
- 请求超过阈值时直接返回降级数据或友好提示

**⑤ 数据库层保护**
- 预热缓存：将热点数据在系统启动或低峰期提前加载到缓存

**来源：**
- Redis 官方文档 - Redis in Action：https://redis.io/docs/
- 美团技术团队 - Redis 缓存雪崩、穿透、击穿：https://tech.meituan.com/2017/03/17/cache.html

## 题目4：Redis集群与高可用

**Redis Cluster的工作原理是什么？它是如何实现数据分片的？Slot（槽）的数量是多少？什么是ASK路由和MOVED重定向？主从复制模式下，如果主节点宕机，从节点如何自动故障转移？Redis Sentinel和Redis Cluster的核心区别是什么？**

### 参考答案

### Redis Cluster 工作原理

Redis Cluster 采用 **去中心化** 架构，所有节点之间通过 Gossip 协议通信，不存在单一的中心节点。

**数据分片：** Redis Cluster 将整个数据集划分为 **16384 个 slot（槽）**。每个 key 通过 `CRC16(key) % 16384` 计算出所属 slot，然后映射到对应节点。

```
slot分布：
节点A：0 ~ 5460
节点B：5461 ~ 10922  
节点C：10923 ~ 16383
```

16384 个 slot 的设计原因：
- slot 数量足够多，能满足数据分片需求
- 节点间心跳包中需要携带所有 slot 信息（2KB 左右的 bitmap），16384 约为 2KB，是网络传输和内存占用的最优平衡点
- 节点数量通常远小于 16384（不宜超过 1000 节点），使用固定 bitmap 而非动态结构更高效

### MOVED重定向 vs ASK重定向

**MOVED 重定向（永久重定向）：**
- 说明该 slot 的负责权已经完全转移到另一个节点
- 客户端收到 MOVED 后，应更新本地 slot -> 节点映射表，下次直接请求正确节点
- 格式：`MOVED <slot> <ip:port>`

**ASK 重定向（临时重定向）：**
- 说明该 slot 正在执行 `cluster setslot migrate` 操作（数据迁移中）
- 当前请求仍然发向旧节点，若该 key 在旧节点存在则返回结果
- 客户端**不会**更新本地映射表，只是本次请求临时转向新节点
- 格式：`ASK <slot> <ip:port>`

### 故障检测与自动故障转移

Redis Cluster 的故障检测由 Gossip 协议实现：
1. 节点间定期互相发送 `PING/PONG` 心跳包
2. 若某个节点超过 `node-timeout` 时间未收到某节点的心跳，则标记为 `PFAIL`（疑似下线）
3. 集群中超过半数主节点都认为某主节点 PFAIL，则该节点被标记为 `FAIL`（客观下线）
4. 该 slot 的从节点发起选举，获得多数投票后升级为新的主节点

**选举机制（类似 Raft）：**
- 从节点发现主节点 FAIL 后，向所有包含该 slot 的主节点发起 `FAILOVER_AUTH_REQUEST` 请求
- 主节点投票：每个 epoch 大的节点优先级高，epoch 越大投票权重越高
- 获得超过半数主节点投票的从节点晋升为新的主节点

### Redis Sentinel vs Redis Cluster

| 维度 | Redis Sentinel | Redis Cluster |
|------|----------------|---------------|
| 定位 | **高可用**：主从自动故障转移 | **分布式**：数据分片 + 高可用 |
| 数据分片 | 无，所有节点存相同数据 | 有，16384 slot 分散到多个节点 |
| 节点角色 | 主节点 + 从节点 + Sentinel 监控节点 | 主节点 + 从节点（每个主节点可有多个从） |
| 客户端连接 | 任意节点均可，驱动自动处理读写分离和故障转移 | 客户端需要知道 slot 分布，只能向正确节点发请求 |
| 扩展性 | 垂直扩展（增加从节点分担读压力） | 水平扩展（增加节点分散 slot） |
| 事务支持 | 支持（单分片内） | 仅支持单 slot 内的操作（`MULTI/EXEC` 跨 slot 会报错） |
| 适用场景 | 小型业务、需要读写分离的场景 | 大型业务、需要水平扩展的场景 |

**Sentinel 的核心职责：** 监控主从节点状态 + 自动故障转移 + 配置中心（客户端订阅通知）

**Cluster 的核心职责：** 数据分片 + 槽迁移 + 节点故障检测和自动转移

**来源：**
- Redis Cluster 官方文档：https://redis.io/docs/management/scaling/
- Redis Cluster 规范：https://redis.io/docs/reference/cluster-spec/

## 题目5：Redis分布式锁

**为什么Redis的SET key value NX EX timeout不是完美的分布式锁？请解释Redis分布式锁的正确实现方式（Redlock算法）以及它面临的问题。SETNX和SET哪个更合适实现分布式锁？为什么？什么是"锁续期"（watchdog）机制？如何用Redisson实现一个可重入锁？**

### 参考答案

### SET key value NX EX timeout 的问题

`SET key value NX EX timeout` 实现了基本的分布式锁，但仍存在以下问题：

1. **值未设置唯一性（原子性不足）**：解锁时用 `DEL key` 直接删除，无法区分是否是自己加的锁——可能误删他人持有的锁。
2. **超时时间固定，无法续期**：任务执行时间超过锁的 TTL，锁自动释放，但任务可能还在执行，导致其他线程获取同一把锁，产生数据竞争。

### 正确的分布式锁实现

#### 第一步：加锁（SETNX + 唯一value）

```java
String lockKey = "order:lock:" + orderId;
String uuid = UUID.randomUUID().toString(); // 唯一标识
Boolean acquired = redis.set(lockKey, uuid, SetParams.nx().ex(30));
```

必须用 **SETNX**（或 `SET NX`）保证原子性：只有一个客户端能成功设置值。

#### 第二步：解锁（Lua脚本保证原子性）

```lua
-- 解锁脚本：只有锁的值等于自己的uuid时才删除
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

```java
public void unlock(String lockKey, String uuid) {
    String script = "if redis.call('GET', KEYS[1]) == ARGV[1] then " +
                    "return redis.call('DEL', KEYS[1]) else return 0 end";
    jedis.eval(script, 1, lockKey, uuid);
}
```

### SETNX vs SET NX

`SET key value NX EX timeout` 是**优于**单独使用 `SETNX` 的：

- `SETNX` 只能设置 key 不存在时才能成功，无法同时设置过期时间
- 若先 `SETNX` 成功后再 `EXPIRE`，在两步之间进程崩溃，则锁永久无法释放
- `SET key value NX EX timeout` 是单条命令，**原子地完成加锁+设置过期时间**，避免中间状态

### Redisson 可重入锁（Watchdog 锁续期机制）

Redisson 使用 **看门狗（Watchdog）** 机制解决锁续期问题：

```java
RLock lock = redissonClient.getLock("order:lock:123");
try {
    // lock.lock() 默认使用 Watchdog 机制
    // 若不指定 leaseTime，则启动 Watchdog：每 10s 自动延长 TTL 为 30s
    // 直到显式 unlock 或客户端崩溃，Watchdog 才停止
    lock.lock();
    // 业务逻辑
} finally {
    lock.unlock();
}
```

```java
// 手动指定 leaseTime（不使用 Watchdog）
lock.lock(10, TimeUnit.SECONDS); // 10秒后自动释放，不续期
```

**Watchdog 工作原理：**
- `lock()` 不传 leaseTime 时生效
- 锁被持有后，Redisson 启动一个定时任务（每 `internalLockLeaseTime / 3`，默认 10s）
- 每次定时任务检测锁是否还被持有，若是则 `PEXPIRE` 延长到 30s
- `unlock()` 时取消定时任务

### Redlock 算法（多节点分布式锁）

Redis 作者 Antirez 提出的 Redlock 算法：**在 N 个 Redis 节点上各获取锁，获取到多数节点成功才算加锁成功**。

```
// Redlock 步骤（N=5为例）
1. 获取当前时间 T1（毫秒）
2. 依次向 5 个节点发送：SET lock_key unique_id NX PX timeout
3. 计算从 T1 到获得所有响应的时间 T2
4. 若 T2 < timeout/2，且成功获取 >= 3 个节点的锁，则锁有效
5. TTL = 原始TTL - T2（扣除获取锁的耗时）
6. 若获取失败，向所有节点发送 DEL 释放锁
```

**Redlock 的问题（争议）：**
- 假设 5 节点中有 2 个发生时钟跳跃（clock jump），可能导致锁提前过期释放
- 依赖系统时间单调性，分布式环境中难以保证
- Martin Kleppmann 等人质疑其安全性：https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html
- Antirez 的反驳和修正：https://antirez.com/news/101

### 生产环境建议

| 场景 | 推荐方案 |
|------|---------|
| 单机 Redis | SET NX + Lua + Watchdog（Redisson） |
| 多节点 | Redisson RedLock（注意时钟问题）或使用 **ZooKeeper / etcd** 实现分布式锁 |
| 追求强一致 | 使用 etcd（如 go-zero 框架中的分布锁）或数据库唯一索引 |
