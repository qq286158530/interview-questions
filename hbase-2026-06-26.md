## 1. HBase 的架构原理是什么？请描述 HBase 的核心组件及其作用。

【答案】

HBase 是一个基于 Hadoop 的分布式列式存储数据库，其架构核心组件包括：

**ZooKeeper：**
- 协调服务，维护 Meta 表位置信息
- 监控 RegionServer 上下线
- 实现分布式锁等协调功能

**HMaster：**
- 负责表的 DDL 操作（创建、删除表）
- 管理 RegionServer 的负载均衡，分配 Region
- 处理 Schema 变更请求
- 监控 RegionServer 健康状态

**RegionServer：**
- 负责数据读写，是真正存储数据的节点
- 管理 MemStore 和 HFile
- 处理客户端的读写请求
- 合并 HFile 以优化存储

**Region：**
- HBase 表的水平分片，每个 Region 包含 [startKey, endKey) 的数据
- 一个表可以有多个 Region，分布在不同的 RegionServer 上

**Store：**
- 每个列族对应一个 Store
- 包含 MemStore 和多个 HFile

**MemStore：**
- 写缓存，先将数据写入 MemStore
- 达到阈值后 flush 成 HFile

**HFile：**
- HBase 的存储文件格式
- 基于 SSTable 结构，支持高效读取

**WAL（Write-Ahead Log）：**
- 预写入日志，用于数据恢复
- 先写 WAL，再写 MemStore

数据读写流程：
- 写：Client → ZooKeeper → RegionServer → WAL → MemStore → flush → HFile
- 读：Client → ZooKeeper → Meta表定位 → RegionServer → MemStore/HFile 合并读取

📖 来源：[HBase 官方文档](https://hbase.apache.org/book.html)

## 2. HBase 的 RowKey 设计应该遵循哪些原则？如何优化 RowKey 的设计？

【答案】

RowKey 设计是 HBase 性能优化的核心，需要遵循以下原则：

**1. 长度原则：**
- RowKey 长度建议在 16-100 字节之间
- 越短越好，减少存储开销
- 避免使用过长的字符串作为 RowKey

**2. 散列原则：**
- RowKey 必须实现高离散度，避免热点
- 反例：时间戳作为 RowKey 前缀会导致热点写入
- 正例：使用 MD5、哈希或盐值打散

**3. 唯一性原则：**
- RowKey 必须唯一标识一行数据
- 类似于主键的概念

**4. 有序性原则：**
- HBase 按 RowKey 字典序排序
- 合理设计可利用顺序读性能

**常见优化方案：**

**方案一：盐值（Salting）**
```
RowKey = MD5(key) % N + original_key
```
将数据分散到 N 个 Region，避免热点

**方案二：哈希打散**
```
RowKey = MD5(original_key) + original_key
```
保证高离散度，同时保留原始信息

**方案三：时间戳反转**
```
RowKey = Long.MAX_VALUE - timestamp + other_info
```
新数据排在前面，适合查询最新数据的场景

**方案四：业务组合**
```
RowKey = userId + orderId + timestamp
```
根据业务查询模式组合多个字段

**避免热点技巧：**
- 不要以时间戳或连续数字开头
- 在高并发写入场景使用随机前缀
- 使用雪花算法生成 ID

## 3. 请描述 HBase 的 MemStore 和 HFile 的工作机制，以及它们之间的关系。

【答案】

**MemStore 工作机制：**

1. **写入流程：**
   - 数据先写入 WAL（Write-Ahead Log）
   - 然后写入 MemStore（内存结构）
   - MemStore 是 SortedMap，按 RowKey+Column+Version 排序

2. **Flush 触发条件：**
   - MemStore 大小达到 `hbase.hregion.memstore.flush.size`（默认 128MB）
   - Region 内所有 MemStore 总大小超过 `hbase.regionserver.global.memstore.size`（默认 40%堆内存）
   - WAL 文件数量超过 `hbase.regionserver.max.logs`
   - 定期 flush（`hbase.regionserver.optionalcacheflushinterval`，默认 1 小时）
   - 手动执行 flush

3. **Flush 过程：**
   - 将 MemStore 内容写出为 HFile
   - 过程原子化，保证数据一致性
   - Flush 期间仍可接收写入

**HFile 工作机制：**

1. **HFile 结构：**
   - 基于 Google SSTable（Sorted String Table）
   - 包含 Data Block、Index Block、Bloom Filter、Meta Block 等

2. **HFile 索引：**
   - 每个 HFile 有根索引和中间索引
   - Bloom Filter 快速判断数据是否存在
   - 支持高效的随机读取

3. **HFile 合并（Compaction）：**
   - Minor Compaction：合并部分小 HFile（可配置参数控制）
   - Major Compaction：合并一个 Store 下所有 HFile，删除过期数据和删除标记
   - 减少 HFile 数量，提升读性能

**MemStore 与 HFile 关系：**

```
写入：WAL → MemStore → (flush) → HFile
读取：MemStore + HFile → 合并返回
```

- MemStore 是写缓存，HFile 是持久化存储
- 读取时需要合并 MemStore 和所有 HFile 的数据
- MemStore 达到阈值后 flush 成 HFile
- 随着 HFile 增多，触发 Compaction 合并

**数据一致性保证：**
- WAL 保证写入可靠性
- Region 恢复时通过 WAL 重放未 flush 的数据

## 4. HBase 协处理器（Coprocessor）是什么？有哪些类型？各自的使用场景是什么？

【答案】

**协处理器概述：**

HBase 协处理器是一种扩展机制，允许用户在 RegionServer 上运行自定义代码，将计算逻辑下推到数据所在节点执行，类似于数据库的触发器和存储过程。

**两种类型：**

**1. Observer 协处理器（观察者）**

类似数据库触发器，基于钩子回调实现：

| 类型 | 钩子时机 | 使用场景 |
|------|----------|----------|
| RegionObserver | GET/PUT/DELETE/SCAN 等操作前后 | 数据验证、访问控制、列值修改 |
| MasterObserver | 创建/删除表、修改表结构等操作前后 | 审计日志、权限控制、级联操作 |
| WALObserver | WAL 写入前后 | WAL 分析、复制拦截 |

**示例场景：**
```java
// 创建二级索引
public class IndexCoprocessor extends RegionObserver {
    @Override
    public void postPut(ObserverContext<RegionCoprocessorEnvironment> c,
                       Put put, WALKey walKey, WALEdit walEdit) {
        // 在 PUT 成功后，自动写入索引表
    }
}
```

**使用限制：**
- 观察者方法不能显示修改授受的参数
- 可通过 Hook 参数间接修改（如预分区）

**2. Endpoint 协处理器（端点）**

类似存储过程，允许在服务器端执行自定义计算：

- 动态添加 RPC 协议
- 服务端执行聚合计算
- 客户端调用获取结果

**经典应用场景：**
```java
// 服务端实现
public class SumAlgorithm implements Coprocessor, AggregationProtocol {
    @Override
    public Long getSum(byte[] column, Scan scan) {
        // 在 RegionServer 端执行 Sum 计算
        // 只返回聚合结果，减少网络传输
    }
}
```

**使用场景对比：**

| 场景 | 推荐类型 | 说明 |
|------|----------|------|
| 二级索引 | Observer | 数据写入时自动更新索引表 |
| 聚合查询 | Endpoint | 在服务端预聚合，减少数据传输 |
| 权限控制 | Observer | 读写前验证权限 |
| 审计日志 | Observer | 操作前后记录日志 |
| 自动分页 | Endpoint | 服务端实现分页逻辑 |

**加载方式：**
1. **配置加载**（所有表）：`hbase.coprocessor.user.classes` 在 hbase-site.xml
2. **表级加载**（指定表）：通过 HTableDescriptor 的 `addCoprocessor()` 方法

**注意事项：**
- 协处理器在 RegionServer 上执行，异常会影响 Region 可用性
- 避免执行耗时过长的操作
- 注意并发控制和资源占用

📖 来源：[HBase Coprocessor 深度解析](https://www.cnblogs.com/foxmailed/p/7357485.html)

## 5. HBase 的读写流程是怎样的？请详细描述数据从客户端到服务器的完整路径。

【答案】

**写数据流程（Put）：**

**Step 1：定位目标 Region**
```
Client → ZooKeeper → /hbase/meta-region-server → Meta表位置
Client → Meta表 → 查询 RowKey 属于哪个 Region → 缓存 Region 位置
```

**Step 2：写入 WAL**
- 数据写入 RegionServer 的 WAL（Write-Ahead Log）
- WAL 顺序写入，保证持久化
- 如果写入失败，数据可从 WAL 恢复

**Step 3：写入 MemStore**
- 数据写入 Region 对应的 MemStore
- MemStore 内部按 RowKey+Column+Version 排序
- 写入 MemStore 即返回成功（异步 flush 到 HFile）

**Step 4：响应客户端**
- RegionServer 返回写入成功确认
- 如果配置了 `autoflush`，则需等 MemStore flush 后才返回

**流程图：**
```
Client.put("row1", "cf1", "col1", "value1")
       ↓
  [定位 Region: ZooKeeper + Meta]
       ↓
  [写入 WAL: 先持久化]
       ↓
  [写入 MemStore: 内存排序结构]
       ↓
  [返回成功给 Client]
```

**读数据流程（Get/Scan）：**

**Step 1：定位目标 Region**（同写入）

**Step 2：并行查询**
- 同时查询 MemStore 和 HFile
- MemStore：最近写入的未 flush 数据
- HFile：已持久化的历史数据

**Step 3：合并结果**
- 根据时间戳（version）合并数据
- 应用删除标记（Delete Marker）
- 返回合并后的完整数据

**Get 流程：**
```
Client.get("row1")
       ↓
  [定位 Region]
       ↓
  [读取 MemStore: 检查 BlockCache]
       ↓
  [读取 HFile: BloomFilter 定位 → BlockIndex → Data]
       ↓
  [合并结果，返回最新版本]
```

**Scan 流程：**
```
Client.scan("row1", "row100")
       ↓
  [定位起始 Region]
       ↓
  [依次扫描 MemStore + HFile]
       ↓
  [返回 ResultScanner（迭代器）]
```

**核心优化机制：**

**1. BlockCache（读缓存）：**
- LRU 策略的读缓存
- 热点数据缓存加速
- 配置：`hfile.block.cache.size`（默认堆内存 40%）

**2. MemStore（写缓存）：**
- 批量内存排序
- flush 时生成 HFile

**3. BloomFilter（布隆过滤器）：**
- 快速判断 HFile 是否包含某 RowKey
- 减少不必要的 HFile 扫描

**4. 缓存查询结果：**
- Scan 结果默认不缓存
- 可通过 `setCacheBlocks(true)` 缓存

**写入一致性：**
- 强一致性：写操作必须在所有副本都写入后才返回
- HBase 0.98 后支持乐观锁（CheckAndPut）

**数据读取优先级：**
```
1. MemStore（最新数据）
2. BlockCache（热点数据）
3. HFile（历史数据，按时间倒序合并）
```
