## 题目一：Spark DAG 是如何生成的？Stage 是如何划分的？

### 参考答案

**DAG 生成过程：**

1. **RDD 依赖链构建**：当代码执行 `rdd.map(...).filter(...).groupByKey()` 时，Spark 根据 Action 算子向前回溯，构建由窄依赖（或宽依赖）连接而成的 RDD 依赖链。

2. **DAG 划分**：Spark Scheduler 根据 RDD 之间的依赖关系（Dependency）将 DAG 划分为多个 Stage。宽依赖（如 `groupByKey`、`join`、`sort`）是 Stage 划分的边界，窄依赖（如 `map`、`filter`）可以在同一个 Stage 内流水线执行。

3. **Stage 类型**：
   - **ResultStage**：包含最后一个 RDD 及对应 Action 算子的 Stage，是 DAG 的终点。
   - **ShuffleMapStage**：为下游 Stage 提供 shuffle 数据的 Stage，输出会写入磁盘。

**Stage 划分算法简述：**
从 DAG 的末尾（ResultStage）出发，从后向前遍历 RDD 依赖链，遇到宽依赖则新建一个 Stage，直到所有 RDD 都归属某个 Stage。最终得到一个 Stage 组成的 DAG，提交给 TaskScheduler 分配 Task 并行执行。

## 题目二：RDD、DataFrame、DataSet 三者的区别与联系？

### 参考答案

| 特性 | RDD | DataFrame | DataSet |
|------|-----|-----------|---------|
| **数据结构** | 分布式元素集合（无结构） | 结构化数据（二维表格，有 schema） | 结构化数据（强类型 JVM 对象） |
| **Schema** | 无 | 有（列名 + 列类型） | 有（编译时检查） |
| **数据不可变性** | 不可变 | 不可变 | 不可变 |
| **优化方式** | 无 Catalyst 优化 | Catalyst 优化器 + Tungsten | 同 DataFrame |
| **API 风格** | 函数式算子 | SQL-like / DSL | Lambda + SQL |
| **序列化** | Java/Kryo | Tungsten（off-heap） | Tungsten（encoder） |
| **创建方式** | `parallelize`、`textFile` | `spark.read.parquet(...)` 等 | `ds.as[CaseClass]` |
| **缺点** | 无结构，GC 开销大 | 运行时才检查类型 | 需 case class，API 较新 |

**三者联系：**
- 底层实现上，**DataFrame 和 DataSet 的逻辑计划最终都转换为 RDD 执行**，即它们是 RDD 的高层抽象。
- `DataFrame = DataSet[Row]`（Row 是通用行对象，无类型），`DataSet[T]` 是带类型 Encoder 的版本。
- DataFrame 享受 Catalyst 优化器和 Tungsten 代码生成，性能普遍优于手写 RDD 算子。

## 题目三：Spark Shuffle 的工作原理是什么？为什么 Shuffle 是性能瓶颈？

### 参考答案

**Shuffle 机制：**

Shuffle 是 Spark 将数据按照 Key 进行重新分区的过程，发生在需要跨节点移动数据的算子处（如 `groupByKey`、`join`、`sort`、`reduceByKey`）。

**执行流程（以 ShuffleMapStage 为例）：**

1. **Map 端**：每个 Task 计算出 (partitionId, data) 键值对，按 partitionId 写入本地磁盘文件（同时生成 index 文件），此过程称为 **Shuffle Write**。

2. **Reduce 端**：下游 Stage 的 Task 根据 shuffle 数据所在的 executor，通过网络抓取（fetch）对应 partition 的数据，写入内存缓冲区，完成聚合/排序，此过程称为 **Shuffle Read**。

3. **磁盘溢写**：如果 Map 端数据量超过阈值，数据会先写入磁盘（sort-based shuffle，Spark 1.2+ 默认）。

**为什么是性能瓶颈？**

- **磁盘 I/O**：Shuffle 数据先落盘再网络传输，磁盘读写成为主要延迟来源。
- **网络 I/O**：大量数据跨节点传输，消耗网络带宽。
- **序列化开销**：数据需要 Kryo/Java 序列化/反序列化。
- **内存压力**：Reduce 端聚合需要将数据放入内存，大数据量易导致 OOM 或频繁 GC。
- **数据倾斜**：个别分区数据量远超其他，造成"木桶效应"。

**优化手段：**
- 合理设置 `spark.shuffle.sort.bypassMergeThreshold`、`spark.sql.shuffle.partitions`
- 使用 `reduceByKey` 替代 `groupByKey`（先局部聚合减少数据量）
- 避免不必要的数据倾斜（加盐、打散 Key）

## 题目四：Spark 的内存管理模型是怎样的？堆内与堆外内存如何划分？

### 参考答案

Spark 2.x 采用 ** Unified Memory Management**（统一内存管理），将执行内存与存储内存打通，但保留动态调度机制。

**内存区域划分（Executor 角度）：**

```
┌──────────────────────────────────────┐
│            JVM 堆内存                  │
│  ┌────────────┬────────────────────┐  │
│  │  存储内存   │    执行内存         │  │
│  │(Storage)   │  (Execution)       │  │
│  │  缓存RDD   │  Shuffle/排序/聚合  │  │
│  │  广播变量   │  等临时对象         │  │
│  └────────────┴────────────────────┘  │
│  ┌──────────────────────────────────┐  │
│  │  用户内存 (User Memory) ~300MB   │  │
│  │  用户数据结构、内部元数据          │  │
│  └──────────────────────────────────┘  │
│  ┌──────────────────────────────────┐  │
│  │  预留内存 (Reserved) ~300MB       │  │
│  └──────────────────────────────────┘  │
└──────────────────────────────────────┘
```

**堆外内存（Off-Heap）：**
- 通过 `spark.memory.offHeap.enabled=true` 开启
- 主要用于 Tungsten 物理执行页（page），绕过 JVM 堆，减少 GC
- 大小由 `spark.memory.offHeap.size` 控制

**核心参数：**
| 参数 | 默认值 | 说明 |
|------|--------|------|
| `spark.memory.fraction` | 0.6 | 堆内用于 Storage+Execution 的比例 |
| `spark.memory.storageFraction` | 0.5 | Storage 占比上限 |
| `spark.executor.memory` | — | Executor 总堆内存 |
| `spark.memory.offHeap.enabled` | false | 是否启用堆外内存 |
| `spark.memory.offHeap.size` | 0 | 堆外内存大小 |

**动态调度规则：**
- Storage 内存不足时，可借用 Execution 内存（正在执行 Task 时被逐出）
- Execution 内存不足时，可驱逐 Storage 内存（但 Storage 正在缓存时不可抢占）
- 通过 `storageRegionRatio` 控制初始比例

**来源：**  
- Apache Spark 官方文档 - Memory Management：https://spark.apache.org/docs/latest/tuning.html#memory-management-overview

## 题目五：广播变量（Broadcast Variable）的作用是什么？有哪些使用注意事项？

### 参考答案

**作用：**
广播变量将一份**只读数据**缓存到每个 Executor 的内存中，供该 Executor 上的所有 Task 共享访问，避免该数据在每个 Task 中重复传输和存储，从而大幅降低网络传输量和内存占用。

典型使用场景：
- 大型配置表 / 维度表（Lookup 表）
- ML 模型的特征向量、权重
- 不变的中间结果

**使用方式：**
```scala
// Scala
val broadcastVar = sc.broadcast(Array(1, 2, 3))
val result = rdd.map(x => x * broadcastVar.value(0))

// Python
broadcastVar = sc.broadcast([1, 2, 3])
result = rdd.map(lambda x: x * broadcastVar.value[0])
```

**核心机制：**
1. Driver 端调用 `broadcast()` 时，数据被序列化并分片
2. 每个 Executor 首次访问时从 Driver 拉取对应分片，存入 `BlockManager`
3. 同一 Executor 后续 Task 共享该缓存，无需重复拉取

**注意事项（面试高频追问）：**

1. **只读不可修改**：广播变量一旦创建不可更改（Spark 不保证修改可见性），若需更新应重新广播。

2. **大小控制**：广播过大的变量反而会占用大量内存甚至 OOM，Spark 默认广播阈值由 `spark.sql.autoBroadcastJoinThreshold`（默认 10MB）控制。

3. **序列化**：广播变量需要可序列化（Kryo/Java），数据过大会导致序列化耗时过长。

4. **广播策略**：Spark 2.x 使用 `TorrentBroadcast`（基于 BitTorrent 的去中心化广播），避免 Driver 单点瓶颈。

5. **不要广播 RDD**：广播 RDD 只能缓存其引用，不会将数据预先拉到 Executor，应使用 `collect()` 收集到 Driver 再广播。

6. **分区策略**：广播数据按 `spark.broadcast.blockSize`（默认 4MB）分片，影响存储和传输效率。

**来源：**  
- Apache Spark 官方文档 - Broadcast Variable：https://spark.apache.org/docs/latest/rdd-programming-guide.html#broadcast-variables
