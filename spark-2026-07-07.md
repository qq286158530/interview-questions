# Spark 面试题（2026-07-07）

## 题目一：Spark 架构与核心组件

**问题：请描述 Apache Spark 的整体架构架构，以及 Driver、Executor、Master、Worker 的作用与相互协作关系。**

### 答案

Spark 采用 **主从架构**，核心组件如下：

| 组件 | 角色 | 职责 |
|------|------|------|
| **Driver（驱动器）** | 主节点 | 解析用户代码，构建 DAG 图，将作业拆分为多个 Stage 的 TaskSet，分发到 Executor |
| **Executor（执行器）** | 从节点 | 驻留在 Worker 节点上，负责执行 Task，向 Driver 汇报结果，拥有独立的 JVM 进程 |
| **Master** | 资源管理主节点 | 协调整个集群的资源分配，管理 Worker 节点的存活状态 |
| **Worker** | 资源管理从节点 | 向 Master 注册自己的资源（CPU、内存），启动和管理 Executor 进程 |

**作业执行流程：**

1. 用户提交 Spark 应用程序，通过 `spark-submit` 向集群发送请求
2. Driver 启动后，创建 `SparkContext`，向 Master 注册申请资源
3. Master 让 Worker 节点启动 Executor（每个应用独立分配 Executor 进程）
4. Executor 向 Driver 反向注册，之后 Driver 开始执行用户代码
5. SparkContext 将代码解析为 DAG，根据 **窄依赖/宽依赖** 划分 Stage，将 Task 分发到各 Executor 并行执行
6. Task 执行完成后，结果逐层汇总至 Driver，Driver 销毁，作业结束

**常考追问：Spark 的 Cluster Manager 有哪些？有什么区别？**

- **Standalone**：Spark 自带，轻量级，适合小规模集群
- **YARN**：复用企业已有 Hadoop 生态，支持 FIFO/Fair 调度，适合资源统一管理
- **Mesos**：更通用的资源调度，支持粗粒度和细粒度两种模式
- **Kubernetes**：云原生趋势，支持容器化弹性伸缩，Spark 2.3+ 开始支持

> **来源参考**：[Apache Spark 官方文档 - Architecture](https://spark.apache.org/docs/latest/cluster-overview.html)

## 题目二：RDD 详解与血统（Lineage）机制

**问题：什么是 RDD？为什么 Spark 要设计 RDD？RDD 的血统（Lineage）机制是如何实现容错的？**

### 答案

**RDD（Resilient Distributed Dataset）** 是 Spark 最核心的数据抽象，表示一个弹性的、分区的、可并行操作的数据集合。

**为什么设计 RDD：**
- **Immutable（不可变）**：保证数据一致性，天然支持并发
- **Lazy Evaluation（惰性求值）**：所有 transformations 不立即执行，直到遇到 Action 才触发计算，大幅减少中间结果落盘
- **Partitioned（分区）**：数据分布在集群节点上，支持并行处理
- **Fault Tolerance（容错）**：通过血统机制而非数据复制实现高效容错

**血统（Lineage）机制 — 如何实现容错：**

RDD 记录了它的"血缘关系"——从初始数据源经过哪些 transformations 形成。当某个 Partition 丢失时：

```
RDD 血缘链（Lineage）
┌─────────────────────────────────────────────────────┐
│ textFile → flatMap → map → reduceByKey → collect   │
└─────────────────────────────────────────────────────┘
```

1. **窄依赖（Narrow Dependency）**：父 RDD 的每个 Partition 最多被子 RDD 的一个 Partition 使用（如 `map`、`filter`）。一个节点故障只需重算该分支，代价小。
2. **宽依赖（Wide Dependency）**：父 RDD 的一个 Partition 被多个子 Partition 使用（如 `groupByKey`、`join`）。形成 **Shuffle**，需要等待父 Partition 全部完成。

当某个 Partition 丢失时，Spark 从源头重新计算丢失的 Partition，沿 Lineage 链回溯。对于窄依赖可以精确到具体 Partition 就近重算；对于宽依赖（Shuffle）则需要重新计算父 RDD 的全部 Partition，这也是为什么 Shuffle 是性能瓶颈。

**为什么不使用数据复制（Checkpoints）容错？**
数据复制开销大（2x 存储），而 Lineage 链重算代价通常很小（尤其是窄依赖链），是更经济的容错策略。

> **来源参考**：[RDD Paper - Matei Zaharia et al.](https://www.usenix.org/system/files/conference/nsdi12/nsdi12-final138.pdf)

## 题目三：DataFrame 与 DataSet 的区别，以及与 RDD 的关系

**问题：Spark 中 DataFrame 和 DataSet 有什么区别？它们与 RDD 之间的关系是什么？何时该用 DataFrame/DataSet 而不是 RDD？**

### 答案

**三者关系：**

```
RDD[Row]  ←→  DataFrame  ←→  DataSet[T]
```

在 Spark 1.x 中三者各自独立；在 Spark 2.x 统一为 **Dataset API**，DataFrame 成为 `Dataset[Row]` 的别名。

| 特性 | RDD | DataFrame | DataSet |
|------|-----|-----------|---------|
| **数据类型** | 任意类型（无schema） | `Dataset[Row]`，固定 Schema | 强类型 `Dataset[T]` |
| **Schema** | 无 | 有（列名+类型） | 有（编译期检查） |
| **编译期检查** | 无 | 无（运行时才发现错误） | 有（编译期发现） |
| **性能优化** | 无自动优化 | Catalyst 优化器 + Tungsten | 同 DataFrame |
| **序列化** | Java/Kryo 手动序列化 | OFF HEAP 内存 + Tungsten | 同 DataFrame |
| **操作方式** | 函数式编程 | SQL-like / DSL | Lambda + DSL |

**为什么 DataFrame/DataSet 性能更好？**

1. **Catalyst 优化器**：将用户代码解析为 Logical Plan → Optimized Logical Plan → Physical Plan，通过谓词下推（Predicate Pushdown）、列剪裁（Column Pruning）等规则减少不必要的数据读写
2. **Tungsten 物理执行引擎**：二进制的内存布局（OFF HEAP）、cache-aware 计算、无锁对象头、code generation 等
3. **Spark SQL 统一引擎**：一处优化，所有语言（Python/Scala/Java/R/SQL）都能受益

**何时用哪个？**

- **RDD**：需要底层精细控制、处理非结构化数据、依赖第三方序列化库
- **DataFrame**：用 Python/Scala 快速处理结构化数据，需要 SQL 能力
- **DataSet**：用 Scala/Java，需要编译期类型安全，同时享受 SQL 优化

**常考追问：DataFrame 的 Join 和 RDD 的 cogroup + flatMap 实现 Join 有什么区别？**

DataFrame 的 join 会被 Catalyst 优化，选择 **Broadcast Join**（小表广播）或 **Sort Merge Join**（大表），而 RDD 的 cogroup 是粗暴的 Shuffle，效率差距极大。

> **来源参考**：[Spark SQL, DataFrames and Datasets Guide](https://spark.apache.org/docs/latest/sql-programming-guide.html)

## 题目四：Shuffle 机制详解与优化策略

**问题：什么是 Shuffle？为什么 Shuffle 是 Spark 作业的性能瓶颈？请列举 Shuffle 优化的常用手段。**

### 答案

**什么是 Shuffle：**

Shuffle 是 Spark 将数据按某种规则**重新分区**的过程。当一个 RDD 的某个 Partition 需要依赖父 RDD 多个 Partition 的数据时，必须跨节点拉取数据——这就是 Shuffle。

**触发 Shuffle 的算子（宽依赖）：**
`groupByKey`、`reduceByKey`、`join`、`cogroup`、`distinct`、`repartition`、`coalesce(..., shuffle=true)` 等。

**为什么是性能瓶颈：**

```
Shuffle 过程：
[Map Stage] → 写磁盘（Map端落盘） → 网络传输（Shuffle Read） → [Reduce Stage]

         ┌─ 磁盘 I/O（写放大）
瓶颈：  ├─ 网络 I/O（跨节点传输）
         ├─ 内存压力（Map 端缓冲区 + Reduce 端缓存）
         └─ 磁盘空间消耗
```

Shuffle 产生的原因：需要按 Key 重新分组数据，但具有相同 Key 的数据起初可能分散在不同的节点和 Partition 中。

**常用优化策略：**

| 优化手段 | 说明 |
|----------|------|
| **Broadcast Join** | 小表（<10MB）广播到每个 Executor，避免 Shuffle，O(n) → O(1) |
| **合理设置分区数** | `spark.sql.shuffle.partitions`（默认200），数据量小则减少分区避免小任务；数据量大可增加 |
| **减少 Shuffle 数据量** | 先 `filter` 再 `join`（谓词下推），避免不必要的数据参与 Shuffle |
| **使用 Repartition 和 Coalesce** | `repartition` 触发 Shuffle 重新分区；`coalesce(shuffle=false)` 缩小分区但不触发 Shuffle（仅限缩小分区） |
| **开启 Shuffle Compression** | `spark.shuffle.compress=true`（默认开启），使用 Snappy/LZF 压缩 |
| **调优 Shuffle Fetch 缓冲区** | `spark.reducer.maxSizeInFlight`（默认 48MB），增大可减少网络往返次数 |
| **使用 Sort Shuffle** | Spark 2.x 以后默认使用 Sort Shuffle（替代 Hash Shuffle），减少文件数 |
| **避免 groupByKey** | 用 `reduceByKey`/`aggregateByKey` 替代，先在 Map 端做预聚合，大幅减少传输数据量 |

> **来源参考**：[Understanding Shuffle in Spark](https://spark.apache.org/docs/latest/rdd-programming-guide.html#shuffle-operations)

## 题目五：Spark 广播变量与内存管理机制

**问题：什么是广播变量（Broadcast Variable）？Spark 的内存管理模型是怎样的？Executor 内存是如何分配的？**

### 答案

**广播变量（Broadcast Variable）：**

广播变量是只读的、封装了常量或查找表的变量，Spark 会将其**缓存在每个 Executor 的内存**中，而非每个 Task 都要传输一份。

```
Without Broadcast：每个 Task 都传输一份 lookup table → N 个 Task = N 倍网络开销
With Broadcast：   每个 Executor 存一份，所有 Task 共享   → N 个 Task = 1 倍网络开销
```

使用方式：
```scala
// 广播
val broadcastVar = sc.broadcast(Array(1, 2, 3))

// 使用
val result = data.map(x => broadcastVar.value.mkString)
// 销毁（可选）
broadcastVar.unpersist()
```

**典型使用场景：** 大维度表 Join（事实表 join 维度表）、机器学习特征工程中的特征映射表。

**Spark 内存管理模型（统一内存管理，Spark 1.6+）：**

Executor 内存分为两大区域（可动态调节比例）：

```
┌──────────────────────────────────────────────────┐
│                  Executor Memory                  │
│  ┌─────────────────┐    ┌─────────────────────┐  │
│  │   Execution     │    │      Storage        │  │
│  │   Memory        │ ↔  │      Memory         │  │
│  │  (执行内存)      │    │   (存储内存/缓存)    │  │
│  │  - Shuffle      │    │  - RDD Cache        │  │
│  │  - Sort         │    │  - Broadcast        │  │
│  │  - Hash Join    │    │  - 元数据           │  │
│  └─────────────────┘    └─────────────────────┘  │
│                                                  │
│  ┌─────────────────┐    ┌─────────────────────┐  │
│  │   User Memory   │    │   Reserved Memory   │  │
│  │  (用户内存)      │    │   (保留内存 300MB)   │  │
│  └─────────────────┘    └─────────────────────┘  │
└──────────────────────────────────────────────────┘
```

- **Execution Memory**：执行时用的内存（Shuffle、Sort、Hash Join 等），可以借用 Storage 内存
- **Storage Memory**：RDD 缓存和广播变量，可以借用 Execution 内存（当 Execution 不足时）
- **User Memory**：用户代码（UDF 等）使用的内存
- **Reserved Memory**：保留内存，固定 300MB，用于系统预留

**内存动态共享规则（统一内存）：**
- Execution 和 Storage 之间可以相互借用空闲内存
- Storage 可以将空闲内存借给 Execution，但 Execution 必须保证可以归还（不能强制驱逐正在使用的 Storage 块）
- 当 Execution 内存不足时，若 Storage 有空闲，Execution 可以强制占用；但若 Storage 需要回收空间时，Execution 正在使用则无法回收，可能导致 OOM

**内存相关调优参数：**
| 参数 | 默认值 | 说明 |
|------|--------|------|
| `spark.memory.fraction` | 0.6 | 堆内存中用于 Execution + Storage 的比例 |
| `spark.memory.storageFraction` | 0.5 | Storage 占统一内存的比例 |
| `spark.executor.memory` | — | Executor 总堆内存大小 |
| `spark.driver.memory` | — | Driver 内存大小 |

> **来源参考**：[Spark Memory Management](https://spark.apache.org/docs/latest/tuning.html#memory-management)
