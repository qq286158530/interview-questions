## 题目一：Flink 的架构是怎样的？请描述 JobManager 和 TaskManager 的核心作用。

### 参考答案

Flink 采用 **主从架构**，由 JobManager（主节点）和 TaskManager（从节点）组成。

**JobManager（JM）核心职责：**
- **JobMaster**：负责单个 JobGraph 的执行，协调 checkpoint、故障恢复等。
- **ResourceManager**：管理 TaskManager 的 slot（插槽）资源，与 YARN/K8s 等资源管理器交互。
- **Dispatcher**：接收用户提交的 Job，为每个 Job 启动一个 JobMaster，并提供 REST 接口。

**TaskManager（TM）核心职责：**
- 每个 TM 是一个 JVM 进程，启动时向 RM 注册可用 slot。
- **Slot** 是 TM 的最小资源单位，决定了并行执行能力。TM 的 slot 数量 = `parallelism`（并行度）。
- **TaskExecutor** 实际执行算子子任务（SubTask），多个 SubTask 可共享一个 slot。

**工作流程简述：**
1. 客户端（Client）将用户代码翻译为 **StreamGraph** → **JobGraph** → 提交给 Dispatcher。
2. Dispatcher 启动 JobManager，JobManager 向 ResourceManager 申请 slot。
3. ResourceManager 让 TaskManager 提供 slot，JobManager 将 JobGraph 分配给对应的 TaskManager 执行。
4. 数据在 TaskManager 之间以 **流（Stream）** 的形式传输。

> **来源：** Apache Flink 官方文档 — Architecture
> https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/flink-architecture/

## 题目二：Flink 的 Watermark（水位线）机制是什么？如何解决乱序数据问题？

### 参考答案

**什么是 Watermark：**

Watermark 是 Flink 用于处理**乱序事件**和**延迟数据**的核心机制，本质是一个**时间戳（Timestamp）**。它携带一个时间值 `W(t)`，表示"所有时间戳 ≤ t 的数据已经全部到达"。

**解决乱序的核心思想：**
- 当 watermark `W >= 窗口结束时间` 时，触发窗口计算并关闭该窗口。
- 在 `[窗口开始时间, 窗口结束时间)` 区间内，watermark 之后到达的"迟到数据"默认被丢弃。
- 可通过 `allowedLateness()` 设置允许的延迟时间，迟到但未超时的数据会触发窗口的**增量更新**。

**Watermark 的生成方式：**
1. **周期性生成**：按时间或数据条数周期性生成，常用于内置数据源。
2. **标点式生成**：在特定数据（如特殊 event）触发时插入 watermark。

**乱序处理的 3 层机制：**
| 机制 | 说明 |
|------|------|
| watermark | 推进事件时间进度，决定窗口何时触发 |
| allowedLateness | 允许迟到时间，触发窗口增量更新 |
| sideOutputTag | 超长迟到数据侧输出，不丢弃任何数据 |

**多并行度下的 watermark 传播：**
多个输入流的 watermark 取**最小值**，保证全局进度一致。

> **来源：** Apache Flink 官方文档 — Watermarks
> https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/time-and-watermarks/

## 题目三：Flink 的 Checkpoint（检查点）和 Savepoint 有何区别？如何实现Exactly-Once语义？

### 参考答案

**Checkpoint vs Savepoint 对比：**

| 维度 | Checkpoint | Savepoint |
|------|------------|-----------|
| 触发方式 | Flink 自动周期性触发 | 手动触发（用户命令） |
| 用途 | 故障恢复、作业重启后继续 | 计划性维护、版本升级 |
| 存储 | JobManager 内存（快速） | 外部存储（HDFS/S3 等，可靠） |
| 删除策略 | 作业删除时自动清理 | 需手动删除 |
| 数据格式 | Flink 自有格式 | Flink 自有格式（可移植） |

**Exactly-Once 实现原理：**

Flink 通过 **两阶段提交（2PC）** 协议实现端到端的 Exactly-Once 语义，核心流程：

1. **Checkpoint Coordinator** 周期向所有算子发送 barrier（屏障）。
2. 算子收到所有输入的 barrier 后，进行**状态快照**（将状态写入 RocksDB/内存），然后向下游广播 barrier。
3. **Sink 算子**（如 Kafka Sink）在 2PC 中的行为：
   - **预提交**：收到所有 barrier 后，将数据写入外部系统，但先不 commit。
   - **Commit**：Checkpoint 完成时，JobManager 通知 Sink 正式提交。如果 Checkpoint 失败，则 abort。

**Exactly-Once 的三个环节：**
- **Source**：通过读取外部存储的 offset（如 Kafka）实现重放。
- **Flink 内部**：通过 Checkpoint 快照算子状态实现。
- **Sink**：通过 2PC 协议确保数据仅写入一次。

> **来源：** Apache Flink 官方文档 — Checkpoints
> https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/checkpoints/

## 题目四：Flink 支持哪几种窗口？滚动窗口、滑动窗口、会话窗口的区别是什么？

### 参考答案

**窗口分类：**

**1. 时间窗口（Time Windows）**
- **滚动窗口（Tumbling Window）**：窗口大小固定，不重叠。适合固定周期统计（如每分钟 PV）。
  ```
  每分钟窗口：[00:00-01:00), [01:00-02:00), ...
  ```
- **滑动窗口（Sliding Window）**：窗口大小和滑动步长可独立设置，可以重叠。适合需要"平滑统计"的场景（如过去5分钟平均销售额，每1分钟更新一次）。
  ```
  窗口大小5分钟，滑动1分钟：
  [00:01-05:01), [00:02-05:02), ...
  ```
- **会话窗口（Session Window）**：基于活动间隙（Gap）动态创建窗口。相同 key 的数据如果到达时间间隔超过 `session gap`，则开启新窗口。适合用户行为分析。

**2. 计数窗口（Count Windows）**
基于数据条数触发，不依赖时间。元素数量达到阈值时触发计算。

**3. 全局窗口（Global Window）**
所有相同 key 的数据进入同一个窗口，需配合自定义触发器使用。

**窗口函数（Window Functions）：**

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| `reduceFunction` |增量聚合，资源占用低| 需要完整聚合值（Sum/Avg） |
| `aggregateFunction` | 增量聚合，支持多输出| 多字段聚合 |
| `processWindowFunction` | 全量窗口数据，完整上下文| 需要访问窗口元数据/所有数据 |

**性能建议**：优先使用 `reduceFunction` 或 `aggregateFunction`（增量聚合），避免使用 `processWindowFunction` 处理海量数据。

> **来源：** Apache Flink 官方文档 — Window Functions
> https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream/operators/windows/

## 题目五：Flink 的状态（State）管理是如何工作的？StateBackend 有哪几种实现？如何选择？

### 参考答案

**什么是 State：**

Flink 是有状态流处理框架，State 是算子在处理每个 key（分区）时需要记住的信息，如聚合中间值、数据缓存等。Flink 的状态由 **Checkpoint** 机制保证持久化和一致性。

**State 的分类：**

| 类型 | 说明 | 作用域 |
|------|------|--------|
| **Keyed State** | 按 Key 分区的状态，只能在 KeyedStream 上使用 | 每个 Key 独立 |
| **Operator State** | 算子级别状态，所有 Key 共享 | 算子实例共享 |
| **Broadcast State** | 广播状态，常见于动态规则/配置场景 | 所有实例共享同一份 |

**KeyedState 常用接口：**
- `ValueState<T>`：单一值
- `ListState<T>`：列表
- `MapState<K, V>`：键值对
- `ReducingState<T>` / `AggregatingState<I, O>`：聚合

**StateBackend 三种实现：**

| Backend | 存储介质 | 适用场景 |
|---------|---------|---------|
| **HashMapStateBackend** | JVM Heap（内存） | 状态小、低延迟；开发测试 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘 + LMDB） | 大状态、增量 Checkpoint；生产环境首选 |
| **AdaptiveRocksDBStateBackend** | RocksDB（自适应参数调优） | 需手动指定 RocksDB 参数时 |

**选型建议：**
- 状态数据 < 10GB、延迟敏感 → `HashMapStateBackend`
- 大状态（GB~TB级）、增量 Checkpoint → `EmbeddedRocksDBStateBackend`（生产标配）

**RocksDB 增量 Checkpoint 优势：**
- 全量 Checkpoint 需要复制全量状态文件，耗时随状态增长线性增加。
- 增量 Checkpoint 仅记录本次变更的 SST 文件，大幅减少 checkpoint 时间和存储量。

> **来源：** Apache Flink 官方文档 — State & Fault Tolerance
> https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/stateful-stream-processing/
