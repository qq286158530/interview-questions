## 题目1：Java基础 - 泛型擦除机制

【答案】

**泛型擦除（Type Erasure）** 是Java编译器的核心机制，在编译阶段将泛型信息移除，生成字节码时将泛型类型替换为上限类型或Object。

### 工作原理

1. **编译时检查**：编译器在编译期间检查泛型类型是否匹配，如果类型不匹配会编译失败
2. **字节码擦除**：生成的.class文件中不包含任何泛型信息，所有泛型类型被替换为原始类型
3. **自动类型转换**：编译器在需要的地方自动插入类型转换代码（checkcast指令）

### 擦除规则

| 泛型类型 | 擦除后类型 |
|---------|-----------|
| `<T>` 无边界 | `Object` |
| `<T extends Number>` | `Number` |
| `<Integer>` | `Integer` |
| `<String>` | `Object`（因为String不是Number的子类） |

### 示例解析

```java
// 源代码
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);

// 编译后（等价于）
List list = new ArrayList();  // 泛型信息被擦除
list.add("hello");
String s = (String) list.get(0);  // 编译器自动插入类型转换
```

### 为什么设计泛型擦除？

- **向后兼容**：Java 1.5引入泛型之前的代码（Raw Type）可以继续运行
- **兼容非泛型集合**：如`ArrayList`底层就是`Object[]`，无需为泛型重新设计
- **类型安全**：在编译期进行类型检查，避免运行期ClassCastException

### 泛型不可用的场景

1. **不能实例化泛型类型**：`new T()` ❌
2. **不能创建泛型数组**：`new T[10]` ❌
3. **不能使用基本类型**：`List<int>` ❌（要用`List<Integer>`）
4. **不能使用instanceof**：`obj instanceof List<String>` ❌

### 桥接方法（Bridge Method）

当子类继承或实现泛型父类/接口时，编译器会自动生成桥接方法：

```java
// 父类
public class Parent<T> {
    public T doSomething(T t) { return t; }
}

// 子类
public class Child extends Parent<String> {
    @Override
    public String doSomething(String s) { return s; }
}

// 编译后实际生成（编译器自动生成桥接方法）
public class Child extends Parent {
    public String doSomething(String s) { return s; }
    
    // 桥接方法，保证协变返回
    public Object doSomething(Object obj) {
        return this.doSomething((String) obj);
    }
}
```

## 题目2：Java进阶 - Java反射机制原理与应用

【答案】

**反射（Reflection）** 是Java在运行时动态获取类的信息、构造对象、调用方法的技术，是"程序在运行时可以访问、检测和修改它本身状态或行为"的能力。

### 核心API

| 类/接口 | 作用 |
|--------|------|
| `Class<T>` | 类的入口，表示类和接口的运行时表示 |
| `Field` | 描述类的属性 |
| `Method` | 描述类的方法 |
| `Constructor<T>` | 描述类的构造器 |

### 获取Class对象的三种方式

```java
// 1. 通过类名.class
Class<?> clazz1 = String.class;

// 2. 通过对象的getClass()
Class<?> clazz2 = "hello".getClass();

// 3. 通过Class.forName()
Class<?> clazz3 = Class.forName("java.lang.String");
```

### 反射创建对象

```java
// 直接创建
Class<?> clazz = Class.forName("com.example.User");
User user = (User) clazz.newInstance();  // 已废弃

// 使用Constructor创建（推荐）
Constructor<?> constructor = clazz.getDeclaredConstructor(String.class, int.class);
User user = (User) constructor.newInstance("张三", 25);
```

### 反射调用方法

```java
Class<?> clazz = User.class;
Method setName = clazz.getDeclaredMethod("setName", String.class);
Method getName = clazz.getDeclaredMethod("getName");

User user = new User();
setName.invoke(user, "李四");  // 相当于 user.setName("李四")
String name = (String) getName.invoke(user);  // 相当于 user.getName()
```

### 反射访问私有成员

```java
// 设置可访问性
Class<?> clazz = User.class;
Field field = clazz.getDeclaredField("password");
field.setAccessible(true);  // 绕过访问控制检查

User user = new User();
field.set(user, "secret123");
String pwd = (String) field.get(user);
```

### 反射性能问题

1. **JIT优化失效**：反射调用绕过了内联优化
2. **安全检查开销**：每次访问都要检查权限
3. **参数装箱拆箱**：可变长参数会有额外开销

### 优化策略

```java
// 1. 缓存Method对象
private static final Map<String, Method> METHOD_CACHE = new ConcurrentHashMap<>();

// 2. MethodHandle（Java 7+）性能更好
MethodHandle handle = MethodHandles.lookup()
    .findVirtual(User.class, "setName", MethodType.methodType(void.class, String.class))
    .bindTo(user);
handle.invokeExact("王五");

// 3. setAccessible(true) 首次调用后缓存
```

### 经典应用场景

1. **Spring IoC**：通过反射+配置实现依赖注入
2. **Jackson/Gson**：将JSON映射为对象
3. **Junit**：通过反射调用@Test标注的方法
4. **MyBatis**：resultMap映射、插件机制
5. **IDE自动补全**：代码提示和重构

### 反射与内省（Introspector）的区别

| 特性 | 反射 | 内省 |
|-----|------|------|
| 来源 | `java.lang.reflect` | `java.beans` |
| 能力 | 绕过访问控制 | 基于JavaBean规范 |
| 性能 | 较慢 | 较反射快 |
| 典型应用 | Spring、MyBatis | Struts、EL表达式 |

## 题目3：Spring Boot - Spring IoC依赖注入流程

【答案】

**IoC（Inversion of Control，控制反转）** 是Spring的核心容器，而**依赖注入（DI）**是实现IoC的具体方式。

### IoC容器初始化流程

```
1. 资源定位（Resource定位）
   └─ 加载配置文件（XML/注解/JavaConfig）

2. Bean定义加载（BeanDefinition加载）
   └─ 将<bean>解析为BeanDefinition对象
   └─ 包含：bean-class、scope、lazy-init、constructor-arg、property等

3. BeanDefinition注册
   └─ 将BeanDefinition存入ConcurrentHashMap
   └─ key = beanName, value = BeanDefinition

4. 依赖注入（Lazy/着急/循环依赖处理）
   └─ 单例预实例化（singletonObjects）
   └─ 循环依赖检测与处理
```

### Bean创建的核心流程

```
BeanFactory.getBean()
    ↓
1. 检查缓存（singletonObjects）
    ↓
2. 如果是prototype，检查是否正在创建（isPrototypeCurrentlyInCreation）
    ↓
3. 获取父BeanFactory，递归获取
    ↓
4. 触发Bean初始化前置处理器（BeanPostProcessor.postProcessBeforeInitialization）
    ↓
5. 调用构造器/工厂方法创建Bean
    ↓
6. 属性填充（PopulateBean）- 依赖注入发生在这里
    ↓
7. 执行Aware接口回调（BeanNameAware、BeanFactoryAware等）
    ↓
8. 初始化前置处理（postProcessBeforeInitialization）
    ↓
9. 执行@PostConstruct / InitializingBean.afterPropertiesSet()
    ↓
10. 初始化后置处理（postProcessAfterInitialization）
    ↓
11. 注册销毁回调（DisposableBean.destroy()）
    ↓
12. 返回Bean实例
```

### 依赖注入方式

#### 1. 构造器注入（推荐）
```java
@Service
public class UserService {
    private final OrderService orderService;
    
    // 推荐：final保证不可变，支持循环依赖检测
    @Autowired
    public UserService(OrderService orderService) {
        this.orderService = orderService;
    }
}
```

#### 2. Setter注入
```java
@Service
public class UserService {
    private OrderService orderService;
    
    @Autowired
    public void setOrderService(OrderService orderService) {
        this.orderService = orderService;
    }
}
```

#### 3. Field注入（不推荐）
```java
@Service
public class UserService {
    @Autowired  // 不推荐：无法保证final、难以测试
    private OrderService orderService;
}
```

### 循环依赖处理

Spring通过**三级缓存**解决单例Bean的循环依赖：

```
┌─────────────────────────────────────────────────────────────┐
│  一级缓存：singletonObjects        已完成初始化的单例Bean      │
│  二级缓存：earlySingletonObjects  已创建但未完成属性填充的Bean│
│  三级缓存：singletonFactories     Bean创建工厂（lambda表达式）│
└─────────────────────────────────────────────────────────────┘

解决流程：
A创建 → 放入三级缓存 → 属性填充发现依赖B
B创建 → 放入三级缓存 → 属性填充发现依赖A
从三级缓存获取A的ObjectFactory → 创建早期引用A → 完成B初始化 → 放入一级缓存
继续完成A的初始化 → 放入一级缓存
```

**注意**：构造器注入的循环依赖无法解决（Spring 6.0已修复部分场景）。

### 循环依赖场景分析

| 注入方式 | 能否解决循环依赖 |
|---------|----------------|
| setter注入（默认） | ✅ 可以 |
| 构造器注入 | ❌ 报错：`BeanCurrentlyInCreationException` |
| @Lazy延迟注入 | ✅ 可以（注入代理对象，懒加载解决） |

## 题目4：Spring - Spring AOP切面编程原理

【答案】

**AOP（Aspect-Oriented Programming，面向切面编程）** 将与业务无关的横切关注点（日志、事务、安全等）抽离出来，达到解耦目的。

### 核心概念

| 概念 | 说明 |
|-----|------|
| Join Point（连接点） | 程序执行的某个位置，Spring AOP中只有方法执行的连接点 |
| Pointcut（切点） | 匹配连接点的表达式，决定哪些连接点需要被增强 |
| Advice（通知/增强） | 切面在特定连接点执行的动作（@Before、@After等） |
| Aspect（切面） | Pointcut + Advice 的组合 |
| Weaving（织入） | 将切面应用到目标对象的过程 |

### Spring AOP的代理实现

Spring AOP默认使用**JDK动态代理**和**CGLIB子类代理**两种方式：

#### JDK动态代理（默认，针对有接口的Bean）
```java
// 原理：生成$Proxy代理类，实现目标接口
public class $Proxy implements UserService {
    private InvocationHandler h;
    
    public $Proxy(InvocationHandler h) {
        this.h = h;
    }
    
    @Override
    public void save() {
        // 调用InvocationHandler.invoke()
        h.invoke(this, method_save, args);
    }
}
```

#### CGLIB代理（针对没有接口的Bean或配置proxy-target-class=true）
```java
// 原理：生成目标类的子类，重写方法
public class UserServiceImpl$$EnhancerByCGLIB$$xxx 
    extends UserServiceImpl {
    
    private MethodInterceptor callback;
    
    @Override
    public void save() {
        // 调用callback.intercept()
        callback.intercept(this, method_save, args, proxy);
    }
}
```

### 两种代理对比

| 特性 | JDK动态代理 | CGLIB代理 |
|-----|------------|----------|
| 实现方式 | 实现接口，生成$Proxy | 继承父类，生成子类 |
| 性能 | 较慢（反射调用） | 较快（直接调用） |
| 适用场景 | 有接口的类 | 无接口或final类 |
| 无法代理 | final方法、静态方法 | 构造方法 |

### 五种通知类型

```java
@Aspect
@Component
public class PerformanceAspect {
    
    // 前置通知：目标方法执行前
    @Before("execution(* com.example.*.*(..))")
    public void before(JoinPoint joinPoint) {
        System.out.println("方法执行前...");
    }
    
    // 后置通知：目标方法执行后（无论是否异常）
    @After("execution(* com.example.*.*(..))")
    public void after() {
        System.out.println("方法执行后...");
    }
    
    // 返回通知：目标方法正常返回后
    @AfterReturning(pointcut = "execution(* com.example.*.*(..))", 
                    returning = "result")
    public void afterReturning(JoinPoint joinPoint, Object result) {
        System.out.println("返回结果：" + result);
    }
    
    // 异常通知：目标方法抛出异常后
    @AfterThrowing(pointcut = "execution(* com.example.*.*(..))", 
                   throwing = "e")
    public void afterThrowing(Exception e) {
        System.out.println("抛出异常：" + e.getMessage());
    }
    
    // 环绕通知：包裹整个方法调用（最强大，可控制是否继续执行）
    @Around("execution(* com.example.*.*(..))")
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = pjp.proceed();  // 执行目标方法
        long cost = System.currentTimeMillis() - start;
        System.out.println("方法耗时：" + cost + "ms");
        return result;
    }
}
```

### 切点表达式

```java
// 语法：execution(<修饰符>? <返回类型> <类路径>.<方法名>(<参数列表>))

// 匹配UserService所有方法
execution(* com.example.UserService.*(..))

// 匹配com.example包下所有类的所有方法
execution(* com.example.*.*(..))

// 匹配以save开头的方法
execution(* com.example.*.save*(..))

// 匹配只有1个参数且参数类型为String的方法
execution(* com.example.*.*(String))

// 匹配public方法
execution(public * com.example.*.*(..))

// 匹配service结尾的Bean
@Pointcut("bean(*Service)")
```

### 多个切面的执行顺序

默认按**字母序**执行。可通过`@Order`注解指定顺序（数字越小优先级越高）：

```java
@Aspect
@Order(1)  // 优先级最高
public class TransactionAspect { }

@Aspect
@Order(2)  // 优先级次之
public class LoggingAspect { }
```

执行顺序：`TransactionAspect` → `LoggingAspect` → 目标方法

### Spring AOP vs AspectJ

| 特性 | Spring AOP | AspectJ |
|-----|-----------|---------|
| 织入时机 | 运行时 | 编译时/加载时 |
| 切点能力 | 方法级别 | 字段、构造器、方法 |
| 性能 | 略慢（代理） | 快（直接修改字节码） |
| 配置 | 简单 | 复杂（需要ajc编译器） |

│           ┌─────────────┴─────────────┐                      │

### 配置管理核心概念

| 概念 | 说明 |

  │                                    │

### 分布式存储一致性协议

**单节点模式**：使用嵌入式数据库（Derby），无需额外配置

**集群模式**：
- **Raft协议**（默认，推荐3节点及以上）：
  - Leader选举

## 题目6：Kafka - ISR副本同步机制

【答案】

**ISR（In-Sync Replicas）** 是Kafka实现数据可靠性和高可用的核心机制。

### 副本架构

```
Topic: order-topic        Partition: 0
┌──────────────────────────────────────────────────────┐
│                                                      │
│   Leader  (Broker 1)    Follower (Broker 2)         │
│   ┌───────────┐         ┌───────────┐               │
│   │  HW=15    │         │  LEO=13   │               │
│   │  LEO=16   │         │           │               │
│   │  [0-15]   │ �──fetch──│  [0-12]  │               │
│   │           │  同步中   │           │               │
│   └───────────┘         └───────────┘               │
│                                                      │
└──────────────────────────────────────────────────────┘

HW (High Watermark): 已提交消息的位置，消费者只能消费到HW之前
LEO (Log End Offset): 日志末尾偏移量，下一条消息写入位置
```

### ISR定义

**ISR** 是与Leader保持同步的副本集合（包含Leader本身）。

判定标准：副本的 `LEO` 落后于 Leader 的 `LEO` 不超过 `replica.lag.time.max.ms`（默认30秒）

```properties
# Broker级别配置
replica.lag.time.max.ms=30000
replica.lag.max.messages=4000  # 已废弃（0.11.0+）
```

### 副本同步流程

```
Producer                    Leader                     Follower
   │                          │                           │
   │──── produce(0,15) ──────▶│                           │
   │                          │ 写入本地日志              │
   │                          │ LEO=16                    │
   │                          │                           │
   │                          │◀─── fetch offset=13 ─────│
   │                          │  返回[13-15]              │
   │                          │                           │
   │                          │ 更新HW=16                 │
   │◀─── ack=1 (成功) ────────│                           │
   │                          │                           │
   │                          │◀─── fetch offset=16 ─────│
   │                          │  (下次拉取从HW开始)        │
```

### 消息提交机制

Kafka使用**水位标记（HW）**机制保证消息只被消费一次：

1. **写入**：消息先写入Leader，LEO更新
2. **同步**：Follower不断拉取并更新本地LEO
3. **提交**：当ISR中所有副本都同步到该位置后，Leader更新HW
4. **消费**：消费者只能读取到HW位置之前的数据

// 收缩后: ISR = {Leader, Follower1}
// Follower2恢复后重新同步

```

|-------|------|-------|-----|

| acks=1 | Leader写入成功即可 | 中等 | 较高 |

### 副本选举（Controller控制器）

当Leader宕机时，Controller负责选举新Leader：

```
选举流程：
1. Controller检测到Leader宕机（通过ZooKeeperWatcher）
2. 从ISR中选择一个Follower作为新Leader
3. 更新 ZooKeeper 中的 /brokers/topics/[topic]/partitions/[partition]/state

## 题目7：RocketMQ - 消息顺序性保证

【答案】

**RocketMQ** 支持两种顺序消息：**分区有序（Partition Order）**和**全局有序（Global Order）**。

### 顺序消息的三种类型

| 类型 | 说明 | 实现方式 | 性能 |
|-----|------|---------|-----|
| 分区有序 | 同一分区内的消息有序，不同分区并行 | 消息发送到同一队列 | 高 |
| 全局有序 | 所有消息按发送顺序消费 | 单队列+单消费者 | 低 |
| 乱序消费 | 不保证顺序 | 并行消费 | 最高 |

### 分区有序实现原理

```
Producer发送顺序：
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  订单消息发送顺序：                                           │
│  ① 创建订单 ──→ ② 支付 ──→ ③ 发货 ──→ ④ 确认收货              │
│                                                             │
│  相同的orderId必须发送到相同的队列：                            │
│  orderId=1001 ──→ Queue 0                                   │
│  orderId=1002 ──→ Queue 1                                   │
│  orderId=1001 ──→ Queue 0  (重试时也必须发送到Queue 0)        │
│                                                             │
└─────────────────────────────────────────────────────────────┘

消费时同一队列串行消费：
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Consumer Group A                                           │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │ Consumer 1  │  │ Consumer 2  │  │ Consumer 3  │        │
│  │ Queue 0 ✓   │  │ Queue 1 ✓   │  │ Queue 2 ✓   │        │
│  │ 顺序消费     │  │ 顺序消费     │  │ 顺序消费     │        │
│  │ 1001的4条消息│  │ 1002的4条消息│  │ 1003的4条消息│        │
│  └─────────────┘  └─────────────┘  └─────────────┘        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 消息定序关键配置

#### 生产者端配置

### 乱序场景及解决方案

#### 场景1：消费者集群部署时消息乱序

```
问题：多消费者并发消费同一队列的不同消息
Consumer 1: 收到①创建订单
Consumer 2: 收到②支付 ──→ 支付在创建之前执行！
Consumer 3: 收到③发货

解决方案：使用MessageListenerOrderly，内部使用分布式锁
```

#### 场景2：重试机制导致乱序

```
问题：消息处理失败后被重新消费
消息①创建订单 ──→ 失败，延迟重试
消息②支付 ──→ 成功
消息③发货 ──→ 成功  ──→ 异常！先发货后创建

解决方案：
1. RocketMQ默认的 consumeThreadMin=20, consumeThreadMax=20
2. 顺序消费时自动绑定线程，但需处理死锁
3. 业务层做状态机校验
```

#### 场景3：Broker故障切换导致乱序

```
问题：Leader切换后消息顺序不一致

解决方案：
1. 使用事务消息（ TransactionMQProducer ）
2. 业务侧做幂等处理
3. 使用消息状态表记录处理进度
```

### 顺序消息与事务消息结合

```java
// 事务消息保证本地事务和消息发送的原子性
public class OrderTransactionProducer {
    public void sendOrderWithTransaction(Order order) {
        TransactionMQProducer producer = new TransactionMQProducer("order_producer");
        
        // 实现事务监听器
        producer.setTransactionListener(new TransactionListener() {
            @Override
            public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
                try {
                    // 1. 执行本地事务（创建订单）
                    orderService.createOrder(order);
                    // 2. 提交半消息
                    return LocalTransactionState.COMMIT_MESSAGE;
                } catch (Exception e) {
                    // 3. 回滚
                    return LocalTransactionState.ROLLBACK_MESSAGE;
                }
            }
            
            @Override
            public LocalTransactionState checkLocalTransaction(MessageExt msg) {
                // 补偿检查：订单状态是否正常
                String orderId = parseOrderId(msg);
                if (orderService.isOrderValid(orderId)) {
                    return LocalTransactionState.COMMIT_MESSAGE;
                }
                return LocalTransactionState.ROLLBACK_MESSAGE;
            }
        });
        
        // 发送事务消息，保证顺序
        Message msg = new Message("order_topic", order.toJson().getBytes());
        producer.sendMessageInTransaction(msg, order.getOrderId());
    }
}
```

## 题目9：算法 - 合并K个升序链表

【答案】

### 题目描述

给你一个链表数组，每个链表都是按升序排列的。请将所有链表合并为一个升序链表并返回。

### 示例

```
输入：lists = [[1,4,5],[1,3,4],[2,6]]
输出：[1,1,2,3,4,4,5,6]

解释：
链表1: 1 → 4 → 5
链表2: 1 → 3 → 4
链表3: 2 → 6
合并后: 1 → 1 → 2 → 3 → 4 → 4 → 5 → 6
```

### 解法一：最小堆（优先队列）

**思路**：维护一个最小堆，每次取出堆中最小的节点，将其接到结果链表后。

**时间复杂度**：O(N log K)，N为总节点数，K为链表数
**空间复杂度**：O(K)

```python
import heapq
from typing import List, Optional

class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def mergeKLists(lists: List[Optional[ListNode]]) -> Optional[ListNode]:
    # 虚拟头节点
    dummy = ListNode(0)
    cur = dummy
    
    # 最小堆：(节点值, 链表索引, 节点)
    heap = []
    
    # 初始化：将每个链表的头节点加入堆
    for i, node in enumerate(lists):
        if node:
            heapq.heappush(heap, (node.val, i, node))
    
    # 逐个取出最小节点
    while heap:
        val, i, node = heapq.heappop(heap)
        cur.next = node
        cur = cur.next
        
        # 将该链表的下一个节点加入堆
        if node.next:
            heapq.heappush(heap, (node.next.val, i, node.next))
    
    return dummy.next
```

### 解法二：分治合并

**思路**：类似归并排序，两两合并链表，逐步减少问题规模。

**时间复杂度**：O(N log K)
**空间复杂度**：O(log K)（递归栈）

```python
def mergeKLists(lists: List[Optional[ListNode]]) -> Optional[ListNode]:
    if not lists:
        return None
    
    def mergeTwoLists(l1, l2):
        dummy = ListNode(0)
        cur = dummy
        while l1 and l2:
            if l1.val <= l2.val:
                cur.next = l1
                l1 = l1.next
            else:
                cur.next = l2
                l2 = l2.next
            cur = cur.next
        cur.next = l1 or l2
        return dummy.next
    
    def divideAndConquer(lists, left, right):
        if left == right:
            return lists[left]
        if left > right:
            return None
        
        mid = (left + right) // 2
        left_list = divideAndConquer(lists, left, mid)
        right_list = divideAndConquer(lists, mid + 1, right)
        return mergeTwoLists(left_list, right_list)
    
    return divideAndConquer(lists, 0, len(lists) - 1)
```

### 解法三：顺序合并

**思路**：逐个合并每个链表。

**时间复杂度**：O(K²N)，最坏情况O(K²N)
**空间复杂度**：O(1）

```python
def mergeKLists(lists: List[Optional[ListNode]]) -> Optional[ListNode]:
    result = None
    for node in lists:
        result = mergeTwoLists(result, node)
    return result

def mergeTwoLists(l1, l2):
    dummy = ListNode(0)
    cur = dummy
    while l1 and l2:
        if l1.val <= l2.val:
            cur.next = l1
            l1 = l1.next
        else:
            cur.next = l2
            l2 = l2.next
        cur = cur.next
    cur.next = l1 or l2
    return dummy.next
```

### 三种解法对比

| 解法 | 时间复杂度 | 空间复杂度 | 适用场景 |
|-----|----------|----------|---------|
| 最小堆 | O(N log K) | O(K) | 工程首选，稳定高效 |
| 分治合并 | O(N log K) | O(log K) | 并行化友好 |
| 顺序合并 | O(K²N) | O(1) | K较小时可接受 |

### 复杂度推导

设 N = 所有链表的总节点数

- **最小堆**：每个节点入堆出堆各一次，O(log K)，共 O(N log K)
- **分治**：树高 log K，每层合并所有节点 O(N)，共 O(N log K)

## 题目10：算法 - 二叉树右视图

【答案】

### 题目描述

给定一个二叉树的根节点 `root`，想象你站在右边看这棵树，返回从右侧能看到的节点值，按从上到下的顺序排列。

### 示例

```
输入: root = [1,2,3,null,5,null,4]
输出: [1,3,4]

    1            ←
   / \
  2   3          ←
   \   \
    5   4        ←

右视图看到的节点：1, 3, 4
```

### 解法一：层序遍历（BFS）

**思路**：按层遍历，每层取最后一个节点的值。

**时间复杂度**：O(N)
**空间复杂度**：O(W)，W为最大宽度

```python
from collections import deque
from typing import List, Optional

class TreeNode:
    def __init__(self, val=0, left=None, right=None):
        self.val = val
        self.left = left
        self.right = right

def rightSideView(root: Optional[TreeNode]) -> List[int]:
    if not root:
        return []
    
    result = []
    queue = deque([root])
    
    while queue:
        level_size = len(queue)
        
        for i in range(level_size):
            node = queue.popleft()
            
            # 每层最后一个节点
            if i == level_size - 1:
                result.append(node.val)
            
            if node.left:
                queue.append(node.left)
            if node.right:
                queue.append(node.right)
    
    return result
```

### 解法二：深度优先搜索（DFS）

**思路**：优先遍历右子树，保证每层第一个访问到的就是右视图节点。

**时间复杂度**：O(N)
**空间复杂度**：O(H)，H为树高

```python
def rightSideView(root: Optional[TreeNode]) -> List[int]:
    result = []
    
    def dfs(node: Optional[TreeNode], depth: int):
        if not node:
            return
        
        # 如果这是该深度第一个访问的节点，加入结果
        if depth == len(result):
            result.append(node.val)
        
        # 先遍历右子树，再遍历左子树
        dfs(node.right, depth + 1)
        dfs(node.left, depth + 1)
    
    dfs(root, 0)
    return result
```

### 算法图解

```
示例树：
    1
   / \
  2   3
   \   \
    5   4

BFS层序遍历过程：

Level 0: [1]        → 右视图: [1]
Level 1: [2, 3]      → 右视图: [1, 3]
Level 2: [5, 4]      → 右视图: [1, 3, 4]

最终结果: [1, 3, 4]

DFS遍历过程（优先右子树）：

dfs(1, 0): result=[1]
  dfs(3, 1): result=[1, 3]      # 右子树先遍历
    dfs(4, 2): result=[1, 3, 4] # 右子树先遍历
    dfs(None, 3)
  dfs(2, 1): result=[1, 3, 4]   # 左子树，但depth已记录
    dfs(5, 2): result=[1, 3, 4] # 不加入，因为depth < len(result)
    dfs(None, 3)
```
