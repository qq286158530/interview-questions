## 题目一：Flink 架构核心组件及其作用

### 参考答案

Flink 运行时由两种类型的进程组成：**JobManager** 和 **TaskManager**。

**JobManager** 负责协调分布式执行，主要包含三个组件：
- **ResourceManager**：负责资源分配和配置，管理 TaskManager 上的 slot
- **Dispatcher**：提供 REST 接口提交 Flink 应用，启动 JobMaster
- **JobMaster**：负责管理单个 JobGraph 的执行

**TaskManager** 是执行任务的工作节点，每个 TaskManager 有一个或多个 **task slot**，slot 是 Flink 集群中的资源调度单位。每个 slot 拥有固定的资源（内存），多个 subtask 可以共享一个 JVM。

**Client** 不是运行时的一部分，负责准备和发送数据流到 JobManager，之后可以断开或保持连接接收进度报告。

```
Client → JobManager (ResourceManager/Dispatcher/JobMaster) → TaskManager(s)
```

**三种集群模式**：
- **Application Cluster**：专用集群，main() 在集群运行，生命周期绑定应用
- **Session Cluster**：长期运行，可接受多个 job，资源竞争
- **Job Cluster（已废弃）**：每个 job 独立集群

> 📚 **来源**：[Apache Flink Architecture](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/flink-architecture/)

## 题目二：Flink 的 Watermark 机制及其在事件时间处理中的作用

### 参考答案

**Watermark（水位线）** 是 Flink 用于处理乱序事件的核心机制。

**核心概念**：
- Watermark 是嵌入数据流中的屏障（barrier），与记录一起流动
- Watermark 永远不会超越记录，严格按顺序流动
- Watermark 将数据流中的记录分为属于当前快照和下一个快照的集合

**事件时间处理**：
- 当 Watermark 到达某个时间 T 时，表示所有时间戳小于 T 的事件都已到达
- Flink 保证在 Watermark 超过窗口结束时间后，窗口会被触发计算
- 允许延迟通过 `allowedLateness()` 设置

**Watermark 生成策略**：
```java
// 固定延迟策略
WatermarkStrategy
    .forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner(...)
```

**关键特性**：
- Watermark 本质上声明了"到某个时间点，所有数据都已到达"
- 处理乱序数据时，需要权衡延迟和完整性
- 当 watermark 越过窗口结束时间 + allowed lateness 后，窗口被清除

> 📚 **来源**：[Apache Flink - Generating Watermarks](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream/operators/watermarks/)

## 题目三：Flink Checkpoint 的实现原理及 exactly-once 语义保障

### 参考答案

**Checkpoint 机制**：

Flink 的容错机制基于 **Chandy-Lamport** 算法的分布式快照实现。

**Barrier（屏障）机制**：
1. Source 节点向数据流中注入 Barrier，标记快照位置（Kafka 中为 offset）
2. Barrier 随数据流动，与记录一起传输但不会超越记录
3. 中间算子收到所有输入的 Barrier 后，触发快照并向下游发送 Barrier
4. Sink 算子收到所有输入 Barrier 后，向 Checkpoint Coordinator 确认完成

**对齐机制（Aligned Checkpoint）**：
- 多输入算子（如 Join）需要等待所有输入的 Barrier
- 未收到全部 Barrier 时，算子不能处理该输入的记录
- 收到全部 Barrier 后，快照状态，发送 Barrier，然后继续处理

**状态快照**：
- 算子在收到所有 Barrier 后，异步将状态写入 State Backend
- 支持 HashMapStateBackend（内存）和 EmbeddedRocksDBStateBackend（RocksDB）

**Exactly-Once 保障**：
```
Checkpoint n 完成 → 系统永远不会要求从 Sn 之前重新读取
```

**Unaligned Checkpointing（1.11+）**：
- 允许 Barrier 超越进行中的数据
- 超前的记录直接进入 operator state
- 适用于长窗口场景，避免对齐耗时过长

**恢复机制**：
1. 选择最新完成的 Checkpoint k
2. 重新部署整个分布式数据流
3. 各算子从 Checkpoint k 恢复状态
4. Source 从位置 Sk 恢复读取

> 📚 **来源**：[Apache Flink - State & Fault Tolerance](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/stateful-stream-processing/)

## 题目四：Flink 窗口函数分类及适用场景

### 参考答案

Flink 支持多种窗口类型，用于处理无限数据流。

### 按时间语义分类

**滚动窗口（Tumbling Windows）**：
- 窗口大小固定，不重叠
- 适合周期性统计，如每 5 分钟计算一次
```java
.window(TumblingEventTimeWindows.of(Time.minutes(5)))
```

**滑动窗口（Sliding Windows）**：
- 窗口大小固定，可重叠
- 适合需要平滑统计的场景
```java
.window(SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(5)))
```

**会话窗口（Session Windows）**：
- 根据活动间隙动态创建窗口
- 无固定开始结束时间
```java
.window(EventTimeSessionWindows.withGap(Time.minutes(10)))
```

**全局窗口（Global Windows）**：
- 所有相同 key 的数据进入同一窗口
- 需要配合自定义 Trigger 使用

### 按函数类型分类

| 函数类型 | 特点 | 适用场景 |
|---------|------|---------|
| **ReduceFunction** | 增量聚合，状态小，性能高 | 简单聚合 |
| **AggregateFunction** | 中间状态可见，支持复杂聚合 | 需要中间结果 |
| **ProcessWindowFunction** | 全量数据，可访问窗口元数据 | 需要窗口上下文 |

### 窗口生命周期

1. **创建**：第一个属于窗口的元素到达时创建
2. **触发**：满足 Trigger 条件（时间/数量/watermark）
3. **清除**：event time 超过窗口结束时间 + allowed lateness 时清除

> 📚 **来源**：[Apache Flink - Windows](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream/operators/windows/)

## 题目五：Flink State Backend 的分类、特点及选型建议

### 参考答案

Flink 提供两种主要的状态后端。

### HashMapStateBackend

**特点**：
- 状态存储在 Java 堆内存的 HashMap 中
- 快速：每次状态访问和更新直接操作堆上对象
- 受限于 JVM 堆内存大小

**适用场景**：
- 状态较小、窗口较短、key/value 状态较小的 Job
- 需要高性能的低延迟场景
- 所有高可用场景

### EmbeddedRocksDBStateBackend

**特点**：
- 状态存储在本地 RocksDB 数据库
- 数据以序列化字节数组形式存储
- 支持增量 Checkpoint
- 可扩展到磁盘空间限制

**限制**：
- 每次读写需要序列化/反序列化，开销较大
- 单 key/value 最大 2^31 字节

**适用场景**：
- 超大状态、长窗口、大 key/value 状态
- 所有高可用场景
- 唯一支持增量 Checkpoint 的后端

### 选型建议

| 维度 | HashMapStateBackend | EmbeddedRocksDBStateBackend |
|------|---------------------|----------------------------|
| **性能** | 高（内存操作） | 较低（需序列化） |
| **容量** | 受限于内存 | 可用磁盘空间 |
| **增量 Checkpoint** | ❌ | ✅ |
| **推荐场景** | 小状态、低延迟 | 大状态、成本敏感 |

### 配置示例

```java
// 代码配置
env.setStateBackend(new HashMapStateBackend());
// 或
env.setStateBackend(new EmbeddedRocksDBStateBackend(true)); // 启用增量Checkpoint

// YAML配置
state.backend: rocksdb
state.checkpoints.dir: hdfs://namenode:40010/flink/checkpoints
```

### RocksDB 内存管理

Flink 默认将 TaskManager 的托管内存分配给 RocksDB 使用，通过 block cache 和 write buffer manager 共享管理。

调优参数：
- `state.backend.rocksdb.memory.write-buffer-ratio`：写缓冲区占比（默认 0.5）
- `state.backend.rocksdb.memory.high-prio-pool-ratio`：索引/过滤器优先级（默认 0.1）

> 📚 **来源**：[Apache Flink - State Backends](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/state_backends/)
