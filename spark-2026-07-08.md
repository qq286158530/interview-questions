## 题目一：Spark 架构核心——Driver 与 Executor 的协作机制

**问题：** 请描述 Spark 应用程序的架构组成，以及 Driver 与 Executor 之间如何协作完成一个 Spark Job？

**答案：**

Spark 采用 **Client-Server** 架构，核心组件包括 **Driver、Executor 和 Cluster Manager**。

### 架构组成

| 组件 | 角色 | 说明 |
|------|------|------|
| **Driver** | 主导进程 | 解析用户代码（DAG），构建 `DAGScheduler` 和 `TaskScheduler`，将 Task 分配给 Executor |
| **Executor** | 工作进程 | 驻留在 Worker 节点上，负责执行分发的 Task，向 Driver 汇报结果 |
| **Cluster Manager** | 资源管理 | YARN/Kubernetes/Standalone，负责分配容器资源 |

### 协作流程

1. **提交阶段**：用户通过 `spark-submit` 提交应用，Driver 启动并创建 `SparkContext`
2. **DAG 构建**：SparkContext 向集群申请 Executor 资源，同时将应用代码发送给 Executor
3. **Task 划分**：Driver 根据 Shuffle 依赖将 DAG 划分为 Stage，Stage 划分为 Task
4. **Task 分发**：`TaskScheduler` 通过 **CoarseGrainedExecutorBackend** 将 Task 序列化发送到 Executor
5. **执行与结果**：Executor 反序列化 Task，执行计算，将结果通过 **Netty** 返回给 Driver

### 关键点

- **Executor 不是一一对应线程**，一个 Executor 可以并发执行多个 Task（由 `spark.executor.cores` 控制）
- **DAGScheduler** 按 **Shuffle 划分 Stage**（宽依赖处切分），**TaskScheduler** 按 **Task 最佳位置优先** 调度
- 通信默认使用 **Netty**，在 2.0+ 移除了 Akka

> 📎 来源：[Apache Spark 官方文档 - Architecture](https://spark.apache.org/docs/latest/cluster-overview.html)

## 题目二：RDD 五大特性与 Lazy 机制

**问题：** RDD 有哪五大特性？Spark 如何实现懒加载（Lazy Evaluation）？这种设计有什么优势？

**答案：**

### RDD 五大特性

| 序号 | 特性 | 说明 |
|------|------|------|
| 1 | **分区列表（Partitions）** | RDD 由多个分区组成，每个分区是数据的逻辑切片，可并行计算 |
| 2 | **计算函数（Compute Function）** | 每个分区都有 compute 函数，用于计算该分区的数据 |
| 3 | **依赖关系（Dependencies）** | RDD 与父 RDD 之间存在依赖（窄依赖或宽依赖） |
| 4 | **分区器（Partitioner）** | 可选，指定 Key-Value RDD 的分区策略（如 HashPartitioner） |
| 5 | **位置偏好（Preferred Locations）** | 可选，记录每个分区的最佳计算位置（HDFS Block 位置），实现数据本地化 |

### Lazy 机制

Spark 的 **懒加载**体现在：RDD 的 **Transformation**（如 `map`、`filter`、`groupByKey`）不会立即执行，只是记录了计算逻辑（构建 DAG）。只有遇到 **Action**（如 `collect`、`count`、`saveAsTextFile`）时，才触发完整的计算链路。

```python
# Transformation（懒加载，只记录逻辑）
rdd2 = rdd1.map(lambda x: x * 2)  # 不执行
rdd3 = rdd2.filter(lambda x: x > 10)  # 不执行

# Action（触发真正计算）
result = rdd3.collect()  # 此刻才开始执行
```

### 设计优势

1. **优化计算**：Spark 可以先完整看到整个 DAG，对计算链路进行**统一优化**（如 pipeline 融合、map端聚合）
2. **减少中间结果落盘**：了解全局后可以尽量减少 Shuffle 次数
3. **容错机制**：通过血统（Lineage）重建丢失分区，而不是复制全量数据

> 📎 来源：[RDD Paper - Resilient Distributed Datasets: A Fault-Tolerant Abstraction for In-Memory Cluster Computing](https://www.usenix.org/system/files/conference/nsdi12/nsdi12-final138.pdf)

## 题目三：Shuffle 机制与优化策略

**问题：** 什么是 Shuffle？为什么 Shuffle 是 Spark 性能瓶颈？请列举至少 3 种 Shuffle 优化策略。

**答案：**

### 什么是 Shuffle

Shuffle 是**将数据按照 Key 进行重新分布**的过程。当 RDD 依赖需要**跨节点聚合数据**时（如 `groupByKey`、`join`、`reduceByKey`），必须进行 Shuffle：

```
Map Phase  →  Shuffle Write（写本地磁盘）→  Shuffle Read（从其他节点拉取）→  Reduce Phase
```

### 为什么是性能瓶颈

1. **磁盘 I/O**：Shuffle 中间结果写入本地磁盘，跨节点网络传输
2. **序列化开销**：数据需要 Java/Kryo 序列化/反序列化
3. **内存压力**：聚合操作在 Reduce 端需要将同一 Key 的数据放入内存，容易导致 OOM
4. **磁盘随机读写**：大量小文件随机写，性能差

### Shuffle 优化策略

#### ① `reduceByKey` 替代 `groupByKey`

`reduceByKey` 会在 Map 端先做**本地聚合（map-side combine）**，再 Shuffle，减少网络传输量：

```python
# ❌ 不推荐：groupByKey 无本地聚合
rdd.groupByKey().mapValues(_.sum)

# ✅ 推荐：reduceByKey 有本地聚合
rdd.reduceByKey(_ + _)
```

#### ② 调整 Shuffle 分区数

分区数过多或过少都会影响性能：
- 分区数 ≈ `core数 * 2~3` 是较优选择
- 通过 `spark.sql.shuffle.partitions`（SQL场景，默认200）调整
- 数据量小时应**减少分区数**，避免小任务开销

#### ③ 使用广播变量（Broadcast Join）优化大表 Join

将小表广播到每个 Executor，避免 Shuffle：

```python
small_df = large_df.sparkSession.createDataFrame(...)
broadcast_small = broadcast(small_df)
result = large_df.join(broadcast_small, "key")
```

#### ④ 开启 Sort Shuffle（Spark 2.x+）

Spark 2.x 后默认使用 **Sort Shuffle**，避免生成大量小文件：
- 开启 `spark.shuffle.sort.bypassMergeThreshold`（默认200）优化简单 Shuffle
- 使用 **Tungsten Sort Shuffle**（钨丝计划）进一步优化内存管理

> 📎 来源：[Spark 官方文档 - Shuffle Behavior](https://spark.apache.org/docs/latest/rdd-programming-guide.html#shuffle-operations)

## 题目四：Spark 内存管理机制——堆内与堆外内存

**问题：** 请详细说明 Spark 的堆内内存（On-Heap）和堆外内存（Off-Heap）的划分方式，以及各自的用途。

**答案：**

Spark 2.x 统一使用 **Unified Memory Management**（统一内存管理），将内存划分为**堆内**和**堆外**两部分。

### 堆内内存（On-Heap Memory）

由 JVM Heap 分配，受 `-Xmx` 控制，由 Spark 自动管理。

| 区域 | 默认占比 | 用途 |
|------|----------|------|
| **Execution 内存** | 30% | Shuffle、排序、HashJoin 等计算过程的临时内存 |
| **Storage 内存** | 30% | 缓存 RDD/Broadcast/广播变量 |
| **User Memory** | 20% | 用户代码（如 Python 进程、UDF 申请） |
| **Reserved Memory** | 300MB | 预留，系统预留 |

### 堆外内存（Off-Heap Memory）

通过 `spark.memory.offHeap.enabled=true` 开启，使用 **JDK DirectByteBuffer** 分配 OS 内存，不受 JVM GC 管理。

- 主要用于**存储**（Storage）和**执行**（Execution）共享的 off-heap 内存区域
- 减少了 GC 开销，适合大内存机器

### 统一内存管理核心机制

Execution 和 Storage 之间可以**互相借用**内存：

```
Storage 内存不足 → 可借用 Execution 内存（需要evict缓存块）
Execution 内存不足 → 可驱逐 Storage 中的缓存块
```

但 Storage 无法从 Execution 抢占**正在被使用的内存**。

### 配置示例

```properties
spark.memory.fraction=0.6        # Execution+Storage 占堆内比例（默认0.6）
spark.memory.storageFraction=0.5  # Storage 占用 memory.fraction 的比例（默认0.5）
spark.memory.offHeap.enabled=true
spark.memory.offHeap.size=4g      # 堆外内存大小
```

### 常见 OOM 原因

- `Execution` 内存不足：Shuffle 聚合时数据量超出限制 → 开启 **外部排序**（`spark.shuffle.spill=true`）
- `Storage` 内存不足：缓存数据量过大 → 减少 `spark.memory.storageFraction` 或清理缓存

> 📎 来源：[Spark 官方文档 - Memory Management](https://spark.apache.org/docs/latest/tuning.html#memory-management)

## 题目五：广播变量（Broadcast Variable）原理与最佳实践

**问题：** 什么是广播变量？它的工作原理是什么？使用广播变量有哪些注意事项？

**答案：**

### 什么是广播变量

广播变量是**只读的、缓存在每个 Executor 节点内存中的共享变量**。当 Task 需要访问同一份小数据集时（如维表数据、配置参数），将其广播到所有节点，避免每个 Task 都复制一份数据。

```python
# 创建广播变量
broadcast_var = spark.sparkContext.broadcast({"bj": 1, "sh": 2, "gz": 3})

# 在 Executor 上访问（每个节点只存一份）
result = rdd.map(lambda x: broadcast_var.value.get(x, -1))
```

### 工作原理

1. **Driver 端**：调用 `sc.broadcast(v)` 时，Spark 将变量序列化为二进制并**按 Executor 分区**存储（每个 Executor 一份）
2. **分发阶段**：当提交 Task 时，广播数据通过 **BitTorrent**（默认）或 **HTTP** 方式传输到 Executor，并存储在 `BlockManager` 中
3. **Executor 端**：Task 执行时，通过 `broadcast.value` 从本地 `BlockManager` 读取数据，无需跨网络

### 广播优势

| 对比项 | 无广播 | 有广播 |
|--------|--------|--------|
| 数据复制 | N 份（N = Task 数） | 1 份/Executor |
| 网络传输 | N × 数据量 | E × 数据量（E = Executor 数） |
| 内存占用 | N × 数据量 | E × 数据量 |

### 注意事项

1. **数据大小阈值**：默认广播阈值 10MB，可通过 `spark.sql.autoBroadcastJoinThreshold`（SQL场景）调整；过大数据广播反而拖累性能

2. **只读不可修改**：广播变量是只读的，修改不会同步到其他节点

3. **序列化方式**：广播变量需要**可序列化**，推荐使用 Kryo 序列化以提升性能：
   ```python
   spark.conf.set("spark.serializer", "org.apache.spark.serializer.KryoSerializer")
   ```

4. **清理时机**：广播变量不会自动清理，需要手动调用 `broadcast.unpersist()` 释放缓存

5. **不适合大表广播**：通常表大小 < 10MB 时才适合广播，否则应使用 **Shuffle Hash Join** 或 **Sort Merge Join**
