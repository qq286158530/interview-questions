## 题目 1：Spark 架构原理 — Driver 与 Executor 的协作机制

**问题：** 请描述 Spark Application 的整体架构，Driver 和 Executor 在 Spark Job 执行过程中各自承担什么角色？它们之间如何协作？

**答案：**

Spark Application 采用 **Master-Slave 架构**，核心组件包括 **Driver**、**Cluster Manager** 和 **Executor**。

| 组件 | 角色 | 说明 |
|------|------|------|
| **Driver** | 主导进程 | 持有 `SparkContext`，负责将用户代码编译为 RDD DAG，向 Cluster Manager 申请资源，将 Task 调度到 Executor 执行 |
| **Cluster Manager** | 资源管理器 | 负责集群资源的管理和调度，常见实现有 Standalone、YARN、Kubernetes |
| **Executor** | 工作进程 | 真正执行 Task 的进程，每个 Executor 拥有自己的 **内存池（JVM Heap）**，负责执行分配给自己的 Task，计算结果写入存储或网络传输 |

**协作流程：**

1. 用户提交 Spark 应用程序，Driver 启动并创建 `SparkContext`
2. Driver 向 Cluster Manager 注册 Application 并申请资源
3. Cluster Manager 通知 Worker 节点启动 Executor
4. Driver 将用户代码转化为**Task DAG**（有向无环图）
5. **DAGScheduler** 将 DAG 划分为多个 Stage，Stage 内部再拆分为具体的 Task
6. TaskScheduler 将 Task 分发到各个 Executor 的线程池中并行执行
7. Executor 执行 Task，将结果汇报给 Driver
8. 执行完毕后，Executor 被释放，Driver 关闭

**关键点：**
- Driver 是整个 Application 的"大脑"，持有元数据，控制全局
- Executor 是"执行者"，以线程池方式并行运行多个 Task
- Driver 与 Executor 之间通过 **CoarseGrainedExecutorBackend** 通信（基于 Netty）

> 📎 来源：[Apache Spark 官方文档 – Cluster Mode Overview](https://spark.apache.org/docs/latest/cluster-overview.html)

## 题目 2：RDD 核心机制 — 血缘关系（Lineage）与容错原理

**问题：** RDD（Resilient Distributed Dataset）是如何实现分布式数据集的容错机制的？它的惰性计算（Lazy Evaluation）有什么意义？

**答案：**

### 1. RDD 的五大属性

| 属性 | 含义 |
|------|------|
| `dependencies` | RDD 的依赖关系（父 RDD） |
| `partitions` | 数据分片列表 |
| `partitioner` | 分区函数（如 HashPartitioner） |
| `compute` | 计算函数，即如何从父 RDD 计算当前 RDD |
| `preferredLocations` | 最优位置（数据本地性） |

### 2. 血缘关系（Lineage）实现容错

RDD 通过**血缘关系（Lineage）**实现容错，而不是像 MapReduce 那样持久化中间结果。具体机制如下：

- 每个 RDD 都知道自己是由哪个父 RDD、通过什么转换（transform）得来的
- 当某个 Partition 数据丢失时，Driver 会根据血缘链**重新计算**该 Partition，而不是全量恢复
- 血缘链越长，重算代价越大，因此 Spark 提供了 **Checkpoint 机制**将 RDD 持久化到磁盘

```scala
// 示例：lineage 示例
val rdd1 = sc.textFile("hdfs://file")
val rdd2 = rdd1.map(_.split(","))
val rdd3 = rdd2.filter(_.length > 3)
// rdd3 的血缘：rdd3 → rdd2 → rdd1
// rdd2 丢失时，只需重算 rdd2 而非从头开始
```

### 3. 惰性计算（Lazy Evaluation）

Spark 的转换操作（`map`、`filter`、`flatMap` 等）都是**惰性的——不会立即执行**，只有遇到**动作操作**（`collect`、`count`、`saveAsTextFile` 等）时，才会触发真正的计算。

**意义：**
- **构建执行计划**：Driver 有机会看到完整的 DAG，进行**管道化（Pipelining）优化**，减少数据流动
- **减少中间结果落盘**：同一 Stage 内的多个 Transform 可以管道执行，不需要写 HDFS
- **统一的优化视图**：Spark 可以从全局视角优化执行路径，而不是逐条执行

> 📎 来源：[RDD Paper – Resilient Distributed Datasets: A Fault-Tolerant Abstraction for In-Memory Cluster Computing](https://www.usenix.org/system/files/conference/nsdi12/nsdi12-final白菜.pdf)（原始论文）  
> 📎 参考：[CSDN – Spark RDD 原理详解](https://blog.csdn.net/gangliao/article/details/108405365)

## 题目 3：DataFrame 与 DataSet 的区别及优势

**问题：** Spark SQL 中的 DataFrame 和 DataSet 有什么区别？它们相比 RDD 在实际开发中有哪些优势？

**答案：**

### 概念区分

| 特性 | DataFrame | DataSet |
|------|-----------|---------|
| **语言支持** | 跨语言（RDD 风格，所有语言通用） | 强类型，仅 Scala/Java |
| **数据结构** | 类表结构，有列名和类型（类似 Pandas / MySQL） | 强类型对象集合（Encoder 序列化） |
| **API 风格** | DSL 风格（column、select、where） | OO 风格（dataset.map(_.field)） |
| **编译时检查** | 弱（运行时错误） | 强（编译时错误） |
| **Encoder** | 无需显式 Encoder | 必须有 Encoder |

> ⚠️ **注意**：在 Spark 2.x 中，`DataFrame = DataSet[Row]`（Row 是弱类型的通用行），在 Spark 3.x 中趋于统一。

### DataFrame vs RDD 的优势

#### 1. **更丰富的抽象**
- DataFrame 把数据当作"带列名的表"，比 RDD 的 `RDD[List[Any]]` 更易理解
- 用户不需要关心数据如何分片、序列化细节

#### 2. **Spark SQL 引擎优化（Catalyst）**
DataFrame 的所有操作都会经过 **Catalyst Optimizer**：
- **谓词下推（Predicate Pushdown）**：将 `filter` 尽可能下推到数据源，减少读取量
- **列裁剪（Column Pruning）**：只读取需要的列，减少 I/O
- **常数折叠（Constant Folding）**：提前计算已知常数
- **向量化（Vectorized HDFS读取）**：使用 ORC/Parquet 列式存储实现向量化扫描

```sql
-- DataFrame API
df.select("name", "age").filter("age > 18").show

-- 底层被 Catalyst 优化为：先 filter 再 select（谓词下推）
-- 再生成最优的物理执行计划
```

#### 3. **性能优势**
- **内存列式存储（Off-Heap）**：通过 Project Tungsten 使用二进制列式内存，避免 JVM 对象开销
- **动态内存管理**：根据查询需求动态分配内存
- **代码生成（Whole-stage Codegen）**：将整个查询阶段生成为单次 Java 代码执行，减少虚函数调用

#### 4. **统一的数据源 API**
通过 `DataFrameReader/DataFrameWriter`，同一套 API 可读写 **Hive、Parquet、ORC、JSON、JDBC、CSV** 等各种数据源。

> 📎 来源：[Databricks – Introducing Apache Spark 2.0](https://www.databricks.com/blog/2016/10/26/introducing-apache-spark-2-0)  
> 📎 参考：[Spark 官方文档 – DataFrame and SQL Guide](https://spark.apache.org/docs/latest/sql-programming-guide.html)

## 题目 4：Shuffle 原理 — 为什么是性能瓶颈？如何优化？

**问题：** 什么是 Shuffle？为什么 Shuffle 会成为 Spark 应用的性能瓶颈？请列举你知道的 Shuffle 优化策略。

**答案：**

### 什么是 Shuffle

Shuffle 是 **分布式数据重新分组的过程**，当一个 RDD/DataFrame 的分区需要依赖父 RDD 的多个分区数据时（如 `groupByKey`、`join`、`reduceByKey`），就必须将数据按照 Key 进行重新分区和传输。

**典型 Shuffle 场景：**
```
WordCount: rdd.flatMap(_.split).map((_,1)).reduceByKey(_+_)
                                 ↑
                           reduceByKey 必须将相同 key 的数据汇聚到同一个 executor
                           → Shuffle 发生
```

### Shuffle 为什么慢

| 原因 | 说明 |
|------|------|
| **磁盘 I/O** | Shuffle 中间结果写入本地磁盘（Map 端落盘） |
| **网络 I/O** | 数据通过千兆网络传输到 Reduce 端 |
| **序列化开销** | 数据需要序列化（Kryo/Java）跨网络传输 |
| **磁盘随机读写** | Reduce 端读取多个小文件，产生磁盘随机读 |
| **内存压力** | Shuffle 数据量大时可能导致 OOM 或频繁 GC |
| **Task 重算** | Shuffle 是宽依赖，一旦父 RDD 分区丢失，需全量重算 |

### Shuffle 优化策略

#### 1. **合理设置分区数**
- 分区过多：每个分区数据量过小，产生过多小文件，增加调度开销
- 分区过少：数据倾斜（Skew）风险高，个别 Task 耗时过长
- 经验值：`shuffle.partitions` 默认 200，可根据数据量调整为 `core数 * 2~5`

#### 2. **使用 map-side 预聚合（High-cardinality 优化）**
`reduceByKey` 优于 `groupByKey`：
- `reduceByKey` 在 Map 端先做局部聚合（map-side 预聚合），再 Shuffle，减少网络传输量
- `groupByKey` 无预聚合，所有数据直接 Shuffle

#### 3. **开启 Shuffle 合并（Shuffle Consolidate）**
```scala
conf.set("spark.shuffle.consolidateFiles", "true")
// 将同一 core 的多个 Shuffle Map Task 合并输出到同一文件，减少小文件数量
```

#### 4. **使用 Kryo 序列化**
```scala
conf.set("spark.serializer", "org.apache.spark.serializer.KryoSerializer")
conf.registerKryoClasses(Array(classOf[MyClass]))
// Kryo 比 Java 序列化快 3~10倍，减少序列化时间和数据大小
```

#### 5. **数据倾斜排查**
- 使用 `spark.sql.shuffle.partitions` 调整
- 对高基数 Key 单独处理，单独开分区
- 过滤掉异常数据（垃圾数据、测试数据）

#### 6. **Spark 3.0+ AQE（Adaptive Query Execution）**
Spark 3.0 引入了自适应执行，在运行时动态优化 Shuffle：
- 动态调整 Shuffle 分区数（解决数据倾斜）
- 自动启用广播 join（避免 Shuffle）
- 自动重算缺失的 Shuffle 分区

> 📎 来源：[Spark 官方文档 – Shuffle Behavior](https://spark.apache.org/docs/latest/configuration.html#shuffle-behavior)  
> 📎 参考：[美团技术团队 – Spark Shuffle 原理与优化](https://tech.meituan.com/2021/01/07/spark-shuffle-optimization.html)

## 题目 5：广播变量（Broadcast Variable）— 原理、用途及注意事项

**问题：** 什么是广播变量？它解决了什么问题？在使用广播变量时有哪些注意事项？

**答案：**

### 什么是广播变量

广播变量是 **一种只读共享变量**，Driver 将一份数据 **高效地分发到所有 Executor**，每个 Executor 只存储一份副本，所有 Task 共享访问，避免了每个 Task 都复制一份数据。

### 工作原理

```
普通变量传递（每个 Task 复制一份）:
  Task1: copy(变量A) → 内存占用 × N（Task 数）
  Task2: copy(变量A)
  Task3: copy(变量A)
  ...

广播变量（每个 Executor 存储一份）:
  Executor1: 存储 变量A副本 × 1
              ↓ Task1, Task2 共用
  Executor2: 存储 变量A副本 × 1
              ↓ Task3, Task4 共用
```

### 典型使用场景

**1. 大维度表与小表 JOIN（广播 join）**
```scala
// 当小表足够小时，自动广播
val smallDF = spark.read.parquet("hdfs://small_table")
val largeDF = spark.read.parquet("hdfs://large_table")

// 手动广播 DataFrame（Spark 2.3 前）
import org.apache.spark.sql.functions.broadcast
largeDF.join(broadcast(smallDF), "key")
```

**2. 跨节点共享配置/字典表**
```scala
val configMap = Map("env" -> "prod", "version" -> "1.0")
val bcConfig = sc.broadcast(configMap)

// 在 Task 中使用
rdd.map(record => {
    val env = bcConfig.value("env")  // 读取广播变量
    process(record, env)
})
```

### 注意事项（面试高频考察点）

| 注意点 | 说明 |
|--------|------|
| **大小限制** | 广播变量不宜过大（建议 < 20MB），过大会导致 Driver OOM 或广播超时 |
| **只读不变** | 广播后不能修改值（只读），修改需重新广播 |
| **广播阈值** | `spark.sql.autoBroadcastJoinThreshold`（默认 10MB），可通过配置调整 |
| **广播失败处理** | 数据超过阈值时 Spark 自动降级为 Shuffle join（全量 Shuffle） |
| **广播变量堆积** | 广播变量与 Task 生命周期相同，未完成 Task 会持续占用 Executor 内存 |
| **广播失效** | 如果广播后该变量在 Driver 侧被 GC，Executor 仍可使用（已传输），但 Driver 需要保留元数据引用 |
| **Spark 3.2+ 优化** | 支持 **P2P 广播**（通过 FetchFromAnyHost），减少 Driver 网络出口压力 |

### 广播 join vs Shuffle join 对比

| 对比项 | 广播 join | Shuffle join |
|--------|-----------|--------------|
| 网络传输 | 仅小表数据 | 全量数据 |
| 磁盘 I/O | 几乎无 | Shuffle 落盘 |
| Driver 压力 | 需广播分发 | 较低 |
| 适用场景 | 小表（< 10MB） | 大表 |
| 延迟 | 低 | 高 |

> 📎 参考：[Databricks – Join Strategy and Broadcast](https://www.databricks.com/blog/2021/10/18/join-optimization-in-apache-spark-3-0.html)

## 📌 面试核心要点速记

| 知识点 | 核心一句话 |
|--------|-----------|
| **Spark架构** | Driver 负责任务 DAG 生成和调度，Executor 负责任务执行 |
| **RDD 容错** | 通过血缘关系（Lineage）重算恢复，不需要持久化中间结果 |
| **惰性计算** | Transform 惰性，Action 触发；意义在于构建 DAG 优化执行计划 |
| **DataFrame 优势** | Catalyst 优化器 + 列式存储 + Whole-stage Codegen，性能远超 RDD |
| **Shuffle 瓶颈** | 磁盘 I/O + 网络 I/O + 序列化开销；优化方向：预聚合、合理分区、Kryo序列化 |
| **广播变量** | 只读共享变量，每个 Executor 一份副本，用于小表 JOIN 和配置共享 |
