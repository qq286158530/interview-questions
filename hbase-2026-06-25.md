## 1. HBase 的过滤器（Filter）机制是什么？常用过滤器有哪些？

### 答案

HBase 提供了**服务器端过滤器（Server-Side Filter）**，在 RegionServer 端完成数据过滤，避免将大量无效数据传输到客户端，从而极大减少网络开销和客户端内存消耗。

#### 工作原理

```
客户端发送 Scan + Filter
        │
        ▼
RegionServer 扫描 HFile/MemStore
        │
        ▼
Filter 在 RegionServer 端过滤数据
        │
        ▼
仅返回符合条件的 KV 给客户端
```

#### 常用过滤器分类

**① 行键过滤器（RowKey Filter）**

| 过滤器 | 类名 | 说明 |
|--------|------|------|
| RowFilter | `RowFilter` | 按行键正则匹配或比较器过滤整行 |
| PrefixFilter | `PrefixFilter` | 返回以指定前缀开头的所有行 |
| KeyOnlyFilter | `KeyOnlyFilter` | 只返回 KeyValue 的 key 部分，不返回值 |

**② 列过滤器（Column Filter）**

| 过滤器 | 类名 | 说明 |
|--------|------|------|
| FamilyFilter | `FamilyFilter` | 按列族名过滤 |
| QualifierFilter | `QualifierFilter` | 按列名（Qualifier）过滤 |
| ColumnPrefixFilter | `ColumnPrefixFilter` | 返回指定列前缀的列 |
| ColumnRangeFilter | `ColumnRangeFilter` | 按列名的字典序范围过滤 |

**③ 值过滤器（Value Filter）**

| 过滤器 | 类名 | 说明 |
|--------|------|------|
| ValueFilter | `ValueFilter` | 按值的内容过滤（如正则匹配） |
| SingleColumnValueFilter | `SingleColumnValueFilter` | 指定列的值满足条件才返回整行 |
| SingleColumnValueExcludeFilter | `SingleColumnValueExcludeFilter` | 与上面类似，但排除指定列 |

**④ 专用过滤器**

| 过滤器 | 类名 | 说明 |
|--------|------|------|
| PageFilter | `PageFilter` | 每页返回指定数量的行 |
| TimestampsFilter | `TimestampsFilter` | 只返回指定时间戳版本的数据 |
| DependentColumnFilter | `DependentColumnFilter` | 以某列的时间戳作为该行的时间戳 |
| SkipFilter | `SkipFilter` | 配合 ValueFilter，遇到不符合条件的 KeyValue 则跳过整行 |
| WhileMatchFilter | `WhileMatchFilter` | 遇到匹配的行停止扫描 |

#### 代码示例

```java
// 示例：查询列族为 "order"、列名前缀为 "item_"、值包含 "手机" 的数据
FilterList filterList = new FilterList(FilterList.Operator.MUST_PASS_ALL);

filterList.addFilter(new FamilyFilter(CompareOp.EQUAL,
    new BinaryComparator("order".getBytes())));
filterList.addFilter(new ColumnPrefixFilter("item_".getBytes()));
filterList.addFilter(new ValueFilter(CompareOp.EQUAL,
    new SubstringComparator("手机")));

Scan scan = new Scan();
scan.setFilter(filterList);
```

#### 注意事项

- **性能影响**：复杂 Filter 组合会导致扫描变慢，尽量精简 Filter 链；
- **功能组合**：可使用 `FilterList` 将多个过滤器用 AND/OR 逻辑组合；
- **Filter 还支持自定义**：实现 `Filter` 接口即可编写业务相关的过滤器。

> 📎 来源：[HBase 官方文档 - Filters](https://hbase.apache.org/book.html#filter)

## 2. HBase 的 Region 分裂（Split）机制是怎样的？如何避免频繁分裂导致的热点？

### 答案

#### Region 分裂原理

HBase 中的表按 RowKey 范围水平拆分成多个 **Region**，每个 Region 由一个 RegionServer 管理。当某个 Region 的数据量增长到一定阈值时，会自动分裂成两个子 Region。

**分裂触发条件**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `hbase.hregion.max.filesize` | 10GB | Region 中 HFile 总大小超过此值触发分裂 |
| `hbase.regionserver.region.split.policy` | 按配置策略 | 分裂策略类，可自定义 |

**分裂流程**：

```
1. RegionServer 向 HMaster 发送分裂请求
2. HMaster 更新元数据（Meta 表）
3. RegionServer 在本地将 Region 拆分为两个子 Region
4. 子 Region 的数据通过 HDFS 短路读取（Short-Circuit）临时共享父 Region 的 HFile 引用
5. 父 Region 下线，子 Region 上线开始服务
6. 大量 Compact 后，父子 HFile 完全独立
```

#### 分裂策略（Split Policy）

HBase 提供多种分裂策略，可通过 `hbase.regionserver.region.split.policy` 配置：

| 策略 | 说明 |
|------|------|
| **ConstantSizeRegionSplitPolicy** | 到达固定大小（10GB）分裂，旧版默认策略 |
| **IncreasingToUpperBoundRegionSplitPolicy** | 分裂大小随 Region 数量动态增长（新手友好） |
| **KeyPrefixRegionSplitPolicy** | 按 RowKey 前缀分组分裂，保证同一前缀的行不被分裂开 |
| **DelimitedKeyPrefixRegionSplitPolicy** | 按分隔符前的字符串前缀分组分裂 |
| **BusyRegionSplitPolicy** | 检测写入繁忙的 Region 提前分裂，避免热点 |
| **DisabledRegionSplitPolicy** | 禁用自动分裂，适合手动管理场景 |

#### 避免频繁分裂导致的热点

**问题**：频繁分裂会产生大量小 Region，导致：
- 分裂时的短期负载不均
- 小文件过多影响读取性能
- 管理开销增大

**解决方案**：

1. **预分区（Pre-Splitting）**：建表时指定分裂点
   ```bash
   create 'table', 'cf', {SPLITS => ['a', 'b', 'c', 'd', 'e']}
   ```

2. **选择合适的分裂策略**：生产环境推荐 `BusyRegionSplitPolicy` 或自定义策略

3. **合理设计 RowKey**：避免热点数据集中在同一 Region

4. **监控 Region 大小**：及时发现异常大 Region 并手动管理

> 📎 来源：[HBase 官方文档 - Region Splitting](https://hbase.apache.org/book.html#ops.region.splits)

## 3. HBase 的 Scan 查询机制是什么？如何理解 BlockCache 和热点缓存策略？

### 答案

#### Scan 机制详解

HBase 的 `Scan` 用于范围查询数据，是读取操作的核心 API。相比 `Get`（单行读取），`Scan` 可以遍历一个 RowKey 范围内的所有数据。

**Scan 的内部流程**：

```
创建 Scan 对象（指定 startRow / stopRow / Filter / Batch / Cache 等）
        │
        ▼
客户端通过 RPC 请求 RegionServer
        │
        ▼
RegionServer 依次扫描：
  ① MemStore（内存，SortedMap，按 RowKey 排序）
        │
        ▼
  ② BlockCache（热点数据缓存，LRU）
        │
        ▼
  ③ HFile（磁盘，按 RowKey 排序的 SSTable）
        │
        ▼
合并所有结果（去重、版本过滤）
        │
        ▼
返回迭代器（ResultScanner）
```

**关键 Scan 参数**：

| 参数 | 说明 |
|------|------|
| `setStartRow` / `setStopRow` | 指定扫描的起止 RowKey |
| `setBatch` | 每行返回的列数限制（适用于稀疏列） |
| `setCaching` | 每次 RPC 批量获取的 Result 条数（默认 2147483647） |
| `setCacheBlocks` | 是否缓存本次扫描的 Block（热点数据设为 true） |
| `setFilter` | 添加过滤器 |

#### BlockCache 机制

BlockCache 是 RegionServer 进程内存中的 **LRU（Least Recently Used）缓存**，用于缓存最近读取的 HFile Data Block。

**BlockCache 结构**：

| 区域 | 说明 |
|------|------|
| **LruBlockCache** | 单个 JvmHeap 大小的 LRU 缓存区 |
| **BucketCache** | 堆外内存（off-heap），可配置为 SSD |
| **CombinedBlockCache** | LruHeap + BucketCache 组合 |

**读写缓存命中流程**：

```
读取请求
   │
   ▼
① L1: MemStore → 命中则返回
   │
   ▼
② L2: BlockCache → 命中则返回 + 移到 LRU 头部
   │
   ▼
③ L3: HFile → 从磁盘读取 + 放入 BlockCache（如果 setCacheBlocks=true）
   │
   ▼
返回客户端
```

#### 热点缓存策略

1. **读多写少场景**：`setCacheBlocks(true)` 充分利用 BlockCache
2. **扫描全表场景**：`setCacheBlocks(false)` 避免污染热点数据缓存
3. **随机读取场景**：合理设置 `setCaching`，避免一次加载过多无用的 Block
4. **监控指标**：通过 HBase UI 或 JMX 监控 `blockCacheHitCount`、`blockCacheMissCount`

#### 最佳实践

```java
// 示例：高效的 Scan 配置
Scan scan = new Scan();
scan.setStartRow("user_0001".getBytes());
scan.setStopRow("user_9999".getBytes());
scan.setCaching(1000);       // 每次 RPC 获取 1000 条，减少 RPC 次数
scan.setCacheBlocks(true);   // 热点数据，开启缓存
scan.setBatch(10);           // 每行最多返回 10 个列

ResultScanner scanner = table.getScanner(scan);
for (Result result : scanner) {
    // 处理每行结果
}
```

## 4. HBase 的数据备份与灾难恢复方案有哪些？Snapshot 和 Replication 的区别是什么？

### 答案

#### 数据备份方案

HBase 提供多种数据备份机制，按备份粒度和业务场景选择：

**① 快照（Snapshot）备份**

快照是 HDFS 层面的技术，不复制数据，只记录 HFile 列表引用，因此创建速度极快（秒级）。

```bash
# 通过 HBase Shell 进行快照操作
# 1. 创建快照
snapshot 'table_name', 'snapshot_name'

# 2. 列出快照
list_snapshots

# 3. 恢复快照到新表
restore_snapshot 'snapshot_name'

# 4. 克隆快照（生成新表）
clone_snapshot 'snapshot_name', 'new_table_name'

# 5. 导出快照到其他集群
hbase org.apache.hadoop.hbase.snapshot.ExportSnapshot \
  -snapshot snapshot_name -copy-to hdfs://target-cluster/hbase
```

**② 导出表（Export）备份**

将 HBase 表数据导出为 HDFS 文件，可跨集群传输：

```bash
hbase org.apache.hadoop.hbase.mapreduce.Export table_name /hdfs/path/to/export \
  -versions 1 -timestamp 'yyyy-MM-dd'
```

**③ 完整 HDFS 备份**

直接备份 HBase 在 HDFS 上的数据目录（`/hbase/data/default/table_name/`）：

```bash
hdfs dfs -cp /hbase/data/default/table_name hdfs:///backup/table_name
```

**④ 复制（Replication）备份**

Replication 是 HBase 内置的异步复制机制，将数据实时复制到另一个集群。

#### Snapshot vs Replication 对比

| 维度 | Snapshot | Replication |
|------|----------|-------------|
| **备份类型** | 定时/手动快照 | 持续增量同步 |
| **数据时效** | 某一时刻的数据 | 近实时数据 |
| **目标位置** | 本集群或其他集群 HDFS | 另一个 HBase 集群 |
| **性能影响** | 极低（仅记录引用） | 中等（写入双写） |
| **故障恢复速度** | 依赖快照新鲜度 | 几乎零丢失 |
| **配置复杂度** | 低 | 中等（需配置 ZK、peer） |
| **适用场景** | 定时备份、测试环境恢复 | 跨机房容灾、高可用 |

#### 生产环境推荐方案

- **日常备份**：每天凌晨使用 Snapshot 创建快照，保留 7 天的快照历史
- **容灾备份**：配置 Replication 将数据实时同步到异地 HBase 集群
- **定期演练**：每季度从快照恢复验证备份有效性

> 📎 来源：[HBase 官方文档 - Snapshots](https://hbase.apache.org/book.html#ops.snapshots) 和 [Replication](https://hbase.apache.org/book.html#replication)

## 5. HBase 协处理器（Coprocessor）的原理是什么？Observer 和 Endpoint 有什么区别？

### 答案

#### 协处理器简介

HBase 协处理器是一种 **运行在 RegionServer 上的插件机制**，让用户可以在 RegionServer 端直接执行业务逻辑，而无需将数据传输到客户端再处理。类似于数据库的触发器和存储过程。

#### 协处理器的两种类型

**① Observer（观察者）**

类似于数据库触发器，hook 在特定的 HBase 操作前后自动执行。

| Observer 接口 | Hook 时机 | 典型用途 |
|--------------|-----------|---------|
| **RegionObserver** | Get / Put / Delete / Scan 等数据操作前后 | 数据验证、二级索引、权限控制 |
| **WALObserver** | WAL 写入前后 | WAL 审计 |
| **MasterObserver** | DDL 操作（create/alter/delete table）前后 | 审计日志、Schema 验证 |
| **RegionServerObserver** | RegionServer 启动/关闭/合并/分裂前后 | 集群管理 |

**② Endpoint（端点）**

类似于存储过程，允许在 RegionServer 上执行自定义的 RPC 通信协议，实现服务端聚合计算。

```java
// 示例：自定义 Endpoint 实现服务端 SUM 聚合
public class SumAggregationProtocol {
    // 定义协处理器协议接口
}

// 在 RegionServer 端实现协议
public class SumAggregationEndpoint extends BaseEndpointCoprocessor 
    implements SumAggregationProtocol {
    
    @Override
    public long sumColumn(byte[] family, byte[] qualifier) {
        // 直接在 RegionServer 内存中扫描计算，无需传输全表数据
        long sum = 0;
        InternalScanner scanner = getScanner(...);
        // ... 累加逻辑
        return sum;
    }
}
```

#### Observer vs Endpoint 对比

| 维度 | Observer | Endpoint |
|------|----------|----------|
| **类比** | 数据库触发器 | 数据库存储过程 |
| **执行方式** | 被动 hook，自动触发 | 主动调用（RPC） |
| **返回值** | 无直接返回值 | 可以返回计算结果 |
| **典型用途** | 数据过滤、权限校验、二级索引 | SUM/COUNT/AVG 聚合、复杂计算 |
| **调用次数** | 每条读写操作自动触发 | 显式调用 |

#### 加载方式

**① 静态加载（所有表生效）**

修改 `hbase-site.xml`：

```xml
<property>
  <name>hbase.coprocessor.user.region.classes</name>
  <value>com.example.MyObserver,com.example.MyEndpoint</value>
</property>
```

**② 动态加载（指定表生效）**

```bash
# 通过 HBase Shell
disable 'my_table'
alter 'my_table', 'coprocessor' => 'hdfs://path/to/coprocessor.jar|com.example.MyObserver|arg1,arg2'
enable 'my_table'

# 通过 Java API
TableDescriptorBuilder.newBuilder(tableName)
    .setCoprocessor("hdfs://path/to/coprocessor.jar", 1, "com.example.MyObserver", args)
    .build()
```

#### 典型应用场景

1. **二级索引**：Put 数据时自动写索引表（Observer）
2. **服务端聚合**：在 RegionServer 端计算 SUM/COUNT，无需全表传输（Endpoint）
3. **数据权限**：在 RegionObserver 中拦截读写请求，判断是否有权限
4. **审计日志**：每次写入记录操作人、操作时间到审计表
5. **数据压缩/加密**：在写入前/读取后进行自定义压缩或加解密
