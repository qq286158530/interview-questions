## 题目一：HBase 的整体架构是怎样的？包含哪些核心组件？

### 参考答案

HBase 是一个基于 **BigTable** 理论开发的列式存储 NoSQL 数据库，采用 **Master-Slave** 架构，主要包含以下核心组件：

### 1. Client
- 访问 HBase 的入口，使用 **RPC** 与 ZooKeeper 和 RegionServer 通信
- 包含缓存机制（Meta 缓存），减少对 ZooKeeper 的请求次数
- 负责定位数据所在的 Region，提供 **region 预切割** 和 **负载均衡** 支持

### 2. ZooKeeper
- 为 HBase 提供 **分布式协调服务**
- 存放整个 HBase 集群的 **元数据**（如 Meta 表位置、Master 地址）
- 实现 **故障检测** 和 **选举机制**（保证只有一个 Active Master）
- 管理所有 RegionServer 的状态（上线/下线）

### 3. Master（RegionServer 管理器）
- 负责管理所有 **RegionServer**，监控其健康状态
- 分配和调度 **Region**（表数据的分片）
- 处理 **DDL 操作**（建表、删表、修改表结构）
- 负责 **负载均衡**，将热点 Region 迁移到空闲 Server
- 不参与实际的数据读写，数据读写由 RegionServer 直接服务客户端

### 4. RegionServer
- **数据存储和读取的核心服务进程**，每个 RegionServer 负责管理一段或多段 Region
- 包含组件：
  - **WAL（Write-Ahead Log）**：预写日志，用于灾难恢复
  - **MemStore**：写缓存，基于 LSM-Tree 结构，先写内存
  - **HFile**：磁盘上的数据文件，基于 SSTable 结构
  - **BlockCache**：读缓存，使用 LRU 策略
- 定期执行 **flush**（MemStore 刷写到 HFile）和 **compact**（合并 HFile）

### 5. Region
- HBase 表的**水平分片**，每张表会被划分为多个 Region
- 每个 Region 包含 `[startKey, endKey)` 的行键范围
- 按行键字典序分布，不同 Region 分布在不同 RegionServer 上

### 架构流程简述

```
Client → ZooKeeper（获取 Meta 表位置） → RegionServer → MemStore → HFile
```

## 题目二：HBase 的 RowKey 设计原则有哪些？如何避免热点问题？

### 参考答案

### RowKey 设计核心原则

#### 1. **唯一性原则**
RowKey 必须唯一标识每一行数据，类似关系型数据库的主键。

#### 2. **长度控制原则**
- RowKey 是一个字符串，建议控制在 **8~1024 字节** 之间
- 越短越好，HBase 按字典序存储和比较，长 Key 会影响写入吞吐和内存占用

#### 3. **散列性原则（避免热点）**
热点（Hot Spot）是指大量请求集中在少数几个 Region 上，导致负载不均。

**常见散列方法：**

| 方法 | 说明 |
|------|------|
| **MD5/Hash** | 对原始 RowKey 取 Hash，缺点是难以阅读 |
| **盐（Salting）** | 在 RowKey 前加随机前缀（如 `a_`, `b_`），打散数据 |
| **反转（Reverse）** | 反转手机号等固定长度字段，使分布更均匀 |
| **时间戳反转** | 时间戳字段倒序，如 `MaxVersion - timestamp` |

#### 4. **查询维度原则**
RowKey 设计要考虑业务**主要查询场景**：
- **点查询**（精确查找）：RowKey 应包含主键字段
- **范围查询**：RowKey 前缀应能覆盖查询条件
- **多条件查询**：考虑用 **二级索引**（协处理器实现）

### 热点场景及解决方案

**场景1：按时间顺序写入**（如日志、IoT 时序数据）
- 问题：所有数据都往最新的 Region 写入，造成热点
- 解决间接寻址方案：在 RowKey 前加随机前缀，或使用哈希后缀

**场景2：用户ID 查询用户数据**
- 问题：同一用户的数据集中在同一 Region
- 解决：结合业务场景，考虑将用户ID + 时间戳组合作为 RowKey

### 最佳实践
```
# 好的设计示例：哈希前缀 + 业务字段
RowKey = MD5(userId).substring(0,6) + "_" + userId + "_" + timestamp
# 或反转
RowKey = reverse(userId) + "_" + timestamp
```

## 题目三：HBase 的存储结构是怎样的？请详细说明 MemStore 和 HFile 的工作原理。

### 参考答案

### 1. HBase 数据存储层次

```
Table (逻辑)
  └── Region (按 RowKey 范围水平切分)
        └── Store (每个 Column Family 对应一个 Store)
              ├── MemStore（写缓存，先写内存）
              └── HFile（磁盘文件，按 KeyValue 存储）
```

### 2. MemStore 工作原理

**核心机制：**
- 客户端写入时，数据先写入 **WAL（预写日志）**，然后写入 MemStore
- MemStore 是 **SkipListMap**（基于跳表的有序Map），保证数据按 RowKey 排序
- 当 MemStore 达到阈值（默认 `hbase.hregion.memstore.flush.size=128MB`）时，触发 **flush** 操作，将数据刷写到磁盘生成 HFile

**为什么先写 MemStore？**
- 内存写入速度远高于磁盘随机写入
- 配合 WAL 实现 **崩溃恢复**：即使 MemStore 数据丢失，也可从 WAL 恢复
- 合并多次小写入为一次大顺序写，提高磁盘 I/O 效率

**MemStore Flush 触发条件（满足任一即触发）：**
1. MemStore 大小超过 `hbase.hregion.memstore.flush.size`
2. Region 中所有 MemStore 总大小超过 `hbase.regionserver.global.memstore.size`（默认堆内存40%）
3. WAL 文件数量超过 `hbase.regionserver.max.logs`

### 3. HFile（HBase 磁盘文件格式）

**HFile 是 HBase 存储数据的基本文件格式，基于 Google BigTable 的 SSTable 结构。**

**HFile 结构（从旧到新版本 v1~v3，原理相同）：**

```
┌─────────────────────────────────────────────┐
│  Data Block（数据块）                        │
│    - 存储 KeyValue 对，按 RowKey 排序        │
│    - 默认 64KB，可配置大小                   │
│    - 支持块压缩（Snappy/GZ）                 │
├─────────────────────────────────────────────┤
│  Meta Block（元数据块，可选）               │
│    - 存储布隆过滤器等信息                    │
├─────────────────────────────────────────────┤
│  File Info（文件信息）                       │
│    - 文件元数据（最后修改时间、键值对数量等）│
├─────────────────────────────────────────────┤
│  Index Block（索引块）                       │
│    - 存储 Data Block 的键值范围索引          │
│    - 用于快速定位数据块                      │
├─────────────────────────────────────────────┤
│  Bloom Filter（布隆过滤器）                  │
│    - 快速判断某个 Key 是否存在于文件中       │
│    - 减少不必要的数据块读取                  │
├─────────────────────────────────────────────┤
│  Trailer（文件尾部）                         │
│    - 记录各区域的偏移量                      │
│    - HFile 版本信息                          │
└─────────────────────────────────────────────┘
```

**KeyValue 格式：**
```
KeyLen(4B) | ValueLen(4B) | RowKey | ColumnFamily | Qualifier | Timestamp | Type | Value
```

### 4. Compact 机制

随着数据不断写入，HFile 文件会越来越多，读操作需要合并多个 HFile，影响性能。HBase 通过 **Compact** 机制合并 HFile：

| 类型 | 说明 |
|------|------|
| **Minor Compaction** | 合并少量 HFile（默认3~10个），不删除过期/删除数据 |
| **Major Compaction** | 合并一个 Region 下的所有 HFile，**物理删除**过期数据和墓碑标记 |

## 题目四：HBase 的读写流程是怎样的？

### 参考答案

### 写流程（Put/Delete）

```
┌──────────────────────────────────────────────────────────────────┐
│                         写流程                                    │
├──────────────────────────────────────────────────────────────────┤
│ 1. Client 请求 ZooKeeper，获取 META 表的 Region 位置              │
│ 2. Client 访问 META RegionServer，获取目标 RowKey 所在的 Region   │
│ 3. Client 将数据写入对应 RegionServer 的 WAL（预写日志）           │
│ 4. WAL 写入成功后，数据写入 MemStore（写缓存）                     │
│ 5. 返回客户端写入成功                                              │
│                                                                   │
│ 触发 Flush 条件：MemStore 达到阈值，刷写成 HFile                  │
└──────────────────────────────────────────────────────────────────┘
```

**关键点：**
- WAL 是顺序写入（Append），先写 WAL 保证持久化
- 数据同时写入 MemStore，即使宕机也可从 WAL 恢复
- 写入是 **最终一致** 的（CAP 理论中选择可用性和分区容错，该模型在HBase中更侧重**可用性**）

### 读流程（Get/Scan）

```
┌──────────────────────────────────────────────────────────────────┐
│                         读流程                                    │
├──────────────────────────────────────────────────────────────────┤
│ 1. Client 请求 ZooKeeper，获取 META 表的 Region 位置               │
│ 2. Client 访问 META RegionServer，获取目标 RowKey 所在的 Region   │
│ 3. Client 读取对应 RegionServer，按以下顺序查找：                  │
│                                                                   │
│    [1] BlockCache（读缓存）→ 命中则直接返回                        │
│    [2] MemStore（写缓存）→ 命中则直接返回                          │
│    [3] 扫描磁盘上的 HFile → 按 KeyValue 合并返回最新版本          │
│                                                                   │
│ 4. 返回客户端查询结果                                              │
└──────────────────────────────────────────────────────────────────┘
```

**读合并（Read Optimization）：**
- 由于 HBase 中同一行可能有多个版本（不同 Timestamp），读取时需要合并 MemStore 和 HFile 数据，**返回最新版本**
- **墓碑标记（tombstone）**：删除操作不立即删除，而是在 Major Compaction 时物理删除

## 题目五：HBase 协处理器（Coprocessor）是什么？有哪两类？各自的使用场景是什么？

### 参考答案

### 协处理器概述

HBase 协处理器是运行在 **RegionServer** 上的扩展机制，允许用户在数据写入/读取时**拦截并执行自定义逻辑**，类似于数据库的触发器和存储过程。

### 两类协处理器

#### 1. Observer（观察者）—— 类似于"触发器"

**工作原理：** 钩子（Hook）机制，在特定事件发生时被调用执行自定义代码。

| 类型 | 触发时机 | 典型场景 |
|------|----------|----------|
| **PrePut** | 数据写入 MemStore **之前** | 数据校验、过滤、修改写入数据 |
| **PostPut** | 数据写入 MemStore **之后** | 更新二级索引、记录审计日志 |
| **PreGet** | 读取数据**之前** | 权限校验、数据过滤 |
| **PostGet** | 读取数据**之后** | 结果后处理、动态列加密 |
| **PreDelete** | 删除前 | 数据删除前的处理逻辑 |
| **PostDelete** | 删除后 | 清理关联的二级索引数据 |

**示例场景：**
```java
// 数据写入前校验字段值合法性
@Override
public void prePut(ObserverContext<RegionCoprocessorEnvironment> e,
                   Put put, WALEdit edit, Durability durability) throws IOException {
    // 校验逻辑，如字段不能为空、值范围校验等
}
```

#### 2. Endpoint（端点）—— 类似于"存储过程"

**工作原理：** 在 **RegionServer 端** 执行自定义计算逻辑，并将结果返回给客户端，支持**分布式并行前景（Projection）**。

**典型场景：**
- **聚合计算**：在每个 Region 上并行执行 Sum/Avg/Max/Min，最后在客户端合并结果
- **自定义过滤器**：在服务端过滤数据，减少网络传输
- **自定义业务逻辑**：在 Region 侧执行业务逻辑（如报表统计）

**示例：自定义聚合 Endpoint**
```java
// 定义协议接口
public interface ColumnSumProtocol extends CoprocessorProtocol {
    long getColumnSum(byte[] family, byte[] qualifier) throws IOException;
}
// 服务端实现，在每个 Region 上执行
public class ColumnSumEndpoint implements RegionObserver, ColumnSumProtocol {
    // 在 Region 侧计算列的和
}
```

### Observer vs Endpoint 对比

| 维度 | Observer | Endpoint |
|------|----------|----------|
| 触发方式 | 事件驱动 | 主动调用（RPC） |
| 返回值 | 无（副作用） | 有（计算结果） |
| 并行性 | 单 Region 内执行 | 多 Region 并行计算后聚合 |
| 数据修改 | 可以修改数据 | 一般只读（可修改） |
| 典型用途 | 数据校验、索引维护、审计 | 聚合计算、复杂查询 |

### 实际应用案例

1. **二级索引**：利用 PostPut/PostDelete 维护索引表
2. **权限控制**：PreGet 检查用户权限
3. **聚合查询**：自定义 Sum/Count 聚合
4. **数据分层存储**：根据时间戳将冷数据迁移到低频存储
