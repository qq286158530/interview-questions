## 题目一：Redis 是单线程还是多线程？为什么 Redis 这么快？

### 参考答案

**Redis 是单线程的吗？**

不完全是。**Redis 4.0 之前完全是单线程模型**，这里的"单线程"指的是**命令处理的核心循环（I/O 多路复用 + 事件分发）是单线程**的。Redis 6.0 引入了 **I/O 多线程**（`io-threads` 配置），将网络 I/O 的读写和解析操作并行化，但**命令执行仍然是单线程**。

**Redis 为什么这么快？**

1. **纯内存操作**：所有数据都在内存中，读取速度远快于磁盘数据库（纳秒级 vs 微秒级）
2. **单线程无锁竞争**：避免了上下文切换和锁开销，也不需要担心同步问题
3. **I/O 多路复用**：使用 epoll/select/kqueue 等机制，单个线程高效监听多个 socket
4. **高效数据结构**：Redis内部使用SDS（简单动态字符串）、跳表（Sorted Set）、压缩列表（Hash/List 小数据量时）等经过优化的数据结构，时间复杂度低
5. **C 语言实现**：贴近操作系统，效率高

>⚠️ 注意：Redis 的单线程指的是命令执行层，网络 I/O 和协议解析在6.0 后可以是多线程的。

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 参考答案

Redis提供了两种持久化机制：

### RDB（Redis Database）

- **原理**：在指定时间间隔内，将内存中的数据快照（Snapshot）以二进制格式写入磁盘，生成 `dump.rdb` 文件
- **优点**：
  - 恢复速度快（直接加载 RDB 文件）
  - 文件紧凑，适合备份和灾难恢复
  - 后台生成，不阻塞主进程
- **缺点**：
  - 可能丢失最近一次快照之后的数据（数据丢失窗口）
  - 若数据量很大，fork 子进程进行快照会消耗 CPU 和内存

### AOF（Append Only File）

- **原理**：将每一个写操作命令追加到 AOF 文件末尾，Redis 重启时通过重放（replay）所有命令来恢复数据
- **优点**：
  - 数据安全性更高（支持 `appendfsync` 配置：always/everysec/no）
  - 写命令采用追加方式，性能好
- **缺点**：
  - 文件体积通常比 RDB 大（可使用 `BGREWRITEAOF` 压缩）
  - 恢复速度比 RDB 慢（需要重放所有命令）
  - 存在 AOF 重写时的阻塞风险

### 对比总结

| 特性 | RDB | AOF |
|------|-----|-----|
| 恢复速度 | 快 | 慢 |
| 数据安全性 | 可能丢失数据 | 更安全 |
| 文件体积 | 小 | 大 |
| 宕机丢失 | 可能丢失快照后的数据 | 取决于 appendfsync 配置 |
| CPU 开销 | fork 时较高 | 持续写入开销 |

### 如何选择？

- **同时开启 RDB + AOF**：在数据安全性要求高的场景下使用，Redis 会优先用 AOF 恢复
- **仅用 RDB**：适合可以容忍少量数据丢失、对恢复速度要求高的场景
- **仅用 AOF**（不推荐）：RDB 快照方便备份和快速恢复，建议同时开启

## 题目三：Redis 的缓存淘汰策略（Eviction Policy）有哪些？什么时候触发淘汰？

### 参考答案

当 Redis 实例的内存使用达到 `maxmemory` 配置上限时，会根据配置的**淘汰策略**来决定如何腾出空间。

### 淘汰策略总览

| 策略 | 含义 |
|------|-----|
| `noeviction` | 不淘汰，返回写操作错误（默认策略） |
| `allkeys-lru` | 所有键中，淘汰最近最少使用（LRU）的键 |
| `volatile-lru` | 仅在设置了过期时间的键中，淘汰 LRU 的键 |
| `allkeys-random` | 所有键中，随机淘汰 |
| `volatile-random` | 仅在设置了过期时间的键中，随机淘汰 |
| `volatile-ttl` | 仅在设置了过期时间的键中，淘汰 TTL 最短（即将过期）的键 |
| `allkeys-lfu` | 所有键中，淘汰访问频率最低（LFU）的键 |
| `volatile-lfu` | 仅在设置了过期时间的键中，淘汰 LFU 的键 |

### 触发条件

- 当执行写命令时（如 `SET`、`INCR` 等），Redis 检测到内存使用量超过 `maxmemory`，就会触发淘汰
- 读操作不会触发淘汰（`noeviction` 下读仍然正常）

### LRU vs LFU

- **LRU（Least Recently Used）**：淘汰最长时间未被访问的对象，不区分访问频率
- **LFU（Least Frequently Used）**：淘汰访问频率最低的对象，适合热点数据区分更明显的场景

### 实际建议

- 若 Redis 作为**缓存**使用，推荐 `allkeys-lru`（不关心键是否设有过期时间）
- 若希望**同时做缓存和存储**，用 `volatile-lru` 让有过期时间的键优先被淘汰
- 需要区分访问热点的场景使用 `allkeys-lfu` 或 `volatile-lfu`

## 题目四：Redis 主从复制原理是什么？说说 Redis Sentinel（哨兵）的工作机制。

### 参考答案

### 主从复制（Replication）

Redis 主从复制采用 **异步复制** 机制，流程如下：

1. **配置**：通过 `replicaof<master-ip> <master-port>` 或配置文件让从节点连接主节点
2. **连接建立**：从节点发送 `PSYNC<runid> <offset>` 命令给主节点
3. **全量同步**：若从节点是全新的（runid 为 `?`），主节点发送完整的 RDB 快照给从节点
4. **增量同步**：之后主节点每次执行写命令，都会将命令发送给所有从节点（`PSYNC` 增量复制）
5. **断线重连**：从节点可基于 `runid` 和 `offset` 尝试增量同步，避免全量复制

**复制积压缓冲区（replication backlog）**：主节点维护一个固定大小的缓冲区，保存最近一段时间的写命令，用于增量复制。

### Redis Sentinel（哨兵）

Sentinel 是 Redis 的**高可用（HA）解决方案**，核心功能：

1. **监控（Monitoring）**：Sentinel 持续检查主节点和从节点是否正常运行
2. **自动故障转移（Failover）**：当主节点不可达时，Sentinel 会选举一个从节点**升为新主节点**，并通知其他从节点重新配置
3. **通知（Notification）**：当发生故障转移时，Sentinel 可以向客户端发送告警

### 重要概念

- **Quorum（法定人数）**：Sentinel 集群中，需要至少 N 个 Sentinel 节点同意才认为主节点客观下线（`down-after-milliseconds` 配置）
- **Leader选举**：多个 Sentinel 竞争对主节点执行故障转移，只有赢得多数票的 Sentinel 才会执行
- **主观下线 vs 客观下线**：
  - 主观下线：一个 Sentinel 节点认为主节点不可达
  - 客观下线：超过 quorum 数量的 Sentinel 都认为主节点不可达

### Sentinel vs Redis Cluster

| 特性 | Sentinel | Cluster |
|------|----------|---------|
| 功能 | 高可用（自动 failover） | 分片 + 高可用 |
| 数据分片 | 无 | 有（16384 个 hash slot） |
| 客户端感知 | 需要订阅 +switch-master 事件 | 客户端直连任意节点 |
| 适用场景 | 单主多从 | 大数据量 + 高并发 |

## 题目五：Redis 的数据类型有哪些？各自的使用场景是什么？

### 参考答案

Redis 支持丰富的数据类型，远不止简单的 Key-Value：

| 数据类型 | 底层实现 | 典型使用场景 |
|----------|----------|-------------|
| **String** | SDS（简单动态字符串） | 缓存、计数器、分布式锁（SETNX） |
| **Hash** | 压缩列表 / 哈希表 | 存储对象（用户信息、配置） |
| **List** | 压缩列表 / 双向链表 | 消息队列（FIFO）、文章列表、时间线 |
| **Set** | 哈希表 / 整数集合 | 标签系统、好友关系、去重 |
| **Sorted Set（Zset）** | 跳表 + 哈希表 | 排行榜、实时评分系统、带权重的任务队列 |
| **Bitmap** | 位数组 | 用户签到、活跃用户统计、权限控制 |
| **HyperLogLog** | 概率数据结构 | UV 统计（独立访客数量） |
| **Geospatial** | GeoHash | 地理位置相关（LBS、附近的人） |
| **Stream** | Radix Tree | 日志流、消息队列（替代 Pub/Sub 的消费确认） |

### 重点数据类型详解

**String**
- 最大存储 512MB（远超 Memcached 的 1MB）
- 常用命令：`SET`、`GET`、`INCR`、`INCRBY`、`SETEX`（设置值+过期时间）
- 应用：Session 共享、API 缓存、分布式 ID 生成（`INCR`）

**Hash**
- 适合存储对象，类比 MySQL 的一条记录
- 常用命令：`HSET`、`HGET`、`HGETALL`
- 注意：字段过多时退化为哈希表，内存开销变大

**Sorted Set（Zset）**
- 每个元素关联一个分数（score），按分数排序
- 典型应用：**排行榜**：用 `ZADD` 更新分数，`ZREVRANGE` 获取 Top N
- 时间复杂度：`ZADD`、`ZREVRANGE` 均为 O(log N)（跳表实现）

**Stream**
- Redis 5.0 引入，类比 Kafka 的消费组模式
- 支持 `XADD`（生产）、`XREAD`（消费）、`XGROUP`（消费组）
- 与 Pub/Sub 的区别：Stream 支持消息持久化和消费确认（ACK）
