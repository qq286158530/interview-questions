## 题目一：Flink 架构原理 — JobManager 与 TaskManager 的职责

**问题：** 请描述 Apache Flink 的核心架构，JobManager 和 TaskManager 分别负责什么？Flink 的任务调度和资源管理是如何工作的？

**答案：**

Flink 采用经典的 Master-Slave 架构，由 **JobManager**（Master）和 **TaskManager**（Worker）组成。

### JobManager（Master）职责

JobManager 是整个 Flink 集群的协调者，负责三项核心组件：

| 组件 | 职责 |
|------|------|
| **ResourceManager** | 管理 TaskManager 上的 slot 资源，负责插槽的分配与释放。对接 YARN、Kubernetes 等多种资源管理器 |
| **Dispatcher** | 提供 REST 接口接收 Flink 应用提交，启动 JobMaster，并运行 Flink WebUI |
| **JobMaster** | 负责管理单个 JobGraph 的执行，协调任务调度、Checkpoint、故障恢复等 |

高可用模式下可以部署多个 JobManager，其中一个为 Leader，其余为 Standby。

### TaskManager（Worker）职责

- 执行 JobManager 分配的具体任务（subtask）
- 缓存和交换数据流
- 以 JVM 进程运行，每个 TaskManager 包含若干 **task slot**，slot 是 Flink 的最小资源调度单位

### 工作流程

1. Client 将用户程序编译为 **JobGraph**，提交给 Dispatcher
2. Dispatcher 启动 JobMaster，JobMaster 向 ResourceManager 请求执行任务所需的 slot 资源
3. ResourceManager 向 TaskManager 申请 slot，TaskManager 汇报可用 slot
4. 分配好 slot 后，JobMaster 将任务分发到对应的 TaskManager 执行
5. TaskManager 之间通过数据流进行数据交换

> **来源：** [Apache Flink 官方文档 - Flink Architecture](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/flink-architecture/)

## 题目二：Watermark（水位线）机制与乱序数据处理

**问题：** 什么是 Flink 中的 Watermark？它是如何解决事件时间（Event Time）下的数据乱序问题的？

**答案：**

### 什么是 Watermark

Watermark（水位线）是 Flink 用于处理**乱序事件**和**延迟数据**的核心机制。它是一个时间戳标记，表示"所有时间戳小于该 watermark 的数据都已经到达"。

### 核心思想

在基于 Event Time 的流处理中，由于网络延迟、分布式系统等原因，数据到达的顺序可能和事件发生的顺序不一致。Watermark 用来告诉算子：**在此之前的数据应该都到齐了，可以触发窗口计算了**。

```
Event Time:  1 ---- 3 ---- 2 ---- 5 ---- 4 ---- 8 ---- 7 ----
Watermark:   -------- w(4) ------------------------------->
             (表示时间 <= 4 的数据都该到了，窗口 [0,5) 可以关闭)
```

### 两种 Watermark 策略

| 策略 | 说明 |
|------|------|
| **有序 Watermark** | 适用于数据基本有序的场景，`WatermarkStrategy.forMonotonousTimestamps()` |
| **乱序 Watermark** | 适用于乱序严重的场景，允许一定的延迟，`WatermarkStrategy.forBoundedOutOfOrderness(Duration.ofSeconds(10))` |

### 窗口触发时机

- 当 watermark 时间 `w >= 窗口结束时间` 时，窗口触发计算
- `allowedLateness` 方法：允许窗口在关闭后额外等待一段时间，处理迟到数据
- 迟到太久（超过 `allowedLateness`）的数据默认被丢弃，或通过 `sideOutputLateData` 侧输出到其他流

### 经典面试点

> **Watermark 是一条单调递增的时间戳。** 当某个窗口的关闭时间戳 T 到达后，Flink 才会触发该窗口的计算。Watermark 的核心价值是**平衡延迟和准确性**——等待时间越长，准确性越高，但延迟也越大。

> **来源：** [Apache Flink 官方文档 - Event Time & Watermarks](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/event-time/)

## 题目三：Checkpoint 与 Flink 的故障恢复机制

**问题：** Flink 的 Checkpoint 机制是如何工作的？为什么需要 Checkpoint？它和 Savepoint 有什么区别？

**答案：**

### 为什么需要 Checkpoint

Flink 是有状态流处理引擎，任务在运行过程中会将中间状态（state）存储在内存或磁盘。当 Flink 作业发生故障（如 TaskManager 宕机）时，需要从某个一致的检查点恢复，否则所有已处理的数据都会丢失。Checkpoint 提供了**exactly-once** 或 **at-least-once** 的语义保障。

### Checkpoint 工作原理

Flink 采用 **Chandy-Lamport 分布式快照算法**的变体（Barrier 对齐机制）：

1. **JobManager** 定期向所有 Source 算子发送 Checkpoint 触发信号
2. Source 算子收到信号后，插入 **Checkpoint Barrier**（检查点屏障），并将自己的状态快照写入持久化存储（HDFS、S3 等）
3. Barrier 随数据流向下游广播，各算子收到所有输入的 Barrier 后进行状态快照
4. 所有算子快照完成后，Checkpoint 成功

```
Source ---barrier A---> Operator1 ---barrier A---> Operator2
         \---barrier B---/ (等待第二个 barrier 对齐)
```

### 关键配置参数

| 参数 | 说明 |
|------|------|
| `checkpoint interval` | Checkpoint 周期，默认 10 分钟 |
| `checkpoint timeout` | Checkpoint 超时时间 |
| `min pause between checkpoints` | 两个 Checkpoint 最小间隔 |
| `exactly-once` vs `at-least-once` | 快照语义，exactly-once 会有 Barrier 对齐开销 |

### Checkpoint vs Savepoint

| 特性 | Checkpoint | Savepoint |
|------|------------|-----------|
| **触发方式** | Flink 自动周期性触发 | 手动触发 |
| **用途** | 故障自动恢复 | 计划性停止/升级/重启 |
| **清理策略** | job 删除时自动清理 | 手动管理 |
| **格式兼容性** | 可能因 job 变化而不兼容 | 格式稳定，向后兼容 |
| **状态后端** | 支持 RocksDB/Heap | 支持 RocksDB/Heap |

### 常见面试追问

- **Barrier 对齐是什么？** 在 exactly-once 模式下，算子需要等待所有输入 Channel 的 Barrier 到达后才进行快照，这会导致短暂的暂停
- **当算子有多个输入时如何处理乱序 Barrier？** Flink 会缓存先到的数据，等所有 Barrier 齐了才触发 Checkpoint

> **来源：** [Apache Flink 官方文档 - Checkpointing](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/checkpoints/)

## 题目四：Flink 窗口函数详解

**问题：** Flink 支持哪些窗口类型？滚动窗口、滑动窗口、会话窗口有什么区别？请举例说明窗口函数的用法。

**答案：**

### 窗口类型概览

Flink 支持四大类窗口：

```
┌─────────────────────────────────────────────────┐
│                    窗口类型                       │
├─────────────┬──────────────┬────────────────────┤
│ Time Window │ Count Window │ Session Window      │
├─────────────┼──────────────┼────────────────────┤
│ 滚动窗口    │ 滚动计数窗口  │ Session Gap 窗口    │
│ 滑动窗口    │ 滑动计数窗口  │ 全局窗口            │
└─────────────┴──────────────┴────────────────────┘
```

### 1. 滚动窗口（Tumbling Window）

- **特点：** 窗口长度固定，不重叠
- **适用场景：** 按固定时间段汇总数据
```java
DataStream<Tuple2<String, Long>> windowed = stream
    .keyBy(0)
    .window(TumblingEventTimeWindows.of(Time.seconds(10)))  // 每10秒一个窗口
    .sum(1);
```

### 2. 滑动窗口（Sliding Window）

- **特点：** 窗口长度固定，但可以重叠（由 slide 参数控制）
- **适用场景：** 需要平滑聚合、频繁更新的场景（如求5分钟窗口内滑动1分钟的均值）
```java
.window(SlidingEventTimeWindows.of(Time.minutes(5), Time.minutes(1)))
// 窗口长度5分钟，每1分钟滑动一次
```

### 3. 会话窗口（Session Window）

- **特点：** 由数据之间的间隔（Gap）动态决定窗口边界，没有固定大小
- **适用场景：** 用户行为分析，按操作间隔划分会话
```java
.window(EventTimeSessionWindows.withGap(Time.minutes(5)))
// 5分钟内无新数据则窗口关闭
```

### 4. 全局窗口（Global Window）

- **特点：** 所有相同 key 的数据进入同一个窗口，需要配合触发器使用
- **适用场景：** 自定义触发逻辑

### 窗口函数（Window Functions）

| 类型 | 说明 | 性能 |
|------|------|------|
| **ReduceFunction** | 增量聚合，收到每条数据时立即更新状态 | 最优 |
| **AggregateFunction** | 增量聚合，支持更灵活的返回值类型 | 优 |
| **ProcessWindowFunction** | 全量窗口数据一起处理，能访问窗口元信息 | 开销大 |

**最佳实践：** 推荐使用 `ReduceFunction` 或 `AggregateFunction` 配合 `ProcessWindowFunction` 的增量+全量混合模式。

> **来源：** [Apache Flink 官方文档 - Windows](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream/operators/windows/)

## 题目五：Flink 状态管理 — 状态后端与键控状态

**问题：** Flink 的状态管理是如何实现的？有哪些状态后端（State Backend）？键控状态（Keyed State）和算子状态（Operator State）有什么区别？

**答案：**

### 为什么需要状态

流处理中的"状态"指**算子在处理过程中需要记住的信息**，例如：
- 窗口的聚合结果
- 数据匹配的模式（CEP）
- 机器学习模型的参数
- 每小时累计的计数

### 状态类型

```
Flink 状态
├── Keyed State（键控状态）
│   ├── ValueState<T>
│   ├── ListState<T>
│   ├── ReducingState<T>
│   ├── AggregatingState<IN, OUT>
│   └── MapState<K, V>
│   (作用域绑定到当前处理的 key，只能在 keyBy 之后使用)
│
└── Operator State（算子状态）
    ├── ListState<T>（列表形式，各子任务状态合并）
    ├── UnionListState<T>（故障恢复时广播给所有子任务）
    └── BroadcastState<K, V>（广播状态，所有并行实例共享）
```

### 三种 State Backend

| 后端 | 存储介质 | 适用场景 |
|------|---------|---------|
| **HashMapStateBackend** | 内存（Java Heap） | 小状态、低延迟，开发/测试环境 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘 + 内存） | 大状态、增量 Checkpoint，生产环境 |
| **NestedStateRocksDBBackend** | 改进版 RocksDB | Flink 1.13+ |

### 各状态后端详解

**1. HashMapStateBackend**
```java
env.setStateBackend(new HashMapStateBackend());
// 状态存储在 TaskManager 的 JVM 堆内存中
// 快速但受内存限制，适合状态很小的场景
```

**2. EmbeddedRocksDBStateBackend**
```java
env.setStateBackend(new EmbeddedRocksDBStateBackend());
// 状态溢出到本地 RocksDB，支持增量 Checkpoint
// 状态可超过内存大小，适合生产环境大状态
// 序列化方式：Flink 的Kyro或自定义Serializer
```

### 状态过期与清理

- **TTL（Time-To-Live）：** 可以为状态设置过期时间，自动清理无效状态
```java
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.hours(1))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
    .build();
ValueStateDescriptor<String> descriptor = new ValueStateDescriptor<>(
    "last-event", String.class);
descriptor.enableTimeToLive(ttlConfig);
```

### 经典面试问题

> **Keyed State 和 Operator State 的核心区别是什么？**
> - **Keyed State：** 按 key 隔离，每个 key 有独立的状态值，只能在 keyBy 之后的算子中使用
> - **Operator State：** 所有并行实例共享同一个状态池，通常用于 Source 算子（如 Kafka offset）或需要与 key 无关的共享数据

> **RocksDB 的优缺点？**
> - ✅ 支持大状态（可超过内存）
> - ✅ 支持增量 Checkpoint
> - ❌ 序列化/反序列化开销大
> - ❌ 延迟比内存状态高

> **来源：** [Apache Flink 官方文档 - State & State Backend](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/stateful-stream-processing/)
