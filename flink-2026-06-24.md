## 题目 1：Flink 的整体架构是怎样的？JobManager 和 TaskManager 的职责是什么？

### 参考答案

Flink 采用**主从架构**，由 JobManager（主节点）和 TaskManager（工作节点）组成。

**JobManager 职责：**
- **JobMaster**：负责单个 Job 的执行，协调整个 Job 的执行流程，包括任务调度、检查点协调等
- **ResourceManager**：管理 TaskManager 的slot（资源槽），负责 slot 的申请与释放
- **Dispatcher**：接收用户提交的 Job，为每个 Job 创建一个 JobMaster，并提供 REST 接口

**TaskManager 职责：**
- 是 Flink 的工作进程，负责**执行具体的任务**（算子实例）
- 以 **Slot** 为单位管理资源，每个 TaskManager 会把自己管理的资源切分成多个 Slot
- 启动后向 ResourceManager 注册自己的 slot
- 与 JobManager 交互，接收任务分配并执行

**作业执行流程：**
1. 用户代码 → StreamGraph → JobGraph → ExecutionGraph
2. JobManager 负责任务调度，将 JobGraph 中的算子（Operator）分配给 TaskManager 执行
3. TaskManager 之间通过 **StreamManager** 进行数据传递（Netty 网络）

**Source:** [Apache Flink 官方文档 - 架构](https://nightlies.apache.org/flink/flink-docs-release-1.18/zh/docs/concepts/flink-architecture/)

## 题目 2：什么是 Watermark（水位线）？它解决的是什么问题？如何生成？

### 参考答案

### 问题背景

在**乱序事件流**场景中，事件到达的顺序可能与事件时间（Event Time）不一致。例如：网络延迟导致本应先到达的 `10:00` 数据反而比 `10:05` 的数据后到。如果基于事件时间做窗口计算，可能导致：
- **窗口已触发关闭**，但迟到数据还未到达 → 数据丢失
- **窗口无限等待** → 计算无法完成

### Watermark 的定义

**Watermark 本质上是一个时间戳**，表示「所有事件时间 ≤ T 的数据都已到达（或已处理）」。

> Watermark(t) = 「当水位线到达时间 T 时，我们相信不会再有时间戳 < T 的事件到达」

### Watermark 的生成方式

**1. Punctuated（周期性标记）**
```java
DataStream<MyEvent> stream = env.addSource(new MySource());
stream.assignTimestampsAndWatermarks(
    new WatermarkStrategy<MyEvent>() {
        @Override
        public WatermarkGenerator<MyEvent> createWatermarkGenerator(WatermarkGeneratorSupplier.Context context) {
            return new MyPunctuatedGenerator(); // 自定义，每条数据都可能发出watermark
        }
    }
);
```

**2. Periodic（周期性水位线）**
```java
.assignTimestampsAndWatermarks(
    WatermarkStrategy
        .<MyEvent>forBoundedOutOfOrderness(Duration.ofSeconds(10)) // 允许最大乱序时间10秒
        .withTimestampAssigner(...)
);
```
- 使用 `BoundedOutOfOrdernessWatermarks`，当 `maxTimestamp - lastEmittedWatermark > maxOutOfOrderness` 时周期性发射

### 窗口与 Watermark 的配合

- 窗口的**关闭条件**：`watermark >= window_end_time`
- 迟到数据（watermark 已过窗口结束时间）处理策略：
  - `allowedLateness(Duration)`：设置额外等待时间
  - `sideOutputLateData(OutputTag)`：将迟到数据侧输出

## 题目 3：Flink 的 Checkpoint（检查点）和 Savepoint 有何区别？如何实现Exactly-Once 语义？

### 参考答案

### Checkpoint vs Savepoint

| 特性 | Checkpoint | Savepoint |
|------|-----------|-----------|
| **触发方式** | Flink 自动周期性触发 | 用户手动触发（CLI/API） |
| **存储位置** | 由 StateBackend 决定（TaskManager 本地或远程） | 指定的外部存储目录 |
| **Job 重启后是否保留** | 通常不保留（可能被清理） | 保留，用于恢复和迁移 |
| **目标** | 快速恢复失败 Job | 计划性维护、版本管理 |
| **格式** | 简化的二进制格式 | 标准的二进制格式 |

### Checkpoint 实现原理

Flink 基于 **Chandy-Lamport 分布式快照算法** 实现 Checkpoint，核心流程：

1. **JobManager 定期向所有 Source 算子发送 Checkpoint 触发消息（CheckpointCoordinator）**
2. **Barrier 对齐机制**：
   - Source 收到 Barrier 后，**暂停发送数据**，向下游广播 Barrier
   - 下游算子等待**所有输入通道的 Barrier 都到达**（对齐），才快照状态
   - 对齐完成后，Snapshot 状态并向下游发送 Barrier
3. **异步快照**：算子将状态写入远程存储（如 HDFS/S3）
4. **Checkpoint 完成**：所有算子快照完成，JobManager 记录 Checkpoint 元数据

### Exactly-Once 语义实现

Flink 通过以下三层保证 Exactly-Once：

1. **数据源 Barrier 对齐**：保证 Source 数据的消费位置被精确记录
2. **算子状态快照**：所有算子状态与 Barrier 对齐点一致
3. **幂等写入或事务写入**：
   - **幂等 Sink**：重复计算结果相同（如 Redis overwrite）
   - **事务 Sink**：使用两阶段提交（2PC），预提交 → Checkpoint 完成 → 真正提交
   - 支持的 Connector：Kafka（支持 Exactly-Once）、JDBC（事务）、FileSink（2PC）

### 常见配置参数

```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
env.enableCheckpointing(60000);  // 60秒触发一次Checkpoint
env.getCheckpointConfig().setCheckpointTimeout(120000);
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30000);
env.getCheckpointConfig().setMaxConcurrentCheckpoints(1);
env.getCheckpointConfig().setTolerableCheckpointFailureNumber(3);
```

**Source:** [Apache Flink 官方文档 - Checkpoint](https://nightlies.apache.org/flink/flink-docs-release-1.18/zh/docs/ops/state/checkpoints/)

## 题目 4：Flink 窗口有哪些类型？滚动窗口、滑动窗口、会话窗口的区别是什么？

### 参考答案

### 窗口分类

```
┌─────────────────────────────────────────────────────────┐
│                      窗口类型                             │
├──────────────┬──────────────┬──────────────┬────────────┤
│  Time Window │ Count Window │ Session Window│ Global Window│
├──────────────┼──────────────┼──────────────┼────────────┤
│ Tumbling     │              │              │            │
│ Sliding      │              │              │            │
│ Count        │              │              │            │
└──────────────┴──────────────┴──────────────┴────────────┘
```

### 1. 滚动窗口（Tumbling Window）

- **特点**：窗口长度固定，**不重叠**，首尾相接
- 适用场景：按固定时间段统计，如「每5分钟的UV」
- 示例：每个整点小时、独立统计

```java
// 滚动时间窗口：每5分钟一个窗口
DataStream<Tuple2<String, Long>> counts = stream
    .keyBy(0)
    .window(TumblingProcessingTimeWindows.of(Time.minutes(5)))
    .sum(1);

// 滚动计数窗口：每100条数据一个窗口
.keyBy(0)
.window(CountWindow.of(100))
.sum(1);
```

### 2. 滑动窗口（Sliding Window）

- **特点**：窗口长度固定，但**可重叠**，由 `窗口长度` 和 `滑动步长` 两个参数决定
- 滑动步长 < 窗口长度：重叠，窗口有重叠区域
- 滑动步长 = 窗口长度：退化为滚动窗口
- 滑动步长 > 窗口长度：跳跃，可能漏数据
- 适用场景：「最近5分钟的平均值，每1分钟更新一次」

```java
// 滑动时间窗口：窗口长度10分钟，滑动步长5分钟
.window(SlidingProcessingTimeWindows.of(Time.minutes(10), Time.minutes(5)))

// 滑动计数窗口：最近100条，每20条滑动一次
.window(SlidingEventTimeWindows.of(Time.of(100, TimeUnit.SECONDS), Time.of(20, TimeUnit.SECONDS)))
```

### 3. 会话窗口（Session Window）

- **特点**：根据**活跃间隙**动态创建窗口，窗口边界由数据本身决定
- 无数据的时间超过指定 `gap` → 窗口关闭
- 适用场景：用户行为分析，用户的操作会话

```java
// 会话窗口：会话间隔超过10秒则关闭当前会话
.window(ProcessingTimeSessionWindows.withGap(Time.seconds(10)))

// 动态间隙会话窗口
.window(EventTimeSessionWindows.withDynamicGap(new SessionWindowTimeGapExtractor<Tuple2<String, Long>>() {
    @Override
    public double extract(Tuple2<String, Long> element) {
        return element.f1; // 根据数据内容动态决定间隙
    }
}))
```

### 窗口函数（Window Functions）

| 类型 | 说明 | 示例 |
|------|------|------|
| `reduce()` | 增量聚合，效率高 | 求和、计数 |
| `aggregate()` | 更加灵活的增量聚合 | 多状态聚合 |
| `process()` | 全量窗口函数，获取窗口全部数据 | 复杂计算、排序 |

### 全窗口 vs 增量窗口对比

```
增量窗口（Reduce/Aggregate）：
  数据来了 → 增量聚合 → 输出
  ✅ 内存占用低、延迟低
  ❌ 无法访问窗口全部数据

全窗口（Process）：
  数据来了 → 缓存 → 等窗口触发 → 全量计算 → 输出
  ✅ 可访问窗口全部数据
  ❌ 内存占用高、延迟高
```

## 题目 5：Flink 的状态管理机制是什么？Operator State 和 Keyed State 有何区别？Flink 如何实现故障恢复？

### 参考答案

### Flink 状态管理概述

Flink 是**有状态流处理**框架，状态是指算子在处理过程中需要**记住的信息**：
- 中间计算结果（如 Sum、Count）
- 数据缓冲（如窗口缓存的数据）
- 与外部系统的交互状态（如 Kafka 偏移量）

### 状态类型

#### 1. Operator State（算子状态）

- **作用域**：整个算子实例，所有数据共享同一份状态
- **常见场景**：Source 算子的 Kafka 分区偏移量、Socket 数据源位置
- **状态分布方式**：
  - `ListState`：算子状态分为多个列表，并行度变化时可重新分配
  - `UnionListState`：故障恢复时，所有并行实例都收到完整状态列表
  - `BroadcastState`：广播状态，所有并行实例持有相同数据

```java
public class MySource extends SourceFunction<String> implements CheckpointedFunction {
    private ListState<Long> offsetState;

    @Override
    public void initializeState(FunctionInitializationContext context) throws Exception {
        offsetState = context.getOperatorStateStore().getListState(
            new ListStateDescriptor<>("offset", Long.class)
        );
    }

    @Override
    public void run(SourceContext<String> ctx) throws Exception {
        // 使用 offsetState 恢复位置
    }
}
```

#### 2. Keyed State（键控状态）

- **作用域**：绑定到 Key（keyBy 后的 Key），每个 Key 独立一份状态
- **只能在 KeyedStream 上使用**（keyBy 之后）
- **状态类型**：
  - `ValueState<T>`：单一值
  - `ListState<T>`：列表
  - `MapState<K, V>`：键值对
  - `ReducingState<T>`：自动聚合
  - `AggregatingState<IN, OUT>`：聚合

```java
DataStream<Tuple2<String, Long>> stream = ...;
stream.keyBy(0)
    .map(new RichMapFunction<Tuple2<String, Long>, Tuple2<String, Long>>() {
        private ValueState<Long> countState;

        @Override
        public void open(Configuration parameters) {
            countState = getRuntimeContext().getState(
                new ValueStateDescriptor<>("count", Long.class)
            );
        }

        @Override
        public Tuple2<String, Long> map(Tuple2<String, Long> value) throws Exception {
            Long count = countState.value() == null ? 0L : countState.value();
            count++;
            countState.update(count);
            return new Tuple2<>(value.f0, count);
        }
    });
```

### Operator State vs Keyed State 对比

| 特性 | Operator State | Keyed State |
|------|----------------|-------------|
| 作用范围 | 算子级别 | Key 级别 |
| 是否需要 keyBy | 否 | 是 |
| 状态存储 | 算子实例共享 | 每个 Key 独立存储 |
| 并行度变化 | 需要状态重分配 | 自动重分区 |
| 常见用途 | Kafka 偏移量、Source 位置 | 聚合计算、CEP 模式匹配 |

### 状态后端（State Backend）

| 后端 | 适用场景 | 存储位置 |
|------|---------|---------|
| **HashMapStateBackend** | 小状态、高并发 | JVM Heap |
| **EmbeddedRocksDBStateBackend** | 大状态、长窗口 | RocksDB（本地磁盘） + 增量 Checkpoint 到远程存储 |

```java
// 配置状态后端
env.setStateBackend(new HashMapStateBackend());
// 或 RocksDB（需引入依赖）
env.setStateBackend(new EmbeddedRocksDBStateBackend());

// 增量Checkpoint（仅RocksDB支持）
env.getCheckpointConfig().enableIncrementalCheckpointing();
```

### 故障恢复机制

Flink 的故障恢复依赖于 **Checkpoint + 状态恢复**：

1. **正常运行时**：JobManager 周期性触发 Checkpoint，各算子将状态快照到远程存储
2. **故障时**：
   - JobManager 通知所有 Source 算子从最近成功的 Checkpoint 元数据恢复
   - Source 算子从快照中读取 Kafka 偏移量，重新消费数据
   - 下游算子从快照中恢复 Keyed State / Operator State
3. **数据一致性**：由于 Source 重新从 Checkpoint 位置消费 + 算子状态恢复，实现了 **Exactly-Once 端到端语义**

### 状态过期与清理

```java
// 设置状态 TTL（Time-To-Live）
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.hours(1))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)  // 每次读写更新TTL
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)  // 不返回过期数据
    .build();

ValueStateDescriptor<String> stateDescriptor = new ValueStateDescriptor<>("myState", String.class);
stateDescriptor.enableTimeToLive(ttlConfig);
```
