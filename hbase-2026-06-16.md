## 题目一：HBase 的整体架构是怎样的？Master 和 RegionServer 分别负责什么？

### 参考答案

HBase 采用 **Master-Slave（主从）架构**，核心组件如下：

| 组件 | 作用 |
|------|------|
| **ZooKeeper** | 负责集群协调、Master 选举、 元数据（-ROOT-、META 表）地址保存 |
| **HMaster** | 负责 Table 的增删改查、Region 分配、负载均衡、RegionServer 故障检测 |
| **RegionServer** | 负责实际的数据读写、MemStore 刷写、HFile 合并、Minor/Major Compaction |
| **Region** | 表按 RowKey 水平切分的数据存储单元，一个 Region 对应一个列族的数据 |

**读写流程简述：**
1. 客户端连接 ZooKeeper，获取 `-ROOT-` 表位置
2. 从 `-ROOT-` 表查到 `META` 表中对应 Region 的位置
3. 从 `META` 表查到数据所在 RegionServer 的地址
4. 直连对应 RegionServer 进行读写（客户端会缓存 meta 位置）

**Master 与 RegionServer 的关系：**
- Master 不参与实际数据读写，只负责元数据管理和调度
- RegionServer 挂掉后，Master 会将其上的 Region 迁移到其他活节点（通过 WAL 回放完成数据恢复）
- 为保证高可用，可以部署多台 Master（Backup Master）

📎 **来源：** [HBase 官方文档 - Architecture](https://hbase.apache.org/book.html#architecture)

## 题目二：HBase 的 RowKey 设计有哪些原则？如何避免热点（Hotspotting）问题？

### 参考答案

### RowKey 设计四大原则

1. **长度原则**
   - RowKey 最大长度 64KB，建议控制在 16~100 字节以内
   - 越短越好，过长的 RowKey 会占用 HFile 索引空间，影响查找效率

2. **唯一性原则**
   - RowKey 必须唯一标识一行数据，类似于关系型数据库的主键

3. **散列（Hash）原则**
   - 避免单调递增或递减的 RowKey（如时间戳 `20260616001`、`20260616002`）
   - 单调递增导致新数据全部写入同一个 Region，造成热点

4. **查询维度前置原则**
   - 将经常用于范围查询的字段放在 RowKey 前缀
   - 利用 HBase 按 RowKey 字典序排序的特性实现高效范围扫描

### 热点解决方案

| 方案 | 说明 |
|------|------|
| **反转（Reverse）** | 对单调字段（如时间戳）做反转，分散写入压力 |
| **Hash 扰动** | 在 RowKey 前加 Hash 前缀（如 `MD5(rowkey).substring(0,3) + original_key`） |
| **加盐（Salting）** | 在 RowKey 前加随机前缀，将数据分散到不同 Region |
| **随机数** | 末尾加随机数，适合只按主键查询、不需要范围查询的场景 |
| **业务组合** | 将多个业务字段组合（如 `userId + timestamp`），利用业务特性打散 |

**示例：避免时间序列热点**
```
# 错误：所有新数据写入同一 Region
rowkey: 20260616100000

# 正确：加盐 + 组合
rowkey: 03_20260616100000  # 前缀 03 为 Hash 取模结果
```

📎 **来源：** [HBase 权威指南 - RowKey Design](https://hbase.apache.org/book.html#rowkey.design)

## 题目三：HBase 的 MemStore 刷写（Flush）机制是什么？何时会触发 Major Compaction？

### 参考答案

### MemStore 刷写机制

当 RegionServer 收到写入请求时，数据首先写入 **WAL（Write-Ahead Log）**，然后写入对应 Region 的 **MemStore**（内存缓存）。当 MemStore 达到以下条件时，会触发刷写（Flush）到磁盘生成 **HFile**：

| 触发条件 | 默认值 | 说明 |
|----------|--------|------|
| **Region 级别** | MemStore 大小 ≥ `hbase.hregion.memstore.flush.size`（默认 128MB） | 单个 Region 的 MemStore 达到阈值 |
| **RegionServer 级别** | 所有 Region 的 MemStore 总大小 ≥ `java.heap.size * hbase.regionserver.global.memstore.size`（默认 40%） | 防止内存溢出 |
| **低水位** | `hbase.regionserver.global.memstore.size.lower.bound`（默认 95%） | 当内存回落至低水位时，优先刷写最大的 MemStore |
| **WAL 超限** | `hbase.regionserver.logroll.multiplier`（默认 0.5） | WAL 文件大小超过阈值时触发 |
| **定期** | `hbase.regionserver.thread.compaction.check.period` | 后台线程定期检查 |

> ⚠️ **注意**：刷写是按 Region 独立进行的，不是整台 RegionServer 一起刷。

### Compaction 机制

| 类型 | 触发条件 | 说明 |
|------|----------|------|
| **Minor Compaction** | HFile 数量 ≥ `hbase.hstore.compaction.min`（默认 3） | 选取多个小的 HFile 合并成少量更大的 HFile，不删除过期数据 |
| **Major Compaction** | `hbase.hregion.majorcompaction`（默认 7 天一次，可配置为 0 禁用自动） | 将一个 Region 的所有 HFile 合并成一个，**删除过期数据、打标删除的数据** |

**Major Compaction 触发时机：**
- 时间窗口：默认在凌晨 `02:00~06:00` 之间随机选择
- 可通过参数 `hbase.hregion.majorcompaction.jitter`（默认 0.5）调整抖动范围
- 业务高峰期建议将自动 Major Compaction 关闭，改为手动定时执行

📎 **来源：** [HBase 官方文档 - Compaction](https://hbase.apache.org/book.html#compaction)

## 题目四：HBase 的存储结构（物理存储）中，HFile 的结构是怎样的？

### 参考答案

HBase 数据最终以 **HFile** 格式存储在 HDFS 上。每个 HFile 的内部结构如下：

```
┌─────────────────────────────────────┐
│          HFile (Trailer)            │
│  ┌───────────────────────────────┐  │
│  │         Trailer               │  │ ← 固定长度，指向各部分偏移量
│  │  (FileInfo、DataIndex、       │  │
│  │   MetaIndex、Meta块)          │  │
│  └───────────────────────────────┘  │
│  ┌───────────────────────────────┐  │
│  │       Data Index Block        │  │ ← 数据块索引（块级布隆过滤器）│
│  └───────────────────────────────┘  │
│  ┌───────────────────────────────┐  │
│  │       Meta Index Block        │  │ ← 元数据块索引
│  └───────────────────────────────┘  │
│  ┌───────────────────────────────┐  │
│  │        Data Blocks            │  │ ← 实际数据（KeyValue 对）
│  │  ┌────┐ ┌────┐ ┌────┐        │  │
│  │  │Block│ │Block│ │Block│ ... │  │
│  │  └────┘ └────┘ └────┘        │  │
│  └───────────────────────────────┘  │
│  ┌───────────────────────────────┐  │
│  │       Meta Blocks             │  │ ← 可选，元数据信息
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘
```

### HFile 核心要素

| 要素 | 说明 |
|------|------|
| **Data Block** | 默认 64KB（`hfile.block.cache.size`），可配置。存储 `KeyValue` 对，按 RowKey 排序 |
| **Block Index** | 数据块的索引，支持按 RowKey 二分查找定位到具体 Block |
| **Bloom Filter** | 布隆过滤器，记录该 HFile 包含哪些 RowKey，可快速判断某行是否不存在，避免无效 IO |
| **Time Range Info** | 记录该 HFile 中数据的最大/最小时间戳，用于跳过无效版本 |
| **KeyValue 结构** | `KeyLength(4B) + ValueLength(4B) + RowKey + ColumnFamily + Qualifier + Timestamp + KeyType` |

### 与 MemStore 的关系

```
写入流程：
MemStore（内存有序） → Flush → HFile（磁盘有序）
                     ↓
            每生成一个 HFile 都会更新
            BlockIndex + BloomFilter
```

📎 **来源：** [HBase 官方文档 - HFile Format](https://hbase.apache.org/book.html#hfile)

## 题目五：HBase 协处理器（Coprocessor）有哪两类？它们分别在什么场景下使用？

### 参考答案

HBase 协处理器是扩展 HBase 功能的插件机制，允许用户在 **Region 级别** 或 **RegionServer 级别** 执行自定义代码，无需修改 HBase 核心代码。

### 两类协处理器对比

| 类型 | 加载方式 | 触发时机 | 典型场景 |
|------|----------|----------|----------|
| **Observer**（观察者） | 继承抽象类，覆写钩子方法 | 事件驱动，类似 AOP | 数据校验、二级索引、审计日志 |
| **Endpoint**（端点） | 动态 RPC，类似存储过程 | 客户端主动调用 | 聚合计算、批量处理、自定义过滤器 |

### 1. Observer（观察者）

类似于数据库触发器，在指定操作前后自动执行：

| 类型 | 触发点 |
|------|--------|
| **RegionObserver** | Get/Put/Delete/Scan 等数据操作前后 |
| **WALObserver** | WAL 写入前后 |
| **MasterObserver** | 表创建/删除、Region 分配等管理操作前后 |

**应用示例：二级索引**
```
在 Put 写入主表数据时，同时向索引表写入索引记录
（利用 RegionObserver 在 postPut() 钩子中实现）
```

### 2. Endpoint（端点）

类似于关系型数据库的存储过程，部署到 RegionServer 上，客户端通过 RPC 调用执行：

**常见用法：**
- 在 RegionServer 端对数据进行聚合（如 `SUM`、`AVG`、`COUNT`），减少网络传输
- 实现自定义的跨行跨列族计算
- 批量数据格式转换

**示例（伪代码）：**
```java
// 定义 Endpoint 接口
public interface AggregateProtocol extends CoprocessorProtocol {
    Long getSum(byte[] family, byte[] qualifier);
}

// 客户端调用
AggregationClient client = new AggregationClient(config);
Long sum = client.aggregate(Bytes.toString("t1"), htable, new SumFunction());
```

### 加载方式

| 方式 | 说明 |
|------|------|
| **静态加载**（配置） | 在 `hbase-site.xml` 中配置，全局生效，重启生效 |
| **动态加载**（Shell/API） | 通过 `alter` 命令为指定表加载，粒度更细 |

⚠️ **注意事项：**
- 协处理器代码 Bug 可能导致 RegionServer 崩溃，务必做好测试
- Observer 执行时间不宜过长，否则影响读写性能
- Endpoint 调用是同步的，需注意超时和异常处理

📎 **来源：** [HBase 官方文档 - Coprocessors](https://hbase.apache.org/book.html#cp)
