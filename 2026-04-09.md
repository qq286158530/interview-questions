### 题目1：HashMap 的底层实现是什么？JDK 1.8 相比 1.7 做了哪些优化？为什么 HashMap 不是线程安全的？

**【答案】**

HashMap 的底层数据结构在 JDK 1.7 中是**数组 + 链表**（Entry 结构），通过 key 的 hashCode 经过扰动函数处理后得到 bucket 位置，如果发生哈希冲突则以链表头插法插入。

JDK 1.8 进行了以下核心优化：

1. **数据结构变化**：当链表长度超过阈值（默认 8）时，链表转为**红黑树**（TreeNode），将最坏查找时间从 O(n) 优化到 O(log n)。
2. **插入方式变化**：不再使用头插法，改为尾插法，避免了并发下扩容时环形链表导致的死循环问题。
3. **扰动函数优化**：JDK 1.8 简化了 hash() 方法（高低位异或），减少碰撞。
4. **扩容机制优化**：扩容后元素位置要么在原索引，要么在原索引 + oldCap，避免了重新计算 hash。

**线程不安全的原因**：
- 多线程同时触发 resize 时，可能导致链表形成环形，产生死循环
- put 时非原子操作，可能覆盖其他线程的 put 结果
- size 字段不是 volatile，读写不是原子的
- Iterator 遍历时如果结构改变，会抛 ConcurrentModificationException

**适用替代方案**：ConcurrentHashMap（JDK 1.7 分段锁，JDK 1.8 CAS + synchronized）、Hashtable（全局锁，效率低）

### 题目2：JVM 的内存模型（运行时数据区）是怎么划分的？每个区域存放什么数据？哪些区域会发生 OOM？

**【答案】**

JVM 运行时数据区按线程是否共享分为两类：

**线程共享区域：**
- **堆（Heap）**：存放对象实例和数组，是 GC 的主要管理区域。JDK 1.7 和之前分成新生代（Eden、Survivor From/To）和老年代；JDK 1.8 之后移除了永久代，用元空间（Metaspace）替代。
- **元空间（Metaspace）**：JDK 1.8+ 替代了永久代，存放类信息、常量池、即时编译器生成的代码等，使用直接内存，不受堆大小限制（但受物理内存限制）。

**线程私有区域：**
- **虚拟机栈（VM Stack）**：每个方法调用创建一个栈帧（Stack Frame），存放局部变量表、操作数栈、动态链接、方法返回地址等。栈深度超过 JVM 定义会抛 StackOverflowError，动态扩展失败会抛 OOM。
- **本地方法栈（Native Method Stack）**：为 native 方法服务，HotSpot 将其与虚拟机栈合二为一。
- **程序计数器（PC Register）**：当前线程执行的字节码行号指示器，线程私有，是唯一不会发生 OOM 的区域。

**会发生 OOM 的区域**：
- **堆**：内存泄漏导致对象无法回收、创建大对象超过老年代
- **虚拟机栈**：递归调用过深（StackOverflowError）、创建过多线程（OOM）
- **元空间**：动态生成大量类（如 CGLib、JSP、OSGI、热部署）
- **直接内存**：NIO 中使用 ByteBuffer.allocateDirect() 分配直接内存，超出限制

### 题目3：volatile 关键字的作用是什么？它能保证原子性吗？synchronized 和 ReentrantLock 的区别是什么？

**【答案】**

**volatile 的作用**：
1. **可见性**：保证变量对所有线程的可见性。当一个线程修改了 volatile 变量的值，其他线程能立即看到最新值（通过 store + invalidate 缓存一致性协议实现）。
2. **有序性**：禁止指令重排序（内存屏障），防止编译器/CPU 将 volatile 变量前的操作重排序到其之后。

**volatile 不能保证原子性**：自增 `i++` 不是原子操作（读取→修改→写入三步），volatile 无法保证这三步不被其他线程打断。因此 volatile 不适合做计数器等需要原子操作的情景。

**synchronized vs ReentrantLock：**

| 特性 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 锁类型 | 隐式锁（自动获取/释放） | 显式锁（手动 lock/unlock） |
| 灵活性 | 非公平锁，不可中断 | 可设置为公平锁，支持 tryLock 可中断 |
| 条件变量 | 无 | 可绑定多个 Condition |
| 粒度 | 对象/类级别 | 代码块级别 |
| 锁获取 | 阻塞式 | 支持非阻塞tryLock()、超时lock() |
| 底层实现 | JVM内置，monenter/monexit | JDK层面，CAS + AQS |

ReentrantLock 是可重入锁，同一线程可多次获取，需要在 finally 中释放。synchronized 在 JDK 1.6 之后做了大量优化（偏向锁、轻量级锁、自旋锁等），性能已与 ReentrantLock 接近，日常推荐优先使用 synchronized。

### 题目4：Spring 事务的传播行为有哪几种？REQUIRED 和 REQUIRES_NEW 的区别是什么？

**【答案】**

Spring 事务的 7 种传播行为定义在 `Propagation` 枚举中：

1. **REQUIRED**（默认）：如果当前存在事务，加入该事务；否则创建新事务。
2. **REQUIRES_NEW**：始终创建新事务，挂起当前事务，两个事务完全独立。
3. **SUPPORTS**：如果当前存在事务，加入事务；否则以非事务方式执行。
4. **NOT_SUPPORTED**：以非事务方式执行，挂起当前事务。
5. **MANDATORY**：必须在事务中执行，否则抛异常。
6. **NEVER**：必须在非事务中执行，否则抛异常。
7. **NESTED**：如果当前存在事务，在嵌套事务中执行（使用 Savepoint）；否则等价于 REQUIRED。

**REQUIRED vs REQUIRES_NEW 的核心区别：**

| | REQUIRED | REQUIRES_NEW |
|--|----------|-------------|
| 事务数量 | 共用同一个事务 | 创建独立新事务 |
| 回滚影响 | 外层方法回滚全部回滚 | 内层回滚不影响外层 |
| 外层捕获异常 | 内层异常会导致外层也回滚 | 可在外层捕获内层异常 |
| 性能/资源 | 一个连接 | 每个事务独立连接 |

**典型场景**：日志记录适合用 REQUIRES_NEW（日志失败不影响主业务）；业务校验后扣款适合用 REQUIRED（业务异常应整体回滚）。

### 题目5：Spring Boot 的启动流程是怎样的？run() 方法核心步骤是什么？

**【答案】**

Spring Boot 启动核心流程（以 SpringApplication.run() 为入口）：

1. **创建 SpringApplication 对象**：
   - 判断应用类型（Servlet/Reactive/None）
   - 初始化 bootstrapRegistryInitializers
   - 加载 spring.factories 中的 ApplicationContextInitializer 和 ApplicationListener

2. **执行 run() 方法**：
   - 创建并启动 StopWatch（计时）
   - 通知 SpringApplicationRunListeners.starting()
   - 准备 ConfigurableEnvironment（加载配置文件如 application.yml）
   - 打印 Banner（Spring ASCII 图）
   - **创建 ApplicationContext**：根据应用类型创建对应的 IoC 容器（AnnotationConfigServletWebServerApplicationContext 等）
   - **准备 Context**：设置 environment、加载 bean sources、应用 initializer
   - **刷新 Context**：调用 AbstractApplicationContext.refresh()，这是最核心的步骤，包含：
     - 加载/解析 Bean 定义（@ComponentScan、@Bean 等）
     - **自动配置（EnableAutoConfiguration）**：加载 spring.factories 中的 AutoConfigurationImportSelector，筛选并注册大量 @Conditional 配置类
     - 注册 BeanFactoryPostProcessor
     - **实例化所有非懒加载的单例 Bean**
     - 启动内嵌 Web 服务器（Tomcat/Jetty/Undertow）
   - 通知 ApplicationContextInitializer.contextPrepared()
   - 加载资源文件
   - 通知 SpringApplicationRunListeners.ready()

3. **执行 Runner**：调用 ApplicationRunner 和 CommandLineRunner（可在此阶段执行初始化数据操作）

### 题目6：Spring Cloud 中服务注册与发现是如何工作的？Nacos 和 Eureka 的区别是什么？

**【答案】**

**服务注册与发现原理：**
1. 服务提供者在启动时向注册中心注册自己的信息（IP、端口、服务名）
2. 服务消费者从注册中心订阅服务，获取提供者列表
3. 注册中心通过心跳机制检测提供者健康状态，不健康的节点会被剔除
4. 服务消费者本地缓存一份提供者列表，通过负载均衡算法选择具体节点

**Nacos vs Eureka 对比：**

| 特性 | Nacos | Eureka |
|------|-------|--------|
| 一致性协议 | CP + AP 可切换（默认CP） | AP（最终一致） |
| 健康检查 | TCP/HTTP/MySQL/GRPC多种 | TCP心跳 |
| 动态上下线感知 | 推送（UDP + 注册中心变更） | 拉取（定时从服务端拉取） |
| .namespace/group/cluster | 支持完整三维隔离 | 仅支持 cluster |
| 配置中心 | 原生支持（配置管理） | 需要结合 Spring Cloud Config |
| 多环境支持 | namespace 天然隔离 | 通过 profile 区分 |
| 维护状态 | 阿里巴巴开源，持续活跃 | Netflix 已停止维护 |

Nacos 同时支持**服务注册发现**和**配置管理**两大功能，一个组件替代了 Eureka + Config 两套体系，是目前国内主流选择。

### 题目7：如何保证分布式事务的一致性？TCC 和 Seata AT 模式各自的原理是什么？

**【答案】**

分布式事务的常见解决方案：

**1. 2PC（两阶段提交）**
- 第一阶段（Prepare）：协调者让所有参与者执行事务但不提交，预提交成功则返回 YES
- 第二阶段（Commit）：所有参与者都预提交成功，协调者发送提交指令
- 问题：同步阻塞、单点故障、数据不一致风险

**2. TCC（Try-Confirm-Cancel）**
- **Try**：预留资源（如冻结库存），所有服务都预留成功才算成功
- **Confirm**：确认执行（真正扣减冻结量）
- **Cancel**：回滚（释放预留资源）
- 优点：性能好，不阻塞；缺点：业务侵入性大，需实现三个接口
- 典型框架：Seata TCC 模式、ByteTCC

**3. Seata AT 模式（推荐）**
- 基于 undo log 的自动补偿机制
- 第一阶段：解析 SQL，生成前后镜像（before/after image），直接提交本地事务
- 第二阶段：异步删除 undo log，如果异常则通过镜像自动回滚
- 对业务无侵入，性能接近本地事务，是目前最流行的分布式事务方案

**4. Saga 模式**
- 适用于长事务，每个参与者执行正向操作，失败则逆向补偿
- 不锁定资源，适合业务流程长的场景

**选型建议**：短事务、并发高 → AT 模式；长流程、跨库多 → TCC 或 Saga；跨系统 → MQ 最终一致性

📖 来源：[Seata 官方文档](https://seata.io/zh-cn/) + 阿里云开发者社区

### 题目8：Kafka 如何实现高吞吐量？它的核心架构和页缓存机制是什么？

**【答案】**

Kafka 高吞吐量的核心原因：

**1. 页缓存（PageCache）机制**
- Kafka 利用操作系统的页缓存来存储数据，写入时先写到页缓存（内存），由 OS 异步刷盘
- 读取时优先从页缓存读，避免直接磁盘 IO
- 顺序写入磁盘：消息追加到文件末尾，利用磁盘顺序读取远快于随机读的特性

**2. 零拷贝（Zero-Copy）**
- 使用 Linux sendfile 系统调用，数据从磁盘到网络跳过用户态切换
- 传统 IO：磁盘→内核缓冲区→用户缓冲区→Socket缓冲区→网卡（4次拷贝）
- Zero-Copy：磁盘→内核缓冲区→网卡（2次拷贝，DMA 直接传输）

**3. 批量处理**
- 生产者将多条消息打包成一个批次（batch）发送，减少网络请求次数
- 消费者以批次拉取和处理，提升吞吐量

**4. 分区（Partition）并行**
- 主题按分区水平扩展，每个分区内的消息有序
- 不同分区的消费可以完全并行（不同消费者实例）

**5. 压缩**
- 支持 gzip、Snappy、LZ4、ZSTD 多种压缩算法，批量压缩后传输，节省网络带宽

**核心架构组件**：
- **Broker**：Kafka 集群中的一个节点，负责接收和存储消息
- **Topic/Partition/Replica**：主题→分区→副本，分区保证并行，副本当选保证高可用
- **Producer**：消息生产者，通过分区策略决定消息落入哪个分区
- **Consumer Group**：消费者组，同一组内消费者分管不同分区，实现负载均衡

### 题目9：RabbitMQ 的消息模型中，Exchange 有哪几种类型？它们的工作原理和适用场景是什么？

**【答案】**

RabbitMQ 核心消息模型：Producer → Exchange → Binding → Queue → Consumer

**Exchange 的四种类型：**

**1. Direct（直接交换机）**
- 根据 routingKey 完全匹配路由到对应队列
- 适合精确匹配的场景（如日志级别：error → error_queue）
- Binding: `queue.bind(exchange, "error")` → routingKey 必须是 "error"

**2. Fanout（扇出交换机）**
- 将消息广播到所有绑定的队列，忽略 routingKey
- 适合群发通知、广播消息（如系统事件通知所有订阅服务）
- 速度最快，因为不需要匹配

**3. Topic（主题交换机）**
- 支持通配符匹配：`*` 匹配一个词，`#` 匹配零个或多个词
- 适合灵活的消息路由（如 `stock.#` 匹配 stock.us.daily 和 stock.eu）
- 例如：`*.stock.#` → "us.stock.daily" ✅ "stock.daily" ✅ "us.stock" ❌

**4. Headers（头交换机）**
- 根据消息头的键值对匹配，而非 routingKey
- 性能较低，较少使用，适合多属性匹配场景

**消息流转总结**：
```
Producer → Exchange（按类型路由）→ Binding（绑定规则）→ Queue → Consumer
```
Binding 相当于路由表，Exchange 根据类型 + routingKey + bindingKey 决定投递到哪些队列。

📖 来源：[RabbitMQ 官方文档](https://www.rabbitmq.com/tutorials) + CyC2018/CS-Notes

### 题目10：RocketMQ 的事务消息是如何实现的？半消息机制的作用是什么？

**【答案】**

RocketMQ 事务消息通过**半消息（Half Message）机制**实现分布式事务：

**完整流程：**

1. **发送半消息**：生产者向 RocketMQ 发送"半消息"（Half Message），该消息对消费者不可见（被标记为 UNKNOWN 状态）
2. **执行本地事务**：生产者执行本地业务逻辑（扣款、下单等）
3. **提交/回滚**：
   - 本地事务成功 → 发送 Commit 指令，Half Message 变为可消费
   - 本地事务失败 → 发送 Rollback，消息被丢弃
4. **回查机制**：如果 RocketMQ 长时间未收到 Commit/Rollback，会定时回查生产者的 `TransactionListener.checkLocalTransaction()` 接口，查询本地事务状态

**半消息的作用**：
- 隔离"本地事务执行"和"消息可见性"两个阶段
- 确保本地事务真正执行成功后，消息才对下游可见
- 本地事务失败时，消息不投递，避免数据不一致

**与普通消息的区别**：
- 发送阶段多了半消息确认，网络超时等情况下会触发回查
- 消费端无感知，收到的是正常消息
- 事务消息不支持延迟投递和批量发送

**典型应用场景**：跨银行转账（扣款方和入账方的一致性）、分布式下单+减库存

📖 来源：[RocketMQ 官方文档](https://rocketmq.apache.org/docs/) + 阿里云消息队列

### 题目11：什么是 RAG（Retrieval-Augmented Generation）？它解决了大模型的哪些问题？向量数据库在其中扮演什么角色？

**【答案】**

**RAG 定义**：
RAG = Retrieval（检索）+ Augmented（增强）+ Generation（生成）。它将外部知识检索与 LLM 生成结合，让大模型能够基于最新、最准确的私有知识回答问题。

**RAG 解决的核心问题**：

1. **知识时效性问题**：大模型训练数据有截止日期，无法回答最新信息；RAG 可以实时从外部库检索最新内容。
2. **幻觉（Hallucination）问题**：LLM 可能编造看似合理但错误的答案；RAG 提供有据可查的上下文，显著降低幻觉率。
3. **领域知识缺失**：通用大模型在垂直领域（企业内部知识、医疗、法律）深度不足；RAG 通过接入领域知识库补充。
4. **数据安全问题**：不需要将私有数据喂给模型训练，检索增强方式更好地保护隐私。
5. **可解释性**：RAG 返回的答案可以直接溯源到源文档，提升可信度。

**向量数据库的角色**：
RAG 的检索环节使用**向量数据库**（如 Milvus、Pinecone、FAISS、ChromaDB）：

1. 将文档切成 chunks（文本片段），通过 Embedding 模型（如 text2vec）转换为高维向量
2. 用户问题同样转成向量，在向量数据库中做**相似度检索**（余弦相似度/点积）
3. 最相似的 Top-K chunks 作为上下文传给 LLM 生成答案

**RAG 流程**：
```
文档 → 分块 → Embedding → 存入向量数据库
                              ↑
用户问题 → Embedding → 相似度检索 Top-K → 上下文拼接 → LLM 生成答案
```

📖 来源：[LangChain 官方文档](https://python.langchain.com/docs/get_started/introduction) + HuggingFace RAG 论文

### 题目12：AI Agent 的记忆系统是如何设计的？短期记忆、长期记忆和向量记忆各有什么用途？

**【答案】**

AI Agent（LLM Agent）的记忆系统通常分为三层：

**1. 短期记忆（Short-Term Memory）**
- 即当前的对话上下文（Conversation Context）
- 通常就是 LLM 的上下文窗口（Context Window），如 128K token
- 用于当前会话的连贯性，Agent 在当前 session 内记住之前的交互
- 会话结束后自动清空

**2. 长期记忆（Long-Term Memory）**
- Agent 的持久化知识，跨会话存在
- 通常存储在文件、数据库或知识图谱中
- 包括：Agent 的角色定义、个人偏好、已学习的技能、长期目标等
- 实现方式：结构化数据（JSON/SQL）、知识图谱（Neo4j）、键值存储

**3. 向量记忆（Vector Memory / Knowledge Base）**
- 将信息以向量形式存入向量数据库（FAISS/Milvus/Pinecone）
- 支持语义相似度检索，适合事实性知识和经验记录
- 工作方式：每次交互后，将重要内容 embedding 后存储；新问题时检索相关记忆作为上下文

**典型的 Agent 记忆架构**：
```
用户输入 → 意图识别
           ↓
    短期记忆（当前上下文）
           ↓
    长期记忆（角色/偏好/技能）→ 向量检索 → 相关历史记录
           ↓
    知识库（RAG）→ 外部知识文档
           ↓
         LLM → 行动计划 → 执行 → 观察结果 → 反思迭代
```

**记忆的管理策略**：
- **重要性过滤**：不是所有对话都值得记住，通过重要性评分决定是否存入长期记忆
- **定期总结**（Summarization）：将长对话压缩为摘要，节省 token
- **遗忘机制**（Forgetting）：定期删除过时、低价值或矛盾的记忆

### 题目13：如何用 LeetCode 例题讲解"二分查找"的边界条件？为什么有的人写二分会把边界写错？

**【答案】**

以 LeetCode 704「二分查找」为例：

**题目**：在有序数组 `nums` 中查找目标值 `target`，返回其索引，不存在返回 -1。

**标准模板（双闭区间 [left, right]）：**
```java
public int search(int[] nums, int target) {
    int left = 0, right = nums.length - 1;
    while (left <= right) {          // 注意是 <=
        int mid = left + (right - left) / 2;  // 防溢出
        if (nums[mid] == target) return mid;
        else if (nums[mid] < target) left = mid + 1;  // 搜索右半边
        else right = mid - 1;        // 搜索左半边
    }
    return -1;
}
```

**常见错误及原因**：

1. **`while (left < right)` 导致漏掉边界**
   - 当 `left == right` 时退出循环，但此时 `left/right` 指向的元素还没比较
   - 适用于查找"最左插入位置"等变体，不适合精确查找

2. **`mid = (left + right) / 2` 整数溢出**
   - 当 left 和 right 很大时（如接近 Integer.MAX_VALUE），相加会溢出
   - 正确写法：`mid = left + (right - left) / 2`

3. **`left = mid` 或 `right = mid` 造成死循环**
   - 当 left=0, right=2, mid=1 时，若 target > nums[mid]，left=mid=1
   - 下一步 left=1, right=2, mid 仍为1，造成死循环
   - 正确：`left = mid + 1` 和 `right = mid - 1`

4. **边界值测试**：
   - `[2]` 找 2 → 应找到
   - `[2]` 找 3 → 应返回 -1
   - `[1,3]` 找 1 → 边界测试

**二分查找变体**：
- LeetCode 34：在排序数组中查找元素的第一个和最后一个位置（左右边界收缩）
- LeetCode 35：搜索插入位置（最左插入点，`left < right` 循环，返回 left）
- LeetCode 74：二维矩阵二分查找

📖 来源：[LeetCode 官方题解](https://leetcode.cn/problems/binary-search/) + CyC2018/CS-Notes

### 题目14：动态规划的核心思想是什么？如何识别一个问题能否用 DP 解决？以「爬楼梯」和「最长公共子序列」为例说明。

**【答案】**

**动态规划核心思想**：
将原问题分解为重叠子问题，通过存储子问题的解（记忆化）避免重复计算，最终由子问题的解推导出原问题的解。适用于**最优子结构 + 重叠子问题**的场景。

**DP 识别三要素**：
1. **最优子结构**：问题的最优解由子问题的最优解构成
2. **重叠子问题**：递归过程中会多次遇到相同的子问题
3. **状态转移**：明确 dp[i] / dp[i][j] 的定义及其转移方程

---

**例1：爬楼梯（LeetCode 70）**

问题：爬 n 层楼梯，每次能爬 1 或 2 级，有多少种方法？

```java
// DP 定义：dp[i] = 爬到第 i 层的方法数
// 转移方程：dp[i] = dp[i-1] + dp[i-2]（最后一步从 i-1 爬1级，或从 i-2 爬2级）
// 初始值：dp[0]=1（可作为初始状态），dp[1]=1
public int climbStairs(int n) {
    if (n <= 2) return n;
    int dp0 = 1, dp1 = 1;
    for (int i = 2; i <= n; i++) {
        int dp2 = dp0 + dp1;
        dp0 = dp1;
        dp1 = dp2;
    }
    return dp1;
}
```

**空间优化**：只用两个变量，从 O(n) 空间降到 O(1)。

---

**例2：最长公共子序列 LCS（LeetCode 1143）**

问题：给定两个字符串，求最长公共子序列的长度。

```java
// DP 定义：dp[i][j] = text1[0..i-1] 和 text2[0..j-1] 的 LCS 长度
// 转移方程：
//   if (text1[i-1] == text2[j-1]) dp[i][j] = dp[i-1][j-1] + 1
//   else dp[i][j] = max(dp[i-1][j], dp[i][j-1])
public int longestCommonSubsequence(String text1, String text2) {
    int m = text1.length(), n = text2.length();
    int[][] dp = new int[m+1][n+1];
    for (int i = 1; i <= m; i++) {
        for (int j = 1; j <= n; j++) {
            if (text1.charAt(i-1) == text2.charAt(j-1))
                dp[i][j] = dp[i-1][j-1] + 1;
            else
                dp[i][j] = Math.max(dp[i-1][j], dp[i][j-1]);
        }
    }
    return dp[m][n];
}
```

**常见 DP 问题分类**：
- 线性 DP：爬楼梯、斐波那契、打家劫舍
- 区间 DP：石子合并、最长回文子序列
- 背包 DP：0-1 背包、完全背包、多重背包
- 状态压缩 DP：TSP 问题、鸡蛋掉落

### 题目15：什么是 Top-K 问题？有哪些解法？快速选择（Quick Select）算法相比堆排序的优势是什么？

**【答案】**

**Top-K 问题定义**：从 n 个元素中找出最大/最小的 K 个元素（K 远小于 n）。

**常见解法及复杂度：**

| 方法 | 时间复杂度 | 空间复杂度 | 特点 |
|------|----------|----------|------|
| 排序后取前 K | O(n log n) | O(log n) | 简单，K 接近 n 时最优 |
| 堆排序（最大堆取 K 次）| O(n + K log n) | O(n) | K 远小于 n 时较好 |
| **快速选择（Quick Select）** | O(n)（期望）| O(1) | 实际最快，不需排序全部 |
| BFPRT eterministic | O(n)（最坏）| O(1) | 理论上最优但常数大 |

---

**快速选择算法（Quick Select）**：

基于快速排序的分区思想，找第 K 小的元素：

```java
public int findKthLargest(int[] nums, int k) {
    // 找第 n-k 小的元素
    int target = nums.length - k;
    int left = 0, right = nums.length - 1;
    while (true) {
        int pivot = partition(nums, left, right);  // 分区，返回 pivot 位置
        if (pivot == target) return nums[pivot];
        else if (pivot < target) left = pivot + 1;  // 第 K 小在右半边
        else right = pivot - 1;  // 在左半边
    }
}

private int partition(int[] nums, int left, int right) {
    int pivot = nums[right];
    int i = left;
    for (int j = left; j < right; j++) {
        if (nums[j] <= pivot) {
            swap(nums, i++, j);
        }
    }
    swap(nums, i, right);
    return i;
}
```

**Quick Select vs 堆排序优势：**

1. **时间复杂度更优**：Quick Select 平均 O(n)，堆排序需要 O(n log n)
2. **额外空间**：Quick Select 原地分区 O(1)，堆排序需要 O(n) 建立堆
3. **cache 友好**：顺序访问模式，CPU 缓存命中率高

**Quick Select 的缺点**：最坏情况 O(n²)（如已排序数组），适合随机化 pivot 优化（Randomized Quick Select），或使用 BFPRT 保证最坏 O(n)。
