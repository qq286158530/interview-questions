## 题目一：HBase 的整体架构是怎样的？请描述 ZooKeeper、RegionServer、Master 的作用。

### 参考答案

HBase 是一个面向列族的分布式 NoSQL 数据库，构建在 HDFS 之上，采用 **LSM-Tree（Log-Structured Merge-Tree）** 作为底层数据结构。其整体架构如下：

#### 1. 核心组件

| 组件 | 职责 |
|------|------|
| **ZooKeeper** | 负责 HBase 的分布式协调服务，保存元数据（如 -ROOT-、META 表位置）、监控 RegionServer 存活、实现分布式锁等 |
| **HMaster** | HBase 的主节点，负责表的管理（创建/删除/修改）、Region 的分配与调度、RegionServer 故障检测与恢复、负载均衡 |
| **RegionServer** | 数据服务节点，每个 RegionServer 负责管理多个 Region；处理读写请求、合并 HFile、刷写 MemStore 等 |
| **HDFS** | 底层存储，HBase 实际数据最终持久化到 HDFS 上 |

#### 2. 表与 Region 的关系

- HBase 表按 **RowKey** 水平切分成多个 **Region**，每个 Region 包含 `[startKey, endKey)` 的行键区间。
- Region 被分配到不同的 **RegionServer** 上，一个 RegionServer 可以管理多个 Region。
- 元数据表结构：`ZooKeeper (-ROOT-) → .META. → 用户表 Region`
  - `-ROOT-` 表记录 `.META.` 表中 Region 的位置（ZooKeeper 中维护）
  - `.META.` 表记录所有用户表 Region 的位置信息

#### 3. 读写数据流程（读）

```
客户端 → ZooKeeper → 获取 .META. 位置 → 获取目标 Region 位置 → RegionServer → MemStore / HFile
```

#### 4. 特点总结

- **无单点故障**（HMaster 可以有多台，通过 ZooKeeper 选举）
- **水平扩展**：通过增加 RegionServer 节点即可扩容
- **强一致性**：同一 Region 的写操作是串行的，保证数据一致性

> 📚 **参考来源**：[HBase 官方文档 - Architecture](https://hbase.apache.org/book.html#architecture)
> 📚 **参考来源**：[Apache HBase 架构详解 - 博客园](https://www.cnblogs.com/smartloli/p/4319563.html)

## 题目二：HBase 中 RowKey 设计的核心原则是什么？请列举常见的设计技巧。

### 参考答案

RowKey 是 HBase 数据的唯一主键，决定了数据的物理存储位置和访问效率。RowKey 设计是 HBase 面试中最重要的问题之一。

#### 核心原则

1. **长度原则**：RowKey 长度建议控制在 **8~1024 字节**，越短越好
   - 过长会占用 HBase 的 MemStore 内存
   - 过短可能无法区分数据

2. **唯一性原则**：RowKey 必须唯一标识一行数据

3. **散列原则**：避免热点（HotSpot），RowKey 要尽量打散分布
   - 反例：时间戳作为 RowKey 前缀 → 所有新数据写入同一个 Region → 热点
   - 正例：md5(rowKey) 前缀 + 原始数据

4. **业务相关性原则**：将经常用于查询的字段放在 RowKey 前缀位置

#### 常见设计技巧

**① 加盐（Salting）**
```
原 RowKey: 20260628001
加盐后:    0_20260628001, 1_20260628001, 2_20260628001
```
通过在 RowKey 前面加随机前缀，让数据分散到不同 Region。

**② 哈希（Hashing）**
```
hash(userId) + userId + timestamp
```
使用 MD5 或其他哈希算法取前几位做前缀，避免单调递增/递减的热点问题。

**③ 反转（Reversing）**
```
原 RowKey:  20260628001
反转后:     1000862602
```
适用于手机号、订单号等有规律后缀的业务场景。

**④ 组合字段**
```
orderId + userId + timestamp
```
将多个查询条件组合在一起，按查询优先级排列字段。

#### 热点场景及解决方案

| 场景 | 问题 | 解决方案 |
|------|------|----------|
| 时间序列数据 | 新数据都写到末尾 Region | 时间戳放最后，加盐/哈希 |
| 用户数据 | 同一用户数据集中在某 Region | userId 加盐/哈希 |
| 订单数据 | 订单号单调递增 | 哈希打散 |

> 📚 **参考来源**：[HBase RowKey 设计指南 - Cloudera](https://www.cloudera.com/documentation/enterprise/5-9-x/topics/admin_hbase_schema_design.html)
> 📚 **参考来源**：[HBase rowkey 设计原则 - CSDN](https://blog.csdn.net/u014727260/article/details/55061908)

## 题目三：HBase 的存储结构是什么？请解释 MemStore 和 HFile 的作用及它们之间的关系。

### 参考答案

HBase 使用 **LSM-Tree（Log-Structured Merge-Tree）** 结构来组织数据，核心组件是 **MemStore**（内存写缓存）和 **HFile**（磁盘存储文件）。

#### 1. HBase 写入流程

```
客户端 → RegionServer → 写入 WAL (Write-Ahead Log) → 写入 MemStore → 返回 ACK
```

1. **WAL（预写日志）**：先写日志再写内存，数据先持久化到 HDFS 的 WAL 文件，防止 MemStore 数据丢失
2. **MemStore**：将数据写入内存中的跳表（Sorted Skip List），按 RowKey 排序
3. 当 MemStore 达到阈值（默认 `128MB` × `hbase.hregion.memstore.flush.size`）时，触发 **刷写（Flush）**
4. Flush 将 MemStore 整块数据写出成 **HFile**，写入 HDFS

#### 2. MemStore 的作用

- 内存中的有序数据结构（基于 ConcurrentSkipListMap），支持高效的范围查询
- 写操作先写内存再落盘，**写性能高**（顺序写内存 + 批量刷磁盘）
- 同一 Region 的多个 Column Family 各自有独立的 MemStore

#### 3. HFile 的结构

HFile 是 HBase 在 HDFS 上的存储格式，结构如下：

```
┌─────────────────────────────────┐
│  Trailer (固定)                  │  ← 记录 Meta Info 偏移量
├─────────────────────────────────┤
│  FileInfo (固定)                │  ← 记录 last key, avg key len 等
├─────────────────────────────────┤
│  Data Index (可变)              │  ← 数据块索引（块位置+最大key）
├─────────────────────────────────┤
│  Meta Index (可变)              │  ← Bloom Filter 索引
├─────────────────────────────────┤
│  Data Blocks (可变)             │  ← 实际数据，每个块 64KB/128KB
│    - KeyValue 对按 RowKey 排序   │
├─────────────────────────────────┤
│  Magic Number (固定)            │
└─────────────────────────────────┘
```

每个 HFile 包含：
- **Bloom Filter**：快速判断某 RowKey 是否存在于该 HFile 中，避免无效 IO
- **Data Index**：记录每个数据块的起始 key，用于快速定位
- **Time Range Information**：记录该文件中的数据时间范围，用于 Region 剪枝

#### 4. MemStore ↔ HFile 的关系

```
MemStore (内存, 排序)  --Flush-->  HFile (磁盘, 排序)
                                          ↓
                                    当 HFile 数量过多时
                                          ↓
                                     Minor Compaction (小合并)
                                     Major Compaction (大合并)
                                          ↓
                                    合并成更大的 HFile
```

#### 5. Compaction（合并）机制

| 类型 | 触发条件 | 合并范围 |
|------|----------|----------|
| **Minor Compaction** | HFile 数量 > `hbase.hstore.compaction.min`（默认3） | 合并部分小 HFile，保留被删除/过期数据 |
| **Major Compaction** | 触发策略（默认7天一次） | 合并一个 Region 的所有 HFile，**物理删除**过期和被标记删除的数据 |

> 📚 **参考来源**：[HBase Storage Architecture - Apache HBase Book](https://hbase.apache.org/book.html#arch.store)
> 📚 **参考来源**：[HBase LSM-Tree 与 Compaction 机制 - 美团技术团队](https://tech.meituan.com/2016/01/22/hbase-log-structured-merge-tree.html)

## 题目四：HBase 的协处理器（Coprocessor）是什么？它有哪两种类型？各自的使用场景是什么？

### 参考答案

HBase 协处理器是一种**允许用户在 RegionServer 上运行自定义代码**的扩展机制，让数据处理逻辑尽可能靠近数据存储层，避免大量网络传输。

#### 1. 两大类型

##### ① Observer Coprocessor（观察者型）

**原理**：类似数据库触发器，在特定事件前后自动触发执行。

```
Hook 点：
  - preGet / postGet        ：GET 查询前后
  - prePut / postPut        ：PUT 写入前后
  - preDelete / postDelete   ：DELETE 删除前后
  - preOpen                 ：Region 打开时
  - preFlush / postFlush    ：MemStore 刷写前后
  - preCompact / postCompact：Compaction 前后
```

**常见使用场景**：
- **二级索引**：在 put 数据时，自动更新索引表（如维护 `email → userId` 的映射）
- **数据权限控制**：在 preGet/prePut 中判断用户是否有权限访问该行
- **审计日志**：在 postPut 中记录谁在什么时间修改了什么数据
- **实时计算**：在 postGet 中对数据做聚合、统计

##### ② Endpoint Coprocessor（端点型）

**原理**：类似数据库存储过程，允许在 RegionServer 上执行自定义的 RPC 服务，通过 RPC 调用触发执行。

**使用场景**：
- **自定义聚合**：在每个 Region 上执行 count/sum/avg，然后合并结果（如统计表的总行数）
- **自定义过滤器**：在 Region 端执行复杂过滤逻辑
- **批量数据处理**：在 Region 侧直接完成数据转换、格式转换

#### 2. 加载方式

| 方式 | 配置位置 | 生效范围 | 使用场景 |
|------|----------|----------|----------|
| **系统级** | `hbase-site.xml` | 全局所有表 | 通用逻辑（如权限管理） |
| **表级** | 表描述符 | 单个表 | 特定表的二级索引 |
| **动态加载** | Shell/API | 单个表/Region | 开发调试 |

```xml
<!-- 系统级加载示例 -->
<property>
  <name>hbase.coprocessor.region.classes</name>
  <value>com.example.IndexCoprocessor,com.example.AuditCoprocessor</value>
</property>
```

#### 3. 注意事项

- 协处理器代码随 RegionServer 加载，需注意**版本兼容性**
- 协处理器中的异常可能导致 Region 不可用，建议做好错误处理
- Endpoint 协处理器需要客户端通过 `CoprocessorRpcChannel` 调用，复杂度较高

> 📚 **参考来源**：[HBase Coprocessor Guide - Apache HBase](https://hbase.apache.org/book.html#cp)
> 📚 **参考来源**：[HBase 协处理器实现二级索引 - IBM Developer](https://developer.ibm.com/zh/articles/os-cn-hbase-coprocessor/)

## 题目五：HBase 如何保证数据一致性？它的读写流程是怎样的？

### 参考答案

HBase 是一个 **强一致性** 的分布式数据库，其一致性保证贯穿整个读写流程。

#### 1. HBase 的一致性模型

HBase 提供的是 **行级别的强一致性**：对同一行数据的并发写操作，RegionServer 会串行处理，保证后写入的数据覆盖先前的版本。

- **写一致性**：数据写入 WAL + MemStore 后才返回成功
- **读一致性**：所有写操作先写 WAL，保证宕机可恢复；MemStore 刷写成 HFile 后数据持久化

#### 2. 写入（Put）流程详解

```
① 客户端请求
      ↓
② 从 ZooKeeper 获取 .META. 表的 Region 位置
      ↓
③ 查询 .META. 表，确定目标 RowKey 属于哪个 Region，获取 RegionServer 地址
      ↓
④ 向目标 RegionServer 发起 Put 请求
      ↓
⑤ RegionServer 执行：
   a. 检查 WAL 是否写成功（写入 HDFS WAL）
   b. 将数据写入 MemStore（按 RowKey 排序）
   c. 返回客户端写入成功
      ↓
⑥ 如果 MemStore 达到阈值，触发 Flush 写成 HFile
      ↓
⑦ 后台进行 Compaction 合并
```

#### 3. 读取（Get/Scan）流程详解

```
① 客户端请求（Get by RowKey 或 Scan 范围扫描）
      ↓
② 定位 Region（同写操作①~③步）
      ↓
③ RegionServer 处理请求：
   a. 先查 MemStore（内存中最新的数据）
   b. 再查 BlockCache（热点数据的缓存）
   c. 最后查 HFile（磁盘文件，按新到旧遍历）
      ↓
④ 合并所有来源的数据，按时间戳返回最新版本
```

#### 4. MVCC（多版本并发控制）

HBase 在**写入时使用 MVCC** 来保证读写之间的一致性：

- 写操作需要获取写锁，写入 MemStore 后分配 MVCC 版本号
- 读操作只能看到 MVCC 版本号 <= 当前读版本的最新数据
- MVCC 确保在 Compaction 期间，读操作不会被正在合并的 HFile 影响

#### 5. 故障恢复机制

| 故障场景 | 恢复机制 |
|----------|----------|
| RegionServer 宕机 | HMaster 通过 ZooKeeper 检测到，将该 Region 重新分配给其他 RegionServer，WAL 切分后重放 |
| MemStore 未刷写 | WAL 日志中记录了未刷写的数据，重放 WAL 恢复 |
| HFile 损坏 | HDFS 默认3副本，读取损坏的 block 时自动从其他副本恢复 |
| 网络分区 | ZooKeeper 选举新 Master，Region 重新分配 |

> 📚 **参考来源**：[HBase 数据一致性保证 - 腾讯云技术社区](https://cloud.tencent.com/developer/article/1337838)
