### 题目1：Spark 的架构是怎样的？Driver 和 Executor 的作用是什么？

【答案】

**Spark 架构核心组件**：

1. **Driver（驱动程序）**
   - 运行 Spark 应用的 `main()` 方法
   - 负责创建 `SparkContext`，将作业转换为 DAG 图
   - 将 DAG 划分为 Stage，构建 TaskSet
   - 调度 Task 到 Executor 执行
   - 收集 Executor 返回的执行结果

2. **Executor（执行器）**
   - 驻留在 Worker Node 上的独立进程
   - 负责执行 Driver 分配的 Task
   - 将数据缓存为 RDD/DataFrame（Storage 层面）
   - 向 Driver 汇报心跳和任务状态

3. **Cluster Manager（集群管理器）**
   - 负责集群资源分配（Standalone、YARN、Mesos、Kubernetes）
   - 在 YARN 模式下分为 Cluster 模式和 Client 模式：
     - **Cluster 模式**：Driver 在 AM 中运行，ApplicationMaster 负责 Driver
     - **Client 模式**：Driver 在客户端运行，AM 仅负责请求资源

4. **Worker Node**
   - 集群中运行 Executor 的节点
   - 启动 Executor 进程或 Driver 进程

**作业执行流程**：
```
用户代码 → SparkContext → DAGScheduler(切分Stage) → TaskScheduler(提交Task) 
→ Worker上的Executor执行Task → 结果返回Driver
```

### 题目2：RDD 是什么？有哪些核心特性？Transformation 和 Action 的区别？

【答案】

**RDD（Resilient Distributed Dataset）弹性分布式数据集**：

是 Spark 最底层的抽象，不可变、分区的集合，支持并行操作。

**RDD 五大核心特性**：
1. **分区列表（Partitions）**：数据分片的列表，每个分区可并行计算
2. **计算函数（Computational Function）**：每个分区上执行的计算逻辑（Split → Iterator[T]）
3. **依赖关系（Dependencies）**：RDD 与父 RDD 的依赖关系（窄依赖/宽依赖）
4. **分区器（Partitioner）**：可选，用于 Key-Value RDD 的 Shuffle（HashPartitioner/RangePartitioner）
5. **位置偏好（Preferred Locations）**：数据块位置偏好（用于数据本地性优化）

**Transformation（转换）—— 懒加载**：
- 不立即执行，只记录血缘关系（DAG）
- 窄依赖：`map`、`filter`、`union`
- 宽依赖：`groupByKey`、`sortByKey`、`join`、`distinct`（触发 Shuffle）

**Action（动作）—— 触发执行**：
- 真正触发作业执行，返回结果或写入外部存储
- 例如：`collect()`、`count()`、`saveAsTextFile()`、`reduce()`

**Lineage（血统）**：
- Spark 通过血统机制实现容错，无需复制数据
- 某个分区丢失时，根据父 RDD 和转换操作重新计算

**Lazy _eval（惰性求值）**：
- 所有的 Transformation 都是惰性的，只有遇到 Action 时才真正计算
- 好处：减少中间结果，优化执行计划

### 题目3：Spark Shuffle 是什么？为什么 Shuffle 操作性能较差？有哪些优化策略？

【答案】

**Shuffle（洗牌）是分布式计算的关键阶段**：

在 Spark 中，Shuffle 是将数据按 Key 重新分区的过程。当一个 RDD 的某个 Partition 需要依赖父 RDD 的多个 Partition 时（如 `groupByKey`、`join`、`sortByKey`），必须跨节点拉取数据。

**Shuffle 性能差的原因**：
1. **磁盘 I/O**：数据先写入磁盘（SortShuffle Writer），再读取
2. **网络 I/O**：跨节点数据传输
3. **序列化/反序列化**：数据需要 Java/Kryo 序列化
4. **内存压力**：Shuffle 数据量大时可能触发 OOM
5. **磁盘带宽**：机械硬盘随机读写成为瓶颈

**Spark Shuffle 演进**：
- **Hash Shuffle（1.1）**：每个 Mapper 为每个 Reducer 产生一个文件，文件数 = M × R，文件数过多
- **Sort Shuffle（1.5+）**：数据先写入内存 buffer，排序后写入单个文件（按 Reducer ID 索引），文件数 = M × 2

**优化策略**：
1. **减少 Shuffle 数据量**：
   - 使用 `map-side aggregation`（`combineByKey` 替代 `groupByKey`）
   - 过滤不必要的分区（`filter` 提前）
2. **增加并行度**：
   - 减小 `spark.sql.shuffle.partitions`（默认 200）
   - 合理设置 Partition 数量
3. **使用广播变量**：
   - 小表广播到每个 Executor（`broadcast`），避免 Shuffle
4. **数据倾斜处理**：
   - 加盐（Salting）：对热点 Key 加随机前缀
   - 两阶段聚合：`map` 预聚合 + `reduceByKey` 本地聚合
5. **使用高性能序列化**：Kryo 替代 Java 序列化
6. **Spark 3.0+ AQE**：自适应查询执行，动态调整 Shuffle 分区数

### 题目4：Spark 内存管理是怎样的？堆内内存和堆外内存有什么区别？如何调优？

【答案】

**Spark 内存管理模型（统一内存管理，Spark 1.6+）**：

Spark 内存分为两大区域：**Execution（执行内存）** 和 **Storage（存储内存）**，二者可以动态共享和抢占。

```
┌──────────────────────────────────────┬──────────────┐
│           Execution Memory           │   Storage    │
│  (Shuffle、Sort、Hash Join、Cache)   │  (RDD Cache) │
│         [可动态借用 Storage]          │ [可动态借用  │
│                                      │  Execution]  │
├──────────────────────────────────────┴──────────────┤
│              Other Memory (系统预留)                 │
└────────────────────────────────────────────────────┘
```

**堆内内存（On-Heap）**：
- JVM Heap 内的内存，受 GC 管理
- `spark.memory.fraction`（默认 0.6）：占堆内内存的比例
- `spark.memory.storageFraction`（默认 0.5）：Storage 预留空间
- 问题：受 JVM GC 影响，大对象易触发 Full GC

**堆外内存（Off-Heap）**：
- `spark.memory.offHeap.enabled=true` 开启
- 使用 `sun.misc.Unsafe` 直接操作堆外内存
- 优点：避免 GC 干扰，适合大内存场景（> 32GB）
- `spark.memory.offHeap.size`：堆外内存大小

**内存分配公式**：
- `Executor 内存 = spark.executor.memory`（-Xmx 设定）
- `JVM Heap = (1 - spark.memory.fraction) * executor_memory`（约 300MB 预留）
- `Spark 统一内存 = spark.memory.fraction * executor_memory`
- `Storage 可用 = Spark 统一内存 * spark.memory.storageFraction`

**内存调优建议**：
| 场景 | 建议 |
|------|------|
| 大数据 Shuffle | 增大 `spark.shuffle.memoryFraction`（0.2 → 0.3） |
| 大量 Cache | 增大 `spark.memory.storageFraction`（0.5 → 0.6） |
| GC 频繁 | 开启堆外内存 + 减小 Heap |
| 超大集群 | 堆外内存 + `spark.memory.offHeap.enabled=true` |

### 题目5：广播变量（Broadcast Variable）是什么？为什么能提升性能？使用注意事项？

【答案】

**广播变量**是 Spark 提供的共享变量机制，将只读数据广播到每个 Executor（而不是每个 Task），避免重复传输。

**原理**：
1. Driver 端将数据集序列化为字节数组
2. 通过 BlockManager 以 Torrent 方式分发给各 Executor（类似 BitTorrent）
3. Executor 缓存广播数据（默认存储为 MEMORY_AND_DISK）
4. 每个节点的 Task 共享同一份广播数据，而非各自复制

**不使用广播（普通 Join）**：
```
Task1: 从 HDFS 拉取全量数据 → 内存压力、网络开销巨大
Task2: 同上（每个 Task 都传输完整小表）
TaskN: 同上
```

**使用广播变量**：
```
小表广播到 Executor 一次 → Task 共享访问 → 显著降低网络传输
```

**性能提升原因**：
1. **减少网络传输**：小表只传输一次，而非每个 Task 传输一次
2. **降低内存消耗**：避免每个 Task 复制一份数据
3. **本地性优化**：数据在本地可用，减少跨节点访问

**使用方式**：
```scala
// Scala
val broadcastVar = sc.broadcast(Array(1, 2, 3, 4, 5))
val result = rdd.map(x => x * broadcastVar.value.sum)

// Python
broadcast_var = sc.broadcast([1, 2, 3, 4, 5])
result = rdd.map(lambda x: x * sum(broadcast_var.value))
```

**注意事项**：
1. **适用场景**：小表广播（通常 < 20MB）；大表广播反而会撑爆内存
2. **广播时机**：Spark 3.0+ 自动广播优化（`spark.sql.autoBroadcastJoinThreshold`，默认 10MB）
3. **广播后不可修改**：Broadcast 变量是只读的
4. **广播顺序**：广播大表前先 `filter` 减少数据量
5. **手动广播**：通过 `broadcast()` 函数手动控制
6. **广播失效**：`spark.sql.broadcastTimeout`（默认 5 分钟）控制超时

**与 Accumulators 的区别**：
| 特性 | Broadcast | Accumulator |
|------|-----------|-------------|
| 用途 | 只读数据共享 | 计数器/累加器 |
| 可修改 | ❌ 只读 | ✅ 只增 |
| Task可见性 | Task 可读 | Task 可增，Driver 可读 |
