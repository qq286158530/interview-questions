## 题目一：HBase 的整体架构是怎样的？请详细说明各组件的作用。

### 答案

HBase 是一个基于 HDFS 的分布式列式存储数据库，采用 Master-Slave 架构，主要包含以下组件：

**1. ZooKeeper**
- 负责选举 Master 节点，协调分布式集群
- 存储所有 RegionServer 的元数据信息（地址、状态）
- 实现分布式锁，保证集群高可用

**2. HMaster（Master 节点）**
- 管理表的 Schema（创建、删除、修改列族）
- 负责 RegionServer 的负载均衡，分配 Region
- 监控 RegionServer 状态，发现故障时进行 Region 迁移
- 接收客户端的建表请求，协调分配 Region

**3. RegionServer（Region 服务器）**
- 负责数据的读写操作，是真正处理请求的节点
- 管理 Region：每个 Region 对应一段连续的 RowKey 范围
- 刷新 MemStore 到 HFile，执行 Compaction 合并
- 向 HMaster 定期汇报自己的健康状态

**4. Region（数据区域）**
- 表的水平分片单元，每个 Region 存储一段 RowKey 范围的数据
- 默认大小约 10GB，超过后会自动分裂（Split）
- 每个 Region 归属于一台 RegionServer

**5. HFile（HBase 数据文件）**
- 存储在 HDFS 上，是 HBase 的实际数据文件格式
- 基于 LSM 树（Log-Structured Merge-Tree）结构
- 包含 Data Block、Index Block、Bloom Filter 等

**6. MemStore（写缓存）**
- 写入数据首先存入 MemStore（内存）
- 当 MemStore 写满后，刷写为 HFile
- 按列族组织，每个列族有独立的 MemStore

**读写流程简述：**
- **写**：Client → ZooKeeper → RegionServer → MemStore → HFile（HDFS）
- **读**：Client → ZooKeeper → 定位 RegionServer → 查 MemStore/HFile → 合并返回

**来源**：[HBase 架构详解 | CSDN](https://blog.csdn.net/weixin_44755073/article/details/125153374)

## 题目二：HBase 的 RowKey 设计应遵循哪些原则？如何避免热点问题？

### 答案

RowKey 是 HBase 中最重要的概念，决定了数据的物理存储位置，直接影响读写性能和负载均衡。

**RowKey 设计核心原则：**

**1. 长度原则**
- RowKey 长度建议控制在 8~1024 字节之间
- 越短越好，过长的 RowKey 会增加存储压力和 MemStore 内存开销
- 避免使用超长字符串作为 RowKey

**2. 唯一性原则**
- RowKey 必须唯一标识一行数据
- 类似关系型数据库的主键

**3. 散列性原则（避免热点）**
- 避免使用时间戳、顺序递增 ID 等作为 RowKey 前缀
- 否则所有请求会集中到某个 RegionServer 上，造成热点

**避免热点的具体方法：**

**方法一：加盐（Salting）**
```
原 RowKey：user_123456
加盐后：   0_user_123456, 1_user_123456, 2_user_123456
```
在 RowKey 前添加随机前缀，将数据分散到不同 Region，但会导致读取时需要扫描更多 Region。

**方法二：哈希（Hash）**
```
原 RowKey：user_123456
哈希后：   md5(user_123456)[0:8]_user_123456
```
使用 RowKey 的哈希值前几位作为前缀，保证数据均匀分布，且读取时可计算定位。

**方法三：反转（Reverse）**
```
原 RowKey：phone_13800138000
反转后：   00000138831_ephon
```
反转低变化部分（如手机号尾部），使热点分散。

**常见业务 RowKey 设计模式：**
- **用户+时间戳**：`userId + "_" + timestamp`（高并发写入可加哈希前缀）
- **订单场景**：`hash(orderId)[0:4] + orderId`（唯一且散列）
- **报表场景**：`stat_date + "_" + stat_type`（按日期范围查询友好）

**来源**：[HBase RowKey 设计原则 | 腾讯云开发者社区](https://cloud.tencent.com/developer/article/1191113)

## 题目三：HBase 的存储结构是怎样的？请说明 MemStore 和 HFile 的工作原理。

### 答案

HBase 采用 LSM 树（Log-Structured Merge-Tree）存储结构，结合内存写入和磁盘顺序写入，兼顾写吞吐和读性能。

**HBase 数据写入流程：**

1. 数据先写入 **WAL（Write-Ahead Log）**，保证宕机可恢复
2. 数据写入 **MemStore**（内存），同时更新内存中的索引
3. MemStore 写满后，触发 **Flush**（刷写）操作，生成 **HFile** 写入 HDFS
4. 随着 HFile 增多，后台定期执行 **Compaction** 合并，减少文件数量

**MemStore（写缓存）：**
- 每个列族对应一个 MemStore 实例
- 数据按 RowKey 排序存储（内存中即有序）
- 默认大小约 128MB，超过后触发 Flush
- Flush 时按列族独立刷写，不是整库刷写

**HFile（底层数据文件）：**
- 基于 Hadoop 的 TSTable 格式
- 内部结构：
  - **Data Block**：存储实际的 KeyValue 数据，默认 64KB
  - **Index Block**：记录每个 Data Block 的起始 RowKey，用于快速定位
  - **Bloom Filter**：快速判断某个 RowKey 是否存在于该 HFile 中
  - **Meta Block**：存储元信息
- HFile 中的数据按 RowKey 字典序排列

**LSM 树核心思想：**
- **写操作**：先写内存（MemStore），后写磁盘，内存顺序写极快
- **读操作**：需要合并 MemStore 和所有 HFile 中的数据
- **Compaction**：后台定期将多个小 HFile 合并成一个大 HFile，淘汰删除数据，减少读放大

**Major Compaction vs Minor Compaction：**
- **Minor Compaction**：将多个小 HFile 合并成较少的大 HFile
- **Major Compaction**：将一个列族下所有 HFile 合并为一个，清理删除和过期数据

## 题目四：HBase 的协处理器（Coprocessor）是什么？有哪两类？各自的使用场景是什么？

### 答案

协处理器是 HBase 提供的扩展机制，允许用户将自定义逻辑运行在 RegionServer 上，直接操作数据，无需多次客户端-服务器往返通信。

**两类协处理器：**

**1. Observer Coprocessor（观察者协处理器）**
- 类似于数据库的触发器（Trigger）
- 在特定的钩子（Hook）点被调用
- 常用钩子：
  - `preGet` / `postGet`：GET 操作前后
  - `prePut` / `postPut`：PUT 写入前后
  - `preDelete` / `postDelete`：DELETE 删除前后
  - `preOpen` / `postOpen`：Region 打开时
  - `preCompact` / `postCompact`：Compaction 前后

- **使用场景**：
  - **二级索引**：写入数据时自动更新索引表
  - **数据验证**：写入前校验数据合法性
  - **审计日志**：记录每次数据变更
  - **实时统计**：写入时更新计数器（如 UV 统计）

**2. Endpoint Coprocessor（端点协处理器）**
- 类似于数据库的存储过程（Stored Procedure）
- 在 RegionServer 上执行自定义计算逻辑
- 客户端调用后，协处理器在每个相关 Region 上执行，结果由客户端汇总
- 常用场景：聚合计算（如 SUM、AVG）、自定义扫描

- **使用场景**：
  - **服务端聚合**：在 RegionServer 上直接计算 SUM/COUNT，减少数据传输
  - **动态列族**：根据数据动态添加列
  - **自定义过滤**：服务端直接过滤不需要的数据

**协处理器的加载方式：**
1. **配置加载**（所有表）：在 `hbase-site.xml` 中配置，作用于所有表
2. **表级别加载**：通过 HBase Shell 或 API 为特定表加载协处理器

**注意事项：**
- 协处理器运行在 RegionServer 上，代码错误可能导致 RegionServer 崩溃
- 生产环境需充分测试后再部署
- Observer 应保持轻量，避免阻塞读写操作

**来源**：[HBase 协处理器详解 | 博客园](https://www.cnblogs.com/huizhang)

## 题目五：HBase 如何实现高可用（HA）？Master 故障后会怎样？

### 答案

HBase 通过多副本和故障转移机制实现高可用，具体体现在以下层面：

**1. ZooKeeper 选举机制**
- ZooKeeper 维护集群元数据（表结构、Region 位置、Server 状态）
- Master 和 RegionServer 都在 ZooKeeper 中注册为临时节点
- 一旦某个节点宕机，对应的临时节点消失，触发集群重新选举

**2. HMaster 高可用**
- 可以部署多个 HMaster，通过 ZooKeeper 选举产生主 Master
- 主 Master 故障后，其他备用 Master 会自动抢主，接管集群管理
- 切换过程中：
  - 元数据表（`hbase:meta`）重新上线
  - Region 重新分配到可用的 RegionServer
  - 整个过程约需几分钟，期间写入可能短暂受影响

**3. RegionServer 高可用**
- 每个 Region 的数据在 HDFS 上默认有 3 副本
- RegionServer 宕机后：
  - ZooKeeper 检测到 Server 消失，通知 HMaster
  - HMaster 将该 Server 上的 Region 重新分配到其他健康节点
  - HDFS 上的数据副本保证数据不丢失
  - 宕机期间，写入该 Server 的数据通过 WAL 恢复

**4. WAL（预写日志）保障数据不丢失**
- 所有写入先写入 WAL，宕机后可通过 WAL 重放恢复
- RegionServer 崩溃时，未刷写的数据会由 HMaster 分配到其他 Server 重放 WAL

**5. 数据本地性（Data Locality）**
- HBase 尽量让 Region 所在 RegionServer 与 HDFS DataNode 共置
- 数据副本优先放在本地节点，减少网络 I/O

**故障恢复流程：**
```
RegionServer 宕机
    ↓
ZooKeeper 检测到心跳消失（Session Timeout）
    ↓
HMaster 收到通知，启动 Region 迁移
    ↓
其他 RegionServer 接收 Region 和 WAL
    ↓
重放 WAL，恢复未刷写的数据
    ↓
集群恢复正常（通常 3~10 分钟）
```

**来源**：[HBase 高可用原理 | 阿里云开发者社区](https://help.aliyun.com/document-detail/927.html)
