### 题目1：HBase 的架构原理是什么？请详细说明 RegionServer 和 HMaster 的作用。

【答案】

HBase 是一个基于 Google BigTable 思想开发的列式存储数据库，采用 Master-Slave 架构：

**核心组件**：

| 组件 | 作用 |
|------|------|
| **HMaster** | 负责表的 CRUD 操作（创建、删除、修改表结构），以及 RegionServer 的负载均衡、故障转移、Region 分裂与合并协调 |
| **RegionServer** | 负责处理实际的数据读写请求，一个 RegionServer 包含多个 Region，每个 Region 存储一段 rowkey 范围的数据 |
| **ZooKeeper** | 负责选举主 HMaster、存储元数据信息（如 -ROOT- 表位置、Meta 表位置），以及检测 RegionServer 存活状态 |
| **HDFS** | HBase 的底层存储，默认使用 HDFS 存储数据文件（HFile），提供数据副本和高可靠性 |

**数据读写流程**：

1. **写数据**：
   - 客户端先访问 ZooKeeper，获取 Meta 表所在的 RegionServer
   - 根据 RowKey 从 Meta 表中找到目标数据所在的 RegionServer
   - 直接连接目标 RegionServer，将数据写入 MemStore（内存缓冲区）
   - 当 MemStore 达到阈值（默认 128MB），触发 Flush 刷写到磁盘生成 HFile
   - 数据先写入 WAL（Write Ahead Log）保证持久性，再写入 MemStore

2. **读数据**：
   - 通过 ZooKeeper 和 Meta 表定位到目标 RegionServer
   - 先在 MemStore 中查找，再在对应的 HFile 中查找
   - HBase 会合并 MemStore 和 HFile 的数据返回给客户端

**Region 分裂**：当某个 Region 过大（默认超过 10GB）时，HMaster 会将其分裂为两个子 Region，实现负载均衡。

### 题目2：HBase 的 RowKey 设计原则有哪些？如何避免热点问题？

【答案】

**RowKey 设计核心原则**：

1. **长度原则**：RowKey 推荐越短越好（不超过 16 字节），因为数据持久化到 HDFS 时，文件名就是 RowKey，过长会浪费 HDFS 存储空间

2. **散列原则**：RowKey 需要保证数据均匀分布在所有 Region 上，避免数据倾斜和热点

3. **唯一原则**：RowKey 必须唯一标识一行数据，类似关系型数据库的主键

**避免热点（HotSpotting）的策略**：

| 方法 | 说明 | 适用场景 |
|------|------|----------|
| **加盐（Salting）** | 在 RowKey 前面添加随机前缀（a、b、c），让数据分散到不同 Region | 写入并发高，但读取时需要知道前缀 |
| **哈希（Hash）** | 对原始 RowKey 取 MD5 或其他哈希值作为前缀 | 数据均匀分布，但无法支持范围查询 |
| **时间戳反转** | 将时间戳反转（如 9999999999 - timestamp） | 按时间倒序查询场景（如消息流） |
| **字符串反转** | 如 `phone_reversed_timestamp` | 手机号等有规律前缀的字段 |
| **随机数** | 在 RowKey 末尾加随机数 | 完全打散，但牺牲有序性 |

**设计示例**：
```bash
# 原始设计（热点）：时间戳直接作为 RowKey
RowKey: 20260624120000001  # 所有新数据集中在最后一个 Region

# 加盐设计
RowKey: 0_20260624120000001  # 数据分散到多个 Region

# 哈希设计
RowKey: md5(original_key).substring(0,6)_original_key
```

**最佳实践**：
- 高并发写入场景：加盐 + 预分区（提前创建多个 Region）
- 查询为主场景：业务主键直接作为 RowKey
- 时间序列场景：反转时间戳 + 业务标识组合

### 题目3：HBase 的 MemStore 和 HFile 是什么？Flush 和 Compaction 机制是什么？

【答案】

**核心存储概念**：

| 概念 | 说明 |
|------|------|
| **MemStore** | 写缓冲区，存在于 RegionServer 内存中，数据先写入 MemStore，顺序写入保证高写入性能 |
| **HFile** | HBase 数据文件的存储格式，基于 Hadoop 的 SSTable，保存在 HDFS 上 |
| **Store** | 每个 Column Family 对应一个 Store，每个 Store 包含一个 MemStore 和多个 HFile |
| **StoreFile** | HFile 在 HBase 层级的抽象，一个 StoreFile 对应一个 HFile |

**Flush 机制（刷写）**：

触发条件（满足任一即触发）：
- MemStore 大小达到 `hbase.hregion.memstore.flush.size`（默认 128MB）
- RegionServer 的 MemStore 总大小超过 `java.heapsize * 0.4 * 0.95`
- WAL 日志数量超过 `hbase.regionserver.maxlogs`（默认 32）
- 定时刷新：超时 `hbase.regionserver.optionalcacheflushinterval`（默认 1 小时）

刷写流程：
1. 数据先写入 WAL（保证持久性）
2. 数据写入 MemStore（内存）
3. 当触发 Flush 时，MemStore 一次性写出为 HFile
4. 刷写完成后，清空 MemStore，WAL 可以被丢弃

**Compaction 机制（合并）**：

| 类型 | 触发条件 | 合并范围 |
|------|----------|----------|
| **Minor Compaction（小合并）** | HFile 数量超过 `hbase.hstore.compaction.min`（默认 3） | 选取几个较小的 HFile 合并成一个，略删除的过期数据 |
| **Major Compaction（大合并）** | 定时触发（默认 7 天一次）或手动触发 | 合并一个 Store 下的所有 HFile，删除过期数据和版本过界的版本 |

合并目的：
- 减少 HFile 数量，降低 BlockCache 缓存失效
- 清理已删除和过期的数据
- 优化读性能（减少需要扫描的 HFile 数量）

⚠️ **注意**：Major Compaction 会产生大量 IO 和网络流量，通常在业务低峰期进行。

### 题目4：HBase 的协处理器（Coprocessor）是什么？Observer 和 Endpoint 有什么区别？

【答案】

**协处理器概念**：

HBase 协处理器是一种扩展机制，允许用户在 RegionServer 上运行自定义代码，在数据写入、读取、删除等生命周期钩子点执行自定义逻辑，类似关系型数据库的触发器和存储过程。

**两种协处理器类型**：

| 类型 | 特点 | 典型用途 |
|------|------|----------|
| **Observer** | 类似触发器，被动响应事件调用 | 数据验证、权限控制、审计日志、二级索引 |
| **Endpoint** | 类似存储过程，主动调用执行 | 聚合计算（如 SUM、AVG）、批量数据处理 |

**Observer 类型**：

| 类型 | 触发时机 | 使用场景 |
|------|----------|----------|
| **RegionObserver** | 数据操作前后（Get、Put、Delete、Scan） | 数据过滤、列值修改、访问控制 |
| **MasterObserver** | 表操作前后（创建、删除、修改表结构） | 权限验证、操作审计 |
| **WALObserver** | WAL 写入前后 | WAL 过滤或复制 |

**Endpoint 示例**（Aggregation Coprocessor）：

```java
// 定义端点接口
public interface SumEndpoint extends CoprocessorProtocol {
    long getSum(byte[] family, byte[] qualifier);
}

// 服务端实现
public class SumEndpointImpl implements SumEndpoint {
    @Override
    public long getSum(byte[] family, byte[] qualifier) {
        // 扫描整个 Region，累加指定列的值
    }
}
```

**加载方式**：
1. **静态加载**（所有表生效）：在 hbase-site.xml 配置，重启生效
2. **动态加载**（指定表生效）：通过 HBase Shell 或 API 为指定表加载

**典型应用场景**：
- **二级索引**：当数据写入 Put 时，自动更新索引表
- **聚合查询**：在 RegionServer 端本地计算 SUM/COUNT，避免全表扫描
- **数据权限**：基于行级别的访问控制
- **审计日志**：记录每次数据访问的操作人、操作时间

### 题目5：HBase 和关系型数据库（如 MySQL）有什么区别？什么场景下应该选择 HBase？

【答案】

**核心区别对比**：

| 维度 | HBase | MySQL/关系型数据库 |
|------|-------|---------------------|
| **数据类型** | NoSQL，仅存储字节数组 | 支持丰富的数据类型 |
| **数据模型** | 列式存储，宽表模型 | 行列存储，严格的 schema |
| **事务支持** | 仅支持单行事务，不支持多行事务 | 支持 ACID 事务 |
| **查询能力** | 仅支持主键查询和范围扫描，无 join | 支持复杂 SQL、JOIN、聚合 |
| **扩展性** | 天然支持水平扩展，PB 级数据 | 垂直扩展为主，水平扩展困难 |
| **写入性能** | 高吞吐顺序写入，适合高并发写入 | 随机写入，性能相对较低 |
| **一致性** | 最终一致性（可配置强一致性） | 强一致性 |
| **索引** | 只有 RowKey 主键索引 | 支持主键、唯一索引、普通索引 |
| **SQL 支持** | 无（可通过 Phoenix 支持） | 原生 SQL |

**HBase 适用场景**：

✅ **应该使用 HBase**：
- **海量数据存储**：数据量达到 TB 甚至 PB 级别
- **高并发写入**：如埋点数据、日志数据、传感器数据（IoT）
- **稀疏数据**：存在大量 NULL 列的宽表（如用户画像）
- **无需复杂查询**：只需根据 RowKey 或范围查询
- **时间序列数据**：如监控数据、订单流水、业务事件
- **实时响应要求高**：百毫秒级响应

❌ **不应使用 HBase**：
- 需要复杂 JOIN 查询
- 需要事务保证（多行操作）
- 需要二级索引和模糊查询
- 数据量较小（GB 级别以下）
- 需要 SQL 完整生态（OLAP 分析）

**典型案例**：
- 京东海量订单存储（万亿级别）
- 小米消息推送系统（每天数十亿条）
- 淘宝用户行为分析
