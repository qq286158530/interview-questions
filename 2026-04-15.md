## 题目1：Java基础 | HashMap 的扩容机制详解

**【答案】**

HashMap 的扩容机制是面试中的高频考点，主要包含以下核心要点：

### 1. 触发条件
- 当 HashMap 中 **元素数量 > 容量 × 负载因子**（默认：16 × 0.75 = 12）时触发扩容
- 也可能因为链表长度 ≥ 8 且桶数组长度 < 64 时触发（树化前优先扩容）

### 2. 扩容过程
- 创建一个新的桶数组，容量翻倍（原 16 → 新 32）
- 遍历原数组中的每个桶，将元素重新哈希到新数组
- 由于新容量是原容量的 2 倍，元素的桶位置只有两种可能：
  - 保持原索引位置（原 hash & (n-1) 不变）
  - 移动到原索引 + 原容量 的位置（原 hash & n）

### 3. 线程安全问题
- **HashMap 不是线程安全的**，多线程下扩容可能导致死循环（Java 7 头插法）、数据覆盖等
- 替代方案：ConcurrentHashMap（分段锁 / CAS）、Hashtable（全局锁）、Collections.synchronizedMap

### 4. Java 7 vs Java 8 区别
| 特性 | Java 7 | Java 8 |
|------|--------|--------|
| 数据结构 | 数组 + 链表 | 数组 + 链表/红黑树 |
| 扩容方式 | 头插法（可能死循环） | 尾插法 + 红黑树优化 |
| 树化阈值 | 无 | 链表长度 ≥ 8 且桶数 ≥ 64 |

## 题目2：Java进阶 | JVM 类加载机制与双亲委派模型

**【答案】**

### 1. 类加载的五个阶段
```
加载(Loading) → 验证(Verification) → 准备(Preparation) → 解析(Resolution) → 初始化(Initialization)
```

### 2. 双亲委派模型（Parent Delegation Model）

**核心原则**：类加载器收到加载请求时，先委派给父类加载器处理，只有父加载器无法完成时，才自己尝试加载。

**三层类加载器**：
- **Bootstrap ClassLoader**（启动类加载器）：加载 `JAVA_HOME/lib` 核心类库
- **Extension ClassLoader**（扩展类加载器）：加载 `JAVA_HOME/lib/ext` 扩展类库
- **Application ClassLoader**（应用类加载器）：加载用户 classpath 下的类

**好处**：保证了 Java 核心类库的稳定运行，避免核心 API 被篡改（如自定义 `java.lang.String` 不会被加载）。

### 3. 打破双亲委派的场景
- **JDBC 驱动加载**（SPI 机制：通过 `ServiceLoader` 委派给线程上下文类加载器）
- **OSGi 模块化框架**
- **热部署/热替换**：Tomcat 自定义 WebAppClassLoader

### 4. 类加载原则
- 主动使用才会触发初始化（new、static 访问、反射等）
- 被动使用不触发（如子类加载时父类不初始化，数组类型由 JVM 自动创建）

## 题目3：Java基础 | synchronized 和 ReentrantLock 的区别

**【答案】**

### 功能对比

| 特性 | synchronized | ReentrantLock |
|------|-------------|----------------|
| 锁类型 | 悲观独占锁 | 悲观独占锁（可重入） |
| 底层实现 | JVM 内置（monitorenter/monitorexit） | AQS + CAS |
| 加锁方式 | 隐式（自动释放） | 显式（手动加锁/解锁） |
| 公平锁 | 非公平（默认） | 支持公平/非公平切换 |
| 等待可中断 | 不可中断 | 可通过 `lockInterruptibly()` 中断 |
| 锁超时 | 不可 | 支持 `tryLock(timeout)` |
| 条件变量 | 内置（wait/notify） | 多个 `Condition` |
| 性能 | JDK 6 后优化（偏向锁/轻量级锁） | 较高（需手动优化） |

### 核心理解
- **synchronized**：JVM 层面的锁，语法简洁，但灵活性不足
- **ReentrantLock**：代码层面的锁，支持公平/非公平、tryLock、超时、多条件变量等高级特性
- **两者皆可重入**：同一线程可多次获取锁，计数累加

### 选用建议
- 一般场景优先使用 `synchronized`（简洁安全）
- 需要超时、终端、公平锁等特性时用 `ReentrantLock`
- 高并发下推荐 `ReentrantReadWriteLock` 读写分离优化

## 题目4：Spring Boot | 自动配置原理（@SpringBootApplication 深度解析）

**【答案】**

### 核心注解组合
```java
@SpringBootApplication = @Configuration + @EnableAutoConfiguration + @ComponentScan
```

### @EnableAutoConfiguration 核心流程

**1. Spring Factories 加载**
- 框架在 `META-INF/spring.factories`（Spring 3.x）或 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`（Spring 6.x）中注册候选配置类

**2. @Import 导入自动配置类**
- `AutoConfigurationImportSelector` 实现了 `DeferredImportSelector`
- 在所有 `@Configuration 类加载完成后`，通过 `SpringFactoriesLoader` 加载 `spring.factories` 中的 `EnableAutoConfiguration` 项

**3. 条件过滤（@Conditional）**
每个自动配置类都有条件注解过滤，排除不满足条件的配置：
- `@ConditionalOnClass`：当类路径存在某类时生效
- `@ConditionalOnMissingClass`：当类路径不存在某类时生效
- `@ConditionalOnBean`：容器中是否存在指定 Bean
- `@ConditionalOnProperty`：配置属性是否匹配

**4. 自动配置生效后的覆盖机制**
- 用户自定义的 `@Bean` / `@Component` 会覆盖自动配置类中的同名 Bean
- 优先级：用户配置 > 自动配置

### 自定义 Starter 示例
1. 编写配置类 + `@Configuration`
2. 在 `META-INF/spring.factories` 中注册
3. 使用 `@ConditionalOnProperty` 等控制开关

## 题目5：Spring Cloud | Nacos 与 Eureka 的区别对比

**【答案】**

### 核心架构对比

| 特性 | Nacos | Eureka |
|------|-------|--------|
| **一致性协议** | CP + AP 双模式 | 仅 AP |
| **服务健康检查** | TCP/HTTP/MySQL/Metrics | 心跳机制 |
| **注册中心/配置中心** | 二合一 | 仅注册中心 |
| **多环境/命名空间** | 支持（namespace/group/cluster） | 仅支持 region |
| **社区活跃度** | 活跃（阿里背书） | 停止维护（Netflix 组件） |
| **运维复杂度** | 需独立部署 | Spring Cloud 内嵌 |
| **数据持久化** | MySQL | 无（内存） |

### Nacos 的 CP/AP 切换
- `nacos.standalone=true` 时为 AP 模式（默认单机）
- 集群模式下通过 RAFT 协议实现 CP
- 服务注册默认 AP，配置管理默认 CP

### 为什么弃用 Eureka？
- Netflix 宣布停止维护 Eureka 2.0
- Eureka 不支持配置中心，需结合 Spring Cloud Config
- Nacos 社区更活跃，功能更全面

### 选型建议
- 新项目优先选 **Nacos**（功能全面，趋势所向）
- 如果只需服务注册且已有成熟体系，可考虑 Consul（多数据中心支持好）
- K8s 环境下可选 K8s Native Service（无需额外注册中心）

## 题目6：Spring Boot | Spring 事务传播行为与隔离级别

**【答案】**

### 7 种传播行为（Propagation）

| 传播行为 | 含义 |
|---------|------|
| **REQUIRED**（默认） | 有事务则加入，无则新建 |
| **REQUIRES_NEW** | 每次新建事务，挂起当前事务 |
| **NESTED** | 嵌套事务（Savepoint），无则新建 |
| **SUPPORTS** | 有则加入，无则以非事务运行 |
| **NOT_SUPPORTED** | 以非事务运行，挂起当前事务 |
| **MANDATORY** | 必须在事务中，否则抛异常 |
| **NEVER** | 必须无事务，否则抛异常 |

**高频面试题**：
- `@Transactional( propagation=REQUIRES_NEW )` 独立事务，不受外层事务回滚影响
- 嵌套事务 `NESTED` 基于 Savepoint，外层回滚会带动内层回滚，但内层可独立回滚

### 4 种隔离级别（Isolation）

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| **READ_UNCOMMITTED** | ✅ 允许 | ✅ 允许 | ✅ 允许 |
| **READ_COMMITTED** | ❌ 禁止 | ✅ 允许 | ✅ 允许 |
| **REPEATABLE_READ**（MySQL默认） | ❌ 禁止 | ❌ 禁止 | ✅ 允许 |
| **SERIALIZABLE** | ❌ 禁止 | ❌ 禁止 | ❌ 禁止 |

**MySQL 的 MVCC 机制**：REPEATABLE_READ 下通过 ReadView 快照读保证不出现幻读（但在两次快照读之间插入数据仍可能出现问题，需配合间隙锁）

### @Transactional 失效场景
1. **方法内部调用**（this.xxx()）- 代理对象不生效
2. **非 public 方法** - Spring AOP 限制
3. **异常被 catch 吞掉** - 不会触发回滚
4. **异常类型不匹配** - 默认只回滚 RuntimeException 和 Error
5. **多数据源未配置** - `@Transactional` 默认作用于主数据源

## 题目7：Spring | Spring Bean 的生命周期与作用域

**【答案】**

### Bean 生命周期完整流程

```
实例化 → 属性赋值 → 初始化前（BeanPostProcessor.postProcessBeforeInitialization）
       → 初始化（@PostConstruct / InitializingBean.afterPropertiesSet）
       → 初始化后（BeanPostProcessor.postProcessAfterInitialization）
       → 使用 → 销毁（@PreDestroy / DisposableBean.destroy()）
```

### 作用域（Scope）

| 作用域 | 范围 | 线程安全 |
|--------|------|---------|
| **singleton**（默认） | 整个容器只有一个 | 否（需设计为无状态） |
| **prototype** | 每次获取创建新实例 | 是 |
| **request** | 一次 HTTP 请求 | 否 |
| **session** | 一次 HTTP 会话 | 否 |
| **application** | ServletContext 生命周期 | 否 |
| **websocket** | WebSocket 生命周期 | 否 |

### 初始化/销毁回调方式（优先级从高到低）
1. `@PostConstruct` / `@PreDestroy`（JSR-250，推荐）
2. `InitializingBean.afterPropertiesSet()` / `DisposableBean.destroy()`
3. `@Bean(initMethod = "xxx", destroyMethod = "xxx")`
4. `BeanFactoryPostProcessor` 和 `BeanPostProcessor`（框架级）

### 循环依赖问题（Spring 解决）
- **构造器注入**：无法解决，原型 Bean 不支持提前暴露
- **Setter 注入（单例）**：通过三级缓存（singletonObjects / earlySingletonObjects / singletonFactory）解决
- **原型 Bean**：不支持循环依赖，直接报错

## 题目8：Spring Cloud | Sentinel 与 Hystrix 的区别

**【答案】**

### 核心对比

| 特性 | Sentinel | Hystrix |
|------|----------|---------|
| **熔断策略** | 基于响应时间、异常比例、异常数 | 基于异常比例 |
| **限流维度** | 多种维度（QPS/并发/调用关系） | 仅线程池限流 |
| **实时指标** | 控制台实时查看 | 需配合 Turbine |
| **规则配置** | 支持推送/拉取多种模式 | 限流规则需重启 |
| **注解支持** | `@SentinelResource` | `@HystrixCommand` |
| **状态** | Alibaba 主推，持续更新 | Netflix 停止维护 |
| **适配框架** | Dubbo/Spring Cloud/ Sentinel GateWay | Spring Cloud Netflix |

### Sentinel 的核心概念

**1. 资源**：被保护的业务代码或接口

**2. 规则类型**：
- **流量控制**（FlowRule）：QPS / 并发数限流
- **熔断降级**（DegradeRule）：慢调用比例/异常比例/异常数
- **系统自适应保护**（SystemRule）：CPU / 线程数 / 入口 QPS
- **热点规则**：参数级别的限流

**3. 滑动窗口算法**：Sentinel 使用滑动窗口（LeapArray）统计实时数据，比 Hystrix 的桶计数更精确

### 选型建议
- 新项目：直接选 **Sentinel**（功能更强，生态活跃）
- 老项目迁移：从 Hystrix 迁移到 Sentinel 成本较低（都支持注解方式）

## 题目9：消息队列 | Kafka 如何保证消息不丢失？

**【答案】**

### 三层保障机制

#### 1. 生产者端（acks 配置）
| acks 值 | 含义 | 可靠性 |
|--------|------|--------|
| acks=0 | 发出去就认为成功 | 最低 |
| acks=1 | Leader 收到并写入就成功 | 中等（可能丢 Follower） |
| acks=all（-1） | ISR 列表所有副本都写入成功 | 最高 |

**推荐配置**：`acks=all` + `retries=3` + `enable.idempotence=true`（幂等生产者）

#### 2. Broker 端
- `replication.factor ≥ 3`：副本数至少 3
- `min.insync.replicas ≥ 2`：ISR 中最少副本数
- `unclean.leader.election=false`：不允许非同步副本成为 Leader（避免数据丢失）
- 刷盘策略：`log.flush.interval.messages=1`（每条消息刷盘，可靠性高但影响性能）

#### 3. 消费者端
- **手动提交 offset**（`enable.auto.commit=false`）
- **先处理业务，再提交 offset**（避免消息丢失但可能重复消费）
- 消费者崩溃时，重平衡后从上次提交 offset 继续消费

### 消息重复消费问题
- 本质上**无法完全避免**，只能通过 **幂等性设计** 解决
- 幂等方案：Redis 分布式锁 / 数据库唯一索引 / 去重表 / 消息表

### Kafka 高效原因
- 顺序写磁盘（顺序 I/O）
- 零拷贝技术（sendfile）
- 分区并行 + 批量压缩

## 题目10：消息队列 | RabbitMQ 镜像队列与仲裁队列的区别

**【答案】**

### 镜像队列（Classic Queue Mirrors）

**架构**：主节点（Master）+ 多个从节点（Slave），主节点负责读写，从节点被动同步

**问题**：
- 主节点故障时，从节点需要重新选主（有短暂不可用）
- 消息堆积时同步压力大
- 队列迁移困难
- 负载不均（Master 承担所有读写）

**配置**：`ha-mode` = all/exactly/nodes

### 仲裁队列（Quorum Queue）

**架构**：基于 RAFT 共识协议，实现真正分布式

**优势**：
| 特性 | 镜像队列 | 仲裁队列 |
|------|---------|---------|
| 数据安全 | 一般 | 强（多数派写入） |
| 故障恢复 | 慢（重新选主） | 快（自动切换） |
| 容量扩展 | 受限 | 支持动态增删节点 |
| 磁盘占用 | 冗余复制 | 优化存储 |
| 跨机房 | 弱 | 强 |

**限制**：
- 不支持某些经典特性（non-durable 队列、excludes、优先级队列）
- 不支持延迟消息
- 需要 Erlang/OTP 支持

**选型建议**：
- 新项目、可靠性要求高 → **仲裁队列**
- 老系统兼容、延迟消息需求 → **镜像队列**

## 题目11：消息队列 | RocketMQ 事务消息原理

**【答案】**

### 两阶段提交 + 事务状态回查

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  Producer   │ ──1──▶ │   Broker    │ ──2──▶ │  消费者      │
│  (执行本地   │      │ (存储半消息) │      │ (消费消息)   │
│   事务)      │ ◀─4── │ (发送commit │ ◀─3── │             │
└─────────────┘      │  /rollback) │      └─────────────┘
                     └─────────────┘
```

### 流程详解

**第一阶段（Half Message）**：
- Producer 发送半消息（Prepare），消息暂不可消费
- Broker 持久化半消息，返回成功

**第二阶段（本地事务）**：
- Producer 执行本地事务（数据库操作等）
- 根据结果提交 `commit` 或 `rollback`

**第三阶段（事务回查）**：
- 如果 Broker 长时间未收到确认，主动回查 Producer 事务状态
- Producer 需要实现 `TransactionListener.checkLocalTransaction()`
- 回查次数默认 15 次（可配置）

### 消息状态
| 状态 | 含义 |
|------|------|
| **COMMIT_MESSAGE** | 提交，消费者可见 |
| **ROLLBACK_MESSAGE** | 回滚，消息删除 |
| **UNKNOWN** | 未知，Broker 触发回查 |

### 注意事项
- 事务消息不支持延迟投递
- 事务回查会加重 Broker 负担，避免本地事务耗时过长
- 消费者需要做好幂等（因为可能重复投递）

## 题目12：AI Agent | LangChain 的核心组件与工作流程

```
│  │  (推理)   │  │ (记忆)   │  │  (编排工作流)      │   │
│  └──────────┘  └──────────┘  └──────────────────┘   │
└──────────────────────────────────────────────────────┘
```

### 六大核心组件

**1. Model I/O**
- 支持 OpenAI、HuggingFace、Anthropic 等模型
- 统一接口调用不同 LLM

**2. Prompt Template**
- 模板化管理提示词
- 支持动态参数注入、Few-shot 示例

- 文档加载 → 分词 → 向量化 → 存储 → 检索

**6. Memory**
- **短期记忆**：对话历史
- **长期记忆**：向量数据库持久化

### ReAct 模式流程
```
Thought → Action → Observation → Thought → ...
```
- Thought：思考当前情况
- Action：决定使用什么工具
- Observation：观察工具返回结果
- 循环直到得到最终答案

## 题目13：AI Agent | RAG（检索增强生成）的核心流程与优化策略

**【答案】**

### RAG 核心流程

```
用户问题 ──▶ 嵌入模型 ──▶ 向量检索 ──▶ 上下文组装 ──▶ LLM 生成 ──▶ 回答
```

**1. 文档处理阶段**
- 文档加载（PDF/TXT/Markdown）
- 文本分块（Chunking）：固定大小 / 语义分句 / 递归字符分割
- 块大小选择：太小→上下文不足，太大→噪音多（通常 500-1000 tokens）

**2. 向量化阶段**
- Embedding 模型选择（text-embedding-ada-002 / BGE / M3E）
- 向量存储到向量数据库（Chroma / FAISS / Milvus / Qdrant）

**3. 检索阶段**
- 用户 query → 同样 Embedding
- Top-K 相似度检索
- 可配合混合搜索（关键词 + 向量）

**4. 生成阶段**
- 将检索到的 chunks 组装到 prompt
- LLM 结合上下文生成答案

### 优化策略

**检索优化**：
| 策略 | 方法 |
|------|------|
| **混合检索** | BM25 + 向量检索融合 |
| **重排序** | 用 Cross-encoder 对 Top-K 重排 |
| **查询扩展** | HyDE / Multi-Query |
| **元数据过滤** | 标签/时间等条件过滤 |

**生成优化**：
- **Context Compression**：压缩检索到的上下文
- **Self-RAG**：让 LLM 自己判断是否需要检索
- **Corrective-RAG**：检测幻觉并修正

**评估指标**：
- 检索：Hit Rate / MRR / NDCG
- 生成：RAGAS / TruLens（答案相关性、忠实度）

## 题目14：算法 | LRU 缓存设计与实现

**【答案】**

### 题目描述
设计并实现一个 LRU（最近最少使用）缓存，要求 `get` 和 `put` 操作都是 **O(1)** 时间复杂度。

### 解题思路
**核心数据结构**：哈希表（HashMap）+ 双向链表
- HashMap：O(1) 查找
- 双向链表：O(1) 增删节点（维护顺序）

### Python 实现

```python
class LRUCache:

    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = {}  # key -> node
        # 虚拟头尾节点简化操作
        self.head = Node(0, 0)  # 最近最常用在头
        self.tail = Node(0, 0)
        self.head.next = self.tail
        self.tail.prev = self.head

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        node = self.cache[key]
        self._move_to_head(node)  # 标记为最近使用
        return node.value

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            node = self.cache[key]
            node.value = value
            self._move_to_head(node)
        else:
            new_node = Node(key, value)
            self.cache[key] = new_node
            self._add_to_head(new_node)
            if len(self.cache) > self.capacity:
                removed = self._remove_tail()
                del self.cache[removed.key]

    def _add_to_head(self, node):
        node.prev = self.head
        node.next = self.head.next
        self.head.next.prev = node
        self.head.next = node

    def _remove_node(self, node):
        node.prev.next = node.next
        node.next.prev = node.prev

    def _move_to_head(self, node):
        self._remove_node(node)
        self._add_to_head(node)

    def _remove_tail(self):
        node = self.tail.prev
        self._remove_node(node)
        return node


class Node:
    def __init__(self, key=0, value=0):
        self.key = key
        self.value = value
        self.prev = None
        self.next = None
```

### 复杂度分析
| 操作 | 时间复杂度 | 空间复杂度 |
|------|-----------|-----------|
| get | O(1) | - |
| put | O(1) | O(capacity) |

### Redis LRU 实现补充
- Redis 使用 `maxmemory-policy allkeys-lru` 或 `volatile-lru`
- 近似 LRU（避免全量排序）：抽样 N 个key，淘汰最少使用

## 题目15：算法 | 字符串最长不重复子串（滑动窗口）

**【答案】**

### 题目
给定一个字符串，找出其中不包含重复字符的最长子串的长度。

### 解法：滑动窗口（Sliding Window）

```python
def length_of_longest_substring(s: str) -> int:
    if not s:
        return 0
    
    char_index = {}  # 字符 -> 最近出现的位置
    max_len = 0
    left = 0  # 窗口左边界
    
    for right, char in enumerate(s):
        # 如果字符已在窗口中（位置 >= left），收缩左边界
        if char in char_index and char_index[char] >= left:
            left = char_index[char] + 1
        
        char_index[char] = right
        max_len = max(max_len, right - left + 1)
    
    return max_len
```

### 执行示例

```
s = "abcabcbb"
右指针移动：
a(0) → ab(2) → abc(3) → abca(发现a重复，左边界移到1) → bcab(4) → bcabc(5) → bcabcb(发现b重复，左边界移到3) → bcabc(5) → ...

max_len = 3 ("abc")
```

### 复杂度
| 维度 | 复杂度 |
|------|--------|
| 时间 | O(n) |
| 空间 | O(min(m, n))，m 为字符集大小（ASCII 128/256 或 Unicode） |

### 变种题
| 题目 | 区别 |
|------|------|
| 无重复最长子串 | 以上解法 |
| 最多 K 次重复字符的最长子串 | 维护 count 数组，窗口内重复字符 ≤ K |
| 连续最长重复 K 字符 | 分组计数 |
