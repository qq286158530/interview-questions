### 题目1：Flink 的架构是怎样的？JobManager 和 TaskManager 分别负责什么？

【答案】

**Flink 架构采用主从模式**，由 JobManager（主节点）和 TaskManager（从节点）组成。

**核心组件**：

1. **JobManager（主节点）**
   - **JobMaster**：负责单个 Job 的执行，协调作业执行，包括任务调度、资源申请、Checkpoint 协调
   - **ResourceManager**：管理 TaskManager 的 Slot，负责资源分配
   - **Dispatcher**：接收用户提交的 Job，为每个 Job 启动一个新的 JobMaster，并提供 REST 接口

2. **TaskManager（从节点）**
   - 每个 TaskManager 是一个 JVM 进程，启动多个 Slot
   - Slot 是资源分配的基本单位，每个 Slot 提供固定量的资源（内存 + CPU）
   - TaskManager 之间通过流式 shuffle 传输数据

**执行流程**：
```
用户代码 → StreamGraph → JobGraph → ExecutionGraph
                                    ↓
                         JobManager 调度到 TaskManager
                                    ↓
                    Source → Transformation → Sink
```

**Slot vs Parallelism**：
- Slot：物理资源，定义并发上限
- Parallelism：算子并发度，运行时实际并发数
- 算子并行度可以动态调整，不受 Slot 数量限制

### 题目2：Flink 的 Watermark 机制是什么？如何处理乱序数据？

【答案】

**Watermark（水位线）是 Flink 实现事件时间处理的核心机制**，用于解决乱序事件和窗口计算问题。

**核心概念**：
1. **Event Time（事件时间）**：数据本身携带的时间戳
2. **Processing Time（处理时间）**：数据到达 Flink 算子的系统时间
3. **Watermark**：一种特殊的时间戳，用于表示"所有时间戳小于此 watermark 的数据都已到达"

**乱序处理原理**：
```
Watermark = 事件时间 - 最大延迟时间
```
- 当 watermark >= 窗口结束时间时，触发窗口计算
- watermark t 表示：不会再有 t 之前的事件到达

**Watermark 传播**：
- Watermark 是按 stream partition 传播的
- 多输入算子取所有输入中最小的 watermark
- 输出时输出当前 watermark

**乱序处理策略**：
```java
// 设置 Watermark 策略
WatermarkStrategy
    .<Tuple2<String, Long>>forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner((event, timestamp) -> event.f1);
```

**Watermark 分类**：
| 类型 | 说明 |
|------|------|
| Periodic Watermark | 周期性生成（默认每200ms） |
| Punctuated Watermark | 每条数据判断是否生成 |

**注意事项**：
- Watermark 设为 Long.MAX_VALUE 表示"全局结束"，会触发所有窗口
- 多流 union 时，以最慢的流为准

📖 来源：https://ci.apache.org/projects/flink/flink-docs-release-1.17/docs/concepts/time/

### 题目3：Flink 的 Checkpoint 和 Savepoint 有什么区别？如何保证 Exactly-Once 语义？

【答案】

**Checkpoint 和 Savepoint 的区别**：

| 特性 | Checkpoint | Savepoint |
|------|------------|------------|
| 触发方式 | Flink 自动周期性触发 | 手动触发 |
| 用途 | 故障恢复 | 计划性备份/升级 |
| 依赖 | 需要 Job 重启 | 独立于 Job 存在 |
| 存储 | RocksDB/堆内状态 | 文件系统（可自定义） |
| 清理 | 自动清理旧版本 | 手动清理 |

**Checkpoint 原理（Chandy-Lamport 算法变体）**：
1. JobManager 定期向所有 Source 算子注入 Barrier（屏障）
2. Barrier 随数据流向下游传递，形成"检查点屏障"
3. 算子收到所有输入的 Barrier 后，触发状态快照
4. 快照完成后，向 JobManager 确认

**Exactly-Once 实现要求**：
1. **Source**：可重置读取位置（如 Kafka offset）
2. **算子状态**：Checkpoint 快照
3. **Sink**：幂等写入或两阶段提交

**两阶段提交 Sink（TwoPhaseCommitSink）**：
```java
public class TwoPhaseCommitSink implements SinkFunction, CheckpointedFunction {
    // 1. checkpoint 开始：预提交
    public void snapshotState() throws Exception;
    // 2. checkpoint 完成：正式提交
    public void notifyCheckpointComplete() throws Exception;
}
```

**Flink 与 Kafka 的 Exactly-Once**：
- Kafka 0.11+ 支持事务
- Flink Kafka Producer 使用两阶段提交协议
- 先预提交到 Kafka，完成 Checkpoint 后再正式提交

**常见 Sink 的语义保证**：
| Sink | 语义 | 说明 |
|------|------|------|
| Kafka Sink | Exactly-Once | 事务保证 |
| File Sink | Exactly-Once | Checkpoint + 原子性重命名 |
| MySQL Sink | At-Least-Once | 需要业务幂等 |
| Print Sink | At-Least-Once | 调试用 |

📖 来源：https://ci.apache.org/projects/flink/flink-docs-release-1.17/docs/ops/state/checkpoints/

### 题目4：Flink 的状态后端有哪些？RocksDB 和 Heap 有何区别？如何选型？

【答案】

**Flink 状态后端（State Backend）**：

| 后端 | 存储 | 适用场景 |
|------|------|----------|
| **HashMapStateBackend** | JVM Heap | 小状态、低延迟（会话窗口、配置缓存） |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘） | 大状态、大 Key/Value、增量 Checkpoint |

**HashMapStateBackend**：
- 状态存储在 JVM Heap 中
- 快速读写，但受限于内存大小
- 状态大小建议 < 100MB
- 重启后状态丢失

**EmbeddedRocksDBStateBackend**：
- 状态溢出到 RocksDB（LSM 树结构）
- 支持增量 Checkpoint
- 支持 Flash 存储，容量大
- 读写有序列化开销
- 支持异步快照

**RocksDB vs Heap 性能对比**：

| 指标 | HashMapStateBackend | RocksDB |
|------|---------------------|---------|
| 读写延迟 | ~1μs | ~10-100μs（含序列化） |
| 内存占用 | 全内存 | 可用磁盘空间 |
| 状态大小 | < 100MB | 无限制 |
| 快照速度 | 快 | 较慢 |
| 增量 Checkpoint | 不支持 | 支持 |

**选型建议**：
```
状态小（<100MB）+ 低延迟 → HashMapStateBackend
状态大 + 需要增量 Checkpoint → RocksDB
超大队列 + 增量 Checkpoint → RocksDB + Flash
```

**配置示例**：
```java
// HashMapStateBackend
env.setStateBackend(new HashMapStateBackend());

// RocksDB + HDFS
env.setStateBackend(new EmbeddedRocksDBStateBackend("hdfs:///checkpoints"));
```

### 题目5：Flink 窗口函数有哪几种？Session Window 如何解决数据稀疏问题？

【答案】

**窗口函数分类**：

| 分类 | 窗口类型 | 说明 |
|------|----------|------|
| **时间窗口** | TumblingEventTimeWindow | 固定大小，不重叠 |
| | SlidingEventTimeWindow | 固定大小，可重叠 |
| | EventTimeSessionWindow | 会话间隙触发 |
| | GlobalWindow | 全局一个窗口 |
| **计数窗口** | TumblingCountWindow | 固定数量，不重叠 |
| | SlidingCountWindow | 固定数量，可重叠 |

**滚动窗口（Tumbling）**：
```
窗口大小=10分钟
[00:00-10:00) | [10:00-20:00) | [20:00-30:00)
```
每个数据只属于一个窗口

**滑动窗口（Sliding）**：
```
窗口大小=10分钟，滑动=5分钟
[00:00-10:00)    | [00:05-10:05)
   [05:00-15:00) |    [05:05-15:05)
```
每个数据可属于多个窗口

**会话窗口（Session Window）**：
- 根据会话间隙动态创建窗口
- 超过指定 gap 没有数据，则窗口关闭

**Session Window 解决数据稀疏问题**：
```java
// 方式1：固定会话间隙
DataStream<String> session = stream
    .keyBy(...)
    .window(EventTimeSessionWindow.withGap(Time.minutes(10)))
    .sum("price");

// 方式2：动态间隙（自定义 Assigner）
DataStream<String> session = stream
    .keyBy(...)
    .window(EventTimeSessionWindow.withDynamicGap(
        new SessionWindowTimeGapExtractor<Tuple2<String, Long>>() {
            public long extract(Tuple2<String, Long> element, long currentTime) {
                return element.f1; // 根据数据动态计算间隙
            }
        }))
    .sum("price");
```

**Session Window 与 Watermark**：
- Session Window 依赖 Watermark 触发
- 如果数据乱序且稀疏，Watermark 可能永远不会超过 gap
- 解决方案：
  1. 设置 `withEventTime` + 合理的 Watermark 延迟
  2. 使用 ProcessingTime Session Window（不依赖 Watermark）
  3. 配置 `allowedLateness` 延迟数据容忍

**窗口API核心方法**：
```java
stream.keyBy(...).window(...).aggregate(new MyAggregateFunction());
// 或
stream.keyBy(...).windowAll(...).reduce(...)
```
