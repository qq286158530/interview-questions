## 题目一：HBase 的整体架构是怎样的？请描述各组件职责及数据写入路径。

### 参考答案

HBase 是一个面向列存储的分布式 NoSQL 数据库，基于 **Hadoop HDFS** 构建，整体架构如下：

**核心组件：**

| 组件 | 职责 |
|------|------|
| **ZooKeeper** | 管理集群元数据（ROOT/META 位置）、分布式锁、故障检测 |
| **HMaster** | 负责表的管理、Region 分配、RegionServer 负载均衡、故障恢复 |
| **RegionServer** | 负责数据读写、MemStore 刷写、HFile 合并等实际存储工作 |
| **HDFS** | 提供底层数据存储，DataNode 默认 3 副本 |

**表数据划分逻辑：**
```
Table → Region → Store → MemStore + HFiles
```

**数据写入路径（Put 流程）：**
1. 客户端连接 ZooKeeper，获取 META 表位置
2. 从 META 表中找到目标 RowKey 所在的 RegionServer
3. 数据先写入 ** WAL（Write-Ahead Log）**，保证故障可恢复
4. 数据写入对应 **MemStore**（内存），同时记录到 WAL
5. 当 MemStore 达到阈值（默认 `hbase.hregion.memstore.flush.size=128MB`），触发 **刷写（Flush）** 生成 HFile
6. 当 HFile 数量过多时，触发 **Compaction** 合并与淘汰

**故障恢复：** 若 RegionServer 宕机，HMaster 通过 ZooKeeper 感知，将该节点的 Region 迁移到其他节点，同时从 WAL 重放未刷写的数据。

**来源：** [HBase 官方文档 - Architecture](https://hbase.apache.org/book.html#architecture)

## 题目二：HBase 中 RowKey 如何设计？请列举常见设计原则与反模式。

### 参考答案

RowKey 是 HBase 数据的唯一主键，类比关系型数据库的主键，但支持范围扫描，其设计直接影响读写性能。

**设计原则：**

1. **长度适中**：建议 `16~24` 字节，过长会增加存储开销并影响索引效率
2. **散列均匀**：避免热点（Hot Spot），防止所有数据集中到少数 RegionServer
3. **查询友好**：将常用于查询的字段放在 RowKey 前缀，便于范围扫描

**常见设计模式：**

```
# 场景：用户订单查询，用户ID + 时间戳 + 订单ID
RowKey = MD5(userId).substring(0,6) + userId + timestamp + orderId
# 好处：MD5 前缀保证散列，后面的 userId 保证同用户订单连续存储
```

**反模式（需避免）：**

| 反模式 | 问题 | 解决方案 |
|--------|------|----------|
| 连续自增 ID 作为 RowKey | 数据写入全部打到末尾 Region，造成热点 | 盐值（Salting）、MD5 散列、翻转数值 |
| 宽表设计（数十列簇） | 读取时解析无效列，浪费网络/内存 | 拆分为多个表或合理拆分列簇 |
| RowKey 过短或过于简单 | 散列不均，热点 Region | 加盐或哈希打散 |
| 列名冗长 | 列名随每行数据存储，浪费存储 | 使用短列名或列族压缩 |

**热点检测：** 可通过 HBase UI 或 `RegionServer` 的 `requests` 指标监控读写分布是否均匀。

**来源：** [HBase 官方文档 - RowKey Design](https://hbase.apache.org/book.html#rowkey.design)

## 题目三：简述 HBase 的 MemStore 刷写机制及 HFile 的合并流程。

### 参考答案

**MemStore 刷写（Flush）：**

当客户端写入数据时，数据首先存储在 **MemStore**（内存中的 sorted map），达到以下任一条件时触发刷写：

- MemStore 大小达到 `hbase.hregion.memstore.flush.size`（默认 **128MB**）
- 所有 MemStore 总大小超过 `hbase.regionserver.global.memstore.size`（默认 **堆内存的 40%**）
- WAL 日志大小超过 `hbase.regionserver.logroll.size`（默认 **HDFS block 大小 128MB**）
- 定期刷写线程（`hbase.regionserver.optionalllogsflushthreshold`，默认 **1小时**）

**HFile 合并（Compaction）：**

刷写生成 HFile 后，随着时间推移同一列族下会积累多个小 HFile，影响读取性能。Compaction 负责合并：

- **Minor Compaction（小合并）：** 选取多个较小的 HFile 合并为 1 个，减少文件数量，不删除过期数据
- **Major Compaction（大合并）：** 将一个 Region 下所有 HFile 合并为 1 个，**删除过期数据和删除标记**，默认在 `7` 天凌晨自动触发

**刷写与合并的配合：**
```
写入 → MemStore → Flush → HFile(n) → Minor Compaction → HFile(m) → Major Compaction → 大HFile
```

> ⚠️ **注意：** Major Compaction 会产生大量 IO，建议在业务低峰期手动触发或配置策略避开高峰期。

**来源：** [HBase 官方文档 - Compaction](https://hbase.apache.org/book.html#compaction)

## 题目四：HBase 的列簇（Column Family）设计有什么注意事项？

### 参考答案

列簇是 HBase 数据建模的核心，决定了数据的物理存储方式。

**基本规则：**
- 建表时必须指定列簇，创建后尽量**不要修改**
- 不同列簇数据存储在不同路径（`/hbase/data/{namespace}/{table}/{region}/{cf}`）
- 列簇下可以有动态的列（qualifier），无需预先定义

**设计原则：**

1. **控制列簇数量**：建议 **≤ 2~3 个**，每个列簇独立 MemStore，过多会导致内存压力
2. **冷热数据分离**：
   - 热点列簇（频繁读写）→ 单独列簇
   - 归档/历史数据（很少访问）→ 单独列簇，配合 TTL 和版本控制
3. **合理设置 `IN_MEMORY=true`**：对于频繁访问的小表（如配置表），设为内存列簇，可提升性能但占用 RegionServer 堆内存
4. **配合 TTL 和版本**：
   - `TTL`（生存时间）：自动清理过期数据，避免手动删除
   - `VERSIONS`：控制同一单元格保留的历史版本数

**示例：**
```bash
create 'user_behavior', {NAME => 'info', TTL => 2592000, VERSIONS => 1}, 
                         {NAME => 'action', TTL => 7776000, VERSIONS => 3}
# info 列簇：用户基本信息，TTL 30天，保留1个版本
# action 列簇：用户行为日志，TTL 90天，保留3个版本
```

**常见误区：**
- 列簇过多 → 每个 MemStore 占用独立内存，GC 压力大
- 列簇过少 → 无法针对冷热数据差异化存储，查询浪费 IO
- 在同一列簇中混合大字段和小字段 → 影响读取效率

**来源：** [HBase 官方文档 - Column Family](https://hbase.apache.org/book.html#cf)

## 题目五：HBase 协处理器（Coprocessor）是什么？有哪两类？各自适用场景？

### 参考答案

协处理器是 HBase 提供的**扩展机制**，允许用户将业务逻辑直接运行在 RegionServer 上，从而在数据存储层完成计算，减少网络传输。

**两类协处理器：**

### 1. Observer（观察者）
类似于数据库触发器，Hook 到指定生命周期事件中执行。

**分类：**
| 类型 | Hook 时机 |
|------|-----------|
| `PreGet` | 读取操作前，可以修改返回值或直接返回跳过读取 |
| `PostGet` | 读取操作后，可以补充列值等 |
| `PrePut` / `PostPut` | 写入前/后触发，可做数据校验、自动化补充字段 |
| `PreDelete` / `PostDelete` | 删除前/后触发 |
| `PreOpen` | Region 打开时（表级别 Observer） |
| `PostOpen` | Region 打开后 |

**适用场景：**
- 数据权限校验（写入前检查用户是否有权限）
- 字段自动填充（如写入时自动补充 `update_time`）
- 简单的行级过滤（减少无效数据返回）
- 审计日志（记录所有写操作）

### 2. Endpoint（端点）
类似于存储过程，需要自定义 RPC 调用，客户端调用后执行服务端逻辑并将结果返回。

**适用场景：**
- 聚合计算（`Sum`、`Count`、`Avg`）在服务端执行，避免大量数据传输
- 批量复杂计算
- 自定义扫描逻辑

**加载方式：**
- **静态加载**：通过 `hbase-site.xml` 配置，重启生效（适合 Observer）
- **动态加载**：通过 HBase Shell `alter` 命令或 API 动态加载/卸载（适合开发调试）

**注意事项：**
- 协处理器运行在 RegionServer 上，bug 可能导致节点崩溃，影响集群稳定性
- 尽量避免复杂、长耗时的计算
- 生产环境建议先在测试环境充分验证

**来源：** [HBase 官方文档 - Coprocessor](https://hbase.apache.org/book.html#cp)
