## 题目一：HBase 的整体架构是怎样的？各组件职责是什么？

### 参考答案

HBase 采用 **Master-Slave 架构**，核心组件包括以下四部分：

**1. HMaster（主节点）**
- 负责集群管理工作：表的创建、删除、修改
- 监控 RegionServer，承担 Region 的负载均衡与分配
- 当 RegionServer 宕机时，负责将失效 Region 迁移到其他健康节点
- 通过 ZooKeeper 选举机制保证高可用（可部署多个 HMaster）

**2. RegionServer（数据节点）**
- HBase 最核心模块，负责管理多个 Region
- 处理客户端对这些 Region 的读写请求
- 内部包含 MemStore（写缓存）和 BlockCache（读缓存）
- 负责在运行过程中对过大的 Region 进行分裂（Split）

**3. ZooKeeper（协调服务）**
- 选举机制保证集群中只有一个 Master 在运行
- 实时监控 RegionServer 的上下线状态并通知 HMaster
- 存储 HBase 的 schema 和所有 Region 的寻址入口

**4. HDFS（底层存储）**
- 持久化存储 HFile（数据文件）和 WAL（预写日志）
- 默认 3 副本机制保障数据可靠性

**数据组织层次：**
```
Table → Region（按 RowKey 水平切分）→ Store（按列族划分）→ MemStore + HFile
```

## 题目二：HBase 的 RowKey 设计原则有哪些？如何避免热点现象？

### 参考答案

**RowKey 设计三大原则：**

**1. 唯一性原则**
- RowKey 是数据的唯一标识，必须保证其唯一性
- 相同的 RowKey 多次写入会被视为更新操作，旧数据可能被覆盖

**2. 长度原则**
- 建议控制在 **10~100 字节**，最大不超过 16KB
- 过长的 RowKey 会快速占满 MemStore 的 128MB 空间，减少实际数据存储量
- 也会导致 HFile 的 Bloom Filter 和 Block Index 膨胀，降低索引效率

**3. 散列原则**
- 避免使用时间戳等连续递增的值作为 RowKey，否则会导致所有数据集中写入同一个 Region（热点现象）

**避免热点的常见方法：**

| 方法 | 做法 | 优缺点 |
|------|------|--------|
| **加盐（Salting）** | 在 RowKey 头部添加随机前缀（如 `hash(userid) % 100 + original_key`） | ✅ 负载分散；❌ 无法范围查询 |
| **哈希（Hashing）** | 使用 MD5 等确定性哈希打散数据 | ✅ 可预测读；❌ 失去有序性 |
| **反转（Reversing）** | 反转固定位数或数字格式的 RowKey | ✅ 简单有效；❌ 完全无序 |
| **预分区** | 建表时通过 `SPLITS` 指定分割点，预创建多个空 Region | ✅ 从源头避免热点 |


**示例：** 用户行为表可设计为 `[userId反转][Long.Max_Value - timestamp]`，既打散负载，又支持按用户和时间范围查询。

## 题目三：描述 HBase 的写数据流程，以及 MemStore 刷写机制。

### 参考答案

**HBase 写入流程（6步）：**

1. **定位 Region**：客户端通过 ZooKeeper 获取 `-ROOT-` 表位置，再访问 `.META.` 表确定目标数据所在的 RegionServer
2. **写入 WAL**：先将数据写入 WAL（预写日志），防止宕机数据丢失
3. **写入 MemStore**：数据进入 MemStore 并按 RowKey 排序
4. **返回响应**：WAL 和 MemStore 都写入成功后向客户端确认
5. **触发刷写**：当 MemStore 大小达到阈值（默认 128MB），会异步刷写到 HDFS 生成 HFile
6. **Compaction**：HFile 越来越多时触发合并（Minor/Major Compaction），优化读取性能

**MemStore 刷写机制（flush 三阶段）：**

| 阶段 | 说明 |
|------|------|
| **prepare** | 添加 UpdateLock 锁，阻塞写请求；遍历 MemStore 生成快照 Snapshot；新建 CellSkipListSet 保存后续写入 |
| **flush** | 将 prepare 阶段的 Snapshot 持久化到 `.tmp` 临时目录，生成新的 HFile 并加入缓存 |
| **commit** | 将临时文件移动到对应 ColumnFamily 目录；更新 storefiles 列表；清空 prepare 阶段的 Snapshot |

**LSM 树核心优势：** 将随机写入转换为顺序写入（先写内存，再批量刷盘），适合写多读少的高吞吐场景。

## 题目四：HBase 读取数据时，数据查找的顺序是什么？为什么先查 BlockCache 再查 MemStore？

### 参考答案

**HBase 读取顺序（从上到下，命中即返回）：**

```
BlockCache（读缓存LRU）→ MemStore（写缓存）→ HFile（磁盘）
```

**各层详解：**

**1. BlockCache**
- LRU（最近最少使用）策略的读缓存
- 缓存频繁访问的 HFile 数据块
- 读取时首先在 BlockCache 查找，未命中才向下查找

**2. MemStore**
- 写缓存，存储已写入但尚未刷写到磁盘的数据
- 数据有序，按 RowKey 排序
- 如果在 BlockCache 中未找到，会直接在 MemStore 中查找

**3. HFile**
- 最终存储文件，HBase 在 HDFS 上的实际存储格式
- 包含多层索引（Bloom Filter、Block Index）加速查找
- 如果以上两层都未命中，才读取 HFile

**为什么要先查 BlockCache？**
- BlockCache 命中时直接返回内存数据，**延迟最低**
- 符合局部性原理：热点数据被缓存后可以重复快速访问
- 减少磁盘 I/O 操作，提升整体读取性能

**数据合并：** 读取时会将 BlockCache、MemStore、HFile 中的结果合并后返回给客户端，确保多版本数据一致性。

## 题目五：HBase 协处理器（Coprocessor）是什么？有哪些类型？用于解决什么问题？

### 参考答案

**协处理器概述：**
HBase 原生不支持二级索引和复杂查询，协处理器（Coprocessor）提供了一种扩展机制，允许用户自定义逻辑在 RegionServer 端执行，从而增强 HBase 的数据处理能力。

**两大类型：**

| 类型 | 触发方式 | 典型场景 |
|------|---------|---------|
| **Observer** | 类似于数据库触发器，在特定事件前后自动执行 | 数据写入时同步更新索引表、审计日志、权限控制 |
| **Endpoint** | 类似于存储过程，需客户端主动调用 | 自定义聚合计算、跨 Region 数据统计 |

**Observer 三种子类型：**
- `PreWrite`：写入 WAL 前执行
- `PostWrite`：数据写入 MemStore 后执行
- `PostGet`：数据读取后执行

**典型应用场景：**

**1. 二级索引实现**
- 在数据写入时通过 Observer 同步更新一张独立的索引表
- 索引表 RowKey 为索引字段值，Value 指向原始数据 RowKey
- 查询时先定位索引，再通过指针获取目标数据

**2. 聚合计算**
- 通过 Endpoint 在 RegionServer 端并行执行 MapReduce 式聚合
- 避免大量数据传输到客户端再聚合，减少网络开销

**3. 权限控制**
- 通过 Observer 在读取前检查用户权限，过滤无权限数据

**注意事项：**
- 协处理器会增加写入延迟（尤其是 Observer）
- 索引表本身可能成为瓶颈，需做好容量规划
- 也可考虑使用 **Phoenix**（HBase SQL 层）内置的全局/本地索引
