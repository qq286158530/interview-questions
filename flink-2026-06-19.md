## 题目一：Flink 的架构是怎样的？JobManager 和 TaskManager 的职责是什么？

### 参考答案

Flink 采用 **Master-Slave 架构**，由以下组件组成：

| 组件 | 职责 |
|------|------|
| **JobManager** | 整个集群的协调者，负责调度任务、协调 Checkpoint、协调故障恢复等 |
| **TaskManager** | 实际执行 Operator 的工作节点，每个 TM 包含多个 Task Slot |

### 核心流程

1. **客户端（Client）** 将用户编写的 Flink 作业（StreamGraph → JobGraph）提交给 JobManager。
2. **JobManager** 将 JobGraph 调度到 TaskManager 上执行。
3. **TaskManager** 将心跳和状态汇报给 JobManager。

### TaskManager 的 Slot

- Slot 是 TM 上的资源单位，默认只隔离内存，CPU 共享。
- Slot 数量 = `parallelism`（并行度）上限。
- Slot 可以复用，同一个 TM 上的多个 Slot 可以共享网络和内存资源。

## 题目二：什么是 Watermark？Flink 如何处理乱序数据？

### 参考答案

**Watermark（水位线）** 是 Flink 中用于处理**事件时间（Event Time）**乱序问题的核心机制。

### 本质

Watermark 本质上是一个**时间戳**，表示"**当 watermark 到达 t 时刻，所有事件时间 < t 的数据都已经到达或不会再被处理**"。

### 核心公式

```
Watermark = 当前最大事件时间 - 最大允许延迟（乱序时间）
```

### 工作机制

1. **乱序问题**：在分布式环境下，由于网络延迟等原因，数据可能不按事件时间顺序到达。
2. **窗口触发**：Flink 窗口会在 **watermark 时间 >= 窗口结束时间** 时触发计算，并清理窗口。
3. **allowedLateness**：即使窗口触发后，仍会保留窗口状态一段时间（`allowedLateness`），接收晚到的数据。
4. **lateData 处理**：超过 `allowedLateness` 的数据默认被丢弃，也可以用 `sideOutputLateData` 收集。

### 示例

```java
WatermarkStrategy
    .<Tuple2<String, Long>>forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner((event, timestamp) -> event.f1);
```

上述代码设置最大乱序时间为 10 秒，watermark 推进速度由已处理数据的最大事件时间驱动。

## 题目三：Flink Checkpoint 的原理是什么？如何实现 EXACTLY_ONCE 语义？

### 参考答案

### Checkpoint 机制

Flink 通过 **Chandy-Lamport 分布式快照算法** 的变体实现 Checkpoint，保证**至少一次（AT_LEAST_ONCE）**和**恰好一次（EXACTLY_ONCE）**语义。

### 核心原理

1. **Barrier（屏障）**：JobManager 周期性地向数据流中注入 `CheckpointBarrier`，将数据流分为快照之前和之后的部分。
2. **对齐（Alignment）**：在 EXACTLY_ONCE 模式下，算子需要等待所有输入 Channel 的 Barrier 到达后才进行快照，避免数据重复或丢失。
3. **状态存储**：算子的状态（Keyed State、Operator State）被异步快照到持久化存储（HDFS、S3 等）。
4. **Barrier 对齐**：多输入算子（如 union、connect）在对齐期间会缓存数据，防止数据乱序。

### EXACTLY_ONCE 的实现

```
Source（记录消费位置）  ──→  Operator（状态快照）  ──→  Sink（输出提交）
         ↑                        ↑                        ↑
     Checkpoint 存储         Checkpoint 存储           两阶段提交
```

Flink 的 Sink（如 Kafka Sink）通过 **两阶段提交协议（2PC）** 确保输出仅提交一次：

1. **预提交**：Checkpoint 完成时，Sink 进入预提交状态。
2. **提交**：收到 Checkpoint 完成通知后，正式提交。

### 配置示例

```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
env.enableCheckpointing(60000);  // 60秒一次
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30000);
env.getCheckpointConfig().setCheckpointTimeout(120000);
```

## 题目四：Flink 窗口函数有哪些？如何选择合适的窗口？

### 参考答案

Flink 支持多种窗口类型，适用于不同的业务场景。

### 窗口分类

#### 1. 按时间划分

| 窗口类型 | 说明 | 代码 |
|----------|------|------|
| **滚动窗口（Tumbling）** | 大小固定，不重叠 | `TumblingEventTimeWindows.of(Time.seconds(10))` |
| **滑动窗口（Sliding）** | 可重叠，有大小和滑动步长 | `SlidingEventTimeWindows.of(Time.seconds(10), Time.seconds(5))` |
| **会话窗口（Session）** | 按活跃间隙动态分组 | `SessionWindows.withGap(Time.seconds(10))` |
| **全局窗口（Global）** | 所有数据进入同一窗口，需自定义触发器 | `GlobalWindows.create()` |

#### 2. 按数据结构划分

- **Keyed Windows**：按 Key 分组后对每个 Key 独立开窗，支持并行。
- **Non-Keyed Windows**：整个算子作为一个窗口，不支持并行（已逐步废弃）。

### 窗口函数（Process Functions）

| 函数类型 | 说明 |
|----------|------|
| **ReduceFunction** | 增量聚合，效率高，适用于可归约的场景 |
| **AggregateFunction** | 通用增量聚合，支持多字段输出 |
| **ProcessWindowFunction** | 全量窗口函数，获取窗口元数据（窗口开始/结束时间、当前 Key 等） |
| **WindowFunction** | 旧版全量窗口函数 |

### 最佳实践

```java
// 最佳实践：组合使用增量聚合 + 全量 Process
stream
    .keyBy(...)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .aggregate(new AvgAggregate(), new ProcessWindowFunction())
```

## 题目五：Flink 状态管理机制是怎样的？状态后端如何选择？

### 参考答案

### Flink 状态分类

| 状态类型 | 说明 | 作用域 |
|----------|------|--------|
| **Keyed State** | 按 Key 分组，只能在 KeyedStream 上使用 | 当前 Key |
| **Operator State** | 算子级别，所有 Key 共享 | 算子实例 |
| **Broadcast State** | 广播状态，跨算子共享 | 所有算子实例 |

### Keyed State 细分

- `ValueState<T>`：单值状态
- `ListState<T>`：列表状态
- `MapState<K, V>`：映射状态
- `ReducingState<T>`：聚合状态
- `AggregatingState<I, O>`：聚合状态（更通用）

### 状态后端（State Backend）

Flink 提供三种状态后端：

| 后端 | 存储位置 | 适用场景 |
|------|----------|----------|
| **HashMapStateBackend** | JVM Heap | 小状态、低延迟；生产环境推荐 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘） | 大状态、海量数据；支持增量 Checkpoint |
| **EmbeddedRocksDBStateBackend（增量）** | RocksDB + 远程存储 | 超大规模状态 |

### 状态过期（TTL）

```java
StateTtlConfig ttlConfig = StateTtlConfig
    .newBuilder(Time.days(7))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)
    .build();

valueState Descriptor withTtl = new ValueStateDescriptor<>("myState", String.class);
myState Descriptor.enableTimeToLive(ttlConfig);
```
