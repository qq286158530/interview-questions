## 题目31：【Java并发】CompletableFuture 相比传统 Future 有哪些优势？请结合实际场景说明如何使用它实现并行计算？

【答案】
CompletableFuture 是 Java 8 引入的异步编程利器，相比传统 Future 有以下核心优势：

**1. 链式调用（thenApply、thenCompose、thenCombine）**
传统 Future 只能通过 get() 阻塞等待结果，CompletableFuture 支持流式API，可以优雅地串联多个异步任务：
```java
CompletableFuture.supplyAsync(() -> 查询用户信息())
    .thenApply(user -> validateUser(user))      // 异步转换
    .thenCombine(CompletableFuture.supplyAsync(() -> 查询订单信息()),
                 (user, orders) -> buildVO(user, orders))  // 并行等待
    .thenAccept(vo -> System.out.println(vo));  // 最终消费
```

**2. 异常处理（exceptionally、handle、whenComplete）**
```java
CompletableFuture.supplyAsync(() -> callRemoteService())
    .exceptionally(ex -> {
        log.error("远程调用失败", ex);
        return fallbackResult();  // 降级处理
    })
    .thenAccept(result -> process(result));
```

**3. 并行组合（allOf、anyOf）**
```java
// 并行调用多个服务，等待全部完成
List<CompletableFuture<Result>> futures = services.stream()
    .map(s -> CompletableFuture.supplyAsync(() -> s.call()))
    .collect(Collectors.toList());

CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]))
    .thenRun(() -> {
        List<Result> results = futures.stream()
            .map(CompletableFuture::join)
            .collect(Collectors.toList());
        mergeResults(results);
    });
```

**4. 实际场景**
- 电商详情页：并行查询商品信息、库存、评价、推荐（4个接口从10s+降到≈max(每个接口耗时)）
- 数据汇总：同时从多个数据源拉取数据，汇总后返回

## 题目32：【JVM】什么是 G1 垃圾收集器？它的工作原理是什么？为什么 G1 能实现可预测的停顿时间？

【答案】
G1（Garbage-First）是从 JDK 9 开始的默认垃圾收集器，它的设计目标是替代 CMS，在提供高吞吐的同时实现可预测的停顿时间。

**核心原理：Region + SATB**

**1. Region 内存布局**
G1 把堆内存划分为多个大小相等的 Region（1MB~32MB），每个 Region 独立管理。采用「标记-整理」算法，不会产生内存碎片。

**2. Young GC 和 Mixed GC**
- Young GC：回收所有年轻代 Region（Eden + Survivor）
- Mixed GC：回收所有年轻代 + 若干个老年代 Region（优先回收垃圾比例最高的 Region，这就是 "Garbage-First" 的含义）

**3. 可预测停顿时间（-XX:MaxGCPauseMillis）**
G1 维护一个停顿预测模型，会优先回收垃圾比例高（回收收益大）的 Region。用户设置 `MaxGCPauseMillis=200ms`，G1 会尽量保证每次 GC 停顿不超过 200ms。

**4. 三色标记 + SATB（Snapshot-At-The-Beginning）**
- 白色：未标记的对象（待回收）
- 灰色：自身已标记但成员未标记的对象
- 黑色：自身和成员都已标记（存活对象）

SATB 保证在并发标记开始时刻的快照一致性，通过「前值快照」解决漏标问题。

**G1 vs CMS 对比**
| 特性 | G1 | CMS |
|------|-----|-----|
| 算法 | 标记-整理 | 标记-清除 |
| 内存碎片 | 无 | 有 |
| 停顿时间 | 可预测 | 不可预测 |
| 并发阶段 | 并发标记、并发清理 | 并发标记、并发清理 |
| 浮动垃圾 | 下一轮回收 | 额外浮动垃圾 |
| 适用场景 | JDK 9+默认，推荐大堆 | 小堆、低停顿 |

## 题目33：【Spring】@Transactional 的失效场景有哪些？如何解决？

【答案】
@Transactional 在以下场景会失效，面试中一定要能说清楚：

**1. 非 public 方法**
Spring AOP 代理是基于接口的 JDK 动态代理或基于子类的 CGLIB 代理，public 方法才能被拦截。
```java
// 失效！非 public 方法
@Transactional
private boolean update() { ... }

// 解决：改为 public
public boolean update() { ... }
```

**2. 自调用（同类内部调用）**
```java
@Service
public class UserService {
    public void methodA() {
        this.methodB();  // 自调用，不经过代理！
    }
    
    @Transactional
    public void methodB() {  // 不会生效
        // 更新操作
    }
}
// 解决：注入自身或使用 AopContext.currentProxy()
```

**3. 异常被 catch 吞掉**
```java
@Transactional
public void update() {
    try {
        // 操作
    } catch (Exception e) {
        // 异常被吞掉，事务不会回滚！
    }
}
// 解决：重新抛出异常，或配置 rollbackFor
@Transactional(rollbackFor = Exception.class)
```

**4. 多数据源问题**
如果没有配置正确的事务管理器，或在 `@Transactional` 方法中切换了数据源，事务会失效。

**5. propagation 配置不当**
```java
@Transactional(propagation = Propagation.SUPPORTS)
public void method() {
    // 如果当前无事务，则以非事务方式执行
}
```

**6. 多线程环境**
```java
@Transactional
public void asyncUpdate() {
    new Thread(() -> {
        // 新线程独立事务，原事务不覆盖！
        // 这个线程内的操作不在事务中
    }).start();
}
```

## 题目34：【消息队列】Kafka 如何保证消息的 Exactly-Once 语义？Kafka Streams 是如何实现端到端 Exactly-Once 的？

【答案】
Kafka 提供了 Exactly-Once 语义，是三种消息语义中最强的：

**三种消息语义对比**
| 语义 | 含义 | 可能丢失？ | 可能重复？ |
|------|------|----------|----------|
| At-Least-Once | 至少一次 | ❌ 不丢 | ✅ 可能重复 |
| At-Most-Once | 至多一次 | ✅ 不丢 | ❌ 不重复 |
| Exactly-Once | 恰好一次 | ❌ 不丢 | ❌ 不重复 |

**Kafka 事务机制（端到端 Exactly-Once）**

**1. 幂等 Producer（单分区 Exactly-Once）**
```java
properties.put("enable.idempotence", true);
```
开启后，Producer 会为每条消息分配唯一 PID（Producer ID）+ Sequence Number，Broker 端会缓存最近5条消息，超出则拒绝重复，实现单分区 Exactly-Once。

**2. 事务 Producer（跨分区 Exactly-Once）**
```java
producer.initTransactions();
producer.beginTransaction();
producer.send("topic1", "msg1");
producer.send("topic2", "msg2");
producer.commitTransaction();  // 原子提交
```
通过两阶段提交协议，保证多个 Topic-Partition 间的写入要么全部成功，要么全部失败。

**3. Kafka Streams 的 Exactly-Once**
```
Kafka Streams 端到端 Exactly-Once = Producer 幂等性 + Consumer 事务 + 处理语义

工作流程：
1. Consumer 使用事务读取（read_committed 模式）
2. Processor 处理逻辑与输出写入同一事务
3. 输出写入 Kafka Topic 或外部系统（通过事务保证）
```

## 题目35：【Spring Boot】如何自定义 Spring Boot Starter？为什么 Spring Boot 能自动配置？

【答案】
**自动配置原理（@EnableAutoConfiguration）**

Spring Boot 的自动配置核心是 `spring.factories`（旧）或 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`（新）文件。

**核心流程：**
1. `@SpringBootApplication` 包含 `@EnableAutoConfiguration`
2. Spring Boot 通过 `AutoConfigurationImportSelector` 加载 `META-INF/spring.factories`
3. 根据 `@ConditionalOnClass`、`@ConditionalOnProperty` 等条件注解判断是否生效
4. 如 `XxxAutoConfiguration` 类的条件都满足，则自动注册对应的 Bean

**自定义 Starter 示例：**

**1. 创建 Starter 模块**
```
my-starter/
├── src/main/java/
│   └── com/example/autoconfigure/
│       ├── MyAutoConfiguration.java
│       └── MyProperties.java
└── src/main/resources/
    └── META-INF/
        └── spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
```

**2. 配置属性类**
```java
@ConfigurationProperties(prefix = "my")
public class MyProperties {
    private String name = "default";
    private int timeout = 3000;
    // getters/setters
}
```

**3. 自动配置类**
```java
@Configuration
@ConditionalOnClass(MyService.class)       // 当 MyService 在 classpath 时生效
@ConditionalOnProperty(prefix = "my", name = "enabled", havingValue = "true", matchIfMissing = false)
@EnableConfigurationProperties(MyProperties.class)
public class MyAutoConfiguration {
    
    @Autowired
    private MyProperties props;
    
    @Bean
    @ConditionalOnMissingBean
    public MyService myService() {
        return new MyService(props.getName(), props.getTimeout());
    }
}
```

**4. 注册自动配置**
```
# META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
com.example.autoconfigure.MyAutoConfiguration
```

**5. 使用 Starter**
```yaml
my:
  enabled: true
  name: "custom"
  timeout: 5000
```
用户只需引入 Starter 依赖，配置属性即可使用。

📖 来源：Spring Boot 官方文档 + 《Spring Boot实战》

## 题目36：【Redis】Redis 的持久化机制有哪几种？RDB 和 AOF 有什么区别？如何选择？

【答案】
Redis 提供了两种持久化机制，面试中经常被问到它们的区别和选择策略。

**RDB（Redis Database）—— 快照持久化**

**原理：** 在指定时间点生成内存数据的全量快照，保存为 `dump.rdb` 文件。
```bash
# 手动触发
BGSAVE  # 后台异步保存
SAVE    # 同步保存（阻塞）

# 自动触发（配置文件）
save 900 1      # 900秒内至少1次写操作
save 300 10     # 300秒内至少10次写操作
save 60 10000   # 60秒内至少10000次写操作
```

**特点：**
- 恢复快（直接加载 RDB 文件）
- fork() 子进程执行，对主进程影响小
- 可能丢失最近一次快照后的数据

**AOF（Append Only File）—— 命令持久化**

**原理：** 每执行一条写命令就追加到 AOF 文件。
```bash
# 配置
appendonly yes
appendfsync everysec  # 每秒同步（推荐，默认）
# appendfsync always   # 每次写都同步（慢但最安全）
# appendfsync no       # 由OS决定何时同步（最快）
```

**特点：**
- 数据安全性更高（everysec 模式最多丢1秒数据）
- AOF 文件会随时间膨胀，需要 Rewrite 压缩
```bash
BGREWRITEAOF  # 异步重写 AOF
```

**RDB vs AOF 对比**
| 特性 | RDB | AOF |
|------|------|-----|
| 持久化方式 | 快照 | 追加命令 |
| 文件大小 | 小（紧凑） | 大（膨胀） |
| 恢复速度 | 快 | 慢 |
| 数据完整性 | 可能丢数据 | 丢数据少 |
| 对性能影响 | fork 子进程，低 | 每秒同步，略高 |

**生产环境推荐策略：**
- **小缓存 / 能容忍数据丢失**：RDB
- **重要数据 / 不能丢**：AOF everysec + RDB 混合持久化（Redis 4.0+ 支持）
```yaml
aof-use-rdb-preamble yes  # Redis 4.0+，AOF 重写时用 RDB 格式
```

## 题目37：【算法】一致性哈希（Consistent Hashing）的原理是什么？它解决了什么问题？

【答案】
一致性哈希是分布式系统中非常重要的算法，常见于 Redis Cluster、Kafka、Cassandra 等组件。

**传统哈希的问题**
```java
// 传统哈希取模
node = hash(key) % N  // N 为节点数

// 当节点数变化时（扩容/缩容）
// hash(key) % 3 vs hash(key) % 4 结果完全不同
// 导致几乎所有缓存失效（数据重新分布）—— 雪崩效应！
```

**一致性哈希原理**

**1. 哈希环**
将哈希值空间组织成一个环（如 0 ~ 2^32-1），每个节点映射到环上的一个位置。
```
        hash(A) = 100
            A
            |
            v
   0 ------------------------------------------------ 2^32
            ^
            |
            C  hash(C) = 3000000000
```
数据 key 顺时针找到第一个节点。

**2. 虚拟节点（Virtual Nodes）**
单个节点在环上只有一个位置会导致负载不均。引入虚拟节点：
```
实际节点 A -> 虚拟节点 A1, A2, A3...
实际节点 B -> 虚拟节点 B1, B2, B3...
```
每个虚拟节点在环上有不同位置，数据分布更均匀。

**3. 解决了什么问题**
- **节点增减时影响范围小**：只有相邻节点间的数据需要迁移
- **支持节点横向扩展**：适合缓存集群、负载均衡器等场景

**Java 实现示例**
```java
public class ConsistentHash<T> {
    private final TreeMap<Long, T> circle = new TreeMap<>();
    
    public void add(T node, int virtualNodes) {
        for (int i = 0; i < virtualNodes; i++) {
            long hash = hash("" + node + "_" + i);
            circle.put(hash, node);
        }
    }
    
    public T get(String key) {
        if (circle.isEmpty()) return null;
        long hash = hash(key);
        // 顺时针找第一个大于等于 hash 的节点
        Map.Entry<Long, T> entry = circle.ceilingEntry(hash);
        return entry != null ? entry.getValue() : circle.firstEntry().getValue();
    }
    
    private long hash(String key) {
        return Math.abs(key.hashCode());
    }
}
```

## 题目38：【Spring Cloud】Seata 的 AT 模式和 TCC 模式有什么区别？各自适用什么场景？

【答案】
Seata 是分布式事务解决方案，提供了 AT、TCC、Saga 三种模式。

**AT 模式（Automatic Transaction）**

**原理：** 基于两阶段提交，Seata 框架自动处理，对业务零侵入。
```
一阶段：
  解析 SQL -> 查询前镜像 -> 执行 SQL -> 查询后镜像 -> 生成 undo_log -> 记录回滚日志
  
二阶段（Commit）：
  异步删除 undo_log（业务表操作已生效）
  
二阶段（Rollback）：
  根据 undo_log 自动生成反向 SQL 并执行
```

**优点：** 对业务代码零侵入，业务层无感知
**缺点：** 需要解析 SQL，依赖数据库 undo_log，性能略低

**TCC 模式（Try-Confirm-Cancel）**

**原理：** 业务层手动实现三个接口，框架只负责调度。
```java
@LocalTCC
public interface OrderTccService {
    
    @TwoPhaseBusinessAction(
        name = "createOrder",
        commitMethod = "confirm",
        rollbackMethod = "cancel"
    )
    void tryCreateOrder(BusinessActionContext context, @BusinessActionContextParameter("orderId") String orderId);
    
    void confirm(BusinessActionContext context);
    
    void cancel(BusinessActionContext context);
}
```

**优点：** 性能高，不依赖数据库 undo_log
**缺点：** 业务侵入大，需要手动实现 Try/Confirm/Cancel 三个接口

**AT vs TCC 对比**
| 特性 | AT 模式 | TCC 模式 |
|------|--------|---------|
| 侵入性 | 零侵入 | 侵入业务代码 |
| 性能 | 中等（undo_log） | 高 |
| 适用场景 | 业务逻辑简单、无强一致性要求 | 对性能要求高、需要强一致性 |
| 隔离级别 | 行级（基于 undo_log） | 业务自定义 |

📖 来源：Seata 官方文档

## 题目39：【MySQL】什么是覆盖索引？如何利用覆盖索引优化查询？为什么覆盖索引能显著提升查询性能？

【答案】
覆盖索引是性能优化利器，面试中经常出现。

**定义：** 如果一个索引包含了查询所需的所有列（SELECT、WHERE、JOIN 等涉及的列），则称该索引为覆盖索引。

**示例**
```sql
-- 假设有联合索引 (name, age, email)
SELECT name, age, email FROM users WHERE name = '张三';
-- 这个查询可以利用覆盖索引，因为 name, age, email 都在索引中
```

**为什么覆盖索引快？**

**普通索引查询过程（回表）：**
1. 在 (name, age) 索引树找到 name='张三' 的记录 → 得到主键 ID
2. **回表**：根据主键 ID 去主键索引树查找完整行数据
3. 如果 SELECT 的列不在索引中，需要额外回表

**覆盖索引优化（无需回表）：**
1. 在 (name, age, email) 索引树中直接找到 name='张三'
2. **无需回表**：所有需要的列都在索引中，直接返回

**性能提升：**
- 减少磁盘 I/O（减少1次主键索引查找）
- 减少内存访问
- 对于 count(*)、limit 等只需索引的查询特别有效

**实战优化示例**
```sql
-- 为高频查询建立覆盖索引
CREATE INDEX idx_covering ON orders(user_id, status, create_time);

-- 优化以下查询：
SELECT create_time, status FROM orders WHERE user_id = 1 ORDER BY create_time DESC LIMIT 10;
-- 索引覆盖了 WHERE、SELECT、ORDER BY 全部列
```

**注意事项：**
- 不要 SELECT *，尽量只查需要的列
- 索引不是越多越好，维护成本高
- 利用 EXPLAIN 的 `Using index` 判断是否命中覆盖索引

## 题目40：【Java基础】什么是 Java 17 的 Sealed Classes？它解决了什么问题？有什么实际应用场景？

【答案】
Sealed Classes（密封类）是 Java 17 引入的重要特性，用于限制类的继承层次。

**解决的问题**
在 Java 17 之前，无法精确控制类的继承范围，导致：
- 类的子类数量不可控
- switch 表达式无法穷举所有子类
- 安全风险（外部可以随意扩展）

**基本语法**
```java
// 密封类：只允许指定的子类继承
public sealed class Shape permits Circle, Rectangle, Triangle {
    // ...
}

// 最终子类（不允许再被继承）
public final class Circle extends Shape { }

// 递归密封（Square 只能是 Rectangle 的子类）
public sealed class Rectangle extends Shape permits Square {
    public static sealed class Square extends Rectangle {
        public final class Fine {}
    }
}
```

**核心优势 1：穷举 switch**
```java
// Java 17 之前：只能用 if-else + instanceof
// Java 17 之后：switch 可以穷举所有密封类的子类
public String getArea(Shape shape) {
    return switch (shape) {
        case Circle c -> "面积: " + Math.PI * c.radius() * c.radius();
        case Rectangle r -> "面积: " + r.width() * r.height();
        case Triangle t -> "面积: " + 0.5 * t.base() * t.height();
        // 编译器保证没有其他子类，添加新子类时不修改此处会编译报错
    };
}
```

**实际应用场景**

**1. 领域驱动设计（DDD）**
```java
public sealed interface Event permits UserCreatedEvent, UserUpdatedEvent, UserDeletedEvent {}

public sealed interface Command permits CreateUserCommand, UpdateUserCommand, DeleteUserCommand {}
```

**2. 策略模式 + 穷举**
```java
public sealed abstract class PaymentStrategy permits AlipayStrategy, WechatPayStrategy, BankCardStrategy {
    abstract void pay(double amount);
}
```

**3. JSON 序列化/反序列化**
Sealed class + record 是完美的 JSON 建模组合：
```java
public sealed record Response<T> permits SuccessResponse, ErrorResponse {
    public record SuccessResponse<T>(T data, int code) extends Response<T> {}
    public record ErrorResponse(String message, int code) extends Response<Null> {}
}
```

## 题目41：【消息队列】Kafka 的分区（Partition）和消费者组（Consumer Group）是如何协同工作的？如何设计分区数才能最大化吞吐量？

【答案】
**分区与消费者组的关系**

```
Producer
  |
  v
Partition 0  -->  Consumer Group A (Consumer 1) --> 处理消息
Partition 1  -->  Consumer Group A (Consumer 2) --> 处理消息  (不同 Consumer Group 都能消费)
Partition 2  -->  Consumer Group A (Consumer 3) --> 处理消息

注意：同一个 Consumer Group 内的 Consumer 不能消费同一个 Partition！
     不同 Consumer Group 之间互不影响，每个都能消费全部消息。
```

**核心规则：**
- **一个 Partition 只能被同一个 Consumer Group 内的一个 Consumer 消费**
- 一个 Consumer 可以消费多个 Partition
- Consumer 数量 > Partition 数量时，多余的 Consumer 会空闲
- Consumer 数量 <= Partition 数量时，并行度 = Consumer 数量

**分区数设计原则**

**分区数不是越多越好！**

| 考量因素 | 说明 |
|---------|------|
| 并行消费 | 分区数决定了最大并发消费数 |
| 文件句柄 | 每个 Partition 打开 3 个文件（索引、时间索引、数据），占用资源 |
| 副本同步 | ISR 列表同步开销随分区数增加 |
| 选举时间 | Broker 挂掉后，Controller 需要为所有 Partition 选举新 Leader |

**经验公式：**
```
分区数 ≈ max(预期吞吐量/单Consumer吞吐量, 预期最大并发数)

例如：
  单Consumer吞吐量 = 10 MB/s
  预期吞吐量 = 100 MB/s
  → 分区数 >= 10

同时要考虑：
  Broker 数量：建议分区数 = Broker数 × (1~3)
  例如：3个Broker，建议分区数 = 3 × 2 = 6（每Broker 2个分区）
```

**最佳实践：**
- **初期预估**：按单 Consumer 10MB/s 估算，预留 2~3 倍余量
- **避免过多分区**：超过 1000 分区后元数据压力明显
- **监控分区膨胀**：Kafka 默认单条消息最大 1MB，分区过多影响恢复时间

📖 来源：Kafka 官方文档 + Confluent 最佳实践

## 题目42：【Spring】Spring Security 的认证流程是怎样的？什么是 OAuth2？它和 SSO 有什么区别？

【答案】
**Spring Security 认证流程**

```
用户请求
    |
    v
Security Filter Chain（过滤器链）
    |
    v
UsernamePasswordAuthenticationFilter
    |-- 提取用户名/密码
    |-- 封装成 UsernamePasswordAuthenticationToken
    |
    v
AuthenticationManager
    |-- 调用 AuthenticationProvider
    |       |-- 调用 UserDetailsService.loadUserByUsername()
    |       |-- 调用 PasswordEncoder.matches() 验证密码
    |
    v
Authentication（认证结果）
    |-- 成功：SecurityContextHolder 设置认证信息
    |-- 失败：抛出 AuthenticationException
```

**OAuth2 四种授权模式**

| 模式 | 适用场景 | 说明 |
|------|---------|------|
| Authorization Code | 有后端的应用 | 最安全，授权码交换 token |
| PKCE | 移动端/SPA | 无后端，纯前端 |
| Client Credentials | 服务间通信 | M2M |
| Password Grant | 极度信任的应用 | 不推荐 |

**OAuth2 vs SSO 对比**

| 维度 | OAuth2 | SSO (Single Sign-On) |
|------|--------|---------------------|
| 定位 | 授权（Authorization） | 认证（Authentication） |
| 授权范围 | 可细粒度（只读某资源） | 访问全部应用 |
| 典型协议 | OAuth2 + OIDC | SAML、OIDC |
| 登录次数 | 一次授权，多次访问 | 一次登录，访问所有应用 |

**Spring Security + OAuth2 核心概念**
```
Resource Owner（用户）
    |
    v
Authorization Server（认证服务器，如 Spring Authorization Server）
    |-- 验证用户身份
    |-- 颁发 Access Token
    |-- 范围（Scope）控制
    |
    v
Client Application（客户端应用）
    |-- 携带 Access Token 访问资源
    |
    v
Resource Server（资源服务器）
    |-- 验证 Token
    |-- 返回受保护资源
```

## 题目43：【AI Agent】什么是 LangChain？它的核心模块有哪些？为什么 LLM 应用需要 LangChain 这样的框架？

【答案】
LangChain 是一个用于构建 LLM 应用的框架，核心思想是将 LLM 与外部工具、数据源、记忆串联起来。

**为什么需要 LangChain？**

纯 LLM 的局限：
- 训练数据有截止日期，无法获取最新信息
- 无法调用外部工具（搜索、数据库、API）
- 无法访问私有知识库
- 无状态（每次对话是独立上下文）

**LangChain 核心模块**

**1. Model I/O（模型交互）**
```python
from langchain_openai import ChatOpenAI
from langchain.schema import HumanMessage

llm = ChatOpenAI(model="gpt-4")
response = llm.invoke([HumanMessage(content="解释量子计算")])
```

**2. Retrieval（检索增强）**
```python
from langchain.retrievers import WikipediaRetriever

retriever = WikipediaRetriever()
docs = retriever.get_relevant_documents("量子计算原理")
# 结合 LLM 生成答案（RAG 模式）
```

**3. Chains（链）**
```python
from langchain.chains import LLMChain
from langchain.prompts import PromptTemplate

prompt = PromptTemplate.from_template("用{language}实现{purpose}")
chain = LLMChain(llm=llm, prompt=prompt)
result = chain.invoke({"language": "Python", "purpose": "快速排序"})
```

**4. Agents（代理）**
```python
from langchain.agents import initialize_agent, Tool
from langchain.tools import DuckDuckGoSearch

tools = [Tool(name="search", func=DuckDuckGoSearch().run, description="搜索")]
agent = initialize_agent(tools, llm, agent="zero-shot-react-description")
agent.run("查一下今天天气怎么样？")
```

**5. Memory（记忆）**
```python
from langchain.memory import ConversationBufferMemory

memory = ConversationBufferMemory(memory_key="chat_history")
# 支持短期记忆、长期记忆、向量记忆
```

## 题目44：【算法】什么是 B+ 树？为什么 MySQL 的 InnoDB 存储引擎选择 B+ 树而不是 B 树作为索引结构？

【答案】
B+ 树是 MySQL InnoDB 引擎的索引结构基础，面试必考。

**B+ 树 vs B 树对比**

```
B树：每个节点都存数据
         [5, 10, 20]
        /   |    \
    [1,3] [6,8] [25,30]   <-- 所有节点都存数据

B+ 树：非叶子节点只存索引，叶子节点存全部数据且有序串联
              [15, 30]
            /    |     \
      [5,10]  [15,20]  [30,40]  <-- 非叶子节点只存索引
        |        |         |
       ...      ...       ...    <-- 叶子节点通过指针串联
```

**MySQL 选择 B+ 树的 6 大原因**

**1. 叶子节点链表 — 范围查询极快**
```sql
SELECT * FROM users WHERE age BETWEEN 20 AND 30;
-- B+ 树叶子节点双向链表，范围查询只需顺序遍历
-- B 树需要中序遍历，效率低
```

**2. 非叶子节点只存索引 — 树更矮，IO 更少**
- InnoDB Page 大小 = 16KB
- 非叶子节点每条索引约 14 字节
- 每节点可存 16KB / 14B ≈ 1200 条索引
- 3 层 B+ 树可存：1200 × 1200 × 1200 ≈ 17亿 条索引
- B 树每节点存数据，空间大，树更高，IO 多

**3. 查询稳定**
B+ 树任何查询都需到叶子节点，复杂度固定为 O(log n)。
B 树可能提前在非叶子节点命中，复杂度不稳定。

**4. 更适合磁盘预读**
顺序遍历叶子节点链表时，只需顺序 IO，磁盘预读命中率高。

**5. 插入删除更稳定**
B+ 树插入删除只在叶子节点进行，非叶子节点只存索引，调整范围小。

**6. 全表扫描效率高**
从最左叶子节点顺序遍历即可扫描全表，无需中序遍历。

📖 来源：《高性能MySQL》+ InnoDB 官方文档

## 题目45：【分布式】什么是 Paxos 算法？它和 Raft 算法有什么区别？为什么 Raft 更适合工程实现？

【答案】
Paxos 和 Raft 都是分布式一致性算法，用于在存在节点故障的分布式系统中达成共识。

**Paxos 算法（Leslie Lamport, 1998）**

**两种角色：**
- **Proposer（提议者）：** 提出提案（Proposal）
- **Acceptor（接受者）：** 批准提案

**两个阶段：**
```
第一阶段（Prepare）：
  Proposer 提出编号 N 的提案，发送给多数 Acceptor
  Acceptor 承诺不再接受编号 < N 的提案

第二阶段（Accept）：
  Proposer 收到多数 Acceptor 承诺后
  发送 value 给 Acceptor 批准
  Acceptor 收到批准后，提案生效
```

Paxos 理论优美但极难实现，ZooKeeper 使用的是 Paxos 的变种（ZAB）。

**Raft 算法（2014）**

Raft 把复杂问题分解为三个子问题：
1. **Leader 选举**（心跳触发，term 递增）
2. **日志复制**（Leader 同步给 Follower，过半确认）
3. **安全**（成员变更处理）

**Leader 选举流程：**
```
节点启动 -> Follower
       |
收到 Leader 心跳（term 不变）-> 继续 Follower
       |
心跳超时（150~300ms 随机）-> 成为 Candidate -> 发起选举
       |
                    |
           收到多数票 -> 成为 Leader
           收到更高 term -> 降为 Follower
```

**Paxos vs Raft 对比**

| 维度 | Paxos | Raft |
|------|-------|------|
| 可理解性 | 极难 | 专为易理解设计 |
| 角色划分 | Proposer/Acceptor/Learner | Leader/Follower/Candidate |
| 日志连续性 | 无强关联，复杂 | Leader 日志连续，易推理 |
| 工程实现 | 基本只有 Chubby/ZAB | etcd, Consul, CockroachDB, TiKV |
| 成员变更 | 复杂 | 支持 joint consensus + 新增节点自动降级 |
| 文档 | 数学论文风格 | paper 标题 "In Search of an Understandable Consensus Algorithm" |

**为什么 Raft 更适合工程？**

1. **日志连续性**：Raft Leader 的日志一定是最新的，复制逻辑简单
2. **没有提案编号冲突**：Raft 用 term + 随机超时避免
3. **日志匹配特性**：如果两个节点日志某处 term 相同，则之前全部相同
4. **大量工业级实现**：etcd/Consul/CockroachDB 验证
