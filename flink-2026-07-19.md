## 题目一：Flink 的架构是怎样的？为什么 Flink 能做到低延迟高吞吐？

### 参考答案

Flink 采用**流式优先（Stream-first）架构**，核心是一个分布式流处理引擎，其架构可以分为以下几层：

**1. 编程 API 层**
- DataStream API（处理无界流）
- DataSet API（处理有界批处理，现已逐步废弃）
- Table API / SQL（统一批流语义）

**2. 运行时（Runtime）层**
- **JobManager**：负责调度 Task、协调 Checkpoint、处理提交的任务。
  - `Dispatcher`：接收用户提交的 Job，为每个 Job 启动一个新的 JobMaster
  - `ResourceManager`：管理 Flink 集群中的 Slot 资源，对接不同的资源管理器（YARN、Kubernetes、Standalone）
  - `JobMaster`：管理单个 JobGraph 的执行，负责 Task 调度、Checkpoint 协调
- **TaskManager**：负责执行具体的 Task，每个 TM 包含多个 Slot，Slot 是资源分配的基本单位

**3. 物理执行层**
- Task 以**流水线（Pipeline）**方式在 Slot 中执行，相邻 Stage 之间通过**流水线模式**而非纯批处理模式通信，实现低延迟
- Flink 支持**流式 shuffle**和**批量 shuffle**两种数据交换模式

**为什么 Flink 能做到低延迟高吞吐？**

| 特性 | 说明 |
|------|------|
| 流水线执行 | 上下游 Task 之间无需等所有数据处理完再传递，数据逐条流动，延迟极低 |
| 轻量级 Checkpoint | 基于 Chandy-Lamport 分布式快照，Checkpoint 开销可控，不阻塞数据处理 |
| 状态后端 | RocksDB 状态后端支持增量 Checkpoint，IO 开销小 |
| 背压机制 | Flink 通过 TCP 滑动窗口实现背压，上游自动降速，无需额外机制 |
| 资源弹性 | Slot 动态分配，Task 可在 TM 间迁移，资源利用率高 |

## 题目二：Flink 的 Watermark 机制是什么？如何处理乱序数据？

### 参考答案

**Watermark（水位线）是 Flink 用于处理**事件时间（Event Time）**下乱序数据**的核心机制。

### 基本概念

```
Watermark W(t) = "当前已知的，所有时间戳 <= t 的数据均已到达"
```

- Watermark 本质是一个**时间戳**，表示"在此之前的数据应该都已经到齐了"
- 引入 Watermark 是因为在分布式网络中，数据可能**乱序**到达

### 两种 Watermark 生成策略

**1. 周期性生成（Periodic Watermark）**
```java
WatermarkStrategy
    .forMonotonousTimestamps()  // 有序流，时间戳 = Watermark
    .withTimestampAssigner(...)
```

**2. 标点式生成（Punctuated Watermark）**
```java
WatermarkStrategy
    .forGenerator(new BoundedOutOfOrdernessGenerator<>())
```

### 窗口触发时机

- 窗口的**结束时间** `<= Watermark` 时，窗口被触发计算
- 迟到（Late）数据可以通过 `allowedLateness()` 设置等待时间，超出后进入 `sideOutputLateData()`

### 处理乱序示例

```java
WatermarkStrategy
    .<Tuple2<String, Long>>forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner((event, timestamp) -> event.f1);
```

这段代码允许最多 **10 秒**的乱序，Watermark 设为当前最大事件时间 - 10 秒。

### 关键点

- Watermark 推进意味着"不再可能有更早的数据"
- 如果乱序时间超过 `allowedLateness`，数据会被当作**迟到数据**丢弃或侧输出
- 多流 JOIN 时，以**最小的 Watermark** 推进（短板效应）

## 题目三：Flink Checkpoint 和 Savepoint 的区别是什么？Flink 如何保证 Exactly-Once 语义？

### 参考答案

### Checkpoint vs Savepoint

| 维度 | Checkpoint | Savepoint |
|------|-----------|-----------|
| **触发方式** | Flink 自动周期性触发（由 JobManager 调度） | 手动触发（用户通过 CLI 或 API） |
| **存储位置** | 由 `state.backend` 配置指定 | 由 `-s` 参数指定路径 |
| **用途** | 故障恢复，保障 HA | 计划内维护、版本迁移 |
| **Job 兼容性** | 对 Job 变化敏感（拓扑变化后可能失效） | 面向版本，支持 Job 升级和拓扑变化 |
| **状态格式** | Flink 内部格式（特定于 Checkpoint 版本） | 外部标准格式（兼容版本升级） |
| **删除机制** | 作业取消或超自动删除 | 手动管理，不自动删除 |

### Flink 的 Exactly-Once 是如何实现的？

Flink 的 Exactly-Once 语义依托于**两阶段提交（2PC）协议**，主要涉及以下组件：

**1. 分布式快照（Chandy-Lamport 算法变体）**
- Checkpoint Barrier 在数据流中**插入**，将数据流切分成快照
- 所有算子收到 Barrier 后做**状态对齐（Barrier Alignment）**，然后做快照

**2. 两阶段提交协议**
- **预提交阶段**：Barrier 到达所有下游算子后，所有算子执行**检查点保存**（将状态写入状态后端），然后进入预提交状态
- **提交阶段**：Source 算子确认 Checkpoint 完成，通知 JobManager，下游算子正式提交（完成两阶段提交）

**3. 端到端 Exactly-Once**
- 需要**数据源**和**数据Sink**支持事务写入
- 例如：
  - **Kafka Source**：可以精准消费偏移量
  - **Kafka Sink**：使用 `FlinkKafkaProducer` 的 `exactly_once_v2` 模式
  - **Hdfs Sink**：使用 `BucketingSink` 配合幂等写入

### 代码示例

```java
env.enableCheckpointing(5000); // 5秒做一次 Checkpoint
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(500);
env.getCheckpointConfig().setCheckpointTimeout(60000);
env.getCheckpointConfig().enableExternalizedCheckpoints(
    ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION);
```

## 题目四：Flink 窗口函数有哪些？滚动窗口、滑动窗口、会话窗口有什么区别？

### 参考答案

### Flink 窗口类型总览

```
┌─────────────────────────────────────────────────┐
│                    窗口类型                      │
├──────────────┬──────────────┬───────────────────┤
│  Time Window │ Count Window │ Custom Window      │
├──────────────┼──────────────┼───────────────────┤
│ 滚动 Tumbling│ 滑动 Sliding │ 会话 Session       │
│ 滑动 Sliding │ 会话 Session │ 全局 Global        │
│ 会话 Session │ 全局 Global  │                    │
└──────────────┴──────────────┴───────────────────┘
```

### 1. 滚动窗口（Tumbling Window）

- 窗口大小固定，**无重叠**，首尾相接
- 适用于"每分钟统计"、"每天统计"

```java
// 每5分钟一个窗口，不重叠
data.keyBy(k -> k.f0)
    .window(TumblingProcessingTimeWindows.of(Time.minutes(5)))
    .sum(1);
```

### 2. 滑动窗口（Sliding Window）

- 窗口大小固定，但**可以重叠**
- 包含两个参数：窗口大小 + 滑动步长
- 适用于"移动平均"等需要重叠数据的场景

```java
// 窗口大小10分钟，滑动步长5分钟
data.keyBy(k -> k.f0)
    .window(SlidingProcessingTimeWindows.of(Time.minutes(10), Time.minutes(5)))
    .sum(1);
```

### 3. 会话窗口（Session Window）

- 根据**活动间隙**动态创建窗口
- 上下两条数据之间的时间间隔超过-gap，窗口就关闭

```java
// 会话间隔为30秒
data.keyBy(k -> k.f0)
    .window(ProcessingTimeSessionWindows.withGap(Time.seconds(30)))
    .sum(1);
```

### 窗口函数（Process Function）

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| `reduce()` | 增量聚合，性能高，数据量小 | 简单聚合（求和、计数、最大最小） |
| `aggregate()` | 更灵活的增量聚合 | 支持多状态输出 |
| `apply()` / `process()` | 全量窗口数据一起处理 | 需要遍历所有数据、复杂逻辑 |

```java
// 使用 process 窗口函数（全量）
windowedstream.process(new ProcessWindowFunction<Tuple2<String,Long>, String, String, TimeWindow>() {
    @Override
    public void process(String key, Context ctx, Iterable<Tuple2<String,Long>> input, Collector<String> out) {
        long count = 0;
        for (Tuple2<String,Long> e : input) count++;
        out.collect("key=" + key + " count=" + count);
    }
});
```

### 迟到数据处理

```java
// 允许5秒的迟到时间
data.keyBy(k -> k.f0)
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .allowedLateness(Time.seconds(5))
    .sideOutputLateData(lateOutputTag)
    .sum(1);
```

## 题目五：Flink 的状态后端（State Backend）有哪些？各自适用什么场景？如何选择？

### 参考答案

### 三种状态后端

| 状态后端 | 配置值 | 存储介质 | 状态大小 | 访问速度 |
|---------|--------|---------|---------|---------|
| **HashMapStateBackend** | `hashmap` | JVM Heap（TM 内存） | 受 TM 内存限制 | 最快（纯内存） |
| **EmbeddedRocksDBStateBackend** | `rocksdb` | RocksDB（本地 RocksDB + 增量 Checkpoint 到 HDFS/S3） | 几乎无限（磁盘） | 较慢（需要序列化/反序列化） |
| **EmbeddedRocksDBStateBackend（之前叫 `rocksdb`）** | 1.13+ 统一为 `rocksdb` | 同上 | 同上 | 同上 |

### HashMapStateBackend

```java
env.setStateBackend(new HashMapStateBackend());
```

**特点：**
- 状态存储在 TaskManager 的 **JVM Heap** 中
- 访问速度极快，适合**状态较小**、对延迟要求极高的场景
- 缺点：状态太大会导致 **OOM**，且 Checkpoint 需要全量序列化到外部存储

**适用场景：**
- 状态数据量小（MB 级别）
- TTL 短，过期数据及时清理
- 需要极低延迟的实时计算

### EmbeddedRocksDBStateBackend

```java
env.setStateBackend(new EmbeddedRocksDBStateBackend());
// 开启增量 Checkpoint（1.13+ 默认开启）
env.getCheckpointConfig().enableIncrementalCheckpointing();
```

**特点：**
- 状态存储在 **RocksDB**（嵌入式 KV 数据库），数据在本地磁盘
- 支持**增量 Checkpoint**，只同步变更的 SST 文件，IO 开销大幅降低
- 状态可以**超出内存**，存储到磁盘，几乎无限扩展
- 缺点：每次读写都需要**序列化/反序列化**（to/from byte[]），有一定 CPU 开销

**适用场景：**
- 状态数据量大（GB 级别）
- 大量 Key，TTL 较长的场景
- 使用 `ListState`、`MapState` 等复杂状态
- Kafka Connector 等需要保存消费偏移量的场景（Kafka Connector 默认使用 RocksDB）

### 如何选择？

```
状态大小 < 1GB         → HashMapStateBackend
状态大小 > 1GB 或不确定  → EmbeddedRocksDBStateBackend
需要 TTL + 大状态      → EmbeddedRocksDBStateBackend
纯内存性能敏感          → HashMapStateBackend
```

### 状态管理最佳实践

1. **及时清理状态**：使用 TTL（Time-To-Live）避免状态无限增长
2. **选择合适的 Key**：Key 设计要均匀，避免数据倾斜导致 TM OOM
3. **监控状态大小**：通过 Flink Dashboard 监控 `numRegisteredKeyGroups` 和 `state size`
4. **开启增量 Checkpoint**：生产环境务必开启，减少 Checkpoint 超时

```java
// 配置 TTL
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.days(7))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
    .cleanupInRocksdbCompactFilter(1000)
    .build();
```
