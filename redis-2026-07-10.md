## 题目一：Redis 有哪些数据类型？适用场景分别是什么？

**参考答案：**

Redis 支持五种基本数据类型，每种类型都有其独特的适用场景：

| 数据类型 | 说明 | 典型使用场景 |
|---------|------|-------------|
| **String** | 最简单的类型，值最大 512MB | 缓存、计数器、分布式锁、限流、Session 存储 |
| **Hash** | 键值对集合，适合存储对象 | 用户信息、商品详情等对象属性存储 |
| **List** | 有序可重复的列表，基于双向链表 | 消息队列、最新消息列表、关注列表 |
| **Set** | 无序不重复集合，支持交并差运算 | 标签系统、好友关系、共同关注、去重 |
| **Sorted Set (Zset)** | 有序不重复集合，每个元素有分数 | 排行榜、热搜榜、延时队列、权重队列 |

此外 Redis 3.0+ 还引入了 **Stream**（消息流）和 **HyperLogLog**（基数统计）、**Bitmap**（位图）、**Geospatial**（地理位置）等高级数据类型。

## 题目二：Redis 的持久化机制有哪些？各有何优缺点？

**参考答案：**

Redis 有两种持久化机制，通常结合使用：

### 1. RDB（Redis Database）

**原理：** 在指定时间间隔内，将内存中的数据快照（Snapshot）写入二进制文件 `dump.rdb`。

**触发方式：**
- 配置文件中的定时任务（如 `save 900 1`）
- 执行 `BGSAVE` / `SAVE` 命令
- 执行 `FLUSHALL` 时（如果配置了 rewrite）

**优点：**
- 文件紧凑，恢复速度快（fork 子进程完成，IO 极小）
- 适合大规模数据备份与灾难恢复
- 性能影响小，不阻塞主进程

**缺点：**
- 可能丢失最后一次快照之后的数据（数据窗口期）
- fork 子进程时如果数据量大，可能有短暂卡顿

---

### 2. AOF（Append Only File）

**原理：** 每次执行写命令后，将命令追加到 `appendonly.aof` 文件末尾。

**刷盘策略（appendfsync）：**
| 策略 | 说明 | 安全性 | 性能 |
|------|------|--------|------|
| always | 每个写命令都同步 | 最高 | 最低 |
| everysec | 每秒同步一次（默认） | 较好 | 较好 |
| no | 由操作系统决定何时刷盘 | 低 | 最高 |

**优点：**
- 数据安全性更高（especially `always` 模式）
- 日志格式，易于理解和修复
- rewrite 机制可压缩文件体积

**缺点：**
- 文件体积通常比 RDB 大
- 恢复速度比 RDB 慢
- 可能比 RDB 慢（取决于刷盘策略）

---

### 3. 混合持久化（Redis 4.0+）

开启混合持久化后，AOF rewrite 时使用 **RDB 格式** 存储历史数据，之后用 **AOF 格式** 记录增量，既保证了恢复速度，又提高了数据安全性。

```conf
aof-use-rdb-preamble yes
```

**推荐策略：** RDB + AOF（everysec）+ 混合持久化，适用于生产环境。

## 题目三：Redis 的过期键删除策略有哪些？

**参考答案：**

Redis 有三种过期键删除策略：

### 1. 定时删除（主动删除）

为每个设置过期时间的 key 创建一个定时器， timer 到期立即删除。

- **优点：** 内存友好，能尽快释放内存
- **缺点：** CPU 不友好，timer 占资源，在 key 数量大时消耗 CPU

### 2. 惰性删除（被动删除）

访问 key 时才检查是否过期，过期则删除。

- **优点：** CPU 友好，只管被访问的 key
- **缺点：** 内存不友好，不访问的过期 key 永远不会被删除

### 3. 定期删除

每隔一段时间（默认 100ms），随机检查一批设置了过期时间的 key，删除其中过期的 key。

```conf
hz 10  # 每秒执行10次定期删除
```

- **优点：** 平衡了 CPU 和内存，是前两种的折中
- **缺点：** 删除不够及时，可能仍有漏网之鱼

---

### 内存淘汰策略（当内存达到上限时）

当 Redis 内存使用达到 `maxmemory` 上限，且没有过期键可删时，触发**内存淘汰策略**：

| 策略 | 说明 |
|------|------|
| `volatile-lru` | 从已设置过期时间的 key 中淘汰最近最少使用的 |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用的 |
| `volatile-ttl` | 从已设置过期时间的 key 中淘汰剩余 TTL 最短的 |
| `volatile-random` | 从已设置过期时间的 key 中随机淘汰 |
| `allkeys-random` | 从所有 key 中随机淘汰 |
| `no-eviction` | 不淘汰，返回错误（默认） |
| `volatile-lfu` | 从已设置过期时间的 key 中淘汰最不经常使用的 |
| `allkeys-lfu` | 从所有 key 中淘汰最不经常使用的 |

**实际建议：** 如果有缓存预热需求，推荐 `allkeys-lru`；如果希望所有 key 都有同等机会，推荐 `allkeys-lfu`。

## 题目四：Redis 主从复制原理是什么？如何实现故障自动切换？

**参考答案：**

### 主从复制原理

Redis 主从复制采用 **CASH（Command Asynchronous Stream Handler）** 机制：

```
                           ┌─────────────┐
                           │   Master    │
                           │  (主节点)    │
                           └──────┬──────┘
                                  │
              ┌──────────────────┼──────────────────┐
              │                  │                  │
              ▼                  ▼                  ▼
        ┌──────────┐       ┌──────────┐       ┌──────────┐
        │ Slave 1  │       │ Slave 2  │       │ Slave N  │
        │ (从节点)  │       │ (从节点)  │       │ (从节点)  │
        └──────────┘       └──────────┘       └──────────┘
```

**复制流程：**

1. **建立连接** — 从节点保存主节点信息，建立 socket 连接
2. **发送 PING** — 验证主节点网络可达
3. **权限验证** — 如果主节点配置了 requirepass，从节点需发送 auth
4. **发送端口信息** — 从节点发送监听端口
5. **全量同步（RDB 同步）**
   - 从节点发送 `PSYNC ? -1`
   - 主节点执行 `BGSAVE`，生成 RDB 文件
   - 主节点将 RDB 文件发送给从节点
   - 从节点清空本地数据，加载 RDB 文件
6. **命令传播** — 主节点将写命令异步发送给从节点（`replication feed`）

**增量复制（部分同步）：**

当从节点短暂断开后重连，使用 `PSYNC <runid> <offset>` 进行增量复制，主节点只需发送断开期间的写命令（依赖 `repl_backlog` 环形缓冲区）。

---

### 故障自动切换（Sentinel 哨兵机制）

Redis Sentinel（哨兵）是 Redis 的高可用方案，核心功能：

| 功能 | 说明 |
|------|------|
| **监控** | 监控主从节点运行状态 |
| **自动故障转移** | 主节点宕机时，自动选举从节点为新主节点 |
| **通知** | 故障转移后通知客户端新主节点地址 |
| **配置提供者** | 客户端可从哨兵获取当前主节点地址 |

**哨兵集群工作流程：**

```
1. 每个哨兵每 10 秒向主从节点发送 INFO 命令
2. 主观下线：哨兵 ping 不通主节点，认为主观下线
3. 客观下线：多个哨兵（quorum）都认为主节点下线
4. 选举领头哨兵（Raft 算法）
5. 领头哨兵从从节点中选择一个作为新主节点
6. 其他从节点指向新主节点（replicaof）
7. 旧主节点恢复后降为从节点
```

**关键配置：**
```conf
sentinel monitor mymaster 127.0.0.1 6379 2  # quorum=2
sentinel down-after-milliseconds mymaster 30000
sentinel failover-timeout mymaster 180000
```

## 题目五：什么是缓存穿透、缓存击穿、缓存雪崩？如何解决？

**参考答案：**

这三个问题是分布式缓存系统中最常见的高并发场景问题：

### 1. 缓存穿透

**问题：** 查询一个**不存在**的数据，由于缓存和数据库都没有，每次请求都打到数据库。

```
请求 → 缓存（无）→ 数据库（无）→ 返回空
       ↓ 大量并发
     数据库崩溃
```

**解决方案：**

| 方案 | 说明 |
|------|------|
| **布隆过滤器（BloomFilter）** | 将所有存在的数据哈希到一个 bitmap，请求时先查布隆过滤器，命中再去缓存/数据库 |
| **缓存空值** | 数据库返回空值时，也在缓存中存一个空值（设置较短 TTL） |
| **参数校验** | 对请求参数做基础校验，拦截非法请求 |

```python
# 布隆过滤器逻辑示例
def get_user(user_id):
    if not bloom_filter.might_contain(user_id):
        return None  # 一定不存在，直接返回
    cache_val = redis.get(f"user:{user_id}")
    if cache_val:
        return json.loads(cache_val)
    db_val = db.query(user_id)
    if db_val:
        redis.setex(f"user:{user_id}", 3600, json.dumps(db_val))
    else:
        redis.setex(f"user:{user_id}", 300, "NULL")  # 缓存空值
    return db_val
```

---

### 2. 缓存击穿

**问题：** 某个**热点 key 过期**，瞬间大量并发请求全部打到数据库。

```
热点 key 过期瞬间
    ↓ 大量并发
  缓存（无）→ 数据库 → 被打爆
```

**解决方案：**

| 方案 | 说明 |
|------|------|
| **互斥锁（Mutex）** | 只允许一个线程从数据库加载数据，其他线程等待 |
| **逻辑过期** | 不给缓存设置真正过期时间，而是设置逻辑过期字段，后台异步更新 |
| **热点数据永不过期** | 通过定时任务主动更新热点数据 |

```python
# 互斥锁方案
def get_user(user_id):
    cache_val = redis.get(f"user:{user_id}")
    if cache_val:
        return json.loads(cache_val)
    
    # 尝试获取锁
    lock = redis.setnx(f"lock:user:{user_id}", "1", nx=True, ex=10)
    if lock:
        db_val = db.query(user_id)
        redis.setex(f"user:{user_id}", 3600, json.dumps(db_val))
        redis.delete(f"lock:user:{user_id}")
        return db_val
    else:
        time.sleep(0.1)
        return get_user(user_id)  # 等待后重试
```

---

### 3. 缓存雪崩

**问题：** 大量缓存 key 在**同一时间过期**，或者 Redis 本身宕机，导致大量请求打到数据库。

```
大量 key 同时过期
    ↓ 大量并发
  缓存（均无）→ 数据库 → 被打爆
```

**解决方案：**

| 方案 | 说明 |
|------|------|
| **过期时间加随机值** | `TTL = base + random(0, 600)`，避免同时过期 |
| **Redis 高可用** | 部署 Redis Cluster / 主从 + Sentinel，避免全盘宕机 |
| **服务熔级限流** | 使用 Hystrix / Sentinel 对数据库访问限流 |
| **预热** | 系统启动时主动加载热点数据到缓存 |
| **多级缓存** | 本地缓存（Guava/Caffeine）+ 分布式缓存，多层保护 |

```python
# 过期时间随机化示例
import random

def set_with_random_ttl(key, value, base_ttl=3600, random_range=600):
    ttl = base_ttl + random.randint(0, random_range)
    redis.setex(key, ttl, value)
```

### 总结对比

| 问题 | 原因 | 核心解决思路 |
|------|------|-------------|
| **缓存穿透** | 查询不存在的数据 | 布隆过滤器 + 缓存空值 |
| **缓存击穿** | 热点 key 过期 | 互斥锁 / 逻辑过期 |
| **缓存雪崩** | 大量 key 同时过期或 Redis 宕机 | 随机 TTL + 高可用架构 + 限流 |
