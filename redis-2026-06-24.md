# Redis 面试题（2026-06-24）

## 题目一：Redis 有哪些数据类型？分别适用于什么场景？

### 参考答案

Redis 支持五种基本数据类型，每种类型都有其独特的应用场景：

| 数据类型 | 可存储值 | 常见应用场景 |
|---------|---------|-------------|
| **String** | 字符串、整数或浮点数 | 缓存、计数器、分布式锁、session存储 |
| **Hash** | 键值对集合 | 存储对象（如用户信息、商品详情）、购物车数据 |
| **List** | 有序、可重复的字符串列表 | 消息队列、最新评论列表、关注列表 |
| **Set** | 无序、不重复的字符串集合 | 标签系统、去重、好友关系、共同关注 |
| **Sorted Set (Zset)** | 有序、不重复，每个元素带分数 | 排行榜、热搜排行、延时队列 |

### 延伸知识点

- **String** 类型最大可以存储 512MB 数据，常用命令：`SET`、`GET`、`INCR`、`DECR`
- **Hash** 适合存储对象，比 String 节省空间，常用命令：`HGET`、`HSET`、`HGETALL`
- **List** 可以实现队列和栈，常用命令：`LPUSH`、`RPOP`、`LRANGE`
- **Set** 支持交集、并集、差集运算，常用命令：`SADD`、`SINTER`、`SUNION`
- **Zset** 通过分数排序，适合排名场景，常用命令：`ZADD`、`ZRANGE`、`ZREVRANGE`

> 📚 参考来源：[Redis 官方文档 - Data Types](https://redis.io/docs/data-types/)

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？

### 参考答案

Redis 提供了两种持久化机制来防止数据丢失：

### RDB（Redis Database）

**原理**：在指定的时间间隔内生成数据集的快照（Point-in-Time Snapshot）。

**配置示例**：
```conf
save 900 1      # 900秒内至少有1个key变化
save 300 10     # 300秒内至少有10个key变化
save 60 10000   # 60秒内至少有10000个key变化
```

**优点**：
- 文件紧凑，适合备份和灾难恢复
- 恢复大数据集时速度较快
- 后台fork子进程进行备份，不影响主进程

**缺点**：
- 可能丢失最后一次快照后的数据
- fork时占用内存较大

### AOF（Append Only File）

**原理**：记录所有写操作命令，以日志形式追加到文件末尾。

**配置示例**：
```conf
appendonly yes
appendfsync everysec   # 每秒同步（推荐）
# appendfsync always   # 每次写操作同步（最安全，性能差）
# appendfsync no       # 由操作系统决定何时同步（性能最好）
```

**优点**：
- 数据安全性更高，丢失最多1秒的数据
- 日志文件是追加形式，即使断电也不会损坏
- 支持重写（rewrite）压缩文件

**缺点**：
- 文件通常比 RDB 大
- 恢复速度比 RDB 慢

### 最佳实践

生产环境通常同时开启两种持久化，兼顾性能和数据安全：
- **RDB** 用于定期备份（如每日凌晨）
- **AOF** 用于保证数据不丢失

> 📚 参考来源：[Redis 官方文档 - Persistence](https://redis.io/docs/management/persistence/)

## 题目三：Redis 如何实现分布式锁？有什么注意事项？

### 参考答案

### 基本实现原理

分布式锁用于在多个进程/服务器之间实现互斥访问共享资源。

### 正确的实现方式（Redisson 模式）

```lua
-- 获取锁的 Lua 脚本
if redis.call('setnx', KEYS[1], ARGV[1]) == 1 then
    redis.call('pexpire', KEYS[1], ARGV[2])
    return 1
elseif redis.call('pttl', KEYS[1]) == -1 then
    redis.call('pexpire', KEYS[1], ARGV[2])
    return 1
end
return 0
```

### Java 代码示例

```java
public class RedisDistributedLock {
    private Jedis jedis;
    private String lockKey;
    private String lockValue;
    private int expireTime = 30000; // 30秒

    public boolean tryLock() {
        lockValue = UUID.randomUUID().toString();
        String result = jedis.set(lockKey, lockValue, 
            new SetParams().nx().px(expireTime));
        return "OK".equals(result);
    }

    public boolean unlock() {
        // 使用 Lua 脚本保证原子性
        String script = 
            "if redis.call('get', KEYS[1]) == ARGV[1] then " +
            "    return redis.call('del', KEYS[1]) " +
            "else return 0 end";
        Object result = jedis.eval(script, 
            Collections.singletonList(lockKey), 
            Collections.singletonList(lockValue));
        return result.equals(1L);
    }
}
```

### 注意事项

| 问题 | 解决方案 |
|-----|--------|
| **锁过期但任务未完成** | 设置合理的过期时间，或使用"看门狗"自动续期 |
| **主从切换导致锁丢失** | 使用 RedLock 算法（多节点），但会增加复杂度 |
| **锁误删** | 使用唯一标识（UUID）识别持有者，Lua脚本保证原子性 |
| **可重入性** | 使用线程ID+计数器实现可重入锁 |

### RedLock 算法

在 N 个独立的 Redis 实例上获取锁，只有当超过 N/2+1 个节点成功获取锁时，才认为成功获取了锁。

> 📚 参考来源：[Redis 官方文档 - Distributed Locks](https://redis.io/docs/manual/patterns/distributed-locks/)

## 题目四：什么是缓存穿透、缓存击穿、缓存雪崩？如何解决？

### 参考答案

这三个是与 Redis 缓存相关的高频问题：

### 1. 缓存穿透

**定义**：查询一个不存在的数据，缓存和数据库都没有，导致请求直接打到数据库。

**危害**：大量无效请求穿透缓存层，可能导致数据库崩溃。

**解决方案**：
- **布隆过滤器（Bloom Filter）**：在缓存层前加一层过滤，快速判断数据是否存在
- **缓存空值**：将查询结果为空的key也缓存起来，设置较短的过期时间
- **参数校验**：对请求参数进行合法性校验

```java
public String getData(String key) {
    // 1. 布隆过滤器判断
    if (!bloomFilter.mightContain(key)) {
        return null; // 肯定不存在，直接返回
    }
    
    // 2. 查询缓存
    String value = redis.get(key);
    if (value != null) {
        return value;
    }
    
    // 3. 查询数据库
    value = db.query(key);
    
    // 4. 缓存空值，防止穿透
    if (value == null) {
        redis.setex(key, 300, ""); // 缓存5分钟
    } else {
        redis.setex(key, 3600, value);
    }
    return value;
}
```

### 2. 缓存击穿

**定义**：某个热点 key 过期瞬间，大量并发请求同时查询该 key，导致数据库压力骤增。

**解决方案**：
- **互斥锁（Mutex）**：只允许一个线程去查询数据库，其他等待
- **热点数据永不过期**：物理不过期，逻辑过期（后台异步更新）
- **提前重建缓存**：异步任务提前刷新热点数据

```java
public String getDataWithLock(String key) {
    String value = redis.get(key);
    if (value != null) {
        return value;
    }
    
    // 尝试获取互斥锁
    String lockKey = "lock:" + key;
    String lockValue = UUID.randomUUID().toString();
    
    if (redis.set(lockKey, lockValue, 
        new SetParams().nx().px(10000))) {
        try {
            // 获取锁成功，查询数据库
            value = db.query(key);
            redis.setex(key, 3600, value);
            return value;
        } finally {
            redis.del(lockKey, lockValue);
        }
    } else {
        // 等待后重试
        Thread.sleep(50);
        return getDataWithLock(key);
    }
}
```

### 3. 缓存雪崩

**定义**：大量缓存数据在同一时间过期，或缓存服务宕机，导致大量请求直接打到数据库。

**解决方案**：
- **过期时间随机化**：给缓存过期时间加随机值，避免同时过期
- **多级缓存**：本地缓存 + Redis + 数据库
- **高可用架构**：Redis Cluster，主从复制
- **服务降级/限流**：熔断器保护数据库

```java
// 设置过期时间时加随机值
int expireTime = baseExpireTime + RandomUtils.nextInt(300);
redis.setex(key, expireTime, value);
```

### 总结对比

| 问题 | 现象 | 解决思路 |
|-----|-----|---------|
| 缓存穿透 | 查询不存在的数据 | 布隆过滤器 + 缓存空值 |
| 缓存击穿 | 热点key过期瞬间 | 互斥锁 + 永不过期 |
| 缓存雪崩 | 大量key同时过期 | 过期时间随机化 + 多级缓存 |

> 📚 参考来源：[Redis 缓存三大问题详解](https://juejin.cn/post/6844904161663040525)

## 题目五：Redis 主从复制原理是什么？如何解决主从延迟和数据一致性？

### 参考答案

### 主从复制架构

```
┌─────────┐     复制     ┌─────────┐     复制     ┌─────────┐
│  Master │ ──────────▶ │ Slave1  │ ──────────▶ │ Slave2  │
└─────────┘             └─────────┘             └─────────┘
```

### 复制原理（PSYNC 2.0 协议）

Redis 复制采用 **增量复制** 机制，分三个阶段：

#### 阶段一：建立连接

1. 从服务器向主服务器发送 `PSYNC ? -1`
2. 主服务器返回 `+FULLRESYNC {runid} {offset}`，开始完整同步
3. 从服务器保存主服务器的 runid 和 offset

#### 阶段二：数据同步（Full Resync）

1. 主服务器 fork 一个子进程执行 `bgsave`，生成 RDB 文件
2. 主服务器将 RDB 文件发送给从服务器
3. 从服务器接收并加载 RDB 文件

#### 阶段三：命令传播（Command Propagation）

1. 主服务器将写命令追加到 **replication buffer** 和 **replication backlog**
2. 主服务器异步发送写命令给从服务器
3. 从服务器执行收到的命令，保持数据一致

### 配置示例

```conf
# 主服务器配置
bind 0.0.0.0
port 6379

# 从服务器配置
replicaof 192.168.1.100 6379
replica-read-only yes
```

### 解决主从延迟问题

**延迟原因**：
- 网络抖动
- 从服务器处理能力不足
- 主服务器写入压力过大

**解决方案**：

| 方案 | 说明 |
|-----|------|
| **监控延迟** | 使用 `INFO replication` 查看 `master_repl_offset` 与从服务器 offset 的差值 |
| **优化网络** | 使用内网互通，减少网络延迟 |
| **提高从服务器配置** | 升级 CPU/内存，使用 SSD |
| **读写分离策略** | 对延迟敏感的业务走主库 |
| **延迟阈值告警** | 设置阈值，如超过 1 秒告警 |

```bash
# 查看复制状态
redis-cli -h <slave-ip> INFO replication
# 输出示例
role:slave
master_link_status:up
master_repl_offset:12345678
slave_repl_offset:12345678
# 两者差值为0，表示无延迟
```

### 数据一致性问题

**CAP 理论**：Redis 集群是 AP 模型（可用性 + 分区容忍性），不是 CP（强一致性）。

**最终一致性**：主从复制是异步的，短时间内可能存在数据不一致。

**保证最终一致性的方案**：

1. **从服务器读取**：对一致性要求不高的场景，读从库
2. **延迟双删**：写操作后删除缓存，数据库更新后再次删除缓存
3. **订阅 keyspace 通知**：从服务器监听主服务器的过期 key 事件

```java
// 延迟双删示例
public void updateData(String key, String value) {
    // 1. 先删除缓存
    redis.del(key);
    
    // 2. 更新数据库
    db.update(key, value);
    
    // 3. 延迟500ms后再次删除（防止并发问题）
    Thread.sleep(500);
    redis.del(key);
}
```

### Redis Sentinel（哨兵模式）

哨兵模式用于 **自动故障转移**，保证高可用：

- 监控主服务器和从服务器状态
- 当主服务器宕机时，自动选举新的主服务器
- 通知应用程序新的主服务器地址

```
┌─────────────┐
│   Sentinel   │ 监控 + 故障转移
└──────┬──────┘
       │
┌──────┴──────┐
│   Master    │ ← 自动切换
└──────┬──────┘
       │ 复制
┌──────┴──────┐
│   Slave1    │
└─────────────┘
```
