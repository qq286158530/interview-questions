## 题目一：Flink 的架构是怎样的？为什么说 Flink 是"真正的"流计算引擎？

### 参考答案

Flink 采用**分层架构**，核心分为四层：

1. **物理执行层（Physical Execution）**
   - TaskManager 是实际执行 Task 的工作节点，每个 TaskManager 是一个 JVM 进程
   - Slot 是 TaskManager 的资源单位，一个 Slot 可以执行一个或多个 SubTask
   - 默认情况下，Slot 数量 = TaskManager CPU 核心数

2. **Runtime 层**
   - JobManager 是协调节点，负责调度 Task、协调 Checkpoint、处理失败恢复
   - StreamGraph → JobGraph → ExecutionGraph 三层图的转换
   - StreamGraph 在客户端生成，对应用户代码的拓扑结构
   - JobGraph 做了算子链化（Operator Chaining），减少数据传输开销
   - ExecutionGraph 是运行时实际调度和执行的图

3. **API 层**
   - DataStream API：用于无界流处理
   - DataSet API：用于有界批处理（已逐步废弃，Flink 1.15 后不再维护）
   - Table API & SQL：统一批流语义的高级抽象

4. **状态后端（State Backend）**
   - **HashMapStateBackend**：状态存储在 TaskManager JVM 堆内存中，高速但受内存限制
   - **EmbeddedRocksDBStateBackend**：状态溢出到 RocksDB（磁盘 KV 存储），支持增量 Checkpoint，适合超大状态场景

### Flink 为什么是"真正的"流计算引擎？

与 Spark Streaming 的微批处理（Mini-Batch）相比，Flink 采用**纯流处理**架构：

- **事件时间（Event Time）处理**：基于数据本身携带的时间戳，而非机器时钟
- **无界流语义**：数据来一条处理一条，延迟低（毫秒级），不像微批有固定批次延迟
- **精确一次（Exactly-Once）语义**：通过 Checkpoint + 两阶段提交协议保证
- **原生流处理**：DataStream API 从一开始就是为流设计，而非后来把批强行转成流

## 题目二：什么是 Watermark？Flink 如何处理乱序数据和迟到事件？

### 参考答案

### Watermark（水位线）的概念

Watermark 是 Flink 用于处理**乱序事件**和时间语义的机制。本质上是一条时间戳 T 的声明：

> **"我承诺，所有时间戳 < T 的数据已经全部到达了"**

Watermark 是一种**启发式猜测**（Heuristic），不是 100% 严格，会导致部分数据被丢弃。

### 两种 Watermark 策略

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| **固定延迟（Fixed Watermark）** | `Watermark = 当前最大事件时间 - 允许延迟` | 简单场景，延迟边界可预估 |
| **单调递增（Monotonous）** | `Watermark = 最大事件时间` | 数据大致有序，偶有乱序 |
| **个性化（Periodic / Punctuated）** | 自定义 WatermarkStrategy，周期性或基于特定事件触发 | 复杂业务逻辑，需精准控制 |

### 代码示例

```java
WatermarkStrategy
    .<Tuple2<String, Long>>forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner((event, timestamp) -> event.f1)  // 从数据中提取时间戳
    .withIdleness(Duration.ofMinutes(1));  // 处理空闲数据源
```

### 迟到事件（Late Data）的处理

Flink 提供三种处理策略：

1. **默认策略（Drop）**：Watermark 之后的数据直接丢弃
2. **allowedLateness**：允许一定范围内的迟到数据，被视为有效输入
   ```java
   .window(...)
       .allowedLateness(Duration.ofMinutes(5))
   ```
3. **sideOutputLateData**：将迟到数据侧输出到旁路流，供后续分析
   ```java
   OutputTag<T> lateTag = new OutputTag<T>("late-data"){};
   .window(...)
       .sideOutputLateData(lateTag)
   ```

### Watermark 传播机制

Watermark 在算子链中**向下游广播传播**，取各输入流 Watermark 的最小值（因为上游任何一个分区的数据延迟都会拖慢整体进度）。

> **来源参考**：Apache Flink 官方文档 — Event Time & Watermarks：<https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/time/#event-time-and-watermarks>
>

## 题目三：Flink Checkpoint 和 Savepoint 的区别是什么？两阶段提交协议如何保证 Exactly-Once？

### 参考答案

### Checkpoint vs Savepoint

| 特性 | Checkpoint | Savepoint |
|------|------------|-----------|
| **触发方式** | Flink 自动周期性触发（JobManager 协调） | 手动触发（用户命令） |
| **用途** | 故障恢复，Job 重启后自动从最近 Checkpoint 恢复 | 计划性维护（升级、扩容），手动恢复 |
| **存储格式** | 轻量，专为快速恢复优化 | 兼容性强，可跨 Flink 版本迁移 |
| **删除机制** | Job 删除时自动清理 | 手动删除，除非配置自动清理策略 |
| **状态后端兼容** | 部分兼容（建议同后端） | 跨后端可能不兼容 |

### 两阶段提交（Two-Phase Commit, 2PC）协议

Flink 的 Exactly-Once 语义通过 **两阶段提交协议** 实现，适用于**含增删改操作**的算子（如 Kafka Sink、File Sink、ProcessFunction）。

#### 四类算子的 Exactly-Once 保证

| 算子类型 | 语义保证 |
|----------|----------|
| 无状态算子（Map、Filter） | 天然 Exactly-Once |
| 有状态算子（Keyed State） | 通过 Checkpoint + 2PC 实现 Exactly-Once |
| **TwoPhaseCommitSink**（Kafka Sink） | Exactly-Once（Flink 1.4+） |
| 普通 Sink（无 2PC 实现） | At-Least-Once |

#### 2PC 执行流程

```
阶段1：预提交（Prepare）
  ├─ Checkpoint 完成时，算子将状态写入 Checkpoint
  ├─ 向下游算子发送 Barrier（屏障）
  ├─ 所有算子完成预提交，返回 ACK 确认

阶段2：正式提交（Commit）
  ├─ JobManager 确认所有算子预提交成功
  ├─ 触发正式提交（如 Kafka transaction.commit()）
  ├─ 若提交失败，Flink 从最近 Checkpoint 重试
```

#### Kafka Exactly-Once 语义

Flink-Kafka Connector 实现了端到端的 Exactly-Once：

```java
// 启用 Exactly-Once
KafkaSink<String> sink = KafkaSink.<String>builder()
    .setBootstrapServers("localhost:9092")
    .setRecordSerializer(new SimpleStringSchema())
    .setDeliveryGuarantee(DeliveryGuarantee.EXACTLY_ONCE)
    .setTransactionalIdPrefix("my-app")
    .build();
```

关键点：Kafka 事务隔离级别设为 `READ_COMMITTED`，Flink 在 Checkpoint 时提交 Kafka 事务，确保数据不重复、不丢失。

## 题目四：Flink 窗口函数有哪些类型？它们各自的语义和使用场景是什么？

### 参考答案

### 窗口分类总览

```
窗口类型
├── 时间窗口（Time Windows）
│   ├── 滚动窗口（Tumbling Window）
│   ├── 滑动窗口（Sliding Window）
│   └── 会话窗口（Session Window）
├── 计数窗口（Count Windows）
│   ├── 滚动计数窗口
│   └── 滑动计数窗口
└── 全局窗口（Global Window）
```

### 1. 滚动窗口（Tumbling Window）

- **特点**：窗口大小固定，不重叠
- **语义**：数据被均匀切分到固定时间段，无数据重叠
- **使用场景**：按分钟/小时统计 UV、GMV、订单量等
- **代码示例**：
  ```java
  DataStream<Tuple2<String, Long>> counts = stream
      .keyBy(t -> t.f0)
      .window(TumblingProcessingTimeWindows.of(Time.seconds(10)))
      .sum(1);
  ```

### 2. 滑动窗口（Sliding Window）

- **特点**：窗口大小固定，但可以重叠（通过 slide 参数控制）
- **语义**：每个窗口独立计算，但数据可能被多个窗口包含
- **使用场景**：移动平均、趋势分析（如最近 5 分钟的的平均值，每 1 分钟更新一次）
- **代码示例**：
  ```java
  .window(SlidingProcessingTimeWindows.of(Time.seconds(30), Time.seconds(10)))
  ```

### 3. 会话窗口（Session Window）

- **特点**：根据数据之间的间隔动态生成窗口，间隔超过阈值则窗口关闭
- **语义**：不固定起止时间，依赖数据到达时间间隔
- **使用场景**：用户行为分析（用户活跃 → 空闲 → 离开的会话划分）、实时监控断连场景
- **代码示例**：
  ```java
  .window(ProcessingTimeSessionWindows.withGap(Time.seconds(5)))
  // 或基于事件时间
  .window(EventTimeSessionWindows.withGap(Time.seconds(5)))
  ```

### 4. 计数窗口（Count Window）

- **特点**：按元素数量触发，而非时间触发
- **语义**：窗口在收集到 N 个元素后触发
- **使用场景**：适合事件驱动型场景，但注意分布式环境下计数窗口的准确性（推荐用 Time Window 替代）
- **代码示例**：
  ```java
  .countWindow(100)        // 滚动计数，每 100 条触发
  .countWindow(100, 50)    // 滑动计数，每 50 条滑动一次
  ```

### 5. 全局窗口（Global Window）

- **特点**：所有数据分配到同一个窗口，需要配合自定义触发器使用
- **语义**：不主动触发，需要自定义 Trigger 决定何时计算
- **使用场景**：自定义业务逻辑，如按天累计达到某阈值才输出结果
- **代码示例**：
  ```java
  .window(GlobalWindows.create())
      .trigger(PurgingTrigger.of(DeltaTrigger.of(1.0, ...)))
  ```

### 增量聚合 vs 全量聚合

| 类型 | 接口 | 说明 | 性能 |
|------|------|------|------|
| **增量聚合** | `AggregateFunction` / `ReduceFunction` | 每条数据来时更新状态，窗口触发时输出最终结果 | 高，内存占用小 |
| **全量聚合** | `WindowFunction` / `ProcessWindowFunction` | 先缓存窗口内所有数据，触发时遍历计算 | 低，适合需要遍历所有数据的场景 |

> **来源参考**：Apache Flink 官方文档 — Windows：<https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream/operators/windows/>
>

## 题目五：Flink 状态管理详解：Keyed State vs Operator State？Flink 如何实现故障恢复？

### 参考答案

### Flink 状态分类

```
Flink State
├── Keyed State（按键分区状态）
│   ├── ValueState<T>
│   ├── ListState<T>
│   ├── ReducingState<T>
│   ├── AggregatingState<IN, OUT>
│   └── MapState<K, V>
│
└── Operator State（算子状态）
    ├── ListState<T>
    ├── UnionListState<T>     // 故障恢复时广播给所有子任务
    ├── BroadcastState<K, V>  // 广播状态，跨算子共享
    └── GenericListState<T>
```

### Keyed State（按键分区状态）

- **绑定到 Key**：状态按 Key 隔离，不同 Key 的状态相互独立
- **只能在 KeyedStream 上使用**（即 `keyBy()` 之后）
- **运行时**：Flink 根据 Key 自动将状态分区，相同 Key 的数据总被同一 SubTask 处理
- **使用示例**：
  ```java
  public class CountWindowAverage extends KeyedProcessFunction<String, Long, Tuple2<String, Double>> {
      private ValueState<Long> countState;
      private ValueState<Long> sumState;

      @Override
      public void open(Configuration parameters) {
          countState = getRuntimeContext().getState(
              new ValueStateDescriptor<>("count", Long.class));
          sumState = getRuntimeContext().getState(
              new ValueStateDescriptor<>("sum", Long.class));
      }

      @Override
      public void processElement(Long value, Context ctx, Collector<Tuple2<String, Double>> out) {
          Long count = countState.value() == null ? 0L : countState.value();
          Long sum = sumState.value() == null ? 0L : sumState.value();
          count++;
          sum += value;
          countState.update(count);
          sumState.update(sum);
          out.collect(new Tuple2<>(ctx.getCurrentKey(), (double) sum / count));
      }
  }
  ```

### Operator State（算子状态）

- **不依赖 Key**：所有算子实例共享同一份状态（分区的算子状态按列表切分）
- **适用场景**：Source（如 Kafka Consumer 的消费位点）、连接到外部系统的算子
- **故障恢复策略**：
  - **LIST_STATE**：均匀分配给所有算子（重新分配时随机 shuffle）
  - **UNION_LIST_STATE**：故障恢复时广播完整状态列表，各算子自行选取自己需要的部分
  - **BROADCAST_STATE**：所有算子收到相同的广播状态
  - **OFFSET_STATE**（Kafka）：一种特殊的 Operator State，存储 Consumer 的分区 offset

### Checkpoint 故障恢复流程

```
1. JobManager 定期（可配置 interval）发起 Checkpoint 协调
2. Source 算子注入 Barrier（屏障），快照当前消费 offset
3. Barrier 随数据流向下游传播，算子完成快照后向 JobManager ACK
4. 所有算子 ACK 完成后，Checkpoint 完成
5. 故障发生时，JobManager 从最近 Checkpoint 元数据恢复：
   ├─ 重启 Source 算子，从 snapshot 的 offset 继续消费
   ├─ 恢复 Keyed State / Operator State
   └─ 重新处理 Checkpoint 之后的数据
```

### 状态过期（TTL）

Flink 支持状态 TTL，避免状态无限膨胀：

```java
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.seconds(3600))
    .setUpdateType(StateTtlConfig.UpdateType.OnCreateAndWrite)  // 或 OnReadAndWrite
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)
    .cleanupInRocksdbCompactFilter(1000)  // RocksDB 增量清理
    .build();

ValueStateDescriptor<T> descriptor = new ValueStateDescriptor<>("my-state", String.class);
descriptor.enableTimeToLive(ttlConfig);
```
