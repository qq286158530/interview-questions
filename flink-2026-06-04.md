## 题目一：Flink 是如何保证 Exactly-Once 语义（端到端一致性）的？

### 参考答案

Flink 通过**两阶段提交（Two-Phase Commit, 2PC）** 实现端到端的 Exactly-Once 语义，需要 Source、Sink 都参与配合：

**1. 事务机制**
- **Source 端**：可回放（如 Kafka Consumer Records 记录偏移量），故障后可从上次提交的位置重新消费数据。
- **Sink 端**：支持事务写入（如 Kafka Sink、JDBC Sink、Iceberg Sink）。Flink 在Checkpoint完成时将数据连同偏移量一起"原子提交"到外部系统。

**2. 两阶段提交流程**
| 阶段 | 操作 | 说明 |
|------|------|------|
| `beginTransaction` | 开启临时目录/缓冲区 | 写入数据先进入"预提交"目录 |
| `preCommit` | 内存数据刷写并关闭 | 与 Checkpoint 屏障对齐 |
| `commit` | 正式提交到目标系统 | 数据从临时目录移动到最终位置 |
| `abort` | 丢弃临时文件 | 失败时清理，不留脏数据 |

**3. Checkpoint 协调**
- Checkpoint 屏障（Checkpoint Barrier）协调各算子的状态快照。
- 所有算子完成快照后，JobManager 才确认 Checkpoint 完成，此时触发 Sink 的 commit 阶段。
- 若在 `preCommit` 后、`commit` 前发生故障，恢复时可以从 `preCommit` 状态重新执行。

**4. 注意事项**
- 如果 Sink 不支持事务（如 HTTP 服务），Flink 的 Exactly-Once 会退化为 **At-Least-Once + 幂等消费**。
- 2PC 会带来 100~500ms 的延迟开销，对延迟敏感的场景需要权衡。

## 题目二：Watermark（水位线）是什么？有哪些生成策略？如何处理迟到数据？

### 参考答案

**1. Watermark 的定义**
Watermark 是流中标记"事件时间进度"的特殊元素，本质是一个时间戳 `T`，表示：**所有事件时间 ≤ T 的数据都已到达**。用于解决乱序和延迟到达问题。

**2. 常见生成策略**

| 策略 | 公式 | 说明 |
|------|------|------|
| **固定延迟** | `Watermark = CurrentMaxEventTime - MaxOutOfOrderness` | 最常用，设 N 秒延迟容忍 |
| **单调递增** | `Watermark = MaxEventTime` | 无乱序场景，简单直接 |
| **标点触发** | 特定事件触发 Watermark | 如每收到一条"订单完成"事件发一个 Watermark |

**3. 窗口触发时机**
- 当 `Watermark >= WindowEndTime` 时，窗口触发计算。
- Watermark 进入窗口后，窗口被认为"关闭"，不会再接收新数据。

**4. 迟到数据处理（Allowed Lateness）**
- `allowedLateness(Time)`：设置一个额外容忍窗口，在窗口关闭后**仍可接收** late event（但只触发窗口的增量更新，不触发窗口fire）。
- `sideOutputLateData(OutputTag)`：超出 allowedLateness 的数据进入侧输出流，避免被丢弃。
- 获取侧输出数据：`DataStream.getSideOutput(tag)`。

**5. 实践建议**
- Watermark 延迟设置太小 → 窗口提前触发，数据丢失
- 延迟设置太大 → 内存压力增大，窗口结果更新延迟高
- 建议根据数据延迟分布来调整，配合 RocksDB StateBackend 缓解内存压力

## 题目三：Flink 的状态后端（State Backend）有哪几种？如何选择？

### 参考答案

Flink 支持两种生产可用的 State Backend：

**1. HeapStateBackend（内存型）**
- 状态存储在 TaskManager 的 JVM 堆内存中
- **优点**：读写速度快（微秒级），无额外 IO 开销
- **缺点**：受堆大小限制（通常 4~32GB/TM），大状态会 OOM
- **适用场景**：小状态、低延迟场景（如简单 ETL、实时聚合）

**2. RocksDBStateBackend（磁盘型）**
- 嵌入式 RocksDB 数据库，状态存储在磁盘，热点数据在堆外内存
- **优点**：状态大小仅受磁盘限制（可达 TB 级），支持增量 Checkpoint
- **缺点**：读写稍慢（毫秒级），增加磁盘 IO
- **适用场景**：大状态场景（如 Session 窗口、特征存储、长期状态）

**3. 选型原则**

```
状态大小 < 1GB/TM   → HeapStateBackend（快）
状态大小 > 10GB/TM  → RocksDBStateBackend（大）
1GB ~ 10GB 之间     → 看延迟需求
```

**4. 关键配置参数**
```java
// Heap
env.setStateBackend(new MemoryStateBackend());

// RocksDB（需加依赖）
env.setStateBackend(new RocksDBStateBackend(filebackend, increamentalCheckpointing));
```

**5. 进阶：增量 Checkpoint**
- 每次 Checkpoint 只写入与上一次相比变化的部分（RocksDB 的 SSTable diff）
- 大状态场景下可将 Checkpoint 时间从数十分钟降到数分钟

## 题目四：Flink 的 Checkpoint 和 Savepoint 有什么区别？如何配置生产级别的 Checkpoint？

### 参考答案

**1. 基本概念**
| | Checkpoint | Savepoint |
|--|-----------|-----------|
| **触发方式** | Flink 自动周期性执行 | 手动触发 |
| **删除机制** | Job 停止时自动删除 | 持久化存储，手动管理 |
| **用途** | 故障恢复（容错） | 升级迁移、版本管理 |
| **兼容性** | 通常兼容（内部格式） | 跨版本可能不兼容 |

**2. Checkpoint 原理**
- 基于 **Chandy-Lamport 分布式快照算法**
- Checkpoint Barrier 从 Source 向下游广播，所有算子在收到 Barrier 时做状态快照
- 全量快照：保存完整状态数据
- 增量快照（RocksDB）：只保存变化部分

**3. 生产级配置建议**

```java
StreamExecutionEnvironment env = ...;

// 启用 Checkpoint，间隔 1 分钟
env.enableCheckpointing(60_000);

// Exactly-Once 语义
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);

// checkpoint 超时时间
env.getCheckpointConfig().setCheckpointTimeout(10 * 60 * 1000);

// 最小间隔（避免密集触发）
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30_000);

// 最多同时 1 个 Checkpoint
env.getCheckpointConfig().setMaxConcurrentCheckpoints(1);

// 开启非对齐 Checkpoint（Flink 1.11+，减少背压）
env.getCheckpointConfig().enableUnalignedCheckpoints();

// 开启 RocksDB 增量 Checkpoint
EmbeddedRocksDBStateBackend rocksDB = new EmbeddedRocksDBStateBackend(true);
env.setStateBackend(rocksDB);
```

**4. 重启策略配置**
```java
// 固定延迟重启：最多重启 3 次，每次间隔 10 秒
env.setRestartStrategy(RestartStrategies.fixedDelayRestart(3, Duration.ofSeconds(10)));

// 失败率重启：1 分钟内失败超过 5 次则判定 Job 失败
env.setRestartStrategy(RestartStrategies.failureRateRestart(
    5, Duration.ofMinutes(1), Duration.ofSeconds(10)));
```

## 题目五：Flink 窗口函数有哪些类型？滑动窗口和会话窗口有什么区别？如何实现亿级 UV 统计？

### 参考答案

**1. 窗口类型总览**

| 窗口类型 | 说明 | 适用场景 |
|---------|------|---------|
| **TumblingWindow（滚动窗口）** | 窗口长度固定，不重叠 | 每分钟/每小时统计 |
| **SlidingWindow（滑动窗口）** | 窗口长度固定，可重叠 | 移动平均、趋势分析 |
| **SessionWindow（会话窗口）** | 按不活动间隙分割 | 用户行为分析 |
| **CountWindow（计数窗口）** | 按元素数量触发 | 凑满 N 条再处理 |

**2. 滚动窗口 vs 滑动窗口**

```java
// 滚动窗口：每 5 分钟一个窗口，无重叠
TimeWindow.of(Time.minutes(5))

// 滑动窗口：窗口长 30 分钟，每 5 分钟滑动一次
TimeWindow.sliding(Time.minutes(30), Time.minutes(5))
```

- 滑动窗口长度 > 滑动距离 时，窗口重叠
- 内存消耗更大（每个元素可能属于多个窗口），性能较差

**3. 会话窗口（Session Window）**
```java
// 不活动间隙超过 10 秒则开启新会话
Window.assignTumbling(EventTimeSessionWindows.withGap(Time.seconds(10)))
```
- 与滑动窗口的本质区别：**窗口边界动态生成**，由数据本身的间隔决定
- 若数据持续发送，会话窗口永远不会触发（可用 GlobalWindow 配合定时器强制关闭）

**4. 亿级 UV 统计解决方案**

**方案一：滑动窗口 + HyperLogLog（推荐）**
```java
// 使用 Flink 内置的 HyperLogLog 聚合
public class UniqueUserAgg implements AggregateFunction<Event, HyperLogLog, Long> {
    @Override
    public HyperLogLog createAccumulator() {
        return new HyperLogLog(); // 固定内存，约 2^12 = 4096 bytes/key
    }
    @Override
    public HyperLogLog add(Event value, HyperLogLog acc) {
        acc.add(value.getUserId());
        return acc;
    }
    @Override
    public Long getResult(HyperLogLog acc) {
        return acc.estimate();
    }
}

// 滑动窗口
env.keyBy(e -> e.getWindowKey())
   .window(SlidingEventTimeWindows.of(Time.hours(24), Time.hours(1)))
   .aggregate(new UniqueUserAgg());
```

**方案二：布隆过滤器（Bloom Filter）+ Redis**
- 布隆过滤器判断用户是否已计入，命中则跳过
- 内存极小（约 10bit/用户，1亿用户仅需约 125MB）
- 需配合 Redis 定期持久化，避免内存丢失

**5. 窗口状态管理注意事项**
- 滑动窗口重叠大时，每个算子实例状态膨胀严重
- 建议使用 RocksDB StateBackend + TTL 自动清理过期状态
- 可通过 `window.assignTumbling(SlidingEventTimeWindows.of(...)).evictor()` 减少状态写入
