## 题目一：Redis 是单线程还是多线程？为什么单线程还能这么快？

### 关键词
`Redis线程模型` `I/O多路复用` `单线程事件循环`

### 题目
Redis 6.0 之前完全使用单线程，为什么单线程的 Redis 还能达到百万 QPS，性能如此之高？

### 答案

Redis 6.0 之前完全采用单线程事件循环模型，但性能却非常高，主要原因有四点：

**1. 纯内存操作**
Redis 是内存数据库，所有数据读写都在内存中完成。内存的随机访问延迟在纳秒级别，而传统数据库每次读写都涉及磁盘 I/O（毫秒级），两者差距达 3 个数量级以上。

**2. 高效的 I/O 模型——I/O 多路复用**
Redis 使用单线程 + I/O 多路复用（Linux 下为 epoll，macOS 为 kqueue）处理网络请求。单个线程可以同时监听和管理成千上万个客户端连接，当某个 socket 可读/可写时，事件循环才去处理，避免了传统网络编程中每个连接一个线程的开销。

**3. 优化的内部数据结构**
Redis 针对不同数据类型实现了高度优化的底层编码（如 ziplist、quicklist、skiplist、intset、hashtable 等），并会根据元素数量、元素大小动态切换最合适的编码。数据结构本身的时间复杂度也是 O(1) 或 O(logN)，不会成为瓶颈。

**4. 简洁高效的通信协议（RESP）**
Redis 自定义了 RESP（REdis Serialization Protocol）协议，简单、解析速度快、二进制安全，相比 JSON 等通用协议序列化/反序列化开销极低。

**Redis 6.0 多线程的真相：**
Redis 6.0 引入了多线程模型，但仅针对**网络数据的读写**（协议解析和响应写入），真正执行用户命令（命令解析、数据操作）仍然是单线程。所以核心的单线程"命令执行"模型没有改变，Redis 的高性能特性依然保持。

> **总结：**Redis 的快不是因为"多线程"，而是因为**纯内存 + I/O 多路复用 + 高效数据结构 + 简洁协议**的组合。

## 题目二：Redis 的过期 key 是如何清理的？谈谈惰性删除和定期删除的原理与配合

### 关键词
`惰性删除` `定期删除` `activeExpireCycle` `expireIfNeeded`

### 题目
Redis 中 key 设置了过期时间后，过期数据是如何被清理的？惰性删除和定期删除有什么区别？如何选择淘汰策略？

### 答案

Redis 采用**惰性删除 + 定期删除**两种策略组合来清理过期 key。

**1. 惰性删除（Lazy Expiration）**
- **原理：** 每次访问 key 时，调用 `expireIfNeeded()` 函数检查该 key 是否已过期。如果过期则立即删除，返回 nil。
- **优点：** 节省 CPU 资源，只有真正访问一个 key 时才检查它，不用额外开销。
- **缺点：** 如果一个 key 已经过期但从未被访问（大量"冷数据"），它会一直占用内存，直到访问发生或内存不足触发淘汰。

**2. 定期删除（Active Expiration）**
- **原理：** Redis 每隔一段时间（默认 **100ms**）执行一次 `activeExpireCycle()`，从设置了过期时间的 key 集合中**随机抽取一部分**进行检查，删除其中已过期的 key。
- **实现细节：**
  - 每轮抽取 `activeExpireCyclesTimelimitExit` 个 key（默认 20 个），耗时超过 25ms 则停止
  - 通过 `ACTIVE_EXPIRE_CYCLE_LOOKUPS_PER_LOOP` 控制每次循环的检查数量
  - 分批渐进式清理，避免一次性 CPU 开销过大
- **优点：** 主动回收过期数据，缓解内存压力。
- **缺点：** 仍有漏网之鱼（未被抽到的过期 key 会保留到下次或访问时）。

**3. 内存淘汰策略（当内存不够用时）**

当 `maxmemory` 到达上限，Redis 会根据配置的淘汰策略清理 key：

| 策略 | 说明 |
|------|------|
| noeviction | 拒绝写入（默认） |
| volatile-lru | 从**过期 key** 中用 LRU 算法淘汰 |
| allkeys-lru | 从**所有 key** 中用 LRU 算法淘汰 |
| volatile-lfu | 从**过期 key** 中用 LFU 算法淘汰 |
| allkeys-lfu | 从**所有 key** 中用 LFU 算法淘汰 |
| volatile-ttl | 淘汰**最快过期**的 key |
| volatile-random | 随机淘汰**过期 key** |
| allkeys-random | 随机淘汰**所有 key** |

**实际建议：**
- 纯缓存场景：推荐 `allkeys-lru`，让热点数据留在内存
- 缓存有 TTL 要求的场景：推荐 `volatile-lru`
- 生产环境务必设置 `maxmemory`，否则可能导致系统内存耗尽

## 题目三：Redis 内存溢出（OOM）后会发生什么？如何监控和优化 Redis 内存使用？

### 关键词
`maxmemory` `内存溢出` `内存优化` `bigkey` `内存碎片`

### 题目
如果 Redis 内存使用达到上限（maxmemory），会触发什么？生产环境中如何监控 Redis 内存使用，以及如何从应用层面减少内存占用？

### 答案

**1. 达到 maxmemory 后会发生什么？**

当 Redis 内存达到 `maxmemory` 配置的上限，Redis 会根据配置的**淘汰策略（eviction policy）**来处理新写入的请求：

- **noeviction（默认）**：拒绝所有写操作（SET、LPUSH 等），读操作正常，返回错误：`OOM command not allowed when used memory > maxmemory`
- **其他策略**：根据策略淘汰 key 后，再执行新写入

**2. 生产环境内存优化策略**

**① 避免 bigkey（最重要的优化）**
bigkey 指单个 key 存储的数据量过大，危害：
- 查询/删除 bigkey 会阻塞事件循环（Redis 单线程）
- 占用大量内存
- 集群迁移时压力极大

发现 bigkey：
```
redis-cli --bigkeys          # 扫描最大的 key
redis-cli --scan | xargs redis-cli OBJECT ENCODING  # 查看编码
```

**② 选择合适的数据类型和编码**

| 数据类型 | 节省内存的建议 |
|---------|--------------|
| String | 尽量控制在 ~10KB 以内，避免 bigkey |
| Hash | field 数量少时用 ziplist 编码（内存高效） |
| Set | 元素少时用 intset 而非 hashtable |
| Zset | 元素少时用 ziplist，元素多用 skiplist |

Redis 3.0+ 支持 `object encoding <key>` 查看 key 的底层编码。

**③ 启用内存压缩**
- `hash-max-ziplist-value`：Hash 类型字段值小于此阈值使用 ziplist（默认 64）
- `set-max-intset-entries`：Set 元素少于此阈值使用 intset（默认 512）

**④ 开启懒释放（unlink）**
`UNLINK` 命令异步删除大 key，不会阻塞事件循环（`DEL` 是同步删除）。

**⑤ 监控内存工具**
```
redis-cli INFO memory          # 查看内存使用详情
redis-cli MEMORY STATS         # Redis 4.0+ 详细内存统计
redis-cli MEMORY DOCTOR        # Redis 6.2+ 内存诊断
```

**⑥ 减少内存碎片的方案**
- 设置 `maxmemory` 时预留 ~20% 空间
- 内存碎片率高时重启 Redis（`INFO memory` 中的 `mem_fragmentation_ratio`）
- Redis 7.2+ 支持自动内存碎片清理：`activedefrag yes`

## 题目四：Redis 事务是怎么实现的？MULTI/EXEC 与 Lua 脚本有什么区别？

### 关键词
`MULTI` `EXEC` `WATCH` `Lua脚本` `原子性` `事务`

### 题目
Redis 事务如何使用？它能保证原子性吗？WATCH 是如何实现乐观锁的？Redis 事务和 Lua 脚本各自适用什么场景？

### 答案

**1. Redis 事务命令**

```
MULTI        # 开启事务，之后的命令进入队列
EXEC         # 执行队列中的所有命令
DISCARD      # 取消事务，清空队列
WATCH key    # 乐观锁，监视 key 变化，变化则事务取消
UNWATCH      # 取消监视
```

**2. Redis 事务能否保证原子性？**

Redis 事务**不保证原子性**（这是它与传统数据库事务最大的区别）：
- `MULTI` 只是将命令加入队列，不做任何检查
- `EXEC` 按顺序执行队列中所有命令
- **如果某个命令执行失败，不会停止其他命令**，也不会回滚已执行成功的命令

```
> MULTI
> INCR count
> INCR count
> EXEC
1) (integer) 1
2) (integer) 2        # 即使某个失败，其他仍会执行
```

**3. WATCH 实现乐观锁**
- `WATCH` 监视一个或多个 key
- 在 `MULTI`/`EXEC` 期间，如果被监视的 key 发生变化（被其他客户端修改），`EXEC` 直接返回 `nil`（事务被打断）
- 这是一种乐观锁实现：假设冲突很少发生，不加锁，只在提交时检查

```
WATCH balance
balance = GET balance
MULTI
DECRBY balance 100
EXEC     # 如果 balance 在 WATCH 后被其他客户端改了，这里返回 nil
```

**4. Lua 脚本——真正的原子操作**

Lua 脚本是 Redis 2.6+ 引入的，脚本内的所有命令在执行期间是**真正的原子操作**：
- Redis 保证脚本执行期间不会被其他命令打断
- 脚本执行过程中，Redis 不会处理其他客户端的请求

```
# 使用 RedisTemplate 的示例
String script = "return redis.call('get', KEYS[1])";
DefaultRedisScript<String> redisScript = new DefaultRedisScript<>(script, String.class);
redisTemplate.execute(redisScript, Collections.singletonList("mykey"));
```

**5. 事务 vs Lua 脚本的选择**

| 特性 | Redis 事务（MULTI/EXEC） | Lua 脚本 |
|------|----------------------|---------|
| 原子性 | 不保证（命令逐条执行） | 真正原子（整个脚本原子） |
| 回滚 | 不支持 | 不支持 |
| 乐观锁 | 支持（WATCH） | 不支持 |
| 适用场景 | 批量命令、流水线 | 复杂逻辑、需要原子的操作 |
| 性能 | 略优（无脚本解析开销） | 需评估脚本复杂度 |

> **生产建议：** 如果需要真正的原子性且逻辑较复杂，使用 **Lua 脚本**；如果只是批量发送命令且可接受最终一致，使用**事务 pipeline**（MULTI 组合 WATCH）。

## 题目五：Redis 和 Memcached 的区别是什么？如何选择？

### 关键词
`Redis vs Memcached` `数据结构` `持久化` `集群` `选择`

### 题目
Redis 和 Memcached 都能做缓存，它们有什么区别？什么时候该选 Redis，什么时候该选 Memcached？

### 答案

**1. 核心区别对比**

| 特性 | Redis | Memcached |
|------|-------|-----------|
| 数据类型 | 支持 5 种基础类型 + 4 种扩展类型 | 只支持**字符串（key-value）** |
| 持久化 | 支持 RDB、AOF、混合持久化 | **不支持**持久化，重启数据丢失 |
| 内存管理 | 自己实现，内存分配更灵活 | slab allocator 固定chunk分配 |
| 过期策略 | 惰性删除 + 定期删除 | 惰性删除（仅此一种） |
| 淘汰策略 | 8 种 LRU/LFU/TTL/Random | LRU（最近最少使用） |
| 集群支持 | 原生支持 Redis Cluster（分片） | 需依赖客户端实现分片 |
| 主从复制 | 支持一主多从复制 | 不支持复制 |
| 性能 | 略低（单线程 + 数据结构丰富） | 略高（纯内存 + 简单结构） |
| 存储方式 | 内存 + 可选磁盘持久化 | 仅内存 |
| 网络协议 | RESP 协议，二进制安全 | ASCII 协议 |

**2. 数据类型能力的差异（最关键的区别）**

Memcached 的 value 只能是字符串：
- 如果要存储对象，需要**手动序列化/反序列化**（JSON）
- 不支持聚合操作（如 Set 的交集、Zset 的排序）

Redis 天然支持丰富数据结构：
- `INCR` 原子计数（无需序列化）
- `HSET` 直接存对象字段
- `ZADD` 直接实现排行榜
- `PFADD` 直接做 UV 统计

**3. 如何选择**

**选 Memcached 的场景：**
- 纯缓存场景，数据可丢失（新闻列表、推荐内容）
- 追求极致简单，不需要复杂数据结构
- 多进程共享内存（如 PHP-FPM 多进程 session 共享）
- 追求略高的简单 GET/SET 性能

**选 Redis 的场景：**
- 需要**丰富的数据类型**（Hash、Set、Zset 等）
- 需要**持久化**或**主从复制**
- 需要** Redis Cluster**做高可用和分片
- 需要**事务、Lua 脚本**保证原子性
- 需要**过期策略、内存优化**的精细控制
- 作为**消息队列、延时队列**使用（List/Stream）

> **一句话总结：** 如果你只需要简单的 KV 缓存且追求极致性能，选 **Memcached**；如果你需要更丰富的数据结构、持久化、集群等能力，选 **Redis**。当前业界 Redis 的使用率远高于 Memcached，Redis 几乎是事实标准。
