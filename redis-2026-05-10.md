## 题目一：Redis 和 Memcached 有什么区别？

### 参考答案

**相同点：**
- 都是基于内存的数据库，通常用作缓存
- 都有过期策略
- 性能都很高

**主要区别：**

| 区别点 | Redis | Memcached |
|--------|-------|-----------|
| 数据类型 | 支持丰富数据类型（String、Hash、List、Set、ZSet） | 仅支持简单的 key-value |
| 持久化 | 支持 RDB 和 AOF 两种持久化方式 | 无持久化，数据全存内存，重启后丢失 |
| 集群 | 原生支持集群模式（主从、哨兵、分片） | 无原生集群，需客户端实现分片 |
| 高级特性 | 支持发布订阅、Lua 脚本、事务等 | 不支持 |

**为什么 Redis 更适合做缓存？**

1. **高性能**：Redis 读写操作直接操作内存，QPS 可达 10W+，是 MySQL 的 10 倍
2. **高并发**：单台 Redis QPS 轻松破 10w，而 MySQL 很难破 1w
3. **丰富的数据结构**：可以满足各种业务场景需求（如排行榜用 ZSet、UV 统计用 HyperLogLog）

## 题目二：Redis 的持久化机制有哪几种？实现原理是什么？

### 参考答案

Redis 有三种持久化方式：**RDB、AOF、混合持久化**。

### RDB（Redis Database）

**原理：** 某个时刻把内存中的所有数据快照写入磁盘，生成 `.rdb` 文件。

**触发方式：**
- 手动触发：`BGSAVE` 命令由子进程执行，不阻塞主进程
- 自动触发：根据配置（如 `save 900 1` 表示 900 秒内至少 1 次修改则触发）

**优点：** 恢复速度快，适合全量备份
**缺点：** 可能丢失最后一次快照之后的数据

### AOF（Append Only File）

**原理：** 每执行一条写操作命令，就将该命令追加到 AOF 文件尾部。

**写回策略（appendfsync）：**
- `Always`：每次执行完命令同步写回，安全性最高但性能最差
- `Everysec`：每秒将 aof_buf 缓冲区内容写回硬盘（默认推荐）
- `No`：由操作系统决定何时写回，性能最好但可能丢失数据最多

**优点：** 数据安全性更高
**缺点：** 文件体积可能很大，恢复速度慢

### 混合持久化（Redis 4.0+）

**原理：** 同时使用 RDB 和 AOF。重写时以 RDB 格式写入数据体，之后增量命令以 AOF 格式追加。

**优点：** 兼顾 RDB 的快速恢复和 AOF 的数据完整性

### AOF 重写机制

当 AOF 文件过大时，后台子进程 `bgrewriteaof` 会：
1. 读取数据库中所有键值对
2. 用一条命令记录每个键值对到新的 AOF 文件
3. 替换旧文件，实现压缩

使用**写时复制（COW）**：父子进程共享物理内存，子进程只读，主进程在修改共享内存时才复制出独立副本。

## 题目三：Redis 的过期删除策略和内存淘汰策略有什么区别？

### 参考答案

### 过期删除策略（删除已过期的 key）

Redis 有两种惰性删除和定时删除策略：

1. **惰性删除**：访问 key 时才检查是否过期，过期则删除。优点是对 CPU 友好，缺点是过期 key 长时间不被访问会占用内存。
2. **定时删除**：每段时间随机检查部分 key，删除已过期的。优点是及时释放内存，缺点是会影响 CPU 性能。

**Redis 默认采用惰性删除 + 定期删除的组合策略。**

### 内存淘汰策略（内存不足时删除 key）

当 Redis 内存使用达到上限（`maxmemory`），会触发淘汰策略：

| 策略 | 说明 |
|------|------|
| `noeviction` | 不淘汰任何 key，返回错误（默认） |
| `volatile-lru` | 从已设置过期时间的 key 中，使用 LRU 算法淘汰 |
| `allkeys-lru` | 从所有 key 中，使用 LRU 算法淘汰 |
| `volatile-lfu` | 从已设置过期时间的 key 中，使用 LFU 算法淘汰 |
| `allkeys-lfu` | 从所有 key 中，使用 LFU 算法淘汰 |
| `volatile-random` | 从已设置过期时间的 key 中随机淘汰 |
| `allkeys-random` | 从所有 key 中随机淘汰 |
| `volatile-ttl` | 淘汰 TTL（剩余存活时间）最短的 key |

**选择建议：**
- 有明确冷数据区分 → `volatile-lru`
- 所有数据都想参与淘汰 → `allkeys-lru`
- 不想淘汰任何有 TTL 的数据 → `allkeys-lru` 或 `noeviction`

## 题目四：Redis 集群有哪几种模式？各有何优缺点？

### 参考答案

Redis 集群主要有三种模式：

### 1. 主从复制模式

**原理：** 一主多从，主节点写数据，从节点读数据。

**优点：**
- 读写分离，提升并发能力
- 主节点故障时从节点可升级为主节点

**缺点：**
- 主节点故障需要手动切换
- 没有自动故障转移
- 主节点是单点，可能成为瓶颈

### 2. 哨兵模式（Sentinel）

**原理：** 哨兵进程监控主从节点，自动完成故障检测和转移。

**核心功能：**
- 监控：检查主从节点是否正常运行
- 通知：故障时通知客户端
- 自动故障转移：从从节点中选举新的主节点

**优点：** 自动故障转移，实现高可用
**缺点：** 只有一个主节点，写入能力受单机限制

### 3. 分片集群模式（Cluster）

**原理：** 将数据分散存储在多个槽（slot）中，每个节点负责一部分数据。

**特点：**
- Redis Cluster 将数据划分为 16384 个槽，每个节点负责一部分
- 每个节点可以有从节点，实现主从复制
- 客户端直接访问任意节点，自动路由

**优点：**
- 数据分片存储，支持水平扩展
- 每个分片都有主从复制，保证高可用
- 节点故障自动切换

**缺点：**
- 不支持跨槽的多键操作（如 `MGET` 的 key 必须都在同一个槽）
- 结构复杂，运维成本高

### 如何选择？

- 数据量小、高可用要求 → 哨兵模式
- 数据量大、需要水平扩展 → Cluster 模式
- 简单场景 → 主从复制模式

## 题目五：如何保证缓存与数据库的双写一致性？

### 参考答案

缓存和数据库双写一致性是分布式系统中经典问题，有多种策略：

### 策略一：Cache Aside（旁路缓存）— 最常用

**读操作：**
1. 先读缓存，命中则直接返回
2. 缓存未命中，读数据库
3. 将数据写入缓存

**写操作：**
1. 先更新数据库
2. 再删除缓存（而不是更新缓存）

**为什么删除而不是更新？** 因为高并发下更新缓存可能导致脏数据，删除让下次读取时重新从数据库加载。

### 策略二：Read Through（读穿透）

缓存负责从数据库加载数据，客户端只和缓存交互。

### 策略三：Write Through（写穿透）

数据写入缓存，缓存同步写入数据库。

### 策略三：Write Behind（异步写回）

数据写入缓存后异步批量写入数据库，性能高但可能丢数据。

### 实际工作中的最佳实践

**延迟双删（解决先删除缓存再更新数据库的并发问题）：**

```python
# 更新数据库
db.update(product)
# 删除缓存
cache.delete(product)
# 延迟一段时间后再次删除（线程休眠或异步任务）
sleep(100ms)
cache.delete(product)
```

**订阅 MySQL BinLog 同步：**
- 应用程序写入 MySQL
- 通过 Canal 等工具监听 BinLog
- 将变更同步到 Redis

### 最终一致性 vs 强一致性

- 大多数业务场景：**最终一致性**（延迟双删 + 过期 TTL）已足够
- 对数据要求严格（如金融交易）：需要**强一致性**，可能需要分布式锁或 TCC 等方案
