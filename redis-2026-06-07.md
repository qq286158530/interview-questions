## 题目一：Redis 有哪些常用数据类型？它们的底层实现分别是什么？

### 参考答案

Redis 提供了 **5 种基础数据类型** 和 **4 种特殊数据类型**：

### 5 种基础数据类型

| 数据类型 | 应用场景 | 底层实现 |
|----------|----------|----------|
| **String（字符串）** | 缓存对象、计数器、分布式锁、Session 共享 | SDS（简单动态字符串），支持二进制安全、O(1) 获取长度、自动扩容 |
| **Hash（哈希）** | 缓存对象、购物车 | 压缩列表（元素少时）或哈希表 |
| **List（列表）** | 消息队列、关注列表 | 双向链表 / 压缩列表（Redis 3.2 后统一为 quicklist） |
| **Set（集合）** | 点赞、共同关注、抽奖 | 哈希表或整数集合（元素全为整数且少时） |
| **Zset（有序集合）** | 排行榜、电话姓名排序 | 压缩列表或跳表（元素少时）或跳表 |

### 4 种特殊数据类型

| 数据类型 | 应用场景 | 版本 |
|----------|----------|------|
| **BitMap（位图）** | 签到、用户在线状态、二值统计 | 2.2+ |
| **HyperLogLog** | 海量数据基数统计（如 UV 计数） | 2.8+ |
| **GEO（地理信息）** | 地理位置存储，如滴滴叫车 | 3.2+ |
| **Stream（流）** | 消息队列，支持消费组和 ACK 机制 | 5.0+ |

### 关键点

- Redis 7.0 后压缩列表（ziplist）已被废弃，改由 **listpack** 实现
- SDS 相比 C 字符串：**二进制安全**、**O(1) 长度获取**、**防缓冲区溢出**
- List 在 Redis 3.2 后统一由 **quicklist** 实现，替代了双向链表 + 压缩列表

## 题目二：Redis 持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 参考答案

Redis 提供两种持久化机制：**RDB** 和 **AOF**，Redis 4.0 后支持混合持久化。

### RDB（Redis DataBase）

- **原理**：定时将内存数据以**快照**形式写入二进制文件 `dump.rdb`
- **优点**：
  - 只有一个文件，方便备份和容灾
  - fork 子进程完成持久化，主进程不受影响
  - 恢复速度比 AOF 快
- **缺点**：间隔持久化，如果故障发生在两次快照之间，会丢失数据

### AOF（Append Only File）

- **原理**：将每次写命令记录到日志文件，重启时重放命令恢复数据
- **优点**：
  - 数据安全性高（可配置 `always/everysec/no`）
  - `append` 模式写文件，宕机后可利用 `redis-check-aof` 修复
- **缺点**：
  - AOF 文件比 RDB 大，恢复速度慢
  - 启动效率低于 RDB

### 混合持久化（Redis 4.0+）

- **同时启用**：以 RDB 作为冷备快速恢复，以 AOF 保证数据不丢失
- RDB 记录数据快照，AOF 记录增量命令，兼顾性能与安全

### 如何选择？

| 场景 | 推荐策略 |
|------|----------|
| 追求快速恢复 | RDB |
| 数据安全性要求高 | AOF |
| 平衡性能与安全 | 混合持久化 |

## 题目三：Redis 为什么快？它的线程模型是怎样的？

### 参考答案

### Redis 为什么这么快？

1. **纯内存操作**：内存访问速度是纳秒级，磁盘是毫秒级，两者相差数个数量级
2. **高效的 I/O 模型**：单线程事件循环 + I/O 多路复用（epoll/select），一个线程处理多个 Socket
3. **优化的内部数据结构**：SDS、quicklist、skiplist、listpack 等高效结构
4. **简洁的通信协议**：RESP 协议实现简单、二进制安全、序列化开销小

### Redis 线程模型

**Redis 6.0 之前**：纯单线程模型

- 主线程负责：**接收连接 → 解析请求 → 读写数据 → 发送响应**
- 后台线程（BIO）处理：关闭文件、AOF 刷盘、内存释放（lazyfree）

**Redis 6.0 之后**：I/O 多线程

- 多线程仅用于**网络 I/O 读写**（接收请求、发送响应）
- **命令执行仍然是单线程**
- 官方测试性能提升至少 **1 倍以上**

### 常见误区

- Redis 不是完全单线程，启动时有后台 BIO 线程处理耗时任务
- 多线程仅针对网络层，核心数据操作仍然是单线程 + 原子性保证

## 题目四：什么是缓存穿透、缓存雪崩、缓存击穿？如何解决？

### 参考答案

### 1. 缓存穿透

**定义**：缓存和数据库中都没有的数据，所有请求都落到数据库。

**解决方案**：
- 接口层增加校验（如用户鉴权、id 基础校验）
- 将查询为空的 key 也缓存起来（key-null），过期时间设短（如 30 秒）
- 使用 **布隆过滤器**，将所有可能存在的数据哈希到 bitmap，拦截不存在数据的请求

### 2. 缓存雪崩

**定义**：大量缓存同一时间大面积失效，导致请求全部打到数据库。

**解决方案**：
- 缓存过期时间加上随机值，避免同时失效
- 使用 **永不过期 + 异步更新** 策略
- 构建多级缓存（本地缓存 + Redis + 数据库）
- 异地多活架构

### 3. 缓存击穿

**定义**：热点 key 过期瞬间，大量并发请求同时访问该 key，打穿数据库。

**解决方案**：
- 热点数据永不过期
- 使用 **互斥锁**（Mutex），只有一个请求去查库并写缓存
- 使用 **分布式锁**，如 Redisson 实现

### 延迟双删策略（保证缓存一致性）

```java
public void write(String key, Object data) {
    redis.delKey(key);       // 1. 先删缓存
    db.updateData(data);     // 2. 更新数据库
    Thread.sleep(1000);     // 3. 延迟 1 秒
    redis.delKey(key);       // 4. 再删缓存（防止旧数据读请求污染）
}
```

## 题目五：Redis 哨兵模式和集群模式的区别是什么？如何保证高可用？

### 参考答案

### 哨兵模式（Sentinel）

- **作用**：监控主从节点，自动故障转移
- **组成**：一组 sentinel 实例监视一个或多个主服务器和多个从服务器
- **核心功能**：
  - 监控（Monitor）：检测主从节点是否正常运行
  - 自动故障转移（Failover）：主节点宕机后自动选举新的主节点
  - 通知（Notification）：故障转移后通知客户端新的主节点地址
  - 配置提供者（Configuration Provider）：客户端通过哨兵获取主节点地址

### 集群模式（Cluster）

- **作用**：数据分片 + 高可用，支持横向扩展
- **原理**：16384 个槽位（slot），每个节点负责一部分槽位，数据分散存储
- **特点**：
  - 支持主从复制和自动故障转移
  - 每个分片可配置一主多从
  - 客户端直连任意节点，自动路由到正确分片

### 两种模式的区别

| 特性 | 哨兵模式 | 集群模式 |
|------|----------|----------|
| 数据分片 | 不支持，所有数据在同一节点 | 支持，16384 槽位分片 |
| 写能力 | 单主节点，写能力受限 | 多主节点，写能力水平扩展 |
| 适用场景 | 小型业务，保证高可用 | 大型业务，需要分片和扩展 |
| 故障转移 | 哨兵自动完成 | 集群自动完成 |

### 选型建议

- **读多写少、高可用要求**：哨兵模式
- **写多、大数据量、需要扩展**：集群模式
