## 题目 1：Spark 的架构与核心组件

**问题：** 请描述 Spark 的整体架构，以及 Driver、Executor、Worker、Catalyst 优化器各自的作用。

**答案：**

Spark 采用 **Master-Slave 架构**，核心组件如下：

| 组件 | 作用 |
|------|------|
| **Driver** | 负责任务调度（构建 DAG、切分 Stage、生成 TaskSet），运行用户的 `main()` 方法 |
| **Executor** | 以容器形式运行在 Worker 节点上，执行具体 Task，将结果返回 Driver；同时缓存数据（RDD / DataFrame） |
| **Worker** | 管理本节点上的 Executor 进程，向 Master 汇报资源状态 |
| **Master** | 资源调度（分配/回收 Worker 上的 CPU 和内存），不参与计算 |
| **Catalyst** | DataFrame/SQL 的查询优化器，将 Logical Plan 逐步优化为 Physical Plan（下推过滤、列剪裁、常量折叠等） |
| **Tungsten** | 内存管理 & 编码优化，利用 off-heap 内存和二进制编码提升序列化效率 |

**运行流程简述：**
1. 用户代码 → Driver 构建 **DAG**（有向无环图）；
2. DAG Scheduler 将 DAG 切分成多个 **Stage**（以 Shuffle 为界）；
3. Task Scheduler 将 Stage 内的 Task 分发到各个 Executor 并行执行；
4. Executor 执行完毕将结果汇报给 Driver。

## 题目 2：RDD 的特性与惰性机制

**问题：** RDD 有哪些核心特性？为什么 Spark 说它是"惰性求值"（Lazy Evaluation）？

**答案：**

### RDD 五大核心特性

1. **分区列表（Partitions）：** RDD 由多个 Partition 组成，每个 Partition 可并行计算。
2. **计算函数（Computations）：** 每个 Partition 上执行 `iterator => Iterator` 函数（如 `map`、`filter`）。
3. **依赖关系（Dependencies）：** RDD 记录其父 RDD 的依赖类型，分为 **窄依赖**（one-to-one，一对一）和 **宽依赖**（shuffle，一对多）。
4. **分区器（Partitioner）：** 对于 Key-Value 类型的 RDD，可指定 Hash 或 Range 分区器。
5. **最优位置（Preferred Locations）：** 尽量将计算调度到数据所在的节点（数据本地性）。

### 惰性求值

- **Transformation**（`map`、`filter`、`flatMap`…）只是**构建 DAG**，不会立即执行。
- **Action**（`collect`、`count`、`saveAsTextFile`…）才会**触发真正的计算**。
- 惰性求值让 Spark 能够：
  - 积累多个 Transformation 形成**管道化（Pipelining）**执行，减少中间结果物化；
  - 在 Action 触发前对 DAG 做整体**优化**（如合并窄依赖阶段的 map、过滤下推）。

## 题目 3：Shuffle 机制与优化策略

**问题：** 什么是 Shuffle？为什么 Shuffle 是 Spark 性能瓶颈？有哪些优化手段？

**答案：**

### 什么是 Shuffle

Shuffle 是 **重新分区数据** 的过程——当一个 RDD/DataFrame 的某个 Partition 需要依赖父 Stage 中**多个 Partition 的数据**时（比如 `groupByKey`、`join`、`reduceByKey`），必须跨节点拉取数据并重新分组写磁盘。

典型 Shuffle 流程：
```
Map 端：写文件（按照 reducer 数量分桶）→ 磁盘
Reduce 端：拉取文件 → 合并 → 计算
```

### 为什么是瓶颈

- **磁盘 I/O：** Shuffle 中间结果写磁盘而非直接内存流水线；
- **网络 I/O：** 跨节点数据传输；
- **内存压力：** 同时缓存 Map 端输出和 Reduce 端缓冲区；
- **磁盘小文件：** 大量 Map Task 产生海量小文件，增加寻道开销。

### 优化策略

| 策略 | 说明 |
|------|------|
| **Map 端聚合（预聚合）** | 使用 `reduceByKey` / `aggregateByKey` 而非 `groupByKey`，在 Map 端先做部分聚合，减少数据量 |
| **控制 Shuffle 并发度** | `spark.sql.shuffle.partitions`（默认 200）可调小避免过多小任务 |
| **使用广播变量** | 小表广播可避免 Shuffle（如 `broadcast join`） |
| **开启 Sort Shuffle** | Spark 2.x+ 默认 Sort Shuffle，比 Hash Shuffle 更稳定（避免海量小文件） |
| **外部排序** | 大数据量场景可开启 `spark.shuffle.sort.bypassMergeThreshold` 等参数调优 |
| **使用 Kryo 序列化** | `spark.serializer=org.apache.spark.serializer.KryoSerializer`，减少数据量 |

## 题目 4：Spark 内存管理模型（Tungsten / Unified Memory）

**问题：** 请描述 Spark 的内存管理模型，包括堆内内存、堆外内存、执行内存、存储内存的区别与动态分配机制。

**答案：**

Spark 采用 **Unified Memory（统一内存管理）** 机制（Spark 1.6+），将堆内存划分为 **Execution（执行）** 和 **Storage（存储）** 两大区域，可动态借用。

### 内存区域划分

```
+------------------------------------------+
|              Heap 堆内内存                |
|  +-----------+----------------+--------+ |
|  |  保留区    |  用户内存      | 备留   | |
|  | (300MB)   |  (20%)        | (20%)  | |
|  +-----------+----------------+--------+ |
|                                          |
|  Storage <== 动态借用 ==> Execution       |
|  (缓存 RDD/广播变量)   (Shuffle/排序等)  |
+------------------------------------------+
|         Off-Heap 堆外内存（可选）         |
|  通过 Unsafe API 直接分配，不受 JVM 管理  |
+------------------------------------------+
```

| 区域 | 用途 | 可动态调整 |
|------|------|-----------|
| **系统保留** | JVM 自身开销（300MB） | 否 |
| **用户内存** | 用户代码（UDF 等），不受 Spark 管理 | 否 |
| **Storage 内存** | RDD 缓存、广播变量 | 可与 Execution 动态互借 |
| **Execution 内存** | Shuffle 排序、Hash 聚合、Map 缓冲区 | 可与 Storage 动态互借 |

### 动态借用规则

- Storage 内存不足时，可**借用** Execution 内存（但被借后若 Execution 需用可**驱逐** Storage）；
- Execution 内存不足时，可**借用** Storage 内存（但 Storage 被驱逐后会**降级**到磁盘）；
- 借用有上限：`spark.memory.fraction`（默认 0.6）× (总堆 - 保留)；
- Storage 可借用比例上限：`spark.memory.storageFraction`（默认 0.5）。

### 堆外内存（Off-Heap）

- 通过 `spark.memory.offHeap.enabled=true` 开启；
- 使用 `sun.misc.Unsafe` 直接向 OS 申请内存，避免 GC 开销；
- 适合大内存场景（JVM > 32GB 时堆内内存效率下降）。

## 题目 5：广播变量（Broadcast Variable）的原理与使用场景

**问题：** 什么是广播变量？它的工作原理是什么？什么场景下使用广播变量能带来显著性能提升？

**答案：**

### 什么是广播变量

广播变量是**只读的、缓存在每个 Executor 进程内存**中的共享变量。Task 无需在每个分区都拷贝一份副本，只需从 Executor 内存中读取，大幅减少网络传输。

```scala
// 定义广播变量
val broadcastVar = sc.broadcast(Array(1, 2, 3, 4, 5))

// 在 Task 中使用（executor 内存读取，无网络传输）
rdd.map(x => x * broadcastVar.value.sum)
```

### 工作原理

1. **Driver 端：** 调用 `sc.broadcast(v)` 时，Spark 将变量序列化（默认 Kryo）并**按 Executor 进程**分发（而非按 Task）。
2. **Executor 端：** 每个 Executor 的 BlockManager **缓存一份**广播变量，所有在该 Executor 上运行的 Task 共享访问。
3. **Task 访问：** Task 直接从本地 Executor 内存读取，**零网络开销**。

### 适用场景

| 场景 | 效果 |
|------|------|
| **小表广播 Join** | 大表 + 小表 Join 时，将小表广播到每个节点，避免 Shuffle（`broadcast(spark.table("small"))`） |
| **配置字典/映射表** | 维度表、配置表等静态数据广播 |
| **UDF 依赖的常量** | 如正则表达式、阈值列表等，避免每个 Task 序列化一份 |
| **过滤条件广播** | 将过滤条件广播，减少网络传输 |

### 注意事项

- **大小控制：** 广播变量过大会撑爆 Executor 内存，建议 < 20MB；超限 Spark 会自动降级为非广播模式。
- **更新代价：** 广播变量是**只读**的，更新需要重新广播。
- **版本一致性：** Driver 广播时会锁定版本，Task 只读取同一版本的广播值。
