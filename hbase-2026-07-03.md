### 题目1：HBase 的架构是怎样的？为什么说它是一个「面向列族」的分布式数据库？

【答案】

**整体架构**：

```
┌─────────────────────────────────────────────┐
│              ZooKeeper                       │
│         (协调服务、选举Master)                 │
└─────────────────┬───────────────────────────┘
                  │
        ┌─────────┴─────────┐
        ▼                   ▼
┌───────────────┐    ┌───────────────┐
│  HMaster      │    │  HMaster(备)  │
│  (主节点)      │    │  (备份)        │
└───────┬───────┘    └───────────────┘
        │     Region Server集群
        │     ┌────┬────┬────┬────┐
        └──→  │ RS1│ RS2│ RS3│ RSn│
              └────┴────┴────┴────┘
                    │
              ┌─────┴─────┐
              ▼           ▼
         Region(一片数据)  Region
              │
         ┌────┴────┐
         ▼         ▼
     Store      Store
    (列族1)    (列族2)
         │         │
    MemStore  MemStore
         │         │
     HFile     HFile
```

**核心组件**：

1. **ZooKeeper**
   - 负责 Region Server 的健康检查与注册
   - 实现 HMaster 的高可用（主备选举）
   - 存储元数据 ROOT 表位置

2. **HMaster**
   - 管理 Table 的 Schema（创建、删除列族）
   - 负责 Region 的分配与负载均衡
   - 监控 Region Server 状态，宕机时重新分配 Region
   - 启动时扫描 .META. 表定位所有 Region

3. **RegionServer**
   - 直接处理读写请求
   - 负责 MemStore 刷写、HFile 合并与拆分
   - 与 HDFS DataNode 交互完成数据持久化

**面向列族的本质**：

- 数据按 **RowKey + ColumnFamily + ColumnQualifier + Timestamp** 四维索引存储
- 同一个列族的数据物理上存储在一起（同一个 Store）
- 不同列族可能存储在不同的 StoreFile 中
- 优点：同列族数据查询快；不同列族可设置不同的存储策略（TTL、压缩、版本数）

### 题目2：HBase 的 RowKey 如何设计？有哪些原则和技巧？

【答案】

**RowKey 设计原则**：

1. **唯一性**：RowKey 必须唯一标识一行数据，相当于主键
2. **长度原则**：控制在 16~100 字节，越短越好（HBase 内部按字典序排序）
3. **散列性**：避免热点，数据均匀分布到各个 Region
4. **业务性**：结合查询场景设计，支持快速定位

**常见设计技巧**：

**1. 加盐（Salting）**
```
原 RowKey：user_123456
加盐后：  a_user_123456  (随机前缀a/b/c)
```
将热点 key 打散到不同 Region，但无法范围查询

**2. 哈希（Hashing）**
```
原 RowKey：order_20260601_12345
哈希后：  MD5(order_20260601_12345)[0:8]_order_20260601_12345
```
打散热点，但破坏原始顺序

**3. 反转（Reversal）**
```
原 RowKey：phone_13800138000
反转后：  00009831_phone
```
适合固定前缀+可变后缀的场景（如手机号、身份证）

**4. 时间戳反转**
```
原：timestamp_1234567890
反转：9999999999 - timestamp_1234567890
```
实现"最新数据在前"的展示效果

**热点场景应对**：

| 场景 | RowKey 设计 | 说明 |
|------|-------------|------|
| 用户订单查询 | `userId + reverseTime` | 用户维度查询+时间倒序 |
| 物联网时序 | `deviceId + (Long.MAX - timestamp)` | 设备查最新数据 |
| 消息Feed | `receiverId + (MAX - timestamp)` | 用户收件箱 |
| 统计报表 | `MD5(date_category)[0:2] + date_category` | 散列+可还原 |

📖 来源：https://cloud.google.com/bigtable/docs/schema/design

### 题目3：HBase 的 MemStore 和 HFile 是什么？它们之间如何协作？

【答案】

**写流程**：

```
Client 写入 → ZooKeeper → RegionServer
                    ↓
              MemStore(内存) ← 数据先写入这里
                    ↓ (达到阈值触发flush)
              HFile(磁盘)   ← 刷写成HFile
                    ↓
              HDFS(DataNode) ← 多副本冗余存储
```

**MemStore**：

- 每个 Store（列族）对应一个 MemStore
- 内存缓冲区，类比 Kafka 的 PageCache
- 按 RowKey 排序（ LSM-Tree 结构）
- 达到以下条件触发刷写：
  - 达到 `hbase.hregion.memstore.flush.size`（默认 128MB）
  - Region 的 MemStore 总大小超过堆内存 × `hbase.regionserver.global.memstore.size`（默认 40%）
  - WAL 日志时间超过 `hbase.regionserver.maxlogs`（默认 30分钟）

**HFile**：

- HBase 在 HDFS 上的存储格式
- 类比 Kafka 的 Segment File + 索引
- 结构：

```
┌─────────────┐
│  Trailer    │  元数据偏移量
├─────────────┤
│  Data Index │  数据块索引（稀疏索引）
├─────────────┤
│  Meta Index │  元数据块索引
├─────────────┤
│   Data      │
│   Block 1   │  KeyValue 数据块（可配置大小，默认 64KB）
│   Block 2   │
│   ...       │
├─────────────┤
│   Meta     │
│   Block     │  元数据块（布隆过滤器信息）
└─────────────┘
```

**核心特性**：

1. **稀疏索引**：每 64KB 数据建立一个索引项，支持快速定位
2. **布隆过滤器（BloomFilter）**：快速判断一个 RowKey 是否存在，避免无效 IO
3. **多版本**：同一 RowKey/Column 可保留多个版本（默认3个）

**刷写时机与合并**：

| 条件 | 说明 |
|------|------|
| MemStore 达到 128MB | 单个 MemStore 刷写 |
| Region 级达到阈值 | 阻塞写，强制刷写 |
|定期刷写 | 所有 MemStore 每小时强制刷写一次 |
| WAL 超限 | HLog 数量过多时优先刷旧数据释放日志 |

** Minor Compaction **：将多个小的 HFile 合并成较大的 HFile
** Major Compaction **：将一个 Store 的所有 HFile 合并成一个，清理过期/删除数据

📖 来源：https://hbase.apache.org/book.html#regionserver

### 题目4：HBase 的存储结构是怎样的？为什么说它是 LSM-Tree 的实现？

【答案】

**LSM-Tree（Log-Structured Merge-Tree）核心思想**：

LSM-Tree 将随机写转换为顺序写，通过分层合并提升读性能：

```
写入：  MemTable（内存） → SStable/ HFile（磁盘，层级 L1~Ln）

读出：  查 MemTable → 查 L0 层的所有文件 → 查 L1 层... → 合并结果
```

**HBase 的存储层次**：

```
┌────────────────────────────────────────┐
│            MemStore (内存)               │
│         写入缓冲，按 RowKey 排序          │
└─────────────────┬──────────────────────┘
                  │  flush（刷写）
                  ▼
┌────────────────────────────────────────┐
│          L0 层 (HFile)                  │
│   每刷写一次生成一个 HFile，无序合并      │
│   文件数量多，可能存在相同 RowKey 的多版本  │
└─────────────────┬──────────────────────┘
                  │  Minor Compaction
                  ▼
┌────────────────────────────────────────┐
│          L1+ 层 (HFile)                 │
│   每层文件 KeyRange 不重叠               │
│   按 Key 排序，全局有序                  │
│   L2 层容量是 L1 的 10 倍                │
└────────────────────────────────────────┘
```

**HBase LSM-Tree 的具体实现**：

1. **写入路径**：
   - 数据先写入 WAL（预写日志，保证持久化）
   - 再写入 MemStore（内存有序）
   - MemStore 刷写后生成 HFile

2. **读路径**：
   - 先查 MemStore（最新数据）
   - 再按层次从新到旧查 HFile
   - 利用 BloomFilter 快速跳过不包含目标 RowKey 的文件

3. **合并策略**：
   - **Size-Tiered Compaction**：类似 Kafka 的 Segment 合并
   - **Level Compaction**（默认）：每层数据量呈指数增长

**LSM-Tree vs B+Tree**：

| 维度 | LSM-Tree (HBase) | B+Tree (MySQL) |
|------|-----------------|----------------|
| 写放大 | 低（顺序写） | 高（随机写+页分裂） |
| 读放大 | 高（多文件合并） | 低（单一路径） |
| 空间放大 | 高（冗余多版本） | 低（一对一） |
| 写吞吐量 | 高 | 低 |
| 适合场景 | 大数据写入 | 高频随机读 |

### 题目5：HBase 协处理器（Coprocessor）是什么？有哪些使用场景？

【答案】

**协处理器是什么**：

协处理器是运行在 RegionServer 上的代码扩展，允许用户在数据写入/读取时自动触发自定义逻辑，无需修改 HBase 核心代码。

**两种类型**：

**1. Observer（观察者）—— 类比 Trigger**

| 类型 | 触发时机 | 典型场景 |
|------|----------|----------|
| **PrePut** | 数据写入前 | 数据校验、过滤、权限控制 |
| **PostPut** | 数据写入后 | 写成功后触发关联更新 |
| **PreGet** | 数据读取前 | 列级权限检查 |
| **PostGet** | 数据读取后 | 动态列转换、脱敏 |
| **PreDelete** | 删除前 | 级联删除关联数据 |
| **PostDelete** | 删除后 | 清理关联缓存 |
| **BulkLoadHFile** | 批量导入后 | 触发索引构建 |

**示例：数据写入前自动添加时间戳**
```java
public class MyObserver extends BaseRegionObserver {
    @Override
    public void prePut(ObserverContext<RegionCoprocessorEnvironment> e,
                       Put put, WALEdit edit, Durability durability) {
        // 自动添加 create_time 列
        put.addColumn(Bytes.toBytes("info"),
                      Bytes.toBytes("create_time"),
                      System.currentTimeMillis());
    }
}
```

**2. Endpoint（端点）—— 类比 Stored Procedure**

- 允许在 RegionServer 上执行自定义计算
- 数据不动，计算移动到数据端
- 支持跨行跨 Region 的聚合操作

**示例：自定义聚合（求最大值）**
```java
// 定义协议接口
public interface MaxFinder extends CoprocessorProtocol {
    long getMax(byte[] column);
}

// RegionServer 实现
public class MaxFinderEndpoint extends BaseEndpointCoprocessor
                             implements MaxFinder {
    @Override
    public long getMax(byte[] column) {
        // 扫描该 Region 的所有行，返回最大值
    }
}

// 客户端调用
HTableInterface table = ...;
byte[] col = Bytes.toBytes("amount");
// 会被分发到所有 Region 并行执行，结果汇总
long maxValue = table.coprocessorExec(MaxFinder.class, startKey, endKey,
    new Batch.Call<MaxFinder, Long>() {
        public Long call(MaxFinder finder) throws IOException {
            return finder.getMax(col);
        }
    });
```

**典型使用场景**：

| 场景 | 协处理器类型 | 说明 |
|------|-------------|------|
| **二级索引** | PostPut/PostDelete | 主表变更时同步更新索引表 |
| **数据聚合** | Endpoint | 在 RegionServer 端并行求和/平均/最大 |
| **权限控制** | PreGet/PrePut | 列级、行级访问控制 |
| **关联查询** | Endpoint | 预计算 JOIN 结果，查询时直接返回 |
| **数据分级存储** | PreGet | 热数据放 SSD，冷数据自动迁移 |
| **审计日志** | PostPut | 记录所有写操作到审计表 |

**加载方式**：

1. **配置加载**（所有表生效）：
   ```xml
   <property>
     <name>hbase.coprocessor.user.region.classes</name>
     <value>com.example.MyObserver</value>
   </property>
   ```

2. **表级加载**（指定表生效）：
   ```java
   HTableDescriptor desc = new HTableDescriptor("myTable");
   desc.addCoprocessor("com.example.MyObserver",
       Priority.PriorityClass.USER, hdfsPath, conf);
   ```
