## 题目一：Redis 有哪五种主要数据类型？各自适合什么场景？

**参考答案：**

Redis 提供了 5 种基础数据类型，分别是 **String、List、Hash、Set、Zset（Sorted Set）**，此外还有 **Bitmap、HyperLogLog、Geospatial** 等高级数据类型。

| 数据类型 | 底层实现 | 典型场景 |
|----------|----------|----------|
| **String** | SDS（简单动态字符串） | 缓存键值对、计数器、Session 存储、限流计数 |
| **List** | Quicklist（压缩列表 + 双向链表） | 消息队列、最新 N 条记录、关注列表、时间轴 |
| **Hash** | 哈希表（ziplist / hashtable） | 存储对象（用户信息、商品详情）、字段级频繁更新的数据 |
| **Set** | 哈希表（intset / hashtable） | 标签系统、去重集合、共同关注/好友推荐 |
| **Zset** | 跳跃表（skiplist + hashtable） | 排行榜、有序集合、延时队列、权重排序场景 |

**补充：** String 最大可存储 512MB，List 支持 LPUSH/RPOP 模拟队列，Zset 的 score 可用于权重排序，是实现排行榜的不二之选。

**来源：** [Redis 官方文档 — Data Types](https://redis.io/docs/data-types/)

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

**参考答案：**

Redis 有两种持久化方式：**RDB（Redis Database）** 和 **AOF（Append Only File）**。

### RDB（快照持久化）

- 定时 fork 子进程，将当前内存数据快照写入磁盘（dump.rdb）
- 可配置 `save 900 1`、`save 300 10` 等规则自动触发
- 恢复快，但可能丢失最近一次快照之后的数据

### AOF（追加日志持久化）

- 每个写操作以 RESP 协议格式追加到文件末尾
- 支持 `always`/`everysec`/`no` 三种同步策略
- 崩溃后仅丢失 1 秒数据（everysec 模式下），但文件较大

### 关键区别

| 维度 | RDB | AOF |
|------|-----|-----|
| 启动速度 | 快（二进制文件） | 慢（需重放命令） |
| 数据完整性 | 可能丢失最近数据 | 最多丢失 1 秒数据 |
| 文件体积 | 小（紧凑快照） | 大（追加日志） |
| 资源消耗 | 周期性 fork 消耗 | 持续磁盘 I/O |

### 生产环境推荐配置

```conf
# RDB + AOF 混合使用
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes   # 开启 RDB-AOF 混合持久化
```

## 题目三：Redis 的过期键删除策略有哪些？它们的区别是什么？

**参考答案：**

Redis 有 **三种过期键删除策略**，分别在不同的时机执行：

### 1. 定时删除（主动删除）

- 在设置键的同时创建一个定时器（Timer），键到期时立即删除
- **优点**：内存友好，及时释放
- **缺点**：CPU 开销大，若大量键同时过期会影响性能
- 实现：`SET key value EX 3600` → 自动在 1 小时后删除

### 2. 惰性删除（被动删除）

- 每次访问键时检查是否过期，过期则删除
- **优点**：CPU 友好，仅在必要时检查
- **缺点**：过期键若永不访问会一直占用内存（内存泄漏风险）
- 实现：Redis 命令执行前调用 `expireIfNeeded()`

### 3. 定期删除

- 每隔一段时间（默认 10 次/秒）检查数据库，删除部分过期键
- 限制每次删除时长不超过 25ms，避免阻塞主线程
- **优点**：平衡了 CPU 和内存

### 内存淘汰策略（当内存不足时）

即使使用了上述删除策略，内存仍可能达到 `maxmemory`，此时 Redis 按配置的淘汰策略处理：

| 策略 | 说明 |
|------|------|
| `volatile-lru` | 从已设置过期时间的键中淘汰最近最少使用 |
| `allkeys-lru` | 从所有键中淘汰最近最少使用 |
| `volatile-ttl` | 淘汰最早过期的键 |
| `volatile-random` | 随机淘汰已设置过期时间的键 |
| `allkeys-random` | 随机淘汰所有键 |
| `noeviction` | 不淘汰，返回写操作错误（默认） |

**来源：** [Redis 官方文档 — Keyspace Notifications / Memory Management](https://redis.io/docs/reference/expire/)

## 题目四：Redis 主从复制原理是什么？哨兵（Sentinel）模式是如何实现故障转移的？

**参考答案：**

### 主从复制原理

Redis 主从复制采用 **PSYNC 协议**，支持全量同步和增量同步：

**全量同步（首次加入或断线重连）：**
1. 从服务器向主服务器发送 `PSYNC ? -1`
2. 主服务器回复 `+FULLRESYNC {runid} {offset}`
3. 主服务器 fork 子进程执行 `BGSAVE`，同时将后续命令写入 **replication buffer**
4. 主服务器将 RDB 文件发送给从服务器
5. 主服务器将 replication buffer 中的增量命令发给从服务器

**增量同步：**
- 主服务器维护一个 `replication backlog` 环形缓冲区
- 从服务器断线重连后发送 `PSYNC {runid} {offset}`
- 主服务器根据 offset 在 backlog 中找到增量数据并发送

### 哨兵（Sentinel）模式

Sentinel 是 Redis 的高可用解决方案，负责监控、选主和通知：

**监控：** Sentinel 每 10 秒向主从服务器发送 `INFO` 命令，发现所有从节点

**故障检测：**
- 每个 Sentinel 每秒向所有实例发送 `PING`
- 主观下线（SDown）：超过 `down-after-milliseconds` 无响应
- 客观下线（ODown）：多数 Sentinel 都认为主服务器下线

**故障转移（选主流程）：**
1. Sentinel 之间通信，达成 ODown 共识
2. 按优先级 `replica-priority` → 复制偏移量 → RunID 最小 选出新主
3. 向候选从服务器发送 `SLAVEOF NO ONE`，升级为新主
4. 向其他从服务器发送 `SLAVEOF 新主IP:PORT`
5. 更新客户端连接的新主地址

**配置建议：**
```conf
sentinel monitor mymaster 127.0.0.1 6379 2
sentinel down-after-milliseconds mymaster 30000
sentinel parallel-syncs mymaster 1
sentinel failover-timeout mymaster 180000
```

**来源：** [Redis 官方文档 — Replication](https://redis.io/docs/management/replication/) | [Sentinel](https://redis.io/docs/management/sentinel/)

## 题目五：什么是缓存穿透、缓存击穿、缓存雪崩？如何应对？

**参考答案：**

### 缓存穿透（Cache Penetration）

**问题：** 查询一个不存在的数据，请求直接打到数据库，数据库也没有对应记录，导致请求绕过缓存。

**解决思路：**

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| **布隆过滤器（BloomFilter）** | 将所有合法 key 放入 BloomFilter，查询前先过滤 | 数据相对固定 |
| **空对象缓存** | 查询数据库无结果时，也缓存一个空值（TTL 短） | 不适合数据频繁变化的场景 |
| **参数校验** | 接口层做基础合法性校验（如 id > 0） | 第一道防线 |

```python
# 空对象方案示例
def get_data(key):
    val = redis.get(key)
    if val is None:
        db_val = db.query(key)
        if db_val is None:
            redis.setex(key, 60, "")  # 短TTL空值
        else:
            redis.setex(key, 3600, db_val)
        return db_val
    return val
```

### 缓存击穿（Cache Smash）

**问题：** 某个热点 key 过期瞬间，大量并发请求同时击穿到数据库。

**解决思路：**

- **互斥锁（Mutex）**：只允许一个线程重建缓存，其他等待
- **热点数据永不过期**：业务层主动续期或永不过期 + 逻辑过期
- **限流熔断**：高并发时返回默认值或降级

```python
# 互斥锁方案
def get_data(key):
    val = redis.get(key)
    if val:
        return val
    lock = redis.setnx("lock:" + key, "1", ex=10)
    if lock:
        db_val = db.query(key)
        redis.setex(key, 3600, db_val)
        redis.delete("lock:" + key)
        return db_val
    else:
        time.sleep(0.1)
        return get_data(key)  # 重试
```

### 缓存雪崩（Cache Avalanche）

**问题：** 大量 key 在同一时间集中过期，或 Redis 宕机，导致数据库压力骤增。

**解决思路：**

| 方案 | 说明 |
|------|------|
| **过期时间随机化** | 设置 TTL 时加上随机偏移量（如 `TTL = base + random(0, 300)`） |
| **多级缓存** | 本地缓存 + Redis + 数据库，层层保护 |
| **高可用架构** | Redis Cluster / 主从 + 哨兵，避免单点故障 |
| **服务熔断降级** | Hystrix / Sentinel 保护数据库 |
| **预热热门数据** | 系统启动时主动加载热点数据到缓存 |
