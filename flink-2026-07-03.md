## 题目一：Flink 的架构是怎样的？请详细说明 JobManager 和 TaskManager 的职责

### 参考答案

Flink 采用**主从架构**，核心组件包括 **JobManager**（主节点）和 **TaskManager**（工作节点）。

### JobManager（主节点）

- **JobMaster**：负责单个 Job 的执行，包括将 JobGraph 转换为 ExecutionGraph、调度任务、协调 Checkpoint、处理异常等。一个 Flink 集群中可以为每个 Job 单独启动一个 JobMaster。
- **ResourceManager**：负责集群资源的管理，与不同的资源管理框架（YARN、Kubernetes、Mesos、Standalone）进行交互，管理 TaskManager 提供的 Slot 资源。
- **Dispatcher**：为每个提交的 Job 启动一个新的 JobMaster，提供 REST 接口供用户提交 Job，支持一个或多个作业。

### TaskManager（工作节点）

- **Slot**：TaskManager 最小的资源分配单位，每个 TaskManager 有一个或多个 Slot，每个 Slot 可以运行一个 Pipeline（由一组算子链化而来）。
- **Task**：实际执行的任务单元，每个 Task 对应一个 Slot 中的一个 Pipeline。
- **Network Manager**：负责 Task 之间的数据交换，通过基于 Netty 的网络栈传输数据。

### 工作流程

```
用户代码 → JobGraph → ExecutionGraph → ExecutionJobVertex → 并行执行
```

Flink On YARN 模式下，JobManager 和 TaskManager 都是 YARN Container 中的进程。JobManager 向 ResourceManager 申请 Slot，TaskManager 注册自己的 Slot，ResourceManager 进行匹配后将 Slot 分配给 JobManager。

## 题目二：什么是 Watermark（水位线）？它是如何解决乱序数据和迟到数据的？

### 参考答案

### Watermark 的定义

Watermark 是 Flink 中用于处理**事件时间（Event Time）**乱序问题的机制，本质上是一个时间戳，代表"所有时间戳小于该水位线的数据都已经到达"。

### 核心思想

```
Watermark(t) = "所有时间戳 <= t 的数据都已到达"
```

- **有序数据流**：Watermark 到达时间 == 事件时间，水位线与事件时间同步。
- **乱序数据流**：由于网络延迟、系统故障等原因，部分数据会迟到（Late Data），Watermark 允许设置一定的容忍度。

### 窗口与 Watermark 的关系

Flink 的基于时间的窗口（如 TumblingEventTimeWindow、SlidingEventTimeWindow）通过比较 `Watermark > 窗口_end_time` 来决定是否触发窗口计算：

```
当 Watermark.time >= 窗口_end_time 时，触发窗口计算
```

### 三种处理策略

1. **默认丢弃（Allowed Lateness）**：迟到数据直接被丢弃，不参与计算。
2. **侧输出流（Side Output）**：将迟到数据输出到侧输出流，供后续单独处理。
3. **Allowed Lateness**：设置一定的容忍时间，在窗口触发后，仍保留窗口状态一段时间，迟到数据在这段时间内仍可参与计算。

### 代码示例

```java
DataStream<String> stream = env
    .addSource(new KafkaSource...)
    .assignTimestampsAndWatermarks(
        WatermarkStrategy
            .<String>forBoundedOutOfOrderness(Duration.ofSeconds(10))
            .withTimestampAssigner((event, timestamp) -> timestamp)
    );
```

## 题目三：Flink 的 Checkpoint 机制是如何工作的？为什么需要 Barrier 对齐？

### 参考答案

### Checkpoint 的核心思想

Flink 通过 **Checkpoint（检查点）** 实现分布式快照，基于 **Chandy-Lamport 算法**的变体——**Asynchronous Barrier Snapshotting（ABS）**，将状态（State）定期持久化存储（如 HDFS、RocksDB），从而实现 Exactly-Once 语义。

### Barrier 对齐流程

1. **JobManager 发起 Checkpoint**：定期向所有 Source 算子发送 Barrier。
2. **Source 发射 Barrier**：Source 算子向下游广播 Barrier，同时保存当前状态快照。
3. **Barrier 向下游传递**：
   - 算子收到第一个输入的 Barrier 后，**阻塞该输入通道**，等待其他输入的 Barrier。
   - 所有输入的 Barrier 都到达后，算子保存状态快照，然后向下游广播 Barrier。
   - 这就是 **Barrier 对齐（Barrier Alignment）** 机制。
4. **Checkpoint 完成**：当 Sink 算子完成快照并确认后，Checkpoint 完成。

### 为什么需要 Barrier 对齐？

- **Exactly-Once 语义**：如果不对齐，算子可能在处理部分数据后再做快照，导致数据被重复处理（部分被快照、部分未被快照）。
- **状态一致性**：对齐确保了所有被处理的数据在快照前都已完成计算，保证快照的一致性。

### Barrier 对齐的代价

- 如果某个输入发生延迟（比如 Kafka 分区数据积压），算子会长时间阻塞，导致处理延迟增加。
- Flink 1.11 之后引入了 **Unaligned Checkpoint**，通过将未对齐的数据也写入 Checkpoint 来减少阻塞时间。

### 状态后端

| 状态后端 | 存储介质 | 适用场景 |
|----------|----------|----------|
| HashMapStateBackend | JVM Heap | 小状态、低延迟 |
| EmbeddedRocksDBStateBackend | RocksDB（磁盘） | 大状态、大吞吐 |

## 题目四：Flink 的窗口函数有哪些？Event Time 窗口和 Processing Time 窗口有什么区别？

### 参考答案

### 窗口类型

#### 1. 时间窗口（Time Windows）

- **滚动窗口（Tumbling Window）**：固定大小、不重叠，如每5分钟一个窗口。
- **滑动窗口（Sliding Window）**：固定大小、可重叠，由窗口大小和滑动步长决定，如窗口大小10分钟、滑动步长5分钟。
- **会话窗口（Session Window）**：根据活动间隙动态创建，如 gap=30秒内无数据则关闭窗口。

#### 2. 计数窗口（Count Windows）

基于元素数量的窗口，如每100个元素一个窗口。

### Event Time vs Processing Time

| 维度 | Event Time | Processing Time |
|------|------------|-----------------|
| **时间来源** | 数据本身携带的事件时间戳 | 机器本地时间（系统时钟） |
| **延迟** | 低（取决于数据本身） | 高（取决于处理速度） |
| **确定性** | ✅ 确定性结果（相同数据无论何时处理结果一致） | ❌ 不确定（处理顺序不同结果不同） |
| **依赖 Watermark** | ✅ 需要（处理乱序） | ❌ 不需要 |
| **乱序处理** | ✅ 支持 | ❌ 不支持 |
| **适用场景** | 日志分析、按事件时间统计 | 实时监控、简单统计 |

### 代码示例

```java
// 滚动事件时间窗口，每5分钟
DataStream<String> windowed = stream
    .keyBy(r -> r.getKey())
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .reduce((a, b) -> a);

// 滑动事件时间窗口
.window(SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(5)))

// 会话窗口，间隙30秒
.window(EventTimeSessionWindows.withGap(Time.seconds(30)))
```

## 题目五：Flink 的状态（State）是如何管理的？KeyedState 和 OperatorState 有什么区别？

### 参考答案

### 状态管理概述

Flink 为有状态流处理提供完善的状态管理机制，状态分为两大类：**KeyedState** 和 **OperatorState**。

### KeyedState（键控状态）

- **作用域**：绑定到某个 Key，只能在 `keyBy` 之后的算子上使用。
- **数据结构**：
  - `ValueState<T>`：单个值
  - `ListState<T>`：列表
  - `MapState<K, V>`：键值对
  - `ReducingState<T>`：聚合状态
  - `AggregatingState<IN, OUT>`：聚合输出状态
- **存储**：按 Key 分区存储，每个 Key 有独立的状态实例。

### OperatorState（算子状态）

- **作用域**：算子级别，所有并行实例共享算子状态，不与 Key 绑定。
- **数据结构**：
  - `ListState<T>`：列表形式，支持重新分配
  - `UnionListState<T>`：广播给所有并行实例，每个实例可选择自己的分区
  - `BroadcastState<K, V>`：广播状态，所有并行实例都有相同的副本
- **使用场景**：Kafka Connector 的分区offset、实时计算中的全局计数。

### 状态一致性

- **Exactly-Once 语义**：Flink Checkpoint 保证了状态的 Exactly-Once 语义。
- **故障恢复**：算子重启后从最近的 Checkpoint 恢复状态，继续处理。

### 代码示例

```java
public class CountWindowAverage extends RichFlatMapFunction<Tuple2<String, Long>, Tuple2<String, Double>> {
    private ValueState<Long> countState;
    private ValueState<Long> sumState;

    @Override
    public void open(Configuration parameters) {
        countState = getRuntimeContext().getState(
            new ValueStateDescriptor<>("count", Long.class));
        sumState = getRuntimeContext().getState(
            new ValueStateDescriptor<>("sum", Long.class));
    }

    @Override
    public void flatMap(Tuple2<String, Long> input, Collector<Tuple2<String, Double>> out) {
        Long currentCount = countState.value() == null ? 0L : countState.value();
        Long currentSum = sumState.value() == null ? 0L : sumState.value();

        countState.update(currentCount + 1);
        sumState.update(currentSum + input.f1);

        if (countState.value() >= 2) {
            out.collect(new Tuple2<>(input.f0, sumState.value().doubleValue() / countState.value()));
            countState.clear();
            sumState.clear();
        }
    }
}
```

### KeyedState vs OperatorState 对比

| 维度 | KeyedState | OperatorState |
|------|-------------|---------------|
| **作用域** | 按 Key 分区 | 算子级别，所有并行实例共享 |
| **使用前提** | 必须在 keyBy 之后 | 无需 keyBy |
| **状态分区** | 每个 Key 独立 | 不分区（List）或广播（Broadcast） |
| **并行度变更时重新分配** | 不需要（Key 自动映射） | 需要重新分配策略 |
| **典型使用** | 窗口聚合、CEP 模式匹配 | Kafka offset、数据源状态 |
