Redis 支持六种高级数据类型，广泛应用于统计、聚合、消息队列等场景：

| 数据类型 |底层实现| 典型使用场景 |
|---------|--------|-------------|
| Bitmap | 字符串 | 用户签到、活跃统计、权限控制 |
| HyperLogLog | 字符串 | UV 统计、独立访客数、基数估计 |
| Geospatial | 有序集合 | 附近的人、门店距离计算、LBS 定位 |
| Stream | Radix Tree | 消息队列、事件流处理、审计日志 |
| Pub/Sub | 字典 | 实时消息推送、订阅发布系统 |

### HyperLogLog 详解

**原理：** 使用概率算法，通过存储哈希后前导零的数量来估算基数。

```
Redis 使用 12KB 内存，可以估算 2^64 个元素的基数，标准误差约 0.81%
```

**常用命令：**

```bash
PFADD page:uv 20240101 20240102 20240103  # 添加用户
PFCOUNT page:uv                               # 统计 UV
PFMERGE pages:merged page1 page2 page3        # 合并多个 PF
```

**应用场景：**

- 每日活跃用户数（DAU）统计
- 网站独立访客数（UV）
- 商品被多少人浏览

**优点：** 极低内存占用（12KB vs 正常计数的大量内存）

**注意：** HyperLogLog 有 ~0.81% 的误差，对精确计数要求高的场景不适用。

来源参考： [Redis 数据类型详解](https://redis.io/docs/data-types/) | [HyperLogLog 原理](https://www.cnblogs.com/54chensongxia/p/15576521.html)

Pipeline（管道）是 Redis 批量执行命令的重要优化手段：

### 原理

普通模式下，每条命令都需要一次 RTT（往返延迟）：

```
Client → Server (命令1) → Client → Server (命令2) → Client
```

Pipeline 将多个命令打包，在一次 RTT 内批量执行：

```bash
# 普通模式：N 次 RTT
GET key1
GET key2
GET key3

# Pipeline 模式：1 次 RTT
GET key1
GET key2
GET key3
```

### 使用示例

```python
# Python redis-py
pipe = redis.pipeline()
pipe.set('k1', 'v1')
pipe.set('k2', 'v2')
pipe.get('k1')
pipe.get('k2')
result = pipe.execute()  # 一次性返回所有结果
```

### 注意事项

- Pipeline 不保证原子性（各命令独立执行）
- 批量过大时客户端内存压力大，建议分批（5000条/批）
- 与事务（MULTI/EXEC）结合需注意 WATCH

**对比：**

| 特性 | Pipeline | 事务（MULTI/EXEC） |
|-----|---------|-------------------|
| 原子性 | ❌ | ✅ |
| RTT | 1次 | 1次 |
| 回滚 | ❌ | ✅ |
| 监控 | ❌ | ✅ WATCH |

来源参考： [Redis Pipeline 文档](https://redis.io/docs/manual/pipelining/) | [Pipeline vs Transactions](https://redis.io/docs/manual/transactions/)

Redis 事务通过 MULTI、EXEC、DISCARD、WATCH 命令实现：

### 基础事务

```bash
MULTI              # 开启事务
SET key1 value1    # 命令入队
INCR counter       # 命令入队
GET key1           # 命令入队
EXEC               # 执行队列中的所有命令
```

###  WATCH - 乐观锁

```bash
WATCH mykey         # 监控 key 变化
MULTI
SET mykey newvalue
EXEC               # 如果 WATCH 的 key 在 WATCH 后被其他客户端修改，事务失败
```

### DISCARD - 取消事务

```bash
MULTI
SET key1 value1
DISCARD            # 清空事务队列，取消事务
```

### Lua 脚本

Redis 内嵌 Lua 解释器，保证脚本原子执行：

```lua
-- 示例：库存扣减（保证原子性）
local stock = redis.call('GET', KEYS[1])
if tonumber(stock) >= tonumber(ARGV[1]) then
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return 'OK'
else
    return 'FAIL'
end
```

```bash
EVAL script 1 stock:100 10   # 参数：脚本 key数量 key1 参数1
```

**EVALSHA：** 脚本缓存后用 SHA1 引用，避免重复传输脚本内容

```bash
SCRIPT LOAD "return redis.call('GET', KEYS[1])"  # 返回 SHA1
EVALSHA sha1 1 mykey
```

### 事务 vs Lua 脚本

| 特性 | 事务 | Lua 脚本 |
|-----|------|---------|
| 原子性 | 全有或全无 | 整个脚本原子 |
| 失败处理 | 无回滚（EXEC 后） | 脚本终止 |
| 性能 | 稍好 | 稍差（解释器开销） |
| 适用场景 | 简单批量操作 | 复杂业务逻辑 |

**注意：** Redis 事务不支持回滚（因 Redis 设计哲学：简单高性能）

来源参考： [Redis Transactions](https://redis.io/docs/manual/transactions/) | [Redis Lua 脚本](https://redis.io/docs/manual/programmability/)

Redis 主从复制是保障数据高可用的基础，哨兵和集群都依赖它：

### 复制流程

```
Master                        Slave
  │                             │
  │←─────── PSYNC ? ────────────│  同步请求
  │─────── +CONTINUE <runid> ──→│  全量复制
  │─────── RDB 文件 ────────────→│  传输快照
  │─────── 缓冲区 ──────────────→│  传输增量
  │                             │
  │←─────── 命令传播 ───────────→│  同步后持续同步
```

**完整同步（Full Sync）：**

1. 主服务器 fork 子进程执行 bgsave，生成 RDB 文件
2. 主服务器将 RDB 文件传输给从服务器
3. 从服务器清空旧数据，加载 RDB 文件
4. 主服务器将复制缓冲区（replication buffer）的命令发送给从服务器

**部分同步（Partial Sync）：Redis 2.8+ 支持：**

- 从服务器断线重连时，如果 offset 在复制积压缓冲区（repl_backlog_buffer）内，则只需补发缺失的命令
- 避免全量同步，提升效率

### 核心配置

```bash
# 从节点配置
replicaof 192.168.1.100 6379    # 指定主节点（Redis 5.0+）
# 或者在配置文件中
replicaof <master-ip> <master-port>

# 主节点配置
repl-diskless-sync yes           # 无盘复制（网络传输 RDB）
repl-backlog-size 10mb          # 复制积压缓冲区大小
min-replicas-to-write 1          # 至少 1 个从节点写入成功
```

### 哨兵模式（Sentinel）

```
┌─────────────┐
│   Client    │
└──────┬──────┘
       │
  ┌────▼────┐
  │ Sentinel │
  │ (Leader) │
  └────┬────┘
       │
   ┌───┴───┐
   │       │
Master   Master
 (主)     (备)
```

哨兵职责：

- **监控：** 定期检测主从节点心跳
- **通知：** 故障时通知应用
- **自动故障转移：** 选举新主节点，通知从节点切换

```bash
# 哨兵配置
sentinel monitor mymaster 192.168.1.100 6379 2
sentinel down-after-milliseconds mymaster 30000
sentinel failover-timeout mymaster 180000
```

### 常见问题

- **复制延迟：** 网络波动导致数据不一致，可通过 `info replication` 查看延迟
- **数据丢失：** 异步复制可能导致故障时少量数据丢失，配置 `min-replicas-to-write` 减少风险
- **脑裂问题：** 网络分区导致两个主节点，可通过 quorum 配置和 `min-replicas-to-write` 缓解

Redis 性能优化是面试中的高频加分项：

### 客户端优化

**连接池：**

```python
# Python 示例
pool = redis.ConnectionPool(
    max_connections=50,
    host='127.0.0.1',
    port=6379,
    decode_responses=True
)
client = redis.Redis(connection_pool=pool)
```

**参数配置：**

```bash
# 连接参数
timeout 10                  # 连接超时
tcp-keepalive 300           # TCP 保活
```

### 服务端优化

```bash
# 内存优化
maxmemory 2gb               # 最大内存
maxmemory-policy allkeys-lru

# 网络优化
tcp-backlog 511             # TCP 全连接队列
client-output-buffer-limit normal 32mb 0 0
```

### 慢查询分析

```bash
# 开启慢查询日志
config set slowlog-log-slower-than 10000   # 10ms
config set slowlog-max-len 128

# 查看慢查询
slowlog get 10

# 分析命令统计
redis-cli --stat
redis-cli --bigkeys
```

### 监控关键指标

```bash
# 核心监控命令
INFO memory                    # 内存使用
INFO stats                     # 命令统计
INFO replication               # 复制状态
INFO clients                   # 客户端连接
INFO cpu                       # CPU 使用
```

**关键指标阈值：**

- `instantaneous_ops_per_sec > 10000` → 请求量高，考虑扩容
- `used_memory_rss > maxmemory * 0.8` → 内存不足，触发淘汰
- `blocked_clients > 0` → 有命令被阻塞
- `repl_lag > 5` → 复制延迟过大

### 常见性能瓶颈

| 瓶颈 | 表现 | 解决方案 |
|-----|------|---------|
| 大 Key | 单次操作耗时长 | 拆分大 Key、使用 HSCAN |
| 热 Key | 单节点压力大 | Hash Tag 打散、本地缓存 |
| 内存碎片 | used_memory_rss 过高 | 启用主动内存整理（activedefrag） |
| 网络带宽 | QPS 突然下降 | 压缩数据、减少大Value |
