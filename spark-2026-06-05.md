## 题目 1：Spark 的架构是怎样的？Driver 和 Executor 之间是什么关系？

### 参考答案

Spark 采用 **Master-Slave 架构**，核心角色如下：

| 角色 | 说明 |
|------|------|
| **Driver（驱动程序）** | 运行用户代码的 main()，负责创建 `SparkContext`、构建 DAG、划分 Stage、生成 TaskSet 并分发给 Executor。是整个应用的"指挥官"。 |
| **Executor（执行器）** | 部署在 Worker 节点上的后台进程，每个节点可启动多个 Executor。负责接收 Driver 分发的 Task，在本节点执行计算，并将结果返回给 Driver。同时通过 `Block Manager` 管理本节点的数据存储。 |
| **Master** | 资源管理器主节点，负责接收 Worker 注册、管理集群资源、分配 CPU 和内存给 Application。 |
| **Worker** | 资源管理器从节点，负责向 Master 注册本节点资源，接收 Master 指令启动 Executor。 |

**Driver 与 Executor 的协作流程：**

1. 用户提交 Spark Application，Driver 创建一个 `SparkContext`。
2. SparkContext 向 Cluster Manager（如 Standalone / YARN / K8s）申请 Executor 资源。
3. Cluster Manager 在 Worker 节点上启动 Executor 进程。
4. Driver 构建 DAG，将 DAG 划分为多个 Stage，每个 Stage 生成一批 Task。
5. Driver 将 Task 分发到各个 Executor 的工作线程中并行执行。
6. Executor 执行完 Task 后，将结果返回给 Driver。
7. 最终 Driver 将结果汇总，写入存储系统。

> **来源：** Apache Spark 官方文档 & 《Spark 权威指南》Chapter 2
> **参考链接：** https://spark.apache.org/docs/latest/architecture.html

## 题目 2：RDD 的特性是什么？惰性求值是如何实现的？

### 参考答案

**RDD（Resilient Distributed Dataset）** 是 Spark 最核心的数据抽象，具有以下五大特性：

| 特性 | 说明 |
|------|------|
| **分区（Partitions）** | 数据集被逻辑划分为多个分区，每个分区是并行的基本单位。 |
| **依赖关系（Dependencies）** | 每个 RDD 记录其父 RDD 的依赖（宽依赖 or 窄依赖），用于容错恢复。 |
| **计算函数（Compute Function）** | 每个 RDD 知道如何通过父 RDD 计算出自己（即 `compute()` 方法）。 |
| **分区器（Partitioner）** | 可选，决定 Key-Value RDD 如何分区（HashPartitioner / RangePartitioner）。 |
| **数据位置（Preferred Locations）** | 可选，记录每个分区最优的计算位置（"数据移动不如计算移动"）。 |

**惰性求值（Lazy Evaluation）：**

Spark 的 Transformation 操作（如 `map`、`filter`、`flatMap`）不会立即执行，只是记录计算逻辑，构建 DAG。只有遇到 **Action 操作**（如 `collect`、`count`、`saveAsTextFile`）时，Spark 才会：

1. 根据依赖关系逆向推导出 **DAG（有向无环图）**。
2. 由 DAG 划分 Stage（依据 **窄依赖划为一组、宽依赖划为边界**）。
3. 生成 **TaskSet**，由 TaskScheduler 分发到 Executor 执行。

> **来源：** Spark 核心概念解析系列
> **参考链接：** https://spark.apache.org/docs/latest/rdd-programming-guide.html

## 题目 3：Spark Shuffle 是什么？为什么要尽量避免过量的 Shuffle？

### 参考答案

**Shuffle（洗牌）** 是 Spark 将数据重新分布的过程。当一个 RDD 的分区需要依赖父 RDD 的多个分区时（常见于 `groupByKey`、`join`、`reduceByKey` 等），必须跨节点传输数据，这一过程称为 Shuffle。

**Shuffle 的工作流程：**

- **Map 阶段**：每个 Mapper 将数据写入本地磁盘，并按目标分区排序。
- **Shuffle Write**：数据写入磁盘文件，可能落盘多次（溢写）。
- **Shuffle Read**：Reducer 启动 Fetch 线程，从各个 Mapper 节点拉取属于自己分区的数据。
- **Merge & Sort**：Reducer 将拉取的数据在本地合并排序后交给计算函数。

**为什么要避免过量 Shuffle？**

1. **磁盘 I/O 开销大**：Shuffle 数据必须落盘，网络传输依赖磁盘文件，无法纯内存操作。
2. **网络开销**：跨节点数据传输，增大集群网络压力。
3. **磁盘空间翻倍**：Map 端输出和 Reducer 端拉取的数据会短时间内共存于磁盘。
4. **启动大量 MapTask/ReduceTask**：短作业频繁调度，Task 启动本身有 overhead。
5. **内存压力**：Reducer 拉取数据后若不够内存排序，会继续溢写磁盘。

**优化建议：**

- 使用 `reduceByKey` 代替 `groupByKey`（先本地聚合再 Shuffle，减少数据量）。
- 避免 `coalesce(shuffle=true)` 来放大分区，尽量在 Shuffle 前后合理设置分区数。
- 使用广播变量避免 Shuffle（将小表广播到各节点）。

> **来源：** 小数点技术社区 & 《Spark内核机制与性能优化》
> **参考链接：** https://spark.apache.org/docs/latest/rdd-programming-guide.html#shuffle-operations

## 题目 4：Spark 的内存管理机制是怎样的？Executor 内存是如何分配的？

### 参考答案

Spark 基于 **JVM 堆内存** 进行管理（Spark 1.6+ 引入 Unified Memory Management），将内存划分为两大区域：

**内存区域划分：**

| 区域 | 用途 | 占比 |
|------|------|------|
| **Execution Memory（执行内存）** | 存储 Shuffle 排序数据、Join 的哈希表、GroupBy 的数据缓冲区等 | ~50%（可动态调整） |
| **Storage Memory（存储内存）** | 缓存 RDD、广播变量、Task 输出结果等 | ~50%（可动态调整） |
| **User Memory（用户内存）** | 用户代码（UDF）及 Spark 内部元数据 | ~33%（固定） |
| **Internal Metadata（内部元数据）** | 系统运行记录 | ~2%（固定） |

**统一内存管理规则：**

- Execution 和 Storage 区域**可互相借用**（storage 可以借用 execution，反之亦然）。
- 借用时，若对方区域有正在使用的内存，则不能强制抢占，需等待释放。
- Storage 区域的数据被借出后，**不会自动回收**，需要主动调用 `unpersist`。

**Spark 1.6 之前的静态内存管理：**
- 内存按比例**静态划分**，互不借用，灵活度低。

**内存相关调优参数：**

```scala
spark.memory.fraction       = 0.6    // Execution + Storage 占 heap 的比例
spark.memory.storageFraction = 0.5   // Storage 占 unified memory 的比例
spark.executor.memory        = 2g    // 每个 Executor 的堆内存大小
```

> **来源：** Apache Spark 官方调优文档
> **参考链接：** https://spark.apache.org/docs/latest/tuning.html#memory-management

## 题目 5：广播变量（Broadcast Variable）的作用是什么？如何使用？有哪些注意事项？

### 参考答案

**广播变量** 是 Spark 提供的优化机制，用于将**只读的大变量**高效地分发到集群中每个 Executor 节点上，避免在每个 Task 中重复传输相同数据。

**核心作用：**

- **节省网络带宽**：一个大表/配置文件等只传输一次到每个节点，而非每个 Task 都传输一次。
- **降低 Driver 压力**：数据存储在 Executor 内存中，Driver 无需重复发送。
- **加速 join**：将小表广播到各节点，将 Shuffle Join 转为 Map Join（无 Shuffle）。

**如何使用：**

```scala
// 在 Driver 端创建广播变量
val broadcastVar = spark.sparkContext.broadcast(Map("key1" -> "value1", "key2" -> "value2"))

// 在 Executor 端访问（使用 .value 获取实际值）
def processRow(row: Row): String = {
    val lookupMap = broadcastVar.value  // 获取广播的数据
    lookupMap.getOrElse(row.getString(0), "default")
}

// 使用完后主动释放（可选）
broadcastVar.unpersist()
```

**注意事项：**

1. **只读**：广播变量一旦创建，内容不可修改，修改不会传播到其他节点。
2. **大小适中**：广播变量存储在 Executor 堆内存中，过大的变量会导致 OOM，建议不超过 `spark.executor.memory * 0.4 * 0.6`。
3. **序列化**：广播变量需要序列化传输，默认使用 Java/Kryo 序列化，Kryo 更高效，推荐配置 `spark.serializer = org.apache.spark.serializer.KryoSerializer`。
4. **广播阈值**：Spark 2.x 起，自动对小表进行广播优化，可通过 `spark.sql.autoBroadcastJoinThreshold` 配置（默认 10MB）。
5. **及时释放**：用完后调用 `unpersist()` 删除缓存中的广播数据，释放 Executor 内存。

> **参考链接：** https://spark.apache.org/docs/latest/rdd-programming-guide.html#broadcast-variables
