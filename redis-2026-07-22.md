## 题目一：Redis 有哪些数据类型？分别适用于什么场景？

### 参考答案

Redis 支持以下五种基本数据类型：

| 数据类型 | 底层实现 | 适用场景 |
|----------|----------|----------|
| **String** | SDS（简单动态字符串） | 缓存、计数器、分布式锁、限流、Session 存储 |
| **List** | QuickList（压缩列表+双向链表） | 消息队列、排行榜、评论列表、最新动态 |
| **Hash** | 压缩列表或 Hash 表 | 对象存储、购物车、用户信息聚合 |
| **Set** | IntSet 或 Hash 表 | 标签系统、好友关系、去重、抽奖 |
| **Sorted Set** | 压缩列表或跳表+字典 | 排行榜、延迟队列、有序通知 |

此外还有三种高级数据类型：
- **Bitmap**：签到统计、用户在线状态、布隆过滤器
- **HyperLogLog**：UV 统计、活跃用户数
- **Geospatial**：附近的人、地理位置服务
- **Stream**：消息队列、事件流处理

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 参考答案

Redis 提供了两种持久化机制：

### RDB（Redis Database）
- **原理**：在指定时间点生成数据集的快照，fork 子进程遍历内存写入二进制文件
- **优点**：恢复速度快、文件紧凑、适合备份
- **缺点**：可能丢失最后一次快照后的数据

### AOF（Append Only File）
- **原理**：记录所有写操作命令，以追加方式写入文件
- **优点**：数据安全性高（可配置同步策略）、崩溃恢复时丢失数据少
- **缺点**：文件体积大、恢复速度慢

### AOF 同步策略
```
appendfsync always    # 每次写入都同步（最安全，性能最差）
appendfsync everysec  # 每秒同步（默认，推荐）
appendfsync no        # 由操作系统决定何时同步
```

### 生产环境推荐配置
```conf
# RDB + AOF 混合持久化（Redis 4.0+）
aof-use-rdb-preamble yes
appendfsync everysec
```

### 选择建议
- **仅使用 Redis 作为缓存**：可不开持久化
- **一般业务**：建议同时开启 RDB + AOF，兼顾性能和数据安全
- **数据敏感**：开启 AOF + everysec 或 always

## 题目三：Redis 的过期键删除策略有哪些？

### 参考答案

Redis 有三种过期键删除策略：

### 1. 定时删除（主动）
- **实现**：为每个设置过期时间的 key 创建定时器，到期立即删除
- **优点**：内存友好，键一过期立即删除
- **缺点**：CPU 开销大，用大量定时器影响性能

### 2. 惰性删除（被动）
- **实现**：访问 key 时检查是否过期，过期则删除
- **优点**：CPU 友好，只在访问时检查
- **缺点**：内存不友好，过期数据可能长期占用内存

### 3. 定期删除
- **实现**：每隔一段时间检查部分 key，删除过期的
- **优点**：折中方案，平衡 CPU 和内存
- **缺点**：难以确定删除间隔和检查比例

### Redis 的实际策略
Redis 采用的是 **惰性删除 + 定期删除** 的组合策略：
- 惰性删除：所有读写命令执行前调用 `expireIfNeeded()` 检查
- 定期删除：`serverCron()` 每 100ms 执行，遍历 `expires` 字典随机抽查

### 内存淘汰策略（当内存不足时）
```
noeviction        # 不淘汰，返回错误（默认）
volatile-lru      # 从已设置过期时间的 key 中淘汰最近最少使用
allkeys-lru       # 从所有 key 中淘汰最近最少使用
volatile-random   # 从已设置过期时间的 key 中随机淘汰
allkeys-random    # 从所有 key 中随机淘汰
volatile-ttl      # 淘汰 TTL 最短的 key
```

## 题目四：Redis 主从复制原理是什么？哨兵模式有什么作用？

### 参考答案

### 主从复制原理

```
+----------------+    复制流    +----------------+
|   Master       | ------------->|   Slave        |
|   (主节点)     | <-------------|   (从节点)     |
+----------------+   PSYNC/PING  +----------------+
```

**复制过程：**
1. **全量同步**：从节点发送 `PSYNC ? -1` → 主节点 `BGSAVE` 生成 RDB → 发送 RDB → 从节点加载
2. **增量同步**：主节点将后续写命令维护在 `replication buffer` 和 `replication backlog` 中
3. 从节点通过网络接收并执行命令，达到数据一致

**关键参数：**
```conf
repl-backlog-size  # 复制积压缓冲区大小（默认 1MB）
repl-ping-slave-period  # 从节点向主节点 ping 频率
```

### 哨兵模式（Sentinel）

哨兵是 Redis 高可用方案，负责：

1. **监控**：检测主从节点是否正常运行
2. **自动故障转移**：主节点宕机时，选举从节点晋升为主节点
3. **通知**：向客户端发送故障转移事件
4. **配置提供者**：提供当前主节点地址

```
+------------------+         +------------------+
|     客户端        | -------->|   Sentinel 集群   |
+------------------+         +--------+---------+
                                  |
                    +-------------+-------------+
                    |                           |
              +------v------+            +------v------+
              |   Master    |            |   Slave     |
              +-------------+            +-------------+
```

**哨兵最少需要 3 个节点**（奇数个），通过 Raft 协议选举领导者完成故障转移。

## 题目五：什么是缓存穿透、缓存击穿、缓存雪崩？如何解决？

### 参考答案

### 1. 缓存穿透
**问题**：查询不存在的数据，缓存和数据库都没有，每次都打到数据库。

**解决方案：**
- **布隆过滤器**：将所有合法数据 key 存入布隆过滤器，拦截非法请求
- **缓存空值**：对查询结果为空的 key 缓存一个空值（如 `"NULL"`），设置较短 TTL
- **参数校验**：对请求参数进行基本校验

### 2. 缓存击穿
**问题**：热点 key 过期瞬间，大量请求同时打到数据库。

**解决方案：**
- **互斥锁**：使用 `SETNX` 保证只有一个请求去查库重建缓存
- **逻辑过期**：不设置 TTL，缓存永不过期，但存储中包含逻辑过期时间，过期时用互斥锁重建
- **热点数据永不过期**：对特别热的数据不设置过期时间

```python
# 互斥锁方案示例
def get_data(key):
    value = redis.get(key)
    if value is None:
        # 尝试获取锁
        if redis.setnx(key + "_lock", "1", ex=10):
            # 查数据库
            value = db.query(key)
            redis.setex(key, 3600, value)
            redis.delete(key + "_lock")
        else:
            # 等待后重试
            time.sleep(0.1)
            return get_data(key)
    return value
```

### 3. 缓存雪崩
**问题**：大量 key 同时过期，或 Redis 宕机，导致数据库压力骤增。

**解决方案：**
- **过期时间随机化**：给 key 的 TTL 加随机值，避免同时过期
- **多级缓存**：本地缓存 + Redis 缓存
- **Redis 高可用**：部署主从 + 哨兵/集群
- **服务熔级限流**：数据库压力过大时拒绝部分请求
- **预热**：系统启动前预加载热点数据

```conf
# 过期时间随机化示例
SET product:1 "xxx" EX 3600  # TTL = 3600 + random(0, 300)
```
