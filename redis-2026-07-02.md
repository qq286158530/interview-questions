## 题目一：Redis 有哪些数据类型？应用场景分别是什么？

### 答案

Redis 支持五种基础数据类型，每种都有其典型的应用场景：

| 数据类型 |底层实现| 典型应用场景 |
|---------|--------|-------------|
| **String** | SDS（简单动态字符串） | 缓存、计数器、分布式锁、限流、Session 存储 |
| **Hash** | 压缩列表 / 哈希表 | 对象存储（如用户信息、商品详情）、购物车 |
| **List** | 快速链表 / 压缩列表 | 消息队列（LPUSH+BRPOP）、最新消息列表、关注列表 |
| **Set** | 整数集合 / 哈希表 | 去重、标签系统、好友关系、共同关注 |
| **Zset** | 压缩列表 / 跳表 + 字典 | 排行榜、有序消息队列、延迟队列 |

**补充：**
- Redis 7.0 还引入了 **Streams**（流数据类型），用于消息队列场景，弥补 List 做消息队列的不足（不支持消费组、消息 ID 等）
- Bitmap 适合签到、活跃用户统计；HyperLogLog 适合 UV 统计；Geospatial 适合附近的人功能

**应用场景举例：**
```
# 分布式锁
SET lock_key value NX PX 30000

# 计数器（incrby）
INCR article:viewcount:1001

# 排行榜（zset）
ZADD hot:articles 1000 "article:1001"
ZREVRANGE hot:articles 0 9 WITHSCORES
```

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 答案

Redis 有两种持久化机制：

### RDB（Redis Database Backup）

- **原理：** 定时将内存中的全部数据快照（Snapshot）写入磁盘，生成 `dump.rdb` 文件
- **触发方式：**
  - `bgsave`：后台异步执行，不阻塞主进程
  - `save`：同步执行，会阻塞（大数据量时不可用）
  - 自动触发：满足 `save m n` 配置（如 `save 900 1` 表示 900 秒内 ≥1 次写操作则触发）
- **优点：** 恢复速度快，适合大规模数据备份
- **缺点：** 可能丢失最后一次快照之后的数据

### AOF（Append Only File）

- **原理：** 每执行一条写命令，就追加到磁盘的 `appendonly.aof` 文件
- **同步策略（appendfsync）：**
  - `always`：每条命令都同步，最安全但最慢
  - `everysec`（默认）：每秒同步一次，最多丢失 1 秒数据
  - `no`：由操作系统决定何时同步，最快但最不安全
- **重写机制：** 当 AOF 文件过大时，后台自动执行 `bgrewriteaof`，对命令合并压缩
- **优点：** 数据安全性更高（everysec 模式下最多丢 1 秒）
- **缺点：** 文件体积通常比 RDB 大，恢复速度慢

### 对比总结

| 维度 | RDB | AOF |
|------|-----|-----|
| 文件体积 | 小 | 大 |
| 恢复速度 | 快 | 慢 |
| 数据安全性 | 可能丢数据 | 丢数据少 |
| 性能影响 | fork 子进程，不影响主进程 | 持续写，略有影响 |
| 场景 | 冷备份、恢复 | 热备份、容灾 |

### 如何选择

- **仅使用 Redis 作为缓存** → 两种都不用开（`save ""`）
- **能容忍数分钟数据丢失** → RDB 即可
- **对数据安全性要求高** → AOF（推荐 `everysec`）
- **最佳实践：** **两者同时开启**，`aof-use-rdb-preamble yes`（Redis 4.0+），混合持久化，重启时优先加载 AOF

## 题目三：Redis 的过期键删除策略有哪几种？内存淘汰策略有哪些？

### 答案

### 一、过期键删除策略

当一个键设置了 TTL（过期时间），Redis 会通过以下策略来处理过期键：

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **定时删除** | 用定时器，过期立即删除 | 节约内存 | CPU 压力大，影响响应时间 |
| **惰性删除** | 访问键时检查是否过期，过期则删除 | CPU 友好 | 内存压力大（过期键占内存） |
| **定期删除** | 每隔一段时间，扫描检查并删除过期键 | 折中方案 | 实现复杂 |

Redis 采用的是 **惰性删除 + 定期删除** 的组合策略：
- **惰性删除**：`get`/`set` 等命令执行时检查并删除
- **定期删除**：每隔一段时间（默认 100ms），随机检查部分键

### 二、内存淘汰策略（Memory Eviction Policy）

当 Redis 内存达到 `maxmemory` 上限时，会触发淘汰策略删除键：

| 策略 | 行为 |
|------|------|
| `noeviction` | 不淘汰，返回错误（默认） |
| `volatile-lru` | 在设置了过期时间的键中，使用 LRU 算法淘汰 |
| `allkeys-lru` | 在所有键中，使用 LRU 算法淘汰 |
| `volatile-lfu` | 在设置了过期时间的键中，使用 LFU 算法淘汰 |
| `allkeys-lfu` | 在所有键中，使用 LFU 算法淘汰 |
| `volatile-random` | 在设置了过期时间的键中，随机淘汰 |
| `allkeys-random` | 在所有键中，随机淘汰 |
| `volatile-ttl` | 在设置了过期时间的键中，淘汰 TTL 最短的 |

**实际建议：**
- 缓存场景推荐 `allkeys-lru` 或 `allkeys-lfu`
- 需要精确控制内存时，建议开启 `maxmemory-policy`

## 题目四：Redis 主从复制原理是什么？哨兵（Sentinel）集群是如何实现故障自动切换的？

### 答案

### 主从复制原理

Redis 主从复制采用 **CouchDB 风格**的复制流程，支持全量同步和增量同步：

**1. 全量同步（Full Sync）：**
```
从服务器 -> 主服务器：PSYNC ? -1         （发送 SYNC 命令，? 表示不知道 master run id，-1 表示请求全量同步）
主服务器 -> 从服务器：+FULLRESYNC <runid> <offset>   （开始全量同步）
主服务器 -> 从服务器：（发送 RDB 文件快照）
从服务器 -> 主服务器：确认接收完成
主服务器 -> 从服务器：（发送缓冲区中的增量命令）
```

**2. 增量同步（Command Sync）：**
- 主服务器维护一个 ** replication buffer**（复制缓冲区）和 **replication backlog**（复制积压缓冲区）
- 每次写命令同时写入这两个缓冲区
- 从服务器以 `repl-offset` 标记自己的位置，持续拉取增量

**3. 关键参数：**
- `repl-backlog-size`：复制积压缓冲区大小（建议设为 `qps * 秒数`，如 qps=1万，时长=60秒，则 600MB）
- `repl-diskless-sync`：无盘复制，适合网络 IO 场景

### 哨兵（Sentinel）故障自动切换（failover）

哨兵集群本身也是一个分布式系统，由一个或多个 Sentinel 节点组成：

**故障检测：**
- 每个 Sentinel 每秒向主/从/其他 Sentinel 发送 `PING`
- 如果超过 `down-after-milliseconds` 未响应，标记为 **主观下线（SDOWN）**
- 超过法定数量（`quorum`）的 Sentinel 都认为主节点下线，则标记为 **客观下线（ODOWN）**

**故障切换（failover）：**
```
1. Sentinel A 发现主节点客观下线，发起选举（向其他 Sentinel 发送 is-master-down-by-addr）
2. 获得半数以上票数的 Sentinel 成为 Leader
3. Leader 从所有从节点中挑选一个：
   - 过滤：排除主观下线、断开连接超过一定时间的
   - 优先级：replica-priority 高的优先
   - 偏移量：offset 最大的（数据最新）
   - runid：字典序最小的
4. Leader 向被选中的从节点发送 SLAVEOF NO ONE，使其升级为新主
5. Leader 向其他从节点发送 SLAVEOF 新主地址，实现主从切换
6. 更新配置，更新.conf 文件，通知客户端新主节点地址
```

## 题目五：Redis Cluster 是如何实现分布式集群的？它是如何保证数据一致性的？

### 答案

### Redis Cluster 架构

Redis Cluster 是 Redis 官方提供的分布式集群方案，采用 **虚拟槽分区（Virtual Slot）**：

- **16384 个槽（slot）**：每个键通过 `CRC16(key) % 16384` 映射到 0~16383 某个槽
- **每个主节点负责一部分槽**：例如 3 主 3 从，则每个主节点负责约 5460 个槽
- **节点间通信**：使用 Gossip 协议（PING/PONG），每 1 秒随机选 5 个节点通信

```
槽 0~5460      -> 节点 A（主） + A-slave
槽 5461~10922  -> 节点 B（主） + B-slave
槽 10923~16383 -> 节点 C（主） + C-slave
```

### 数据一致性保证

#### 1. 主从同步（复制）

- 每个主节点有 1~N 个从节点（建议 1 个）
- 从节点异步复制主节点数据（与普通主从复制相同）
- 主节点故障时，从节点自动升级为主节点

#### 2. 故障转移（failover）

- 类似于哨兵机制：从节点发现主节点不可用后，发起选举，获得多数票则晋升
- 节点故障检测：Gossip 协议检测，超过 `cluster-node-timeout` 无响应则标记为不可达

#### 3. 数据分片（sharding）

- **写操作**：只有槽所在的主节点接受写请求
- **读操作**：可以配置只在主节点读（默认）或从节点读（`READONLY` 命令开启）
- **MOVED 重定向**：客户端访问错误节点时，收到 MOVED 错误，客户端自动更新槽映射表

#### 4. 网络分区（脑裂）处理

- `cluster-require-full-coverage yes`：当任意槽不可用时，整个集群停止服务（默认）
- 脑裂期间可能存在数据丢失（主从切换时，旧主可能还有未同步的写操作）

### 集群限制

- 不支持跨槽的多键操作（如 `SUNION` 需要在同一个槽）
- 不支持多键事务（`MULTI/EXEC`）
- 不支持 Lua 脚本操作多个槽的键
- 不支持 `KEYS` 命令（应在客户端过滤）

### 与 Codis/Redis Cluster 对比

| 特性 | Redis Cluster | Codis |
|------|--------------|-------|
| 架构 | 无代理，客户端直连 | Proxy 代理中间件 |
| 槽数 | 16384（固定） | 1024（可配置） |
| 迁移 | 在线迁移，客户端配合 | 在线迁移，对客户端透明 |
| 成熟度 | 官方，成熟 | 豌豆荚开源，较成熟 |
| 适用场景 | 中小型集群（≤ 1000 节点） | 超大规模集群 |
