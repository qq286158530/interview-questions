## 题目一：Redis 有哪几种数据类型？分别适用于什么场景？

**参考答案：**

Redis 支持以下五种基本数据类型：

| 数据类型 | 底层实现 | 典型场景 |
|---------|---------|---------|
| **String** | SDS（简单动态字符串） | 缓存、计数器、分布式锁（SETNX） |
| **Hash** | 压缩列表或哈希表 | 对象存储（如用户信息）、购物车 |
| **List** | 压缩列表或链表 | 消息队列（LPUSH+BRPOP）、最新N条评论 |
| **Set** | 哈希表或整数集合 | 标签系统、共同好友、去重 |
| **Zset（Sorted Set）** | 压缩列表或跳跃表 | 排行榜、热搜榜、延时队列 |

**扩展类型**（Redis 3.2+）：
- **Geo**：地理位置，适用 LBS 服务
- **HyperLogLog**：UV 统计
- **BitMap**：签到统计、活跃用户统计

**答案要点：** 答出每种类型的底层实现差异（压缩列表 vs 哈希表/链表），以及为什么 Zset 用跳跃表而不是红黑树——跳跃表区间查询效率高，内存占用低于红黑树。

## 题目二：Redis 的持久化机制有哪些？各自的优缺点是什么？

**参考答案：**

Redis 有两种持久化机制：

### 1. RDB（Redis Database）

- **原理**：通过 `bgsave` 在后台生成某一时刻的内存快照，保存为 `dump.rdb` 文件。
- **优点**：恢复速度快，适合大规模数据备份；子进程完成，不阻塞主进程。
- **缺点**：可能丢失最近一次快照之后的数据；fork 子进程有内存开销，大数据量时可能卡顿。
- **配置示例**（`redis.conf`）：
  ```
  save 900 1      # 900秒内至少1次写操作则触发bgsave
  save 300 10
  save 60 10000
  ```

### 2. AOF（Append Only File）

- **原理**：每个写操作以 RESP 协议追加到文件末尾，文件只增不减。
- **优点**：数据安全性更高（可选 `always` 策略），适合做灾难恢复。
- **缺点**：文件体积通常比 RDB 大，恢复速度慢；高并发时 `rewrite` 可能阻塞。
- **配置示例**：
  ```
  appendonly yes
  appendfsync everysec   # 每秒同步（默认），可选项：always/no
  ```

### 3. 混合持久化（Redis 4.0+）

- 开启混合持久化后，AOF rewrite 时先将 RDB 内容写入 AOF 文件，再追加增量 AOF 指令。
- 兼具快速恢复和大数据安全性。

**常见追问**：
- `bgsave` 和 `bgrewriteaof` 能否同时执行？**不能**，Redis 禁止同时运行。
- 主从复制能否替代持久化？**不能**，主从复制是副本机制，持久化是本地恢复手段，两者必须共存。

## 题目三：Redis 的过期键删除策略有哪几种？

**参考答案：**

Redis 有两种过期删除策略：

### 1. 惰性删除（Lazy Expiration）

- **机制**：客户端访问某个 key 时，Redis 才检查它是否过期，过期则删除。
- **优点**：节省 CPU，不需要定时扫描。
- **缺点**：若大量过期键长期不被访问，会占用内存（内存泄漏风险）。

### 2. 定期删除（Active Expiration）

- **机制**：Redis 每隔一段时间（默认 100ms）随机抽取部分设置了过期时间的 key，检查并删除已过期的。
- **优点**：主动释放内存，控制内存上限。
- **缺点**：删除是不精准的（只抽检部分），可能有漏网之鱼。

### 3. 内存淘汰策略（Memory Eviction）

当内存达到 `maxmemory` 上限时，Redis 还会触发内存淘汰（不是删除，而是驱逐）：

| 策略 | 说明 |
|------|------|
| `volatile-lru` | 从已设置过期时间的键中，淘汰最久未使用的（LRU） |
| `allkeys-lru` | 从所有键中淘汰最久未使用的 |
| `volatile-ttl` | 淘汰剩余 TTL 最短的键 |
| `volatile-random` | 随机淘汰已设置过期时间的键 |
| `allkeys-random` | 随机淘汰所有键 |
| `noeviction` | 不淘汰，返回写操作错误（默认） |

**实际建议**：生产环境通常用 `volatile-lru` 或 `allkeys-lru`，并配合 `maxmemory` 设置内存上限。

## 题目四：Redis 主从复制原理是什么？说说你对 `cap定理` 和 `哨兵机制` 的理解。

**参考答案：**

### 主从复制原理

1. **同步（Sync）**：从服务器向主服务器发送 `SYNC` 命令；主服务器 fork 子进程执行 `bgsave`，同时将后续写命令写入缓冲区；`bgsave` 完成后将 RDB 文件发送给从服务器，从服务器加载。
2. **命令传播（Command Propagation）**：主服务器将后续写命令以 PING 方式发送给所有从服务器，保持数据一致。
3. **全量同步 → 增量同步（Redis 2.8+）**：用 `PSYNC` 替代 `SYNC`，支持断线重连后的增量同步（复制积压缓冲区），避免全量同步开销。

### 哨兵机制（Sentinel）

哨兵是 Redis 的高可用方案，核心作用：

- **监控（Monitoring）**：哨兵进程持续检测主从服务器是否存活。
- **自动故障转移（Failover）**：主服务器下线时，哨兵从从服务器中选举出新主节点，其他从服务器指向新主节点，并通知应用方。
- **配置提供者（Configuration Provider）**：客户端连接哨兵获取当前主节点地址，而非直连主节点。

> **CAP 定理**：分布式系统不可能同时满足一致性（Consistency）、可用性（Availability）和分区容忍性（Partition tolerance）。Redis 主从复制是 AP 模型——优先保证可用性和分区容忍，数据一致性在一定时间窗口内可能不一致。

## 题目五：Redis 缓存穿透、缓存击穿、缓存雪崩分别是什么？如何解决？

**参考答案：**

这是 Redis 最经典的高频面试题，三种问题的定义和解决思路各不相同：

### 1. 缓存穿透

- **定义**：查询一个**不存在**的数据，由于缓存和数据库都没有，每次请求都打到数据库。
- **危害**：大量不存在数据的请求击穿数据库。
- **解决方案**：
  - **布隆过滤器（Bloom Filter）**：在缓存层前加一层，对已存在 key 做快速判断，避免无效查询打到数据库。
  - **缓存空值**：将查不到的结果也缓存（设置短 TTL，如 60 秒）。
  - **参数校验**：对 ID 格式等做基础校验，过滤非法请求。

### 2. 缓存击穿

- **定义**：某个**热点 key 过期**的瞬间，大量并发请求同时涌入，直接打到数据库。
- **解决方案**：
  - **互斥锁（SETNX）**：只有一个线程去查数据库并重建缓存，其他线程等待。
  - **逻辑过期**：不给缓存设真正的过期时间，而是在 value 中存储逻辑过期时间，读取时异步更新。
  - **热点数据永不过期**：对极度热点数据不设过期时间，由后台任务主动更新。

### 3. 缓存雪崩

- **定义**：大量 key **在同一时刻过期**，或 Redis 本身宕机，导致数据库瞬间压力过大。
- **解决方案**：
  - **过期时间随机化**：给 key 的 TTL 加上随机值（如 `baseTTL + random(0, 300)` 秒）。
  - **多级缓存**：引入本地缓存（如 Caffeine）作为 Redis 前的二级缓存。
  - **Redis 高可用**：搭建 Redis Cluster 或 Sentinel 集群，避免单点故障。
  - **熔断/限流**：数据库层做熔断和限流，防止瞬时压力压垮数据库。
