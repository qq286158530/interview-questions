## 题目一：HBase 的整体架构是怎样的？请描述 HMaster、RegionServer、ZooKeeper 的作用及其协作关系。

### 参考答案

HBase 采用 **Master-Slave 架构**，整体由三类核心组件构成：

| 组件 | 作用 |
|------|------|
| **HMaster** | 负责 Table/Region 的元数据管理、Schema 变更、Region 负载均衡、RegionServer 失效检测与恢复。不参与实际数据读写。 |
| **RegionServer** | 负责数据读写、MemStore 与 HFile 管理、 compaction 合并、与 HDFS DataNode 交互存储数据。 |
| **ZooKeeper** | 分布式协调服务，负责 HMaster 高可用选举（active/standby）、RegionServer 心跳监测、元数据位置信息（`-ROOT-` → `-META-`）存储。 |

**协作流程：**
1. 客户端启动后，先访问 ZooKeeper 获取 `-ROOT-` 表位置；
2. 通过 `-ROOT-` 找到 `-META-` 表中目标 RowKey 所属的 Region 位置；
3. 直连对应 RegionServer 进行数据读写，无需经过 Master。

**高可用机制：** ZooKeeper 选举产生 active HMaster，宕机后可快速切换到 standby HMaster。

## 题目二：HBase 的读写流程是怎样的？请从客户端到服务端完整描述一次 Get 和一次 Put 的执行路径。

### 参考答案

### 写流程（Put）

```
1. 客户端 → ZooKeeper 获取 -ROOT- → -META- → RegionServer 地址
2. 客户端直接联系目标 RegionServer
3. RegionServer 将数据写入 WAL（Write-Ahead Log，预写日志）
4. 数据写入 MemStore（内存中的有序缓冲区）
5. 满足刷写条件后，MemStore → HFile（StoreFile）刷写到 HDFS
6. 返回客户端写入成功
```

**刷写（Flush）触发条件：**
- MemStore 大小达到 `hbase.hregion.memstore.flush.size`（默认 128MB）
- WAL 日志数量过多（`hbase.regionserver.maxlogs`）
- 定期刷写（`hbase.regionserver.flush.wait.period`）

### 读流程（Get / Scan）

```
1. 客户端通过 ZooKeeper → -META- 获取目标 RegionServer
2. RegionServer 先查 MemStore（最新数据）→ 查到则返回
3. 未查到则查 BlockCache（LRU 缓存，存储热点 HFile 数据块）
4. 仍未查到则读取 HFile（通过 BloomFilter 跳过无关文件）合并返回
```

**Merge（读合并）：** 由于数据可能分布在多个 HFile 中，读取时需要将各 HFile 结果与 MemStore 合并，形成最终结果。

## 题目三：HBase 的 RowKey 设计原则有哪些？如何设计 RowKey 以实现高效查询和避免热点？

### 参考答案

### RowKey 设计核心原则

**1. 唯一性原则**
RowKey 必须唯一标识一行数据，类似主键，通常由业务主键拼接构成。

**2. 散列（避免热点）原则**
避免连续 RowKey 导致所有请求打到同一 RegionServer（热点写入）。常用方法：

| 方法 | 说明 |
|------|------|
| **反转（Reversal）** | 将手机号 `13800138000` 反转为 `00008310083` |
| **哈希（Hashing）** | `MD5(rowKey).substring(0, 5) + originalKey` |
| **随机前缀** | 拼接随机数或时间戳打散 |
| **高位取模** | 取 RowKey 哈希后高位计算 Region 分布 |

**3. 长度控制**
- 推荐 ≤ 16 字节（Java 中 `byte[]` 数组），最大 64KB
- 越短越好，减少内存和 HDFS 存储开销

**4. 查询维度前置（利用字典序特性）**
HBase 按 RowKey 字典序排序，Scan 范围查询依赖前缀匹配：
```
# 适合范围查询：将时间戳放在 RowKey 末尾
orderId + timestamp   # 前缀匹配 orderId 可快速定位
userId + deviceId     # 适合查询某用户的所有设备记录
```

**5. 避免使用枚举或固定值**
枚举类 RowKey（如 `status=1`）会导致数据集中分布到少数 Region。

### 热点案例

```
# ❌ 热点 RowKey：时间戳开头
20260620100001:data  → 所有新数据集中在最新 Region

# ✅ 哈希打散 RowKey
MD5(20260620100001).substring(0,4) + 20260620100001:data
```

## 题目四：MemStore 和 HFile 是如何工作的？为什么 HBase 支持高并发随机写？

### 参考答案

### 核心原理

HBase 之所以支持高并发随机写，核心在于 **"顺序写 MemStore + 批量刷写 HFile"** 的机制：

**MemStore（内存写）**
- 每个 Store（列族）对应一个 MemStore，按 RowKey 字典序排序存储
- 所有写入先入 MemStore（内存顺序写入，极快）
- 写入 WAL 保证宕机可恢复

**HFile（磁盘持久化）**
- 当 MemStore 达到阈值，刷写为 HFile（StoreFile）
- HFile 是 **Hadoop SequenceFile** 格式，按 KeyValue 紧凑存储
- 数据按 RowKey 排序，支持高效范围 Scan

### 为什么高并发随机写快？

| 对比 | 传统关系型数据库 | HBase |
|------|----------------|-------|
| 写入方式 | 随机 IO（直接更新 B+树索引） | 顺序写 MemStore（内存） |
| 索引维护 | 实时更新聚簇索引 | 无（数据按 RowKey 排序写入） |
| 并发写入 | 锁竞争严重 | 写入 MemStore 无锁并发 |

### compaction 合并

随着时间推移，一个 Store 下会产生多个 HFile，影响读性能。HBase 通过 **compaction** 合并：

- **Minor Compaction**：选取几个小 HFile 合并为一个（保留最新版本）
- **Major Compaction**：一个 Store 下所有 HFile 合并为一个，清理删除和过期数据

刷写与 compaction 形成 **"小批量顺序写代替随机写"**，这是 HBase 高性能的核心。

## 题目五：HBase 协处理器（Coprocessor）是什么？Observer 和 Endpoint 的区别和使用场景是什么？

### 参考答案

### 协处理器概述

HBase 协处理器是运行在 RegionServer 上的扩展框架，允许用户在 **数据写入/读取/查询** 的生命周期中注入自定义逻辑，无需修改 HBase 核心代码。

### 两类协处理器对比

#### 1. Observer（观察者）— 类似数据库触发器

**Hook 机制：** 在指定事件前后自动触发

| 类型 | Hook 点 | 场景 |
|------|---------|------|
| **RegionObserver** | Get/Put/Delete/Scan 等操作前后 | 数据校验、访问控制、实时过滤 |
| **WALObserver** | WAL 写入前后 | WAL 加密/压缩 |
| **MasterObserver** | Table/DDL 操作前后 | 权限审计、Schema 变更通知 |

**示例场景：**
```java
// 需求：写入数据时自动对敏感字段加密
public class EncryptObserver extends RegionObserver {
    @Override
    public void prePut(ObserverContext<RegionCoprocessorEnvironment> c,
                       Put put, WALEdit edit, Durability durability) {
        // 在写入 MemStore 前，对特定列族/列进行加密
    }
}
```

#### 2. Endpoint（端点）— 类似数据库存储过程

**主动调用机制：** 客户端主动调用 RegionServer 上的自定义方法

- 使用 **Protobuf** 定义 RPC 接口
- Endpoint 在每个 Region 上执行，结果返回客户端
- 客户端负责聚合所有 Region 的返回结果

**典型使用场景：**
- **自定义聚合**：在每个 Region 上计算 Sum/Count/Max，然后客户端汇总（如求某列的总和）
- **二级索引**：数据变更时自动更新索引表
- **动态过滤**：在 Region 侧做复杂条件过滤，减少网络传输

**与 Observer 的核心区别：**
| 对比 | Observer | Endpoint |
|------|----------|----------|
| 触发方式 | 被动（事件驱动） | 主动（客户端调用） |
| 返回结果 | 仅修改数据，无返回值 | 执行后返回数据给客户端 |
| 数据范围 | 仅当前 Region | 可跨多个 Region 并行执行 |

### 使用注意事项

- 协处理器代码随 Region 加载，需注意 **版本兼容性** 和 **线程安全**
- 避免在 Hook 中执行耗时操作，否则影响读写延迟
- 建议使用 **Observer 做轻量逻辑**（如过滤、校验），**Endpoint 做复杂聚合**
