## 题目一：Spark 的架构设计与核心组件

### 问题
请描述 Spark 的整体架构，以及 Driver、Executor、Cluster Manager、Worker Node 各自的作用和协作关系。

### 答案

**Spark 架构概览**

Spark 采用主从架构，核心组件如下：

| 组件 | 作用 |
|------|------|
| **Driver（驱动程序）** | 负责任务调度（DAG 切分 Stage、生成 TaskSet）、与 Cluster Manager 通信申请资源。用户代码运行在 Driver 进程中。 |
| **Executor（执行器）** | 运行在 Worker 节点上的进程，负责执行分发的 Task，将结果写入存储或回传给 Driver。一个 Application 可以有多个 Executor。 |
| **Cluster Manager（集群管理器）** | 负责集群资源的管理和分配。常见实现：Standalone（Spark 自带）、YARN、Mesos、Kubernetes。 |
| **Worker Node** | 集群中的实际计算节点，运行 Executor 进程。 |

**协作流程：**
1. 用户提交 Spark Application，Driver 启动并创建 SparkContext。
2. SparkContext 向 Cluster Manager 申请 Executor 资源。
3. Cluster Manager 在 Worker 节点上启动 Executor 进程。
4. Driver 将应用代码发送给 Executor，并将 Task 分发到各 Executor 执行。
5. Executor 执行 Task，将结果汇报给 Driver。
6. 应用结束后，Driver 向 Cluster Manager 注销资源。

**补充：** Spark 在 YARN 模式下还有 **ApplicationMaster** 角色，负责与 YARN ResourceManager 协商资源。

## 题目二：RDD 的特性与 lineage（血统）机制

### 问题
什么是 RDD？为什么 Spark 选择 RDD 作为核心抽象？RDD 的 lineage（血统）机制是如何工作的，它在容错中起到什么作用？

### 答案

**RDD（Resilient Distributed Dataset）** 是 Spark 最核心的数据抽象，表示一个弹性的、分布式的数据集，具有以下特性：

| 特性 | 说明 |
|------|------|
| **分区（Partitioned）** | 数据集逻辑上划分为多个分区，每个分区分布在集群不同节点 |
| **只读（Immutable）** | RDD 创建后不可修改，只能通过 transformations 生成新的 RDD |
| **血缘（Lineage）** | 每个 RDD 记录了其父 RDD 的依赖关系，形成 DAG |
| **容错（Fault Tolerant）** | 如果某个分区丢失，通过 lineage 信息重新计算恢复，无需全局快照 |

**Lineage 机制的工作原理：**

```
RDD A → map → RDD B → filter → RDD C → collect → RDD D
```

当 RDD D 的某个分区丢失时：
1. Spark 从 lineage DAG 中回溯，找到 RDD A（源头分区）。
2. 从 RDD A 开始，沿路径重新执行 transformations，恢复丢失的分区。
3. 不需要像 MapReduce 那样做 Checkpoint 或将数据写回磁盘。

**为什么选择 RDD 而不是 DSM（Distributed Shared Memory）：**
- **只读模型简化了容错**：无需记录所有写入操作，只需记录血统。
- **lineage 实现细粒度恢复**：只重算丢失的分区，而不是整个作业。
- **粗粒度 transformations**：适合数据并行场景，性能更优。

**注意：** RDD 的容错机制不适用于 shuffle 过程中丢失的数据（因为 Shuffle 中间文件可能未持久化），此时需要启用 Checkpoint。

## 题目三：DataFrame 和 DataSet 的区别与联系

### 问题
Spark 中 DataFrame 和 DataSet 有什么区别？它们与 RDD 相比有哪些优势？在什么场景下应该优先选择 DataFrame 或 DataSet？

### 答案

**概念对比**

| | DataFrame | DataSet |
|---|---|---|
| **数据结构** | `DataFrame = Dataset[Row]` | 强类型的 `Dataset[T]` |
| **类型安全** | 无编译时类型检查（Row 为通用类型） | 有编译时类型检查 |
| **API 风格** | 纯 DSL（函数式/链式调用） | 兼具 SQL 和 OOP 风格 |
| **序列化** | Tungsten（Binary + off-heap） | Encoders |
| **操作方式** | `.select()`, `.filter()` 等 | `.map()`, `.filter()` + Lambda |

**DataFrame / DataSet 相比 RDD 的优势：**

1. **性能优化（Catalyst 优化器 + Tungsten）**
   - Catalyst 对执行计划进行两阶段优化：Logical Plan → Physical Plan → 选择最优执行策略。
   - 自动进行谓词下推（Predicate Pushdown）、列裁剪（Column Pruning）、常量替换等优化。
   - Tungsten 使用 off-heap 内存和二进制格式，减少 GC 开销和序列化开销。

2. **易用性**
   - 支持 SQL 语法，Python/R 用户无需学习 Scala 类型系统。
   - 统一的 API 同时支持 DSL 和 SQL 查询。

3. **统一的数据处理**
   - 支持结构化数据（通过 Spark SQL），处理起来更直观。

**选择建议：**

| 场景 | 推荐 |
|------|------|
| Python/Java 用户，处理结构化数据 | **DataFrame**（无编译时检查，灵活） |
| Scala 用户，需要类型安全 | **DataSet**（编译期检查，更安全） |
| 需要极致性能（结构化数据） | **DataFrame/DataSet**（Catalyst 优化） |
| 非结构化数据、低层次控制 | **RDD** |
| 需要函数式 API + 类型安全的混合 | **DataSet** |

**趋势：** Spark 3.x 中 DataFrame 和 DataSet 已逐渐统一，大多数场景推荐使用 DataFrame，除非需要强类型。

## 题目四：Spark Shuffle 机制详解

### 问题
什么是 Shuffle？为什么 Spark 需要 Shuffle？Shuffle 过程中数据是如何存储和传输的？有哪些常见的优化策略？

### 答案

**什么是 Shuffle**

Shuffle 是 Spark 将数据按某种策略重新分区（Repartition）的过程。**所有需要跨节点重新分配数据的操作都会触发 Shuffle**，典型操作包括：
- `repartition()`、`coalesce()`
- `groupByKey()`、`reduceByKey()`（但 `reduceByKey` 会在 map 端先做本地聚合，优化过）
- `join()`、`cogroup()`
- `distinct()`、`intersection()` 等

**为什么需要 Shuffle**

某些 operation 的输出数据的分区方式，取决于输入数据的分布。例如 `groupByKey` 需要将所有相同 key 的数据汇聚到同一个节点，不经过 shuffle 无法完成跨节点数据移动。

**Shuffle 的工作流程（Spark 2.x+）**

```
Map Task（写） → Shuffle File（本地磁盘） → Reduce Task（读）
```

1. **Map 阶段**：Map Task 将数据写入本地磁盘，生成 shuffle file（按 reducer 分区，每个 reducer 一个文件）。文件格式为 `bucket`，使用排序或 hash 方式组织。
2. **Shuffle Fetch**：Reduce Task 通过网络从各 Map Task 所在的节点拉取属于自己的分区数据（HTTP / Block Manager）。
3. **Reduce 阶段**：Reduce Task 对拉取的数据进行聚合、排序等操作。

**常见优化策略：**

| 优化项 | 说明 |
|------|------|
| **Map 端聚合** | 使用 `reduceByKey` 代替 `groupByKey`，减少数据量 |
| **Consolidate Shuffle Files** | Spark 1.5+ 支持将同一 Executor 内多个 Task 的 shuffle 文件合并为少量文件，减少文件数 |
| **Sort-based Shuffle** | 使用 SortShuffleWriter，对每个分区数据排序后写入，替代 hash-based，减少内存压力 |
| **Bypass Sort Shuffle** | 当 reducer 数量 ≤ `spark.shuffle.sort.bypassMergeThreshold`（默认200）时，使用 bypass 模式跳过排序 |
| **广播小表** | 小表可以在 map 端广播（Broadcast Hash Join），避免 shuffle |
| **合理设置分区数** | 分区过多导致小文件问题，分区过少导致数据倾斜 |

**Shuffle 的性能瓶颈：** 网络 I/O、磁盘 I/O、 JVM GC（大量读写时）。

## 题目五：Spark 内存管理与广播变量

### 问题
请详细描述 Spark 的内存管理模型（堆内 vs 堆外内存），以及广播变量（Broadcast Variable）的工作原理和使用场景。

### 答案

**Spark 内存管理模型（Spark 2.x+ Unified Memory）**

Spark 统一管理两片内存区域：**Execution（执行）内存** 和 **Storage（存储）内存**。

| 内存区域 | 用途 |
|------|------|
| **Execution Memory** | 用于 Shuffle、排序、Hash Join 等计算的临时内存 |
| **Storage Memory** | 用于缓存 RDD、广播变量、Internal Meta 信息 |
| **User Memory** | 用户代码的内存分配（非 Spark 管理） |
| **Reserved Memory** | 预留内存（默认 300MB），用于系统预留 |

**堆内（On-heap）和堆外（Off-heap）内存：**

- **堆内内存**：由 JVM 管理的堆空间，受 GC 影响。
- **堆外内存**：直接向 OS 申请的堆外空间（通过 `spark.memory.offHeap.enabled=true`），避免 GC 干扰，适合需要精确控制内存的场景。

**内存动态共享：** Execution 和 Storage 内存可以互相借用（soft boundary）。当 Execution 内存不足时，可以抢占 Storage 内存；反之亦然。但 Storage 内存被占用后，需要等待 Execution 释放才能恢复。

**内存配置关键参数：**
```scala
spark.memory.fraction = 0.6       // Spark 内存占总内存的比例
spark.memory.storageFraction = 0.5 // Storage 内存占 Spark 内存的比例
spark.memory.offHeap.enabled = true // 启用堆外内存
```

---

**广播变量（Broadcast Variable）**

**原理：** 广播变量将只读的、小而常用的数据封装成一个对象，发送到每个 Executor 的内存中，所有 Task 共享这份数据，避免每个 Task 都持有一份副本，减少网络传输和内存消耗。

**使用方式：**
```scala
// 创建广播变量
val broadcastVar = sc.broadcast(Array(1, 2, 3, 4, 5))

// 在 Task 中使用（只读）
rdd.map(x => x * broadcastVar.value.sum)

// 销毁广播变量（不再使用时）
broadcastVar.unpersist()
```

**工作机制：**
1. Driver 端调用 `.broadcast()` 时，将数据序列化并分发到各 Executor。
2. Executor 收到后存放在 `BlockManager` 中（Storage Level = MEMORY_AND_DISK）。
3. Task 在执行时通过 `broadcastVar.value` 读取（无需网络传输）。
4. 广播变量一旦创建，**不可修改**（只读语义）。

**使用场景：**

| 场景 | 说明 |
|------|------|
| **大表 join 小表** | 将小表广播到每个节点，在 map 端完成 join（Broadcast Hash Join），避免 shuffle |
| **静态配置数据** | 维度表、配置表、字典表等只读数据 |
| **函数闭包中的大对象** | 避免闭包序列化放大（每个 Task 序列化一份） |
| **机器学习特征向量** | 训练时共享的特征权重、模型参数 |

**注意事项：**
- 广播变量适合**小而常用**的数据；过大的数据广播会撑爆 Executor 内存。
- 使用 `spark.sql.autoBroadcastJoinThreshold`（默认 10MB）可让 Spark 自动对小表进行广播。
- 广播后数据不可更改，如果数据更新需要重新广播。
