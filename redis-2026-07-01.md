## 题目一：什么是缓存穿透、缓存击穿和缓存雪崩？如何解决？

### 参考答案

**缓存穿透：**

指查询一个**不存在的数据**，由于缓存中不存在，每次都要查询数据库。如果有人恶意大量请求不存在的数据，会导致数据库压力过大。

**解决方案：**
- **布隆过滤器（Bloom Filter）**：在缓存层前加一个布隆过滤器，将所有存在的数据哈希到一个 bitmap 中，请求过来先查布隆过滤器，如果不存在直接返回
- **缓存空值**：即使查询结果为空，也将空值缓存起来，并设置较短的过期时间
- **接口参数校验**：做好参数校验，减少非法请求

---

**缓存击穿：**

指**某个热点 key 过期**的瞬间，大量并发请求直接打到数据库，导致数据库压力激增。

**热点 key**：通常是访问频率非常高的 key，比如促销商品、热搜新闻等。

**解决方案：**
- **互斥锁（Mutex Key）**：缓存失效时，不立即更新，先获取互斥锁，只有获得锁的线程去更新缓存，其他线程等待缓存更新后重新读取
- **热点数据永不过期**：对热点 key 设置较长的过期时间或定期更新
- **逻辑过期**：不设置真正的过期时间，而是在 value 中记录逻辑过期时间，到期后用单独线程异步重建缓存

---

**缓存雪崩：**

指**大量缓存 key 在同一时间过期**，导致大量请求同时打到数据库。

**两种场景：**
1. **大量 key 集中过期**： Redis 中的 key 批量过期
2. **Redis 宕机**：整个缓存服务不可用

**解决方案：**

针对第一种（key 过期）：
- **过期时间随机化**：给缓存过期时间加上随机值，避免同一时间大量 key 过期
- **热点数据不设置过期时间**：让热点数据永久有效
- **后端服务实现熔断/限流**：当数据库压力过大时，限制部分请求

针对第二种（Redis 宕机）：
- **Redis 集群部署**：保证高可用
- **本地缓存 + Redis 双保险**：使用本地缓存作为兜底
- **服务降级**：当 Redis 不可用时，返回默认值或友好提示

## 题目二：Redis 的过期键删除策略和内存淘汰策略有什么区别？

### 参考答案

这是两个不同层面的机制，必须区分清楚：

**一、过期键删除策略（定时删除 + 惰性删除 + 定期删除）**

针对「key 何时被删除」，Redis 有三种策略：

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **定时删除** | 设置 key 的过期时间时，同时创建一个定时器（Timer），到期立即删除 | 对内存最友好，保证过期 key 及时被删除 | 对 CPU 不友好，过期键多时占用 CPU 时间 |
| **惰性删除（Lazy Free）** | 每次访问 key 时才检查是否过期，过期则删除 | 对 CPU 友好，只在访问时检查 | 过期 key 长时间不访问会一直占用内存 |
| **定期删除** | 每隔一段时间，扫描一定数量的 key，删除其中过期的 | 折中方案，CPU 和内存兼顾 | 难以确定合理的扫描间隔和数量 |

Redis 默认采用 **惰性删除 + 定期删除** 组合策略。

---

**二、内存淘汰策略（当内存达到 maxmemory 时触发）**

针对「内存不足时哪些 key 该被清理」，Redis 8.0 提供了多种策略：

**基于淘汰方式的策略：**

| 策略 | 说明 |
|------|------|
| `noeviction` | 不淘汰任何 key，满后返回错误（默认） |
| `allkeys-lru` | 所有 key 中淘汰最近最少使用的（LRU） |
| `allkeys-lfu` | 所有 key 中淘汰使用频率最少的（LFU，Redis 4.0+） |
| `allkeys-random` | 所有 key 中随机淘汰 |
| `volatile-lru` | 已设置过期时间的 key 中淘汰 LRU |
| `volatile-lfu` | 已设置过期时间的 key 中淘汰 LFU（4.0+） |
| `volatile-random` | 已设置过期时间的 key 中随机淘汰 |
| `volatile-ttl` | 已设置过期时间的 key 中淘汰 TTL 最短的（即最早过期的） |

**选择建议：**
- 如果希望保证所有数据都不丢失 → `noeviction`
- 如果主要缓存非热点数据 → `allkeys-lru` 或 `allkeys-lfu`
- 如果只淘汰要过期的数据 → `volatile-lru` / `volatile-lfu`
- 如果只作为临时缓存，不需要旧数据 → `allkeys-random`

**LRU vs LFU：**
- **LRU（Least Recently Used）**：最近最少使用，只看访问时间
- **LFU（Least Frequently Used）**：最不常使用，看访问频率，Redis 4.0 引入，能更好地识别真正的热点数据

## 题目三：如何用 Redis 实现分布式锁？Redlock 又是怎么回事？

### 参考答案

**一、为什么需要分布式锁？**

在单机多线程环境下，Java 的 `synchronized` 或 `ReentrantLock` 可以保证线程安全。但在分布式环境下，不同机器上的进程需要协调，就必须用分布式锁。

**二、Redis 分布式锁的实现原理**

核心思想：**用 SET NX EX 命令抢锁，谁抢到谁持有。**

```redis
# 加锁（原子操作，防止误删他人的锁）
SET lock_key unique_value NX EX 30

# 释放锁（必须判断是自己加的锁才能删除，防止误删）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**关键点：**
- `NX`：保证只有一个客户端能加锁成功（原子性）
- `EX 30`：设置过期时间，防止持有锁的客户端崩溃后锁无法释放
- `unique_value`：唯一标识，用于识别锁是谁加的，释放时做校验

**三、锁的自动续期**

如果持锁时间超过 TTL，但业务还没执行完怎么办？
→ 可以用 watchdog 机制：持有锁的客户端每 1/3 TTL 续期一次，直到业务完成主动解锁。

**四、单机分布式锁的问题**

如果 Redis 主从架构中，主节点加了锁但还没同步给从节点就宕机了，会导致锁丢失。

**五、Redlock（红锁）解决方案**

Redlock 由 Redis 作者 Antirez 提出，原理：

1. 选取 N（N≥5）个独立的 Redis 节点
2. 按顺序在每个节点上使用相同的 key 和 value 加锁
3. 计算获取锁的总时间，如果能在过半节点成功加锁，且总耗时小于锁的 TTL，则认为加锁成功
4. 如果获取锁失败，立刻在所有节点上释放锁

```
成功条件：
- 在 N/2+1 个节点上成功加锁
- 总耗时 < TTL
- 已加锁成功的节点数 >= N/2+1
```

**Redlock 的争议：**
- 需要部署多个独立 Redis 实例，成本高
- 部分分布式系统专家认为，在时钟漂移严重的场景下 Redlock 不可靠
- 建议优先使用成熟的分布式锁组件（如 Redisson、ZooKeeper）

## 题目四：Redis 事务是什么？MULTI/EXEC/WATCH 是如何工作的？

### 参考答案

**Redis 事务是什么？**

Redis 事务是将一系列命令打包，然后一次性、按顺序执行，在执行期间不会插入其他客户端命令。

**Redis 事务的三个阶段：**

1. **开启事务** (`MULTI`)：客户端进入事务状态
2. **命令入队**：所有后续命令进入队列但不会立即执行
3. **执行事务** (`EXEC`)：一次性执行队列中的所有命令

```
redis> MULTI
OK
redis> SET name "Alice"
QUEUED
redis> GET name
QUEUED
redis> INCR counter
QUEUED
redis> EXEC
1) OK
2) "Alice"
3) (integer) 1
```

**WATCH 命令——乐观锁机制：**

`WATCH key [key ...]` 用于监视一个或多个 key，在 EXEC 执行前，如果这些 key 被其他客户端修改过，事务自动取消并返回空。

```redis
WATCH account_balance
GET account_balance
# 此时其他客户端可能修改了 balance
MULTI
SET account_balance 1000
EXEC  # 如果 balance 被改过，这里返回 nil（事务取消）
```

**应用场景：** 扣款场景，先 WATCH 余额，有变化则重试。

**Redis 事务的三个重要特性：**

1. **无隔离级别**：Redis 事务不支持回滚，命令入队错误不阻止后续命令入队
2. **不保证原子性**：事务中某条命令失败不会影响其他命令执行（与 MySQL 不同）
3. **不支持回滚**：Redis 设计哲学是「简单高效」，不支持真正的回滚

**Redis 事务 vs Lua 脚本：**

| 特性 | Redis 事务 | Lua 脚本 |
|------|-----------|---------|
| 原子性 | 无真正原子性 | 有真正原子性（整个脚本执行完才处理其他命令） |
| 回滚 | 不支持 | 不支持 |
| 灵活性 | 动态入队 | 脚本固定 |
| 性能 | 一般 | 更优（减少网络往返） |
| 适用场景 | 简单批量操作 | 复杂业务逻辑 |

## 题目五：Redis 的 BigKey 是什么？会产生什么问题？如何发现和优化？

### 参考答案

**什么是 BigKey？**

BigKey 是指 Redis 中存储的数据量过大的 key，通常指：
- String 类型 value > 10MB
- List/Set/ZSet 元素数 > 10000 个
- Hash 字段数过多

**BigKey 会产生的问题：**

1. **内存不均**：单个 key 占用内存过大，导致 Redis 内存分配不均衡
2. **操作阻塞**：对 BigKey 执行 `DEL`、`HGETALL`、`SMEMBERS` 等操作会阻塞主线程（尤其是主从架构下，从节点也会被阻塞）
3. **网络阻塞**：读取 BigKey 会产生大量流量
4. **迁移困难**：集群环境下，BigKey 会导致节点 rehash 时卡顿

**如何发现 BigKey？**

```bash
# 方法1：redis-cli --bigkeys 统计各类型的最大 key
redis-cli --bigkeys

# 方法2：使用 SCAN + OBJECT ENCODING 分析
redis-cli --scan | head -10000 | xargs -r redis-cli pipeline --score 0 | grep -v '^[0-9]* '

# 方法3：MEMORY USAGE 查看 key 内存占用
redis-cli MEMORY USAGE big_key_name

# 方法4：Redis 5.0+ 支持 Redis-cli --bigkeys
redis-cli --bigkeys -i 0.1  # 每 0.1 秒采样
```

**如何优化 BigKey？**

1. **拆分**：将大 key 拆成小 key
   - 例如：`user:1000:followers`（10000个粉丝）→ `user:1000:followers:1`、`user:1000:followers:2`（每组1000个）

2. **使用合适的数据结构**：
   - String → 压缩或拆分
   - Hash → 定期检查 field 数量，考虑拆分为多个 Hash
   - List → 控制长度上限

3. **lazy delete**：删除 BigKey 时用 `UNLINK` 替代 `DEL`，异步在后台线程释放内存，不阻塞主线程

4. **设置合理过期时间**：不常用的大 key 及时删除

5. **监控告警**：通过 `redis-cli --bigkeys` 定期扫描，对接近阈值的 key 提前处理
