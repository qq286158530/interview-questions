## 题目一：Redis 的数据类型有哪些？各自的使用场景？

**参考答案：**

Redis 支持 5 种基础数据类型和 3 种高级数据结构：

| 类型 | 数据结构 | 典型使用场景 |
|------|---------|-------------|
| **String** | 简单字符串、整数、浮点数 | 缓存、计数器、分布式锁（SETNX）、限流 |
| **Hash** | 键值对哈希表 | 存储对象（如用户信息、商品详情） |
| **List** | 有序可重复的链表 | 消息队列（LPUSH+BRPOP）、最新列表、时间线 |
| **Set** | 无序不重复集合 | 标签系统、去重、共同关注、共同好友 |
| **Zset** | 有序不重复集合（score 排序） | 排行榜、延时队列、权重队列 |

**3 种高级数据结构：**
- **HyperLogLog**：基数统计（UV 统计），空间 O(1)
- **Bitmap**：位图，适合签到、状态监控
- **Geospatial**：地理位置附近的人检索

**常见追问：**
- String 最大值是多少？ → **512MB**
- Zset 的 score 可以重复吗？ → 可以，member 唯一
- 为什么 Redis 的 List 既可以做队列又可以做栈？ → LPUSH/LPOP 是栈，LPUSH/BRPOP 是队列

**来源：** Redis 官方文档 https://redis.io/docs/data-types/

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

**参考答案：**

Redis 提供了两种持久化机制：

### RDB（Redis Database）
- **原理**：定时将内存中的全量数据快照（point-in-time snapshot）写入磁盘，生成 `dump.rdb` 文件
- **触发方式**：
  - 定时触发（`save` / `bgsave` 命令）
  - 自动触发（`stop-writes-on-bgsave-error` + 条件）
  - 主从复制时自动触发
- **优点**：恢复速度快（全量恢复），适合备份/灾难恢复
- **缺点**：可能丢失最后一次快照后的数据（数据丢失窗口）

### AOF（Append Only File）
- **原理**：将每个写操作追加到日志文件末尾，重启时重放日志恢复数据
- **策略**（`appendfsync`）：
  - `always`：每条命令同步刷盘，数据最安全但性能最差
  - `everysec`（默认）：每秒刷盘，最多丢失 1 秒数据
  - `no`：依赖 OS 刷盘，性能最好但可能丢失较多数据
- **优点**：数据安全性更高，支持重写（rewrite）压缩文件
- **缺点**：文件体积大，恢复速度慢

### 混合持久化（Redis 4.0+）
- 开启 `aof-use-rdb-preamble yes` 后，AOF 重写时先用 RDB 格式写入，再追加增量 AOF 日志
- 兼具 RDB 快速恢复和 AOF 数据完整性的优点

### 如何选择？

| 场景 | 推荐策略 |
|------|---------|
| 数据安全性要求极高 | AOF + everysec/always |
| 允许分钟级数据丢失 | RDB |
| 做备份/容灾 | RDB + AOF 双开 |
| 容器/无持久化需求 | 关闭持久化（`appendonly no`）|

**来源：** Redis 持久化官方文档 https://redis.io/docs/management/persistence/

## 题目三：Redis 的过期键删除策略有哪几种？

**参考答案：**

Redis 有**三种过期键删除策略**：

### 1. 定时删除（主动删除）
- **原理**：对每个设置了过期时间的 key，创建定时器，到期立即删除
- **优点**：内存友好，键一过期立即删除，释放内存
- **缺点**：CPU 不友好，如果过期 key 很多，会占用大量 CPU 时间
- **实际应用**：Redis **不采用**这种方式（太耗性能）

### 2. 惰性删除（被动删除）
- **原理**：每次访问 key 时检查是否过期，过期则删除
- **优点**：对 CPU 友好，只在访问时才检查
- **缺点**：对内存不友好，过期键长期占用内存（内存泄漏风险）
- **实际应用**：Redis **采用**，通过 `expireIfNeeded()` 函数实现

### 3. 定期删除（主动+被动折中）
- **原理**：每隔一段时间（默认 100ms），随机检查一部分过期 key 进行删除
- **流程**：
  1. 从所有带过期时间的 key 中随机抽取 `ACTIVE_EXPIRE_CYCLE_LOOKUPS_PER_LOOP`（默认 20）个
  2. 删除其中过期的 key
  3. 如果过期 key 超过 25%，继续抽取检查
- **优点**：折中 CPU 和内存压力
- **实际应用**：Redis **采用**，是惰性删除的补充

### 内存淘汰策略（当内存达到上限时）
当内存不足时，Redis 还会通过 **8 种淘汰策略** 主动淘汰 key：

| 策略 | 说明 |
|------|------|
| `volatile-lru` | 从已设置过期时间的 key 中淘汰最近最少使用 |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用 |
| `volatile-lfu` | 从已设置过期时间的 key 中淘汰最不经常使用 |
| `allkeys-lfu` | 从所有 key 中淘汰最不经常使用 |
| `volatile-random` | 从已设置过期时间的 key 中随机淘汰 |
| `allkeys-random` | 从所有 key 中随机淘汰 |
| `volatile-ttl` | 淘汰即将过期的 key |
| `no-eviction` | 不淘汰，返回错误（默认） |

**来源：** Redis 过期键官方文档 https://redis.io/docs/data-types/expires/

## 题目四：Redis 主从复制原理是什么？如何处理故障转移？

**参考答案：**

### 主从复制架构
Redis 支持一主多从的复制架构，主节点（master）负责写，从节点（slave）负责读（读写分离）。

### 复制原理（3 个阶段）

#### 阶段 1：建立连接 & 数据同步
1. 从节点保存主节点信息（`replicaof` / `slaveof` 命令）
2. 从节点向主节点发送 `PING`，确认主节点可达
3. **全量同步**（`PSYNC` / 旧版 `SYNC`）：
   - 从节点发送 `PSYNC ? -1`
   - 主节点执行 `bgsave`，生成 RDB 文件
   - 主节点将 RDB 文件发送给从节点
   - 从节点加载 RDB 文件完成全量数据同步
4. **增量同步**：主节点将同步期间产生的写命令（replication buffer）发送给从节点

#### 阶段 2：命令传播（长连接）
- 主节点维护一个 **replication stream**（命令传播队列）
- 每次执行写命令后，通过长连接将命令同步给从节点
- 从节点持续接收并执行这些命令，保持与主节点数据一致

#### 阶段 3：断线重连
- 从节点断开后重新连接，尝试增量同步（`PSYNC <runid> <offset>`）
- 若主节点的 replication buffer 还在（未过期），则增量同步
- 若 buffer 已清除，则触发全量同步

### 故障转移（哨兵 / Sentinel）
当主节点宕机时，Sentinel 集群自动完成故障转移：

1. **主观下线**：Sentinel 检测到主节点无响应（`down-after-milliseconds`）
2. **客观下线**：多数 Sentinel 都认为主节点下线
3. **选举 leader**：Raft 协议选举出一个 Sentinel 节点作为 leader
4. **选出新主**：leader 从从节点中选出一个「数据最新的」作为新主
5. **通知应用**：Sentinel 广播新主节点地址，客户端通过 Sentinel 获取新地址

### 选新主的优先级（从高到低）：
1. 排除网络断开的从节点
2. 选择 `priority` 配置最高的从节点
3. 选择 `offset`（复制偏移量）最大的从节点（数据最新）
4. 选择 `runid` 最小的从节点

**来源：** Redis 主从复制官方文档 https://redis.io/docs/management/replication/

## 题目五：什么是缓存穿透、缓存击穿、缓存雪崩？如何解决？

**参考答案：**

这是面试中最常考的高并发场景题，合称「缓存三穿透」。

### 1. 缓存穿透
**定义**：查询一个**不存在**的数据，由于缓存没有命中，去查数据库，数据库也没有，导致每次请求都打到数据库。

**危害**：恶意攻击者用大量不存在的 ID 请求，直接击垮数据库。

**解决方案：**
- **布隆过滤器（Bloom Filter）**：将所有存在的 key 存入布隆过滤器，查询前先过滤掉不存在的 key
- **缓存空值（null）**：将查询结果为空的数据也缓存起来，key = 唯一标识，value = null，并设置较短过期时间（如 30 秒）
- **参数校验**：对请求参数做基础合法性校验（如 ID > 0）

### 2. 缓存击穿
**定义**：某个**热点 key 突然过期**，大量请求同时涌入，直接打到数据库。

**危害**：瞬时数据库压力过大，可能导致数据库崩溃。

**解决方案：**
- **互斥锁（SETNX）**：缓存失效时，只允许一个请求去查数据库并重建缓存，其他请求等待
- **热点数据永不过期**：不设置过期时间，改为「逻辑过期」（value 中包含过期时间戳，访问时异步更新）
- **提前重建**：异步任务提前发现热点 key 即将过期，主动重建

### 3. 缓存雪崩
**定义**：大量缓存 key 在**同一时间集中过期**，导致大量请求打到数据库。

**危害**：数据库瞬间压力过大，与缓存击穿类似但规模更大。

**解决方案：**
- **过期时间随机化**：给缓存过期时间加上随机值（如 `TTL + random(0, 3600)`），避免同时过期
- **热点数据分散**：避免将大量相关数据集中在同一时间点过期
- **多级缓存**：本地缓存 + Redis 多级缓存架构，减少对 Redis 的直接压力
- **服务降级/熔断**：当数据库压力过大时，返回兜底数据（fallback）
- **Redis 高可用**：使用 Redis Cluster / Sentinel 保证缓存服务可用性

### 对比总结

| 现象 | 原因 | 关键词 | 解决方案 |
|------|------|--------|---------|
| 缓存穿透 | 查询不存在的数据 | 查不到 | 布隆过滤器、缓存空值、参数校验 |
| 缓存击穿 | 热点 key 过期 | 集中查一个 | 互斥锁、逻辑过期、提前重建 |
| 缓存雪崩 | 大量 key 同时过期 | 集中查一批 | 随机过期时间、多级缓存、熔断降级 |

**来源：** Redis 缓存设计实践 https://redis.io/docs/manual/patterns/
