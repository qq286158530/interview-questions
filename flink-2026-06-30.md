## 题目一：Flink 的架构是怎样的？请描述 Flink 的核心组件及其职责

### 参考答案

Flink 采用 **主从架构**，核心组件包括：

| 组件 | 进程 | 职责 |
|------|------|------|
| **JobManager** | Master | 负责调度 Task、协调 Checkpoint、处理异常。整个 Flink 作业只有一个 JobManager（高可用模式下有多个）。 |
| **TaskManager** | Worker | 负责执行具体的 Task、存储状态、管理资源（Slot）。可以有多个。 |
| **Slot** | 资源单位 | TaskManager 的最小资源分配单元，每个 Slot 提供固定内存（无 CPU 隔离）。 |
| **Dispatcher** | 独立进程 | 接收用户提交的 Job，为每个 Job 启动一个新的 JobManager，并提供 REST 接口。 |
| **ResourceManager** | 独立进程 | 负责管理 Slot 资源，与 YARN/K8s/Mesos 等资源管理器交互。 |

**作业执行流程：**
1. 用户通过 CLI 或 REST API 提交 Job，Dispatcher 接收并启动 JobManager。
2. JobManager 将 JobGraph 转换为 ExecutionGraph，确定并行度和调度策略。
3. ResourceManager 向资源管理器申请 Slot（若资源不足则等待）。
4. TaskManager 提供 Slot，JobManager 将算子子任务分配到各个 Slot 中执行。
5. 数据在算子之间通过流进行传递，形成数据流图。

**与 Spark 的区别：** Spark 是批流一体的微批模型，Flink 是原生流处理模型（Dataflow Model），延迟更低、事件时间支持更好。

## 题目二：什么是 Watermark？Flink 如何使用 Watermark 处理乱序数据？

### 参考答案

**Watermark（水位线）** 是 Flink 用于处理**事件时间（Event Time）**乱序问题的核心机制。

### 基本概念

- **Event Time**：数据本身携带的时间戳，代表事件实际发生的时间。
- **Watermark**：一种**时间戳断言**，表示"当 Watermark = T 时，所有时间戳 ≤ T 的数据都已经到达（或可认为不会再到达）"。
- **乱序问题**：由于网络延迟等原因，数据可能不按时间戳顺序到达。

### Watermark 的工作机制

```
Watermark T 的含义：不再等待时间戳小于 T 的数据
```

Flink 窗口根据 Watermark 决定何时触发计算并销毁窗口：

1. **窗口收集数据**：窗口持续接收时间戳在 [start, end) 范围内的数据。
2. **Watermark 推进**：当 Watermark >= window_end_time 时，窗口触发计算。
3. **乱序数据处理**：时间戳 < 当前 Watermark 的数据被认为是"迟到数据"，默认被丢弃（可通过 sideOutput 收集）。

### 生成方式

| 方式 | 说明 |
|------|------|
| **周期 Watermark** | 周期性（默认 200ms）从数据流中提取最大时间戳并发射 `maxTimestamp - maxOutOfOrderness` |
| **标点 Watermark** | 遇到特定数据（如特殊事件）时立即发射当前 Watermark |

### 代码示例

```java
// 自定义 Watermark 策略：允许 5 秒乱序
WatermarkStrategy<MyEvent> wms = WatermarkStrategy
    .<MyEvent>forBoundedOutOfOrderness(Duration.ofSeconds(5))
    .withTimestampAssigner((event, timestamp) -> event.getTimestamp());

DataStream<MyEvent> stream = env.fromSource(
    kafkaSource, wms, "Kafka Source"
);
```

### 总结

- Watermark 解决了"如何在有乱序数据的情况下，正确触发基于事件时间的窗口计算"。
- 设置 `maxOutOfOrderness`（最大乱序度）是关键参数，太大导致延迟高，太小导致数据丢失。
- 在低吞吐场景下，周期性 Watermark 可能推进较慢，需要结合 `withIdleness` 处理空闲源。

## 题目三：Flink 的 Checkpoint 和 Savepoint 有什么区别？如何选择？

### 参考答案

Flink 的 Checkpoint 和 Savepoint 都是用于**状态持久化**的机制，但目的和触发方式不同。

### 对比表

| 维度 | Checkpoint | Savepoint |
|------|-----------|-----------|
| **触发方式** | Flink 自动周期性触发 | 用户手动触发 |
| **目的** | 故障恢复，保证 exactly-once | 计划性备份，用于升级、迁移 |
| **依赖** | 依赖 Checkpoint 存储（默认 S3/HDFS） | 手动指定存储位置 |
| **清理策略** | Job 取消/配置保留数自动删除 | 除非手动删除，否则永久保留 |
| **作业图兼容性** | 必须从同一个 JobGraph 恢复 | 可从不同 JobGraph 恢复（需要手动映射） |
| **性能影响** | 增量 Checkpoint 可将对性能影响降到最低 | 同步/异步执行，会暂停部分工作 |

### Checkpoint 原理

1. **Barrier 对齐**：JobManager 定期向数据流中注入 Barrier（屏障），Barrier 随数据流动。
2. **状态快照**：算子收到所有输入的 Barrier 后，将自身状态快照写入持久化存储。
3. **Chandy-Lamport 变体**：Flink 的分布式快照算法，完美匹配流处理的有向无环图（DAG）结构。
4. ** Exactly-Once 语义**：Barrier 对齐确保每个数据只被处理一次。

```java
// 启用 Checkpoint，间隔 10 秒，exactly-once 模式
env.enableCheckpointing(10_000, CheckpointingMode.EXACTLY_ONCE);
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(5000);
env.getCheckpointConfig().setCheckpointTimeout(60_000);
```

### Savepoint 用途

- 应用升级（不停服）
- 集群迁移
- A/B 测试
- 审计和回溯

### 建议

- **日常容灾**：使用 Checkpoint，开启增量 Checkpoint 减少存储和性能开销。
- **版本发布**：在发布前手动触发 Savepoint，以便回滚。
- **状态后端选择**：生产环境推荐 RocksDB（支持增量 Checkpoint + 大状态）。

> - Flink Savepoint 详解：<https://www.cnblogs.com/do-more/p/16323996.html>

## 题目四：Flink 的窗口函数有哪些？滚动窗口、滑动窗口、会话窗口的区别是什么？

### 参考答案

### 窗口类型总览

Flink 支持两大类窗口：

1. **Keyed Windows**（按键分区窗口）：按 Key 分组后对每个 Key 分别开窗。
2. **Non-Keyed Windows**（非按键分区窗口）：所有数据合并到同一窗口（并行度=1）。

### 按时间语义分类

| 类型 | 说明 |
|------|------|
| **滚动窗口（Tumbling Window）** | 固定大小、不重叠的窗口 |
| **滑动窗口（Sliding Window）** | 固定大小、可重叠的窗口 |
| **会话窗口（Session Window）** | 由不活动时间间隙触发的动态窗口 |

### 详细对比

#### 1. 滚动窗口（Tumbling Windows）

- **特点**：窗口大小固定，窗口之间不重叠，首尾相接。
- **场景**：按固定周期统计，如"每 5 分钟统计一次 PV"。

```java
// 每 5 分钟一个窗口，统计次数
DataStream<Tuple2<String, Long>> counts = stream
    .keyBy(r -> r.f0)
    .window(TumblingProcessingTimeWindows.of(Time.minutes(5)))
    .sum(1);
```

#### 2. 滑动窗口（Sliding Windows）

- **特点**：窗口大小固定，但可以设置滑动步长，窗口之间可重叠。
- **参数**：窗口大小（window size）+ 滑动步长（slide interval）。
- **场景**：需要平滑统计或重叠分析，如"每 5 分钟计算一次过去 10 分钟的平均值"。

```java
// 窗口大小 10 分钟，滑动步长 5 分钟
.window(SlidingProcessingTimeWindows.of(Time.minutes(10), Time.minutes(5)))
```

| 配置 | 结果 |
|------|------|
| size=10min, slide=5min | 窗口重叠，窗口数 = 2×窗口数 |
| size=10min, slide=10min | 等同于滚动窗口 |
| size=10min, slide=20min | 窗口之间有间隙 |

#### 3. 会话窗口（Session Windows）

- **特点**：由不活动时间间隙（Gap）动态决定窗口边界，无固定大小。
- **场景**：用户行为分析，如"用户操作间隔超过 30 秒则视为新会话"。

```java
// 会话间隔 30 秒
.window(ProcessingTimeSessionWindows.withGap(Time.seconds(30)))
```

### 窗口函数（Process Function）

| 类型 | 说明 |
|------|------|
| **ReduceFunction** | 对窗口内数据两两聚合，输出单条结果，效率高 |
| **AggregateFunction** | 更通用的聚合接口，可输出任意类型，支持中间状态 |
| **ProcessWindowFunction** | 全量窗口数据作为迭代器，可访问窗口元信息（时间、Key），适合需要上下文场景 |
| **WindowFunction** | ProcessWindowFunction 的旧版接口 |

### 最佳实践

- 优先使用 **ReduceFunction / AggregateFunction**（增量聚合，性能更好）。
- 需要窗口元信息或复杂逻辑时使用 **ProcessWindowFunction**。
- 大状态窗口推荐 **RocksDB** 作为状态后端。

## 题目五：Flink 的状态后端（State Backend）有哪些？各自适用场景是什么？

### 参考答案

Flink 的**状态后端（State Backend）**决定了状态如何存储、访问和持久化。Flink 支持三种主要状态后端：

### 三种状态后端对比

| 状态后端 | 存储介质 | 状态大小 | 性能 | 适用场景 |
|---------|---------|---------|------|---------|
| **HashMapStateBackend** | JVM Heap | 小（受 JVM 堆内存限制） | 最高（纯内存读写） | 小状态、低延迟场景、测试 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘） | 大（可横向扩展） | 略低（序列化开销） | 大状态、生产环境 |
| **EmbeddedRocksDBStateBackend（增量 Checkpoint）** | RocksDB + 远程存储 | 非常大 | 中等 | 超大状态、需长期容灾 |

### 1. HashMapStateBackend

```java
env.setStateBackend(new HashMapStateBackend());
```

- **存储位置**：状态直接存储在 TaskManager 的 JVM 堆内存中。
- **优点**：读写性能最高，无序列化开销。
- **缺点**：受 JVM 堆大小限制，不适合大状态；故障恢复需从 Checkpoint 完整拉回。
- **适用**：状态很小的 Job（如 ETL 简单去重）、Flink SQL 作业。

### 2. EmbeddedRocksDBStateBackend

```java
env.setStateBackend(new EmbeddedRocksDBStateBackend());
```

- **存储位置**：状态先写入本地 RocksDB（磁盘），Checkpoin 时将 SST 文件上传到远程存储（如 HDFS/S3）。
- **优点**：支持大状态（TB 级别）；增量 Checkpoint 减少写入量。
- **缺点**：每次读写需序列化/反序列化（Kyro 或 protobuf）；磁盘 I/O 带来一定延迟。
- **适用**：生产环境的大状态作业，如实时数仓、复杂聚合。

### 3. 增量 Checkpoint（RocksDB）

```java
EmbeddedRocksDBStateBackend rocksDB = new EmbeddedRocksDBStateBackend(true); // true = 增量
env.setStateBackend(rocksDB);
```

- 每次 Checkpoint 仅上传新产生的 SST 文件，而非全量状态。
- 大幅减少Checkpoint 完成时间和网络开销。

### 状态生命周期

```
Keyed State（按键分区状态）：
  - ValueState<T>
  - ListState<T>
  - ReducingState<T>
  - AggregatingState<IN, OUT>
  - MapState<K, V>

Operator State（算子状态）：
  - ListState<T>
  - UnionListState<T>
  - BroadcastState<K, V>
```

### 选型建议

| 场景 | 推荐后端 |
|------|---------|
| 小状态、低延迟、实时计算 | HashMapStateBackend |
| 大状态、生产环境 | EmbeddedRocksDBStateBackend + 增量 Checkpoint |
| 超大状态（TB 级）| EmbeddedRocksDBStateBackend + 增量 Checkpoint + HDFS/S3 |
