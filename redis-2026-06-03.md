## 题目一：Redis 和 Memcached 有什么区别？

### 参考答案

**共同点：**
- 都是基于内存的数据库，一般都用来当做缓存使用
- 都有过期策略
- 两者的性能都非常高

**区别：**

| 特性 | Redis | Memcached |
|------|-------|-----------|
| 数据类型 | 支持丰富数据类型（String、Hash、List、Set、ZSet） | 只支持最简单的 key-value 数据类型 |
| 持久化 | 支持数据持久化，可将内存数据保存在磁盘，重启后加载使用 | 无持久化功能，数据全在内存，重启后丢失 |
| 集群 | 原生支持集群模式（主从、哨兵、切片集群） | 无原生集群模式，需依靠客户端实现分片 |
| 高级特性 | 支持发布订阅、Lua脚本、事务等 | 不支持 |

**为什么Redis更适合作为MySQL缓存？**
- **高性能**：Redis直接操作内存，QPS可达10W+，是MySQL的10倍
- **高并发**：扛住大量并发请求，减轻数据库压力

## 题目二：Redis 的五种数据类型以及使用场景是什么？

### 参考答案

**五种基本数据类型：**

| 类型 | 使用场景 |
|------|---------|
| **String** | 缓存对象、常规计数、分布式锁、共享session信息 |
| **Hash** | 缓存对象、购物车 |
| **List** | 消息队列（需注意：需自行实现全局唯一ID，不支持消费组形式消费） |
| **Set** | 聚合计算（并集、交集、差集），如点赞、共同关注、抽奖活动 |
| **Zset** | 排序场景，如排行榜、电话姓名排序 |

**四种扩展数据类型（Redis 2.2+）：**

| 类型 | 版本 | 使用场景 |
|------|------|---------|
| **BitMap** | 2.2+ | 二值状态统计，如签到、判断用户登录状态、连续签到用户总数 |
| **HyperLogLog** | 2.8+ | 海量数据基数统计，如百万级网页UV计数 |
| **GEO** | 3.2+ | 存储地理位置信息，如滴滴叫车 |
| **Stream** | 5.0+ | 消息队列，支持自动生成全局唯一消息ID，支持消费组形式消费 |

## 题目三：Redis 单线程模型是怎样的？为什么单线程还能这么快？

### 参考答案

**Redis单线程指的是：**
「接收客户端请求 → 解析请求 → 进行数据读写等操作 → 发送数据给客户端」这个过程由一个线程（主线程）完成。

**注意：** Redis 在 2.6 版本后会启动后台线程（BIO）处理耗时任务：
- 2.6版本：2个后台线程处理关闭文件、AOF刷盘
- 4.0+版本：新增lazyfree线程，异步释放内存（大key删除用unlink代替del）

**单线程模式工作流程：**
1. 调用 `epoll_create()` 创建 epoll 对象，调用 `socket()` 创建服务端 socket
2. 调用 `bind()` + `listen()` 绑定监听
3. 调用 `epoll_ctl()` 将 listen socket 加入 epoll，注册连接事件处理函数
4. 进入事件循环：
   - 先处理发送队列（write函数发送数据）
   - 调用 `epoll_wait()` 等待事件
   - 连接事件 → accept → 注册读写事件
   - 读事件 → read → 解析命令 → 处理命令 → 写入发送队列
   - 写事件 → write 发送数据

**单线程为什么这么快？**

1. **内存操作**：大部分操作在内存中完成，采用高效数据结构，瓶颈是内存/带宽而非CPU
2. **无竞争开销**：单线程避免了多线程切换和锁竞争，不存在死锁问题
3. **I/O多路复用**：一个线程处理多个IO流（select/epoll），可同时监听多个Socket

Redis 6.0引入了多线程用于网络I/O加速，但命令执行仍是单线程。

## 题目四：Redis 持久化机制是什么？RDB和AOF有什么区别？

### 参考答案

**RDB（Redis Database）：** 定时生成数据快照
- 原理： fork一个子进程，遍历内存数据写入RDB文件
- 优点：恢复速度快，适合灾难恢复
- 缺点：可能丢失最后一次快照后的数据

**AOF（Append Only File）：** 记录每次写命令
- 原理：将每个写命令追加到AOF缓冲区，再异步刷盘
- 三种刷盘策略：
  - `always`：每次写命令同步刷盘，最安全但最慢
  - `everysec`（默认）：每秒异步刷盘，可能丢失1秒数据
  - `no`：由操作系统决定何时刷盘
- 优点：数据安全性更高
- 缺点：AOF文件比RDB大，恢复速度慢

**混合持久化（Redis 4.0+）：**
- 开启后，AOF重写时会先用RDB格式写入数据，再用AOF格式追加增量命令
- 兼顾快速恢复和数据完整性

**生产环境推荐配置：**
- 开启RDB + AOF混合持久化
- 主从复制实现读写分离，从节点开启RDB

## 题目五：Redis 过期删除策略和内存淘汰策略有什么区别？

### 参考答案

**过期删除策略（针对设置了过期时间的key）：**

| 策略 | 原理 | 优缺点 |
|------|------|-------|
| **定时删除** | 定时器主动扫描并删除过期key | 内存友好，CPU不友好，可能影响响应时间 |
| **惰性删除** | 访问key时检查是否过期，过期则删除 | CPU友好，内存不友好，过期key堆积占用内存 |
| **定期删除** | 每隔一段时间扫描检查，删除过期key | 折中方案，平衡CPU和内存 |

Redis采用「惰性删除 + 定期删除」组合策略。

**内存淘汰策略（内存满时的处理）：**

当内存达到 `maxmemory` 上限时，Redis通过淘汰策略腾出空间：

| 策略 | 行为 |
|------|------|
| `volatile-lru` | 从已设置过期时间的key中淘汰最近最少使用的 |
| `volatile-ttl` | 从已设置过期时间的key中淘汰即将过期的 |
| `volatile-random` | 从已设置过期时间的key中随机淘汰 |
| `allkeys-lru` | 从所有key中淘汰最近最少使用的（最常用） |
| `allkeys-random` | 从所有key中随机淘汰 |
| `no-eviction` | 不淘汰，返回错误（默认） |

**如何选择：**
- 缓存场景推荐 `allkeys-lru` 或 `volatile-lru`
- 所有key都有过期时间用 `volatile-lru`
- 不确定用 `allkeys-lru`
