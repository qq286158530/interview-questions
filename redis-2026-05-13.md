## 题目一：Redis 数据类型以及使用场景

### ✅ 答案

Redis 提供了丰富的数据类型，常见的有五种：**String（字符串）、Hash（哈希）、List（列表）、Set（集合）、Zset（有序集合）**。

随着 Redis 版本的更新，后面又支持了四种数据类型：**BitMap、HyperLogLog、GEO、Stream**。

### 五种数据类型应用场景

| 类型 | 应用场景 |
|------|----------|
| **String** | 缓存对象、常规计数、分布式锁、共享 session 信息等 |
| **Hash** | 缓存对象、购物车等 |
| **List** | 消息队列（需注意：生产者需自行实现全局唯一 ID，不能以消费组形式消费数据） |
| **Set** | 聚合计算（并集、交集、差集）场景，比如点赞、共同关注、抽奖活动等 |
| **Zset** | 排序场景，比如排行榜、电话和姓名排序等 |

### 四种高级数据类型应用场景

| 类型 | 应用场景 |
|------|----------|
| **BitMap（2.2版）** | 二值状态统计的场景，比如签到、判断用户登陆状态、连续签到用户总数等 |
| **HyperLogLog（2.8版）** | 海量数据基数统计的场景，比如百万级网页 UV 计数等 |
| **GEO（3.2版）** | 存储地理位置信息的场景，比如滴滴叫车 |
| **Stream（5.0版）** | 消息队列，相比于 List 实现的消息队列，有自动生成全局唯一消息ID、支持以消费组形式消费数据两个特有特性 |

## 题目二：Redis 单线程模型为什么这么快？

### ✅ 答案

Redis 官方基准测试的结果是单线程的 Redis 吞吐量可以达到 **10W QPS**。Redis 采用单线程却这么快的原因有以下几点：

1. **内存操作**：Redis 的大部分操作都在内存中完成，并且采用了高效的数据结构（SDS、跳表等），因此 Redis 瓶颈可能是机器的内存或者网络带宽，而并非 CPU。

2. **避免竞争开销**：Redis 采用单线程模型可以避免了多线程之间的竞争，省去了多线程切换带来的时间和性能上的开销，而且也不会导致死锁问题。

3. **I/O 多路复用**：Redis 采用了 I/O 多路复用机制处理大量的客户端 Socket 请求。IO 多路复用机制是指一个线程处理多个 IO 流，即 select/epoll 机制。在 Redis 只运行单线程的情况下，该机制允许内核中同时存在多个监听 Socket 和已连接 Socket，内核会一直监听这些 Socket 上的连接请求或数据请求，一旦有请求到达就会交给 Redis 线程处理，实现了**一个线程处理多个 IO 流**的效果。

> ⚠️ 注意：Redis 6.0 之前单线程指网络 I/O 和执行命令是单线程，但 Redis 启动时会启动后台线程（BIO）处理关闭文件、AOF 刷盘、释放内存等耗时任务，避免阻塞主线程。

## 题目三：Redis 如何实现数据持久化？RDB 和 AOF 区别？

### ✅ 答案

Redis 共有三种数据持久化的方式：

### 1. AOF 日志（Append Only File）

- **原理**：每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里。Redis 重启时，会读取该文件记录的命令，然后逐一执行命令来进行数据恢复。
- **写回策略**：

| 策略 | 说明 | 安全性 | 性能 |
|------|------|--------|------|
| **Always** | 每次写操作命令执行完后，同步将 AOF 日志数据写回硬盘 | 高 | 低 |
| **Everysec** | 每次写操作命令执行完后，先将命令写入到内核缓冲区，每隔一秒将缓冲区内容写回硬盘 | 中 | 中 |
| **No** | 由操作系统控制写回时机 | 低 | 高 |

- **AOF 重写机制**：当 AOF 文件过大时，Redis 会启用后台子进程 `bgrewriteaof` 来压缩 AOF 文件。重写过程中主进程仍可处理命令，增量命令写入 AOF 重写缓冲区，待重写完成后合并到新 AOF 文件。

### 2. RDB 快照（Redis DataBase）

- **原理**：记录某一时刻的内存数据，以二进制方式写入磁盘。恢复数据时直接将 RDB 文件读入内存，速度比 AOF 快。
- **命令**：
  - `save`：在主线程生成 RDB 文件，会阻塞主线程
  - `bgsave`：创建子进程生成 RDB 文件，避免阻塞
- **配置自动触发**：通过 `save 900 1`、`save 300 10`、`save 60 10000` 等条件自动执行 bgsave。
- **快照期间数据可修改**：利用 **写时复制（Copy-On-Write）** 技术，fork 出子进程时父子进程共享内存，主线程修改数据时会复制副本，子进程将副本写入 RDB。

### 3. 混合持久化（Redis 4.0+）

- **原理**：结合 RDB 和 AOF 的优点。重写时先将以 RDB 格式写入 AOF 文件，再将增量命令以 AOF 格式追加到文件后半部分。
- **优点**：Redis 重启时先加载 RDB 部分（快速启动），再加载增量 AOF 部分（减少数据丢失）。
- **缺点**：AOF 文件可读性差，兼容性低（无法用于 4.0 之前版本）。

### RDB vs AOF 对比

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件格式 | 二进制 | 命令日志文本 |
| 恢复速度 | 快 | 慢（全量命令执行） |
| 数据完整性 | 差（取决于快照频率） | 好（可配置写回策略） |
| 文件大小 | 小 | 大 |
| 性能影响 | 快照时阻塞（可用 bgsave） | 低（后台刷盘） |

## 题目四：Redis 主从复制原理是什么？如何实现高可用？

### ✅ 答案

### 主从复制

Redis 主从复制采用**一主多从**模式，主服务器进行读写操作，从服务器只读并接收主服务器同步过来的写操作命令。

**命令传播阶段**：主服务器收到新的写命令后，会异步发送给从服务器。主服务器不会等待从服务器实际执行完命令就向客户端返回结果，因此**无法保证强一致性**。

### 哨兵模式

当主从服务器出现故障宕机时，哨兵模式可以自动完成故障恢复。哨兵的核心功能包括：

1. **监控**：哨兵进程周期性检查主从服务器是否正常运行
2. **自动故障转移**：当主服务器下线时，哨兵会选举出一个从服务器升级为新的主服务器，并通知其他从服务器更新主服务器配置
3. **提醒**：故障转移完成后，哨兵通过 API 向管理员发送通知

哨兵以特殊"只读"模式运行，通过 `INFO` 命令周期性获取主服务器信息并检测主观下线（SDOWN）和客观下线（ODOWN）状态。当确认主服务器下线后，通过 **Raft 算法**选举出领头的哨兵节点来执行故障转移。

### 集群模式（切片集群）

当数据量非常大时，单机无法满足需求，Redis 提供了**切片集群**（Redis Cluster）方案：

- **数据分片**：将数据划分为 16384 个 slot，分布到多个主节点上
- **高可用**：每个主节点可以配置多个从节点，节点故障时自动选举
- **客户端路由**：客户端请求直接访问对应的节点，无需经过代理层

## 题目五：如何解决 Redis 和 MySQL 双写一致性问题？

### ✅ 答案

将数据放在 Redis 中缓存时，当 MySQL 中数据改变后需要同步更新 Redis 缓存，这就会涉及双写一致性问题。常见有以下几种方案：

### 1. Cache Aside（旁路缓存）— 最常用

- **读**：先读 Redis，命中则返回；未命中则读 MySQL，然后将数据写入 Redis
- **写**：直接更新 MySQL，**删除** Redis 中的缓存（旧值删除，而不是更新）。下次访问时再从 MySQL 加载到 Redis

> ✅ 推荐：删除缓存而非更新，因为删除操作原子性更强，避免并发下的数据不一致。

### 2. Read/Write Through（读/写穿透）

- **读穿透**：缓存服务负责读和写，缓存与数据库同步更新
- **写穿透**：应用只写缓存，缓存服务同步更新数据库

### 3. Write Behind（异步写入）

- 应用只写缓存，缓存服务**异步批量**将数据写入数据库
- 优点：写入性能高；缺点：数据可能丢失

### 延迟双删策略

针对删除缓存后到数据库更新完成期间，其他请求可能把旧值重新写入缓存的问题（并发场景），可以使用延迟双删：

```python
# 1. 先删除 Redis 缓存
redis.delete(key)
# 2. 更新 MySQL
mysql.update(data)
# 3. 延迟一段时间（估计数据库更新完成的时间）后再删除 Redis 缓存
time.sleep(0.5)
redis.delete(key)
```

### 最终一致性保证

- **重试机制**：使用消息队列（如 RocketMQ）保证更新数据库和删除缓存的最终一致性
- **订阅 binlog**：解析 MySQL binlog，将变更数据写入 Redis
