## 题目一：Redis 为什么这么快？

### 参考答案

Redis 之所以快，主要有以下几个原因：

**1. 纯内存操作**
Redis 数据读写直接发生在内存中，访问速度是纳秒级别；而传统数据库读写磁盘是毫秒级别，两者相差数个数量级。内存的 I/O 性能远高于磁盘，这是 Redis 高性能的最主要原因。

**2. 高效的 I/O 模型**
Redis 使用单线程事件循环（Event Loop）+ I/O 多路复用技术，让单个线程可以同时处理多个网络连接上的 I/O 事件（如读写），避免了多线程的上下文切换和锁竞争问题。结合内存操作的高效性和 I/O 多路复用，Redis 能轻松处理大量并发请求。

> Redis 6.0 之后，在网络数据的读写阶段引入了多线程，以进一步提升 I/O 性能，但命令执行仍是单线程。

**3. 优化的内部数据结构**
Redis 提供多种数据类型（String、List、Hash、Set、Zset 等），内部采用高度优化的编码方式（如 ziplist、quicklist、skiplist、hashtable 等）。Redis 会根据数据大小和类型动态选择最合适的内部编码，在性能和空间效率之间取得最佳平衡。

**4. 简洁高效的通信协议**
Redis 使用自己设计的 RESP（REdis Serialization Protocol）协议，实现简单、解析性能好且二进制安全，序列化/反序列化开销很小，有助于提升整体交互速度。

**为什么不用 Redis 当主数据库？**
主要因为内存成本太高，且 Redis 持久化仍存在数据丢失风险。

## 题目二：Redis 有哪些持久化机制？各有何优缺点？

### 参考答案

Redis 提供两种持久化机制：**RDB** 和 **AOF**，Redis 4.0 之后采用混合持久化。

### RDB（Redis DataBase）

RDB 是 Redis 默认的持久化方式，按照一定时间将内存中的数据以**快照**形式保存到硬盘，生成 `dump.rdb` 文件。

**优点：**
- 只有一个文件，方便备份和灾难恢复
- fork 子进程进行持久化，主进程不参与 I/O 操作，性能最大化
- 恢复大数据集时速度比 AOF 快

**缺点：**
- 可能丢失最后一次快照之后的数据（实时性较差）
- fork 子进程时会占用一定内存

### AOF（Append Only File）

AOF 以**日志**形式记录每个写操作，重启时重放日志来恢复数据。

**优点：**
- 数据安全性更高，丢数据风险更低
- 日志文件可读，适合紧急恢复

**缺点：**
- 日志文件通常比 RDB 大，恢复速度较慢
- 持续写入时 I/O 压力较大

### 混合持久化（Redis 4.0+）

Redis 4.0 之后引入混合持久化，AOF 重写时使用 RDB 格式写入 AOF 文件开头，后续增量操作以 AOF 格式追加。兼顾 RDB 快速恢复和 AOF 数据不丢失的优点。

**实际生产推荐配置：** 用 AOF 保证数据不丢失作为数据恢复首选，用 RDB 做不同程度的冷备。

## 题目三：Redis 的数据类型有哪些？各自适用场景是什么？

### 参考答案

Redis 支持 5 种基础数据类型和 3 种特殊数据类型：

### 5 种基础数据类型

| 类型 | 数据结构 | 典型使用场景 |
|------|----------|------------|
| **String** | 字符串 | 缓存简单数据、Session、计数器 |
| **List** | 双向链表 | 消息队列、任务队列、评论列表 |
| **Set** | 哈希表 | 标签系统、去重、关注列表 |
| **Hash** | 哈希表 | 存储对象（如用户信息、商品详情） |
| **Zset** | 跳表 + 哈希表 | 排行榜、热搜榜、有序消息队列 |

### 3 种特殊数据类型

| 类型 | 使用场景 |
|------|----------|
| **HyperLogLog** | 基数统计（UV 统计） |
| **Bitmap** | 活跃用户统计、签到功能 |
| **Geospatial** | 地理位置附近的人/商户查询 |

### 实战场景举例

- **用户活跃排行榜** → Zset，按 score 排序取前 N
- **作者白名单** → Set，O(1) 复杂度判断是否在集合中
- **用户 Session** → String，SET/GET 用户 token
- **热点数据缓存** → String/Hash，文章详情、商品信息
- **分布式锁** → SETNX + 过期时间，或 Redisson

## 题目四：如何保证缓存与数据库双写时的数据一致性？

### 参考答案

缓存与数据库双写一致性是分布式系统中的经典难题，主要有以下几种策略：

### 策略一：Cache Aside（旁路缓存，最常用）

**读操作：** 先查缓存，缓存未命中则查数据库，并将结果写入缓存。
**写操作：** 先更新数据库，再删除缓存。

> 推荐：写操作时**删除缓存**而不是更新缓存，因为删除缓存成本更低，且能避免并发写导致的数据不一致。

### 策略二：Read/Write Through（读写穿透）

应用只与缓存交互，缓存负责同步更新数据库。应用不感知数据库，相当于缓存做了更多工作。

### 策略三：Write Behind（异步写入）

写操作只更新缓存，缓存异步批量更新到数据库。性能最高但数据安全性最低，可能丢数据。

### 常见问题与解决方案

**问题：并发情况下先删缓存再更新数据库，导致脏读**
解决：延迟双删（删除缓存 → 更新数据库 → 延迟一段时间再删缓存）

**问题：缓存删除失败**
解决：引入消息队列（如 RabbitMQ、RocketMQ）重试删除操作，或将删除任务写入数据库定时处理。

**问题：缓存雪崩**
解决：为缓存数据设置不同的过期时间，避免大量 key 同时过期；使用 Redis 持久化机制保障数据恢复。

**问题：缓存击穿**
解决：使用互斥锁（如 SETNX）或将热点数据设为永不过期 + 异步重建。

## 题目五：Redis 的高可用方案有哪些？说说 Sentinel 和 Cluster 的区别？

### 参考答案

Redis 高可用方案主要有三种：**主从复制**、**哨兵模式（Sentinel）**、**集群模式（Cluster）**。

### 主从复制

- 原理：主节点（Master）处理读写请求，从节点（Replica）异步复制主节点数据
- 优点：读写分离、提升可用性（主节点挂了从节点可接管）
- 缺点：无自动故障转移，需要手动干预

### Sentinel（哨兵模式）

哨兵模式在主从复制基础上增加了监控、自动故障转移和通知功能。

**核心功能：**
- 监控：Sentinel 节点持续检测主从节点是否宕机
- 自动故障转移：当主节点不可用时，Sentinel 自动将从节点升级为新主节点，并通知客户端
- 配置提供：客户端通过 Sentinel 获取当前主节点地址

**部署架构：** 一主两从 + 三个 Sentinel 节点（最常见），故障转移时间约 10-15 秒。

### Cluster（集群模式）

Redis Cluster 是 Redis 原生提供的**分布式数据库方案**，通过哈希槽（16384 个槽）自动分片。

**核心特点：**
- 数据自动分片：每个主节点存储一部分数据（16384 / N 个槽）
- 高可用：每个主节点可配置从节点（3 主 3 从最常见），主节点挂了从节点自动升级
- 水平扩展：直接增加节点即可扩展集群容量
- 故障转移：通常在几秒内完成

### 对比总结

| 特性 | Sentinel | Cluster |
|------|----------|---------|
| 数据分片 | 无（全量复制） | 有（16384 槽自动分片） |
| 客户端连接 | 连任意 Sentinel 获取主节点地址 | 连任意节点，自动路由到正确节点 |
| 故障转移 | 哨兵自动完成 | 从节点自动升级为主节点 |
| 适用场景 | 中小规模、数据量较小 | 大规模数据、需要水平扩展 |
| 复杂度 | 较低 | 较高 |
