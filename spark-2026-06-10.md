## 1️⃣ Spark 的 Stage 划分机制是什么？Task 是如何被调度执行的？

### 参考答案

**Stage划分依据 ——窄依赖与宽依赖**

Spark 根据 RDD 之间的依赖关系划分 Stage：
- **窄依赖（Narrow Dependency）**：一个父分区最多被一个子分区使用，可以 pipeline 执行，无需 shuffle。同一 Stage 内多个 transformation 可以合并成一个 Task。
- **宽依赖（Shuffle Dependency）**：一个父分区可能被多个子分区使用，必须等父 Stage全部完成才能启动子 Stage，需要 shuffle。

**划分算法（回溯法）：**
```
从 DAG 末端的 Action 开始回溯
→ 遇到宽依赖（Shuffle）则划分一个新 Stage
→ 窄依赖则继续合并到当前 Stage
→ 直到溯源到起点
```

**Task调度执行流程：**

```
Application → SparkContext → DAGScheduler（划分 Stage）
                                        ↓
              ResultStage（最后一个Stage）→ TaskSet → TaskScheduler
                                                  ↓
                              TaskSetManager调度 Task 到 Executor
                                                  ↓
                              Executor 反序列化 Task → 运行 Task
                                                  ↓
                              ShuffleMapStage 结果写入 Shuffle
                              ResultStage 结果返回给 Driver
```

**ShuffleMapStage vs ResultStage：**
- `ShuffleMapStage`：中间结果需要 shuffle，输出到磁盘，供后续 Stage消费
- `ResultStage`：最后一个 Stage，输出结果给 Driver 程序

**关键点补充：**
- 一个 Stage 内部包含多个 Task，数量等于该 Stage 最后一个 RDD 的分区数
- Task 分为 `ShuffleMapTask`（需要 shuffle）和 `ResultTask`（最终结果）
- Task 被调度到Executor后，Executor 通过线程池并行运行多个 Task

- Apache Spark 官方文档 - "Spark Architecture"：https://spark.apache.org/docs/latest/cluster-overview.html
- 《Spark The Definitive Guide》- Chapter 20: How Spark Schedules Jobs

## 2️⃣ RDD、DataFrame、DataSet 三者的区别与联系是什么？如何选择？

### 参考答案

**一、定义与特点**

| 特性 | RDD（Resilient Distributed Dataset） | DataFrame | DataSet（Strong Typed DataFrame） |
|------|------|------|------|
| **数据结构** | 分布式元素集合（无结构） | 类似关系型数据库表，有列名和类型 | 强类型的 DataFrame（JVM 对象） |
| **类型安全** | ✅ 编译时类型安全 | ❌ 编译时不安全 | ✅ 编译时类型安全 |
| **数据组织** | 每行是自定义对象 | 每行是 `Row` 对象 | 每行是指定的样例类对象 |
| **优化** | 无 Catalyst优化 | 有 Catalyst + Tungsten 优化 | 有 Catalyst + Tungsten 优化 |
| **API** | 底层函数式 API（map/filter...） | 高级 DSL（select/where/groupBy...） | 结合两者，类型安全 SQL |
| **出场版本** | Spark 1.0 | Spark 1.3 | Spark 1.6 |

**二、内部实现统一性**

实际上 **DataFrame 和 DataSet 在 Spark 2.0 之后已经统一**，DataSet 只是 DataFrame 的强类型版本，二者在内部都表示为 `Dataset<Row>`：

```scala
// Spark 2.0+ DataSet 本质是强类型的 DataFrame
type DataSet[T] = Dataset[T]  // where Dataset[T] = Dataset[Row] with typed encoder
```

**三、优化原理（Catalyst 优化器）**

```
User Code（DSL/SQL）
    ↓
Catalyst Parser（解析为 Logical Plan）
    ↓
Analyzer（绑定列名、数据类型）
    ↓
Logical Plan
    ↓
Cost Optimizer（选择最优执行计划）
    ↓
Physical Plan（多个 Physical Plan 比较成本）
    ↓
Code Generation（Tungsten 生成紧凑字节码）
    ↓
Executor 执行
```

**四、选择建议**

- **选择 RDD**：需要细粒度控制、性能敏感、底层调优、无结构化数据
- **选择 DataFrame**：结构化/半结构化数据、SQL 查询、追求性能、自动优化
- **选择 DataSet**：需要类型安全、想要强类型 API + SQL 混合（ Scala）

**参考来源：**
- Spark Summit 2016 - "Apache Spark 2.0 Preview"：https://databricks.com/blog/2016/07/26/introducing-apache-spark-datasets.html
- 《Spark The Definitive Guide》- Chapter 9: DataFrames vs RDDs vs DataSets

## 3️⃣ Spark 广播变量（Broadcast）的原理是什么？使用场景和注意事项有哪些？

### 参考答案

**一、为什么需要广播变量？**

当 Task 需要访问只读的共享数据（如字典表、配置表）时，如果不使用广播，每个 Task都会在每个 Partition 中复制一份数据，造成极大内存浪费。一个1000 分区的 Job，如果广播一份10MB 数据，实际传输 10GB+。

**二、广播原理**

```
Driver 端：
  broadcast.collect() → 广播变量注册到 Driver
                              ↓
  BlockManager（粗粒度广播模式）：
    广播数据分块（默认 4MB/块）→ 分块存储在 Driver 的 BlockManager
                              ↓
        向 Driver 所在节点的 Map Output Tracker 注册分块信息

Executor 端：
  Task 首次请求广播变量 → 向 Map Output Tracker 请求分块位置
                                    ↓
  获取到位置后，通过 BlockManager 获取分块（优先本地，无则远程）
                                    ↓
  缓存到本地 Executor 的 BlockManager，后续 Task 直接从本地读取
```

**Spark 2.x 之后使用 P2P（Peer-to-Peer）广播模式**，不再走 HTTP Server，通过 BlockManager 的 `BlockFetcher` 直接从其他 Executor 获取数据分块。

**三、广播变量的使用**

```scala
// 创建广播变量
val broadcastList = sc.broadcast(List("北京", "上海", "广州"))

// 在 Task 中使用（必须使用 .value 获取）
def filterFunc(x: String) = broadcastList.value.contains(x)

val rdd = sc.parallelize(List("北京", "深圳", "上海"))
rdd.filter(filterFunc).collect()

// 释放广播（不再使用时主动释放，节省内存）
broadcastList.unpersist()
```

**四、使用场景**

| 场景 | 说明 |
|------|------|
| **大表 join 小表** | 将小表广播到各节点，避免 shuffle（Map Side Join） |
| **静态配置表** | 维表、字典表等只读数据 |
| **过滤条件列表** | IN 语句中的候选项列表 |
| **机器学习特征元数据** | 特征名、特征类型等元信息 |

**五、注意事项**

| 注意点 | 说明 |
|--------|------|
| **数据量** | 建议小于 20MB，过大会导致广播失败或 OOM |
| **只读性** | 广播后不可修改（修改无效且危险） |
| **序列化** | 数据必须可序列化（Java/Kryo） |
| **手动 unpersist** | 广播数据默认永久缓存，任务完成后需手动释放 |
| **spark.sql.autoBroadcastJoinThreshold** | Spark SQL 中，小表小于此阈值（默认 10MB）自动广播 |

**参考来源：**
- Apache Spark 官方文档 - "Broadcasting Variables"：https://spark.apache.org/docs/latest/rdd-broadcasting.html
- 《Learning Spark》- Chapter 6: Broadcasting and Accumulator

## 4️⃣ Spark 的内存管理机制是怎样的？堆内内存和堆外内存的区别是什么？

### 参考答案

**一、内存区域划分**

Spark 1.6 之后引入 **Unified Memory Management（统一内存管理）**，将存储内存和执行内存共享，动态抢占。

```
┌──────────────────────────────────────────────────────┐
│                    Heap Memory                        │
│  ┌────────────────┐  ┌────────────────────────────┐  │
│  │  用户内存      │  │     执行内存 + 存储内存     │  │
│  │（User Memory） │  │  (Execution + Storage)      │  │
│  │  300MB │  │  默认各占 50%（可动态调整） │  │
│  └────────────────┘  └────────────────────────────┘  │
│                                                      │
│  ┌──────────────────────────────────────────────┐ │
│  │ 预留内存（Reserved Memory = 300MB）     │   │
│  └──────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│               Off-Heap Memory（堆外内存）              │
│  需开启 spark.memory.offHeap.enabled = true │
│  通过 Unsafe API 直接操作系统内存，不受 JVM 管理 │
└──────────────────────────────────────────────────────┘
```

**二、各内存区域作用**

| 区域 | 作用 | 配置参数 |
|------|------|---------|
| **Reserved Memory** | 保留 300MB给系统，保证最基本运行 | 固定 300MB |
| **User Memory** | 用户代码使用的内存，RDD 管理元数据等 | `spark.memory.fraction × (Heap - 300MB)` |
| **Storage Memory** | RDD Cache、广播变量、Task 输出临时数据 | 可与 Execution 动态交换 |
| **Execution Memory** | Shuffle 排序、Hash 聚合、JOIN 的 hash table | 可与 Storage 动态交换 |
| **Off-Heap Memory** | 堆外内存，不受 GC 影响，适合大内存场景 | `spark.memory.offHeap.enabled=true` |

**三、动态占用机制**

```
存储内存不足时：
  检查执行内存是否空闲 → 空闲则借用（可恢复）
  
执行内存不足时：
  检查存储内存是否空闲 → 空闲则借用（可恢复）
  若存储内存也在使用 → 吐出自身已缓存数据直到腾出足够空间
  
借出方需要内存时：
  将自身数据吐到磁盘（ spill），归还借用的内存
```

**四、存储级别（Storage Levels）**

| 级别 | 说明 |
|------|------|
| `MEMORY_ONLY` | 仅内存，容易 OOM |
| `MEMORY_AND_DISK` | 内存不足则溢写到磁盘 |
| `MEMORY_ONLY_SER` | 序列化后存内存，省空间但 CPU 开销大 |
| `DISK_ONLY` | 仅磁盘 |
| `OFF_HEAP` | 堆外内存，需开启 |

**五、内存计算实例**

假设 Executor 堆内存 `spark.executor.memory = 4GB`：
- Reserved：300MB
- User Memory：`0.4 × (4096 - 300) ≈ 1520MB`
- 可用 Unified Memory：`4096 - 300 - 1520 ≈ 2276MB`（Execution 和 Storage 各约 1138MB，动态调整）

**参考来源：**
- Apache Spark 官方文档 - "Memory Management"：https://spark.apache.org/docs/latest/tuning.html#memory-management
- 《Spark权威指南》- 第10章：Spark 内存管理

## 5️⃣ Spark Checkpoint 和 RDD persist 的区别是什么？什么场景下应该用 Checkpoint？

### 参考答案

**一、核心区别**

| 维度 | Checkpoint | Persist（Cache） |
|------|------------|-----------------|
| **数据存储位置** | 写入 HDFS/S3 等可靠存储（通常 HDFS） | 存储在 Executor 的内存/磁盘（BlockManager） |
| **血缘关系** | 截断 RDD 的血缘依赖（DAG 从 Checkpoint 处断开） | 不截断血缘，出现问题可从源头重新计算 |
| **容灾可靠性** | ✅ 高（数据可靠存储，即使 Driver 重启也不丢） | ❌ 一般（依赖 Executor 不重启，漂移后数据丢失） |
| **计算开销** | 需要重新计算一个 Stage（写入成本） | 直接缓存，无额外计算 |
| **使用方式** | 显式调用 `sc.setCheckpointDir` + `rdd.checkpoint()` | 调用 `rdd.persist(LEVEL)` |
| **适用场景** | 长血缘链路、DAG 复杂的迭代计算 | 中间结果复用（短期） |

**二、Checkpoint 执行原理**

```
触发时机：
  当 RDD 被 action触发时（注意：只声明 checkpoint()不会真正执行）
  
执行流程：
  1. SparkContext 设置 CheckpointDir（通常 HDFS）
  2. 触发 Action 时，RDD 依赖链被重新计算到 Checkpoint 所在 Stage
  3. 计算完成后，将数据写入 HDFS（多副本，可靠性高）
  4. 截断血缘：后续 RDD 不再依赖父 RDD，只依赖 Checkpoint 数据
  5. 下次使用时，直接从 HDFS 读取
```

**三、persist 不截断血缘的原因**

```
RDD A → B → C → D (persist C) → E
              ↓
         如果 C 丢失：可从 B重新计算 → 恢复 C
         
RDD A → B → [C: Checkpointed] → D → E
              ↓
         C 的血缘已截断，D 直接依赖 HDFS 上的 C 数据
```

**四、何时使用 Checkpoint**

| 场景 | 说明 | 示例 |
|------|------|------|
| **长迭代链路** | 经过多次 transformation，血缘链很长，计算代价高 | ML 迭代训练 |
| **DAG 复杂** | 宽依赖多，任意节点失败重新计算代价大 | 图计算（GraphX） |
| **Spark Streaming迭代** | 微批之间有依赖关系，需要可靠状态 | 状态流处理 |
| **共享中间结果** | 同一 RDD 被多个分支复用，且计算成本高 | 特征工程中多路复用 |
| **防止 Driver 重启丢失数据** | 已缓存数据在 Driver 重启后消失 | 长时间运行的任务 |

**五、最佳实践**

```scala
// 1. 设置检查点目录（建议 HDFS，支持高可用）
sc.setCheckpointDir("hdfs://namenode:9000/spark-checkpoint")

// 2. 对需要截断血缘的 RDD 标记 checkpoint（懒执行）
val resultRDD = processedRDD.checkpoint()

// 3. 第一次 action 时触发 checkpoint 计算和写入
// 建议先 cache，第二次 action 时从 cache 读取（避免重复计算）
resultRDD.cache()
resultRDD.count()  // 第一次：计算 + checkpoint
println(resultRDD.collect().mkString)  // 第二次：从 cache/HDFS 直接读
```

**六、Checkpoint vs Persist 联合使用**

对长血缘 RDD，先 cache避免重复计算，再 checkpoint截断血缘，实现"容错 + 性能"兼顾：

```scala
val intermediateRDD = data.map(...).filter(...).groupBy(...)
intermediateRDD.checkpoint()  // 截断血缘
intermediateRDD.cache()      // 避免重复计算
val result = intermediateRDD.reduceByKey(_ + _)
```
