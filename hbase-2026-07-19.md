## 题目一：HBase 的整体架构是怎样的？Regionserver 和 HDFS 的关系是什么？

### 参考答案

HBase 采用 **Master-Slave 架构**，核心组件包括：

| 组件 | 职责 |
|------|------|
| **HMaster** | 负责表的管理、Region 分配、负载均衡、Schema 变更。不参与实际数据读写，是纯粹的协调者。|
| **RegionServer** | 负责数据读写、与 HDFS DataNode 交互处理实际的 I/O 请求。每个 RegionServer 管理一组 Region。|
| **ZooKeeper** | 负责 Master election、RegionServer 心跳、 元数据存储（`-ROOT-`、`META` 表位置）。|

### Region 与 HDFS 的关系

- HBase 表按 **RowKey 范围** 水平切分为多个 **Region**，每个 Region 分配给某一台 RegionServer 托管。
- RegionServer 将数据以 **HFile** 格式存储在 **HDFS** 上。作为 HDFS 的客户端，RegionServer 的每次写入实际是写入 HDFS 的 DataNode 副本（默认 3 副本）。
- 因此 RegionServer 本质上是 **HDFS 的客户端进程**，不负责副本管理，副本的持久化和恢复由 HDFS namenode 决定。

**数据写入流程简述：**
1. 客户端连接 ZooKeeper，获取 `-ROOT-` → `META` 表位置。
2. 根据 RowKey 从 `META` 表找到目标 Region 所在的 RegionServer。
3. 直接请求该 RegionServer 写入数据（先写 WAL，再写 MemStore）。

## 题目二：HBase 中 RowKey 设计的核心原则是什么？请举例说明常见的 RowKey 设计策略。

### 参考答案

RowKey 是 HBase 检索数据的**唯一主键**，其设计直接影响读写性能和负载均衡。

### 三大核心原则

1. **长度原则**：RowKey 最大 64KB，建议越短越好（≤16 字节最佳）。因为：
   - MemStore 和 HFile 中的 KeyValue 对象会存储 RowKey，越长存储成本越高。
   - 短 RowKey 减少 HDFS 块扫描范围。

2. **唯一性原则**：RowKey 必须全局唯一，承载业务主键（如 `userId_orderId`）。

3. **散列性原则**：避免热点（Hot Spotting）。热点现象指大量请求集中在少数几个 Region 上，导致负载不均。

### 常见 RowKey 设计策略

| 策略 | 做法 | 适用场景 |
|------|------|----------|
| **加盐（Salting）** | RowKey 头部拼接随机前缀（如 `0_`, `1_`, `2_`），将数据打散到多个 Region | 写入并发高、读请求随机的场景 |
| **哈希（Hash）** | 对业务主键做 MD5/Hash，取前几位作为前缀 | 既要散列又要可还原的场景 |
| **反转（Reverse）** | 将时间戳反转（如 `9999999999 - timestamp`） | 以时间开头、容易产生热点的场景 |
| **业务字段拼接** | `userId + deviceId + timestamp`，利用业务维度天然散列 | 多维查询、范围扫描场景 |

**反热点示例：**
```
# 容易热点（时间戳开头）
RowKey: 20260719_000001

# 反转时间戳（解决了写入热点）
RowKey: 9999999999_20260719_000001

# 哈希打散
RowKey: 3a5f_20260719_000001
```

## 题目三：简述 HBase 的存储结构——MemStore 和 HFile 分别是什么？它们之间的关系是怎样的？

### 参考答案

HBase 采用 **LSM-Tree（Log-Structured Merge Tree）** 思想，数据写入和持久化的核心流程围绕 MemStore 和 HFile 展开。

### MemStore（写缓存）

- 位于 **RegionServer 内存**中，每个 Store（列族）对应一个 MemStore。
- 数据写入时先追加到 **WAL（Write-Ahead Log）**，然后写入 MemStore（按 RowKey 排序）。
- 当 MemStore 达到阈值（默认 ≥ 128MB 或 ≥ 1 小时），触发 **Flush** 操作，将内存中的有序数据落盘为 HFile。
- 作用：缓冲写入、合并随机写为顺序写，大幅提升写吞吐。

### HFile（HBase 的存储文件格式）

- HFile 是 HBase 在 HDFS 上的**最终存储格式**，位于 `/hbase/data/<namespace>/<table>/<region>/<store>/` 目录下。
- HFile 内部采用 **B+Tree 变体**结构（类似 HFile index），包含：
  - **Data Block**（数据块，默认 64KB）：存储 KeyValue 对，按 RowKey 有序。
  - **Meta Block**（元数据块）：布隆过滤器（BloomFilter）信息。
  - **File Info**：最大/最小 RowKey、时间范围等。
  - **Index Block**：数据块索引，支持快速定位。

### MemStore → HFile 的合并（Compaction）

- **Minor Compaction**：将多个较小的 HFile 合并为一个，删除标记为删除的 KeyValue（TTL/版本过期）。
- **Major Compaction**：将一个 Store 下的所有 HFile 合并为一个，**物理删除**标记过期的数据，性能消耗大，通常放在凌晨等低峰期。
- **目的**：减少 HFile 数量，降低读放大（Read Amplification），提升查询性能。

### 整体关系图

```
写入请求 → WAL → MemStore（内存排序）
         ↓ Flush（阈值触发）
      HFile（N 个，持续 Compaction 合并）
```

## 题目四：HBase 的 WAL（Write-Ahead Log）机制是什么？它在数据可靠性中扮演什么角色？

### 参考答案

### WAL 概念

WAL 是 HBase 实现 **高可靠写入**的核心机制。每次数据写入必须先记录到 WAL 日志，才算写入成功。

### 工作原理

1. **写入链路**：客户端发起 Put → 数据写入 **RegionServer 的 WAL**（分布式文件系统 HDFS 上的一个 HLog 文件）→ 写入 MemStore。
2. **崩溃恢复**：如果 RegionServer 在数据写入 MemStore 后、Flush 到 HFile 前宕机，WAL 中的数据可以通过**重放（Replay）**恢复。Master 会将该 Region 分配给其他 RegionServer，后者读取 WAL 并重放未持久化的操作。
3. **WAL 切分**：每个 RegionServer 有一个 WAL 文件（简单理解为按 Region 分区的序列文件）。当 WAL 过大或宕机时，WAL 被切分并分配给对应的新 RegionServer 处理。

### 可靠性保证

- WAL 写入 HDFS 默认 3 副本，天然支持 **宕机恢复**。
- 如果关闭 WAL（`setWriteToWAL(false)`），写入会大幅加快，但**丢失数据的风险由应用承担**。
- 批量写入（Batch Put）可以共享同一个 WAL 实例，减少 I/O 开销。

### WAL 与 MemStore 的关系

```
异常宕机 → WAL 重放 → 恢复 MemStore 未 Flush 的数据
         ↓
   无 WAL → 数据永久丢失
```

## 题目五：HBase 协处理器（Coprocessor）是什么？它有哪两类？各自的使用场景是什么？

### 参考答案

### 协处理器是什么

HBase 协处理器是一种**运行在 RegionServer 上的扩展机制**，允许用户将自定义代码注入到 HBase 的读写路径中，从而在**数据写入/读取时自动执行业务逻辑**，无需在客户端做额外调用。

### 两类协处理器

#### 1. Observer（观察者，类似数据库触发器）

**原理**：在特定的钩子点（Hook）插入业务逻辑，事件触发时执行。

| 类型 | 触发时机 | 典型场景 |
|------|----------|----------|
| **PreGet** | 读取数据之前 | 数据权限校验、动态列过滤 |
| **PostGet** | 读取数据之后 | 审计日志、敏感数据脱敏 |
| **PrePut** | 写入数据之前 | 数据格式校验、字段补全 |
| **PostPut** | 写入数据之后 | 写入从表（同步更新关联数据）、触发通知 |
| **PreDelete** | 删除数据之前 | 权限校验、防误删 |
| **PostDelete** | 删除数据之后 | 级联删除关联数据、删除日志 |

#### 2. Endpoint（端点，类似数据库存储过程）

**原理**：在 **Region 端**（而非客户端）执行计算逻辑，将计算推向数据所在位置，减少网络传输。

- **使用场景**：
  - 对一个大表的所有 RowKey 做聚合（Count、Sum、Avg）。
  - 在服务端完成复杂的过滤和计算，只返回结果集。
- **调用方式**：通过 HBase RPC 调用，客户端需定义 Protocol Buffer 接口。

### 加载方式

| 方式 | 配置位置 | 生效范围 |
|------|----------|----------|
| **表级加载** | `hbase-table.xml`（表描述中指定） | 特定表 |
| **系统级加载** | `hbase-site.xml` 的 `hbase.coprocessor.enabled` | 所有表 |

### 典型使用示例（PrePut 观察者）

```java
@Override
public void prePut(ObserverContext<RegionCoprocessorEnvironment> e, Put put, WALEdit edit)
    throws IOException {
    // 在数据写入前，强制设置一个审核字段
    put.addColumn(Bytes.toBytes("cf"), Bytes.toBytes("audit_flag"),
                  Bytes.toBytes("audited"));
    e.bypass(); // 如需跳过原 Put 操作
}
```

> ⚠️ **注意事项**：协处理器代码异常会导致 RegionServer 崩溃，需做好充分测试和异常捕获。
