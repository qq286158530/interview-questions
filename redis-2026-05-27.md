## Redis持久化：RDB、AOF以及混合持久化有什么区别？分别适合什么场景？

### 三种持久化机制

**1. RDB（Redis Database）——定时快照**

- 核心原理：Redisfork出一个子进程，遍历整个内存数据，将数据序列化写入 RDB 文件
- 触发方式：
  - 手动：`SAVE`（阻塞）或 `BGSAVE`（后台异步）
  - 自动：`save m n`（m 秒内存在 n 次写操作时触发）
- 优点：文件紧凑（压缩二进制）、恢复快、适合做冷备
- 缺点：可能丢失最近一次 BGSAVE 之后的数据；频繁 fork 子进程影响性能

**2. AOF（Append Only File）——命令日志**

- 核心原理：Redis将每一次写操作以 RESP 协议命令格式追加写入 AOF 文件
- 同步策略（`appendfsync`）：
  - `always`：每条命令同步写入（最多丢失一条，但性能最差）
  - `everysec`：每秒 fsync 一次（推荐，默认策略）
  - `no`：由操作系统决定何时同步（最快，但丢失数据不确定）
- 重写机制（AOF Rewrite）：当 AOF 文件过大时，后台自动进行 AOF 重写，将多条命令合并为一条（e.g., `INCR key` × 100 → `SET key 100`），减小文件体积
- 优点：数据安全性更高（everysec 最多丢失 1 秒数据）
- 缺点：AOF 文件通常比 RDB 大，恢复慢

**3. 混合持久化（Redis 4.0+）——兼顾两者**

- 开启：`aof-use-rdb-preamble yes`
- 原理：
  - AOF 重写时，先以 RDB 格式写入全量数据，再以 AOF 格式追加增量命令
  - 文件结构：RDB 头部 + AOF 增量
- 优点：兼顾 RDB 的快速恢复 + AOF 的数据安全性
- 缺点：只支持 RDB 格式恢复，无法用旧版本 Redis 读取

### 如何选择

| 场景 | 推荐策略 |
|------|----------|
| 缓存（数据可丢失） | RDB，关掉 AOF |
| 追求快速恢复 + 数据安全 | 混合持久化 |
| 内存纯缓存 + 严格不丢失 | AOF everysec + 主从复制 |
| 做冷备（异地备份） | 定期 BGSAVE 生成 RDB 文件 |

生产环境建议：主从复制 + 混合持久化（everysec）。

## Redis主从复制原理是什么？数据同步过程是怎样的？如果从库断开了增量复制期间主库有新数据该怎么办？

### 主从复制架构

Redis 支持一主多从的复制拓扑，主库负责写，从库负责读（读写分离）。

### 全量复制（Full Sync）

当从库首次连接主库，或断线重连后数据落后太多时，触发全量复制：

1. **从库发送 `PSYNC ? -1`**：向主库请求全量复制
2. **主库执行 `BGSAVE`**： fork 子进程将当前数据保存为 RDB 文件
3. **主库同时使用复制缓冲区（replication buffer）**：记录 BGSAVE 期间所有写命令
4. **主库将 RDB 文件发送给从库**：从库接收并加载到内存
5. **主库将缓冲区中的增量命令发送给从库**：从库执行这些命令，完成最终同步
6. **之后进入增量复制阶段**

### 增量复制（Command Sync）

全量复制完成后，进入增量复制阶段：

- 主库为每个从库维护一个 **replication backlog buffer**（循环缓冲区，默认 1MB）
- 主库每次处理写命令，同时将命令广播给所有从库
- 从库持续发送 `PSYNC <runid> <offset>`，告知主库自己执行到了哪个位置
- 如果 offset 在缓冲区范围内，主库只发送缺失的部分命令

### 断线重连的处理

```
从库断线 → 重连 → 发送 PSYNC runid offset
                ↓
        主库检查 offset 是否在 backlog 内
            ↓              ↓
         是（仍在）        否（太旧）
          增量复制         全量复制
```

### 主从复制相关重要参数

| 参数 | 说明 |
|------|------|
| `repl-backlog-size` | 复制缓冲区大小，建议 > `单从库数据量 × 网络中断时长` |
| `repl-timeout` | 复制超时时间（默认 60s），超时应答视为断开 |
| `repl-diskless-sync` | 开启无盘复制（通过 socket 直接传 RDB） |

### 常见问题

- **从库越多复制越慢**：每个从库都占用主库带宽，建议最多 2-3 个从库
- **主库宕机**：需要人工介入或借助 Sentinel/Cluster 自动故障转移

## Redis哨兵（Sentinel）是如何实现自动故障转移的？主观下线和客观下线有什么区别？

### 哨兵架构概述

Redis Sentinel 是 Redis 的高可用解决方案，由一组（建议至少 3 个）Sentinel 进程组成，负责监控主从库的可用性并在主库故障时自动完成故障转移。

### 故障转移流程（核心步骤）

**第一步：主观下线（SDown）**
单个 Sentinel 进程 ping 主库，如果超过 `down-after-milliseconds` 没有响应，则该 Sentinel 认为主库主观下线。

**第二步：客观下线（ODown）**
主库被一个 Sentinel 标记为 SDown 后，该 Sentinel 向其他 Sentinel 发送 `SENTINEL is-master-down-by-addr` 询问。如果达到 `quorum` 数量（ configurable，最少 2 个）的 Sentinel 都认为主库下线，则主库进入客观下线状态。

**第三步：Sentinel  Leader 选举**
当主库客观下线后，所有 Sentinel 需要选举出一个 leader 来执行故障转移：
- 采用 Raft 算法的变种：每个候选的 Sentinel 给自己投票，向其他 Sentinel 拉票
- 获得超过半数票的 Sentinel 成为 leader
- 如果票数相同，则等待重新选举（超时后重新发起）

**第四步：故障转移（由 leader 执行）**
1. 从所有从库中挑选一个最优的成为新主库：
   - 优先级 `replica-priority` 最高的
   - 优先级相同，则选择 `offset` 最新的（复制进度最完整）
   - 还相同，则选择 runid 字典序最小的
2. 向新主库执行 `SLAVEOF NO ONE`，将其升级为主库
3. 向其他从库发送 `SLAVEOF 新主库IP:PORT`，让他们指向新主库
4. 旧主库重新上线后，Sentinel 将其降为从库指向新主库

### 哨兵集群的职责

| 职责 | 说明 |
|------|------|
| 监控（Monitoring） | 每个 Sentinel 不断检查主从库是否正常运行 |
| 自动故障转移（Automatic Failover） | 主库不可用时，自动将某个从库升级为主库 |
| 通知（Notification） | 故障转移完成后，通过 API/客户端通知管理员 |
| 配置提供者（Configuration Provider） | 客户端连接 Sentinel 获取当前主库地址 |

### 重要配置参数

```bash
sentinel monitor mymaster <主库IP> <端口> <quorum票数>
sentinel down-after-milliseconds mymaster 30000  # 30s没响应则认为主观下线
sentinel failover-timeout mymaster 180000        # 故障转移超时时间
sentinel parallel-syncs mymaster 1               # 同时同步新主库的从库数量
```

### 主观下线 vs 客观下线

| | 主观下线（SDown） | 客观下线（ODown） |
|---|---|---|
| 触发者 | 单个 Sentinel | 多个 Sentinel 协商 |
| 条件 | 一个 Sentinel ping 不通主库 | `quorum` 数量的 Sentinel 都认为主库下线 |
| 结果 | 只是该 Sentinel 的本地判断 | 触发真正的故障转移流程 |

## Redis Cluster是如何实现数据分片的？槽（slot）和节点之间的关系是什么？如果某个节点挂了会发生什么？

### Redis Cluster 数据分片原理

Redis Cluster 通过 **哈希槽（Hash Slot）** 实现数据分片：

- **共 16384 个槽**（slot），编号 0 ~ 16383
- 每个 key 通过 `CRC16(key) % 16384` 计算出它属于哪个槽
- 这 16384 个槽分布在 Cluster 的所有主节点上

```
举例：3 个主节点的分片
节点A：槽 0 ~ 5460
节点B：槽 5461 ~ 10922
节点C：槽 10923 ~ 16383
```

### 槽信息在集群中的传播

- 每个节点都持有 **集群总线协议（Cluster Bus Protocol）**，通过 Gossip 协议（随机扩散）在节点间互相传播槽信息
- 新节点加入或节点下线后，槽信息会通过 Gossip 协议逐步收敛到最终一致
- 客户端可以连接任意节点，节点会根据槽信息计算 key 所属节点并重定向客户端（MOVED 重定向）

### 故障转移机制（从节点晋升）

当某个主节点及其所有从节点都宕机时，该槽范围内没有可用的节点，集群会进入**不可用状态**。

正常运行时的故障转移流程：

1. **从库检测到主库主观下线**：通过心跳机制，超过 `cluster-node-timeout` 没收到主库消息
2. **从库申请成为主库**：向集群中其他节点广播 `FAILOVER_REQUEST` 消息
3. **其他节点投票**：如果从库获得超过半数的投票，则晋升为新的主库
4. **新主库接管槽位**：原来的主库下线后，其负责的槽位由新主库接管

### 集群节点间的通信——Gossip 协议

Redis Cluster 节点之间通过 Gossip 协议（类似 Cassandra/P2P 网络）进行通信：
- 每秒随机选择一个节点发送 ping，包含自己的状态信息和部分其他节点的信息
- 被 ping 的节点回复 pong
- 消息体很小（约 1KB），即使集群有 1000 个节点，开销也很低
- 节点间通信端口：`客户端端口 + 10000`（如客户端 6379，则集群总线端口 16379）

### 集群伸缩——槽迁移

添加新节点时的槽迁移步骤：
1. 新节点加入集群，开始接收少量槽
2. 使用 `redis-cli --cluster reshard <任意节点IP:端口>` 指定将哪些槽迁移到新节点
3. 在迁移过程中，被迁移的 key 会先从源节点获取（返回 ASK 重定向，客户端继续到目标节点）
4. 迁移完成后，更新槽映射表并通过 Gossip 广播

### 为什么是 16384 个槽？

- 节点间心跳消息中需要携带所有槽信息，16384 ≈ 2KB，而 65536 ≈ 8KB
- 16384 是 2^14，足够满足绝大多数业务分片需求
- Redis Cluster 官方认为 16384 是性能和网络开销的最优平衡点

## 什么是缓存穿透、缓存击穿、缓存雪崩？如何解决？Redis有什么具体的实现方案？

### 缓存穿透（Cache Penetration）

**定义**：查询一个既不存在于缓存、也不存在于数据库的数据（"查不到"的数据）。大量请求直接打到数据库，可能导致数据库崩溃。

**常见场景**：
- 恶意请求：用不存在的主键 ID 频繁查询
- 业务逻辑错误：查询了不应该存在的 key

**解决方案**：

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| **布隆过滤器（Bloom Filter）** | 将所有合法 key 存入布隆过滤器，查询前先检查是否存在 | 内存占用低，但有误判率（可接受） |
| **缓存空值（Null Value）** | 将"查不到"的结果也缓存，value 设为 null，过期时间短一些 | 简单，但浪费内存 |
| **参数校验** | 接入层做 ID 合法性校验，过滤非法请求 | 从源头解决问题 |

### 缓存击穿（Cache Breakdown）

**定义**：某个热点 key 突然过期，大量请求同时涌入，直接打到数据库。常见于秒杀活动、热榜等场景。

**解决方案**：

| 方案 | 原理 |
|------|------|
| **互斥锁（SETNX）** | 缓存失效时，用 `SET key value NX EX 10` 只允许一个请求去查库，其他等待 |
| **逻辑过期（不过期）** | value 中存储 `data + expireTime`，后台异步更新数据，查询时返回旧数据 |
| **永不过期** | 对于热点数据，不设置过期时间，由业务逻辑控制更新 |

Redis 实现互斥锁示例：
```python
# Redis 分布式锁
lock = redis.set("lock:product:123", "1", nx=True, ex=10)
if lock:
    data = db.query("SELECT * FROM products WHERE id = 123")
    redis.setex("product:123", 300, json.dumps(data))
    redis.delete("lock:product:123")
else:
    time.sleep(0.1)
    return get_from_cache("product:123")
```

### 缓存雪崩（Cache Avalanche）

**定义**：大量缓存 key 在同一时间集中过期，或者 Redis 本身宕机，导致所有请求打到数据库。

**两种情况**：

1. **大量 key 同时过期**：
   - 原因：统一设置了相同的过期时间（e.g., `expire = now + 3600`）
   - 解决方案：过期时间加上随机偏移量（e.g., `expire = base + rand(0, 300)`）

2. **Redis 宕机**：
   - 原因：Redis 挂了，所有请求穿透到数据库
   - 解决方案：
     - 架构层面：Redis 主从 + Sentinel/Cluster 高可用
     - 本地缓存兜底：Caffeine/Guava Cache 做本地缓存
     - 流量保护：接口限流（rate limiter）

### 三种问题的对比

| 问题 | 原因 | 核心解决思路 |
|------|------|-------------|
| 缓存穿透 | 查询不存在的数据 | 布隆过滤器 / 缓存空值 |
| 缓存击穿 | 热点 key 突然过期 | 互斥锁 / 逻辑过期 |
| 缓存雪崩 | 大量 key 同时过期 或 Redis 宕机 | 过期时间随机化 + 高可用架构 + 本地缓存兜底 |
