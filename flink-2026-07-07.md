# Flink 面试题 - 2026-07-07

## 题目一：Flink 的架构是怎样的？为什么 Flink 能做到精确一次语义？

**参考答案：**

Flink 采用**分层架构**，主要分为四层：

1. **API & Libraries 层**：提供 DataStream API（流处理）和 DataSet API（批处理）
2. **Runtime 层**：核心执行引擎，采用 **DAG**（有向无环图）结构
3. **Physical Layer**：将 DAG 转换为 Task 的物理部署（TaskManager、Slot）
4. **物理部署层**：JobManager（协调）+ TaskManager（执行）

**为什么 Flink 能做到精确一次（Exactly-Once）？**

Flink 的精确一次语义依托于 **Chandy-Lamport 分布式快照算法**的变体——**Barrier（屏障）机制**：

1. **Checkpoint 屏障**：JobManager 定期向数据流中插入 Barrier 标记
2. **对齐（Align）阶段**：当 Source 收到 Barrier 时，保存当前状态快照
3. **异步快照**：将状态快照异步写入持久化存储（如 HDFS）
4. **两阶段提交**：Sink 端采用两阶段提交协议，确保输出只在 checkpoint 完成后才对外可见

相比其他框架，Flink 的核心优势是：**状态后端 + Checkpoint + 两阶段提交** 三者配合，真正实现端到端的精确一次。

## 题目二：Flink 的 Watermark（水位线）机制是什么？如何处理乱序数据？

**参考答案：**

### 什么是 Watermark？

Watermark 是 Flink 用于处理**事件时间（Event Time）**乱序问题的机制。本质上是一个**时间戳 T**，表示"所有时间戳 ≤ T 的数据已经全部到达"。

### 核心概念

| 概念 | 说明 |
|------|------|
| Event Time | 数据自带的事件时间（业务时间） |
| Watermark T | 含义：不再接收时间戳 < T 的数据 |
| 乱序窗口 | Watermark 触发窗口计算前，等待迟到数据 |

### 处理乱序的策略

**1. Watermark 机制（WaterMarkEventTimeStrategy）**
```
Watermark = 当前最大事件时间 - 允许延迟时间
```
- 当 Watermark ≥ 窗口结束时间，窗口触发计算
- 超出 Watermark 的数据被**丢弃**或送往**侧输出流**

**2. 窗口allowedLateness 延迟机制**
```java
window.timeWindow(Time.minutes(5))
      .allowedLateness(Time.minutes(1))  // 允许额外迟到1分钟
```
- 窗口触发后，保留状态 1 分钟
- 迟到但未超期的数据仍可触发窗口重新计算

**3. 侧输出流（Side Output）**
```java
OutputTag<String> lateOutput = new OutputTag<>("late-data");
DataStream<String> result = stream
    .keyBy(...)
    .window(...)
    .sideOutputLateData(lateOutput)  // 迟到数据侧输出
```

### 常见 Watermark 生成策略
- **FixedWatermarkStrategy**：固定延迟
- **MonotonousWatermarkStrategy**：单调递增（无乱序场景）
- **PeriodicWatermarkStrategy**：周期性生成
- **PunctuatedWatermarkStrategy**：基于特定事件触发

## 题目三：Flink 的 Checkpoint 和 Savepoint 有什么区别？

**参考答案：**

| 对比维度 | Checkpoint | Savepoint |
|---------|------------|-----------|
| **触发方式** | Flink 自动周期性触发 | 手动触发（用户命令） |
| **用途** | 容错恢复，自动从失败恢复 | 计划性维护，版本迁移 |
| **删除机制** | Job 取消/失败时自动删除 | 手动管理，不自动删除 |
| **存储位置** | State Backend（内存/HDFS/RocksDB） | 外部文件系统（HDFS/S3/NAS） |
| **兼容性** | 同一 Job ID，状态格式兼容 | 不同版本可能不兼容 |
| **性能影响** | 异步执行，影响较小 | 需要暂停 Barrier 对齐，有暂停时间 |

### Checkpoint 工作流程

```
1. JobManager 定期向 Source 插入 Checkpoint Barrier
2. Source 快照自身状态，发送 Barrier 到下游
3. 中间算子收到所有输入的 Barrier 后快照状态
4. Sink 确认快照完成，通知 JobManager Checkpoint 完成
5. 失败时从最近成功的 Checkpoint 恢复
```

### Savepoint 使用场景

```bash
# 创建 Savepoint
flink savepoint - jobId <job_id> /path/to/savepoint

# 从 Savepoint 恢复
flink run -s /path/to/savepoint/.../sv_xx -d <jar>

# Job 升级/迁移
# 修改代码后从 Savepoint 恢复
```

### 推荐配置

```yaml
execution.checkpointing.interval: 1min          # Checkpoint 间隔
execution.checkpointing.mode: EXACTLY_ONCE      # 精确一次
execution.checkpointing.timeout: 10min          # 超时时间
execution.checkpointing.externalized-checkpoint: DELETE_ON_CANCELLATION  # 取消时策略
```

## 题目四：Flink 的状态后端（State Backend）有哪些？如何选择？

**参考答案：**

### 三种 State Backend 对比

| 类型 | 存储介质 | 适用场景 | 优缺点 |
|------|---------|---------|--------|
| **HashMapStateBackend** | JVM Heap | 小状态、低延迟场景 | 速度快，但受内存限制 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘） | 大状态、长窗口 | 支持增量 Checkpoint，但延迟略高 |
| **EmbeddedRocksDBStateBackend（远程）** | RocksDB + 序列化到远程 | 超大规模状态 | 需要序列化开销 |

### 详细说明

**1. HashMapStateBackend**
```java
// 代码配置
env.setStateBackend(new HashMapStateBackend());

// 特点
- 状态存储在 TaskManager 的 JVM Heap
- 性能最高，适合 TB 级状态以下
- 状态大小受内存限制
```

**2. EmbeddedRocksDBStateBackend**
```java
// 代码配置
env.setStateBackend(new EmbeddedRocksDBStateBackend());

// 特点
- 状态溢出到本地 RocksDB
- 支持增量 Checkpoint（只同步增量变更）
- 适合大状态（TB 级）
- 序列化/反序列化开销
```

### 选择建议

| 数据量 | 推荐方案 |
|--------|---------|
| < 1GB | HashMapStateBackend |
| 1GB ~ 100GB | EmbeddedRocksDBStateBackend |
| > 100GB | EmbeddedRocksDBStateBackend + 增量 Checkpoint |

### 状态分类

- **Keyed State**：基于 Key 的状态（keyBy 之后），包括 ValueState、ListState、MapState、ReducingState 等
- **Operator State**：算子状态（非 Keyed），需要做 Checkpoint 时实现 CheckpointedFunction

## 题目五：Flink 窗口函数有哪些？如何处理窗口计算？

**参考答案：**

### 窗口分类

```
Flink 窗口
├── 时间窗口（Time Window）
│   ├── 滚动窗口（Tumbling）
│   ├── 滑动窗口（Sliding）
│   └── 会话窗口（Session）
└── 计数窗口（Count Window）
    ├── 滚动计数窗口
    └── 滑动计数窗口
```

### 1. 滚动窗口（Tumbling Window）
```java
// 每5分钟一个窗口，不重叠
stream.keyBy(...)
      .window(TumblingProcessingTimeWindows.of(Time.minutes(5)))
      .sum("field");
```

### 2. 滑动窗口（Sliding Window）
```java
// 窗口大小10分钟，滑动步长5分钟
stream.keyBy(...)
      .window(SlidingProcessingTimeWindows.of(Time.minutes(10), Time.minutes(5)))
      .sum("field");
```

### 3. 会话窗口（Session Window）
```java
// 间隔10秒内的数据归为同一会话
stream.keyBy(...)
      .window(ProcessingTimeSessionWindows.withGap(Time.seconds(10)))
      .sum("field");
```

### 窗口函数类型

| 函数类型 | 说明 | 示例 |
|---------|------|------|
| **reduce()** | 增量聚合，输出一个元素 | 求和、极值 |
| **aggregate()** | 通用增量聚合 | 复杂聚合逻辑 |
| **apply()** | 全量窗口计算 | 全量数据处理 |
| **process()** | 底层 ProcessWindowFunction | 最灵活，支持上下文 |

### 示例：aggregate() 实现平均计算
```java
DataStream<Tuple2<String, Long>> input = ...

input.keyBy(x -> x.f0)
     .window(TumblingProcessingTimeWindows.of(Time.minutes(1)))
     .aggregate(new AverageAggregate());

// 实现 AggregateFunction
public static class AverageAggregate 
    implements AggregateFunction<Tuple2<String, Long>, Tuple2<Long, Long>, Double> {
    
    @Override
    public Tuple2<Long, Long> createAccumulator() {
        return new Tuple2<>(0L, 0L);
    }
    
    @Override
    public Tuple2<Long, Long> add(Tuple2<String, Long> value, 
                                   Tuple2<Long, Long> acc) {
        return new Tuple2<>(acc.f0 + value.f1, acc.f1 + 1);
    }
    
    @Override
    public Double getResult(Tuple2<Long, Long> acc) {
        return (double) acc.f0 / acc.f1;
    }
    
    @Override
    public Tuple2<Long, Long> merge(Tuple2<Long, Long> a, 
                                     Tuple2<Long, Long> b) {
        return new Tuple2<>(a.f0 + b.f0, a.f1 + b.f1);
    }
}
```

### 窗口触发时机

- **ProcessingTime**：按照机器时间触发（系统时钟）
- **EventTime**：按照数据自带的时间戳触发（需要 Watermark）
