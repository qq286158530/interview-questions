## 题目一：MySQL 慢查询如何分析和优化？

### 参考答案

**分析手段：**

1. **慢查询日志**
   - 通过 `slow_query_log = ON` 开启慢查询日志
   - 设置 `long_query_time` 阈值（默认 10 秒，生产环境通常设为 1-3 秒）
   - 使用 `mysqldumpslow` 工具分析日志

2. **EXPLAIN 执行计划分析**
   - 重点关注：`type`（扫描类型）、`key`（实际用到的索引）、`rows`（扫描行数）、`Extra`
   - 常见 type 从好到差：`const > eq_ref > ref > range > index > all`
   - Extra 出现 `Using filesort` / `Using temporary` 需要优化

3. **SHOW PROFILE**
   - 开启：`SET profiling = 1`
   - 查看：`SHOW PROFILES` + `SHOW PROFILE FOR QUERY N`
   - 可分析 CPU、IO、内存等资源消耗

**优化步骤：**

1. 运行 SQL 时加 `SQL_NO_CACHE`，确认是否真的慢
2. 用 `EXPLAIN` 查看执行计划，确认是否按预期走索引
3. 对 WHERE 条件字段逐一排查，找出区分度最高的字段优先建立索引
4. `ORDER BY LIMIT` 形式的语句优先排序
5. 参照建索引原则（最左前缀、覆盖索引、避免回表等）
6. 观察结果，不符合预期则重新分析

## 题目二：MySQL 索引优化的核心原则有哪些？

### 参考答案

1. **复合索引优于单列索引组合**
   - 尽量创建联合索引，让多个查询条件复用同一个索引

2. **利用覆盖索引减少回表**
   - 如果查询字段在索引中已包含，则无需回表查询主键索引
   - 示例：`SELECT user_id, status FROM orders WHERE user_id = 100` 优于 `SELECT * FROM orders WHERE user_id = 100`

3. **主键使用自增 ID**
   - 避免页分裂（page split），保证数据顺序写入，插入效率高

4. **避免 `SELECT *`**
   - 只查需要的字段，减少网络传输和回表开销

5. **使用短索引**
   - 减少非叶子节点占用空间，降低 B+ 树层高，减少磁盘 IO

6. **避免索引失效**
   - 不在索引列上使用函数或运算：`WHERE YEAR(create_time) = 2026`
   - 避免左模糊 `%like`：`WHERE name LIKE '%abc'`
   - 遵循最左前缀原则
   - OR 前后字段不同时有索引会导致索引失效

7. **选择区分度高的列建索引**
   - 区分度 = 不同值数量 / 总行数，越接近 1 越好（如 UUID、订单号）
   - 性别、状态等低区分度字段不适合建索引

## 题目三：什么是聚簇索引和二级索引？InnoDB 为什么选择 B+ 树？

### 参考答案

**聚簇索引（Clustered Index）**
- 主键索引即聚簇索引，叶子节点存储完整行数据
- 每张表只有一个聚簇索引

**二级索引（Secondary Index）**
- 也称辅助索引，非主键字段上的索引
- 叶子节点存储主键值，而非完整数据
- 查询时需要先查二级索引找到主键，再回表查主键索引获取完整数据

**为什么用 B+ 树？**

| 对比项 | B+ 树 | B 树 | 二叉树/红黑树 | Hash |
|--------|-------|------|-------------|------|
| 单节点数据量 | 大（只存索引） | 小（同时存数据） | 小 | - |
| 范围查询 | 叶子链表，O(1) | 需要中序遍历 | 无法高效范围 | 不支持 |
| 查询效率 | O(logd N)，d>100 | O(logd N) | O(log N) | O(1) |
| 千万级数据 IO 次数 | **3-4 次** | 较多 | 很多 | - |

- B+ 树只有叶子节点存数据，非叶子节点可容纳更多索引条目，相同数据量下树高更矮，磁盘 IO 更少
- 叶子节点双向链表连接，适合范围查询和顺序访问
- MySQL InnoDB 将主键和数据一起存在 B+ 树叶子节点，实现了聚簇索引

## 题目四：什么是索引下推？MySQL 5.6 之后如何优化联合索引查询？

### 参考答案

**索引下推（Index Condition Pushdown，ICP）**

在联合索引 `(a, b)` 查询 `WHERE a > 1 AND b = 2` 时：

- **MySQL 5.6 之前**：先在二级索引中找到所有满足 `a > 1` 的主键值，然后逐一回表，再在主键索引中判断 `b = 2`。每个满足条件的主键都需要回表。
- **MySQL 5.6 之后**：索引下推优化允许在遍历二级索引时，**直接在索引层面对 `b` 字段进行判断**，过滤掉不满足 `b = 2` 的记录，只将符合条件的主键值回表，大幅减少回表次数。

**如何验证：**
通过 `EXPLAIN` 查看执行计划，Extra 列出现 `Using index condition` 即表示使用了索引下推。

**最左匹配原则的特殊情况：**
- `>`、`<`、`<=`、`>=` 会导致最左匹配**停止**，后续字段无法利用索引
- `BETWEEN ... AND ...`、like 前缀匹配、`IN`、`=`，不会停止最左匹配

## 题目五：MySQL 主从复制原理是什么？如何通过读写分离优化性能？

### 参考答案

**主从复制原理：**

1. 主库将写操作记录到 **binlog**（二进制日志）
2. 从库通过 **I/O 线程**读取主库的 binlog，写入本地 **relay log**（中继日志）
3. 从库的 **SQL 线程**读取 relay log，在从库重放执行

```
Master → binlog → I/O Thread → relay log → SQL Thread → Slave
```

**读写分离优化：**
- 将读请求分发到从库，写请求访问主库
- 适用于读多写少场景，有效分担主库压力
- 常见实现：基于应用层配置（多数）、或中间件（MySQL Proxy / Atlas / ShardingSphere）

**常见问题：**
- **主从延迟**：从库同步滞后导致读取到旧数据，可采用延迟读或强制读主库
- **数据一致性**：半同步复制（semi-sync）可保证至少一个从库收到日志

**分库分表进阶方案：**
- 单表超过 **500 万行**或容量超 **2GB** 时建议分表
- 水平分表：按主键 ID 范围或 Hash 分散数据
- 垂直分表：将大字段（TEXT/BLOB）独立成表，减少主表 IO
