## 题目一：HBase 的架构组成及各组件作用

**来源：**
- https://www.tutorialspoint.com/hbase/hbase_interview_questions.htm
- https://data-flair.training/blogs/hbase-interview-questions/

**题目：** 请描述 HBase 的整体架构，主要包含哪些组件？各自的作用是什么？

**参考答案：**

HBase 是一个基于 HDFS 的列式存储数据库，其架构主要由以下三个核心组件构成：

### 1. HMaster（主服务器）
- 负责监控集群中所有的 RegionServer 实例
- 处理表的创建、删除、修改等元数据操作
- 负责将 Region 分配给 RegionServer，并进行负载均衡
- 在分布式集群中，通常运行在 NameNode 节点上
- 处理 Region 的分裂和迁移

### 2. RegionServer（区域服务器）
- 负责管理和服务其所拥有的 Region
- 直接处理客户端的读写请求
- 每个 RegionServer 在分布式集群中通常运行在 DataNode 节点上
- 包含以下关键组件：
  - **MemStore**：写缓存，先写入内存再刷写到磁盘
  - **Store**：对应一个列族，包含 MemStore 和多个 HFile
  - **HFile**：HBase 中实际存储数据的文件格式

### 3. ZooKeeper
- 维护集群的配置信息
- 提供分布式同步机制
- 负责 HMaster 和 RegionServer 之间的通信
- 监控 RegionServer 的上下线状态，实现故障转移
- 存储 `-ROOT-` 和 `.META.` 表的位置信息

**架构流程简述：**
```
客户端 → ZooKeeper → 获取 .META. 位置 → 访问 RegionServer → 操作数据
```

## 题目二：HBase 的 RowKey 设计原则与实践

**来源：**
- https://data-flair.training/blogs/hbase-interview-questions/
- https://www.tutorialspoint.com/hbase/hbase_interview_questions.htm

**题目：** 在 HBase 中，RowKey 设计的核心原则是什么？如何避免热点问题？

**参考答案：**

### RowKey 的核心特性
- 每行数据由 RowKey 唯一标识
- RowKey 是字节数组，没有固定类型
- HBase 中的数据按 RowKey 的字典序排序存储

### 设计原则

**1. 唯一性原则**
RowKey 必须唯一标识一行数据，类似主键的概念。

**2. 散列性原则（避免热点）**
热点问题（Hotspotting）是指大量请求集中在少数几个 Region 上，导致负载不均。避免方法：

- **加盐（Salting）**：在 RowKey 头部添加随机前缀，使数据分散到不同 Region
- **哈希（Hash）**：对业务主键做 MD5 或其他哈希后再作为 RowKey
- **反转（Reverse）**：反转固定长度或高可预测性的字段

**3. 长度控制**
- RowKey 建议控制在 16~100 字节以内
- 越短越好，过长的 RowKey 会占用过多 MemStore 内存

**4. 业务查询友好**
RowKey 设计要结合实际查询场景：
- 如果按时间范围查询，时间戳应放在 RowKey 末尾
- 如果按用户 ID 查询，用户 ID 应放在 RowKey 前面

### 常见 RowKey 设计模式
```
# 用户ID + 时间戳反转（实现按用户查询最近记录）
userId + String.valueOf(Long.MAX_VALUE - timestamp)

# 哈希 + 业务ID（打散避免热点）
MD5(userId).substring(0,6) + userId
```

## 题目三：MemStore 与 HFile 的工作机制

**来源：**
- https://www.tutorialspoint.com/hbase/hbase_interview_questions.htm

**题目：** 请描述 HBase 写数据的完整流程，以及 MemStore 刷写和 HFile 合并的工作机制。

**参考答案：**

### HBase 写数据完整流程

```
1. 客户端发起写入请求
       ↓
2. 数据首先写入 WAL（Write Ahead Log），保证故障恢复
       ↓
3. 数据写入 MemStore（内存中的写缓存）
       ↓
4. 返回客户端写入成功
       ↓（满足条件后）
5. MemStore 刷写（Flush）到磁盘，生成 HFile
```

### MemStore 刷写条件
以下任一条件满足时触发刷写：
- MemStore 大小达到 `hbase.hregion.memstore.flush.size`（默认 128MB）
- Region 中所有 MemStore 总大小超过堆内存的 40%
- WAL 日志数量超过阈值
-定时刷写（默认 1 小时）

### HFile 的结构
HFile 是 HBase 实际存储数据的文件格式，特点：
- 基于 SSTable（BigTable 论文）设计
- 按 KeyValue 对存储，支持随机读取
- 包含数据块、索引块、布隆过滤器等

### Compaction（合并）机制

HBase 有两种 Compaction 类型：

**Minor Compaction（小合并）：**
- 将多个相邻的小 HFile 合并为一个较大的 HFile
- 不会删除已标记删除的数据
- 由系统自动触发

**Major Compaction（大合并）：**
- 将一个 Store 中所有 HFile 合并为一个
- 会清理已删除和过期的数据版本
- 通常需要手动触发或在低峰期自动执行

### 数据读取流程
```
MemStore（内存） + 多个 HFile（磁盘） → 合并返回结果
```

## 题目四：HBase 的协处理器（Coprocessor）是什么？如何使用？

**来源：**
- Apache HBase 官方文档：https://hbase.apache.org/book.html#cp

**题目：** 什么是 HBase 协处理器？它有哪几种类型？各自的使用场景是什么？

**参考答案：**

### 协处理器简介
协处理器是运行在 RegionServer 上的代码片段，允许用户在数据写入和读取时拦截并执行自定义逻辑，无需修改 HBase 核心代码。

### 两种协处理器类型

#### 1. Observer（观察者）
类似于数据库触发器，在特定事件发生时被调用。

**三种 Observer：**
- **RegionObserver**：拦截 GET、PUT、DELETE、SCAN 等操作
  - 使用场景：数据验证、访问控制、动态列值注入
- **WALObserver**：拦截 WAL 写入操作
  - 使用场景：WAL 复制到外部系统
- **MasterObserver**：拦截 DDL 操作（创建/删除表等）
  - 使用场景：审计日志、表结构验证

#### 2. Endpoint（端点）
类似于存储过程，在 RegionServer 上执行计算并返回结果。

**使用场景：**
- 在服务器端执行聚合计算（COUNT、SUM、AVG）
- 减少网络传输的数据量
- 支持自定义过滤和计算逻辑

### 使用方式

协处理器需要通过 HBase 配置加载：

**方式一：通过 HTableDescriptor 加载**
```java
HTableDescriptor td = new HTableDescriptor("myTable");
td.addCoprocessor("com.example.MyObserver");
```

**方式二：在 hbase-site.xml 中配置**
```xml
<property>
  <name>hbase.coprocessor.region.classes</name>
  <value>com.example.MyObserver</value>
</property>
```

### 注意事项
- 协处理器运行在 RegionServer 上，代码错误可能导致 RegionServer 不稳定
- 生产环境使用前需充分测试
- 优先考虑使用 Filter 而非 Coprocessor 进行简单过滤

## 题目五：HBase 如何实现高可用性？故障恢复机制是怎样的？

**来源：**
- https://data-flair.training/blogs/hbase-interview-questions/
- https://www.tutorialspoint.com/hbase/hbase_interview_questions.htm

**题目：** HBase 的高可用性是如何实现的？当 RegionServer 崩溃时，数据如何恢复？

**参考答案：**

### HBase 高可用性实现机制

#### 1. Region 副本机制（Region Replication）
- HBase 支持为每个 Region 创建多个副本
- 副本分布在不同的 RegionServer 上
- 读请求可以分发到多个副本，提升读取吞吐量
- ZooKeeper 确保同一 Region 不会同时在多个 RegionServer 上作为主副本

#### 2. ZooKeeper 监控
- ZooKeeper 实时监控所有 RegionServer 的健康状态
- 当 RegionServer 宕机时，ZooKeeper 会通知 HMaster
- HMaster 自动将该 RegionServer 上的所有 Region 重新分配给其他存活的 RegionServer

### 故障恢复机制

#### 步骤一：WAL 日志保证数据不丢失
当数据写入 HBase 时：
```
1. 数据先写入 WAL（Write Ahead Log）
2. 再写入 MemStore
```
如果 RegionServer 在 MemStore 数据刷写前崩溃，WAL 中的数据不会丢失。

#### 步骤二：故障检测与 Region 重分配
```
RegionServer 崩溃
       ↓
ZooKeeper 检测到心跳超时，通知 HMaster
       ↓
HMaster 将该 RegionServer 上的 Region 标记为离线
       ↓
HMaster 从 -ROOT- 和 .META. 表获取 Region 元信息
       ↓
HMaster 将 Region 分配给其他存活的 RegionServer
```

#### 步骤三：WAL 重放（Log Splitting）
- HMaster 从崩溃的 RegionServer 上读取 WAL 文件
- 将 WAL 中属于每个 Region 的操作日志分割出来
- 在新分配的 RegionServer 上重放这些操作，恢复 MemStore 数据

#### 步骤四：MemStore 数据恢复
- WAL 中的操作被重放后，MemStore 被重建
- 如果 MemStore 中的数据已经刷写为 HFile，则从 HFile 恢复

### 恢复流程图
```
RegionServer 崩溃
       ↓
WAL 文件保留在 HDFS 上
       ↓
HMaster 触发 Log Splitting
       ↓
按 Region 分组 WAL 日志
       ↓
在新 RegionServer 上重放日志
       ↓
MemStore 恢复完成
```

### 其他高可用措施
- **HDFS 副本**：HBase 数据默认有 3 份副本，存储在 HDFS 上
- **Master 高可用**：可配置多个 HMaster，通过 ZooKeeper 选主
- **WAL 复制**：通过 HDFS 的副本机制，WAL 也被复制多份
