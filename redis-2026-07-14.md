## 题目一：Redis 为什么采用单线程模型？为什么单线程还能达到很高的 QPS？

### 参考答案

Redis 的单线程指的是**「接收客户端请求→解析请求→执行命令→返回结果」**这一过程由主线程完成，但 Redis 本身并不是完全单线程的，它在启动时会启动多个后台线程（BIO）来处理耗时任务。

**Redis 采用单线程的原因：**

1. **CPU 不是瓶颈**：Redis 的主要操作都在内存中完成，采用高效的数据结构（SDS、跳表、哈希表等），性能瓶颈往往是内存或网络带宽，而非 CPU。既然 CPU 不是瓶颈，单线程就能满足需求。
2. **避免竞争开销**：多线程模型会引入线程切换、锁竞争、死锁等问题，增加系统复杂度，而单线程完全不存在这些开销。
3. **I/O 多路复用**：Redis 使用 `epoll/select` 等 I/O 多路复用机制，一个线程可以同时处理多个 Socket 连接请求，实现了高性能。

**Redis 单线程依然很快的原因：**

- 操作在内存中完成，没有磁盘 I/O 的阻塞
- 采用了高效的数据结构，时间复杂度低
- I/O 多路复用机制避免了线程阻塞
- 单线程避免了上下文切换和锁开销

## 题目二：Redis 有哪几种持久化方式？各自的优缺点是什么？

### 参考答案

Redis 有三种持久化方式：**RDB、AOF、混合持久化**。

### 1. RDB（Redis Database）快照

RDB 快照是将某一时刻的内存数据以**二进制**格式写入磁盘。

- **优点**：恢复速度快（二进制直接加载），文件紧凑，适合全量备份
- **缺点**：无法做到实时持久化（可能丢失最近一次快照后的数据），fork 子进程时如果数据量大会有压力
- **触发方式**：`save`（主线程执行，会阻塞）或 `bgsave`（子进程执行，不阻塞）

### 2. AOF（Append Only File）日志

AOF 通过记录每一次**写操作命令**到文件末尾来实现持久化。

- **优点**：数据安全性更高，丢失数据少
- **缺点**：恢复速度慢（需要逐一执行命令），AOF 文件体积可能很大
- **三种写回策略**：
  - `always`：每次写命令执行完后同步写回磁盘，可靠性最高，性能最差
  - `everysec`（默认）：每秒同步一次，折中方案
  - `no`：由操作系统决定何时写回，性能最好，可靠性最低

AOF 还支持**重写机制**，在后台通过子进程 `bgrewriteaof` 压缩冗余命令，解决文件过大的问题。

### 3. 混合持久化（Redis 4.0+）

开启混合持久化后，AOF 重写时前半部分以 **RDB 格式**保存全量数据，后半部分以 **AOF 格式**保存增量命令。

- **优点**：结合了 RDB 快速恢复和 AOF 数据完整性，Redis 重启时先快速加载 RDB 部分，再执行增量 AOF
- **缺点**：AOF 文件可读性变差，且不兼容 Redis 4.0 之前的版本

## 题目三：Redis 的过期删除策略是什么？内存淘汰策略有哪些？

### 参考答案

### 过期删除策略

Redis 采用**「惰性删除 + 定期删除」**组合策略：

1. **惰性删除**：每次访问 key 时才检查是否过期，过期才删除。优点是对 CPU 友好（几乎不占用额外资源），缺点是对内存不友好（过期 key 会一直占用内存）。
2. **定期删除**：每隔一段时间随机抽取一批 key 检查，删除其中过期的 key。流程是随机取 20 个 key，若过期比例超过 25% 就继续抽取，否则停止。每次执行时间不超过 25ms，避免阻塞主线程。

两种策略配合使用，在 CPU 时间和内存占用之间取得平衡。

### 内存淘汰策略（maxmemory-policy）

当 Redis 内存达到 `maxmemory` 上限时，会触发数据淘汰，共有 6 种策略：

| 策略 | 说明 |
|------|------|
| `volatile-lru` | 从已设置过期时间的数据中，淘汰最近最少使用的 |
| `volatile-ttl` | 从已设置过期时间的数据中，淘汰即将过期的 |
| `volatile-random` | 从已设置过期时间的数据中，随机淘汰 |
| `allkeys-lru` | 从所有数据中，淘汰最近最少使用的（最常用） |
| `allkeys-random` | 从所有数据中随机淘汰 |
| `noeviction` | 不淘汰，返回写操作错误（默认） |

Redis 4.0 还新增了 `volatile-lfu` 和 `allkeys-lfu`，基于访问频率淘汰。

## 题目四：Redis 的五种基本数据类型是什么？它们的底层数据结构分别是什么？

### 参考答案

Redis 有 5 种基本数据类型，每种类型都有对应的底层数据结构：

| 数据类型 | 底层数据结构 |
|----------|------------|
| **String** | SDS（简单动态字符串），相比 C 字符串支持 O(1) 长度获取、二进制安全、自动扩容 |
| **List** | Redis 3.2 后统一使用 **quicklist**（压缩列表+双向链表的组合） |
| **Hash** | 压缩列表（元素少时）或**哈希表** |
| **Set** | 整数集合（元素全为整数且少时）或**哈希表** |
| **ZSet** | 压缩列表（元素少时）或**跳表 + 哈希表**（跳表实现有序，哈希表实现 O(1) 查找） |

**补充：Redis 3.0 vs 7.0 的变化**
- Redis 3.0 中 List/Hash/ZSet 在元素少时用压缩列表（ziplist），多时切换为对应结构
- Redis 7.0 中压缩列表（ziplist）已被**废弃**，改由 **listpack** 数据结构替代，节省内存并提高性能

## 题目五：什么是 Redis 集群脑裂问题？如何解决？

### 参考答案

### 什么是脑裂？

在 Redis 主从架构中，当主节点与从节点之间的**网络出现故障**，但主节点与客户端之间的网络正常时，会发生以下现象：

1. 客户端继续向主节点写入数据（主节点以为自己还在正常工作）
2. 哨兵（Sentinel）检测到主节点失联，判定为「客观下线」
3. 哨兵在从节点中选举出一个新主节点，**集群出现两个主节点**
4. 网络恢复后，旧主节点被降级为从节点，从新主节点执行全量同步，**导致之前客户端写入的数据全部丢失**

这就是**集群脑裂（Split-Brain）**导致的数据丢失问题。

### 解决方案

通过配置 Redis 的 **`min-slaves-to-write`** 和 **`min-slaves-max-lag`** 两个参数：

```conf
min-slaves-to-write 1    # 主节点必须至少有 1 个从节点
min-slaves-max-lag 10    # 主从复制延迟不能超过 10 秒
```

**原理**：当主节点发现从节点数量不足或复制延迟过长时，**直接拒绝客户端的写请求**，而不是继续接收数据。这样即使网络断开，旧主节点也无法接收新数据，避免脑裂后的数据丢失。
