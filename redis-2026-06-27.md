## 题目一：Redis 和 Memcached 有什么区别？为什么 Redis 作为缓存比 Memcached 更优秀？

### 答案

**共同点：**
- 都是基于内存的数据库，通常用作缓存
- 都支持过期策略
- 性能都非常高

**核心区别：**

| 特性 | Redis | Memcached |
|------|-------|-----------|
| 数据类型 | 支持 String、Hash、List、Set、ZSet 等丰富类型 | 只支持简单的 key-value |
| 持久化 | 支持 RDB、AOF、混合持久化 | 无持久化，数据全存内存 |
| 集群 | 原生支持 Cluster 模式 | 无原生集群，需客户端实现分片 |
| 高级特性 | 支持 Lua 脚本、事务、发布订阅 | 不支持 |
| 数据结构 | 底层实现丰富（SDS、跳表、压缩列表等） | 纯内存，无复杂结构 |

**为什么选 Redis 作为 MySQL 缓存：**

1. **高性能**：Redis 读写直接操作内存，QPS 可达 10W+，是 MySQL 的 10 倍
2. **高并发**：单台 Redis QPS 轻松破 10w，MySQL 难以破 1w
3. **丰富数据类型**：满足多样化业务场景（排行榜、计数器、消息队列等）
4. **数据持久化**：重启后可恢复数据
5. **集群支持**：天然支持主从、哨兵、Cluster 水平扩展

## 题目二：Redis 的过期键删除策略是什么？惰性删除和定期删除有什么区别？

### 答案

Redis 使用「**惰性删除 + 定期删除**」组合策略来清理过期键。

### 惰性删除（Passive/Lazy Expiration）

**原理：** 当访问某个 key 时，才检查它是否过期，过期则删除。

**优点：**
- CPU 友好，只在访问时检查，不占用额外资源

**缺点：**
- 内存不友好，如果过期键长期不被访问，就会一直占用内存

```python
# 伪代码逻辑
def get(key):
    if key in db and key.is_expired():
        delete(key)
    return db[key]
```

### 定期删除（Active Expiration）

**原理：** 每隔一段时间，随机从数据库中抽取一批 key 进行检查，删除其中过期的。

**执行流程：**
1. 从过期字典中随机抽取 20 个 key
2. 删除其中已过期的 key
3. 如果过期 key 数量 > 25%（即 > 5个），继续步骤 1
4. 单次执行时间上限 25ms，避免阻塞

**优点：**
- 减少过期键占用的内存
- 对 CPU 时间友好，可控

**缺点：**
- 难以精确控制删除频率

### 为什么不只用一种？

- 惰性删除对 CPU 友好但内存不友好
- 定期删除对内存友好但可能浪费 CPU
- 两者组合，取长补短，实现平衡

### 持久化时对过期键的处理

**RDB：**
- 生成阶段：过期键不会被写入 RDB 文件
- 加载阶段：主服务器会过滤过期键，从服务器会加载所有键（主从同步时会清空）

**AOF：**
-写入阶段：过期键未被删除时会保留，被删除时追加 DEL 命令
- 重写阶段：已过期键不会被保存到新的 AOF 文件

## 题目三：Redis 的持久化机制有哪些？RDB 和 AOF 各自的优缺点是什么？

### 答案

Redis 共有三种持久化方式：**RDB（快照）**、**AOF（日志）**、**混合持久化**。

### RDB 快照

**原理：** 记录某一时刻的内存数据，以二进制格式写入磁盘。

**触发方式：**
- `save` 命令：主线程执行，会阻塞
- `bgsave` 命令：子进程执行，不阻塞
- 配置自动触发：`save 900 1 / save 300 10 / save 60 10000`

**核心技术：Copy-On-Write（写时复制）**
- bgsave 时，子进程与父进程共享物理内存
- 父进程执行写操作时，复制数据页给子进程写入 RDB
- 保证快照期间，主进程仍可处理请求

**优点：** 恢复速度快（二进制直接加载）
**缺点：** 可能丢失较多数据（取决于快照频率）

### AOF 日志

**原理：** 每执行一条写命令，以追加方式写入日志文件。

**三种写回策略（appendfsync）：**

| 策略 | 含义 | 优点 | 缺点 |
|------|------|------|------|
| Always | 同步写回，每个命令执行完即写盘 | 零丢失 | 占用大量 I/O |
| Everysec | 每秒写回，命令执行后写入缓冲区 | 性能适中 | 最多丢1秒数据 |
| No | 操作系统控制写回时机 | 性能最好 | 丢失数据不可控 |

**AOF 重写机制：**
- 解决 AOF 文件过大的问题
- 后台子进程 bgrewriteaof 完成
- 读取当前数据库所有键值对，用一条命令记录到新文件
- 重写期间，主进程将新命令同时写入「AOF 缓冲区」和「AOF 重写缓冲区」

**优点：** 丢失数据少（可配置）
**缺点：** 恢复速度慢（需逐条执行命令）

### 混合持久化（Redis 4.0+）

**原理：** RDB + AOF，取两者优点。

**工作流程：**
- AOF 重写时，先以 RDB 格式写入全量数据
- 后续增量操作以 AOF 格式追加
- 文件前半部分是 RDB，后半部分是 AOF

**优点：** 启动快 + 数据丢失少
**缺点：** AOF 可读性差，兼容性低（4.0 前版本不支持）

### 如何选择？

- 数据重要且要求高可靠性：混合持久化
- 可接受少量数据丢失，追求恢复速度：RDB
- 要求最高可靠性：AOF + everysec 或 always

## 题目四：Redis 单线程模型是什么？为什么单线程的 Redis 还能这么快？

### 答案

### Redis 单线程指的是什么？

Redis 单线程指的是**命令执行过程**是单线程：
- 接收客户端请求 → 解析请求 → 读写数据 → 发送响应

但 Redis 启动时会创建**后台线程**处理耗时任务：
- `bio_close_file`：异步关闭文件
- `bio_aof_fsync`：异步 AOF 刷盘
- `bio_lazy_free`：异步释放内存

### Redis 6.0 之前（经典单线程模型）

```
客户端 → [事件循环] → 命令执行 → 响应
              ↓
         epoll 监听
        多个 Socket
```

**初始化流程：**
1. `epoll_create()` 创建 epoll 对象
2. `socket() + bind() + listen()` 监听端口
3. `epoll_ctl()` 注册连接事件
4. 主线程进入事件循环：
   - 处理发送队列
   - `epoll_wait()` 等待事件
   - 连接事件 → accept → 注册读事件
   - 读事件 → read → 解析 → 执行 → 写入发送队列
   - 写事件 → write 发送数据

### 为什么单线程还能这么快？

1. **内存操作**：Redis 数据全在内存，瓶颈是内存或带宽，不是 CPU
2. **高效数据结构**：SDS、跳表、压缩列表等，时间复杂度优秀
3. **无竞争**：单线程无锁、无上下文切换开销
4. **I/O 多路复用**：epoll 机制让一个线程处理成千上万个 Socket

### Redis 6.0 多线程做了什么？

主要优化**网络 I/O**，命令执行仍是单线程：
- 默认 I/O 线程只处理响应数据发送
- 开启读多线程：`io-threads-do-reads yes`
- 官方建议：4 核 CPU 设 2-3 线程，8 核设 6 线程
- 性能提升至少 **100%+**

### 常见误解

> "Redis 单线程无法利用多核 CPU"

实际上：
- 单节点 Redis 可用 `redis-server --port 6380` 启动多实例
- 生产环境推荐 **Redis Cluster** 水平扩展
- Redis 官方 FAQ 明确：CPU 不是 Redis 瓶颈

**来源：** https://xiaolincoding.com/redis/base/redis_interview.html

## 题目五：Redis 主从复制、哨兵模式和 Cluster 集群有什么区别？如何选型？

### 答案

### 主从复制（Replication）

**架构：** 一主多从，写操作在主节点，从节点只读。

**原理：**
1. 从节点发起 `PSYNC / SYNC` 同步请求
2. 主节点执行 bgsave 生成 RDB 快照
3. 主节点将 RDB 发送给从节点
4. 从节点加载 RDB 完成全量同步
5. 后续主节点将写命令异步发送给从节点（命令传播）

**特点：**
- 主从数据异步复制，不保证强一致性
- 主节点挂了，需要手动切换

**适用场景：** 数据量不大、要求读写分离的场景

### 哨兵模式（Sentinel）

**架构：** 在主从基础上 + 哨兵进程监控主从节点。

**核心功能：**
1. **监控**：哨兵周期性检查主从节点是否存活
2. **自动故障转移**：主节点宕机后，哨兵选举从节点升级为主节点
3. **通知**：告诉客户端新的主节点地址

**故障转移流程：**
1. 哨兵发现主节点主观下线
2. 询问其他哨兵，超过 quorum 认定客观下线
3. 选举一个从节点为新主节点
4. 更新配置，通知客户端

**适用场景：** 要求高可用、自动故障切换的业务

### Redis Cluster（切片集群）

**架构：** 数据分布在 16384 个哈希槽中，多个主节点分片存储。

**数据寻址：**
1. 对 key 使用 CRC16 计算哈希值
2. 对 16384 取模，得到哈希槽编号
3. 根据槽分配表找到对应节点

**特点：**
- 数据分布在多个节点，突破单机内存上限
- 每个节点可有从节点（主从复制）
- 无中心化设计，客户端直连任意节点
- 16384 个槽官方建议不超过 1000 节点

**适用场景：** 海量数据、高并发、需要水平扩展的业务

### 三者对比

| 特性 | 主从复制 | 哨兵模式 | Cluster |
|------|---------|---------|--------|
| 高可用 | ❌ 手动切换 | ✅ 自动切换 | ✅ 自动切换 |
| 横向扩展 | ❌ | ❌ | ✅ 分片 |
| 数据量 | 单机容量 | 单机容量 | 多机之和 |
| 写能力 | 单机 | 单机 | 多机（分片） |
| 复杂度 | 低 | 中 | 高 |

### 如何选型？

1. **单机 Redis 足够**：主从复制（读写分离）
2. **要求高可用**：哨兵模式
3. **数据量大、需扩展**：Redis Cluster
4. **极端场景**：Cluster + 哨兵（每个 Cluster 节点配从节点）

### 脑裂问题与解决方案

**脑裂现象：**
- 主节点网络故障，但与客户端连通
- 哨兵误判主节点宕机，选举新主
- 原主节点恢复后降级为从节点
- 新主从节点全量同步，导致数据丢失

**解决方案：**
```conf
min-slaves-to-write 1      # 主节点至少1个从节点
min-slaves-max-lag 10      # 复制延迟不超过10秒
```
- 配置后，主节点检测到从节点不达标，禁止写入
- 即使原主节点假故障，也无法写入，保证数据安全
