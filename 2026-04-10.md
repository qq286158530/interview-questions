**题目1：Object 类有哪些主要方法？clone() 方法有什么限制？finalize() 为什么不推荐使用？**

【答案】

Object 类是 Java 中所有类的根父类，它包含以下主要方法：

**主要方法：**
1. **getClass()** - 返回运行时类的 Class 对象
2. **hashCode()** - 返回对象的哈希码值
3. **equals(Object obj)** - 判断两个对象是否相等
4. **toString()** - 返回对象的字符串表示
5. **clone()** - 创建并返回对象的副本
6. **finalize()** - 在对象被垃圾回收前调用（已废弃）
7. **notify() / notifyAll() / wait()** - 用于线程同步

**clone() 的限制：**
- clone() 是浅拷贝，只拷贝基本类型和引用地址
- 如果类包含可变对象引用，克隆后两个对象会共享同一引用
- 需要实现 Cloneable 接口，否则抛出 CloneNotSupportedException
- 构造函数不会被调用
- 深拷贝需要手动实现

**为什么不推荐使用 finalize()：**
1. 不确定性：GC 何时调用 finalize() 不确定
2. 性能开销：增加 GC 负担
3. 安全性问题：可能被恶意利用
4. 已被 Java 9 正式废弃
5. 更好的替代方案：使用 try-with-resources 或 PhantomReference

**题目2：什么是 happens-before 原则？JVM 如何保证可见性和有序性？**

【答案】

**happens-before 原则定义：**
JVM 规定了两项操作之间的偏序关系（Partial Order）。如果操作 A happens-before 操作 B，那么 A 的执行结果对 B 可见，且 A 的执行顺序在 B 之前。这是 JVM 内存模型（JMM）的核心规则。

**8大happens-before 规则：**
1. **程序顺序规则**：同一个线程中，前面的操作 happens-before 后面的操作
2. **监视器锁规则**：解锁操作 happens-before 对同一个锁的加锁操作
3. **volatile 变量规则**：对 volatile 变量的写操作 happens-before 对该变量的读操作
4. **线程启动规则**：Thread.start() happens-before 被启动线程中的任何操作
5. **线程终止规则**：线程中的所有操作 happens-before 其他线程检测到该线程终止
6. **传递性**：如果 A happens-before B，B happens-before C，则 A happens-before C
7. **中断规则**：对 interrupt() 的调用 happens-before 被中断线程检测到中断
8. **终结器规则**：对象的构造函数 happens-before finalize()

**JVM 如何保证可见性：**
- **MESI 缓存一致性协议**：CPU 核心修改数据后，其他核心能感知到变化
- **内存屏障（Memory Barrier）**：防止指令重排序
- **synchronized**：解锁时强制刷新工作内存到主存
- **volatile**：写操作后插入 Store 屏障刷新到主存，读操作前插入 Load 屏障失效化其他核心缓存

**JVM 如何保证有序性：**
- 通过内存屏障禁止特定类型的指令重排序
- happens-before 规则本身也是有序性保证

**题目3：Java 中有哪几种引用类型？WeakReference 和 SoftReference 有什么区别？各适合什么场景？**

【答案】

**Java 的四种引用类型：**

| 引用类型 | 回收时机 | 适用场景 |
|---------|---------|---------|
| **强引用（Strong Reference）** | 永不回收（GC ルート可达） | 常规对象引用 |
| **软引用（Soft Reference）** | 内存不足时回收 | 缓存 |
| **弱引用（Weak Reference）** | 下次 GC 时回收 | 规范映射（Map缓存） |
| **虚引用（Phantom Reference）** | 随时可能被回收 | 对象生命周期追踪 |

**WeakReference vs SoftReference 的区别：**

| 特性 | WeakReference | SoftReference |
|------|---------------|---------------|
| **回收时机** | 下次 GC 时，无视内存状态 | 内存不足时才回收 |
| **存活周期** | 更短 | 更长，只要内存充足就保留 |
| **典型应用** | ThreadLocal、WeakHashMap | 内存敏感的缓存 |

**典型使用场景：**

**WeakReference 场景：**
```java
// 防止内存泄漏：ThreadLocal 的 key 使用 WeakReference
ThreadLocal<Map<K, V>> threadLocal = new ThreadLocal<>();

// WeakHashMap：缓存不会阻止 GC
Map<Key, Value> cache = new WeakHashMap<>();
```

**SoftReference 场景：**
```java
// 图片缓存 - 内存不足时自动清理
SoftReference<Bitmap> bitmapCache = new SoftReference<>(loadBitmap());

// 大对象缓存
SoftReference<Object> largeObjectCache = new SoftReference<>(heavyObject);
```

**注意事项：**
- 引用队列（ReferenceQueue）可以监听对象被回收的时机
- 软引用和弱引用的 get() 方法可能返回 null
- 虚引用必须与 ReferenceQueue 配合使用，主要用于堆外内存管理

**题目4：@Resource、@Autowired 和 @Inject 注解有什么区别？各自注入方式是什么？**

【答案】

**三个注解的对比：**

| 特性 | @Autowired | @Resource | @Inject |
|-----|------------|-----------|---------|
| **来源** | Spring 独有 | Java EE (JSR-250) | Java CDI (JSR-330) |
| **包名** | org.springframework.beans.factory.annotation | javax.annotation | javax.inject |
| **默认注入方式** | byType + byName | byName + byType | byType |
| **required 属性** | 支持 | 不支持（可以用在 JDK 9+） | 支持 |
| **位置限制** | 构造函数、setter、字段 | 字段、setter | 字段、setter、构造函数 |

**@Autowired 注入流程：**
1. 首先按 **byType** 查找类型匹配的 Bean
2. 如果有多个同类型 Bean，则按 **byName**（属性名或参数名）查找
3. 可以配合 **@Qualifier** 显式指定 Bean 名称
4. 可以配合 **@Primary** 指定首选 Bean
5. 找不到时会抛出异常（除非 required=false）

**@Resource 注入流程：**
1. 首先按 **byName** 查找（默认使用属性名）
2. 如果找不到，再按 **byType** 查找
3. 可以通过 name 属性显式指定 Bean 名称
4. 找不到时不会报错（行为取决于具体实现）

**@Inject 注入流程：**
1. 按 **byType** 查找类型匹配的 Bean
2. 需要配合 **@Named** 实现 byName（类似 @Qualifier）
3. 需要导入 `javax.inject` 依赖

**推荐使用场景：**
- Spring 生态项目：@Autowired + @Qualifier 组合最灵活
- 跨框架兼容：@Resource 是标准注解，更通用
- 需要强类型检查：@Inject 需要显式导入依赖

**最佳实践：**
```java
// 构造器注入（推荐）- 保证依赖不可变
@Service
public class UserService {
    private final UserRepository userRepository;
    
    @Autowired  // 构造器注入可以省略（Spring 4.3+）
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}
```

**题目5：Spring Framework 6.x 有哪些重要新特性？与 Spring 5.x 相比有什么重大变化？**

【答案】

**Spring Framework 6.x 重大变化：**

**1. 最低 Java 版本提升**
- Spring 6.x 要求 **Java 17+**（推荐 Java 21）
- Spring 5.x 支持 Java 8-17
- 这意味着企业必须升级 JDK 才能使用 Spring 6.x

**2. 迁移到 Jakarta EE 9+**
- 包名从 `javax.*` 迁移到 `jakarta.*`
- 需要升级依赖库的版本（如 Spring Boot 3.x + Jakarta EE 9+）
- 影响：Servlet、Validation、XML Parsing 等规范

**3. AOT 编译支持（GraalVM Native Image）**
- 支持通过 Spring Native 编译为原生镜像
- 启动时间从秒级降到毫秒级
- 内存占用大幅降低
- 提前编译牺牲了动态代理的灵活性

**4. 响应式编程增强**
- WebFlux 性能优化
- 更好的背压（Backpressure）处理
- R2DBC 支持增强

**5. HTTP Interface**
- 可以用 `@HttpExchange` 声明式定义 HTTP 客户端
- 类似于 Feign 但内置支持响应式

**6. 观察性增强（Observability）**
- Micrometer 与 Spring 的集成更好
- 支持统一 Tracing API

**7. 配置文件变化**
- `spring.config.import` 支持更灵活
- 环境预处理增强

**升级注意事项：**
```
javax → jakarta 迁移清单：
1. Spring Boot 2.7.x → 3.0.x
2. Hibernate ORM 5.x → 6.x
3. Spring Security 5.x → 6.x
4. 所有 javax.* 包名替换为 jakarta.*
```

**题目6：Nacos 的配置管理机制是什么？如何实现多环境（dev/test/prod）配置隔离？**

【答案】

**Nacos 配置管理核心概念：**

**1. 配置模型（Configuration Model）**
- **Namespace（命名空间）**：最顶层隔离，典型用于环境隔离
- **Group（分组）**：业务维度分组，默认 DEFAULT_GROUP
- **Data ID（配置 ID）**：具体配置文件标识

**2. 配置获取流程：**
```
应用启动 → 读取 bootstrap.yml → 
Nacos Client 拉取配置 → 解析配置 → 
监听配置变更（Long Polling / UDP）→ 实时更新
```

**多环境配置隔离方案：**

**方案一：Namespace 隔离（推荐）**
```yaml
# bootstrap.yml
spring:
  cloud:
    nacos:
      server-addr: nacos-server:8848
      namespace: dev-env-id  # dev/test/prod 不同的 namespace ID
      group: DEFAULT_GROUP
      username: nacos
      password: nacos
```

**方案二：Data ID 规则隔离**
```
# 命名规范：{application}-{profile}.{file-extension}
user-service-dev.yaml
user-service-test.yaml
user-service-prod.yaml
```

**方案三：Profile + Label 组合**
```yaml
spring:
  cloud:
    nacos:
      config:
        # 支持拉取多个配置
        shared-configs:
          - data-id: common.yaml
            group: COMMON_GROUP
            refresh: true
        # 扩展配置（profile 相关）
        extension-configs:
          - data-id: ${spring.application.name}-${spring.profiles.active}.yaml
            group: ${spring.cloud.nacos.config.group}
            refresh: true
```

**配置优先级：**
```
properties: 
  1. 应用自身配置文件 > shared-configs > extension-configs
  2. 相同 data-id，后声明的覆盖先声明的
  3. namespace > group > data-id
```

**配置热更新机制：**
```java
@RefreshScope  // 标注后，配置变更会自动刷新
@Configuration
public class AppConfig {
    @Value("${custom.timeout:30}")
    private int timeout;
}
```

**Nacos 配置变更监听原理：**
1. **长轮询（Long Polling）**：默认 30 秒超时
2. **推送（UDP）**：配置变更时服务端主动推送
3. 客户端收到通知后重新拉取配置并刷新 Bean

**题目7：Sentinel 的限流算法有哪些？滑动窗口算法和令牌桶算法的区别是什么？各适合什么场景？**

【答案】

**Sentinel 支持的限流算法：**

**1. 直接拒绝（Default）**
- 请求量超过阈值时直接拒绝
- 优点：简单、无需额外计算
- 缺点：不平滑，可能造成流量突刺

**2. 滑动窗口算法（Sliding Window）**
- 原理：将时间窗口划分为多个小窗口，统计所有小窗口的请求数
- Sentinel 默认使用滑动窗口记录请求统计
- 优点：比固定窗口更精确，避免边界流量突刺
- 实现：`StatisticSlot` 使用滑动窗口计数器

**3. 漏桶算法（Leaky Bucket）**
- 原理：以固定速率处理请求，超出的请求在桶中排队
- 优点：流量绝对平滑，以恒定速率处理
- 缺点：无法应对突发流量（排队等待）

**4. 令牌桶算法（Token Bucket）**
- 原理：桶中存放令牌，请求必须获取令牌才能通过
- 优点：允许一定程度的突发流量（桶满时）
- 典型实现：Guava RateLimiter、Sentinel 的 WarmUpController

**滑动窗口 vs 令牌桶：**

| 特性 | 滑动窗口 | 令牌桶 |
|------|---------|-------|
| **流量平滑度** | 较平滑（细分窗口） | 更平滑（匀速发放） |
| **突发流量** | 允许（在窗口内） | 允许（桶满时） |
| **实现复杂度** | 中等 | 简单 |
| **资源控制** | 精确 QPS 控制 | 精确 QPS + 突发量控制 |
| **典型场景** | Sentinel 默认 | 限流插件、API 限流 |

**Sentinel 核心概念：**
- **Resource**：需要保护的资源
- **Entry**：每次资源访问的入口
- **Slot**：插槽链（统计、限流、降级、系统保护）
- **Context**：上下文，保存调用链信息

**Sentinel 系统自适应限流：**
- 根据系统 Load、CPU 使用率、平均 RT、入口 QPS 自动调整阈值
- 保护系统不被拖垮

**题目8：Kafka 的分区副本（Replica）机制是什么？如何实现故障转移（Failover）？ISR 是如何管理的？**

【答案】

**Kafka 分区副本核心概念：**

**1. 副本角色**
- **Leader Replica**：处理所有读写请求
- **Follower Replica**：被动复制 Leader 数据
- **ISR（In-Sync Replicas）**：与 Leader 保持同步的副本集合

**2. 副本同步原理**
```
Producer → Leader Replica → 写入本地磁盘 → 
            ↓
       Follower 定时拉取（Fetch）→ 写入本地磁盘 → 发送 ACK
```

**3. ISR 管理机制**
```
ISR 动态变化条件：
- Follower 副本的 LEO（Log End Offset）追上 Leader 的 HW（High Watermark）
- 判定追上：Lag = 0 且 在 replica.lag.time.max.ms 时间内无落后

触发条件导致 Follower 踢出 ISR：
- 网络抖动、GC 停顿、Consumer 消费慢
- Follower 故障（进程崩溃）
```

**4. 故障转移流程（Leader Election）**
```
1. Controller 感知到 Broker 宕机（ZooKeeper Watch）
2. Controller 从 ISR 中选择新的 Leader（选举策略：AR+1偏移量最小的）
3. 更新 LeaderAndIsr 消息
4. 所有 Broker 收到更新后切换
5. Producer 收到 NotLeaderException 重试
```

**5. 数据一致性和可靠性保证**
```
HW（High Watermark）：消费者能看到的最大偏移量
LEO（Log End Offset）：日志末端偏移量

数据一致性保证：
- Consumer 只能消费到 HW 之前的数据
- Leader 切换时，Follower 会截断（Truncate）多余消息
- 保证了 no data loss
```

**6. 副本分配策略**
- 副本因子（replication.factor）：建议设置为 3
- 分区分配要跨机架（broker.rack）防止机架故障
- Leader 分布要均衡（--rebalance-algorithms）

**7. Unclean Leader Election**
- 开启 `unclean.leader.election.enable=true` 允许非 ISR 副本成为 Leader
- 可能造成数据丢失，但保证了可用性
- 生产环境建议关闭

**题目9：如何保证消息队列消费的幂等性？常见的幂等性实现方案有哪些？**

【答案】

**消息消费幂等性问题的本质：**
消费者处理消息可能存在重复执行的情况（如网络重试、Consumer 宕机恢复），导致业务数据重复。

**常见幂等性实现方案：**

**方案一：数据库唯一索引（最常用）**
```sql
-- 订单消息处理
INSERT INTO order_log (msg_id, order_id, status) 
VALUES ('msg-123', 'order-456', 'PAID')
ON DUPLICATE KEY UPDATE status = status;  -- 重复插入忽略
```
- 适用场景：业务数据表有天然唯一约束
- 优点：实现简单、可靠
- 缺点：依赖数据库

**方案二：Redis 分布式锁**
```java
// 扣减库存
String lockKey = "stock:lock:" + msgId;
if (redis.set(lockKey, "1", "NX", "PX", 3000)) {
    try {
        doDeductStock(orderId);
    } finally {
        redis.del(lockKey);
    }
} else {
    // 已处理，跳过
}
```
- 优点：性能高、支持集群
- 缺点：依赖 Redis 可用性

**方案三：Redis + MySQL 组合**
```java
// 先查 Redis 是否处理过
if (redis.exists("processed:msg:" + msgId)) {
    return; // 已处理
}
// 业务处理
doBusiness();
// 标记已处理
redis.setex("processed:msg:" + msgId, 86400, "1");
```
- 解决 Redis 不可用时的数据丢失问题

**方案四：状态机幂等**
```java
// 订单状态流转：CREATED → PAID → SHIPPED → COMPLETED
// 只允许正向流转，逆向操作需校验
if (order.getStatus() == OrderStatus.PAID) {
    // 执行发货
}
```
- 适用场景：状态有明确流转顺序

**方案五：Token 机制**
```
1. Producer 发送消息前获取 Token
2. 消息包含 Token
3. Consumer 处理前检查 Token 是否已消费
4. 使用 Redis SetNx 或数据库唯一键保证
```

**最佳实践建议：**
```
幂等性设计原则：
1. 业务表设计时考虑幂等字段（如 msg_id）
2. 乐观锁（version 字段）防止并发更新
3. 消费前先检查，业务处理后再更新状态
4. 使用唯一键约束 + 异常捕获
5. 补偿机制兜底（定时任务核对）
```

**Kafka 消费幂等特别处理：**
```java
// 开启 Kafka 幂等性
props.put("enable.idempotence", true);
// 配合 transactional.id 实现 Exactly Once
```

**题目10：RocketMQ 的事务消息是如何实现的？半消息（Half Message）机制的作用是什么？**

【答案】

**RocketMQ 事务消息原理：**

**核心流程（两阶段提交）：**

```
┌─────────────────────────────────────────────────────────┐
│                    Producer 发送事务消息                  │
└─────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────┐
│            第一阶段：发送 Half Message（预处理消息）        │
│  - 消息状态设置为 Prepared（对 Consumer 不可见）           │
│  - 写入 RocketMQ 内部队列                                │
└─────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────┐
│            本地事务执行                                   │
│  - 执行订单创建、库存扣减等业务                            │
│  - 事务结果：COMMIT / ROLLBACK                           │
└─────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────┐
│            第二阶段：事务状态回查                         │
│  - 如果提交：RocketMQ 标记消息为可投递                    │
│  - 如果回滚：RocketMQ 删除消息                            │
│  - 如果超时：Broker 主动回查 Producer                    │
└─────────────────────────────────────────────────────────┘
```

**Half Message（半消息）机制：**

**作用：**
1. **暂时隔离**：消息对 Consumer 不可见，防止提前消费
2. **事务控制**：只有本地事务成功后才投递
3. **可回滚**：本地事务失败时可以删除，不会投递

**Half Message 本质：**
- 消息的 `transactionId` 属性被设置
- 消费监听器（MessageListener）不返回 CONSUME_SUCCESS
- RocketMQ 内部维护一个 `transaction_half` 主题

**事务状态回查机制：**
```java
// Producer 需要实现 TransactionListener
public class TransactionListenerImpl implements TransactionListener {
    
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        // 执行本地事务
        try {
            orderService.create(msg);
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }
    
    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 回查本地事务状态
        // 幂等性：检查订单是否已创建
        if (orderService.exists(msg.getTransactionId())) {
            return LocalTransactionState.COMMIT_MESSAGE;
        }
        return LocalTransactionState.UNKNOW;  // 继续回查
    }
}
```

**Broker 端回查逻辑：**
```
回查触发条件：
1. Producer 在 transactionTimeout（默认 6s）内未提交/回滚
2. 定时任务扫描 prepared 消息（每 60s）
3. 最多回查 15 次（可配置）

回查策略：
- UNKNOW → 延迟重新回查（指数退避）
- 达到最大次数 → 自动回滚
```

**与普通消息的区别：**
| 特性 | 普通消息 | 事务消息 |
|------|---------|---------|
| 发送方式 | one-way / sync | 两阶段提交 |
| 消费时机 | 发送后立即可消费 | 事务提交后才可消费 |
| 本地事务 | 无 | 必须实现 |
| 性能开销 | 低 | 较高（多一次 RPC） |
| 使用场景 | 普通异步消息 | 分布式事务 |

**应用场景：**
- 订单系统：下单 + 扣库存 + 发消息，保证一致性
- 转账系统：扣款 + 发消息，保证原子性
- 积分系统：消费 + 积分发放，保证不丢失

**Function Calling 是什么？**

Function Calling（大模型函数调用）是 LLM（大语言模型）的一种能力，允许模型在生成响应时主动调用预定义的外部函数/工具，而不是仅输出文本。

```

                                        ↓
                               2. LLM 分析请求 + 工具列表
                                        ↓
                               3. 返回函数调用请求
                                        ↓
┌─────────────┐     4. 函数执行结果   ┌─────────────┐
│   Tools     │ ◀─────────────────  │  AI Agent   │
└─────────────┘                     └─────────────┘
                                        ↓
                               5. LLM 整合结果生成回复
                                        ↓
┌─────────────┐     6. 最终回复       ┌─────────────┐
│   User      │ ◀─────────────────  │  AI Agent   │
└─────────────┘                     └─────────────┘
```

**题目12：什么是 Agentic RAG？与传统 RAG 相比，Agentic RAG 有哪些优势？它如何解决复杂多跳问题？**

【答案】

**传统 RAG（Retrieval-Augmented Generation）的问题：**

1. **单次检索**：只做一次检索，无法处理复杂查询
2. **缺乏规划**：不能分解问题为多个步骤
3. **无法反思**：不检查检索结果是否正确
4. **多跳困难**：无法回答需要多个事实的问题

**Agentic RAG 是什么？**

Agentic RAG = RAG + Agent 能力（规划、反思、工具使用）

```
传统 RAG 流程：
Query → Retriever → Context → LLM → Answer

Agentic RAG 流程：
Query → Agent（规划）
            ↓
       ┌── 检索器1（查相关实体）──→ 子答案1
       ↓
       ┌── 检索器2（查关联关系）──→ 子答案2
       ↓
       ┌── 知识图谱检索 ─────────→ 子答案3
       ↓
       LLM 整合所有子答案 → 最终答案
```

**Agentic RAG 的核心能力：**

**1. 查询分解（Query Decomposition）**
```
用户问题：张三在哪个公司工作？该公司有多少员工？

Agent 分解为：
- 子问题1：张三在哪家公司工作？→ 答案：ABC公司
- 子问题2：ABC公司有多少员工？→ 答案：5000人

最终整合：张三在ABC公司工作，该公司有5000名员工。
```

**2. 迭代检索（Iterative Retrieval）**
```
Agent 发现答案不完整：
- 初始检索：找到相关文章
- 第一次分析：发现缺少关键信息
- 第二次检索：针对缺失信息检索
- 最终回答：整合所有信息
```

**3. 路由（Routing）**
```python
# Agent 根据问题类型选择不同策略
if question.type == "simple":
    strategy = "direct_retrieval"
elif question.type == "multi-hop":
    strategy = "decomposition"  # 多跳问题：分解
elif question.type == "comparison":
    strategy = "parallel_retrieval"  # 比较问题：并行检索
```

**4. 反思与校验（Self-Reflection）**
```
Agent 校验流程：
1. 检索到候选答案
2. 问自己：答案是否完整？
3. 发现不完整 → 继续检索
4. 问自己：答案是否矛盾？
5. 整合不矛盾的答案
```

**主流框架实现（LangGraph）：**

```python
from langgraph.graph import StateGraph
from typing import TypedDict

class AgentState(TypedDict):
    question: str
    plan: List[str]
    retrieved_contexts: List[str]
    answer: str

# 构建 Agentic RAG 图
graph = StateGraph(AgentState)

graph.add_node("planner", plan_node)      # 规划节点
graph.add_node("retriever", retrieve_node)  # 检索节点
graph.add_node("reasoner", reason_node)  # 推理节点
graph.add_node("grader", grade_node)     # 评估节点

# 边：根据评估结果决定下一步
graph.add_edge("planner", "retriever")
graph.add_conditional_edges(
    "grader",
    should_continue,  # 决定是否继续检索
    ["retriever", "reasoner"]
)

graph.set_entry_point("planner")
```

**解决多跳问题的例子：**

```
问题：特斯拉和SpaceX的CEO是同一个人吗？

传统 RAG：无法回答（需要关联两个实体的关系）

Agentic RAG：
1. 检索"特斯拉 CEO" → 马斯克
2. 检索"SpaceX CEO" → 马斯克
3. 比较判断 → 是同一个人
4. 生成答案 → 是的，都是埃隆·马斯克
```

**Agentic RAG vs 传统 RAG 对比：**

| 维度 | 传统 RAG | Agentic RAG |
|-----|---------|-----------|
| 检索次数 | 1次 | 多次/迭代 |
| 问题分解 | 不支持 | 支持 |
| 自我校验 | 不支持 | 支持 |
| 多跳问题 | 效果差 | 效果好 |
| 响应质量 | 一般 | 更高 |
| 延迟 | 低 | 较高（多次调用） |
| 成本 | 低 | 较高 |

**题目13：什么是一致性哈希（Consistent Hashing）？它是如何解决分布式系统中节点增减时的数据迁移问题的？虚拟节点有什么作用？**

【答案】

**为什么需要一致性哈希？**

普通哈希（取模）的痛点：
```python
# 普通哈希
node_index = hash(key) % node_count
# 当节点数从 3 变为 4 时，几乎所有 key 都要重新映射
# 节点故障：3台 → 2台，60%的数据都要迁移
```

**一致性哈希原理：**

**1. 环形哈希空间**
```
将所有节点和 key 映射到一个 0 ~ 2^32 的哈希环上：

        0
         ↑
    4000000000
         ↑
         ↓
   节点C (3000000000)
         ↑
         ↓        节点A (1000000000)
    节点D (2000000000) → key1 (1500000000) 会路由到节点A
         ↑                   
         ↓  key2 (500000000) 会路由到节点D
    节点B (500000000)
```

**2. 数据路由规则**
```
key 的路由：沿环顺时针方向，找到第一个节点
- key1 的哈希值：1500000000 → 顺时针找到 A
- key2 的哈希值：500000000 → 顺时针找到 B
```

**3. 节点增减时的数据迁移**
```
原状态：节点 A、B、C 分布在环上
增加节点 D：插入到 C 和 A 之间

影响范围：只有落在新增节点 D 所跨区间的 key 需要迁移
迁移量 = 1/N（N 为节点数），而不是全部
```

**虚拟节点（Virtual Nodes）的作用：**

**问题：节点硬件配置不同**
- 高配机器（16核64G）vs 低配机器（4核8G）
- 如果只用 1 个物理节点，会造成负载不均

**解决方案：虚拟节点**
```
物理节点 A → 虚拟节点 A1、A2、A3、A4（共4个槽位）
物理节点 B → 虚拟节点 B1、B2、B3（共3个槽位）

映射到环上时：
A1: hash("A1")
A2: hash("A2")
A3: hash("A3")
A4: hash("A4")
B1: hash("B1")
B2: hash("B2")
B3: hash("B3")

客户端路由时：
- 先找到虚拟节点
- 再从虚拟节点映射到物理节点
```

**虚拟节点的优势：**
| 优势 | 说明 |
|-----|-----|
| 负载均衡 | 虚拟节点越多，分布越均匀 |
| 节点异构 | 根据硬件配置分配不同数量虚拟节点 |
| 故障隔离 | 单个虚拟节点故障不影响其他 |

**一致性哈希的典型应用：**

**1. 分布式缓存（Redis Cluster）**
```
Redis Cluster 使用 hash slot（16384 个槽）
客户端：CRC16(key) % 16384 = slot
槽位 → 节点映射
```

**2. 负载均衡器（Nginx）**
```
upstream 配置中使用 consistent_hash $request_uri;
```

**3. 分布式数据库（Cassandra、DynamoDB）**
```
DynamoDB 使用改进版一致性哈希 + 虚拟节点
```

**代码实现示例：**
```python
import hashlib

class ConsistentHash:
    def __init__(self, nodes=None, virtual_nodes=150):
        self.virtual_nodes = virtual_nodes
        self.ring = {}
        self.sorted_keys = []
        
        if nodes:
            for node in nodes:
                self.add_node(node)
    
    def _hash(self, key):
        return int(hashlib.md5(str(key).encode()).hexdigest(), 16)
    
    def add_node(self, node):
        for i in range(self.virtual_nodes):
            vnode = f"{node}#{i}"
            hash_key = self._hash(vnode)
            self.ring[hash_key] = node
            self.sorted_keys.append(hash_key)
        self.sorted_keys.sort()
    
    def get_node(self, key):
        hash_key = self._hash(key)
        # 二分查找第一个 >= hash_key 的位置
        for h in self.sorted_keys:
            if h >= hash_key:
                return self.ring[h]
        return self.ring[self.sorted_keys[0]]  # 闭环
```
