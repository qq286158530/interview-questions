## 题目一：Flink 的架构是怎样的？为什么说 Flink 是「真正的流计算」？

### 参考答案

Flink 采用**分层架构**，自底向上包括：

1. **物理部署层**：支持 Standalone、YARN、Kubernetes 等多种部署模式。
2. **执行引擎层（Runtime）**：用 Java/NIO 实现，负责接收 DAG（Directed Acyclic Graph），调度 Task 到 TaskManager 上执行。
3. **API 层**：
   - **DataStream API**：用于无界流处理（ unbounded stream）
   - **DataSet API**：用于有界批处理（bounded dataset，Flink 1.12 后统一为 DataStream）
4. **上层库**：CEP（复杂事件处理）、Table API / SQL、Gelly（图计算）等。

**「真正的流计算」**的原因：

| 特性 | 批处理（MapReduce/Spark） | Flink |
|------|--------------------------|-------|
| 数据模型 | micro-batch（微批次） | 真正的记录级别的流（record-at-a-time） |
| 延迟 | 较高（等待批次凑满） | 低（来一条处理一条） |
| 语义 | 至少一次/恰好一次（通过微批） | 精确一次（end-to-end exactly-once） |
| 窗口 | 依托微批，窗口边界对齐批次 | 任意对齐的滚动/滑动窗口，水位线驱动 |

Flink 将**无界流**作为一等公民，通过 **Chandy-Morgan** 分布式快照算法实现端到端的精确一次语义，而不是靠微批模拟流计算。

## 题目二：Watermark（水位线）的作用是什么？如何处理乱序数据？

### 参考答案

### Watermark 的作用

Watermark 是 Flink 用于处理**事件时间（Event Time）**乱序问题的核心机制。它是一条时间戳声明：

> **"当 Watermark 到达时间 T 时，所有时间戳 ≤ T 的数据都已经到达"**

Flink 基于 Watermark 决定何时**触发窗口计算**和**清理过期状态**，从而在乱序场景下仍能保证结果的正确性。

### 核心概念

- **Event Time（事件时间）**：数据本身携带的时间戳（业务时间）。
- **Processing Time（处理时间）**：数据到达算子的系统时钟时间。
- **Watermark**：携带时间戳 T，告诉算子「可以安全地认为不会再有时间戳 ≤ T 的数据了」。

### 乱序处理的两种策略

**1. Watermark + 窗口策略**

```
Watermark = max(数据时间戳) - 最大乱序等待时间
```

窗口只有在其 Watermark ≥ 窗口结束时间时才触发。例如 `TumblingEventTimeWindow` 大小为 5 分钟：

- 窗口 [10:00, 10:05) 的 Watermark 必须 ≥ 10:05 才触发
- 时间戳 10:03 但晚到的数据，只要 Watermark 未超过 10:05 仍可被处理

**2. 迟到数据处理**

```java
// 允许迟到 30 秒，超时后数据进入旁路输出
widgetWindow
    .allowedLateness(Time.seconds(30))
    .sideOutputLateData(lateTag);
```

- **allowedLateness**：窗口触发后仍保留状态 30 秒，期间的迟到数据仍可触发窗口重新计算。
- **sideOutputLateData**：超过 allowedLateness 的数据被标记为迟到，输出到旁路。

### 注意事项

- Watermark 的生成方式（周期性或标点式）需要根据数据源特点配置。
- 多流 Join 时，以**最小的 Watermark** 推进（因为需要所有流的数据）。
- 如果数据源本身有序，可以不设置 Watermark，但生产环境几乎都有乱序。

## 题目三：Flink Checkpoint 与 Savepoint 的区别是什么？Flink 如何保证精确一次（Exactly-Once）语义？

### 参考答案

### Checkpoint vs Savepoint

| 对比项 | Checkpoint | Savepoint |
|--------|-----------|-----------|
| 触发方式 | Flink 自动周期性触发（JobManager 协调） | 手动触发（用户命令） |
| 存储位置 | 由 `state.backend` 配置决定 | 由 `-s` 参数指定路径 |
| 用途 | 故障恢复，保证持续运行 | 计划性备份、升级、扩缩容 |
| 状态格式 | 特定状态后端格式（rocksdb/changelog） | 统一的外部化格式 |
| 保留策略 | 由 Flink 自动管理（旧 checkpoint 自动清理） | 手动管理，不自动删除 |
| 依赖 | 依赖 Checkpoint 协调器 | 独立于运行中的 Job |

### Flink 的精确一次实现

Flink 通过 **Chandy-Morgan**（分布式快照）算法实现端到端精确一次语义：

**核心原理（三阶段提交）：**

1. **Checkpoint Coordinator** 向所有 Source 算子发送 `CheckpointTrigger` 消息。
2. **Source 算子**广播 **Barrier（屏障）**，同时**快照自身状态**（偏移量）并写入持久化存储，然后向下游发送 Barrier。
3. **中间算子**收到所有上游的 Barrier 后（对齐机制），做快照，然后继续向下游传递。
4. **Sink 算子**完成快照后向 Coordinator 确认，所有算子快照完成则 Checkpoint 成功。

**对齐机制（Barrier Alignment）：**
当有多个上游输入时，算子需要等**所有上游的 Barrier** 都到达后才开始快照，在此期间会缓存先到的数据。这保证了快照的一致性。

**端到端精确一次的必要条件：**

- Source 支持回退（如 Kafka Consumer 可以回溯 offset）
- Sink 支持幂等写入或两阶段提交（如 Kafka Producer 的事务、RocksDB 的 ingest-external-db）

**Flink 1.11 后的改进（Unaligned Checkpoint）：**
对于高反压场景，Unaligned Checkpoint 将未处理的 Stream 数据也纳入快照，减少对齐等待时间，提升大规模集群的 Checkpoint 稳定性。

## 题目四：Flink 窗口函数有哪些类型？Session Window 是如何工作的？

### 参考答案

### 窗口函数分类

**按数据划分方式：**

| 类型 | 说明 | 示例 |
|------|------|------|
| **Tumbling Window（滚动窗口）** | 大小固定，不重叠 | 每 5 分钟统计一次 |
| **Sliding Window（滑动窗口）** | 大小固定，可重叠 | 每 5 分钟窗口，每 1 分钟滑动一次 |
| **Session Window（会话窗口）** | 根据活动间隙动态划分 | 活跃间隔 < 10 分钟视为同一会话 |
| **Global Window（全局窗口）** | 所有数据归入同一窗口 | 自定义触发逻辑 |

**按时间语义：**

- **EventTimeWindow**：基于事件时间（需要 Watermark）
- **ProcessingTimeWindow**：基于处理时间（简单但结果不确定）

**按函数类型：**

- **增量聚合函数**：`reduceFunction`、`aggregateFunction`、`sum/max/min` — 数据来一条处理一条，存储部分结果，内存高效。
- **全量聚合函数**：`applyFunction`、`processFunction` — 先缓存窗口内所有数据，窗口触发时统一计算，适合需要访问全部数据的场景。

### Session Window 的工作原理

Session Window 没有固定大小，由**会话间隙（Gap）**动态决定窗口边界。

**核心参数：**
- **Gap**：两条数据之间的最大时间间隔，超过则开启新窗口。
- **Gap Duration**：可以是固定值或从数据中动态提取（如从用户行为数据中提取会话超时时间）。

```java
DataStream<String> stream = ...;

// 固定 Gap：会话间隔 5 分钟内视为同一会话
stream.keyBy(x -> x.getUserId())
      .window(EventTimeSessionWindow.withGap(Time.minutes(5)))
      .sum("price");

// 动态 Gap：从数据中提取
stream.keyBy(x -> x.getUserId())
      .window(EventTimeSessionWindow.withDynamicGap(
          new SessionWindowTimeGapExtractor<Tuple2<String, Long>>() {
              @Override
              public double extract(Tuple2<String, Long> element) {
                  return element.f1; // 返回动态 gap 值
              }
          }))
      .sum(1);
```

**Session Window 的特点：**

1. **动态边界**：窗口何时关闭不确定，只有当「 gap 时间内无新数据」时才触发。
2. **迟到数据敏感**：如果数据乱序严重，session gap 太小会导致本应属于同一会话的数据被分到不同窗口。
3. **结合 allowedLateness**：可以进一步处理乱序数据。

**示例场景：**
- 用户行为分析：用户 10:00 活跃，10:02 还在操作，10:10 才停止，10:11 无操作。设置 gap=5 分钟，则会生成窗口 [10:00, 10:11)。

## 题目五：Flink 的状态管理机制是怎样的？为什么需要状态后端（State Backend）？

### 参考答案

### 状态管理的必要性

Flink 是有状态的流计算引擎。在流处理中，很多场景需要「记住」历史数据：

- 窗口计算：需要缓存窗口内的数据或聚合结果。
- 维表 Join：需要关联外部数据源。
- 复杂事件处理（CEP）：需要匹配事件序列。
- 去重：需要记录已经见过的数据 ID。

如果状态丢失，重新从数据源读取是不现实的（数据可能已过期或不可回溯），因此状态必须被可靠地存储。

### 状态类型

**按算子作用域：**

| 类型 | 说明 | 示例 |
|------|------|------|
| **Keyed State（键控状态）** | 每个 Key 独立的状态，自动按 Key 分区 | `valueState`、`mapState` |
| **Operator State（算子状态）** | 所有算子实例共享的状态 | `ListState`、`UnionListState`、`BroadcastState` |

**Keyed State 常用类型：**
- `ValueState[T]`：单个值
- `ListState[T]`：列表
- `MapState[K, V]`：键值映射
- `ReducingState[T]`、`AggregatingState[I, O]`：聚合类型

### 状态后端（State Backend）

Flink 提供三种状态后端：

| 后端 | 存储介质 | 适用场景 | 特点 |
|------|---------|---------|------|
| **HashMapStateBackend** | JVM Heap | 小状态、低延迟 | 性能最高，状态存储在 TaskManager 堆内存，JM 重启会丢失 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地 SSD） | 大状态（GB 级） | 增量 Checkpoint，LSM 树结构，支持序列化到远程存储 |
| **HashMap + RocksDB（混合）** | 两者结合 | 超大状态 + 高性能 | 热数据在 HashMap，冷数据溢出到 RocksDB |

**选择建议：**
- 状态 < 10MB：直接用 `HashMapStateBackend`。
- 状态大（GB 级）或需要增量 Checkpoint：用 `EmbeddedRocksDBStateBackend`。
- Flink 1.13+ 推荐使用 `HashMapStateBackend` + `state.backend.incremental:true`。

### 状态过期与清理

```java
// 设置状态 TTL（Time-To-Live），自动清理过期状态
ValueStateDescriptor<T> desc = new ValueStateDescriptor<>(
    "my-state",
    TypeInformation.of(new TypeHint<T>() {})
);
desc.enableTimeToLive(StateTtlConfig.newBuilder(Time.hours(1))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)
    .build());
```

**清理策略：**
- **OnReadAndWrite**：每次读写时检查并清理过期数据。
- **OnCreateAndWrite**：仅写入时清理（更轻量，但读到的数据可能是过期的）。
- **NeverReturnExpired**：不返回过期数据，但不主动清理（依赖下次写入时清理）。
