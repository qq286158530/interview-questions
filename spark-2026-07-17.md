## 题目一：Spark 架构与核心组件

**请描述 Spark 的整体架构，以及 Driver、Executor、Master、Worker 的作用是什么？它们之间如何协作？**

### 答案

Spark 采用 **Master-Slave** 架构，核心组件如下：

| 组件 | 角色 | 职责 |
|------|------|------|
| **Driver** | 主导节点 | 负责构建 `SparkContext`，将用户代码转换为 DAG，向集群申请资源，调度 Task 分发到 Executor |
| **Executor** | 工作节点 | 运行在 Worker 节点上，负责执行 Task，缓存数据，向 Driver 汇报结果 |
| **Master** | 资源管理器 | 接收 Worker 注册，管理集群资源，将 Driver 的资源请求分配给合适的 Worker |
| **Worker** | 节点代理 | 启动和管理 Executor 进程，向 Master 注册自己 |

**协作流程：**
1. 用户提交应用，Driver 创建 `SparkContext`，构建 DAG。
2. DAG Scheduler 将 DAG 划分为 Stage，Stage 再拆分为 Task。
3. Task Scheduler 将 Task 发送到 Executor 执行。
4. Executor 以多线程方式运行 Task，结果返回 Driver。
5. Spark 基于 Standalone / YARN / Kubernetes 等集群管理器进行资源调度。

## 题目二：RDD 特性与算子

**RDD 有哪些核心特性？`map`、`filter`、`reduceByKey`、`join` 属于什么类型的算子，它们是如何执行的你了解吗？**

### 答案

**RDD 五大核心特性：**
- **分区列表**：数据被划分为多个分区，每个分区是并行计算的基本单位
- **函数计算**：每个分区上执行相同的计算函数
- **依赖关系**：RDD 之间存在 Narrow Dependency（窄依赖，一对一/范围依赖）或 Wide Dependency（宽依赖，一对多，如 Shuffle）
- **分区器**：可选，决定 key-value 类型 RDD 的分区策略（HashPartitioner / RangePartitioner）
- **位置偏好**：可选，计算每个分区最优位置（数据本地性）

**算子分类：**

| 类型 | 例子 | 特点 |
|------|------|------|
| **Transformation（转换算子）** | `map`、`filter`、`flatMap`、`groupByKey`、`reduceByKey` | 延迟执行，返回新的 RDD |
| **Action（行动算子）** | `collect`、`count`、`saveAsTextFile`、`reduce` | 触发计算，返回结果或写入外部存储 |

**执行过程：**
- `map` / `filter` 属于 **窄依赖**，在同一个分区内部完成，数据无需 Shuffle。
- `reduceByKey` 属于 **宽依赖**，需要在 Shuffle 阶段将相同 key 的数据汇聚到同一分区，再执行聚合。
- `join` 若两个 RDD 的分区器不同，也会产生 Shuffle。

## 题目三：Spark Shuffle 机制

**什么是 Shuffle？为什么 Spark 需要 Shuffle？Shuffle 过程中数据如何写入磁盘？有哪些常见优化手段？**

### 答案

**Shuffle 定义：**
Shuffle 是将数据按 key 进行重新分布的过程。当一个 RDD 的某个分区需要依赖上一个 RDD 中不同分区的数据时（如 `groupByKey`、`reduceByKey`、`join`），必须经过 Shuffle。

**为什么需要 Shuffle：**
窄依赖只需本地计算，无需跨分区数据交换。但宽依赖（如按 key 聚合）必须将相同 key 的数据汇聚到同一分区，才能执行计算——这个跨分区数据交换就是 Shuffle。

**Shuffle 写磁盘流程（Spark 2.x Sort-based Shuffle）：**
1. Map 端：数据写入内存缓冲区（`bufferSize`），达到阈值后 Spill 到磁盘，生成多个临时文件。
2. 本地排序：每个分区文件内按 key 排序（sort-based）。
3. 合并文件：Map Task 完成后，将所有 Spill 文件和内存数据归并排序，写入最终的 Shuffle 数据文件（index + data 文件对）。
4. Reduce 端：拉取对应分区的数据块（Block），进行聚合/连接等操作。

**常见优化手段：**
- **开启 Consolidate机制**（`spark.shuffle.consolidateFiles=true`）：多个 Map Task 共用同一Executor 的 Shuffle 文件，减少文件数量。
- **合理设置 Shuffle Partition 数量**：`spark.sql.shuffle.partitions` 默认 200，生产环境根据数据量调整为 100~500。
- **使用广播变量** 避免 Shuffle：大表 join 时将小表广播，变为 Map-side Join。
- **避免 groupByKey**：优先使用 `reduceByKey`/`aggregateByKey`，它们会在 Map 端先做预聚合，减少 Shuffle 数据量。

## 题目四：Spark 内存管理

**Spark 的内存是如何管理的？Executor 内存分为哪几部分？堆内与堆外内存有什么区别？JVM GC 对 Spark 应用有什么影响，如何调优？**

### 答案

**Executor 内存结构（`spark.executor.memory`）：**

| 区域 | 占比（默认） | 用途 |
|------|-------------|------|
| **Execution 内存** | 约 30% | Shuffle Sort、Hash Join、HashMap 等计算过程 |
| **Storage 内存** | 约 30% | 缓存 RDD/Broadcast/广播变量、临时数据 |
| **User Memory** | 约 20% | 用户数据结构与元数据（NGC 列表等） |
| **Reserved Memory** | 约 300MB | 系统预留 |

**堆内 vs 堆外内存：**
- **堆内内存（On-heap）**：由 JVM 管理，受 GC 影响，存在碎片化问题。
- **堆外内存（Off-heap）**：`spark.memory.offheap.enabled=true`，绕过 JVM 直接分配，不受 GC 管理，适合缓存大对象，可减少 GC 暂停。

**JVM GC 影响与调优：**
- Spark 任务执行时会产生大量短生命周期对象（Task 每次执行都会创建新对象），频繁 Young GC 可能影响性能。
- **调优方向：**
  1. 降低 `spark.executor.cores`（建议 2~4），减少并发对象创建压力。
  2. 增大 `spark.executor.memory` 减少 OOM，同时配合 `spark.memory.fraction` 调整 Execution/Storage 比例。
  3. 开启 G1GC：`spark.executor.extraJavaOptions=-XX:+UseG1GC`。
  4. 对于大缓存场景，开启堆外内存避免 GC 干扰。
  5. 监控 GC 日志（`spark.executor.extraJavaOptions` 配置 `GC_LOG`），分析 Young/Old 比例是否合理。

## 题目五：广播变量与 DataFrame/DataSet

**什么是广播变量？它解决了什么问题？在 DataFrame 和 DataSet 中如何高效使用广播？Broadcast Join 的原理是什么？**

### 答案

**广播变量的作用：**
当 Executor 需要访问一份**小而只读**的数据（如维度表、配置表）时，如果每个 Task 都从外部拉取，会产生巨大的网络开销。广播变量将数据复制到每个 Executor 的内存中，所有 Task 共享一份，避免重复 Shuffle。

**DataFrame / Dataset 中的广播：**
- Spark SQL 自动识别小表并广播：**`spark.sql.autoBroadcastJoinThreshold`**（默认 10MB），小于该阈值的表自动广播。
- 手动广播：`import org.apache.spark.sql.functions._; val broadcastDF = broadcast(df)`
- 查看广播计划：`df.explain(true)` 中看到 `BroadcastHashJoin` 即表示使用了广播。

**Broadcast Join 原理：**
1. Driver 端将小表数据收集为 `BroadcastExchange` 算子，将数据序列化后发送到各 Executor 节点。
2. 大表各分区独立与本地广播数据进行 Hash Join，无需跨节点 Shuffle。
3. 适合场景：**大表 join 小表**、维度表 Lookup。

**注意事项：**
- 广播变量数据必须能放入 Executor 内存，过大会导致 OOM。
- 广播后数据为只读，修改无效。
- Spark 3.x 起默认使用 **AQE（Adaptive Query Execution）**，运行时动态调整 Broadcast 阈值，进一步优化性能。
