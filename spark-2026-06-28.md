## 题目一：Spark 架构与核心组件

**请描述 Spark 的整体架构，以及 Driver、Executor、Master、Worker 的作用是什么？**

### 参考答案

Spark 采用 **主从架构**，核心组件如下：

| 组件 | 角色 | 职责 |
|------|------|------|
| **Driver** | 主节点 | 负责提交 Spark Application，将任务转换为 Task 分发给 Executor；持有 SparkContext |
| **Executor** | 从节点 | 每个 Worker 节点上的进程，负责执行具体 Task 并汇报结果；拥有独立的 JVM 进程 |
| **Master** | 集群管理器 | 负责资源管理（Standalone 模式下），分配 Worker 资源 |
| **Worker** | 节点监控器 | 监控本节点资源（CPU、内存），与 Master 通信汇报 |

**工作流程：**

1. 用户提交 Application → Driver 创建 SparkContext
2. Driver 向 Cluster Manager 申请资源
3. Cluster Manager 在 Worker 节点上启动 Executor
4. Driver 解析代码生成 DAG，切分为 Stage 和 Task
5. Task 被发送到各 Executor 并行执行
6. 结果汇总返回 Driver

## 题目二：RDD 特性与惰性机制

**Spark 中 RDD 是什么？为什么 RDD 是惰性求值的？这种设计有什么优缺点？**

### 参考答案

**RDD（Resilient Distributed Dataset）** 是 Spark 最底层的抽象，表示不可变、可分区、可以并行计算的数据集合。

**五大特性：**

1. **分区列表（partitions）**：数据被分成多个分区
2. **计算函数（compute）**：每个分区如何计算
3. **依赖关系（dependencies）**：RDD 之间的依赖（窄依赖/宽依赖）
4. **分区器（partitioner）**：可选，K-V 型 RDD 的分区策略（HashPartitioner / RangePartitioner）
5. **优先位置（preferredLocations）**：数据在哪里偏好在哪个节点计算

**惰性求值（Lazy Evaluation）：**

- RDD 的 Transformation（如 `map`、`filter`、`groupByKey`）不会立即执行，只是记录逻辑
- 只有遇到 **Action**（如 `collect`、`count`、`saveAsTextFile`）时，才会触发真正的计算
- Driver 会根据依赖关系将 DAG 优化为 **DAG 有向无环图**，再生成 Task 执行

**优点：**
- 可以做**全局优化**（如 pipeline 合并、减少 shuffle）
- 减少中间结果的物化，节省内存

**缺点：**
- 调试困难，错误往往在 Action 时才暴露
- 不适合流式/增量计算场景

## 题目三：Shuffle 机制与优化策略

**什么是 Shuffle？为什么 Shuffle 是 Spark 性能瓶颈？如何优化 Shuffle？**

### 参考答案

**Shuffle 是指将数据根据 Key 进行重新分布的过程**，即 Map 端的数据重新分区到 Reduce 端，需要跨节点传输数据。

**经典场景：** `groupByKey`、`reduceByKey`、`join`、`repartition` 等操作会触发 Shuffle。

**为什么是瓶颈：**

1. **磁盘 I/O**：Shuffle 数据先写磁盘再网络传输
2. **网络 I/O**：跨节点数据复制
3. **内存压力**：需要缓存 shuffle 数据，可能触发磁盘溢出
4. **JVM 开销**：大量的序列化/反序列化、GC

**优化策略：**

| 策略 | 说明 |
|------|------|
| 减少 Shuffle 数据量 | 使用 `mapPartitions` 替代 `map`；先 filter 再 join |
| 合理设置分区数 | 分区数 ≈ 并行度，一般 2-4 个 CPU 核 / task；避免过多小分区或过少大分区 |
| 使用广播变量 | 小表广播，避免 Shuffle（`broadcast(rdd)`） |
| 开启 Shuffle 压缩 | `spark.shuffle.compress=true` |
| 选择合适的分区器 | 避免数据倾斜，可用自定义分区器 |
| 开启 Sort-based Shuffle | Spark 2.x+ 默认 `sort-based shuffle`，减少文件数 |

## 题目四：广播变量与内存管理

**Spark 广播变量的原理是什么？它解决了什么问题？广播变量一定比普通变量快吗？**

### 参考答案

**广播变量（Broadcast Variable）** 的原理：

- 将**只读**的大变量（如 lookup 表、配置数据）提前发送到各节点的 Executor 内存中
- 所有 Task 共享同一份副本，避免每个 Task 都持有一份拷贝
- 底层使用 **BitTorrent 类似的广播机制**（先发给少量节点，再由这些节点分发给其他节点）

**解决了什么问题：**
- 避免 Shuffle 时相同数据重复传输
- 减少网络 I/O 开销
- 降低Executor 内存占用（不用每个 Task 拷贝）

**广播变量一定更快吗？不一定！**

| 场景 | 结论 |
|------|------|
| 大表 join 小表（小表可广播） | ✅ 广播更快，避免 Shuffle |
| 数据量很大，无法放入 Executor 内存 | ❌ 广播会 OOM，反而更慢 |
| Task 数量很多，重复数据多 | ✅ 广播节省显著 |
| 数据量适中，本身 shuffle 开销不大 | ⚠️ 差异不明显 |

**正确做法：** 通过 `spark.sql.autoBroadcastJoinThreshold`（默认 10MB）自动判断是否广播，或手动 `broadcast()`。

## 题目五：DataFrame vs DataSet vs RDD

**Spark 中 DataFrame、DataSet 和 RDD 的区别是什么？什么时候该用哪一种？**

### 参考答案

| 特性 | RDD | DataFrame | DataSet |
|------|-----|-----------|---------|
| **数据结构** | 无结构，每行自己解析 | 有 Schema（列名+类型） | 有 Schema + 类型安全 |
| **语言支持** | 各语言原生 | 各语言（Row 通用） | 只支持 Java/Scala |
| **优化** | 无 Catalyst 优化 | 有 Catalyst 优化 | 有 Catalyst 优化 |
| **序列化** | 手动/Java 序列化 | Tungsten（UnsafeRow） | Tungsten + Encoder |
| **编译时检查** | 无 | 无 | 有（泛型约束） |
| **操作方式** | 函数式 map/filter | SQL / DSL | 混合（Lambda + DSL） |

**DataFrame = DataSet[Row]**（Scala 语法糖）

**为什么 DataFrame/DataSet 更快？**

Spark 2.x+ 使用 **Tungsten 内存管理 + Catalyst 优化器**：
- Catalyst 会将 DataFrame/DataSet 操作解析为 Logical Plan → Physical Plan，进行谓词下推、列剪裁、常量折叠等优化
- Tungsten 使用 off-heap 内存和二进制格式，避免 JVM 序列化开销

**选择建议：**

- **结构化数据（CSV、JSON、Parquet）** → DataFrame / DataSet（清晰、高效）
- **非结构化数据（日志、文本）** → RDD（灵活）
- **需要类型安全（Java/Scala）** → DataSet
- **性能优先，结构清晰** → DataFrame（最常用）
