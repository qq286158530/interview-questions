## 题目一：Redis 为什么这么快？它的线程模型是怎样的？

**参考答案：**

Redis 之所以这么快，主要有以下四个原因：

1. **纯内存操作**：Redis 数据读写都在内存中完成，内存访问速度是纳秒级，而磁盘访问是毫秒级，两者相差数个数量级。

2. **高效的 I/O 多路复用 + 单线程事件循环**：Redis 使用单线程事件循环配合 I/O 多路复用（epoll/kqueue），让单个线程同时处理多个网络连接的 I/O 事件，避免了多线程的上下文切换和锁竞争问题。

3. **优化的内部数据结构**：Redis 内部实现了高度优化的数据结构，如 SDS（简单动态字符串）、quicklist（替代双向链表+压缩列表）、skiplist（跳表）、hashtable 等，会根据数据大小动态选择最合适的编码。

4. **简洁高效的通信协议**：Redis 使用自己设计的 RESP（REdis Serialization Protocol）协议，实现简单、解析性能好、二进制安全。

**关于 Redis 线程模型需要澄清两个常见误区：**

- **Redis 4.0 之前**：完全是单线程模型，主线程负责接收请求、解析命令、执行操作、返回响应。
- **Redis 4.0 之后**：引入了后台线程（bio_close_file、bio_aof_fsync、bio_lazy_free），用于处理关闭文件、AOF 刷盘、异步删除大 key 等耗时任务，避免阻塞主线程。
- **Redis 6.0 之后**：引入了多线程 I/O，仅用于处理网络数据的读写（read/write socket），而命令执行仍然是单线程。所以 Redis 6.0 官方声称性能提升至少一倍以上。

**为什么早期 Redis 使用单线程？** 因为 Redis 的瓶颈不在 CPU，而在内存和网络 I/O，单线程避免了多线程竞争带来的性能损耗，更易维护。

## 题目二：RDB 和 AOF 持久化的区别是什么？生产环境如何选择？

**参考答案：**

Redis 提供了两种持久化机制来防止数据丢失：

### RDB（Redis Database Backup）

- **原理**：通过生成某一时刻的内存数据快照（二进制文件）来实现全量备份。
- **触发方式**：
  - 手动触发：`BGSAVE`（异步，在子进程执行）
  - 自动触发：根据配置（如 `save 900 1` 表示 900 秒内≥1 次修改则触发）
- **优点**：文件紧凑、恢复速度快（全量加载）
- **缺点**：可能丢失最后一次快照之后的数据；大数据量时 `fork()` 可能造成主线程短暂阻塞

### AOF（Append Only File）

- **原理**：记录每一次写操作命令到日志文件，恢复时逐一执行命令。
- **三种写回策略**：

| 策略 | 说明 | 安全性 | 性能 |
|------|------|--------|------|
| `always` | 每次写命令执行后同步写盘 | 最高 | 最差 |
| `everysec`（默认） | 每秒将缓冲区内容写回硬盘 | 中等 | 较好 |
| `no` | 由操作系统决定何时写盘 | 最差 | 最好 |

- **优点**：数据安全性更高（特别是 `everysec` 可实现秒级持久化）；支持大文件重写（bgrewriteaof）压缩体积
- **缺点**：文件较大；恢复速度慢（逐条执行命令）

### 混合持久化（Redis 4.0+）

开启混合持久化后，AOF 重写时先以 RDB 格式写入全量数据，后续增量命令以 AOF 格式追加。兼顾恢复速度和安全性。

### 生产环境推荐配置

```conf
# 开启 AOF
appendonly yes
appendfsync everysec

# 开启混合持久化
aof-use-rdb-preamble yes

# RDB 作为备份
save 900 1
save 300 10
save 60 10000
```

**总结**：如果对数据安全性要求高，首选 AOF `everysec` + 混合持久化；如果更注重性能且能容忍少量数据丢失，可以只开 RDB。

## 题目三：Redis 的五种基本数据类型是什么？各自的应用场景？

**参考答案：**

Redis 提供了 **5 种基本数据类型**和 **4 种特殊数据类型**：

### 5 种基本数据类型

| 类型 | 底层数据结构 | 典型应用场景 |
|------|------------|------------|
| **String** | SDS（简单动态字符串） | 缓存、计数器（INCR）、分布式锁（SETNX）、Session 共享 |
| **List** | quicklist（3.2+版本） | 消息队列（LPUSH+BRPOP）、最新消息列表、评论列表 |
| **Hash** | hashtable / listpack | 存储对象（用户信息、商品详情）、购物车 |
| **Set** | hashtable / intset | 标签系统、共同好友（ZINTER）、去重 |
| **ZSet（Sorted Set）** | skiplist + hashtable | 排行榜（ZADD）、延迟队列（按时间戳 score 排序） |

### 4 种特殊数据类型

| 类型 | 应用场景 |
|------|---------|
| **Bitmap** | 二值状态统计：用户签到、登录状态、连续签到统计 |
| **HyperLogLog** | 海量数据基数统计：百万级 UV 去重计数（内存仅 12KB） |
| **GEO** | 地理位置存储：滴滴叫车、附近的人/商家 |
| **Stream** | 消息队列：支持消费者组、消息 ID 自动生成、类似 Kafka |

### 各数据类型经典面试追问

**Q：String 还是 Hash 存储对象更好？**
- String：整体序列化读写，简单直接，适合对象结构固定且主要整体读写的场景。
- Hash：字段级读写，可单独修改某个字段，更省内存（小型 Hash 用 ziplist 优化），适合需要频繁修改部分字段的场景。

**Q：ZSet 为什么用跳表而不是平衡树/红黑树/B+树？**
- 跳表实现简单，插入/删除只需维护相邻层指针，无需旋转操作；
- 范围查询（ZRANGE）效率高，从高层开始逐层降维查找；
- 跳表的空间复杂度 O(n) 更低，而 B+ 树需要维护完整索引结构。

**Q：购物车信息为什么用 Hash 存储？**
- 用户 ID 为 key，商品 ID 为 field，商品数量为 value，增删改查都方便，无需操作整个对象。

## 题目四：什么是缓存穿透、缓存击穿、缓存雪崩？如何解决？

**参考答案：**

这三个问题都是缓存机制常见的高并发异常问题：

### 1. 缓存穿透

**定义**：查询一个数据库中根本不存在的数据，缓存层和数据库层都查不到，每次请求都穿透到数据库。

**常见场景**：恶意攻击（如爬虫、故意传入大量不存在的 ID 查询）

**解决方案：**
- **布隆过滤器（BloomFilter）**：将所有存在的 key 存入布隆过滤器，请求到达时先检查布隆过滤器，如果不存在直接返回，避免无效查询打到数据库。
- **缓存空值**：即使查询结果为空，也将 `null` 缓存起来（需设置较短的过期时间），下次直接返回 null 而不查数据库。

### 2. 缓存击穿

**定义**：某个热点 key 过期后，恰好有大量并发请求同时访问这个 key，由于缓存不存在，所有请求都直接打到数据库。

**解决方案：**
- **互斥锁**：使用 Redis SETNX 或分布式锁（如 Redisson），只允许一个请求去查数据库并回写缓存，其他请求等待锁释放后从缓存读取。
- **永不过期**：对热点 key 不设置 TTL，而是通过逻辑过期时间（value 中包含过期时间戳），后台异步更新缓存。

### 3. 缓存雪崩

**定义**：大量 key 集中过期（或者 Redis 宕机），导致大量请求直接打到数据库，造成数据库瞬时压力过大。

**解决方案：**
- **过期时间随机化**：给不同的 key 设置不同的 TTL（如在原 TTL 基础上加 `Math.random()*60` 秒），避免集中过期。
- **Redis 高可用架构**：使用 Redis Sentinel（哨兵模式）或 Redis Cluster，保证 Redis 本身的高可用性，避免单点故障。
- **熔断降级**：当 Redis 不可用时，前端服务自动切换到降级逻辑（返回默认值或静态数据）。
- **预热**：在缓存过期前，主动刷新热点数据到缓存。

### 对比总结

| 问题 | 原因 | 核心解决思路 |
|------|------|------------|
| 缓存穿透 | 查询不存在的数据 | 布隆过滤器拦截 + 缓存空值 |
| 缓存击穿 | 热点 key 过期瞬间高并发 | 互斥锁 / 永不过期 |
| 缓存雪崩 | 大量 key 集中过期 / Redis 宕机 | 过期时间随机化 + 高可用架构 + 熔断降级 |

## 题目五：Redis Cluster 是如何实现数据分片的？主从模式下数据如何同步？

**参考答案：**

### Redis Cluster 数据分片机制

Redis Cluster 采用 **哈希槽（Hash Slot）** 进行数据分片：

- **槽位总数**：共 16384（16k）个槽位
- **分片原理**：客户端通过 `CRC16(key) % 16384` 计算 key 对应的槽号，然后直接路由到对应节点
- **节点分配**：每个 Redis 节点负责一部分槽位（如 3 节点集群中，节点 A 负责 0-5460，节点 B 负责 5461-10922，节点 C 负责 10923-16383）
- **特点**：客户端可直连对应节点（smart 客户端），无需经过代理层，性能更高

### Redis 主从复制原理

主从复制用于实现数据备份和读写分离：

**全量同步（Full Sync）**：
1. 从节点发送 `PSYNC ? -1`（或 `SYNC`，第一次连接）
2. 主节点执行 `BGSAVE` 生成 RDB 快照，同时将新命令追加到 **复制积压缓冲区（replication backlog）**
3. 将 RDB 文件发送给从节点
4. 从节点清空本地数据，加载 RDB 文件
5. 主节点将缓冲区中的增量命令发送给从节点

**增量同步（Partial Sync）**：
- 从节点断线重连后，发送 `PSYNC <run_id> <offset>`
- 如果主节点的复制积压缓冲区中包含断线期间的写命令，则发送这些增量命令给从节点
- 否则触发全量同步

### 故障转移（Failover）过程

1. 哨兵/从节点定时向其他节点发送 `PING`
2. 如果超过一半节点认为某个主节点不可达，触发故障转移
3. 从节点向集群中其他节点广播请求投票（FAILOVER_AUTH_REQUEST）
4. 获得多数票的从节点升级为新的主节点
5. 原主节点恢复后自动变为新主节点的从节点

### 脑裂问题（Split-Brain）及解决方案

**问题**：网络分区导致集群出现两个主节点，客户端可能同时向两个主节点写入数据。

**危害**：
- 原主节点上写入的数据在恢复后被清空同步到新主节点，造成数据丢失
- 两个主节点数据不一致，无法合并

**解决方案**：
```conf
min-slaves-to-write 1      # 主节点至少要有 1 个可写的从节点才允许写入
min-slaves-max-lag 10      # 从节点复制延迟不超过 10 秒
```
此外还应合理配置 `down-after-milliseconds`（避免网络抖动误判）和 `quorum`（半数以上确认才触发故障转移）。

### 集群方案对比

| 方案 | 适用场景 | 特点 |
|------|---------|------|
| 主从复制 | 小规模、读写分离 | 手动故障转移 |
| 哨兵模式（Sentinel） | 中等规模、自动故障转移 | 1 主多从，提供监控和自动 failover |
| Redis Cluster | 大规模、数据分片 | 16384 槽自动分片，支持横向扩展 |
