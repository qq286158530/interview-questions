# Redis 面试题

> 本文档整理了 5 道高质量的 Redis 面试题，涵盖数据类型、持久化、过期删除与内存淘汰、缓存策略、集群等核心知识点。

## 题目一：Redis 有哪些数据类型？各自的使用场景是什么？

### 答案

Redis 提供了丰富的数据类型，常见的有 **5 种基础数据类型**和**4 种扩展数据类型**。

#### 5 种基础数据类型

| 数据类型 | 使用场景 |
|----------|----------|
| **String（字符串）** | 缓存对象、常规计数、分布式锁、共享 session 信息等 |
| **Hash（哈希）** | 缓存对象、购物车等 |
| **List（列表）** | 消息队列（需注意：生产者需自行实现全局唯一 ID，不能以消费组形式消费数据） |
| **Set（集合）** | 聚合计算（并集、交集、差集）场景，如点赞、共同关注、抽奖活动等 |
| **Zset（有序集合）** | 排序场景，如排行榜、电话和姓名排序等 |

#### 4 种扩展数据类型（Redis 2.2+）

| 数据类型 | 版本 | 使用场景 |
|----------|------|----------|
| **BitMap（位图）** | 2.2+ | 二值状态统计，如签到、判断用户登录状态、连续签到用户总数等 |
| **HyperLogLog（基数统计）** | 2.8+ | 海量数据基数统计，如百万级网页 UV 计数等 |
| **GEO（地理信息）** | 3.2+ | 存储地理位置信息，如滴滴叫车等 |
| **Stream（流）** | 5.0+ | 消息队列，相比 List 实现的消息队列，有自动生成全局唯一消息 ID、支持消费组形式消费数据两大特性 |

#### 底层数据结构

- **String** → SDS（简单动态字符串），支持二进制安全、O(1) 长度获取、自动扩容
- **List** → Redis 3.2+ 统一使用 quicklist（双向链表 + 压缩列表）
- **Hash** → 压缩列表（元素少时）或哈希表
- **Set** → 整数集合（纯整数且少时）或哈希表
- **ZSet** → 压缩列表（元素少时）或跳表（Skip List）
- Redis 7.0+ 压缩列表已废弃，改为 listpack

## 题目二：Redis 持久化机制是什么？AOF 和 RDB 有什么区别？

### 答案

Redis 共有 **三种**数据持久化方式：

#### 1. AOF 日志（Append Only File）

每执行一条写操作命令，就把该命令以追加的方式写入一个文件。Redis 重启时读取该文件，逐条执行命令来恢复数据。

**AOF 写回策略（appendfsync 配置项）：**

| 策略 | 含义 | 优点 | 缺点 |
|------|------|------|------|
| **Always** | 每次写命令执行完，同步写回硬盘 | 数据不丢失 | 开销最大，阻塞主线程 |
| **Everysec**（默认） | 每次写命令执行完，先写入内核缓冲区，每秒刷盘一次 | 性能适中，最多丢 1 秒数据 | 可能丢 1 秒数据 |
| **No** | 由操作系统决定何时刷盘 | 性能最好 | 数据丢失不可控 |

**优点：** 安全性相对较高，支持丰富的写回策略。  
**缺点：** 日志文件体积可能很大，数据恢复速度慢（需要逐条执行命令）。

**AOF 重写机制：** 当 AOF 文件过大时，后台子进程 `bgrewriteaof` 会读取数据库中所有键值对，用一条命令记录每个 key 的最新值，从而压缩文件。

#### 2. RDB 快照（Redis Database）

将某一时刻的内存数据以**二进制格式**写入磁盘。恢复时直接读取文件，速度极快。

- `save` 命令：在主线程执行，会阻塞
- `bgsave` 命令：创建子进程执行，不阻塞主线程

Redis 默认配置满足以下任一条件时自动触发 `bgsave`：
- 900 秒内至少 1 次修改
- 300 秒内至少 10 次修改
- 60 秒内至少 10000 次修改

RDB 快照是**全量快照**，通过 COW（Copy-On-Write，写时复制）技术，在生成快照期间主线程仍可处理写操作。

**优点：** 恢复速度快。  
**缺点：** 快照频率难把握——太频繁影响性能，太少则丢数据多。

#### 3. 混合持久化（Redis 4.0+）

结合 AOF 和 RDB 的优点。在 AOF 重写时，先以 RDB 格式写入全量数据，再以 AOF 格式追加增量命令。

- AOF 文件前半部分是 RDB 格式的全量数据 → 加载快
- 后半部分是 AOF 格式的增量数据 → 丢数据少

**优点：** 启动快 + 数据丢失少。  
**缺点：** AOF 可读性差，不兼容 Redis 4.0 之前版本。

## 题目三：Redis 的过期删除策略和内存淘汰策略有什么区别？

### 答案

#### 过期删除策略

Redis 对 key 设置过期时间后，会将 key 和过期时间存入**过期字典（expires dict）**。Redis 采用**「惰性删除 + 定期删除」**配合使用的策略：

**惰性删除：** 每次访问 key 时才检查是否过期，过期则删除。
- ✅ 对 CPU 友好，几乎不占系统资源
- ❌ 对内存不友好，过期 key 长期占用内存

**定期删除：** 每隔一段时间随机抽取一批 key 检查，删除其中过期的 key。
- 流程：从过期字典随机抽 20 个 key → 删除已过期的 → 若过期比例 > 25% 则继续，否则停止
- 为防止阻塞，单次执行时间上限为 25ms
- ✅ 对内存友好，可释放过期内存
- ❌ 执行频率难以精确控制

#### 内存淘汰策略

当 Redis 运行内存达到 `maxmemory` 配置的阈值时，触发**内存淘汰机制**。共 8 种策略，分为两类：

**不淘汰（1 种）：**
- `noeviction`：不淘汰任何数据，直接返回错误（Redis 3.0+ 默认）

**淘汰设置了过期时间的 key（4 种）：**
- `volatile-random`：随机淘汰
- `volatile-ttl`：优先淘汰更早过期的
- `volatile-lru`：淘汰最久未使用的（Redis 3.0 前默认）
- `volatile-lfu`：淘汰使用频率最低的（Redis 4.0+）

**淘汰所有 key（3 种）：**
- `allkeys-random`：随机淘汰
- `allkeys-lru`：淘汰最久未使用的
- `allkeys-lfu`：淘汰使用频率最低的（Redis 4.0+）

> LRU（Least Recently Used）和 LFU（Least Frequently Used）的区别：  
> - LRU 基于「最近访问时间」，适合热点数据较稳定的场景  
> - LFU 基于「访问频次」，能更好地解决缓存污染问题（如一次读取大量数据）

## 题目四：如何解决缓存雪崩、缓存击穿和缓存穿透？

### 答案

这是三个经典的缓存问题，核心都是"大量请求绕过缓存直达数据库"。

#### 缓存雪崩

**问题：** 大量缓存数据在同一时间过期，此时有大量用户请求都无法在 Redis 中处理，全部打到数据库。

**解决方案：**
1. **缓存过期时间随机化：** 在原有失效时间基础上 + 随机值（如 1~10 分钟），避免集体失效
2. **设置缓存不过期：** 通过后台服务异步更新缓存数据

#### 缓存击穿

**问题：** 某个热点数据过期（如秒杀商品），此时大量请求直接打穿到数据库。

**解决方案：**
1. **互斥锁：** 使用 `setNX` 加锁，保证同一时间只有一个线程重建缓存，其他线程等待或返回默认值
2. **热点数据永不过期：** 由后台异步更新，或在热点数据过期前主动刷新

#### 缓存穿透

**问题：** 请求的数据既不在缓存中，也不在数据库中（如恶意攻击大量请求不存在的 ID），数据库压力骤增。

**解决方案：**
1. **非法请求拦截：** 在 API 入口校验请求参数，过滤恶意请求
2. **缓存空值/默认值：** 对查询不存在的数据，在缓存中设置空值或默认值
3. **布隆过滤器（Bloom Filter）：** 写入数据库时在布隆过滤器中标记，查询前先通过布隆过滤器快速判断数据是否存在

#### 常见缓存更新策略（Cache Aside）

- **写操作：** 先更新数据库，再删除缓存
- **读操作：** 缓存命中 → 直接返回；未命中 → 从数据库读取并写入缓存，再返回

> ⚠️ 不能先删缓存再更新数据库，因为在「读+写」并发时会导致数据不一致。

## 题目五：Redis 如何实现高可用？主从复制、哨兵模式、集群模式有什么区别？

### 答案

#### 主从复制

一主多从模式，主节点负责读写，从节点只读。主节点自动将写操作同步给从节点。

- 读写分离：主节点写，从节点读
- 主从复制是**异步**的——主节点执行完命令即返回客户端，不等待从节点执行完
- 无法保证强一致性，数据不一致难以避免

#### 哨兵模式（Sentinel）

解决主从复制中"主节点宕机需手动恢复"的问题。哨兵是一个独立的进程，负责：

1. **监控：** 监控主从节点的运行状态
2. **自动故障转移：** 主节点失联后，在从节点中选举出新主节点
3. **通知：** 告诉客户端新的主节点地址

#### Redis Cluster（切片集群）

当数据量单台机器无法承载时，使用切片集群将数据分布到多个节点。

**核心机制——哈希槽（Hash Slot）：**
- 整个集群共有 **16384 个哈希槽**
- 每个 key 通过 `CRC16(key) % 16384` 映射到对应槽位
- 槽位分配给各节点（平均分配或手动分配）

**集群脑裂问题：**  
当主节点网络故障但仍与客户端连通时，哨兵可能选举出新主节点，形成"两个主节点"的脑裂现象。网络恢复后，旧主节点降级为从节点并执行全量同步，之前客户端写入的数据会丢失。

**解决方案：** 配置 `min-slaves-to-write` 和 `min-slaves-max-lag`，确保主节点至少有 N 个从节点且延迟小于 T 秒时才接收写请求。

#### 三种高可用方案对比

| 方案 | 适用场景 | 特点 |
|------|----------|------|
| **主从复制** | 数据量不大、要求简单的场景 | 一主多从，读写分离 |
| **哨兵模式** | 对可用性要求高，需要自动故障转移 | 自动选主 + 故障转移 |
| **Cluster 模式** | 数据量大，需横向扩展 | 哈希槽分片，支持动态扩缩容 |
