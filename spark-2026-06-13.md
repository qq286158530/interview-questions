## 题目一：Spark 架构与核心组件

**问题：** 请描述 Spark 的整体架构，以及 Driver、Executor、Master、Worker 的作用与相互关系。

**答案：**

Spark 采用 **Master-Slave** 架构，核心组件如下：

| 组件 | 角色 | 职责 |
|------|------|------|
| **Driver** | 主控节点 | 负责将用户代码转换为 DAG，构建 TaskSet，协调任务调度，与 Executor 通信 |
| **Executor** | 工作节点 | 负责执行具体的 Task，将结果返回 Driver，同时管理各自节点上的内存和磁盘 |
| **Master** | 资源管理主节点 | 负责集群资源管理（YARN Cluster 模式下由 ResourceManager 承担），接收 Worker 注册并分配资源 |
| **Worker** | 工作节点守护进程 | 负责向 Master 注册本节点资源，启动和管理 Executor 进程 |

**工作流程：**
1. 用户提交 Spark 应用程序，Driver 启动并创建 `SparkContext`；
2. Driver 将代码解析为 DAG，根据 `shuffle` 划分 Stage；
3. Driver 向 Cluster Manager 申请资源；
4. Worker 节点上的 Executor 启动后向 Driver 注册；
5. Driver 将 Task 分发到各个 Executor 执行；
6. Executor 执行完毕后将结果汇报给 Driver。

## 题目二：RDD 特性与算子分类

**问题：** 什么是 RDD？RDD 有哪些核心特性？请列举常见的 Transformation 和 Action 算子，并说明两者的区别。

**答案：**

**RDD（Resilient Distributed Dataset）** 是 Spark 最基本的数据抽象，表示不可变、分区、可并行计算的数据集合。

**RDD 五大核心特性：**
1. **`分区列表`**（Partitions）：数据被划分成多个分区，每个分区是并行计算的基本单元；
2. **`计算函数`**（Compute Function）：每个分区上执行的计算逻辑；
3. **`依赖关系`**（Dependencies）：RDD 之间的依赖关系（窄依赖 / 宽依赖），是容错的基础；
4. **`分区器`**（Partitioner）：可选，决定 Shuffle 时数据的分区方式（HashPartitioner / RangePartitioner）；
5. **`最佳位置`**（Preferred Locations）：可选，调度任务时优先将计算移到数据所在节点（数据本地性）。

**Transformation 算子（转换算子）：**
- 属于**懒加载**，不会立即执行，只有遇到 Action 才会触发计算；
- 常见：`map`、`filter`、`flatMap`、`groupByKey`、`reduceByKey`、`join`、`union`、`distinct`、`sortByKey` 等。

**Action 算子（行动算子）：**
- 会**触发实际计算**，将结果写入外部存储或返回给 Driver；
- 常见：`collect`、`count`、`take`、`reduce`、`collectAsMap`、`saveAsTextFile`、`foreach` 等。

**两者的核心区别：** Transformation 延迟执行，Action 触发执行并生成 Job。

## 题目三：Spark Shuffle 机制与优化

**问题：** 什么是 Shuffle？Shuffle 在 Spark 中是如何实现的？有哪些常见的优化策略？

**答案：**

**Shuffle（洗牌）** 是 Spark 中将数据按 Key 重新分布的过程，发生在 `groupByKey`、`join`、`reduceByKey` 等需要跨分区聚合的场景。Shuffle 是 Spark 最昂贵的操作之一，会触发磁盘 I/O、网络 I/O 和序列化。

**实现机制（Hash Shuffle → Sort Shuffle → Tungsten Sort Shuffle）：**

1. **`Hash Shuffle`（早期版本）：** 每个 Map 任务为每个 Reduce 任务产生一个文件，导致文件数量爆炸（`M × R` 个文件），小文件问题严重，已基本废弃。

2. **`Sort Shuffle`（Spark 1.2+）：** 所有数据写入同一个文件（按 Key 排序），同时生成索引文件。每个 Mapper 只需产生 `R` 个文件（`R` = Reduce 分区数），文件数大幅减少。

3. **`Tungsten Sort Shuffle`（钨丝计划，Spark 1.5+）：** 基于 `sun.misc.Unsafe` 直接操作堆外内存，省去序列化开销，直接对二进制数据排序，性能更高。当 Shuffle 数据量小于 `spark.shuffle.sort.bypassMergeThreshold`（默认200）时，回退到 Hash Shuffle 模式。

**常见优化策略：**

| 优化方向 | 具体措施 |
|----------|----------|
| **减少 Shuffle 数据量** | 避免 `groupByKey`，优先用 `reduceByKey` / `aggregateByKey`（在 Map 端预聚合） |
| **合理设置分区数** | 分区数过少导致并行度低、OOM；过多则小任务开销大；推荐 `cores * 2~3` |
| **使用广播变量** | 小表广播可避免 Shuffle（如 `Broadcast Join`） |
| **开启 Shuffle 压缩** | 设置 `spark.shuffle.compress=true`，使用 `spark.io.compression.codec`（推荐 LZ4） |
| **避免数据倾斜** | 加盐（Salt）、两阶段聚合、打散大 Key |

## 题目四：Spark 内存管理机制

**问题：** Spark 1.6 之后引入了 Unified Memory（统一内存管理），请详细说明其内存模型，包括存储内存、执行内存、用户内存和系统内存的划分与动态调度。

**答案：**

Spark 内存模型将堆内内存划分为以下几大区域：

```
+----------------------------------+--------------------+
|          Storage Memory          |      Execution     |
|    （存储内存：缓存 RDD / 广播）    |  Memory（执行内存）  |
|          动态共享                  |     动态共享        |
+----------------------------------+--------------------+
|         User Memory（用户内存）     |    Reserved Memory |
|   用户代码、Spark 内部元数据等      |     系统预留 300MB  |
+----------------------------------+--------------------+
```

**各区域说明：**

| 区域 | 默认占比 | 说明 |
|------|----------|------|
| **Storage Memory** | ~45%（可动态调节） | 用于缓存 RDD 持久化数据（`cache()` / `persist()`）和广播变量 |
| **Execution Memory** | ~45%（可动态调节） | 用于 Shuffle 排序、Hash 聚合、Hash Join 等执行操作的临时内存 |
| **User Memory** | ~40% | 用户代码（UDF、数据结构等）和 Spark 内部元数据，不可被 Spark 回收 |
| **Reserved Memory** | 300MB | 系统预留，极其有限，不可改变 |

**动态调度机制：**

- 当 Storage Memory 不足时，可以**借用** Execution Memory（反之亦然）；
- 借用后若原持有者需要内存，必须等待对方释放到安全阈值以下才能回收；
- 存在 `storageRegionProportion = 0.5` 的默认比例上限，防止某一方过度侵占；
- 当 `spark.memory.fraction = 0.6`（JVM Old Gen 的 60%），`spark.memory.storageFraction = 0.5`（存储/执行各占 50%）。

**存储级别：** `MEMORY_ONLY`、`MEMORY_AND_DISK`、`DISK_ONLY`、`MEMORY_ONLY_2`（副本）等。

**来源：** [Apache Spark 官方文档 - Memory Management](https://spark.apache.org/docs/latest/tuning.html#memory-management)

## 题目五：广播变量（Broadcast Variable）

**问题：** 什么是广播变量？它解决了什么问题？使用广播变量有哪些注意事项？

**答案：**

**广播变量**是 Spark 提供的优化机制，允许将**只读的小数据**高效地分发到每个 Executor，避免该数据在每个 Task 中重复传输和存储。

**解决的问题：**
- 场景：在大数据 Join 小表时，小表数据会随每个 Task 复制 N 份（`N = partitions`），造成巨大的网络开销和内存浪费；
- 广播后，小表数据只在每个 Executor 加载**一份**，所有在该 Executor 运行的 Task 共享访问，大幅降低网络 I/O。

**使用方式：**

```scala
// 广播小表
val broadcastVar = sc.broadcast(List("北京", "上海", "深圳"))

// 在算子中访问
rdd.filter(x => broadcastVar.value.contains(x))
```

**注意事项：**

| 注意点 | 说明 |
|--------|------|
| **数据大小** | 广播适合小到中等规模数据（建议 < 20MB），过大会撑爆 Executor 堆外内存 |
| **只读性** | 广播变量不可修改，否则结果不可预期 |
| **序列化** | 广播数据必须可序列化（Java / Scala 序列化或 Kryo） |
| **版本一致性** | SparkContext 生命周期内不要重复创建同名广播变量 |
| **与 Join 的选择** | 大表 Join 小表 → 广播；大表 Join 大表 → Shuffle Hash Join 或 Sort Merge Join |
| **清理时机** | 任务结束后广播变量不会自动释放，需显式调用 `broadcastVar.unpersist()` |

**来源：** [Spark 官方文档 - Broadcast Variables](https://spark.apache.org/docs/latest/rdd-programming-guide.html#broadcast-variables)
