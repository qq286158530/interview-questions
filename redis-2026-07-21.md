## 题目一：Redis 和 Memcached 有什么区别？为什么 Redis 更适合做缓存？

### 参考答案

**共同点：**
- 都是基于内存的数据库，通常用来做缓存使用
- 都有过期策略
- 两者性能都非常高

**核心区别：**

| 对比维度 | Redis | Memcached |
|---------|-------|-----------|
| 数据类型 | 支持 String、Hash、List、Set、ZSet 等丰富类型 | 只支持简单的 key-value 类型 |
| 持久化 | 支持 RDB、AOF、混合持久化，可将数据保存到磁盘 | 无持久化功能，数据全部存内存，重启后丢失 |
| 集群 | 原生支持主从复制、哨兵模式、切片集群 | 无原生集群，需依赖客户端实现分片 |
| 高级特性 | 支持发布/订阅、Lua 脚本、事务 | 不支持 |

**为什么 Redis 更适合做缓存？**

1. **高性能**：Redis 读写操作直接操作内存，单机 QPS 可达 10W+，是 MySQL 的 10 倍以上；Memcached 虽也快，但数据类型单一，功能受限。
2. **高并发**：直接访问 Redis 承受的并发量远大于直接访问 MySQL，可承接大量请求，减轻数据库压力。
3. **数据安全性**：Redis 支持持久化，宕机后可恢复数据；Memcached 宕机数据全丢，风险高。
4. **功能丰富**：Redis 支持丰富的数据结构，可满足多样化的业务场景需求。

## 题目二：Redis 的过期删除策略是什么？惰性删除和定期删除各有什么优缺点？

### 参考答案

Redis 同时使用**惰性删除**和**定期删除**两种策略配合工作，以平衡 CPU 时间和内存占用。

### 惰性删除（Passive/Lazy Expiration）

**原理：** 不主动删除过期键，每次访问 key 时才检查是否过期——若过期则删除，否则正常返回。

**优点：**
- 每次访问才检查，对 CPU 时间最友好，系统资源消耗极少
- 实现简单，不需要额外的定时扫描任务

**缺点：**
- 如果过期键一直未被访问，则该键永不被删除，**占用的内存不会被释放**，造成内存浪费
- 对内存不友好

### 定期删除（Active Expiration）

**原理：** 每隔一段时间，随机从数据库中取出一定数量的 key 检查，删除其中已过期的 key。

**具体流程：**
1. 从过期字典中随机抽取 20 个 key
2. 检查这 20 个 key 是否过期，删除已过期的
3. 如果已过期 key 超过 5 个（>25%），继续重复步骤 1
4. 如果已过期 key 比例小于 25%，停止删除，等待下一轮

**优点：**
- 通过限制删除操作执行时长和频率，减少对 CPU 的影响
- 能主动删除一部分过期数据，减少过期键对空间的无效占用

**缺点：**
- 难以确定执行时长和频率——执行太频繁浪费 CPU，太稀疏则过期 key 堆积
- 定期删除循环时间上限默认不超过 25ms，防止卡死

### 两者配合

惰性删除对 CPU 友好但对内存不友好，定期删除对内存友好但对 CPU 不友好。Redis 两者结合使用，取长补短，在 **CPU 时间和内存空间** 之间取得平衡。

## 题目三：Redis 持久化有哪些方式？RDB 和 AOF 各自优缺点是什么？混合持久化有何优势？

### 参考答案

### 三种持久化方式

1. **RDB（Redis Database）**：将某一时刻的内存数据以二进制格式写入磁盘
2. **AOF（Append Only File）**：每执行一条写操作命令，以追加方式写入日志文件
3. **混合持久化**：Redis 4.0 新增，结合 RDB 和 AOF 的优点

### RDB 详解

**触发方式：**
- 手动：`save`（主线程执行，会阻塞）和 `bgsave`（子进程执行，不阻塞）
- 自动：通过 `save` 配置项（如 `900 1`、`300 10`、`60 10000`），满足任一条件则自动执行 `bgsave`

**核心机制——写时复制（COW）：**
- 执行 `bgsave` 时，父子进程共享同一片物理内存
- 父进程执行写操作时，被修改的数据复制出一份副本，bgsave 子进程将副本写入 RDB 文件
- 此过程中，父进程仍可正常修改原数据

**优点：** 恢复速度快（全量二进制，直接读入内存）
**缺点：** 快照是全量快照，频率难以把握——太频繁影响性能，太低导致数据丢失多

### AOF 详解

**写回策略（appendfsync）：**

| 策略 | 含义 | 优点 | 缺点 |
|------|------|------|------|
| Always | 每次写命令执行完后同步写回硬盘 | 数据完全不丢失 | I/O 开销大，性能差 |
| Everysec（默认） | 每秒将缓冲区内容写回硬盘 | 性能适中，最多丢 1 秒数据 | 可能丢 1 秒数据 |
| No | 由操作系统决定何时写回 | 性能最好 | 数据丢失不可控 |

**AOF 重写机制：**
- 当 AOF 文件过大时，后台子进程 `bgrewriteaof` 读取数据库所有键值对，用一条命令记录到新的 AOF 文件，实现压缩
- 重写期间主进程接收的新命令同时写入 "AOF 缓冲区" 和 "AOF 重写缓冲区"，保证数据一致性

**优点：** 丢失数据少；**缺点：** 恢复速度慢（需逐一执行命令）

### 混合持久化

**工作原理：** AOF 重写时，先将以 RDB 格式的全量数据写入 AOF 文件，再将重写期间的增量命令以 AOF 格式追加其后。

**优点：**
- 开头为 RDB 内容，Redis 重启时加载速度极快
- 后半部分为 AOF 增量数据，丢失数据少
- 兼顾了 RDB 的快速恢复和 AOF 的数据完整性

**缺点：** AOF 文件可读性差；兼容性差，Redis 4.0 之前版本无法使用。

## 题目四：Redis 是单线程吗？为什么单线程的 Redis 还能这么快？

### 参考答案

### Redis 确实是"单线程"

Redis 单线程指的是：**接收客户端请求 → 解析请求 → 进行数据读写等操作 → 发送数据给客户端** 这一核心过程由**一个主线程**完成。

但 Redis 程序本身**并非单线程**，它在启动时会启动多个后台线程（BIO）：
- **Redis 2.6**：2 个后台线程——关闭文件、AOF 刷盘
- **Redis 4.0+**：新增 lazy free 线程，异步释放内存（如 `unlink`、`flushdb async`）
- **Redis 6.0+**：额外启动 3 个 I/O 线程分担网络 I/O 压力

### Redis 为什么这么快？

#### 1. 内存操作，CPU 非瓶颈
Redis 大部分操作在内存中完成，采用高效数据结构（SDS、跳表、压缩列表等），性能瓶颈往往是**内存大小或网络带宽**，而非 CPU，因此单线程足以应对。

#### 2. 避免锁竞争和线程切换开销
多线程模型虽能利用多核 CPU，但会引入：
- 线程切换带来的性能损耗
- 加锁解锁的死锁风险
- 并发读写的竞态问题

单线程天然**无锁**，也**无线程切换开销**，代码逻辑更简单，可维护性更高。

#### 3. I/O 多路复用机制
Redis 使用 `epoll`（Linux）/ `kqueue`（macOS）等 I/O 多路复用机制，**一个线程管理多个 Socket 连接**：
- 内核同时监听多个 Socket 的连接请求或数据请求
- 有请求到达时才交给 Redis 线程处理
- 实现了一个线程高效处理大量并发连接的效果

### Redis 6.0 之后引入的多线程

Redis 6.0 对**网络 I/O** 引入多线程处理（默认仅针对 write client socket），但**命令执行仍是单线程**。官方表示此特性对性能提升至少 1 倍以上。线程数建议：4 核 CPU 设 2~3 个，8 核 CPU 设 6 个。

## 题目五：Redis 如何实现服务高可用？哨兵模式和集群切片模式有什么区别？

### 参考答案

### Redis 高可用方案概述

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| 主从复制 | 一主多从，读写分离 | 数据容灾、读写分离 |
| 哨兵模式 | 主从复制 + 自动故障转移 | 自动化高可用 |
| 切片集群（Redis Cluster） | 数据分片到多个节点 | 海量数据、高并发 |

### 1. 主从复制

- **写操作在主节点**，自动同步到所有从节点
- 从节点默认**只读**接收主节点的命令复制
- 主从之间命令复制是**异步**的——主节点执行完命令后立即返回客户端，不等待从节点执行完毕
- **无法保证强一致性**，数据不一致难以避免

### 2. 哨兵模式（Sentinel）

**解决的问题：** 主从架构下主节点宕机时，需要手动恢复的麻烦。

**核心功能：**
- **监控**：哨兵持续检查主从节点是否正常运行
- **自动故障转移**：主节点失效时，哨兵自动从从节点中选举出新主节点，并更新其他从节点的复制目标
- **通知**：将故障转移结果通知客户端

**原理简述：**
- 哨兵自身也由多个节点组成（哨兵集群），避免单点故障
- 通过 `INFO` 命令周期性获取主从节点状态
- 主观下线（单个哨兵检测失败）→ 客观下线（哨兵集群投票达成共识）→ 选举领头的哨兵 → 从从节点中选新主

### 3. 切片集群（Redis Cluster）

**解决的问题：** 单台 Redis 内存无法承载全部数据的场景。

**核心机制：**
- 整个集群共有 **16384 个哈希槽（Hash Slot）**
- 每个键值对根据 key 通过 `CRC16` 算法计算哈希值，再对 16384 取模，映射到对应哈希槽
- 哈希槽分配到集群的多个节点上存储

**数据分布方式：**
- **平均分配**：Redis 自动将 16384 个槽均匀分布到各节点
- **手动分配**：通过 `cluster meet` 和 `cluster addslots` 命令手动指定每个节点的槽

**优点：**
- 数据分布在多台服务器，降低单节点内存压力
- 支持横向扩展，理论上无上限
- 每个节点都可处理读写请求，**高并发**

### 脑裂问题及解决方案

**什么是脑裂？** 主节点网络短暂断开，哨兵未及时感知，选举出新主节点后出现两个主节点。旧主节点恢复后降级为从节点，清空本地数据做全量同步，导致之前写入的数据丢失。

**解决方案（配置两个参数）：**
```
min-slaves-to-write 1      # 主节点必须至少有 1 个从节点连接
min-slaves-max-lag 10      # 主从复制延迟不能超过 10 秒
```
即：主节点连接的从节点中，至少有 1 个的 ACK 延迟 ≤ 10 秒，否则主节点拒绝接收写请求，防止假故障期间写入数据丢失。
