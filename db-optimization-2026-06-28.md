## 题目一：MySQL 索引失效的场景有哪些？如何避免？

### 答案

MySQL 索引失效的常见场景包括：

1. **使用函数或运算**
   ```sql
   -- 失效：WHERE YEAR(create_time) = 2026
   -- 正确：WHERE create_time >= '2026-01-01' AND create_time < '2027-01-01'
   ```

2. **隐式类型转换**
   ```sql
   -- id 为 VARCHAR 类型时
   -- 失效：WHERE id = 123  -- 整型被转成字符串
   -- 正确：WHERE id = '123'
   ```

3. **最左前缀原则不匹配**
   ```sql
   -- 索引为 (a, b, c)
   -- 失效：WHERE b = 1
   -- 失效：WHERE c = 1
   -- 生效：WHERE a = 1
   -- 生效：WHERE a = 1 AND b = 2
   ```

4. **LIKE 以通配符开头**
   ```sql
   -- 失效：WHERE name LIKE '%张%'
   -- 生效：WHERE name LIKE '张%'
   ```

5. **使用 OR 连接条件**
   ```sql
   -- 失效：WHERE name = '张三' OR age = 20（OR两侧任一字段无索引）
   -- 解决：拆分为 UNION 或给 age 建索引
   ```

6. **使用 NOT、!=、<> 操作符**
   - 可以通过覆盖索引或改写为 `> AND <` 组合条件来部分优化

**避免策略**：避免在 WHERE 子句中对字段进行运算；尽量使用覆盖索引；用 EXPLAIN 分析执行计划确认索引使用情况。

## 题目二：一条 SQL 查询很慢，如何进行性能分析和优化？

### 答案

排查步骤如下：

1. **使用 EXPLAIN 分析执行计划**
   ```sql
   EXPLAIN SELECT * FROM orders WHERE user_id = 100;
   ```
   关注：`type`（最好到 ref/range）、`key`（是否命中索引）、`rows`（扫描行数）、`Extra`（Using filesort/Using temporary 则需优化）。

2. **检查索引是否合理**
   - 确认 WHERE 条件字段有索引且符合最左前缀原则
   - 检查索引选择性（`SELECT COUNT(DISTINCT col)/COUNT(*) FROM table`）

3. **分析慢查询日志**
   ```sql
   -- 开启慢查询
   SET GLOBAL slow_query_log = 'ON';
   SET GLOBAL long_query_time = 1; -- 超过1秒记录
   ```
   使用 `mysqldumpslow` 或 `pt-query-digest` 分析日志。

4. **优化手段**
   - 添加合适索引（复合索引遵循最左前缀）
   - 避免 SELECT *，只查所需字段
   - 使用 JOIN 代替子查询（MySQL 5.7+ 优化器已较成熟，但需注意驱动表选择）
   - 分页优化：`WHERE id > last_id LIMIT 20` 代替 `OFFSET`
   - 读写分离：将主从复制用于读多写少场景

5. **业务层面**
   - 对大表加缓存（Redis）
   - 分库分表（按时间或用户ID水平拆分）
   - 定期 `OPTIMIZE TABLE` 整理表空间

## 题目三：MySQL InnoDB 与 MyISAM 存储引擎的区别？如何选型？

### 答案

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| 事务支持 | ✅ 支持 ACID 事务 | ❌ 不支持 |
| 行锁 | ✅ 支持行级锁，并发高 | ❌ 只支持表级锁 |
| 外键 | ✅ 支持外键约束 | ❌ 不支持 |
| 崩溃恢复 | ✅ 支持 MVCC + redo log | ❌ 需手动修复 |
| 全文索引 | ✅ 5.6+ 支持 | ✅ 原生支持 |
| 存储空间 | 同等数据量下占用更大 | 约 InnoDB 的 1/2~1/3 |
| 计数查询 | COUNT(*) 全表扫描 | 有专门计数器，快速 |

**选型建议**：

- **选用 InnoDB**：生产环境几乎必选。需要事务、行锁、外键、崩溃恢复能力；并发写入多的场景；主从架构中作为主库。

- **选用 MyISAM**：只读或极少修改的日志表；需要全文检索且版本 < 5.6；表数据量小且不在意事务和并发。

> **当前主流实践**：MySQL 8.0+ InnoDB 已是默认引擎，MyISAM 基本退出主流场景。

## 题目四：什么是数据库连接池？为什么要用连接池？常见连接池有哪些？

### 答案

**连接池（Connection Pool）** 是预先建立一批数据库连接并维护在内存中的技术。应用程序需要访问数据库时，从池中获取空闲连接，用完后归还而非关闭，避免重复创建/销毁连接的开销。

**为什么需要连接池**：
- 数据库连接是 TCP socket，建立需要 TCP 三次握手 + 认证，耗时 10~100ms
- 频繁创建销毁连接会产生大量短连接，导致 `too many connections` 错误
- 连接池将连接复用，把连接建立成本均摊到请求级别

**常见连接池**：
| 语言/框架 | 连接池 |
|-----------|--------|
| Java | Druid、HikariCP（性能最优）、c3p0 |
| Python | SQLAlchemy（内置连接池）、DBUtils |
| Go | database/sql（标准库内置）、GORM |
| Node.js | node-mysql2（内置）、Prisma |

**配置关键参数**：
- `最大连接数`（max_connections）：根据数据库服务器配置和并发量设定
- `最小空闲连接`：预热建立，保证响应时间
- `连接超时`：防止等待过久
- `连接生命周期`：定期检测连接活性（心跳检测），避免死连接

## 题目五：Redis 缓存与数据库一致性问题如何解决？有哪些策略？

### 答案

缓存与数据库双写时存在一致性问题，常见解决方案如下：

### 策略一：Cache Aside（最常用）
- **读**：先读缓存，缓存未命中则读数据库并写入缓存
- **写**：先更新数据库，再删除缓存（而非更新缓存，避免并发问题）

```
Read: Cache → Miss → DB → Write Cache
Write: Update DB → Delete Cache
```

### 策略二：Read Through / Write Through
- 由缓存中间件负责读写数据库，应用代码只操作缓存
- 实现复杂，一般不推荐

### 策略三：延迟双删（解决并发问题）
```python
# 线程A：更新数据库
update_db()
delete_cache()  # 第一次删除

# 线程B：读取（可能把旧数据写入缓存）
time.sleep(0.5)  # 等待线程B读完并写缓存
delete_cache()  # 第二次删除，确保旧缓存被清除
```

### 策略四：基于 MQ 的最终一致性
- 更新数据库 → 发送消息到 MQ → 消费者更新/删除缓存
- 保证最终一致，但不保证实时性

### 一致性级别对比

| 策略 | 实时一致性 | 复杂度 | 适用场景 |
|------|-----------|--------|----------|
| Cache Aside | 准实时 | 低 | 大多数业务场景 |
| 延迟双删 | 准实时 | 中 | 对一致性要求较高的写多场景 |
| MQ 最终一致 | 最终一致 | 高 | 高并发写入，允许短暂不一致 |

> **建议**：对绝大多数场景采用 **Cache Aside + 设置过期时间**（如 TTL=5分钟）即可满足需求，兼顾性能与维护成本。
