## 题目1：HBase 的整体架构是怎样的？RegionServer 和 Region 的关系是什么？

### 参考答案

HBase 采用经典的 **Master-Slave架构**，核心组件如下：

| 组件 | 职责 |
|------|------|
| **ZooKeeper** | 做分布式协调，负责选举 Master、存储 meta 表位置、维护集群元数据 |
| **HMaster** | 监控所有 RegionServer，负责表的 CRUD、Region负载均衡、故障转移 |
| **RegionServer** | 直接管理数据读写的进程，一个 RS 负责多个 Region |
| **Region** | HBase 表的水平分片，是数据调度的最小单位 |

**Region 分配流程：**
1. 客户端连接 ZooKeeper，获取 `hbase:meta` 表（记录所有 Region 的位置）的位置；
2. 从 `hbase:meta` 查询目标 RowKey 落在哪个 Region；
3. 直连对应 RegionServer 进行读写；
4. Region 过大（超过阈值）会触发**分裂（Split）**，一分为二。

**写入流程：**
```
Client → ZooKeeper → Meta → RegionServer
       → MemStore（内存）→ WAL（HLog）→ Flush成HFile → HDFS
```

> **来源参考：** [HBase 官方文档 - Architecture](https://hbase.apache.org/book.html#architecture)

## 题目 2：HBase 中 RowKey 设计的原则是什么？如何避免热点问题？

### 参考答案

RowKey 是 HBase 中数据的**唯一主键**，设计好坏直接影响读写性能和集群均衡。

### 核心设计原则

1. **长度原则**：RowKey 不要过长，建议控制在 10～100 字节。过长的 RowKey 会占用更多 MemStore 和 HFile 存储空间。
2. **散列原则**：让数据均匀分散到不同 RegionServer，避免所有请求集中到某几台节点（**热点问题**）。
3. **唯一性原则**：唯一标识一行数据。
4. **业务导向原则**：将查询条件中最频繁使用的字段放在 RowKey 的前面。

### 避免热点的常用技巧

| 技巧 | 说明 |
|------|------|
| **加盐（Salting）** | 在 RowKey 头部添加随机前缀（如 `hash(key) % N`），强制打散数据到不同 Region |
| **Hash散列** | 对长字符串（如手机号、订单号）做 MD5 或其他 Hash，取前几位或完整值作为 RowKey |
| **时间戳反转** | 将时间戳用 `Long.MAX_VALUE - timestamp`，这样新数据排在前面，便于获取最新记录 |
| **随机字符串** | 使用 UUID 替换自然键，避免连续写入热点 |
| **业务字段组合** | 将多个业务字段拼接（如 `cityId + userId + timestamp`），同时满足多维度查询 |

**热点示例：**
- 反面：将时间戳直接作为 RowKey（如 `20260607120000`），大量请求会集中在某个时间段的 Region 上。
- 正面：`md5(phone)[0:2] + phone + reverse(ts)` 避免写入集中。

> **来源参考：** [HBase 官方文档 - RowKey Design](https://hbase.apache.org/book.html#rowkey.design)

## 题目 3：HBase 的存储结构是什么？HFile 和 MemStore 的作用是什么？

### 参考答案

HBase 采用** LSM-Tree（Log-Structured Merge-Tree）** 结构，写入流程如下：

```
① 写入请求 → ② 写入 WAL（HLog） → ③ 写入 MemStore → ④ 异步 Flush 成 HFile
```

### MemStore

- **内存缓冲区**：数据先写入 MemStore（每个 Column Family独立一个），满足内存写带来极快的写入速度。
- **排序**：写入 MemStore 前会按 RowKey 字典序排序，Flush 时直接生成有序 HFile。
- **阈值触发**：当 MemStore 达到 `hbase.hregion.memstore.flush.size`（默认 128MB）时，触发 Flush 生成新的 HFile。

### HFile（HBase File）

- **持久化的有序文件**：存储在 HDFS 上，文件内部数据按 RowKey 排序。
- **结构**：
  - `Data Block`：存储 KV 数据，默认为 64KB，可配置。
  - `Meta Block`：存储 bloomfilter，快速判断某 RowKey 是否存在。
  - `Index Block`：构建 Block 索引，支持快速定位。
  - `Trailer`：存储文件元数据（magic、字段数等）。
- **合并（Compaction）**：随着时间推移，小 HFile 会被合并成大 HFile，减少读取时的文件数量，提升查询性能。

###读取流程

```
读请求 → BlockCache（热点数据缓存）→ MemStore → HFile（多文件合并后返回）
```

> **来源参考：** [HBase 官方文档 - Storage](https://hbase.apache.org/book.html#regions.arch)

## 题目 4：HBase 的 WAL（Write-Ahead Log）机制是什么？为什么需要 WAL？

### 参考答案

### WAL 定义

WAL（Write-Ahead Logging）是 HBase 用于**保证数据持久性**的机制。所有写入操作在写入 MemStore 之前，**必须先写入 WAL 日志**，然后才能返回客户端写入成功。

### 为什么需要 WAL

HBase 写入流程是：内存（MemStore）→ Flush → HFile。如果数据还在 MemStore 时节点突然宕机，这部分数据就会丢失。**WAL 通过将操作日志先持久化到磁盘，确保即使内存数据丢失，也能从 WAL恢复。**

### 工作机制

- 每个 RegionServer 维护**共享的 WAL 文件**（实际存储在 HDFS 上），包含该 RS 上所有 Region 的操作记录。
- 当 MemStore 数据 Flush 到 HFile 成功后，对应的 WAL 日志可以被归档删除。
- **故障恢复**：RS 重启时，Master 会分配新的 RS 并重放 WAL，将未 Flush 的数据重新写入 MemStore。

### 关键参数

| 参数 | 说明 |
|------|------|
| `hbase WAL enabled` | 是否开启 WAL（默认 true，关闭会丢数据） |
| `hbase.wal.streamimplementation` | WAL 实现类 |
| `hbase.wal.provider` | WAL 提供者 |

> **来源参考：** [HBase 官方文档 - WAL](https://hbase.apache.org/book.html#wal)

## 题目 5：HBase 协处理器（Coprocessor）是什么？有哪些使用场景？

### 参考答案

### 协处理器是什么

HBase 协处理器是一种**运行在 RegionServer 上的扩展机制**，允许用户将自己的代码逻辑注入到 HBase 的读写路径中，无需修改 HBase 源码。

### 两种类型

| 类型 | 说明 |
|------|------|
| **Observer（观察者）** | 类似于数据库触发器，在特定事件前后自动执行回调（`preGet`、`postPut` 等） |
| **Endpoint（终端）** | 类似于存储过程，允许在 RegionServer 上执行自定义的聚合/计算逻辑，支持动态扩展 |

### Observer 使用场景

- **二级索引**：在 `postPut` 时将数据同步写入索引表，实现按非主键字段查询。
- **列值验证**：在 `prePut` 中检查数据合法性，不合法则抛出异常拒绝写入。
- **审计日志**：在 `preGet`/`postGet` 中记录所有读取操作到审计表。
- **动态过滤**：在读取时动态过滤列或行。

### Endpoint 使用场景

- `postScan` 回调：在每个 Region 上执行聚合计算（如 Sum、Count），再合并返回客户端。
- 跨行事务：实现更复杂的多行原子操作。

### 配置方式

1. **加载方式一（表级别）**：在表创建时指定
   ```java
   HTableDescriptor desc = new HTableDescriptor("myTable");
   desc.setValue("COPROCESSOR$1", "path/to.jar|com.example.ObserverClass|priority");
   ```

2. **加载方式二（HDFS路径）**：通过 HDFS 加载协处理器 jar 包。

> **来源参考：** [HBase 官方文档 - Coprocessors](https://hbase.apache.org/book.html#cp)
