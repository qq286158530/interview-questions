## 题目 1：Redis 的 5 种主要数据类型？它们各自的底层实现原理是什么？

### 参考答案

Redis 是典型的键值对（Key-Value）数据库，支持 5 种核心数据类型：

| 数据类型 | 底层实现 | 典型使用场景 |
|---------|---------|-------------|
| **String** | SDS（Simple Dynamic String） | 缓存、计数器、分布式锁 |
| **List** | QuickList（3.2 之前是 ZipList + LinkedList） | 消息队列、排行榜 |
| **Hash** | Listpack（2.6 之前是 ZipList，当字段少时）或 HashTable | 对象存储、购物车 |
| **Set** | IntSet（纯整数且小于阈值时）或 HashTable | 标签、好友关系、去重 |
| **Sorted Set（ZSet）** | Listpack + 跳表（SkipList） | 排行榜、有序集合 |

### 详细原理

#### String（SDS）
- 预分配内存，支持惰性释放，避免频繁 `malloc`/`free`
- 记录长度，可 O(1) 获取字符串长度
- 二进制安全，支持包含 `\0` 的数据

#### Hash
- 字段数 < `hash-max-ziplist-entries`（默认 512）且每个字段值 < `hash-max-ziplist-value`（默认 64 字节）时，用 Listpack 编码（内存紧凑）
- 超过阈值自动转换为 HashTable（渐进式 `rehash`）

#### ZSet
- Score 相同时按字典序排列
- 跳表层数 = `MAX(1, N/64)`，查找复杂度 O(log N)
- 与 HashTable 结合，同时维护「成员→分数」映射

## 题目 2：Redis 的持久化机制有哪些？RDB 和 AOF 的区别是什么？如何选择？

### 参考答案

Redis 提供两种持久化机制：

#### RDB（Redis Database Backup）
- 定时将内存中的**全量数据**快照写入磁盘（`dump.rdb`）
- 触发方式：`bgsave`（fork 子进程）、`save`（阻塞）、自动触发（`save m n` 配置）
- 优点：恢复快（直接加载 rdb 文件），fork 的子进程不影响父进程性能
- 缺点：最后一次快照后的数据会丢失

#### AOF（Append Only File）
- 记录所有**写命令**到日志文件，只追加不修改
- 三种同步策略：
  - `always`：每条命令都同步（最多丢失一条）
  - `everysec`（默认）：每秒同步，最多丢失 1 秒数据
  - `no`：由操作系统决定何时同步，最多丢失不可控
- `bgrewriteaof`：压缩 AOF 文件（去除冗余命令）
- 优点：数据安全性更高
- 缺点：AOF 文件比 RDB 大，恢复慢

#### 混合持久化（Redis 4.0+）
- 开启 `aof-use-rdb-preamble yes`
- AOF 重写时，先写 RDB 快照，再追加增量 AOF 命令
- 兼具快速恢复和增量安全

#### 如何选择？

| 场景 | 推荐策略 |
|------|---------|
| 允许分钟级数据丢失 | RDB 单独使用 |
| 对数据安全性要求高 | AOF everysec + RDB 定期备份 |
| 追求最优性能 + 可接受秒级丢失 | AOF everysec |
| 最高安全要求 | AOF always + RDB 定期备份 |

## 题目 3：Redis 的过期 key 删除策略有哪几种？什么是惰性删除和定时删除？

### 参考答案

Redis 采用**两种策略结合**的方式清理过期 key：

#### 惰性删除（Lazy Expire）
- **原理**：每次访问 key 时，检查是否过期，过期则删除
- **优点**：对 CPU 友好，只在访问时检查，不浪费 CPU 资源
- **缺点**：如果 key 永远不被访问，则永远不会被删除（内存泄漏风险）

#### 定时删除（Active Expiration）
- **原理**：Redis 每秒随机抽取一批设置了过期时间的 key，检查并删除已过期的
- 实现：`ACTIVE_EXPIRE_CYCLE_LOOKUP_PER_LOOP = 20`（每次最多检查 20 个 key 所在的 slot）
- 采用**稀疏扫描**：只检查部分 slot，避免阻塞主线程

#### 定期删除（Adaptive Expiration）
- Redis 在 serverCron 中（每 100ms 一次）遍历每个数据库，检查 `expires` 字典中的过期 key
- 注意：这里的「定期」和上面的「定时删除」是同一个机制的不同描述

#### 内存淘汰策略（当内存达到 maxmemory 时）

当内存不足以写入新数据时，Redis 触发**内存淘汰策略**：

| 策略 | 行为 |
|------|------|
| `noeviction` | 拒绝写入（默认） |
| `volatile-lru` | 从已设置过期时间的 key 中淘汰最近最少使用 |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用 |
| `volatile-lfu` | 从已设置过期时间的 key 中淘汰使用频率最低 |
| `allkeys-lfu` | 从所有 key 中淘汰使用频率最低 |
| `volatile-random` | 从已设置过期时间的 key 中随机淘汰 |
| `allkeys-random` | 从所有 key 中随机淘汰 |
| `volatile-ttl` | 淘汰最近要过期的 key |

> **实战建议**：通常使用 `allkeys-lru` 或 `volatile-lru`，适合作为缓存场景。

## 题目 4：Redis 主从复制原理是什么？什么是哨兵（Sentinel）和集群（Cluster）？

### 参考答案

#### 主从复制原理

```
1. 主节点（master）执行写操作
2. 将写命令记录到 ring buffer（replication buffer）
3. 从节点（slave）发送 PSYNC 命令（增量）或 FULLSYNC（首次全量）
4. 主节点 fork 子进程执行 BGSAVE，生成 RDB 快照
5. 主节点将 RDB 文件发送给从节点
6. 从节点加载 RDB 到内存
7. 之后主节点通过 push 方式将增量命令发送给从节点
```

关键配置：`replica-read-only yes`（从节点默认只读）

#### 哨兵（Sentinel）模式

哨兵是 Redis 的**高可用（HA）解决方案**：

- 监控主从节点健康状态（`PING` 机制）
- 主观下线（SDOWN）和客观下线（ODOWN）判定
- 自动故障转移：当主节点不可达，哨兵通过投票（Raft 协议）选出一个从节点升为主节点
- 通知客户端：客户端订阅哨兵频道获取新的主节点地址

```
应用端 ──→ 哨兵集群（3个以上） ──→ Redis 主从集群
          ↓
      自动故障转移
```

#### 集群（Cluster）模式

Redis Cluster 是**分布式数据存储方案**：

- **数据分片**：16384 个 slot（槽），每个节点负责一部分 slot
- **Gossip 协议**：节点间通过 Gossip 协议传播集群状态信息
- **MOVED 重定向**：客户端请求到错误节点时，返回 MOVED 错误，客户端自动转向正确节点
- **ASK 重定向**：正在迁移的 slot 分片内的 key，客户端需要先发 ASK 命令
- **failover**：主节点宕机后，从节点自动发起选举，获得多数投票后升为主节点
- **去中心化**：无中心Proxy，每个节点都知道全局拓扑

> **核心区别**：哨兵解决的是高可用（主节点故障自动切换），集群解决的是数据分片和水平扩展。

## 题目 5：什么是缓存穿透、缓存击穿、缓存雪崩？如何应对？

### 参考答案

这是 Redis / 缓存系统最经典的高频面试题。

#### 缓存穿透（Cache Miss Penetration）

- **问题**：查询一个**不存在**的 key（数据库中也没有），每次请求都穿透到数据库
- **危害**：大量不存在数据的请求打垮数据库
- **解决方案**：
  1. **布隆过滤器（Bloom Filter）**：在缓存层前加一个 Bloom Filter，判定 key 是否可能存在
  2. **缓存空值**：将查询结果为空也缓存起来（设置较短 TTL，如 60 秒）
  3. **参数校验**：对查询参数做基础校验，过滤明显恶意的 ID

#### 缓存击穿（Cache Breakdown）

- **问题**：某个**热点 key 过期**的瞬间，大量并发请求同时涌入，直接打向数据库
- **特点**：并发量集中在一个 key 上，与穿透不同（穿透是 key 根本不存在）
- **解决方案**：
  1. **互斥锁（分布式锁）**：只允许一个请求去查询数据库并回填缓存，其他等待
  2. **热点 key 永不过期**：物理不过期 + 逻辑过期（用额外字段记录过期时间）
  3. **提前重建**：主动在 key 即将过期前异步重建缓存

#### 缓存雪崩（Cache Avalanche）

- **问题**：大量 key **同时过期**，导致大量请求同时打到数据库
- **解决方案**：
  1. **过期时间随机化**：在 TTL 基础上加随机偏移量，避免同时过期
  2. **多级缓存**：L1（本地缓存）+ L2（Redis）+ 数据库
  3. **熔断/限流**：数据库访问达到阈值时拒绝部分请求
  4. **高可用架构**：Redis Cluster + 数据库水平扩展

#### 三者对比

| 类型 | 原因 | 应对重点 |
|------|------|---------|
| 穿透 | key 不存在 | 布隆过滤器 / 缓存空值 |
| 击穿 | 热点 key 过期 | 互斥锁 / 永不过期 |
| 雪崩 | 大量 key 同时过期 | 随机 TTL / 多级缓存 |
