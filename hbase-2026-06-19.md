## 题目一：HBase 的整体架构是怎样的？请详细说明 HMaster 和 RegionServer 的职责。

### 参考答案

HBase 采用 **Master-Slave** 架构，主要包含以下组件：

#### 1. HMaster（主节点）
- **职责**：负责表的 CRUD 操作（创建、删除、修改表结构）
- 管理 RegionServer 的负载均衡，调度 Region 分布
- 处理 schema 变更请求
- 监控 RegionServer 存活状态，当 RegionServer 宕机时重新分配其上的 Region
- 在 ZooKeeper 中进行监控和协调

> 注意：HMaster 是轻量级进程，不参与实际的数据读写，真正的数据读写发生在 RegionServer 上。

#### 2. RegionServer（数据节点）
- **职责**：负责数据的读写操作
- 管理 Region（一个 Region 包含多个 Store，每个 Store 对应一个列族）
- 写入流程：先写 WAL（Write-Ahead Log），再写 MemStore，满足条件后刷写成 HFile
- 读取流程：先查 MemStore，未命中则查 BlockCache，最后查 HFile
- 与 HMaster 通信，报告负载和状态

#### 3. ZooKeeper
- 存储 HBase 的元数据信息（ROOT 表位置、META 表位置）
- 监控 RegionServer 的心跳
- 实现 HMaster 的高可用（主备切换）

#### 4. 数据寻址流程
```
Client → ZooKeeper → ROOT 表 → META 表 → RegionServer → Region → Store → MemStore/HFile
```

> **来源**：[HBase 官方文档](https://hbase.apache.org/book.html#architecture)

## 题目二：HBase 的 RowKey 设计原则有哪些？如何设计一个好的 RowKey？

### 参考答案

#### RowKey 设计原则

##### 1. **唯一性原则**
RowKey 必须唯一标识每一行数据，作为 HBase 表的主键。

##### 2. **长度适中原则**
- RowKey 最大支持 64KB，建议越短越好（10-100 字节）
- 原因：MemStore 刷写成 HFile 时，RowKey 会被存储多次
- 过长的 RowKey 会占用大量内存，影响存储和查询效率

##### 3. **散列原则（避免热点）**
- 避免使用时间戳或单调递增的字符串作为 RowKey（如 `20260619`、`000001`）
- 热点问题：所有请求集中到某个 RegionServer，导致负载不均
- **解决方案**：
  - 对 RowKey 进行 Hash 打散（如 MD5）
  - 拼接随机前缀
  - 盐值（Salt）策略：在 RowKey 前加随机数

##### 4. **业务相关性原则**
- 将查询条件中常用的字段放在 RowKey 前缀
- 利用 HBase 的字典序排序特性，将需要一起查询的数据放在一起

#### 经典设计方案

| 场景 | RowKey 设计 | 说明 |
|------|-------------|------|
| 用户订单查询 | `userId + reverse(orderId)` | 用户ID在前，支持按用户查订单；反转订单号打散热点 |
| 时间序列数据 | `deviceId + (Long.MAX - timestamp)` | 利用字典序倒排，最近数据排在前面 |
| 多条件查询 | `联合字段 + 分隔符` | 如 `areaId#cityId#shopId` |

#### 示例
```java
// 避免热点：使用 MD5 打散
String badRowKey = "user_123456";  // 单调递增，会热点

String goodRowKey = MD5(userId).substring(0, 6) + userId;
// 效果：6位随机前缀 + userId，实现负载均衡
```

> **来源**：[HBase RowKey 设计最佳实践](https://hbase.apache.org/book.html#rowkey.design)

## 题目三：HBase 的 MemStore 和 HFile 是什么？请详细说明数据刷写机制。

### 参考答案

#### 1. MemStore（写缓存）

- **位置**：RegionServer 的内存中
- **作用**：当数据写入 HBase 时，先写入 WAL，然后写入 MemStore（类似 MySQL 的 Buffer Pool）
- **特点**：
  - 按 RowKey 排序（LSM-Tree 结构）
  - 支持单列族、多版本
  - 达到阈值后刷写成 HFile

#### 2. HFile（存储文件）

- **位置**：HDFS 上（`/hbase/data/.../*.hfile`）
- **结构**：
  - Data Block：存储实际的 KeyValue 数据
  - Meta Block：存储元信息
  - File Info：文件级别元数据
  - Index Block：数据索引，加速查询
- **特点**：
  - 只写入不修改（ immutable）
  - 支持布隆过滤器（Bloom Filter）快速判断 Key 是否存在

#### 3. 数据刷写机制（Flush）

**触发条件**（满足任一即可）：
| 条件 | 默认值 | 说明 |
|------|--------|------|
| MemStore 大小 | `hbase.hregion.memstore.flush.size=128MB` | 单个 MemStore 达到阈值 |
| Region 级别 | `hbase.regionserver.global.memstore.size` | RegionServer 上所有 MemStore 总和达到堆内存比例 |
| 时间 | `hbase.regionserver.flush.interval=1h` | 达到指定时间间隔 |
| WAL 阻塞 | `hbase.regionserver.max.logs=32` | WAL 文件数量超过阈值 |

**刷写流程**：
```
MemStore 达到阈值 
  → 创建 snapshot 防止数据丢失
  → 将 MemStore 按 Key 排序输出为 HFile
  → 写入 HDFS
  → 删除旧的 WAL 日志
  → 清理 MemStore
```

#### 4. 合并机制（Compact）

**小合并（Minor Compaction）**：
- 将多个小的 HFile 合并成更大的 HFile
- 合并时删除已标记删除的数据

**大合并（Major Compaction）**：
- 将一个 Store 下所有 HFile 合并成一个
- 默认 7 天执行一次
- 会删除所有过期版本和已标记删除的数据

> **来源**：[HBase 数据刷写与合并机制](https://hbase.apache.org/book.html#regionserver.arch)

## 题目四：HBase 的存储结构是怎样的？请说明 Store、Region、StoreFile 的关系。

### 参考答案

#### HBase 存储层次结构

```
Table (表)
  └── Region (区域)
        └── Store (存储，按列族划分)
              ├── MemStore (内存缓存)
              └── StoreFile (HFile) × N (多个文件)
```

#### 1. Table（表）
- 由一个或多个 Region 组成
- 通过 `create 'table', 'cf1', 'cf2'` 创建，可指定多个列族

#### 2. Region（区域）
- **概念**：表在行方向上的分片
- 每个 Region 包含 `[startKey, endKey)` 范围内的数据
- 特点：
  - 按 RowKey 字典序连续分布
  - 一个表初始只有一个 Region，随着数据增长自动分裂
  - 分裂后两个子 Region 会被调度到不同的 RegionServer

#### 3. Store（存储）
- **概念**：按列族（Column Family）划分
- 每个 Region 下的每个列族对应一个 Store
- 包含一个 MemStore 和多个 StoreFile

#### 4. StoreFile / HFile
- HBase 的实际存储文件格式
- 存储在 HDFS 上，副本数为 3
- 内部采用 LSM-Tree（Log-Structured Merge-Tree）结构

#### 5. KeyValue 结构
HBase 中每行数据的每个单元格存储为 KeyValue 格式：
```
KeyValue = RowKey + ColumnFamily + ColumnQualifier + Timestamp + Value
```
特点：
- 包含完整的坐标信息（可独立寻址）
- 同一行同一列的多个版本通过时间戳区分

#### 数据写入流程
```
Client PUT 
  → ZooKeeper 定位 RegionServer 
  → 写入 WAL 
  → 写入 MemStore 
  → 返回 ACK
     ↓
 MemStore 刷写 → HFile
```

#### 数据读取流程
```
Client GET 
  → 定位 RegionServer 
  → 先查 MemStore（最新数据）
  → 未命中则查 BlockCache
  → 还未命中则查 HFile（使用 Bloom Filter 加速）
  → 合并结果返回
```

> **来源**：[HBase 存储结构文档](https://hbase.apache.org/book.html#datamodel)

## 题目五：HBase 协处理器（Coprocessor）是什么？有哪几种类型？有什么作用？

### 参考答案

#### 1. 什么是协处理器？

协处理器是 HBase 提供的 **扩展机制**，允许用户将自定义代码部署到 RegionServer 上运行，从而在 **数据层面** 实现自定义逻辑，而无需客户端参与。

#### 2. 协处理器类型

##### ① 观察者（Observer）- 触发器模式

类似于数据库的触发器，在特定事件发生时自动执行回调。

| 类型 | 触发时机 | 使用场景 |
|------|----------|----------|
| `RegionObserver` | GET/PUT/DELETE 前后 | 数据验证、过滤、修改返回值 |
| `WALObserver` | WAL 写入前后 | WAL 加密、审计 |
| `MasterObserver` | 表创建/删除/修改前后 | 权限控制、元数据同步 |

**示例：实现数据验证**
```java
public class DataValidatorObserver extends RegionObserver {
    @Override
    public void prePut(ObserverContext<RegionCoprocessorEnvironment> c, 
                       Put put, WALEdit edit, Durability durability) {
        // 在数据写入前验证，如：年龄必须为正数
        // 验证失败抛出 IOException 拒绝写入
    }
}
```

##### ② 端点（Endpoint）- 存储过程模式

类似于数据库的存储过程，在 RegionServer 上执行计算，减少数据传输。

- 使用 ProtoBuf 定义接口
- 在每个 Region 上执行，然后合并结果
- 常用于：聚合计算、批量扫描、自定义过滤

**示例：自定义聚合**
```java
// 定义接口
public interface SumEndpoint extends Coprocessor, CoprocessorService {
    // 在每个 Region 上计算局部和
    Service getService();
}
```

#### 3. 使用场景

| 场景 | 协处理器类型 | 说明 |
|------|--------------|------|
| 数据权限控制 | Observer | 写入/读取前检查权限 |
| 二级索引 | Observer + Endpoint | 数据变更时同步更新索引表 |
| 聚合查询 | Endpoint | 在 RegionServer 端聚合，减少网络传输 |
| 数据压缩/加密 | WALObserver | 在 WAL 层处理 |
| TTL 自动清理 | RegionObserver | 结合 TTL 实现更复杂的清理逻辑 |

#### 4. 加载方式

```xml
<!-- 方式1：hbase-site.xml 全局加载 -->
<property>
  <name>hbase.coprocessor.region.classes</name>
  <value>com.example.MyObserver,com.example.MyEndpoint</value>
</property>

<!-- 方式2：Shell 按表加载 -->
hbase> alter 'myTable', METHOD => 'table_att', 
      'COPROCESSOR' => 'hdfs://path/to/coprocessor.jar|com.example.Observer|参数'
```

#### 5. 注意事项

- 协处理器运行在 RegionServer 上，代码异常可能导致 RegionServer 宕机
- 避免执行耗时操作（如网络请求、外部文件 IO）
- 生产环境建议做好监控和异常捕获

> **来源**：[HBase 协处理器官方文档](https://hbase.apache.org/book.html#cp)
