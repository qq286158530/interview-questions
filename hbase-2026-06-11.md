## 题目一：HBase 的整体架构是怎样的？请描述 RegionServer、Region、HMaster 的作用及数据寻址流程。

### 参考答案

**架构组成**

| 组件 | 作用 |
|------|------|
| **HMaster** | 负责 Region 的分配、表的DDL操作（创建/删除/修改）、负载均衡、故障恢复。不参与实际数据读写。 |
| **RegionServer** | 负责实际的数据读写请求。每个 RegionServer 包含多个 Region，是 HBase 的工作节点。 |
| **ZooKeeper** | 负责选举主 HMaster、存储 Meta 表位置、心跳检测，保证集群高可用。 |
| **Region** | HBase 表的水平分片，每个 Region 存储一段行键区间内的数据，默认为 128MB。 |

**数据寻址流程（3步）**

1. **客户端连接 ZooKeeper**，获取 Meta 表所在的 RegionServer 地址。Meta 表（hbase:meta）记录了所有用户表 Region 的位置信息。
2. **客户端请求 Meta RegionServer**，查询目标行键所在的 Region 位置，然后客户端缓存该信息。
3. **客户端直接请求目标 RegionServer**，完成数据读写。以后请求直接命中缓存，不再查询 Meta。

**写入流程**：Client → ZooKeeper → Meta → 获取 Region 位置 → 直接写 MemStore → 刷写 HFile

## 题目二：HBase 的 RowKey 设计原则有哪些？如何避免热点问题？

### 参考答案

**RowKey 设计核心原则**

1. **长度原则**：RowKey 不要过长，建议控制在 16~100 字节内。RowKey 会长久存储在内存和 HFile 中，过长会浪费内存和磁盘。
2. **散列原则**：RowKey 的前缀要足够分散，避免数据集中在少数 RegionServer 上（热点问题）。
3. **唯一性原则**：RowKey 必须唯一标识一行数据，类似于主键。
4. **排序原则**：HBase 按 RowKey 字典序排序存储，设计时可利用此特性将经常一起读取的数据放在一起。

**避免热点的常见策略**

| 策略 | 说明 |
|------|------|
| **加盐（Salting）** | 在 RowKey 前面添加随机字节，使数据分散到不同 Region。缺点是读操作无法利用缓存。 |
| **哈希散列** | 对 RowKey 取 MD5 或其他 hash 值作为前缀，保证数据均匀分布。 |
| **翻转（Reversal）** | 反转固定格式的电话号码或时间戳，避免单调递增前缀导致的热点。 |
| **时间戳反转** | 如手机号 `138xxxx1234` 反转为 `4321xxxx831`，使单调递增的行键被打散。 |

**示例**：如果业务需要按时间查询且时间单调递增，直接用时间戳做 RowKey 会导致所有新数据集中在最右侧的 Region，造成写热点。正确做法是 `MD5(user_id) + timestamp` 或在时间戳前加随机前缀。

## 题目三：HBase 的 MemStore 刷写机制是什么？为什么需要先写 MemStore 而不是直接写 HFile？

### 参考答案

**MemStore 的作用**

HBase 采用 LSM-Tree（Log-Structured Merge-Tree）存储结构，写入数据时先写入 MemStore（内存），而不是直接落盘。这种设计带来以下优势：

1. **顺序写入**：内存写速度远快于随机磁盘写入，充分利用磁盘顺序 I/O。
2. **批量合并**：MemStore 累积一定量数据后批量刷写到 HFile，减少磁盘 I/O 次数。
3. **写路径短**：客户端写操作只需写入内存即可返回成功，降低延迟。

**刷写（Flush）触发条件**

| 条件 | 说明 |
|------|------|
| MemStore 大小超过阈值 | 单个 MemStore 默认达到 **128MB** 时触发刷写 |
| Region 的所有 MemStore 总大小 | 超过 `hbase.hregion.memstore.flush.size` (默认 128MB) × 列族数 |
| WAL 日志数量过多 | WAL 中待刷写的日志数量超过 `hbase.regionserver.max.logs`（默认 32）时，强制刷写释放日志 |
| 定时刷写 | `hbase.regionserver.flush.policy.interval` 定时检查是否需要刷写 |
| 手动命令 | 用户可通过 `flush` 命令主动触发刷写 |

**刷写流程**

```
MemStore 达到阈值 → 创建快照 → 新数据写入新 MemStore → 快照刷写为 HFile → 删除快照
```

**与 HFile 的关系**：MemStore 是数据写入 HFile 前的缓冲，HFile 是最终持久化的存储结构。多个 HFile 会在后台通过 Minor Compaction（小合并）合并成更大的 HFile，Major Compaction（大合并）会合并所有 HFile 并清理已删除数据。

## 题目四：HBase 的 HFile 存储结构是怎样的？为什么设计成这样？

### 参考答案

**HFile 结构（V2 版本）**

HFile 由多层索引构成，设计目标是支持高效的单行查询和范围扫描：

```
┌────────────────────────────────────────────┐
│  Data Block（数据块）                       │
│  ├─ 存储 KeyValue 数据                      │
│  ├─ 默认 64KB，可配置                       │
│  └─ 每个 Block 有独立的 Leaf Index Entry    │
├────────────────────────────────────────────┤
│  Leaf Index（叶索引）                        │
│  ├─ 索引 Data Block 的起始行键               │
│  └─ 存储在 Block 中                         │
├────────────────────────────────────────────┤
│  Intermediate Index（中间索引）              │
│  └─ 索引 Leaf Index Block                   │
├────────────────────────────────────────────┤
│  Root Index（根索引）                        │
│  └─ 指向 Intermediate Index                 │
├────────────────────────────────────────────┤
│  Bloom Filter（布隆过滤器）                  │
│  └─ 快速判断某 RowKey 是否存在于该 HFile     │
├────────────────────────────────────────────┤
│  Meta Block（元数据块）                      │
├────────────────────────────────────────────┤
│  File Info（文件信息）                       │
├────────────────────────────────────────────┤
│  Trailer（ trailer 指针块）                  │
│  └─ 指向各区域的偏移量，用于快速定位         │
└────────────────────────────────────────────┘
```

**KeyValue 结构**

HBase 中每行数据以 KeyValue 形式存储，结构为：
```
[KeyLength][ValueLength][RowKey][ColumnFamily:Qualifier][Timestamp][Value]
```
KeyValue 是变长的，没有固定结构，这样可以高效存储稀疏数据。

**设计原因**

| 特性 | 优势 |
|------|------|
| 多层索引 | 查询时先从 Root Index → Intermediate → Leaf Index → Data Block，只需加载少量索引即可定位数据 |
| Bloom Filter | 快速判断某 RowKey 是否在 HFile 中，避免不必要的磁盘扫描 |
| Block 大小（64KB） | 适中：索引数据量可控，同时保证顺序扫描的吞吐量 |
| Trailer | 存储各区域偏移量，HFile 打开时只需读取 Trailer 即可快速建立完整索引结构 |

**与 MemStore 的配合**：MemStore 数据刷写到 HFile 时会按 RowKey 排序，因此 HFile 本身也是有序的，支持二分查找和范围扫描。

## 题目五：HBase 协处理器（Coprocessor）是什么？Observer 和 Endpoint 的区别和使用场景是什么？

### 参考答案

**协处理器是什么**

协处理器是 HBase 提供的扩展机制，允许用户在 RegionServer 上运行自定义代码，在数据写入、读取、删除时"钩住"执行逻辑，实现类数据库存储过程的功能。

**两种协处理器类型对比**

| 类型 | 继承类 | 触发时机 | 典型场景 |
|------|--------|----------|----------|
| **Observer** | `BaseRegionObserver` | 钩子回调，不主动发起响应 | 数据验证、权限控制、二级索引、审计日志 |
| **Endpoint** | `CoprocessorProtocol` | 客户端调用触发 | 聚合计算（如 SUM/COUNT）、自定义压缩、跨 Region 数据处理 |

**Observer 示例（二级索引）**

```java
public class IndexObserver extends BaseRegionObserver {
    @Override
    public void postPut(ObserverContext e, Put put, WALEdit edit, boolean writeToWAL) throws IOException {
        // 在数据写入后自动更新索引表
        byte[] rowKey = put.getRow();
        // 解析数据，构建索引 RowKey，写入索引表
    }
}
```

**Endpoint 示例（跨 Region 聚合）**

```java
// 服务端：定义聚合协议
public class SumEndpoint implements CoprocessorProtocol {
    public long getSum(byte[] family, byte[] qualifier) { ... }
}

// 客户端：调用所有 Region 的 Endpoint 并汇总结果
AggregationClient client = new AggregationClient(conf);
long total = client.sum(TableName.valueOf("t"), new SumOperation());
```

**使用注意事项**

1. **加载方式**：可通过配置 `hbase.coprocessor.*` 在所有 Region 上加载（系统级），或通过 HBase Shell 在特定表上加载（表级）。
2. **性能影响**：协处理器运行在 RegionServer 上，代码质量差会直接影响集群读写性能。
3. **故障隔离**：建议使用独立的 HBase 集群或做好充分测试后再在生产环境部署。

**常见应用场景**：Phoenix 的二级索引、HBase 写入审计、Gaea 的数据验证中间件都基于协处理器实现。
