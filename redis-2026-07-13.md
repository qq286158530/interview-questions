## 题目一：Redis 的过期删除策略和内存淘汰策略有什么区别？

### 参考答案

**过期删除策略**是指当 Redis 中的某个 key 设置了过期时间时，Redis 如何删除已经过期的 key。

Redis 使用的过期删除策略是「**惰性删除 + 定期删除**」两种策略配合使用：

1. **惰性删除**：不主动删除过期键，每次从数据库访问 key 时，才检测 key 是否过期，如果过期则删除。
   - 优点：对 CPU 时间最友好，只会在访问时检查
   - 缺点：如果 key 一直不被访问，就会占用内存，造成内存浪费

2. **定期删除**：每隔一段时间，随机从数据库中取出一定数量的 key 进行检查，删除其中的过期 key。
   - 每轮随机抽取 20 个 key
   - 如果过期 key 超过 5 个（25%），则继续抽取删除
   - 循环时间上限默认 25ms，避免阻塞

**内存淘汰策略**是指当 Redis 运行内存达到 `maxmemory` 阈值时，Redis 如何选择删除哪些 key 来释放空间。

内存淘汰策略共有八种：

| 策略 | 说明 |
|------|------|
| noeviction | 不淘汰任何数据，拒绝写入（默认） |
| volatile-random | 随机淘汰设置了过期时间的任意 key |
| volatile-ttl | 优先淘汰更早过期的 key |
| volatile-lru | 淘汰所有设置了过期时间的 key 中最久未使用的 |
| volatile-lfu | 淘汰所有设置了过期时间的 key 中最少使用的 |
| allkeys-random | 随机淘汰任意 key |
| allkeys-lru | 淘汰整个键值中最久未使用的 |
| allkeys-lfu | 淘汰整个键值中最少使用的 |

**两者的区别**：
- 过期删除策略是针对「已过期但尚未删除」的 key，是在访问时或定期检查时被动执行
- 内存淘汰策略是针对「内存不足时需要释放空间」，是主动执行

## 题目二：什么是缓存雪崩、击穿、穿透？如何解决？

### 参考答案

### 缓存雪崩

**定义**：大量缓存数据在同一时间过期（失效），导致大量请求直接访问数据库，造成数据库压力骤增甚至宕机。

**解决方案**：
1. **将缓存失效时间随机打散**：在原有失效时间基础上增加一个随机值（1~10分钟），降低集体失效的概率
2. **设置缓存不过期**：通过后台服务更新缓存数据，避免因缓存失效造成的雪崩

### 缓存击穿

**定义**：某个热点数据过期（失效）时，大量并发请求同时访问该热点数据，无法从缓存读取，直接冲垮数据库。

**解决方案**：
1. **互斥锁方案**：使用 `setNX` 保证同一时间只有一个业务线程请求缓存
2. **热点数据不过期**：由后台异步更新缓存，或提前续约过期时间

### 缓存穿透

**定义**：请求的数据既不在缓存中，也不在数据库中（比如恶意攻击或业务误操作），导致请求直接打到数据库。

**解决方案**：
1. **非法请求限制**：在 API 入口处判断请求参数是否合理
2. **设置空值/默认值**：对不存在的数据在缓存中设置空值或默认值
3. **布隆过滤器**：在写入数据库时使用布隆过滤器标记，查询时快速判断数据是否存在

## 题目三：Redis 是单线程吗？为什么单线程还这么快？

### 参考答案

**Redis 单线程指的是**：接收客户端请求 → 解析请求 → 进行数据读写等操作 → 发送数据给客户端 这一过程由主线程完成。

**但 Redis 程序并不是单线程的**，Redis 启动时会启动后台线程（BIO）：
- Redis 2.6：2 个后台线程，处理关闭文件、AOF 刷盘
- Redis 4.0+：新增 lazyfree 线程，异步释放内存

**Redis 单线程为什么还这么快？**

1. **内存操作**：大部分操作都在内存中完成，采用高效的数据结构
2. **瓶颈不在 CPU**：Redis 瓶颈通常是内存大小或网络带宽，而非 CPU
3. **无竞争开销**：单线程避免了多线程切换、锁竞争、死锁等问题
4. **I/O 多路复用**：采用 `select/epoll` 机制，一个线程处理多个 IO 流

**Redis 6.0 引入多线程**：
- Redis 6.0 对网络 I/O 采用了多线程处理（默认 3 个 I/O 线程）
- 但命令执行仍然是单线程，所以 Redis 仍然是单线程处理命令

## 题目四：Redis 如何实现数据持久化？RDB 和 AOF 有什么区别？

### 参考答案

Redis 有三种数据持久化方式：

### RDB（Redis Database）快照

- 将某一时刻的内存数据以**二进制**方式写入磁盘
- 恢复数据时直接读取 RDB 文件，速度快
- 命令：`save`（主线程执行，会阻塞）、`bgsave`（子进程执行，不阻塞）
- 触发机制：按配置自动触发（900秒内1次修改/300秒内10次修改/60秒内10000次修改）

**原理**：使用 Copy-On-Write（写时复制），fork 子进程共享父进程内存，父进程修改数据时复制副本给子进程写入 RDB。

### AOF（Append Only File）日志

- 每执行一条写操作命令，就以**追加方式**写入到日志文件
- 恢复时逐条执行命令，速度慢但数据更完整

**三种写回策略**：
| 策略 | 说明 | 安全性 | 性能 |
|------|------|--------|------|
| Always | 每次写命令执行完后同步写回硬盘 | 最安全 | 最低 |
| Everysec | 每秒将缓冲区内容写回硬盘 | 较快 | 较高 |
| No | 由操作系统控制写回时机 | 最差 | 最高 |

**AOF 重写机制**：当 AOF 文件过大时，后台子进程 `bgrewriteaof` 会读取数据库中的所有键值对，用一条命令记录到新的 AOF 文件，压缩文件体积。

### 混合持久化（Redis 4.0+）

- RDB 格式的全量数据 + AOF 格式的增量数据
- 开头为 RDB 格式，加载速度快
- 后续为 AOF 格式，保证数据完整性

### RDB vs AOF 对比

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件体积 | 小（二进制） | 大（文本命令） |
| 恢复速度 | 快 | 慢 |
| 数据完整性 | 可能丢失较多数据 | 可配置（always 最完整） |
| 优先级 | 可与 AOF 混合使用 | 单独使用 |

## 题目五：如何用 Redis 实现分布式锁？Redlock 是什么？

### 参考答案

### 基本分布式锁实现

使用 `SET` 命令的 `NX` 和 `PX` 参数：

```bash
SET lock_key unique_value NX PX 10000
```

**加锁需要满足三个条件**：
1. **原子性**：使用 `NX` 保证 key 不存在时才插入
2. **过期时间**：使用 `PX/EX` 设置锁的过期时间，避免锁无法释放
3. **唯一值**：使用唯一标识值区分不同客户端的锁，释放锁时验证

**解锁**（使用 Lua 脚本保证原子性）：
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

### 单节点分布式锁的问题

如果 Redis 主节点获取锁后宕机，从节点晋升时锁数据未同步，会导致多个客户端同时获取锁。

### Redlock（红锁）解决方案

Redis 官方推荐的分布式锁算法，基于多个独立 Redis 节点：

**算法步骤**：
1. 客户端获取当前时间戳（t1）
2. 依次向 N 个 Redis 节点执行加锁操作（SET + NX + PX + 超时时间）
3. 如果成功获取锁的条件：
   - 超过 N/2 + 1 个节点成功
   - 总耗时 < 锁的过期时间
4. 计算锁实际有效时间 = 锁过期时间 - 总耗时

**优点**：
- 多个独立节点避免单点故障
- 超过半数节点成功才认为加锁成功

**建议**：至少部署 5 个 Redis 主节点
