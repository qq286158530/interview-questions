## 题目一：Flink 与 Spark 的核心区别

### 📋 题目
Apache Flink 和 Apache Spark 都是大数据计算引擎，它们有哪些核心区别？各自的优势是什么？

### ✅ 参考答案

#### 核心架构差异

| 维度 | Spark | Flink |
|------|-------|-------|
| **计算模型** | 微批处理（Micro-Batch） | 流原生（Native Streaming） |
| **延迟** | 亚秒级~秒级 | 毫秒级（真正流处理） |
| **架构** | Spark Core → SparkSQL/ML/GraphX | DataStream API + DataSet API |
| **时间语义** | 主要处理批处理，支持有限事件时间 | 原生支持事件时间、处理时间、注入时间 |

#### 批处理 vs 流处理

**Spark 的批处理思维：**
- 把流当作"有界数据集"处理
- 一切皆为批，Structured Streaming 是对微批的封装
- 延迟较高（mini-batch 周期通常 0.5s~数秒）

**Flink 的流处理思维：**
- 把批当作"无界流的特例"处理
- 真正的流原生架构，每条数据独立处理
- 延迟低至毫秒级

#### 窗口函数支持

| 特性 | Spark Structured Streaming | Flink |
|------|----------------------------|-------|
| 滚动窗口 | ✅ | ✅ |
| 滑动窗口 | ✅ | ✅ |
| 会话窗口 | ❌（支持有限） | ✅（原生支持） |
| 全窗口（ProcessFunction） | 有限 | ✅（完整支持） |
| 迟到数据处理 | watermark + late rows | watermark + allowedLateness + sideOutput |

#### 状态管理

- **Spark**：状态存储依赖外部系统（如 Redis），流状态管理能力较弱
- **Flink**：内置完整状态管理，支持 keyed-state、operator-state，RocksDB 状态后端

#### 生态集成

| 维度 | Spark | Flink |
|------|-------|-------|
| SQL 支持 | SparkSQL（成熟） | FlinkSQL（1.10后成熟） |
| ML 生态 | MLlib（丰富） | FlinkML（相对较少） |
| 图计算 | GraphX | Gelly |
| 生态兼容 | Hadoop 生态完美兼容 | 完美兼容 Kafka、Hadoop |

#### 各自优势

**Spark 优势：**
- 批处理场景更成熟，生态更完整
- 与 Hadoop/Hive/HBase 无缝集成
- ML 和 SQL 能力更强
- 社区活跃，文档丰富

**Flink 优势：**
- 真正的流处理，延迟更低
- 窗口函数更丰富灵活
- 状态管理更强大
- 事件时间处理更精确
- 适合实时数仓、流式 ETL、实时风控等场景

#### 面试加分点
- 目前行业趋势：离线用 Spark，实时用 Flink 已是主流架构
- 双流 Join、状态 TTL 等高级特性，Flink 支持更好
- 对于 Exactly-Once 语义，Flink 的 Checkpoint 实现更轻量

## 题目二：Flink 背压（Back Pressure）机制

### 📋 题目
什么是 Flink 的背压机制？它是如何工作的？生产环境中如何排查和解决背压问题？

### ✅ 参考答案

#### 背压定义
背压（Back Pressure）是指下游算子处理速度跟不上上游数据发送速度时，产生的反压现象。如果不加以控制，会导致数据积压、checkpoint 失败、甚至程序崩溃。

#### Flink 背压工作原理

Flink 采用**基于信用（Credit-Based）的流量控制机制**：

```
上游（TaskManager A）              下游（TaskManager B）
┌──────────────────────┐           ┌──────────────────────┐
│  Task 1              │           │  Task 2              │
│  ┌────────────────┐  │           │  ┌────────────────┐  │
│  │ ResultPartition│──┼──TCP连接──┼─▶│ InputChannel   │  │
│  │ (缓冲区队列)    │  │  credit   │  │ (消费队列)      │  │
│  └────────────────┘  │           │  └────────────────┘  │
└──────────────────────┘           └──────────────────────┘
```

**核心机制：**
1. 下游 TaskManager 维护一个**接收缓冲区（Floating Buffer Pool）**
2. 上游发送数据前，先检查下游是否有可用信用（credit）
3. 如果下游有足够缓冲区，数据才被发送
4. 背压时，下游缓冲区耗尽，信用变为 0，上游自动停止发送

#### 背压传播链

```
Source（被背压） → Map → Filter → KeyBy → Window → Sink
       ↑                                              ↑
   背压从 Sink 传播回 Source，形成完整的反压链路
```

#### 背压等级判断

通过 Flink Web UI 查看背压状态：
- **OK**：正常，下游消费速度跟得上上游
- **LOW**：轻微背压，可接受
- **HIGH**：严重背压，需要排查
- **LOW** 或 **HIGH** 状态持续存在 = 问题

#### 背压原因排查步骤

**① 确认背压位置**
```bash
# 通过 Web UI 或 REST API 查看
http://<jobmanager>:8081/#/job/<job_id>/back-pressure
```

**② 常见原因分析**

| 原因 | 表现 | 解决方案 |
|------|------|----------|
| 数据倾斜（Data Skew） | 部分 SubTask CPU 100% | 重新设计 KeyBy，使用随机前缀 |
| 下游 Sink 写入慢 | Sink 并行度低或外部系统瓶颈 | 增加并行度，优化 Sink 配置 |
| 状态过大 | RocksDB 读写慢 | 减少状态大小，优化序列化 |
| GC 停顿 | TaskManager GC 频繁 | 增大 TM 内存，优化 GC |
| 网络瓶颈 | shuffle 速度慢 | 增加网络带宽，调整 TM 配置 |

**③ 定位数据倾斜**
```java
// 查看各 SubTask 的记录数
env.setParallelism(4);
DataStream<String> stream = ...

// 加一个侧输出流输出分区信息
output.collect(value);
```

**④ 常用优化手段**

```java
// 1. 增加并行度
env.setParallelism(8);

// 2. 使用 rebalance() 打散数据
stream.rebalance();

// 3. 使用 rescale() 本地化调度
stream.rescale();

// 4. 调整 TM 配置
taskmanager.numberOfTaskSlots: 8
taskmanager.heap.size: 4096m

// 5. 使用 RocksDB 增量 Checkpoint 减少状态 IO
env.getCheckpointConfig().setPreferCheckpointForRecovery(true);
```

#### 生产环境监控建议

- 开启 Flink Metrics，监控 `recordsReadPerSecond`、`recordsWrittenPerSecond`
- 配置告警：背压持续 N 分钟 > HIGH 状态
- 定期检查 Checkpoint 时长和状态大小

## 题目三：Flink 实时 ETL 场景实战

### 📋 题目
如何使用 Flink 实现一个实时数据清洗 ETL 任务？请实现：从 Kafka 读取原始日志，进行数据清洗、去重、聚合，最终写入 ClickHouse。

### ✅ 参考答案

#### 架构设计

```
Kafka (原始日志) → Flink ETL → ClickHouse (清洗后数据)
                      ↓
               侧输出 (异常数据)
```

#### 完整实现代码

```java
public class RealTimeETLJob {
    
    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = 
            StreamExecutionEnvironment.getExecutionEnvironment();
        
        // 启用 Checkpoint（Exactly-Once）
        env.enableCheckpointing(60 * 1000);
        env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);
        env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30 * 1000);
        
        // 设置状态后端（RocksDB，适合大状态）
        env.setStateBackend(new RocksDBStateBackend("hdfs:///checkpoints"));
        
        // Kafka 数据源
        KafkaSource<String> kafkaSource = KafkaSource.<String>builder()
            .setBootstrapServers("kafka:9092")
            .setGroupId("flink-etl-group")
            .setTopics("raw-logs")
            .setStartingOffsets(OffsetsInitializer.committedOffsets())
            .setValueOnlyDeserializer(new SimpleStringSchema())
            .build();
        
        DataStream<String> rawStream = env.fromSource(
            kafkaSource, WatermarkStrategy.noWatermarks(), "Kafka Source");
        
        // 旁路输出标签（用于异常数据）
        OutputTag<String> dirtyDataTag = new OutputTag<>("dirty-data"){};
        OutputTag<String> duplicateDataTag = new OutputTag<>("duplicate-data"){};
        
        // 数据清洗主流程
        SingleOutputStreamOperator<CleanedLog> etlStream = rawStream
            .process(new LogETLProcessFunction(dirtyDataTag, duplicateDataTag))
            .name("Log ETL Process");
        
        // 异常数据侧输出
        DataStream<String> dirtyData = etlStream.getSideOutput(dirtyDataTag);
        DataStream<String> duplicateData = etlStream.getSideOutput(duplicateDataTag);
        
        // 写入 ClickHouse
        etlStream.addSink(new ClickHouseSink());
        
        // 异常数据写入备用表
        dirtyData.addSink(new DirtyDataSink());
        duplicateData.addSink(new DuplicateDataSink());
        
        env.execute("RealTime ETL Job");
    }
}
```

#### 日志清洗函数实现

```java
public class LogETLProcessFunction 
    extends ProcessFunction<String, CleanedLog> {
    
    private OutputTag<String> dirtyDataTag;
    private OutputTag<String> duplicateDataTag;
    
    // 状态：用于去重（使用 MapState）
    private MapState<String, Long> seenKeys;
    // 状态：统计每分钟处理量
    private ValueState<Long> minuteCount;
    
    @Override
    public void open(Configuration parameters) {
        // 初始化 MapState 用于去重
        MapStateDescriptor<String, Long> seenKeysDesc = 
            new MapStateDescriptor<>("seen_keys", String.class, Long.class);
        seenKeys = getRuntimeContext().getMapState(seenKeysDesc);
        
        // 初始化计数状态
        ValueStateDescriptor<Long> minuteCountDesc = 
            new ValueStateDescriptor<>("minute_count", Long.class);
        minuteCount = getRuntimeContext().getState(minuteCountDesc);
    }
    
    @Override
    public void processElement(String raw, Context ctx, Collector<CleanedLog> out) {
        try {
            // 1. JSON 解析
            JSONObject logJson = JSON.parseObject(raw);
            
            // 2. 数据校验
            String userId = logJson.getString("userId");
            String eventId = logJson.getString("eventId");
            Long timestamp = logJson.getLong("timestamp");
            
            if (userId == null || eventId == null || timestamp == null) {
                ctx.output(dirtyDataTag, raw);  // 脏数据
                return;
            }
            
            // 3. 去重检查（基于 eventId）
            if (seenKeys.containsKey(eventId)) {
                ctx.output(duplicateDataTag, raw);  // 重复数据
                return;
            }
            seenKeys.put(eventId, timestamp);
            
            // 4. 数据清洗
            CleanedLog cleaned = new CleanedLog();
            cleaned.setUserId(userId.trim().toLowerCase());
            cleaned.setEventId(eventId.trim());
            cleaned.setEventTime(LocalDateTime.ofInstant(
                Instant.ofEpochMilli(timestamp), ZoneId.systemDefault()));
            cleaned.setEventType(logJson.getString("eventType"));
            cleaned.setDeviceId(logJson.getString("deviceId"));
            cleaned.setSessionId(logJson.getString("sessionId"));
            
            // 5. 敏感数据脱敏
            if (cleaned.getPhone() != null) {
                cleaned.setPhone(desensitizePhone(cleaned.getPhone()));
            }
            
            out.collect(cleaned);
            
        } catch (Exception e) {
            ctx.output(dirtyDataTag, raw);  // 解析异常
        }
    }
    
    private String desensitizePhone(String phone) {
        if (phone.length() == 11) {
            return phone.substring(0, 3) + "****" + phone.substring(7);
        }
        return phone;
    }
}
```

#### ClickHouse Sink 实现

```java
public class ClickHouseSink extends RichSinkFunction<CleanedLog> {
    
    private Connection connection;
    private PreparedStatement statement;
    
    @Override
    public void open(Configuration parameters) {
        try {
            Class.forName("ru.clickhouse.jdbc.ClickHouseDriver");
            connection = DriverManager.getConnection(
                "jdbc:clickhouse://clickhouse:8123/default");
            statement = connection.prepareStatement(
                "INSERT INTO cleaned_logs VALUES (?, ?, ?, ?, ?, ?, ?)");
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
    
    @Override
    public void invoke(CleanedLog log, Context context) {
        try {
            statement.setString(1, log.getUserId());
            statement.setString(2, log.getEventId());
            statement.setString(3, log.getEventTime().toString());
            statement.setString(4, log.getEventType());
            statement.setString(5, log.getDeviceId());
            statement.setString(6, log.getSessionId());
            statement.setTimestamp(7, Timestamp.valueOf(log.getEventTime()));
            statement.addBatch();
        } catch (SQLException e) {
            throw new RuntimeException(e);
        }
    }
    
    @Override
    public void close() {
        try {
            if (statement != null) statement.close();
            if (connection != null) connection.close();
        } catch (SQLException e) {
            e.printStackTrace();
        }
    }
}
```

## 题目四：Flink CDC（Change Data Capture）实战

### 📋 题目
什么是 Flink CDC？它如何实现数据库的实时同步？请用 Flink CDC 实现 MySQL 到 ClickHouse 的实时同步方案。

### ✅ 参考答案

#### CDC 概念

CDC（Change Data Capture）是一种捕获数据库变更的技术，Flink CDC 通过 Debezium 采集 MySQL binlog，实现：
- **全量同步**：首次读取数据库全量数据
- **增量同步**：实时监听 binlog，捕获 DML/DDL 变更
- **断点续传**：支持 Checkpoint，任务失败可恢复

#### 架构原理

```
MySQL (binlog) → Debezium → Kafka → Flink CDC Connector → ClickHouse
                      ↓
               变更记录流
```

Flink CDC Connector 的优势：
- 无需额外部署 Kafka，直接 Debezium → Flink
- 支持全量 + 增量一体化同步
- 支持 Schema Evolution（Schema 变更处理）

#### Maven 依赖

```xml
<dependency>
    <groupId>com.ververica</groupId>
    <artifactId>flink-connector-mysql-cdc</artifactId>
    <version>2.4.0</version>
</dependency>
```

#### 完整实现

```java
public class MySQLToClickHouseCDC {
    
    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = 
            StreamExecutionEnvironment.getExecutionEnvironment();
        
        // 启用 Checkpoint
        env.enableCheckpointing(30 * 1000);
        env.getCheckpointConfig().setMinPauseBetweenCheckpoints(10 * 1000);
        
        // MySQL CDC Source 配置
        MySqlSource<String> mySqlSource = MySqlSource.<String>builder()
            .hostname("mysql-host")
            .port(3306)
            .databaseList("source_db")       // 监控的数据库
            .tableList("source_db.orders")   // 监控的表
            .username("flink_cdc")
            .password("password")
            .deserializer(new JsonDebeziumDeserializationSchema())  // JSON 格式
            .startupOptions(StartupOptions.initial())  // 全量 + 增量
            .includeSchemaChanges(true)  // 包含 Schema 变更
            .build();
        
        DataStream<String> cdcStream = env.fromSource(
            mySqlSource, WatermarkStrategy.noWatermarks(), "MySQL CDC Source");
        
        // 解析 CDC 数据并转换
        SingleOutputStreamOperator<Order> transformStream = cdcStream
            .process(new CDCMaintenanceProcessFunction())
            .name("CDC Transform");
        
        // 写入 ClickHouse（Upsert）
        transformStream.addSink(new ClickHouseUpsertSink());
        
        env.execute("MySQL to ClickHouse CDC Sync");
    }
}
```

#### CDC 数据解析

```java
public class CDCMaintenanceProcessFunction 
    extends ProcessFunction<String, Order> {
    
    @Override
    public void processElement(String json, Context ctx, Collector<Order> out) {
        try {
            JSONObject record = JSON.parseObject(json);
            String operation = record.getString("op");
            
            // 跳过 DDL 操作（schema 变更）
            if ("c".equals(operation) || "r".equals(operation)) {
                // INSERT / READ（全量阶段）
                Order order = parseOrder(record.getJSONObject("after"));
                out.collect(order);
            } else if ("u".equals(operation)) {
                // UPDATE：使用 after 数据
                Order order = parseOrder(record.getJSONObject("after"));
                out.collect(order);
            } else if ("d".equals(operation)) {
                // DELETE：使用 before 数据
                Order order = parseOrder(record.getJSONObject("before"));
                order.setDeleted(1);  // 标记删除
                out.collect(order);
            }
            // else: 's' = schema change, skip
            
        } catch (Exception e) {
            // 记录异常
            System.err.println("CDC parse error: " + json);
        }
    }
    
    private Order parseOrder(JSONObject json) {
        Order order = new Order();
        order.setId(json.getLong("id"));
        order.setOrderNo(json.getString("order_no"));
        order.setUserId(json.getLong("user_id"));
        order.setAmount(json.getBigDecimal("amount"));
        order.setStatus(json.getString("status"));
        order.setCreatedAt(json.getTimestamp("created_at"));
        return order;
    }
}
```

#### ClickHouse 物化视图同步方案

对于 ClickHouse，推荐使用 **MergeTree + 物化视图** 实现 Upsert：

```sql
-- ClickHouse 建表
CREATE TABLE orders (
    id UInt64,
    order_no String,
    user_id UInt64,
    amount Decimal(10,2),
    status String,
    created_at DateTime,
    sign Int8 DEFAULT 1  -- 删除标记
) ENGINE = ReplacingMergeTree(sign)
ORDER BY (id, created_at);
```

#### 常见问题处理

| 问题 | 解决方案 |
|------|----------|
| 全量阶段太慢 | 使用整库同步（databaseList）+ 分片并发读取 |
| Schema 变更丢失数据 | 开启 includeSchemaChanges，配置 Schema 演化 |
| 重复数据 | ClickHouse 使用 ReplacingMergeTree 去重 |
| 延迟过高 | 调整 TM 并行度和内存，减少状态后端 IO |

## 题目五：Flink 性能调优与生产实践

### 📋 题目
生产环境中如何对 Flink 作业进行全面的性能调优？请从资源规划、并行度、状态管理、网络、Checkpoint 等维度说明。

### ✅ 参考答案

#### 一、资源规划调优

**JobManager 配置：**
```yaml
# flink-conf.yaml
jobmanager.heap.size: 2048m  # JM 内存不宜过大
jobmanager.rpc.address: <jm_host>
```

**TaskManager 配置：**
```yaml
taskmanager.heap.size: 4096m  # 至少 4GB，根据状态大小调整
taskmanager.numberOfTaskSlots: 8  # 通常 4-8
taskmanager.memory.managed.fraction: 0.4  # 托管内存占比
```

**推荐配置比例：**
| 场景 | TM 内存 | Slot 数 | 并行度建议 |
|------|---------|---------|-----------|
| 小状态 ETL | 4GB | 4 | 4-8 |
| 大状态聚合 | 8GB | 4 | 4-8 |
| 超大状态（RocksDB） | 16GB+ | 2-4 | 2-4 |

#### 二、并行度（Parallelism）调优

**原则：**
- Source 并行度 = Kafka 分区数（或 topic 数）
- 中间算子：与 Source 保持一致或成倍数关系
- Sink 并行度：不超过 Sink 集群的写入能力

```java
// 全链路并行度统一设置
env.setParallelism(8);

// 分算子独立设置
source.setParallelism(4);
process.setParallelism(8);
sink.setParallelism(4);

// 数据倾斜时：增大热点算子并行度
keyBy.setParallelism(16);
```

#### 三、状态管理调优

**减少状态大小的技巧：**

```java
// 1. 使用精简的序列化器（Kyro -> Protobuf/FastJSON）
env.getConfig().setExecutionConfig(new ExecutionConfig()
    .setSerializerInfos(
        new KryoSerializerProvider(new ProtobufSerializer())
    ));

// 2. 状态 TTL，定期清理过期数据
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.days(1))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)
    .build();

// 3. 使用 MapState 替代 ValueState<Map>
MapStateDescriptor<String, Long> mapDesc = 
    new MapStateDescriptor<>("keyed_state", String.class, Long.class);
```

**RocksDB 调优：**
```java
// RocksDB 配置
RocksDBStateBackend rocksDBBackend = new RocksDBStateBackend(checkpointDir);
rocksDBBackend.setDbStoragePath("file:///tmp/rocksdb");  // 本地 SSD
rocksDBBackend.setEnableIncrementalCheckpointing(true);  // 增量 Checkpoint
rocksDBBackend.setMemorySize(1024 * 1024 * 1024L);  // 1GB 写缓存
env.setStateBackend(rocksDBBackend);
```

#### 四、网络调优

```yaml
# flink-conf.yaml
taskmanager.network.memory.fraction: 0.1  # 网络缓冲区占比
taskmanager.network.memory.min: 64mb
```
