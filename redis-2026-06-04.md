## 题目一：Redis 有哪些数据类型？应用场景是什么？

### 参考答案

Redis 提供了丰富的数据类型，常见的有五种：**String（字符串）**、**Hash（哈希）**、**List（列表）**、**Set（集合）**、**Zset（有序集合）**。

随着 Redis 版本的更新，后面又支持了四种数据类型：**BitMap（2.2版）**、**HyperLogLog（2.8版）**、**GEO（3.2版）**、**Stream（5.0版）**。

### 五种基本数据类型的应用场景

| 类型 | 应用场景 |
|------|----------|
| **String** | 缓存对象、常规计数、分布式锁、共享Session信息 |
| **Hash** | 缓存对象、购物车 |
| **List** | 消息队列（需自行实现全局唯一ID，不支持消费组形式消费） |
| **Set** | 聚合计算（并集、交集、差集），如点赞、共同关注、抽奖活动 |
| **Zset** | 排序场景，如排行榜、电话和姓名排序 |

### 四种高级数据类型的应用场景

| 类型 | 应用场景 |
|------|----------|
| **BitMap** | 二值状态统计，如签到、判断用户登录状态、连续签到用户总数 |
| **HyperLogLog** | 海量数据基数统计，如百万级网页UV计数 |
| **GEO** | 存储地理位置信息，如滴滴叫车 |
| **Stream** | 消息队列，相比List支持自动生成全局唯一ID，支持消费组形式消费数据 |

### 各数据类型的底层实现

- **String**：底层实现为 SDS（简单动态字符串），有 int、embstr、raw 三种编码
- **List**：Redis 3.2+ 底层为 quicklist，Redis 7.0+ 使用 listpack 替代压缩列表
- **Hash**：底层为压缩列表或哈希表，Redis 7.0+ 使用 listpack
- **Set**：底层为哈希表或整数集合
- **Zset**：底层为压缩列表或跳表，Redis 7.0+ 使用 listpack

## 题目二：Redis 持久化机制是什么？RDB 和 AOF 有什么区别？

### 参考答案

Redis 共有三种数据持久化的方式：**AOF 日志**、**RDB 快照**、**混合持久化**。

### AOF 日志（Append Only File）

AOF 日志是每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里。Redis 重启时，会读取该文件记录的命令，然后逐一执行来进行数据恢复。

**AOF 写回策略有三种：**

| 策略 | 说明 | 优缺点 |
|------|------|--------|
| `always` | 每次写操作命令执行完后，同步将AOF日志数据写回硬盘 | 数据不会丢失，但IO开销大 |
| `everysec`（默认） | 每次写操作命令执行完后，先将命令写入AOF内核缓冲区，每隔一秒将缓冲区内容写回硬盘 | 性能适中，最多丢失1秒数据 |
| `no` | 由操作系统决定何时将缓冲区内容写回硬盘 | 性能最好，但数据丢失不可控 |

AOF 文件过大会触发**AOF 重写机制**，后台子进程 `bgrewriteaof` 会读取数据库中所有键值对，用一条命令记录到新的 AOF 文件，压缩文件体积。

### RDB 快照（Redis Database）

RDB 快照是记录某一个瞬间的内存数据，以二进制方式写入磁盘。Redis 提供了两个命令：

- `save`：在主线程生成 RDB 文件，会阻塞主线程
- `bgsave`：创建子进程生成 RDB 文件，避免阻塞

Redis 可通过配置文件实现自动快照（`save 900 1` 等条件）。

RDB 快照使用**写时复制（Copy-On-Write）**技术，在 `bgsave` 期间，主线程可以继续处理操作命令。

### 混合持久化

Redis 4.0 提出了混合持久化，结合了 RDB 和 AOF 的优点。

混合持久化工作原理：在 AOF 重写时，fork 出来的子进程先将与主线程共享的内存数据以 **RDB 格式**写入到 AOF 文件，然后主线程处理的操作命令会被记录到**重写缓冲区**，增量命令以 **AOF 格式**写入到 AOF 文件。

### RDB 与 AOF 对比

| 特性 | RDB | AOF |
|------|-----|-----|
| 数据恢复速度 | 快（直接读取二进制） | 慢（需执行所有操作命令） |
| 数据完整性 | 可能有丢失（取决于快照频率） | 更完整（可配置最多丢失1秒数据） |
| 文件体积 | 小（二进制格式） | 大（文本命令日志） |
| 快照频率 | 重量级操作，不宜频繁 | 可配置，everysec较常用 |
| 优先级 | 恢复优先加载RDB | 无 |

## 题目三：什么是缓存雪崩、击穿、穿透？如何解决？

### 参考答案

### 缓存雪崩

**定义：** 大量缓存数据在同一时间过期或失效，导致大量请求直接打到数据库，造成数据库压力激增甚至崩溃。

**解决方案：**

1. **给缓存数据加随机过期时间**：避免大量缓存同时过期
2. **互斥锁/限流**：只允许一个线程重建缓存，其他线程等待或返回旧数据
3. **多级缓存**：设置多级缓存策略（如L1+L2缓存）
4. **Redis 永久生效**：对热点数据不设置过期时间，由程序控制更新

### 缓存击穿

**定义：** 某个热点数据过期，瞬间大量请求直接打到数据库，和雪崩类似但是是单点问题。

**解决方案：**

1. **互斥锁**：保证只有一个线程重建缓存，其他线程等待
2. **热点数据永不过期**：物理不过期 + 逻辑过期（后台异步更新）
3. **提前预热**：在缓存即将过期前，主动更新缓存

### 缓存穿透

**定义：** 请求的数据既不在缓存中，也不在数据库中，导致每次请求都直接打到数据库。常见于恶意请求不存在的数据。

**解决方案：**

1. **布隆过滤器（Bloom Filter）**：在缓存层前加一层过滤器，请求的数据不存在则直接返回
2. **空值缓存**：将数据库查询结果为空的数据也缓存起来（设置较短过期时间）
3. **参数校验**：对请求参数进行合法性校验，拦截非法请求
4. **黑名单机制**：对恶意IP或用户进行限制

### 一句话总结

| 问题 | 原因 | 解决思路 |
|------|------|----------|
| 缓存雪崩 | 大量缓存同时过期 | 随机过期时间、多级缓存、限流 |
| 缓存击穿 | 热点缓存过期 | 互斥锁、热点数据永不过期 |
| 缓存穿透 | 数据既不在缓存也不在数据库 | 布隆过滤器、空值缓存、参数校验 |

## 题目四：Redis 主从复制是怎么实现的？哨兵模式有什么用？

### 参考答案

### 主从复制

主从复制是 Redis 高可用服务的基础，实现方案是将数据从主服务器（master）同步到多台从服务器（slave），即 **一主多从** 模式。

**工作原理：**

1. **全量同步**：从服务器连接主服务器时，发起 `PSYNC` 命令，如果从服务器没有复制过主服务器的数据，主服务器会执行 `BGSAVE` 生成 RDB 快照，同时将后续操作命令记录到**复制缓冲区**，发送给从服务器
2. **增量同步**：主服务器将 replication buffer 中的操作命令发送给从服务器执行

**读写分离：** 主服务器进行读写操作，从服务器只读并接受主服务器同步过来的写操作命令。主从服务器之间的命令复制是**异步进行**的。

### 哨兵模式（Sentinel）

哨兵模式用于自动故障转移，解决了主从服务器故障时需要手动恢复的问题。

**哨兵的核心功能：**

1. **监控**：哨兵周期性向主从服务器发送 PING 命令，检测节点是否存活
2. **自动故障转移**：当主服务器下线时，哨兵通过选举机制选出一个从服务器升级为新的主服务器，其他从服务器自动指向新主服务器
3. **通知**：将故障转移的结果通知给客户端

**哨兵如何判断主服务器下线？**

- 主观下线：单个哨兵检测到主服务器无响应
- 客观下线：超过配置数量的哨兵都认为主服务器下线（通过 `sentinel is-master-down-by-addr` 命令协商）

**选举机制：** 哨兵通过 Raft 算法进行leader选举，选举出的leader负责执行故障转移。

### 主从复制的问题

主从复制无法保证**强一致性**（数据时时刻刻保持一致），因为命令复制是异步的。如果需要强一致性，需要使用 **Redis Cluster** 或手动干预。

## 题目五：Redis 为什么是单线程的？为什么单线程还能这么快？

### 参考答案

### Redis 是单线程的吗？

Redis **单线程**指的是「接收客户端请求→解析请求→进行数据读写等操作→发送数据给客户端」这个过程是由**一个线程（主线程）**来完成的。

但 Redis 程序**并不是单线程的**，Redis 在启动时会启动**后台线程（BIO）**：

- **Redis 2.6 版本**：启动 2 个后台线程，分别处理关闭文件、AOF 刷盘任务
- **Redis 4.0 版本之后**：新增 `lazyfree` 线程，用于异步释放内存（如执行 `unlink key`、`flushdb async`）

后台线程相当于消费者，从任务队列中取出任务执行，避免阻塞主线程。

### Redis 6.0 之后的多线程

Redis 6.0 版本引入了**多线程 I/O** 来处理网络请求，主要是为了提高网络 I/O 的并行度。但**命令执行仍然是单线程**的。

默认情况下，I/O 多线程只针对发送响应数据（write client socket），可以通过配置 `io-threads-do-reads yes` 开启多线程处理读请求。

### 为什么单线程的 Redis 还能这么快？

1. **内存操作**：Redis 大部分操作都在内存中完成，并采用了高效的数据结构（SDS、跳表等），瓶颈是内存或网络带宽，而非 CPU
2. **避免了多线程竞争**：单线程模型省去了多线程切换带来的时间和性能开销，也不会产生死锁问题
3. **I/O 多路复用**：Redis 采用 `select/epoll` 机制，一个线程处理多个 IO 流，使得内核可以同时监听多个 Socket 上的请求，有请求到达时才调用 Redis 线程处理，避免了阻塞

### 官方基准测试

Redis 官方基准测试结果显示，单线程的 Redis 吞吐量可以达到 **10W/秒**，性能非常出色。
