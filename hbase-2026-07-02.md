## 题目一：HBase 的整体架构是怎样的？Regionserver 和 HDFS 的关系是什么？

### 参考答案

HBase 采用 **Master-Slave 架构**，核心组件如下：

| 组件 | 职责 |
|------|------|
| **HMaster** | 负责表的管理、Region 的分配与调度、负载均衡，以及故障恢复。不参与实际数据读写。 |
| **RegionServer** | 负责实际数据的读写请求，是 HBase 的工作节点。每个 RegionServer 管理多个 Region。 |
| **ZooKeeper** | 负责选举 Master、存储元数据路径（如 `/hbase/meta-region-server`）、检测 RegionServer 存活状态。 |
| **HDFS** | HBase 的底层存储层，HFile 最终以文件形式存在于 HDFS 上。RegionServer 与 HDFS DataNode 通常部署在同一节点上，利用数据本地性提升读写性能。 |

**读写流程简述：**
1. 客户端连接 ZooKeeper，获取 `-ROOT-` 表（已废弃）或 `hbase:meta` 表的位置。
2. 从 `hbase:meta` 中找到目标 RowKey 所在的 Region 及其 RegionServer 地址。
3. 直接与目标 RegionServer 通信，进行读写操作（无需经过 Master）。

**RegionServer 与 HDFS 的关系：**
- RegionServer 本身不持久化数据，所有数据最终以 HFile 格式写入 HDFS。
- 为了读取性能，HBase 会利用 **BlockCache**（读缓存）和 **MemStore**（写缓存）。
- RegionServer 与 DataNode 共置（Co-locality），使得数据写入时天然具备"数据本地性"（写入本地 HDFS 节点），读取时也尽可能从本地读取。

## 题目二：HBase 的存储结构是什么？MemStore 和 HFile 的作用分别是什么？

### 参考答案

HBase 的存储层次从上到下为：

```
Table → Region → Store → MemStore + HFiles
```

**核心概念：**

- **Region**：一张表按 RowKey 范围水平切分成多个 Region，每个 Region 负责一段连续 RowKey 区间的数据。
- **Store**：每个列族对应一个 Store，一个 Region 中可以有多个 Store。
- **MemStore**：写缓冲区，发生在每个 Store 中。
- **HFile**：HBase 在 HDFS 上的实际存储文件格式。

**写入流程（Write-Ahead Log 机制）：**

1. 数据先写入 **WAL（Write-Ahead Log）**，确保宕机可恢复。
2. 数据随后写入 **MemStore**（内存中），此时客户端收到 ACK。
3. 当 MemStore 达到阈值（默认约 128MB × 列族数），触发 **Flush** 操作，将内存数据Dump 成 HFile 落盘。

**HFile 结构：**

HFile 是 HBase 自定义的二进制存储格式，结构如下：

```
HFile
├── BlockIndex（块索引区）
├── Data Block（数据块，默认 64KB）
│   ├── KeyValue 格式：RowKey(变长) + ColumnFamily + Column Qualifier + Timestamp + Value
│   └── 按 RowKey 有序排列
├── FileInfo（文件元信息）
└── Trailer（文件尾部指针）
```

**为什么需要 MemStore？**
- HDFS 顺序写入、不支持随机写的特性，使得每次写入都直接落盘代价很高。MemStore 作为内存写缓冲，批量 flush 成 HFile，显著提升写入性能。
- 同时，MemStore 内部按 RowKey 排序，flush 后的 HFile 也是有序的，利于后续 Compaction 和读取。

**Minor Compaction 与 Major Compaction：**

- **Minor Compaction**：将多个小 HFile 合并成少量较大的 HFile（控制文件数量）。
- **Major Compaction**：将一个 Store 下所有 HFile 合并成一个，清理过期版本和删除数据，显著影响读取性能但会消耗大量 IO。

## 题目三：如何设计 HBase 的 RowKey？有哪些原则和技巧？

### 参考答案

RowKey 是 HBase 中数据检索的唯一手段，设计质量直接影响读写性能。核心原则如下：

### 三大原则

**1. 散列性原则（避免热点）**
热点问题：大量请求集中在一个或少数几个 Region 上，导致负载不均。

- **反转时间戳**：将时间戳反转（如 `Long.MAX_VALUE - timestamp`），使新数据散列到不同 Region。
- **哈希打散**：对 RowKey 取 MD5 或 Hash，截取前几位作为前缀，例如 `hash(rowKey) % N` 将数据打散到 N 个 Region。
- **随机前缀**：在 RowKey 头部拼接随机数。

**2. 唯一性原则**
RowKey 必须唯一标识一行数据，复合 RowKey 是常见做法，例如：

```
orderId_userId_timestamp  →  20260702001_10086_1719900000
```

**3. 长度原则**
RowKey 存储在每个 KeyValue 中，过长的 RowKey 会浪费存储空间、增加 IO 开销。建议：
- 业务 ID 类：定长或控制长度在 16~50 字节以内。
- 用 Long 类型替代字符串存储时间戳（8 字节 vs 13 字节字符串）。

### 常见设计模式

| 场景 | RowKey 设计方案 | 示例 |
|------|----------------|------|
| 用户订单查询 | `userId + reverseTimestamp` | `10086_9999999999999`（新时间小值排前面） |
| 消息收件箱 | `receiverId + timestamp` | `10086_1719900000` |
| 时序数据 | `deviceId + reverseTimestamp` | `sensor_001_9999999999999` |
| 交易记录 | `merchantId + orderId` | `M001_20260702001` |

### 注意事项
- 避免使用单调递增的 RowKey（如纯时间戳），会导致所有写入集中在最后一个 Region，形成 **写热点**。
- 列族名、列名尽量短（HBase 内部会为每个单元格存储这些元信息）。
- 结合业务特点，利用 **前缀过滤** 减少扫描范围。

## 题目四：HBase 的协处理器（Coprocessor）是什么？Observer 和 Endpoint 的区别和使用场景？

### 参考答案

HBase 协处理器是加载在 RegionServer 上的扩展框架，允许用户在数据写入/读取路径上执行自定义逻辑，类似于数据库的触发器和存储过程。

### 两类协处理器

#### 1. Observer（观察者，类似触发器）

**特点：** 在特定事件发生时自动执行回调，不主动发起响应，只能在数据操作流程中"观察"并修改行为。

**三种 Observer 类型：**

| 类型 | 触发时机 | 典型场景 |
|------|----------|----------|
| **RegionObserver** | GET/PUT/DELETE/SCAN 等数据操作前后 | 数据验证、访问控制、修改写入内容 |
| **WALObserver** | WAL 写入前后 | WAL 加密、审计日志 |
| **MasterObserver** | DDL 操作（CREATE TABLE/DROP TABLE 等）前后 | 权限控制、审计 |

**使用示例（RegionObserver 过滤非法数据）：**

```java
public class DataValidator extends BaseRegionObserver {
    @Override
    public void prePut(final ObserverContext<RegionCoprocessorEnvironment> e,
                       final Put put, final WALEdit edit) throws IOException {
        // 获取列族 "order" 下的列 "amount"
        KeyValue kv = put.get(Bytes.toBytes("order"), Bytes.toBytes("amount"));
        if (kv != null) {
            long amount = Bytes.toLong(kv.getValue());
            if (amount < 0) {
                throw new IOException("订单金额不能为负数！");
            }
        }
    }
}
```

#### 2. Endpoint（端点，类似存储过程）

**特点：** 客户端主动调用，由 RegionServer 在目标 Region 上执行自定义计算逻辑，最后将结果返回给客户端（类似 RPC）。

**典型使用场景：**
- 在服务端执行聚合计算（COUNT、SUM、AVG），减少网络传输数据量。
- 实现跨列族、跨列的复杂查询逻辑。

**使用示例（服务端聚合）：**

```java
// 定义 Coprocessor 接口
public interface OrderCounter extends Coprocessor {
    long getOrderCount(byte[] startRow, byte[] endRow);
}

// 服务端实现
public class OrderCounterImpl extends BaseEndpointCoprocessor implements OrderCounter {
    @Override
    public long getOrderCount(byte[] startRow, byte[] endRow) {
        // 扫描指定 RowKey 范围的订单
        InternalScanner scanner = getEnvironment().getRegion().getScanner(...);
        // 统计数量并返回
    }
}

// 客户端调用
Table table = connection.getTable(TableName.valueOf("orders"));
byte[] startRow = ...;
byte[] endRow = ...;
OrderCounter counter = table.coprocessorProxy(OrderCounter.class, startRow, endRow);
long total = counter.getOrderCount(startRow, endRow);
```

### 部署方式

| 方式 | 配置位置 | 生效范围 | 适用场景 |
|------|----------|----------|----------|
| **Table Coprocessor** | 表的 `conf` 中配置 | 单张表 | 临时验证、按需加载 |
| **Global Coprocessor** | `hbase-site.xml` | 所有表/所有 Region | 全局拦截、审计 |

### 注意事项
- Observer 代码异常会导致数据操作失败，影响可用性，务必做好异常捕获。
- Endpoint 调用会显著增加 RegionServer CPU 负载，避免大数据量扫描。
- 协处理器类必须无状态（Stateless），因为 HBase 不保证同一客户端请求落在同一个 RegionServer 上。

## 题目五：HBase 如何保证数据一致性和高可用？与关系型数据库相比有哪些优劣？

### 参考答案

### 数据一致性

HBase 的一致性模型为 **最终一致性（Eventual Consistency）**，默认情况下：

- **写一致性**：写入 WAL + MemStore 后即返回客户端成功（ACK），此时数据在内存中，尚未刷写到 HFile。多副本通过 HDFS 复制保证数据不丢失。
- **读一致性**：同一 Region 内读操作可以指定 `Consistency.TIMELINE`（强一致性，读取最新数据）或默认一致性（可能被过期 BlockCache 响应）。

**强一致性读**可以通过以下方式实现：
```java
Get get = new Get(rowKey);
get.setConsistency(Consistency.STRONG);
Result result = table.get(get);
```

### 高可用保障

| 机制 | 说明 |
|------|------|
| **HDFS 多副本** | 每个 HFile 块默认 3 副本，任意 DataNode 宕机不丢数据。 |
| **WAL 持久化** | 数据先写 WAL，宕机后可从 WAL 恢复 MemStore 未 flush 的数据。 |
| **Master 高可用** | ZooKeeper 选举备用 Master，Master 宕机后自动切换。 |
| **Region 故障恢复** | ZooKeeper 检测到 RegionServer 宕机后，Master 将其 Region 快速迁移到其他节点（约数十秒）。 |
| **Replication** | 可开启 HBase 异步主从复制，实现跨数据中心容灾。 |

### 与关系型数据库的对比

| 维度 | HBase | 传统 RDBMS（如 MySQL） |
|------|-------|------------------------|
| **数据模型** | 列族式 NoSQL，宽表稀疏列 | 严格行式关系模型 |
| **一致性** | 最终一致（可配置强一致性） | 强 ACID 事务 |
| **扩展性** | 水平扩展，天然支持 PB 级数据 | 垂直扩展为主，分库分表复杂度高 |
| **写入性能** | 高（顺序写入 HDFS + MemStore） | 中等（B+树随机写） |
| **读取性能** | 弱（随机点查快，复杂扫描弱） | 强（索引优化，复杂查询优） |
| **事务** | 单行事务，无跨行跨表事务 | 支持 ACID 多表事务 |
| **JOIN** | 不支持，需在应用层或借助 MapReduce 实现 | 原生支持多表 JOIN |
| **延迟** | 毫秒级写入，查询延迟波动大 | 毫秒级稳定 |
| **适用场景** | 日志/物联网时序数据、推荐系统、画像存储 | 强事务业务、OLTP、报表查询 |

**总结：** HBase 擅长海量数据的随机写入与简单点查/范围查询，适合作为数据仓库或大数据场景下的在线存储；RDBMS 擅长复杂查询和强事务场景。两者并非替代关系，常见架构中会结合使用（如 HBase 存储明细，ElasticSearch/Hive 做复杂检索）。
