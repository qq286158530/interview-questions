## 题目一：Spark 与 Hadoop 的核心区别是什么？为什么 Spark 更快？

### 参考答案

| 维度 | Apache Spark | Hadoop MapReduce |
|------|-------------|------------------|
| **速度** | 内存计算比 MapReduce 快 10~100 倍 | 速度较慢 |
| **数据处理方式** | 实时+批量处理，内存计算，支持缓存 | 仅批量处理，依赖磁盘 |
| **编程难度** | 提供高级 API（Scala/Python/Java/R），易学 | 需要手写 Map/Reduce，较复杂 |
| **故障恢复** | 通过 RDD 实现分区恢复 | 通过副本机制保证容错 |
| **交互模式** | 支持交互式 shell | 需借助 Pig/Hive，无迭代计算 |

**Spark 更快的核心原因：**

1. **内存计算（In-Memory Computation）**：Spark 将数据存储在内存中，而非每次操作都读写磁盘，大幅减少 I/O 开销
2. **DAG 执行引擎**：Spark 将所有 transformation 构成有向无环图（DAG），直到遇到 Action 才触发执行，避免了不必要的中间结果写入磁盘
3. **Lazy Evaluation（延迟执行）**：transformation 不会立即执行，Spark 会进行整体优化后再执行
4. **RDD 抽象**：RDD 支持在内存中缓存常用数据，减少重复计算

## 题目二：什么是 RDD？RDD 有哪些核心特性？请解释 Spark 的 Lazy Evaluation。

### 参考答案

**RDD（Resilient Distributed Dataset）** 是 Spark 的核心抽象，表示一个弹性的、分布式的数据集。RDD 有以下核心特性：

### RDD 核心特性

1. **分区（Partitioned）**：数据被划分为多个分区，并行分发到集群各节点
2. **不可变性（Immutable）**：RDD 创建后不可修改，保证数据一致性
3. **容错性（Fault Tolerant）**：通过 lineage（血统）图记录 transformations，当节点故障时可根据 lineage 重新计算，无需数据副本
4. **并行计算（Parallel）**：各分区可在不同节点并行处理

### RDD 的创建方式

```python
# 方式1：通过并行化集合
data = [1, 2, 3, 4, 5]
rdd = sc.parallelize(data)

# 方式2：读取外部数据源
rdd = sc.textFile("hdfs://path/to/data.txt")
```

### Spark 的 Lazy Evaluation（延迟执行）

- **Transformation（转换操作）**：如 `map`、`filter`、`flatMap`、`reduceByKey` 等，不会立即执行，只是记录要执行的操作并构建 DAG
- **Action（动作操作）**：如 `collect()`、`count()`、`saveAsTextFile()` 等，触发实际的计算执行

```python
# 延迟执行示例
lines = sc.textFile("data.txt")        # 不加载数据
filtered = lines.filter(lambda x: "error" in x)  # 不执行过滤
result = filtered.collect()              # 触发执行
```

**延迟执行的优势**：
- Spark 可以整体优化执行计划，减少不必要的数据 shuffle
- 避免中间结果写入磁盘，提高执行效率

## 题目三：什么是 Spark Shuffle？Shuffle 过程中会发生什么？如何优化 Shuffle？

### 参考答案

### 什么是 Shuffle

Shuffle 是 Spark 在 **需要跨分区重新分配数据** 时发生的操作。当执行 `groupByKey`、`reduceByKey`、`join`、`cogroup` 等需要重新分区数据的算子时，Spark 需要将数据按照 Key 进行重新分配，这个过程即为 Shuffle。

### Shuffle 的执行过程

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  Map Stage  │ ───► │  Shuffle   │ ───► │ Reduce Stage│
│ (写入本地)   │      │ (跨节点传输) │      │ (聚合计算)   │
└─────────────┘      └─────────────┘      └─────────────┘
```

1. **Map 阶段**：每个 Mapper 将数据写入本地磁盘文件，并根据 Key 计算目标分区
2. **Shuffle Write**：数据按照分区写入磁盘，生成多个小文件
3. **Shuffle Read**：每个 Reducer 通过网络拉取属于自己分区的数据
4. **Reduce 阶段**：对拉取的数据进行聚合、排序等操作

### Shuffle 的主要开销

- **磁盘 I/O**：中间结果写入本地磁盘
- **网络 I/O**：数据在不同节点间传输
- **序列化/反序列化**：对象需序列化后传输，再反序列化

### 如何优化 Shuffle

1. **使用 Broadcast 广播小表**
   ```python
   small_df = broadcast(large_df.join(small_df))
   ```
   避免小表在 Shuffle 时重复传输

2. **减少 Shuffle 次数**
   - 使用 `map-side aggregation`（如 `reduceByKey` 而非 `groupByKey`）
   - `reduceByKey` 在 Map 端先做预聚合，比 `groupByKey` 更高效

3. **合理设置分区数**
   - 分区过多导致小文件问题，分区过少导致并行度不足
   - 建议：`2 ~ 4 × CPU core 数`

4. **使用 Coalesce 而非 Repartition（减少分区时）**
   - `coalesce()` 减少分区时不会全量 shuffle，比 `repartition()` 更高效

5. **使用排序合并而非哈希连接**
   - 大表 join 时，开启 `spark.sql.shuffle.partitions` 合理配置

## 题目四：Spark DataFrame 和 DataSet 的区别是什么？与 RDD 相比有什么优势？

### 参考答案

### DataFrame vs DataSet vs RDD

| 特性 | RDD | DataFrame | DataSet |
|------|-----|-----------|---------|
| **数据类型** | 无 schema，每行是泛型 | 每行是 `Row` 对象 | 每行是强类型对象 |
| **Schema** | 无 | 有（列名+类型） | 有（case class/POJO） |
| **优化** | 无（手动优化） | Tungsten + Catalyst 优化器 | Tungsten + Catalyst 优化器 |
| **编译时检查** | 无 | 无（运行时才发现错误） | 有（编译时检查类型） |
| **性能** | 较低 | 高 | 高 |
| **语言支持** | Scala/Java/Python/R | Scala/Java/Python/R | Scala/Java |

### DataFrame 优势

1. **自动优化（Catalyst Optimizer）**：DataFrame 通过 Catalyst 优化器生成最优执行计划，开发者无需关心执行细节
   ```sql
   df.filter("age > 30").join(broadcast(deptDF), "deptId")
   ```

2. **统一 API**：DataFrame 统一了 Spark SQL、机器学习、图计算的 API

3. **内存列式存储（ Tungsten）**：使用列式内存格式，减少数据访问开销

4. **代码示例（Scala）**：
   ```scala
   case class Person(name: String, age: Int)
   
   // DataFrame
   val df = spark.read.parquet("people.parquet")
   df.filter($"age" > 30).show()
   
   // DataSet（强类型）
   val ds = spark.read.parquet("people.parquet").as[Person]
   ds.filter(_.age > 30).show()
   ```

### DataSet 的优势

- **类型安全**：编译时即可检查类型错误
- **面向对象**：直接使用 lambda 或方法，操作更自然
- **DataFrame 的优化**：享受 DataFrame 所有优化

### 选择建议

- **Python 用户**：只能使用 DataFrame（Python 没有 DataSet 类型安全）
- **Scala/Java 用户**：推荐 DataSet（有类型安全 + 优化）
- **结构化数据**：使用 DataFrame 或 DataSet，而非 RDD

## 题目五：Spark 的内存管理机制是怎样的？Executor 内存如何划分？什么是广播变量？

### 参考答案

### Spark 内存管理机制

Spark 使用 **Unified Memory Management**（统一内存管理），将堆内存统一划分为两部分：**存储内存** 和 **执行内存**，两者可以动态借用。

### Executor 内存区域划分

```
┌────────────────────────────────────────────────┐
│                   Executor 内存                  │
├──────────────────────┬─────────────────────────┤
│    存储内存 (Storage)    │     执行内存 (Execution)     │
│   - RDD 缓存           │   - Shuffle 排序/聚合       │
│   - Broadcast 变量      │   - Hash Join           │
│   - DataFrame 列式存储   │   - UDF 执行              │
├──────────────────────┼─────────────────────────┤
│              系统/用户内存/预留                    │
└──────────────────────┴─────────────────────────┘
```

**关键配置参数**：
- `spark.memory.fraction = 0.6`（Executor 内存中用于存储+执行的比例，默认 60%）
- `spark.memory.storageFraction = 0.5`（存储/执行区域各自的初始比例，可动态调整）
- `spark.executor.memory`（Executor 总内存大小）

### 动态内存管理规则

- 当 **执行内存 > 存储内存** 时，执行内存可以借用存储内存（反之亦然）
- 存储内存被占用时，缓存的 RDD 数据会被淘汰（LRU 策略）
- 执行内存被占用时，无法借用存储内存（避免执行失败）

### 广播变量（Broadcast Variables）

广播变量用于将 **只读的大数据** 广播到每个节点，避免在每个 Task 中重复传输。

**使用示例**：
```python
# 广播小表
broadcasted_df = broadcast(small_df)

# 在 join 中使用
result = large_df.join(broadcasted_df, "key")
```

**优势**：
- 减少网络传输开销（数据只传输一次到每个节点）
- 加快 join 操作速度
- 节省集群带宽

**注意事项**：
- 广播变量不适合大到单机放不下的数据
- 广播的变量是只读的，更新需要重新广播
- Spark 自动选择是否广播（`spark.sql.autoBroadcastJoinThreshold`，默认 10MB）

### RDD Persistence（持久化）级别

| 级别 | 空间 | CPU | 内存 | 磁盘 |
|------|------|-----|------|------|
| MEMORY_ONLY | 高 | 低 | ✅ | ❌ |
| MEMORY_AND_DISK | 高 | 中 | ✅ | ✅ |
| MEMORY_ONLY_SER | 低 | 高 | ✅ | ❌ |
| DISK_ONLY | 低 | 高 | ❌ | ✅ |
| OFF_HEAP | 低 | 高 | ✅ | ✅ |
