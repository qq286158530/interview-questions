## 题目一：Flink 的架构是怎样的？为什么说 Flink 是真正的流计算引擎？

### 参考答案

Flink 架构采用 **Master-Slave 分布式架构**，核心组件包括：

| 组件 | 职责 |
|------|------|
| **JobManager** | 作业调度、Checkpoint 协调、TaskManager 资源管理 |
| **TaskManager** | 实际执行 Task（算子实例），提供 slot 资源 |
| **ResourceManager** | 管理 TaskManager slots，动态分配资源 |
| **Dispatcher** | 接收作业提交，启动 JobManager |

**作业执行流程：**
1. 用户代码 → StreamGraph（算子拓扑）
2. StreamGraph → JobGraph（算子链化、减少数据传输）
3. JobGraph 提交给 JobManager
4. JobManager 向 ResourceManager 申请 slot
5. TaskManager 提供 slot，JobManager 分发 JobGraph 到 TaskManager 执行

**为什么 Flink 是真正的流计算引擎：**
- **原生流处理**：DataStream API 基于流处理模型，数据到达即处理，无需转为微批
- **事件时间语义**：支持 Event Time，提供精确一次（Exactly-Once）保障
- **乱序处理**：通过 Watermark 机制处理乱序数据
- **低延迟高吞吐**：采用流水线数据传输，延迟可达毫秒级

> **来源：** [Apache Flink 官方文档 - 架构](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/flink-architecture/)

## 题目二：Watermark（水位线）的作用是什么？如何处理乱序数据？

### 参考答案

**Watermark 本质**：Watermark 是时间戳上的一个标记，表示"所有时间戳 ≤ Watermark 的数据都已到达"。

**核心作用**：解决分布式流处理中的**乱序延迟**问题。

**乱序场景**：
- 网络抖动导致数据延迟
- 分布式源并行读取导致顺序不确定
- 机器故障/重启导致数据回溯

**Watermark 机制**：
```
Watermark = max(当前最大事件时间) - 允许延迟时间
```

**窗口触发时机**：当 Watermark 时间 ≥ 窗口结束时间，窗口才触发计算。

**代码示例**：
```java
// 允许 5 秒延迟的 Watermark 策略
WatermarkStrategy<Tuple2<String, Long>> wms = WatermarkStrategy
    .<Tuple2<String, Long>>forBoundedOutOfOrderness(Duration.ofSeconds(5))
    .withTimestampAssigner((event, timestamp) -> event.f1);

// 在数据流上应用 Watermark
DataStream<Tuple2<String, Long>> stream = env
    .fromSource(kafkaSource, wms, "Kafka Source");
```

**三种 Watermark 策略**：
| 策略 | 说明 |
|------|------|
| **FixedWatermark** | 固定延迟 |
| **BoundedOutOfOrderness** | 允许一定乱序 |
| **MonotonousWatermark** | 单调递增（完美顺序） |

**超时处理**：窗口长时间未触发可通过 `allowedLateness()` 设置允许延迟时间，超时后通过 `sideOutputLateData()` 侧输出。

> **来源：** [Apache Flink 官方文档 - Watermark](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/time-and-watermarks/)

## 题目三：Flink Checkpoint 和 Savepoint 的区别是什么？如何实现精确一次（Exactly-Once）语义？

### 参考答案

**Checkpoint vs Savepoint 对比**：

| 特性 | Checkpoint | Savepoint |
|------|------------|-----------|
| **触发方式** | Flink 自动周期性触发 | 手动触发 |
| **用途** | 故障恢复 | 作业升级、迁移、暂停 |
| **依赖** | 依赖 Checkpoint 机制 | 依赖 Savepoint 机制 |
| **删除** | Job 完成后自动删除 | 手动删除 |
| **兼容性** | 状态可能不兼容 | 通常兼容（除非算子改变） |
| **存储** | 临时目录 | 用户指定目录 |

**实现 Exactly-Once 的核心机制**：

1. **分布式快照（Chandy-Lamport 变体）**：
   - Barrier（屏障）在数据流中携带
   - 所有算子收到所有输入的 Barrier 后做快照
   - 快照完成后才向下游发送 Barrier

2. **两阶段提交（2PC）**：
   - **预提交**：算子完成计算，准备提交
   - **正式提交**：所有算子确认后提交
   - Kafka Sink 使用 `TwoPhaseCommitSinkFunction` 实现

3. **Barrier 对齐**：
   ```
   算子收到 InputGate1 的 Barrier
          ↓
   缓存 InputGate1 数据，等待 InputGate2 Barrier
          ↓
   收到所有 Barrier → 做快照 → 发送 Barrier
   ```

**代码示例（启用 Checkpoint）**：
```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
env.enableCheckpointing(10_000); // 每 10 秒一次
env.getCheckpointConfig().setCheckpointStorage("hdfs:///checkpoint/");
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(500);
env.getCheckpointConfig().setCheckpointTimeout(60_000);
env.getCheckpointConfig().setTolerableCheckpointFailureNumber(3);

// 精确一次语义
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);
```

> **来源：** [Apache Flink 官方文档 - Checkpoint](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/checkpoints/)

## 题目四：Flink 窗口函数有哪些？滚动窗口、滑动窗口、会话窗口的区别？

### 参考答案

**窗口类型总览**：

| 窗口类型 | 特点 | 适用场景 |
|----------|------|----------|
| **滚动窗口（Tumbling）** | 固定大小、不重叠 | 固定周期的聚合统计 |
| **滑动窗口（Sliding）** | 固定大小、可重叠 | 需要平滑连续的聚合 |
| **会话窗口（Session）** | 动态大小、间隙触发 | 用户行为分析 |
| **全局窗口（Global）** | 所有数据一个窗口 | 全量聚合 |

**1. 滚动窗口（TumblingWindow）**
```java
// 每 5 分钟统计一次
DataStream<Tuple2<String, Long>> windowed = stream
    .keyBy(0)
    .window(TumblingProcessingTimeWindows.of(Time.minutes(5)))
    .sum(1);
```

**2. 滑动窗口（SlidingWindow）**
```java
// 窗口大小 10 分钟，滑动步长 5 分钟
DataStream<Tuple2<String, Long>> windowed = stream
    .keyBy(0)
    .window(SlidingProcessingTimeWindows.of(Time.minutes(10), Time.minutes(5)))
    .sum(1);
```

**3. 会话窗口（SessionWindow）**
```java
// 会话间隔 3 分钟
DataStream<Tuple2<String, Long>> windowed = stream
    .keyBy(0)
    .window(ProcessingTimeSessionWindows.withGap(Time.minutes(3)))
    .sum(1);

// 动态间隙会话窗口
.window(SessionWindows.withDynamicGap(new SessionWindowTimeGapExtractor<Tuple2<String, Long>>() {
    @Override
    public double extract(Tuple2<String, Long> element) {
        return element.f1; // 根据数据动态确定间隙
    }
}));
```

**窗口函数类型**：
| 函数 | 说明 |
|------|------|
| **ReduceFunction** | 增量聚合，效率高，输出类型需与输入一致 |
| **AggregateFunction** | 更加灵活，可定义中间状态类型 |
| **ProcessWindowFunction** | 全量窗口数据，获取窗口上下文信息 |

**最佳实践**：滚动窗口适合周期性统计，滑动窗口适合趋势分析，会话窗口适合用户行为路径分析。

> **来源：** [Apache Flink 官方文档 - Windows](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream_api/windows/)

## 题目五：Flink 状态管理机制是怎样的？KeyedState 和 OperatorState 的区别？

### 参考答案

**状态类型划分**：

```
Flink State
├── KeyedState（键控状态）
│   ├── ValueState<T>
│   ├── ListState<T>
│   ├── ReducingState<T>
│   ├── AggregatingState<IN, OUT>
│   └── MapState<UK, UV>
│
└── OperatorState（非键控状态）
    ├── ListState<T>
    ├── UnionListState<T>
    ├── BroadcastState<K, V>
    └── ValueState<T>
```

**KeyedState vs OperatorState 对比**：

| 特性 | KeyedState | OperatorState |
|------|------------|---------------|
| **作用域** | 绑定到 Key，只能在 `keyBy` 后使用 | 算子全局作用域 |
| **存储方式** | 按 Key 分区存储 | 按算子实例存储 |
| **状态分发** | 根据 Key 自动路由到对应分区 | 支持 List、Union、Broadcast 模式 |
| **使用场景** | 状态需要按 Key 隔离 | 状态与 Key 无关（如 Kafka offset） |

**代码示例**：
```java
public class CountWindowAverage extends RichFlatMapFunction<Tuple2<String, Long>, Tuple2<String, Double>> {
    
    // KeyedState：按 key 维护状态
    private ValueState<Count> state;
    
    @Override
    public void open(Configuration parameters) {
        state = getRuntimeContext().getState(
            new ValueStateDescriptor<>("myCount", Count.class)
        );
    }
    
    @Override
    public void flatMap(Tuple2<String, Long> input, Collector<Tuple2<String, Double>> out) {
        Count current = state.value();
        if (current == null) {
            current = new Count();
            current.count = 0;
            current.sum = 0;
        }
        
        current.count += 1;
        current.sum += input.f1;
        state.update(current);
        
        if (current.count == 10) {
            out.collect(Tuple2.of(input.f0, current.sum / (double) current.count));
            state.clear();
        }
    }
}
```

**OperatorState 示例（Kafka Offset）**：
```java
public class KafkaOffsetConsumer extends RichSourceFunction<String> implements CheckpointedFunction {
    
    private transient ListState<Long> checkpointedOffset;
    private long offset = 0;
    
    @Override
    public void initializeState(FunctionInitializationContext context) throws Exception {
        // OperatorState
        checkpointedOffset = context.getOperatorStateStore().getListState(
            new ListStateDescriptor<>("kafkaOffset", Long.class)
        );
    }
    
    @Override
    public void snapshotState(FunctionSnapshotContext context) throws Exception {
        checkpointedOffset.clear();
        checkpointedOffset.add(offset);
    }
}
```

**状态后端选择**：
| 后端 | 适用场景 | 特点 |
|------|----------|------|
| **HashMapStateBackend** | 小状态、低延迟 | 内存存储，RocksDB 更适合大状态 |
| **EmbeddedRocksDBStateBackend** | 大状态、长窗口 | 增量 Checkpoint，磁盘存储 |
