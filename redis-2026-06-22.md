## 题目一：Redis 数据类型有哪些？各自的使用场景是什么？

### 参考答案

Redis 提供了丰富的数据类型，主要分为 **5种基本数据类型** 和 **4种扩展数据类型**：

### 五种基本数据类型

| 类型 | 使用场景 |
|------|---------|
| **String** | 缓存对象、常规计数、分布式锁、共享 session 信息 |
| **List** | 消息队列（需自行实现全局唯一ID，不支持消费组形式） |
| **Hash** | 缓存对象、购物车 |
| **Set** | 聚合计算（并集、交集、差集）场景，如点赞、共同关注、抽奖 |
| **Zset** | 排序场景，如排行榜、电话姓名排序 |

### 四种扩展数据类型

| 类型 | 使用场景 |
|------|---------|
| **BitMap** | 二值状态统计，如签到、判断用户登录状态、连续签到用户总数 |
| **HyperLogLog** | 海量数据基数统计，如百万级网页 UV 计数 |
| **GEO** | 存储地理位置信息，如滴滴叫车 |
| **Stream** | 消息队列，相比 List 支持自动生成全局唯一消息ID，支持消费组形式消费 |

### 底层数据结构

Redis 五种数据类型的底层实现：

- **String**：SDS（简单动态字符串），O(1) 获取长度，支持二进制数据
- **List**：Redis 3.2+ 仅用 quicklist（双向链表+压缩列表）
- **Hash**：压缩列表（元素<512且每个<64字节）或哈希表
- **Set**：整数集合（全是整数且<512个）或哈希表
- **Zset**：压缩列表（元素<128且每个<64字节）或跳表

## 题目二：Redis 持久化机制是什么？RDB 和 AOF 有什么区别？

### 参考答案

Redis 共有 **三种数据持久化方式**：

### 1. AOF 日志（Append Only File）

每执行一条写操作命令，就把该命令以追加的方式写入文件。Redis 重启时读取 AOF 文件的命令逐一执行来恢复数据。

**写回策略（appendfsync）**：

| 策略 | 说明 | 安全性 | 性能 |
|------|------|--------|------|
| **Always** | 每次写命令执行完后同步写回硬盘 | 最安全，数据零丢失 | 低 |
| **Everysec** | 每秒将缓冲区内容写回硬盘 | 最多丢失1秒数据 | 中 |
| **No** | 由操作系统控制何时写回 | 依赖系统，可能丢失较多 | 高 |

**AOF 重写机制**：当 AOF 文件过大时，后台子进程 `bgrewriteaof` 会读取当前数据库中所有键值对，用一条命令记录到新的 AOF 文件，从而压缩文件体积。重写期间主进程的新命令同时写入 "AOF缓冲区" 和 "AOF重写缓冲区"，保证数据一致性。

### 2. RDB 快照（Redis Database）

记录某一时刻的内存数据，以二进制方式写入磁盘。

- **save 命令**：在主线程执行，会阻塞
- **bgsave 命令**：创建子进程生成 RDB，避免阻塞（采用 COW 写时复制技术）

### 3. 混合持久化（Redis 4.0+）

结合 RDB 和 AOF 的优点。AOF 重写时，先以 RDB 格式写入全量数据，再以 AOF 格式增量写入重写缓冲区中的命令。新 AOF 文件前半部分是 RDB 格式（快速加载），后半部分是 AOF 格式（减少数据丢失）。

### RDB vs AOF 对比

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件体积 | 小（二进制） | 大（命令文本） |
| 恢复速度 | 快 | 慢（全量执行命令） |
| 数据安全性 | 可能丢失较多数据 | 可配置（always到everysec到no） |
| 资源消耗 | 快照瞬间可能阻塞 | 持续AOF写入消耗 |
| 优先级 | 恢复快 | 丢失少 |

## 题目三：Redis 是单线程吗？为什么单线程还这么快？

### 参考答案

### Redis 单线程指的是什么

Redis 单线程指的是 **「接收客户端请求→解析请求→进行数据读写等操作→发送数据给客户端」** 这一过程由一个线程（主线程）来完成。

但 Redis 程序并不是完全单线程的，启动时会启动后台线程（BIO）：

- **Redis 2.6**：2个后台线程 — 关闭文件、AOF刷盘
- **Redis 4.0+**：新增 lazyfree 线程，异步释放内存（unlink / flushdb async）

### 为什么单线程还这么快？

1. **内存操作**：Redis 大部分操作都在内存中完成，且采用高效数据结构（SDS、跳表等），瓶颈是内存或网络带宽而非 CPU
2. **无竞争开销**：单线程避免了多线程切换开销和锁竞争，不会有死锁问题
3. **I/O 多路复用**：使用 epoll/select 机制，一个线程处理多个 Socket 请求，内核同时监听多个 Socket 上的连接请求或数据请求

### Redis 6.0 引入多线程

Redis 6.0 对**网络 I/O** 引入多线程（默认3个 I/O 线程），分担网络读写压力。但**命令执行仍然是单线程**的。I/O 多线程默认只处理响应数据（write），读请求（read）需配置 `io-threads-do-reads yes` 开启。

## 题目四：Redis 的过期删除策略和内存淘汰策略有什么区别？

### 参考答案

### 过期删除策略

Redis 使用 **惰性删除 + 定期删除** 组合策略：

**惰性删除**：每次访问 key 时才检查是否过期，过期则删除。
- ✅ 对 CPU 时间友好，开销小
- ❌ 对内存不友好，过期 key 长期占用内存

**定期删除**：每隔一段时间随机从数据库中抽取 20 个 key 检查，删除已过期的。如果过期 key 比例超过 25%，继续抽取；低于 25% 则停止。单次执行时间不超过 25ms。
- ✅ 释放过期内存，对 CPU 和内存都较友好
- ❌ 删除频率不好把控

### 内存淘汰策略（maxmemory-policy）

当 Redis 运行内存达到 `maxmemory` 阈值时，触发淘汰机制，共有 8 种策略：

| 策略 | 行为 |
|------|------|
| `noeviction` | 不淘汰，返回错误（默认） |
| `volatile-lru` | 从已设置过期时间的 key 中淘汰最近最少使用 |
| `volatile-lfu` | 从已设置过期时间的 key 中淘汰最不经常使用 |
| `volatile-ttl` | 从已设置过期时间的 key 中淘汰即将过期的 |
| `volatile-random` | 从已设置过期时间的 key 中随机淘汰 |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用 |
| `allkeys-lfu` | 从所有 key 中淘汰最不经常使用 |
| `allkeys-random` | 从所有 key 中随机淘汰 |

### 两者区别

- **过期删除**：针对设置了 TTL 的 key，由惰性删除和定期删除配合，主动删除过期键
- **内存淘汰**：当内存达到 maxmemory 限制时触发，被动淘汰任何 key（包括未设置过期时间的）

## 题目五：Redis 主从复制、哨兵模式和集群模式有什么区别？如何应对脑裂问题？

### 参考答案

### 三种高可用方案对比

| 模式 | 原理 | 读写分离 | 故障自动恢复 |
|------|------|---------|-------------|
| **主从复制** | 一主多从，数据异步同步 | 主写从读 | ❌ 需手动恢复 |
| **哨兵模式** | 基于主从复制 + 哨兵监控 | 主写从读 | ✅ 自动故障转移 |
| **集群模式** | 16384 哈希槽分片 | 各自处理槽内读写 | ✅ 自动分片和故障转移 |

### 主从复制

主节点处理写操作并异步同步给从节点，从节点默认只读。命令复制是异步的，无法保证强一致性，可能丢失数据。

### 哨兵模式（Sentinel）

哨兵监控主从节点健康，当主节点客观下线时，从从节点中选举一个新主节点并完成故障转移。客户端通过哨兵获取当前主节点地址。

### Redis Cluster（切片集群）

共有 **16384 个哈希槽**，通过 `CRC16(key) % 16384` 决定 key 所属槽位，槽位分配到不同节点。数据分布在不同节点上，降低单节点压力，提高读写性能。

### 脑裂问题及解决方案

**脑裂现象**：主节点网络故障但与客户端连接正常，哨兵误判主节点宕机并选举新主节点，此时存在两个主节点。当网络恢复后，旧主节点降级为从节点并执行全量同步，导致之前客户端写入的数据全部丢失。

**解决方案**：配置 `min-slaves-to-write` 和 `min-slaves-max-lag`

```conf
min-slaves-to-write 1    # 主节点至少要有1个从节点
min-slaves-max-lag 12    # 复制和 ACK 延迟不能超过12秒
```

当主节点发现从节点数量不足或延迟超过阈值时，直接拒绝客户端写请求，防止在"假故障"期间写入数据。等新主库上线后，旧主库降级为从库，数据从新主库同步，不会有新数据丢失。
