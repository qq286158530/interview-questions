## 题目 1：HBase 的整体架构是怎样的？请描述 RegionServer、Region、HMaster 的作用及数据读写流程。

### 参考答案

HBase 采用 **Master-Slave 架构**，核心组件如下：

| 组件 | 作用 |
|------|------|
| **HMaster** | 负责表的 CRUD 操作、Region 的分配与调度、负载均衡、故障恢复 |
| **RegionServer** | 负责数据的读写服务，每个 RegionServer 管理多个 Region |
| **Region** | 表的数据分片，按 RowKey 范围划分，一个 Region 默认约 1-2GB |
| **ZooKeeper** | 负责分布式协调，保存元数据位置、Master 地址等关键信息 |

**读写流程：**

1. **写流程：**
   - 客户端先访问 ZooKeeper，获取 `.META.` 表所在的 RegionServer 地址
   - 根据 RowKey 从 `.META.` 表找到目标数据所在的 RegionServer
   - 数据先写入 **MemStore**（内存缓冲区），同时写入 **HLog（WAL）** 防止宕机丢失
   - MemStore 写满后触发 **Flush** 成 **HFile** 落盘

2. **读流程：**
   - 客户端通过 ZooKeeper → `.META.` 表定位目标 RegionServer
   - 在 BlockCache（读缓存）+ MemStore + HFile 中合并查找数据
   - 先查 MemStore（未 flush 的最新数据），再查 HFile（按时间倒序合并）

**特点：** HBase 读写都不经过 Master，Master 仅负责元数据管理和调度，这是其能做到高性能的原因之一。

## 题目 2：HBase 中 RowKey 设计有哪些原则？为什么 RowKey 设计至关重要？

### 参考答案

**RowKey 设计四大原则：**

1. **长度原则**：RowKey 定长，建议不超过 16 字节，越短越好
   - HBase 索引（RowKey + ColumnFamily + Qualifier + Timestamp）都存在内存中，过长的 RowKey 浪费内存

2. **散列原则**：RowKey 必须保证分布均匀，避免热点（Hot Spotting）
   - ❌ 错误示例：以时间戳开头 → 所有请求集中在一个 Region
   - ✅ 正确做法：反转时间戳、或在开头加随机前缀、或使用 Hash

3. **唯一原则**：RowKey 是 HBase 中数据的唯一主键，类似于 RDBMS 的主键

4. **业务查询原则**：将高频查询条件放在 RowKey 前缀
   - 例如：查询某用户订单 → `userId_orderId`，而非 `orderId_userId`

**热点场景及解决方案：**

| 热点场景 | 解决方案 |
|----------|----------|
| 时间顺序写入（如 IoT 设备日志） | 盐值（Salting）+ 随机前缀 |
| 固定前缀查询（如某商家订单） | 哈希打散 + 业务字段组合 |
| 大范围扫描 | 设计多层索引或使用 Coprocessor |

## 题目 3：简述 HBase 的存储结构（MemStore、HFile），以及 Flush 的触发条件和流程。

### 参考答案

**核心存储结构：**

```
RegionServer
├── MemStore（写缓存，JVM 堆内）
├── BlockCache（读缓存，LIRS 算法）
└── HFiles（磁盘文件，LSM-Tree 结构）
    └── HFile = Data Block + Index Block + Bloom Filter + Meta Block
```

**MemStore：**
- 是 Region 级别的写缓存，按 ColumnFamily 分开
- 数据按 RowKey 排序（内存中维护有序结构）
- 达到以下任一条件触发 Flush：
  - `hbase.regionserver.global.memstore.size.lower.global`（默认堆 35%）
  - 单个 MemStore 大小超过 `hbase.hregion.memstore.flush.size`（默认 128MB）
  - WAL 文件数量超过 `hbase.regionserver.max.logs`（默认 32）
  - 定期刷新线程（超时 `hbase.regionserver.optionalcacheflushinterval`，默认 1 小时）

**HFile：**
- Flush 后生成 HFile（Immutable，顺序写）
- HBase 使用 **LSM-Tree（Log-Structured Merge-Tree）** 结构：
  - 写入：先写 WAL，再写 MemStore（顺序写入，速度快）
  - 合并：Minor Compaction 合并多个小 HFile，Major Compaction 合并所有 HFile 并删除 tombstone 标记的删除数据

**为什么 MemStore 要排序后写入 HFile？**
- 保证 HFile 中的数据按 RowKey 有序，支持高效的范围查询和二分查找
- 同时生成 Bloom Filter 和 Index Block，加速查找

> - 《LSM-Tree 与 HBase  compaction》：https://www.alibabacloud.com/blog/lsm-tree-based-storage-techniques_594473

## 题目 4：什么是 HBase 协处理器（Coprocessor）？有哪些类型？分别适用于什么场景？

### 参考答案

协处理器是 HBase 提供的 **运行在 RegionServer 端的扩展机制**，允许用户自定义代码在数据读写路径上执行，无需修改 HBase 核心源码。

**两种主要类型：**

### 1. Observer Coprocessor（观察者，类似触发器）

- 插入到读写链路的关键 Hook 点，自动触发执行
- 典型 Hook 点：
  - `preGet` / `postGet` — GET 请求前后
  - `prePut` / `postPut` — PUT 写入前后
  - `preDelete` / `postDelete` — DELETE 删除前后
  - `preOpen` / `postOpen` — Region 打开时
  - `preSplit` / `postSplit` — Region 分裂前后

**应用场景：**
- **二级索引**：在 put 时自动写入索引表（Phoenix 的实现方式）
- **数据验证**：写入前校验数据合法性（如金额不能为负）
- **审计日志**：记录每次修改的操作人、时间
- **实时聚合**：维护计数器（如统计 PV/UV）

### 2. Endpoint Coprocessor（端点，类似存储过程）

- 支持在 RegionServer 上执行自定义的 RPC 调用，类似分布式计算
- 典型场景：
  - 服务器端聚合计算（求和、平均）
  - 自定义 Scan 逻辑

**加载方式：**
| 方式 | 配置位置 | 作用域 |
|------|----------|--------|
| 加载到所有表 | `hbase-site.xml` | 全局 |
| 加载到单张表 | 表描述符（TableDescriptor）| 表级别 |

> 📎 参考来源：
> - HBase 官方 Coprocessor 文档：https://hbase.apache.org/book.html#cp
> - 《HBase 协处理器实战》：https://blog.csdn.net/zouhuanli/article/details/123419819

## 题目 5：HBase 如何保证数据一致性和高可用？WAL 的作用是什么？

### 参考答案

**WAL（Write-Ahead Log）作用：**

HBase 采用 **先写日志再写内存（Write-Ahead Logging）** 的策略：
```
客户端写数据 → ① 写入 WAL（HLog） → ② 写入 MemStore → 返回成功
```
- WAL 保存在 HDFS 上，默认 3 副本复制
- 当 MemStore 未 Flush 就宕机时，WAL 可用于恢复数据
- **是 HBase 容错能力的核心保障**

**数据一致性：**

HBase 提供 **强一致性** 保证：
- 写入：数据先写入 WAL + MemStore，Flush 后变成 HFile，多副本保证不丢失
- 读取：Region 级别强一致（同一 Region 的读写串行执行）
- 原子性：单行数据（同一 RowKey + 所有列）的读写是原子的（Per Row）

**高可用（HA）架构：**

```
┌──────────┐     ┌──────────────────┐
│  HMaster │◄────│   ZooKeeper      │
└──────────┘     │ (Master 元数据)  │
      ▲          └──────────────────┘
      │                   ▲
      │            ┌──────┴──────┐
      │            │  RegionServers │
      │            │  (数据节点)    │
      │            └─────────────┘
```

- **HMaster HA**：通过 ZooKeeper 选举，主 Master 宕机后备 Master 自动接管
- **RegionServer HA**：每个 Region 的数据在 HDFS 上有 3 副本，某 RS 宕机后 Master 将其 Region 迁移到其他节点
- **WAL 保活**：宕机的 RegionServer 上的 WAL 会被其他 RS 重放（Replay），恢复未 Flush 的数据

**故障恢复流程：**
1. ZooKeeper 检测到 RegionServer 心跳超时
2. HMaster 将该 RS 上的 Region 分配到其他节点
3. 其他 RS 重放（Replay）WAL 文件，恢复 MemStore 数据
4. 客户端重新定位到新的 RegionServer 继续读写

> - 《HBase WAL 与数据恢复机制》：https://cloud.tencent.com/developer/article/1807497
