# Spark 面试题 — 2026-07-22

## 题目一：Spark 架构核心组件及作业执行流程

### 📌 问题
请描述 Spark 的整体架构包含哪些核心组件，以及一个 Spark 作业从提交到执行完成的完整流程。

### ✅ 答案

**Spark 核心组件：**

| 组件 | 作用 |
|------|------|
| **Driver（驱动器）** | 运行用户代码的 main() 函数，创建 SparkContext，负责与 Cluster Manager 通信，调度 Task 分发到 Executor |
| **Executor（执行器）** | 运行在 Worker 节点上的进程，负责执行 Task 并将结果返回给 Driver，同时缓存计算数据 |
| **Cluster Manager** | 集群资源管理器（Standalone / YARN / Kubernetes），负责分配集群资源 |
| **Worker Node** | 集群中运行 Executor 的节点 |
| **SparkContext** | Spark 应用的入口点，负责与 Cluster Manager 通信并获取资源 |

**作业执行流程：**

1. **提交阶段**：用户通过 `spark-submit` 提交应用，Driver 启动并创建 SparkContext
2. **资源申请**：SparkContext 向 Cluster Manager 注册并申请 Executor 资源
3. **任务调度**：Cluster Manager 分配 Worker 节点，Executor 启动并向 Driver 注册
4. **DAG 构建**：Driver 根据用户代码构建 DAG（有向无环图），将 DAG 划分为 Stage
5. **Stage 划分**：DAGScheduler 将 DAG 划分为多个 Stage，每个 Stage 包含多个可以并行执行的 Task
6. **Task 分发**：TaskScheduler 将 Task 分发到各个 Executor 执行
7. **结果回收**：Executor 执行 Task，结果逐级汇总返回给 Driver
8. **完成退出**：SparkContext 注销，资源释放

**shuffle 操作会将 DAG 划分为不同的 Stage，这是面试中的关键追问点。**

## 题目二：RDD 的特性、创建方式及 Transformation 与 Action 的区别

### 📌 问题
什么是 RDD？RDD 有哪些核心特性？如何创建 RDD？Transformation 和 Action 操作有什么区别？请举例说明。

### ✅ 答案

**RDD（Resilient Distributed Dataset）** 是 Spark 最基本的数据抽象，表示弹性分布式数据集，不可变、可分区、可并行计算。

**RDD 五大核心特性：**

1. **分区列表（Partitions）**：RDD 由多个分区组成，每个分区可以并行计算
2. **计算函数（Compute Function）**：每个分区上执行的计算逻辑
3. **依赖关系（Dependencies）**：RDD 之间的依赖关系（窄依赖 / 宽依赖）
4. **分区器（Partitioner）**：对于 Key-Value RDD，可指定分区策略（如 HashPartitioner）
5. **位置偏好（Preferred Locations）**：数据块的物理位置偏好，利于数据本地性

**RDD 创建方式：**

```python
# 1. 从外部存储创建
rdd = sc.textFile("hdfs://path/to/file")

# 2. 从 Scala/Python 集合并行化创建
rdd = sc.parallelize([1, 2, 3, 4, 5], 3)  # 第二个参数为分区数

# 3. 从其他 RDD 转换而来
rdd2 = rdd.map(lambda x: x * 2)
```

**Transformation vs Action 区别：**

| 特征 | Transformation | Action |
|------|---------------|--------|
| 返回类型 | 新的 RDD | 其他数据类型 |
| 执行时机 | 懒执行（Lazy） | 立即执行（Eager） |
| 示例 | `map`, `filter`, `flatMap`, `groupByKey` | `collect`, `count`, `reduce`, `saveAsTextFile` |
| 链式调用 | 可以链式调用，形成 DAG | 触发 DAG 执行 |

```python
# Transformation - 懒执行，不立即执行
lines = sc.textFile("data.txt")          # 不执行
errors = lines.filter(lambda s: "error" in s)  # 不执行

# Action - 触发真正的计算
count = errors.count()                   # 此时才执行
```

**面试追问**：为什么 Spark 采用懒执行？—— 为了优化执行计划，通过 DAG 分析进行 Pipeline 优化、避免中间结果物化，节省资源。

## 题目三：DataFrame 和 DataSet 的区别，以及与 RDD 的对比

### 📌 问题
请说明 Spark 中 DataFrame 和 DataSet 的区别，以及它们与 RDD 相比各自的优势是什么？

### ✅ 答案

**三者对比：**

| 特性 | RDD | DataFrame | DataSet |
|------|-----|-----------|---------|
| **数据结构** | 无结构，每行元素自行解析 | 类表格结构，有 Schema（列名+类型） | 有类型的 DataFrame，强类型 |
| **API 风格** | 底层函数式 API | 高级声明式 DSL | 高级声明式 DSL + 类型安全 |
| **编译时检查** | 无 | 无（运行时才发现错误） | 有（编译时检查） |
| **性能优化** | 无自动优化（依赖用户） | 有（Catalyst + Tungsten） | 有（与 DataFrame 相同） |
| **数据处理方式** | 逐行处理 | 按列式存储批量处理 | 按列式存储批量处理 |
| **序列化** | Java/Kotlin 序列化，效率低 | Off-Heap  Tungsten 序列化，高效 | 与 DataFrame 相同 |

**DataFrame vs DataSet 的关键区别：**

```
DataFrame = DataSet[Row]  （无类型的行集合）
DataSet   = DataFrame 的强类型版本
```

- **DataFrame**：每一行是 `Row` 类型，列访问通过 `df("colName")`，编译时不检查列名
- **DataSet**：每行是具体的样例类（如 `case class Person(name: String, age: Int)`），编译时类型安全

```scala
// DataFrame - 弱类型
val df = spark.read.json("people.json")
df.filter(df("age") > 21).show()

// DataSet - 强类型
case class Person(name: String, age: Int)
val ds = spark.read.json("people.json").as[Person]
ds.filter(_.age > 21).show()
```

**Catalyst 优化器工作流程：**
1. Unresolved Logical Plan（未解析的逻辑计划）
2. → Analyzed Logical Plan（分析阶段）
3. → Optimized Logical Plan（优化阶段，如谓词下推、列裁剪）
4. → Physical Plan（物理执行计划）
5. → RDD operations（生成最终 RDD）

**优势**：用户无需关心执行计划优化，Spark 自动完成谓词下推、列裁剪、常量替换等优化。

## 题目四：Shuffle 原理、Shuffle Manager 及常见优化策略

### 📌 问题
什么是 Shuffle？为什么 Shuffle 是 Spark 性能的关键瓶颈？请说明 Shuffle 的工作原理，以及常见的优化策略。

### ✅ 答案

**什么是 Shuffle：**

Shuffle 是 Spark 中**重新分区数据**的操作，将数据按 Key 进行重新分布，涉及跨节点的数据传输。当一个 RDD 依赖链中出现「同一个 Key 的数据可能分布在不同的分区」时，需要 Shuffle。

**触发 Shuffle 的算子**： `groupByKey`、`reduceByKey`、`join`、`sortByKey`、`cogroup`、`distinct` 等。

**Shuffle 工作原理（Spark 2.x 之后基于 Sort-Based Shuffle）：**

```
          Mapper 端                                    Reducer 端
    ┌─────────────────┐                          ┌─────────────────┐
    │  ShuffleMapTask │                          │  ShuffleMapTask │
    │  (每个分区一个)   │                          │  (每个分区一个)   │
    └────────┬────────┘                          └────────┬────────┘
             │ 写入本地磁盘                                  │
             │ 按 partitionId 分区                            │
             │ 每个分区内按 Key 排序                          │
             ▼                    网络传输 ▼                  ▼
    ┌──────────────────────────────────────────────────────────┐
    │              Shuffle Files（磁盘文件）                     │
    │  partition 0: [key1, key2, ...]                          │
    │  partition 1: [key1, key3, ...]                          │
    └──────────────────────────────────────────────────────────┘
```

**Shuffle Manager 三种实现：**

| Manager | 特点 | 备注 |
|---------|------|------|
| **Hash Shuffle Manager** | 每个 Mapper 为每个 Reducer 生成一个文件，文件数 = M × R | 小文件过多，内存压力大，已废弃 |
| **Sort Shuffle Manager** | 每个 Mapper 生成一个文件，按 Partition ID 排序 | 主流选择，文件数 = 2 × M |
| **Tungsten Sort Shuffle** | 基于 Tungsten 优化的排序 Shuffle | 高效序列化，减少内存开销 |

**Shuffle 常见性能问题：**

- **磁盘 I/O 开销**：大量小文件读写
- **网络传输**：跨节点数据传输
- **内存压力**：Map 端排序缓存、Aggregator 内存

**常见优化策略：**

1. **减少 Shuffle 数据量**
   - 使用 `mapSideCombine` 替代 `groupByKey`（`reduceByKey` 会在 Map 端先聚合）
   - 过滤无关数据，尽量在 Shuffle 前过滤

2. **增加并行度**
   ```scala
   spark.conf.set("spark.sql.shuffle.partitions", "200")  // 默认200，可调大
   ```

3. **使用广播变量**
   - 小表广播到各个节点，避免 Shuffle（适用于 Join 场景）
   ```scala
   val broadcastVar = sc.broadcast(hugeMap.value)
   ```

4. **选择合适的分区器**
   - `HashPartitioner`：默认，按 Key 的 Hash 值分区
   - `RangePartitioner`：适合有序/范围分布的数据
   - 自定义分区器：针对特定业务逻辑

5. **启用 Bypass 机制**
   - 当 Reducer 数量 <= `spark.shuffle.sort.bypassMergeThreshold`（默认200）时，Sort Shuffle 会退化为 Hash Shuffle，减少排序开销

6. **使用 Kryo 序列化**
   ```scala
   conf.set("spark.serializer", "org.apache.spark.serializer.KryoSerializer")
   ```

## 题目五：Spark 内存管理机制 — 堆内 vs 堆外内存、 Execution 与 Storage 内存分配

### 📌 问题
请详细说明 Spark 的内存管理模型，包括堆内内存（On-heap）和堆外内存（Off-heap）的区别，Execution 内存和 Storage 内存的关系，以及内存溢出（OOM）的常见原因和解决方法。

### ✅ 答案

**统一内存管理（Spark 2.x+）模型：**

```
┌─────────────────────────────────────────────────────────┐
│                   JVM 堆内存 (On-heap)                    │
│  ┌───────────────┬───────────────┬────────────────────┐ │
│  │   Reserved    │   User Memory │     Spark Memory    │ │
│  │   300MB       │   (约 20%)   │      (约 60%)       │ │
│  └───────────────┴───────────────┴────────────────────┘ │
│                                       ┌───────────────┐ │
│                                       │  Unified      │ │
│                                       │  ┌─────────┐  │ │
│                                       │  │Execution│  │ │
│                                       │  └─────────┘  │ │
│                                       │  ┌─────────┐  │ │
│                                       │  │ Storage │  │ │
│                                       │  └─────────┘  │ │
│                                       └───────────────┘ │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                 堆外内存 (Off-heap)                       │
│  - 通过 Java Unsafe API 直接操作系统内存                    │
│  - 主要用于 Tungsten 序列化存储                            │
│  - 不受 JVM GC 影响                                       │
└─────────────────────────────────────────────────────────┘
```

**内存区域详解：**

| 区域 | 用途 | 说明 |
|------|------|------|
| **Reserved Memory** | 系统预留，约 300MB | 固定不变 |
| **User Memory** | 用户代码的内存分配 | 用于 Spark 内部元数据、UDF 等 |
| **Spark Memory** | Spark 核心内存池 | 统一管理 Execution + Storage |
| **Off-heap** | Tungsten 序列化数据 | 可选开启，减少 GC |

**Execution vs Storage 内存：**

| 内存类型 | 用途 | 特点 |
|----------|------|------|
| **Execution Memory** | Shuffle 排序、Hash 聚合、Join 的哈希表等 | 严格要求，非共享 |
| **Storage Memory** | 缓存 RDD/Broadcast/广播变量 | 可借用 Execution 空闲空间 |

**动态占用机制（Unified Memory）：**

```
当 Execution 内存不足时，可以申请 Storage 内存（归还时需回收）
当 Storage 内存不足时，可以申请 Execution 空闲内存（受阈值限制）
```

- `spark.memory.fraction = 0.6`（Spark 内存占堆内比例）
- `spark.memory.storageFraction = 0.5`（Storage 占 Spark 内存的比例）

**内存溢出（OOM）常见原因：**

1. **Executor 内存配置过小**：`spark.executor.memory` 不足
2. **Shuffle 数据量过大**：groupByKey / reduceByKey 处理了过多数据
3. **DataFrame / Dataset 大规模 Join**：笛卡尔积导致内存爆炸
4. **Broadcast 过大**：广播变量超过限制
   - `spark.sql.broadcastTimeout = 300`（默认5分钟）
   - `spark.sql.autoBroadcastJoinThreshold = 10MB`（默认）

**解决方法：**

```python
# 1. 增加 Executor 内存
--executor-memory 8G

# 2. 减少 Shuffle 分区数（避免过多小任务）
spark.sql.shuffle.partitions = 400  # 适当调大，减少小任务开销

# 3. 使用广播 Join 优化小表 Join
spark.sql.autoBroadcastJoinThreshold = 100MB  # 调大阈值

# 4. 开启 Off-heap 内存
spark.memory.offHeap.enabled = true
spark.memory.offHeap.size = 4g

# 5. 使用 Kryo 序列化减少对象开销
spark.serializer = org.apache.spark.serializer.KryoSerializer

# 6. 合理使用持久化级别
# MEMORY_ONLY -> MEMORY_AND_DISK 系列，避免全量内存存储
```

**面试加分项**：理解 Spark 3.x 的 AQE（Adaptive Query Execution）特性，可根据运行时 Statistics 动态调整 Shuffle 分区数和 Join 策略，进一步减少 OOM 风险。
