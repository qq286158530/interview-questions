## 题目一：Redis 和 Memcached 有什么区别？

### 参考答案

**共同点：**
- 都是基于内存的数据库，一般都用来充当缓存使用
- 都有过期策略
- 两者的性能都非常高

**区别：**

| 维度 | Redis | Memcached |
|------|-------|------------|
| 数据类型 | 支持丰富数据类型（String、Hash、List、Set、ZSet） | 只支持最简单的 key-value 类型 |
| 持久化 | 支持 RDB + AOF 混合持久化，数据可保存到磁盘 | 无持久化功能，数据全存内存，宕机后数据丢失 |
| 集群 | 原生支持集群模式（哨兵、切片集群） | 无原生集群模式，需依赖客户端实现分片 |
| 高级特性 | 支持发布/订阅、Lua 脚本、事务等 | 不支持 |

## 题目二：Redis 单线程模式是怎样的？为什么单线程还这么快？

### 参考答案

**单线程模式工作流程：**

Redis 6.0 之前，"接收客户端请求 → 解析请求 → 数据读写 → 发送响应" 这一整个过程由主线程完成，采用 I/O 多路复用（epoll）处理大量 Socket：

1. 主线程初始化 `epoll_create()`、绑定端口、监听 socket，将 listen socket 加入 epoll 注册连接事件处理函数
2. 进入事件循环：
   - **连接事件**：调用 `accept()` 获取已连接 socket → 注册读事件
   - **读事件**：调用 `read()` 读取数据 → 解析命令 → 处理命令 → 将结果写入发送缓冲区
   - **写事件**：调用 `write()` 将响应发送给客户端

**单线程快的三个原因：**

1. **内存操作**：大部分操作在内存中完成，采用高效数据结构（SDS、跳表、哈希表），瓶颈是内存/带宽而非 CPU
2. **无竞争开销**：单线程避免了多线程切换、锁竞争、死锁等问题
3. **I/O 多路复用**：一个线程处理多个 IO 流（select/epoll），内核同时监听多个 Socket，有请求到达才交给 Redis 处理

## 题目三：Redis 有哪几种持久化方式？简述 AOF 重写机制。

### 参考答案

**三种持久化方式：**

- **RDB 快照**：将某一时刻的内存数据以二进制格式写入磁盘，恢复快但可能丢数据
- **AOF 日志**：每执行一条写命令就以追加方式写入文件，恢复慢但丢数据少
- **混合持久化**（Redis 4.0+）：开头是 RDB 全量数据，后续是 AOF 增量命令，兼顾两者优点

**AOF 重写机制：**

当 AOF 文件大小超过阈值（`auto-aof-rewrite-percentage`），Redis 启用后台子进程 `bgrewriteaof` 进行重写：

1. **fork 子进程**：子进程共享父进程物理内存（只读），父子进程数据共享
2. **读取并转换**：子进程扫描数据库所有键值对，将每条数据转换为一条写命令写入新的 AOF 文件
3. **主进程继续处理命令**：同时将新命令追加到「AOF 缓冲区」和「AOF 重写缓冲区」
4. **写时复制（COW）**：若主进程修改了某个 key，该 key 复制出副本，子进程操作副本，不影响父进程
5. **合并完成**：子进程重写完成后发送信号，主进程将「AOF 重写缓冲区」内容追加到新 AOF 文件
6. **替换旧文件**：新 AOF 文件覆盖旧文件，重写完成

## 题目四：Redis 数据类型及其底层实现是什么？

### 参考答案

**五种基本数据类型：**

| 类型 | 应用场景 | 底层实现（< 阈值时） | 底层实现（> 阈值时） |
|------|----------|---------------------|---------------------|
| **String** | 缓存对象、计数器、分布式锁、共享 session | SDS（简单动态字符串） | SDS |
| **List** | 消息队列、列表 | quicklist（Redis 3.2+替代了双向链表+压缩列表） | quicklist |
| **Hash** | 缓存对象、购物车 | listpack（Redis 7.0 替代压缩列表）或哈希表 | 哈希表 |
| **Set** | 点赞、共同关注、抽奖 | 整数集合（元素均为整数且 < 512 个） | 哈希表 |
| **Zset** | 排行榜、排序 | listpack（Redis 7.0 替代压缩列表）或跳表 | 跳表 |

**SDS 相比 C 字符串的优势：**
- O(1) 获取长度（`len` 属性）
- 二进制安全（用 `len` 而非 `\0` 判断结束）
- 动态扩容不会缓冲区溢出

**四种新数据类型：**
- **BitMap**（2.2+）：二值状态统计（签到、登录状态）
- **HyperLogLog**（2.8+）：海量数据基数统计（UV 计数）
- **GEO**（3.2+）：地理位置（滴滴叫车）
- **Stream**（5.0+）：消息队列，支持消费组

## 题目五：Redis 如何实现服务高可用？哨兵模式是如何工作的？

### 参考答案

**Redis 高可用方案演进：**

1. **主从复制**（基础）：一主多从，读写分离，主服务器写命令异步同步给从服务器。问题是故障需手动恢复。
2. **哨兵模式**（Sentinel）：自动故障转移
3. **切片集群**（Cluster）：数据分片存储

**哨兵模式工作原理：**

哨兵集群由一个或多个哨兵节点组成，负责监控主从节点的健康状态：

**监控：**
- 哨兵周期性向主从节点发送 `PING` 命令
- 若节点在 `down-after-milliseconds` 时间内未响应，标记为「主观下线」
- 哨兵集群中多数节点都认为主节点主观下线时，标记为「客观下线」

**自动故障转移：**
1. 哨兵集群从主节点的从节点中选举出一个新的主节点（基于优先级/复制偏移量/RunID）
2. 被选中的从节点向新的主节点发起 `SLAVEOF NO ONE`，升级为主节点
3. 其他从节点向新主节点发起 `SLAVEOF` 重新同步
4. 哨兵集群更新配置，客户端通过哨兵获取新主节点地址

**配置通知：**
- 哨兵故障转移完成后，会更新主节点配置，并通过发布订阅通知客户端新地址
