## 题目一：Redis 有哪些数据类型？底层是如何实现的？

### 参考答案

Redis 提供了丰富的数据类型，主要分为 **5 种基础数据类型** 和 **4 种高级数据类型**。

### 5 种基础数据类型

| 类型 | 应用场景 | 底层数据结构 |
|------|----------|-------------|
| **String** | 缓存对象、计数器、分布式锁 | SDS（简单动态字符串） |
| **Hash** | 缓存对象、购物车 | 压缩列表 / 哈希表 |
| **List** | 消息队列、列表 | 双向链表 / 压缩列表 / quicklist |
| **Set** | 点赞、共同关注、抽奖 | 哈希表 / 整数集合 |
| **Zset** | 排行榜、电话姓名排序 | 跳表 / 压缩列表 |

### 4 种高级数据类型

- **BitMap（2.2版）**：二值状态统计（签到、判断登录状态）
- **HyperLogLog（2.8版）**：海量数据基数统计（百万级UV计数）
- **GEO（3.2版）**：地理位置存储（滴滴叫车）
- **Stream（5.0版）**：消息队列，支持消费组

### 底层实现细节

- **SDS vs C字符串**：SDS 使用 `len` 属性记录长度（O(1)获取），支持二进制数据，拼接安全不会缓冲区溢出。
- **List（3.2+）**：只由 quicklist 实现，替代了双向链表和压缩列表。
- **Hash**：元素 < 512 个且每个 < 64 字节用压缩列表，否则用哈希表。
- **Zset**：元素 < 128 个且每个 < 64 字节用压缩列表，否则用跳表。
- **Redis 7.0+**：压缩列表已废弃，交由 listpack 实现。

> 📚 来源：[图解Redis - 数据类型篇](https://xiaolincoding.com/redis/data_struct/command.html)

## 题目二：Redis 持久化机制有哪些？RDB 和 AOF 区别是什么？如何选择？

### 参考答案

Redis 共有 **三种持久化方式**：

### 1. RDB 快照（Redis Database）

- 记录某一时刻的内存数据，以**二进制**方式写入磁盘
- 通过 `bgsave` 命令创建子进程生成 RDB 文件，主线程不被阻塞
- 快照过程中数据可修改，借助 **写时复制（COW）** 技术
- 触发方式：`save`（同步，阻塞）或配置文件自动触发（`bgsave`）

### 2. AOF 日志（Append Only File）

- 每执行一条写操作命令，以**追加**方式写入日志文件
- Redis 启动时读取 AOF 文件，逐条执行命令恢复数据
- **写回策略**（`appendfsync`）：

| 策略 | 含义 | 优点 | 缺点 |
|------|------|------|------|
| Always | 每次写命令同步写回硬盘 | 数据不丢失 | I/O压力大 |
| Everysec | 每秒一次异步写回 | 性能适中 | 最多丢1秒数据 |
| No | 由操作系统决定 | 性能最好 | 数据丢失不可控 |

- **AOF 重写**：当文件过大时，后台子进程 `bgrewriteaof` 读取当前数据库所有键值对，用一条命令记录，压缩文件体积

### 3. 混合持久化（Redis 4.0+）

- 开头为 RDB 格式（全量数据），后半部分为 AOF 格式（增量命令）
- 结合两者优点：恢复快 + 丢失少

### 对比总结

| 维度 | RDB | AOF |
|------|-----|-----|
| 文件体积 | 小（二进制） | 大（命令文本） |
| 恢复速度 | 快 | 慢 |
| 数据安全性 | 可能丢失较多数据 | 可配置（每秒/每次） |
| 对性能影响 | bgsave 时fork子进程 | 持续写IO |

### 如何选择？

- **混合使用**：生产环境推荐 `bgsave` + AOF `everysec`
- **数据敏感**：开启混合持久化，兼顾恢复速度和安全性

> 📚 来源：[图解Redis - 持久化篇](https://xiaolincoding.com/redis/storage/aof.html)

## 题目三：Redis 主从复制原理是什么？哨兵模式解决了什么问题？

### 参考答案

### 主从复制

Redis 主从复制采用 **一主多从** 架构，实现数据多节点冗余，具体原理：

**复制过程：**
1. 从节点保存主节点地址（`replicaof masterip port`）
2. 从节点向主节点发送 `PSYNC` 命令（首次为全量同步 `psync ? -1`）
3. 主节点执行 `bgsave`，生成 RDB 文件并发送给从节点
4. 从节点清空本地数据，加载 RDB 文件
5. 主节点将同步期间的写命令缓存起来，发送给从节点（增量同步）
6. 后续主节点的写操作异步同步给从节点

**注意事项：**
- 命令复制是**异步**的，主节点不等待从节点执行完毕就返回结果
- 主从数据存在不一致窗口，**无法保证强一致性**

### 哨兵模式（Sentinel）

**解决的问题：**
- 主节点故障时，需要人工手动恢复的问题
- 提供 **监控、自动故障转移、通知** 三大功能

**工作原理：**
1. 哨兵节点监控主从节点的运行状态（心跳检测）
2. 主节点客观下线（`down-after-milliseconds` 内无响应）后，触发选举
3. 哨兵集群通过 Raft 算法选举出领头的哨兵（leader）
4. leader 哨兵在从节点中选举出新主节点
5. 其他从节点向新主节点同步（`replicaof no one`）
6. 哨兵通知客户端新主节点地址

### 脑裂问题

集群脑裂：主节点网络故障，哨兵误判为主观下线，选举出新主节点，形成两个主节点。故障恢复后旧主节点降级为从节点，会清空本地数据导致之前写入的数据丢失。

**解决方案：** 配置 `min-slaves-to-write` 和 `min-slaves-max-lag`，限制主节点在从节点过少或延迟过大时禁止写入。

> 📚 来源：[图解Redis - 主从复制与哨兵](https://xiaolincoding.com/redis/cluster/master_slave_replication.html)

## 题目四：什么是缓存雪崩、击穿、穿透？如何解决？

### 参考答案

### 缓存雪崩

**问题：** 大量缓存数据**同一时间过期**，导致大量请求直接打到数据库。

**解决方案：**
- 设置**过期时间时加随机值**（`TTL + random`），避免同时过期
- 使用 **Redis 集群**，保证缓存服务高可用
- 对数据库请求加 **互斥锁** 或 **队列**，避免瞬时并发过大
- 业务层实现**缓存预热**，提前加载热点数据

### 缓存击穿

**问题：** 某个**热点数据**过期，导致大量请求打到数据库（类似雪崩，但只有1个key）。

**解决方案：**
- 对热点数据设置**永不过期**（由业务逻辑控制更新）
- 使用 **互斥锁**（`SETNX`），只允许一个线程重建缓存
- 逻辑过期：热点数据永不过期，但字段中包含逻辑过期时间，到期后异步更新

### 缓存穿透

**问题：** 查询**数据库和缓存中都不存在**的数据（如恶意攻击），每次都打到数据库。

**解决方案：**
1. **布隆过滤器（BloomFilter）**：在缓存层前加一个过滤器，判断数据是否存在，不存在直接返回
2. **缓存空值/默认值**：将查询结果为NULL的数据也缓存起来（设置较短TTL）
3. **参数合法性校验**：如ID为负数、非法格式直接拦截
4. **接口限流**：防止大量恶意请求

### 对比总结

| 类型 | 原因 | 解决思路 |
|------|------|---------|
| 雪崩 | 大量Key同时过期 | 随机TTL + 预热 + 集群 |
| 击穿 | 1个热点Key过期 | 永不过期 + 互斥锁 |
| 穿透 | 数据压根不存在 | 布隆过滤 + 空值缓存 |

> 📚 来源：[图解Redis - 缓存问题](https://xiaolincoding.com/redis/cluster/cache_problem.html)

## 题目五：Redis 如何实现分布式锁？有什么注意事项？

### 参考答案

### 基本实现

使用 `SET key value NX PX milliseconds` 命令加锁：

```bash
# 加锁（key不存在才设置，原子操作）
SET lock_key unique_value NX PX 10000

# 释放锁（Lua脚本：只有自己才能删除自己）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**关键点：**
- `NX`：保证互斥，只有不存在时才能设置成功
- `PX`：设置过期时间，防止锁忘记释放导致死锁
- `value` 使用唯一值（如UUID）：释放锁时验证是持有者本人

### 注意事项

#### 1. 锁过期但任务未完成

场景：任务执行时间超过锁TTL，锁自动释放，导致其他线程获取到锁。

**解决：** 守护线程续命（Watchdog）：Redisson框架自动在后台每 `leaseTime/3` 为锁续期。

#### 2. 主从切换导致锁丢失

场景：主节点刚写入锁，未同步到从节点时主节点宕机，新主节点没有锁数据。

**解决：** 使用 **RedLock（红锁）**，在多个独立Redis实例上获取锁，超过半数成功才算成功。

```bash
# RedLock示意
for (RedisInstance : instances) {
    result = SET(key, value, NX, PX, lockTimeout)
}
if (成功数量 > N/2 + 1) {
    // 持有锁
}
```

#### 3. 可重入性

同一线程可多次获取同一把锁，需要维护计数器。

#### 4. 公平锁 vs 非公平锁

- **非公平锁**（默认）：先到不一定先获取，吞吐量高
- **公平锁**：按请求顺序获取，实现复杂（ZSet实现）

### 高并发场景建议

- 优先使用 **Redisson** 等成熟库，避免自己实现踩坑
- 业务层面考虑锁粒度（粗锁 vs 细锁）
- 异常时确保 finally 中释放锁
