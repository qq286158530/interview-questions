## 1️⃣ Spark Shuffle 的原理是什么？为什么 Shuffle 会影响性能？

### 参考答案

**Shuffle 是 Spark 中将数据按 Key 进行重新分区的过程**，发生在 `groupBy`、`join`、`reduceByKey` 等需要跨分区聚合的操作时。

**执行流程：**

```
RDD[Task] → Shuffle Write → 本地磁盘文件
                            ↓
            Shuffle Read ← 其他节点的 Block Manager
                            ↓
           RDD[Task] 重新计算
```

- **Shuffle Write**：map 任务结束后，将数据写入本地磁盘，根据目标分区算法形成多个文件（每个对应一个 reduce 分区）
- **Shuffle Read**：reduce 任务启动后，去各个 map 节点拉取属于自己分区的数据，进行聚合

**影响性能的原因：**

| 因素 | 说明 |
|------|------|
| **磁盘 I/O** | 中间结果写入本地磁盘而非内存，网络传输需读写磁盘 |
| **网络 I/O** | 数据需要跨节点传输，数据量越大开销越大 |
| **序列化/反序列化** | 数据需要在网络间传输，必须序列化；拉取后需反序列化 |
| **磁盘空间** | Shuffle 文件会落盘，磁盘空间压力大 |
| **频繁 GC** | 大规模 Shuffle 会产生大量临时对象，触发 GC |

**优化思路：**
- 减少 Shuffle 次数（减少不必要的 `repartition`、`coalesce`）
- 使用 `reduceByKey` 替代 `groupByKey`（map 端预聚合）
- 合理设置 `spark.shuffle.manager`（SortShuffleManager优于 HashShuffleManager）
- 调大 `spark.shuffle.service.enabled`（开启外部 Shuffle Service）
- 增加 `spark.executor.memory` 减少 OOM

## 2️⃣ Spark RDD、DataFrame、DataSet 的区别与联系？

### 参考答案

| 特性 | RDD（Resilient Distributed Dataset） | DataFrame | DataSet |
|------|------------------------------------|-----------|---------|
| **数据结构** | 分布式元素集合（无结构） | 分布式行集合（结构化，有 schema） | 强类型分布式数据集（JVM 对象） |
| **Schema** | 无 | 有（列名、类型） | 有（编译时检查） |
| **数据安全** | Python/Java 对象 | Row 对象（二进制） | 强类型 T 对象 |
| **操作方式** | 函数式算子（map/filter/reduce） | SQL / DataFrame API | Lambda + Dataset API |
| **优化** | 无 Catalyst优化 | Catalyst优化器 | Catalyst + Tungsten |
| **序列化** | Java/Kryo | Row序列化 | Encoder |
| **编译时检查** | 无 | 无 | 有（类型安全） |
| **易用性** | 低（需手写逻辑） | 高（类似 SQL） | 中（结合两者优点） |

**联系：**
- `DataFrame = DataSet[Row]` —— DataFrame 是特殊类型的 DataSet
- 三者底层都是基于 RDD 实现的，都是惰性求值
- DataFrame/DataSet 享受 Catalyst 优化器的优化（列裁剪、谓词下推、常量替换等）
- 相互转换：`rdd.toDF()` / `ds.toDF()` / `df.as[T]` 等

**什么时候选哪个？**
- 业务逻辑简单、数据结构不规则 → **RDD**
- 结构化数据、熟悉 SQL、需要优化 → **DataFrame**
- 需要类型安全、熟悉 JVM 语言 → **DataSet**
- **实际工程中推荐优先使用 DataFrame/DataSet**，性能更好

## 3️⃣广播变量（Broadcast Variable）的作用和原理是什么？使用时应注意什么？

### 参考答參考答案

**作用：** 将一个只读的变量**广播**到所有 Executor节点，每个节点缓存一份，而非每个 Task 传递一份。从而**避免在每个任务中重复传输大表数据**，大幅减少网络开销。

**原理：**

```
Driver 端：broadcast.collect() →广播管理器（Driver）
                         ↓ P2P / BitTorrent-like
各 Executor 节点：本地 BlockManager 缓存广播数据
                         ↓
           Task 运行时直接读取本地缓存，无需网络拉取
```

-广播变量在 Driver 端创建（`spark.sparkContext.broadcast(v)`）
- 广播后调用 `.value` 获取实际值
- 广播变量一旦创建**不可修改**（只读）
- Spark 自动选择最优广播策略（小数据用 `Array` 序列化，大数据用 P2P）

**使用示例：**

```scala
// 定义广播变量
val broadcastVar = spark.sparkContext.broadcast(Map("北京" -> 1, "上海" -> 2))

// 在算子内使用
rdd.map { case (city, count) =>
  val factor = broadcastVar.value.getOrElse(city, 0)
  (city, count * factor)
}
```

**注意事项：**

| 问题 | 说明 |
|------|------|
| **只读** | 广播后不可修改，如需更新需重新广播 |
| **大小控制** | 广播数据不宜过大（>20MB 会警告），过大需考虑其他方案 |
| **节点一致** | 广播更新后需重新创建，否则各 Executor 仍用旧值 |
| **序列化** | 广播数据需可序列化（Java/Kryo） |
| **时机** | 多任务重复使用同一份小/中数据时才值得广播 |
| **清理** | 广播变量不用时应 `.destroy()` 释放内存 |

## 4️⃣ Spark 的内存管理机制是怎样的？请说明堆内内存和堆外内存的区别？

### 参考答案

Spark 内存管理采用**统一内存管理（Unified Memory Management）**，主要分为两部分：

**内存划分（Spark 2.x+）：**

```
┌────────────────────────────────────────────────────┐
│                   Heap Memory │
│  ┌──────────────┐ ┌─────────────────────────────┐ │
│  │  Execution │  │        Storage              │ │
│  │   Memory     │  │       Memory                │ │
│  │  (执行内存)   │  │      (存储内存)              │ │
│  │             │◄─│动态共享，统一管理 │ │
│  └──────────────┘  └─────────────────────────────┘ │
│  ┌──────────────────────────────────────────────┐  │
│  │      User Memory (用户内存)                   │  │
│  │   (RDD残留、用户数据结构、内部元数据)            │  │
│  └──────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────┘
```

**堆内内存（Heap Memory）：**

- 由 JVM 管理的堆内存，受 `spark.executor.memory` 控制
- 包含 Execution Memory、Storage Memory、User Memory
- Spark 内部对象在此分配

**堆外内存（Off-Heap Memory）：**

- 由 `spark.memory.offHeap.enabled=true` 开启
- 通过 `spark.memory.offHeap.size` 设置大小（默认关闭）
- **不受 JVM GC 管理**，减少 Full GC 频率
- 主要用于：Tungsten 序列化（ByteBuffer）、广播变量、Shuffle 缓存等
- 更适合生产环境，可避免因 Java 对象开销导致的 OOM

**Execution Memory vs Storage Memory 动态划分：**

- 两者**共享统一内存池**，可互相借用
- 执行内存不足时，可驱逐存储内存中的缓存（LRU）
- 存储内存不足时，不能借用执行内存（执行内存不驱逐）
- 动态比例由 `spark.memory.fraction`（默认 0.6）和 `spark.memory.storageFraction`（默认 0.5）控制

**内存参数配置建议：**

```properties
spark.executor.memory = 4g
spark.memory.fraction = 0.6          # 统一内存占heap比例
spark.memory.storageFraction = 0.5  # Storage初始占比
spark.memory.offHeap.enabled = true # 开启堆外内存
spark.memory.offHeap.size = 2g      # 堆外内存大小
```

## 5️⃣ Spark 任务调度流程是什么？从提交 Spark作业到 Task 执行，经历了哪些阶段？

### 参考答案

**完整调度流程如下：**

```
用户提交 Spark 作业
        ↓
  Driver 端初始化
    ├─ 创建 SparkContext
    ├─ 向 Cluster Manager 注册 Application
    └─ 申请 Executor资源
        ↓
  DAGScheduler（DAG 调度器）
    ├─ 将 Job 划分为多个 Stage（依据 Shuffle 边界）
    └─ 将 Stage 转换为 TaskSet（包含多个 Task）
        ↓
  TaskScheduler（任务调度器）
    ├─ 将 TaskSet 提交给 Executor
    ├─ 跟踪 Task 执行状态
    └─ 失败重试、优先调度本地性高的 Task
        ↓
  Executor端
    ├─ 接收 Task（启动 CoarseGrainedExecutorBackend）
    ├─ 反序列化 Task 代码
    ├─ 调用 Task.run() 执行计算
    └─ 将结果返回给 Driver
        ↓
  完成 ResultTask 或 ShuffleMapTask
```

**各阶段详解：**

1. **Application**：用户编写的 Spark 程序（Driver + Executors）
2. **Job**：一个 Action 算子触发一个 Job（如 `count()`、`collect()`）
3. **Stage**：一个 Job 按 Shuffle 边界划分为多个 Stage（宽依赖分界）
4. **Task**：一个 Stage 包含多个 Task（每个 Partition 对应一个 Task）
5. **TaskSet**：同一 Stage 的所有 Task 组成 TaskSet

**DAG 划分 Stage原则：**
- **窄依赖**（map、filter）→ 同一 Stage 内，流水线执行
- **宽依赖**（groupByKey、join、reduceByKey）→ 划分 Stage 边界，需要 Shuffle

**Task 种类：**
- **ShuffleMapTask**：不是最终 stage，需要将数据写入 Shuffle 文件，供下游 Stage读取
- **ResultTask**：最终 stage，直接返回结果给 Driver

**失败重试机制：**
- Task 失败 → TaskScheduler 重试（默认4 次）→ 失败则标记 Stage失败
- Stage 失败 → DAGScheduler 重试（重新计算整个 Stage）
- Job 失败 → 整个 Application 失败
