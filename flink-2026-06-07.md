## 题目1：Flink 的整体架构是怎样的？JobManager 和 TaskManager 的职责是什么？

### 参考答案

Flink 采用主从架构，整体分为 **JobManager**（主节点）和 **TaskManager**（从节点）两部分。

**JobManager 职责：**
- 负责调度任务，将 JobGraph 转换为 ExecutionGraph
- 协调 Checkpoint 机制
- 分配 TaskSlot 给 TaskManager
- 负责故障恢复和资源协调
- 包含三个核心组件：
  - **Dispatcher**：接收用户提交的任务，启动 JobMaster
  - **ResourceManager**：管理 TaskManager 的 Slot（槽位），进行资源分配
  - **JobMaster**：负责执行单个 JobGraph，包括 Task调度和 Checkpoint 协调

**TaskManager 职责：**
- 负责执行具体的 Task（算子实例）
- 提供计算资源（TaskSlot）
- 与 JobManager 通信汇报任务状态
- 管理本地状态（State Backend）
- 数据在算子之间流动时，TaskManager 负责数据传递

**工作流程：**
1. 用户代码 → StreamGraph → JobGraph
2. JobManager 接收 JobGraph，转换为 ExecutionGraph
3. ResourceManager 分配 TaskSlot
4. JobMaster 调度 Task 到 TaskManager 执行
5. TaskManager 之间通过数据流进行数据交换

## 题目 2：什么是 Watermark（水位线）？它在 Flink 中是如何解决事件时间乱序问题的？

### 参考答案

**Watermark概念：**
Watermark 是 Fllink 用于处理**事件时间（Event Time）**乱序问题的核心机制。它是一个时间戳标记，表示"所有时间戳小于该 Watermark 的事件都已经到达"。

**核心作用：**
- 解决数据乱序：由于网络延迟、分布式系统等因素，数据可能不按时间戳顺序到达
- 触发窗口计算：当 Watermark 超过窗口结束时间时，窗口才会触发计算
- 允许一定程度的延迟：Watermark 设置的延迟时间即为系统允许的最大乱序程度

**Watermark 生成方式：**

```java
// AssignerWithPeriodicWatermarks - 周期性生成（默认200ms）
DataStream<Tuple2<Long, String>> withWatermarks = stream
    .assignTimestampsAndWatermarks(
        WatermarkStrategy
            .forBoundedOutOfOrderness(Duration.ofSeconds(10))
    );

// 使用固定延迟策略
WatermarkStrategy.forMonotonousTimestamps();

// 使用严格递增策略（不允许乱序）
WatermarkStrategy.forStrictlyMonotonicTimestamps();
```

**Watermark 传播机制：**
- 下游算子会继承上游算子的 Watermark
- 取上游 Watermark 的最小值
- 每个分区独立计算 Watermark

**乱序处理示例：**
```
窗口 [10:00 - 10:05)

事件时间戳: 10:01, 10:04, 10:02(乱序), 10:03(乱序)
Watermark(10:03)到达 → 窗口可以触发计算
事件10:02 和 10:03之前的数据都已收到
```

**注意事项：**
- Watermark 延迟设置过大会增加处理延迟
- 设置过小可能导致数据丢失
- 需要根据数据特点和网络状况调整

## 题目 3：Flink 的 Checkpoint 和 Savepoint有什么区别？如何实现 Exactly-Once 语义？

### 参考答案

**Checkpoint vs Savepoint 对比：**

| 特性 | Checkpoint | Savepoint |
|------|------------|-----------|
| **触发方式** | Flink 自动周期性触发 | 手动触发（用户命令） |
| **用途** | 故障恢复 | 计划性维护、版本升级 |
| **存储位置** | JobManager 管理的状态后端 | 用户指定的外部存储 |
| **删除方式** | 作业失败/取消时自动删除 | 需手动删除 |
| **格式兼容性** | 可能因状态数据结构变化而不兼容 | 通常兼容 |
| **对作业影响** | 不影响作业运行 | 可能暂停作业 |

**Checkpoint 工作原理：**

1. **Barrier（屏障）机制**：
   - Checkpoint Coordinator发送 Barrier 到数据流中
   - Barrier 随数据流动，被算子接收
   - 算子收到所有输入的 Barrier 后进行状态快照

2. **分布式快照算法**：
   - 采用 Chandy-Lamport 变体算法
   -Barrier 对齐：算子需等待所有输入 Channel 的 Barrier 到达
   - 快照完成前，状态不可更改

3. **状态存储**：
   - **HashMapStateBackend**：内存存储，速度快但受限于 JVM 堆内存
   **EmbeddedFileStateBackend**：RocksDB 状态后端，支持大状态
   - **rocksdb + s3**：增量 Checkpoint，适合大规模状态

**Exactly-Once 实现条件：**

1. **Source 端**：
   - 需支持数据回退和重放（如 Kafka 的 offset提交）
   - 故障恢复时从上次 Checkpoint 位置重新消费

2. **Sink 端**：
   - 需支持幂等写入或事务写入
   - 幂等：重复写入同一数据不改变结果
   - 事务：预提交 + Checkpoint 完成时提交（如 Kafka Transaction）

3. **Flink 内部**：
   - Checkpoint 机制确保内部状态一致性
   -Barrier 对齐确保数据不丢失不重复

**配置示例：**

```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
//启用 Checkpoint，间隔10 秒
env.enableCheckpointing(10000);
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(500);
env.getCheckpointConfig().setCheckpointTimeout(60000);
env.getCheckpointConfig().setMaxConcurrentCheckpoints(1);
```

## 题目 4：Flink 的窗口函数有哪些类型？Session Window 和 Tumbling Window 的区别是什么？

### 参考答案

**窗口函数分类：**

**1. 按时间划分：**
- **Time Window（时间窗口）**：基于时间戳划分
- **Count Window（计数窗口）**：基于数据条数划分

**2. 按滑动方式：**
- **Tumbling Window（滚动窗口）**：无重叠，等长切分
- **Sliding Window（滑动窗口）**：有重叠，由窗口大小和滑动步长决定
- **Session Window（会话窗口）**：基于数据间隔动态创建

**3. 按语义：**
- **Keyed Window（带键窗口）**：按 Key 分组，每个 Key 独立计算
- **Non-Keyed Window（非带键窗口）**：所有数据聚合到一个任务处理

**滚动窗口 vs 滑动窗口：**

```java
// 滚动窗口：每 5 秒一个窗口，不重叠
stream.keyBy(x -> x.field)
     .window(TumblingEventTimeWindows.of(Time.seconds(5)))
     .sum(1);

// 滑动窗口：窗口大小 10 秒，每 5 秒滑动一次
stream.keyBy(x -> x.field)
     .window(SlidingEventTimeWindows.of(Time.seconds(10), Time.seconds(5)))
     .sum(1);
```

**会话窗口（Session Window）：**

```java
// 会话窗口：数据间隔超过 30 秒则开启新窗口
stream.keyBy(x -> x.field)
     .window(SessionWindow.withGap(Time.seconds(30)))
     .sum(1);

//动态间隙会话窗口
stream.keyBy(x -> x.field)
     .window(DynamicSessionWindow.withGap(new SessionWindowTimeGapExtractor<T>() {
         @Override
         public long extract(T element) {
             return element.getGapMillis();
         }
     }))
```

**各窗口特点对比：**

| 窗口类型 | 特点 | 适用场景 |
|----------|------|----------|
| Tumbling Window | 无重叠，简单高效 | 固定周期统计 |
| Sliding Window | 有重叠，状态存储较大 | 需要平滑统计 |
| Session Window | 动态窗口长度 | 用户行为分析、动态事件 |
| Count Window | 不关心时间，只关心数量 | 凑够 N 个处理 |

**窗口计算流程：**
1. 窗口分配器（Window Assigner）决定数据属于哪个窗口
2. 窗口函数（Window Function）处理窗口数据
   - **增量计算**：每来一条数据就计算（如 reduce、aggregate）
   - **全量计算**：等窗口关闭后再计算（如 apply、process）
3. 窗口清除器决定何时清除窗口状态

**注意事项：**
- 使用 EventTime 时必须配置 Watermark
- 窗口状态大小需要合理控制，避免内存溢出
- Session Window 的 gap 设置要合理

## 题目 5：Flink 的状态管理机制是怎样的？State Backend 有哪些类型？如何选择合适的状态后端？

### 参考答案

**状态管理架构：**

Flink 的状态管理由两部分组成：
- **Keyed State**：按 Key 隔离的作用域状态，只能在 KeyedStream 上使用
- **Operator State**：算子级别的状态，所有算子实例共享

**Keyed State 类型：**

| 状态类型 | 说明 | 使用场景 |
|----------|------|----------|
| ValueState<T> | 单值状态，存储单个值 | 存储简单标志位 |
| ListState<T> | 列表状态 | 存储历史数据列表 |
| MapState<K, V> | 映射状态 | 存储键值对映射 |
| ReducingState<T> | 自动聚合状态 | 持续聚合计算 |
| AggregatingState<IN, OUT> | 聚合状态 | 复杂聚合逻辑 |

**Operator State 类型：**
- **ListState**：算子状态列表
- **UnionListState**：故障恢复时分发全部状态
- **BroadcastState**：广播状态，所有并行实例共享

**State Backend 类型：**

1. **HashMapStateBackend**：
   - 状态存储在 JVM 堆内存中
   - 速度快，但受限于内存大小
   - 状态大小建议< 10MB
   - 默认状态后端

2. **EmbeddedFileSystemStateBackend（RocksDB）**：
   - 状态存储在 RocksDB 中（本地磁盘）
   - 支持增量 Checkpoint
   - 可处理较大状态（TB级别）
   - 性能较 HashMap 低（需序列化）

3. **EmbeddedFileSystemStateBackend +远程存储**：
   - RocksDB + S3/HDFS
   - 支持状态容灾和迁移

**配置示例：**

```java
// 方式1：HashMapStateBackend（默认）
env.setStateBackend(new HashMapStateBackend());

// 方式2：RocksDB + 本地存储
env.setStateBackend(new EmbeddedFileSystemStateBackend("file:///tmp/flink-checkpoints"));

// 方式3：RocksDB + S3（需配置 S3 credentials）
env.getCheckpointConfig().setCheckpointStorage("s3://my-bucket/flink-checkpoints");
```

**状态使用示例：**

```java
public class CountWindowAverage
    extends KeyedProcessFunction<Long, Tuple2<Long, Long>, Tuple2<Long, Double>> {
    
    // 定义状态
    private ValueState<Long> count;
    private ValueState<Long> sum;
    
    @Override
    public void open(Configuration parameters) {
        count = getRuntimeContext().getState(
            ValueStateDescriptor("count", Long.class)
        );
        sum = getRuntimeContext().getState(
            ValueStateDescriptor("sum", Long.class)
        );
    }
    
    @Override
    public void processElement(
        Tuple2<Long, Long> value, 
        Context ctx, 
        Collector<Tuple2<Long, Double>> out) throws Exception {
        
        Long currentCount = count.value();
        currentCount = currentCount == null ? 0L : currentCount;
        count.update(currentCount + 1);
        
        Long currentSum = sum.value();
        currentSum = currentSum == null ? 0L : currentSum;
        sum.update(currentSum + value.f1);
        
        if (currentCount + 1 >= 5) {
            out.collect(new Tuple2<>(value.f0, Double.valueOf(currentSum / currentCount)));
            count.update(0L);
            sum.update(0L);
        }
    }
}
```

**状态后端选择建议：**

| 场景 | 推荐状态后端 |
|------|--------------|
| 状态较小，低延迟 | HashMapStateBackend |
| 状态较大（GB-TB级） | RocksDB + 远程存储 |
|增量 Checkpoint | RocksDB |
| 快速原型开发 | HashMapStateBackend |

**状态过期与清理：**
- 使用 `.ttl()` 配置状态 TTL（生存时间）
- `StateTtlConfiguration` 可设置清理策略
- 适用于状态历史数据清理场景
