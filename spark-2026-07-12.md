## 题目一：Spark 架构与核心组件

**问题：** 请描述 Spark 的整体架构，以及 Driver、Executor、Cluster Manager 之间的协作关系。

**答案：**

Spark 采用 **主从架构**，核心组件如下：

| 组件 | 作用 |
|------|------|
| **Driver（驱动器）** | 运行用户代码，创建 `SparkContext`，将 Job 划分为多个 Stage 和 Task，调度任务分发到 Executor |
| **Executor（执行器）** | 驻留在 Worker 节点上，每个 Executor 启动后保持运行，负责执行分发的 Task、存储结果数据 |
| **Cluster Manager** | 负责集群资源管理（YARN、Standalone、Mesos、Kubernetes），在 Worker 节点上分配资源并启动 Executor |
| **Worker Node** | 集群中的实际工作节点，运行 Executor 进程 |

**协作流程：**
1. 用户提交 Spark 应用，Driver 向 Cluster Manager 申请资源。
2. Cluster Manager 在 Worker 节点上启动 Executor。
3. Driver 将应用代码（Jar/Python 文件）发送给 Executor，并将 Task 分发到各 Executor 并行执行。
4. Executor 执行 Task，将结果返回给 Driver或写入存储。
5. Task 执行完毕，Executor 被释放，Driver 进程结束。

> **面试亮点：** 可补充 Spark 3.x 的「动态资源分配」特性，以及 Executor 失败时的 Task 重试机制（默认 4 次）。

## 题目二：RDD 五大特性

**问题：** RDD（Resilient Distributed Dataset）的五大特性是什么？请结合源码说明其作用。

**答案：**

RDD 是 Spark 最底层的抽象，具备以下五大特性：

1. **` partitions ` — 分区列表**
   每个 RDD 由多个分区组成，分区是并行计算的基本单位。分区数量决定并发 Task 数量。

2. **` computeFn(partition) ` — 计算函数**
   RDD 通过 `compute` 方法对每个分区执行计算逻辑，返回迭代器。

3. **` dependencies ` — 依赖关系**
   RDD 之间通过依赖（窄依赖 / 宽依赖）构成 DAG。窄依赖（如 `map`、`filter`）支持流水线执行，宽依赖（如 `groupByKey`、`join`）触发 Shuffle。

4. **` partitioner ` — 分区器（可选）**
   仅键值对 RDD（K-V RDD）有分区器，决定 Shuffle 时数据如何路由到目标分区。常见分区器有 `HashPartitioner` 和 `RangePartitioner`。

5. **` preferredLocations ` — 优先位置（可选）**
   记录每个分区的最佳计算位置，调度器尽可能将 Task 调度到数据所在节点，实现**数据本地性（Data Locality）**优化。

> **面试亮点：** 宽依赖的 `Shuffle` 是 Spark 性能瓶颈的根源，理解依赖类型才能理解 Stage 划分和流水线优化。

## 题目三：DataFrame vs DataSet vs RDD

**问题：** Spark 中 DataFrame 和 DataSet 以及 RDD 三者有何区别？何时该用哪种？

**答案：**

| 特性 | RDD | DataFrame | DataSet |
|------|-----|-----------|---------|
| **数据结构** | 无结构，每行是 JVM 对象 | 有 Schema，每列有名字和类型 | 有 Schema，且是类型安全的 JVM 对象 |
| **语言支持** | Scala/Java/Python/R | Scala/Java/Python/R | Scala/Java（Python 不支持 Dataset） |
| **序列化** | Java/Kryo 序列化 | Tungsten 二进制编码 + 堆外内存 | 同 DataFrame |
| **优化** | 无 Catalyst 优化 | Catalyst 优化器 + Tungsten | 同 DataFrame（加上类型安全） |
| **API 表达力** | 灵活但冗长 | 简洁 DSL | 兼具类型安全与 DSL |

**核心区别：**
- **RDD**：适合低层操作、自定义分区、无法用 DataFrame/DataSet 表达的场景。
- **DataFrame**：适合结构化/半结构化数据，Catalyst 可做谓词下推、列剪裁等优化，显著提升性能。
- **DataSet**：在 DataFrame 基础上增加了编译时类型检查，适合需要类型安全的 Scala/Java 代码。

**选择建议：**
- 大多数场景优先选 **DataFrame**（易用+优化）。
- 若需编译期类型安全且用 Scala/Java，选 **DataSet**。
- 底层控制、图计算等特殊场景用 **RDD**。

> **面试亮点：** 可以提到 Spark 2.x 将 DataFrame 和 DataSet 统一，DataFrame 实际是 `Dataset[Row]` 的别名。

## 题目四：Shuffle 机制与优化策略

**问题：** 什么是 Shuffle？为什么 Shuffle 是 Spark 性能瓶颈？有哪些优化策略？

**答案：**

**Shuffle（混洗）** 是 Spark 中将数据重新分布的过程——当一个 RDD 的分区需要依赖上一个 RDD 中多个分区的数据时（如 `groupByKey`、`join`、`reduceByKey`），必须跨节点拉取数据，这就是 Shuffle。

**为什么是瓶颈：**
1. **磁盘 I/O**：Shuffle 数据先写入磁盘再读取，磁盘读写成为主要延迟来源。
2. **网络 I/O**：跨节点数据传输消耗带宽。
3. **内存压力**：Shuffle 过程中需要缓存中间数据，可能触发 OOM 或溢写磁盘。
4. **计算放大**：宽依赖导致前后 Stage 无法流水线执行，必须等 Shuffle 完成后才能继续。

**优化策略：**

| 策略 | 说明 |
|------|------|
| **减少 Shuffle 数据量** | 使用 `mapSideCombine`（如 `reduceByKey` 优于 `groupByKey`），提前聚合 |
| **合理设置分区数** | 分区过多增加小文件，分区过少数据倾斜；一般每个 CPU 核 2-4 个分区 |
| **使用广播变量** | 将小表广播到每个 Executor，避免 Shuffle（适合大表 join 小表） |
| **开启 Shuffle 压缩** | 设置 `spark.shuffle.compress=true`，使用 `spark.io.compression.codec`（推荐 LZ4/Zstd） |
| **升级到 SortShuffle** | Spark 2.x+ 默认 SortShuffle，避免 hash shuffle 的文件膨胀 |
| **使用外部排序** | `spark.shuffle.service.enabled=true`（配合 YARN）将 Shuffle 数据卸载到外部服务 |
| **避免数据倾斜** | 通过加盐、随机前缀等方法打散热点 key |

> **面试亮点：** 结合具体参数（如 `spark.sql.shuffle.partitions` 默认 200）及背后的原理来回答，显得更有深度。

## 题目五：广播变量（Broadcast Variable）原理与使用场景

**问题：** 什么是广播变量？它是如何实现的？适合用在哪些场景？

**答案：**

**广播变量**是将一份只读数据缓存到每个 Executor 的内存中，供该 Executor 上所有 Task 共享的机制，避免每个 Task 都持有一份副本从而节省大量内存和网络传输。

**实现原理：**

1. Driver 端：调用 `spark.sparkContext.broadcast(value)` 创建广播变量，对象被序列化后存储在 Driver 的内存中。
2. 广播阶段：Driver 将广播数据分块（默认 4MB/块），通过 BitTorrent 风格的分发策略发送给各 Executor。
3. Executor 端：Executor 收到数据块后存储在堆内/堆外内存中（`BroadcastBlock`），Task 通过广播变量的 `value` 属性访问。
4. 清理：应用结束后 Driver 通知 Executor 删除广播数据。

**核心特点：**
- **只读**：一旦广播，无法修改（修改不会同步）。
- **高效共享**：每个 Executor 只有一份副本，而不是每个 Task 一份。
- **容错**：如果某个 Executor 丢失，广播数据会重新分发。

**使用示例（Python）：**

```python
# 将小表广播到每个 Executor
small_df = spark.read.parquet("hdfs://small_table")
broadcast_df = spark.sparkContext.broadcast(small_df.collect())

# 在 UDF 或 join 中使用
def join_udf(key):
    return broadcast_df.value.get(key)

result = large_df.withColumn("joined", udf(join_udf)("key"))
```

**使用场景：**

| 场景 | 说明 |
|------|------|
| **大表 join 小表** | 小表广播后避免 Shuffle，时间从 O(n) 降到 O(n)（无 Shuffle） |
| **配置表/维度表** | 将维表广播到各节点做 lookup |
| **固定枚举值/字典** | 将字典数据广播，所有 Task 直接读取，无需重复传输 |
| **过滤条件广播** | 将大表的过滤条件广播，减少 Shuffle 数据量 |

**注意事项：**
- 广播变量不适合**大表**，否则会导致 Executor OOM（默认堆内存储）。
- Spark SQL 中 `broadcast hint`（`/*+ BROADCAST(t) */`）可自动识别小表并广播。

> **面试亮点：** 可补充 Spark 3.x 对广播变量的优化（如自适应广播），以及广播变量与 `accumulators`（累加器）的区别（广播变量只读，累加器可写）。
