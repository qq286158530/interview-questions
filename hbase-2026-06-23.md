## 1. HBase 的数据写入流程是怎样的？为什么需要 WAL？

### 答案

HBase 的数据写入采用 **Write-Ahead Log (WAL)** + **MemStore** 两阶段机制，流程如下：

#### 写入流程

```
客户端写请求
     │
     ▼
① RegionServer 获取 WAL（预写日志）
     │  数据先写入 WAL（顺序写磁盘）
     ▼
② 数据写入 MemStore（内存 SortedMap）
     │
     ▼
③ 返回客户端 "写入成功"（异步刷盘）
```

1. **WAL 写入**：数据首先 Append 到 RegionServer 的 WAL 文件（一个 HDFS 上的 HLog），顺序写入保证高性能；
2. **MemStore 写入**：数据同时写入 MemStore，按 RowKey 排序（Java TreeMap 结构）；
3. **异步 Flush**：当 MemStore 达到阈值（默认 128MB）后，Flush 成 HFile 持久化到 HDFS；
4. **成功响应**：客户端在数据写入 MemStore 后即收到 ACK，实际已保证持久化。

#### WAL 的核心价值

| 场景 | WAL 的作用 |
|------|-----------|
| **RegionServer 宕机** | MemStore 数据未刷盘丢失，从 WAL 重放恢复 |
| **网络抖动/写入失败** | WAL 未关闭时可重试，不丢数据 |
| **集群故障转移** | 其他 RegionServer 接管时从 WAL 重放未持久化的数据 |

> 📎 来源：[HBase 官方文档 - Write Path](https://hbase.apache.org/book.html#wal)

## 2. HBase 的 Compaction 机制是什么？Minor Compaction 和 Major Compaction 有何区别？

### 答案

Compaction 是 HBase 用于合并 HFile、清理过期数据、提升读性能的后台进程。

#### 两种 Compaction 类型

| 类型 | 触发条件 | 合并范围 | 特点 |
|------|---------|---------|------|
| **Minor Compaction** | HFile 数量达到阈值（默认 3） | 合并部分小 HFile | 轻量快速，不清理所有过期数据 |
| **Major Compaction** | 手动触发或自动调度（7天默认） | 一个 Region 的所有 HFile | 重写所有数据，清理所有过期/删除标记 |

#### Minor Compaction 流程

1. 挑选待合并的 HFile（按时间/大小策略）；
2. 读取多个小 HFile 内容，按 Key 排序合并；
3. 写入新的合并后 HFile；
4. 删除被合并的旧 HFile。

#### Major Compaction 流程

1. 扫描该 Region 所有 HFile；
2. 读取全部数据，按 RowKey 排序重写为少量新 HFile；
3. **执行物理删除**：清理带 Delete 标记的 Cell；
4. **清理 TTL 到期数据**；
5. 可能触发大量磁盘 I/O（也叫 "Compaction Storm"）。

#### 实际影响

- **读性能**：HFile 数量越少，扫瞄成本越低；
- **写放大**：Major Compaction 会重写大量数据，影响写入；
- **生产建议**：关闭自动 Major Compaction，改为业务低峰期手动触发。

> 📎 来源：[HBase 官方文档 - Compaction](https://hbase.apache.org/book.html#compaction)

## 3. HBase 协处理器（Coprocessor）有哪两类？各自的使用场景是什么？

### 答案

HBase 协处理器是一种扩展机制，允许用户在 RegionServer 上运行自定义代码，就近处理数据，避免大量网络传输。

#### 两类协处理器

##### ① Observer（观察者）— 类似于数据库触发器

在特定事件前后触发，用于**拦截和增强**现有操作。

```
Observer 类型：
├── RegionObserver      操作某个 Row/Region 时触发（Get/Put/Delete/Scan）
├── WALObserver          WAL 写入/滚动时触发
├── MasterObserver       Master 操作（创建表、删除表）时触发
└── Endpoint            对应二级索引
```

**典型使用场景**：
- **二级索引**：Put 数据时自动写入索引表；
- **数据校验**：Put 前校验数据合法性；
- **审计日志**：记录所有写操作到独立表；
- **TTL 变更**：动态修改 Cell 级别 TTL。

##### ② Endpoint（端点）— 类似于存储过程

允许在 RegionServer 上执行**聚合计算**，将计算推到数据端，减少网络传输。

**典型使用场景**：
- Sum/Count/Avg 聚合（类似 MapReduce 但实时）；
- 自定义过滤和转换；
- 跨行跨列族的复杂计算。

#### 加载方式

| 方式 | 配置位置 | 适用场景 |
|------|---------|---------|
| **系统级** | hbase-site.xml | 所有表，所有 RegionServer |
| **表级** |alter table 't1', METHOD => 'table_att' | 特定表 |

```java
// 表级加载示例
hbase> alter 'myTable', METHOD => 'table_att',
  'COPROCESSOR' => 'hdfs://path/to/coprocessor.jar|com.example.Observer|1001|arg1=val1'
```

> 📎 来源：[HBase 官方文档 - Coprocessors](https://hbase.apache.org/book.html#cp)

## 4. HBase 如何保证数据一致性？读操作是否满足 CAP 定理的哪一项？

### 答案

#### HBase 的一致性模型

HBase 提供了 **行级别的事务一致性**，即同一 RowKey 的读写满足以下保证：

- **写一致性**：写入 WAL + MemStore 后返回成功；
- **读一致性**：读取时无分布式锁，通过 MVCC（多版本并发控制）保证同一事务内的读取一致。

#### MVCC 机制

```
1. 写入数据分配写入编号（WriteNumber）
2. 数据写入 MemStore，标记为 "待提交"
3. 读取时只读取 "已提交" 的最大版本数据
4. 提交后旧版本成为可清理垃圾
```

#### CAP 定理中的定位

HBase 是一个 **CP（一致性 + 分区容错性）** 系统：

| 特性 | 说明 |
|------|------|
| **C（一致性）** | 同一 RowKey 多次读取返回相同数据 |
| **A（可用性）** | 分区故障时集群不可用（短暂不可用） |
| **P（分区容错）** | 依赖 HDFS 的副本机制容忍节点故障 |

#### 读流程中的强一致性保证

```
客户端读请求
     │
     ▼
① MemStore（未刷盘数据）→ 有则返回
     │ 无
     ▼
② BlockCache（热点缓存）→ 有则返回
     │ 无
     ▼
③ HFile（磁盘）→ BloomFilter 定位文件 → 扫描返回
     │
     ▼
结果合并 → 返回客户端（一致性视图）
```

> 📎 来源：[HBase 官方文档 - Consistency](https://hbase.apache.org/book.html#strong.examples)

## 5. HBase 的存储结构（MemStore + HFile）中，HFile 的内部结构是怎样的？

### 答案

HFile 是 HBase 数据在 HDFS 上的实际存储格式，是一种 **SSTable（Sorted String Table）** 结构。

#### HFile 物理结构

```
┌────────────────────────────────────────────┐
│  Magic（4字节）│   固定头部标志              │
├────────────────────────────────────────────┤
│  Data Block（数据块）│ 存储 KeyValue 对       │
│  - 最小子单元，默认 64KB                     │
│  - 每个 Block 有自己的 Leaf Index           │
│  - 可配置大小（hfile.block.size）           │
├────────────────────────────────────────────┤
│  Meta Block（可选）│ BloomFilter 等元数据    │
├────────────────────────────────────────────┤
│  File Info│ 末尾元信息（KEY_COUNT等）        │
├────────────────────────────────────────────┤
│  Index Block│ Data Block 的 Root Index     │
├────────────────────────────────────────────┤
│  Trailer│ 偏移量信息、版本号、Magic          │
└────────────────────────────────────────────┘
```

#### HFile 核心组件详解

| 组件 | 作用 |
|------|------|
| **Data Block** | 存储实际的 KeyValue 对，按 RowKey 排序，支持块压缩 |
| **Meta Block** | 存储 BloomFilter（判断 Key 是否存在）和根因追踪信息 |
| **Index Block** | 两级索引：Data Block 级别索引 + 文件级别 Root Index |
| **File Info** | 包含 KeyCount、时间范围、最大时间戳等元信息 |
| **Trailer** | 记录各区块的起始偏移量，是 HFile 读取的入口 |

#### KeyValue 对的结构

```
KeyValue = RowKeyLength(2B) + RowKey + ColumnFamilyLength(1B) 
         + ColumnFamily + ColumnQualifier + Timestamp(8B) 
         + KeyType(Put/Delete) + ValueLength(4B) + Value
```

#### 读取加速手段

- **BloomFilter**：写入时建立，读取时快速判断某 RowKey 是否存在于某 HFile，避免无效扫描；
- **BlockCache**：热点数据缓存在 RegionServer 内存（LRU 策略）；
- **Short-Circuit Read**：HDFS 允许绕过 DataNode 直接读本地磁盘数据。

> 📎 来源：[HBase 官方文档 - HFile Format](https://hbase.apache.org/book.html#hfile)
