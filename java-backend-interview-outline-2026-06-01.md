## 一、Java基础与核心特性

### 1. 面向对象特性

Java的四大面向对象特性为**抽象、封装、继承和多态**：
- **抽象**：将共性行为或属性提取成类的过程
- **封装**：通过访问修饰符（如private）隐藏内部实现，仅暴露必要接口
- **继承**：允许子类复用父类代码，并支持方法重写
- **多态**：满足三个前提——存在继承或实现关系、有方法重写、父类引用指向子类对象，实现运行时动态绑定

**接口与抽象类的核心区别：**
- **接口**强调"能做什么"，支持多实现，方法默认为`public abstract`，属性默认为`public static final`，适用于定义能力契约
- **抽象类**侧重"是什么"，用于代码复用，遵循单继承原则，可包含构造器、非抽象方法及成员变量，适合构建具有共同基础逻辑的类族

---

### 2. String 类家族

| 类 | 线程安全 | 性能 | 适用场景 |
|----|---------|------|---------|
| **String** | 线程安全 | 低（每次修改创建新对象） | 少量操作 |
| **StringBuffer** | 线程安全 | 较低（synchronized修饰） | 多线程大量字符串操作 |
| **StringBuilder** | 非线程安全 | 高（无同步控制） | 单线程大量字符串拼接（推荐） |

---

### 3. == 与 equals 区别

- **`==`**：比较基本数据类型时判断值是否相等；比较引用类型时判断两个引用是否指向同一内存地址
- **`equals()`**：由Object类定义，默认行为与 `==` 相同（比较内存地址）；String、Integer等包装类重写了此方法，实现基于内容的相等性判断

```java
// String 重写了 equals
String s1 = new String("hello");
String s2 = new String("hello");
s1 == s2        // false（不同内存地址）
s1.equals(s2)   // true（内容相同）
```

---

### 4. 泛型与反射

**泛型擦除机制：**
- 编译器在编译期间移除泛型信息，替换为原始类型（如Object）或指定的上界类型
- 兼容JDK 1.5之前的版本，字节码层面不存在泛型约束
- 无法通过反射获取实际的泛型参数类型

**反射的应用场景：**
- **Spring IoC**：动态创建Bean实例并注入依赖
- **AOP**：生成代理对象织入切面逻辑
- **JDBC驱动加载**：`Class.forName()` 加载数据库驱动

## 二、集合框架与并发编程

### 1. ArrayList vs LinkedList

| 对比项 | ArrayList | LinkedList |
|--------|-----------|------------|
| **底层结构** | 动态数组 | 双向链表 |
| **查询性能** | O(1)，通过索引直接访问 | O(n)，需从头或尾遍历 |
| **尾部插入删除** | 快，但扩容时需复制元素 | 极快，只需调整指针 |
| **中间插入删除** | 慢，需移动后续所有元素 | 快，仅修改前后节点指针 |
| **内存占用** | 较少，仅存储数据 | 较多，每个节点额外存储前驱后继指针 |

---

### 2. HashMap 与 ConcurrentHashMap

| 项目 | HashMap | ConcurrentHashMap |
|------|---------|-------------------|
| **线程安全** | 否 | 是 |
| **JDK 1.8 结构** | 数组 + 链表/红黑树 | Node数组 + CAS + synchronized |
| **锁粒度** | 无锁 | 头节点级别，细粒度锁 |
| **扩容机制** | 单线程扩容 | 多线程协助迁移，提高效率 |
| **size() 准确性** | 并发下可能不准 | 使用CounterCell分段统计，保证准确性 |

---

### 3. 线程池核心参数与设置

线程池五大核心参数：
1. **corePoolSize**：核心线程数，即使空闲也不会被回收
2. **maximumPoolSize**：最大线程数，当队列满且任务继续提交时，创建新线程直至达到此上限
3. **keepAliveTime**：非核心线程的空闲存活时间，超时后将被终止
4. **workQueue**：任务队列，缓存等待执行的任务
5. **handler**：拒绝策略，线程池饱和且队列满时处理新任务的方式

**线程数设置建议：**
- **CPU密集型任务**：核心线程数 = CPU核心数 + 1
- **IO密集型任务**：核心线程数 = CPU核心数 × 2 或更高

---

### 4. volatile 关键字

volatile 保障两方面语义：

- **可见性**：强制线程读取变量时从主内存刷新最新值，写入时立即同步回主内存
- **有序性**：插入内存屏障防止指令重排序

**不保证原子性**：`i++` 这样的自增操作需要借助 `AtomicInteger` 等原子类解决

---

### 5. synchronized 锁升级过程

`synchronized` 的锁升级过程：

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
```

| 阶段 | 条件 | 实现方式 |
|------|------|---------|
| **偏向锁** | 无竞争 | 锁标记给首个获取它的线程，避免重复CAS开销 |
| **轻量级锁** | 轻微竞争 | CAS操作尝试获取锁，减少系统调用成本 |
| **重量级锁** | 竞争激烈 | 依赖操作系统Mutex实现阻塞，线程进入休眠释放CPU |

---

### 6. ThreadLocal 原理与内存泄漏

**原理：** 每个线程维护一个独立的 `ThreadLocalMap`，key为ThreadLocal实例的**弱引用**，value为用户数据的**强引用**。

**内存泄漏原因：** key的弱引用在外部强引用断开后能被GC，但value因强引用无法自动清理，若线程长期复用（如线程池），便会造成泄漏。

**解决方案：** 使用完ThreadLocal后务必调用 `remove()` 方法，主动清除map中的entry。

## 三、JVM 与性能调优

### 1. JVM 运行时数据区

| 区域 | 是否线程共享 | 主要功能 |
|------|------------|---------|
| **堆（Heap）** | 是 | 存放对象实例和数组，GC的主要区域 |
| **方法区（元空间）** | 是 | 存储类信息、常量、静态变量、即时编译后的代码 |
| **虚拟机栈** | 否 | 执行Java方法，管理栈帧的入栈出栈 |
| **本地方法栈** | 否 | 为Native方法服务 |
| **程序计数器** | 否 | 记录当前线程所执行字节码的行号 |

---

### 2. 垃圾回收算法

| 算法 | 原理 | 优缺点 |
|------|------|-------|
| **标记-清除** | 标记存活对象，清除未标记对象 | 产生内存碎片 |
| **复制算法** | 内存分两块，存活对象复制到另一块 | 无碎片，但内存利用率仅50%，适用于新生代 |
| **标记-整理** | 标记后将存活对象向一端移动 | 解决碎片问题，适用于老年代 |
| **G1混合回收** | 采用Region分区，可预测停顿时间 | 兼顾吞吐与延迟 |

---

### 3. 常见垃圾回收器对比

| 收集器 | 适用代 | 特点 |
|--------|--------|------|
| **Serial** | 新生代 | 单线程收集，Client模式默认，简单高效 |
| **Parallel Scavenge** | 新生代 | 多线程，追求高吞吐量，适合后台批处理 |
| **CMS** | 老年代 | 并发标记清除，低延迟，但已弃用，易产生碎片 |
| **G1** | 整堆 | 分Region管理，可预测停顿，支持大堆内存 |
| **ZGC** | 整堆 | 超低暂停时间，支持TB级堆，适用于实时性要求极高的场景 |

### 4. OOM 排查与调优

**常用工具：**
- **jps**：列出Java进程
- **jstack**：打印线程堆栈，定位死锁或死循环
- **jmap**：导出堆转储文件
- **MAT（Memory Analyzer Tool）**：分析dump文件，找出内存泄漏根源

**常用调优参数：**
```bash
-Xms 和 -Xmx          # 设置堆初始与最大大小，建议两者相等
-XX:+UseG1GC          # 启用G1收集器

```

## 四、Spring 全家桶

### 1. IoC 与 AOP

- **IoC（控制反转）**：将对象的创建和依赖管理权交给Spring容器，开发者不再手动new对象，实现组件间解耦
- **DI（依赖注入）**：IoC的具体实现方式，容器自动装配依赖到目标Bean
- **AOP（面向切面编程）**：通过动态代理（JDK动态代理或CGLIB），在不修改业务代码的前提下，将日志、事务、权限等横切关注点统一织入，提升代码可维护性和复用性

---

### 2. Spring Bean 生命周期

```
实例化 → 属性填充 → 初始化前回调(@PostConstruct)
       → 初始化(afterPropertiesSet/init-method)
       → AOP代理生成（初始化后置处理器）
       → Bean放入单例池 → 销毁(destroy-method)
```

---

### 3. 循环依赖解决方案

Spring通过**三级缓存**解决单例Bean的循环依赖：

| 缓存 | 内容 | 作用 |
|------|------|------|
| **一级** | 完全初始化好的Bean | singletonObjects |
| **二级** | 早期暴露的对象 | earlySingletonObjects，解决普通循环依赖 |
| **三级** | ObjectFactory | singletonFactories，延迟创建代理对象，解决AOP循环依赖 |

---

### 4. Spring Boot 自动装配原理

`@EnableAutoConfiguration` 注解是自动装配入口：
1. 导入 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 中的所有自动配置类
2. 通过 `@ConditionalOn...` 注解（如`@ConditionalOnClass`、`@ConditionalOnMissingBean`）进行条件判断
3. 仅当条件满足时，该配置才生效，实现按需加载

---

### 5. Spring 事务失效场景

以下八种场景会导致声明式事务失效：

1. 方法被定义为 `private`、`final` 或 `static`，导致AOP代理失败
2. 在同一类中，通过 `this` 直接调用带有 `@Transactional` 的方法，绕过代理
3. 异常被捕获但未抛出，事务管理器认为方法成功执行
4. 抛出非 `RuntimeException` 的受检异常，而 `rollbackFor` 未配置
5. 事务传播行为配置错误（如在已有事务中使用 `NEVER`）
6. 目标对象不是Spring容器管理的Bean
7. 数据库引擎不支持事务（如MyISAM）
8. 在 `readOnly=true` 的事务中执行写操作

## 五、MySQL 高频考点

### 1. DATETIME vs TIMESTAMP

| 对比项 | DATETIME | TIMESTAMP |
|--------|----------|-----------|
| **存储范围** | 1000-9999年 | 1970-2038年 |
| **时区处理** | 与时区无关，存储字面值 | 存储UTC时间戳，查询时根据会话时区转换 |
| **存储空间** | 8字节 | 4字节 |
| **自动更新** | 需显式设置ON UPDATE | 可设置自动更新 |

---

### 2. 字符集选择

- **utf8**（utf8mb3）：最多支持3字节字符，无法存储emoji等4字节字符
- **utf8mb4**：真正的UTF-8编码，兼容utf8并支持完整的4字节字符，**现代应用推荐**

---

### 3. 金额字段类型

金额字段**必须使用 DECIMAL(M,D)** 类型：

```sql
price DECIMAL(10,2)  -- 总长10位，小数点后2位
```

**禁止使用 FLOAT 或 DOUBLE**：基于IEEE 754，存在精度丢失风险（如 `0.1 + 0.2 ≠ 0.3`）

---

### 4. CHAR vs VARCHAR

| 对比项 | CHAR(n) | VARCHAR(n) |
|--------|---------|------------|
| **存储方式** | 定长，不足n个字符用空格填充 | 变长，按实际长度存储 |
| **性能** | 读写更快 | 稍慢，需记录长度信息 |
| **适用场景** | 固定长度数据（手机号、身份证号） | 长度变化大的数据（用户名、地址） |

---

### 5. B+树索引优势

- **非叶子节点仅存储索引键和指针**，不存数据，单个节点能容纳更多键，降低树高，减少磁盘I/O
- **所有数据存储在叶子节点**，通过双向链表相连，天然支持高效范围查询和全表扫描
- **平衡性保证查询性能稳定**

---

### 6. 索引失效场景

常见索引失效场景：
- 对索引列使用函数或表达式（如 `WHERE YEAR(create_time) = 2026`）
- 隐式类型转换（如varchar字段与数字比较）
- LIKE模糊查询以 `%` 开头（`LIKE '%张三'`）
- OR连接的多个条件中存在无索引字段
- 对索引列进行运算
- 使用 `NOT IN`、`!=` 等负向查询
- 未遵循最左前缀原则的联合索引查询

## 六、Redis 核心面试题

### 1. 五大基本数据类型及场景

| 类型 | 典型用途 |
|------|---------|
| **String** | 缓存对象（JSON序列化）、Token、计数器 |
| **Hash** | 存储对象属性，如用户信息、购物车 |
| **List** | 消息队列、最新动态列表 |
| **Set** | 去重、标签系统、共同好友 |
| **ZSet** | 排行榜、延迟队列 |

---

### 2. 缓存三大问题及解决方案

| 问题 | 场景 | 解决方案 |
|------|------|---------|
| **缓存穿透** | 请求数据库和缓存都不存在的数据 | 布隆过滤器拦截非法Key，或缓存空值并设置短TTL |
| **缓存击穿** | 热点Key过期瞬间，大量请求直达DB | 互斥锁（如Redisson）或逻辑过期（异步刷新） |
| **缓存雪崩** | 大量Key同时过期或Redis宕机 | 设置随机TTL、部署多级缓存、实施熔断降级 |

---

### 3. Redis 为什么快？

1. **所有数据操作都在内存中**，避免磁盘I/O瓶颈
2. **单线程模型**，避免多线程上下文切换和锁竞争开销
3. **基于IO多路复用**（如epoll）处理网络事件，高效支撑海量连接
4. **底层数据结构精心设计**（跳跃表、压缩列表），保证操作高效性

---

### 4. 分布式锁实现

| 方案 | 说明 |
|------|------|
| **SETNX + EXPIRE** | 基础方案，通过Lua脚本保证原子性 |
| **Redisson WatchDog** | 自动续期，防止锁因业务执行时间过长而意外释放 |
| **RedLock** | 向多个独立Redis实例加锁提升可靠性（复杂，有争议） |
| **ZooKeeper / etcd** | 生产环境更推荐，CP系统实现分布式锁 |

## 七、分布式与微服务

### 1. CAP 定理

分布式系统无法同时满足：
- **C（Consistency）**：一致性
- **A（Availability）**：可用性
- **P（Partition Tolerance）**：分区容错性

由于网络分区不可避免（P必须保证），只能在C和A之间权衡：
- **AP系统**（如Eureka）：优先保证服务可用，允许短暂数据不一致
- **CP系统**（如ZooKeeper）：优先保证数据强一致，牺牲部分可用性

---

### 2. 分布式事务解决方案

| 方案 | 特点 | 适用场景 |
|------|------|---------|
| **2PC** | 两阶段提交，强一致性，但性能差、存在单点故障 | 传统银行核心系统 |
| **TCC** | Try-Confirm-Cancel，业务侵入性强，但灵活性高 | 支付、订单等复杂业务流程 |
| **Saga** | 长事务拆分为多个子事务，通过补偿机制实现最终一致 | 订单履约、跨服务业务流 |
| **本地消息表** | 通过可靠消息实现最终一致，实现简单 | 异步通知、积分发放 |

---

### 3. 分布式ID生成

| 方案 | 原理 | 优缺点 |
|------|------|-------|
| **Snowflake** | 64位ID（时间戳+机器ID+序列号） | 性能优异，但需注意时钟回拨问题 |
| **Leaf-Segment** | 数据库号段缓冲，双buffer预加载 | 高并发下性能好，适用于金融级场景 |

---

### 4. 接口幂等性保障

| 方案 | 原理 |
|------|------|
| **Token机制** | 客户端先申请唯一Token，提交时携带，服务端校验并删除 |
| **分布式锁** | 执行关键操作前获取锁，防止并发 |
| **唯一索引** | 利用数据库唯一约束，防止重复插入 |
| **乐观锁** | 更新时带版本号（version），仅版本匹配才执行 |

## 八、系统设计与场景题

### 1. 秒杀系统设计要点

- **分层削峰**：前端Nginx限流 → 网关层鉴权过滤 → Redis预减库存 → MQ异步下单
- **热点隔离**：秒杀商品建立独立Redis集群，避免影响主业务
- **超卖控制**：使用Redis的 `DECR` 命令或Lua脚本保证库存扣减原子性
- **兜底方案**：Redis不可用时，降级至数据库检查库存

---

### 2. 百万点赞设计

- **Redis ZSet**：记录用户与被赞对象的关系，Score存储点赞时间，支持按时间排序
- **分片**：按用户ID分片，将压力分散到多个Redis实例
- **异步持久化**：最终数据通过定时任务或MQ异步落MySQL，保证最终一致性

---

### 3. 短链接生成服务

- **算法**：长URL → MD5哈希 → 取部分作为ID → Base62编码（0-9,a-z,A-Z）转短字符串
- **高并发**：预生成ID池，批量分配
- **存储**：Redis缓存映射提供快速响应 + MySQL持久化
