## 题目一：Spark 架构与核心组件

**请描述 Apache Spark 的整体架构，并说明 Driver、Executor、Worker Node、Cluster Manager 各自的作用以及它们之间的协作关系。**

### 参考答案

Spark 采用 **主从（Master-Slave）架构**，核心组件如下：

| 组件 | 作用 |
|------|------|
| **Driver（驱动器）** | 运行用户 `main()` 函数，创建 `SparkContext`，负责将用户代码转换为 DAG、划分 Stage、调度 Task 到 Executor |
| **Executor（执行器）** | 运行在 Worker Node 上的进程，负责执行分发的 Task、缓存数据、向 Driver 汇报结果 |
| **Worker Node** | 集群中实际存储数据和处理计算的工作节点 |
| **Cluster Manager** | 集群资源管理服务（Standalone / YARN / Mesos），负责分配集群资源、启动 Application |

**工作流程：**

1. 用户提交 Spark Application，Driver 创建一个 `SparkContext`
2. Driver 向 Cluster Manager 申请资源，启动 Executor
3. Driver 将应用代码（jar/py 文件）发送给各个 Executor
4. `SparkContext` 将用户代码编译为 DAG，再拆分为多个 Stage，每个 Stage 包含若干 Task
5. Task 被分发到各个 Executor 并行执行
6. 执行结果返回 Driver，Driver 汇总后输出

## 题目二：RDD  Transformation 与 Action 的区别

**Spark 中 Transformation 和 Action 有什么区别？请举例说明，并解释 Spark 为什么对 Transformation 采用懒评估（Lazy Evaluation）策略。**

### 参考答案

| 特性 | Transformation | Action |
|------|---------------|--------|
| **返回值** | 新的 RDD | 其他数据类型（Int/List/Void） |
| **执行时机** | 懒评估，不会立即执行 | 触发 Job 执行 |
| **示例** | `map`, `filter`, `flatMap`, `groupByKey` | `collect`, `count`, `saveAsTextFile`, `reduce` |

**懒评估的原理：**
- Transformation 只是记录操作轨迹（Lineage），构建 DAG，不触发实际计算
- 直到遇到 Action 时，Spark 才完整构建 DAG 并提交执行
- 这种设计可以**减少中间结果落盘**，优化执行计划（如 pipeline 合并、减少 shuffle）

**举例说明：**

```python
# Transformation（懒评估，只记录操作）
rdd = sc.textFile("data.txt")         # 读取文件（此时未真正加载）
rdd2 = rdd.map(lambda x: x.upper())   # 记录 map 操作
rdd3 = rdd2.filter(lambda x: "ERROR" in x)  # 记录 filter 操作

# Action（触发真正的计算）
result = rdd3.collect()  # 此时才真正读取文件并执行 map 和 filter
```

## 题目三：Spark Shuffle 机制与性能优化

**什么是 Shuffle？为什么 Shuffle 是 Spark 性能瓶颈？请说明 map output 和 reduce input 之间的数据流转过程，并给出 3 种以上的优化策略。**

### 参考答案

**Shuffle 的定义：**
当 RDD 依赖关系中需要**重新分区数据**（如 `groupByKey`、`reduceByKey`、`join`、`sortByKey` 等）时，Spark 需要将数据按照 Key 进行重新分布——这个跨节点拉取数据的过程称为 **Shuffle**。

**性能瓶颈原因：**
- **跨节点网络 IO**：数据需要从 map 端写入磁盘再由 reduce 端拉取，网络开销巨大
- **磁盘 IO**：Shuffle 数据通常需要先落盘
- **内存压力**：大量中间数据占用内存，可能触发 GC

**数据流转过程：**

```
Map Task 输出 → 本地磁盘（分区文件）→ Fetch 请求 → Reduce Task 输入缓冲区 → 执行 reduce
```

Shuffle 过程中，map 端按照 `partitioner` 将数据写入本地磁盘（通常有合并机制减少文件数），reduce 端通过 HTTP 请求从各 map 节点拉取属于自己的分区数据。

**优化策略：**

1. **使用合理数量的分区**：分区过少导致并行度不够，过多则增加 shuffle 文件数。建议每个 CPU 核 2-4 个分区
2. **使用 map-side 预聚合**：用 `reduceByKey`/`aggregateByKey` 替代 `groupByKey`，在 map 端先做本地聚合，减少网络传输数据量
3. **启用排序合并（Sort Shuffle）**：Spark 2.x 后的默认模式，比 hash-based shuffle 更稳定
4. **使用 Kryo 序列化**：`conf.set("spark.serializer", "org.apache.spark.serializer.KryoSerializer")`，减少网络传输数据量
5. **广播小表**：对于 join 中较小的表，使用 `broadcast()` 广播到每个 Executor，避免 shuffle

> **来源：** [Apache Spark 官方文档 - Tuning Guide](https://spark.apache.org/docs/latest/tuning.html#shuffle-operations)

## 题目四：广播变量（Broadcast Variable）原理与使用场景

**什么是广播变量？它解决了什么问题？请说明广播变量的工作原理，并与普通变量进行对比，最后给出一个实际使用场景。**

### 参考答案

**定义：**
广播变量是 Spark 提供的一种**只读共享变量**，可将一份数据缓存到**每个 Executor 的内存**中，而不是每个 Task 复制一份。从而避免每次 Task 执行时重复传输大变量。

**解决的问题：**
- 减少网络传输：每个 Executor 只传输一次，而非每个 Task 传输一次
- 节省内存：Executor 内所有 Task 共享同一份数据副本

**工作原理：**

```
Driver 端：broadcast(data) → Spark 将数据分成 broadcast blocks（默认 4MB）
           → 每个 Executor 在首次访问时通过 BitTorrent 风格下载到本地
           → 后续 Task 直接从 Executor 内存读取，不再重复传输
```

**普通变量 vs 广播变量：**

| 特性 | 普通变量 | 广播变量 |
|------|---------|---------|
| 作用域 | 每个 Task 独立副本 | Executor 内共享 |
| 传输方式 | 随 Task 一起序列化到各节点 | 一次性广播到 Executor |
| 修改 | 可修改 | **只读** |
| 适用场景 | 小数据 / 频繁变化 | 大数据 / 只读数据 |

**使用示例：**

```python
# 普通方式：dim_table 会在每个 task 中复制一份，网络开销大
result = facts.map(lambda x: (x.key, x.value * dim_table.get(x.key)))

# 使用广播变量：dim_table 只传输一次到每个 Executor
dim_table_bc = sc.broadcast(dim_table_dict)
result = facts.map(lambda x: (x.key, x.value * dim_table_bc.value.get(x.key)))
```

**使用场景：**
- ETL 中维度表（如日期维表、城市码表）的关联
- 机器学习中共享特征映射表
- 配置信息、查询参数等只读数据的分发

> **来源：** [Apache Spark 官方文档 - RDD Programming Guide](https://spark.apache.org/docs/latest/rdd-programing-guide.html#broadcast-variables)

## 题目五：Spark 内存管理机制

**请详细描述 Spark 的内存管理模型，包括 Execution Memory 和 Storage Memory 的区别、如何调优 `spark.memory.fraction` 和 `spark.memory.storageFraction` 参数，以及如何排查和解决 OOM 问题。**

### 参考答案

**统一内存管理模型（Spark 1.6+）：**

Spark 将 JVM 堆内存划分为两部分共享区域 M：

```
JVM 堆内存 = M（执行+存储统一区域） + 用户数据结构 + 保留空间（300MB）
     M = spark.memory.fraction × (堆内存 - 300MB)  ≈ 60%
```

| 内存类型 | 用途 | 特性 |
|---------|------|------|
| **Execution Memory** | Shuffle、Join、Sort、聚合等计算 | 可抢占：存储内存空闲时可借用 |
| **Storage Memory** | RDD 缓存（`persist`/`cache`）、广播变量 | 有保底空间 R：`spark.memory.storageFraction × M`（默认 50%） |

**关键参数：**

```properties
spark.memory.fraction=0.6       # M 占堆内存比例（默认 0.6）
spark.memory.storageFraction=0.5 # R（保底存储空间）占 M 的比例（默认 0.5）
```

**RDD 存储级别（StorageLevel）：**
- `MEMORY_ONLY`：纯内存，高效但占用大
- `MEMORY_ONLY_SER`：序列化后存内存，省空间但 CPU 开销大
- `MEMORY_AND_DISK`：内存不足时溢写到磁盘
- `DISK_ONLY`：纯磁盘，适用于大数据集

**OOM 排查与解决思路：**

1. **估算内存消耗**：`SizeEstimator.estimate()` 可估算 RDD 对象内存占用
2. **使用序列化存储**：减少对象开销，`MEMORY_ONLY_SER` + Kryo 序列化
3. **优化数据结构**：避免嵌套对象，使用原始类型数组（`fastutil` 库）
4. **降低 `spark.memory.fraction`**：当 GC 频繁时，说明用户数据结构占用了过多空间
5. **增加分区数**：更多小分区减少单分区内存压力
6. **开启 JVM 压缩指针**：` -XX:+UseCompressedOops`（< 32GB 内存时自动开启）
