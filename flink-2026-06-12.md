## 题目一：Flink 的整体架构是怎样的？请解释 JobManager 和 TaskManager 的作用

### 参考答案

Flink 采用**主从架构**，由 JobManager（主节点）和 TaskManager（工作节点）组成。

**JobManager** 负责：
- 接收 Flink Job（算子 DAG 图）
- 将 Job 解析成 **ExecutionGraph**（执行图）并分发到 TaskManager
- 协调 Checkpoint 和故障恢复
- 调度 Task 到 TaskManager 的 Slot 中执行

**TaskManager** 负责：
- 真正执行 Task（算子的 Subtask）
- 管理 **Slot**（资源单位），每个 Slot 可运行一个或多个 Subtask
- 向 JobManager 汇报心跳和状态

**StreamGraph → JobGraph → ExecutionGraph 的转换过程：**
1. 用户代码生成 **StreamGraph**（算子拓扑）
2. StreamGraph 优化生成 **JobGraph**（带链化优化）
3. JobManager 将 JobGraph 转换为 **ExecutionGraph**（并行化执行计划）

```
Source ──map──> KeyBy/Window ──Sink
   │            │               │
StreamGraph → JobGraph → ExecutionGraph
```

**并行度机制：** 一个算子的并行度决定了其 Subtask 的数量。Stream 分区（one-to-one 或 redistributing）决定了上下游 Subtask 之间的数据流向。

📎 **来源：** [Apache Flink 官方文档 - 架构概述](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/learn-flink/overview/)

## 题目二：什么是 Watermark？为什么需要 Watermark？如何生成？

### 参考答案

**Watermark（水位线）是 Flink 用于处理**事件时间（Event Time）**乱序问题的核心机制。

### 为什么需要 Watermark？

在现实场景中，由于网络延迟、机器故障等原因，事件到达的顺序往往与事件自带的时间戳不一致（即**乱序**）。Flink 需要判断某个时间窗口的数据是否已经到齐，才能触发窗口计算。

Watermark 本质上是一个**时间戳**，表示"时间 ≤ T 的所有事件都已经到达（或者说不会再有早于 T 的事件到达）"。

### Watermark 的生成方式

**1. 在 Source 层面生成（推荐）：**
```java
DataStream<String> stream = env
    .source(new FlinkKafkaConsumer<>("topic", new SimpleStringSchema(), props))
    .assignTimestampsAndWatermarks(
        WatermarkStrategy
            .<String>forBoundedOutOfOrderness(Duration.ofSeconds(10))
            .withTimestampAssigner((event, timestamp) -> extractTimestamp(event))
    );
```

**2. 周期性生成（Periodic）：** 通过 `AssignerWithPeriodicWatermarks` 周期性生成 Watermark，适用于数据量稳定场景。

**3. 标点生成（Punctuated）：** 通过 `AssignerWithPunctuatedWatermarks` 在特定事件触发时生成，适用于事件驱动的精确 Watermark。

### Watermark 与窗口的结合

Flink 的窗口机制会等 Watermark 越过窗口结束时间后才触发窗口计算：
- **延迟触发：** 当 `Watermark.watermark_timestamp >= window_end_time` 时，窗口触发
- **乱序容忍：** `forBoundedOutOfOrderness(Duration.ofSeconds(10))` 表示最多容忍 10 秒乱序

📎 **来源：** [Apache Flink 官方文档 - 事件时间与 Watermark](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/learn-flink/event_time/)

## 题目三：Flink Checkpoint 的原理是什么？如何实现 Exactly-Once 语义？

### 参考答案

### Checkpoint 核心原理

Flink 的 Checkpoint 基于 **Chandy-Lamport 分布式快照算法**，在不影响数据流处理的前提下，对分布式状态做一致性快照。

**核心流程：**
1. **JobManager 定期发起 Checkpoint**（通过 `checkpoint coordinator`）
2. **Checkpoint Coordinator 向所有 Source 算子注入 `Barrier`（屏障）**
3. Source 收到 Barrier 后**保存当前读取的 Kafka offset**，然后将 Barrier 随数据向下游广播
4. **每个算子收到所有输入的 Barrier 后**，对自己的状态做快照，然后继续处理数据
5. **状态快照持久化到存储**（HDFS/S3/RocksDB），完成后通知 JobManager

```
Source ──[Barrier]──> Operator ──[Barrier]──> Operator
  │保存offset           │快照状态         │快照状态
  └─────────────────── ↓ ──────────────── ↓
                    Checkpoint Storage (HDFS/S3)
```

### Exactly-Once 实现的关键点

- **Barrier 对齐（Barrier Alignment）：** 当有多个输入流时，算子必须等所有输入的 Barrier 都到达才做快照，确保全局一致性
- **两阶段提交（Two-Phase Commit）：** Sink 算子（尤其是写入外部系统的 Sink）使用 Two-Phase Commit 协议，在 Checkpoint 完成后才真正提交数据，保证输出不重复
- **Source 端可重放：** Kafka 等消息队列支持 offset 重放，Source 可从 Checkpoint 保存的位置重新读取

**常用 Checkpoint 配置：**
```java
env.enableCheckpointing(60_000);              // 每60秒一次
env.getCheckpointConfig().setCheckpointTTL(Duration.ofHours(2));
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(500);
```

### 与 Spark 对比

| 特性 | Flink Checkpoint | Spark Streaming |
|------|-----------------|-----------------|
| 增量 | 支持 RocksDB 增量 Checkpoint | micro-batch 不可增量 |
| 延迟 | 亚秒级（异步快照不阻塞） | 较慢（需等批次完成） |
| 精确一次 | 端到端 Exactly-Once | 端到端 At-Least-Once |

📎 **来源：** [Apache Flink 官方文档 - Checkpoint](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/learn-flink/fault_tolerance/)

## 题目四：Flink 窗口函数的分类？Session Window 和 Tumbling Window 的区别？

### 参考答案

### 窗口函数分类

Flink 支持两大类窗口：

**1. 时间窗口（Time Windows）**
- **Tumbling Window（滚动窗口）：** 大小固定、不重叠
  ```java
  stream.keyBy(...).window(TumblingEventTimeWindows.of(Time.seconds(10)))
  ```
- **Sliding Window（滑动窗口）：** 大小固定、可重叠
  ```java
  stream.keyBy(...).window(SlidingEventTimeWindows.of(Time.seconds(10), Time.seconds(5)))
  ```
- **Session Window（会话窗口）：** 由 inactivity gap 分隔
  ```java
  stream.keyBy(...).window(SessionWindow.withGap(Time.seconds(10)))
  ```

**2. 计数窗口（Count Windows）**
- 按接收到的元素数量触发
  ```java
  stream.keyBy(...).countWindow(100)  // 每100条触发
  stream.keyBy(...).countWindow(100, 50) // 滑动计数窗口
  ```

### Tumbling Window vs Session Window 核心区别

| 特性 | Tumbling Window | Session Window |
|------|-----------------|----------------|
| **触发时机** | 固定时间/数量，到点即触发 | 超过指定 gap 无活动才触发 |
| **窗口长度** | 固定，事前已知 | 动态，事后才知道 |
| **重叠性** | 不重叠 | 不重叠，但 gap 可重叠 |
| **适用场景** | 周期性统计（每分钟 PV） | 用户行为分析（用户会话） |
| **资源占用** | 稳定可预估 | 动态波动 |

### Session Window 的特殊点

- **Gap（会话间隔）：** 当两个事件之间的时间差超过 gap，窗口关闭
- **动态大小：** 每个窗口长度取决于该会话的实际活跃时长
- **可合并：** 如果后续事件与已有窗口 gap 重叠，窗口会自动合并

**示例：用户点击行为分析**
```
用户点击 → 点击 → 点击 → [gap 5秒无操作] → 窗口关闭
Tumbling: 每10秒一个桶，无论用户有没有操作都触发
Session:  用户停止操作5秒才关闭窗口，精准反映真实会话
```

📎 **来源：** [Apache Flink 官方文档 - 窗口函数](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/learn-flink/streaming/)

## 题目五：Flink 状态管理机制？State Backend 有哪些？如何选型？

### 参考答案

### 状态管理机制

Flink 是**有状态的流处理框架**，状态（State）指算子在处理过程中需要记住的信息（如聚合和、计数器、缓存等）。

**状态分类：**
- **Keyed State：** 按 Key 分组，只能在 `keyBy` 之后使用，作用于当前 Key 的数据
  - `ValueState<T>`、`MapState<K,V>`、`ListState<T>`、`ReducingState<T>` 等
- **Operator State：** 作用于整个算子实例，所有 Key 共享
  - `ListState<T>`、`UnionListState<T>`、`BroadcastState<K,V>`

**状态访问特点：** 状态始终本地存储（TaskManager 内存或 RocksDB），高性能访问，不需要跨网络。

### State Backend（状态后端）选型

Flink 有三种 State Backend：

| Backend | 存储介质 | 适用场景 | 容量 | 性能 |
|---------|---------|---------|------|------|
| **HashMapStateBackend** | JVM Heap | 小状态、低延迟 | 受限于 JVM 堆内存 | 最高 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘） | 大状态、长窗口 | 磁盘容量决定，可扩展 | 略低（SSD 推荐） |

**选型建议：**
- **小状态（MB 级）、低延迟：** 用 `HashMapStateBackend`（默认）
- **大状态（GB 级）、长窗口、巨大 Key 基数：** 用 `RocksDBStateBackend`

### 典型使用示例

```java
// 配置 RocksDB State Backend（适合大状态）
Configuration config = new Configuration();
EmbeddedRocksDBStateBackend rocksDB = new EmbeddedRocksDBStateBackend(true);
env.setStateBackend(rocksDB);
env.getCheckpointConfig().setCheckpointStorage("hdfs://namenode:8020/flink-checkpoints");
```

### 状态过期与清理

- **TTL（Time-To-Live）：** 可为状态设置过期时间，自动清理
  ```java
  StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.days(7))
      .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
      .build();
  ```
- **RocksDB 增量 Checkpoint：** 只 Checkpoint 新增的 SST 文件，大幅减少 Checkpoint 数据量
