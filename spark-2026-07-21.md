## 题目一：Spark 架构原理——Driver 与 Executor 的工作机制

### 📝 题目

请描述 Spark 的整体架构，Driver 和 Executor 在 Spark 作业中各自承担什么角色？它们之间如何协作完成一个 Spark 任务？

### ✅ 答案

**Spark 整体架构**

Spark 采用 **Master-Slave** 架构，核心组件包括 Driver、Executor、Cluster Manager 和 Worker。

```
┌─────────────────────────────────────────────────────┐
│                  Cluster Manager                     │
│              (Standalone / YARN / K8s)              │
└──────────────┬──────────────────────┬────────────────┘
               │                      │
        ┌──────▼──────┐        ┌──────▼──────┐
        │   Worker    │        │   Worker    │
        │             │        │             │
        │ ┌─────────┐ │        │ ┌─────────┐ │
        │ │Executor │ │        │ │Executor │ │
        │ │ Task1   │ │        │ │ Task3   │ │
        │ │ Task2   │ │        │ │ Task4   │ │
        │ └─────────┘ │        │ └─────────┘ │
        └─────────────┘        └─────────────┘
               ▲                      ▲
               │      ┌────────┐      │
               └──────►  Driver ◄─────┘
                      (SparkContext)
```

**Driver（驱动器）**
- 运行用户编写的 `SparkContext` 代码（即 `main` 方法所在节点）
- 负责将用户代码转换为 **DAG（有向无环图）**，将 DAG 划分为 **Stage**，再将 Stage 拆分为 **Task**
- 负责任务的调度、分发和结果汇总
- 与 Cluster Manager 通信，申请 Executor 资源
- **是整个作业的协调者和控制者**

**Executor（执行器）**
- 是 Worker 节点上的 **常驻进程**，负责执行具体的 Task
- 每个 Executor 启动时向 Driver 反向注册，拥有独立的 **JVM 进程**
- Executor 将数据存储在 **内存或磁盘** 中（RDD Cache）
- 一个 Executor 可以并行执行多个 Task（线程池）

**协作流程**
1. 用户提交 Spark 程序，Driver 初始化 `SparkContext`
2. Driver 向 Cluster Manager 申请资源（Executor）
3. Cluster Manager 在 Worker 节点上启动 Executor
4. Executor 向 Driver 反向注册
5. Driver 根据 Shuffle 将 DAG 划分为 Stage，生成 TaskSet
6. Driver 将 Task 分发到各个 Executor 的线程池中执行
7. Executor 执行 Task，将结果返回给 Driver
8. Driver 汇总结果，完成作业

> **来源参考**：  
> - 《Spark设计与实现》 - https://spark.apache.org/docs/latest/architecture.html  
> - 美团技术团队：https://tech.meituan.com/spark-tuning-book.html

## 题目二：RDD 原理——为什么 RDD 是惰性求值？Transformation 与 Action 的区别？

### 📝 题目

Spark 为什么设计 RDD 为惰性求值（Lazy Evaluation）？Transformation 和 Action 操作有什么区别？请举例说明。

### ✅ 答案

**惰性求值（Lazy Evaluation）**

RDD 的 Transformation 操作（如 `map`、`filter`、`flatMap`）**不会立即执行**，只是记录了数据的转换逻辑（Lineage 血统 lineage）。只有当遇到 **Action 操作**（如 `collect`、`count`、`saveAsTextFile`）时，Spark 才会触发真正的计算。

```
RDD 转换流程：

Transformation（不执行）
  rdd1 = sc.textFile("data.txt")           # 只是创建了 RDD 描述
  rdd2 = rdd1.map(_.toUpperCase)           # 只是记录转换逻辑
  rdd3 = rdd2.filter(_.startsWith("A"))    # 只是记录转换逻辑

Action（触发执行）
  result = rdd3.collect()                   # ← 此时才真正执行！
```

**为什么设计为惰性？**
1. **减少中间结果的物化开销**：避免为每个 Transformation 都生成中间数据集
2. **支持 Pipeline 优化**：多个窄依赖的 Transformation 可以合并成一个 Stage，减少数据移动
3. **全链路优化**：Spark 能够看到整个 DAG，进行全局优化（如 Chain 合并、重复利用缓存）

**Transformation vs Action**

| 特征 | Transformation | Action |
|------|--------------|--------|
| 执行时机 | 惰性（不立即执行） | 触发执行（惰性求值） |
| 返回值 | 新的 RDD | 其他类型（Array、Long 等） |
| 示例 | `map`, `filter`, `groupByKey`, `join` | `collect`, `count`, `saveAsTextFile`, `reduce` |
| 链式处理 | 可链式调用 | 终止链式调用 |

**窄依赖 vs 宽依赖（决定 Stage 划分）**
- **窄依赖（Narrow Dependency）**：父 RDD 的每个分区最多被子 RDD 的一个分区使用 → 可以 Pipeline 优化
  - 示例：`map`、`filter`
- **宽依赖（Shuffle Dependency）**：父 RDD 的分区可能被多个子 RDD 分区使用 → 需要 Shuffle，必须划分新的 Stage
  - 示例：`groupByKey`、`join`、`sortByKey`

> **来源参考**：  
> - 官方文档：https://spark.apache.org/docs/latest/rdd-programming-guide.html  
> - 《Learning Spark》- O'Reilly

## 题目三：Shuffle 原理——Spark Shuffle 的工作机制与调优

### 📝 题目

什么是 Shuffle？为什么 Spark 需要 Shuffle？Shuffle 过程中数据如何落地？请说明 Shuffle 的基本原理以及常见的调优策略。

### ✅ 答案

**什么是 Shuffle？**

Shuffle 是 Spark 在 **重新分区数据** 时，需要将数据按照 Key 进行重新分布的操作。当一个 RDD 需要根据 Key 进行 **聚合、连接、排序** 时，必须将相同 Key 的数据汇聚到同一个节点，这就产生了 Shuffle。

```
Shuffle 本质：数据跨节点重新分区

分区 0: (K1, v1), (K3, v3)
分区 1: (K2, v2), (K1, v4)    → Shuffle →   节点 A: K1, K1（合并）
分区 2: (K3, v5), (K2, v6)                    节点 B: K2, K2
                                                节点 C: K3, K3
```

**为什么需要 Shuffle？**
- `groupByKey`、`reduceByKey`、`join`、`sortByKey` 等操作都需要将相同 Key 的数据汇聚
- Shuffle 是 Spark 最昂贵的操作之一，涉及 **磁盘 I/O、网络 I/O、序列化/反序列化**

**Shuffle 的写入流程（Map 端）**
1. Map 任务将数据写入**本地磁盘缓冲区**（buffer）
2. 缓冲区达到阈值后，先**溢写（Spill）** 到磁盘
3. Map 任务完成后，将所有溢写文件**合并（Merge）**成一个数据文件
4. 同时生成**索引文件**（记录每个 Reduce 分区在数据文件中的位置）

**Shuffle 的读取流程（Reduce 端）**
1. Reduce 任务根据索引文件，发起 **Fetch 请求**到各 Map 节点拉取数据
2. 数据拉回后先放入 **内存缓冲区**，达到阈值后溢写磁盘
3. 所有数据拉取完成后，进行**聚合/排序**

**Shuffle 常见调优参数**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `spark.shuffle.file.buffer` | 32KB | Map 端写入文件的缓冲区大小，调大可减少溢写次数 |
| `spark.shuffle.manager` | sort | Shuffle 策略（sort、hash、tungsten-sort） |
| `spark.shuffle.service.enabled` | false | 开启 External Shuffle Service，减少 Executor 崩溃影响 |
| `spark.shuffle.sort.bypassMergeThreshold` | 200 | 当 Reduce 分区数小于此值时，hash Shuffle 绕过合并 |
| `spark.sql.shuffle.partitions` | 200 | Shuffle 时的默认分区数（SQL/DF 操作） |

**最佳实践**
1. 减少 Shuffle 次数：尽量在一次 Shuffle 中完成聚合
2. 合理设置分区数：避免过多小分区或过少大分区（一般 200-500 为宜）
3. 使用 `reduceByKey` 而非 `groupByKey`（前者有 Map 端预聚合）
4. 开启 `spark.shuffle.service.enabled`（生产环境推荐）

## 题目四：广播变量（Broadcast Variable）——原理、用途与注意事项

### 📝 题目

什么是广播变量？它在 Spark 中解决什么问题？如何使用广播变量？使用广播变量有什么注意事项？

### ✅ 答案

**什么是广播变量？**

广播变量是 Spark 提供的一种**只读变量**优化机制，允许将一份**只读的小表/配置/数据**高效地分发到每个 Executor，而不是为每个 Task 单独复制一份。

```
不使用广播变量：
  Task1: 复制 100MB 数据
  Task2: 复制 100MB 数据  →  100MB × N 个 Task = 大量冗余传输
  Task3: 复制 100MB 数据

使用广播变量：
  Executor 共享一份 100MB 数据  →  只传输一次，Executor 内共享
```

**解决的问题**
- 避免大变量在每个 Task 复制一份，造成 **内存浪费和网络传输开销**
- 典型场景：**大表 join 小表** 时，将小表广播到每个节点，避免 Shuffle

**如何使用**

```scala
// 定义广播变量
val smallRDD = sc.parallelize(Seq(("A", 1), ("B", 2)))
val broadcastVar = sc.broadcast(smallRDD.collectAsMap())

// 在算子中使用广播变量（用 .value 读取）
val result = largeRDD.map { case (k, v) =>
  val lookupTable = broadcastVar.value  // 读取广播的数据
  (k, lookupTable.getOrElse(k, 0) + v)
}

// 销毁广播变量（不再使用时）
broadcastVar.unpersist()
```

**注意事项**

1. **只适合小数据量**：广播变量存储在 Executor 的 **堆内存** 中，过大的数据会导致 OOM
2. **只读不可修改**：广播后不可更改值（Spark 不保证同步更新）
3. **在 Action 前销毁**：大广播变量用完后及时 `unpersist()` 释放内存
4. **序列化问题**：广播的变量必须支持 **Kryo 或 Java 序列化**
5. **广播阈值的配置**：
   ```scala
   // 小于这个阈值自动广播（默认 10MB）
   spark.sql.autoBroadcastJoinThreshold = 10485760
   ```
6. **广播与 Shuffle 的选择**：
   - 小表（< 10MB）→ 优先广播
   - 大表 → 使用 Shuffle join 或 Bucketing

> - Databricks 博客：https://www.databricks.com/blog/2016/05/27/apache-spark-scale-a-101-guide-to-broadcast-joins.html

## 题目五：Spark 内存管理——堆内内存与堆外内存、内存划分

### 📝 题目

Spark 的内存管理机制是怎样的？请详细说明 Executor 的内存划分（堆内 vs 堆外），以及 Storage 内存、Execution 内存、User Memory 的作用。如果 Storage 内存和 Execution 内存发生冲突，Spark 如何处理？

### ✅ 答案

**Executor 内存结构**

Executor 的总内存 = **堆内内存（On-heap）** + **堆外内存（Off-heap）**

```
┌──────────────────────────────────────────┐
│           Executor Total Memory           │
├────────────────────┬─────────────────────┤
│    On-heap Memory  │  Off-heap Memory    │
│   (JVM Heap)       │  (Native Memory)    │
└────────────────────┴─────────────────────┘
```

**堆内内存划分（默认占比，以 spark.memory.fraction=0.6 为基准）**

```
┌──────────────────────────────────────────────────────────┐
│                   On-heap Memory (JVM Heap)              │
├─────────────────────┬────────────────────────────────────┤
│  System Reserved    │        Spark Memory (60%)          │
│  (300MB 预留)       │  ┌────────────┬──────────────────┐  │
│                    │  │ Execution  │    Storage       │  │
│                    │  │   Memory   │    Memory        │  │
│                    │  │  ( shuffle │  (RDD Cache      │  │
│                    │  │  sort hash │  broadcast cache │  │
│                    │  │  etc. )    │  etc. )          │  │
│                    │  └────────────┴──────────────────┘  │
├─────────────────────┴────────────────────────────────────┤
│                   User Memory (40%)                       │
│   (用户代码：UDF、数据结构、序列化等)                       │
└──────────────────────────────────────────────────────────┘
```

**各部分作用**

| 区域 | 占比 | 作用 |
|------|------|------|
| **System Reserved** | 固定 300MB | JVM 内部使用（代码缓存、线程空间等） |
| **Execution Memory** | ~30% | Shuffle 排序、Hash Join 聚合、UDF 运行时内存 |
| **Storage Memory** | ~30% | RDD Cache、广播变量、Unroll 展开 |
| **User Memory** | ~40% | 用户代码、UDF 创建的对象、字符串等 |

**堆外内存（Off-heap）**
- 通过 `spark.memory.offHeap.enabled=true` 开启
- 通过 `spark.memory.offHeap.size` 设置大小（默认关闭）
- 主要用于 **序列化后的数据**（Tungsten/S Unsafe）
- 优点：避免 JVM GC 开销，适合大内存场景

**Storage vs Execution 内存冲突——动态内存管理**

Spark 采用 **统一内存管理**（Unified Memory Manager，Spark 1.6+），两者共享一个内存池，可以动态占用彼此的空闲空间：

```scala
// 核心规则
if (Execution 不足 && Storage 有空闲):
    Storage 借出内存给 Execution  // Execution 可以强制占用
elif (Storage 不足 && Execution 有空闲):
    Storage 可以借用 Execution 的空闲内存  // 但 Execution 需要时优先回收
```

**关键机制——软边界（Soft Boundary）**
- **Execution 内存不足**时，可以**强制抢占** Storage 的空间（不会等待）
- **Storage 内存不足**时，只能使用 Execution 的空闲空间，且 Execution 回收空间时**优先归还 Storage**
- 这意味着 **Execution 优先级更高**——Shuffle 等计算任务不会因为缓存而饿死

**调优建议**
- 大数据量 Shuffle 场景：适当增大 `spark.memory.fraction`（如 0.7）和 `spark.memory.storageFraction`（如 0.5）
- 缓存为主的场景：增大 Storage 比例
- 减少 GC 开销：开启堆外内存 `spark.memory.offHeap.enabled=true`

> - 官方内存管理文档：https://spark.apache.org/docs/latest/tuning.html#memory-management
