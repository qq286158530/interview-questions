## 题目 1：Spark 的 Shuffle 机制是什么？为什么 Shuffle 是性能瓶颈？

### 参考答案

**Shuffle 是 Spark 中将数据重新分布的过程**，当一个 RDD 的分区需要依赖父 RDD 所有分区的数据时（如 groupByKey、reduceByKey、join 等），就必须经过 Shuffle。

**Shuffle 执行流程：**

1. **Map 端**：按照 key 进行分区，将数据写入本地磁盘文件（Shuffle Write）
2. **Map 任务完成**后，每个 Reducer 会从所有 Map 节点拉取属于自己的数据（Shuffle Read）
3. **Reduce 端**：对拉取来的数据进行聚合、排序等操作

**Shuffle 的问题（为什么是瓶颈）：**

| 问题 | 描述 |
|------|------|
| **磁盘 I/O** | 中间数据写入磁盘而非内存，I/O 开销巨大 |
| **网络 I/O** | 数据需要跨节点传输，网络带宽成为瓶颈 |
| **内存压力** | Reduce 端需要在内存中缓冲数据，容易 OOM |
| **磁盘大量小文件** | 每个 Shuffle 操作产生大量小文件，给文件系统带来压力 |

**优化方向：**
- `reduceByKey` 优于 `groupByKey`（Map 端先做预聚合）
- 合理设置 `spark.sql.shuffle.partitions`（默认 200）
- 开启 `spark.shuffle.compress` 压缩中间数据
- 使用 `consolidateFiles` 合并小文件

> 📎 来源：[Apache Spark 官方文档 - Shuffle Behavior](https://spark.apache.org/docs/latest/rdd-programming-guide.html#shuffle-operations)

## 题目 2：Spark 的广播变量（Broadcast Variable）是什么？如何使用？有什么注意事项？

### 参考答案

**广播变量**是一种只读变量，在每个节点上只缓存一份副本，而非每个 Task 都发送一份，从而大幅减少网络传输和内存消耗。

**使用方式：**

```scala
// 定义广播变量
val broadcastVar = sc.broadcast(Array(1, 2, 3, 4, 5))

// 在算子内使用 .value 读取
val result = rdd.map(x => x * broadcastVar.value.sum)
```

**广播变量的典型使用场景：**

- **维度表 join**：将小表广播到每个节点，用 map-side join 替代 shuffle join
- **配置参数**：跨节点共享只读配置
- **IP 黑名单/白名单**：过滤场景中常用的数据

**注意事项：**

| 注意点 | 说明 |
|--------|------|
| 大小控制 | 广播变量不宜过大（建议 < 20MB），过大反而占内存 |
| 只读不变 | 广播后不可修改，否则结果不可预期 |
| 序列化 | 数据必须支持 Java 序列化，或使用 Kryo 获得更好性能 |
| 时机选择 | 尽量在 Shuffle 之前广播，避免重复广播 |

**替代方案：** 对于极大数据集，考虑 `spark.sql.adaptive.enabled` 开启自适应查询执行，让 Spark 自动选择最优 Join 策略。

> 📎 来源：[Apache Spark 官方文档 - Broadcast Variable](https://spark.apache.org/docs/latest/rdd-programming-guide.html#broadcast-variables)

## 题目 3：Spark 的内存管理模型是怎样的？堆内与堆外内存如何划分？

### 参考答案

Spark 2.x 引入 **统一内存管理（Unified Memory Management）**，将内存划分为存储内存和执行内存两部分，二者可动态共享。

**内存区域划分：**

```
┌────────────────────────────────────┐
│           系统预留 (20%)            │
├──────────────┬─────────────────────┤
│   存储内存   │     执行内存          │
│ (Storage)   │    (Execution)      │
│   可借用     │     可借用            │
├──────────────┴─────────────────────┤
│         用户内存 (User Memory)      │
│     用于 RDD 转换、UDF 等           │
└────────────────────────────────────┘
```

**各区域说明：**

| 区域 | 默认占比 | 用途 |
|------|---------|------|
| **存储内存** | ~45% | 缓存 RDD / DataFrame、广播变量 |
| **执行内存** | ~45% | Shuffle 排序、Hash 表、代码缓存 |
| **用户内存** | ~20% | RDD 转换、UDF、Python 进程等 |
| **系统预留** | ~10% | JVM 自身开销 |

**堆外内存（Off-Heap）：** 通过 `spark.memory.offHeap.enabled=true` 开启，将数据存储在 JVM 堆之外，减少 GC 开销，适合大内存机器。

**内存配置参数：**

```properties
spark.memory.fraction=0.6        # 统一内存占堆内存比例
spark.memory.storageFraction=0.5  # 存储内存最小保留比例
spark.memory.offHeap.enabled=true
spark.memory.offHeap.size=8g      # 堆外内存大小
```

**常见 OOM 原因：** 执行内存被存储内存抢占，或存储了过多缓存数据导致执行内存不足。应结合 `spark.cleaner.referenceTracking.cleanerTypes` 及时清理缓存。

> 📎 来源：[Apache Spark 官方文档 - Memory Management](https://spark.apache.org/docs/latest/sql-performance-tuning.html#memory-management)

## 题目 4：RDD、DataFrame、DataSet 三者的区别是什么？何时该用哪种？

### 参考答案

**三者对比：**

| 特性 | RDD | DataFrame | DataSet |
|------|-----|----------|---------|
| **数据结构** | 无结构，每个元素是 JVM 对象 | 有 Schema，每行是 Row 对象 | 有类型，每行是 JVM 对象 |
| **编译器检查** | 无 | 无（运行时才发现错误） | 有（编译期即检查） |
| **序列化方式** | Java/Scala 序列化 | Tungsten（字节码） | Tungsten + Encoder |
| **操作方式** | 函数式算子 | SQL / 链式 API | 链式 API + Lambda |
| **性能** | 低（对象开销大） | 高（Catalyst + Tungsten 优化） | 高 |
| **GC 压力** | 高 | 低 | 低 |

**DataFrame = DataSet[Row]**，DataSet 是带类型的 DataFrame。

**选择建议：**

- **用 RDD**：需要低层次控制、自定义分区、数据不在表格结构中、需与低层 API 交互
- **用 DataFrame**：做 ETL / SQL 类操作、列式操作、享受自动优化（Catalyst + Tungsten）
- **用 DataSet**：需要编译时类型安全、同时享受 DataFrame 的优化（Spark 2.0+）

**示例对比：**

```scala
// RDD：处理无结构数据
val rdd: RDD[String] = sc.textFile("hdfs://data/logs")

// DataFrame：结构化表格，支持 SQL
val df: DataFrame = spark.read.parquet("hdfs://data/users")
df.filter($"age" > 25).groupBy("city").count().show()

// DataSet：类型安全
case class User(name: String, age: Int)
val ds: Dataset[User] = spark.read.parquet("hdfs://data/users").as[User]
ds.filter(_.age > 25).groupBy(_.city).count().show()
```

**性能优化原理：** DataFrame/DataSet 受益于 **Catalyst 优化器**（逻辑计划→物理计划）和 **Tungsten 内存管理**（Off-Heap 字节码操作，避免 JVM 对象开销和 GC）。

> 📎 来源：[Apache Spark 官方文档 - DataFrames, Datasets, and RDDs](https://spark.apache.org/docs/latest/sql-programming-guide.html)

## 题目 5：Spark 的 Catalyst 优化器是如何工作的？为什么能提升 SQL 执行性能？

### 参考答案

**Catalyst 是 Spark SQL 的查询优化器**，负责将用户编写的 DataFrame/Dataset/SQL 代码转换为最优的物理执行计划。

**Catalyst 工作流程（4个阶段）：**

```
用户代码（DSL / SQL）
    ↓ 解析
Unresolved Logical Plan（未解析的逻辑计划）
    ↓ 分析 & 解析
Analyzed Logical Plan（分析后的逻辑计划）
    ↓ 优化规则（谓词下推、列裁剪、常量折叠...）
Optimized Logical Plan（优化后的逻辑计划）
    ↓ Cost Model 选择物理执行算子
Physical Plans（物理执行计划）
    ↓ 选择最优
Selected Physical Plan（选中的物理计划）→ 执行
```

**核心优化策略：**

| 优化策略 | 说明 |
|---------|------|
| **谓词下推（Predicate Pushdown）** | Filter 条件下推到数据源，减少读取数据量 |
| **列裁剪（Column Pruning）** | 只读取 SELECT 的列，减少 I/O |
| **常量折叠（Constant Folding）** | 编译期计算常量表达式 |
| **表达式简化** | 移除恒真/恒假条件 |
| **Join 重排** | 根据表大小自动重排多表 Join 顺序 |
| **内存列式格式** | Tungsten 将数据以二进制列式存储，减少 GC |

**示例说明：**

```sql
SELECT name FROM users WHERE age > 30 AND city = 'Beijing'
```

Catalyst 优化后会：
1. 先在数据源层面执行 `city = 'Beijing'` 过滤（谓词下推）
2. 再执行 `age > 30` 过滤
3. 只读取 `name` 列（列裁剪）

**Whole-Stage Codegen：** 将整个执行阶段代码动态编译为 Java 字节码，减少虚函数调用和数据拷贝开销，进一步提升性能。

> 📎 来源：[Apache Spark 官方文档 - Catalyst Optimizer](https://spark.apache.org/docs/latest/sql-performance-tuning.html#caching-data-in-memory)
