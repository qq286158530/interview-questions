# 每日程序员面试题推送 - 2026-06-18

## 题目1：HBase 的整体架构是怎样的？请详细说明 HMaster、HRegionServer、ZooKeeper 的作用。

【答案】

HBase 是一个分布式的、面向列的 NoSQL 数据库，构建于 HDFS 之上，其架构主要由三部分组成：**HMaster**、**HRegionServer** 和 **ZooKeeper**。

### 1. ZooKeeper（HBase 的协调服务）

- **作用**：HBase 使用 ZooKeeper 作为分布式协调服务，用于**选主（Master Election）**和**元数据管理**。
- ZooKeeper 维护了一张 `META` 表的位置信息，当 RegionServer 宕机时，ZooKeeper 会检测到并通知 HMaster 进行故障恢复。
- HBase 默认将 ZooKeeper 集成在自己的进程里，也可以配置使用外部 ZooKeeper 集群。

### 2. HMaster（主节点）

- **作用**：负责**表的管理**（创建、删除表）和 **Region 的调度**（split、compact、负载均衡）。
- HMaster 通过 ZooKeeper 选举产生（主备模式），宕机时 ZooKeeper 会重新选主。
- HMaster 不参与实际的数据读写，真正的读写请求由 HRegionServer 处理。

### 3. HRegionServer（数据节点）

- **作用**：负责**管理 Region**，处理客户端的读写请求，是 HBase 中真正存储数据的地方。
- 每个 HRegionServer 包含多个 Region，每个 Region 对应一张表的一个 **RowKey 区间**。
- HRegionServer 的核心组件包括：
  - **WAL（Write-Ahead Log）**：写入前先写日志，用于宕机恢复。
  - **MemStore**：写缓存，先将数据写入 MemStore，满了之后 flush 成 HFile。
  - **HFile**：HBase 的实际存储文件，存储在 HDFS 上。
  - **BlockCache**：读缓存，热点数据缓存于此。

### 架构图

```
                    ┌──────────┐
                    │ HMaster │
                    └────┬─────┘
                         │
           ┌─────────────┼─────────────┐
           │             │             │
    ┌──────┴─────┐ ┌────┴────┐ ┌──────┴─────┐
    │RegionServer│ │RegionServer│ │RegionServer│
    │  (Node1)   │ │  (Node2)   │ │  (Node3)   │
    └──────┬─────┘ └────┬────┘ └──────┬─────┘
           │             │             │
           └─────────────┴─────────────┘
                         │
               ┌─────────┴─────────┐
               │     ZooKeeper      │
               │   (协调服务)        │
               └───────────────────┘
```

### 一句话总结

HMaster 负责元数据管理和 Region 调度，HRegionServer 负责实际的数据读写和存储，ZooKeeper 负责协调和选主。三者配合实现 HBase 的高可用和分布式存储。

## 题目2：HBase 的读写流程是怎样的？请详细描述写入数据时 MemStore 和 HFile 的关系，以及读取数据时 BlockCache 的作用。

【答案】

### 写入流程（Write Path）

HBase 的写入流程如下：

```
客户端 → ZooKeeper → 获取 META 表位置 → 定位 Region → HRegionServer
```

1. **写入 WAL**：客户端先将数据写入 WAL（Write-Ahead Log），WAL 用于宕机恢复。如果写入 WAL 失败，整个写入操作失败。
2. **写入 MemStore**：数据写入 WAL 成功后，数据被放入 MemStore（一个写缓存，按 RowKey 排序）。
3. **MemStore Flush**：当 MemStore 大小达到阈值（默认 128MB），会触发 **flush** 操作，将 MemStore 中的数据以 **HFile** 格式写入 HDFS。
4. **返回客户端**：写入 MemStore 成功后，HBase 返回客户端写入成功。

### 读取流程（Read Path）

```
客户端 → ZooKeeper → 获取 META 表位置 → 定位 Region → HRegionServer
```

1. **扫描 BlockCache**：HBase 先在读缓存 BlockCache 中查找是否有热点数据命中。
2. **扫描 MemStore**：如果 BlockCache 没有命中，再到 MemStore（未 flush 到磁盘的数据）中查找。
3. **扫描 HFile**：如果 MemStore 也没有，则通过 **BloomFilter** 定位可能的 HFile，在 HFile 中查找数据。
4. **合并返回**：将各部分查到的数据合并（注意时间戳/version），返回给客户端。

### MemStore 与 HFile 的关系

| 特性 | MemStore | HFile |
|------|-----------|-------|
| 位置 | 内存（写缓存） | HDFS（磁盘文件） |
| 形态 | 有序的 kv 内存结构 | 有序的 kv 文件结构 |
| 作用 | 缓冲写入、排序 | 持久化存储 |
| Flush 条件 | 大小达到阈值或时间 | MemStore 满时触发 |
| 数量 | 每个 Region 一个 | 每个 ColumnFamily 多个 |

### BlockCache 的作用

- **BlockCache** 是 HBase 的读缓存，采用 **LRU（最近最少使用）** 淘汰策略。
- 热点数据会被加载到 BlockCache 中，加速后续读取。
- 读取时优先查 BlockCache → MemStore → HFile，三级查找。

### WAL（Write-Ahead Log）的作用

- WAL 记录所有未 flush 到磁盘的操作，用于 **RegionServer 宕机后恢复数据**。
- 写入操作必须先写入 WAL 成功，才算写入完成。

## 题目3：HBase 的 RowKey 设计应该遵循哪些原则？为什么 RowKey 设计如此重要？

【答案】

RowKey 是 HBase 中数据的唯一标识，类似于关系型数据库的主键。RowKey 设计是 HBase 性能优化的**最关键因素之一**，直接决定数据分布和读写效率。

### RowKey 设计核心原则

#### 1. **长度原则（Length）**
- RowKey 应该尽量**短小**，因为：
  - MemStore 的 key-value 结构中 RowKey 是核心字段，越短占用内存越少。
  - 每次 RPC 传输都会携带 RowKey，越短网络开销越小。
  - 建议：RowKey 最大 16 字节，通常 8~16 字节为宜。

#### 2. **散列原则（Salt）**
- RowKey 应该尽量**均匀分布**，避免热点（HotSpot）问题。
- 如果写入的 RowKey 呈现时间顺序递增（如时间戳），会导致所有写入集中在同一个 Region，造成热点。
- 解决方案：在 RowKey 前加**随机前缀（Salt）**，打散数据分布。
- 示例：`MD5(rowKey) % N + originalKey`

#### 3. **唯一原则**
- RowKey 必须全局唯一，HBase 根据 RowKey 定位数据。

#### 4. **排序原则**
- HBase 中的数据按 **RowKey 字典序**排序存储。
- 利用这个特性，可以设计 RowKey 实现**范围查询**（scan）。
- 示例：时间戳倒序设计（最新数据在前）

#### 5. **避免热点策略**

| 策略 | 做法 | 适用场景 |
|------|------|----------|
| 加盐（Salting） | RowKey 前加随机前缀 | 写入均匀但读取无法预知 |
| 哈希（Hashing） | MD5/SHA 哈希后取 N 位 | 写入均匀，读取可计算 |
| 反转（Reversing） | 反转固定格式的电话号码 | 打破递增规律 |
| 时间戳反转 | 倒序时间戳（MAX - timestamp） | 查询最新数据 |

### 典型 RowKey 设计案例

```bash
# 案例1：用户行为数据（用户ID + 时间戳 + 行为类型）
# 错误设计（热点）：
user_actions_{userId}_{timestamp}

# 正确设计（加盐）：
MD5(userId) % 10 + userId + timestamp + actionType

# 案例2：订单数据
# 正确设计（结合业务查询模式）：
order_{倒序时间戳}_{orderId}  # 查询最近订单
```

### 为什么 RowKey 设计重要？

1. **决定数据分布**：RowKey 决定数据落在哪个 Region，直接影响集群负载均衡。
2. **影响读写性能**：热点 Region 会导致单节点压力过大，成为瓶颈。
3. **决定查询效率**：基于 RowKey 的 scan 操作才能利用 HBase 的有序特性。

### 一句话总结

RowKey 设计要遵循"**短小、散列、唯一、有序**"四字原则，核心目标是避免热点、保证数据均匀分布、提升读写效率。

## 题目4：HBase 的协处理器（Coprocessor）是什么？它有哪几种类型？各有什么用途？

【答案】

HBase 协处理器是运行在 **HRegionServer** 上的自定义代码，允许用户在**数据存储层面**直接执行自定义逻辑，无需在客户端做额外处理。类似于数据库的触发器（Trigger）和存储过程（Stored Procedure）。

### 协处理器的两种类型

#### 1. Observer（观察者）

类似于数据库的**触发器（Trigger）**，在特定事件发生时被调用。

**三种 Observer**：

| 类型 | 触发时机 | 用途 |
|------|----------|------|
| **RegionObserver** | Get/Scan/Put/Delete 等操作前后 | 数据过滤、访问控制、修改返回值 |
| **WALObserver** | WAL 写入前后 | WAL 过滤或分析 |
| **MasterObserver** | 表创建/删除/修改等操作前后 | 权限管理、审计日志 |

**应用场景**：
- 在写入数据时自动加密（写入前加密）
- 在读取数据时自动解密（读取后解密）
- 实现列级访问控制（ACL）
- 记录审计日志

**代码示例**：
```java
public class MyRegionObserver extends RegionObserver {
    @Override
    public void preGet(ObserverContext<RegionCoprocessorEnvironment> e, Get get, 
                        List<Cell> result) throws IOException {
        // 在 Get 操作前执行：例如访问权限校验
        if (!hasPermission(get.getRow())) {
            throw new AccessDeniedException("No permission to access this row");
        }
    }
    
    @Override
    public void postPut(ObserverContext<RegionCoprocessorEnvironment> e, Put put,
                        WALEdit edit, Durability durability) throws IOException {
        // 在 Put 操作后执行：例如触发关联更新
        triggerRelatedUpdate(put);
    }
}
```

#### 2. Endpoint（端点）

类似于数据库的**存储过程**，在 HBase 服务器端执行计算逻辑，并将结果返回给客户端。

- 客户端调用 Endpoint，HBase 在 **所有相关 Region 上并行执行** 自定义计算。
- 常用于：聚合计算（求和、统计）、自定义过滤、分布式计算。
- Endpoint 使用 **Protocol Buffers** 定义接口。

**应用场景**：
- 在服务器端直接求某列的总和（避免传输全量数据）
- 分布式 join（跨 Region 的关联查询）
- 自定义聚合函数

**代码示例**：
```java
public class MyEndpoint implements Coprocessor, CoprocessorService {
    @Override
    public void getService(RpcController controller, MyRequest request,
                           RpcCallback<MyResponse> done) {
        // 在每个 Region 上执行自定义计算
        List<MyResult> results = new ArrayList<>();
        // 计算逻辑...
        MyResponse response = MyResponse.newBuilder().addAllResult(results).build();
        done.run(response);
    }
}
```

### Observer vs Endpoint 对比

| 特性 | Observer | Endpoint |
|------|----------|----------|
| 触发方式 | 事件驱动 | 客户端调用 |
| 执行位置 | 单一 Region | 所有相关 Region 并行执行 |
| 返回值 | 修改操作结果 | 自定义计算结果 |
| 类比 | 数据库触发器 | 数据库存储过程 |
| 典型用途 | 加密、ACL、审计 | 聚合、统计、分布式计算 |

### 加载方式

协处理器有两种加载方式：
1. **静态加载（配置）**：在 `hbase-site.xml` 中配置，重启后生效。
2. **动态加载（代码）**：通过 HBase Shell 或 Java API 动态加载，无需重启。

https://hbase.apache.org/book.html

## 题目5：HBase 的 Compaction（压缩合并）机制是什么？Minor Compaction 和 Major Compaction 有什么区别？

【答案】

HBase 的 Compaction 是指将多个 **HFile 合并成更少、更大的 HFile** 的过程，是 HBase 性能优化的核心机制之一。

### 为什么需要 Compaction？

- **读取性能**：HBase 读取时需要扫描多个 HFile，文件越多 IO 越大。合并后文件数减少，读取更快。
- **内存释放**：MemStore flush 后生成 HFile，大量小 HFile 会占用过多块缓存。
- **清理过期数据**：删除和过期的数据在 Compaction 时真正被清理（物理删除）。

### Compaction 的类型

#### 1. Minor Compaction（小型合并）

- **触发时机**：当一个 ColumnFamily 下的 HFile 数量达到阈值（默认 3~10 个，可配置）。
- **合并范围**：选择**少量相邻的较小 HFile** 进行合并。
- **特点**：
  - 速度快，IO 开销小。
  - **不会**清理所有过期/删除的数据。
  - 合并后的 HFile 大小有上限（默认不过 10GB）。
  - 后台线程自动执行，不阻塞读写。

#### 2. Major Compaction（大型合并）

- **触发时机**：手动触发或自动调度（默认每 7 天自动执行一次）。
- **合并范围**：一个 ColumnFamily 下的**所有 HFile** 合并成一个。
- **特点**：
  - 合并后每个 ColumnFamily 只有 **1~3 个 HFile**。
  - **会真正删除** 被标记为删除和过期的数据（版本淘汰）。
  - IO 开销大，通常放在业务低峰期执行。
  - Major Compaction 会重写所有数据，可以利用此机会调整 HFile 的 BlockSize 等参数。

### 两者对比

| 特性 | Minor Compaction | Major Compaction |
|------|-----------------|------------------|
| 合并范围 | 少量小文件 | 所有文件 |
| 触发频率 | 频繁（天级/小时级） | 稀疏（周级/月级） |
| 删除处理 | 不清理 | 真正物理删除 |
| IO 开销 | 较小 | 很大（重写全量数据） |
| 阻塞读写 | 不阻塞 | 通常手动执行，不阻塞 |
| 结果 | 减少小文件数量 | HFile 数量最少化 |
| TTL/版本 | 不处理 | 清理过期版本和 TTL 外数据 |

### Compaction 策略

HBase 提供了多种 Compaction 策略：
- **RatioBasedCompactionPolicy**（默认）：基于文件大小比率合并。
- **ExploringCompactionPolicy**：更智能地选择待合并文件，减少 IO。
- **StripeCompactionPolicy**：适用于大表，减少 Major Compaction 开销。

### Compaction 执行时机的影响

- **优点**：减少 HFile 数量、提升读取性能、清理垃圾数据。
- **缺点**：执行期间 IO 压力增大，可能影响读写性能。
- **优化建议**：
  - Major Compaction 放在业务低峰期。
  - 合理配置 `hbase.hstore.compactionThreshold` 控制触发时机。
  - 大表建议使用 Stripe Compaction 策略。

### Compaction 与 Flush 的关系

```
MemStore → flush → HFile → Minor Compaction → 多个小HFile → Major Compaction → 大HFile
     │                         │                            │
 写入时触发                  定期触发                      定期/手动触发
```
