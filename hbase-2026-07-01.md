## 题目一：HBase 的整体架构是怎样的？包含哪些核心组件？

### 参考答案

HBase 是一个基于 Google BigTable 论文设计的分布式 NoSQL 列式存储数据库，构建在 Hadoop HDFS 之上。其整体架构采用 **Master-Slave（主从）模式**，核心组件如下：

**1. ZooKeeper**
- 负责选举 Master 节点，协调分布式集群的运行状态。
- 存放所有 Region 的地址信息（-ROOT- 和 .META. 表的位置）。
- 检测 RegionServer 的上下线，实现故障感知。

**2. HMaster（Master 节点）**
- 管理所有 RegionServer，负责表的 DDL 操作（创建、删除表等）。
- 分配和负载均衡 Region 到各个 RegionServer。
- 监控 RegionServer 健康状态，Region 故障时触发恢复与数据迁移。
- 协调 GC（垃圾回收）等元数据管理工作。

**3. RegionServer（数据节点）**
- 负责数据读写，是实际存储和处理数据的节点。
- 一个 RegionServer 包含多个 Region，每个 Region 对应一张表的一个数据分片。
- 核心子组件：
  - **WAL（Write-Ahead Log）**：预写日志，用于记录还未写入 MemStore 的操作，保证数据可靠性，宕机时可从 WAL 恢复。
  - **MemStore**：写缓存，数据先写入 MemStore，达到阈值后 flush 成 HFile。
  - **HFile**：HBase 的实际存储文件，存储在 HDFS 上。
  - **BlockCache**：读缓存，热点数据缓存于此，提升读取性能。

**4. 寻址流程**
- 客户端连接 ZooKeeper → 获取 -ROOT- 表位置 → 访问 .META. 表 → 根据 RowKey 找到对应 Region → 直连 RegionServer 进行读写。
- （新版 HBase 已简化寻址路径，但原理类似。）

## 题目二：HBase 的存储结构是怎样的？什么是 MemStore 和 HFile？

### 参考答案

HBase 采用 **LSM 树（Log-Structured Merge Tree）** 作为存储结构的核心思想，数据写入和持久化的流程如下：

**1. MemStore（写缓存）**
- 数据写入时，先将数据 Append 到 WAL，然后写入 MemStore（内存中的有序缓冲区）。
- MemStore 内部对数据按 RowKey 排序，保证数据有序。
- 当 MemStore 达到阈值（默认约 128MB）时，触发 **flush** 操作，将数据写成 HFile。

**2. HFile（HBase 数据文件）**
- HFile 是 HBase 实际存储数据的文件格式，构建在 HDFS 之上。
- HFile 结构（由 Google BigTable 论文定义）：
  - **Data Block**：存储 KeyValue 数据对，Block 大小默认 64KB。
  - **Meta Block**：存储元信息。
  - **File Info**：文件级别的元信息。
  - **Index Block**：数据块索引，加速随机读取。
- HFile 一旦写入 HDFS，就是只读的，不修改。

**3. Minor Compaction（小合并）**
- 将多个小的 HFile 合并成更少的大 HFile，清理被删除和过期的数据版本。

**4. Major Compaction（大合并）**
- 将一个 Store 下所有 HFile 合并成一个 HFile。
- 合并时删除标记为 delete 的数据和过期版本的数据。
- **注意**：Major Compaction 会带来大量 IO 和网络开销，通常在业务低峰期手动触发。

**LSM 树的核心思想**：将随机写转化为顺序写（先写内存，再定期刷盘合并），从而获得优秀的写入性能。

## 题目三：HBase 的 RowKey 如何设计？有哪些原则和技巧？

### 参考答案

RowKey 是 HBase 中用于唯一标识一行数据的主键，设计好坏直接决定读写性能和负载均衡。设计原则如下：

**1. 长度原则**
- RowKey 长度建议控制在 16 字节以内（最长不超过 64KB）。
- 过长的 RowKey 会占用大量存储空间（RowKey 会在每个 Cell 中重复存储），影响 MemStore 和 HFile 的效率。

**2. 散列原则（避免热点）**
- 如果 RowKey 按时间戳或有序递增设计（如 `userId + timestamp`），会导致所有新数据集中在同一个 RegionServer 上写入，造成 **热点问题（Hot Spotting）**。
- 常用散列方法：
  - **加盐（Salting）**：在 RowKey 头部加随机前缀（如 Hash(userId) % N），让数据分散到不同 Region。
  - **哈希（Hashing）**：对 RowKey 取 MD5 或其他哈希值，截取部分作为前缀。
  - **反转（Reversing）**：反转固定长度的前缀，如手机号 `13800138000` → `00008310083`。

**3. 查询原则**
- RowKey 即索引，设计时考虑查询模式（get / scan）。
- 如果经常按 **单一 RowKey 查询**，直接使用该字段作为 RowKey。
- 如果经常按 **范围查询**（如时间区间），将时间字段放在 RowKey 后缀或作为组合键的一部分。

**4. 唯一性原则**
- RowKey 必须唯一标识一行数据。

**典型案例**：用户行为轨迹表
```
RowKey 设计：`{userId的hash值前4位}_{userId}_{时间戳倒序}`
优点：数据按用户分散，无热点；同一用户的数据按时间倒序排列，最新数据先被查到。
```

## 题目四：HBase 的协处理器（Coprocessor）是什么？有哪些使用场景？

### 参考答案

**协处理器（Coprocessor）** 是 HBase 提供的扩展机制，允许用户将自定义代码部署到 RegionServer 节点上，在数据写入/读取的**本地节点**上执行自定义逻辑，无需经过 Master 协调，从而实现高性能的分布式计算。

HBase 提供两种协处理器：

**1. Observer（观察者）**
- 类似于数据库触发器，在特定事件前后触发执行。
- 子类：
  - **PreWriteTrigger / PreMutateTrigger**：写操作前触发，可用于数据验证、权限检查。
  - **PostGetTrigger**：GET 读取后触发，可用于动态列填充、访问审计。
  - **PostPutTrigger / PostMutateTrigger**：写入成功后触发，可用于更新二级索引、异步记录日志。
  - **PostScanTrigger**：Scan 扫描完成后触发。

- 使用场景：
  - 写入数据时自动生成索引列（类似关系库的触发器索引维护）。
  - 实现列级别的访问权限控制。
  - 写入时自动记录审计日志。
  - 实现 HBase 的 "所见即所得" 二级索引（如 Phoenix 的全局索引实现）。

**2. Endpoint（端点）**
- 类似于数据库存储过程，允许在 RegionServer 上执行自定义的聚合计算。
- 客户端调用协处理器方法，HBase 将请求分发到相关 Region 并行执行，最后汇总结果。
- 使用场景：
  - 在服务器端执行 COUNT、SUM、AVG 等聚合计算，减少网络传输数据量。
  - 实现自定义的跨行跨表复杂事务逻辑。
  - 执行服务器端的过滤和转换逻辑。

**3. 加载方式**
- **静态加载**：配置 `hbase-site.xml` 中的 `hbase.coprocessor.region.classes` 参数，重启集群生效。
- **动态加载**：通过 HBase Shell 或 Java API 将协处理器 jar 包注册到表上，无需重启集群。

**注意事项**：协处理器运行在 RegionServer 上，代码 bug 可能导致节点崩溃，需充分测试后再上线。

## 题目五：HBase 和关系型数据库（如 MySQL）的区别是什么？什么场景下适合用 HBase？

### 参考答案

**1. 数据模型对比**

| 维度 | HBase | MySQL（关系型数据库） |
|------|-------|----------------------|
| 数据模型 | 列式存储，NoSQL，Schema-less | 行式存储，关系型，固定 Schema |
| 事务支持 | 仅支持单行事务，不支持跨行跨表事务 | 支持完整 ACID 事务 |
| 查询语言 | 无 SQL，仅通过 API（Get/Scan/Put） | 完整 SQL 支持，强大查询能力 |
| 索引 | 只有 RowKey 索引，二级索引需自行实现 | 支持主键、唯一索引、普通索引、全文索引等 |
| 扩展性 | 天然分布式，水平扩展能力强 | 水平扩展较难，通常靠垂直扩展 |

**2. 存储与性能对比**

| 维度 | HBase | MySQL |
|------|-------|-------|
| 写入性能 | 极高（顺序写+内存写），适合批量写入 | 中等，受限于随机 IO 和事务日志 |
| 读取性能 | 适合批量顺序读取，单行随机读取性能优秀 | 单行随机读取性能好 |
| 海量数据 | 支持 PB 级数据，高并发 | 通常支撑 TB 级 |
| 数据类型 | 存储字节数组，无需强类型 | 强类型，丰富的 SQL 数据类型 |

**3. 适合使用 HBase 的场景**
- **海量数据存储**：数据量达到 TB 甚至 PB 级别，业务增长快。
- **高并发写入**：如埋点数据、日志数据、IoT 传感器数据、消息记录等。
- **无事务需求**：不需要跨行 ACID 事务的业务。
- **Schema 灵活**：列动态增减，不需要固定表结构。
- **实时随机读取**：通过 RowKey 可以毫秒级获取单行数据。

**4. 不适合使用 HBase 的场景**
- 需要复杂 SQL 查询（JOIN、子查询、聚合）、全文检索。
- 需要强事务一致性（跨行跨表）。
- 数据量不大（GB 级以下），传统 RDBMS 完全够用。
- 需要二级索引（需要借助 Phoenix 等组件实现）。

**典型使用案例**：
- 京东云存储用户行为数据（双十一写入峰值达千万/秒）。
- 滴滴出行行程轨迹存储。
- 淘宝商品详情页存储（天猫、淘宝部分核心数据）。
