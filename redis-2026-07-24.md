## 1. Redis 的数据类型有哪些？分别适用于什么场景？

### 答案

Redis 支持 5 种基础数据类型和 3 种高级数据结构：

**5 种基础数据类型：**

| 类型 | 底层实现 | 典型场景 |
|------|---------|---------|
| `String` | SDS（简单动态字符串） | 缓存、计数器、分布式锁、限流 |
| `Hash` | 压缩列表 / 哈希索引 | 对象存储、购物车、用户信息 |
| `List` | 压缩列表 / 双向链表 | 消息队列、粉丝列表、文章评论 |
| `Set` | 整数集合 / 哈希表 | 标签系统、去重、好友关系 |
| `Sorted Set` | 压缩列表 / 跳表 | 排行榜、延时队列、权重队列 |

**3 种高级数据结构：**

| 类型 | 说明 |
|------|------|
| `Bitmap` | 位图，适合签到、在线状态统计 |
| `HyperLogLog` | 基数估算，适合 UV 统计 |
| `Geospatial` | 地理位置，适合附近的人、距离计算 |

**String 类型最大容量？**
- Value 最大 512MB，但建议控制在 100MB 以内
- 计数器场景用 `INCR`/`DECR`，原子操作，支持上亿级计数

**Hash vs String 如何选？**
- 对象字段少于阈值（64个）且值小于阈值（512字节）时，用 ziplist 压缩，节省内存
- 字段不固定或需要单独操作字段时，用 Hash；整体读取/写入时用 String（序列化）

## 2. Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 答案

Redis 提供两种持久化机制：

### RDB（Redis Database Backup）

- **原理**：定时 fork 子进程，将内存数据快照写入二进制文件（dump.rdb）
- **触发方式**：
  - 手动：`SAVE`（阻塞）或 `BGSAVE`（后台异步）
  - 自动：按配置时间间隔自动执行（如 `save 900 1`）
- **优点**：恢复速度快，适合备份和灾难恢复
- **缺点**：可能丢失最近一次快照之后的数据

### AOF（Append Only File）

- **原理**：每个写操作追加到文件末尾（appendonly.aof）
- **同步策略**：
  - `always`：每条写命令同步写入（最安全，性能最差）
  - `everysec`（默认）：每秒同步一次（性能与安全性折中）
  - `no`：由操作系统决定何时同步（性能最好，可能丢失较多数据）
- **重写（Rewrite）**：AOF 文件过大时，自动压缩/合并命令，减小文件体积
- **优点**：数据安全性更高，丢失最多 1 秒数据
- **缺点**：文件体积通常比 RDB 大，恢复速度较慢

### 对比总结

| 维度 | RDB | AOF |
|------|-----|-----|
| 文件体积 | 小 | 大 |
| 恢复速度 | 快 | 慢 |
| 数据完整性 | 可能丢数据 | 最多丢 1 秒 |
| 资源消耗 | fork 瞬时阻塞 | 持续占用磁盘 IO |
| 启动优先级 | 低 | 高 |

### 生产环境推荐配置

```conf
# RDB + AOF 混合使用
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes  # 开启 RDB+AOF 混合持久化
save 900 1
save 300 10
save 60 10000
```

## 3. Redis 的过期 key 删除策略有哪些？内存淘汰机制是什么？

### 答案

### 过期 key 删除策略

Redis 有 3 种过期删除策略：

**1. 定时删除（主动）**
- 为每个 key 创建定时器，到期立即删除
- 优点：内存友好，key 一到期就释放
- 缺点：CPU 不友好，过期 key 过多时占用大量 CPU

**2. 惰性删除（被动）**
- 访问 key 时检查是否过期，过期则删除
- 优点：CPU 友好，只处理被访问的 key
- 缺点：内存不友好，过期 key 长期占用内存

**3. 定期删除（主动 + 被动折中）**
- 每隔一段时间（默认 100ms），随机检查部分 key，删除过期的
- 优点：折中 CPU 和内存
- **Redis 默认采用惰性删除 + 定期删除的组合策略**

### 内存淘汰机制（6 + 4 种策略）

当内存达到上限（`maxmemory`）时，Redis 提供了 6 种数据淘汰策略：

**在所有 key 中淘汰：**
| 策略 | 说明 |
|------|------|
| `volatile-lru` | 从设置过期时间的 key 中淘汰最近最少使用的 |
| `volatile-ttl` | 从设置过期时间的 key 中淘汰即将过期的 |
| `volatile-random` | 从设置过期时间的 key 中随机淘汰 |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用的 |
| `allkeys-random` | 从所有 key 中随机淘汰 |
| `no-eviction` | 不淘汰，返回错误（默认） |

**在 LFU 模式下额外策略（Redis 4.0+）：**
- `volatile-lfu`：从设置过期时间的 key 中淘汰最不经常使用的
- `allkeys-lfu`：从所有 key 中淘汰最不经常使用的

**生产环境推荐：**
- 缓存场景：`volatile-lru` 或 `allkeys-lru`
- 禁止使用：`no-eviction`（会导致 OOM）
- 设置合理 `maxmemory`，建议预留 20-30% 内存

## 4. Redis 的主从复制原理是什么？哨兵模式是如何实现故障转移的？

### 答案

### 主从复制原理

Redis 主从复制采用 **PSYNC 增量同步** 机制（Redis 2.8+），分为全量同步和增量同步：

**全量同步（首次同步或无法增量时）：**
1. 从节点发送 `PSYNC ? -1` 请求同步
2. 主节点执行 `BGSAVE` 生成 RDB 快照
3. 主节点将 RDB 文件发送给从节点
4. 从节点加载 RDB 文件完成全量复制
5. 主节点将缓冲区的写命令同步给从节点

**增量同步（后续同步）：**
- 主节点维护一个环形缓冲区（replication buffer）
- 从节点通过 `PSYNC <runid> <offset>` 持续同步
- 主节点根据 offset 发送差异命令

**关键配置：**
```conf
replicaof masterip masterport  # 从节点配置主节点
replica-read-only yes          # 从节点只读
```

### 哨兵模式（Sentinel）

哨兵是 Redis 的高可用解决方案，用于监控、选主和通知：

**核心功能：**
1. **监控（Monitoring）**：哨兵定期检查主从节点是否存活
2. **自动故障转移（Automatic Failover）：** 主节点宕机时自动选举新主节点
3. **通知（Notification）**：故障转移后通知客户端新主节点地址
4. **配置提供（Configuration Provider）：** 客户端向哨兵获取当前主节点地址

**故障转移流程：**
1. 主观下线（SDown）：单个哨兵检测到主节点无响应
2. 客观下线（ODown）：多个哨兵投票，超过 quorum 阈值确认
3. 选举领头哨兵（Raft 算法）：所有哨兵投票，选出负责故障转移的哨兵
4. 选出新主节点：从所有从节点中根据优先级/同步进度等综合评选
5. 旧主节点降为从节点：指向新主节点，重新同步

**哨兵配置：**
```conf
sentinel monitor mymaster 127.0.0.1 6379 2  # 监控主节点，2个哨兵同意则客观下线
sentinel down-after-milliseconds mymaster 5000  # 5秒无响应判定为下线
```

## 5. Redis 缓存穿透、缓存击穿、缓存雪崩是什么？如何解决？

### 答案

这是 Redis 面试中最常见的高频问题，三者症状不同，解决思路也不同。

### 缓存穿透

**问题描述：** 查询一个不存在的数据，缓存没有命中，每次都去查数据库。

**危害：** 大量请求打到数据库，可能导致数据库崩溃。

**解决方案：**

| 方案 | 原理 |
|------|------|
| **布隆过滤器（BloomFilter）** | 在缓存层前加一层过滤器，不存在的 key 直接返回 |
| **缓存空值（Null Value）** | 将 NULL 结果也缓存起来，key-Null，过期时间短一些 |
| **参数校验** | 接口层对请求参数做合法性校验，过滤明显非法参数 |

```python
# 布隆过滤器示例
def get_user(user_id):
    if not bloom_filter.exists(user_id):
        return None  # 一定不存在
    user = redis.get(f"user:{user_id}")
    if user:
        return user
    user = db.query(user_id)
    if user:
        redis.setex(f"user:{user_id}", 3600, user)
    return user
```

### 缓存击穿

**问题描述：** 某个热点 key 过期瞬间，大量并发请求同时涌入，直接打垮数据库。

**解决方案：**

| 方案 | 原理 |
|------|------|
| **互斥锁（Mutex）** | 只允许一个线程重建缓存，其他线程等待 |
| **逻辑过期** | 不设置实际过期时间，业务层控制逻辑过期 |
| **永不过期** | 对热点数据不设置过期时间，后台异步更新 |

```python
# 互斥锁方案
def get_user(user_id):
    key = f"user:{user_id}"
    user = redis.get(key)
    if user:
        return user
    
    # 尝试获取锁
    lock_key = f"lock:{key}"
    if redis.setnx(lock_key, 1, expire=10):
        user = db.query(user_id)
        redis.setex(key, 3600, user)
        redis.delete(lock_key)
    else:
        time.sleep(0.1)
        return get_user(user_id)  # 重试
    return user
```

### 缓存雪崩

**问题描述：** 大量 key 同时过期，或缓存服务宕机，导致数据库压力剧增。

**解决方案：**

| 方案 | 原理 |
|------|------|
| **过期时间加随机值** | `TTL = baseTTL + random(0, 5min)` |
| **多级缓存** | L1(本地) + L2(Redis)，避免同时失效 |
| **高可用架构** | Redis Cluster + 哨兵，保证缓存服务可用性 |
| **服务熔级降级** | 数据库压力大时，返回兜底数据 |
| **预热** | 系统启动时主动加载热点数据到缓存 |

```python
# 过期时间随机化
def set_cache(key, value, base_ttl=3600):
    import random
    ttl = base_ttl + random.randint(0, 300)  # 3600~3900秒
    redis.setex(key, ttl, value)
```

### 三者对比总结

| 问题 | 原因 | 特点 | 核心解决思路 |
|------|------|------|-------------|
| 穿透 | 查询不存在的数据 | 缓存 + 数据库都查不到 | 布隆过滤器 / 缓存空值 |
| 击穿 | 热点 key 过期瞬间 | 只有 1 个热点 key | 互斥锁 / 逻辑过期 |
| 雪崩 | 大量 key 同时过期 | 大量 key | 过期时间随机化 / 多级缓存 |
