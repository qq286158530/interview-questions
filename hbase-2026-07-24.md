## 题目一：HBase 的整体架构是怎样的？包含哪些核心组件？

### 参考答案

HBase 是一个基于 **Google BigTable** 思想设计的分布式列式存储系统，构建在 Hadoop HDFS 之上，其核心架构如下：

### 1. Region Server（RS）
- **职责**：负责数据的读写操作，是 HBase 中实际管理数据的服务进程
- 每个 Region Server 可以管理多个 **Region**
- 承担数据的 CRUD、Compaction、Split 等操作

### 2. HMaster
- **职责**：负责表的元数据管理、Region 的分配与调度、垃圾回收等
- 不存储实际业务数据，只管理元信息和协调工作
- 支持多个 HMaster 实现高可用（Active-Standby）

### 3. Zookeeper
- 负责 **HMaster 的选举** 和 **元数据（Meta 表）的位置信息** 存储
- 协调 Region Server 的上下线（Watch 机制）
- 保存整个集群的状态信息

### 4. HDFS
- HBase 的底层存储层，所有数据最终持久化到 HDFS
- 利用 HDFS 的副本机制保证数据高可用

### 5. 数据存储流程
```
Client → Zookeeper → .META.表 → Region Server → MemStore → HFile → HDFS
```

## 题目二：HBase 的 RowKey 设计原则有哪些？为什么重要？

### 参考答案

RowKey 是 HBase 中数据访问的**核心**，设计优劣直接影响读写性能。核心原则如下：

### 1. **长度原则**
- RowKey 建议控制在 **16~100 字节** 以内
- 越短越好，过长的 RowKey 会占用大量 MemStore 内存和 HFile 存储空间

### 2. **唯一性原则**
- RowKey 必须唯一标识一行数据，类似关系型数据库的主键
- 如果有多列族，RowKey 是跨所有列族的统一主键

### 3. **散列性原则（避免热点）**
- 避免使用单调递增或递减的 RowKey（如时间戳 `20240101000000`）
- 热点会导致所有请求集中到某个 RegionServer，造成负载不均
- **解决方案**：使用雪花算法（Snowflake）、MD5 哈希、逆序时间戳等打散数据

### 4. **查询场景匹配原则**
- **点查询**（get）：RowKey 要能唯一确定一行
- **范围查询**（scan）：将经常一起访问的数据放在相近的 RowKey 范围内
- 多条件查询时考虑 **拼接 RowKey**（如 `userId_orderId`）

### 5. **业务导向原则**
- 将查询最频繁的字段放在 RowKey 的**高位**
- 利用 HBase 的字典序排序特性优化范围查询

### 热点场景示例：
```java
# 错误示例（热点）：使用时间戳开头
RowKey: 20240724001, 20240724002, 20240724003  → 请求全落在同一Region

# 正确示例（散列）：哈希打散
RowKey: MD5(userId+timestamp).substring(0,6) + userId + timestamp
```

## 题目三：HBase 的 MemStore 和 HFile 是怎么回事？它们之间的关系是什么？

### 参考答案

HBase 采用 **LSM-Tree（Log-Structured Merge Tree）** 思想进行存储，MemStore 和 HFile 是其核心机制。

### MemStore（写缓存）
| 特性 | 说明 |
|------|------|
| **位置** | 内存中 |
| **作用** | 写入数据时，先写入 MemStore，累积到阈值后刷写成 HFile |
| **大小** | 由 `hbase.hregion.memstore.flush.size` 控制（默认 128MB） |
| **排序** | 数据在 MemStore 中按 RowKey 字典序排列 |
| **列族** | 每个列族有独立的 MemStore |

**写流程**：
1. 数据先写入 **WAL（Write-Ahead Log）** 保证宕机恢复
2. 数据写入 MemStore（按 RowKey 排序）
3. 返回客户端写入成功

### HFile（HBase 的存储文件）
| 特性 | 说明 |
|------|------|
| **位置** | HDFS 上 |
| **作用** | HBase 的实际存储文件格式 |
| **结构** | Data Block + Meta Block + Index Block + Bloom Filter |
| **大小** | 由 `hbase.hregion.max.filesize` 控制（默认 10GB） |

### 两者关系
```
MemStore（内存，排序） --flush--> HFile（HDFS，持久化）
                              ↓
          当 HFile 过多时触发 Compaction（合并）
```

### Compaction 机制
- **Minor Compaction**：小范围合并几个 HFile
- **Major Compaction**：将一个 Region 的所有 HFile 合并为一个，删除标记过的数据

### 为什么这么设计？
- **写性能优化**：写入只需写内存和 WAL，无需随机 IO
- **读性能优化**：HFile 定期合并，保证读请求落在少量文件中
- **LSM-Tree 思想**：牺牲一定的读性能换取极致的写性能

## 题目四：HBase 的协处理器（Coprocessor）是什么？有哪些使用场景？

### 参考答案

HBase **协处理器** 是一种扩展机制，允许用户在 RegionServer 上运行自定义代码，类似于数据库的触发器和存储过程。

### 两类协处理器

#### 1. Observer（观察者）
类似于触发器，在特定事件前后执行回调。

| 类型 | 触发时机 |
|------|----------|
| **PreGetOp** | 数据读取之前 |
| **PostGetOp** | 数据读取之后 |
| **PrePutOp** | 数据写入之前 |
| **PostPutOp** | 数据写入之后 |
| **PreDeleteOp** | 数据删除之前 |
| **PostDeleteOp** | 数据删除之后 |
| **PreSplitOp** | Region 分裂之前 |
| **PostSplitOp** | Region 分裂之后 |

#### 2. Endpoint（端点）
类似于存储过程，允许在 RegionServer 上执行自定义计算，然后汇总结果。

### 使用场景

#### 场景一：二级索引
- HBase 原生只支持 RowKey 索引
- 通过 Observer 在 Put 时同步写索引表，实现多条件查询
- 业界常用方案：Phoenix / Elasticsearch 协变

#### 场景二：权限控制
- 在 PreGetOp / PrePutOp 中检查用户权限，拒绝未授权操作

#### 场景三：聚合计算
- 使用 Endpoint 在每个 Region 并行计算 SUM/COUNT/AVG，然后汇总
- 避免大量数据网络传输，提高聚合效率

#### 场景四：数据校验
- 在 PrePutOp 中校验数据格式、范围等，不合法则拒绝写入

#### 场景五：审计日志
- 在 PostPutOp 中记录所有数据变更操作

### 示例（Observer 伪代码）
```java
public class MyObserver extends RegionObserver {
    @Override
    public void postPut(ObserverContext<RegionCoprocessorEnvironment> c,
                        Put put, WALEdit edit) throws IOException {
        // 在数据写入后，同步写入索引表
        Table indexTable = c.getEnvironment().getTable(TableName.valueOf("idx_table"));
        // ... 构建索引行
        indexTable.put(indexPut);
    }
}
```

## 题目五：HBase 和关系型数据库（如 MySQL）的区别是什么？什么场景下选择 HBase？

### 参考答案

### 核心区别对比

| 维度 | HBase | MySQL |
|------|-------|-------|
| **数据库类型** | NoSQL，列式存储 | RDBMS，行式存储 |
| **事务** | 仅支持单行事务 | 支持完整 ACID 事务 |
| **数据结构** | 稀疏、多版本、动态列 | 固定Schema |
| **扩展性** | 横向扩展，支持 PB 级 | 纵向扩展，有瓶颈 |
| **查询能力** | 仅支持 RowKey 查询和范围扫描 | 强大的 SQL 查询、JOIN、索引 |
| **延迟** | 写入延迟低，读延迟略高 | 读写延迟都较低 |
| **一致性** | 最终一致性（可配置强一致性） | 强一致性 |

### HBase 的优势
1. **海量数据存储**：支持 TB → PB 级数据横向扩展
2. **高并发写入**：写入性能极佳，适合日志类、时序类数据
3. **稀疏数据**：列可以动态增加，适合数据结构多变的场景
4. **多版本**：天然支持数据版本管理
5. **高可用**：基于 HDFS，数据副本机制完善

### HBase 的劣势
1. **不支持复杂查询**：没有 SQL，没有 JOIN，需要自己实现二级索引
2. **单行事务**：跨行跨表事务支持弱
3. **学习成本**：RowKey 设计、架构原理较复杂
4. **延迟**：不适合毫秒级低延迟要求的简单查询

### 选型建议

**选择 HBase**：
- 数据量达到 TB 级别，且持续增长
- 写多读少，或者写读都多但查询简单
- 需要存储半结构化、动态 Schema 数据
- 日志存储、时序数据、消息记录、用户行为数据

**选择 MySQL**：
- 数据量在 GB 级别
- 需要复杂查询、事务、JOIN
- 数据结构固定，查询条件多样
- 需要强一致性和完整 SQL 支持

**业界常见方案**：MySQL + HBase 混用，MySQL 存业务元数据，HBase 存海量明细数据，通过 ID 关联。
