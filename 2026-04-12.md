### 题目1：Java 21 的 Virtual Threads（虚拟线程）相比传统线程有哪些优势？它的原理是什么？适用于哪些场景？

【答案】

Virtual Threads（虚拟线程）是 Java 21 引入的轻量级线程实现，是 Project Loom 的核心成果。

**核心优势：**

1. **极低的创建成本**：虚拟线程不再占用 OS 线程资源，可以轻松创建数百万个虚拟线程，而传统线程受限于 OS 线程数量。
2. **高效的资源利用**：大量虚拟线程可以共享少量 OS 线程（Carrier Threads），显著提高系统吞吐量。
3. **简化的并发编程**：无需使用复杂的线程池或异步编程模型，可以用同步的方式写并发代码。
4. **无感的内存占用**：虚拟线程的堆栈大小是动态增长的，默认最大 1MB，而传统线程堆栈在创建时就固定分配（通常 1MB）。

**工作原理：**

虚拟线程由 JVM 管理，不直接对应 OS 线程。JVM 将多个虚拟线程映射到少量的 OS 线程（Carrier Threads）上执行。当虚拟线程执行阻塞操作（如 I/O）时，它会从 OS 线程上"摘下"，让其他虚拟线程复用该 OS 线程，从而避免线程阻塞导致的资源浪费。

**适用场景：**

- 高并发短任务：如 HTTP 请求处理、微服务调用
- 异步 I/O 密集型应用：数据库查询、文件操作、网络通信
- 消息消费：Kafka/RocketMQ 消费者
- 爬虫、数据采集等 IO 密集型任务

**不适用场景：**

- CPU 密集型计算任务（不如物理线程池）
- 需要精确控制线程数量的场景
- 与 ThreadLocal 配合使用时需要注意生命周期管理

📖 来源：Oracle 官方 JEP 444 (Virtual Threads) + Java 21 Documentation

### 题目2：什么是 Java 的 Sealed Classes（密封类）？它解决了什么问题？和 interface 的 default 方法有何区别？

【答案】

Sealed Classes（密封类）是 Java 17 引入的正式特性（JEP 409），用于限制类或接口的继承层次。

**核心概念：**

通过 `sealed` 修饰符声明一个类，并通过 `permits` 关键字明确指定哪些类可以继承它。

```java
public sealed class Shape permits Circle, Rectangle, Triangle {
    // ...
}

final class Circle extends Shape { }        // 完整实现，不允许再被继承
final class Rectangle extends Shape { }    // 完整实现
non-sealed class Triangle extends Shape { } // 开放继承，可以被任何类继承
```

**解决的问题：**

1. **穷举式类型检查**：在 switch 表达式或 instanceof 模式匹配中，编译器可以确保所有可能的情况都被处理，无需 default 分支。
2. **增强的封装性**：明确声明哪些类可以扩展，防止未知子类破坏设计假设。
3. **模式匹配优化**：让编译器更智能地优化密封类相关的模式匹配代码。

**与 interface default 方法的区别：**

| 特性 | Sealed Classes | Interface Default Methods |
|------|----------------|-------------------------|
| 继承限制 | 明确 permits 哪些类可继承 | 无限制，任何类都可实现 |
| 目的 | 限制扩展，保持类型安全 | 提供默认实现，保持向后兼容 |
| 抽象方法 | 可以有抽象方法 | 可以有 default 方法 |
| 构造器 | 可以有构造器 | 不能有构造器（Java 8-20） |

📖 来源：Oracle 官方 JEP 409 + Java 17 Documentation

### 题目3：MySQL 的 InnoDB 引擎为什么选择 B+ 树而不是 B 树或红黑树作为索引结构？B+ 树有哪些关键特性？

【答案】

**为什么是 B+ 树？**

1. ** vs B 树：**
   - B+ 树的非叶子节点不存储数据，只存储索引，这意味着同样大小的节点可以容纳更多的索引项，树的高度更低，磁盘 I/O 更少。
   - B 树的叶子节点层级不一致，查询效率不稳定；B+ 树所有叶子节点在同一层级，查询效率稳定 O(log n)。
   - B+ 树的叶子节点通过双向链表串联，便于范围查询。

2. ** vs 红黑树（内存结构）：**
   - 红黑树是内存数据结构，适合内存操作，但不适合磁盘存储。
   - 磁盘 I/O 的单位是页（Page，通常 16KB），红黑树的高度很高，每个节点一次 I/O，效率低下。
   - B+ 树的多叉结构使树高通常在 3-4 层，大幅减少 I/O 次数。

**B+ 树的关键特性：**

1. **多叉平衡树**：所有叶子节点在同一层级，保持查询时间复杂度 O(log n)。
2. **节点大小等于页大小**：InnoDB 默认页大小 16KB，磁盘 I/O 以页为单位，预读取效率高。
3. **非叶子节点仅存索引**：可以容纳更多索引项，减少树高。
4. **叶子节点双向链表**：支持高效的范围查询（ BETWEEN、>、< 等）。
5. **主键索引与辅助索引**：主键索引叶子节点存储完整行数据，辅助索引叶子节点存储主键值（回表）。

**实际案例：**

假设每行数据 1KB，每页可存 16 条数据，索引项 8 字节：
- B+ 树根节点：可存 2000 个索引 → 第二层 2000 个节点
- 每个第二层节点：可存 2000 个索引 → 第三层 400 万个节点
- 每个第三层节点：可存 16 条数据 → 存储 6400 万条数据
- 树高只需 3 层，3 次 I/O 即可定位任何数据。

## Spring/Spring Boot/Spring Cloud

### 题目4：Spring Boot 3.x 相比 2.x 有哪些重大变化？升级时需要注意哪些兼容性问题？

【答案】

**重大变化：**

1. **Java 版本要求**：Spring Boot 3.x 最低要求 Java 17（Spring Boot 2.x 最低 Java 8），这意味着可以全面使用虚拟线程、模式匹配等新特性。

2. **Jakarta EE 迁移**：从 `javax.*` 命名空间迁移到 `jakarta.*`，所有 Spring Boot 2.x 的 Servlet、Validation、Persistence 等 API 都已迁移。

3. **Spring Framework 6.x**：基于 Spring Framework 6.0，支持 GraalVM 原生镜像，启动更快、内存占用更低。

4. **GraalVM 原生镜像支持**：通过 Spring Native 项目，应用程序可以编译为原生可执行文件，启动时间从秒级降到毫秒级。

5. **Observability 升级**：采用 Micrometer 2.0 标准，统一 metrics、tracing、logging 的集成。

6. **Spring Security 6.x**：默认禁止请求链式调用，CSRF 保护默认启用，部分旧配置需要调整。

**升级注意事项：**

1. **Jakarta 迁移**：所有 `javax.servlet`、`javax.validation`、`javax.persistence` 包需要改为 `jakarta.*`。
2. **Spring Security 配置**：`WebSecurityConfigurerAdapter` 已移除，需要使用组件扫描方式配置。
3. **Property 属性变化**：`server.servlet.xx` 改为 `server.servlet.xx`（部分移除前缀）。
4. **第三方库兼容性**：确保所有依赖支持 Java 17+ 和 Spring 6.x。
5. **弃用警告**：Spring Boot 2.x 的许多 Deprecated API 在 3.x 中已移除。

### 题目5：Spring Cloud Alibaba 在国内为什么比 Spring Cloud Netflix 更流行？它有哪些核心组件？Nacos 的注册与配置中心是如何实现的？

【答案】

**Nacos 相比 Eureka 的优势：**

1. **功能整合**：Nacos 同时支持服务注册发现和配置管理，Eureka 仅支持注册发现，需要配合 Spring Cloud Config 使用。
2. **配置变更推送**：Nacos 支持配置变更的主动推送（配置监听），而 Eureka 需要依赖 Spring Cloud Bus 等组件。
3. **国产化支持**：Nacos 是阿里开源的国产项目，在国内有大量中文文档、技术支持和社区维护。
4. **性能与扩展性**：Nacos 支持集群模式，具备更强的一致性协议支持。

**Spring Cloud Alibaba 核心组件：**

| 组件 | 功能 |
|------|------|
| Nacos | 服务注册发现 + 配置中心 |
| Sentinel | 流量控制、熔断降级 |
| Seata | 分布式事务 |
| Dubbo | RPC 通信框架 |
| RocketMQ | 消息队列 |
| Alibaba Cloud OSS | 对象存储 |

**Nacos 实现原理：**

1. **服务注册**：客户端启动时，通过 HTTP 调用 Nacos Server 的 `/nacos/v1/ns/instance` 接口，将服务元数据（IP、Port、权重等）注册到 Server。
2. **心跳机制**：客户端定时（默认 5 秒）向 Server 发送心跳，如果 Server 15 秒未收到心跳，则将实例标记为不健康；30 秒未收到，则剔除实例。
3. **配置管理**：客户端监听某个配置后，Server 在配置变更时通过 HTTP 长轮询或 WebSocket 主动推送最新配置。
4. **命名空间隔离**：通过命名空间（Namespace）实现多环境隔离，配置按 Namespace + Group + DataId 定位。

### 题目6：Spring Security 6.x 的认证流程是怎样的？JWT Token 认证在 Spring Security 中如何实现？

【答案】

**Spring Security 认证核心流程：**

```
用户请求 → Security Filter Chain → UsernamePasswordAuthenticationFilter
                                              ↓
                                    AuthenticationManager
                                              ↓
                                    AuthenticationProvider（验证）
                                              ↓
                                    SecurityContextHolder（存储认证信息）
```

1. **Security Filter Chain**：Spring Security 使用多个 Filter 组成过滤链，按顺序执行。
2. **AuthenticationFilter**：拦截登录请求，提取用户名密码，封装为 `UsernamePasswordAuthenticationToken`。
3. **AuthenticationManager**：委托给 `AuthenticationProvider` 进行认证。
4. **UserDetailsService**：`AuthenticationProvider` 调用 `UserDetailsService.loadUserByUsername()` 加载用户信息。
5. **PasswordEncoder**：验证密码（通常使用 BCrypt）。
6. **SecurityContext**：认证成功后，将 `Authentication` 对象存入 `SecurityContextHolder`。

**JWT Token 认证实现：**

1. **登录认证**：验证用户名密码后，生成 JWT Token 返回给客户端。

```java
// 生成 Token
String jwt = Jwts.builder()
    .setSubject(username)
    .setExpiration(new Date(System.currentTimeMillis() + 86400000))
    .signWith(key, SignatureAlgorithm.HS256)
    .compact();
```

2. **Token 验证过滤器**：新增一个 `OncePerRequestFilter`，从请求头 `Authorization: Bearer <token>` 中提取 Token 并验证。

3. **配置过滤器链**：将 Token 验证过滤器添加到 `SecurityFilterChain`，放在 `UsernamePasswordAuthenticationFilter` 之前。

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class)
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/login").permitAll()
            .anyRequest().authenticated()
        );
    return http.build();
}
```

4. **SecurityContext 填充**：验证成功后，将用户信息（从 Token 解码）封装为 `Authentication` 存入 `SecurityContext`。

### 题目7：如何理解分布式系统的一致性？CAP 定理和 BASE 理论是什么关系？Seata AT 模式的实现原理是什么？

【答案】

**CAP 定理：**

分布式系统最多只能同时满足以下两个特性：

- **Consistency（一致性）**：所有节点在同一时刻看到相同的数据。
- **Availability（可用性）**：每个请求都能在合理时间内得到响应。
- **Partition Tolerance（分区容错）**：系统能在网络分区的情况下继续运行。

由于网络分区不可避免，分布式系统必须在 C 和 A 之间权衡：
- **CP 系统**：优先保证一致性（如 Zookeeper、HBase）
- **AP 系统**：优先保证可用性（如 Cassandra、Eureka）

**BASE 理论：**

是对 CAP 中一致性和可用性权衡的实践总结：

- **Basically Available（基本可用）**：允许系统在故障时降级，保证核心功能可用。
- **Soft state（软状态）**：允许系统数据在不同节点间存在中间状态。
- **Eventually consistent（最终一致性）**：系统在一段时间后达到一致状态。

**Seata AT 模式原理：**

AT 模式是 Seata 默认的事务模式，基于两阶段提交（2PC）实现：

1. **一阶段（执行阶段）**：
   - TM（Transaction Manager）向 TC（Transaction Coordinator）申请全局事务，获取 XID。
   - 各分支事务执行 SQL，同时记录undo log（回滚日志）。
   - 报告分支事务状态。

2. **二阶段（提交阶段）**：
   - TC 检查所有分支事务状态，全部成功则发送提交命令。
   - 分支事务异步删除 undo log（后置提交）。
   - 如果任何分支失败，TC 发送回滚命令，各分支事务根据 undo log 恢复数据。

**AT vs TCC：**

| 模式 | 侵入性 | 性能 | 一致性 | 适用场景 |
|------|--------|------|--------|----------|
| AT | 低（自动处理） | 高 | 最终一致 | 低并发、复杂事务 |
| TCC | 高（业务编写） | 中 | 强一致 | 高并发、金融级场景 |

## 消息队列

### 题目8：Kafka 3.x 的 KRaft 模式相比 Zookeeper 模式有哪些优势？Kafka 是如何实现高吞吐量和低延迟的？

【答案】

**KRaft vs Zookeeper：**

| 特性 | KRaft 模式 | Zookeeper 模式 |
|------|------------|----------------|
| 依赖 | 无需外部依赖 | 需要部署 Zookeeper 集群 |
| 部署复杂度 | 低（进程内共识） | 高（3+ 节点） |
| 故障恢复 | 更快，无外部依赖 | 依赖 ZK 恢复 |
| 元数据管理 | Controller 直接管理 | 通过 ZK 间接管理 |
| 规模 | 支持更多分区（百万级） | 受 ZK 性能限制 |

**Kafka 高吞吐量实现：**

1. **顺序写磁盘**：追加写入，只更新索引，避免随机 I/O。
2. **页缓存（Page Cache）**：利用 OS 内存缓存数据，写入实际落盘由 OS 后台异步完成。
3. **零拷贝（Zero Copy）**：数据从磁盘到网络的传递不经过用户空间，直接通过 DMA 传输。
4. **批量处理**：消息可批量发送、消费，减少网络往返。
5. **压缩**：支持 GZIP、Snappy、LZ4、ZSTD 等压缩算法，减少网络带宽。
6. **分区并行**：通过分区实现水平扩展，消费并发度等于分区数。

**低延迟机制：**

1. **时间轮（Timing Wheel）**：用于延迟操作调度，避免堆的精确延迟。
2. **Sendfile 系统调用**：减少上下文切换和数据拷贝次数。
3. **请求处理管道**：Network/Encode/Decorate/Process 分层处理。

### 题目9：RabbitMQ 消息积压是如何产生的？如何处理？死信队列（DLX）和延迟队列在生产环境中如何使用？

【答案】

**消息积压原因：**

1. **消费能力不足**：消费者数量少于分区/队列数量，或消费者处理速度慢。
2. **消费阻塞**：消费者代码存在阻塞操作（同步调用、外部接口超时）。
3. **异常处理不当**：消息处理失败后不断重试，导致更多消息堆积。
4. **流量突增**：促销、秒杀等场景导致消息产生速度远超消费速度。

**处理方案：**

1. **紧急扩容**：增加消费者实例，提升消费并发度。
2. **监控告警**：配置积压阈值监控，提前发现处理。
3. **消费者优化**：异步处理、非阻塞 I/O、批量消费。
4. **丢弃旧消息**：在极端情况下，可配置 TTL 丢弃旧消息。
5. **重写队列**：创建新队列，消费者先处理积压消息，新消息路由到新队列。

**死信队列（DLX）使用场景：**

当消息被拒绝（reject/nack）、超时或超过重试次数时，进入死信队列：

```java
// 设置死信交换机和路由键
map.put("x-dead-letter-exchange", "dlx.exchange");
map.put("x-dead-letter-routing-key", "dlx.routing.key");
channel.queueDeclare("normal.queue", true, false, false, map);

// 消费端拒绝消息
channel.basicReject(deliveryTag, false); // 不重新入队，进入 DLX
```

**延迟队列使用场景：**

1. **订单超时取消**：下单后，延迟 30 分钟检查订单状态，未支付则取消。
2. **定时任务**：替代 Linux Cron，实现更灵活的任务调度。
3. **重试机制**：消息处理失败后，延迟一段时间再重试。

实现方式：
- **TTL + DLX**：为消息设置 TTL，过期后进入 DLX，再路由到原队列。
- **RabbitMQ 插件**：`rabbitmq_delayed_message_exchange` 插件支持延迟消息。

### 题目10：RocketMQ 5.x 的 Proxy 模式和传统 Broker 模式有什么区别？事务消息的半消息机制是如何工作的？

【答案】

**Proxy 模式 vs 传统 Broker 模式：**

| 模式 | 架构 | 优势 | 适用场景 |
|------|------|------|----------|
| 传统模式 | 客户端直连 Broker | 延迟最低、性能最好 | 超大规模集群 |
| Proxy 模式 | 客户端连接 Proxy，Proxy 连接 Broker | 解耦、协议适配、安全增强 | 多协议接入、混合云 |

**RocketMQ Proxy 优势：**

1. **协议适配**：Proxy 支持 gRPC、HTTP、WebSocket 等多协议，旧系统无需改动即可接入。
2. **资源解耦**：计算资源（Proxy）和存储资源（Broker）独立扩展。
3. **安全增强**：支持 TLS 加密、认证授权等企业级功能。
4. **多语言支持**：Python、Go 等语言可通过 gRPC 接入。

**事务消息半消息机制：**

RocketMQ 事务消息基于两阶段提交和消息回查实现：

```
1. 生产者发送半消息（Half Message）到 Broker
   ↓
2. Broker 持久化半消息，返回成功
   ↓
3. 生产者执行本地事务
   ↓
4. 生产者提交事务状态（COMMIT 或 ROLLBACK）
   ↓
   ├─ COMMIT：Broker 投递消息给消费者
   ├─ ROLLBACK：Broker 删除消息
   └─ 未知：Broker 启动回查，调用生产者检查事务状态
```

**回查机制：**

当 Broker 超过一定时间未收到事务状态，会主动回查：
- 发送回查请求给生产者
- 生产者查询本地事务状态
- 返回 COMMIT/ROLLBACK

**应用场景：**

- 分布式事务：订单系统 + 库存系统，通过事务消息保证一致性。
- 批量任务：先记录任务，再执行，执行失败则回滚。
- 跨系统同步：两个系统数据同步，保证要么都成功，要么都失败。

### 题目11：什么是 Function Calling（函数调用）？AI Agent 如何通过函数调用实现复杂任务？它和 Tool Use 有何区别？

【答案】

**Function Calling 概念：**

Function Calling 是大语言模型（LLM）的一种能力，允许模型在生成回复时主动调用预定义的外部函数/工具，并将函数返回结果纳入上下文继续推理。

**工作流程：**

```
用户请求 → LLM 识别需要调用函数 → 输出函数调用请求（JSON格式）
    ↓
系统执行函数 → 返回结果给 LLM
    ↓
LLM 结合函数结果 → 生成最终回复
```

**Function Calling 实现示例：**

```json
// LLM 输出函数调用
{
  "function_call": {
    "name": "get_weather",
    "arguments": {"city": "上海"}
  }
}

// 执行函数
def get_weather(city: str):
    return {"city": "上海", "temp": "18°C", "weather": "多云"}

// LLM 结合结果生成回复
"上海今天天气多云，气温18°C，适合出行。"
```

**Function Calling vs Tool Use：**

| 特性 | Function Calling | Tool Use |
|------|-----------------|----------|
| 本质 | LLM 输出结构化调用请求 | 更宽泛的概念，包括各种工具使用 |
| 接口规范 | 有标准协议（如 OpenAI JSON Schema） | 无标准，工具形态多样 |
| 执行控制 | LLM 决定是否调用 | LLM 或 Agent 控制 |
| 典型场景 | 数据库查询、API 调用 | 网页搜索、代码执行、文件操作 |

**实现复杂任务案例：**

一个订票 Agent 可以通过函数调用链完成任务：
1. `search_flights()` - 查询航班
2. `book_flight()` - 预订航班
3. `send_email()` - 发送确认邮件
4. `create_calendar_event()` - 添加到日历

每个函数调用结果成为下一步的输入，LLM 负责决策调用顺序和参数。

### 题目12：AI Agent 的规划能力（Planning）是如何实现的？Chain of Thought、Tree of Thought 和 ReAct 有什么区别？

【答案】

**规划能力的重要性：**

复杂任务需要分解为多个步骤，Agent 需要具备"规划"能力来决定执行顺序、回退重试、评估结果。

**Chain of Thought（CoT，思维链）：**

让 LLM 显式输出推理步骤，再基于推理得出结论。

```
问题：小明有5个苹果，小红给了他3个，小明吃掉了2个，还剩几个？
思考过程：
1. 小明原有5个苹果
2. 小红给了3个 → 5 + 3 = 8个
3. 吃掉了2个 → 8 - 2 = 6个
答案：6个
```

**Tree of Thought（ToT，思维树）：**

在 CoT 基础上，允许分支探索，评估多个可能的推理路径：

```
问题：如何用最少的硬币找零 63 元？
        ┌─ 尝试 [60+3] ─┐
起点 ──┤ 尝试 [50+10+3] ├─ 评估最优解
        └─ 尝试 [20*3+3] ┘
```

适用于需要探索和回溯的复杂问题（如游戏 AI、战略规划）。

**ReAct（Reasoning + Acting）：**

将推理和行动结合，让 Agent 在推理过程中主动调用外部工具：

```
问题：上海今天的天气如何？
思考：我需要调用天气查询工具来获取这个信息。
行动：调用 get_weather(city="上海")
观察：返回 {temp: 18°C, weather: "晴"}
思考：根据返回结果，上海今天晴天，气温18°C。
行动：生成回复告诉用户
```

**三者对比：**

| 方法 | 核心思想 | 外部交互 | 适用场景 |
|------|----------|----------|----------|
| CoT | 显式推理步骤 | 无 | 数学计算、逻辑推理 |
| ToT | 多路径探索评估 | 无 | 复杂规划、战略决策 |
| ReAct | 推理+行动循环 | 有（工具调用） | 真实世界任务、对话 Agent |

**ReAct 的优势：**

1. **真实性**：通过外部工具获取实时信息，避免幻觉。
2. **可控性**：每步行动都有观察反馈，便于调试。
3. **可扩展性**：可接入任何工具，扩展 Agent 能力边界。

📖 来源：ReAct 论文（arxiv:2210.03629）+ LangChain Agent 实现文档

### 题目13：什么是字典树（Trie）？它的时间和空间复杂度是多少？请用代码实现一个 Trie 并说明其在实际场景中的应用。

【答案】

**Trie（字典树/前缀树）概念：**

Trie 是一种树形数据结构，用于高效存储和检索字符串集合，特别是前缀匹配场景。

**核心思想：**

每个节点代表一个字符，从根到某节点的路径代表一个前缀，每个节点的孩子表示下一个可能的字符。

**复杂度分析：**

| 操作 | 时间复杂度 | 空间复杂度 |
|------|------------|------------|
| 插入 | O(m)，m 为字符串长度 | O(m) 最坏情况 |
| 查找 | O(m)，m 为字符串长度 | O(1) |
| 前缀匹配 | O(m + n)，n 为匹配数量 | O(1) |

**代码实现：**

```python
class TrieNode:
    def __init__(self):
        self.children = {}
        self.is_end = False

class Trie:
    def __init__(self):
        self.root = TrieNode()
    
    def insert(self, word):
        node = self.root
        for char in word:
            if char not in node.children:
                node.children[char] = TrieNode()
            node = node.children[char]
        node.is_end = True
    
    def search(self, word):
        node = self._search_prefix(word)
        return node is not None and node.is_end
    
    def startsWith(self, prefix):
        return self._search_prefix(prefix) is not None
    
    def _search_prefix(self, prefix):
        node = self.root
        for char in prefix:
            if char not in node.children:
                return None
            node = node.children[char]
        return node
```

**实际应用场景：**

1. **搜索引擎自动补全**：Google 搜索框输入时，实时显示前缀匹配的建议词。
2. **IP 路由最长前缀匹配**：路由器使用 Trie 匹配 IP 地址前缀。
3. **字符串检索**：拼写检查、敏感词过滤。
4. **电话号码簿**：通过前缀快速检索联系人。

**优化变种：**

- **压缩前缀树（Compressed Trie）**：合并只有一个孩子的节点，减少空间。
- **Radix Tree**：二进制版本，常用于 IP 路由。

### 题目14：如何设计一个 LRU（最近最少使用）缓存？LinkedHashMap 的原理是什么？请实现一个线程安全的 LRU 缓存。

【答案】

**LRU 缓存设计原理：**

LRU 是一种常见的缓存淘汰策略，核心思想是：最近使用的元素在未来也最可能被使用，淘汰最久未使用的元素。

**数据结构选择：**

需要同时支持 O(1) 的：
1. **查找**：判断 key 是否存在
2. **插入/删除**：在头部插入，在尾部删除

最佳组合：**哈希表 + 双向链表**
- 哈希表：O(1) 查找
- 双向链表：O(1) 插入/删除，且能维护访问顺序

**代码实现：**

```python
from collections import OrderedDict

class LRUCache:
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = OrderedDict()  # 保持插入顺序
    
    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        self.cache.move_to_end(key)  # 标记为最近使用
        return self.cache[key]
    
    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            self.cache.move_to_end(key)
        self.cache[key] = value
        if len(self.cache) > self.capacity:
            self.cache.popitem(last=False)  # 删除最旧元素
```

**线程安全版本（使用读写锁）：**

```python
import threading

class ThreadSafeLRUCache:
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = {}
        self.access_order = []  # 维护访问顺序
        self.lock = threading.RLock()
    
    def get(self, key: int) -> int:
        with self.lock:
            if key not in self.cache:
                return -1
            self.access_order.remove(key)
            self.access_order.append(key)  # 移到末尾（最近使用）
            return self.cache[key]
    
    def put(self, key: int, value: int) -> None:
        with self.lock:
            if key in self.cache:
                self.access_order.remove(key)
            elif len(self.cache) >= self.capacity:
                oldest = self.access_order.pop(0)
                del self.cache[oldest]
            self.cache[key] = value
            self.access_order.append(key)
```

**LinkedHashMap 原理：**

LinkedHashMap 继承 HashMap，内部维护一个双向链表（accessOrder=true 时）或保持插入顺序（accessOrder=false 时）。

```java
// 源码简化
public class LinkedHashMap<K,V> extends HashMap<K,V> {
    // 双向链表维护顺序
    // accessOrder=true 按访问顺序，false 按插入顺序
}
```

**应用场景：**

1. **浏览器缓存**：网页缓存、CDN 缓存
2. **数据库连接池**：复用数据库连接
3. **CPU 缓存行**：LruCache 用于内存管理
4. **分布式缓存**：Redis 的 LRU 淘汰策略

### 题目15：什么是单调栈（Monotonic Stack）？它在解决什么问题时有优势？请举例说明如何用单调栈解决 Next Greater Element 问题。

【答案】

**单调栈概念：**

单调栈是一种特殊栈结构，栈内元素保持单调递增或单调递减顺序。

**核心思想：**

遍历数组时，维护一个栈，栈中存储候选元素的索引。当遇到一个元素时：
- 如果它比栈顶大，说明栈顶元素的"下一个更大元素"找到了，弹出栈顶并记录结果。
- 如果它比栈顶小或相等，则入栈，作为栈内元素的候选。

**时间复杂度：**

每个元素最多入栈、出栈各一次，整体 O(n)。

**Next Greater Element 问题：**

**问题描述**：对于数组中每个元素，找到它右边第一个比它大的元素，如果不存在则记为 -1。

**示例**：
```
输入：[2, 1, 4, 3, 5]
输出：[4, 4, 5, 5, -1]
解释：
- 2 的右边第一个更大的是 4
- 1 的右边第一个更大的是 4
- 4 的右边第一个更大的是 5
- 3 的右边第一个更大的是 5
- 5 右边没有更大元素，-1
```

**单调栈解法：**

```python
def next_greater_element(nums):
    n = len(nums)
    result = [-1] * n
    stack = []  # 存储索引，栈底到栈顶递减（即值递增）
    
    for i in range(n):
        # 当前元素比栈顶大，说明找到了栈顶的下一个更大元素
        while stack and nums[i] > nums[stack[-1]]:
            top = stack.pop()
            result[top] = nums[i]
        stack.append(i)  # 当前元素入栈
    
    return result

# 验证
print(next_greater_element([2, 1, 4, 3, 5]))  # [4, 4, 5, 5, -1]
```

**执行过程图解：**

```
数组：[2, 1, 4, 3, 5]

i=0, num=2:  stack=[], result=[-1,-1,-1,-1,-1]
            stack=[0]  (2入栈)

i=1, num=1:  1 < 2，stack=[0,1] (1入栈)

i=2, num=4:  4 > 1 → pop 1, result[1]=4
            4 > 2 → pop 0, result[0]=4
            stack=[2]  (4入栈)

i=3, num=3:  3 < 4，stack=[2,3] (3入栈)

i=4, num=5:  5 > 3 → pop 3, result[3]=5
            5 > 4 → pop 2, result[2]=5
            stack=[4]  (5入栈)

最终：result=[4, 4, 5, 5, -1]
```

**单调栈能解决的问题类型：**

1. **Next Greater/Smaller Element**：下一个更大/更小元素
2. **每日温度（LeetCode 739）**：计算每天要等多少天才能等到更暖和的日子
3. **柱状图最大矩形（LeetCode 84）**：最大矩形面积
4. **去除重复字母（LeetCode 316）**：保持字典序的去重
5. **接雨水问题（LeetCode 42）**： trap 雨水总量
