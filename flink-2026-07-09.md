## 题目一：Flink 的 Task Slot（任务槽）和 Parallelism（并行度）是什么？它们之间的关系是怎样的？

### 参考答案

**1. 基本概念**

| 概念 | 定义 | 作用 |
|------|------|------|
| **Slot** | TaskManager 的资源分配单位 | 决定 TM 上可同时运行的算子实例数 |
| **Parallelism** | 算子/任务的并行度 | 决定一个算子有多少个并行实例 |

**2. Slot vs Parallelism 的关系**

```
JobManager
   └── Operator Chain（算子链）
         ├── Task 1  ──→  SubTask 1.1  (Slot 1)
         ├── Task 2  ──→  SubTask 2.1  (Slot 2)
         └── Task 3  ──→  SubTask 3.1  (Slot 3)
```

- **Slot 数量** = `TaskManager 数量 × 每 TM 的 Slot 数`
- **Parallelism** = 算子的并行实例数，不能超过可用 Slot 总数
- 一个 Slot 可以运行**多个 SubTask**（来自链化的不同算子）

**3. Slot 配置示例**

```yaml
# flink-conf.yaml
taskmanager.numberOfTaskSlots: 8   # 每个 TM 的 slot 数
parallelism.default: 2              # 默认 job 并行度
```

```java
// 代码中设置并行度
env.setParallelism(4);                    // 全局并行度
env.readTextFile("path").setParallelism(8); // 单独算子并行度

// 注意：Source/Sink 的并行度需要和 Kafka 分区数匹配
KafkaSource.builder().setProperty("fetch.message.max.bytes", "10240000");
```

**4. 槽位共享（Slot Sharing）**

默认情况下，**同一 Job 的不同算子可以共享同一个 Slot**：

```java
// 禁用槽位共享（每个算子独占 slot）
env.disableOperatorChaining();

// 查看 slot 使用情况
ExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
System.out.println("Max Parallelism: " + env.getMaxParallelism());
```

**5. 生产调优建议**

```
最佳实践：
- Slot 数 = CPU 核心数（避免上下文切换）
- 并行度设置为 Slot 的 1~2 倍（利用率高时性能最佳）
- 如果并行度 > Slot 数 → 任务会排队等待

计算公式：
所需 Slot 数 = max(各算子 Parallelism)  // 槽位共享时
所需 Slot 数 = sum(各算子 Parallelism) // 无槽位共享时
```

## 题目二：Flink 是如何处理背压（Back Pressure）问题的？有哪些排查和解决方案？

### 参考答案

**1. 什么是背压**

背压是指下游算子处理速度跟不上上游发送速度，导致上游算子被迫降速的现象。在流处理中表现为：**Source → Operator A → Operator B**，当 B 变慢时，A 也会被拖慢。

**2. Flink 的背压处理机制**

Flink 使用**基于 Credit 的流量控制**：

```
Upstream (Producer)          Downstream (Consumer)
     │                               │
     │──[Buffer Request]─────────────>│  下游发出 credit（缓冲区可用量）
     │<──[Buffer Availability]───────│  上游根据 credit 发送数据
     │                               |
```

- 下游消费慢时，credit 减少，上游自动限速
- 无需暂停整个 pipeline，局部背压可控

**3. 背压排查方法**

**方法一：Flink Web UI（最常用）**
```
Job Manager Web UI → Jobs → Click Job → 查看 SubTasks 的 Status
背压的算子会显示 "BACK_PRESSURED" 状态（红色高亮）
```

**方法二：Rest API 查看背压状态**
```bash
curl http://<jobmanager>:8081/jobs/<jobid>/vertices/<vertexid>/backpressure
```

**方法三：Flink Metrics 监控**
```
metrics:
  - name: outPoolUsage
    type: Gauge
  - name: inPoolUsage
    type: Gauge
# inPoolUsage > 0.8 表示背压严重
```

**4. 背压常见原因及解决方案**

| 原因 | 解决方案 |
|------|---------|
| 数据倾斜（Skew） | 重新分桶（keyBy 加随机数前缀） |
| 外部系统 IO 慢 | 异步 IO 或增加并发 |
| 状态过大（RocksDB 读写慢） | 优化状态后端，增量 Checkpoint |
| 代码逻辑复杂 | 拆解算子，减少单算子计算量 |
| GC 停顿 | 增加 TM 内存，调整 GC 参数 |
| 网络瓶颈 | 增加 TM 数量，调整网络缓冲区 |

**5. 代码层面优化**

```java
// 1. 异步 IO 改造（访问外部系统）
DataStream<String> result = input
    .keyBy(r -> r.getKey())
    .process(new AsyncIOFunction<String, String>() {
        @Override
        public void asyncInvoke(String input, ResultFuture<String> result) {
            // 异步查询外部服务
            CompletableFuture.supplyAsync(() -> queryExternal(input))
                .thenAccept(result::complete);
        }
    });

// 2. 算子链拆分（避免单算子过重）
env.disableOperatorChaining(); // 全局禁用
// 或
.someTransformation().disableChaining(); // 单算子禁用

// 3. 反压时快速失败（设置超时）
env.getConfig().setAutoWatermarkInterval(100);
```

## 题目三：Flink Kafka Connector（Source 和 Sink）如何使用？如何保证 Kafka 消费与 Flink Checkpoint 的一致性？

### 参考答案

**1. Kafka Source 使用**

```java
// Kafka Source（最新 API：Flink 1.17+）
KafkaSource<String> source = KafkaSource.<String>builder()
    .setBootstrapServers("localhost:9092")
    .setGroupId("flink-consumer-group")
    .setTopics("input-topic")
    .setValueOnlyDeserializer(new SimpleStringSchema())
    .setStartingOffsets(OffsetsInitializer.committedOffsets())
    .setProperty("failOnDataLoss", "false")  // 关键：数据丢失时尽量不失败
    .build();

DataStream<String> stream = env.fromSource(source, WatermarkStrategy.noWatermarks(), "KafkaSource");
```

**2. Kafka Sink 使用**

```java
// Kafka Sink
KafkaSink<String> sink = KafkaSink.<String>builder()
    .setBootstrapServers("localhost:9092")
    .setRecordSerializer(KafkaRecordSerializationSchema.builder()
        .setTopic("output-topic")
        .setValueSerializationSchema(new SimpleStringSchema())
        .build())
    .setDeliveryGuarantee(DeliveryGuarantee.EXACTLY_ONCE)  // 开启 Exactly-Once
    .setTransactionalIdPrefix("flink-")                      // 事务前缀
    .build();

stream.sinkTo(sink);
```

**3. Exactly-Once 语义保证原理**

Kafka + Flink 的端到端 Exactly-Once 实现：

```
┌──────────────────────────────────────────────────────────┐
│                     Checkpoint 流程                       │
│                                                           │
│  1. Flink Checkpoint Barrier 到达 Source                  │
│  2. Source 暂停发出数据，记录当前 Kafka 偏移量 (offset)    │
│  3. 偏移量 + 状态一起做 Checkpoint（写入 Kafka 特殊 topic） │
│  4. Checkpoint 完成 → 继续消费                            │
│                                                           │
│  故障恢复时：                                              │
│  - 从 Checkpoint 读取上次提交的 offset                      │
│  - 从该 offset 重新消费（Kafka 天然幂等）                   │
└──────────────────────────────────────────────────────────┘
```

**4. 关键配置参数**

```yaml
# Kafka Broker 配置（保证幂等性和事务）
broker:
  transaction.id.extraction: "producer_id"
  transactional.id.expiration.ms: 900000

# Flink Kafka Consumer 配置
kafka:
  consumer:
    isolation.level: read_committed   # 只能读到已提交事务的数据
    enable.auto.commit: false         # 关闭自动提交，由 Flink 管理偏移量
    auto.offset.reset: earliest
```

**5. 常见问题与处理**

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 重复消费 | Checkpoint 间隔太长 | 减少 Checkpoint 间隔 |
| 数据丢失 | `failOnDataLoss=true` 且 Kafka 重平衡 | 设置 `failOnDataLoss=false` |
| 事务超时 | Checkpoint 超时 | 增加 `checkpointing.timeout` |
| 消费顺序乱 | 多分区并行消费 | 设置 `setPartitionDiscoveryInterval` |

**6. 生产环境完整配置示例**

```java
// 完整的 Exactly-Once Kafka Source + Sink 配置
KafkaSource<String> source = KafkaSource.<String>builder()
    .setBootstrapServers("kafka-1:9092,kafka-2:9092,kafka-3:9092")
    .setGroupId("flink-etl-group")
    .setTopics("source-topic")
    .setValueOnlyDeserializer(new SimpleStringSchema())
    .setStartingOffsets(OffsetsInitializer.committedOffsets(OffsetResetStrategy.EARLIEST))
    .setProperty("isolation.level", "read_committed")
    .setProperty("failOnDataLoss", "false")
    .setProperty("partition.discovery.interval.ms", "30000")
    .build();

env.enableCheckpointing(30_000); // 30秒 Checkpoint
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(15_000);
```

## 题目四：Flink SQL 如何实现双流 Join（Interval Join 和 Regular Join）？两者有什么区别？

### 参考答案

**1. 双流 Join 类型总览**

| Join 类型 | 特点 | 数据保留 | 延迟 |
|---------|------|---------|------|
| **Regular Join** | 两表全部历史数据 | 有状态，永不过期 | 受状态大小限制 |
| **Interval Join** | 限制时间范围 | 超过范围自动清理 | 更可控 |
| **Lookup Join** | 维表关联（异步） | 实时查询 | 低 |
| **Temporal Join** | 时态表 Join（版本化） | 基于时间版本 | 可控 |

**2. Regular Join（无时间窗口限制）**

```sql
-- 两流所有历史数据 Join（状态永不清理，需设置状态 TTL）
SELECT 
    o.orderId,
    o.amount,
    p.productName,
    p.inventory
FROM Orders AS o
JOIN Products AS p
    ON o.productId = p.productId

-- ⚠️ 必须设置状态过期时间，否则状态无限增长
env.getConfig().setAutoWatermarkInterval(60000);
TableConfig config = tableEnv.getConfig();
config.setIdleStateRetention(Duration.ofHours(2));  // 状态 TTL
```

**3. Interval Join（区间 Join，最常用）**

```sql
-- 限制时间范围：订单和支付在 15 分钟内的 Join
SELECT
    o.orderId,
    o.amount,
    p.payId,
    p.payTime
FROM Orders o
JOIN Payment p
    ON o.orderId = p.orderId
    AND p.payTime BETWEEN o.orderTime AND o.orderTime + INTERVAL '15' MINUTE

-- 或者用 Lambda 表达式（DataStream API）
DataStream<Tuple2<Order, Payment>> result = orderStream
    .keyBy(o -> o.orderId)
    .intervalJoin(paymentStream.keyBy(p -> p.orderId))
    .between(Time.minutes(-15), Time.minutes(0))
    .upperBound(UpperBoundary.CURRENT_TIME)  // 上界为当前时间
    .process(new IntervalJoinFunction<...>() { ... });
```

**4. Temporal Join（时态表 Join）**

```sql
-- 订单与下单时刻的产品价格关联（价格可能随时间变化）
SELECT
    o.orderId,
    o.productId,
    o.orderTime,
    o.price AS orderPrice,
    p.price AS currentPrice
FROM Orders o
LEFT JOIN Products FOR SYSTEM_TIME AS OF o.orderTime AS p
    ON o.productId = p.productId
```

**5. Lookup Join（维表关联）**

```sql
-- 订单关联用户维表（异步查询外部数据库）
SELECT
    o.orderId,
    u.userName,
    u.userLevel
FROM Orders o
LEFT JOIN LATERAL TABLE(async_lookup_user(o.userId)) AS u
    ON TRUE

-- 定义异步查找函数
tableEnv.createTemporaryFunction("async_lookup_user", 
    new AsyncRedisLookupFunction());  // 需实现 AsyncTableFunction
```

**6. 状态清理策略（防止 OOM）**

```java
// 方式1：设置状态 TTL（Flink SQL 自动清理）
TableConfig config = tableEnv.getConfig();
config.setIdleStateRetention(Duration.ofHours(2));

// 方式2：ProcessFunction 手动清理
@Override
public void processElement(ValueState<String> state, Context ctx, Collector<...> out) {
    // 定期清理状态
    long currentTime = ctx.timestamp();
    if (currentTime - stateLastUpdate > CLEANUP_THRESHOLD) {
        state.clear();
    }
}
```

**7. 生产选型建议**

```
选型决策树：

是否有明确时间范围？
├── 是 → Interval Join（推荐，性能好，状态可控）
└── 否 → 需要全量历史？
         ├── 是 → Regular Join（加状态 TTL）
         └── 否 → Temporal Join 或 Lookup Join
```

## 题目五：Flink 如何处理数据倾斜（Data Skew）？常见的场景和解决方案有哪些？

### 参考答案

**1. 什么是数据倾斜**

数据倾斜是指数据在各个分区/算子间分布不均匀，导致部分任务处理数据量远大于其他任务，造成**木桶效应**（整体速度由最慢的任务决定）。

```
正常情况：        数据倾斜：
Task1: ████        Task1: ██████████████████████████
Task2: ████        Task2: ██
Task3: ████        Task3: ██
Task4: ████        Task4: ██
（处理均匀）       （Task1 处理 90% 数据，严重拖慢整体）
```

**2. 数据倾斜的常见场景**

| 场景 | 原因 | 表现 |
|------|------|------|
| **keyBy 后聚合** | 少数 key 数据量极大 | 某些 SubTask 状态爆炸 |
| **窗口计算** | 特定时间窗口数据集中 | 整点/午夜等时刻窗口压力大 |
| **Kafka 分区热点** | 某些分区数据量远超其他 | 消费不均 |
| **维表 join** | 热点维度值频繁出现 | 算子处理热点数据时阻塞 |
| **数据源本身倾斜** | 原始数据分布不均 | 源头就不均匀 |

**3. 解决方案汇总**

**方案一：Key 打散（加随机前缀/后缀）**

```java
// 在 keyBy 之前加随机数分散热点 key
input
    .map(event -> {
        // 热点 key（如大V用户ID）加随机后缀
        String actualKey = event.getUserId().hashCode() % 10 == 0
            ? event.getUserId() + "_" + (int)(Math.random() * 10)
            : event.getUserId();
        return new Tuple2<>(actualKey, event);
    })
    .keyBy(r -> r.f0)
    .sum(1);

// 窗口聚合后再按真实 key 合并
```

**方案二：两阶段聚合（预聚合 + 最终聚合）**

```java
// 第一阶段：先按 (真实key, 时间桶) 预聚合
input
    .keyBy(e -> Tuple2.of(e.getUserId(), e.getEventTime() / 60000)) // 按分钟分桶
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .aggregate(new PreAggregator())

// 第二阶段：按真实 key 合并（数据量大幅减少）
.keyBy(e -> e.getUserId())
.aggregate(new FinalAggregator());
```

**方案三：自定义分区器（Custom Partitioner）**

```java
// 自定义分区逻辑：将数据更均匀地分配
input.partitionCustom(new Partitioner<String>() {
    @Override
    public int partition(String key, int numPartitions) {
        // 使用 key 的哈希取模 + 随机扰动
        int hash = (key.hashCode() & Integer.MAX_VALUE) % numPartitions;
        return (hash + (int)(Math.random() * 3)) % numPartitions;
    }
}, r -> r.getUserId());
```

**方案四：调整并行度 + 数据重分布**

```java
// 热点数据单独处理，设置更大并行度
input
    .filter(r -> isHotKey(r.getUserId()))  // 热点数据
    .setParallelism(32)                     // 热点并行度加大
    .keyBy(r -> r.getUserId())
    .windowAll(TumblingEventTimeWindows.of(Time.minutes(1)))

// 非热点数据正常处理
    .union(hotDataStream);
```

**方案五：RocksDB + 增量 Checkpoint（应对状态倾斜）**

```java
// 即使发生倾斜，RocksDB 可以扩展到磁盘，不会 OOM
EmbeddedRocksDBStateBackend rocksDB = new EmbeddedRocksDBStateBackend(true);
env.setStateBackend(rocksDB);
env.enableCheckpointing(60_000);
```

**4. 数据倾斜排查步骤**

```
1. 查看 Flink Web UI → SubTasks 页面
   - 查看每个 SubTask 的输入Records数量
   - 倾斜严重的 SubTask 会明显高出平均值

2. 使用 Metrics 监控
   - numRecordsInPerSecond
   - currentInputWatermark
   - backlogbytes (积压量)

3. 使用 FlameGraph 定位热点算子
   - JProfiler / async-profiler 生成 CPU 火焰图
```

**5. 生产实战案例**

```java
// 案例：实时统计 DAU 时遇到大V用户数据倾斜
// 问题：某大V用户每秒产生 10万条事件，导致统计严重不均

// 解决：按用户ID尾号分区 + 两阶段聚合
DataStream<UserEvent> stream = env.addSource(kafkaSource);

DataStream<UVResult> finalResult = stream
    // Step1: 按 用户ID尾号 + 分钟 进行预聚合
    .map(e -> Tuple2.of(
        e.getUserId().substring(e.getUserId().length() - 1), // 取最后一位尾号
        e.getEventTime() / 60000,                            // 分钟桶
        e.getUserId()))
    .keyBy(t -> t.f0 + "_" + t.f1)  // "0_1700000001" 组合 key
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .aggregate(new DistinctCountAgg())  // 分钟级去重
    
    // Step2: 按分钟桶最终合并
    .keyBy(r -> r.getMinuteBucket())
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .aggregate(new FinalMergeAgg());  // 合并分钟结果
```
