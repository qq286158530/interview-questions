## 题目一：Redis 有哪些常见数据类型？它们的典型使用场景是什么？

### 参考答案

Redis 支持五大常用数据类型，各有不同的底层实现和适用场景：

| 数据类型 | 底层实现 | 典型场景 |
|----------|----------|----------|
| **String** | 动态字符串（SDS） | 缓存、计数器、分布式锁（SETNX）、session 共享 |
| **Hash** | 压缩列表 / 字典 | 存储对象属性（如用户信息、商品详情），比 JSON 序列化更高效 |
| **List** | Quicklist（双向链表+压缩列表） | 消息队列（LPUSH+BRPOP）、最新 N 条记录、排行榜 |
| **Set** | 整数集合 / 字典 | 标签系统、去重、好友关系、共同关注（SDIFF/SINTER） |
| **Zset** | 跳跃表（Skip List） | 排行榜、有序通知、延迟队列（按 score 排序） |

**补充：** Redis 7.0 还引入了 **Streams**（Streams 是 List 的替代方案，支持消费组、消息 ID 持久化），以及 **Bitmap**、**HyperLogLog**、**Geospatial** 等扩展类型。

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？生产环境如何选择？

### 参考答案

Redis 提供两种持久化方式：

#### RDB（Redis Database）
- **原理：** 定时 fork 子进程，将当前内存快照写入磁盘，生成 `.rdb` 文件。
- **触发方式：** `SAVE`（同步阻塞）或 `BGSAVE`（后台异步）；也可通过 `save` 配置项定时触发（如 `save 900 1` 表示 900 秒内有 1 次写操作则触发）。
- **优点：** 文件紧凑，恢复速度快（fork 子进程不影响主进程处理请求）。
- **缺点：** 两次持久化之间如果宕机会丢失数据，fork 时会消耗额外内存。

#### AOF（Append Only File）
- **原理：** 将每个写操作追加到 `.aof` 文件，Redis 重启时通过重放命令恢复数据。
- **刷盘策略：** `always`（每条命令都 fsync）、`everysec`（每秒一次，推荐）、`no`（依赖 OS）。
- **优点：** 数据安全性更高，everysec 模式最多丢 1 秒数据；AOF 重写（BGREWRITEAOF）可压缩文件。
- **缺点：** 文件通常比 RDB 大，恢复速度慢。

#### 生产环境选择建议

| 场景 | 推荐策略 |
|------|----------|
| Cache（允许少量数据丢失） | 仅 RDB 或 仅 AOF everysec |
| Storage（不允许数据丢失） | **RDB + AOF 同时开启**（AOF 为主），且设置 `appendfsync everysec` + `auto-aof-rewrite-percentage 100` |
| 大数据量、高写入场景 | RDB + AOF everysec，避免 always 性能损耗 |

> ⚠️ **注意：** Redis 4.0+ 支持混合持久化（`aof-use-rdb-preamble yes`），重写时先写 RDB 再追加 AOF，启动时优先加载 RDB，兼具两者优点。

## 题目三：什么是 Redis 缓存穿透、缓存击穿、缓存雪崩？如何应对？

### 参考答案

这三个问题都是缓存层与数据库交互中常见的高并发风险：

#### 缓存穿透
- **问题：** 请求的数据既不在缓存也不在数据库中（如恶意请求或业务漏洞），导致大量请求直接打穿到 DB。
- **应对方案：**
  1. **参数校验：** 对 ID 格式、范围做基础校验，过滤非法值。
  2. **布隆过滤器（Bloom Filter）：** 将所有存在的 key 存入 Bitmap，请求先经过布隆过滤器过滤不存在的数据。
  3. **空值缓存：** 数据库查不到的结果也缓存一个空值（如 `null`），设置较短 TTL（如 60s）。

#### 缓存击穿
- **问题：** 某个热点 key 过期瞬间，大量并发请求同时击穿到数据库。
- **应对方案：**
  1. **互斥锁（SETNX）：** 只允许一个线程去查 DB 并重建缓存，其他线程等待。
  2. **热点数据永不过期：** 物理不过期，逻辑上用单独线程异步更新。
  3. **预热：** 业务低峰期提前加载热点数据。

#### 缓存雪崩
- **问题：** 大量缓存 key 在同一时间失效（或 Redis 宕机），导致请求全部打到数据库。
- **应对方案：**
  1. **过期时间随机化：** 设置 TTL 时加上随机偏移（如 `TTL = base + random(0, 600)`）。
  2. **多级缓存：** 本地缓存（如 Caffeine/Guava）+ Redis，避免全部击穿。
  3. **Redis 高可用：** 使用 Redis Sentinel 或 Cluster 保障可用性。
  4. **限流熔断：** 使用 Sentinel、Hystrix 等做流量限制，防止 DB 被压垮。
  5. **预过期（lazy expire）：** 主动提前更新热点 key，而非等其过期。

## 题目四：Redis 主从复制原理是什么？复制风暴（replication storm）是怎么回事？如何优化？

### 参考答案

#### 主从复制原理

Redis 复制采用 **PSYNC**（Partial Sync，部分同步）协议，支持全量复制和增量复制：

1. **从库连接主库：** 从库发送 `PSYNC ? -1` 请求全量复制，或 `PSYNC <runid> <offset>` 尝试增量复制。
2. **主库 Fork 子进程：** 主库 fork 子进程执行 `BGSAVE`，同时将期间的写命令缓存到 **replication buffer**（Redis 3.0+ 使用 `repl-backlog-size` 环形缓冲区）。
3. **RDB 传输：** 主库将 RDB 文件发送给从库，从库先清空本地数据再加载 RDB。
4. **增量同步：** RDB 传输期间主库的写命令通过 buffer 同步给从库，保证最终一致。

#### 复制风暴（Replication Storm）

当主库宕机后，所有从库会在重启或故障转移后同时向新主库发起 `PSYNC` 全量复制请求。如果从库数量多或网络拥塞，可能导致新主库被大量复制请求打垮。

常见场景：
- 一主多从架构，主库宕机后大量从库同时发起全量复制。
- 从库重启时从主库拉取 RDB，造成主库瞬间负载飙升。

#### 优化方案

1. **架构层面：**
   - 使用 **Sentinel（哨兵）** 做自动故障转移，减少人工干预延迟。
   - 部署 **Redis Cluster**，将数据分片，多个主节点分担复制压力。
2. **复制参数调优：**
   - `repl-diskless-sync yes`：无盘复制，避免 fork 子进程写 RDB 到磁盘导致的 I/O 瓶颈。
   - `repl-backlog-size`：调大环形缓冲区（默认 1MB），尽量支持增量复制。
   - `repl-timeout`：适当增大超时时间，防止网络抖动导致复制中断。
3. **从库启动优化：**
   - 从库开启 `replica-serve-stale-data no`，在复制中途拒绝服务，倒逼先完成同步。
   - 避免大量从库同时重启。

## 题目五：Redis Cluster 的工作原理是什么？它如何实现数据分片和高可用？

### 参考答案

#### 数据分片原理

Redis Cluster 将整个数据空间划分为 **16384 个 slot（槽）**，每个节点负责一部分 slot：

- **key → slot：** `slot = CRC16(key) % 16384`
- 客户端可以连接任意一个节点，节点之间通过 **Gossip 协议** 传播集群拓扑信息。
- 当客户请求的 key 所在的 slot 不在当前节点时，节点会返回 `MOVED` 重定向错误，客户端自动重定向到正确节点（大多数 Redis 客户端 SDK 会自动处理）。

#### 高可用（故障检测与转移）

1. **主观下线（SDOWN）：** 某个节点在 `cluster-node-timeout` 时间内未收到目标节点的响应，认为其已下线。
2. **客观下线（ODOWN）：** 持有该节点负责的 slot 的主节点中，超过半数认为其已下线，则触发故障转移。
3. **故障转移流程：**
   - 从持有该 slot 的从节点中选出一个新主节点（基于 raft 风格的投票机制）。
   - 新主节点向集群广播 `PONG` 消息，其他节点更新拓扑。
   - 恢复后旧主节点变为新主节点的从节点。
4. **cluster-require-full-coverage：** 默认开启，要求所有 slot 都至少有一个主从节点存活，否则整个集群停止服务。可关闭但会导致部分 slot 不可用。

#### 常用命令

```bash
# 查看集群状态
CLUSTER INFO
CLUSTER NODES

# 查看 slot 分布
CLUSTER SLOTS

# 添加节点到集群（需先 meet）
CLUSTER MEET <ip> <port>
CLUSTER REPLICATE <node_id>   # 将当前节点设为某节点的从库

# 迁移 slot
CLUSTER SETSLOT <slot> MIGRATING <node_id>
CLUSTER SETSLOT <slot> IMPORTING <node_id>
```

#### 注意事项

- **不支持跨 slot 的事务（MULTI/EXEC）**，需要使用 Lua 脚本实现原子性。
- **不支持跨 slot 的批量操作**（如 `MGET`、`MSET` 跨多个节点）。
- **最小高可用单元：** 生产环境每个 slot 至少 1 主 1 从，建议 1 主 2 从。
