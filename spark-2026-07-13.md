## 题目一：RDD 是什么？它有哪些核心特性？

**答案：**

RDD（Resilient Distributed Dataset）是 Apache Spark 的核心抽象，代表一个弹性的、可分区的、分布式的数据集合。它具有以下五大核心特性：

1. **分区（Partitioned）**：数据被划分为多个分区，并行分布在集群的不同节点上，实现数据的分布式处理。
2. **只读（Immutable）**：RDD 是不可变的，一旦创建就不能直接修改，只能通过转换（Transformation）产生新的 RDD。
3. **容错（Fault Tolerant）**：RDD 通过血统（Lineage）机制实现容错。当某个分区丢失时，Spark 可以根据血统记录重新计算该分区，而不需要像 Hadoop 那样做数据副本。
4. **惰性求值（Lazy Evaluation）**：RDD 的转换操作（如 `map`、`filter`）不会立即执行，只有遇到行动操作（如 `collect`、`count`）时才会触发真正的计算。
5. **持久化（Cacheable）**：可以将 RDD 缓存在内存或磁盘中，避免重复计算，提高迭代计算和交互式查询的性能。

RDD 支持两种创建方式：① 从外部存储（HDFS、S3 等）读取；② 通过已有 RDD 转换得到。

## 题目二：Spark 的 Shuffle 机制是什么？为什么 Shuffle 是性能瓶颈？

**答案：**

### 什么是 Shuffle

Shuffle 是 Spark 中将数据重新分布的过程，发生在 Map 阶段和 Reduce 阶段之间。当执行 `groupByKey`、`reduceByKey`、`join`、`distinct` 等需要跨分区聚合或关联的操作时，相同 Key 的数据必须被拉到同一个分区中——这个数据跨节点传输的过程就是 Shuffle。

### Shuffle 的工作流程

1. **Map 端**：每个 Map Task 将数据写入本地磁盘文件，并根据目标分区策略（如 Hash Partitioner）生成多个输出文件。
2. **数据传输**：Spark 通过 Block Manager 在节点之间传输数据片段，接收节点将数据写入本地磁盘。
3. **Reduce 端**：每个 Reduce Task 从相关 Map Task 所在节点拉取属于它的数据文件，进行聚合或连接操作。

### 为什么是性能瓶颈

- **磁盘 I/O**：Shuffle 过程中数据会写入磁盘再读取，磁盘读写成为主要开销。
- **网络传输**：数据需要在集群节点之间跨网络传输，带来了显著的网络开销。
- **内存压力**：Shuffle 中间结果会占用大量堆内内存，可能触发 GC。
- **小文件问题**：默认产生 200 个 Shuffle 分区，会产生大量小文件，增加文件系统压力。

### 优化策略

- 合理设置 `spark.sql.shuffle.partitions`（默认 200），根据数据量调整。
- 使用 `coalesce` 或 `repartition` 减少分区数。
- 避免使用 `groupByKey`，尽量使用 `reduceByKey` 或 `aggregateByKey` 在 Map 端预聚合。
- 对于倾斜的数据（Skew），可以采用加盐（Salting）策略。

## 题目三：DataFrame 和 RDD 有什么区别？DataFrame 的优势是什么？

**答案：**

| 对比维度 | RDD | DataFrame |
|---|---|---|
| **数据结构** | 无结构，每个元素是原始数据类型 | 有结构，每列有名称和类型（相当于数据库表） |
| **优化** | 无自动优化，依赖用户手动优化 | Catalyst 优化器自动生成最优执行计划 |
| **API 风格** | 低层次函数式编程（map/filter/reduce） | 高层次声明式 API（SQL 风格或类似 pandas） |
| **编译时检查** | 无，运行时才发现错误 | 可以利用 Tungsten 进行序列化优化 |
| **语言支持** | Scala、Python、Java、R | Scala、Python、Java、R |
| **数据处理方式** | 逐行处理 | 批量向量化处理 |

### DataFrame 的优势

1. **自动优化（Catalyst 优化器）**：DataFrame 的所有操作都会经过 Catalyst 优化器，优化器会自动完成谓词下推（Predicate Pushdown）、列剪裁（Column Pruning）、常量替换等优化，减少不必要的数据读写。
2. ** Tungsten 序列化**：使用堆外内存和二进制编码，大幅减少序列化和反序列化的开销。
3. **向量化处理**：针对列式存储格式（如 Parquet、ORC）可以进行批量读取和处理，提升 I/O 效率。
4. **统一的 API**：可以用 SQL、DataFrame API 或 Dataset API 三种方式处理数据，且底层使用同一套执行引擎。
5. **外部数据源集成**：可以方便地读取 Hive、Parquet、JSON、JDBC 等多种数据源。

> 在 Scala/Java 中，`DataFrame` 实际上是 `Dataset[Row]` 的别名，两者在底层共享同一套优化机制。

## 题目四：Spark 的广播变量（Broadcast Variable）是什么？用它来优化什么场景？

**答案：**

### 广播变量的概念

广播变量是 Spark 提供的一种共享变量机制，允许将一个只读的变量高效地缓存到集群的每个节点上，而不是在每个 Task 中都重复传输一份副本。

### 工作原理

- 当 Spark 启动一个广播操作时，会将变量值序列化后存储在集群的每个节点的磁盘或内存中。
- 各个 Task 在运行时直接读取本地缓存的广播变量，而不需要每次都从 Driver 端获取。
- 广播变量是**只读**的，一旦创建后不能修改其值。

### 适用场景

**典型场景：维度表 join（大表 join 小表）**

在事实表与维度表进行 join 时，如果维度表足够小，可以将其作为广播变量广播到每个 Executor，避免 Shuffle 操作。

```scala
// 广播维度表
val dimTable = spark.sparkContext.broadcast(dimData)

// 在 Map 端使用广播变量
val result = factData.map(row => {
    val dim = dimTable.value.getOrElse(row.getInt(0), defaultRow)
    // join 处理
})
```

### 性能优势

- **节省网络带宽**：维度表只传输一次，而不是每个 Task 都传输一遍。
- **避免 Shuffle**：大表不需要与其他节点交换数据，直接在本地完成 join。
- **减少 GC 压力**：序列化/反序列化次数大幅减少。

### 注意事项

- 广播变量适合数据量较小的场景（一般建议 < 20MB），过大的广播变量会占用过多 Executor 内存。
- 可以通过 `spark.sql.broadcastTimeout` 设置广播超时时间（默认 5 分钟）。
- Spark 3.0 起引入了自适应查询执行（AQE），可以自动将小表广播，进一步简化了用户操作。

## 题目五：Spark 的内存管理模型是怎样的？Storage Memory 和 Execution Memory 如何划分？

**答案：**

### 统一内存管理（Spark 1.6+，默认）

Spark 使用统一内存管理模型，将堆内内存划分为两大区域：**Storage Memory** 和 **Execution Memory**，两者可以动态借用。

```
┌──────────────────────────────────────────────┐
│                   Heap Memory                 │
│  ┌──────────────────┬──────────────────────┐ │
│  │  Execution Memory │   Storage Memory     │ │
│  │  (Shuffle 排序/   │  (RDD Cache/         │ │
│  │   join/agg 等)    │   Broadcast 等)      │ │
│  └──────────────────┴──────────────────────┘ │
│  ┌─────────────────────────────────────────┐ │
│  │         User Memory (数据结构/GC)        │ │
│  └─────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────┐ │
│  │         Reserved Memory (300MB)          │ │
│  └─────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
```

### 内存区域详解

| 区域 | 用途 | 特点 |
|---|---|---|
| **Reserved Memory** | 系统预留，默认 300MB | 固定不变，用于 Spark 内部对象 |
| **Execution Memory** | Shuffle 排序、Join 操作的中间数据、Hash 表等 | 可借用 Storage Memory，满了会 spill 到磁盘 |
| **Storage Memory** | RDD 缓存（`cache()`、`persist()`）、广播变量 | 可借用 Execution Memory，但不能将正在使用的缓存逐出 |
| **User Memory** | 用户数据结构（HashMap 等）和元数据、Internal metadata | 完全由用户控制，Spark 不管理 |

### 动态占用机制

当 Execution Memory 不足时，可以借用 Storage Memory；当 Storage Memory 不足时，也可以借用 Execution Memory。但存在以下约束：

- **Storage → Execution**：Storage 中的缓存数据如果正在被使用（active），则不能被逐出借用。
- **Execution → Storage**：Execution 可以驱逐 Storage 中的缓存数据来获取所需内存。

### 内存配置参数

```properties
spark.memory.fraction = 0.6        # 堆内用于 Execution+Storage 的比例
spark.memory.storageFraction = 0.5 # Storage 在总内存中的初始占比
spark.executor.memory = 4g         # Executor 总堆内存
```

### 常见内存问题排查

- **`OutOfMemoryError: GC overhead limit exceeded`**：通常是 User Memory 被耗尽，说明应用程序创建了过多对象。
- **Shuffle OOM**：Execution Memory 不足，可以调低 `spark.memory.fraction` 或增加 `spark.shuffle.service.enabled`（开启 External Shuffle Service）。
