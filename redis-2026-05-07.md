## 题目一：Redis 有哪些数据类型？它们的典型使用场景是什么？

### 答案

Redis 支持以下五种基础数据类型：

| 数据类型 | 底层实现 | 典型使用场景 |
|---------|---------|-------------|
| **String** | SDS（简单动态字符串） | 缓存、计数器、分布式锁、限流、Session |
| **Hash** | 压缩列表 / 哈希表 | 存储对象（如用户信息）、购物车、商品详情 |
| **List** | 压缩列表 / 双向链表 | 消息队列、任务队列、最新动态列表、文章评论列表 |
| **Set** | 整数集合 / 哈希表 | 标签系统、去重、关注/粉丝、共同好友、抽奖 |
| **Zset** | 压缩列表 / 跳表 + 字典 | 排行榜、有序消息队列、延迟队列、权重队列 |

**补充说明：**

- **Bitmap**：基于 String 类型，用于签到、用户活跃度统计、布隆过滤器基础
- **HyperLogLog**：用于 UV 统计，内存极低（12KB），误差约 0.81%
- **Geospatial**：地理位置，附近的人、滴滴打车
- **Stream**：消息队列，替代 List 的消费组模式

**String 类型注意**：value 最大 512MB，但生产环境建议控制在 10MB 以内，否则影响 RDB 和主从同步性能。

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 各自有什么优缺点？如何选择？

### 答案

Redis 提供了两种持久化机制：

### 1. RDB（Redis Database Backup）

- **原理**：在指定时间点生成数据集的快照（point-in-time snapshot）
- **触发方式**：
  - `save`（同步阻塞）/ `bgsave`（后台异步）
  - 自动触发：`save m n`（m 秒内 n 次修改）
  - `redis-cli shutdown` 时自动触发
  - `FLUSHALL` + `FLUSHDB AOF` 时也会触发
- **文件**：`dump.rdb`
- **优点**：紧凑压缩，适合备份/灾难恢复；恢复速度快
- **缺点**：最后一次快照后的数据会丢失；fork 子进程时开销较大

### 2. AOF（Append Only File）

- **原理**：记录每次写操作命令，以 Redis 协议格式追加到文件末尾
- **配置**：`appendonly yes`，同步策略：
  - `always`：每条命令都 fsync，性能最差但最安全
  - `everysec`（默认）：每秒一次，最多丢 1 秒数据
  - `no`：由 OS 控制，性能最好但可能丢较多数据
- **文件**：`appendonly.aof`
- **重写机制**：`bgrewriteaof` 压缩 AOF 文件，去除无效命令（如多次 SET 同一 key）
- **优点**：数据安全性更高；AOF 日志是追加写入的意外中断也不会损坏
- **缺点**：AOF 文件通常比 RDB 大；高并发下 QPS 会下降

### 3. 混合持久化（Redis 4.0+）

```
aof-use-rdb-preamble yes
```
混合使用 RDB + AOF：重写时先以 RDB 格式写入全量数据，后续增量用 AOF 追加。

### 4. 生产环境选择建议

| 场景 | 推荐策略 |
|-----|---------|
| 缓存场景，数据可丢 | RDB，或不持久化 |
| 金融/订单等关键数据 | AOF everysec + RDB 备份 |
| 追求最佳数据安全 | AOF always + RDB 备用 |
| 通用生产环境 | 混合持久化 + everysec |

## 题目三：Redis 的过期 key 删除策略有哪些？内存淘汰机制是什么？

### 答案

### 1. 过期 key 删除策略

Redis 有两种过期 key 删除策略：

#### ① 惰性删除（Lazy Expunge）

- **原理**：在访问 key 时才检查是否过期，过期则删除
- **优点**：对 CPU 友好，不用额外开销
- **缺点**：过期的 key 长时间不被访问会占用内存

#### ② 定期删除（Active Expire）

- **原理**：每隔一段时间（`hz` 参数，默认 10），随机抽取带过期时间的 key 进行检查，删除已过期的 key
- **流程**：抽取 `active-expire-effort`（默认 1）个 key，不只检查一次上限为 20 个/次
- **优点**：定期清理，减少内存占用
- **缺点**：对 CPU 有一定压力

> Redis 默认同时使用惰性删除 + 定期删除两种策略。

### 2. 内存淘汰机制（MaxMemory Policy）

当 Redis 内存达到 `maxmemory` 时，会根据以下策略淘汰数据：

| 策略 | 说明 |
|-----|------|
| `volatile-lru` | 从已设置过期时间的 key 中，淘汰最近最少使用的 |
| `allkeys-lru` | 从所有 key 中，淘汰最近最少使用的（最常用） |
| `volatile-lfu` | 从已设置过期时间的 key 中，淘汰使用频率最低的 |
| `allkeys-lfu` | 从所有 key 中，淘汰使用频率最低的 |
| `volatile-random` | 从已设置过期时间的 key 中，随机淘汰 |
| `allkeys-random` | 从所有 key 中，随机淘汰 |
| `volatile-ttl` | 从已设置过期时间的 key 中，淘汰最近要过期的（TTL 最小的） |
| `no-eviction` | 不淘汰，写入时会报错（默认策略） |

### 3. 实战建议

- **热点数据**：使用 `allkeys-lru`，让常用数据留在内存
- **缓存 + 过期时间**：使用 `volatile-lru`
- **TTL 精确控制**：使用 `volatile-ttl`
- **配置示例**：`maxmemory 2gb` + `maxmemory-policy allkeys-lru`

## 题目四：Redis 主从复制原理是什么？哨兵模式有什么作用？

### 答案

### 1. 主从复制原理

Redis 主从复制采用 **长连接同步（PSYNC）** 机制，支持全量同步和增量同步：

#### 全量同步流程：
1. **连接建立**：从节点保存主节点地址，发送 `PSYNC ? -1`
2. **主节点 fork**：主节点启动 bg save 子进程生成 RDB 文件
3. **发送快照**：主节点将 RDB 文件发送给从节点，从节点清空旧数据并加载 RDB
4. **同步缓冲**：主节点在生成 RDB 期间新写入的命令记录到 **replication buffer**
5. **同步缓冲**：将 buffer 中的写命令发送给从节点

#### 增量同步流程：
- 主从节点建立长连接后，所有写操作通过 **replication stream**（命令传播）实时同步给从节点
- 从节点断线重连后，发送 `PSYNC <runid> <offset>`，主节点根据偏移量补发缺失数据

#### 关键配置：
```
replicaof masterip masterport      # 从节点配置主节点
masterauth <password>              # 主节点密码认证
replica-read-only yes              # 从节点只读（默认）
```

### 2. 哨兵模式（Sentinel）

哨兵是 Redis 的高可用方案，核心功能：

| 功能 | 说明 |
|-----|------|
| **监控** | 周期性 ping 主/从/哨兵节点，判断节点是否下线 |
| **自动故障转移** | 主节点主观下线 → 客观下线 → 选举新主节点 → 旧主节点变为从节点 |
| **通知** | 下线事件通过 pub/sub 通知客户端（如哨兵地址变更） |
| **配置提供** | 客户端向哨兵询问当前主节点地址 |

#### 哨兵集群：
- 至少需要 **3 个哨兵 + 1 主 2 从**
- 哨兵之间互相监控，通过 **Raft 算法** 选举领导者完成故障转移
- 配置建议：`sentinel monitor mymaster 127.0.0.1 6379 2`（2 表示至少 2 个哨兵同意才进行故障转移）

### 3. 主从延迟问题

- 从节点延迟是正常的（异步复制）
- 延迟大的原因：网络慢、从节点负载高、主节点写入 QPS 高
- 解决方案：流水线打包写入、调整 `repl-diskless-sync`、从节点数量控制

## 题目五：Redis 缓存穿透、缓存击穿、缓存雪崩是什么？如何解决？

### 答案

这三个问题是分布式缓存系统中最常见的高可用问题：

### 1. 缓存穿透

**问题**：查询一个**不存在**的数据，缓存没有命中，请求直接打到数据库。

**危害**：大量恶意/异常请求穿透缓存直接访问数据库，可能导致数据库宕机。

**解决方案：**

| 方案 | 说明 |
|-----|------|
| **布隆过滤器（BloomFilter）** | 在缓存层前加一个布隆过滤器，所有 key 存入过滤器，命中则放行，不命中直接返回 null |
| **缓存空值** | 将 null 结果也缓存起来，设置短 TTL（如 30s） |
| **参数校验** | 对非法 id 参数做基础校验（负数、非法格式等） |
| **接口限流** | 对异常 QPS 做限流，防止大量穿透 |

**布隆过滤器实现**：
```java
// Redis 自带 Bitmap 实现布隆过滤器
SETBIT bloom_filter 10000000 1  // 添加 key 哈希位
GETBIT bloom_filter 10000000    // 检查是否存在
```

### 2. 缓存击穿

**问题**：某个**热点 key 过期**，大量并发请求同时穿透到数据库。

**特点**：与穿透不同，击穿是针对热点数据过期，数据库瞬时压力骤增。

**解决方案：**

| 方案 | 说明 |
|-----|------|
| **互斥锁（分布式锁）** | 只允许一个请求去查库，其他等待（如 SETNX 实现互斥） |
| **热点数据永不过期** | 不设置 TTL，通过后台异步更新或逻辑过期 |
| **逻辑过期** | key 不设过期时间，但 value 中包含逻辑过期时间，过期后用单独线程重建缓存 |

**互斥锁代码示例**：
```java
String value = redis.get(key);
if (value == null) {
    // 尝试获取锁
    if (redis.setnx("lock:" + key, "1", 10)) {
        // 查库
        value = db.get(key);
        redis.setex(key, 3600, value);
        redis.del("lock:" + key);
    } else {
        // 等待后重试或降级
        Thread.sleep(50);
        return redis.get(key);
    }
}
```

### 3. 缓存雪崩

**问题**：大量缓存 key **在同一时间失效**，所有请求直接打向数据库。

**原因**：大量 key 设置了相同的过期时间、Redis 重启、缓存服务器宕机。

**解决方案：**

| 方案 | 说明 |
|-----|------|
| **过期时间加随机值** | `TTL = baseTTL + random(0, baseTTL * 0.2)` |
| **多级缓存** | 本地缓存（Caffeine/Guava）+ Redis，层层挡请求 |
| **服务熔丝/限流** | Hystrix/Sentinel 保护数据库，超时快速失败 |
| **高可用架构** | Redis Cluster + 主从，确保缓存服务可用性 |
| **预热** | 系统启动时主动加载热点数据到缓存 |

**最佳实践**：
```
# 过期时间设计
expire_time = base_time + random(0, 300) // 基础时间 + 0~5分钟随机
```

### 4. 三者对比总结

| 问题 | 原因 | 特征 | 核心方案 |
|-----|------|-----|---------|
| 缓存穿透 | 查询不存在的数据 | 缓存无、DB 无 | 布隆过滤器 |
| 缓存击穿 | 热点 key 过期 | 缓存无、DB 有（压力骤增） | 互斥锁 / 永不过期 |
| 缓存雪崩 | 大量 key 同时过期 | 缓存无、DB 有（大批量） | 随机 TTL + 多级缓存 |
