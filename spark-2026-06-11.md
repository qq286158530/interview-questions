## 题目一：Spark 架构与核心组件

### 📌 面试题
> Spark 的整体架构是怎样的？请详细说明 Driver、Executor、Cluster Manager 三者的作用及它们之间的协作关系。

### ✅ 答案

Apache Spark 采用 **主从架构**，核心组件包括 **Driver（驱动程序）**、**Executor（执行器）** 和 **Cluster Manager（集群管理器）**。

#### 1. Driver（驱动程序）
- 运行应用 `main()` 方法，创建 `SparkContext`
- 负责将用户代码转换为 **DAG（有向无环图）**，将 DAG 划分为多个 **Stage**，再将 Stage 拆分为 **Task**
- 调度 Task 到 Executor 上执行
- 接收 Executor 返回的结果
- 必须与 Worker 节点网络可达（通常与集群在同一局域网）

#### 2. Executor（执行器）
- 由 Cluster Manager 在 Worker 节点上启动，是 **长期运行的进程**
- 负责：
  - 运行分配给它的 Task（多线程并行）
  - 将结果返回给 Driver
  - 通过 `BlockManager` 管理所在节点的缓存和磁盘存储
- 每个应用有独立的 Executor，不同应用之间隔离

#### 3. Cluster Manager（集群管理器）
Spark 支持三种集群管理器：

| 模式 | 说明 |
|------|------|
| **Standalone** | Spark 自带的简单集群管理器，易于部署 |
| **YARN** | Hadoop 生态的资源和调度管理器 |
| **Kubernetes** | 容器化编排，支持原生微服务部署 |

#### 协作流程
```
用户提交代码 → Driver 创建 SparkContext
  → 连接 Cluster Manager
  → Cluster Manager 在 Worker 节点上分配资源、启动 Executor
  → Driver 将应用代码（JAR/Python）发送给 Executor
  → Driver 发送 Task 给 Executor 并行执行
  → Executor 执行完成，结果返回 Driver
```

## 题目二：RDD、DataFrame、Dataset 三者区别

### 📌 面试题
> Spark 中 RDD、DataFrame 和 Dataset 三种数据抽象有什么区别？各自适用于什么场景？

### ✅ 答案

#### 核心区别

| 特性 | RDD | DataFrame | Dataset |
|------|-----|-----------|---------|
| **数据类型** | 无schema，强类型 JVM 对象 | 有 schema（列名+类型），类似数据库表 | 有 schema，类型安全的 JVM 对象 |
| **API 语言** | Scala/Java/Python/R | Scala/Java/Python/R | 仅 Scala/Java |
| **优化引擎** | 无（底层） | **Catalyst 优化器 + Tungsten** | **Catalyst 优化器 + Tungsten** |
| **执行效率** | 低（绕过 SQL 优化层） | 高（代码生成、向量化） | 高 |
| **编译时检查** | 无 | 无（运行时才发现错误） | 有（编译时检查类型） |
| **序列化方式** | Java Serialization | Kryo / Java Serializer | Encoder（Kryo + 优化） |
| **创建方式** | `parallelize()` / `textFile()` | `read.json()` 等 / `toDF()` | `toDS()` / 创建 Case Class |

#### 代码示例（Scala）

```scala
// RDD - 底层 API，无 schema
val rdd: RDD[(String, Int)] = sc.textFile("data.txt")
  .flatMap(_.split(" "))
  .map((_, 1))
  .reduceByKey(_ + _)

// DataFrame - 有 schema，行式存储
val df: DataFrame = spark.read.json("people.json")
df.filter(df("age") > 30).select("name").show()

// Dataset - 强类型（有 schema 的 RDD）
case class Person(name: String, age: Int)
val ds: Dataset[Person] = spark.read.json("people.json").as[Person]
ds.filter(_.age > 30).map(_.name).show()
```

#### 选型建议
- **用 RDD**：需要细粒度控制、低级 API、与旧代码兼容
- **用 DataFrame**：大多数场景，SQL 查询、数据分析、ETL
- **用 Dataset**：需要编译时类型安全、复杂业务逻辑

> **注意**：在 Scala/Java 中，`DataFrame = Dataset[Row]`，是同一概念的不同别名。

## 题目三：Shuffle 机制深度剖析

### 📌 面试题
> 什么是 Shuffle？为什么 Spark 中 Shuffle 操作代价高昂？请列举常见的 Shuffle 操作并说明如何优化。

### ✅ 答案

#### 什么是 Shuffle？

Shuffle（洗牌）是 Spark 将 **数据重新分布** 的过程。当某个 RDD/DataFrame 的一个分区需要依赖前一个 RDD 的多个分区数据时，就发生了 Shuffle。

典型场景：**按键分组聚合**（如 `groupByKey`、`reduceByKey`、`join`）。

#### Shuffle 执行流程

```
Stage 0 (Map Stage)：
  Partition 0 ──┐
  Partition 1 ──┼──→ Shuffle Write（写入本地磁盘文件）──→ 按 partitionId 合并文件
  Partition 2 ──┤                                           │
  Partition 3 ──┘                                           ▼
Stage 1 (Reduce Stage)：                              Shuffle Read
  Partition 0 ← 从各节点读取所需数据 ←────────────────┘
  Partition 1
```

- **Shuffle Write**：Map 任务完成后，将数据按 Key 哈希写入本地磁盘，生成 `shuffleBlock` 文件
- **Shuffle Read**：Reduce 任务启动时，从所有 Map 节点拉取属于自己分区的数据，进行聚合

#### 常见 Shuffle 操作

| 类型 | 操作 |
|------|------|
| **聚合类** | `groupByKey`, `reduceByKey`, `aggregateByKey`, `foldByKey` |
| **关联类** | `join`, `cogroup`, `leftOuterJoin`, `rightOuterJoin` |
| **重分区** | `repartition`, `coalesce`, `distinct`, `sortByKey` |
| **集合类** | `intersection`, `subtract`, `union`（需去重时） |

#### 为什么 Shuffle 代价高昂？

1. **磁盘 I/O**：Shuffle 数据必须落盘（无法纯内存），涉及大量磁盘读写
2. **网络 I/O**：数据需要在节点间复制传输
3. **序列化/反序列化**：跨网络传输需要 Kryo 或 Java 序列化
4. **中间文件**：Shuffle 文件会持久化到磁盘，占用空间
5. **磁盘文件数爆炸**：每个 Map 任务会产生 `R × M` 个文件（R = Reduce 分区数，M = Map 分区数）

#### 优化策略

1. **使用 `reduceByKey` 而非 `groupByKey`**
   - `reduceByKey` 在 Map 端先做本地聚合，减少网络传输数据量
   - `groupByKey` 传输所有原始数据

2. **避免不必要的数据重分区**
   ```
   pipeline 尽量在同一分区操作，减少跨分区依赖
   ```

3. **合理设置 Shuffle 分区数**
   - 分区过多 → 文件数爆炸、任务调度开销大
   - 分区过少 → 并行度低、OOM 风险
   - 经验公式：`Shuffle 分区数 ≈ 集群 CPU 总核数 × 2~4`

4. **使用广播变量优化 Join**
   - 小表广播到每个 Executor 内存，避免 Shuffle
   ```scala
   val smallTable = collectAsMap()  // 收集到 Driver
   val broadcastVar = sc.broadcast(smallTable)
   ```

5. **启用外部 Shuffle 服务**（Spark 3.x）
   - 使用 `spark.shuffle.service.enabled=true`
   - Shuffle 文件由独立服务管理，Executor 退出后文件仍可用

## 题目四：广播变量（Broadcast Variables）

### 📌 面试题
> 什么是广播变量？它解决了什么问题？使用广播变量有哪些注意事项？

### ✅ 答案

#### 什么是广播变量？

广播变量是 Spark 提供的 **只读共享变量**，将一份数据缓存到每个 Executor 的内存中，而不是每个 Task 传递一份副本。

```
Without Broadcast：
  Task 1 ──→ copy of lookup table (1MB)
  Task 2 ──→ copy of lookup table (1MB)
  Task 3 ──→ copy of lookup table (1MB)
  ...
  1000 Tasks = 1000 MB 网络传输！

With Broadcast：
  Executor 1 ──→ 1 copy of lookup table in memory
                Task 1 / Task 2 / Task 3 / ... (all local access)
  Executor 2 ──→ 1 copy of lookup table in memory
                Task 1 / Task 2 / Task 3 / ... (all local access)
```

#### 解决的问题

- **减少网络传输**：大表/配置数据只传输一次，而非每个 Task 传输一次
- **加速 Join**：将小表广播到内存，加速 `broadcast join`
- **降低 Driver 内存压力**：数据不堆积在 Driver 端

#### 使用方法

```scala
// Python (PySpark)
broadcast_var = sc.broadcast({"key1": "value1", "key2": "value2"})

// 在 Task 中通过 .value 访问
result = broadcast_var.value["key1"]
```

```scala
// Scala
val broadcastVar = sc.broadcast(Map("key1" -> "value1", "key2" -> "value2"))
val result = broadcastVar.value("key1")
```

#### 注意事项

1. **广播的数据必须小于 8GB**（Spark 限制）
   - 超过限制需使用 `spark.sql.files.maxPartitionBytes` 调整或分片广播

2. **只适合小表广播**：大表广播会导致 Executor OOM
   - 经验：广播表建议 < 20MB

3. **只读不可修改**：广播变量创建后不可更改值

4. **自动最优化的广播 Join**
   ```scala
   // Spark SQL 中，小表自动广播
   // 手动控制：
   spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 10485760) // 10MB
   ```

5. **版本兼容性**：Spark 2.x 之后广播实现更稳定，3.x 引入 External Shuffle Service 优化

## 题目五：Spark 内存管理与存储级别

### 📌 面试题
> Spark 的内存模型是怎样的？请说明统一内存管理（Spark 2.x+）的组成，以及 Storage 和 Execution 内存之间的动态共享机制。

### ✅ 答案

#### Spark 内存模型（统一内存管理，Spark 2.x+）

Spark 使用 **统一内存管理（Unified Memory Management）**，将堆内存分为两部分：

```
┌──────────────────────────────────────────────┐
│                  JVM 堆内存                     │
│  ┌──────────────────┐  ┌──────────────────┐  │
│  │    Storage       │  │    Execution     │  │
│  │    Memory        │◄─►│    Memory        │  │
│  │                  │  │                  │  │
│  │  缓存 RDD/广播变量 │  │  Shuffle 缓冲区   │  │
│  │  序列化数据        │  │  哈希连接/排序缓冲 │  │
│  └──────────────────┘  └──────────────────┘  │
│                                              │
│  ┌────────────────────────────────────────┐  │
│  │           Other Memory                  │  │
│  │  JVM 内部开销、系统预留、用户代码          │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

#### 三大内存区域

| 区域 | 用途 | 配置参数 |
|------|------|---------|
| **Storage Memory** | 缓存 RDD/DataFrame、广播变量、`persist()` 的数据 | `spark.memory.storageFraction`（默认 0.5） |
| **Execution Memory** | Shuffle 排序/聚合缓冲区、哈希join、迭代计算 | 与 Storage 动态共享 |
| **Other Memory** | JVM 元数据、内部开销、UDF、用户代码 | 剩余空间 |

#### Storage vs Execution 动态共享

- **Storage 空间空闲时**：可借用给 Execution（用于 Shuffle）
- **Execution 空间空闲时**：可借用给 Storage（用于缓存）
- **强制回收**：任何一方可用空间不足时，**Storage 优先归还**被借用的空间（防止 OOM）

#### Spark 存储级别（Storage Levels）

| 级别 | 说明 | 使用场景 |
|------|------|---------|
| `MEMORY_ONLY` | 仅内存（反序列化） | 内存充足，数据不重复使用时 |
| `MEMORY_ONLY_SER` | 仅内存（序列化） | 内存紧张，省空间 |
| `MEMORY_AND_DISK` | 内存+磁盘（内存不够溢写到磁盘） | 数据量大，内存不够 |
| `DISK_ONLY` | 仅磁盘 | 中间结果复用频率低 |
| `MEMORY_AND_DISK_SER` | 序列化后存内存+磁盘 | 内存紧张 + 需序列化 |

```scala
// Python
rdd.persist(StorageLevel.MEMORY_AND_DISK)
rdd.unpersist()  // 手动释放
```

#### 内存管理关键配置

```properties
spark.memory.fraction        = 0.6    # 用于 Storage + Execution 的比例（占堆内存）
spark.memory.storageFraction  = 0.5    # Storage 初始占比（Execution 初始为另一半）
spark.executor.memory         = 4g     # 每个 Executor 堆内存大小
```

#### OOM 常见原因与排查

1. **数据量远超内存**：增加分区数、调整存储级别
2. **序列化开销**：使用 `MEMORY_AND_DISK_SER` 而非 `MEMORY_ONLY`
3. **Shuffle 文件过多**：增大 `spark.shuffle.service.enabled`
4. **Driver 端数据堆积**：避免在 Driver 端 `collect()` 大量数据
