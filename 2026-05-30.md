## 题目1：Java并发编程 | volatile关键字的作用与局限性

【答案】
volatile是Java并发编程中最轻量的同步机制，主要解决共享变量的**可见性**问题。

**核心作用：**
1. **保证可见性**：一个线程对volatile变量的修改，对其他线程立即可见。当volatile变量被写入时，会强迫CPU将数据写回主内存，并使其他CPU缓存中该变量的副本失效，读取时必须从主内存重新获取最新值。
2. **禁止指令重排序**：volatile变量的读写前后会插入内存屏障（Memory Barriers），防止编译器/CPU进行指令重排序优化。

**volatile不能保证原子性：**
- volatile++操作（如 `i++`）不是原子操作，它包含三个步骤：①读取i的值 ②i+1 ③写回i。中间任何一步都可能被其他线程打断。
- 例如：两个线程同时读取i=0，各自计算得到i=1，然后分别写回，最终i=1而非预期的i=2。
- 解决方案：使用synchronized或java.util.concurrent中的原子类（如AtomicInteger）。

**典型使用场景：**
- 状态标志位（如 `volatile boolean running = true`）
- 双重检查锁定（Double-Checked Locking）中的单例模式
- 何时不用：计数器、累加器等需要原子操作的场景

## 题目2：JVM垃圾回收 | 三色标记与并发标记算法

【答案】
**三色标记是垃圾回收器用于追踪存活对象的着色算法，将对象分为三种颜色：**

- **白色（White）**：未被标记的对象，最终作为垃圾回收
- **灰色（Gray）**：被GC线程标记过，但自身还未完成对子节点的遍历
- **黑色（Black）**：已完成遍历的对象，其所有引用都已扫描完毕

**并发标记产生的问题——漏标/错标：**
在并发标记阶段（用户线程与GC线程同时运行），可能发生：
- **漏标**：原本存活的对象被误标为死亡（导致对象被错误回收）
- **错标**：原本死亡的对象被误标为存活（导致内存泄漏，但相对影响较小）

**漏标的两个必要条件：**
1. 灰色对象删除了对白色对象的引用
2. 黑色对象重新引用了该白色对象

**CMS/G1的解决策略：**

**CMS（Concurrent Mark and Sweep）**：
- 使用**增量更新（Incremental Update）**：当黑色对象新增对白色对象的引用时，将该黑色对象记录下来，在重新标记阶段再扫描这些黑色对象（可能漏掉部分）

**G1（Garbarge-First）**：
- 使用**SATB（Snapshot-At-The-Beginning）**：在并发标记开始时对堆做快照，标记开始时存活的对象在标记结束后都认为是存活的，即使期间变成了垃圾也被保留。优点是100%不会漏标，但会积累浮动垃圾。

## 题目3：Spring Boot | 自动配置原理与@Conditional注解

【答案】
**自动配置是Spring Boot的核心特性，它让"零配置"成为可能。**

**核心原理：**
1. **@SpringBootApplication注解**：包含@EnableAutoConfiguration，是自动配置的入口
2. **spring.factories文件**：在`META-INF/spring.factories`中配置了EnableAutoConfiguration的实现类列表
3. **@Configuration + @Bean**：每个自动配置类都是一个Java配置类，通过@Bean声明各种Bean
4. **@Conditional条件注解**：每个配置类都有一系列@Conditional条件，只有条件都满足时该配置才会生效

**@Conditional家族常用注解：**
- @ConditionalOnClass：类路径中存在指定类才生效（如HikariDataSource）
- @ConditionalOnMissingBean：容器中不存在指定Bean才生效（避免覆盖用户自定义Bean）
- @ConditionalOnProperty：配置文件中有指定属性才生效
- @ConditionalOnBean：容器中存在指定Bean才生效
- @ConditionalOnJava：Java版本满足条件才生效

**典型自动配置示例（DataSourceAutoConfiguration）：**
```java
@Configuration
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseBuilder.class })
@ConditionalOnMissingBean(DataSource.class)
@EnableConfigurationProperties(DataSourceProperties.class)
public class DataSourceAutoConfiguration {
    // 根据类路径和配置文件自动创建DataSource
}
```

**执行顺序：**
1. Spring Boot扫描spring.factories中的AutoConfiguration
2. 按@Conditional条件过滤，排除不满足的配置类
3. 通过@EnableConfigurationProperties绑定配置属性
4. 最后通过@Import将配置类注册为Bean

## 题目4：Spring Cloud | Nacos注册中心与配置中心原理

【答案】
**Nacos是阿里的开源项目，同时具备服务注册/发现和配置管理功能。**

**作为注册中心的原理：**

**服务注册：**
1. 应用启动时，Nacos Client将服务信息（IP、Port、权重等）注册到Nacos Server
2. 注册信息存储在distro一致性协议的存储层
3. 健康检查机制：Nacos Client定时发送心跳，若超时未收到则标记为不健康

**服务发现：**
1. 消费者从Nacos获取服务提供者列表
2. 本地缓存服务列表，并订阅服务变更（推送机制）
3. 服务下线时，Nacos Server主动推送更新给消费者

**作为配置中心的原理：**

**配置管理：**
1. 在Nacos控制台创建配置（Data ID + Group + Namespace）
2. 客户端通过Namespace+Group+Data ID唯一确定配置
3. 配置变更时，Nacos Server通过长连接推送配置变更通知
4. 客户端收到通知后，拉取最新配置并更新到本地

**核心概念：**
- **Namespace**：用于环境隔离（dev/test/prod）
- **Group**：用于配置分组（如按业务线划分）
- **Data ID**：配置文件名称（如application.yml）
- **Annotation @NacosConfigurationProperties**：自动刷新配置

**与Spring Cloud的集成：**
- @EnableDiscoveryClient：启用服务注册
- @NacosConfigListener：监听配置变更
- bootstrap.yml：优先加载Nacos配置

## 题目5：Spring Cloud | Sentinel熔断降级与限流算法

【答案】
**Sentinel是阿里开源的流量控制组件，提供熔断、降级、限流三大功能。**

**限流算法：**

**1. 计数器算法（固定窗口）：**
- 统计时间窗口内的请求数，超过阈值则拒绝
- 问题：无法处理突发流量，窗口边界可能产生瞬间穿透

**2. 滑动窗口算法：**
- 将时间窗口分成多个小窗口，每个小窗口独立计数
- 比计数器更精确，避免窗口边界问题

**3. 令牌桶算法（Token Bucket）：**
- 以固定速率往桶中放入令牌，获取到令牌才可处理请求
- 特点：允许一定程度的突发流量
- 典型实现：Guava RateLimiter

**4. 漏桶算法（Leaky Bucket）：**
- 请求以任意速率进入桶，桶以固定速率输出
- 特点：平滑流量，但无法处理突发

**熔断降级策略：**

**熔断器三种状态：**
- **Closed（关闭）**：正常调用，请求通过
- **Open（打开）**：触发熔断，后续请求直接拒绝/执行fallback
- **Half-Open（半开）**：试探恢复，限流通过部分请求

**熔断触发条件：**
- 慢调用比例：响应时间超过阈值，且比例超过设定值
- 异常比例/异常数：异常请求占比/数量超过阈值

**降级策略：**
- 默认降级：流控/熔断/系统自适应降级
- 自定义降级：@SentinelResource指定fallback方法
- 热点参数降级：根据参数值进行流控

**核心概念：**
- **资源（Resource）**：被保护的业务方法
- **规则（Rule）**：流控规则、降级规则、熔断规则
- **SlotChain**：责任链模式，每个Slot负责不同功能

## 题目6：消息队列 | Kafka架构与消息存储机制

【答案】
**Kafka是LinkedIn开源的分布式消息系统，以高吞吐量和持久化著称。**

**核心架构组件：**

- **Broker**：Kafka实例，每台服务器可运行多个Broker
- **Topic**：消息主题，用于分类消息
- **Partition**：Topic的分区，实现负载均衡和数据并行处理
- **Replica**：副本机制，保证数据高可用
- **Producer**：消息生产者
- **Consumer**：消息消费者
- **Consumer Group**：消费者组，同一组内消费者消费不同Partition，同一Partition只能被组内一个消费者消费

**消息存储机制：**

**Partition结构：**
- 每个Partition对应一个物理目录
- 目录下包含多组Segment文件（log + index + timeindex）
- Segment采用稀疏索引，快速定位消息

**写入流程：**
1. Producer向Leader Partition写入消息（不直接写Follower）
2. 通过ACK机制保证可靠性：
   - acks=0：不需要确认，可能丢失数据
   - acks=1：仅Leader确认
   - acks=all：Leader+所有Follower确认

**消息查找流程：**
1. 利用二分查找定位Segment文件
2. 通过index稀疏索引定位消息物理偏移量
3. 顺序扫描找到目标消息

**存储策略：**
- 基于时间：默认7天
- 基于大小：默认1GB
- cleanup.policy配置清理策略

**消费机制：**
- 消费者通过offset记录消费位置
- offset保存在__consumer_offsets这个Topic中
- 消费者组内自动负载均衡

## 题目7：消息队列 | RabbitMQ五种队列模式详解

【答案】
**RabbitMQ通过AMQP协议实现消息传递，五种队列模式覆盖各种业务场景。**

**1. 简单队列（Simple Queue）：**
- 最基础的模式：P（生产者）→ [Queue] → C（消费者）
- 一对一通信，队列中的消息只能被一个消费者消费
- 缺点：生产者无法得知消费者处理结果

**2. Work模式（Work Queue）：**
- 一个生产者和多个消费者
- 同一消息只能被一个消费者获取
- 实现负载均衡（如，消费者1处理快，消费者2处理慢，自动分配）
- **能者多劳**：配置prefetch=1，让处理快的消费者多处理任务

**3. 订阅模式（Fanout Exchange）：**
- 生产者发送到交换机，交换机将消息广播到所有绑定的队列
- 适用场景：系统通知、广播消息
- 示例：日志系统同时写入数据库和文件

**4. 路由模式（Direct Exchange）：**
- 交换机根据路由键（Routing Key）精确匹配队列
- 队列绑定时指定路由键，消息发送时指定路由键
- 完全匹配：`error`路由键只发送到`log_error`队列

**5. 主题模式（Topic Exchange）：**
- 支持通配符匹配：
  - `*`：匹配一个词
  - `#`：匹配零个或多个词
- 示例：`log.#`可匹配`log.info`、`log.error.mysql`
- 适用场景：灵活的消息路由，如日志分级处理

**ACK确认机制：**
- 自动ACK：消息消费后立即确认
- 手动ACK：处理完成后确认，支持消息拒绝和重入队列

**消息持久化：**
- 交换机持久化：`durable=true`
- 队列持久化：`durable=true`
- 消息持久化：`delivery_mode=2`

## 题目8：AI Agent | RAG检索增强生成核心原理与主流方案

【答案】
**RAG（Retrieval-Augmented Generation）是解决大模型幻觉和知识时效性问题的主流架构。**

**核心思想：**
RAG让大模型在回答问题前，先从外部知识库检索相关内容，再基于检索结果生成答案。就像开卷考试：先翻书找资料，再根据资料答题。

**标准RAG流程（Naive RAG）：**
1. **离线索引**：文档切块 → Embedding模型转向量 → 存入向量数据库
2. **在线查询**：用户问题转向量 → 向量库相似度搜索 → 召回Top-K文档块 → 拼成Prompt → LLM生成答案

**进阶RAG方案：**

**检索优化：**
- **Multi-Query RAG**：将用户问题改写成多种表述分别检索，合并去重（适合口语化场景）
- **HyDE**：让LLM先写一个"假答案"，用假答案的向量检索（因为假答案文体更接近真实文档）
- **语义分块**：按句子边界切分，保证语义完整性
- **Hybrid Search**：向量检索+BM25关键词检索融合，兼顾语义和精确匹配

**生成优化：**
- **Reranking精排**：粗召回150个 → 轻量模型初筛20个 → Cross-Encoder精排5个
- **Corrective RAG**：检索后让LLM评判文档相关性，低质量时回退到Web搜索
- **Self-RAG**：在生成过程中设置检查点，让模型自我审视答案是否有文档支撑

**复杂推理方案：**
- **GraphRAG**：将文档转为知识图谱，支持多跳推理（如"张三是哪个中心？"需要串联多个实体）
- **Text-to-SQL RAG**：将自然语言转为SQL查询结构化数据
- **Agentic RAG**：AI Agent自主决策使用哪种检索工具（向量库/Web搜索/数据库）

**核心评估指标（RAGAS框架）：**
- Context Recall：召回率
- Context Precision：精确率
- Faithfulness：答案忠实度（无幻觉）
- Answer Relevance：答案相关性

## 题目9：算法 | 二分查找变体——查找第一个大于等于目标值的位置

【答案】
**这是二分查找最经典的变体之一，考察对边界条件的处理。**

**题目：** 给定一个升序数组和目标值，找到第一个大于等于目标值的元素下标，若不存在返回-1。

**代码实现：**
```java
public int findFirstGreaterOrEqual(int[] nums, int target) {
    int left = 0, right = nums.length - 1;
    int result = -1; // 初始化为-1表示未找到
    
    while (left <= right) {
        int mid = left + (right - left) / 2; // 防止溢出
        
        if (nums[mid] >= target) {
            // 找到符合条件的，记录下来，然后继续向左找
            result = mid;
            right = mid - 1; // 收缩右边界
        } else {
            left = mid + 1; // mid位置不满足，向右找
        }
    }
    
    return result;
}
```

**核心思路：**
1. 当`nums[mid] >= target`时，说明`mid`可能是答案，但左边可能还有更小的符合条件的
2. 因此记录`result = mid`，并继续向左收缩`right`
3. 最终`result`就是第一个大于等于目标值的位置

**测试用例：**
- `nums = [1, 3, 5, 7, 9], target = 4` → 返回2（nums[2]=5是第一个>=4的）
- `nums = [1, 3, 5, 7, 9], target = 1` → 返回0（第一个>=1的是1）
- `nums = [1, 3, 5, 7, 9], target = 10` → 返回-1（没有>=10的）
- `nums = [], target = 5` → 返回-1（空数组）

**变形题目：**
- 查找最后一个小于等于目标值的位置：将`nums[mid] >= target`改为`nums[mid] > target`，方向相反
- 查找目标值第一次出现的位置：用`nums[mid] >= target`向左找

## 题目10：算法 | 合并区间——合并所有重叠区间

【答案】
**这是一道经典的地狱难度算法题，考察排序和区间合并的逻辑。**

**题目：** 给定一个区间的集合，合并所有重叠的区间。

**示例：**
```
输入：[[1,3],[2,6],[8,10],[15,18]]
输出：[[1,6],[8,10],[15,18]]
解释：区间[1,3]和[2,6]重叠，合并为[1,6]
```

**代码实现：**
```java
public int[][] merge(int[][] intervals) {
    if (intervals == null || intervals.length == 0) {
        return new int[0][];
    }
    
    // 按起点排序
    Arrays.sort(intervals, (a, b) -> a[0] - b[0]);
    
    List<int[]> result = new ArrayList<>();
    int[] current = intervals[0]; // 初始化当前区间
    
    for (int i = 1; i < intervals.length; i++) {
        int[] next = intervals[i];
        
        if (current[1] >= next[0]) {
            // 有重叠，合并：起点保留较小值，终点点取较大值
            current[1] = Math.max(current[1], next[1]);
        } else {
            // 无重叠，将当前区间加入结果，开始新的区间
            result.add(current);
            current = next;
        }
    }
    
    // 添加最后一个区间
    result.add(current);
    
    return result.toArray(new int[0][]);
}
```

**核心思路：**
1. **排序**：按区间起点升序排列，保证相关区间相邻
2. **遍历合并**：
   - 如果当前区间的终点 >= 下一个区间的起点，则有重叠，合并（扩大终点）
   - 否则无重叠，将当前区间保存，开始新区间
3. **时间复杂度**：O(n log n)（排序），空间复杂度O(n)

**关键细节：**
- 排序后只需要一次线性扫描
- 合并时取`Math.max(current[1], next[1])`确保区间正确扩展
- 最后要加入最后一个区间

**变种题目：**
- 插入区间：先二分找到插入位置，再合并
- 无重叠区间：贪心判断每个区间是否应该被保留
- 区间交集：找出两个区间数组的交集
