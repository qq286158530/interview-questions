## 题目一：HBase 的整体架构是怎样的？请说明 HMaster 和 RegionServer 的主要职责。

### 参考答案

HBase 采用 **Master-Slave（主从）架构**，整体由以下核心组件构成：

#### 1. HMaster（主节点）
- **职责**：负责表的 CRUD 操作（创建、删除、修改表）、Region 的分配与调度、负载均衡、以及 RegionServer 的故障检测。
- HMaster 通过 Zookeeper 感知各 RegionServer 的心跳状态，当某个 RegionServer 宕机时，HMaster 会将其负责的 Region 迁移到其他健康节点。
- HMaster 可以部署多个（主备模式），但同一时刻只有一个处于 Active 状态。

#### 2. RegionServer（从节点）
- **职责**：负责数据的读写操作，直接与客户端交互。
- 每个 RegionServer 管理多个 **Region**（一段连续的数据区间），每个 Region 对应一张表的一个 Row Key 区间。
- 数据写入时，先写 WAL（Write Ahead Log），再写入 MemStore；达到阈值后刷写为 HFile 持久化到 HDFS。

#### 3. Zookeeper
- 充当分布式协调服务，保存所有 RegionServer 的元数据（ROOT、META 表位置），帮助 HMaster 感知集群状态。
- HMaster 和 RegionServer 都在 Zookeeper 中注册，心跳机制维持活性检测。

#### 4. HDFS
- HBase 底层数据最终存储在 **HDFS** 上，HFile 即 HDFS 中的数据文件。
- HDFS 提供了数据副本机制（默认 3 副本），保证了 HBase 数据的高可靠性。

## 题目二：HBase 中 RowKey 设计的核心原则有哪些？为什么 RowKey 设计至关重要？

### 参考答案

RowKey 是 HBase 行的唯一标识，数据按照 RowKey 的字典序排列存储。优秀的 RowKey 设计直接决定了查询效率和数据分布的均匀性。

#### 核心设计原则

**1. 长度适中**
- RowKey 建议控制在 8~1024 字节之间，过长会占用过多 MemStore 内存和 HFile 索引空间。

**2. 唯一性**
- RowKey 必须全局唯一，类似关系型数据库的主键。

**3. 散列（Hash）原则**
- 避免热点问题：如果 RowKey 呈顺序或时间戳递增（如 `timestamp_long`），所有写入会集中到同一个 RegionServer，造成热点。解决方案是对 RowKey 取 MD5 或 Hash 打散。

**4. 查询友好**
- 如果经常按时间范围查询，RowKey 前缀可包含时间信息（如 `userId + reverseDate`），使相近时间的数据相邻存储，便于区间扫描。

**5. 利用 Coprocessor 优化**
- 在协处理器中配合 RowKey 进行过滤，可以减少网络 IO。

#### 常见反模式
| 反模式 | 问题 | 解决方案 |
|--------|------|----------|
| 纯时间戳作为 RowKey | 写入热点集中在末尾 Region | 加盐（Salting）或 Hash 打散 |
| 自增 ID 作为 RowKey | 写入单调递增，造成热点 | 拼接随机前缀或 Hash |
| RowKey 过长 | 内存占用高，索引大 | 适当截断或编码压缩 |

## 题目三：描述 HBase 的数据写入流程，以及 MemStore 和 HFile 的作用。

### 参考答案

#### 数据写入流程

```
客户端 → ZooKeeper → 获取 META 表位置 → 定位 RegionServer
       ↓
  写入 WAL（Write Ahead Log）
       ↓
  写入 MemStore（内存）
       ↓
  返回 ACK 给客户端
       ↓
  [后台] MemStore 刷写 → HFile
```

**详细步骤：**
1. **定位路由**：客户端通过 Zookeeper 找到 `.META.` 表，获取目标 RowKey 所在的 RegionServer。
2. **写入 WAL**：先将数据追加写入 WAL（Write Ahead Log），确保数据持久化到磁盘，防止服务器崩溃导致数据丢失。
3. **写入 MemStore**：同时将数据写入 MemStore（内存中的有序缓冲区，按 RowKey 排序），立即返回客户端写入成功。
4. **后台刷写**：当 MemStore 达到阈值（默认 128MB 或 1 小时），触发 **Flush** 刷写到磁盘，生成 HFile。
5. **Compaction 合并**：后台定期执行 Minor/Major Compaction，将多个小 HFile 合并成大 HFile，删除标记为删除的数据（Tombstone Marker）。

#### MemStore 的作用
- 内存缓冲区，承接所有写入请求，提供高速写入能力。
- 数据在 MemStore 中按键字典序排序，为后续生成有序 HFile 打好基础。
- 属于写缓存，大小直接影响写入性能。

#### HFile 的作用
- HBase 数据在 HDFS 上的最终存储格式。
- HFile 内部采用 B+ 树或 SkipList 结构，包含 Data Block、Index Block、Bloom Filter 等。
- 每个 HFile 对应一个 Column Family 的部分数据。

## 题目四：HBase 的 WAL（Write Ahead Log）是什么？它在容错中扮演什么角色？

### 参考答案

#### WAL 概念
WAL（Write Ahead Log，预写日志）是 HBase 用于保证数据持久性的机制。在数据真正写入 MemStore 之前，所有操作都会先 append 到 WAL 文件中。

#### 工作原理
```
写入请求 → 写入 WAL 日志文件 → 写入 MemStore → 返回客户端
```

WAL 文件存储在 HDFS 上（默认 3 副本），即使 RegionServer 崩溃，只要 WAL 还在，就能恢复未刷写到磁盘的数据。

#### 崩溃恢复流程
1. RegionServer 宕机后，HMaster 检测到心跳超时。
2. HMaster 将该 RegionServer 管理的 Region 分配给其他节点。
3. 新节点重放（Replay）WAL 中未持久化的操作，重新写入 MemStore 并刷写为 HFile。
4. 客户端再次写入时，会从新的 RegionServer 读写数据。

#### WAL 的关键参数
| 参数 | 说明 |
|------|------|
| `hbase.wal.enabled` | 是否启用 WAL，默认 true |
| `hbase.wal.provider` | WAL 实现方式（默认 AsyncFSWAL） |
| `hbase.regionserver.wal.maximum` | 单个 RegionServer 最大 WAL 文件数 |

#### 注意事项
- **性能权衡**：WAL 写入会增加磁盘 IO，高并发写入场景可考虑适当调整刷写策略。
- **异步 vs 同步**：HBase 支持异步 WAL（默认），对于高可靠性场景可开启同步模式（每条日志强制刷盘）。

## 题目五：HBase 协处理器（Coprocessor）是什么？有哪两类协处理器？各自的典型使用场景？

### 参考答案

#### 协处理器概述
协处理器是 HBase 提供的扩展机制，允许用户将自定义代码 **部署到 RegionServer 上运行**，在数据写入/读取的路径上拦截并执行用户逻辑，无需在客户端做额外处理。

#### 两类协处理器

**1. Observer（观察者）协处理器**
- 类似于数据库的触发器（Trigger）。
- 在特定事件前后插入钩子（Hook），可干预数据操作但不改变原有流程。
- **典型使用场景：**
  - **PrePut / PostPut**：数据写入前/后执行自定义逻辑，如记录审计日志、数据脱敏、生成二级索引。
  - **PreGet / PostGet**：读取前/后拦截，可用于访问控制、结果过滤。
  - **PreDelete / PostDelete**：删除前/后执行清理逻辑。
  - **RegionObserver**：针对特定 Region 级别的事件处理。

**2. Endpoint（端点）协处理器**
- 类似于数据库的存储过程（Stored Procedure）。
- 将计算逻辑推送到数据所在节点执行，减少数据传输。
- 允许在服务器端动态执行自定义操作，返回聚合结果给客户端。
- **典型使用场景：**
  - 服务端聚合（如 Sum、Average、Count）。
  - 复杂计算（服务器端直接完成，避免大量数据返回客户端）。
  - 动态列计算。

#### 示例：使用 Observer 实现二级索引
```java
public class IndexObserver extends RegionObserver {
    @Override
    public void postPut(ObserverContext<RegionCoprocessorEnvironment> e,
                        Put put, WALEdit edit) {
        // 写入主表后，自动向索引表写入索引数据
        // 实现 ES/HBase 二级索引同步等场景
    }
}
```

#### 部署方式
- **静态加载**：修改 `hbase-site.xml`，重启集群后生效，适用于所有表。
- **动态加载（表级）**：通过 HBase Shell 或 API 将 Coprocessor 绑定到指定表，无需重启集群。
