## 题目一：HBase 的整体架构是怎样的？Region、RegionServer、HMaster 的职责是什么？

### 参考答案

HBase 是一个基于 **Google BigTable** 论文设计的分布式 NoSQL 列族数据库，采用 **ZooKeeper** 进行协调，**HDFS** 作为底层存储。其核心架构如下：

**1. HMaster（主节点）**
- 负责表的 DDL 操作（创建、删除表，修改表结构）
- 负责 Region 的分配和调度，将 Region 分配给 RegionServer
- 监控集群中所有 RegionServer 的状态，发现故障时进行负载均衡和恢复
- 接收客户端的元数据请求，返回对应的 RegionServer 地址

**2. RegionServer（从节点）**
- 负责管理 Region，真正处理读写请求
- 每台 RegionServer 包含多个 Region，每个 Region 对应一段连续的 RowKey 区间
- 响应客户端的 CRUD 请求，与 HDFS DataNode 交互完成数据读写
- 定期向 HMaster 汇报心跳和负载情况

**3. Region（数据分片）**
- HBase 表的水平分片单元，每张表可能被切分成多个 Region
- 每个 Region 有[startkey, endkey) 的 RowKey 区间
- 首次插入数据时，一张表只有一个 Region，随着数据增长自动拆分

**4. ZooKeeper**
- 存储元数据（如 `.META.` 表的位置、Master 地址）
- 实现分布式协调（选主、故障检测）

**5. 写入流程简述**
```
Client → ZooKeeper（获取 .META. 位置）→ .META. 表（获取目标 Region）→ RegionServer
```

## 题目二：HBase 的存储结构是什么？MemStore 和 HFile 的作用是什么？它们之间的关系？

### 参考答案

HBase 采用 **LSM-Tree（Log-Structured Merge-Tree）** 思想进行存储，数据写入流程如下：

**1. MemStore（写缓存）**
- 数据写入时，先写入 **MemStore**（一个内存中的有序 SkipList 结构）
- 满足条件后（达到阈值或时间触发），MemStore 会 **flush** 成 HFile 文件
- MemStore 中的数据已经按 RowKey 排序，因此 HFile 是有序的

**2. HFile（HBase 的存储文件）**
- HFile 是 HDFS 上的数据文件，存储在 DataNode 上
- 采用 **HFile V3** 格式，包含：Data Block、Meta Block、FileInfo、Index Block 等
- 支持 **B+树 索引**，通过 BloomFilter 快速定位 Key

**3. 它们的关系**
| 维度 | MemStore | HFile |
|------|----------|-------|
| 存储位置 | 内存 | HDFS |
| 数据有序性 | 按 RowKey 排序 | 按 RowKey 排序 |
| 写入方式 | 实时写入 | 定期 flush |
| 持久化 | 否（掉丢） | 是（掉电不丢） |
| 合并 | flush 后形成 | 触发 Major/Minor Compaction 合并 |

**4. Minor Compaction 与 Major Compaction**
- **Minor Compaction**：小规模合并，将多个小的 HFile 合并成少量较大的 HFile
- **Major Compaction**：将一个 Region 下的所有 HFile 合并成一个，删除过期版本和被删除的数据

## 题目三：HBase 的 RowKey 如何设计？有哪些设计原则和常用技巧？

### 参考答案

RowKey 是 HBase 查询的核心，决定了数据分布和查询效率。

**1. RowKey 设计原则**

- **唯一性**：RowKey 必须唯一标识一行数据
- **长度适中**：建议 16~100 字节，越短越好（Region 按 RowKey 范围分片，过长的 Key 会增加索引体积）
- **散列性**：避免热点（所有数据集中在少数 RegionServer 上），确保写入均匀分布

**2. 防止热点的策略**

- **加盐（Salting）**：在 RowKey 头部拼接随机前缀，使数据分散到不同 Region，例如：`MD5(rowKey) % N + originalKey`
- **哈希打散**：对 RowKey 做 MD5 或 Hash 取模后拼接原 Key
- **反转（Reversing）**：将固定前缀的字段反转，如手机号 `138xxxx` → `xxxx831`，打散基于手机号的查询

**3. 常见 RowKey 设计模式**

| 场景 | RowKey 设计方案 | 示例 |
|------|----------------|------|
| 用户订单查询 | 用户ID + 时间倒序 | `userId_20260706_orderId` |
| 时序数据 | 时间戳 + 设备ID（反转） | `reverse(timestamp)_deviceId` |
| 互斥查询 | 加盐前缀 + 业务ID | `salt_md5(id)_id` |
| 多租户系统 | 租户ID + 业务ID | `tenantId_bizId` |

**4. 注意事项**
- 如果使用加盐，读取时需要扫描多个 Region（读放大）
- 避免单调递增的 RowKey（如时间戳开头），会导致写入热点

## 题目四：HBase 的读写流程是怎样的？BloomFilter 和 BlockCache 在其中起到什么作用？

### 参考答案

**1. 写入流程（Write-Ahead Log 机制）**

```
① Client 发起写入 → 连接 ZooKeeper 获取 .META. 表位置
② 查询 .META. 确定目标 RowKey 属于哪个 Region
③ 数据先写入对应 RegionServer 的 WAL（Write-Ahead Log），确保数据可恢复
④ 数据写入 MemStore（内存排序结构）
⑤ 返回客户端 "写入成功"（此时数据已在内存和 WAL 中）
⑥ 异步 flush：MemStore 写满后 flush 为 HFile
```

**2. 读取流程（Read Flow）**

```
① Client 查询 ZooKeeper → 获取 .META. 表位置 → 确定目标 RegionServer
② 先查询 BlockCache（读缓存，LruCache 实现）
   - 命中 → 直接返回
   - 未命中 → 继续
③ 查询 MemStore（如果数据还在内存中，直接返回）
④ 如果仍未命中，遍历 HFile 文件读取数据（可能触发大量磁盘 IO）
⑤ 将结果写入 BlockCache，供后续查询使用
⑥ 返回客户端
```

**3. BloomFilter 的作用**
- 每个 HFile 包含 **BloomFilter** 元数据，快速判断一个 Key **是否不存在于该 HFile 中**
- 避免无效的 IO 查询：如果 BloomFilter 返回不存在，则直接跳过该 HFile
- 适用场景：稀疏列、随机查询（点查）场景效果显著

**4. BlockCache 的作用**
- 采用 **LRU（Least Recently Used）** 策略，缓存热点 HFile 数据块
- 读请求优先查缓存，减少磁盘 IO
- 配置参数：`hfile.block.cache.size`（默认堆内存的 40%）
- 写操作不会直接写入 BlockCache（只有被读取的 HFile 才会缓存）

## 题目五：HBase 协处理器（Coprocessor）是什么？Observer 和 Endpoint 有什么区别？典型使用场景？

### 参考答案

**1. 什么是 Coprocessor**

协处理器是 HBase 提供的扩展机制，允许用户将自定义代码 **部署到 RegionServer 上运行**，在数据读写过程中进行拦截和增强处理，类似数据库的触发器和存储过程。

**2. 两类协处理器**

| 类型 | Observer（观察者） | Endpoint（端点） |
|------|-------------------|-----------------|
| 机制 | 钩子函数，事件驱动 | RPC 调用，类 RPC 服务 |
| 作用 | 在特定事件前后执行拦截逻辑 | 跨行跨 Region 的分布式计算 |
| 触发时机 | Get/Put/Delete/Open/Close 等 Hook | 需客户端显式调用 |
| 典型用途 | 权限校验、二次索引、审计日志 | 聚合计算（Sum/Avg）、自定义过滤 |

**3. Observer 的 Hook 种类**
- **RegionObserver**：行级别操作（preGet、postGet、prePut、postPut、preDelete、postDelete 等）
- **MasterObserver**：DDL 操作（createTable、deleteTable、addColumn 等）
- **WALObserver**：WAL 写入相关

**4. 典型使用场景**

**场景 A：二级索引（Index Observer）**
```
在 postPut 钩子中，解析写入的列值，构建索引行写入另一张索引表
例如：订单表按 userId 建索引 → 索引表 RowKey = userId，Value = orderId
```

**场景 B：聚合计算（Endpoint）**
```
在 RegionServer 上执行 region 内的聚合（Count/Sum/Max）
客户端汇总各 Region 的结果，减少网络传输
类似 MapReduce 的思路，但直接在 RegionServer 层执行
```

**场景 C：权限控制（Access Controller）**
```
在 preGet/prePut 中判断当前用户是否有权访问该行，非法访问直接抛出 AccessDeniedException
```

**5. 注意事项**
- 协处理器代码随 Region 加载，异常会影响 Region 可用性，需做好异常处理
- HBase 2.x 对协处理器 API 有较大改动，从 1.x 迁移需注意兼容性问题
