## 题目一：Redis 有哪些数据类型？各自的使用场景是什么？

### 参考答案

Redis 提供了丰富的数据类型，主要分为 **5 种基础数据类型** 和 **4 种扩展数据类型**。

#### 五种基础数据类型

| 数据类型 | 使用场景 |
|---------|---------|
| **String（字符串）** | 缓存对象、常规计数（PV/UV）、分布式锁、共享 session 信息 |
| **Hash（哈希）** | 缓存对象（如用户信息）、购物车 |
| **List（列表）** | 消息队列（需自行实现全局唯一 ID、不支持消费组）、文章列表、任务队列 |
| **Set（集合）** | 聚合计算场景：点赞、共同关注、抽奖活动（利用并集/交集/差集） |
| **Zset（有序集合）** | 排序场景：排行榜、电话和姓名排序 |

#### 四种扩展数据类型

| 数据类型 | 版本 | 使用场景 |
|---------|------|---------|
| **BitMap（位图）** | 2.2+ | 二值状态统计：签到、判断用户登录状态、连续签到用户总数 |
| **HyperLogLog** | 2.8+ | 海量数据基数统计：百万级网页 UV 计数（内存占用极低） |
| **GEO（地理信息）** | 3.2+ | 存储地理位置：滴滴叫车、外卖配送范围 |
| **Stream（流）** | 5.0+ | 消息队列，相比 List 支持自动生成全局唯一消息 ID 和消费组形式消费 |

#### 五种数据类型的底层实现

- **String**：底层基于 SDS（简单动态字符串），支持二进制数据、O(1) 复杂度获取长度、安全拼接。
- **List**：Redis 3.2 之前用双向链表或压缩列表；3.2+ 统一用 quicklist 实现。
- **Hash**：元素少时用压缩列表（ziplist），元素多时切换为哈希表。
- **Set**：全是整数且元素少时用整数集合（intset），否则用哈希表。
- **Zset**：元素少时用压缩列表，否则用跳表（skiplist）+ 哈希表组合实现。

## 题目二：Redis 的持久化机制是什么？AOF 和 RDB 有什么区别？

### 参考答案

Redis 共有三种数据持久化方式：**AOF 日志**、**RDB 快照** 和 **混合持久化**。

#### AOF（Append Only File）

- **原理**：每执行一条写操作命令，就把该命令以追加的方式写入 AOF 文件。Redis 重启时读取 AOF 文件中的命令，逐一执行来恢复数据。
- **写回策略**（由 `appendfsync` 配置）：
  - `Always`：每次写命令执行完后同步写回硬盘，安全性最高但性能最差。
  - `Everysec`（默认）：每秒将缓冲区内容写回硬盘，折中方案。
  - `No`：由操作系统决定何时写回，性能最好但数据安全性最低。
- **AOF 重写机制**：当 AOF 文件过大时，后台子进程 `bgrewriteaof` 读取当前数据库所有键值对，用一条命令记录每个 key 的最新值，从而压缩文件体积。重写过程中主进程通过 **AOF 重写缓冲区** 保证数据一致性。

#### RDB（Redis Database）

- **原理**：将某一时刻的内存数据以二进制格式写入磁盘，记录的是实际数据而非命令。
- **快照命令**：
  - `save`：在主线程执行，会阻塞。
  - `bgsave`：创建子进程生成 RDB 文件，不阻塞主线程。
- **触发方式**：可配置 `save m n` 自动触发，满足"m 秒内 n 次修改"即执行 bgsave。
- **COW 机制**：bgsave 使用写时复制（Copy-On-Write），子进程共享父进程内存，只在数据被修改时才复制副本。

#### 混合持久化（Redis 4.0+）

- 开启混合持久化后，AOF 重写时先以 RDB 格式写入全量数据，再以 AOF 格式追加增量命令。
- 优点：Redis 启动时快速加载 RDB 部分，后续增量通过 AOF 重放恢复，数据丢失少。
- 缺点：AOF 可读性差，不兼容 Redis 4.0 之前版本。

#### AOF vs RDB 对比

| 特性 | AOF | RDB |
|------|-----|-----|
| 文件格式 | 命令日志 | 二进制快照 |
| 恢复速度 | 慢（需逐条执行命令） | 快（直接读入内存） |
| 数据安全性 | 取决于写回策略 | 取决于快照频率 |
| 文件体积 | 较大 | 较小 |
| 对性能影响 | 写操作稍微变慢 | bgsave 不影响，频率过高影响 |

## 题目三：Redis 为什么是单线程的？为什么还这么快？

### 参考答案

#### Redis 单线程指的是什么

Redis 单线程是指 **「接收客户端请求→解析请求→执行数据读写→发送响应」** 这一完整流程由主线程完成。但 Redis 实际上并非完全单线程：

- **Redis 2.6** 起会启动 **2 个后台线程（BIO）**：分别处理关闭文件、AOF 刷盘任务。
- **Redis 4.0+** 新增第三个后台线程 **lazyfree**：异步释放内存，执行 `unlink key`、`flushdb async` 等命令时不会阻塞主线程。
- **Redis 6.0+** 引入了 **I/O 多线程**，用于并行处理网络 I/O（write client socket），但命令执行仍是单线程。

#### Redis 为什么这么快

1. **内存操作**：大部分操作在内存中完成，CPU 不是瓶颈，内存和网络带宽才是。
2. **高效数据结构**：SDS（动态字符串）、跳表、压缩列表等都是为性能优化的数据结构。
3. **无锁设计**：单线程避免了多线程竞争和上下文切换开销，也不存在死锁问题。
4. **I/O 多路复用**：使用 `select/epoll` 机制，一个线程同时管理多个 Socket，内核监听所有连接请求，有请求到达才交给 Redis 处理，实现了一个线程处理大量并发 I/O。

#### Redis 6.0 之前为什么坚持用单线程

根据 Redis 官方 FAQ：CPU 不是性能瓶颈，更多受限于内存大小和网络 I/O。单线程模型可维护性更高，多线程带来的并发问题（竞争条件、锁、死锁）会增加复杂度。如果需要利用多核 CPU，可以在同一台服务器上启动多个 Redis 实例，或使用分片集群。

#### Redis 6.0 之后引入多线程的目的

随着网络硬件性能提升，网络 I/O 成为了新的瓶颈。Redis 6.0 对**网络 I/O 读写**引入多线程，但**命令执行仍是单线程**，避免破坏数据一致性。官方实测性能提升至少一倍。

## 题目四：Redis 如何实现高可用？主从复制和哨兵模式有什么区别？

### 参考答案

#### 主从复制（Replication）

- **原理**：一主多从架构，主服务器负责读写操作，从服务器只读并同步主服务器的写操作命令（异步进行）。
- **读写分离**：所有写操作在主服务器执行，读操作可分散到从服务器，提升并发能力。
- **数据一致性**：无法保证强一致性，因为主服务器不会等待从服务器实际执行完命令才返回结果。
- **同步方式**：从服务器连接主服务器时发送 `PSYNC` 命令，主服务器返回 RDB 文件并持续推送增量命令。

#### 哨兵模式（Sentinel）

- **解决的问题**：主从模式下，如果主服务器宕机，需要手动恢复。哨兵模式实现自动故障转移。
- **核心功能**：
  - **监控**：哨兵进程监控主从服务器是否正常运行。
  - **自动故障转移**：主服务器下线时，哨兵选举出一个从服务器升级为新的主服务器。
  - **通知**：将故障转移结果通知客户端（通过发布订阅）。
- **哨兵集群**：Redis 支持多个哨兵组成集群，避免单哨兵误判，提高可靠性。

#### 主从复制 vs 哨兵模式对比

| 特性 | 主从复制 | 哨兵模式 |
|------|---------|---------|
| 高可用能力 | 基础，数据冗余 | 自动故障转移 |
| 故障恢复 | 手动介入 | 自动切换 |
| 写能力 | 仅主节点可写 | 仅主节点可写 |
| 读能力 | 主从均可 | 主从均可 |
| 自动化程度 | 低（手动切换） | 高（自动监控和切换） |

## 题目五：Redis 缓存更新策略有哪些？Cache Aside 模式是什么？

### 参考答案

缓存更新策略主要有四种模式：**Cache Aside**、**Read Through**、**Write Through**、**Write Behind**。

#### Cache Aside（旁路缓存）- 最常用

- **读**：先查缓存，命中则返回；未命中则查数据库，写入缓存，再返回。
- **写**：先更新数据库，再删除缓存（而非更新缓存）。后续读操作会重新将新数据写入缓存。
- **优点**：避免了并发写操作导致缓存和数据库数据不一致的问题。
- **潜在问题**：读缓存时缓存失效 + 并发写的corner case下可能短暂出现脏数据，但概率极低。
- **Facebook** 在论文 *Scaling Memcache at Facebook* 中使用了此策略。

> 来源：https://coolshell.cn/articles/17416.html

#### Read Through（读穿透）

- 缓存失效时，由缓存服务自己加载数据，对应用层透明。
- 应用层认为后端就是一个单一的存储。

#### Write Through（写穿透）

- 写操作时，缓存命中则同时更新缓存和数据库（同步）；缓存未命中则直接更新数据库。
- 数据强一致，但写操作延迟较高。

#### Write Behind（写回 / Write Back）

- 写操作只更新缓存，异步批量将数据写回数据库。
- 性能最高，但存在数据丢失风险（系统异常时未刷盘的数据会丢失）。
- 类似 Linux 文件系统 Page Cache 的 lazy write 机制。

#### 四种策略对比

| 策略 | 一致性 | 性能 | 复杂度 | 数据安全性 |
|------|--------|------|--------|-----------|
| Cache Aside | 最终一致 | 高 | 中 | 中（取决于写回策略） |
| Read Through | 最终一致 | 高 | 低 | 中 |
| Write Through | 强一致 | 低 | 中 | 高 |
| Write Behind | 弱一致 | 最高 | 高 | 低 |

#### 实际建议

- 大多数场景推荐 **Cache Aside** 模式，是工程实践中被广泛验证的最佳选择。
- 为缓存设置合理的 **过期时间（TTL）**，作为兜底策略。
- 对一致性要求高的场景，可使用 **分布式锁** 或 **两阶段提交协议（2PC/Paxos）** 保障一致性，但会牺牲性能。
