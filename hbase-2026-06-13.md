## 题目一：HBase 的整体架构是怎样的？

### 参考答案

HBase 是一个基于 Google BigTable 设计的分布式列式存储系统，其整体架构采用 **Master-Slave** 模式，主要包含以下组件：

### 1. **架构角色**

| 角色 | 说明 | 备注 |
|------|------|------|
| **HMaster** | 主节点，负责管理 Region 调度、负载均衡、Schema 变更 | 可配置多主备部署 |
| **HRegionServer** | 数据节点，负责处理读写请求、管理 MemStore 和 HFile | 数据存储和处理核心 |
| **ZooKeeper** | 协调服务，管理集群元数据（Master 地址、Region 位置） | 奇数节点部署 |
| **HDFS** | 底层存储，HBase 的数据最终持久化到 HDFS | 通常 3 副本存储 |

### 2. **Region 概念**

- **Region** 是 HBase 负载均衡的基本单位
- 每个 Region 存储一段 **RowKey 区间** 的数据
- 当 Region 过大时（默认 10GB），会触发 **分裂（Split）**
- Region 由 HRegionServer 管理，可在不同节点间迁移

### 3. **写数据流程**

```
Client → ZooKeeper → 找到对应 HRegionServer → 写入 WAL → 写入 MemStore → 响应
```

### 4. **读数据流程**

```
Client → ZooKeeper → 找到对应 HRegionServer → 合并 MemStore + HFile → 返回结果
```

## 题目二：HBase 的 RowKey 设计原则有哪些？

### 参考答案

RowKey 是 HBase 中最重要的设计要素，直接影响查询效率和读写性能。

### 1. **RowKey 设计原则**

| 原则 | 说明 | 避免 |
|------|------|------|
| **长度原则** | RowKey 最大 64KB，建议越短越好（不超过 100 字节） | 过长的 RowKey 占用内存和 HDFS 空间 |
| **唯一性** | 必须唯一标识一行数据 | 重复的 RowKey 会覆盖旧数据 |
| **散列性** | 避免热点，数据均匀分布 | 连续 RowKey 导致单一 Region 压力过大 |

### 2. **常见 RowKey 设计策略**

#### (1) 盐值（Salting）
```
# 在 RowKey 前面添加随机前缀
RowKey = (hash(node_id) % N) + original_key
```

#### (2) 哈希（Hashing）
```
RowKey = MD5(original_key).substring(0, 8) + original_key
```

#### (3) 反转（Reversal）
```
# 反转固定格式的 RowKey，如手机号、时间戳
RowKey = reverse(phone_number)
```

### 3. **热点问题解决方案**

- **加随机数**：在 RowKey 末尾添加随机数
- **时间戳反转**：如 `long.MAX_VALUE - timestamp`
- **业务层面拆分**：将热点 key 拆分到多个 Region

## 题目三：HBase 的 MemStore 和 HFile 是什么？它们之间如何协作？

### 参考答案

### 1. **MemStore（写缓存）**

- 位于 **HRegionServer** 内存中
- 接收所有 **写请求**（先写 WAL，再写 MemStore）
- 当 MemStore 达到 **128MB**（可配置）时，触发 **刷盘（Flush）**
- 按 RowKey 排序存储，支持高效的范围查询

### 2. **HFile（HBase 的存储文件）**

- 存储在 **HDFS** 上，是 HBase 的实际数据文件
- 采用 **LSM 树（Log-Structured Merge-Tree）** 结构
- 由 Data Block、Index Block、Bloom Filter 等组成
- 默认块大小 64KB，支持压缩

### 3. **协作流程**

```
写请求:
  ① 数据写入 WAL（日志）
  ② 数据写入 MemStore（内存）

刷盘触发（满足以下任一条件）:
  - MemStore 大小达到 128MB
  - HDFS 阻塞或超时
  - Region 下线前

刷盘过程:
  MemStore → 生成 HFile → 写入 HDFS
           → 删除对应的 WAL 日志

读请求:
  合并 MemStore（内存）+ HFile（磁盘）→ 返回结果
```

### 4. **Minor Compaction 与 Major Compaction**

| 类型 | 说明 | 触发条件 |
|------|------|---------|
| **Minor Compaction** | 合并少量小 HFile | 自动触发，默认 3 个以上小文件 |
| **Major Compaction** | 合并所有 HFile，清理删除数据 | 手动或自动，默认 7 天一次 |

## 题目四：什么是 HBase 协处理器（Coprocessor）？有哪两种类型？

### 参考答案

### 1. **协处理器概述**

协处理器是 HBase 提供的扩展机制，允许用户在 **Region 级别** 执行业务逻辑，类似于数据库的触发器和存储过程。

### 2. **两种协处理器类型**

#### (1) **Observer（观察者）**

类似数据库触发器，在特定事件前后执行钩子方法：

| 类型 | 触发时机 | 使用场景 |
|------|---------|---------|
| **PrePut** | 数据写入前 | 数据验证、权限控制 |
| **PostPut** | 数据写入后 | 异步索引、审计日志 |
| **PreGet** | 数据读取前 | 列级权限控制 |
| **PostGet** | 数据读取后 | 数据脱敏 |
| **PreDelete** | 删除前 | 关联删除 |
| **PostDelete** | 删除后 | 清理关联数据 |

#### (2) **Endpoint（端点）**

类似数据库存储过程，允许在服务端执行自定义计算：

- 使用 **Protobuf** 定义接口
- 客户端调用协处理器方法，服务端在 Region 上执行
- 返回计算结果（如聚合、分组操作）

### 3. **加载方式**

```xml
<!-- hbase-site.xml -->
<property>
  <name>hbase.coprocessor.user.region.classes</name>
  <value>com.example.MyObserver,com.example.MyEndpoint</value>
</property>
```

### 4. **使用场景**

- **二级索引**：在写入时自动更新索引表
- **聚合查询**：在服务端计算 SUM、COUNT 等
- **数据验证**：写入前校验数据合法性
- **审计日志**：记录所有操作到独立表

## 题目五：HBase 的存储结构是怎样的？如何实现高效查询？

### 参考答案

### 1. **数据模型**

```
Table
├── RowKey1
│   ├── Column Family: cf1
│   │   ├── Column: col1 → Value1 (版本1, timestamp)
│   │   ├── Column: col1 → Value2 (版本2, timestamp)
│   │   └── Column: col2 → Value3
│   └── Column Family: cf2
│       └── Column: col3 → Value4
├── RowKey2
│   └── ...
```

- **Table**：表，schema 由 Column Family 定义
- **RowKey**：行键，字节数组，唯一标识一行
- **Column Family**：列族，列的分组，必须在建表时定义
- **Column Qualifier**：列限定符，动态添加
- **Cell**：单元格，存储 (RowKey + CF + CQ + Timestamp) → Value
- **Timestamp**：版本号，默认支持多版本

### 2. **HFile 结构**

```
HFile
├── FileInfo (元信息)
├── Data Block 1
│   ├── Key Length (4B)
│   ├── Value Length (4B)
│   ├── Key (RowKey + CF + CQ + Timestamp)
│   └── Value
├── Data Block 2
├── ...
├── Data Block Index (数据块索引)
├── Meta Block Index (元数据块索引)
├── Bloom Filter (布隆过滤器)
└── Trailer (文件尾部信息)
```

### 3. **Bloom Filter 原理**

- **作用**：快速判断某个 RowKey 是否存在于 HFile 中
- **原理**：使用多个哈希函数映射到位数组，判断时存在不一定在，不存在一定不在
- **优势**：减少不必要的 HFile 读取，提升读性能 10-100 倍

### 4. **查询优化机制**

| 机制 | 说明 |
|------|------|
| **Block Cache** | 读缓存，LRU 策略，热点数据缓存加速 |
| **Bloom Filter** | 快速判断行/列是否存在 |
| **Index** | 数据块索引，通过 RowKey 定位数据块 |
| **Compaction** | 合并 HFile，减少文件数量 |

### 5. **LSM 树（Log-Structured Merge-Tree）**

```
MemStore (内存有序)
    ↓ Flush
Level 0 (若干 HFile)
    ↓ Minor Compaction
Level 1 (更大的 HFile)
    ↓ Major Compaction
Level 2 ...
```

- **写优化**：顺序写入 MemStore，避免随机 I/O
- **读优化**：通过 Compaction 合并，减少文件数量
- **空间放大**：写入放大，需要定期 Compaction
