## 题目一：Redis 和 Memcached 有什么区别？为什么 Redis 更适合作为 MySQL 的缓存？

### 参考答案

**Redis 与 Memcached 的共同点：**
- 都是基于内存的数据库，一般都用来当做缓存使用
- 都有过期策略
- 两者的性能都非常高

**Redis 与 Memcached 的区别：**

| 区别点 | Redis | Memcached |
|--------|-------|-----------|
| 数据类型 | 支持 String、Hash、List、Set、ZSet 等丰富数据类型 | 只支持最简单的 key-value 数据类型 |
| 持久化 | 支持 RDB 和 AOF 两种持久化方式 | 没有持久化功能，数据全部存在内存中 |
| 集群模式 | 原生支持 Cluster 集群模式 | 没有原生集群模式，需依赖客户端实现分片 |
| 高级特性 | 支持发布订阅、Lua 脚本、事务等 | 不支持 |

**为什么 Redis 更适合作为 MySQL 的缓存？**

主要是因为 Redis 具备「高性能」和「高并发」两种特性：

1. **高性能**：用户第一次访问 MySQL 中的数据较慢（从硬盘读取），将数据缓存到 Redis 后，下一次访问可以直接从内存获取，速度相当快。

2. **高并发**：Redis 单机 QPS 能轻松破 10W，而 MySQL 单机 QPS 很难破 1W。直接访问 Redis 能承受的请求远大于直接访问 MySQL，可以将部分数据库请求转移到缓存层。

## 题目二：Redis 的过期删除策略是什么？惰性删除和定期删除有什么区别？

### 参考答案

Redis 使用的过期删除策略是「**惰性删除 + 定期删除**」两种策略配合使用。

**惰性删除策略：**
- 不主动删除过期键，每次从数据库访问 key 时才检测是否过期
- 如果过期则删除该 key
- **优点**：对 CPU 时间最友好，只占用很少的系统资源
- **缺点**：如果过期 key 一直没有被访问，占用的内存就不会释放，对内存不友好

**定期删除策略：**
- 每隔一段时间随机从数据库中取出一定数量的 key 进行检查，删除其中的过期 key
- 具体流程：
  1. 从过期字典中随机抽取 20 个 key
  2. 检查这 20 个 key 是否过期，并删除已过期的 key
  3. 如果已过期 key 超过 5 个（25%），继续重复步骤 1；否则停止
- Redis 为了保证定期删除不会循环过度导致卡死，设置了时间上限，默认不超过 25ms
- **优点**：减少过期键对空间的无效占用，对 CPU 友好
- **缺点**：难以确定删除操作的时长和频率

惰性删除对内存不友好，定期删除对 CPU 不友好，两者配合使用可以在合理使用 CPU 时间和避免内存浪费之间取得平衡。

## 题目三：Redis 的持久化机制有哪些？RDB 和 AOF 各自有什么优缺点？

### 参考答案

Redis 共有三种数据持久化方式：

### 1. RDB 快照（Redis Database）

RDB 快照是记录某一时刻的内存数据，以二进制方式写入磁盘。

- **执行命令**：`save`（主线程执行，会阻塞）和 `bgsave`（子进程执行，不阻塞）
- **触发条件**：通过配置文件设置，满足以下任一条件即执行 bgsave：
  - 900 秒内至少 1 次修改
  - 300 秒内至少 10 次修改
  - 60 秒内至少 10000 次修改
- **快照过程中数据可修改**：利用写时复制（COW）技术，bgsave 期间主线程可以继续处理命令

### 2. AOF 日志（Append Only File）

每执行一条写操作命令，就把该命令以追加的方式写入文件。

**AOF 写回策略（appendfsync 配置项）：**

| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| Always | 每次写操作后同步写回硬盘 | 数据不丢失 | 磁盘 I/O 开销大 |
| Everysec | 每秒将缓冲区内容写回硬盘 | 性能适中 | 最多丢失 1 秒数据 |
| No | 由操作系统决定何时写回 | 性能最好 | 数据丢失不可控 |

**AOF 重写机制**：当 AOF 文件过大时，后台子进程 bgrewriteaof 会读取数据库所有键值对，用一条命令记录到新的 AOF 文件，实现压缩。

### 3. 混合持久化（Redis 4.0+）

结合 RDB 和 AOF 的优点：
- AOF 文件前半部分是 RDB 格式的全量数据
- 后半部分是 AOF 格式的增量数据
- 启动时加载 RDB 部分快速恢复，结合 AOF 保证数据不丢失

**RDB vs AOF 对比：**

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件体积 | 小（二进制） | 大（文本命令） |
| 恢复速度 | 快 | 慢 |
| 数据安全性 | 可能丢失较多数据 | 可配置（每秒/每次） |
| 性能影响 | fork 子进程，轻微 | 持续写入，略有影响 |

## 题目四：Redis 如何实现服务高可用？什么是哨兵模式？什么是 Cluster 集群？

### 参考答案

Redis 通过三种方案实现服务高可用：

### 1. 主从复制（Master-Slave）

- 一主多从模式，主服务器负责读写，从服务器只读
- 主服务器收到写操作后异步同步给从服务器
- **注意**：主从复制是异步的，无法保证强一致性

### 2. 哨兵模式（Sentinel）

解决主从服务器故障时需要手动恢复的问题：
- 监控主从服务器运行状态
- 当主服务器下线时，自动进行故障转移
- 选举从服务器中表现优秀的作为新的主服务器
- 通知客户端新的主服务器地址

### 3. Redis Cluster（切片集群）

当数据量大到单台服务器无法缓存时使用：
- 将数据分布在不同的服务器上
- 采用 **16384 个哈希槽（Hash Slot）** 处理数据与节点的映射
- 数据定位：key → CRC16 校验 → 对 16384 取模 → 找到对应哈希槽 → 找到节点
- 支持自动分配或手动分配哈希槽

**集群脑裂问题：**
- 原因：主节点网络故障但与客户端连接正常，哨兵选举出新主节点后产生两个主节点
- 解决：配置 `min-slaves-to-write` 和 `min-slaves-max-lag`，限制主节点在从节点不足或延迟过高时禁止写入

## 题目五：什么是缓存雪崩、缓存击穿、缓存穿透？如何避免？

### 参考答案

这是 Redis 最经典的高频面试题之一，考察对缓存系统异常情况的理解。

### 1. 缓存雪崩

**定义**：大量缓存数据在同一时间过期失效，此时有大量用户请求都无法在 Redis 中处理，全部直接访问数据库。

**解决方案：**
- **过期时间随机化**：在原有失效时间基础上增加随机值（1~10分钟），降低缓存集体失效概率
- **设置缓存不过期**：通过后台服务更新缓存数据，避免因过期造成缓存雪崩

### 2. 缓存击穿

**定义**：某个热点数据过期，而大量请求恰好访问该热点数据，导致直接查询数据库。

**解决方案：**
- **互斥锁**：使用 `setNX` 保证同一时间只有一个业务线程请求缓存
- **热点数据永不过期**：由后台异步更新缓存，或在热点数据过期前主动更新

### 3. 缓存穿透

**定义**：用户请求的数据既不在缓存中，也不在数据库中（如恶意攻击或业务误操作）。

**常见场景：**
- 业务误操作，缓存和数据库数据都被删除
- 黑客恶意攻击大量访问不存在的数据

**解决方案：**
- **非法请求限制**：在 API 入口处校验请求参数
- **设置空值/默认值**：缓存中设置空值或默认值，避免查询数据库
- **布隆过滤器**：写入数据库时标记到布隆过滤器，查询时先通过布隆过滤器判断数据是否存在

**三种异常对比：**

| 异常类型 | 原因 | 特点 |
|----------|------|------|
| 缓存雪崩 | 大量缓存同时过期 | 缓存层失效，直接打满数据库 |
| 缓存击穿 | 热点缓存过期 | 某个热点 key 失效，并发请求打到数据库 |
| 缓存穿透 | 数据既不在缓存也不在数据库 | 恶意请求或业务问题，正常请求也被拒绝 |
