## 题目一：Redis为什么这么快？

### 参考答案

Redis 之所以这么快，主要有以下几个原因：

**1. 纯内存操作**
Redis 数据读写操作都发生在内存中，访问速度是纳秒级别，而传统数据库频繁读写磁盘的速度是毫秒级别，两者相差数个数量级。

**2. 高效的 I/O 模型**
Redis 使用单线程事件循环配合 I/O 多路复用技术（epoll/select/kqueue），让单个线程可以同时处理多个网络连接上的 I/O 事件，避免了多线程模型中的上下文切换和锁竞争问题。

**3. 优化的内部数据结构**
Redis 提供多种数据类型（String, List, Hash, Set, Sorted Set等），内部实现采用高度优化的编码方式（如 ziplist、quicklist、skiplist、hashtable 等），并会根据数据大小和类型动态选择最合适的内部编码。

**4. 简洁高效的通信协议**
Redis 使用自己设计的 RESP（REdis Serialization Protocol）协议，实现简单、解析性能好、二进制安全，序列化/反序列化开销很小。

> ⚠️ 注意：Redis 6.0 引入了多线程模型，但仅针对网络数据的读写，命令执行仍然是单线程的，所以核心特性不变。

## 题目二：Redis的数据类型有哪些？ZSet底层是如何实现的？

### 参考答案

**Redis 数据类型分为 5 种基础类型和 4 种扩展类型：**

基础类型：
- **String（字符串）**：缓存对象、常规计数、分布式锁
- **Hash（哈希）**：缓存对象、购物车
- **List（列表）**：消息队列（需自行实现全局唯一ID）
- **Set（集合）**：聚合计算（并集、交集、差集），如点赞、共同关注
- **Zset（有序集合）**：排行榜、电话姓名排序

扩展类型（Redis 2.2+）：
- **BitMap**：二值状态统计（签到、用户登录状态）
- **HyperLogLog**：海量数据基数统计（百万级网页UV计数）
- **GEO**：存储地理位置（滴滴叫车）
- **Stream（5.0+）**：消息队列，支持消费组形式消费

**ZSet 底层实现：**
ZSet 类型的底层数据结构是由**压缩列表（ziplist）**或**跳表（skiplist）**实现的：

- 当有序集合的元素个数 **< 128 个**，且每个元素的值 **< 64 字节**时，使用压缩列表
- 否则使用跳表

> 📌 在 Redis 7.0 中，压缩列表已被废弃，交由 listpack 数据结构实现。

**跳表原理：**
跳表是在链表基础上改进的多层有序链表，通过在头节点维护多层指针（Level 0 / Level 1 / Level 2...），实现类似二分查找的快速定位，查找复杂度为 O(logN)。

**Redis 为什么用跳表而不是 B+ 树？**
- 跳表本质是多层链表，实现简单（几百行C代码），修改指针即可完成插入/删除
- B+ 树实现复杂，插入/删除可能引发节点分裂合并，需要全局重平衡
- Redis 是纯内存操作，没有磁盘 I/O 开销，B+ 树为降低磁盘 I/O 次数而设计的复杂机制在内存场景下显得多余

## 题目三：什么是缓存雪崩、击穿、穿透？如何解决？

### 参考答案

这是 Redis 面试中出现频率最高的问题之一，三个概念容易混淆。

**缓存雪崩**
指缓存在同一时间大面积失效，导致后续请求全部落到数据库上，造成数据库短时承受大量请求而崩溃。

解决方案：
- Redis 高可用（主从+哨兵，Redis Cluster），避免全盘崩溃
- 本地缓存（ehcache）+ 限流降级（Hystrix），保护数据库
- 缓存过期时间设置随机值，防止同一时间大量数据过期
- 逻辑上永不过期，给缓存数据增加标记，失效则更新数据缓存

**缓存击穿**
指并发查询同一条数据，缓存中没有数据（过期或不存在），同时大量并发请求打到数据库。和雪崩的区别：雪崩是不同数据都过期，击穿是同一条数据被并发访问。

解决方案：
- 热点数据设置永不过期，由异步线程更新
- 使用互斥锁（SETNX），只有一个线程去查数据库，其他线程等待
- 缓存预热，系统上线前将可预期的热点数据直接加载到缓存

**缓存穿透**
指缓存和数据库中都没有的数据（如恶意请求不存在的ID），导致所有请求都打到数据库。

解决方案：
- 接口层增加校验（如id<=0直接拦截，用户鉴权）
- 将空数据也缓存起来，设置短TTL（如30秒），防止恶意攻击
- 使用布隆过滤器拦截不存在的数据

## 题目四：Redis持久化机制有哪些？RDB和AOF有什么区别？

### 参考答案

Redis 提供两种持久化机制：**RDB** 和 **AOF**，Redis 4.0 后采用混合持久化。

**RDB（Redis DataBase）**
RDB 是 Redis 默认的持久化方式，通过指定时间间隔将内存中的数据快照写入磁盘（dump.rdb）。

优点：
- 只有一个文件 dump.rdb，方便持久化
- 容灾性好，可保存到安全磁盘
- fork 子进程进行持久化，主进程继续处理命令，性能最大化
- 数据集大时，启动效率比 AOF 高

缺点：
- 数据安全性低，是间隔持久化，宕机会丢失最后一次快照后的数据

**AOF（Append Only File）**
AOF 通过记录每次写命令到日志文件来持久化，重启时重新执行日志中的命令恢复数据。

优点：
- 数据安全性高，可配置 appendfsync（always/everysec/no）
- 通过 append 模式写文件，即使中途宕机也可通过 redis-check-aof 工具恢复

缺点：
- AOF 文件比 RDB 大，恢复速度慢
- 数据集大时，启动效率比 RDB 低

**混合持久化（Redis 4.0+）**
- 以 AOF 为主保证数据不丢失
- 用 RDB 做不同程度的冷备
- 重启时优先加载 AOF 文件

| 对比项 | RDB | AOF |
|--------|-----|-----|
| 文件大小 | 小（快照） | 大（命令日志） |
| 恢复速度 | 快 | 慢 |
| 数据安全性 | 低（可能丢数据） | 高（可配置） |
| 性能影响 | 低（fork子进程） | 略高（每次写命令） |

## 题目五：Redis主从复制原理是什么？哨兵机制如何实现故障自动恢复？

### 参考答案

**主从复制原理**

Redis 主从复制有两种同步方式：**全量同步** 和 **增量同步**。

**全量同步**（适用于首次同步或从节点落后太多）：
1. 从节点向主节点发送 `PSYNC` 命令
2. 主节点 fork 一个子进程生成 RDB 快照
3. 主节点将 RDB 文件发送给从节点（期间新写入的命令存入缓冲区）
4. 从节点加载 RDB 文件完成全量同步
5. 主节点将缓冲区中的写命令发送给从节点

**增量同步**（适用于主从短暂断开后的同步）：
- 主节点维护一个复制偏移量（offset）和积压缓冲区（replication buffer）
- 从节点断线重连后，发送 `PSYNC {runId} {offset}`
- 主节点根据 offset 从积压缓冲区中获取差异数据发送给从节点

**哨兵机制（Sentinel）**

哨兵是 Redis 的高可用解决方案，用于监控主从节点的健康状态并自动完成故障转移。

核心功能：
- **监控**：哨兵节点定期向主从节点发送 PING，若超过阈值无响应则判定为主观下线
- **选主**：当主节点主观下线后，哨兵通过投票机制（Raft协议）判定为客观下线，然后从所有从节点中根据 **优先级 + 复制偏移量** 选出新主节点
- **通知**：故障转移后，通过发布订阅机制通知客户端新的主节点地址

一个典型的哨兵集群配置：**一主两从 + 三个 Sentinel 节点**（至少3个哨兵保证多数选举）。
