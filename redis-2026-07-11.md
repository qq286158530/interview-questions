## 题目一：Redis 有哪些常用数据结构？底层分别是如何实现的？

### 参考答案

Redis 提供了 5 种基础数据结构，以及几种高级数据结构：

**基础数据结构：**

| 数据结构 | 底层实现 | 典型场景 |
|---|---|---|
| `String` | SDS（Simple Dynamic String，简单动态字符串） | 缓存、计数器、分布式锁 |
| `List` | QuickList（压缩列表 + 双向链表） | 消息队列、排行榜 |
| `Hash` | HT（Dict）+ 压缩列表（元素少时） | 对象存储、购物车 |
| `Set` | HT（Dict）+ 整数集合（纯整数时） | 标签、好友关系、去重 |
| `Zset` | SkipList（跳跃表）+ HT | 有序排行榜、延时队列 |

**关键实现细节：**

- **SDS**：相比 C 字符串，SDS 支持 O(1) 长度获取、自动扩容（空间预分配 + 惰性释放）、二进制安全（允许包含 `\0`）。
- **QuickList**：每个节点是一个压缩列表（ziplist），节点间双向链接。压缩列表节省内存，但修改操作可能触发连锁更新。
- **跳跃表**：平均 O(log N) 查找，Redis 的 Zset 用跳跃表实现有序 + 字典实现 O(1) 按成员查分值的复合结构。
- **整数集合（IntSet）**：当 Set 全部元素为整数且数量不多时，用有序数组存储，避免哈希开销。

**高级数据结构：**

| 数据结构 | 说明 |
|---|---|
| `Bitmap` | 位图，适合签到、活跃用户统计 |
| `HyperLogLog` | 基数统计，误差约 0.81%，适合 UV 统计 |
| `GEO` | 地理位置，附近的人功能 |
| `Stream` | 消息流，支持消费组，替代 List 做消息队列 |

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别，如何选择？

### 参考答案

Redis 支持两种持久化机制：

**1. RDB（Redis Database）**

- 原理：在指定时间点将内存数据集快照写入二进制文件（`dump.rdb`）。
- 触发方式：
  - `SAVE`（同步阻塞） 或 `BGSAVE`（fork 子进程异步）
  - 配置自动触发：`save 900 1`、`save 300 10`、`save 60 10000`
  - `FLUSHALL` + `SHUTDOWN` 时也会触发
- 优点：恢复快（直接加载 RDB 文件），fork 在数据量不大时影响小。
- 缺点：可能丢失最近一次快照之后的数据。

**2. AOF（Append Only File）**

- 原理：每条写命令追加到日志文件末尾。
- 同步策略（`appendfsync`）：
  - `always`：每条命令都 fsync，最安全但最慢。
  - `everysec`（默认）：每秒执行一次 fsync，最多丢 1 秒数据。
  - `no`：由操作系统决定何时同步，最快但最不安全。
- AOF 重写（BGREWRITEAOF）：后台压缩冗余命令，生成精简文件。
- 优点：数据安全性更高（everysec 模式最多丢 1 秒）。
- 缺点：AOF 文件通常比 RDB 大，恢复速度较慢。

**3. 混合持久化（Redis 4.0+）**

- 开启 `aof-use-rdb-preamble yes` 后，AOF 重写用 RDB 格式写入开头，后续增量用 AOF 格式。
- 兼顾快速恢复和实时性，是目前生产环境的推荐方案。

**如何选择：**

| 场景 | 推荐策略 |
|---|---|
| 缓存场景，允许少量数据丢失 | 仅 RDB |
| 数据可靠性要求高 | AOF + 混合持久化 |
| 既要快恢复又要高可靠 | 混合持久化（默认推荐） |

## 题目三：Redis 的过期 key 删除策略是什么？内存淘汰策略有哪些？

### 参考答案

**一、过期 key 删除策略**

Redis 同时使用**惰性删除**和**定期删除**两种策略：

1. **惰性删除（Lazy Free）**：每次访问 key 时检查是否过期，过期则删除。对 CPU 友好，但过期 key 长时间不被访问会一直占用内存。
2. **定期删除（Active Expiration）**：每 100ms 随机抽取一批设置了过期时间的 key，删除其中过期的。若超过 25% 的 key 已过期，重复此过程。

**二、内存淘汰策略（Memory Eviction）**

当内存达到 `maxmemory` 上限时，Redis 通过以下策略淘汰 key：

| 策略 | 行为 |
|---|---|
| `noeviction` | 不淘汰，返回错误（默认） |
| `volatile-lru` | 从已设置过期时间的 key 中淘汰最近最少使用的 |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用的 |
| `volatile-lfu` | 从已设置过期时间的 key 中淘汰使用频率最低的 |
| `allkeys-lfu` | 从所有 key 中淘汰使用频率最低的 |
| `volatile-random` | 从已设置过期时间的 key 中随机淘汰 |
| `allkeys-random` | 从所有 key 中随机淘汰 |
| `volatile-ttl` | 淘汰 TTL 最短的 key（即最快过期的） |

**实际建议：**
- 生产环境推荐 `allkeys-lru` 或 `volatile-lru`，结合合理的 `maxmemory` 配置。
- 如果有缓存 + 持久数据混合场景，用 `volatile-lru` 避免误删持久数据。

## 题目四：Redis 主从复制原理是什么？哨兵和集群有什么区别？

### 参考答案

**一、主从复制原理**

Redis 复制采用 **PING-PONG 机制**（命令流）：

1. **全量同步**（PSYNC2）：
   - 主节点 fork 子进程生成 RDB 快照，同时将期间的写命令写入 **复制缓冲区（replication buffer）**。
   - 将 RDB 文件发送给从节点，从节点清空本地数据并加载 RDB。
   - 主节点将复制缓冲区中的增量命令发给从节点，完成同步。
2. **增量同步（命令传播）**：
   - 主节点每执行一条写命令，实时将命令流发送给从节点。
   - 从节点通过 **replication feed the slaves** 机制持续接收并执行。
3. **断线重连**：从节点携带自己的 `master_replid` 和 `offset` 尝试增量同步，若主节点 repl_backlog 缓冲区仍包含所需数据则增量同步，否则全量。

**二、哨兵（Sentinel）vs 集群（Cluster）**

| 对比项 | 哨兵模式 | 集群模式 |
|---|---|---|
| 数据分片 | 无，所有节点存相同数据 | 有，数据按 slot 分片（16384 个槽） |
| 节点角色 | 主节点 + 从节点 + 哨兵 | 每个分片一主多从 |
| 故障切换 | 哨兵自动选举新主节点 | 各分片独立选举 |
| 扩展性 | 垂直扩展（加从节点） | 水平扩展（加节点分片） |
| 事务支持 | 支持（单分片） | 仅支持跨槽多 key 操作 |
| 典型场景 | 少量数据，高可用 | 大数据量，高可用 + 高性能 |
| 配置复杂度 | 较低 | 较高 |

**总结**：数据量小（< 10GB）且主要需求是**高可用**，用哨兵模式；数据量大或需要**水平扩展**，用集群模式。

## 题目五：什么是缓存穿透、缓存击穿、缓存雪崩？如何应对？

### 参考答案

这是面试中最常考的高并发场景题，三者针对的问题不同：

**一、缓存穿透**

- **问题**：查询一个**不存在**的数据，缓存没有，数据库也没有，每次请求都打到数据库。
- **应对方案**：
  - **布隆过滤器（BloomFilter）**：将所有合法 key 存入布隆过滤器，查询前先过滤。
  - **缓存空值**：对不存在的 key 也缓存一个空值（如 `"NULL"`），设置较短 TTL。
  - **参数校验**：做好接口参数合法性检查。

**二、缓存击穿**

- **问题**：某个**热点 key 过期**的瞬间，大量并发请求直接打到数据库。
- **应对方案**：
  - **互斥锁**：缓存未命中时，只允许一个请求去查数据库并回填缓存（如 SETNX）。
  - **永不过期**：不给热点 key 设置过期时间，或后台异步更新。
  - **逻辑过期**：存储时设置 `value = {data, expireTime}`，逻辑过期则异步重建缓存。

**三、缓存雪崩**

- **问题**：大量 key **同时过期**，导致大量请求同时击穿缓存打到数据库。
- **应对方案**：
  - **过期时间随机化**：在 TTL 基础上加随机秒数：`TTL = base + rand(0, 3600)`。
  - **多级缓存**：本地缓存 + Redis + 数据库。
  - **Redis 高可用**：主从 + 哨兵/集群，避免 Redis 全量宕机。
  - **服务降级 / 限流**：数据库扛不住时，返回兜底数据或友好提示。

**总结对比：**

| 场景 | 核心问题 | 关键词 |
|---|---|---|
| 缓存穿透 | 查询不存在的数据 | 布隆过滤器、空值缓存 |
| 缓存击穿 | 热点 key 过期瞬间 | 互斥锁、永不过期 |
| 缓存雪崩 | 大量 key 同时过期 | TTL 随机、多级缓存、高可用 |
