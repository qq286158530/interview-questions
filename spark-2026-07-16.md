## 题目一：RDD、DataFrame、Dataset 三者的区别与联系？

**参考答案：**

| 维度 | RDD | DataFrame | Dataset |
|------|-----|-----------|---------|
| **数据结构** | 原始 JVM 对象，无结构 | Row 对象（结构化表格） | 强类型 JVM 对象 |
| **编译时类型安全** | ✅ 有 | ❌ 无（仅运行时检查） | ✅ 有（泛型） |
| **语法提示** | 差，需-know 数据结构 | 好，列名提示 | 好，属性提示 |
| **性能** | 低（对象序列化开销大） | 高（Spark SQL 优化器 Tungsten） | 高（同 DataFrame） |
| **操作方式** | map/filter/reduce... | SQL / DSL | Lambda + typed API |
| **数据处理** | 底层灵活，控制力强 | 上层抽象，优化好 | 兼顾类型安全与优化 |

**核心联系：**

- **DataFrame = Dataset[Row]**：DataFrame 本质上是 `Dataset<Row>` 的别名，消除了类型参数。
- **三者底层都是 RDD**：无论使用哪种抽象，Spark 最终都会将代码转换为 RDD 上的操作，再交给 DAGScheduler 调度执行。
- **统一 API**：Spark 2.x 之后，通过 SparkSession 提供统一入口，三者可相互转换。

```scala
// 互相转换示例
val rdd: RDD[Person] = spark.sparkContext.parallelize(...)
val df: DataFrame = rdd.toDF()           // RDD -> DataFrame
val ds: Dataset[Person] = df.as[Person]  // DataFrame -> Dataset
val rdd2: RDD[Row] = df.rdd              // DataFrame -> RDD
```

**典型选择建议：**
- 需要编译期类型安全 → Dataset
- 需要 SQL 查询和强大优化 → DataFrame
- 需要底层灵活性（如非结构化数据） → RDD

## 题目二：Spark Shuffle 机制是什么？为什么 Shuffle 是性能瓶颈？

**参考答案：**

### 什么是 Shuffle？

Shuffle 是 Spark 中**跨分区数据重分布**的过程。当一个 RDD 依赖需要**重新分区**（如 `groupByKey`、`reduceByKey`、`join`、`sortByKey` 等）时，Spark 必须将具有相同 key 的数据汇总到同一个分区，这一跨节点传输数据的过程称为 Shuffle。

### Shuffle 的工作流程

```
[Map Stage]  →  每个 Mapper 将数据写入本地磁盘文件（同时生成索引文件）
     ↓
[Shuffle Write]  数据按 partition 分区写入本地磁盘
     ↓
[Shuffle Fetch]  每个 Reducer 通过网络抓取（fetch）所需分区数据
     ↓
[Reduce Stage]  Reducer 对抓取来的数据在内存/磁盘进行聚合
```

### 为什么是性能瓶颈？

1. **磁盘 I/O**：Shuffle 数据先落盘再网络传输，相比内存操作慢数十倍。
2. **网络 I/O**：跨节点数据传输，依赖网络带宽和延迟。
3. **内存压力**：Shuffle 期间数据可能落盘（spark.shuffle.spill = true），导致频繁 GC。
4. **启动开销**：每个 Shuffle 任务都需要启动新的线程/进程，有固定 overhead。
5. **相关性聚合限制**：Shuffle 后同一 key 的数据由单一 Reducer 处理，容易出现数据倾斜（Data Skew）。

### 优化建议

- **使用 map-side 预聚合**：用 `reduceByKey`/`aggregateByKey` 替代 `groupByKey`，在 map 端先做局部聚合，减少网络传输量。
- **合理设置分区数**：分区过多 → 小任务 overhead 大；分区过少 → 并行度低。一般每个 CPU 核分配 2~3 个分区。
- **广播小表**：对于 Join 操作，若较小表可放入内存，优先使用广播（Broadcast Join）避免 Shuffle。
- **开启 Shuffle 合并**：`spark.shuffle.manager = sort`（Spark 2.x+ 默认），利用 Shuffle Consolidation 合并文件减少小文件。

## 题目三：广播变量（Broadcast Variable）的作用与原理是什么？

**参考答案：**

### 作用

广播变量允许将**只读的大变量**高效地分发到集群每个节点，**避免在每个任务中重复传输**。通常用于：
- 大容量查找表（如字典、配置表）
- 小表广播 Join（将小表广播到所有节点，避免 Shuffle）
- 共享配置数据

### 原理

```
Driver 端：
  broadcast.collect() → 将数据序列化 → 按 BitTorrent 风格分发到各 Executor 节点

Executor 端：
  首次使用时从 Driver/其他 Executor 拉取 → 存入 BlockManager（内存+磁盘）
  后续任务复用本地缓存，不再重复传输
```

### 使用方式

```scala
// 定义广播变量
val broadcastVar = spark.sparkContext.broadcast(Map(
  "BJ" -> "北京", "SH" -> "上海", "GZ" -> "广州"
))

// 在算子中使用（自动解密）
def translate(cityCode: String): String = {
  broadcastVar.value.getOrElse(cityCode, "未知")
}

// 销毁广播变量（显式释放）
broadcastVar.unpersist()
```

### 注意事项

| 注意点 | 说明 |
|--------|------|
| **只读不可修改** | 广播变量一旦创建，在 Executor 端只能读，不能修改。若需更新须重新广播 |
| **大小控制** | 广播对象太大会导致 Executor OOM，建议 < 20MB，可调 `spark.broadcast.blockSize` |
| **避免闭包捕获** | 普通变量会随任务序列化复制，广播变量独立管理，不重复传输 |
| **自动释放** | Task 完成后默认保留在内存，可调用 `unpersist()` 手动删除 |

## 题目四：Spark 的内存管理模型是怎样的？堆内与堆外内存如何划分？

**参考答案：**

Spark 2.x 采用**统一内存管理（Unified Memory Management）**，将堆内和堆外内存统一考虑。

### 内存区域划分

```
┌──────────────────────────────────────────────────────┐
│                    堆内内存 (On-Heap)                  │
│  ┌──────────────────┐  ┌──────────────────────────┐  │
│  │   Execution      │  │   Storage                │  │
│  │   执 · 行 · 区   │  │   存 · 储 · 区            │  │
│  │   (shuffle等)    │  │   (cache/Broadcast)      │  │
│  └──────────────────┘  └──────────────────────────┘  │
│  ┌──────────────────┐  ┌──────────────────────────┐  │
│  │   User Memory    │  │   Reserved Memory        │  │
│  │   (用户代码)      │  │   (300MB 系统预留)        │  │
│  └──────────────────┘  └──────────────────────────┘  │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│                   堆外内存 (Off-Heap)                  │
│  主要用于：NIO direct memory (Tungsten 序列化)          │
└──────────────────────────────────────────────────────┘
```

### 各区域作用

| 区域 | 用途 | 配置参数 |
|------|------|----------|
| **Execution 内存** | Shuffle sort、hash join、agg 聚合等算子执行时的临时内存 | `spark.memory.fraction`（默认 0.6） |
| **Storage 内存** | RDD Cache、广播变量、Broadcast Join 等缓存数据 | 与 Execution 共用，可动态借用 |
| **User Memory** | 用户数据结构、自定义类、元数据等 | `1 - spark.memory.fraction` |
| **Reserved Memory** | 固定 300MB，系统预留 | 不可配置 |

### 动态内存共享机制

- Execution 和 Storage 之间**可以互相借用**空闲内存（通过 `spark.memory.storageFraction` 控制借出下限）。
- Storage 内存被借出后，被驱逐的数据会**降级到磁盘**（如果开启了 `MEMORY_AND_DISK`）。

### 堆外内存（Off-Heap）

- 通过 `--conf spark.memory.offHeap.enabled=true` 开启
- 用于 Tungsten 序列化（`spark.serializer=org.apache.spark.serializer.KryoSerializer`）
- 减少 GC 开销，适合大内存场景

### 常见内存问题排查

```bash
# 查看内存使用情况
spark.sparkContext.uiWebUrl  # 访问 UI 查看 Storage 页面

# 常见报错：
# - ExecutorLostFailure (OOM): 减小广播变量大小或关闭堆外内存
# - Shuffle FetchFailedException: Execution 内存不足，增加 spark.shuffle.memoryFraction
```

## 题目五：Spark DAG 是如何生成的？DAG、Stage、Task 的关系是什么？

**参考答案：**

### DAG 生成过程

```
用户代码 (RDD 算子链)
     ↓
SparkContext 提交 Job
     ↓
RDD DAG Builder 构建 DAG（有向无环图）
     ↓
DAGScheduler 将 DAG 划分为 Stage（按 Shuffle 边界）
     ↓
TaskScheduler 将 Stage 包装为 TaskSet，分发到 Executor
     ↓
Executor 执行 Task（ShuffleMapTask / ResultTask）
```

### 核心概念

**DAG（Directed Acyclic Graph）**
- RDD 的依赖关系图，包含**窄依赖**（Narrow Dependency）和**宽依赖**（Shuffle Dependency / Wide Dependency）。
- 描述了数据从源头到最终结果的完整转换路径。

**窄依赖 vs 宽依赖**

| 依赖类型 | 特点 | 例子 | 意义 |
|----------|------|------|------|
| **窄依赖** | 父分区 ≤ 子分区，一对一或一对多 | map, filter | 可在同一个 Stage 内流水线执行 |
| **宽依赖** | 一个父分区对应多个子分区（Shuffle） | groupByKey, join | 是划分 Stage 的边界 |

**Stage**
- 以 **Shuffle 宽依赖**为边界，将 DAG 切分成多个 Stage。
- Stage 类型：`ShuffleMapStage`（产生 Shuffle 数据）→ `ResultStage`（最终结果）。
- Stage 内部所有 Task 可以并行执行。

**Task**
- Stage 内每个分区对应一个 Task。
- `ShuffleMapTask`：负责读取数据、执行算子、输出 Shuffle 数据。
- `ResultTask`：负责执行最后一个 Stage，输出最终结果给 Driver。

### DAG 调度流程

```
Job (Action 触发)
  └── DAGScheduler.run()
        └── 构建 DAG（从 ResultStage 反向回溯）
        └── 按拓扑顺序从右向左划分 Stage
        └── 为每个 Stage 创建 TaskSet
              └── TaskScheduler.submitTasks()
                    └── TaskSet 在 Executor 上并行执行
```

### 为什么 DAG 很重要？

1. **优化执行计划**：窄依赖可流水线执行，避免不必要的数据物化。
2. **容错机制**：某个 Partition 丢失，只需根据 DAG 从父 RDD 重新计算对应路径，而无需从头重算。
3. **资源调度依据**：Stage 划分决定了并行度（Task 数量）和数据本地性。

```scala
// 查看 DAG（通过算子依赖关系推演）
// map + filter 是窄依赖 → 同一 Stage
// groupByKey 是宽依赖 → 触发 Stage 切分
val rdd = sc.textFile("hdfs://...")
  .map(_.split(","))
  .filter(_.length > 3)
  .groupBy(_(0))
  .mapValues(_.toList)
```
