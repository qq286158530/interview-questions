## 题目一：Redis 是单线程还是多线程？为什么单线程还能这么快？

### 参考答案

Redis 在 **4.0 版本之前**采用的是典型的单线程模型，这里的"单线程"指的是 **网络 I/O 线程（连接处理）和读写命令（Set、Get）是由一个线程完成的**。但这不意味着 Redis 整个服务只有一个线程——持久化（AOF 刷盘、RDB 生成）、主从复制等操作是由其他线程执行的。

**4.0 版本之后**，Redis 引入了多线程支持，主要体现在大数据的**异步删除**操作上，如 `unlink key`、`flushdb async`、`flushall async` 等。

Redis 单线程还能非常快的原因主要有三点：

1. **纯内存操作**：数据存放在内存中，读写不需要磁盘 I/O，速度极快。
2. **非阻塞 I/O + 事件驱动**：Redis 使用 `epoll`（Linux）/ `kqueue`（macOS）等 I/O 多路复用机制，单线程可以高效处理大量并发连接。
3. **避免了锁竞争和上下文切换开销**：单线程模型不存在死锁和线程切换的性能损耗，且数据结构内部实现高效（如 SDS 字符串、跳表、有序集合等）。

## 题目二：Redis 持久化机制有哪几种？RDB 和 AOF 各自的原理是什么？如何选择？

### 参考答案

Redis 提供了三种持久化机制：

#### 1. RDB（Redis DataBase）—— 内存快照

- **原理**：在指定的时间间隔内，将 Redis 内存中的**数据快照**以二进制格式写入磁盘，生成 `dump.rdb` 文件。
- **触发方式**：
  - `save`（同步，会阻塞主线程，不推荐）
  - `bgsave`（异步 fork 子进程，推荐）
  - 自动触发（根据 `save` 配置规则）
- **优点**：恢复速度快（直接读取二进制），对主线程影响小。
- **缺点**：可能丢失最后一次快照之后的数据。

#### 2. AOF（Append Only File）—— 命令日志

- **原理**：记录所有**写操作命令**到日志文件（默认 `appendonly.aof`），Redis 重启时通过重放日志来恢复数据。
- **三种同步策略**：
  - `appendfsync always`：每次写命令都同步，安全性最高但性能差
  - `appendfsync everysec`（默认）：每秒同步一次，折中方案
  - `appendfsync no`：由操作系统决定何时同步，性能最好但可能丢失较多数据
- **优点**：数据安全性更高（尤其 `always` 模式下）。
- **缺点**：日志文件体积可能较大，恢复速度慢于 RDB。

#### 3. 混合持久化（Redis 4.0+）

同时使用 RDB 和 AOF，取两者之长：数据以 RDB 格式写入文件，后续操作以 AOF 格式追加。兼顾快速恢复和较低数据丢失风险。

**如何选择**：
- 若业务对数据完整性要求极高（如金融类），推荐 **AOF + 混合持久化**。
- 若追求高性能且能容忍少量数据丢失，可仅用 **RDB**。
- 生产环境推荐 **RDB + AOF 同时开启**。

## 题目三：什么是缓存穿透、击穿和雪崩？如何解决？

### 参考答案

这三个问题都是高并发场景下缓存失效导致的数据库压力异常，区别在于失效的原因和范围不同。

#### 🔴 缓存穿透

**定义**：查询的数据**既不在缓存中，也不在数据库中**（数据压根不存在）。大量恶意请求直接打到数据库。

**常见解决方案**：
1. **缓存空值/缺省值**：对不存在的数据也在 Redis 中缓存一个默认值（如 `NULL`），防止重复查库。
2. **布隆过滤器（Bloom Filter）**：在缓存层前加一层布隆过滤器，快速判断数据是否存在，将"不存在"的请求直接拦截在数据库之外。
3. **请求合法性检测**：在前端或网关层对请求参数做校验，过滤掉恶意/非法请求。

#### 🟡 缓存击穿

**定义**：某个**访问非常频繁的热点数据**，缓存中某一时刻突然过期，大量并发请求全部打到数据库。

**常见解决方案**：
1. **热点数据不过期**：对极度热点数据设置较长的过期时间或永不过期，由业务逻辑主动更新。
2. **分布式锁**：缓存失效时，只允许一个请求去查库并更新缓存，其他请求等待或轮询。可用 `SET key value EX seconds NX`（SETNX）实现。
3. **使用 Redisson**：利用其"看门狗"机制自动续期锁的 TTL，解决锁提前释放的问题。

#### 🔵 缓存雪崩

**定义**：大量缓存的 **key 在同一时间集中过期**，导致大量并发同时打到数据库。

**常见解决方案**：
1. **过期时间随机化**：给不同 key 的 TTL 增加一个较小的随机数（如 `TTL + random(60~180秒)`），避免同时失效。
2. **服务熔断/限流**：当数据库压力过大时，在入口层限流（如每秒只允许 1000 个请求通过），保护数据库。
3. **热点数据分散失效时间**：从业务层面在更新缓存时人为制造时间差，减少并发叠加。
4. **预热**：在缓存即将过期前，主动提前加载数据到缓存。

## 题目四：Redis Cluster 是如何将数据分配到不同节点的？它和哨兵模式有什么区别？

### 参考答案

#### Redis Cluster 槽位分片机制

Redis Cluster 将整个数据库分为了 **16384（2^14）个 Slot（槽位）**，采用**类一致性哈希**的算法来分配数据：

1. 对每个键的 key 使用 `CRC16` 算法计算出一个 **16 bit 的值**。
2. 用这个值对 **16384 取模**，得到 0~16383 之间的模数，每个模数对应一个哈希槽编号。
3. 每个主节点（Master）负责一部分槽位。例如三节点集群：
   - 节点 A：0 ~ 5000
   - 节点 B：5001 ~ 10000
   - 节点 C：10001 ~ 16383

当客户端访问某个 key 时，会根据上述算法**直接定位到对应的节点**，无需经过代理层。

#### 哨兵模式 vs 集群模式

| 维度 | 哨兵模式 | 集群模式 |
|------|---------|---------|
| **架构** | 主从复制 + 哨兵监控 | 多主多从 + 槽位分片 |
| **扩展能力** | 只扩展从节点读能力，无法扩展写能力和存储能力 | 同时扩展读写能力和存储能力 |
| **故障恢复** | 哨兵自动选举新 master | 自动槽位重新分配和故障转移 |
| **适用场景** | 读多写少、对数据完整性要求高的场景 | 写并发高、数据量大的场景 |

**为什么有了哨兵还需要集群？**  
哨兵模式本质还是主从架构，写能力和存储能力受限于 master 节点的天花板，无法水平扩展。集群模式通过数据分片，让每个 master 节点只负责一部分数据，从而突破了单节点的瓶颈。

## 题目五：Redis 支持哪些数据类型？ZSet 的底层为什么要用跳表而不是 B+ 树？

### 参考答案

#### Redis 五种核心数据类型

| 数据类型 | 结构 | 典型应用场景 |
|---------|------|------------|
| **String** | 字符串（SDS） | 缓存、计数器、分布式锁（SETNX） |
| **List** | 双向链表 / 压缩列表 | 消息队列、排行榜、Timeline |
| **Hash** | 哈希表 / 压缩列表 | 存储对象、购物车、用户信息 |
| **Set** | 整数集合 / 哈希表 | 标签系统、去重、共同好友 |
| **ZSet**（Sorted Set） | 跳表 + 压缩列表 | 排行榜、带权重的消息队列 |

#### ZSet 为什么用跳表而不是 B+ 树？

这是 Redis 经典高频面试题。核心原因在于**数据结构特性与 Redis 读写场景的天然匹配**：

1. **读操作（范围查询）性能**：跳表的范围查询只需找到起始元素，然后沿指针顺序遍历即可，时间复杂度 O(log n)。B+ 树的范围查询虽然也只需要遍历叶子节点链表，但单次跳表的指针跳转开销更小。

2. **写操作（插入/删除）**：跳表的插入和删除只需要修改局部指针，时间复杂度 O(log n)，实现简单。**B+ 树的插入/删除涉及节点的分裂和合并，开销更大，实现复杂。**

3. **内存开销**：跳表每个节点的层数是随机生成的（服从泊松分布），平均每个节点约 1.33 个指针，空间效率高。B+ 树每个节点都有完整的指针域和冗余数据，内存开销更大。

4. **实现简洁**：跳表实现远比 B+ 树简单，Redis 可完全用 C 语言手写，而 B+ 树需要复杂的磁盘调度逻辑。
