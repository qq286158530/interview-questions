### 题目1：Spark 的架构是怎样的？为什么比 Hadoop MapReduce 快？

【答案】

**Spark 架构**：
- **Driver**：运行用户代码，创建 SparkContext，负责 DAG 切分、任务调度
- **Executor**：运行在 Worker 节点上的进程，执行 Task，向 Driver 汇报结果
- **Cluster Manager**：集群管理器（Standalone / YARN / Mesos），负责资源分配
- **SparkContext**：应用入口，连接集群，协调分布式操作

**核心组件**：
```
Driver
  └─ SparkContext
       ├─ DAGScheduler（划分 Stage）
       └─ TaskScheduler（提交 Task 到 Executor）
```

**为什么比 MapReduce 快？**

| 对比点 | Hadoop MapReduce | Spark |
|--------|-----------------|-------|
| 编程模型 | 批量离线计算 | 统一的批/流/交互计算 |
| 内存使用 | 每步写磁盘（HDFS） | 基于 DAG 内存计算 |
| 任务启动 | 每次 Shuffle 都要启动新进程 | Executor 常驻，复用 JVM |
| Shuffle | 大量磁盘 I/O | 内存+网络，但可配置 spills |
| 中间结果 | 每步落盘 | 内存流水线（pipelining） |
| 数据格式 | 原始字节流 | Tungsten 内存二进制格式 |

**关键原因**：Spark 使用 DAG 执行引擎，将多个 Stage 串联成一张图，只需一次 shuffle 落地，而 MR 每步都必须写磁盘，且每次 shuffle 都是独立的 MapReduce 任务。

📖 来源：https://spark.apache.org/docs/latest/architecture.html

### 题目2：RDD 是什么？有哪些特性？RDD 和 DataFrame/DataSet 的区别？

【答案】

**RDD（Resilient Distributed Dataset）**：
弹性分布式数据集，是 Spark 最底层的抽象，不可变、可分区、并行运行。

**五大特性**：
1. **`partitions`**：数据分区的列表，每个分区对应一个 Task
2. **`computeFunction`**：计算函数（plit → Iterator），作用于每个分区
3. **`dependencies`**：RDD 依赖关系（宽依赖/窄依赖），决定 DAG 划分 Stage
4. **`partitioner`**：分区器（仅 KV 型 RDD），决定 Shuffle 时数据如何分区
5. **`preferredLocations`**：数据本地性偏好，让 Task 尽量在数据所在节点执行

**宽依赖 vs 窄依赖**：
- **窄依赖**：父 RDD 每个分区最多被子 RDD 一个分区使用（map、filter）→ 同一节点内流水线计算
- **宽依赖**：父 RDD 每个分区被子 RDD 多个分区使用（groupByKey、join）→ 必须 Shuffle，划分 Stage 边界

**RDD vs DataFrame vs DataSet**：

| 特性 | RDD | DataFrame | DataSet |
|------|-----|-----------|---------|
| 数据结构 | 无结构（原始类型） | 有结构（Row 对象） | 有结构（强类型 JVM 对象） |
| 编译时检查 | 无 | 无（运行才发现错误） | 有（编译时检查） |
| 运行时安全 | 差 | 中等 | 好 |
| 语法糖 | 低（函数式） | 高（DSL 类似 SQL） | 高（面向对象） |
| 性能优化 | 无自动优化 | 有 Catalyst 优化器 | 有 Catalyst + Tungsten |
| 序列化 | Java/Kryo | Tungsten（off-heap） | Tungsten（off-heap） |
| 适用场景 | 非结构化/低级控制 | 结构化数据/SQL 查询 | 结构化数据/类型安全 |

DataFrame = DataSet[Row]，在 Spark 2.x 之后统一，Spark 3.0 进一步统一了 API。

📖 来源：https://spark.apache.org/docs/latest/rdd-programming-guide.html

### 题目3：Spark Shuffle 是什么？为什么 Shuffle 是性能瓶颈？如何优化？

【答案】

**Shuffle 定义**：
将数据按照 Key 进行重新分布的过程，类似于 MapReduce 的 partition-sort 阶段。当一个 RDD 的分区数据需要根据 Key 重新聚合/分组到不同节点时，就必须 Shuffle。

**触发 Shuffle 的算子**：
`groupByKey`、`reduceByKey`、`join`、`cogroup`、`distinct`、`repartition`、`sortByKey` 等

**Shuffle 流程（Spark 2.x Sort-based Shuffle）**：

```
Map 端（Shuffle Write）：
  1. 数据写入内存缓冲区（buffer spill）
  2. 按 partitionId 排序
  3. 溢写磁盘（spill to disk）
  4. 每个分区生成一个 data 文件 + index 文件

Reduce 端（Shuffle Read）：
  1. 先读取 index 文件获知各分区位置
  2. 通过网络抓取（fetch）对应分区的数据
  3. 按 Key 排序或聚合
```

**为什么是瓶颈？**
- **大量磁盘 I/O**：Shuffle 数据先落盘再网络传输
- **网络 I/O**：跨节点数据复制，带宽竞争
- **内存压力**：Shuffle 数据可能撑爆内存，触发 OOM
- **磁盘空间**：大量 spill 文件占用磁盘
- **启动开销**：每个 Stage 边界都需要重新启动 Task

**优化策略**：

| 优化方向 | 具体方法 |
|---------|---------|
| 减少 Shuffle 数据量 | 1. 使用 map-side 预聚合（`reduceByKey` 优于 `groupByKey`）<br>2. 过滤无用数据<br>3. 减少 join 前的数据膨胀 |
| 增加并行度 | 1. `repartition(num)` 增加分区数<br>2. `coalesce(num, shuffle=true)` 减少分区<br>3. 配置 `spark.sql.shuffle.partitions`（默认 200） |
| 内存优化 | 1. 增大 `spark.shuffle.memoryFraction`（默认 20%）<br>2. 使用 `spark.shuffle.service.enabled`（External Shuffle Service）<br>3. 开启 `spark.shuffle.compress` 压缩 |
| 存储优化 | 1. 使用 `spark.shuffle.manager` = sort（默认）<br>2. 尝试 `tungsten-sort`<br>3. 避免频繁小文件 Shuffle |
| 代码层面 | 1. 广播小表（`broadcast` join）避免 Shuffle<br>2. 避免 `groupByKey` 后再 `reduceByKey`（连续两次 Shuffle） |

📖 来源：https://spark.apache.org/docs/latest/tuning.html

### 题目4：Spark 的内存管理机制是怎样的？堆内 vs 堆外内存？

【答案】

**统一内存管理（Spark 1.6+，默认）**：

Spark 将内存划分为两大区域，**执行内存（Execution）** 和 **存储内存（Storage）** 共享同一个区域，可以动态占用对方空闲空间。

**内存划分**：

```
总内存 = 系统预留（300MB）+ 用户内存（1 - spark.memory.fraction ≈ 40%）
                              ↓
           执行内存 + 存储内存（≈ 60%，由 spark.memory.storageFraction 分配）
```

| 区域 | 用途 | 说明 |
|------|------|------|
| **系统预留** | JVM 自身运行（代码缓存、线程等） | 约 300MB，不可干涉 |
| **用户内存** | 用户数据结构、元数据、RDD 广播变量 | 默认 40% |
| **统一内存** | 执行（Shuffle sort buffer、hash table）<br>存储（cache RDD block、broadcast） | 默认 60% |

**堆内 vs 堆外内存**：

| 特性 | 堆内内存（On-heap） | 堆外内存（Off-heap） |
|------|-------------------|---------------------|
| 管理方式 | JVM GC 管理 | 直接分配/释放（Unsafe API） |
| 序列化 | Java 对象序列化 | 直接二进制，不经过 JVM |
| 优点 | JVM 生态成熟 | 避免 GC 开销，精确控制，适合大数据 |
| 缺点 | GC 停顿，大对象易触发 Full GC | 调试困难，不受 JVM 保护 |
| 配置 | `spark.memory.offHeap.enabled=false` | `spark.memory.offHeap.enabled=true` |

**存储内存淘汰策略**：
1. LRU（最近最少使用）：存储内存不足时，淘汰最近未使用的 RDD block
2. `spark.memory.storageFraction` 控制存储内存占比（默认 50%）
3. 被执行内存占用的存储空间，淘汰后无法恢复，只能等执行内存释放

**常见 OOM 原因**：
- Shuffle 数据量 > 统一内存大小
- 大量小文件导致 block 元数据过多（存储在用户内存区）
- 广播变量过大，撑爆存储内存

📖 来源：https://spark.apache.org/docs/latest/tuning.html#memory-tuning

### 题目5：广播变量（Broadcast Variable）是什么？它解决了什么问题？如何使用？

【答案】

**广播变量的定义**：
广播变量是只读的、缓存在每个节点 Executor 内存中的共享变量。使用广播变量后，每个 Executor 只会在每个批次任务开始前拉取一次数据，而不是每个 Task 都拉取一次。

**解决的问题**：
- **网络传输优化**：假设 1000 个分区，每个 Task 都需要一份相同的小表（100KB），原始方式需要传输 1000 × 100KB = 100MB；使用广播后只需要 1 × 100KB = 100KB
- **降低 Driver 压力**：避免重复发送相同数据给所有 Task
- **减少 GC 开销**：数据在 Executor 内共享，避免重复创建对象

**适用场景**：
- 大表 join 小表（小表广播）
- 维度表缓存
- 配置参数、查找表等只读数据

**使用方法**：

```scala
// 定义广播变量
val broadcastVar = spark.sparkContext.broadcast(Map(
  "key1" -> "value1",
  "key2" -> "value2"
))

// 在算子内通过 .value 访问
val result = data.map(row => {
  val lookupTable = broadcastVar.value
  lookupTable.getOrElse(row.key, "default")
})

// 用完后销毁（可选）
broadcastVar.unpersist()
```

```python
# PySpark
from pyspark.sql.functions import broadcast

# broadcast join（小表广播）
result = large_df.join(broadcast(small_df), "key")
```

**注意事项**：
- 广播变量只读，修改无效（Spark 不支持修改广播变量）
- 广播的变量不宜过大（超过 20MB 的表建议评估必要性，否则可能 OOM）
- 广播变量缓存在 Executor 的堆内/堆外内存中
- 可以调用 `unpersist()` 手动删除缓存，调用 `destroy()` 完全销毁
- Spark SQL 中 `broadcast hint`（`/*+ BROADCAST(t) */`）等价于广播变量

**广播变量的执行流程**：
1. Driver 端对广播变量进行分片（默认 4MB 一片）
2. Executor 通过 BlockManager 获取广播分片，存储在本地
3. 每个 Task 运行时直接从 Executor 本地获取，不需要 Driver 再发送

📖 来源：https://spark.apache.org/docs/latest/rdd-programming-guide.html#broadcast-variables
