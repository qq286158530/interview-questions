## 题目一：HBase 的整体架构是怎样的？请详细说明各组件的作用

### 参考答案

HBase 是一个基于 HDFS 的列式存储数据库，采用 **Master-Slave 架构**，主要包含以下核心组件：

#### 1. HMaster（主节点）
- 负责表的创建、删除、修改等元数据操作
- 管理 Region 的分配和调度，决定将哪个 Region 分配给哪个 Region Server
- 监控集群中所有 Region Server 的状态，实现负载均衡和故障转移
- 在集群中有多个备份，通过 Zookeeper 进行协调

#### 2. Region Server（数据节点）
- 负责处理数据的读写请求，是真正存储数据的地方
- 每个 Region Server 可以管理 1000+ 个 Region
- 包含以下关键组件：
  - **MemStore**：写缓存，先写入内存，满了之后 flush 成 HFile
  - **HFile**：HBase 的实际存储文件格式
  - **Write Ahead Log (WAL)**：预写日志，保证数据可靠性

#### 3. Zookeeper（协调服务）
- 维护集群中所有服务器的心跳和状态
- 帮助 HMaster 选举主节点
- 存储 HBase 的元数据信息（如 .META. 表位置）
- 实现分布式协调功能

#### 4. 数据寻址流程
```
Client → Zookeeper → .META.表 → Region Server → MemStore/HFile
```

## 题目二：HBase 的 RowKey 设计应该遵循哪些原则？为什么？

### 参考答案

RowKey 是 HBase 中最重要的设计要素，直接影响读写性能和负载均衡。设计原则如下：

#### 1. 长度原则
- RowKey 应尽可能短，建议不超过 16 个字节
- 原因：RowKey 会在内存和磁盘中反复存储，过长的 RowKey 会浪费存储空间和内存
- 短 RowKey 能提升 MemStore 写入和 HFile 读取的效率

#### 2. 唯一性原则
- RowKey 必须唯一标识一行数据
- 类似于传统数据库的主键

#### 3. 散列性原则（关键！）
- RowKey 设计要避免数据热点问题
- **不要使用单调递增的 RowKey**（如时间戳），否则所有数据会写入同一个 Region
- 常用方法：
  - 哈希打散：`MD5(rowKey) + 固定前缀`
  - 盐值（Salting）：在 RowKey 前加随机前缀
  - 反转：反转时间戳字符串

#### 4. 有序性原则
- HBase 按照 RowKey 的字典序排序存储
- 可以利用这一特性，将经常一起查询的数据放在相近位置
- 常见模式：`[业务前缀]_[时间戳]_[用户ID]`

#### 5. 查询模式匹配
- RowKey 设计要与业务查询模式匹配
- 支持三种scan方式：get（单行）、scan（范围）、filter（过滤）

## 题目三：请描述 HBase 的存储结构，包括 MemStore 和 HFile 的工作原理

### 参考答案

HBase 采用 LSM-Tree（Log-Structured Merge-Tree）存储架构，核心是 MemStore 和 HFile 的配合。

#### 1. MemStore（写缓存）
- **位置**：Region Server 的内存中
- **作用**：接收所有写请求，先写入 MemStore，满了之后 flush 成 HFile
- **特点**：
  - 按 RowKey 和 Column Family 排序
  - 同一个 Column Family 的所有列存在一起
  - 写性能高，因为是顺序写入内存
- **Flush 触发条件**：
  - MemStore 大小达到阈值（默认 128MB）
  - HLog（WAL）数量过多
  - Region 完成分裂

#### 2. HFile（存储文件）
- **位置**：HDFS 上的实际存储文件
- **结构**：
  - **Data Block**：存储 KeyValue 数据对，默认 64KB
  - **Meta Block**：存储布隆过滤器信息
  - **File Info**：文件元信息
  - **Index Block**：数据块索引，加速随机读取
- **特点**：
  - 写入后不可修改（Append-only）
  - 通过 Compaction 合并小文件

#### 3. 写入流程
```
1. Client 发起 PUT 请求
2. 数据先写入 WAL（HLog）
3. 数据写入 MemStore
4. MemStore 满了，触发 Flush，生成 HFile
5. HFile 落盘，完成写入
```

#### 4. Compaction（合并压缩）
- **Minor Compaction**：合并小 HFile，不删除过期数据
- **Major Compaction**：合并所有 HFile，删除过期和删除的数据
- 默认 Major Compaction 会在夜间自动执行

## 题目四：什么是 HBase 的协处理器（Coprocessor）？它有哪几种类型？各自的使用场景是什么？

### 参考答案

HBase 协处理器是一种扩展机制，允许用户在 Region Server 上运行自定义代码，实现服务器端数据处理。

#### 1. 协处理器的两种类型

##### (1) Observer Coprocessor（观察者）
- **类似数据库触发器**
- 在特定事件前后自动执行回调
- 三种子类型：
  - **PreGet**：读操作之前执行，可用于列级权限控制
  - **PostGet**：读操作之后执行，可用于动态列过滤
  - **PrePut/PostPut**：写操作前后执行
  - **PreDelete/PostDelete**：删除操作前后执行

- **使用场景**：
  - 数据验证和过滤
  - 访问控制（列级或行级权限）
  - 审计日志
  - 生成派生列或汇总数据

##### (2) Endpoint Coprocessor（端点）
- **类似数据库存储过程**
- 在 Region Server 上执行自定义计算
- 支持在服务器端进行聚合、连接等操作
- 必须主动调用才会执行

- **使用场景**：
  - 服务器端聚合（如 SUM、AVG）
  - 复杂计算（减少网络传输）
  - 实现二级索引
  - 自定义扫描和过滤逻辑

#### 2. 加载方式
- **动态加载**：通过 HBase Shell 动态挂载到表
- **静态加载**：配置到 hbase-site.xml，集群范围生效

#### 3. 注意事项
- 协处理器代码bug可能导致 Region Server 崩溃
- 需要做好异常处理
- 注意资源消耗，避免影响正常读写

## 题目五：HBase 的数据删除机制是怎样的？为什么删除的数据还能被读取？

### 参考答案

这是一个经典的 HBase 面试问题，涉及墓碑标记（Tombstone Marker）和 Compaction 机制。

#### 1. 删除流程

##### 步骤一：写入删除标记
- 删除数据时，**不会立即物理删除**
- 而是在对应位置写入一个 **删除标记（Tombstone Marker）**
- 删除标记包含：被删除数据的 RowKey、Column Family、Column Qualifier、删除版本

##### 步骤二：标记可见性
- 设置 `Delete` 类型后，该版本数据在读取时不可见
- 但数据仍然存在于 HFile 中

##### 步骤三：Compaction 时物理删除
- **Major Compaction** 时，真正删除带有删除标记的数据
- 合并 HFile，清理墓碑标记
- 释放磁盘空间

#### 2. 三种删除类型
| 删除类型 | 作用范围 |
|---------|---------|
| **Version Delete** | 删除某个特定版本 |
| **Column Delete** | 删除某个列的所有版本 |
| **Family Delete** | 删除整个 Column Family |

#### 3. 为什么删除后还能读到数据？
- **写入顺序问题**：删除标记和后续写入可能乱序
- **HBase 读取原则**：返回最新可见版本
- 如果删除标记还没有被处理，可能读到旧数据
- 读取时会检查时间戳和删除标记

#### 4. 实际案例
```
写入顺序：
1. Put 'row1', 'cf:name' → 'Alice'
2. Delete 'row1', 'cf:name'   ← 墓碑标记
3. Put 'row1', 'cf:name' → 'Bob'

读取结果：
- 如果在 Compaction 前读取：可能返回 'Bob' 或 空（取决于时间戳）
- Compaction 后读取：只能返回 'Bob'
```
