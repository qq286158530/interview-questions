## 题目一：Redis有哪些数据类型？它们的底层实现分别是什么？

### 参考答案

Redis 提供了丰富的数据类型，常见的有**五种基础数据类型**：

| 数据类型 | 应用场景 | 底层实现（Redis 3.0） | 底层实现（Redis 7.0） |
|---------|---------|----------------------|----------------------|
| String（字符串） | 缓存对象、常规计数、分布式锁、共享session | SDS（简单动态字符串） | SDS |
| Hash（哈希） | 缓存对象、购物车 | 压缩列表 / 哈希表 | listpack / 哈希表 |
| List（列表） | 消息队列 | 双向链表 / 压缩列表 | quicklist（listpack） |
| Set（集合） | 聚合计算（点赞、共同关注、抽奖） | 哈希表 / 整数集合 | 哈希表 / 整数集合 |
| Zset（有序集合） | 排行榜、电话姓名排序 | 压缩列表 / 跳表 | listpack / 跳表 |

**关键细节：**

- **SDS 优势**：O(1) 获取长度、二进制安全（可存图片/音频）、自动扩容不缓冲区溢出
- **List 底层**：Redis 3.2 后仅用 quicklist 实现；Redis 7.0 后 quicklist 节点内部用 listpack 替代压缩列表
- **Hash 底层**：压缩列表在 Redis 7.0 已废弃，统一用 listpack
- **String 内部编码**：有 `int`、`embstr`、`raw` 三种，小于等于32字节用embstr（一次内存分配，访问更高效）

**来源：** https://xiaolincoding.com/redis/data_struct/command.html

## 题目二：Redis持久化机制有哪些？RDB和AOF各自的优缺点是什么？

### 参考答案

Redis 共有三种持久化方式：**RDB快照**、**AOF日志**、**混合持久化**。

### RDB 快照

将某一时刻的内存数据以**二进制**方式写入磁盘。提供了 `save`（主线程）和 `bgsave`（子进程）两个命令。Redis 还可以通过配置文件实现每隔一段时间自动快照。

**优点：** 恢复速度快（二进制文件直接加载）
**缺点：** 无法做到秒级持久化，快照频率不好把握（太频繁影响性能，太低丢失数据多）

### AOF 日志

每执行一条写操作命令，就把该命令以**追加方式**写入文件。Redis 重启时通过执行日志中的命令来恢复数据。

AOF 有三种写回策略：

| 策略 | 效果 | 优点 | 缺点 |
|-----|------|-----|------|
| Always | 每次写命令执行完后同步写回硬盘 | 数据不丢失 | I/O压力大 |
| Everysec | 每秒将缓冲区内容写回硬盘（默认） | 性能适中 | 最多丢失1秒数据 |
| No | 由操作系统决定何时写回 | 不用管 | 最多丢失大量数据 |

AOF 文件过大会触发**AOF重写机制**，通过后台子进程 `bgrewriteaof` 完成，用一条命令记录最终键值对，解决文件膨胀问题。

**优点：** 丢失数据少
**缺点：** 文件大、恢复慢（日志是命令而非实际数据）

### 混合持久化

Redis 4.0 新增。结合两者优点：AOF 重写时，先以 RDB 格式写入前半部分，再用 AOF 格式记录增量命令。恢复时先加载 RDB 部分加速，再用 AOF 部分补齐。

**优点：** 启动快 + 丢失数据少
**缺点：** AOF可读性差，不兼容4.0之前版本

**来源：** https://xiaolincoding.com/redis/base/redis_interview.html

## 题目三：Redis哨兵模式是如何工作的？主从故障转移的流程是怎样的？

### 参考答案

哨兵（Sentinel）是 Redis 2.8 后提供的机制，用于实现**主从节点故障转移**。

### 哨兵集群工作三件事

1. **监控**：哨兵每隔1秒向主从节点发送 PING 命令，超时未响应则标记"主观下线"
2. **选主**：当足够多哨兵（达到 quorum 配置值）都认为主节点主观下线 → 标记"客观下线"，然后选举 leader 执行故障转移
3. **通知**：通过发布/订阅机制将新主节点信息通知给从节点和客户端

### 主观下线 vs 客观下线

- **主观下线**：单个哨兵检测到主节点超时无响应
- **客观下线**：多个哨兵投票，达到 quorum 阈值后认定主节点真的故障了（仅针对主节点）

### 主从故障转移流程（4步）

1. **选新主节点**：过滤已下线从节点 → 过滤历史网络连接不好的从节点 → 三轮考察（优先级 → 复制进度 → ID号），选出最优从节点，发送 `SLAVEOF no one` 将其升级为新主节点
2. **重指向从节点**：向所有从节点发送 `SLAVEOF 新主节点`，让它们复制新主节点
3. **通知客户端**：通过 `+switch-master` 频道发布新主节点 IP 和端口，客户端订阅后自动更新连接
4. **旧主节点变为从节点**：旧主节点重新上线后，降级为新主节点的从节点

### 哨兵 leader 选举规则

候选者需满足：
- 拿到**半数以上**赞成票
- 票数 **≥ quorum** 配置值

**为什么至少要3个哨兵节点？** 保证挂掉1个后，剩下2个仍能完成选举（奇数个节点是共识，防止脑裂）

## 题目四：什么是缓存雪崩、击穿、穿透？各自的解决方案是什么？

### 参考答案

缓存异常有三种常见问题：

### 缓存雪崩

**原因：** 大量缓存数据**同时过期**，或 Redis **故障宕机**，导致大量请求直接访问数据库，数据库压力骤增甚至宕机。

**解决方案：**
- **均匀设置过期时间**：给过期时间加随机数，避免集中过期
- **互斥锁**：保证同一时间只有一个请求构建缓存
- **后台更新缓存**：缓存不设置有效期，由后台线程定时更新 + 实时检测失效并重新加载
- **Redis高可用集群**：主从架构防止 Redis 故障宕机导致的雪崩
- **服务熔断/请求限流**：缓存故障时暂停或限制请求，保护数据库

### 缓存击穿

**原因：** 某个**热点数据**过期，此时大量请求访问该热点数据，直接冲击数据库。

**解决方案：**
- **互斥锁**：与雪崩类似
- **热点数据永不过期**：由后台异步更新，或提前续期

### 缓存穿透

**原因：** 用户访问的数据**既不在缓存也不在数据库中**（如业务误删或恶意攻击），导致请求直达数据库。

**解决方案：**
- **非法请求限制**：API入口校验参数合理性
- **缓存空值/默认值**：在缓存中设置空值或默认值，拦截穿透请求
- **布隆过滤器**：写入数据库时在布隆过滤器做标记，查询时先判断数据是否存在（存在不一定真在，但不存在一定不在）

### 三者对比总结

| 问题 | 原因 | 核心方案 |
|-----|------|---------|
| 缓存雪崩 | 大量数据同时过期 / Redis宕机 | 均匀过期 + 高可用集群 + 熔断限流 |
| 缓存击穿 | 热点数据过期 | 互斥锁 / 热点数据永不过期 |
| 缓存穿透 | 数据既不在缓存也不在数据库 | 布隆过滤器 / 缓存空值 |

## 题目五：Redis为什么快？单线程模型是怎样的？6.0之前和之后有什么区别？

### 参考答案

### Redis 为什么这么快？

1. **基于内存**：大部分操作在内存中完成，瓶颈是内存或网络带宽而非 CPU
2. **高效数据结构**：SDS、跳表、哈希表等高效数据结构
3. **单线程无竞争**：避免多线程切换和锁竞争开销，无死锁问题
4. **I/O多路复用**：使用 epoll 机制，一个线程处理多个 Socket 请求（select/epoll）

### Redis 单线程模型（6.0之前）

主线程负责：接收连接 → 解析请求 → 读写数据 → 发送响应。事件循环基于 epoll。

**后台线程（4.0后）：**
- `bio_close_file`：异步关闭文件
- `bio_aof_fsync`：异步 AOF 刷盘
- `bio_lazy_free`：异步释放内存（执行 `unlink` / `flushdb async`）

### Redis 6.0 之后的变化

- 引入 **多线程 I/O**，默认只针对**发送响应数据**（write socket），可通过配置开启多线程读
- **命令执行仍然是单线程**，所以 Redis 仍然是单线程，只是 I/O 并行化了
- 官方表示多线程 I/O 性能提升**至少一倍**

### 线程数设置建议

- 4核CPU → 线程数设 2 或 3
- 8核CPU → 线程数设 6
- 线程数 **不是越大越好**，应小于机器核数

### 关键结论

- Redis 单线程指的是**命令执行**单线程，不是"整个Redis进程"单线程
- 6.0后网络 I/O 多线程，命令执行仍是单线程
- CPU 不是 Redis 瓶颈，内存和网络才是
