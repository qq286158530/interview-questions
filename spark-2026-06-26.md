## 题目一：Spark 架构与核心组件

### 问题
请描述 Apache Spark 的整体架构及其核心组件各自的作用。

### 答案

Spark 采用 **Master-Slave 架构**，核心组件包括：

| 组件 | 作用 |
|------|------|
| **Driver（驱动器）** | 运行 `main()` 方法，创建 `SparkContext`，负责任务分发和结果收集。将作业转换为 Task，再分发给 Executor。 |
| **Executor（执行器）** | 运行在 Worker 节点上的进程，负责执行具体的 Task 任务，拥有自己的内存池和 CPU 资源。 |
| **Cluster Manager（集群管理器）** | 负责集群资源的管理和调度，如 Standalone Master、YARN ResourceManager、Kubernetes Master 等。 |
| **Worker Node** | 集群中的实际工作节点，运行 Executor 进程。 |

**作业提交流程：**
1. 用户提交 Spark Application，Driver 创建 `SparkContext`；
2. `SparkContext` 向 Cluster Manager 申请资源；
3. Cluster Manager 在 Worker 节点上启动 Executor；
4. Driver 将应用代码发送给 Executor；
5. `SparkContext` 将 Job 划分为 Stage，再划分为 Task，分发到各 Executor 执行；
6. Executor 执行 Task，结果返回 Driver；
7. 任务完成后释放资源。

## 题目二：RDD 详解

### 问题
什么是 RDD？RDD 有哪些核心特性？请详细说明。

### 答案

**RDD（Resilient Distributed Dataset）** 是 Spark 最底层的抽象，代表一个不可变、可分区、可以并行计算的分布式数据集。

### 核心特性

1. **分区（Partitioned）**
   - RDD 数据被划分为多个分区，每个分区可以在集群的不同节点上并行计算。

2. **只读（Immutable）**
   - RDD 是只读的，任何 Transformation 都会生成一个新的 RDD，不修改原数据，保证数据一致性。

3. **依赖关系（Dependencies）**
   - RDD 之间存在依赖关系，分为两类：
     - **窄依赖（Narrow Dependency）**：父 RDD 的每个分区最多被一个子分区使用（如 `map`、`filter`），支持 Pipeline 执行。
     - **宽依赖（Wide Dependency）**：父 RDD 的一个分区可能被多个子分区使用（如 `groupByKey`、`join`），会触发 Shuffle。

4. **计算函数（Compute Function）**
   - 每个 RDD 知道如何通过父 RDD 计算出自己，支持透明容错。

5. **分区器（Partitioner）**（可选）
   - 对于键值对 RDD，可以指定分区器（如 `HashPartitioner`、`RangePartitioner`），影响 Shuffle 行为。

6. **最优位置（Preferred Locations）**
   - RDD 知道数据存储的位置，会尽量将计算调度到数据所在节点，实现数据本地性。

### 常见 Transformation 与 Action

```python
# Transformation（返回新的 RDD）
rdd = sc.textFile("hdfs://path/to/data")
rdd2 = rdd.map(lambda x: x.split(","))      # 窄依赖
rdd3 = rdd.filter(lambda x: "error" in x)   # 窄依赖
rdd4 = rdd.groupByKey()                      # 宽依赖，触发 Shuffle

# Action（触发计算，返回结果）
result = rdd.collect()     # 返回所有元素到 Driver
count = rdd.count()        # 返回元素个数
first = rdd.first()        # 返回第一个元素
```

## 题目三：Shuffle 机制

### 问题
什么是 Shuffle？为什么 Spark 需要 Shuffle？Shuffle 有哪些优化？

### 答案

**Shuffle（混排）** 是 Spark 中将数据按-key重新分区的过程，发生在宽依赖操作时（如 `groupByKey`、`reduceByKey`、`join`、`sortByKey` 等）。

### 为什么需要 Shuffle

当需要对数据进行聚合、关联或排序时，必须将相同 key 的数据汇聚到同一个节点。这个数据跨节点传输的过程就是 Shuffle。

### Shuffle 的问题

- **大量磁盘 I/O**：Shuffle 数据先写磁盘再读磁盘；
- **网络传输**：数据跨节点传输，消耗网络带宽；
- **内存压力**：Shuffle 数据可能占用大量堆内内存；
- **磁盘压力大**：大量小文件随机读写。

### Spark 3.x Shuffle 优化

1. **Sort-Based Shuffle（默认）**
   - 数据先写入内存缓冲区，达到阈值后排序写入磁盘文件；
   - 每个 Map 任务只产生少量（最多 2 个）有序的磁盘文件，减少文件数量。

2. **统一 SortShuffleManager**
   - Spark 2.x 起合并了 SortShuffleWriter 和 BypassMergeSortShuffleWriter，统一管理。

3. **参数调优**
   ```python
   # 增大 Shuffle 文件缓冲，减少磁盘写入
   conf.set("spark.shuffle.file.buffer", "64kb")   # 默认 32KB
   
   # 使用 SortShuffle（已默认）
   conf.set("spark.shuffle.manager", "sort")
   
   # 开启_consolidateFiles_（Spark 2.x 以前有效）
   conf.set("spark.shuffle.consolidateFiles", "true")
   ```

4. **Spark 3.x AQE（Adaptive Query Execution）**
   - 运行时动态调整 Shuffle 分区数；
   - 动态启用 Sort Merge Join 或 Broadcast Join。

## 题目四：广播变量（Broadcast Variables）

### 问题
什么是广播变量？它解决了什么问题？使用广播变量有哪些注意事项？

### 答案

**广播变量**是 Spark 用来高效分发大只读变量到每个节点节点的机制。

### 解决的问题

假设在 `map` 操作中使用一个查询表（如字典）：
```python
# ❌ 不用广播变量：每个 Task 都复制一份表到内存，网络传输 N 次
lookup_table = {"北京": "华北", "上海": "华东", ...}
rdd.map(lambda x: lookup_table.get(x.city, "未知"))
```

使用广播变量后：
```python
# ✅ 使用广播变量：每个 Executor 只复制一次，所有 Task 共享
broadcast_table = sc.broadcast({"北京": "华北", "上海": "华东", ...})
rdd.map(lambda x: broadcast_table.value.get(x.city, "未知"))
```

### 优势

- **减少网络传输**：大变量只传输到每个 Executor，而不是每个 Task；
- **降低内存占用**：Executor 内所有 Task 共享同一份数据副本；
- **加速计算**：减少数据获取的延迟。

### 注意事项

1. **变量大小**：只适合中等大小的只读数据（几十 MB 到几百 MB），过大会导致 Executor OOM；
2. **不可修改**：广播变量创建后不可修改；
3. **释放时机**：不再使用时 Spark 会自动释放，或手动调用 `broadcast.unpersist()`；
4. **序列化**：广播变量必须可序列化（实现 `Serializable`）；
5. **合理分区**：对于超大广播变量，可考虑手动分区或使用外部存储（如 Redis）。

### 最佳实践

```python
# 广播大变量
broadcast_data = sc.broadcast(load_large_data())

# 在 Task 中访问广播变量的 .value 属性
def process(row):
    return broadcast_data.value.get(row.key, "default")

rdd.map(process).collect()

# 手动删除缓存（可选）
broadcast_data.unpersist()
```

## 题目五：Spark 内存管理

### 问题
Spark 的内存模型是怎样的？堆内内存（On-Heap）和堆外内存（Off-Heap）有什么区别？请说明 Spark 2.x 统一内存管理模型的组成。

### 答案

### 内存模型概述

Spark 2.x 引入**统一内存管理（Unified Memory Management）**，将堆内和堆外内存统一管理。

### 堆内内存（On-Heap Memory）

由 JVM 堆分配，受 GC 影响，分为以下区域：

| 区域 | 用途 | 配置参数 | 默认占比 |
|------|------|----------|---------|
| **Execution（执行内存）** | Shuffle 排序、Hash Join 聚合、缓存 RDD 计算结果 | `spark.memory.fraction` | 约 60% |
| **Storage（存储内存）** | 缓存 RDD/Broadcast/广播变量、Task 结果暂存 | `spark.memory.storageFraction` | 约 40% |
| **User Memory（用户内存）** | 用户数据结构与元数据、RDD 依赖元信息 | 由 `spark.memory.fraction` 间接控制 | 约 40% |
| **Reserved Memory（预留内存）** | 系统预留，300MB，存放 Spark 内部对象 | 固定 | 300MB |

### 堆外内存（Off-Heap Memory）

- 通过 `sun.misc.Unsafe` 直接分配，不受 JVM GC 管理；
- 通过 `spark.memory.offHeap.enabled=true` 开启；
- 通过 `spark.memory.offHeap.size` 设置大小；
- 主要用于存储序列化的 Shuffle 数据，减少 GC 开销。

###  Execution 与 Storage 内存动态共享

统一内存管理下，Execution 和 Storage 区域**可以相互借用**：

```
┌──────────────────────────────────────┐
│           堆内内存（On-Heap）          │
│  Reserved  │   User    │ Unified     │
│  300MB     │  Memory   │ Memory      │
│            │           │ ┌────────┐ │
│            │           │ │Execution│ │
│            │           │ │ Storage│ │
│            │           │ │(动态)   │ │
│            │           │ └────────┘ │
└──────────────────────────────────────┘
```

- 当 Execution 内存不足时，可以借用 Storage 空闲内存；
- 当 Storage 内存不足时，**不能**抢占 Execution 内存，只能等待 Execution 释放；
- 通过 `spark.memory.storageFraction` 设置初始比例。

### 常用调参

```python
conf.set("spark.memory.fraction", "0.6")          # 堆内用于 Execution+Storage 的比例
conf.set("spark.memory.storageFraction", "0.5")    # Storage 初始占比
conf.set("spark.memory.offHeap.enabled", "true")   # 开启堆外内存
conf.set("spark.memory.offHeap.size", "2g")        # 堆外内存大小
```

### 内存溢出（OOM）常见原因

1. 缓存过多 RDD，数据长期占据 Storage 内存；
2. 大表 Join 时 Shuffle 数据过大，撑爆 Execution 内存；
3. Task 数量过少，单个 Task 处理数据量过大；
4. 广播变量过大，超出 Executor 可用内存。
