## 题目一：Redis 的过期 key 删除策略有哪些？它们的区别是什么？

### 参考答案

Redis 处理过期 key 主要有三种策略：

### 1. 定时删除（TTL Timer）
- **原理**：为每个 key 创建一个定时器，key 过期时立即删除
- **优点**：内存友好，能及时释放内存
- **缺点**：CPU 不友好，过期 key 较多时会消耗大量 CPU 资源
- **实际场景**：几乎不用，性能开销太大

### 2. 惰性删除（Lazy Expire）
- **原理**：访问 key 时才判断是否过期，过期则删除
- **优点**：CPU 友好，只处理被访问的 key
- **缺点**：内存不友好，如果 key 从未被访问则永不释放
- **实际场景**：Redis 默认策略，行为上接近 LRU

### 3. 定期删除（Active Expire）
- **原理**：每隔一段时间扫描 Redis 的过期 key 集合，删除其中过期的 key
- **实现**：Redis 每 100ms 运行一次 `activeDefragCycle`，随机抽样检测并删除过期 key
- **优点**：平衡了 CPU 和内存，开销可控
- **缺点**：仍有可能遗漏部分过期 key

### 两种 Redis 过期策略的配合
生产环境通常采用 **惰性删除 + 定期删除** 的组合：
- **惰性删除**：每次访问 key 时检查并删除
- **定期删除**：后台定时任务补充清理

### 追问：RDB 对过期 key 的处理
- `save` 和 `bgsave` 生成 RDB 文件时，已过期的 key 不会被写入
- `redis-check-rdb` 检测 RDB 时会跳过过期 key

## 题目二：Redis 的内存淘汰策略有哪些？如何选择合适的策略？

### 参考答案

当 Redis 内存达到上限（`maxmemory`）时，会触发内存淘汰策略删除 key。

### 八种淘汰策略

| 策略 | 含义 |
|-----|------|
| `noeviction` | 不淘汰，返回写操作错误（默认） |
| `volatile-lru` | 从过期 key 集合中淘汰最近最少使用的 |
| `volatile-lfu` | 从过期 key 集合中淘汰使用频率最低的 |
| `volatile-ttl` | 从过期 key 集合中淘汰 TTL 最短的 |
| `volatile-random` | 从过期 key 集合中随机淘汰 |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用的 |
| `allkeys-lfu` | 从所有 key 中淘汰使用频率最低的 |
| `allkeys-random` | 从所有 key 中随机淘汰 |

### LRU vs LFU
- **LRU（Least Recently Used）**：淘汰最长时间未被访问的
- **LFU（Least Frequently Used）**：淘汰访问次数最少的，更适合热点数据保留

### 如何选择

**选择建议：**
- **热点数据场景**：`allkeys-lru` — 优先保留热点数据
- **有明确过期时间**：`volatile-lru` 或 `volatile-lfu` — 只淘汰会过期的
- **缓存业务**：`allkeys-lru` + `maxmemory-policy`
- **会话存储**：`volatile-lru` — 配合 TTL 使用
- **禁止数据淘汰**：`noeviction` — 仅用于不允许丢数据的场景

### 生产配置示例
```bash
maxmemory 3gb
maxmemory-policy allkeys-lru
maxmemory-samples 5
```
`maxmemory-samples 5` 表示抽样 5 个 key 进行 LRU/LFU 计算。

### 追问：如何查看 Redis 内存使用情况
```bash
info memory        # 查看内存详情
memory stats       # 详细内存统计
CLIENT LIST        # 查看客户端内存占用
```

## 题目三：Redis 集群有哪些模式？各自适用什么场景？

### 参考答案

Redis 集群主要有四种模式，从简单到复杂：

### 1. 单机模式（Single）
- 最基础的模式，不推荐生产使用
- 适用：开发测试、极小数据量

### 2. 主从复制（Master-Slave）
```
Master（主） → Slave1（从）
            → Slave2（从）
```
- **原理**：主节点写，从节点自动同步读
- **优点**：读写分离、备份简单
- **缺点**：主节点挂了需要手动切换
- **适用**：读多写少场景，数据备份

### 3. 哨兵模式（Sentinel）
```
           ┌→ Sentinel（监控）
Master → Slave
    ↑    ↑
    └← Sentinel（故障检测）←→ 客户端
```
- **原理**：哨兵进程监控主从，自动故障转移
- **功能**：监控、通知、自动故障转移
- **优点**：自动切换，客户端只需连接哨兵
- **缺点**：只有主节点能写，哨兵本身没有集群功能
- **适用**：中小规模，搭建快

### 4. Cluster 模式（Redis 原生集群）
```
        ┌→ Node1 (master) → Replica
Slot0~5460
Slot5461~10922  → Node2 (master) → Replica
Slot10923~16383 → Node3 (master) → Replica
```
- **原理**：16384 个 slot 分布在多个节点
- **优点**：数据分片、自动故障转移、水平扩展
- **缺点**：架构复杂，不支持跨 slot 的多键操作
- **适用**：大规模生产环境

### 模式对比

| 模式 | 自动故障转移 | 写入能力 | 数据分片 | 复杂度 |
|-----|------------|---------|---------|-------|
| 主从复制 | ❌ | 单master | ❌ | 低 |
| 哨兵模式 | ✅ | 单master | ❌ | 中 |
| Cluster | ✅ | 多master | ✅ | 高 |

### 生产建议
- **< 10 台服务器**：哨兵模式足够
- **> 10 台或大数据量**：Cluster 模式
- **最低要求**：主从 + 哨兵，不能裸奔

**来源参考：** [Redis 集群官方文档](https://redis.io/docs/management/scaling/) | [Redis Sentinel vs Cluster](https://www.cnblogs.com/rjzheng/p/10837543.html)

## 题目四：如何实现一个可靠的 Redis 分布式锁？

### 参考答案

分布式锁是分布式系统中保证多进程互斥访问的核心手段。

### 基本实现思路
```python
# 获取锁
lock = redis.set("lock_key", "unique_value", nx=True, ex=10)
if lock:
    # 业务逻辑
    redis.delete("lock_key")
else:
    # 获取锁失败
```

### 可靠分布式锁的核心要求

1. **互斥性**：同一时刻只能有一个客户端持有锁
2. **防死锁**：锁要有自动过期时间，防止进程崩溃无法释放
3. **原子性**：加锁和释放锁必须原子操作
4. **可重入**：同一客户端可多次获取锁

### 常见问题与解决方案

#### 问题一：锁误删（释放了他人的锁）
**场景**：A 获取锁，超时未完成，B 获得同一把锁，A 完成并释放锁时删除了 B 的锁
**解决**：使用唯一值（UUID）标识锁，释放时判断值是否匹配
```python
# 错误写法
redis.delete("lock_key")  # 可能误删

# 正确写法
if redis.get("lock_key") == "uuid_123":
    redis.delete("lock_key")
```

但上述写法不是原子的，正确方式是使用 Lua 脚本：
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

#### 问题二：锁续期（锁提前过期导致任务未完成）
**解决**：Redisson 的看门狗机制自动续期

#### 问题三：主从切换导致锁丢失
**解决**：RedLock 算法，在 N 个独立 Redis 实例上获取锁

### Redisson 分布式锁示例
```java
RLock lock = redissonClient.getLock("order_lock");
try {
    // 等待锁，最多等待10秒，锁定后自动30秒释放
    lock.lock(30, TimeUnit.SECONDS);
    // 业务逻辑
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

### 总结：分布式锁实现要点
| 要点 | 实现 |
|-----|------|
| 互斥 | `SETNX` / `SET NX EX` |
| 防死锁 | 设置过期时间 |
| 原子释放 | Lua 脚本 |
| 可重入 | 计数 + 唯一值 |
| 高可用 | RedLock |

**来源参考：** [Redis 分布式锁实现](https://xiaolincoding.com/redis/lock/distributed_lock.html) | [Redisson 官方文档](https://github.com/redisson/redisson/wiki/8.-Distributed-locks-and-synchronizers/)

## 题目五：如何排查和优化 Redis 慢查询？

### 参考答案

慢查询是 Redis 性能问题的常见原因，需要系统化排查和优化。

### 排查步骤

#### 第一步：开启慢查询日志
```bash
# 开启慢查询
config set slowlog-log-slower-than 10000   # 10ms
config set slowlog-max-len 128              # 保留128条

# 查看慢查询
slowlog get 10        # 最近10条
slowlog get -- 50     # 最多50条

# 清空日志
slowlog reset
```

#### 第二步：分析慢查询原因

**常见慢查询原因：**
1. **大 Key 操作**：`HGETALL`、`SMEMBERS`、`LRANGE` 返回大量数据
2. **全量扫描**：`KEYS *`、`FLUSHDB`、`FLUSHALL`
3. **阻塞命令**：`SORT`（无索引排序）、`SUNIONSTORE`
4. **内存不足**：触发 Swap 或 OOM

#### 第三步：定位大 Key
```bash
# 扫描大 key（Redis 4.0+）
redis-cli --bigkeys

# 扫描指定 key
redis-cli --scan | xargs redis-cli --bigkeys
```

### 优化策略

| 问题 | 优化方案 |
|-----|---------|
| 大 Key | 拆分为小 Key、使用 Hscan/Sscan/Lrange 分批读取 |
| KEYS * | 使用 SCAN 替代（`SCAN cursor MATCH pattern COUNT 100`） |
| 热 Key | 拆分到不同节点、使用本地缓存兜底 |
| 全量操作 | 避免 `FLUSHDB`，改为按 key 前缀逐步删除 |

### 生产监控建议

**必做监控项：**
```bash
# CPU 和内存
info cpu        # CPU 使用率
info memory     # 内存使用率

# 命令统计
info stats | grep ops      # 每秒命令执行数
info commandstats           # 各命令耗时统计

# 客户端
client list                 # 查看连接数和来源
```

**关键指标阈值：**
- `instantaneous_ops_per_sec > 10000` → 考虑扩容
- `used_memory_rss > maxmemory * 0.8` → 触发内存淘汰，影响性能
- `blocked_clients > 0` → 有命令被阻塞

### 追问：Redis 命令耗时排名
```bash
redis-cli info commandstats
# Output example:
# cmdstat_get:calls=1000000,usec=500000,usec_per_call=0.50
# cmdstat_hgetall:calls=100,usec=200000,usec_per_call=2000.00
```
`HGETALL` 平均耗时 2ms，`GET` 仅 0.5μs，差距 4000 倍！
