## 题目 1：Spark 支持哪几种部署模式？Driver 和 Executor 的作用是什么？

### 参考答案

**部署模式：**

| 模式 | 说明 |
|------|------|
| Local | 本地单 JVM 运行，用于开发和测试 |
| Standalone | Spark 自带的独立集群管理器 |
| YARN | 运行在 Hadoop YARN 之上，支持 yarn-client/yarn-cluster |
| Mesos | 运行在 Apache Mesos 集群上 |
| Kubernetes | 运行在 K8s 集群中 |

**Driver（驱动器）：**

- 是 Spark 应用程序的入口点，运行 `main()` 方法
- 负责构建 `SparkContext`，将用户代码转换为 DAG 并划分为 Stage 和 Task
- 协调整个作业的执行，将 Task 分发到 Executor
- 在 yarn-cluster 模式下，Driver 在 ApplicationMaster 中运行

**Executor（执行器）：**

- 是 Worker 节点上运行的进程，负责执行分配给它的 Task
- 每个 Executor 启动时向 Driver 注册，并保持通信
- 将数据存储在内存或磁盘中（Cache）
- 任务完成后将结果汇报给 Driver

> **来源：** Spark Official Documentation - Overview / Components
> https://spark.apache.org/docs/latest/cluster-overview.html

## 题目 2：RDD 是什么？为什么 Spark 要设计 RDD？

### 参考答案

**RDD（Resilient Distributed Dataset）** 是 Spark 最核心的数据抽象，表示一个可分区的、容错的、可以并行操作的数据集合。

**RDD 的五大属性：**

1. **` partitions`**：数据分区的列表
2. **` computeFn(partition)`**：计算每个分区的函数
3. **` dependencies`**：RDD 依赖关系（宽依赖/窄依赖）
4. **` partitioner`**：可选，分区器（用于 PairRDD）
5. **` preferredLocations`**：每个分区偏好的位置（数据本地性）

**为什么设计 RDD：**

- **容错机制**：通过 lineage（血统）链记录转换关系，某个分区丢失时从源头重新计算，而不是复制数据
- **数据本地性**：计算向数据移动，避免大量网络 IO
- **内存计算**：支持将数据缓存到内存，加速迭代计算
- **简单易用**：提供简洁的高级 API，屏蔽底层分布式细节

> **来源：** Matei Zaharia et al., "Resilient Distributed Datasets: A Fault-Tolerant Abstraction for In-Memory Cluster Computing" (Spark 原始论文)
> https://www.usenix.org/system/files/conference/nsdi12/nsdi12-final138.pdf

## 题目 3：解释 Spark 的 Shuffle 机制，为什么 Shuffle 是性能瓶颈？

### 参考答案

**Shuffle（ shuffle ）是什么：**

当 RDD 转换操作需要重新分区数据时（如 `groupByKey`、`join`、`sortByKey`），数据需要跨节点重新分布，这个过程称为 Shuffle。

**执行流程（Spark 2.x 以后）：**

1. **Map 阶段**：每个 Task 将数据写入本地磁盘文件，按 Partition ID 和 Reducer ID 组织
2. **Shuffle Write**：数据溢写（Spill）到磁盘，生成多个小文件
3. **Shuffle Fetch**：每个 Reduce Task 去所有 Map 节点拉取属于自己的数据
4. **Reduce 阶段**：数据在 Executor 中归并聚合

**为什么是性能瓶颈：**

| 问题 | 说明 |
|------|------|
| **磁盘 IO** | Shuffle 数据先落盘再网络传输，磁盘读写成为主要延迟 |
| **网络 IO** | 数据跨节点传输，数据量大时网络带宽成为瓶颈 |
| **内存压力** | Map 端缓存、聚合需要大量内存，容易触发 OOM |
| **大量小文件** | Shuffle 产生大量小文件，给文件系统带来压力 |
| **序列化开销** | 数据需要序列化/反序列化，CPU 密集 |

**优化方法：**

- 减少 Shuffle 数据量（先 map-side join、后 filter 减少数据）
- 合理设置 `spark.sql.shuffle.partitions`（默认 200，可调）
- 开启 `spark.shuffle.compress` 压缩
- 使用 `coalesce`/`repartition` 减少分区数

> **来源：** Spark Official Documentation - Shuffle Behavior
> https://spark.apache.org/docs/latest/rdd-programming-guide.html#shuffle-operations

## 题目 4：Spark 的内存管理模型是怎样的？堆内内存和堆外内存有什么区别？

### 参考答案

** Unified Memory（统一内存管理，Spark 1.6+）**：

Spark 统一管理 **存储内存** 和 **执行内存**，两者可以动态借用。

```
┌──────────────────────────────────────┐
│            JVM Heap (堆内)             │
│  ┌──────────┬──────────┬───────────┐  │
│  │  系统预留  │  存储内存  │  执行内存  │  │
│  │ (~300MB) │  (可动态扩展)│  (可动态扩展)│  │
│  └──────────┴──────────┴───────────┘  │
└──────────────────────────────────────┘
```

**堆内内存（On-Heap Memory）：**

- 由 JVM 管理，大小由 `spark.memory.fraction`（默认 0.6）控制
- 对象存储在 JVM Heap 中，有 GC 压力
- 存储和执行内存都在堆内，可以相互动态借用
- `spark.memory.storageFraction`（默认 0.5）决定双方的底部边界

**堆外内存（Off-Heap Memory）：**

- 通过 `spark.memory.offHeap.enabled=true` 开启
- 使用 `sun.misc.unsafe` 直接操作堆外内存，绕过 JVM
- **执行内存**可在堆外分配（`spark.memory.offHeap.size`）
- 优点：减少 GC 开销，适合大内存场景；缺点：不稳定

**内存分配策略：**

- **存储内存**：存放 cached RDD/DataFrame、广播变量、Unroll 内存
- **执行内存**：存放 Shuffle 排序/聚合的缓冲区、元数据
- 两者可以互相借用，但执行内存被驱逐时会导致数据重新计算

> **来源：** Spark Official Documentation - Memory Management
> https://spark.apache.org/docs/latest/tuning.html#memory-management

## 题目 5：广播变量（Broadcast）和累加器（Accumulator）的作用与使用场景是什么？

### 参考答案

### 广播变量（Broadcast Variable）

**概念：** 将一个只读的大变量（表、配置、字典）广播到所有节点，每个节点缓存一份，而非每个 Task 传递一份。

**使用方式：**

```scala
val broadcastVar = sc.broadcast(List("a", "b", "c"))
val result = rdd.map(x => broadcastVar.value.contains(x))
```

**优势：**

- 大幅减少网络传输：数据只在 Driver → Executor 时传输一次
- 减少 Task 序列化开销：每个 Task 不需要携带大对象
- 典型场景：维度表 join（map-side join）、大配置表查找

**注意事项：**

- 只能广播只读数据，广播后不可修改
- 广播大变量时需合理设置 `spark.broadcast.blockSize`

---

### 累加器（Accumulator）

**概念：** 提供分布式计数器功能，允许多个 Task 安全地对同一个变量进行累加操作。

**使用方式：**

```scala
val accum = sc.longAccumulator("My Counter")
rdd.foreach(x => accum.add(x))
println(accum.value) // 在 Driver 读取
```

**特性：**

- Task 只能对累加器执行 `add`（累加）操作
- Driver 只能 `read`（读取）操作
- 支持自定义类型（继承 `AccumulatorV2`）
- 重复执行同一个 job，累加器会重复累加（需注意）

**使用场景：**

- 统计日志行数、错误记录数
- 实现分布式 counter
- 调试时追踪数据处理情况

> **来源：** Spark Official Documentation - Shared Variables
> https://spark.apache.org/docs/latest/rdd-programming-guide.html#shared-variables
