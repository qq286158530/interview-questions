**题目1：Java基础 — String为什么是不可变的？有什么好处？**

【答案】

String是Java中最常用的类，其不可变性（Immutable）是由以下设计决定：

**不可变性的实现机制：**
1. `String`类被`final`修饰，不能被继承，防止子类破坏其行为
2. `String`内部用于存储字符的`char[]`（JDK9后是`byte[]`）也被`final`修饰，数组引用不可改变
3. String类没有提供任何公开的修改字符数组内容的方法（如setCharAt）
4. 所有String的"修改"操作（如`concat`、`substring`、`replace`）都是返回一个新的String对象，而不是修改原对象

```java
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {
    private final char[] value;  // JDK9+是 byte[] + coder
    // 没有setCharAt等修改方法
}
```

**不可变性的好处：**
1. **安全性**：字符串常用于网络连接、文件路径、数据库连接等场景，不可变性确保值不会被意外或恶意修改
2. **线程安全**：不可变对象天然线程安全，无需同步措施，适合并发场景
3. **哈希码缓存**：String的hashCode可以被缓存（因为值不变），适合作为HashMap的key
4. **字符串常量池**：JVM对String有特殊处理，相同字面量的字符串可以共享内存，减少内存占用
5. **类加载器安全性**：不可变性确保类加载时使用的字符串不会被篡改

**面试追问：**
- StringBuilder和StringBuffer的区别？答：前者非同步（性能高），后者同步（线程安全）
- String在JDK9后为什么改成byte[]存储？答：节省内存，Latin-1字符集只需1字节

**题目2：Java进阶 — 什么是AQS？ReentrantLock可重入锁如何实现？**

【答案】

**AQS（AbstractQueuedSynchronizer）概述：**
AQS是Java并发包`java.util.concurrent.locks`下的抽象类，是Doug Lea大师设计的同步框架核心。它使用一个volatile的int类型的state变量表示同步状态，配合FIFO双向队列管理获取锁失败的线程。

**核心结构：**
```java
public abstract class AbstractQueuedSynchronizer
    extends AbstractOwnableSynchronizer {
    // 同步状态
    private volatile int state;
    // FIFO双向等待队列头
    private transient volatile Node head;
    // FIFO双向等待队列尾
    private transient volatile Node tail;
}
```

**ReentrantLock可重入原理：**
ReentrantLock内部定义了Sync类继承AQS：

1. **state表示重入次数**：state=0表示无锁，state>0表示已有线程持有且重入次数为state值

2. **非公平锁获取锁（tryAcquire）：**
```java
// NonfairSync.tryAcquire核心逻辑
final void lock() {
    if (compareAndSetState(0, 1))
        setExclusiveOwnerThread(Thread.currentThread());
    else
        acquire(1);
}

public final void acquire(int arg) {
    if (!tryAcquire(arg) && acquireQueued(addWaiter(Node.EXCLUSIVE), arg))
        selfInterrupt();
}

// tryAcquire核心逻辑
protected final boolean tryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        // 尝试CAS修改state，成功则标记持有线程
        if (compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    } else if (current == getExclusiveOwnerThread()) {
        // 若当前线程已是持有者，state累加重入次数
        int nextc = c + acquires;
        if (nextc < 0) // overflow
            throw new Error("Maximum lock count exceeded");
        setState(nextc);
        return true;
    }
    return false;
}
```

3. **可重入实现**：当同一线程再次获取锁时，因为`current == getExclusiveOwnerThread()`成立，state会累加，允许同一线程多次持有锁

4. **释放锁（tryRelease）：**
```java
protected final boolean tryRelease(int releases) {
    int c = getState() - releases;
    if (Thread.currentThread() != getExclusiveOwnerThread())
        throw new IllegalMonitorStateException();
    boolean free = false;
    if (c == 0) {
        // 只有state减到0时才真正释放锁
        setExclusiveOwnerThread(null);
        free = true;
    }
    setState(c);
    return free;
}
```

**可重入的意义：** 允许同一线程在持有锁的情况下多次调用同步方法，而不会自己死锁自己。比如：
```java
synchronized void m1() {
    m2();  // 可以再次获取锁，不会死锁
}
synchronized void m2() {
    // ...
}
```

**题目3：Spring Boot — 事务失效的常见场景有哪些？如何避免？**

【答案】

Spring声明式事务基于AOP代理实现，事务失效的本质是代理被绕过或失效。常见场景：

**1. 方法内部自调用（最常见）：**
```java
@Service
public class UserService {
    public void transfer(String from, String to, int amount) {
        // 走代理，事务生效
        updateAccount(from, -amount);
        // 内部调用不走代理，事务失效！
        doSomethingElse();
    }
    
    @Transactional
    public void updateAccount(String account, int amount) {
        // 这个方法的事务不会生效，因为是内部调用
    }
}
```
**原因**：内部调用不走代理对象，而是直接调用目标方法，绕过了AOP增强。

**解决方案**：
- 注入自身代理对象调用
- 使用`TransactionTemplate`编程式事务
- 将方法拆分到另一个Service

**2. 异常被吞掉或未抛出：**
```java
@Transactional
public void update() {
    try {
        doSomething();
    } catch (Exception e) {
        // 异常被捕获未抛出，事务不回滚
        log.error(e);
    }
}
```
Spring事务默认只对`RuntimeException`和`Error`回滚，需通过`rollbackFor`指定。

**3. 非public方法：**
```java
@Transactional
void doSomething() {  // private方法事务不生效
}
```
CGLIB代理要求方法是public的。

**4. 多数据源未正确配置：**
使用多数据源时，未指定TransactionManager，Spring可能使用默认的不相关数据源事务。

**5. 事务传播行为配置错误：**
```java
@Transactional(propagation = Propagation.SUPPORTS)
public void methodA() {  // SUPPORTS不创建事务，可能不符合预期
}
```

**6. 同一个类内部无代理对象调用工具类，工具类方法的事务不生效。**

**7. 锁与事务顺序问题**：`@Transactional`在锁外面，事务提交在锁释放前，锁释放后数据已提交一般没问题，但若锁范围小而事务范围大可能出现不一致。

📖 来源：基于Spring Framework官方文档及《Spring源码深度解析》整理

**题目4：Spring Cloud — Sentinel是如何实现熔断降级的？核心原理是什么？**

【答案】

**Sentinel核心概念：**
Sentinel是阿里开源的流量控制、熔断降级组件，与Hystrix类似但更轻量。

**1. 核心原理 — 滑动窗口算法：**
Sentinel使用滑动窗口（LeapArray）统计请求数据，而不是每次请求都做统计：
- 将时间划分为固定大小的桶（默认1秒一个桶，共60个桶）
- 每个桶记录该时间窗口内的通过/拒绝/异常等指标
- 滑动窗口可以快速计算任意时间窗口的QPS/响应时间

```java
// 核心思想：环形数组存储统计信息
private final AtomicReferenceArray<MetricBucket> array;
```

**2. 熔断器原理（StateMachine）：**
Sentinel有三个状态：CLOSED（正常）、OPEN（熔断）、HALF_OPEN（半开）：
- **CLOSED→OPEN**：当异常比例/异常数量/响应时间超过阈值，触发熔断
- **OPEN→HALF_OPEN**：经过熔断时长（默认10s），进入半开状态
- **HALF_OPEN→OPEN/CLOSED**：允许N个请求通过，若成功则恢复CLOSED，否则重新OPEN

```java
// 状态转换核心逻辑
public class CircuitBreaker {
    // 触发熔断
    private void transformToOpen(float snapshot) {
        currentState.set(CircuitBreakerState.OPEN);
        nextRetryTime = System.currentTimeMillis() + recoveryIntervalMs;
    }
}
```

**3. 流量控制原理：**
- **QPS流控**：计算当前QBS与阈值比较，超限则拒绝
- **并发线程数流控**：统计当前正在处理的请求数
- **warm-up预热**：根据请求量逐步提升阈值，防止冷启动压垮系统

**4. 与Hystrix的区别：**
| 维度 | Sentinel | Hystrix |
|------|----------|---------|
| 线程隔离 | 信号量隔离（默认） | 线程池隔离 |
| 熔断策略 | 基于异常比例/数量/响应时间 | 基于错误比例 |
| 实时性 | 滑动窗口，实时计算 | 依赖Hystrix命令统计 |
| 性能 | 轻量，无额外线程池开销 | 有线程池开销 |

📖 来源：基于Sentinel官方文档 https://github.com/alibaba/Sentinel 整理

**题目5：Spring Cloud — Feign远程调用的核心原理是什么？**

【答案】

**Feign是什么：**
Feign是Netflix开源的声明式HTTP客户端，开发者只需定义接口并在方法上标注注解，即可完成HTTP调用，无需手写HTTP客户端代码。

**核心原理：**

**1. 启动时的处理（spring.factories / @EnableFeignClients）：**
```java
@EnableFeignClients
public @interface EnableFeignClients {
    // 扫描@FeignClient注解的接口
}
```

**2. @FeignClient注解解析：**
每个被`@FeignClient`标注的接口，Feign会为其生成一个代理实现类（通过JDK动态代理或CGLIB）：
```java
@FeignClient(name = "user-service", url = "http://localhost:8080")
public interface UserClient {
    @GetMapping("/user/{id}")
    User getUser(@PathVariable Long id);
}
```

**3. 请求构建（Contract解析 + RequestTemplate）：**
Feign内部有一个`Contract`解析器，解析Java接口上的注解（如`@GetMapping`、`@PathVariable`），生成`RequestTemplate`对象：
```java
// 核心调用链
ReflectiveFeign#newInstance()
  -> HardcodedTarget<?> target  // 包含name和url
  -> Factory#create()            // 创建InvocationHandler
    -> SynchronousMethodHandler#create()
      -> MethodHandler.invoke()  // 处理每次方法调用

// 实际发送请求
SynchronousMethodHandler.invoke() {
    // 1. 解析注解，构建RequestTemplate
    RequestTemplate template = this.buildTemplate.resolve(args);
    // 2. 负载均衡选择server（如果有Ribbon）
   IClientFEIGN client = this.client;
    // 3. 发送HTTP请求
    Response response = client.executeWithTemplate();
}
```

**4. 负载均衡集成：**
当Spring Cloud启用Ribbon时，Feign会使用`LoadBalancerFeignClient`包装原始Client：
```java
public class LoadBalancerFeignClient implements Client {
    @Override
    public Response execute(Request request, Options options) {
        // 1. 从Ribbon获取服务实例列表
        Server server = lbClientFactory.chooseServer(serviceName);
        // 2. 截取URL中的服务名，替换为具体host:port
        Request newRequest = transformRequest(request, server);
        // 3. 实际发送请求到目标服务器
        return delegate.execute(newRequest, options);
    }
}
```

**5. 核心流程总结：**
```
启动扫描@FeignClient接口
    ↓
为每个接口生成JDK动态代理（InvocationHandler）
    ↓
方法调用 → Contract解析注解 → 构建RequestTemplate
    ↓
LoadBalancerFeignClient → Ribbon选择服务实例
    ↓
ConvertedRequest → HTTP Client发送请求
    ↓
ResponseDecoder解码响应 → 返回Java对象
```

📖 来源：基于Feign开源代码 https://github.com/OpenFeign/feign 及 Spring Cloud官方文档整理

**题目6：Kafka — Rebalance（再平衡）机制是什么？触发条件有哪些？**

【答案】

**什么是Rebalance：**
Kafka的Consumer Group中的消费者实例变化（加入、离开、崩溃）时，会触发分区所有权从消费者A到消费者B的转移，这个过程叫Rebalance。Rebalance期间整个Consumer Group暂停消费，直到再平衡完成。

**触发条件：**
1. 消费者组成员发生变化（新增消费者、消费者崩溃、消费者主动取消订阅）
2. 消费者订阅的主题数量发生变化（正则订阅场景）
3. 分区数量发生变化（如管理员增加主题分区）
4. 消费者组超时未发送心跳，被Coordinator移除

**Rebalance流程：**
```
[Consumer1]  [Consumer2]  [Consumer3]
         \       |        /
          [Coordinator]
              
1. JoinGroup阶段：
   所有消费者向Coordinator发送JoinGroup请求
   Coordinator等待所有成员加入
   选举出Group Leader（第一个加入的消费者）
   
2. SyncGroup阶段：
   Leader消费者收集所有消费者元数据，生成 partitionAssignor 方案
   Leader将分配方案发送给Coordinator
   Coordinator下发给所有消费者
   
3. 开始正常消费
```

**分区分配策略（PartitionAssignor）：**
- **RangeAssignor**（默认）：按主题逐个分配，每个消费者获得连续的分区
- **RoundRobinAssignor**：轮询所有分区，均匀分配
- **StickyAssignor**：保持原有的分配关系不变，只在必要时调整

**Rebalance问题及优化：**

1. **Session.timeout vs Heartbeat.interval**：
   - `session.timeout.ms`：超过此时间无心跳，消费者被踢出（默认10s）
   - `heartbeat.interval.ms`：心跳间隔（默认3s，应该是session.timeout的1/3）
   - `max.poll.interval.ms`：两次poll之间的最大间隔（默认5min）

2. **再平衡风暴**：大量消费者同时重启会导致频繁Rebalance
   解决：错峰重启，或使用静态成员（instance.id）避免Rebalance

3. **再平衡期间不可消费**：可以通过设置`partition.assignment.strategy`为Sticky减少影响

**题目7：RocketMQ — 事务消息的实现原理是什么？如何保证本地事务与消息发送的一致性？**

【答案】

**事务消息的核心问题：**
普通消息发送后无法回滚。如果先扣款后发消息，消息发送失败导致数据不一致；如果先发消息后扣款，消费者会读到未扣款的消息。

**RocketMQ事务消息的两阶段提交：**

**第一阶段：发送半消息（Half Message）**
```java
// 生产者发送半消息（对消费者不可见）
TransactionMQProducer producer = new TransactionMQProducer("producer_group");
producer.sendMessageInTransaction(msg, new LocalTransactionExecuter() {
    @Override
    public LocalTransactionState executeLocalTransactionBranch(Message msg, Object arg) {
        // 执行本地业务逻辑（扣款等）
        boolean success = accountService.deduct(userId, amount);
        return success ? LocalTransactionState.COMMIT_MESSAGE 
                       : LocalTransactionState.ROLLBACK_MESSAGE;
    }
}, null);
```

**半消息的特点：**
- 存储在RMQ_SYS_TRANS_HALF_TOPIC主题中
- 对消费者不可见（通过索引机制实现）
- 如果一阶段超时未响应，会回查事务状态

**第二阶段：提交/回滚**
- 本地事务执行成功后，返回COMMIT_MESSAGE，Broker将半消息从HALF_TOPIC复制到真正的Topic
- 本地事务失败，返回ROLLBACK_MESSAGE，半消息被标记为rollback，不再处理

**TransactionListener回查机制：**
如果生产者崩溃或网络问题，Broker无法确定事务状态，会主动回查：
```java
public interface TransactionListener {
    // 执行本地事务
    LocalTransactionState executeLocalTransaction(Message msg, Object arg);
    // Broker回查本地事务状态
    LocalTransactionState checkLocalTransaction(MessageExt msg);
}
```

**完整流程：**
```
1. 生产者发送半消息（Half Message）到Broker ✓
2. Broker响应写入成功
3. 执行本地事务（扣款）  ✓
4. 提交事务（COMMIT_MESSAGE）
5. Broker将消息从HALF_TOPIC复制到真实Topic
6. 消费者订阅并消费

异常场景：
- 步骤4失败（网络超时）→ Broker回查 → 生产者返回COMMIT/ROLLBACK
- 生产者宕机 → Broker回查 → 根据本地事务状态决定提交或回滚
```

**与Kafka事务的区别：**
Kafka事务是针对Producer端的多分区原子性写入，RocketMQ事务是针对"本地事务+消息"的一致性保证。

**题目8：Spring基础 — Spring框架是如何通过三级缓存解决循环依赖的？**

【答案】

**什么是循环依赖：**
A依赖B，B依赖A，形成闭环：
```java
@Service class A { @Autowired B b; }
@Service class B { @Autowired A a; }
```

**三级缓存原理：**
Spring使用三个Map（singletonObjects、earlySingletonObjects、singletonFactories）来解决循环依赖：

```java
// DefaultSingletonBeanRegistry.java
// 一级缓存：完整创建好的单例bean（可直接使用）
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

// 二级缓存：提前暴露的bean（已实例化但未完成属性填充和初始化）
private final Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>(16);

// 三级缓存：Bean工厂对象（用于创建代理）
private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);
```

**解决循环依赖流程（A、B互相依赖）：**
```
getBean(A):
  1. 一级缓存没有，二级也没有，调用getSingleton从三级缓存获取
  2. 三级缓存也没有，创建A的ObjectFactory并存入三级缓存
  3. A执行属性注入，发现依赖B，调用getBean(B)
  
getBean(B):
  4. B的一级缓存没有，二级也没有
  5. B尝试从三级缓存获取A（因为A已开始创建）
  6. 三级缓存找到了A的ObjectFactory，调用它得到A的早期引用
  7. 将A从三级缓存移到二级缓存（earlySingletonObjects）
  8. B完成属性注入和初始化，将B存入一级缓存
  9. 返回B给A，完成A的属性注入
  
getBean(A)继续：
  10. A注入B成功（此时B已完成）
  11. A完成初始化，将A从二级缓存移到一级缓存
```

**为什么需要三级缓存？**
- 一级缓存直接可用，二级缓存存放早期引用
- 三级缓存存放ObjectFactory，用于在需要时创建代理对象（延迟代理创建）

**哪些情况无法解决：**
- 构造函数注入的循环依赖（Spring无法提前暴露未构造完成的对象）
- 多例Bean的循环依赖（prototype作用域每次创建新实例）
- `@Async`注解的Bean通过动态代理会打破二级缓存的假设

📖 来源：基于Spring Framework源码 org.springframework.beans.factory.support.DefaultSingletonBeanRegistry 整理

**题目9：AI Agent — 什么是RAG（检索增强生成）？它的核心工作流程是怎样的？**

【答案】

**RAG是什么：**
RAG（Retrieval-Augmented Generation，检索增强生成）是一种结合检索系统和LLM大模型的技术架构。RAG通过从外部知识库检索相关文档，然后将这些文档作为上下文（Context）提供给LLM，让LLM基于真实、最新的外部知识生成回答，而不是仅依赖训练数据。

**为什么需要RAG：**
1. LLM知识有截止日期，无法回答最新问题
2. LLM可能产生"幻觉"（一本正经地胡说八道）
3. 企业私有数据不在LLM训练集中
4. RAG提供可解释性——答案来源于哪个文档一目了然

**RAG核心工作流程：**

**第一阶段：数据准备（离线）**
```
文档 → 分块(Chunking) → 向量化(Embedding) → 存储到向量数据库
```

关键步骤：
- **分块（Chunking）**：将长文档切分成小块（如每500字符或按段落），块太小丢失上下文，块太大增加检索噪声
- **向量化（Embedding）**：使用Embedding模型（如text2vec、bge）将文本块转换为向量
- **向量数据库**：存储向量和原始文本，典型数据库包括Milvus、Pinecone、Chroma

**第二阶段：检索与生成（在线）**
```
用户问题 → 向量化用户问题 → 向量数据库检索 Top-K 相关文档 → 构建Prompt → LLM生成回答
```

1. **Query Encoding**：用户问题同样通过Embedding模型转换为向量
2. **Retrieval**：在向量数据库中做相似度搜索（余弦相似度/点积），返回Top-K最相关文档块
3. **Rerank**：可选，对检索结果重排序，提升相关性
4. **Prompt构建**：将用户问题和检索到的文档块组合成Prompt：
   ```
   已知信息：
   {retrieved_docs}
   
   基于以上信息，请回答：{user_question}
   ```
5. **LLM Generation**：LLM根据Prompt和检索内容生成答案

**RAG优化策略：**
- **Hybrid Search**：同时做关键词搜索（如BM25）和向量搜索，结合两者优点
- **Query改写**：将用户口语化问题改写为更适合检索的形式
- **Parent Document Retrieval**：先检索小chunk，再召回其所属的大文档块保证完整上下文
- **Self-RAG**：让LLM自己判断检索是否必要

**题目10：算法 — 用Java实现一个算法：给定一个数组，找出其中第K大的元素（要求时间复杂度O(n)）**

【答案】

**题目要求：** 在未排序的数组中找到第K大的元素，要求平均时间复杂度O(n)。

**思路：快速选择算法（Quick Select）**

快速选择算法是快速排序的变体，平均时间复杂度O(n)，最坏O(n²)。

**核心思想：** 利用快速排序的partition过程，每次 partition 后基准元素的位置是最终排序后的位置。如果基准位置 == n-k，则找到答案；如果基准位置 > n-k，则在左半部分继续找；如果基准位置 < n-k，则在右半部分继续找。

**Java实现：**

```java
public class QuickSelect {
    
    public int findKthLargest(int[] nums, int k) {
        // 第K大 = 第(n-k+1)小，所以找第 n-k 小
        return quickSelect(nums, 0, nums.length - 1, nums.length - k);
    }
    
    private int quickSelect(int[] nums, int left, int right, int targetIndex) {
        if (left == right) {
            return nums[left];
        }
        
        // partition 返回基准元素最终位置
        int pivotIndex = partition(nums, left, right);
        
        if (pivotIndex == targetIndex) {
            return nums[pivotIndex];
        } else if (pivotIndex < targetIndex) {
            // 目标在右半部分
            return quickSelect(nums, pivotIndex + 1, right, targetIndex);
        } else {
            // 目标在左半部分
            return quickSelect(nums, left, pivotIndex - 1, targetIndex);
        }
    }
    
    private int partition(int[] nums, int left, int right) {
        int pivot = nums[right];  // 选择最右元素作为基准
        int i = left;  // i是小于基准元素的区域的右边界
        
        for (int j = left; j < right; j++) {
            if (nums[j] <= pivot) {
                // 交换 nums[i] 和 nums[j]
                swap(nums, i, j);
                i++;
            }
        }
        // 将基准元素放到正确位置
        swap(nums, i, right);
        return i;
    }
    
    private void swap(int[] nums, int i, int j) {
        int temp = nums[i];
        nums[i] = nums[j];
        nums[j] = temp;
    }
    
    public static void main(String[] args) {
        QuickSelect qs = new QuickSelect();
        int[] nums = {3, 2, 1, 5, 6, 4};
        System.out.println(qs.findKthLargest(nums, 2));  // 输出5（第2大的元素）
        
        int[] nums2 = {3, 2, 3, 1, 2, 4, 5, 5, 6};
        System.out.println(qs.findKthLargest(nums2, 4));  // 输出4（第4大的元素）
    }
}
```

**复杂度分析：**
- **时间复杂度**：平均 O(n)，每次 partition 将数组分成两半，共 O(n + n/2 + n/4 + ...) = O(2n) = O(n)
- **最坏 O(n²)**：当数组有序且每次partition选择最大/最小值时
- **空间复杂度**：O(log n)（递归栈深度），或 O(1)（如果用迭代）

**优化 — BFPRT算法（最坏情况也是O(n)）：**
通过精心选择pivot（用中位数的中位数），保证每次partition都是均匀划分，但实现较复杂。Java标准库`Arrays.sort()`使用三路切分快速排序，对重复元素效率高。

**Top-K问题的通用解法对比：**

| 方法 | 时间复杂度 | 空间复杂度 | 适用场景 |
|------|-----------|-----------|---------|
| 排序 | O(n log n) | O(1) | 数据量小 |
| 快速选择 | O(n) avg, O(n²) worst | O(log n) | 只需找第K大 |
| 堆排序 | O(n log k) | O(k) | 海量数据找Top-K |
