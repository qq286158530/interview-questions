## 题目一：Redis 是单线程还是多线程？为什么 Redis 能达到这么高的 QPS？

### 答案

**Redis 的线程模型并非纯单线程，而是采用了「主线程 + 后台线程」的设计：**

### 1. Redis 6.0 之前的核心模块（单线程）
- **网络 I/O 和命令执行**全部在主线程中完成，采用 I/O 多路复用（epoll/select/kqueue）监听多个 socket 连接
- 每次从 socket 读取请求、解析命令、执行命令、返回结果，全部串行执行，无需加锁

### 2. Redis 6.0 开始的多线程（I/O 线程化）
- **网络 I/O 读写**可以交给多个 I/O 线程并行处理（`io-threads-do-reads yes`）
- **命令执行仍然是单线程**的，保证数据操作的原子性
- 多线程仅用于加速网络层的数据读写，不改变核心执行逻辑

### 3. 后台线程（异步任务）
Redis 还有一些后台线程处理耗时操作：
- `bio_close_file`：异步关闭大文件
- `bio_aof_fsync`：异步 AOF 刷盘
- `bio_lazy_free`：异步释放大内存对象

### 为什么 Redis 这么快？

| 关键因素 | 说明 |
|---------|------|
| **内存操作** | 数据全部在内存中，访问速度比磁盘快几个数量级 |
| **单线程无锁** | 避免了线程切换和锁竞争的开销 |
| **I/O 多路复用** | 一个线程高效管理成千上万个并发连接 |
| **高效数据结构** | SDS、跳表、压缩列表等针对性能和内存优化 |
| **纯哈希表 O(1)** | key-value 查找复杂度为 O(1)，不会随数据量退化 |

**常见误解澄清**：Redis 单线程指的是「命令执行」单线程，不是「整个进程」单线程。即使在单线程模式下，Redis QPS 也能轻松达到 10w+。

## 题目二：Redis 分布式锁如何实现？有什么常见问题？

### 答案

### 1. 基于 SETNX 实现分布式锁（基础版）

```bash
# 加锁
SET lock_key unique_value NX PX 30000

# 释放锁（必须匹配 value，防止误删他人的锁）
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

- `NX`：只有 key 不存在时才设置成功
- `PX 30000`：30秒自动过期，防止进程崩溃无法释放锁
- `unique_value`：用 UUID 或时间戳，防止误删

### 2. Redisson 分布式锁（生产级方案）

Redisson 封装了更完善的分布式锁实现：

- **可重入锁**（`RLock`）：同一线程可多次获取锁，内部用计数器实现
- **自动续期**（watchdog）：锁快过期时自动续期，防止任务未完成锁就释放
- **公平锁**：按加锁顺序排队
- **读写锁**：`RReadWriteLock`

```java
RLock lock = redisson.getLock("order_lock");
try {
    lock.lock(30, TimeUnit.SECONDS);
    // 业务逻辑
} finally {
    lock.unlock();
}
```

### 3. 常见问题与解决方案

| 问题 | 说明 | 解决方案 |
|------|------|---------|
| **锁误删** | 业务超时，锁自动释放后删了别人的锁 | 用 Lua 脚本匹配 value 再删除 |
| **锁不可重入** | 同一线程无法多次获取 | 用计数器实现可重入锁 |
| **主从切换丢锁** | 主节点加锁后宕机，从节点上位，锁丢失 | 使用 RedLock（多节点多数投票） |
| **锁过期但任务未完成** | 自动过期导致并发 | watchdog 自动续期机制 |

### 4. RedLock 算法

为解决主从架构下锁丢失的问题，Redis 官方提出 RedLock：

- 向 N（N ≥ 3）个独立 Redis 节点加锁
- 超过半数节点加锁成功才认为获取成功
- 失败后向所有节点释放锁
- 代价：实现复杂，延迟增加，通常推荐使用单节点 + 自动续期即可满足大多数场景

## 题目三：Redis 的 Zset 底层实现是什么？为什么要用跳表而不是平衡树？

### 答案

### 1. Zset 的底层数据结构

Redis Zset（有序集合）的底层实现根据数据量不同采用两种结构：

**① 数据量小时：压缩列表（ziplist）**
- 当元素数量 < `zset-max-ziplist-entries`（默认 128）时使用
- 所有元素连续存储，节省内存
- 插入/删除需要移动后续元素，复杂度 O(N)

**② 数据量大时：跳表 + 哈希表（skiplist + dict）**
- 当元素数量 ≥ 128 时自动转换为跳表
- 跳表：支持 O(log N) 的范围查询和排序
- 哈希表：支持 O(1) 的单元素查找（value → score）

### 2. 跳表（Skip List）原理

跳表是一种多层链表结构，从上到下索引逐渐变密：

```
Level 2:  ----1-----------------8------
Level 1:  --1----4--------8----9------
Level 0:  1-3--4-5--6--7--8--9--10--
```

- **查找**：从最高层开始，向右找到小于目标的最大节点，逐层下降
- **时间复杂度**：平均 O(log N)，最坏 O(N)
- **空间复杂度**：O(N log N)，每层索引约为下层的 1/2

### 3. 为什么用跳表而不是平衡树（AVL/红黑树）？

| 对比维度 | 跳表 | 平衡树（AVL/红黑树） |
|---------|------|-------------------|
| 查询复杂度 | O(log N) | O(log N) |
| 范围查询 | 天然支持（链表遍历） | 需要中序遍历，不友好 |
| 插入/删除 | 只需局部修改指针 | 可能触发大量旋转 |
| 实现复杂度 | 简单（只需随机层数） | 复杂（平衡条件严格） |
| 内存占用 | 略高（多层索引） | 较低（每个节点1个指针） |
| 有序性 | 链表天然有序 | 严格有序 |
| Redis 定制 | 易扩展（增加 maxLevel） | 难以定制 |

**Redis 选择跳表的核心原因**：
1. **范围查询友好**：Zset 需要按分数区间查询（如 ZRANGEBYSCORE），跳表的链表结构天然支持 O(log N) 范围遍历，平衡树需要中序遍历，不够高效
2. **实现简单**：跳表比红黑树容易实现，减少出 bug 的可能
3. **遍历友好**：跳表从高层往低层遍历，可以顺序访问，平衡树是 pointer-based，无法高效顺序遍历

Redis 作者 antirez 在 Hacker News 中解释过：跳表实现简单，调试容易，空间换时间，是"足够好"的方案。

**来源：** https://xiaolincoding.com/redis/base/redis_interview.html

## 题目四：Redis 的 Pipeline 是什么？与事务（MULTI/EXEC）有什么区别？

### 答案

### 1. Pipeline（管道）

**Pipeline 是客户端-side 的批量打包技术**，将多个 Redis 命令一次性发送到服务器，服务器依次执行后批量返回结果。

```java
// Jedis Pipeline 示例
Pipeline pipeline = jedis.pipelined();
for (int i = 0; i < 10000; i++) {
    pipeline.set("key" + i, "value" + i);
    pipeline.get("key" + i);
}
List<Object> results = pipeline.exec();
```

**原理**：客户端将命令缓存在本地缓冲区，达到一定数量或时间后一次性发送，减少网络往返次数（RTT）。

**性能提升**：从 O(N) × RTT 降为 O(1) × RTT（仅一次往返）

### 2. Redis 事务（MULTI/EXEC）

Redis 事务提供**服务端-side 的原子性保证**：

```bash
MULTI
SET name "Alice"
INCR age
HSET user:1 city "Beijing"
EXEC
```

- `MULTI` 开启事务，后续命令进入队列
- `EXEC` 原子性执行队列中所有命令
- `DISCARD` 清空命令队列，取消事务

**特性**：
- **原子性**：EXEC 执行期间不会执行其他客户端命令
- **无回滚**：Redis 事务不支持回滚（因为 Redis 设计哲学是简单高性能）
- **无隔离**：执行过程中其他命令可以穿插执行（不是严格隔离）

### 3. Pipeline 与事务的关键区别

| 特性 | Pipeline | 事务（MULTI/EXEC） |
|------|---------|-------------------|
| **原子性** | 无，多命令独立执行 | 有，EXEC 原子执行 |
| **回滚** | 不支持 | 不支持（Redis 风格） |
| **一致性** | 命令可能穿插执行 | 命令执行期间不被打断 |
| **网络优化** | 批量打包，减少 RTT | 不改变命令数量，只是队列缓冲 |
| ** WATCH** | 不支持 | 支持乐观锁（CAS） |
| **Lua 脚本** | 无特殊保证 | 可替代事务，实现更复杂的原子操作 |

### 4. Watch 命令（乐观锁）

```bash
WATCH account:1
balance = GET account:1
if balance >= 100:
    MULTI
    DECRBY account:1 100
    INCRBY account:2 100
    EXEC     # 如果 watch 的 key 在 WATCH 和 EXEC 之间被修改，事务失败
else:
    UNWATCH
```

- WATCH 监控一个或多个 key，如果在事务执行前被其他客户端修改，事务自动失败
- 失败后需要重试（适合低竞争场景）

### 5. 最佳实践

- **Pipeline**：批量读取/写入，同一批次内命令无原子性要求（如批量导入数据）
- **事务**：需要原子执行的命令（如扣款），配合 WATCH 实现乐观锁
- **Lua 脚本**：需要复杂原子逻辑，且希望减少网络往返（脚本内多个命令一次发送执行）

## 题目五：Redis 集群的槽（slot）迁移过程中如何保证数据一致性？

### 答案

### 1. Redis Cluster 槽分配原理

Redis Cluster 将整个 keyspace 划分为 **16384 个槽（slot）**，每个节点负责一部分槽：

```
节点A：slot 0 - 5460
节点B：slot 5461 - 10922
节点C：slot 10923 - 16383
```

客户端连接任意节点，节点计算 `CRC16(key) % 16384` 得出槽编号，如果该槽不在本节点，则返回 `MOVED` 错误，客户端自动重定向到正确节点。

### 2. 槽迁移过程

当需要扩缩容时，会进行槽迁移（slot migration），分三步：

**Step 1：目标节点准备接收（ IMPORTING）**
```
CLUSTER SETSLOT <slot> IMPORTING <source_node_id>
```
告诉目标节点：这个槽的数据将从 source 迁入，先做好准备

**Step 2：源节点准备迁出（MIGRATING）**
```
CLUSTER SETSLOT <slot> MIGRATING <target_node_id>
```
告诉源节点：这个槽要迁到 target，源节点对迁移中的 key 做特殊处理

**Step 3：实际迁移数据**
```
CLUSTER GETKEYSINSLOT <slot> <count>   # 批量获取该槽的 key
MIGRATE <target_ip> <target_port> "" 0 <timeout> KEYS <keys>  # 迁移 key
```

源节点对每个 key 发送 `MIGRATE` 命令，将数据序列化发送到目标节点，目标节点确认后源节点删除该 key。

### 3. 迁移过程中的客户端访问

**这是面试最关心的问题**：迁移过程中，客户端访问正在迁移的槽会发生什么？

| 场景 | 源节点处理 | 目标节点处理 |
|------|----------|------------|
| key 还在源节点 | 正常处理 | 返回 `ASKING`，告诉客户端去源节点查 |
| key 已迁移到目标节点 | 返回 `ASK` 重定向 | 正常处理 |
| key 不存在 | 返回 nil | 返回 nil |

**ASK 重定向**：客户端收到 `ASK <slot> <node>` 后，先发 `ASKING` 命令（一次性标记），再发原本要执行的命令到目标节点。

**为什么不是 MOVED**：MOVED 是永久重定向（更新路由表），ASK 是临时重定向（不更新路由表），因为槽迁移还未完全结束，路由还没稳定。

### 4. 迁移一致性保证

Redis Cluster 在槽迁移期间通过以下机制保证数据一致性：

1. **ASK 语义**：确保迁移中请求打到正确的节点
2. **MIGRATE 原子性**：使用 `MIGRATE` 的 `COPY` 或 `REPLACE` 模式控制迁移原子性
3. **旧节点保留过期 key**：迁移完成后，源节点在一定时间内仍会保留已迁移 key 的元信息，直到目标节点确认完成
4. **禁止部分写入**：cluster-require-full-coverage 配置为 yes 时，如果存在槽不可用，整个集群拒绝写入（保证数据完整性）

### 5. 迁移超时处理

- 如果 `MIGRATE` 超时，目标节点会放弃本次迁移
- 可以用 `CLUSTER SETSLOT <slot> STABLE` 中止迁移
- 大量 key 迁移时建议使用 `redis-cli --cluster reshard` 工具，自动分批迁移（每次迁移少量 key），减少对线上影响
