## 题目一：Redis 有哪些数据类型？底层是怎么实现的？

### 答案

Redis 提供了丰富的数据类型，常见的有五种：**String（字符串）、Hash（哈希）、List（列表）、Set（集合）、Zset（有序集合）**。随着版本更新，又支持了 BitMap、HyperLogLog、GEO、Stream 四种数据类型。

### 五种数据类型的应用场景

| 类型 | 应用场景 |
|------|----------|
| String | 缓存对象、常规计数、分布式锁、共享 session 信息 |
| List | 消息队列（需自己实现全局唯一 ID）、文章列表、点赞列表 |
| Hash | 缓存对象、购物车 |
| Set | 聚合计算（并集、交集、差集）：点赞、共同关注、抽奖活动 |
| Zset | 排序场景：排行榜、电话姓名排序 |

### 底层数据结构实现

- **String**：底层实现为 SDS（简单动态字符串），相比 C 字符串支持二进制数据、O(1) 获取长度、安全拼接不缓冲区溢出。
- **List**：Redis 3.2 之前用双向链表或压缩列表，之后统一用 quicklist。
- **Hash**：底层用压缩列表（元素少且小）或哈希表。
- **Set**：底层用整数集合（全是整数且元素少）或哈希表。
- **Zset**：底层用压缩列表（元素少且小）或跳表（Sorted Set）。

## 题目二：Redis 是单线程吗？为什么单线程还这么快？

### 答案

**Redis 单线程指的是「接收客户端请求→解析请求→执行数据读写→返回响应」这一流程由主线程完成。** 但 Redis 程序本身并非完全单线程，4.0 后有后台线程处理 lazy free 等任务；6.0 后有 I/O 多线程处理网络 I/O。

### Redis 为什么这么快？

1. **内存操作**：大部分操作在内存中完成，采用高效数据结构，瓶颈是内存或网络而非 CPU。
2. **单线程无竞争**：避免多线程切换开销和锁竞争，不会有死锁问题。
3. **I/O 多路复用**：使用 epoll 机制，一个线程处理多个 Socket 请求。

### Redis 6.0 之后的变化

6.0 引入了多线程 I/O 来处理网络请求（默认 3 个 I/O 线程），但**命令执行仍是单线程**。官方表示多线程 I/O 性能提升至少一倍。

## 题目三：Redis 持久化机制是什么？RDB 和 AOF 有什么区别？

### 答案

Redis 有三种持久化方式：**RDB（快照）、AOF（日志）、混合持久化**。

### RDB 快照

- 将某一时刻的内存数据以二进制格式写入磁盘。
- `bgsave` 命令会 fork 子进程生成 RDB 文件，主线程不受阻塞（利用写时复制）。
- 恢复速度快，但快照频率难把握，频率太低丢失数据多，太高影响性能。

### AOF 日志

- 每执行一条写操作命令，就将命令追加到文件末尾。
- 三种写回策略：
  - `Always`：每次执行完同步写回，性能差但最安全。
  - `Everysec`：每秒写回一次，推荐配置。
  - `No`：由操作系统决定何时写回。

### 混合持久化（Redis 4.0+）

- 重写 AOF 时，先以 RDB 格式写入全量数据，再以 AOF 格式追加增量命令。
- 兼具快速启动和低数据丢失风险。

### RDB vs AOF 对比

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件体积 | 小（二进制） | 大（命令文本） |
| 恢复速度 | 快 | 慢（需逐条执行） |
| 数据安全性 | 可能丢失较多数据 | 可配置（always 最安全） |
| 性能影响 | fork 子进程，轻微阻塞 | 持续写操作，略高 |

## 题目四：什么是缓存雪崩、击穿、穿透？如何解决？

### 答案

### 缓存雪崩

**大量缓存数据同时过期，导致大量请求直接打到数据库。**

**解决方案：**
- 给过期时间加上随机值，避免同时过期。
- 使用互斥锁或队列控制数据库写请求。
- 构建多级缓存（本地缓存 + Redis）。
- 定时更新缓存热点数据。

### 缓存击穿

**某个热点 key 过期，瞬间大量请求打到数据库。**

**解决方案：**
- 使用互斥锁或 Redisson 分布式锁，只允许一个线程重建缓存。
- 热点数据永不过期（逻辑过期，需额外逻辑处理）。
- 提前预热热点数据。

### 缓存穿透

**请求的数据既不在缓存也不在数据库，大量无效请求穿透到数据库。**

**解决方案：**
- 参数校验，拦截非法请求。
- 布隆过滤器（Bloom Filter）判断数据是否存在。
- 缓存空值或默认值（设置短过期时间）。
- 记录请求特征，拉入黑名单。

## 题目五：Redis 主从复制、哨兵、Cluster 集群有什么区别？如何保证高可用？

### 答案

### 主从复制

- **一主多从**模式，主节点可读可写，从节点只读。
- 主节点新写操作同步到从节点，异步进行，**无法保证强一致性**。
- 优点：读写分离提升吞吐量；缺点：主节点挂了需要手动恢复。

### 哨兵模式（Sentinel）

- 监控主从节点健康状态。
- 主节点挂了时，自动从从节点选举新主节点。
- 客户端连接哨兵获取主节点地址，实现故障自动转移。

### Redis Cluster（切片集群）

- **数据分片**：16384 个哈希槽，键通过 CRC16 取模映射到槽。
- 每个主节点负责一部分槽，支持自动故障转移。
- 优点：水平扩展能力强；缺点：不支持跨槽操作（需用 hash tag）。

### 高可用对比

| 方案 | 适用场景 | 故障自动切换 |
|------|----------|-------------|
| 主从复制 | 读多写少，数据备份 | ❌ 需手动 |
| 哨兵 | 需要自动故障转移 | ✅ 自动 |
| Cluster | 数据量大，需分片 | ✅ 自动 |

### 脑裂问题

集群脑裂会导致数据丢失，可通过配置 `min-slaves-to-write` 和 `min-slaves-max-lag` 限制主节点在从节点不健康时禁止写入。
