## 题目一：MySQL 查询执行流程是怎样的？如何定位慢查询？

### 参考答案

MySQL 查询执行的主要流程：

1. **连接器**：客户端建立 TCP 连接，验证用户身份
2. **查询缓存**（MySQL 8.0 前）：命中则直接返回结果；8.0 后已移除
3. **解析器**：词法分析 + 语法分析，生成解析树
4. **预处理器**：权限检查、消解表/列名
5. **优化器**：生成执行计划（基于成本模型），选择索引、决定 JOIN 顺序
6. **执行器**：调用存储引擎接口，逐行获取数据
7. **存储引擎**：InnoDB/MyISAM 按执行计划读写数据页

**定位慢查询的方法：**

- `SHOW FULL PROCESSLIST` — 查看当前连接及执行状态
- 开启慢查询日志：`slow_query_log=1`，设置 `long_query_time`（如 1 秒）
- 分析慢查询日志工具：`mysqldumpslow -t 10 /path/slow.log`
- 使用 `EXPLAIN` / `EXPLAIN ANALYZE` 分析执行计划
- 开启 `performance_schema`：监控语句执行耗时统计

## 题目二：什么是回表查询？如何通过覆盖索引避免回表？

### 参考答案

**回表查询**：当查询的字段无法通过索引直接获取时，需要先通过索引找到主键值，再根据主键去主键索引（B+树）中查找完整的行记录，这个过程叫**回表**。

例如：
```sql
-- 假设 idx_name 是普通索引，只包含 name 字段
SELECT name, age FROM user WHERE name = '张三';
```
执行计划 `type` 为 `ref`，但 `Extra` 中不会出现 `Using index`，说明发生了回表。

**覆盖索引**：将查询所需字段全部放入索引中，查询时直接从索引返回，无需回表。

```sql
-- 创建覆盖索引（包含查询的所有列）
CREATE INDEX idx_name_age ON user(name, age);

-- 此时查询直接走索引，无需回表
SELECT name, age FROM user WHERE name = '张三';
-- EXPLAIN Extra 列显示 Using index
```

**注意事项**：
- 覆盖索引不适用于 `SELECT *`
- 索引不是越多越好，维护成本高
- 复合索引遵循最左前缀原则

## 题目三：InnoDB 与 MyISAM 存储引擎有哪些核心区别？各自适用场景是什么？

### 参考答案

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| **事务支持** | 支持 ACID 事务 | 不支持事务 |
| **锁粒度** | 行级锁 + 间隙锁 | 表级锁 |
| **并发能力** | 高（行锁） | 低（表锁） |
| **外键约束** | 支持 | 不支持 |
| **崩溃恢复** | 自动恢复（redo log） | 需手动修复（myisamchk） |
| **索引结构** | 聚簇索引（数据在索引叶子节点） | 非聚簇索引（索引与数据分离） |
| **COUNT(\*)** | 全表扫描（无专门计数器） | 维护了行数计数器 |
| **全文索引** | MySQL 5.6+ 支持 | 原生支持 |

**适用场景**：
- **InnoDB**：OLTP 业务、需要事务支持、并发写入多、数据可靠性要求高
- **MyISAM**：只读/极少修改的日志表、全文搜索需求（老系统迁移场景）

## 题目四：如何优化大表分页查询（LIMIT offset, N）？

### 参考答案

大表分页问题是经典的性能瓶颈。当 `offset` 很大时，MySQL 需要先扫描并丢弃前 offset 条记录，效率极低。

**优化方案一：延迟关联**
```sql
-- 原始慢查询
SELECT id, title FROM article ORDER BY id LIMIT 1000000, 10;

-- 优化：先用索引定位 ID，再关联取字段
SELECT a.id, a.title
FROM article a
INNER JOIN (
    SELECT id FROM article ORDER BY id LIMIT 1000000, 10
) t ON a.id = t.id;
```

**优化方案二：游标分页（基于上一页最后一条 ID）**
```sql
-- 第一页
SELECT id, title FROM article ORDER BY id LIMIT 10;
-- 下一页，传入上一页最后一条的 id
SELECT id, title FROM article
WHERE id > #{last_id}
ORDER BY id LIMIT 10;
```
性能稳定，复杂度 O(1)，适合无限滚动场景。

**优化方案三：覆盖索引 + 延迟关联**
```sql
-- 确保 ORDER BY 字段和 WHERE 条件都在索引中
CREATE INDEX idx_id_title ON article(id, title);
```

**通用原则**：
- 避免跳页查询，业务上引导用户使用游标分页
- 大 OFFSET 分页是典型的反模式

## 题目五：Redis 缓存与 MySQL 如何配合使用？缓存穿透、缓存击穿、缓存雪崩是什么？如何解决？

### 参考答案

**缓存模式**：
- **Cache-Aside（旁路缓存）**：读时先查缓存，miss 再查 DB 并写入缓存；写时先更新 DB，再删除缓存
- **Read-Through / Write-Through**：由缓存层自行处理 DB 交互，对应用透明

**缓存三大的问题及解决方案**：

### 缓存穿透（查询不存在的数据）
大量请求查询 DB 和缓存中都不存在的 key，穿透到 DB。

**解决**：
- 布隆过滤器（存在性判断，过滤掉不存在的 key）
- 缓存空值（NULL）或默认值，设置较短 TTL
- 接口参数校验

### 缓存击穿（热点 key 过期）
某个热点 key 过期瞬间，大量并发请求击穿到 DB。

**解决**：
- 互斥锁（SETNX）+ 双检（先查缓存，加锁后再次检查）
- 热点数据永不过期（被动过期 + 异步重建）
- 逻辑过期（不设 TTL，设置逻辑过期时间字段，本地处理过期逻辑）

### 缓存雪崩（大量 key 同时过期）
大量 key 在同一时间过期，导致 DB 压力骤增。

**解决**：
- 过期时间加随机偏移量（`TTL + random(0, 60)`）
- 服务熔断降级 / 请求限流
- Redis 持久化 + 高可用架构（RDB+AOF 主从）
- 预热热点数据到缓存

**缓存一致性问题**：推荐**先更新数据库，再删除缓存**（而不是更新缓存），配合消息队列或延迟双删处理并发问题。
