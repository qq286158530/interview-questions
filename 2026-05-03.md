## 题目1：ThreadLocal 内存泄漏是如何发生的？如何避免？

【答案】

### ThreadLocal 原理

每个线程内部都有一个 `ThreadLocalMap`（本质是定制化的 HashMap），key 是 ThreadLocal 实例的弱引用（`WeakReference<ThreadLocal<?>>`），value 是我们存入的强引用值。

```java
// Thread 类中的 ThreadLocal 存储
ThreadLocal.ThreadLocalMap threadLocals = null;

// ThreadLocal.set() 内部实现
public void set(T value) {
    Thread t = Thread.currentThread();
    ThreadLocalMap map = getMap(t);
    if (map != null)
        map.set(this, value); // this 就是 ThreadLocal 实例作为 key
    else
        createMap(t, value);
}
```

### 内存泄漏原因

内存泄漏的发生需要**同时满足两个条件**：

1. **key 是弱引用**：ThreadLocal 实例不再被任何强引用指向时，GC 会回收 key，导致 Entry 的 key 变为 `null`
2. **value 是强引用**：即使 key 被 GC 回收，value 仍然被 Entry 强引用，无法被 GC 回收

```java
static class Entry extends WeakReference<ThreadLocal<?>> {
    Object value; // 强引用
    Entry(ThreadLocal<?> k, Object v) {
        super(k); // key 是弱引用
        value = v;
    }
}
```

在线程池等线程复用场景下，线程生命周期很长，ThreadLocalMap 持续存在，导致 key 为 null 的 entry 无法被回收——即内存泄漏。

虽然 `get()`、`set()`、`remove()` 会主动清理 key 为 null 的 entry，但这种清理是被动的，不完全可靠。

### 为什么 key 设计为弱引用？

如果 key 是强引用，即使 ThreadLocal 不再被使用，只要线程存活，key 就不会被回收，导致 ThreadLocal 实例本身也无法被回收，造成更严重的内存泄漏。弱引用设计保证了：**ThreadLocal 实例失去外部强引用后可以被 GC 回收**，只是 value 可能泄漏。

### 如何避免？

**在每次使用完 ThreadLocal 后，务必调用 `remove()` 方法**，这是最安全的做法：

```java
try {
    threadLocal.set(value);
    // 业务逻辑
} finally {
    threadLocal.remove(); // 显式移除，防止内存泄漏
}
```

在线程池场景下，使用 `try-finally` 确保即使发生异常，`remove()` 也一定会执行。

## 题目2：Java 异常体系中，Checked Exception 和 Unchecked Exception 有什么区别？

【答案】

### 异常分类

Java 异常继承结构：
- `Throwable` → `Error`（系统级错误，无法处理）
- `Throwable` → `Exception`
  - `RuntimeException` 及其子类 → **Unchecked Exception**
  - 其他 Exception 子类（如 `IOException`、`SQLException`） → **Checked Exception**

### 核心区别

| 特征 | Checked Exception | Unchecked Exception |
|------|-----------------|---------------------|
| 编译检查 | 编译时强制处理，否则无法通过编译 | 编译时不检查，可以不处理 |
| 处理方式 | 必须通过 `try-catch` 或 `throws` 捕获/声明 | 可以不处理 |
| 设计意图 | 业务逻辑的一部分，调用方必须处理 | 通常是代码 Bug，应该暴露并修复 |

### 常见 Unchecked Exception（需熟记）

- `NullPointerException`（空指针）
- `IllegalArgumentException`（参数错误）
- `ArrayIndexOutOfBoundsException`（数组越界）
- `ClassCastException`（类型转换错误）
- `ArithmeticException`（算术错误，如除零）
- `UnsupportedOperationException`（不支持的操作）

### 最佳实践

**默认使用 Unchecked Exception**，只在以下情况使用 Checked Exception：该异常是业务逻辑的一部分，且**调用方必须处理**。

例如余额不足不是 Bug，而是正常业务分支，用 Checked Exception 强制调用者处理（如提示用户充值）。

把 Unchecked Exception 看作代码 Bug——对待 Bug 最好的方式是让它暴露出来，而不是用 try-catch 掩盖。

## 题目3：ArrayList 与 LinkedList 有何区别？分别在什么场景下使用？

【答案】

### 底层数据结构

| | ArrayList | LinkedList |
|---|---|---|
| 底层实现 | 动态 Object[] 数组 | 双向链表（JDK1.6 前是循环链表，JDK1.7 取消循环） |
| 内存连续性 | 连续内存，利于 CPU 缓存命中 | 不连续，通过指针串联 |
| 随机访问 | O(1)，通过下标直接计算地址 | O(n)，需要遍历查找 |
| 插入/删除 | O(n)，需要移动元素 | O(1)，只需要改变前后指针（已知位置时） |

### ArrayList 的特点

- 动态扩容：容量不足时扩容 1.5 倍（`newCapacity = oldCapacity + (oldCapacity >> 1)`）
- 扩容时需要**复制整个数组**，成本较高
- 查找快（O(1) 随机访问），增删慢（需要搬移元素）
- 适合读多写少的场景

### LinkedList 的特点

- 双向链表，每个节点存储 `item`、`prev`、`next`
- 插入/删除快（已知位置时 O(1)）
- 查找慢（O(n)，需要遍历）
- 内存开销比数组大（每个节点额外存储两个指针）
- 适合写多读少的场景

### 如何选用？

```java
// 需要快速随机访问（按下标访问）→ ArrayList
// 需要频繁在中间插入/删除 → LinkedList
// 需要当作栈/队列使用 → ArrayDeque（比 LinkedList 效率更高）
```

注意：除非需要在集合**中间**频繁插入删除，否则优先选择 ArrayList——现代 JVM 对连续内存的访问有很好的优化，链表节点的CPU缓存命中很差。

## 题目4：CompletableFuture 相比 Future 有哪些改进？为什么需要它？

【答案】

### Future 的局限性

1. **不支持异步任务编排组合**：多个 Future 之间无法串联、组合
2. **`get()` 方法是阻塞调用**：调用线程会一直等待结果
3. **无法手动完成**：无法提前设置结果，只能等待任务执行完毕

### CompletableFuture 的改进

`CompletableFuture` 同时实现了 `Future` 和 `CompletionStage` 接口，提供了：

1. **函数式编程能力**：支持 lambda 表达式、链式调用
2. **异步任务编排组合**：可以将多个异步任务串联成一个完整的链式调用
3. **非阻塞获取结果**：提供 `thenApply`、`thenAccept` 等回调方法

### 常见创建方式

```java
// 使用静态工厂方法（推荐自定义线程池）
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
    // 异步任务，返回 String
    return "result";
}, customExecutor);

// 基于 new 关键字
CompletableFuture<String> future = new CompletableFuture<>();
future.complete("手动设置结果"); // 主动完成
```

### 任务编排示例

```java
// 场景：获取用户信息 -> 获取商品详情 -> 获取物流信息（串行依赖）
CompletableFuture
    .supplyAsync(() -> getUserInfo(), executor)          // 第1步：获取用户
    .thenCompose(user -> getProductDetails(user))        // 第2步：依赖用户信息
    .thenCompose(product -> getLogistics(product))       // 第3步：依赖商品信息
    .thenAccept(result -> System.out.println(result));    // 第4步：处理结果

// 并行执行多个无依赖任务
CompletableFuture.allOf(
    CompletableFuture.supplyAsync(() -> getAds()),
    CompletableFuture.supplyAsync(() -> getRecommendations()),
    CompletableFuture.supplyAsync(() -> getHotSearch())
).join(); // 等待全部完成
```

### 与 Thread.start() 的区别

`Thread.start()` 是创建线程的底层方式，而 `CompletableFuture` 是封装好的异步编程抽象，提供了任务编排、组合、异常处理等高级能力，实际项目中**优先使用 CompletableFuture**。

## 题目5：synchronized 与 ReentrantLock 有何区别？

【答案】

### 核心区别

| 特性 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 类型 | 关键字，JVM 内置 | JDK 层面的类（java.util.concurrent） |
| 锁类型 | 非公平锁 | 支持公平/非公平（构造参数决定） |
| 灵活性 | 语义固定，不可中断 | 可中断、可超时 |
| 条件等待 | `Object.wait()/notify()` | `Condition.await()/signal()` |
| 锁释放 | 自动释放（代码块结束） | 必须手动 `unlock()`（建议放在 finally） |
| 多次加锁 | 可重入（同一个线程可以反复加锁） | 可重入（计数） |

### synchronized 的特点

```java
synchronized (obj) {
    // 原子操作
} // 自动释放锁
```
- **非公平锁**：无法指定公平策略
- **自动释放**：出了代码块自动释放锁，使用简单
- **可重入**：同一线程可重复获取，计数器递增

### ReentrantLock 的特点

```java
ReentrantLock lock = new ReentrantLock(true); // true = 公平锁
try {
    lock.lock();              // 获取锁
    // 原子操作
} finally {
    lock.unlock();            // 必须手动释放
}
```
- **支持公平锁**：等待时间最长的线程优先获取锁
- **可中断**：`lockInterruptibly()` 支持等待时被中断
- **可超时**：`tryLock(timeout)` 支持超时自动放弃
- **多个条件**：`newCondition()` 创建多个条件队列，精确控制线程等待

### 如何选型？

- **优先使用 synchronized**：JVM 内置，实现更简洁，JIT 编译器会做优化
- **需要高级特性时**（公平锁、可中断、超时、多条件队列）使用 ReentrantLock
- 实际项目中两者性能差异不大，现代 JVM 对 synchronized 优化很好（偏向锁、轻量级锁等）

## 题目6：Spring Boot 自动装配是如何实现的？

【答案】

### 核心原理

自动装配（Auto Configuration）是 Spring Boot 最核心的特性之一，核心是 `@EnableAutoConfiguration` 注解 + `spring.factories` 文件 + `@Conditional` 条件注解的组合。

### 执行流程

1. **Spring Boot 启动** → 加载 `@SpringBootApplication`（其中包含 `@EnableAutoConfiguration`）
2. **`@EnableAutoConfiguration`** 内部使用 `AutoConfigurationImportSelector`
3. `AutoConfigurationImportSelector` 读取 **所有 JAR 包中 `META-INF/spring.factories` 文件**
4. 文件中声明了需要自动注册的 `AutoConfiguration` 类
5. 每个 `AutoConfiguration` 类上标注了各种 `@Conditional` 注解
6. Spring 根据条件判断**是否真正注册**这些 Bean

### spring.factories 示例

```properties
# spring-boot-autoconfigure/META-INF/spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration,\
org.springframework.boot.autoconfigure.web.WebMvcAutoConfiguration,\
...
```

### @Conditional 条件注解

只有满足条件才注册 Bean，常见条件包括：

| 注解 | 条件 |
|------|------|
| `@ConditionalOnClass` | classpath 中存在指定类才注册 |
| `@ConditionalOnMissingBean` | 不存在指定 Bean 才注册 |
| `@ConditionalOnProperty` | 配置文件中存在指定配置才注册 |
| `@ConditionalOnBean` | 存在指定 Bean 才注册 |

### 一个典型的自动配置类

```java
@Configuration
@ConditionalOnClass(Tomcat.class)           // classpath 有 Tomcat 才生效
@ConditionalOnMissingBean(ServerProperties.class) // 没手动配置才生效
@EnableConfigurationProperties(ServerProperties.class)
public class TomcatAutoConfiguration {
    @Bean
    public TomcatServletWebServerFactory tomcatServletWebServerFactory(...) {
        // 创建 Tomcat 服务器
    }
}
```

### 手动替换自动配置

如果自动配置的 Bean 不符合需求，可以**手动定义同类型 Bean** 覆盖（因为 `@ConditionalOnMissingBean` 的存在）。

## 题目7：Spring Cloud Gateway 的路由转发原理是什么？

【答案】

### 核心概念

Spring Cloud Gateway 是基于 **Spring WebFlux** 实现的响应式网关，使用 **Netty** 异步非阻塞处理请求，核心组件：

- **Route**（路由）：目标 URL + 过滤器链
- **Predicate**（断言）：匹配请求条件（路径、时间、Header 等）
- **Filter**（过滤器）：在请求前后执行逻辑

### 请求处理流程

```
请求 → Route Predicate Handler Mapping（路由断言映射器）
       ↓ 匹配成功
Route（路由）
       ↓
Filter Web Handler（过滤器链）
       ↓ (pre 过滤器)
       → Proxy Service（调用下游服务）
       ↓ (post 过滤器)
响应
```

### 配置示例

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service          # 路由唯一标识
          uri: lb://user-service    # lb = 负载均衡，去注册中心找服务
          predicates:
            - Path=/api/user/**     # 路径匹配
            - After=2026-01-01T00:00:00+08:00  # 时间之后才生效
          filters:
            - StripPrefix=2         # 去掉前2个路径前缀
            - RequestRateLimiter=100 # 限流
```

### 核心原理

1. **Predicate 匹配**：根据配置的规则（路径、时间、Header等）判断请求是否符合条件
2. **Route 转发**：匹配成功后，请求被转发到 `uri` 指定的目标地址
3. **Filter 链**：分为 `pre`（转发前）和 `post`（转发后）两阶段，可做认证、日志、限流、熔断等
4. **负载均衡**：`lb://` 前缀会从注册中心（Nacos/Eureka）获取服务实例并负载均衡

### 与 Zuul 的区别

- Zuul 1.x 是同步阻塞（Servlet 模型），Gateway 是响应式异步（WebFlux + Netty）
- Gateway 性能比 Zuul 高 20%~50%（官方数据）
- Gateway 功能更丰富（内置限流、熔断、重试等），Zuul 需要配合 Hystrix

## 题目8：Kafka 如何保证消息的可靠性？

【答案】

### 消息丢失的三个环节

消息从生产到消费可能丢失的环节：
1. **生产者发送阶段** - 消息未成功发送到 Broker
2. **Broker 存储阶段** - Leader 副本崩溃，数据未同步到 Follower
3. **消费者消费阶段** - 消费了消息但 offset 未提交

### 1. 生产者端：acks 配置 + retries

```java
// 关键配置
properties.setProperty("acks", "all");       // 所有 ISR 副本都确认才算成功
properties.setProperty("retries", "3");      // 发送失败自动重试
properties.setProperty("enable.idempotence", "true"); // 开启幂等性，防止重复
```

- `acks=0`：发出去就当成功，高吞吐但可能丢数据
- `acks=1`：Leader 副本写入成功即确认，可能丢数据（Follower 未同步）
- `acks=all`（或 `-1`）：ISR 所有副本都写入成功才确认，最可靠

### 2. Broker 端：副本机制 + ISR

```properties
# 服务端配置
min.insync.replicas=2           # 最少 ISR 数量
replication.factor=3             # 分区副本数
unclean.leader.election.enable=false  # 不允许从非 ISR 中选 Leader
```

**ISR（In-Sync Replicas）**：与 Leader 保持同步的副本集合。只有在 ISR 中的副本才可能成为新的 Leader。

### 3. 消费者端：手动提交 offset

```java
// 正确做法：处理完消息后再提交 offset
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        process(record);                    // 处理业务
        consumer.commitSync();              // 处理完再提交
    }
}
```

不要使用自动提交（`enable.auto.commit=true`），否则可能消费了消息但 offset 没提交，导致重复消费或丢消息。

### 综合最佳实践

| 配置项 | 推荐值 |
|--------|--------|
| acks | all |
| retries | ≥ 3 |
| enable.idempotence | true |
| replication.factor | ≥ 3 |
| min.insync.replicas | ≥ 2 |
| consumer | 手动提交 offset |

## 题目9：RabbitMQ 消息确认机制（Confirm 和 Ack）是如何工作的？

【答案】

### 两种确认机制

RabbitMQ 有两套确认机制，分别对应生产者 → Broker 和 Broker → 消费者两个方向。

### 1. 生产者 Confirm（发布确认）

确认消息是否成功到达 Broker：

```java
// 开启 Confirm 模式
channel.confirmSelect();

channel.basicPublish("exchange", "routingKey", 
    MessageProperties.PERSISTENT_TEXT_PLAIN,
    message.getBytes());

// 等待 Broker 确认
try {
    channel.waitForConfirmsOrDie(5000);  // 5秒超时
    System.out.println("消息已确认");
} catch (Exception e) {
    // 确认失败，消息可能丢失，需要重发
    System.out.println("消息确认失败");
}
```

**三种确认模式**：
- **同步单条确认**：每发一条等待 `waitForConfirmsOrDie()`，性能差
- **批量确认**：发一批后调用 `waitForConfirmsOrDie()`，性能中等
- **异步 Confirm**：注册 `ConfirmListener` 回调，性能好，推荐

### 2. 消费者 Ack（消费确认）

确认消费者是否成功处理消息：

```java
// 手动确认模式
channel.basicConsume("queue", false, // autoAck = false
    new DefaultConsumer(channel) {
        @Override
        public void handleDelivery(String consumerTag,
                Envelope envelope, AMQP.BasicProperties properties,
                byte[] body) throws IOException {
            try {
                process(body);                      // 处理消息
                channel.basicAck(envelope.getDeliveryTag(), false); // 确认
            } catch (Exception e) {
                channel.basicNack(envelope.getDeliveryTag(), false, true); // 拒绝，重新入队
            }
        }
    });
```

| 确认方式 | 行为 | 适用场景 |
|---------|------|---------|
| `basicAck` | 确认成功，删除消息 | 正常处理完成 |
| `basicNack(requeue=true)` | 拒绝，重新入队 | 处理失败，需要重试 |
| `basicNack(requeue=false)` | 拒绝，丢弃消息 | 处理失败，不需要重试 |
| `basicReject` | 同 Nack，但只能单条拒绝 | 简单拒绝 |

### 消息可靠性总结

```
生产者端：开启 Confirm 机制，重试 + 幂等性
Broker 端：持久化（durable=true）+ 镜像队列
消费者端：手动 Ack，处理完再确认
```

## 题目10：LangChain 的核心组件有哪些？Agent 是如何工作的？

【答案】

### LangChain 六大核心模块

LangChain 是一个 LLM 编程框架，主体分为 **6 个模块**：

| 模块 | 作用 |
|------|------|
| **Model I/O** | 管理大语言模型输入输出（提示词模板、模型调用、输出解析） |
| **Data Connection** | 打通外部数据管道（文档加载、分割、嵌入、向量存储） |
| **Chains** | 将 LLM 和多组件连接，实现复杂任务 |
| **Memory** | 在多次调用之间保持状态（LLM 没有长期记忆） |
| **Agents** | 使用 LLM 做出决策，执行特定任务 |
| **Callbacks** | 事件钩子，追踪 token 使用等 |

### Model I/O（输入输出管理）

```python
# 提示词模板
from langchain.prompts import PromptTemplate
prompt = PromptTemplate(
    template="请把以下内容翻译成中文：{content}",
    input_variables=["content"]
)

# 模型调用
```

### Agent 的工作原理

**Agent = Action + Observation + Decision**，三个部分循环执行：

1. **接收用户输入** → LLM 判断需要执行什么 Action
2. **Action（工具调用）**：例如调用搜索、计算器、数据库查询等外部工具
3. **Observation（观察结果）**：获取 Action 的执行结果
4. **Decision（决策）**：根据结果判断是否继续，还是直接回复用户

### 题目11：合并 K 个升序链表（LeetCode 23）

【答案】

**方法一：分治合并（最优）**

```java
public ListNode mergeKLists(ListNode[] lists) {
    if (lists == null || lists.length == 0) return null;
    return merge(lists, 0, lists.length - 1);
}

private ListNode merge(ListNode[] lists, int left, int right) {
    if (left == right) return lists[left];
    int mid = (left + right) / 2;
    ListNode l1 = merge(lists, left, mid);      // 合并左半部分
    ListNode l2 = merge(lists, mid + 1, right);  // 合并右半部分
    return mergeTwoLists(l1, l2);               // 合并两个有序链表
}

private ListNode mergeTwoLists(ListNode l1, ListNode l2) {
    ListNode dummy = new ListNode(0);
    ListNode cur = dummy;
    while (l1 != null && l2 != null) {
        if (l1.val <= l2.val) {
            cur.next = l1;
            l1 = l1.next;
        } else {
            cur.next = l2;
            l2 = l2.next;
        }
        cur = cur.next;
    }
    cur.next = (l1 != null) ? l1 : l2;
    return dummy.next;
}
```

**时间复杂度**：O(N log K)，N 是总节点数，K 是链表数量
**空间复杂度**：O(log K)，递归栈深度

**方法二：最小堆**

```java
public ListNode mergeKLists(ListNode[] lists) {
    PriorityQueue<ListNode> heap = new PriorityQueue<>(
        (a, b) -> a.val - b.val
    );
    // 所有头节点入堆
    for (ListNode node : lists) {
        if (node != null) heap.offer(node);
    }
    ListNode dummy = new ListNode(0);
    ListNode cur = dummy;
    while (!heap.isEmpty()) {
        ListNode node = heap.poll();
        cur.next = node;
        if (node.next != null) heap.offer(node.next);
        cur = cur.next;
    }
    return dummy.next;
}
```

### 题目12：字符串匹配（KMP 算法，LeetCode 28）

【答案】

KMP 的核心思想是**利用已匹配的信息，避免重复匹配**。

```java
// KMP：构建 LPS（Longest Proper Prefix which is Suffix）数组
public int strStr(String haystack, String needle) {
    if (needle.isEmpty()) return 0;
    
    // 1. 构建 LPS 数组
    int[] lps = new int[needle.length()];
    buildLPS(needle, lps);
    
    // 2. 双指针匹配
    int i = 0; // haystack 指针
    int j = 0; // needle 指针
    while (i < haystack.length()) {
        if (haystack.charAt(i) == needle.charAt(j)) {
            i++;
            j++;
            if (j == needle.length()) {
                return i - j; // 找到匹配，返回起始下标
            }
        } else if (j > 0) {
            // 匹配失败，根据 LPS 回退 j
            j = lps[j - 1];
        } else {
            i++; // j 为 0，直接移动 i
        }
    }
    return -1;
}

private void buildLPS(String needle, int[] lps) {
    int len = 0;        // 前一个最长匹配前缀的长度
    int i = 1;
    lps[0] = 0;
    
    while (i < needle.length()) {
        if (needle.charAt(i) == needle.charAt(len)) {
            len++;
            lps[i] = len;
            i++;
        } else if (len > 0) {
            // 不匹配，回退到前一个可能的位置
            len = lps[len - 1];
        } else {
            lps[i] = 0;
            i++;
        }
    }
}
```

**LPS 数组含义**：`lps[i]` 表示模式串 `needle[0..i]` 中，前缀和后缀相同的最大长度。

**时间复杂度**：O(m + n)，m=haystack长度，n=needle长度
**空间复杂度**：O(n)

**举例**：needle = "AAACAAAA"
```
i:    0  1  2  3  4  5  6  7
lps:  0  1  2  0  1  2  3  1

needle[7]='A'，lps[6]=3，说明 needle[0..2] == needle[4..6] == "AAA"
```
