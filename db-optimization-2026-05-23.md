## 题目一：B+树索引原理与索引失效场景

### 题目
B+树是 MySQL InnoDB 存储引擎的默认索引结构，请描述 B+树相比 B 树的优点，并列举 5 种导致索引失效的常见场景。

### 答案

**B+树相比 B 树的优势：**

1. **查询复杂度稳定**：B+树所有数据都集中在叶子节点，且叶子节点之间通过双向链表相连，无论查找成功与否，都必须走到叶子节点，查询复杂度稳定为 O(log n)。
2. **范围查询高效**：叶子节点组成的链表使范围查询（如 `BETWEEN`、`>`、`<`）只需定位起点，然后顺序遍历链表即可，无需回溯树结构。
3. **节点可容纳更多指针**：非叶子节点不存储数据，只存储索引（键值），每个节点可容纳更多指针，树的高度更低，磁盘 I/O 更少。
4. **更适合数据库系统**：数据只出现在叶子节点，符合数据库顺序读取和缓存预取的特性。

**5 种导致索引失效的常见场景：**

| 场景 | 说明 | 示例 |
|------|------|------|
| ① 使用函数或运算 | 在索引列上使用函数、表达式、类型转换 | `WHERE LEFT(name,3)='Tom'` / `WHERE age + 1 = 20` |
| ② 使用 NOT / `<>` / `!=` | 极值条件下优化器可能放弃索引 | `WHERE status != 'active'` |
| ③ 使用 LIKE 前缀通配 | `LIKE '%keyword'` 以通配符开头 | `WHERE name LIKE '%om'` |
| ④ 类型不匹配 | 字段类型与值类型不一致导致隐式转换 | `WHERE code = 12345`（code 为 VARCHAR） |
| ⑤ 使用 OR 连接不同条件 | OR 两边字段不同且不满足最左前缀原则 | `WHERE name = 'Tom' OR age = 20` |

## 题目二：Explain 执行计划分析

### 题目
执行 `EXPLAIN SELECT ...` 后，字段 `type`、`key_len`、`Extra` 各表示什么？请列举 Extra 列中三个能说明索引被有效使用的关键值。

### 答案

**各字段含义：**

- **type（连接类型）**：从最优到最差依次为 `system > const > eq_ref > ref > range > index > ALL`。生产环境应极力避免 `ALL`（全表扫描）。
  - `const`：最多 1 条匹配行（主键或唯一索引）。
  - `ref`：使用普通索引等值匹配。
  - `range`：使用索引范围扫描（`BETWEEN`、`<>`、`IN` 等）。
  - `ALL`：全表扫描，性能最差。

- **key_len（索引长度）**：表示本次查询中使用的索引长度（字节数）。可用于判断复合索引使用了前几列。

- **Extra（附加信息）**：说明查询的具体执行方式。包含多种信息，用逗号分隔。

**三个说明索引被有效使用的关键值：**

| Extra 值 | 含义 |
|----------|------|
| `Using index` | 覆盖索引生效——查询的列全部被索引覆盖，无需回表（性能最优）。 |
| `Using index condition` | 索引下推（Index Condition Pushdown，ICP）生效——在索引层完成部分 WHERE 条件过滤，减少回表次数。 |
| `Using where; Using index` | 使用了索引且在存储引擎层之后还需用 WHERE 条件过滤（可接受，说明索引起到过滤作用）。 |

## 题目三：慢查询分析与优化

### 题目
线上发现一条 SQL 每次执行耗时超过 3 秒，业务方要求优化。请描述完整的排查思路和优化步骤。

### 答案

**完整排查思路（4 步法）：**

**Step 1：开启慢查询日志定位慢 SQL**

```sql
-- 查看慢查询配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 开启慢查询日志（临时）
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 超过 1 秒记录

-- 分析慢查询日志
mysqldumpslow -s t -t 10 /var/lib/mysql/slow.log
```

**Step 2：EXPLAIN 分析执行计划**

```sql
EXPLAIN SELECT ...
EXPLAIN ANALYZE ...  -- MySQL 8.0+，查看实际执行成本
```

关注：type、key、rows、Extra 是否出现 `Using filesort`（需额外排序）或 `Using temporary`（需建临时表）。

**Step 3：针对性优化**

| 问题 | 优化手段 |
|------|----------|
| 全表扫描 | 添加合适索引、改写 SQL |
| 文件排序 | 覆盖索引、调整 `max_length_for_sort_data` |
| 临时表 | 避免 `DISTINCT`、`GROUP BY` 隐式类型转换 |
| 深分页 | `WHERE id > last_id LIMIT 100` 替换 `OFFSET` |
| 子查询 | 改为 JOIN，利用索引 |

**Step 4：业务层优化**

- 读多写少场景：引入 Redis 缓存热点数据。
- 分页查询：改用游标分页（基于主键 ID 或时间戳）。
- 数据量大：做分库分表或历史数据归档。

## 题目四：MySQL InnoDB 与 MyISAM 的核心差异

### 题目
InnoDB 为什么能成为 MySQL 5.5 之后的默认存储引擎？请从事务支持、并发能力、崩溃恢复、存储结构四个维度对比 InnoDB 与 MyISAM。

### 答案

| 维度 | InnoDB | MyISAM |
|------|--------|--------|
| **事务支持** | 支持 ACID 事务，提供 `COMMIT`、`ROLLBACK` | 不支持事务，不保证原子性 |
| **并发能力** | 行级锁（Row Lock），支持高并发写入 | 表级锁，并发写入性能差 |
| **崩溃恢复** | 自动崩溃恢复（Crash Recovery），通过 redo log 实现 | 崩溃后可能损坏，需手动修复 |
| **存储结构** | 聚簇索引（主键索引和数据行在一起），辅助索引存储主键值 | 非聚簇索引（索引与数据分离），支持全文索引 |
| **外键支持** | 支持外键约束 | 不支持外键 |
| **COUNT(*) 性能** | 全表扫描（行级锁不阻塞读，但需实时计算） | 有专门计数器，速度快 |
| **适用场景** | 核心业务系统、写入频繁场景 | 非核心只读报表、CDN 静态资源表 |

**为何 InnoDB 成为默认引擎：**
MySQL 5.5 之前默认使用 MyISAM，但 MyISAM 不支持事务和行级锁，在数据安全和并发写入方面存在明显短板。InnoDB 通过 MVCC（多版本并发控制）解决了读写冲突，配合聚簇索引大幅提升随机 I/O 性能，因此成为现代 MySQL 的默认选择。

## 题目五：数据库缓存策略与缓存三大问题

### 题目
在 Web 应用中引入 Redis 缓存时，常遇到缓存穿透、缓存击穿、缓存雪崩三个问题。请分别解释这三个问题的含义，并给出对应的解决方案。

### 答案

**三大缓存问题的定义与解决方案：**

### 1. 缓存穿透（Cache Miss Storm）
**含义**：查询的数据在数据库中根本不存在，导致请求每次都穿透缓存直接打到数据库。恶意攻击者利用此漏洞大量请求不存在的数据，导致数据库压力过大。

**解决方案：**
- **布隆过滤器（Bloom Filter）**：将所有合法数据的特征码存入布隆过滤器，查询前先校验，非法请求直接拦截。
- **空值缓存**：即使查询结果为空，也将 `NULL` 写入缓存，并设置较短 TTL（如 30 秒），避免同一无效 key 反复穿透。

### 2. 缓存击穿（Hotspot Key Expiration）
**含义**：某个热点 key 过期瞬间，大量并发请求同时涌入，直接击穿缓存打到数据库，造成数据库瞬时压力峰值。

**解决方案：**
- **互斥锁（Mutex）**：缓存失效时，只允许一个请求去查询数据库并重建缓存，其他请求等待该请求完成后再从缓存读取。
- **永不过期 + 异步更新**：对热点 key 不设置过期时间，由后台异步任务负责定期更新缓存。
- **逻辑过期时间**：将过期时间作为 value 的一部分，业务层读取时判断逻辑过期时间，决定是否异步刷新。

### 3. 缓存雪崩（Cache Avalanche）
**含义**：大量缓存 key 在同一时间集中过期，或缓存服务本身宕机，导致短时间内大量请求直接打到数据库。

**解决方案：**
- **过期时间随机化**：为 key 的 TTL 添加随机偏移量（如 `TTL = base + rand(0, 1800)`），避免集中过期。
- **多级缓存架构**：L1（本地缓存）+ L2（Redis）+ 数据库，本地缓存失效不会直接打到 Redis。
- **熔断 / 限流**：缓存服务不可用时，启动服务降级，数据库层实现限流保护。
- **Redis 高可用**：使用 Redis Cluster 或 Sentinel 保证缓存服务的高可用性，避免全量缓存失效。
