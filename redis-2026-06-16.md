## 题目一：Redis 有哪些数据类型？分别用于什么场景？

### 参考答案

Redis 支持以下五种基本数据类型：

| 数据类型 | 可存储值 | 典型应用场景 |
|----------|----------|--------------|
| **String** | 字符串、整数或浮点数 | 缓存简单值、计数器、分布式锁（SETNX）、session 共享 |
| **Hash** | 键值对集合（field-value map） | 存储对象（如用户信息、商品详情）、结构化缓存 |
| **List** | 有序字符串列表（双向链表） | 消息队列（FIFO）、最新消息列表、关注列表 |
| **Set** | 无序唯一字符串集合 | 标签系统、去重、共同好友、抽奖抽奖池 |
| **Sorted Set (ZSet)** | 有序唯一字符串集合，每个元素有 score | 排行榜、延时队列、带权重的任务调度 |

此外还有三种高级数据类型：
- **Bitmap**：位图，适合签到、在线状态等位操作场景
- **HyperLogLog**：基数统计，适合 UV 统计（内存极小）
- **Geospatial**：地理位置，适合附近的人/商铺查询

**使用建议**：
- 优先选择合适的数据结构，避免用 String JSON 存储复杂对象（无法针对单个字段操作）
- ZSet 的 score 可用双精度浮点数，注意精度问题
- List 的 LPUSH/LPOP 实现队列，RPUSH/RPOP 实现栈

📎 来源：[Redis 官方文档 – Data Types](https://redis.io/docs/data-types/)

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 参考答案

Redis 提供了两种持久化机制：

### RDB（Redis Database）
- **原理**：在指定时间间隔内，将内存中的数据快照（snapshot）写入二进制文件 `dump.rdb`
- **触发方式**：定时自动触发（`save` 配置）、手动 `BGSAVE` 命令、Redis 关闭时
- **优点**：恢复速度快（直接加载 RDB 文件）、适合大规模数据备份
- **缺点**：最后一次快照后的数据会丢失、fork 子进程时有一定内存开销

### AOF（Append Only File）
- **原理**：记录每个写操作命令，以追加方式写入 `appendonly.aof`
- **同步策略**：`always`（每个命令同步）、`everysec`（每秒同步，默认）、`no`（由系统决定）
- **优点**：数据安全性更高（最多丢 1 秒数据）、日志格式易读
- **缺点**：文件体积通常比 RDB 大、恢复速度较慢

### 重写（Rewrite）机制
当 AOF 文件过大时，Redis 会执行 **BGREWRITEAOF**，对现有数据进行重写，生成精简的 AOF 文件。

### 如何选择？

| 场景 | 推荐策略 |
|------|----------|
| 追求最高数据安全性 | AOF + everysec 或 always |
| 允许少量数据丢失，追求性能 | RDB |
| 两者兼得 | RDB + AOF（同时启用，Redis 优先用 AOF 恢复） |

> ⚠️ 如果对数据安全性要求极高，建议同时开启 RDB 和 AOF，并合理配置 `appendfsync everysec`。

📎 来源：[Redis 官方文档 – Persistence](https://redis.io/docs/management/persistence/)

## 题目三：Redis 的过期键删除策略有哪几种？有什么区别？

### 参考答案

Redis 有三种过期键删除策略：

### 1. 定时删除（主动删除）
- 为每个设置过期时间的键创建一个定时器（Timer），到期立即删除
- **优点**：内存友好，过期键及时清理
- **缺点**：CPU 不友好，若过期键较多会消耗大量 CPU 资源，影响性能

### 2. 惰性删除（被动删除）
- 访问某个键时，检查是否过期，过期则删除
- **优点**：对 CPU 友好，只在需要时才检查
- **缺点**：内存不友好，大量过期键堆积不被清理（内存泄漏风险）

### 3. 定期删除（折中方案）
- 每隔一段时间（如 100ms），随机检查一部分有过期键的数据库，删除其中已过期的键
- 通过 `hz` 配置调整检查频率
- **优点**：折中 CPU 和内存开销
- **缺点**：如果执行时间过长，仍会影响 Redis 响应

### Redis 采用的策略
Redis 采用 **惰性删除 + 定期删除** 组合策略：
- **惰性删除**：所有读写命令执行前调用 `expireIfNeeded()` 检查
- **定期删除**：Redis 服务器定时任务 `activeExpireCycle()` 周期性运行

### 内存淘汰机制
当内存达到 `maxmemory` 上限时，Redis 还提供 8 种**内存淘汰策略**（`maxmemory-policy`）：
- `volatile-lru` / `allkeys-lru`：对设置了过期时间的键 / 所有键进行 LRU 淘汰
- `volatile-ttl`：淘汰剩余 TTL 最小的键
- `volatile-random` / `allkeys-random`：随机淘汰
- `no-eviction`：不淘汰，返回错误（默认）

📎 来源：[Redis 官方文档 – Expires](https://redis.io/commands/expire/)

## 题目四：Redis 主从复制原理是什么？什么是全量同步和增量同步？

### 参考答案

### 复制流程

Redis 主从复制支持 **一主多从** 架构，主节点负责写，从节点负责读（读写分离）。

```
主节点（Master）  -->  PSYNC 命令  -->  从节点（Slave）
       |                                    |
       |<----------- 复制偏移量 offset ----------->|
       |                                    |
       |<------- Rdb 文件（全量）或 命令流（增量） -----|
```

### 全量同步（Full Resync）

当从节点首次连接主节点，或主从断线重连后数据差异过大时触发：

1. **主节点**执行 `BGSAVE`，生成 RDB 快照文件
2. 主节点将 RDB 文件发送给从节点（**期间新写命令写入 replication buffer**）
3. 从节点清空本地数据，加载 RDB 文件
4. 主节点将 replication buffer 中的增量命令同步给从节点

### 增量同步（Partial Resync / PSYNC）

当从节点短时断开后重连，且复制积压缓冲区（replication backlog）仍包含断连期间的命令时：

1. 从节点发送 `PSYNC <runid> <offset>`
2. 主节点判断 runid 匹配且 offset 在 backlog 范围内
3. 直接发送积压区的增量命令给从节点

> **replication backlog** 是一个固定长度的环形缓冲区（默认 1MB），记录最近一段时间的写命令。

### 相关配置

| 配置项 | 说明 |
|--------|------|
| `replicaof <masterip> <masterport>` | 配置主节点 |
| `repl-diskless-sync yes` | 无磁盘复制（直接通过网络发送 RDB） |
| `repl-backlog-size 1mb` | 积压缓冲区大小，建议设大一些 |
| `min-replicas-to-write 2` | 至少 N 个从节点确认后才算写入成功 |

📎 来源：[Redis 官方文档 – Replication](https://redis.io/docs/management/replication/)

## 题目五：Redis 如何实现分布式锁？有什么注意事项？

### 参考答案

### 基于 SET 命令的简单实现

```lua
-- 加锁（SET NX + PX 原子操作）
SET lock_key unique_value NX PX 30000

-- 解锁（Lua 脚本，保证原子性）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**关键点**：
- `NX`：保证只有一个客户端能获取锁（非原子性竞争）
- `PX`：设置锁的自动过期时间，防止锁永久持有
- **value 必须唯一**（如 UUID），防止误删他人的锁
- 解锁必须用 Lua 脚本，保证「检查 + 删除」原子性

### 完整分布式锁需要解决的问题

| 问题 | 解决方案 |
|------|----------|
| 锁过期但任务未完成 | 开启** watchdog** 续期机制（Redisson 实现） |
| 主从切换导致锁丢失 | 使用 **RedLock**（多节点多数派） |
| 误删他人锁 | value 用唯一标识（如 UUID），Lua 脚本先检查再删除 |
| 可重入性 | value 中记录重入次数和线程信息 |

### RedLock 算法（红锁）

为解决主从复制带来的锁丢失问题，RedLock 算法建议：
1. 向 N 个独立 Redis 节点（无主从关系）获取锁
2. 超过 ⌊N/2⌋+1 个节点成功获取锁，才算成功
3. 各节点使用相同的 key 和随机 value

> ⚠️ RedLock 并非绝对安全，在网络分区等极端情况下可能存在争议，实际使用需评估业务场景。

### 成熟的分布式锁库推荐
- **Redisson**（Java）：实现了 watchdog 自动续期、可重入锁、公平锁等
- **Python-redlock**（Python）：Python 实现的 RedLock

📎 来源：[Redis 官方文档 – Distributed Locks](https://redis.io/docs/manual/patterns/distributed-locks/)
