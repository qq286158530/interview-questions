## 题目一：Flink 的架构是怎样的？请描述 Flink 的核心组件及数据流转过程

### 参考答案

Flink 采用**流式计算模型**，核心架构分为 **Client、JobManager、TaskManager** 三大组件：

```
┌─────────────────────────────────────────────────────────┐
│                      Flink 架构                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   Client                                                 │
│     │                                                   │
│     ▼                                                   │
│   JobManager (Master)                                   │
│   ├── Dispatcher ──── REST API                         │
│   ├── ResourceManager ── 资源管理 (YARN/K8s/Standalone) │
│   └── JobMaster ─────── 作业调度与执行                  │
│          │                                              │
│          ▼                                              │
│   TaskManager (Worker)                                 │
│   ├── Task Slot ──── 资源隔离单元                       │
│   └── Memory Manager ── 内存管理                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 核心组件职责

| 组件 | 职责 |
|------|------|
| **Client** | 负责将用户编写的 Flink 应用程序编译为 JobGraph，提交给 JobManager |
| **JobManager** | 整个集群的协调者，负责接收 JobGraph、分配资源、调度 Task 到 TaskManager |
| **TaskManager** | 实际执行 Task 的工作节点，每个 TaskManager 包含多个 Task Slot |

### 数据流转过程

1. **用户代码** → 转换为 **StreamGraph**（API层面）
2. **StreamGraph** → 优化为 **JobGraph**（Flink 优化器）
3. **JobGraph** → 提交给 **JobManager**
4. **JobManager** 向 **ResourceManager** 申请资源
5. **ResourceManager** 分配 **TaskManager** 和 **Task Slot**
6. **JobMaster** 将 JobGraph 拆分为 **ExecutionGraph**，并调度 Task
7. **TaskManager** 接收 Task 并启动 **StreamTask** 执行

### 扩展问题

**Flink 和 Spark Streaming 的区别？**

| 维度 | Flink | Spark Streaming |
|------|-------|-----------------|
| 计算模型 | 纯流式，逐条处理 | 微观批量（micro-batch） |
| 延迟 | 低延迟（毫秒级） | 较高（秒级） |
| 容错 | Checkpoint + Exactly-Once | Checkpoint + Exactly-Once |
| 状态管理 | 原生状态后端 | 基于 RDD |

## 题目二：什么是 Watermark？Flink 如何处理乱序数据和迟到数据？

### 参考答案

### Watermark（水位线）概念

**Watermark** 是 Flink 用于处理**乱序事件**和**迟到事件**的机制，本质是一个**时间戳**，表示"所有早于此时间戳的数据都已到达"。

```
时间线示意：
Event Time ──────────────────────────────▶
         │
         │  Watermark = 10:05:00
         │  (表示 10:05:00 之前的数据都已到达)
         │
    ┌────┴────┐
    │  窗口    │  窗口范围：[10:00:00 - 10:05:00)
    │ 10:00-  │
    │ 10:05   │
    └─────────┘
```

### 核心机制

#### 1. 解决乱序问题

当 Watermark 到达窗口结束时间时，窗口才触发计算。即使部分数据迟到，只要 Watermark 未超过窗口结束时间，数据仍会被处理。

```java
// 方式一：固定 Watermark 策略
WatermarkStrategy
    .forBoundedOutOfOrderness(Duration.ofSeconds(10))  // 允许10秒乱序
    .withTimestampAssigner(...)
```

#### 2. 处理迟到数据

| 策略 | 行为 | 配置方式 |
|------|------|----------|
| **默认丢弃** | 超过 Watermark 的数据直接丢弃 | — |
| **allowedLateness** | 额外等待一段时间，允许迟到数据进入 | `.allowedLateness(Duration.ofMinutes(1))` |
| **sideOutputLateData** | 将迟到数据输出到侧输出流 | `.sideOutputLateData(outputTag)` |

```java
WatermarkStrategy
    .forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner((event, timestamp) -> event.getTimestamp())
    .withAllowedLateness(Duration.ofMinutes(1));  // 额外等待1分钟

// 窗口计算
window
    .allowedLateness(Time.minutes(1))
    .sideOutputLateData(lateOutputTag)
    .sum(...)
```

### 完整示例

```java
public class WatermarkExample {
    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = 
            StreamExecutionEnvironment.getExecutionEnvironment();
        env.setStreamTimeCharacteristic(TimeCharacteristic.EventTime);
        
        // 定义迟到数据侧输出标签
        OutputTag<String> lateTag = new OutputTag<>("late-data"){};
        
        DataStream<String> stream = env
            .socketTextStream("localhost", 9999)
            .map(new MapFunction<String, Tuple2<Long, String>>() {
                @Override
                public Tuple2<Long, String> map(String value) {
                    String[] parts = value.split(",");
                    return new Tuple2<>(Long.parseLong(parts[0]), parts[1]);
                }
            })
            .assignTimestampsAndWatermarks(
                WatermarkStrategy
                    .<Tuple2<Long, String>>forBoundedOutOfOrderness(
                        Duration.ofSeconds(5))
                    .withTimestampAssigner((e, ts) -> e.f0)
            )
            .keyBy(e -> e.f1)
            .window(SlidingEventTimeWindows.of(Time.minutes(5), Time.minutes(1)))
            .allowedLateness(Time.minutes(1))
            .sideOutputLateData(lateTag)
            .sum(1);
        
        // 获取侧输出流（迟到数据）
        DataStream<String> lateStream = stream.getSideOutput(lateTag);
        
        env.execute();
    }
}
```

### Watermark 传播机制

Watermark 在算子间传递时，遵循**最小值原则**：

```
Source (W=100) ──▶ Map (W=min(100, 80)=80) ──▶ KeyBy (W=min(80, 60)=60) ──▶ Window
     │                  │                         │
  W=100              W=80                       W=60
```

**关键点**：下游算子的 Watermark = 上游传递的 Watermark 与本地已接收 Watermark 的**最小值**。

## 题目三：Flink 的 Checkpoint 机制是如何工作的？为什么需要 Barrier？如何实现 Exactly-Once 语义？

### 参考答案

### Checkpoint 简介

Checkpoint 是 Flink **容错机制**的核心，通过**分布式快照**保存状态数据，确保故障恢复后能从上次快照继续执行，保证 **Exactly-Once** 语义。

### Checkpoint 执行流程（基于 Chandy-Lamport 算法）

```
                    ┌─────────────────────────────────────┐
                    │         Checkpoint Coordinator      │
                    │            (JobManager)             │
                    └──────────────┬──────────────────────┘
                                   │ 发送 Checkpoint 触发消息
                    ┌──────────────▼──────────────────────┐
                    │     Source Operator                │
                    │  ┌─────────┐  ┌─────────┐          │
   Barrier_n ──────▶│  │ Check-  │  │ Check-  │──▶ Barrier_n
                    │  │ point 1 │  │ point 2 │          │
                    │  └────┬────┘  └────┬────┘          │
                    └───────┼────────────┼────────────────┘
                            │            │
              ┌─────────────┘            └──────────────┐
              ▼                                          ▼
   ┌───────────────────┐                    ┌───────────────────┐
   │  Map Operator     │                    │  Map Operator     │
   │  ┌───────────┐    │                    │  ┌───────────┐    │
   │  │  State 1  │──┼──┼────────────────────┼─▶│  State 2  │    │
   │  └───────────┘    │                    │  └───────────┘    │
   └───────────────────┘                    └───────────────────┘
              │                                          │
              └──────────────────┬───────────────────────┘
                                 ▼
                    ┌───────────────────┐
                    │   Window/Sink     │
                    │  ┌───────────┐    │
                    │  │  State 3  │    │
                    │  └───────────┘    │
                    └───────────────────┘
```

### Barrier（屏障）机制

**Barrier** 是 Checkpoint 快照的"分界线"，将数据流中的记录分为**快照前**和**快照后**两部分。

#### 对齐阶段（Alignment）

当算子有**多个输入流**时（如 KeyBy 后的双流 join），需要等待所有流的 Barrier 到达才能做快照：

1. 算子从 InputGate 接收数据
2. 当某个 InputGate 的 Barrier 先到达，**缓存**后续数据直到所有 Barrier 到达
3. Barrier 全部到达后，**阻塞该通道**，继续处理其他通道数据
4. 完成对齐后，**并行执行**快照操作

```
输入通道 A:  D1 D2 │Bar_A│ D5 D6 ...
输入通道 B:  D3    │Bar_B│ D4 D7 ...

对齐过程：
- Bar_A 先到，暂停 A 通道，处理 B 通道
- Bar_B 到达后，两个通道同时放行
- 对齐完成，开始快照
```

### Exactly-Once 实现

Flink 的 Exactly-Once 语义通过**三件事**保证：

| 步骤 | 机制 | 说明 |
|------|------|------|
| 1 | **Source 端** | 记录消费位置（offset），故障恢复从上次 Checkpoint 位置重读 |
| 2 | **算子间** | Barrier 对齐 + 状态快照，所有算子状态一致 |
| 3 | **Sink 端** | 事务提交（2PC），先预提交，Checkpoint 完成后正式提交 |

### Sink 端 2PC 提交

```
              Checkpoint_n 完成
                    │
                    ▼
         ┌──────────────────────┐
         │   预提交 (Pre-commit) │
         │   Sink 写入数据      │
         │   但不正式生效        │
         └──────────┬───────────┘
                    │
                    ▼
         ┌──────────────────────┐
         │  正式提交 (Commit)   │
         │  落盘/对外可见        │
         └──────────────────────┘
```

### 配置示例

```java
StreamExecutionEnvironment env = 
    StreamExecutionEnvironment.getExecutionEnvironment();

// 启用 Checkpoint，间隔 10 秒
env.enableCheckpointing(10_000L);

// Checkpoint 存储后端
env.setStateBackend(new RocksDBStateBackend("hdfs:///checkpoint"));

// 高级配置
CheckpointConfig config = env.getCheckpointConfig();
config.setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);
config.setMinPauseBetweenCheckpoints(5_000L);    // 两次 Checkpoint 最小间隔
config.setCheckpointTimeout(120_000L);            // Checkpoint 超时时间
config.setTolerableCheckpointFailureNumber(3);   // 允许失败次数
config.enableExternalizedCheckpointCleanup(
    ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION  // 取消时保留
);
```

### Checkpoint 与 Savepoint 的区别

| 维度 | Checkpoint | Savepoint |
|------|------------|-----------|
| **触发者** | Flink 自动（定时） | 用户手动触发 |
| **用途** | 故障恢复 | 计划性停机/升级 |
| **删除策略** | Job 取消时删除（可配置） | 手动删除 |
| **格式** | Flink 内部格式 | 兼容性好，可跨版本 |

## 题目四：Flink 的窗口函数有哪些？Session Window 是如何工作的？如何处理窗口结果？

### 参考答案

### 窗口函数分类

Flink 支持多种窗口类型，按**数据驱动**和**时间驱动**两个维度分类：

```
                    ┌─────────────────┬─────────────────┐
                    │   时间窗口       │   计数窗口       │
        ┌───────────┼─────────────────┼─────────────────┤
        │ Tumbling  │ 滚动窗口        │ 计数滚动窗口     │
        │ (不重叠)   │ 固定大小、无重叠 │ 固定计数、无重叠 │
        ├───────────┼─────────────────┼─────────────────┤
        │ Sliding   │ 滑动窗口        │ 计数滑动窗口     │
        │ (可重叠)   │ 固定大小、可重叠 │ 固定计数、可重叠 │
        ├───────────┼─────────────────┼─────────────────┤
        │ Session    │ 会话窗口        │ 会话计数窗口     │
        │ (动态)     │ 动态大小、间隙大 │ 动态、gap内无数据│
        └───────────┴─────────────────┴─────────────────┘
```

### 常用窗口详解

#### 1. 滚动窗口（Tumbling Window）

```java
// 每5分钟统计一次
DataStream<Tuple2<String, Long>> windowed = stream
    .keyBy(0)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .sum(1);

// 滚动计数窗口：每100条统计一次
.window(SlidingProcessingTimeWindows.of(100, 0))  // 注意：这里用 0 表示不重叠
// 或者用 TumblingEventTimeWindows.of(Time.of(100, TimeUnit.SECONDS)) 但这是时间窗口

// 正确写法（Flink 1.13+）
.window(TumblingCountWindow.of(100))
```

#### 2. 滑动窗口（Sliding Window）

```java
// 窗口大小10分钟，滑动步长5分钟
DataStream<Tuple2<String, Long>> windowed = stream
    .keyBy(0)
    .window(SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(5)))
    .sum(1);
```

#### 3. 会话窗口（Session Window）

**Session Window** 根据**会话间隙**（Gap）动态创建窗口，当两条数据之间的间隔超过指定 Gap 时，开启新窗口。

```java
// 动态会话窗口，gap 为 5 秒无数据则关闭窗口
DataStream<Tuple2<String, Long>> windowed = stream
    .keyBy(0)
    .window(EventTimeSessionWindows.withGap(Time.seconds(5)))
    .sum(1);

// 动态 Gap（基于数据本身）
.window(EventTimeSessionWindows.withDynamicGap(new SessionWindowTimeGapExtractor<Tuple2<String, Long>>() {
    @Override
    public double extract(Tuple2<String, Long> element) {
        return element.f1;  // 根据数据动态计算 Gap
    }
}))
```

#### 会话窗口工作原理

```
数据流：
  A@1s ─── A@3s ──────────── A@10s ────── A@15s
           │                  │            │
        start │              start │      start │
          │   │                │   │        │   │
       Gap=5s ▼                ▼   │     ▼   │   │
                                  │         │   │
                            关闭 │         │   │
                              ▼   │         ▼   │
                            [A@3s]        [A@10s,A@15s]

说明：
- A@1s, A@3s 间隔2秒 → 属于同一会话
- A@3s 到 A@10s 间隔7秒 > Gap(5s) → 新会话开始
- A@10s 到 A@15s 间隔5秒 = Gap → 属于同一会话
```

### 窗口函数类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **增量聚合函数** | 每条数据触发一次，性能高 | `sum()`, `min()`, `max()`, `reduce()` |
| **全量聚合函数** | 窗口关闭时一次性计算 | `apply()`, `process()` |
| **两条数据一起计算** | 处理两个流的 join | `coGroup()` |

### 窗口计算示例

```java
// 增量聚合：实时累加
DataStream<Tuple2<String, Long>> result = stream
    .keyBy(e -> e.f0)
    .window(SlidingEventTimeWindows.of(Time.minutes(5), Time.minutes(1)))
    .reduce((v1, v2) -> new Tuple2<>(v1.f0, v1.f1 + v2.f1));

// 全量聚合：窗口关闭时计算
windowedStream
    .apply(new WindowFunction<Tuple2<String, Long>, String, String, TimeWindow>() {
        @Override
        public void apply(
                String key,
                TimeWindow window,
                Iterable<Tuple2<Long, String>> input,
                Collector<String> out) {
            long sum = 0;
            for (Tuple2<Long, String> e : input) {
                sum += 1;
            }
            out.collect("key: " + key + " window: [" + 
                window.getStart() + "," + window.getEnd() + 
                ") count: " + sum);
        }
    });

// 使用 ProcessFunction（全量，更灵活）
windowedStream
    .process(new ProcessWindowFunction<Tuple2<String, Long>, String, String, TimeWindow>() {
        @Override
        public void process(
                String key,
                Context ctx,
                Iterable<Tuple2<String, Long>> input,
                Collector<String> out) {
            long count = 0;
            for (Tuple2<String, Long> e : input) {
                count++;
            }
            out.collect("窗口 " + key + " 元素个数: " + count);
            // 还可以访问窗口信息
            ctx.window().getStart();
            ctx.window().getEnd();
        }
    });
```

### 多流合并窗口

```java
// 两条流合并，事件时间对齐
stream1
    .union(stream2)
    .assignTimestampsAndWatermarks(...)
    .keyBy(e -> e.key)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .reduce(...);
```

## 题目五：Flink 的状态后端有哪些？如何选择合适的状态后端？状态是如何存储和管理的？

### 参考答案

### Flink 状态后端概述

Flink 的**状态后端**（State Backend）决定了**状态如何存储、快照如何保存**，是影响性能和稳定性的关键组件。

### 三种状态后端

| 状态后端 | 存储位置 | 适用场景 | 特点 |
|----------|----------|----------|------|
| **HashMapStateBackend** | JVM Heap | 小状态、低延迟 | 最快，但受限于 JVM 堆内存 |
| **EmbeddedRocksDBStateBackend** | RocksDB + 磁盘 | 大状态、超大窗口 | 可存储 TB 级状态，序列化开销 |
| **EmbeddedRocksDBStateBackend + 增量 Checkpoint** | RocksDB + 磁盘 | 超大状态 | 增量快照，减少 Checkpoint 时间 |

### 架构对比

```
┌──────────────────────────────────────────────────────────────┐
│                     HashMapStateBackend                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   TaskManager                                                │
│   ┌────────────────────────────────────┐                    │
│   │         JVM Heap                    │                    │
│   │  ┌─────────┐  ┌─────────┐          │                    │
│   │  │ State 1 │  │ State 2 │  ...     │                    │
│   │  └─────────┘  └─────────┘          │                    │
│   └────────────────────────────────────┘                    │
│                                                              │
│   优点: 访问速度最快                                         │
│   缺点: 状态大小受限于内存                                   │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                 EmbeddedRocksDBStateBackend                   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   TaskManager                                                │
│   ┌────────────────────────────────────┐                    │
│   │  RocksDB (本地磁盘)                 │                    │
│   │  ┌─────────┐  ┌─────────┐          │                    │
│   │  │ State 1 │  │ State 2 │  ...     │                    │
│   │  │ (SSTable)│  │ (SSTable)│         │                    │
│   │  └─────────┘  └─────────┘          │                    │
│   └────────────────────────────────────┘                    │
│          │                                                   │
│          ▼ (异步)                                            │
│   ┌──────────────┐                                          │
│   │  Checkpoint  │  (HDFS/S3/...)                           │
│   └──────────────┘                                          │
│                                                              │
│   优点: 支持超大状态                                          │
│   缺点: 序列化/反序列化开销，每次访问需读写磁盘                 │
└──────────────────────────────────────────────────────────────┘
```

### 配置方式

#### 1. 全局配置（flink-conf.yaml）

```yaml
# HashMapStateBackend（默认）
state.backend: hashmap

# RocksDBStateBackend
state.backend: rocksdb
state.backend.incremental: true
state.checkpoints.dir: hdfs:///checkpoint
state.savepoints.dir: hdfs:///savepoint
```

#### 2. 代码配置

```java
StreamExecutionEnvironment env = 
    StreamExecutionEnvironment.getExecutionEnvironment();

// 方式一：HashMapStateBackend
env.setStateBackend(new HashMapStateBackend());

// 方式二：RocksDBStateBackend（推荐大状态使用）
env.setStateBackend(new EmbeddedRocksDBStateBackend());

// 配置 Checkpoint 存储
env.getCheckpointConfig().setCheckpointStorage(
    "hdfs:///checkpoint"
    // 或 "file:///tmp/checkpoint"（本地测试）
    // 或 "s3://bucket/checkpoint"（S3）
);
```

#### 3. 增量 Checkpoint（RocksDB）

```java
EmbeddedRocksDBStateBackend rocksDBBackend = 
    new EmbeddedRocksDBStateBackend(true);  // true = 开启增量

env.setStateBackend(rocksDBBackend);
env.getCheckpointConfig().setCheckpointStorage("hdfs:///checkpoint");

// 增量 Checkpoint 只保存增量部分，大大减少 Checkpoint 时间
```

### 状态类型

| 状态类型 | 说明 | 使用场景 |
|----------|------|----------|
| **Keyed State** | 按 Key 隔离的状态 | `keyBy` 之后使用，底层自动按 Key 管理 |
| **Operator State** | 算子级别的共享状态 | Kafka Connector 的 Offset、模型参数 |
| **Broadcast State** | 广播状态，两流 join | 规则流广播给数据流 |

### Keyed State 示例

```java
public class CountWindowAverage 
    extends KeyedProcessFunction<String, Tuple2<Long, Long>, Tuple2<String, Double>> {
    
    // 声明状态
    private ValueState<Long> sumState;
    private ValueState<Long> countState;
    
    @Override
    public void open(Configuration parameters) {
        // 初始化状态
        sumState = getRuntimeContext().getState(
            new ValueStateDescriptor<>("sum", Long.class));
        countState = getRuntimeContext().getState(
            new ValueStateDescriptor<>("count", Long.class));
    }
    
    @Override
    public void processElement(
            Tuple2<Long, Long> value,
            Context ctx,
            Collector<Tuple2<String, Double>> out) throws Exception {
        
        // 读取当前状态
        Long currentSum = sumState.value();
        Long currentCount = countState.value();
        
        // 更新状态
        sumState.update(currentSum == null ? value.f1 : currentSum + value.f1);
        countState.update(currentCount == null ? 1L : currentCount + 1);
        
        // 输出结果
        if (countState.value() > 0) {
            out.collect(new Tuple2<>(value.f0, 
                (double) sumState.value() / countState.value()));
        }
    }
}
```

### 状态过期与清理

```java
// 状态 TTL（过期时间）
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.hours(1))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)  // 读写时更新
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)  // 不返回过期状态
    .cleanupInRocksdbCompactFilters(1000)  // RocksDB 紧凑过滤时清理
    .build();

ValueStateDescriptor<String> stateDescriptor = 
    new ValueStateDescriptor<>("my-state", String.class);
stateDescriptor.enableTimeToLive(ttlConfig);
```

### 选型建议

| 状态规模 | 推荐后端 | 原因 |
|----------|----------|------|
| **< 1GB** | HashMapStateBackend | 纯内存，延迟最低 |
| **1GB - 100GB** | EmbeddedRocksDBStateBackend | 支持大状态，性价比高 |
| **> 100GB** | RocksDB + 增量 Checkpoint | 避免超长 Checkpoint 时间 |
