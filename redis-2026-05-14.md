## 题目一：Redis 有哪些常用数据类型？它们的典型使用场景是什么？

### 参考答案

Redis 支持五种基础数据类型，每种都有其独特的应用场景：

| 数据类型 | 底层实现 | 典型使用场景 |
|---------|---------|-------------|
| **String** | SDS（简单动态字符串） | 缓存、计数器、分布式锁、限流 |
| **List** | QuickList（压缩链表） | 消息队列、排行榜、关注列表 |
| **Hash** | 压缩列表 + 哈希表 | 存储对象、购物车、会话缓存 |
| **Set** | 哈希表 + 整数集合 | 去重、标签系统、好友关系 |
| **Zset** | 压缩列表 + 跳表 | 排行榜、有序集合、延迟队列 |

**进阶追问：**
- `Zset` 为什么用跳表而不是红黑树？  
  答：跳表支持范围查询更高效，实现简单易懂，内存占用与红黑树相当但插入/删除只需局部更新。

- SDS 与 C 字符串的区别？  
  答：SDS 支持 O(1) 长度获取、有空间预分配和惰性回收，比 C 字符串更安全高效。

**来源参考：** [Redis设计与实现](http://redisbook.com/) | [Redis 文档](https://redis.io/docs/data-types/)

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 参考答案

Redis 提供了两种持久化机制：

### RDB（Redis Database）
- **原理**：定时生成数据集的快照，fork 子进程进行全量备份
- **触发方式**：`save`（同步）或 `bgsave`（异步），也可配置自动触发
- **优点**：恢复快、文件紧凑、适合备份
- **缺点**：可能丢失最近一次快照后的数据

### AOF（Append Only File）
- **原理**：记录所有写操作命令，以日志形式追加
- **同步策略**：`always`/`everysec`/`no`
- **优点**：数据安全性更高、只追加写入
- **缺点**：文件较大、恢复慢

### 对比

| 特性 | RDB | AOF |
|-----|-----|-----|
| 文件体积 | 小 | 大 |
| 恢复速度 | 快 | 慢 |
| 数据完整性 | 可能丢失数据 | 取决于同步策略 |
| 性能影响 | fork开销 | 持续IO开销 |

### 生产环境建议
- **小数据量**：纯 AOF
- **大数据量**：RDB + AOF（主从复制保证数据完整性）
- `appendonly yes` + `appendfsync everysec` 是常用配置

## 题目三：什么是缓存穿透、缓存击穿、缓存雪崩？如何解决？

### 参考答案

这是 Redis 面试中最常见的高频问题，简称「缓存三剑客」：

### 1. 缓存穿透
**问题**：查询不存在的数据，绕过缓存直击数据库
**危害**：恶意请求会压垮数据库

**解决方案：**
- 布隆过滤器（Bloom Filter）：将已存在的 key 存入 bitmap
- 缓存空值：对查询结果为空的值也缓存（设置短 TTL）
- 参数校验：接口层做好参数合法性校验

### 2. 缓存击穿
**问题**：热点 key 过期瞬间，大量请求涌入数据库
**危害**：数据库压力骤增

**解决方案：**
- 互斥锁（Mutex Key）：只允许一个线程重建缓存
- 逻辑过期：热点数据不过期，后台异步更新
- 永不过期：对重要数据设置较长的逻辑过期时间

### 3. 缓存雪崩
**问题**：大量 key 同时过期或 Redis 宕机
**危害**：数据库请求激增，系统崩溃

**解决方案：**
- 过期时间随机化：` TTL + random()`
- Redis 高可用：主从 + 哨兵 / Cluster
- 限流降级：Hystrix / Sentinel 保护数据库
- 预热：系统启动时加载热点数据

### 一句话总结
| 问题 | 原因 | 关键词 |
|-----|------|-------|
| 穿透 | 查询不存在 | 布隆过滤器 |
| 击穿 | 热点 key 过期的瞬间 | 互斥锁 |
| 雪崩 | 大量 key 同时过期 | 过期随机 + 高可用 |

**来源参考：** [缓存穿透、击穿、雪崩详解](https://blog.csdn.net/zeb_perfect/article/details/124482615) | [Redis 缓存问题](https://juejin.cn/post/6844903807266087943)

## 题目四：Redis 是单线程还是多线程？为什么这么快？

### 参考答案

### Redis 的线程模型

**Redis 6.0 之前**：纯单线程（核心网络 I/O 和命令处理）
**Redis 6.0 之后**：I/O 线程多线程化，但核心执行仍是单线程

### 为什么单线程还能这么快？

1. **内存操作**  
   数据全在内存，纳秒级访问，比磁盘快 10 万倍

2. **高效数据结构**  
   SDS、QuickList、SkipList、ZipList 等经过优化的数据结构

3. **非阻塞 I/O**  
   基于 Epoll（Linux）/ Kqueue（macOS）事件轮询，单线程处理高并发

4. **避免锁开销**  
   单线程无需上下文切换和锁竞争

5. **Pipeline 批量操作**  
   减少网络往返次数，提高吞吐

### Redis 6.0 I/O 多线程
```
主线程接受连接 → 分发给 I/O 线程 → 命令执行仍在主线程
```
这只优化了网络层的并发，实际数据操作仍是单线程顺序执行。

### 生产环境注意事项
- 单线程不代表只能使用一个 CPU 核心（可绑定多核）
- 避免执行 O(N) 级别操作（如 `KEYS *`、`HGETALL` 大 key）
- 使用 `SCAN` 替代 `KEYS`，监控 big key

**来源参考：** [Redis 单线程模型](https://redis.io/docs/reference/internals/redis-event-loop/) | [Redis 为什么这么快](https://tech.meituan.com/2022/04/01/redis-fast.html)

## 题目五：如何保证 Redis 与 MySQL 的数据一致性？

### 参考答案

这是分布式系统中经典的问题，没有完美方案，只有权衡取舍。

### 常见方案

#### 方案一：Cache Aside（旁路缓存）
```
读：先读缓存，没有则读库并写入缓存
写：先更新库，再删除缓存（不是更新）
```
**优点**：简单易懂，命中率低时性能好  
**缺点**：删除缓存失败会导致数据不一致

#### 方案二：Read Through
```
读：缓存负责读库并回填，调用方只和缓存交互
```

#### 方案三：Write Through
```
写：同步更新缓存和数据库
```

#### 方案四：Write Behind
```
写：异步批量写数据库
```

### 一致性保障建议

| 场景 | 推荐方案 | 原因 |
|-----|---------|------|
| 数据一致性要求高 | 延迟双删 | 简单有效 |
| 高并发读多写少 | Cache Aside + 过期 | 性能优先 |
| 强一致性 | 分布式锁 + 串行写 | 牺牲性能换安全 |

### 延迟双删实现
```python
def update_user(user_id, data):
    # 1. 先更新数据库
    db.update(user_id, data)
    # 2. 删除缓存
    redis.delete(f"user:{user_id}")
    # 3. 延迟一段时间后再次删除（异步）
    sleep(0.5)
    redis.delete(f"user:{user_id}")
```

### 分布式锁保底
对一致性要求极高的场景，使用 Redisson 分布式锁：
```java
RLock lock = redisson.getLock("update_user");
try {
    lock.lock(10, TimeUnit.SECONDS);
    // 1. 更新数据库
    // 2. 更新缓存
} finally {
    lock.unlock();
}
```

**核心原则**：根据业务对一致性容忍度选择方案，通常最终一致性即可满足大多数场景。
