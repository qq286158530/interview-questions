## 🔥 Spark 核心基础

---

### 题目1：Spark 架构是怎样的？DAG、Stage、Task 是如何划分的？

【答案】

**Spark 架构核心组件**：

| 组件 | 作用 |
|------|------|
| Driver | 运行用户代码，创建 SparkContext，负责 DAG 切分、Task 调度 |
| Executor | 真正执行 Task 的 worker 进程，每个节点一个或多个，负责数据存储和计算 |
| Cluster Manager | 资源管理器（Standalone/YARN/Kubernetes），负责资源分配 |
| SparkContext | 程序的入口点，连接 Driver 与 Cluster Manager |

**DAG 划分 Stage 的流程**：

1. 用户代码 → SparkContext → DAGScheduler
2. DAGScheduler 将 DAG 划分为多个 Stage（依据是否发生 Shuffle）
3. **窄依赖**（map、filter）→ 同一 Stage 内，流水线（pipelining）执行
4. **宽依赖**（groupByKey、join、reduceByKey）→ 触发 Shuffle，形成新的 Stage
5. 每个 Stage 包含多个 Task（并行度 = 分区数），Task 发送给 Executor 执行

**关键原则**：
- 窄依赖同一 Stage 内可流水线执行，无需 Shuffle
- 宽依赖是划分 Stage 的边界
- 一个 Partition 对应一个 Task

📖 来源：https://github.com/apache/spark/blob/master/core/src/main/scala/org/apache/spark/scheduler/DAGScheduler.scala

### 题目2：RDD 是什么？为什么 RDD 需要 Action 才会真正执行？

【答案】

**RDD（Resilient Distributed Dataset）** 是 Spark 的核心数据结构，是一个不可变、可分区、分布式的数据集，支持粗粒度转换操作（map、filter 等）。

**RDD 五大特性**：
1. **分区列表**：RDD 由多个分区组成，每个分区可在不同节点上并行计算
2. **依赖关系**：每个 RDD 记录其父 RDD 的依赖关系（窄依赖/宽依赖）
3. **计算函数**：每个 RDD 知道如何在自己的分区上计算
4. **分区器**（可选）：对于 Key-Value RDD，可指定 Hash 或 Range 分区器
5. **位置偏好**（可选）：优先将数据放在特定节点（数据本地性）

**为什么 Action 才触发执行？**

```
RDD 转换操作（Transformation） → 懒加载（Lazy Evaluation）
       ↓ 记录 lineage（DAG）
Action（collect、count、saveAsTextFile...） → 触发真正计算
```

- **Transformation 只是记录计算逻辑**：如 `rdd.map(...).filter(...)` 只是构建了 DAG，并未真正执行
- **Action 触发 Job**：调用 Action 时，SparkContext 才会提交 Job，DAGScheduler 切分 Stage，TaskScheduler 分发 Task 给 Executor
- **惰性求值的好处**：减少中间结果落盘，优化执行计划（如 pipelining、pipelineFused）

**经典面试追问**：
- `rdd.collect()` 和 `rdd.count()` 的区别？——collect 会拉取全量数据到 Driver，count 只返回数字
- `rdd.take(n)` vs `rdd.collect()` ——take 只取前 n 条，避免 OOM

📖 来源：https://spark.apache.org/docs/latest/rdd-programming-guide.html

### 题目3：DataFrame 和 RDD 有什么区别？DataFrame 的 Catalyst 优化器是如何工作的？

【答案】

**DataFrame vs RDD 对比**：

| 特性 | RDD | DataFrame |
|------|-----|-----------|
| 抽象层次 | 低级（原始数据） | 高级（类 SQL 结构化） |
| 数据结构 | JVM 对象（Java/Scala 对象） | Row 对象的分布式数据集 |
| 类型安全 | 编译时检查 | 运行时检查（Dataset[T] 提供编译时检查） |
| 优化 | 无自动优化（依赖用户） | Catalyst 优化器自动优化 |
| 操作方式 | 函数式转换 | DSL / SQL |
| 性能 | 较低 | 较高（有优化） |
| 序列化 | Java/Kryo | Tungsten 二进制序列化 |

**Catalyst 优化器工作流程**：

```
User Code (DSL/SQL)
       ↓
Unresolved Logical Plan（未解析的逻辑计划）
       ↓ [Analyzer + Rules]
Logical Plan（逻辑计划）
       ↓ [Optimization Rules]
Optimized Logical Plan（优化后的逻辑计划）
       ↓ [Physical Planning]
Multiple Physical Plans
       ↓ [Cost Model]
Selected Physical Plan（选择的物理计划）
       ↓ [Code Generation]
Executed RDDs
```

**常见优化规则**：
- **谓词下推（Predicate Pushdown）**：filter 尽早执行，减少数据量
- **列裁剪（Column Pruning）**：只读取需要的列
- **常量折叠（Constant Folding）**：提前计算常量表达式
- **广播 Join（Broadcast Join）**：小表广播，避免 Shuffle
- **空值降级**：提前过滤 null 值

📖 来源：https://databricks.com/blog/2015/04/13/deep-dive-into-spark-sqls-catalyst-optimizer.html

### 题目4：Spark Shuffle 的过程是什么？为什么 Shuffle 会影响性能？如何优化？

【答案】

**Shuffle 本质**：将数据按照 Key 进行重新分区，跨节点拉取数据，涉及磁盘 I/O、网络 I/O、序列化/反序列化。

**Shuffle 写（Map 端）**：
1. Map Task 计算结果写入本地磁盘（Executor 所在节点）
2. 按照 Reduce 端分区数，分成多个 bucket（文件）
3. 可能触发溢写（Spill to Disk）：内存达到阈值时溢写到磁盘

**Shuffle 读（Reduce 端）**：
1. Reduce Task 向多个 Map 节点拉取数据（基于 ShuffleReader）
2. 先拉取到 Reduce 端内存（可能 Spill）
3. 合并、排序（如果需要）

**Shuffle 文件管理**：
- Spark 1.2 之前：Hash Shuffle（每个 Map 生成 M*R 个文件，小文件过多）
- Spark 1.2+：Sort Shuffle（每个 Map 只生成 R 个文件）

**Shuffle 性能影响原因**：
1. **磁盘 I/O**：数据先落盘再网络传输
2. **网络 I/O**：跨节点数据传输
3. **内存压力**：Shuffle 数据需要缓存，容易 OOM
4. **HDFS 短写问题**：HDFS 不擅长大量小文件写入

**优化方案**：
| 优化方向 | 具体措施 |
|----------|----------|
| 减少 Shuffle 数据量 | 预聚合（reduceByKey 优于 groupByKey） |
| 广播小表 | `broadcast(小表)` 避免 Shuffle |
| 增加并行度 | `spark.sql.shuffle.partitions`（默认200） |
| 数据本地性 | 调整 `spark.locality.wait` |
| 使用 Kryo 序列化 | `spark.serializer=org.apache.spark.serializer.KryoSerializer` |
| 开启 Consume 端压缩 | `spark.shuffle.compress=true` |

📖 来源：https://github.com/apache/spark/blob/master/core/src/main/scala/org/apache/spark/shuffle/

### 题目5：Spark 广播变量是什么？如何使用？广播Join的原理？

【答案】

**广播变量（Broadcast Variable）**：将只读变量高效地分发到所有 Executor，每个 Executor 只需加载一份副本到内存，避免每个 Task 都传输一份数据。

**使用方式**：
```scala
// 广播小表
val broadcastVar = sc.broadcast(Array(1, 2, 3))

// 在 Task 中使用 .value 获取值
val result = broadcastVar.value.map(_ * 2)

// 广播 DataFrame
import org.apache.spark.sql.functions.broadcast
val result = bigDF.join(broadcast(smallDF), "key")
```

**广播 Join 原理**：

| 步骤 | 说明 |
|------|------|
| 1. 判断小表 | Spark 通过 `spark.sql.autoBroadcastJoinThreshold`（默认10MB）判断 |
| 2. 广播小表 | 小表数据广播到所有 Executor 内存 |
| 3. 广播传输 | 通过 TorrentBroadcast 或 HttpBroadcast 传输（基于 BitTorrent 思想） |
| 4. 执行 Join | 大表每个分区的 Task 在本地与广播变量 Join，无 Shuffle |

**为什么比 Shuffle Join 快**：
- Shuffle Join：M 个 Map 分区 × R 个 Reduce 分区 = M×R 次网络传输
- Broadcast Join：只需广播一次，所有分区本地 Join，无跨节点 Shuffle

**注意事项**：
- 广播变量不宜过大（超过 `spark.broadcast.blockSize` 会失败）
- 广播后变量只读，修改不会生效
- 广播前可先 `broadcast(df.cache())` 缓存

📖 来源：https://spark.apache.org/docs/latest/tuning.html#broadcasting-large-variables

## 💾 内存管理与调优

---

### 题目6：Spark 内存模型是怎样的？堆内与堆外内存如何划分？

【答案】

**Spark 内存管理（统一内存管理，Spark 1.6+）**：

```
┌─────────────────────────────────────────────────┐
│              Execution Memory                   │
│    (Shuffle、Sort、Hash Join 等操作)             │
│         spark.memoryfraction × 0.5              │
├──────────────────────┬──────────────────────────┤
│   Execution Memory    │    Storage Memory        │
│  (运行内存，50%)       │   (存储内存，50%)         │
│                       │   (缓存 RDD/Broadcast)    │
│   可借用 Storage       │   可借用 Execution        │
├───────────────────────┴─────────────────────────┤
│            User Memory (用户代码，40%)           │
│    (RDD 创建对象、用户数据结构等)                 │
│   (1 - spark.memory.fraction) × 0.6              │
├─────────────────────────────────────────────────┤
│            Reserved Memory (300MB 预留)          │
└─────────────────────────────────────────────────┘
```

**堆内 vs 堆外内存**：

| 类型 | 配置 | 用途 |
|------|------|------|
| 堆内内存（On-heap） | `--executor-memory` | 执行和存储共用的统一内存区 |
| 堆外内存（Off-heap） | `spark.memory.offHeap.enabled`（默认 false） | Tungsten 直接操作堆外内存，避免 GC |

**内存参数调优**：
```bash
spark.executor.memory=4g
spark.memory.fraction=0.6          # 执行+存储占总内存比例
spark.memory.storageFraction=0.5    # Storage 占 Execution 的比例
spark.memory.offHeap.enabled=true  # 启用堆外内存
spark.memory.offHeap.size=2g       # 堆外内存大小
```

**常见内存问题**：
- **OOM**：数据量超过存储内存 + 借用执行内存上限
- **Spill**：执行内存不足时溢写到磁盘
- **GC 频繁**：堆内内存碎片，可开启堆外内存

📖 来源：https://spark.apache.org/docs/latest/configuration.html#memory-management
