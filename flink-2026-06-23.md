## 题目一：Flink 的架构是怎样的？请详细说明各组件及其作用

### 参考答案

Apache Flink 是一个分布式流处理框架，其核心架构采用 **Master-Slave** 模式，主要包含以下组件：

#### 1. JobManager（Master 节点）
- **作用**：负责整个集群的调度和协调，是 Flink 集群的领导者
- **核心组件**：
  - **Dispatcher**：接收用户提交的 Job，为每个 Job 启动一个新的 JobMaster
  - **ResourceManager**：管理集群资源（TaskManager 的注册、注销、Slots 分配）
  - **JobMaster**：负责单个 JobGraph 的执行，协调 JobExecution
- **高可用**：支持基于 ZooKeeper 的 HA 配置

#### 2. TaskManager（Worker 节点）
- **作用**：负责执行具体的 Task 和算子，是真正的工作节点
- **核心机制**：
  - 每个 TaskManager 是一个 JVM 进程
  - 通过 Slot 来管理资源，每个 Slot 可以执行一个 Task
  - 多个 Task 可以共享一个 Slot（适合资源密集型算子）

#### 3. Slot 与资源管理
- **Slot 是 Flink 资源分配的基本单位**
- 每个 Slot 有固定的资源（内存 + CPU）
- 一个 TaskManager 默认有多少个 Slot 取决于配置
- **Slot 共享**：同一作业的不同算子可以在同一个 Slot 中运行

#### 4. 数据流执行
```
DataStream API / Table API / SQL
         ↓
    JobGraph（用户代码编译而来）
         ↓
    ExecutionGraph（JobMaster 转换）
         ↓
    物理执行计划（分配到 TaskManager 执行）
```

**执行特点**：
- Streaming Job 默认为流水线（pipelined）执行
- Batch Job 支持流水线或换行（batched）执行
- 支持 exactly-once、at-least-once、at-most-once 三种语义

## 题目二：什么是 Watermark？Flink 如何处理乱序数据和迟到事件？

### 参考答案

#### 1. Watermark（水位线）的概念

**Watermark 是 EventTime 窗口触发的一个判定机制**，本质是一个时间戳，用于告诉 Flink：「在这个时间戳之前的数据都已经到达了」。

```java
// Watermark 生成示例
WatermarkStrategy
    .forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner(...)
```

#### 2. 为什么需要 Watermark？

在现实场景中：
- 网络延迟
- 消息重试
- 数据乱序

这些问题导致数据**不按事件时间顺序**到达。Watermark 就是用来**处理乱序**的机制。

#### 3. Watermark 的两种策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **固定延迟（Fixed Watermark）** | 固定延迟 N 秒 | 数据延迟可控 |
| **乱序容忍度（BoundedOutOfOrderness）** | 允许最多延迟 N 秒 | 数据乱序不确定 |
| **单调递增（Monotonous Watermark）** | 只取当前最大时间戳 | 严格递增数据流 |

#### 4. Flink 处理迟到事件的机制

**Watermark 策略**：
```java
// 允许最大乱序时间为 5 秒
WatermarkStrategy
    .forBoundedOutOfOrderness(Duration.ofSeconds(5))
```

**窗口延迟关闭**：
```java
// 窗口触发后，额外等待 3 秒再关闭窗口
window.allowLateMerging(true)
```

**三种处理策略**：

| 策略 | 效果 | 配置方式 |
|------|------|----------|
| **默认（丢弃）** | 直接丢弃迟到数据 | - |
| **allowedLateness** | 数据保留一段时间后再丢弃 | `window.allowLateMerging()` |
| **sideOutputLateData** | 将迟到数据输出到侧输出流 | `window.sideOutputLateData()` |

#### 5. 经典面试题示例

```java
// 完整示例：处理乱序 + 迟到数据
WatermarkStrategy
    .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(20))  // 乱序容忍度
    .withTimestampAssigner((event, timestamp) -> event.getTimestamp())
    .withIdleness(Duration.ofMinutes(1));  // 处理空流

// 窗口设置
window
    .allowedLateness(Time.seconds(5))  // 允许迟到 5 秒
    .sideOutputLateData(lateOutputTag);  // 侧输出迟到数据
```

## 题目三：Flink 的 Checkpoint 机制是如何工作的？为什么需要 Checkpoint？

### 参考答案

#### 1. 为什么需要 Checkpoint？

**流处理作业是 7×24 小时运行的**，如果程序崩溃重启：
- **没有 Checkpoint**：从上次保存状态的位置重新消费，从头开始处理 → **数据重复处理/丢失**
- **有 Checkpoint**：从最近快照恢复状态，继续处理 → **精确一次（exactly-once）语义**

#### 2. Checkpoint 的核心原理

Flink 采用 **Chandy-Lamport 分布式快照算法**的变体来实现 Checkpoint。

**核心概念**：

| 概念 | 说明 |
|------|------|
| **Barrier（屏障）** | 从 Source 插入的特殊标记，区分 checkpoint 的数据范围 |
| **Checkpoint Coordinator** | JobManager 中的协调者，负责触发和协调 |
| **CheckpointID** | 每个 Checkpoint 的唯一标识 |
| **State Backend** | 状态后端，存储快照数据 |

#### 3. Checkpoint 执行流程

```
1. Checkpoint Coordinator 向 Source 发送 Checkpoint 触发消息
2. Source 向下游广播 Barrier，同时保存自身状态到 State Backend
3. 每个算子收到所有输入的 Barrier 后，保存状态并向下游传递
4. 所有算子完成快照后，向 Coordinator 确认
5. Checkpoint 完成，保存到指定存储（如 HDFS/S3）
```

#### 4. Barrier 对齐机制（Exactly-Once）

```
多输入算子（如 Join）的 Barrier 对齐：
     Stream1 [barrier] ──┐
                         ├──→ 算子（必须等所有 barrier 到达后才能处理）
     Stream2 [  数据  ] ──┘  ↑ 等待中...
                               ↑
                               此时 Stream1 的数据会被缓存
```

**注意**：Barrier 对齐期间，缓存的数据会导致反压。

#### 5. Checkpoint 配置

```java
// 启用 Checkpoint，间隔 10 秒
env.enableCheckpointing(10_000);

// 高级配置
CheckpointConfig config = env.getCheckpointConfig();
config.setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);
config.setMinPauseBetweenCheckpoints(5_000);  // 两次 checkpoint 最小间隔
config.setCheckpointTimeout(120_000);  // checkpoint 超时时间
config.setTolerableCheckpointFailureNumber(3);  // 容忍失败次数
config.setMaxConcurrentCheckpoints(1);  // 最大并发数
```

#### 6. State Backend 选择

| State Backend | 存储介质 | 适用场景 |
|--------------|---------|----------|
| **HashMapStateBackend** | JVM Heap | 小状态、低延迟 |
| **EmbeddedRocksDBStateBackend** | RocksDB + 磁盘 | 大状态、长窗口 |
| **EmbeddedRocksDBStateBackend（增量）** | RocksDB 增量快照 | 超大状态 |

#### 7. Unaligned Checkpoint（不透明检查点）

Flink 1.11+ 引入的优化，当 Barrier 对齐时间长时：
- **对齐**：将未处理的数据也纳入快照
- **优点**：减少反压影响，提高 Checkpoint 成功率
- **配置**：`env.enableCheckpointing(..., CheckpointingMode.EXACTLY_ONCE, true)`

## 题目四：Flink 窗口函数有哪些类型？滚动窗口、滑动窗口、会话窗口有什么区别？

### 参考答案

#### 1. 窗口概述

窗口是 Flink 处理无限流的核心机制，将无限流切分为有限桶进行聚合计算。

```
无界数据流 → [窗口1] [窗口2] [窗口3] → 有界结果
            ↓          ↓         ↓
         [w=1,2]   [w=3,4]   [w=5,6]
```

#### 2. 窗口类型

##### 2.1 时间窗口（Time Window）

| 类型 | 说明 | 示例 |
|------|------|------|
| **滚动窗口（Tumbling）** | 固定大小，不重叠 | 每 5 分钟一个窗口 |
| **滑动窗口（Sliding）** | 固定大小，可重叠 | 每 5 分钟窗口，滑动步长 1 分钟 |
| **会话窗口（Session）** | 动态大小，按活跃间隙划分 | 间隔 10 秒无数据则关闭窗口 |

##### 2.2 计数窗口（Count Window）

基于元素数量的窗口，不常用（生产环境推荐时间窗口）。

#### 3. 滚动窗口 vs 滑动窗口

**滚动窗口（TumblingWindow）**：
```
数据流：  t1  t2  t3  t4  t5  t6  t7  t8  t9  t10
窗口(5s): [  t1-t5  ][  t6-t10 ]

特点：无重叠，每个元素只属于一个窗口
```

**滑动窗口（SlidingWindow）**：
```
数据流：  t1  t2  t3  t4  t5  t6  t7  t8  t9  t10
窗口(5s,2s): [  t1-t5  ]
               [  t3-t7  ]
                  [  t5-t9  ]
                     [  t7-t10 ]

特点：有重叠，适合"最近 N 小时内平均值"等场景
```

#### 4. 会话窗口（SessionWindow）

**核心特点**：动态窗口大小，根据活动间隙自动调整

```java
// 会话窗口，间隔 5 秒无数据则关闭
window
    .windowAll(SessionWindows.withGap(Time.seconds(5)))
    .reduce((a, b) -> merge(a, b));
```

**动态间隙会话窗口**：
```java
// 根据数据本身动态决定会话间隔
window.windowAll(DynamicProcessingTimeSessionWindows
    .withDynamicGap((element) -> element.getInterval()));
```

#### 5. 全窗口函数 vs 增量聚合函数

| 类型 | 函数 | 特点 |
|------|------|------|
| **增量聚合** | `reduce()`, `fold()`, `aggregate()` | 来一条处理一条，状态小，性能好 |
| **全窗口** | `apply()`, `process()` | 先收集所有数据，最后一起处理，状态大 |
| **混合** | `aggregate()` + `ProcessWindowFunction` | 增量聚合 + 额外上下文信息 |

#### 6. 完整代码示例

```java
// 滑动窗口：统计最近 5 分钟，每 1 分钟更新一次
DataStream<SensorReading> input = ...
input
    .keyBy(r -> r.getId())
    .window(SlidingEventTimeWindows.of(Time.minutes(5), Time.minutes(1)))
    .aggregate(new AvgTemperatureFunction());

// 会话窗口
input
    .keyBy(r -> r.getId())
    .window(EventTimeSessionWindows.withGap(Time.seconds(10)))
    .sum("temperature");
```

#### 7. 窗口分配器（Window Assigner）

```java
// 常用窗口分配器
TumblingEventTimeWindows.of(Time.minutes(5))    // 滚动
SlidingEventTimeWindows.of(Time.minutes(5), Time.minutes(1))  // 滑动
EventTimeSessionWindows.withGap(Time.minutes(10))  // 会话
CountWindow.of(100)  // 计数滚动窗口
```

## 题目五：Flink 的状态管理机制是怎样的？算子状态和键控状态有什么区别？

### 参考答案

#### 1. Flink 状态概述

**状态（State）**是 Flink 实现精确一次语义的基础，是算子在处理过程中需要记住的信息。

```
算子处理：Stream → [读取状态] → [计算] → [更新状态] → 输出
```

#### 2. 状态类型

##### 2.1 按数据结构分类

| 状态类型 | 说明 | 示例 |
|---------|------|------|
| **ValueState** | 单值状态 | `state.update(newValue)` |
| **ListState** | 列表状态 | `state.add(newElement)` |
| **MapState** | 键值对状态 | `state.put(key, value)` |
| **ReducingState** | 聚合后状态 | 自动聚合 |
| **AggregatingState** | 自定义聚合状态 | 中间结果类型可不同 |

##### 2.2 按作用范围分类（核心区别）

| 状态类型 | 作用域 | 并行度影响 | 使用场景 |
|---------|--------|-----------|----------|
| **算子状态（Operator State）** | 当前算子实例 | 重启后重新分配 | Source 偏移量、聚合中间结果 |
| **键控状态（Keyed State）** | 当前 Key | 每个 Key 独立 | 按 Key 聚合统计 |

#### 3. 算子状态（Operator State）

**特点**：
- 作用域是**算子实例**（一个并行度实例）
- 重启后需要重新分配（ListState / UnionListState）
- 常用于 Source 的消费偏移量

```java
public class SourceFunctionWithState implements SourceFunction<String> {
    private transient ListState<Long> offsetState;
    
    @Override
    public void initializeState(Context context) {
        offsetState = context.getOperatorStateStore()
            .getListState(new ListStateDescriptor<>("offset", Long.class));
    }
}
```

**ListState 重新分配策略**：
- **Even-split redistribution**：平均分配给新实例
- **Union redistribution**：每个实例获取完整列表
- **Broadcast state**：所有实例都有相同副本

#### 4. 键控状态（Keyed State）

**特点**：
- 作用域是**当前 Key**
- 隐式按 Key 分组，只能在 `keyBy()` 后使用
- 每个 Key 维护独立状态

```java
public class TemperatureAggregator
    extends KeyedProcessFunction<String, SensorReading, String> {
    
    // 键控状态声明
    private ValueState<Double> lastTemp;
    private ValueState<Long> count;
    private ListState<Double> allTemps;
    
    @Override
    public void open(Configuration parameters) {
        lastTemp = getRuntimeContext().getState(
            new ValueStateDescriptor<>("lastTemp", Double.class));
        count = getRuntimeContext().getState(
            new ValueStateDescriptor<>("count", Long.class));
        allTemps = getRuntimeContext().getListState(
            new ListStateDescriptor<>("allTemps", Double.class));
    }
    
    @Override
    public void processElement(SensorReading in, Context ctx, Collector<String> out) {
        // 使用状态
        Double prev = lastTemp.value();
        lastTemp.update(in.getTemperature());
        count.update(count.value() + 1);
    }
}
```

#### 5. 状态后端（State Backend）

| 后端 | 存储 | 性能 | 适用场景 |
|------|------|------|----------|
| **HashMapStateBackend** | JVM Heap | 最快 | 小状态、低延迟 |
| **EmbeddedRocksDBStateBackend** | RocksDB（内存+磁盘） | 中等 | 大状态、大窗口 |
| **EmbeddedRocksDBStateBackend（增量）** | RocksDB 增量快照 | 可控 | 超大状态 |

```java
// 配置状态后端
env.setStateBackend(new HashMapStateBackend());
env.setStateBackend(new EmbeddedRocksDBStateBackend());

// 或在 flink-conf.yaml
state.backend: rocksdb
state.backend.incremental: true
```

#### 6. 状态 TTL（Time-To-Live）

```java
// 状态过期清理
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.hours(1))
    .setUpdateType(StateTtlConfig.UpdateType.OnCreateAndWrite)  // 创建/写入时更新
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)  // 不返回过期数据
    .cleanupFullSnapshot()  // 全量快照时清理
    // .cleanupInRocksdbCompactFilter(1000)  // RocksDB compaction 时清理
    .build();

ValueStateDescriptor<Double> stateDesc = 
    new ValueStateDescriptor<>("temp", Double.class);
stateDesc.enableTimeToLive(ttlConfig);
```

#### 7. 状态管理最佳实践

1. **避免大状态**：状态过大会拖慢 Checkpoint
2. **选择合适的 State Backend**：根据状态大小选择
3. **及时清理状态**：使用 TTL 清理无用状态
4. **避免使用 Pojo 类型**：使用原始类型减少序列化开销
5. **监控状态大小**：`Metric: numKeyedStates, stateSize`
