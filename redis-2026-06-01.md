## 题目一：Redis 有哪些数据类型？

### 参考答案

Redis 支持五种基本数据类型和三种特殊数据结构：

**基础数据类型：**
- **String（字符串）**：最基础的类型，可存储字符串、整数或浮点数。底层实现为 SDS（Simple Dynamic String），支持动态扩容和二进制安全。
- **List（列表）**：按插入顺序存储字符串元素，底层是双向链表或压缩列表（ziplist），常用于消息队列、任务队列。
- **Set（集合）**：无序且不重复的字符串集合，支持交、并、差运算，底层是 intset（全为整数时）或 hashtable。
- **Hash（哈希）**：键值对集合，适合存储对象，底层是 ziplist（字段少时）或 hashtable。
- **Zset（有序集合）**：每个元素关联一个分数，按分数排序，支持范围查询，底层是跳表（skiplist）配合 hashtable。

**特殊数据结构：**
- **HyperLogLog**：用于基数统计，内存占用极低（约 12KB），适合 UV 统计。
- **Geospatial**：地理空间索引，支持地理位置存储与距离计算。
- **Bitmaps**：位图，按位操作，适合签到、状态监控等场景。

**来源：** [Redis 数据类型详解 — Redis 官方文档](https://redis.io/docs/data-types/)

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？

### 参考答案

Redis 提供了两种持久化机制：

**RDB（Redis Database）：**
- 定时将内存中的所有数据快照（Snapshot）保存到磁盘上的 `dump.rdb` 文件。
- 触发方式：手动执行 `BGSAVE` 命令，或配置 `save` 策略（如 `save 900 1` 表示 900 秒内有 1 次写操作则触发）。
- **优点**：恢复速度快，适合大规模数据备份。
- **缺点**：最后一次快照到故障之间的数据会丢失。

**AOF（Append Only File）：**
- 记录所有写操作命令，以追加方式写入 `appendonly.aof` 文件。
- 同步策略：`always`（每条命令都同步）、`everysec`（每秒同步，默认）、`no`（由操作系统决定）。
- 支持重写（Rewrite）：合并冗余命令，压缩文件体积。
- **优点**：数据安全性更高，Redis 4.0 后支持混合持久化（RDB + AOF）。
- **缺点**：文件体积通常比 RDB 大，恢复速度较慢。

**生产环境推荐配置：** 开启 AOF + RDB 混合持久化，同时配置 `appendonly yes` 和合适的 `save` 策略。

**来源：** [Redis 持久化详解 — Redis 官方文档](https://redis.io/docs/management/persistence/)

## 题目三：什么是缓存穿透、缓存击穿、缓存雪崩？如何解决？

### 参考答案

**缓存穿透：**
- 指查询一个**不存在的数据**，缓存和数据库都没有，导致请求直接打到数据库。
- **解决方案：**
  1. 布隆过滤器（BloomFilter）：在缓存层前加一层过滤器，存在则查缓存，不存在直接返回。
  2. 缓存空值（null）：将空结果也缓存起来，设定较短的过期时间。
  3. 参数校验：拦截非法参数请求。

**缓存击穿：**
- 指**一个热点 key 过期**，瞬间大量并发请求击穿缓存直接打数据库。
- **解决方案：**
  1. 互斥锁（Mutex）：只允许一个线程从数据库加载数据，其他线程等待。
  2. 逻辑过期：不设过期时间，异步更新缓存，保证数据始终可用。
  3. 热点数据永不过期（配合异步更新）。

**缓存雪崩：**
- 指**大量缓存 key 同时过期**，或缓存服务宕机，导致大量请求打到数据库。
- **解决方案：**
  1. 过期时间随机化：为 key 的 TTL 添加随机值，避免同时过期。
  2. 多级缓存：本地缓存 + Redis 多级缓存架构。
  3. Redis 高可用：哨兵（Sentinel）或集群（Cluster）保证可用性。
  4. 服务降级/熔断：数据库压力大时返回兜底数据。

**来源：** [Redis 缓存问题及解决方案 — 美团技术团队](https://tech.meituan.com/2017/03/17/cache.html)

## 题目四：Redis 的过期 key 删除策略是什么？

### 参考答案

Redis 采用两种过期 key 删除策略配合使用：

**惰性删除（Lazy Expiration）：**
- 指**每次访问 key 时**才检查是否过期，过期则删除。
- 优点：对 CPU 友好，不会在 key 过期时立即删除。
- 缺点：如果 key 永不被访问，会一直占用内存（内存泄漏风险）。

**定期删除（Active Expiration）：**
- Redis 每隔一段时间（默认 100ms，配置 `hz` 参数可调整）随机抽取一批设置了过期时间的 key 检查删除。
- 优点：主动清理过期 key，释放内存。
- 缺点：对 CPU 有一定开销。

**内存淘汰策略（当内存达到上限时）：**
当内存不足以容纳新数据时，Redis 通过 `maxmemory-policy` 配置选择淘汰策略：
- `noeviction`：不淘汰，返回错误（默认）。
- `volatile-lru`：在设置了过期时间的 key 中淘汰最近最少使用的（LRU）。
- `allkeys-lru`：所有 key 中淘汰最近最少使用的。
- `volatile-ttl`：淘汰最早过期的 key。
- `volatile-random`：随机淘汰设置了过期时间的 key。
- `allkeys-random`：随机淘汰所有 key。
- `volatile-lfu` / `allkeys-lfu`：淘汰最少使用的（LFU，Redis 6.0+）。

**来源：** [Redis 过期策略详解 — Redis 官方文档](https://redis.io/commands/expire/)

## 题目五：Redis 主从复制原理是什么？如何实现读写分离？

### 参考答案

**主从复制原理：**

Redis 主从复制采用 **PSYNC（Partial Sync）** 机制，支持全量同步和增量同步：

1. **建立连接：** 从节点（replica）保存主节点（master）地址，向主节点发送 `PSYNC` 命令。
2. **全量同步：**
   - 主节点 fork 一个子进程执行 `BGSAVE`，生成 RDB 快照。
   - 主节点将快照期间的操作命令记录到 **复制缓冲区（replication buffer）**。
   - 将 RDB 文件发送给从节点，从节点加载到内存。
   - 主节点再将缓冲区中的命令发送给从节点，完成同步。
3. **增量同步（断线重连时）：**
   - 从节点记录已处理的命令 offset，主节点根据 offset 补发缺失的命令。
4. **心跳检测：** 主从节点每 10 秒发送心跳包（`replconf ack {offset}`），检测连接状态。

**读写分离：**
- 架构上，所有**写操作**走主节点，**读操作**可分配到从节点。
- 好处：减轻主节点压力，提升整体 QPS。
- 注意事项：
  - 从节点数据可能存在**延迟**，不适合强一致性场景。
  - 可配置 `read-only yes` 开启从节点只读。
  - 建议配合 **哨兵（Sentinel）** 或 **Codis** 等实现故障自动切换。

**Redis 6.0 多线程 IO：**
- Redis 6.0 引入了多线程 IO 模型，用于处理网络请求的读取和解析，但仍保持单线程命令执行。

**来源：** [Redis 主从复制 — Redis 官方文档](https://redis.io/docs/management/replication/)
