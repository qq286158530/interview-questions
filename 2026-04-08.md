### 题目1：String 为什么是不可变的？底层原理是什么？

【答案】
String 不可变主要有以下原因：

1. **final 关键字修饰**：String 类被 `final` 修饰，不能被继承，防止子类修改
2. **private final char[] value**：String 内部使用 `private final char[] value` 存储字符数组，且不对外提供修改入口
3. **字符数组引用不可变**：value 数组被 final 修饰，引用不可变
4. **无修改方法**：String 类没有提供公开的修改内部数组的方法

好处：
- **线程安全**：不可变对象天然线程安全
- **字符串常量池**：节省内存，相同字符串可复用
- **HashCode 缓存**：不可变才能缓存 hashCode
- **安全性**：防止类加载器混淆，常用于网络连接、文件路径等

### 题目2：synchronized 和 ReentrantLock 有哪些区别？

【答案】

| 特性 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 底层实现 | JVM 内置关键字 | JDK 实现（AbstractQueuedSynchronizer） |
| 锁类型 | 悲观锁/可重入锁 | 悲观锁/可重入锁（可重入次数可设置） |
| 等待可中断 | 不可中断 | 可通过 `lockInterruptibly()` 中断 |
| 公平锁 | 非公平（默认） | 可选公平/非公平 |
| 条件变量 | 内置 wait/notify/notifyAll | 可用 `newCondition()` 多个条件队列 |
| 释放方式 | 自动释放（monitorenter/exit） | 必须 finally 中 unlock() |
| 锁绑定 | 无 | 可绑定 `Condition` |

使用建议：
- 简单同步用 synchronized（更简洁）
- 需要灵活控制用 ReentrantLock（定时锁、公平锁、中断）

### 题目3：JVM 双亲委派模型是什么？有什么作用？

【答案】

**双亲委派机制**：
类加载请求向上传递直到 Bootstrap ClassLoader，只有父加载器无法完成时才自己加载。

**类加载器层次**：
1. Bootstrap ClassLoader（C++编写，加载 JAVA_HOME/lib）
2. Extension ClassLoader（加载 JAVA_HOME/lib/ext）
3. Application ClassLoader（加载 classpath 指定）
4. 自定义 ClassLoader

**作用**：
1. **防止核心类被篡改**：用户自定义的 `java.lang.String` 不会被加载，保证核心类安全
2. **保证类的唯一性**：父加载器已加载的类不会被子加载器重复加载
3. **安全性基础**：防止恶意类冒充系统类

**打破双亲委派**：
- 热部署：OSGI、Tomcat
- JDBC 4.0后通过 `ServiceLoader`
- JDK 9 模块化系统

### 题目4：IoC 和 DI 是什么？依赖注入有哪些方式？

【答案】

**IoC（控制反转）**：
将对象的创建和管理权反转给 Spring 容器，之前是程序主动 new，现在是被动接收。

**DI（依赖注入）**：IoC 的一种实现方式，在对象创建时自动注入其所依赖的其他对象。

**注入方式**：
1. **构造器注入**（推荐）：通过构造函数注入，强制依赖，利于不可变对象
2. **Setter 注入**：通过 setter 方法注入，可选依赖
3. **字段注入**（@Autowired）：直接注入字段，不推荐（测试困难）

**Spring 支持方式**：
- `@Autowired`：Spring 官方
- `@Inject`：Java 标准（JSR-330）
- `@Resource`：Java 标准（JSR-250）

### 题目6：Spring Bean 的作用域有哪些？线程安全如何保证？

【答案】

**作用域（Scope）**：
| 作用域 | 说明 |
|--------|------|
| singleton | 默认，整个容器只有一个实例 |
| prototype | 每次获取创建新实例 |
| request | HTTP 请求中一个实例（Web） |
| session | HTTP Session 中一个实例（Web） |
| application | ServletContext 中一个实例（Web） |
| websocket | WebSocket 中一个实例 |

**线程安全**：
Spring 容器中的 Bean 默认是 singleton，多线程下共享存在线程安全问题。

**解决方案**：
1. **无状态 Bean**：不存储可变成员变量（推荐）
2. **ThreadLocal**：使用 ThreadLocal 存储线程局部变量
3. **避免并发**：将 scope 改为 prototype（性能开销大）
4. **同步控制**：使用 `synchronized` 或 `ReentrantLock`

**最佳实践**：保持 Bean 无状态，设计为不可变对象。

### 题目8：Kafka 为什么能实现高吞吐量？架构层面如何设计？

【答案】

**高吞吐量原因**：

1. **顺序写磁盘**：利用顺序 I/O，写入速度接近内存
2. **页缓存（Page Cache）**：Linux 系统的内存缓存，读取时优先从缓存读
3. **零拷贝（Zero Copy）**：使用 sendfile 系统调用，数据直接在内核空间传输，避免用户态/内核态切换
4. **批量处理**：消息批量发送、批量压缩（支持 gzip、snappy、lz4）
5. **分区（Partition）**：水平扩展，并行处理
6. **稀疏索引**：稀疏哈希索引，快速定位消息

**架构设计**：
```
Producer → Topic(多个Partition) → Consumer Group
              ↓
         Partition副本（ISR机制）
```

**关键参数优化**：
- `batch.size`：批量大小
- `linger.ms`：等待时间
- `compression.type`：压缩算法
- `acks`：可靠性级别

### 题目9：RabbitMQ 的 Exchange 有哪四种类型？各自用途？

【答案】

| 类型 | 路由规则 | 典型场景 |
|------|----------|----------|
| **direct** | 完全匹配 routing key | 精确路由 |
| **fanout** | 广播到所有绑定队列 | 发布/订阅 |
| **topic** | 通配符匹配（*  #） | 模式匹配 |
| **headers** | 根据 headers 属性匹配 | 多条件路由 |

**详细说明**：

**1. direct（直连交换机）**
```
routing key = "order.created" → 只投递到 order.queue
```

**2. fanout（广播交换机）**
```
所有绑定队列都收到消息（类似 Observer 模式）
```

**3. topic（主题交换机）**
```
*.order.#     → 匹配 xxx.order.xxx
user.*.created → 匹配 user.xxx.created
```

**4. headers（头交换机）**
```java
Map<String, Object> headers = new HashMap<>();
headers.put("format", "pdf");
headers.put("type", "report");
```

**选择建议**：
- 路由键确定 → direct
- 需要模式匹配 → topic
- 广播通知 → fanout
- 多条件路由 → headers

### 题目10：消息可靠性如何保证？三端保障机制是什么？

【答案】

**三端保障机制**：

**1. 生产者端（Producer）**
- `publisher confirms`：确认消息已写入 broker
- `publisher returns`：消息无法路由时回调
- `事务模式`（性能差，不推荐）

**2. Broker 端**
- `副本机制（ISR）`：同步复制到多个副本
- `刷盘策略`：同步刷盘（不丢但慢）/ 异步刷盘（快但可能丢）
- `ack 确认级别`：
  - `acks=0`：不等待， fastest
  - `acks=1`：leader 写入即返回
  - `acks=all`：ISR 全部写入

**3. 消费者端（Consumer）**
- `手动 ack`：处理完业务再确认
- `手动确认 + 重试`：失败后重新入队
- `死信队列（DLQ）`：处理失败的消息进入 DLQ

**最佳实践**：
```
1. 使用 RabbitMQ：事务 + confirm 模式
2. 使用 Kafka：acks=all + 重试机制
3. 业务幂等设计：数据库唯一索引 / Redis 防重
4. 定时任务补偿：定期扫描未确认消息
```

### 题目12：ReAct 模式是什么？ReAct 与 CoT、ToT 有何区别？

【答案】

**ReAct = Reasoning + Acting**

ReAct 让 LLM 同时进行：
1. **推理（Reasoning）**：分析当前状态、思考下一步
2. **行动（Acting）**：调用外部工具、执行动作

**ReAct 流程**：
```
Question → Think → Action → Observation → Think → Action → ... → Final Answer
              ↑______________↓______________↑
                    循环直到得到答案
```

**示例**：
```
问题：北京今天多少度？
Thought：我需要先搜索北京天气
Action：search[北京天气]
Observation：今天北京25度，晴
Thought：已获取到天气信息

```

**ReAct vs CoT vs ToT**：

| 模式 | 特点 | 适用场景 |
|------|------|----------|
| **CoT** | 连续推理，无行动 | 数学推导、逻辑分析 |
| **ToT** | 树形探索，多路径 | 复杂规划、创意写作 |
| **ReAct** | 推理+行动结合 | 需要外部知识的任务 |

**ReAct 优势**：
- 结合推理和行动，信息获取更准确
- 可干预、可解释性强

### 题目13：快速排序的原理是什么？如何优化？

【答案】

**基本原理**：
1. 选择基准元素（pivot）
2. 分区（partition）：将数组分为两部分，左边 < pivot，右边 > pivot
3. 递归排序左右两部分

**时间复杂度**：
- 平均：O(n log n)
- 最坏：O(n²)（数组有序时）

**Java 实现**：
```java
public void quickSort(int[] arr, int low, int high) {
    if (low < high) {
        int pivotIndex = partition(arr, low, high);
        quickSort(arr, low, pivotIndex - 1);
        quickSort(arr, pivotIndex + 1, high);
    }
}

public int partition(int[] arr, int low, int high) {
    int pivot = arr[low];  // 选择第一个元素作为基准
    while (low < high) {
        while (low < high && arr[high] >= pivot) high--;
        arr[low] = arr[high];
        while (low < high && arr[low] <= pivot) low++;
        arr[high] = arr[low];
    }
    arr[low] = pivot;
    return low;
}
```

**优化策略**：

1. **三数取中（Median-of-Three）**
   ```java
   int mid = (low + high) >>> 1;
   if (arr[mid] > arr[high]) swap(arr, mid, high);
   if (arr[low] > arr[high]) swap(arr, low, high);
   if (arr[mid] > arr[low]) swap(arr, mid, low);
   ```
   避免最坏情况

2. **小区间优化**：数据量小时用插入排序

3. **尾递归优化**：减少递归深度

4. **随机基准**：随机选择 pivot

### 题目14：堆排序的原理是什么？建堆时间复杂度为什么是 O(n)？

【答案】

**堆的特性**：
- 完全二叉树
- 大顶堆：父节点 >= 子节点
- 小顶堆：父节点 <= 子节点
- 数组存储时：父节点 i → 子节点 2i+1, 2i+2

**算法流程**：
1. **建堆**：将数组调整为堆结构（从最后一个非叶子节点下沉）
2. **排序**：将堆顶与末尾交换，断开除末尾外部分继续调整

**下沉调整（siftDown）**：
```java
public void heapify(int[] arr, int i, int size) {
    int left = 2 * i + 1;
    int right = 2 * i + 2;
    int largest = i;
    if (left < size && arr[left] > arr[largest]) largest = left;
    if (right < size && arr[right] > arr[largest]) largest = right;
    if (largest != i) {
        swap(arr, i, largest);
        heapify(arr, largest, size);
    }
}
```

**建堆 O(n) 证明**：

高度 h 的节点最多下沉 h 层

```
T(n) = Σ(2^h * (h-1))  for h=1 to log₂n
     = n * Σ((h-1)/2^h)
     = n * 2 = O(n)
```

（等比数列求和：h/2^h 的和收敛于常数）

**复杂度**：
- 建堆：O(n)
- 排序：O(n log n)
- 总空间：O(1)（原地排序）

### 题目15：如何判断链表是否有环？找到环的入口点？

【答案】

**判断有环：快慢指针（Floyd 算法）**
```java
public boolean hasCycle(ListNode head) {
    ListNode slow = head;
    ListNode fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) return true;
    }
    return false;
}
```
- 快指针每次走2步，慢指针每次走1步
- 有环：一定会相遇
- 无环：快指针先到终点

**找环入口点**：
```java
public ListNode detectCycle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) break;
    }
    if (fast == null || fast.next == null) return null;
    
    // 重新从头出发，相遇点即为环入口
    slow = head;
    while (slow != fast) {
        slow = slow.next;
        fast = fast.next;
    }
    return slow;
}
```

**数学证明**：
- 设起点到入口距离为 a，环长度为 b
- 相遇时：slow 走了 a + x，fast 走了 a + x + nb
- 因为 fast 速度是 slow 的2倍：2(a+x) = a + x + nb
- 推出：a = nb - x = (n-1)b + (b-x)
- 所以从 head 走 a 步 = 从 head 走 (n-1)b + (b-x) 步
- 从相遇点走 (b-x) 步也到入口
- 结论：两指针分别从 head 和相遇点出发，再次相遇就是环入口

**复杂度**：
- 时间：O(n)
- 空间：O(1)
