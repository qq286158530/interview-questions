## 题目一：Flink 的状态管理机制是什么？请解释 Keyed State 和 Operator State 的区别。

### 参考答案

Flink 提供了一套完整的状态管理机制，用于在流处理过程中保存计算中间结果，确保故障恢复和精确一次语义。

**状态类型分类：**

1. **Keyed State（键控状态）**
   - 仅限 KeyedStream 上使用，基于当前处理数据的 key 来维护和访问
   - 自动限制在当前 key 的范围内
   - 支持多种数据结构：ValueState、ListState、MapState、ReducingState、AggregatingState、FoldingState
   - 状态随 key 分布存储在各 task manager 中

2. **Operator State（算子状态）**
   -作用于整个算子实例，不依赖 key
   - 所有 operator 实例共享状态（列表模式）或一个实例一个状态
   - 支持三种模式：
     - **ListState**：算子实例故障后均匀分配给其他实例
     - **UnionListState**：故障恢复时所有实例都收到完整状态副本
     - **BroadcastState**：特殊状态，广播到所有分区

**状态存储：**
- 状态后端（State Backend）决定状态的存储方式
- HashMapStateBackend：内存存储，性能高
- EmbeddedRocksDBStateBackend：RocksDB 存储，支持大状态

## 题目二：Flink 的 Watermark 机制是什么？如何处理乱序事件？

### 参考答案

**Watermark（水位线）概念：**

Watermark 是 Flink 用于处理事件时间（Event Time）乱序的一种机制，本质是一个时间戳，用于告诉算子"在此之前的事件都已到达"。

**核心原理：**

```
Watermark(t) 表示：所有时间戳 <= t 的事件都已到达
```

当算子收到 Watermark(t) 时，意味着不会再收到时间戳小于 t 的事件，可以触发窗口计算。

**乱序处理策略：**

1. **Watermark 生成**
   - 周期性生成：按时间或数量触发
   - 不规则生成：基于特定业务事件

2. **窗口触发时机**
   - 当 watermark 时间 >= 窗口结束时间时，触发窗口计算
   - 设置 `allowedLateness` 延迟触发时间

3. **处理乱序的策略**
   ```java
   // 设置允许延迟
   .allowedLateness(Time.minutes(5))
   // 延迟数据处理方式
   .sideOutputLateData(lateOutputTag)
   ```

**Watermark 传播：**

- 单流输入：多个上游 partition 取最小 watermark
- 双流 Join：两个流 watermark 交汇后取较小值

## 题目三：Flink 的 Checkpoint 和 Savepoint 有什么区别？

### 参考答案

**Checkpoint（检查点）：**

- **目的**：自动故障恢复，保证 exactly-once 语义
- **触发方式**：Flink 自动周期性执行，由 `checkpointing-interval` 参数控制
- **存储位置**：可配置（Memory / FileSystem / RocksDB）
- **清理机制**：完成后自动清理旧检查点（除非配置为 retain）
- **数据格式**：增量存储，仅保存自上次 checkpoint 以来的变更

**Savepoint（保存点）：**

- **目的**：手动备份、程序升级、集群迁移、A/B测试
- **触发方式**：手动触发 `bin/flink savepoint` 命令
- **存储位置**：用户指定目录，不会自动清理
- **数据格式**：完整快照，包含所有算子状态
- **兼容性**：与 JobManager 重启无关，面向作业而非集群

**核心区别：**

| 特性 | Checkpoint | Savepoint |
|------|-----------|-----------|
| 触发者 | 系统自动 | 用户手动 |
| 用途 | 故障恢复 | 计划性备份 |
| 清理策略 | 自动 | 手动 |
| 数据完整性 | 增量 | 完整 |
| 作业重启后 | 继续自动执行 | 可从指定点恢复 |

**代码配置示例：**
```java
env.enableCheckpointing(10000); // 10秒执行一次 checkpoint
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(5000);
env.getCheckpointConfig().setCheckpointTimeout(60000);
```

## 题目四：Flink 窗口函数有哪些类型？各自适用场景是什么？

### 参考答案

**窗口分类：**

**1. 按时间角度划分**

| 窗口类型 | 说明 | 适用场景 |
|---------|------|---------|
| TumblingWindow（滚动窗口） | 无重叠，等大小 | 每分钟/每小时统计 |
| SlidingWindow（滑动窗口） | 有重叠，可设置滑动步长 | 平滑统计、趋势分析 |
| SessionWindow（会话窗口） | 基于不活跃间隔划分 | 用户行为分析 |
| GlobalWindow（全窗口） | 统一所有数据到同一窗口 | 需要手动触发场景 |

**2. 按数据来源划分**

- **Keyed Windows**：先 keyBy 再 window，可并行
- **Non-Keyed Windows**：整个流作为一个窗口，并行度为1

**3. 窗口函数**

- **增量聚合函数**：`reduceFunction`、`aggregateFunction`
  - 窗口内每来一条数据就计算一次，保留轻量状态
- **全量聚合函数**：`processWindowFunction`
  - 等窗口关闭后一次性计算，可访问窗口完整信息

**最佳实践：**
```java
// 推荐组合：增量+全量
stream
  .keyBy(<key selector>)
  .window(<window assigner>)
  .aggregate(new AggregateFunction<...>(), new ProcessWindowFunction<...>())
```

**CountWindow（计数窗口）**

- TumblingCountWindow：固定数量触发
- SlidingCountWindow：滑动计数

## 题目五：Flink 如何保证精确一次（Exactly-Once）语义？

### 参考答案

**Exactly-Once 语义定义：**

每条数据只被处理一次，即使发生故障，最终结果也是准确的。这是最严格的一致性保证。

**Flink 实现 Exactly-Once 的三个关键机制：**

**1. 分布式快照（Checkpoint）**
- 基于 Chandy-Lamport 分布式快照算法
- 通过 Barrier 对数据流进行对齐切分
- 每个 Checkpoint 包含完整的算子状态快照

**2. 两阶段提交（TwoPhaseCommitSink）**
- 在 Checkpoint 完成时触发预提交
- 所有算子完成预提交后，外部系统再正式提交
- 需要 Sink 端支持事务或幂等操作

**3. 状态一致性**
- 算子状态与 checkpoint 保持同步
- 故障恢复时从最近有效 checkpoint 恢复

**适用场景示例：**

```java
// Kafka 作为 source 和 sink 时的 exactly-once 配置
KafkaSource<String> source = KafkaSource.<String>builder()
    .setProperty("isolation.level", "read_committed")
    .build();

DataStream<String> stream = env.fromSource(source, WatermarkStrategy.noWatermarks(), "KafkaSource");

// 使用 Exactly-Once 的 Sink（如文件系统）
stream.sinkTo(ElementsAdapter.sink(new TwoPhaseCommitSink<>() {...}));
```

**常见 Sink 的 Exactly-Once 支持：**

| Sink | 支持情况 | 说明 |
|------|---------|------|
| Kafka | 支持 | 使用事务 producer |
| FileSystem | 支持 | 幂等写入 + checkpoint |
| MySQL | 不支持 | 只能做到 at-least-once，需业务补偿 |
| Elasticsearch | 不支持 | 需手动幂等 |
