## 1. Spark 架构与核心组件

**题目：** 请描述 Spark 的整体架构，Driver、Executor、Master、Worker 各自的作用是什么？Spark 应用执行的核心流程是怎样的？

**参考答案：**

Spark 采用 **Master-Slave** 架构，核心组件如下：

| 组件 | 角色 | 职责 |
|------|------|------|
| **Driver** | 主控节点 | 运行 `main()`，创建 `SparkContext`，负责任务调度、Stage 划分、DAG 构建 |
| **Executor** | 工作节点 | 运行在 Worker 节点上，执行 Task，向 Driver 汇报结果，拥有独立的 JVM |
| **Master** | 资源管理器 | 分配资源给应用，负责 Application 的注册和调度（Standalone 模式下） |
| **Worker** | 节点管理器 | 管理本节点资源，接收 Master 指令启动 Executor |

**执行流程：**
1. 用户提交 Spark Application，创建 `SparkContext`
2. Driver 向 Master 注册应用，Master 调度资源
3. Worker 节点启动 Executor，Executor 向 Driver 反向注册
4. SparkContext 划分 DAG → 优化生成物理执行计划 → 分解为 Stage → Task 调度到 Executor
5. Executor 以 **多线程** 方式执行 Task，Task 结果返回 Driver
6. 所有 Task 完成，Driver 结束，SparkContext 关闭

## 2. RDD 的理解与特性

**题目：** 什么是 RDD？RDD 有哪些核心特性？RDD 与 DataFrame/DataSet 的区别是什么？

**参考答案：**

**RDD（Resilient Distributed Dataset）** 是 Spark 最底层的抽象，是一个不可变的、可分区的、分布式的数据集合。

**RDD 五大核心特性：**

| 特性 | 说明 |
|------|------|
| **分区列表（Partitions）** | 数据集被划分为多个分区，每个分区在不同的节点上存储 |
| **分区函数（Partitioner）** | 可选，决定 KV 类型 RDD 的分区策略（HashPartitioner、RangePartitioner） |
| **父 RDD 依赖（Dependencies）** | 记录 RDD 之间的 lineage（血统）关系，用于故障恢复 |
| **计算函数（Compute）** | 对每个分区执行的计算逻辑 |
| **优先位置（Preferred Locations）** | 调度Task时尽量移动计算到数据所在节点（数据本地性） |

**RDD vs DataFrame vs DataSet：**

| 对比维度 | RDD | DataFrame | DataSet |
|----------|-----|-----------|---------|
| **数据结构** | 无schema，每行类型由用户决定 | 有schema，Row 无类型 | 有schema，每行是强类型对象 |
| **编译时检查** | 无 | 无 | 有（Spark 2.x） |
| **API语言** | Scala/Java/Python/JavaScript | 多语言 | Scala/Java |
| **序列化** | Java/Kryo | Row（ Tungsten 二进制） | Encoder（Kryo/序列化） |
| **性能优化** | 无自动优化 | 有 Catalyst 优化器 | 有 Catalyst 优化器 |
| **数据处理方式** | 函数式算子 | SQL-like / DSL | Lambda + SQL |

## 3. Shuffle 机制与优化

**题目：** 什么是 Shuffle？为什么 Spark 中需要 Shuffle？Shuffle 有哪些常见的优化策略？

**参考答案：**

**Shuffle（混排）** 是指当一个 RDD/Stage 的分区数据需要按照某种规则重新分配到下一个 Stage 的分区时，跨节点的数据传输过程。

**为什么需要 Shuffle：**
- `groupByKey`、`reduceByKey`、`join`、`distinct`、`repartition` 等操作需要将相同 Key 的数据汇聚到同一个分区
- 这意味着数据需要跨网络传输到其他节点

**Spark Shuffle 执行流程（Hash Shuffle）：**
1. Map 端：为每个 Mapper 生成 `M` 个文件 + `M` 个索引文件
2. 数据写入本地磁盘文件，按目标分区编号组织
3. Reduce 端：每个 Reducer 根据分区编号拉取对应的数据文件
4. Reducer 合并所有拉取到的数据并执行聚合

**Shuffle 优化策略：**

| 优化策略 | 说明 |
|----------|------|
| **Hash Shuffle → Sort Shuffle** | Spark 1.2 引入，减少文件数量（`M×R` → `2M`） |
| **Consolidation 机制** | 同一 Executor 内的 Mapper 共享输出文件，减少磁盘文件数 |
| **Map端聚合（Map-side Combine）** | `reduceByKey` 等在 Map 端先做预聚合，减少数据量 |
| **合理设置分区数** | 分区数不宜过多或过少，一般 2-4 倍 CPU 核数 |
| **使用广播变量** | 小表广播，避免 Shuffle |
| **过滤空值/空Key** | 减少 Shuffle 数据量 |
| **自定义 Partitioner** | 根据业务 Key 特征设计分区策略 |

> 📎 来源：[Spark Shuffle 机制深度解析 - 美团技术团队](https://tech.meituan.com/2016/05/03/spark-internals.html)

## 4. Spark 内存管理机制

**题目：** Spark 的内存是如何管理的？堆内内存（On-Heap）和堆外内存（Off-Heap）的区别是什么？Task 执行内存和存储内存如何划分？

**参考答案：**

Spark 内存管理采用 **统一内存管理（Unified Memory Management）** 机制（Spark 1.6+），堆内和堆外各自独立管理。

**内存区域划分：**

```
┌─────────────────────────────────────────────┐
│                 JVM Heap                    │
│  ┌─────────────────┐  ┌──────────────────┐  │
│  │  Execution 内存 │  │   Storage 内存    │  │
│  │  (Task执行/Shuffle)│  │(缓存RDD/Broadcast)│ │
│  └─────────────────┘  └──────────────────┘  │
│  ┌─────────────────┐  ┌──────────────────┐  │
│  │   User Memory   │  │    System Meta    │  │
│  │  (用户代码/内部元数据) │ │   (HDFS元数据等) │ │
│  └─────────────────┘  └──────────────────┘  │
└─────────────────────────────────────────────┘
```

**堆内内存（On-Heap）：**
- 由 JVM 统一管理，受 `spark.executor.memory` 控制
- 分为 Execution、Storage、User Memory、System Meta 四个区域

**堆外内存（Off-Heap）：**
- 通过 `spark.memory.offHeap.enabled=true` 开启
- 由 `spark.memory.offHeap.size` 指定大小
- 主要用于 **Tungsten 序列化**（直接操作二进制，避免 JVM GC 开销）
- 适合生产环境，可规避 JVM 堆内存碎片问题

**Execution 与 Storage 动态共享：**
- 两者共享统一内存池，可相互借用
- Storage 内存被借走后，缓存数据可驱逐（Eviction）
- Execution 内存被借走后，磁盘临时文件溢出

**关键配置参数：**

| 参数 | 说明 |
|------|------|
| `spark.executor.memory` | Executor 总堆内内存 |
| `spark.memory.fraction` | Execution + Storage 占 Heap 的比例（默认 0.6） |
| `spark.memory.storageFraction` | Storage 占统一内存的比例（默认 0.5） |
| `spark.memory.offHeap.enabled` | 是否启用堆外内存 |
| `spark.memory.offHeap.size` | 堆外内存大小 |

> 📎 来源：[Spark 内存管理机制 - 官方文档](https://spark.apache.org/docs/latest/tuning.html#memory-management)

## 5. 广播变量（Broadcast Variable）

**题目：** 什么是广播变量？广播变量适用于什么场景？使用广播变量有哪些注意事项？

**参考答案：**

**广播变量** 是 Spark 提供的只读共享变量，通过 `SparkContext.broadcast()` 创建，会被分发到各个 Executor 节点，供该节点上所有 Task 共享读取，避免数据重复传输。

**适用场景：**
- **大表 join 小表**：将小表数据广播到各个节点，避免 Shuffle（`BroadcastHashJoin`）
- **公共配置/字典表**：每个节点都需要读取的配置数据
- **静态只读数据**：机器学习模型参数、特征映射表等

**基本使用：**

```scala
// 创建广播变量
val broadcastVar = SparkContext.broadcast(Map("a" -> 1, "b" -> 2))

// 在 Task 中使用（自动广播到每个节点）
val result = broadcastVar.value.get("a")  // .value 获取实际数据
```

```python
# PySpark
broadcast_var = sc.broadcast([1, 2, 3, 4, 5])
result = broadcast_var.value
```

**BroadcastHashJoin 示例：**

```scala
import org.apache.spark.sql.functions.broadcast

// 小表广播
val result = bigTable.join(broadcast(smallTable), "key")
```

**注意事项：**

| 注意点 | 说明 |
|--------|------|
| **大小控制** | 广播变量不宜过大（一般 < 20MB），过大导致 Executor OOM |
| **只读不可修改** | 广播变量创建后不可修改，修改不会同步到其他节点 |
| **及时释放** | 不再使用时调用 `.unpersist()` 清理缓存 |
| **序列化方式** | 默认使用 Kryo 序列化，需确保数据可序列化 |
| **触发时机** | 广播发生在 Action 触发时，而非 Transformation |
| **Spark 3.x 优化** | 自动广播小表（`spark.sql.autoBroadcastJoinThreshold`，默认 10MB） |

> 📎 来源：[Spark 广播变量官方文档](https://spark.apache.org/docs/latest/rdd-programming-guide.html#broadcast-variables)
