## 题目1：volatile 与 synchronized 的区别是什么？

**【答案】**

volatile 和 synchronized 都是 Java 多线程编程中的关键字，但它们的原理和使用场景截然不同。

### 1. 本质区别

| 特性 | volatile | synchronized |
|------|----------|--------------|
| **作用对象** | 变量 | 方法/代码块 |
| **原理** | 内存屏障 + 禁止指令重排 | Monitor（监视器锁） |
| **原子性** | 仅保证可见性，不保证原子性 | 保证原子性、可见性、有序性 |
| **性能** | 轻量级，无阻塞 | 重量级，可能阻塞线程 |

### 2. 可见性保证

- **volatile**：写操作后立即刷新到主内存，读操作从主内存读取。配合内存屏障（Memory Barrier）防止指令重排。
- **synchronized**：通过 Monitor 机制，在进入锁时清空本地内存缓存，退出锁时刷新到主内存，保证可见性。

### 3. 有序性保证

- **volatile**：通过插入内存屏障，禁止编译器/CPU 对 volatile 变量操作进行重排。
- **synchronized**：在同一锁的保护下，代码块内的指令有序（同一锁的解锁 happens-before 加锁）。

### 4. 能否保证原子性

- **volatile**：不保证原子性。例如 `i++` 这种读-改-写操作，volatile 无法保证线程安全。
- **synchronized**：保证原子性，同一时刻只有一个线程能进入同步块。

### 5. 使用场景

- **volatile**：适用于一个线程写、多个线程读的场景，如状态标记位。
- **synchronized**：适用于需要原子操作、复合操作的场景，如计数器、账户余额。

### 6. JVM层面的实现

- volatile 写操作后，会插入 `StoreStore` + `StoreLoad` 内存屏障。
- synchronized 通过 `monitorenter` / `monitorexit` 指令实现，依赖对象头的 Mark Word。

### 7. JDK 1.5 之后的改进

`java.util.concurrent` 包中的原子类（如 `AtomicInteger`）使用 CAS + volatile 实现，比 synchronized 性能更好。

## 题目2：HashMap 与 ConcurrentHashMap 的区别是什么？JDK 1.8 前后有何变化？

**【答案】**

### 1. 线程安全性

- **HashMap**：非线程安全。多线程并发操作可能导致数据丢失或死循环（在 JDK 1.7 的扩容场景下）。
- **ConcurrentHashMap**：线程安全。采用分段锁或 CAS 实现并发安全。

### 2. 底层数据结构（JDK 1.7 vs JDK 1.8）

| 版本 | HashMap | ConcurrentHashMap |
|------|---------|-------------------|
| JDK 1.7 | 数组 + 链表 | Segment（分段锁） + 数组 + 链表 |
| JDK 1.8 | 数组 + 链表 + 红黑树 | Node（CAS） + 数组 + 链表 + 红黑树 |

### 3. JDK 1.7 的分段锁机制

ConcurrentHashMap 由 Segment 数组组成，每个 Segment 继承 ReentrantLock，每个 Segment 下包含一个 HashEntry 数组。默认并发度为 16，即最多 16 个线程同时操作。

### 4. JDK 1.8 的改进

- 取消分段锁，采用 `Node + CAS + synchronized` 实现。
- 当链表长度超过 8 且数组长度 >= 64 时，转换为红黑树，提升查询效率（O(n) → O(log n)）。
- 使用 CAS（Compare-And-Swap）保证并发插入的原子性。
- 在链表/红黑树节点上加 synchronized 锁，保证数据一致性。

### 5. 关键区别

| 对比项 | HashMap | ConcurrentHashMap |
|--------|---------|-------------------|
| null 键/值 | 允许一个 null 键 | 不允许 null 键和 null 值（避免多线程歧义） |
| 迭代器 | 弱一致性问题 | 不会抛出 ConcurrentModificationException |
| 性能 | 单线程性能高 | 多线程并发性能高 |

### 6. put 操作的并发控制

- JDK 1.7：对 Segment 加锁，冲突时锁粒度较大。
- JDK 1.8：
  1. 通过 CAS 尝试插入。
  2. 若 CAS 失败（其他线程已插入），进入 synchronized (tab[i]) 块再次尝试。
  3. 插入后判断是否需要树化。

### 7. 应用场景

- **HashMap**：单线程环境或需要 null 键值。
- **ConcurrentHashMap**：高并发场景，如缓存、计数器、Session 存储。

## 题目3：Spring Boot 自动配置（Auto Configuration）是如何实现的？

**【答案】**

### 1. 核心注解

`@SpringBootApplication` 包含三个注解：
- `@SpringBootConfiguration`（等同于 `@Configuration`）
- `@EnableAutoConfiguration`：启用自动配置
- `@ComponentScan`：组件扫描

### 2. @EnableAutoConfiguration 原理

该注解通过 `@Import(AutoConfigurationImportSelector.class)` 导入自动配置选择器。

### 3. AutoConfigurationImportSelector 工作流程

1. 调用 `SpringFactoriesLoader.loadFactoryNames()` 读取 `META-INF/spring.factories`。
2. 也读取 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`（Spring Boot 2.7+）。
3. 筛选出符合条件的自动配置类，返回给 Spring 容器。

### 4. spring.factories 文件示例

```properties
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
  org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration,\
  org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration
```

### 5. 自动配置类结构

每个自动配置类（如 `DataSourceAutoConfiguration`）通常包含：

```java
@Configuration
@ConditionalOnClass({ DataSource.class })      // 条件：classpath 存在 DataSource
@ConditionalOnMissingBean(DataSource.class)     // 条件：不存在 DataSource bean
@EnableConfigurationProperties(DataSourceProperties.class)
public class DataSourceAutoConfiguration {
    // ...
}
```

### 6. @Conditional 系列条件注解

| 注解 | 条件 |
|------|------|
| `@ConditionalOnClass` | classpath 中存在指定类 |
| `@ConditionalOnMissingBean` | 不存在指定的 bean |
| `@ConditionalOnProperty` | 配置文件中存在指定属性 |
| `@ConditionalOnWebApplication` | 是 Web 应用环境 |
| `@ConditionalOnBean` | 存在指定的 bean |

### 7. 自定义 Starter 的标准流程

1. 创建 `xxx-spring-boot-starter` 模块。
2. 创建 `xxx-autoconfigure` 模块。
3. 在 autoconfigure 模块中编写自动配置类，标注 `@Configuration` 和条件注解。
4. 在 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 中注册。
5. 在 starter 的 `pom.xml` 中依赖 autoconfigure 模块。

### 8. 排除自动配置

```java
@SpringBootApplication(exclude = { DataSourceAutoConfiguration.class })
// 或在 application.yml 中配置
spring:
  autoconfigure:
    exclude:
      - org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
```

📖 来源：[Spring Boot 自动配置原理深度解析](https://blog.csdn.net/fim77/article/details/147687588)

## 题目4：Spring Cloud 中 Eureka 与 Nacos 的区别是什么？如何选型？

**【答案】**

### 1. 基本概念

| 特性 | Eureka | Nacos |
|------|--------|-------|
| **定位** | Netflix 提供的服务发现组件 | 阿里巴巴提供的更全面的服务平台 |
| **功能** | 仅服务注册/发现 | 服务注册/发现 + 配置管理 + 元数据管理 |
| **一致性协议** | AP（最终一致性） | CP + AP 可切换 |
| **上线时间** | 2012年 | 2018年 |

### 2. 架构对比

**Eureka 架构**：
- 包含 Eureka Server（注册中心）和 Eureka Client（客户端）。
- Server 支持集群部署，各节点相互同步注册表。
- 采用纯 AP 模型，节点之间通过异步复制保持最终一致。

**Nacos 架构**：
- 同时承担服务发现和配置管理角色。
- 支持两种模式：持久化（CP）和非持久化（AP）。
- 默认是持久化模式，支持配置变更的实时推送。

### 3. Nacos 的优势

1. **配置中心功能**：Nacos 原生支持配置管理，无需集成 Config + Bus。
2. **双协议支持**：CP（Raft 协议）+ AP（Distro 协议）可切换。
3. **更友好的 UI**：提供可视化的管理界面。
4. **更活跃的社区**：阿里背书，更新迭代快。
5. **性能更好**：Nacos 2.0 基于 gRPC 进行服务通信，性能优于 Eureka。

### 4. Eureka 的局限

1. 服务列表同步存在延迟。
2. 不支持配置中心功能。
3. 已进入维护模式（Netflix 已不再积极开发）。
4. 集群节点之间同步效率较低。

### 5. 选型建议

| 场景 | 推荐 |
|------|------|
| 仅需要服务注册/发现，老项目迁移 | Eureka |
| 微服务 + 配置中心一体化 | Nacos |
| 需要配置实时变更推送 | Nacos |
| 对一致性要求极高的金融场景 | Nacos CP 模式 / Zookeeper |
| 中小型项目，快速上手 | Nacos（文档更完善） |

### 6. Nacos 配置示例

```yaml
# application.yml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848
        namespace: dev
      config:
        server-addr: 127.0.0.1:8848
        file-extension: yaml
```

📖 来源：[Spring Cloud Alibaba 微服务原理与实战](https://blog.csdn.net/fim77/article/details/147687588)

## 题目5：Spring Cloud Gateway 网关的路由转发原理是什么？

**【答案】**

### 1. 工作流程

```
请求 → Gateway Handler Mapping（路由映射） → Gateway Web Handler（处理器） → 过滤器链 → 目标服务
```

### 2. 核心组件

| 组件 | 作用 |
|------|------|
| **RouteLocator** | 路由定位器，配置路由信息（ID、URI、 predicates、filters） |
| **RoutePredicateFactory** | 谓词工厂，用于匹配请求条件（Path、Query、Header 等） |
| **GatewayFilter** | 网关过滤器，单个路由生效 |
| **GlobalFilter** | 全局过滤器，所有路由生效 |
| **NettyRoutingFilter** | 实际执行 HTTP 请求的过滤器 |

### 3. 路由配置示例

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: http://localhost:8080
          predicates:
            - Path=/api/user/**
          filters:
            - StripPrefix=1   # 去掉第一层路径
            - RequestRateLimiter=100,10  # 限流
```

### 4. 请求匹配流程

1. **RoutePredicateHandlerMapping** 遍历所有 Route。
2. 对每个 Route 的 predicates 进行匹配（Path、Method、Header 等）。
3. 匹配成功后，进入 **GatewayWebHandler**，执行过滤器链。
4. 过滤器链顺序：Ordered 高的先执行（Pre 阶段），然后转发请求，再执行低优先级的（Post 阶段）。

### 5. 内置 predicates（谓词）

| 谓词 | 匹配条件 |
|------|----------|
| `Path` | 请求路径 |
| `Query` | 请求参数 |
| `Header` | 请求头 |
| `Method` | HTTP 方法 |
| `Cookie` | Cookie 值 |
| `Before / After / Between` | 时间条件 |

### 6. 过滤器执行顺序

- 所有过滤器按 `Ordered.getOrder()` 值升序排列。
- 数值越小，越先执行。
- 自定义过滤器实现 `Ordered` 接口或返回 `int` 值。

### 7. 常用过滤器

- **AddRequestHeader**：添加请求头
- **AddResponseHeader**：添加响应头
- **StripPrefix**：去掉路径前缀
- **PrefixPath**：添加路径前缀
- **RequestRateLimiter**：限流（配合 Redis）
- **Retry**：重试机制

### 8. 自定义全局过滤器示例

```java
@Component
public class AuthGlobalFilter implements GlobalFilter, Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (token == null) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() {
        return -100; // 优先级
    }
}
```

### 9. 跨域（CORS）配置

```yaml
spring:
  cloud:
    gateway:
      globalcors:
        cors-configurations:
          '[/**]':
            allowedOrigins: "*"
            allowedMethods: "*"
```

📖 来源：[Spring Cloud Gateway 官方文档](https://blog.csdn.net/fim77/article/details/147687588)

## 题目6：Kafka 如何保证消息不丢失？

**【答案】**

### 1. 消息丢失的三个环节

| 环节 | 场景 | 解决方案 |
|------|------|----------|
| **生产者** | 发送失败、网络抖动 | 配置 `acks=all` + 重试机制 |
| **Broker** | 分区副本丢失 | 配置 `replication.factor >= 3` |
| **消费者** | 自动提交 offset | 手动提交 offset， processing 完成后再提交 |

### 2. 生产者端配置

```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer");
// 关键配置
props.put("acks", "all");           // 所有副本确认
props.put("retries", 3);           // 重试次数
props.put("enable.idempotence", true);  // 幂等性
```

**acks 参数说明**：

| 值 | 含义 | 可靠性 |
|----|------|--------|
| `0` | 不等待broker确认 | 最低 |
| `1` | Leader 确认即可 | 中等 |
| `all`（或 `-1`） | 所有 ISR 副本确认 | 最高 |

### 3. Broker 端配置

```properties
# server.properties
# 副本因子 >= 3
offsets.topic.replication.factor=3

# 最小 ISR 数量
min.insync.replicas=2

# 禁用手动删除 topic（防止数据丢失）
delete.topic.enable=false

# 消息刷盘策略（性能 vs 可靠性权衡）
flush.messages=1
flush.ms=1000
```

### 4. ISR（In-Sync Replicas）机制

- ISR 是与 Leader 保持同步的副本集合。
- 当 `min.insync.replicas=2` 时，必须有至少 2 个副本（包括 Leader）确认，消息才算写入成功。
- 若 ISR 数量低于阈值，Kafka 会停止写入（`min.insync.replicas` 配置生效）。

### 5. 消费者端配置

```java
props.put("enable.auto.commit", false); // 关闭自动提交
// 手动提交 offset
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        process(record);  // 处理业务逻辑
    }
    consumer.commitSync();  // 处理完成后手动提交
}
```

### 6. 幂等性配置（防止重复消息）

```java
props.put("enable.idempotence", true);  // 开启幂等
props.put("max.in.flight.requests.per.connection", 5);  // 限制未确认请求数
```

### 7. 事务机制（Exactly Once）

```java
props.put("transactional.id", "transaction-id-1");  // 开启事务
props.put("enable.idempotence", true);

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
producer.initTransactions();
producer.beginTransaction();
producer.send(new ProducerRecord<>("topic", "key", "value"));
producer.commitTransaction();
```

### 8. 数据持久化策略

| 配置 | 说明 |
|------|------|
| `log.flush.interval.messages` | 消息达到一定数量后刷盘 |
| `log.flush.interval.ms` | 定时刷盘 |
| `log.retention.hours` | 日志保留时间 |

### 9. 监控与告警

- 监控 consumer lag（消费滞后）。
- 监控 `UnderReplicatedPartitions`（未充分副本分区）。
- 配置告警阈值，及时处理。

## 题目7：RabbitMQ 有哪些交换机类型？它们有什么区别？

**【答案】**

### 1. 四种交换机类型

| 类型 | 说明 | 路由规则 |
|------|------|----------|
| **Direct** | 直接交换机 | 精确匹配 routing key |
| **Fanout** | 广播交换机 | 忽略 routing key，发送给所有绑定队列 |
| **Topic** | 主题交换机 | 支持通配符匹配 |
| **Headers** | 头交换机 | 匹配消息头的键值对（较少用） |

### 2. Direct 交换机

**工作原理**：根据精确的 routing key 进行匹配。

```java
// 声明交换机
channel.exchangeDeclare("direct.exchange", BuiltinExchangeType.DIRECT, true);

// 绑定队列
channel.queueBind("queue1", "direct.exchange", "order.create");
channel.queueBind("queue2", "direct.exchange", "order.pay");

// 发送消息（精确匹配）
channel.basicPublish("direct.exchange", "order.create", null, message.getBytes());
```

**应用场景**：需要根据不同业务类型精确路由的场景，如订单创建、订单支付。

### 3. Fanout 交换机

**工作原理**：忽略 routing key，将消息广播给所有绑定的队列。

```java
// 声明交换机
channel.exchangeDeclare("fanout.exchange", BuiltinExchangeType.FANOUT, true);

// 绑定多个队列
channel.queueBind("queue1", "fanout.exchange", "");
channel.queueBind("queue2", "fanout.exchange", "");

// 发送消息（所有队列都能收到）
channel.basicPublish("fanout.exchange", "", null, message.getBytes());
```

**应用场景**：
- 系统通知（广播给所有相关模块）
- 日志收集
- 多语言系统间的消息同步

### 4. Topic 交换机

**工作原理**：支持通配符匹配 routing key。

| 通配符 | 含义 | 示例 |
|--------|------|------|
| `*` | 匹配一个单词 | `order.*` 匹配 `order.create` |
| `#` | 匹配零个或多个单词 | `order.#` 匹配 `order.create.pay` |

```java
// 绑定示例
channel.queueBind("queue1", "topic.exchange", "order.create");
channel.queueBind("queue2", "topic.exchange", "order.*");
channel.queueBind("queue3", "topic.exchange", "order.#");

// 发送
channel.basicPublish("topic.exchange", "order.create.pay", null, message.getBytes());
// queue1、queue2、queue3 都能收到
```

**应用场景**：灵活的路由规则，如日志级别路由（error.*、info.*）。

### 5. Headers 交换机

**工作原理**：根据消息头的键值对进行匹配，匹配模式为 `all`（全部匹配）或 `any`（任一匹配）。

```java
Map<String, Object> headers = new HashMap<>();
headers.put("format", "pdf");
headers.put("type", "report");

channel.basicPublish("headers.exchange", "", new AMQP.BasicProperties.Builder()
    .headers(headers)
    .build(), message.getBytes());
```

**应用场景**：复杂的消息过滤条件（不常用，性能较差）。

### 6. 交换机与队列的绑定关系

```
[Producer] → [Exchange] → [Queue] → [Consumer]
              ↗    ↗
            Queue1  Queue2
```

### 7. 死信交换机（Dead Letter Exchange）

当消息被拒绝（`basicReject`/`basicNack`）或超时，会进入死信队列：

```java
Map<String, Object> args = new HashMap<>();
args.put("x-dead-letter-exchange", "dlx.exchange");
args.put("x-dead-letter-routing-key", "dead.order");
channel.queueDeclare("order.queue", true, false, false, args);
```

### 8. 镜像队列（RabbitMQ Cluster）

- 镜像队列将主节点的数据同步到从节点，提高可用性。
- 但要注意，镜像队列不是负载均衡，而是高可用方案。

## 题目8：什么是 RAG（检索增强生成）？它的核心组件有哪些？

**【答案】**

### 1. 什么是 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）是一种结合了信息检索技术与语言生成模型的技术，旨在解决大模型「幻觉」问题和知识时效性问题。

### 2. RAG 的工作流程

```
用户提问 → 检索（向量数据库） → 知识召回 → Prompt 组装 → LLM 生成答案
```

### 3. 核心组件

| 组件 | 说明 |
|------|------|
| **Embedding 模型** | 将文本向量化，常见如 m3e、bge、OpenAI embedding |
| **向量数据库** | 存储向量并做相似度检索，如 Milvus、Chroma、FAISS |
| **文档分割器** | 将长文档切分为小块（Chunk），需考虑语义完整性 |
| **检索器** | 负责从向量数据库中召回相关文档 |
| **重排序器** | 对召回结果进行二次排序，提升相关性 |
| **LLM** | 根据上下文生成最终答案 |

### 4. 数据准备阶段

```
文档提取 → 文本分割 → 向量化（Embedding） → 数据入库
```

- **文档提取**：支持 PDF、Word、Markdown、HTML 等多格式。
- **文本分割**：固定长度分割 / 语义分割，需考虑 token 限制。
- **向量化**：将文本转化为高维向量。
- **数据入库**：写入向量数据库，构建索引。

### 5. 应用阶段

```
用户提问 → 向量化问题 → 检索 → 召回 Top-K → 注入 Prompt → LLM 生成
```

### 6. 高级 RAG 优化技术

1. **混合检索**：结合向量检索 + 关键词检索（BM25），结果通过 RRF 算法融合。
2. **重排序（Rerank）**：使用 Cross-Encoder 或 LLM 对初筛结果重排序。
3. **查询改写**：利用 LLM 将复杂问题拆分为子问题。
4. **HyDE（假设性文档嵌入）**：让 LLM 生成假设性回答，再用假设回答去检索。
5. **句子窗口检索**：检索最相关句子，扩展上下文窗口后送给 LLM。
6. **父文档检索器**：先找小块，再关联到更大的父块，保证信息完整性。

### 7. RAG vs Fine-tuning

| 对比项 | RAG | Fine-tuning |
|--------|-----|-------------|
| 成本 | 低，无需训练 | 高，需GPU训练 |

| 可解释性 | 高，可溯源 | 低 |
| 适用场景 | 知识库问答、实时信息 | 特定领域优化 |

## 题目9：二分查找变体 —— 「搜索旋转排序数组」

**【答案】**

### 题目描述

假设一个按升序排序的数组在某个 pivot 处进行了旋转（如 `[0,1,2,4,5,6,7]` 变成 `[4,5,6,7,0,1,2]`）。给定一个目标值，找出它在哪里，若不存在返回 -1。

### 解题思路

数组在旋转后分为两段有序区间，利用二分查找的变体，每次判断 mid 落在哪个有序区间：

1. 若 `nums[mid] >= nums[left]`，说明左半段有序。
2. 若 `nums[mid] < nums[left]`，说明右半段有序。
3. 根据目标值和边界的关系，缩小搜索范围。

### 代码实现

```java
public int search(int[] nums, int target) {
    int left = 0, right = nums.length - 1;

    while (left <= right) {
        int mid = left + (right - left) / 2;

        if (nums[mid] == target) {
            return mid;
        }

        // 判断哪一段是有序的
        if (nums[mid] >= nums[left]) {
            // 左半段有序 [left, mid]
            if (target >= nums[left] && target < nums[mid]) {
                right = mid - 1;  // 目标在左半段
            } else {
                left = mid + 1;   // 目标在右半段
            }
        } else {
            // 右半段有序 [mid, right]
            if (target > nums[mid] && target <= nums[right]) {
                left = mid + 1;   // 目标在右半段
            } else {
                right = mid - 1;  // 目标在左半段
            }
        }
    }

    return -1;
}
```

### 复杂度分析

- **时间复杂度**：`O(log n)`，每次折半
- **空间复杂度**：`O(1)`，原地操作

### 边界情况处理

| 场景 | 示例 | 处理 |
|------|------|------|
| 单元素数组 | `[1]` | 直接比较 |
| 未旋转数组 | `[1,2,3]` | 按标准二分处理 |
| 重复元素 | `[2,2,2,1,2]` | 可能退化为 O(n)，需特殊处理 |

### 变体：找旋转点（最小值）

```java
public int findMin(int[] nums) {
    int left = 0, right = nums.length - 1;

    while (left < right) {
        int mid = left + (right - left) / 2;

        if (nums[mid] > nums[right]) {
            left = mid + 1;   // 最小值在右半段
        } else {
            right = mid;      // mid 可能是最小值
        }
    }

    return nums[left];
}
```

## 题目10：合并两个有序链表

**【答案】**

### 题目描述

给定两个升序链表，将其合并为一个升序链表，返回合并后的链表。

**示例**：
- 输入：`1 -> 2 -> 4`，`1 -> 3 -> 4`
- 输出：`1 -> 1 -> 2 -> 3 -> 4 -> 4`

### 解法一：迭代法

```java
public ListNode mergeTwoLists(ListNode l1, ListNode l2) {
    ListNode dummy = new ListNode(-1);  // 虚拟头节点
    ListNode cur = dummy;

    while (l1 != null && l2 != null) {
        if (l1.val <= l2.val) {
            cur.next = l1;
            l1 = l1.next;
        } else {
            cur.next = l2;
            l2 = l2.next;
        }
        cur = cur.next;
    }

    // 连接剩余部分
    cur.next = (l1 != null) ? l1 : l2;

    return dummy.next;
}
```

**复杂度**：
- 时间：`O(n + m)`，n 和 m 分别为两个链表长度
- 空间：`O(1)`

### 解法二：递归法

```java
public ListNode mergeTwoLists(ListNode l1, ListNode l2) {
    if (l1 == null) return l2;
    if (l2 == null) return l1;

    if (l1.val <= l2.val) {
        l1.next = mergeTwoLists(l1.next, l2);
        return l1;
    } else {
        l2.next = mergeTwoLists(l1, l2.next);
        return l2;
    }
}
```

**复杂度**：
- 时间：`O(n + m)`
- 空间：`O(n + m)`，递归调用栈深度

### 关键点分析

| 点 | 说明 |
|----|------|
| **虚拟头节点（dummy）** | 简化边界处理，避免处理空头 |
| **指针移动** | l1、l2、cur 三个指针协同 |
| **剩余部分** | 直接拼接，不需要再遍历 |

### 链表节点定义

```java
public class ListNode {
    int val;
    ListNode next;
    ListNode() {}
    ListNode(int val) { this.val = val; }
    ListNode(int val, ListNode next) { this.val = val; this.next = next; }
}
```

### 扩展：合并 K 个有序链表

```java
public ListNode mergeKLists(ListNode[] lists) {
    // 使用最小堆（小顶堆）优化
    PriorityQueue<ListNode> heap = new PriorityQueue<>(
        (a, b) -> a.val - b.val
    );

    // 初始化堆
    for (ListNode node : lists) {
        if (node != null) heap.offer(node);
    }

    ListNode dummy = new ListNode(-1);
    ListNode cur = dummy;

    while (!heap.isEmpty()) {
        ListNode node = heap.poll();
        cur.next = node;
        if (node.next != null) heap.offer(node.next);
        cur = cur.next;
    }

    return dummy.next;
}
```

**复杂度**：
- 时间：`O(N log k)`，N 为总节点数，k 为链表数
- 空间：`O(k)`，堆大小
