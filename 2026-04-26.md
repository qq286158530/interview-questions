## 题目1：Java基础 - HashMap的put方法执行流程是什么？JDK 1.8有哪些优化？

【答案】

**HashMap的put方法执行流程：**

1. **计算hash值**：通过key的hashCode()方法计算出hash值，然后进行高低位异或扰动处理（hash ^= (h >>> 16)），减少碰撞概率

2. **定位桶位置**：通过 `(n-1) & hash` 计算数组下标，确定该key应该落在哪个桶（数组槽位）中

3. **遍历桶内链表/红黑树**：
   - 如果桶位置为空，直接插入新节点
   - 如果桶位置有节点，遍历链表/红黑树，查找是否有相同的key
   - 如果找到相同key，则覆盖value

4. **判断是否需要扩容**：
   - 如果插入后 size > threshold（容量×负载因子），触发resize()扩容
   - 扩容为原来容量的2倍，重新计算每个元素的位置

**JDK 1.8的主要优化：**

1. **引入红黑树**：当链表长度超过8（TREEIFY_THRESHOLD）时，链表转换为红黑树，提升查询效率从O(n)到O(logn)

2. **优化hash扰动**：JDK 1.7使用4次位运算+5次异或，JDK 1.8简化为1次异或（hash ^= (h >>> 16)），在保证散列性的同时减少计算量

3. **插入方式改变**：JDK 1.7头插法（可能导致环形链表），JDK 1.8尾插法（避免环形链表问题，同时保持插入顺序）

4. **扩容优化**：JDK 1.8在扩容时，不需要重新计算每个元素的hash位置，而是通过 `oldCap（旧容量）` 判断元素的hash新增位是0还是1，为0则位置不变，为1则新位置 = 旧位置 + oldCap，大幅减少重新计算

## 题目2：Java进阶 - 你了解JVM的类加载机制吗？什么是双亲委派模型？为什么要这样做？

【答案】

**JVM类加载机制（类的生命周期）：**

加载（Loading）→ 验证（Verification）→ 准备（Preparation）→ 解析（Resolution）→ 初始化（Initialization）→ 使用（Using）→ 卸载（Unloading）

其中验证、准备、解析统称为链接（Linking）。

**加载阶段的三件事：**
1. 通过类的全限定名获取类的二进制字节流
2. 将字节流转化为方法区的运行时数据结构
3. 在堆中生成java.lang.Class对象，作为方法区数据的访问入口

**双亲委派模型（Parent Delegation Model）：**

类加载器分为三层级结构：
- **Bootstrap ClassLoader**（启动类加载器）：加载JAVA_HOME/lib目录下的核心类库
- **Extension ClassLoader**（扩展类加载器）：加载JAVA_HOME/lib/ext目录下的扩展类
- **Application ClassLoader**（应用程序类加载器）：加载classpath下编写的类

**委派流程**：当需要加载一个类时，当前加载器首先委派给父加载器处理，父加载器再委派给自己的父加载器，直到Bootstrap。如果父加载器找不到，才由子加载器自己加载。

**为什么要用双亲委派？**

1. **安全性**：避免用户自定义类替换核心API类。例如java.lang.String由Bootstrap加载，用户无法自定义java.lang.String来替代它，防止恶意代码注入

2. **避免重复加载**：父加载器已加载的类，子加载器不需要重新加载，保证类的唯一性

3. **保证类的正常使用**：类的静态属性等能够被正确共享

**破坏双亲委派的情况：**
- JDK 1.2之前没有双亲委派，为兼容新增了findClass()方法
- SPI（Service Provider Interface）机制，如JDBC驱动，由ThreadContextClassLoader加载
- OSGI热部署、Tomcat自定义类加载器等

## 题目3：Spring/Spring Boot - @Autowired和@Resource注解有什么区别？

【答案】

**@Autowired和@Resource的区别：**

| 区别 | @Autowired | @Resource |
|------|-----------|-----------|
| 来源 | Spring原生注解 | Java标准注解（JSR-250） |
| 包位置 | org.springframework.beans.factory.annotation | javax.annotation |
| 注入方式 | byType（按类型）优先，可配合@QualifierbyName | byName优先，可指定name属性 |
|  Required属性 | 可设置required=false | 不支持 |
|  注解声明位置 | 字段、构造函数、setter方法 | 字段、setter方法 |

**@Autowired详解：**

```java
// 默认按类型匹配
@Autowired
private UserService userService;

// 指定名称（配合@Qualifier）
@Autowired
@Qualifier("userServiceImpl")
private UserService userService;

// 构造函数注入
public class UserController {
    private final UserService userService;
    @Autowired
    public UserController(UserService userService) {
        this.userService = userService;
    }
}
```

**@Resource详解：**

```java
// 默认按名称匹配（字段名）
@Resource
private UserService userService;

// 指定名称
@Resource(name = "userServiceImpl")
private UserService userService;

// 指定类型
@Resource(type = UserServiceImpl.class)
private UserService userService;
```

**执行顺序总结：**
- @Autowired：先按类型找，找不到再按名称（byName）
- @Resource：先按名称找，找不到再按类型（byType）

**实际建议**：
- 如果代码需要跨Spring框架使用，优先用@Resource（Java标准）
- 如果只需要在Spring内使用，@Autowired更灵活，可配合@Qualifier精确指定

## 题目4：Spring/Spring Boot - Spring Boot的配置文件加载顺序是怎样的？application.yml和bootstrap.yml有什么区别？

【答案】

**Spring Boot配置文件优先级（从高到低）：**

1. 命令行参数（--spring.config.additional-location）
2. 打包外的application-{profile}.yml
3. 打包外的application.yml
4. 打包内的application-{profile}.yml
5. 打包内的application.yml
6. @PropertySource注解加载的配置文件

**注意**：相同配置项，高优先级覆盖低优先级。

**application.yml vs bootstrap.yml 的区别：**

| 区别 | bootstrap.yml | application.yml |
|------|--------------|----------------|
| 加载时机 | Spring Boot启动早期（上下文引导阶段） | 应用上下文初始化阶段 |
| 使用场景 | 配置中心（Spring Cloud Config）、加密密钥等引导性配置 | 业务配置、应用级配置 |
| 属性来源 | 由父ApplicationContext加载 | 由主ApplicationContext加载 |
| 支持profile | 支持 | 支持 |
| 典型用途 | spring.application.name、spring.cloud.config.uri、bootstrapsecret | server.port、logging.level等业务配置 |

**bootstrap.yml适用场景：**

```yaml
# bootstrap.yml - 用于配置中心引导
spring:
  application:
    name: user-service
  cloud:
    config:
      uri: http://config-server:8888
      label: master
      profile: dev

# application.yml - 业务配置
server:
  port: 8080
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/mydb
```

**为什么需要bootstrap.yml？**

Spring Cloud需要从配置中心获取配置，但主应用上下文还没启动，所以需要独立的引导上下文（Bootstrap Context）来加载配置中心的配置。bootstrap.yml由BootstrapApplicationListener触发加载，早于application.yml。

## 题目5：Spring/Spring Boot/Spring Cloud - Nacos的服务注册与发现原理是什么？心跳机制如何实现？

【答案】

**Nacos服务注册与发现原理：**

**服务注册流程：**
1. 服务启动时，向Nacos Server发送注册请求（POST /nacos/v1/ns/instance）
2. 请求包含：服务名、IP、端口、权重、clusterName等元数据
3. Nacos Server将服务实例信息存储到注册表（Map<serviceName, Cluster>）
4. 注册表采用CAP中的AP模式（优先可用性），保证高可用

**服务发现流程：**
1. 消费者启动时，从Nacos Server获取服务提供者列表（GET /nacos/v1/ns/instance/list）
2. 缓存在本地，同时订阅服务变更（UDP推送）
3. 当服务列表变更时，Nacos Server主动推送更新给消费者
4. 消费者更新本地缓存

**心跳机制实现：**

1. **心跳周期**：默认5秒（Nacos Server向所有注册服务发送心跳）
2. **实例续约**：客户端每隔5秒发送心跳（如果15秒没收到，则标记为不健康；30秒没收到，则剔除）
3. **健康检查**：Nacos支持三种模式：
   - **客户端主动上报**（TCP/HTTP/MYSQL）：客户端定时上报健康状态
   - **服务端探测**：服务端主动探测客户端健康
   - **混合模式**：结合主动上报和探活

**Nacos注册表结构：**

```java
// 服务Map
Map<String, Service> -> serviceName -> Service
Service {
    Map<String, Cluster> clusters;  // 集群
    Map<String, Instance> ips;     // 实例列表（按权重）
}
```

**关键参数配置：**
```yaml
# 心跳相关参数
spring:
  cloud:
    nacos:
      discovery:
        heart-beat-interval: 5000      # 心跳间隔（毫秒）
        heart-beat-timeout: 15000      # 心跳超时（毫秒）
        ip-delete-timeout: 30000       # 实例删除超时（毫秒）
```

## 题目6：消息队列 - Kafka如何保证消息的顺序性？分区分配策略是怎样的？

【答案】

**Kafka保证消息顺序性的方法：**

**1. 单分区内有序**

Kafka Topic的一个分区（Partition）内，消息是严格有序的。原因是：每个分区由Leader副本处理，消息追加到末尾时保持顺序。

**2. 多分区全局有序**

如果需要全局有序，只能将所有消息发送到同一个分区（如设置key相同，或只建1个分区），但这会严重影响吞吐量。

**3. 生产端保证有序**

- 使用单分区生产者（partition.num履历=1）
- 或通过partitioner确保相同key的消息发到同一分区

**分区分配策略（Partition Assignment Strategy）：**

Kafka有两种消费者分配策略：

**策略1：Range（范围分配）**
```
假设有2个Topic（t0, t1），各有3个分区，消费者C0、C1

t0 p0, t0 p1, t0 p2
t1 p0, t1 p1, t1 p2

分配结果：
C0: t0 p0, t0 p1, t1 p0, t1 p1
C1: t0 p2, t1 p2

问题：可能导致分配不均匀
```

**策略2：RoundRobin（轮询分配）**
```
将所有分区分组后轮询分配

C0: t0 p0, t0 p2, t1 p1
C1: t0 p1, t1 p0, t1 p2

更均匀，但打乱分区顺序
```

**策略3：StickyAssignor（黏性分配，Kafka 2.4+）**
- 最大化均匀分配
- 当消费者变更时，尽量少迁移分区（减少Rebalance影响）
- 优先保持原有分配不变

**消费者Rebalance触发条件：**
- 消费者加入或离开group
- 分区数变更（增加）
- 订阅主题变更

**如何保证顺序消费：**
```java
// 场景：订单消息，需要按订单号有序处理
// 方案：将订单号作为key发送到同一分区
kafkaTemplate.send("order-topic", orderId, orderMessage);
// 消费者端：使用单线程顺序消费，或按partition顺序处理
```

## 题目7：消息队列 - RabbitMQ有哪几种交换机类型？它们有什么区别和应用场景？

【答案】

**RabbitMQ四种交换机类型：**

**1. Direct Exchange（直接交换机）**

**路由规则**：根据 routingKey（路由键）精确匹配，消息被投递到指定队列。

```
QueueA <-- binding key: "order"
QueueB <-- binding key: "payment"

消息routingKey="order" --> QueueA
消息routingKey="payment" --> QueueB
```

**应用场景**：点对点消息传递，如订单处理、支付回调等。

**2. Fanout Exchange（扇出交换机）**

**路由规则**：无视routingKey，将消息广播到所有绑定的队列。

```
QueueA <-- 绑定到FanoutExchange
QueueB <-- 绑定到FanoutExchange

消息 --> FanoutExchange --> 同时投递到QueueA和QueueB
```

**应用场景**：发布-订阅模式，如系统通知、消息广播、日志收集等。

**3. Topic Exchange（主题交换机）**

**路由规则**：根据通配符模式匹配routingKey，实现灵活的消息路由。

```
# 通配符规则：
# * 匹配一个单词
# # 匹配零个或多个单词

绑定关系：
QueueA binding: "order.*"    (匹配 order.created, order.paid 等)
QueueB binding: "order.#"   (匹配 order, order.created, order.created.confirmed 等)

"order.created" --> QueueA 和 QueueB（都会被匹配）
```

**应用场景**：复杂的消息路由，如不同业务类型、日志级别等。

**4. Headers Exchange（头交换机）**

**路由规则**：根据消息头的键值对匹配，而不是routingKey。

```java
// 生产者发送消息时指定headers
Map<String, Object> headers = new HashMap<>();
headers.put("content-type", "application/json");
headers.put("priority", 10);
channel.basicPublish("headers-exchange", "", new AMQP.BasicProperties.Builder()
    .headers(headers)
    .build(), body);

// 绑定队列时指定header匹配规则
channel.queueBind("myQueue", "headers-exchange", "",
    Map.of("content-type", "application/json"));
```

**应用场景**：基于消息属性的路由，如根据内容类型、版本号等路由。

**对比总结：**

| 类型 | 路由依据 | 性能 | 适用场景 |
|------|---------|------|---------|
| Direct | 精确routingKey | 最高 | 点对点 |
| Fanout | 无（广播） | 高 | 发布-订阅 |
| Topic | 通配符模式 | 中 | 复杂路由 |
| Headers | 消息头属性 | 较低 | 属性路由 |

## 题目8：AI Agent - 什么是LangChain？它的核心组件有哪些？如何协同工作？

【答案】

**LangChain是什么？**

LangChain是一个用于构建LLM应用的开发框架，通过组件化和链式调用的方式，让开发者能够方便地组合大语言模型、工具、外部数据源来构建复杂的AI应用。

**LangChain核心组件（6大模块）：**

**1. Model I/O（模型输入输出）**
- 支持多种LLM：OpenAI GPT、Anthropic Claude、本地模型等
- PromptTemplate：模板化管理提示词
- OutputParser：解析模型输出为结构化数据

```python
from langchain.prompts import PromptTemplate
from langchain.output_parsers import StructuredOutputParser

prompt = PromptTemplate.from_template(
    "请提取以下文本中的关键信息：{text}"
)
```

**2. Retrieval（检索）**
- DocumentLoaders：文档加载（PDF、HTML、Markdown等）
- TextSplitters：文本分割（递归字符分割、语义分割）
- Embeddings：向量化（OpenAI Embeddings、Cohere等）
- VectorStores：向量数据库（Pinecone、Milvus、Chroma）

**3. Chains（链）**
- LLMChain：最基本的LLM调用链
- SequentialChain：顺序执行多个链
- RetrievalQAChain：检索问答链

**4. Agents（智能体）**
- Agent：根据LLM决定执行哪些工具
- Tool：定义可调用的外部工具（搜索、计算、数据库查询）
- Toolkit：工具集

**5. Memory（记忆）**
- ConversationBufferMemory：对话缓冲记忆
- ConversationSummaryMemory：摘要记忆（节省token）
- VectorStoreRetrievedMemory：向量检索记忆

**6. Callbacks（回调）**
- 用于日志、监控、流式输出等

**组件协同工作示例（构建RAG应用）：**

```python
# 1. 文档加载与分割
loader = WebBaseLoader("https://example.com/article")
documents = loader.load()
splitter = RecursiveCharacterTextSplitter(chunk_size=500)
docs = splitter.split_documents(documents)

# 2. 向量化存储
embeddings = OpenAIEmbeddings()
vectorstore = Chroma.from_documents(docs, embeddings)

# 3. 构建检索QA链
retriever = vectorstore.as_retriever()
qa_chain = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(),
    chain_type="stuff",
    retriever=retriever
)

# 4. 执行查询
result = qa_chain.invoke({"query": "文章的主要观点是什么？"})
```

**LangChain的作用**：
- 降低LLM应用开发门槛
- 组件复用，提高开发效率
- 统一接口，方便切换不同LLM/向量库

## 题目9：算法 - 请用最优算法实现「合并区间」（LeetCode 56变体）

【答案】

**题目**：给定一个区间的集合 intervals，合并所有重叠的区间。

**示例**：
```
输入：intervals = [[1,3],[2,6],[8,10],[15,18]]
输出：[[1,6],[8,10],[15,18]]

输入：intervals = [[1,4],[4,5]]
输出：[[1,5]]
```

**解题思路：**

1. 按起点排序
2. 依次遍历，维护当前合并区间
3. 如果下一个区间起点 <= 当前区间终点，则合并（更新终点为max）
4. 否则开新区间

**代码实现（Java）**：

```java
import java.util.Arrays;
import java.util.ArrayList;
import java.util.List;

public class MergeIntervals {
    public int[][] merge(int[][] intervals) {
        if (intervals == null || intervals.length <= 1) {
            return intervals;
        }
        
        // 按起点排序
        Arrays.sort(intervals, (a, b) -> a[0] - b[0]);
        
        List<int[]> result = new ArrayList<>();
        int[] current = intervals[0];
        
        for (int i = 1; i < intervals.length; i++) {
            // 当前区间起点 <= 合并区间终点，重叠
            if (intervals[i][0] <= current[1]) {
                // 更新终点为两者的较大值
                current[1] = Math.max(current[1], intervals[i][1]);
            } else {
                // 不重叠，保存当前区间，开启新区间
                result.add(current);
                current = intervals[i];
            }
        }
        // 别忘了加入最后一个区间
        result.add(current);
        
        return result.toArray(new int[0][]);
    }
}
```

**代码实现（Python）**：

```python
def merge(intervals):
    if not intervals:
        return []
    
    # 按起点排序
    intervals.sort(key=lambda x: x[0])
    
    result = [intervals[0]]
    
    for start, end in intervals[1:]:
        # 如果新区间起点 <= 上一区间终点，则合并
        if start <= result[-1][1]:
            result[-1][1] = max(result[-1][1], end)
        else:
            result.append([start, end])
    
    return result
```

**复杂度分析**：
- 时间复杂度：O(n log n)（排序开销）
- 空间复杂度：O(n)（结果存储，不考虑排序栈空间）

**关键点**：
1. 排序是核心，不排序无法正确合并
2. 合并时取Math.max而不是直接用新区间终点（如[1,5]和[2,3]合并，结果是[1,5]）
3. 最后一个区间容易遗漏，要记得加入结果

## 题目10：算法 - 请实现「LRU缓存」（LeetCode 146），要求get和put都是O(1)

【答案】

**题目**：设计并实现一个LRU（最近最少使用）缓存的数据结构，支持get和put操作，且均为O(1)时间复杂度。

**核心思路**：使用「哈希表 + 双向链表」实现

- **哈希表**：O(1)查找key对应的节点
- **双向链表**：O(1)维护访问顺序，头部是最近使用，尾部是最久未使用

**LRU规则**：容量满时，删除链表尾部（最久未使用），最近访问的放到链表头部。

**代码实现（Java）**：

```java
import java.util.*;

public class LRUCache {
    // 双向链表节点
    class Node {
        int key;
        int value;
        Node prev;
        Node next;
        
        Node(int key, int value) {
            this.key = key;
            this.value = value;
        }
    }
    
    private int capacity;
    private Map<Integer, Node> cache;  // key -> Node
    private Node head;  // 伪头部（最近使用）
    private Node tail;  // 伪尾部（最久未使用）
    
    public LRUCache(int capacity) {
        this.capacity = capacity;
        this.cache = new HashMap<>(capacity);
        // 初始化伪头伪尾
        this.head = new Node(-1, -1);
        this.tail = new Node(-1, -1);
        head.next = tail;
        tail.prev = head;
    }
    
    public int get(int key) {
        Node node = cache.get(key);
        if (node == null) return -1;
        
        // 移动到头部（最近使用）
        moveToHead(node);
        return node.value;
    }
    
    public void put(int key, int value) {
        Node node = cache.get(key);
        
        if (node != null) {
            // 更新值，移动到头部
            node.value = value;
            moveToHead(node);
        } else {
            // 新增节点
            Node newNode = new Node(key, value);
            cache.put(key, newNode);
            addToHead(newNode);
            
            // 容量超限，删除最久未使用的
            if (cache.size() > capacity) {
                Node removed = removeTail();
                cache.remove(removed.key);
            }
        }
    }
    
    // 移动节点到头部
    private void moveToHead(Node node) {
        removeNode(node);
        addToHead(node);
    }
    
    // 删除节点
    private void removeNode(Node node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }
    
    // 添加到头部
    private void addToHead(Node node) {
        node.next = head.next;
        node.prev = head;
        head.next.prev = node;
        head.next = node;
    }
    
    // 删除尾部节点（最久未使用）
    private Node removeTail() {
        Node node = tail.prev;
        removeNode(node);
        return node;
    }
}
```

**使用示例**：

```java
LRUCache cache = new LRUCache(2);  // 容量2

cache.put(1, 1);   // 缓存: {1=1}
cache.put(2, 2);   // 缓存: {1=1, 2=2}
cache.get(1);      // 返回1，缓存变为{2=2, 1=1}
cache.put(3, 3);   // 删除key=2，缓存:{1=1, 3=3}
cache.get(2);      // 返回-1（未找到）
cache.put(4, 4);   // 删除key=1，缓存:{3=3, 4=4}
cache.get(1);      // 返回-1
cache.get(3);      // 返回3
cache.get(4);      // 返回4
```

**Redis中的LRU实现**：

Redis也使用类似策略，当内存超过maxmemory时，Redis通过LRU淘汰策略删除最近最少使用的key。

**复杂度**：get O(1)，put O(1)
