### 题目1：Spark 的架构是怎样的？Driver、Executor、Master、Worker 的作用是什么？

【答案】

**Spark 架构核心组件：**

1. **Driver（驱动器）**
   - 运行 `main()` 方法，创建 `SparkContext`
   - 负责将用户代码转换成 DAG（有向无环图）
   - 将 DAG 划分为 Stage，构建 TaskSet
   - 调度 Task 到 Executor 执行
   - 协调整个应用的运行

2. **Executor（执行器）**
   - 运行在 Worker 节点上的进程
   - 负责执行 Task，并返回结果
   - 将数据缓存到内存或磁盘
   - 向 Driver 汇报自身状态

3. **Master**
   - 集群管理器，负责资源分配
   - 监控 Worker 和 Application 状态
   - 在 Standalone 模式下生效（YARN/Kubernetes 模式下由相应组件管理）

4. **Worker**
   - 运行在集群节点上的进程
   - 负责启动和管理 Executor
   - 监控本节点的资源使用情况

**工作流程：**
```
用户代码 → SparkContext → DAG → Scheduler划分Stage → TaskSet → Worker节点Executor执行
```

**通信机制：**
- Driver 与 Executor 之间通过 Akka 通信（早期）或 Netty（3.x+）
- Executor 启动后向 Driver 注册，Driver 掌握全量 Executor 拓扑

📖 来源：https://spark.apache.org/docs/latest/cluster-overview.html

### 题目2：RDD 是什么？为什么 Spark 要设计 RDD？RDD 有哪些核心特性？

【答案】

**RDD（Resilient Distributed Dataset）** 是 Spark 最核心的数据抽象，表示一个弹性、分布式、可分区的数据集。

**为什么设计 RDD：**
- 传统 MapReduce 磁盘 IO 开销大，迭代计算效率低
- 需要一种高效的内存计算抽象，支持迭代计算和交互式查询
- 提供容错机制，无需大量复制数据

**RDD 五大核心特性：**
1. **分区列表（Partitions）**：数据被分成多个分区，可并行计算
2. **计算函数（Computations）**：每个分区都有 compute 函数，用于计算该分区的数据
3. **依赖关系（Dependencies）**：RDD 与父 RDD 之间的依赖关系（窄依赖/宽依赖）
4. **分区器（Partitioner）**：对于 KV 型 RDD，指定分区策略（如 HashPartitioner）
5. **位置偏好（PreferredLocations）**：数据块的优先位置，计算向数据移动

**窄依赖 vs 宽依赖：**
- **窄依赖**：一个父分区最多被子一个子分区使用（如 map、filter）→ 可流水线执行，容错容易
- **宽依赖**：一个父分区可能被多个子分区使用（如 groupByKey、join）→ 需要 Shuffle，必须等父 Stage 完成

**RDD 惰性执行**：transformation 算子都是惰性的，只有遇到 action 算子时才会触发真正的计算。

📖 来源：https://spark.apache.org/docs/latest/rdd-programming-guide.html

### 题目3：DataFrame 和 DataSet 有什么区别？与 RDD 相比有什么优势？

【答案】

**DataFrame：**
- 类似于传统数据库的二维表结构，有列名和类型
- 底层采用 RDD + Schema 实现，本质是 `Row` 类型的 RDD
- 提供 SQL 风格的操作 API（select、filter、groupBy）
- 相比 RDD 有更丰富的优化（Catalyst 优化器、 Tungsten 序列化）

**DataSet：**
- 是 Spark 1.6 引入的新抽象，兼具 DataFrame 的优化和 RDD 的强类型
- 是类型安全的，编译时会检查类型
- DataFrame 是 DataSet[Row] 的别名（Untyped），DataSet 是强类型（Typed）
- 支持 Encoder 自动序列化

**DataFrame vs DataSet 对比：**

| 特性 | DataFrame | DataSet |
|------|-----------|---------|
| 类型安全 | 不安全（运行期检查） | 安全（编译期检查） |
| Schema | 有固定 Schema | 有 Schema |
| 操作方式 | SQL / DSL | DSL（强类型 lambda） |
| 序列化 | Tungsten 自动序列化 | Encoder 序列化 |
| 性能优化 | Catalyst 优化 | Catalyst + Tungsten |
| null 处理 | 原生支持 | 原生支持 |

**与 RDD 相比的优势：**
1. **Schema 预定义**：支持指定列名和数据类型，计算更高效
2. **Catalyst 优化器**： DSL 算子会被 Catalyst 优化为执行计划（如谓词下推、列剪裁）
3. **Tungsten 序列化**：使用堆外内存和高效二进制序列化，减少 GC 开销
4. **易用性**：可以用 SQL、DSL 两种方式操作数据

📖 来源：https://spark.apache.org/docs/latest/sql-programming-guide.html

### 题目4：Spark Shuffle 是什么？Hash Shuffle 和 Sort Shuffle 有什么区别？如何优化 Shuffle？

【答案】

**Shuffle 是什么：**
- 当 RDD/DataFrame 的一个分区的数据需要根据 Key 重新分配到多个分区时（如 groupByKey、join、repartition），就会发生 Shuffle
- Shuffle 是 Spark 最昂贵的操作之一，会触发数据跨节点传输和磁盘 IO

**Hash Shuffle（Spark 1.x 早期）：**
- 每个 Mapper 会为每个 Reducer 单独写一个文件
- 假设 M 个 Mapper，R 个 Reducer，会产生 M×R 个小文件
- 文件数量巨大，大量小文件导致文件系统压力和 GC 问题

**Sort Shuffle（Spark 1.1+ 引入，1.2 成为默认）：**
- 每个 Mapper 将数据写入一个本地文件，按 Key 排序
- 如果需要合并（需要全局排序），会使用归并排序
- 每个 Mapper 只产生 2 个文件（数据文件 + 索引文件）
- 大幅减少文件数量（从 M×R 降为 2M）

**Sort Shuffle 的两种模式：**
1. **Bypass 模式**：当 shuffle read partition 数量 <= `spark.shuffle.sort.bypassMergeThreshold`（默认200）且没有 map-side combine 时，走简化路径，不排序直接输出
2. **普通模式**：正常排序后输出

** Tungsten Sort Shuffle（Spark 2.0+）：**
- 使用 `TungstenEncoder` 直接序列化数据到字节数组
- 写入排序后的序列化数据，通过 `ShuffleExternalSorter` 管理
- 进一步提升性能，减少内存和磁盘开销

**Shuffle 优化建议：**
1. **减少 shuffle 数据量**：尽量使用 map-side 预聚合（如 reduceByKey 而非 groupByKey）
2. **合理设置分区数**：`spark.sql.shuffle.partitions`（默认200）根据数据量调整，避免过多小分区
3. **使用广播变量**：小表 join 时用广播避免 shuffle
4. **序列化优化**：使用 Kryo 序列化减少数据量
5. **开启 shuffle 压缩**：`spark.shuffle.compress=true`（默认开启）

📖 来源：https://spark.apache.org/docs/latest/rdd-programming-guide.html#shuffle-operations

### 题目5：Spark 的内存管理是怎样的？Executor 内存是如何划分的？

【答案】

**Spark 2.x 统一内存管理（Unified Memory Management）：**

Executor 的堆内内存主要由以下部分组成：

1. **Execution 内存（执行内存）**
   - 用于 Shuffle、Join、Sort 等计算操作的临时内存
   - 可以动态借用 Storage 内存
   - 不足时可溢出到磁盘

2. **Storage 内存（存储内存）**
   - 用于缓存 RDD/DataFrame 数据（cache、persist）
   - 可以动态借用 Execution 内存
   - 存储广播变量

3. **User Memory（用户内存）**
   - 用户代码使用的内存（如创建对象、数据结构）
   - 默认占 25%（`spark.memory.fraction=0.6`，剩余的 40% 中 User 占 1/6 ≈ 25.6%）
   - 不可借用给其他区域

4. **Reserved Memory（保留内存）**
   - 300MB 固定保留，用于系统内部对象
   - 不可被用户代码使用

**内存比例（堆内）：**
```
Total = Reserved(300MB) + User Memory(25%) + Spark Memory(75%)
Spark Memory = Execution(30%~60%) + Storage(30%~60%)
```

**堆外内存（Off-heap，Tungsten）：**
- 通过 `spark.memory.offHeap.enabled=true` 开启
- 使用 `sun.misc.Unsafe` 直接操作堆外内存
- 减少 GC 开销，适合大内存场景
- 大小由 `spark.memory.offHeap.size` 指定

**Storage 与 Execution 内存动态共享：**
- 空闲时互相借用
- 的一方需要空间时，可以强制回收对方占用的内存（基于 LRU 策略）
- Storage 内存被 Execution 抢占后，数据会被降级（从内存移到磁盘）

**内存溢出（OOM）常见原因：**
1. 数据倾斜导致单分区数据过大
2. 分区数过少，单分区数据量超过 Execution 内存
3. 缓存过多数据，Storage 内存不足
4. 使用 collect() 将数据拉回 Driver，Driver 内存不足

📖 来源：https://spark.apache.org/docs/latest/configuration.html#memory-management
