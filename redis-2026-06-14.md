## 题目一：Redis 有哪些主要的数据类型？它们各自的使用场景是什么？

### 参考答案

Redis 提供了 5 种基础数据类型，以及几种高级数据结构：

| 数据类型 | 底层实现 | 典型使用场景 |
|---------|---------|------------|
| **String** | SDS（简单动态字符串） | 缓存简单值、计数器、分布式锁（SETNX）、限流（incr） |
| **Hash** | 压缩列表（ziplist） 或 哈希表（hashtable） | 存储对象/结构体数据，如用户信息、商品详情；适合部分字段更新的场景 |
| **List** | 压缩列表 或 双向链表 | 消息队列（lpush + brpop）、最新列表（lpush + ltrim）、时间线（朋友圈动态） |
| **Set** | 整数集合（intset） 或 哈希表 | 去重集合、标签系统、共同好友、抽奖（SRANDMEMBER） |
| **Zset** | 压缩列表 或 跳表（skiplist）+ 哈希表 | 有序排行榜（游戏积分、热度排行）、延迟队列、权重队列 |
| **Bitmap** | 位数组 | 签到统计、在线状态、UV 统计（BITOP） |
| **HyperLogLog** | PFADD/PFCOUNT | 大规模去重计数（UV、DAU），误差约 0.81% |
| **Geospatial** | zset（score = geohash） | 附近的人/商家、LBS 应用 |
| **Stream** | Rax 树 + 消费者组 | 消息队列（支持 ACK、阻塞消费），替代 List 实现可靠消息 |

**高频追问：SDS 与 C 字符串的区别？**
- SDS 支持 O(1) 长度获取（`len` 字段）、空间预分配（减少内存分配次数）、惰性空间释放（避免频繁 realloc）。
- C 字符串以 `\0` 结尾，拼接操作是 O(n)，存在缓冲区溢出风险。

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 参考答案

Redis 有两种持久化方式：

#### RDB（Redis Database Backup）
- **原理**：在指定时间点生成内存数据的快照，fork 一个子进程将数据写入临时 RDB 文件，完成后替换旧文件。
- **触发方式**：`bgsave`（后台异步）、`save`（同步阻塞，线上禁用）、自动触发（`save m n` 配置）。
- **优点**：恢复速度快（直接加载 `.rdb` 文件），适合冷备份/灾难恢复。
- **缺点**：fork 时机到故障之间的数据会丢失（最多丢失一个快照周期），fork 时内存越大延迟越高。

#### AOF（Append Only File）
- **原理**：将每个写操作以 Redis 协议格式追加到文件末尾。重启时重放命令恢复数据。
- **同步策略**：`always`（每条命令刷盘，最安全但最慢）、`everysec`（每秒刷盘，默认推荐）、`no`（由 OS 决定）。
- **重写机制**：`bgrewriteaof` 命令压缩 AOF 文件（去除冗余命令，如多次 SET 同一 key 保留最后一次）。
- **优点**：数据安全性更高（`everysec` 最多丢失 1 秒数据），AOF 重写后可生成更小的文件。
- **缺点**：AOF 文件通常比 RDB 大，恢复速度慢。

#### 关键区别对比

| 维度 | RDB | AOF |
|------|-----|-----|
| 文件格式 | 二进制快照 | 文本（Redis 协议命令） |
| 恢复速度 | 快 | 慢 |
| 数据完整性 | 可能有丢失 | 丢失少 |
| 文件体积 | 小 | 大 |
| 开销 | fork + 写盘 | 持续写盘 |
| 压缩重写 | 每次全量生成 | 后台增量重写 |

#### 生产环境选择建议
- **小内存实例 + 能接受分钟级数据丢失**：`RDB` 即可。
- **对数据安全性要求高**：`RDB + AOF` 混合持久化（`aof-use-rdb-preamble yes`），Redis 7.0 默认开启。重放时优先用 RDB 快速恢复，再用 AOF 补缺。
- **绝对不能丢数据**：`always` 刷盘的 AOF，或配合主从复制做热备份。

## 题目三：什么是缓存穿透、缓存击穿、缓存雪崩？如何解决？

### 参考答案

这是缓存系统的三大经典问题：

#### 1. 缓存穿透
- **定义**：查询一个**不存在**的数据，缓存和数据库都没有，每次请求都打到 DB。
- **危害**：大量无效请求击穿数据库。
- **解决思路**：
  - **布隆过滤器（BloomFilter）**：将所有存在的 key 存入 Bitmap，请求进来先查布隆过滤器，存在才查缓存/DB。适用于数据相对稳定的场景。
  - **缓存空值（null）**：查询 DB 无结果时，在 Redis 缓存一个 `null` 或约定值（如 `"NULL"`），并设置较短过期时间（防脏数据）。
  - **参数校验**：接口层做基础合法性校验（如 id <= 0 直接拒绝）。

#### 2. 缓存击穿
- **定义**：**热点 key 过期**的瞬间，大量并发请求同时涌入，直接打穿到 DB。
- **特点**：只有 1 个（或少数）热点 key，不适用于大量 key 同时过期的情况。
- **解决思路**：
  - **互斥锁（SETNX）**：缓存失效时，只允许一个请求去查 DB 并回填缓存，其他请求等待锁后从缓存取。实现简单，但会损失部分并发能力。
  - **逻辑永不过期**：不给热点 key 设置 TTL，而是用异步任务根据数据版本号主动更新缓存。
  - **热点数据永驻**：将过期时间加上随机偏移量，避免大量同时过期。

#### 3. 缓存雪崩
- **定义**：**大量 key 同时过期**或**Redis 宕机**，导致大量请求直接打到数据库。
- **解决思路**：
  - **过期时间随机化**：给 TTL 加随机值（如 `baseTTL + random(0, 300s)`），错开过期时间。
  - **Redis 高可用**：主从 + 哨兵（Sentinel）或集群模式，避免单机故障。
  - **本地缓存**：在应用层加一层本地缓存（如 Caffeine、Guava Cache），减少对 Redis 的冲击。
  - **服务熔丝/限流**：Sentinel、Hystrix 等，数据库被打爆前先降级。
  - **预热**：系统启动时主动加载热点数据到缓存。

## 题目四：Redis 主从复制原理是什么？哨兵（Sentinel）和集群（Cluster）有什么区别？

### 参考答案

#### 主从复制原理
1. 从节点执行 `replicaof <master-ip> <port>`（Redis 5.0 前是 `slaveof`），向主节点发送 `PSYNC` 命令。
2. 主节点触发 `bgsave`，生成 RDB 文件，同时将后续写命令存入 **复制积压缓冲区（replication backlog）**。
3. 主节点将 RDB 文件发送给从节点，从节点加载到内存。
4. 主节点将 backlog 中的增量命令发送给从节点，完成同步。
5. 此后主节点每执行一条写命令，就异步发送给从节点（**命令传播**阶段）。
6. 若从节点断开重连，支持**部分重同步**（PSYNC2），利用 backlog 增量补发；若 backlog 不够则触发全量重同步。

#### 哨兵（Sentinel）模式
- **定位**：Redis 的**高可用（HA）**方案，解决单点故障问题。
- **原理**：Sentinel 是一个独立进程，监控主从节点的健康状态。当主节点宕机，Sentinel 自动完成**故障转移（failover）**——从从节点中选出一个新的主节点，其他从节点指向新主节点，客户端配置自动更新。
- **特点**：仍然只有**一主多从**，没有分片，数据不分离；主节点写压力是瓶颈。
- **最小部署**：3 个 Sentinel 节点（需要多数投票避免脑裂）。

#### Redis Cluster 模式
- **定位**：Redis 的**分布式数据存储**方案，解决数据量大和并发高的问题。
- **原理**：16384 个 slot 分片，节点管理自己负责的 slot，数据按 slot 路由（`MOVED` 重定向）。每个分片可以有一主多从（副本）。
- **特点**：
  - 数据分布式存储（水平扩展），每个节点只存部分数据。
  - 支持自动故障转移（节点副本自动提升）。
  - 客户端直接连任意节点，通过 `MOVED/ASK` 重定向找到正确节点。
  - 不支持跨 slot 的事务（`MULTI/EXEC` 需在同一个节点）。
  - 官方推荐至少 6 个节点（3 主 3 从）。
- **适用场景**：数据量大（单实例放不下）、并发高（单实例 QPS 瓶颈）。

#### 对比

| 维度 | 哨兵（Sentinel） | 集群（Cluster） |
|------|----------------|----------------|
| 数据分片 | 无，仍然是主从复制 | 有，16384 slot 分片 |
| 写扩展性 | 差（只有一主） | 好（多主可并发写） |
| 数据量 | 受单机内存限制 | 可水平扩展 |
| 故障转移 | Sentinel 自动完成 | 集群内部自动完成 |
| 客户端支持 | 智能客户端/Sentinel | 支持 Cluster 的客户端 |
| 跨 slot 操作 | 支持 | 不支持 |

## 题目五：如何实现一个可靠的 Redis 分布式锁？需要注意哪些问题？

### 参考答案

#### 基本实现（SETNX）
```python
# 获取锁
lock_key = "product:lock:123"
lock_value = uuid  # 唯一标识，防止误删他人的锁
expire_seconds = 30

if redis.set(lock_key, lock_value, nx=True, ex=expire_seconds):
    try:
        # 业务逻辑
        pass
    finally:
        # 释放锁（Lua 脚本保证原子性）
        lua_script = """
        if redis.call("get", KEYS[1]) == ARGV[1] then
            return redis.call("del", KEYS[1])
        else
            return 0
        end
        """
        redis.eval(lua_script, 1, lock_key, lock_value)
else:
    # 获取锁失败，重试或直接返回
    pass
```

#### 需要注意的核心问题

1. **锁误删（最关键）**：不能用 `DEL` 直接删锁——必须用 Lua 脚本验证 `value` 是否与加锁时一致，防止删除其他客户端的锁。
2. **锁过期时间**：业务执行时间可能超过 TTL，需评估好最大执行时间。业务执行前先判断剩余时间，必要时续期（watchdog 机制，Redisson 框架采用此方案）。
3. **不可重入**：同一线程不能重复获取同一把锁（除非支持重入）。可改进为 `HINCRBY` 计数 + 线程标识实现可重入锁。
4. **单点故障**：单机 Redis 挂了则锁失效。解决方案：**RedLock（红锁）**：向 N 个独立 Redis 实例（建议 ≥ 5 台）获取锁，超过半数成功才算加锁成功。
5. **异步复制丢失**：主节点加锁成功后宕机，从节点未同步该数据。RedLock 通过多数派机制缓解但不能完全消除（CAP 理论）。
6. **Clock Drift**：多节点时间不同步导致 TTL 计算偏差，RedLock 建议使用逻辑时钟而非物理时钟。

#### 生产级方案选型

| 方案 | 框架 | 适用场景 |
|------|------|---------|
| 单机 Redis SETNX | 手写/Lua | 简单场景，对可用性要求不高 |
| RedLock | Redisson / Redisson | 对正确性要求高的分布式场景 |
| Zookeeper 分布式锁 | Curator | 强一致性场景（CP），性能低于 Redis |
| 数据库唯一索引 | 手写 | 低并发、错误容忍度高的场景 |

**一句话总结**：分布式锁的核心是**互斥 + 防误删 + TTL + 高可用**，生产环境推荐使用 Redisson 等成熟框架，不要自己手写一个"看起来对"的实现。
