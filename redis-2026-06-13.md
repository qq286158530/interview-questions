## 题目 1：Redis 的持久化机制有哪些？RDB 和 AOF 各自的优缺点是什么？混合持久化是什么原理？

### 答案

Redis 提供了三种数据持久化方式：**RDB 快照**、**AOF 日志**、以及两者结合的**混合持久化**。

### RDB 快照

RDB 快照是记录某一瞬间的内存数据，以二进制格式写入磁盘。Redis 提供了两个生成 RDB 文件的命令：

- **save**：在主线程执行，会阻塞主线程；
- **bgsave**：创建子进程在后台生成 RDB 文件，主线程不受影响（推荐）。

bgsave 使用了 **写时复制（Copy-On-Write, COW）** 技术：fork 子进程时父子进程共享物理内存，只有当主线程修改了共享数据时，才会复制出独立副本。这样 bgsave 期间主线程仍可正常处理请求。

Redis 默认配置满足以下任一条件即触发 bgsave：
- 900 秒内至少 1 次修改
- 300 秒内至少 10 次修改
- 60 秒内至少 10000 次修改

### AOF 日志

AOF（Append Only File）通过记录每次写操作命令来实现持久化。Redis 执行完写命令后，将命令追加写入 `server.aof_buf` 缓冲区，再通过系统调用写入磁盘。

AOF 提供三种写回策略（`appendfsync` 配置项）：

| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| Always | 每次写命令后同步写回磁盘 | 数据不丢失 | 磁盘 IO 成为瓶颈，性能最低 |
| Everysec（默认） | 每秒将缓冲区内容写回磁盘 | 性能和数据安全兼顾 | 可能丢失 1 秒数据 |
| No | 由操作系统决定何时写入 | 性能最好 | 数据依赖操作系统，可能丢失较多 |

AOF 文件过大会触发**重写机制**（bgrewriteaof），后台子进程读取当前数据库中所有键值对，用一条命令记录每个 key 的最新值，实现文件压缩。重写期间主线程会将新命令同时写入 AOF 缓冲区和 AOF 重写缓冲区，保证数据一致性。

### RDB vs AOF

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件格式 | 二进制 | 文本（命令操作日志） |
| 恢复速度 | 快（直接读入内存） | 慢（需逐一执行命令） |
| 数据完整性 | 两次快照之间数据可能丢失 | 可精确到秒级 |
| 快照频率 | 不能太频繁（重量级操作） | 可每秒记录 |

### 混合持久化

Redis 4.0 引入了混合持久化，结合两者优点。开启配置 `aof-use-rdb-preamble yes`。

原理：在 AOF 重写期间，fork 的子进程先将共享内存数据以 RDB 格式写入 AOF 文件前半部分，主线程的增量操作命令记录到重写缓冲区，最终以 AOF 格式追加到后半部分。

优点：Redis 重启加载时，先快速加载 RDB 部分，再用 AOF 部分补充后续增量数据，兼顾启动速度和数据完整性。

## 题目 2：Redis 数据类型的底层实现是什么？SDS、压缩列表、哈希表、跳表各自的适用场景是什么？

### 答案

### String 类型

String 的底层实现主要有两种：**int** 和 **SDS（简单动态字符串）**。

- 如果保存的是整数值且可用 long 类型表示，则使用 int 编码，直接将整数值保存在 ptr 属性中；
- 如果是字符串且长度 ≤ 32~44 字节（Redis 版本不同阈值不同），使用 **embstr** 编码——在一次内存分配中连续存放 redisObject 和 SDS，访问效率更高；
- 如果是长字符串，则使用 **raw** 编码，redisObject 和 SDS 分别分配内存。

SDS 相比 C 原生字符串的优势：
- **O(1) 复杂度获取长度**（C 字符串需遍历 O(n)）
- **二进制安全**（以 len 判断结束，可存图片/音频等二进制数据）
- **空间自动扩容**，不会缓冲区溢出

### List 类型

- 元素个数 < 512 且每个元素 < 64 字节 → **压缩列表（ziplist）**
- 否则 → **双向链表（linkedlist）**

Redis 3.2 后，底层统一使用 **quicklist**（压缩列表 + 双向链表的混合体），替代了单纯的 ziplist 和 linkedlist。Redis 7.0 后，ziplist 进一步被 **listpack** 替代。

### Hash 类型

- 元素个数 < 512 且每个 value < 64 字节 → **压缩列表（ziplist）**
- 否则 → **哈希表（hashtable）**

Redis 7.0 后 ziplist 被 listpack 替代。

### Set 类型

- 元素全是整数且个数 < 512 → **整数集合（intset）**，节省内存
- 否则 → **哈希表（hashtable）**

### Zset 类型

- 元素个数 < 128 且每个元素 < 64 字节 → **压缩列表（ziplist）**
- 否则 → **跳表（skiplist）**

**跳表**是 Zset 实现排序功能的核心数据结构，本质是多层链表，最底层包含所有元素，每一层是有序子集，通过随机层数实现 O(log N) 的查找效率。

Redis 7.0 后 ziplist 被 listpack 替代。

## 题目 3：什么是缓存雪崩、击穿、穿透？各自有哪些解决方案？

### 答案

引入 Redis 作为缓存层后，面临三类典型的缓存异常问题。

### 缓存雪崩

大量缓存数据**同一时间过期**或 **Redis 故障宕机**，导致大量请求直接打到数据库，数据库压力骤增甚至宕机，引发连锁反应导致系统崩溃。

**解决方案：**

- **大量数据过期**：给过期时间加随机数，避免同时失效；使用互斥锁保证同一时间只有一个请求构建缓存；后台定时更新缓存替代过期机制。
- **Redis 故障宕机**：启用服务熔断或请求限流；构建主从高可用集群保证故障自动切换。

### 缓存击穿

热点数据在缓存中**过期**，此时大量请求恰好访问该热点数据，全部打到数据库。这是缓存雪崩的一个子集。

**解决方案：**
- 互斥锁，保证只有一个请求重建缓存；
- 不给热点数据设置过期时间，由后台异步更新；
- 热点数据即将过期时，提前通知后台线程更新并重新设置过期时间。

### 缓存穿透

要访问的数据**既不在缓存也不在数据库**中（如业务误删、黑客恶意攻击），导致请求每次都访问数据库。

**解决方案：**
- **非法请求限制**：API 入口校验请求参数，过滤恶意请求；
- **缓存空值/默认值**：查询不到的数据在缓存中设置空值或默认值；
- **布隆过滤器（Bloom Filter）**：写入数据库时在布隆过滤器做标记，查询时先通过布隆过滤器判断数据是否存在——存在不一定在 DB，但不存在则一定不在 DB。

| 问题 | 原因 | 核心应对 |
|------|------|----------|
| 缓存雪崩 | 大量数据同时过期 / Redis 宕机 | 过期时间随机化 + 高可用集群 |
| 缓存击穿 | 热点数据过期 | 互斥锁 / 后台异步更新 |
| 缓存穿透 | 数据既不在缓存也不在 DB | 布隆过滤器 / 缓存空值 |

## 题目 4：Redis 是单线程吗？为什么单线程的 Redis 还能达到每秒数万 QPS？

### 答案

### Redis 的单线程与多线程

Redis **单线程**指的是"接收客户端请求 → 解析请求 → 数据读写等操作 → 发送响应"这个核心流程由**一个主线程**完成。

但 Redis 启动后会启动多个**后台线程**（Bio 子线程）：
- Redis 2.6：2 个后台线程——处理关闭文件、AOF 刷盘
- Redis 4.0+：新增 lazy free 线程，异步释放内存（如 `unlink key`、`flushdb async`）
- Redis 6.0+：引入了 **I/O 多线程**，专门处理网络 I/O 的发送（write client socket），但命令执行仍然是单线程

所以准确地说：
- **Redis 6.0 之前**：核心网络 I/O 和命令执行是单线程 + 2~3 个 Bio 后台线程
- **Redis 6.0 之后**：主线程单线程执行命令 + I/O 多线程分担网络 I/O + Bio 后台线程

### 单线程 Redis 为什么这么快？

1. **内存操作**：Redis 数据在内存中，操作瓶颈是内存或网络带宽而非 CPU，单线程足够应对；
2. **高效数据结构**：SDS、跳表等高效内部数据结构，避免了复杂操作；
3. **无锁竞争**：单线程不存在多线程切换和锁竞争的开销；
4. **I/O 多路复用**：使用 epoll/select/kqueue 等机制，一个线程高效监听处理大量 Socket 请求；
5. **避免阻塞**：后台线程处理耗时任务（关闭文件、AOF 刷盘、释放内存），不阻塞主线程。

Redis 官方基准测试中，单线程 Redis 吞吐量可达 **10W QPS** 以上。

## 题目 5：Redis 主从复制、哨兵模式、Cluster 集群的区别是什么？分别解决了什么问题？

### 答案

### 主从复制（Master-Slave Replication）

一主多从架构，主节点负责读写，从节点只读。主节点写入数据后异步同步给从节点。

**特点**：
- 命令复制是**异步**的，主节点执行完命令即返回客户端，不等待从节点执行；
- 无法保证强一致性，数据在主从间存在不一致窗口；
- 从节点故障需要手动切换。

**解决的问题**：读写分离 + 数据冗余备份。

### 哨兵模式（Sentinel）

在主从复制基础上，增加了**自动故障转移**能力。哨兵是一个独立进程，监控主从节点健康状态。

**核心功能**：
- **监控**：持续检测主节点和从节点是否存活；
- **自动故障转移**：主节点宕机时，哨兵自动将从节点升级为新的主节点，并通知所有从节点切换；
- **通知**：将故障转移结果通知给客户端（通过订阅发布机制）。

**解决的问题**：主节点故障时需要人工介入的问题，实现**高可用（HA）**。

### Cluster 集群（切片集群）

当数据量单节点内存无法容纳时，需要**水平扩展**。Redis Cluster 通过**分片（slot）** 实现数据分不到多个节点。

**核心机制**：
- 将整个数据空间划分为 **16384 个 slot**，分配到多个节点；
- 每个节点负责一部分 slot 的数据；
- 客户端请求任意节点，节点根据 key 的 hash 值路由到正确节点；
- 支持**故障自动转移**（部分迁移自 Sentinel 机制）；
- 支持**异地多活**部署。

**解决的问题**：单节点内存瓶颈 + 水平扩展 + 高可用。

### 三者对比

| 特性 | 主从复制 | 哨兵模式 | Cluster 集群 |
|------|----------|----------|--------------|
| 数据复制 | 异步 | 异步 | 异步（分片） |
| 故障自动切换 | ❌ | ✅ | ✅ |
| 水平扩展 | ❌ | ❌ | ✅ |
| 节点角色 | 主/从 | 主/从/哨兵 | 多主多从 |
| slot 分片 | ❌ | ❌ | ✅ |
| 客户端路由 | 任意节点 | 任意节点 | 智能路由 |
