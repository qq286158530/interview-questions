## 题目一：Flink 的架构是怎样的？为什么说 Flink 是真正的流计算引擎？

### 参考答案

Flink 架构主要分为三层：**分布式数据集（Distributed Dataflow）**、**流处理（Stream Processing）** 和 **API 层**。

核心组件：
- **JobManager**：协调整个作业的执行，负责调度任务、协调 Checkpoint、处理失败恢复等。
- **TaskManager**：执行具体的算子任务，一个 TaskManager 可以运行一个或多个 subtask（并行的任务实例）。
- **Slot**：TaskManager 的资源单位，每个 Slot 有固定的内存资源，CPU 资源共享。默认情况下 Slot 数量 = TaskManager 数量 × 每个 TM 的 Slot 数。

**为什么 Flink 是真正的流计算引擎：**

1. **原生流处理模型**：Flink 将批处理视为流处理的一个特例（有限流），而非反过来。数据在网络中一直以数据流的形式传递，不需要像 Spark 那样微批处理（mini-batch）。
2. **低延迟**：由于是逐条记录处理，Flink 可以提供毫秒级延迟，而 Spark Structured Streaming 存在 micro-batch 的固有延迟。
3. **精确一次（Exactly-Once）语义**：Flink 通过分布式快照（Chandy-Lamport 算法变体）实现端到端的精确一次语义，优于 Spark 的至少一次（At-Least-Once）。
4. **事件时间处理**：Flink 支持事件时间（Event Time）和处理时间（Processing Time），通过 Watermark 机制处理乱序数据，这是流计算的核心能力。

**Flink 数据流执行流程：**
```
Stream -> Source -> Transformation -> Sink
              ↑
         (Parallelism)
```

每个算子可以设置并行度，数据在算子之间通过 StreamGraph 进行传递，形成 DAG 结构。

## 题目二：什么是 Watermark？Flink 如何处理乱序和迟到数据？

### 参考答案

**Watermark（水位线）** 是 Flink 用于处理**事件时间（Event Time）**乱序到达的一种机制，本质上是一个时间戳，表示"所有时间戳小于该 Watermark 的数据都已经到达"。

### 核心概念

```
Event Time（事件时间）：数据本身携带的时间戳（业务时间）
Processing Time（处理时间）：数据被 Flink 算子处理时的时间
Watermark T：意味着所有 event_time < T 的数据都已经到达（或者说不会再来了）
```

### 解决乱序问题的原理

1. **窗口触发时机**：当 Watermark 到达窗口结束时间时，窗口才会触发计算。即 ` watermark >= window_end_time`
2. **乱序数据处理**：
   - **_allowedLateness**：允许数据迟到的时间窗口。迟到但在允许范围内的数据，仍会触发窗口计算。
   - **sideOutputTag**：超过允许迟到时间的"极端迟到"数据，可以被侧输出流捕获，避免丢失。

### 代码示例

```java
// 定义 Watermark 策略：允许 5 秒乱序
WatermarkStrategy
    .<Tuple2<String, Long>>forBoundedOutOfOrderness(Duration.ofSeconds(5))
    .withTimestampAssigner((event, timestamp) -> event.f1) // 从数据中提取时间戳
    .withIdleness(Duration.ofMinutes(1)); // 处理空闲数据源
```

### 多流 Watermark 合并

当两个流做 join 时，每个流的 Watermark 独立生成，在算子处取两个流的最小 Watermark，以保证两个流的数据都就绪后再处理。

### 常见 Watermark 策略

| 策略 | 描述 | 适用场景 |
|------|------|---------|
| `forBoundedOutOfOrderness` | 固定乱序容忍度 | 已知数据延迟上界 |
| `forMonotonousTimestamps` | 无延迟，Watermark = 当前最大时间戳 | 测试或低延迟场景 |
| `forGenerator` | 自定义 Watermark 生成器 | 特殊业务逻辑 |

## 题目三：Flink Checkpoint 和 Savepoint 的区别是什么？Flink 是如何实现 Checkpoint 的？

### 参考答案

### Checkpoint vs Savepoint

| 特性 | Checkpoint | Savepoint |
|------|-----------|-----------|
| **触发方式** | Flink 自动周期性触发 | 手动触发（用户命令） |
| **用途** | 故障恢复，作业失败时自动从最近 Checkpoint 恢复 | 计划性停机维护、作业升级、A/B 测试 |
| **存储位置** | JobManager 内存（快速失败恢复） | 外部存储系统（HDFS/S3/NAS） |
| **删除策略** | 作业取消后默认删除 | 手动删除 |
| **格式** | 简化的状态格式 | 标准化的可移植格式 |
| **兼容性** | 不保证跨版本兼容 | 保证跨版本兼容（通常） |

### Checkpoint 实现原理——Chandy-Lamport 分布式快照

Flink 采用 **Chandy-Lamport 算法变体**实现分布式快照，核心思想：

1. **JobManager 周期性发起 Checkpoint barrier**：在数据流中插入特殊的 barrier（屏障）标记。
2. **barrier 对齐（Barrier Alignment）**：每个算子收到多个输入的 barrier 时，需要等所有 barrier 到达后才向下游发送 barrier（避免快照不一致）。
3. **状态快照**：算子将当前状态异步快照到状态后端（如 RocksDB），同时向 Checkpoint 协调器确认完成。
4. **增量 Checkpoint**：只存储与上一次 Checkpoint 的差异，适合大状态场景。

### Checkpoint 配置关键参数

```java
env.enableCheckpointing(60_000);           // 60秒做一次 Checkpoint
env.getCheckpointConfig().setCheckpointTimeout(10 * 60 * 1000); // Checkpoint 超时 10 分钟
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30_000); // 两次 Checkpoint 最小间隔 30 秒
env.getCheckpointConfig().setMaxConcurrentCheckpoints(1);        // 最大并发 Checkpoint 数
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE); // 精确一次语义
```

### 状态后端选择

| 状态后端 | 适用场景 | 特点 |
|---------|---------|------|
| HashMapStateBackend | 小状态、低延迟 | 内存存储，状态上限 = JVM 堆内存 |
| EmbeddedRocksDBStateBackend | 大状态、增量 Checkpoint | RocksDB 本地磁盘 + 异步快照 |

## 题目四：Flink 窗口函数有哪些？滚动窗口、滑动窗口、会话窗口的区别是什么？

### 参考答案

### Flink 窗口类型

#### 1. 时间窗口（Time Windows）

- **滚动窗口（Tumbling Window）**：窗口长度固定，不重叠。
  ```java
  TumblingEventTimeWindows.of(Time.seconds(10))  // 每10秒一个窗口
  ```
- **滑动窗口（Sliding Window）**：窗口长度固定，但可以重叠。
  ```java
  SlidingEventTimeWindows.of(Time.seconds(10), Time.seconds(5)) // 窗口长10秒，步长5秒
  ```
- **会话窗口（Session Window）**：基于数据之间的间隔动态创建窗口。
  ```java
  SessionWindow.withGap(Time.seconds(10))  // 10秒无数据则关闭窗口
  ```

#### 2. 计数窗口（Count Windows）

- **滚动计数窗口**：固定数量数据触发一次窗口。
  ```java
  countWindow(100)  // 每100条数据触发
  ```
- **滑动计数窗口**：
  ```java
  countWindow(100, 50)  // 每100条触发，步长50
  ```

#### 3. 全局窗口（Global Window）

将所有相同 key 的数据分配到同一个窗口，需要自定义触发器。

```java
GlobalWindows.create()
```

### 窗口函数（Process Window Function）

| 类型 | 特点 | 性能 |
|------|------|------|
| **ReduceFunction** | 增量聚合，状态最小化 | 最优 |
| **AggregateFunction** | 增量聚合，支持多输出 | 优 |
| **FoldFunction** | 旧版 API，逐渐被淘汰 | 一般 |
| **ProcessWindowFunction** | 全量窗口数据，状态大 | 差（配合其他聚合使用） |

### 窗口水印与迟到数据处理组合

```java
stream
    .assignTimestampsAndWatermarks(watermarkStrategy)
    .keyBy(<key selector>)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .allowedLateness(Time.minutes(1))   // 允许迟到1分钟
    .sideOutputLateData(lateOutputTag)  // 超时数据侧输出
    .aggregate(new MyAggregateFunction(), new MyProcessWindowFunction())
```

### 关键区别总结

| 特性 | 滚动窗口 | 滑动窗口 | 会话窗口 |
|------|---------|---------|---------|
| 窗口长度 | 固定 | 固定 | 动态（由数据间隔决定） |
| 重叠 | 不重叠 | 重叠 | 不重叠 |
| 触发次数/周期 | 固定周期 | 固定周期（更频繁） | 不固定（依赖数据到达） |
| 适用场景 | 固定周期统计 | 平滑统计、趋势分析 | 用户行为分析、会话统计 |

## 题目五：Flink 的状态管理机制是什么？为什么需要状态管理？有哪些状态后端？

### 参考答案

### 什么是状态（State）

**状态**是 Flink 流处理中，算子在处理过程中需要"记住"的数据。例如：
- 窗口聚合的当前累加值
- 匹配事件模式的已匹配事件列表
- 实时计算中的历史累计值

### 状态类型

#### 按 keyed 划分
- **Keyed State**：按 key 分区，只能在 `keyBy` 后的算子中使用，作用域是当前 key。
  - `ValueState[T]`：单一值状态
  - `ListState[T]`：列表状态
  - `MapState[K, V]`：映射状态
  - `ReducingState[T]`：自动聚合的单值状态
  - `AggregatingState[IN, OUT]`：聚合状态

#### 按算子划分
- **Operator State（算子状态）**：算子内所有并行实例共享，常见于 Source 算子（如 Kafka offset）。
  - `ListState`
  - `UnionListState`（故障恢复时广播给所有子任务）
  - `BroadcastState`（广播状态，所有子任务一致）

### 状态后端（State Backend）

| 状态后端 | 存储位置 | 适用场景 |
|---------|---------|---------|
| **HashMapStateBackend** | JVM Heap（内存） | 小状态、低延迟测试环境 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘） | 大状态的生产环境 |
| **HeavyBytesStateBackend** | 远程状态存储 | （已废弃） |

**RocksDB 优势**：
- 增量 Checkpoint 支持（只写入增量变更）
- 状态大小不受 JVM 堆内存限制
- 适合 TB 级别的超大状态

### 状态清理策略

```java
// 窗口使用增量快照 + 定时清理
.window(TumblingEventTimeWindows.of(Time.minutes(5)))
.trigger(ContinuousEventTimeTrigger.of(Time.minutes(1))) // 每分钟触发一次
.evictor(TimeEvictor.of(0, true))  // 清理窗口外的数据
```

### 状态管理在故障恢复中的作用

Flink 的 **精确一次语义** 依赖状态管理：
1. Checkpoint 时将所有算子的状态快照持久化
2. 故障恢复时，从最近的 Checkpoint 恢复所有算子状态
3. 重新消费 Kafka 的 offset（配合 Kafka 事务）实现端到端精确一次

### 使用状态的最佳实践

1. **避免大状态**：合理设计 keyBy，避免状态过大
2. **及时清理状态**：使用 `processElement` 和 `onTimer` 回调清理无用状态
3. **选择合适的状态后端**：生产环境优先 RocksDB
4. **监控状态大小**：通过 Flink UI 监控状态，防止 OOM
