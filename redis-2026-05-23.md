## 题目一：Redis 数据类型以及使用场景分别是什么？

### 参考答案

Redis 提供了丰富的数据类型，常见的有五种：**String（字符串）、Hash（哈希）、List（列表）、Set（集合）、Zset（有序集合）**。

随着 Redis 版本的更新，后面又支持了四种数据类型：**BitMap（2.2 版新增）、HyperLogLog（2.8 版新增）、GEO（3.2 版新增）、Stream（5.0 版新增）**。

### 五种数据类型的应用场景

| 类型 | 应用场景 |
|------|----------|
| **String** | 缓存对象、常规计数、分布式锁、共享 session 信息等 |
| **List** | 消息队列（需要自行实现全局唯一 ID，不能以消费组形式消费） |
| **Hash** | 缓存对象、购物车等 |
| **Set** | 聚合计算（并集、交集、差集）场景，比如点赞、共同关注、抽奖活动等 |
| **Zset** | 排序场景，比如排行榜、电话和姓名排序等 |

### 四种扩展数据类型的应用场景

| 类型 | 应用场景 |
|------|----------|
| **BitMap** | 二值状态统计的场景，比如签到、判断用户登陆状态、连续签到用户总数等 |
| **HyperLogLog** | 海量数据基数统计的场景，比如百万级网页 UV 计数等 |
| **GEO** | 存储地理位置信息的场景，比如滴滴叫车 |
| **Stream** | 消息队列，相比于基于 List 类型实现的消息队列，有这两个特有的特性：自动生成全局唯一消息 ID，支持以消费组形式消费数据 |

## 题目二：Redis 持久化有哪些方式？RDB 和 AOF 有什么区别？

### 参考答案

Redis 共有三种数据持久化的方式：

### 1. RDB 快照

将某一时刻的内存数据，以二进制的方式写入磁盘。Redis 提供了两个命令来生成 RDB 文件：
- **save**：在主线程生成 RDB 文件，会阻塞主线程
- **bgsave**：创建子进程来生成 RDB 文件，避免阻塞

Redis 可以通过配置文件实现每隔一段时间自动执行 bgsave，例如：
- 900 秒之内，对数据库进行了至少 1 次修改
- 300 秒之内，对数据库进行了至少 10 次修改
- 60 秒之内，对数据库进行了至少 10000 次修改

RDB 的优点是数据恢复速度快，缺点是快照频率不好把握，容易丢失数据。

### 2. AOF 日志

每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里。Redis 重启时，会读取 AOF 文件记录的命令，然后逐一执行来恢复数据。

AOF 写回策略有三种：

| 策略 | 说明 | 优缺点 |
|------|------|--------|
| **Always** | 每次写操作命令执行完后，同步将 AOF 日志数据写回硬盘 | 数据不会丢失，但开销大 |
| **Everysec** | 每次写操作命令执行完后，先将命令写入到 AOF 文件的内核缓冲区，然后每隔一秒将缓冲区里的内容写回到硬盘 | 性能适中，最多丢失 1 秒数据 |
| **No** | 不由 Redis 控制写回硬盘的时机，转交给操作系统控制 | 性能最好，但数据丢失不确定 |

### 3. 混合持久化（Redis 4.0+）

结合了 RDB 和 AOF 的优点。在 AOF 重写时，先将以 RDB 格式写入 AOF 文件，然后增量命令以 AOF 格式写入。这样既保证了启动速度，又降低数据丢失风险。

### RDB vs AOF 对比

| 特性 | RDB | AOF |
|------|-----|-----|
| 数据恢复速度 | 快 | 慢 |
| 数据完整性 | 较差（取决于快照频率） | 较好 |
| 文件大小 | 小（二进制） | 大（命令日志） |
| 性能影响 | bgsave 时可能有延迟 | 取决于写回策略 |
| 兼容性 | 跨版本兼容 | Redis 4.0+ 混合模式旧版本不兼容 |

## 题目三：Redis 是单线程吗？为什么单线程还这么快？

### 参考答案

**Redis 单线程指的是「接收客户端请求 -> 解析请求 -> 进行数据读写等操作 -> 发送数据给客户端」这个过程是由一个线程（主线程）来完成的**，这也是我们常说 Redis 是单线程的原因。

但是，Redis 程序并不是完全单线程的，Redis 在启动的时候会启动后台线程（BIO）：

- **Redis 2.6 版本**：启动 2 个后台线程，分别处理关闭文件、AOF 刷盘这两个任务
- **Redis 4.0 版本之后**：新增了 lazyfree 线程，用来异步释放 Redis 内存

### 为什么单线程的 Redis 这么快？

1. **内存操作**：Redis 的大部分操作都在内存中完成，并且采用了高效的数据结构（SDS、跳表等），因此 Redis 瓶颈可能是机器的内存或者网络带宽，而并非 CPU

2. **无锁开销**：采用单线程模型避免了多线程之间的竞争，省去了多线程切换带来的时间和性能上的开销，也不会导致死锁问题

3. **I/O 多路复用**：Redis 采用了 I/O 多路复用机制（select/epoll），允许内核中同时存在多个监听 Socket 和已连接 Socket，内核会一直监听这些 Socket 上的连接请求或数据请求，一旦有请求到达，就会交给 Redis 线程处理，这就实现了一个 Redis 线程处理多个 IO 流的效果

### Redis 6.0 之后的多线程改进

Redis 6.0 版本引入了多线程 I/O 来处理网络请求（默认只针对发送响应数据 write client socket），但对于命令的执行仍然使用单线程。所以 Redis 6.0 并不是有多线程同时执行命令，而是分摊网络 I/O 的压力。

## 题目四：Redis 如何实现服务高可用？

### 参考答案

Redis 实现高可用主要有三种方案：**主从复制、哨兵模式、切片集群**。

### 1. 主从复制

实现一主多从的模式：
- 主服务器进行读写操作
- 从服务器只读，并接受主服务器同步过来的写操作命令
- 主从服务器之间的命令复制是异步进行的

主从复制的优点是数据冗余、读写分离，但无法自动故障恢复。

### 2. 哨兵模式

Redis 在 2.8 版本以后提供了哨兵（Sentinel）机制，用于实现主从节点故障转移。哨兵一般以集群方式部署，至少需要 3 个哨兵节点。

**哨兵集群主要负责三件事情**：
- **监控**：哨兵节点每隔 1 秒给所有主从节点发送 PING 命令，检测节点是否存活
- **选主**：当主节点不可用时，选举一个从节点升级为主节点
- **通知**：将新主节点的信息通过发布/订阅机制通知给客户端

**故障转移过程**：
1. 挑选新主节点：过滤已下线从节点 → 过滤网络状态不好的从节点 → 按优先级、复制进度、ID 号三轮排序
2. 将其他从节点指向新主节点
3. 通过发布/订阅机制通知客户端
4. 旧主节点重新上线时，设置为新主节点的从节点

**quorum 配置建议**：设置为哨兵个数的二分之一加 1，例如 3 个哨兵设置为 2，且哨兵节点数量应为奇数。

### 3. 切片集群（Redis Cluster）

Redis Cluster 是 Redis 原生的分布式解决方案，通过数据分片（slot）来分布数据，每个节点负责一部分数据，支持自动故障转移。

## 题目五：为什么用 Redis 作为 MySQL 的缓存？Redis 和 Memcached 有什么区别？

### 参考答案

### 为什么用 Redis 作为 MySQL 的缓存？

主要是因为 Redis 具备「高性能」和「高并发」两种特性：

**高性能**：用户第一次访问 MySQL 中的某些数据时较慢（从硬盘读取），将数据缓存在 Redis 后，下一次访问就可以直接从缓存中获取，操作缓存就是直接操作内存，速度相当快。

**高并发**：单台设备的 Redis 的 QPS（每秒处理完请求的次数）是 MySQL 的 10 倍，Redis 单机的 QPS 能轻松破 10w，而 MySQL 单机的 QPS 很难破 1w。

### Redis 与 Memcached 对比

| 特性 | Redis | Memcached |
|------|-------|-----------|
| **数据类型** | 丰富（String、Hash、List、Set、ZSet 等） | 只支持简单的 key-value 类型 |
| **持久化** | 支持 RDB、AOF 混合持久化 | 不支持，数据全部存在内存 |
| **集群模式** | 原生支持主从、哨兵、Cluster | 没有原生集群，需要客户端实现分片 |
| **高级特性** | 支持发布订阅、Lua 脚本、事务等 | 不支持 |
| **数据大小** | Value 最大 512MB | 最大 1MB |
| **内存管理** | 自己实现内存管理（zmalloc） | Slab Allocation |

### 总结

选择 Redis 作为缓存的主要原因：
1. 丰富的数据类型能支持更复杂的业务场景
2. 支持数据持久化，重启后可恢复
3. 原生集群支持，易于扩展
4. 具备高并发高性能特性，能承担更大的访问压力
