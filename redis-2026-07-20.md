## 题目一：Redis 数据类型有哪些？各自的底层实现是什么？

### 参考答案

Redis 提供了丰富的数据类型，主要分为**五大基础数据类型**和**四种扩展数据类型**。

#### 五大基础数据类型

| 类型 | 使用场景 |
|------|----------|
| **String** | 缓存对象、常规计数、分布式锁、共享 session 信息 |
| **Hash** | 缓存对象、购物车 |
| **List** | 消息队列（存在无法以消费组消费等问题） |
| **Set** | 聚合计算（并集、交集、差集），如点赞、共同关注、抽奖 |
| **Zset** | 排序场景，如排行榜、电话姓名排序 |

#### 四种扩展数据类型

| 类型 | 版本 | 使用场景 |
|------|------|----------|
| **BitMap** | 2.2+ | 二值状态统计（签到、判断登录状态、连续签到总数） |
| **HyperLogLog** | 2.8+ | 海量数据基数统计（百万级网页 UV） |
| **GEO** | 3.2+ | 地理信息存储（滴滴叫车等） |
| **Stream** | 5.0+ | 消息队列（支持消费组、自动生成全局唯一 ID） |

#### 底层数据结构实现

- **String**：底层基于 SDS（简单动态字符串），支持二进制安全、O(1) 获取长度、避免缓冲区溢出。
- **List**：Redis 3.2 之前由双向链表或压缩列表实现；3.2 之后统一由 quicklist 实现。
- **Hash**：小数据量时用压缩列表，大数据量时用哈希表。Redis 7.0 废弃压缩列表，改用 listpack。
- **Set**：元素为整数且小于 512 个时用整数集合，否则用哈希表。
- **Zset**：元素小于 128 个且每个小于 64 字节时用压缩列表，否则用跳表（Skip List）。

## 题目二：Redis 持久化机制有哪些？AOF 和 RDB 各自有什么优缺点？

### 参考答案

Redis 共有三种持久化方式：**AOF 日志**、**RDB 快照**、**混合持久化**。

#### AOF 日志（Append Only File）

- **原理**：每执行一条写操作命令，就把该命令以追加方式写入一个文件。Redis 重启时读取该文件逐一执行命令来恢复数据。
- **写回策略**（由 `appendfsync` 配置）：

| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **Always** | 每次执行完命令后同步写回 | 数据不丢失 | 性能差 |
| **Everysec** | 每次写命令后先写缓冲区，每秒刷盘 | 性能与安全性平衡 | 可能丢失 1 秒数据 |
| **No** | 由操作系统决定何时刷盘 | 性能最好 | 数据丢失风险大 |

- **AOF 重写**：当 AOF 文件过大时，后台子进程 `bgrewriteaof` 读取当前数据库所有键值对，用一条命令记录每个键值对，实现文件压缩。

#### RDB 快照

- **原理**：将某一时刻的内存数据以二进制方式写入磁盘。记录的是实际数据，而非命令。
- **命令**：`save`（主线程执行，会阻塞）、`bgsave`（子进程执行，不阻塞）。
- **配置自动触发**：如 `save 900 1` 表示 900 秒内至少 1 次修改则触发 bgsave。
- **写时复制（COW）**：bgsave 期间主线程可继续处理请求，写操作会复制数据副本，不影响 RDB 写入。

#### 混合持久化（Redis 4.0+）

- **原理**：AOF 重写时，先将以 RDB 格式写入 AOF 文件前半部分，再将增量命令以 AOF 格式写入后半部分。
- **优点**：重启恢复速度快（RDB 部分快速加载）+ 丢失数据少（AOF 增量部分）。
- **缺点**：AOF 可读性差；兼容性差，无法用于 4.0 之前版本。

#### 对比总结

| 特性 | RDB | AOF | 混合持久化 |
|------|-----|-----|-----------|
| 文件格式 | 二进制 | 命令日志 | RDB+AOF |
| 恢复速度 | 快 | 慢 | 快 |
| 数据完整性 | 可能丢失较多 | 可配置（always 最完整） | 丢失少量 |
| 快照频率 | 难以把握（太频繁影响性能，太少丢数据多） | 依赖刷盘策略 | 适中 |

## 题目三：Redis 过期删除策略和内存淘汰策略有什么区别？

### 参考答案

#### 过期删除策略

Redis 对已设置过期时间的 key 采用**惰性删除 + 定期删除**配合使用：

1. **惰性删除**：每次访问 key 时检查是否过期，过期则删除。优点是对 CPU 友好，缺点是过期 key 长期不被访问会浪费内存。
2. **定期删除**：每隔一段时间随机从过期字典中抽取 20 个 key 检查，删除已过期的。如果已过期 key 超过 25%，继续抽取。为防止循环耗时，单次执行上限为 25ms。

#### 内存淘汰策略

当 Redis 运行内存达到 `maxmemory` 时，触发内存淘汰机制，共八种策略：

**不淘汰（noeviction，默认）：** 内存满时直接返回错误，不服务新请求。

**在设置了过期时间的数据中淘汰：**

| 策略 | 规则 |
|------|------|
| volatile-ttl | 优先淘汰更早过期的 key |
| volatile-random | 随机淘汰任意设置了过期时间的 key |
| volatile-lru | 淘汰所有过期 key 中最久未使用的（Redis 3.0 前默认） |
| volatile-lfu | 淘汰所有过期 key 中最少使用的（Redis 4.0+） |

**在所有数据范围内淘汰：**

| 策略 | 规则 |
|------|------|
| allkeys-random | 随机淘汰任意 key |
| allkeys-lru | 淘汰整个键值中最久未使用的 |
| allkeys-lfu | 淘汰整个键值中最少使用的（Redis 4.0+） |

#### LRU vs LFU

- **LRU（Least Recently Used）**：淘汰最近最少使用的数据。Redis 使用近似 LRU 算法（随机采样 5 个，淘汰最久未用的），避免维护完整链表的性能开销，但无法解决缓存污染问题。
- **LFU（Least Frequently Used）**：淘汰访问频次最低的数据。Redis 对象头中 24bit 的 lru 字段在高 16bit 存时间戳、低 8bit 存访问计数器，能更好地解决一次性数据占用缓存的问题。

## 题目四：Redis 如何实现高可用？主从复制、哨兵、Cluster 有什么区别？集群脑裂如何处理？

### 参考答案

#### 高可用方案

1. **主从复制（一主多从）**
   - 主节点处理读写操作，从节点只读并同步主节点写操作命令。
   - 采用读写分离，主从命令复制是异步的，**无法保证强一致性**。
   - 数据同步分为全量同步（RDB）和增量同步（命令传播）。

2. **哨兵模式（Sentinel）**
   - 解决主从模式下故障需手动恢复的问题。
   - 哨兵监控主从节点状态，进行**自动故障转移**：主观下线 → 客观下线 → 选举新主节点 → 旧主降级。

3. **切片集群（Redis Cluster）**
   - 解决单节点内存无法承载全部数据的问题。
   - 采用 **16384 个哈希槽**（Hash Slot），通过 `CRC16(key) % 16384` 确定数据归属槽位，槽位再映射到具体节点。
   - 支持自动数据分片和部分节点故障时的自动迁移。

#### 集群脑裂问题

**什么是脑裂**：主节点网络故障但与客户端网络正常，客户端继续写入数据到主节点，同时哨兵选举出新主节点，集群出现两个主节点。网络恢复后，旧主被降级为从节点，其数据因全量同步被清空，导致之前写入的数据丢失。

**解决方案**：配置以下两个参数：

```properties
min-slaves-to-write 1      # 主节点至少要有 1 个从节点连接
min-slaves-max-lag 10      # 主从复制延迟不能超过 10 秒
```

满足条件：主库至少连接 1 个从库，且 ACK 延迟 ≤ 10 秒，否则主节点拒绝接收客户端写请求。即使原主假故障，也无法响应心跳和完成 ACK，主库就会被限制写请求，避免脑裂。

## 题目五：什么是缓存雪崩、击穿、穿透？如何解决？

### 参考答案

#### 缓存雪崩

**定义**：大量缓存数据在同一时间过期失效，此时大量请求无法命中缓存，全部涌入数据库，导致数据库压力骤增甚至宕机。

**解决方案**：
- 将缓存失效时间**随机打散**（基础过期时间 + 随机 1~10 分钟），降低集体失效概率。
- 设置缓存**永不过期**，通过后台异步更新缓存数据。

#### 缓存击穿

**定义**：某个热点数据过期时，大量请求并发访问该数据，直接击穿缓存层打到数据库。

**解决方案**：
- 使用**互斥锁**（如 `SETNX`），保证同一时间只有一个线程重建缓存。
- 热点数据**不设置过期时间**，由后台异步更新，或在过期前主动刷新。

#### 缓存穿透

**定义**：请求的数据既不在缓存中，也不在数据库中（如业务误删、恶意攻击），导致请求每次都打到数据库。

**常见原因**：业务误操作删除数据、黑客恶意大量访问不存在的数据。

**解决方案**：
- **非法请求限制**：API 入口校验参数合理性，拦截恶意请求。
- **设置空值/默认值**：缓存层写入空值或默认值（如 `null`），避免频繁查库。
- **布隆过滤器（BloomFilter）**：写入数据库时在布隆过滤器做标记，请求到达时先查布隆过滤器快速判断数据是否存在，大量无效请求只查过滤器和 Redis 而不打到数据库。

#### 三种问题对比

| 问题 | 原因 | 核心解决思路 |
|------|------|-------------|
| 缓存雪崩 | 大量 key 同时过期 | 随机过期时间、永久缓存 |
| 缓存击穿 | 热点 key 过期 | 互斥锁、不过期策略 |
| 缓存穿透 | 数据本就不存在 | 过滤空值、布隆过滤器 |
