## 题目1：HashMap 的底层数据结构是什么？JDK1.8 相比 1.7 有什么优化？

【答案】

**底层数据结构：**

JDK 1.8 之前，HashMap 由**数组 + 链表**组成（链表散列）。数组是 HashMap 的主体，链表是为了解决哈希冲突而存在的（"拉链法"解决冲突）。

JDK 1.8 及之后，当链表长度大于等于阈值（默认为 8）且数组长度大于等于 64 时，会将链表转化为**红黑树**，以减少搜索时间。如果数组长度小于 64，则优先进行数组扩容。

**核心属性：**

- `DEFAULT_INITIAL_CAPACITY = 16`：默认初始化容量
- `DEFAULT_LOAD_FACTOR = 0.75f`：默认负载因子
- `TREEIFY_THRESHOLD = 8`：链表转红黑树的阈值
- `UNTREEIFY_THRESHOLD = 6`：红黑树转链表的阈值
- `MIN_TREEIFY_CAPACITY = 64`：链表转红黑树的最小数组容量

**扰动函数：**

HashMap 的 hash 方法（扰动函数）用于防止一些实现比较差的 hashCode() 方法，减少碰撞。JDK 1.8 的 hash 方法源码：

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

相比 JDK 1.7 的扰动了 4 次，JDK 1.8 的 hash 方法更加简化。

**put 元素时如何定位桶位置：**

通过 `(n - 1) & hash` 判断当前元素存放的位置，其中 n 是数组长度。

## 题目2：ConcurrentHashMap 在 Java 7 和 Java 8 中的实现有什么区别？

【答案】

**Java 7 的实现：**

- 使用**Segment 数组 + HashEntry 数组 + 链表**的结构。
- 每一个 Segment 是一个类似于 HashMap 的结构，可以单独扩容。
- Segment 的个数默认是 16 个，通过 `ReentrantLock` 实现分段锁。
- 默认支持最多 16 个线程并发。

**Java 8 的实现：**

- 使用**Node 数组 + 链表/红黑树**的结构。
- 摒弃了 Segment，采用 `synchronized` 锁 + CAS 机制实现线程安全。
- 当链表长度达到 `TREEIFY_THRESHOLD(8)` 时转化为红黑树。
- 使用 `sizeCtl` 控制数组初始化和扩容：
  - -1 表示正在初始化
  - -N 表示正在进行扩容，高 16 位是扩容标识戳，低 16 位是正在扩容的线程数
  - 0 表示未初始化
  - >0 表示扩容阈值

**put 流程：**

1. 根据 key 计算 hash 值
2. 判断是否需要初始化
3. 通过 CAS 尝试写入
4. 如果 hash == MOVED(-1)，说明正在扩容，协助扩容
5. 否则使用 synchronized 加锁写入数据
6. 如果数量大于阈值则树化

## 题目3：Spring Boot 自动装配的原理是什么？

【答案】

**什么是自动装配：**

Spring Boot 定义了一套接口规范，在启动时会扫描外部引用 jar 包中的 `META-INF/spring.factories` 文件（Spring Boot 3.0 改为 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`），将文件中配置的类型信息加载到 Spring 容器，并执行类中定义的各种操作。

**核心注解：**

`@SpringBootApplication` 是核心注解，实际上是 `@Configuration`、`@EnableAutoConfiguration`、`@ComponentScan` 的组合。

`@EnableAutoConfiguration` 是实现自动装配的核心注解，通过 `@Import(AutoConfigurationImportSelector.class)` 加载自动装配类。

**自动装配流程：**

1. `AutoConfigurationImportSelector` 实现 `ImportSelector` 接口
2. `selectImports()` 方法调用 `getAutoConfigurationEntry()`
3. 从 `META-INF/spring.factories` 获取所有候选配置类
4. 通过 `@ConditionalOnXXX` 条件注解按需加载

**条件注解包括：**

- `@ConditionalOnClass`：当类路径下有指定类时
- `@ConditionalOnMissingClass`：当类路径下没有指定类时
- `@ConditionalOnBean`/`@ConditionalOnMissingBean`：容器中是否存在指定 Bean
- `@ConditionalOnProperty`：指定的属性是否有指定的值

## 题目4：RabbitMQ 有哪些核心概念？消息是如何路由的？

【答案】

**核心概念：**

- **Producer（生产者）**：发送消息的应用
- **Consumer（消费者）**：接收消息的应用
- **Broker（代理）**：RabbitMQ 服务节点
- **Queue（队列）**：存储消息的缓冲区
- **Exchange（交换机）**：接收生产者消息，根据规则路由到队列
- **Binding（绑定）**：Exchange 和 Queue 之间的关联关系
- **Routing Key（路由键）**：用于 Exchange 判断哪些队列接收消息

**Exchange 类型：**

1. **Direct Exchange**：完全匹配 Routing Key，精确路由
2. **Fanout Exchange**：广播模式，忽略 Routing Key，发送到所有绑定的队列
3. **Topic Exchange**：通配符匹配，支持 `*`（匹配一个单词）和 `#`（匹配零个或多个单词）
4. **Headers Exchange**：根据消息头部的属性进行匹配（较少使用）

**消息流转流程：**

1. 生产者将消息发送到 Exchange
2. Exchange 根据类型和 Binding 规则将消息路由到 Queue
3. 消费者从 Queue 取出消息进行消费
4. 多个消费者可以订阅同一个队列，消息会被轮询分发

**与 Kafka 的区别：**

- RabbitMQ 是消息代理，Kafka 是分布式流处理平台
- RabbitMQ 支持多种 Exchange，Kafka 是基于 Topic 和 Partition 的
- RabbitMQ 消费模式是 push，Kafka 是 pull
- Kafka 支持消息回溯，RabbitMQ 支持消息确认和重试

## 题目5：Kafka 为什么能做到高吞吐量？它的核心架构是怎样的？

【答案】

**高吞吐量的原因：**

1. **顺序写磁盘**：Kafka 采用顺序写，比随机写性能高很多
2. **页缓存（Page Cache）**：利用操作系统的页缓存，避免 JVM 垃圾回收开销
3. **零拷贝（Zero Copy）**：使用 Linux 的 sendfile 系统调用，数据直接从磁盘拷贝到网卡，无需经过应用层
4. **批量处理**：消息可以批量发送和压缩，减少网络 IO
5. **分区（Partition）**：支持水平扩展，多个分区并行处理
6. **高效的数据结构**：使用稀疏索引和顺序读取

**核心架构：**

- **Topic（主题）**：消息的分类，每个 Topic 包含多个 Partition
- **Partition（分区）**：物理上的存储单位，每个 Partition 内的消息是有序的
- **Replica（副本）**：每个 Partition 可以有多个副本，实现数据冗余和高可用
- **Leader 和 Follower**：读写操作通过 Leader 完成，Follower 同步 Leader 数据
- **Producer**：生产者，支持负载均衡和 ACK 确认机制
- **Consumer**：消费者，通过 Consumer Group 消费，一个 Partition 只能被一个 Consumer Group 内的一个 Consumer 消费
- **Consumer Group**：消费者组，实现消息的负载均衡和顺序消费

**ACK 机制：**

- `acks=0`：Producer 不等待确认，可能丢失数据
- `acks=1`：等待 Leader 确认
- `acks=all`：等待 Leader 和所有 Follower 确认

## 题目6：RAG（检索增强生成）是什么？它的工作流程是怎样的？

【答案】

**RAG 概念：**

RAG（Retrieval-Augmented Generation，检索增强生成）是一种结合检索系统和生成模型的技术。它通过从外部知识库中检索相关文档，来增强大语言模型（LLM）的回答质量和准确性。

**为什么需要 RAG：**

1. LLM 的知识有截止日期，无法获取最新信息
2. LLM 可能产生"幻觉"，生成不准确的内容
3. 企业私有知识无法直接用于 LLM
4. RAG 可以提供可解释性，答案有据可查

**工作流程：**

1. **文档处理（Document Processing）**：
   - 对原始文档进行分块（Chunking）
   - 常用策略：固定长度分块、重叠分块、基于语义的分块

2. **向量化（Embedding）**：
   - 使用 Embedding 模型将文本块转换为向量
   - 存储到向量数据库（如 Milvus、Pinecone、Chroma）

3. **检索（Retrieval）**：
   - 将用户问题转换为向量
   - 在向量数据库中检索最相关的 Top-K 个文档块

4. **增强（Augmentation）**：
   - 将检索到的相关文档与用户问题组合成提示词（Prompt）

5. **生成（Generation）**：
   - LLM 基于增强后的 Prompt 生成回答

**关键组件：**

- **Embedding 模型**：BGE、Text2Vec、OpenAI Embeddings 等
- **向量数据库**：存储和检索向量
- **重排序（Re-ranker）**：对检索结果进一步排序（可选）

## 题目7：Spring Cloud Alibaba Nacos 的服务发现和配置管理原理是什么？

【答案】

**服务发现原理：**

1. **服务注册**：
   - 服务提供者启动时向 Nacos Server 注册自己的信息
   - 包括 IP、端口、服务名、元数据等
   - 通过心跳机制维持活性，默认每 5 秒发送一次心跳

2. **服务同步**：
   - Nacos Server 集群之间通过共享存储（如 MySQL）或异步复制同步服务信息
   - 支持临时实例（ephemeral）和持久实例（persistent）

3. **服务查询**：
   - 服务消费者从 Nacos Server 获取服务提供者列表
   - 本地缓存 + 定时更新机制
   - 支持加权负载均衡

**配置管理原理：**

1. **配置存储**：
   - 配置存储在 Nacos Server，支持 MySQL 等数据库持久化
   - 配置以 (dataId, group, namespace) 唯一标识

2. **配置推送**：
   - Nacos 采用长轮询机制推送配置变更
   - 客户端定时（如每 30 秒）检查配置更新
   - 服务端有变更时立即响应

3. **配置变更通知**：
   - 通过 `Listener` 机制通知客户端
   - 客户端执行回调处理配置更新

**Nacos 与 Eureka 的区别：**

| 特性 | Nacos | Eureka |
|------|-------|--------|
| 一致性协议 | CP/AP 可切换 | AP |
| 配置管理 | 原生支持 | 需要结合 Config |
| 多环境 | namespace + group | - |
| 健康检查 | TCP/HTTP/MySQL | 心跳 |

## 题目8：算法题 - 两数之和

【题目】给定一个整数数组 `nums` 和一个整数 `target`，找出数组中两个数之和等于 `target` 的索引。

**示例：**
```
输入: nums = [2, 7, 11, 15], target = 9
输出: [0, 1]
解释: nums[0] + nums[1] = 2 + 7 = 9
```

【答案】

**方法一：暴力枚举 O(n²)**

```java
public int[] twoSum(int[] nums, int target) {
    for (int i = 0; i < nums.length; i++) {
        for (int j = i + 1; j < nums.length; j++) {
            if (nums[i] + nums[j] == target) {
                return new int[]{i, j};
            }
        }
    }
    return new int[]{};
}
```

**方法二：哈希表 O(n) - 最优解**

```java
public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> map = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (map.containsKey(complement)) {
            return new int[]{map.get(complement), i};
        }
        map.put(nums[i], i);
    }
    return new int[]{};
}
```

**思路分析：**

- 遍历数组，对于每个元素 `nums[i]`，计算 `complement = target - nums[i]`
- 在哈希表中查找是否存在 `complement`
- 如果存在，返回两个索引；如果不存在，将当前元素存入哈希表
- 时间复杂度从 O(n²) 优化到 O(n)，空间复杂度 O(n)

## 题目9：算法题 - 有效的括号

【题目】给定一个只包含 `'('`, `')'`, `'{'`, `'}'`, `'['`, `']'` 的字符串，判断括号是否有效。

**示例：**
```
输入: "()[]{}"  输出: true
输入: "(]"       输出: false
输入: "([)]"     输出: false
```

【答案】

**方法：栈 O(n)**

```java
public boolean isValid(String s) {
    if (s.length() % 2 != 0) return false;
    
    Stack<Character> stack = new Stack<>();
    Map<Character, Character> mapping = new HashMap<>();
    mapping.put(')', '(');
    mapping.put('}', '{');
    mapping.put(']', '[');
    
    for (char c : s.toCharArray()) {
        if (mapping.containsKey(c)) {
            // 是右括号
            char top = stack.isEmpty() ? '#' : stack.pop();
            if (top != mapping.get(c)) {
                return false;
            }
        } else {
            // 左括号入栈
            stack.push(c);
        }
    }
    return stack.isEmpty();
}
```

**思路分析：**

- 遍历字符串中的每个字符
- 遇到左括号 `(`、`{`、`[` 时，入栈
- 遇到右括号 `)`、`}`、`]` 时，检查栈顶元素是否匹配
- 如果匹配，栈顶元素出栈；不匹配返回 false
- 最终栈为空表示所有括号都匹配

**时间复杂度：O(n)**，**空间复杂度：O(n)**

## 题目10：LinkedList 和 ArrayList 的区别是什么？如何选择使用？

【答案】

**底层数据结构：**

- **ArrayList**：基于动态数组实现，支持随机访问
- **LinkedList**：基于双向链表实现，不支持随机访问

**时间复杂度：**

| 操作 | ArrayList | LinkedList |
|------|-----------|------------|
| 按索引访问 | O(1) | O(n) |
| 头部插入/删除 | O(n) | O(1) |
| 尾部插入/删除 | O(1) amortized | O(1) |
| 指定位置插入/删除 | O(n) | O(n) 平均 |

**为什么 LinkedList 不能实现 RandomAccess 接口：**

`RandomAccess` 是标记接口，表明支持随机访问。LinkedList 底层是链表，内存地址不连续，只能通过指针遍历，不支持随机快速访问，所以不能实现该接口。

**LinkedList 的插入删除时间复杂度：**

- 头部插入/删除：O(1)
- 尾部插入/删除：O(1)
- 指定位置插入/删除：O(n)，需要先移动到指定位置

**如何选择：**

- **ArrayList**：适合频繁随机访问、尾部操作的场景
- **LinkedList**：适合频繁在头部或中间插入删除、作为栈或队列使用

**实际项目中：**

LinkedList 作者 Josh Bloch 自己都说从来不会使用 LinkedList。因为即使在头尾操作，ArrayList 的尾部 O(1) amortized 也足够好，而 LinkedList 的节点开销大、CPU 缓存不友好。几乎所有场景都可以用 ArrayList 代替。
