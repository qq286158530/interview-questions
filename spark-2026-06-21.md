# Spark 高频面试题（2026-06-21）

## 题目一：Spark 架构与核心组件

**问题：请描述 Apache Spark 的整体架构，以及 Driver、Executor、Master、Worker 的作用是什么？**

### 答案

Spark 是一个分布式计算框架，采用 **Master-Slave** 架构，主要包含以下核心组件：

| 组件 | 作用 |
|------|------|
| **Driver（驱动程序）** | 负责运行用户代码，创建 SparkContext，将 Job 划分为多个 Stage 和 Task，协调整个应用程序的执行 |
| **Executor（执行器）** | 运行在 Worker 节点上的进程，负责执行分配给它的 Task，向 Driver 汇报计算结果 |
| **Master** | 集群管理器，负责资源分配和调度，接受提交的应用，申请 Executor 资源 |
| **Worker** | 运行在集群节点上的进程，负责管理本节点的资源，接收 Master 的命令启动 Executor |

### Spark 执行流程

1. 用户提交 Application，Driver 创建 SparkContext
2. SparkContext 向 Cluster Manager 申请资源
3. Cluster Manager 在 Workers 上启动 Executors
4. SparkContext 发送 Task 到 Executors 执行
5. Task 执行结果返回给 Driver
6. Application 结束，SparkContext 关闭

### 重要补充

- **SparkContext** 是 Spark 应用的入口点，负责与 Cluster Manager 通信
- **DAGScheduler** 将 Job 划分为 Stage（Shuffle 边界划分）
- **TaskScheduler** 将 Stage 内的 Task 分发到 Executor 执行

## 题目二：RDD 特性与惰性机制

**问题：什么是 RDD？Spark 为什么选择 RDD 作为核心数据结构？RDD 的惰性计算是什么意思？**

### 答案

**RDD（Resilient Distributed Dataset）** 是 Spark 提供的一种容错、可并行计算的数据集合，是 Spark 最基本的数据抽象。

### RDD 的五大特性

1. **分区列表（Partitions）**：RDD 由多个分区组成，每个分区是数据的一个逻辑片段
2. **计算函数（Compute Function）**：每个分区都有计算函数（iterator）
3. **依赖关系（Dependencies）**：RDD 之间存在依赖关系（窄依赖/宽依赖）
4. **分区器（Partitioner）**：可选，用于 Key-Value RDD 的分区控制
5. **位置优先（Preferred Locations）**：可选，数据块的位置偏好

### 为什么选择 RDD

| 特性 | 优势 |
|------|------|
| **内存计算** | 中间结果缓存在内存，减少磁盘 IO |
| **容错机制** | 通过 lineage（血统）恢复丢失分区，无需复制数据 |
| **不可变性** | 线程安全，支持并发操作 |
| **统一的抽象** | 覆盖批处理、流处理、SQL 等多种场景 |

### 惰性计算（Lazy Evaluation）

```
 transformations（转换操作） → 记录操作轨迹，构建 DAG
 actions（行动操作）     → 触发真正的计算执行
```

**示例：**

```scala
// 转换操作 - 惰性，不执行
val rdd1 = sc.textFile("data.txt")           // 不执行
val rdd2 = rdd1.map(line => line.split(","))  // 不执行
val rdd3 = rdd2.filter(fields => fields(0) > 10) // 不执行

// 行动操作 - 触发计算
val count = rdd3.count()  // 此时才真正执行
```

**好处：**
- 减少中间结果的物化，节省内存
- 优化执行计划，减少数据移动
- 支持管道化执行，提高效率

## 题目三：Shuffle 机制与优化

**问题：什么是 Shuffle？为什么 Spark 中 Shuffle 是性能瓶颈？如何优化 Shuffle 性能？**

### 答案

**Shuffle** 是 Spark 中将数据重新分布的过程，当一个 RDD 的某个分区需要依赖前一个 RDD 的所有分区时（如 `groupByKey`、`join`），就需要跨节点移动数据。

### Shuffle 执行过程

```
Map 阶段          Shuffle Write         Shuffle Read          Reduce 阶段
  │                    │                      │                     │
  │ 写入本地磁盘        │ 网络传输              │ 读取远程数据         │
  │ (partitioned file) │ ──────────────────>   │ (aggregated buffer) │
```

### Shuffle 为什么是瓶颈

| 问题 | 原因 |
|------|------|
| **磁盘 IO** | Shuffle 数据写入本地磁盘 |
| **网络 IO** | 数据跨节点传输 |
| **序列化** | 数据需要序列化/反序列化 |
| **内存压力** | Map 端缓冲、Reduce 端聚合都消耗内存 |
| **磁盘压力** | 大量小文件随机读写 |

### Shuffle 优化策略

#### 1. 减少 Shuffle 数据量

```scala
// 避免使用 groupByKey，用 reduceByKey/aggregateByKey 替代
rdd.groupByKey()           // ❌ 内存压力大
rdd.reduceByKey(_ + _)     // ✅ 先局部聚合，减少传输
```

#### 2. 合理设置分区数

```scala
// 分区数 = max(并发任务数, 数据量 / 每个分区理想大小)
// 一般每个分区 128-256MB 为宜
rdd.repartition(100)  // 增大分区（触发 Shuffle）
rdd.coalesce(10)      // 减少分区（不触发 Shuffle，只能减少）
```

#### 3. 配置参数调优

```properties
spark.shuffle.file.buffer 32k     # Map 端写文件的缓冲区大小
spark.shuffle.manager sort        # 使用 SortShuffleManager
spark.shuffle.sort.bypassMergeThreshold 200  # 超过此分区数绕过合并
```

#### 4. 使用广播变量

```scala
// 将小表广播，避免大表 Shuffle
val broadcastVar = sc.broadcast(smallRdd.collectAsMap())
val result = largeRdd.mapPartitions(iter => {
  val map = broadcastVar.value
  iter.map(kv => (kv._1, kv._2 + map.getOrElse(kv._1, 0)))
})
```

## 题目四：广播变量与广播Join

**问题：什么是广播变量？什么时候应该使用广播变量？如何用广播变量实现大表与小表的 Join？**

### 答案

**广播变量（Broadcast Variable）** 是 Spark 提供的优化机制，允许将只读数据缓存到每个 Executor 的内存中，避免在任务间重复传输。

### 基本用法

```scala
// 定义广播变量
val broadcastData = sc.broadcast(Map("a" -> 1, "b" -> 2, "c" -> 3))

// 在 Task 中使用
rdd.map(record => {
  val lookupTable = broadcastData.value  // 获取广播的数据
  lookupTable.getOrElse(record.key, 0)
})
```

### 广播变量的优势

| 对比 | 无广播 | 使用广播 |
|------|--------|----------|
| **数据传输** | 每个 Task 复制一份数据 | 每个 Executor 共享一份 |
| **网络传输** | N 个 Task × 数据大小 | 1 份数据 |
| **内存占用** | N × 数据大小 | 1 × 数据大小 |

### 大表 Join 小表优化（Broadcast Join）

```scala
// 场景：大表 orders + 小表 products（< 10MB）

// 方法1：自动广播（Spark 自动识别小表）
val result = orders.join(products)

// 方法2：手动广播（显式控制）
val productsBroadcast = sc.broadcast(products.collectAsMap())
val result = orders.mapPartitions(iter => {
  val productMap = productsBroadcast.value
  iter.map(order => (order._1, (order._2, productMap.get(order._1))))
})
```

### 使用注意事项

1. **小表前提**：广播变量存储在 Executor 内存，太大的表会 OOM
2. **阈值设置**：`spark.sql.autoBroadcastJoinThreshold`（默认 10MB）
3. **手动广播**：当 Spark 没有自动识别时，可手动调用 `broadcast()`

## 题目五：Spark 内存管理与存储策略

**问题：Spark 的内存管理机制是怎样的？Executor 内存如何划分？Storage Memory 和 Execution Memory 之间如何协调？**

### 答案

Spark 采用 **统一内存管理**（Unified Memory Management），这是 1.6+ 版本引入的改进。

### Executor 内存区域划分

```
┌─────────────────────────────────────────┐
│           Executor Memory                │
│  ┌───────────────┬─────────────────────┐ │
│  │   User Memory  │    40%             │ │
│  │  (自定义数据结构) │                   │ │
│  ├───────────────┼─────────────────────┤ │
│  │  Execution +   │    60%             │ │
│  │  Storage Memory│                   │ │
│  │  (可动态协调)   │                   │ │
│  └───────────────┴─────────────────────┘ │
│  ┌───────────────┬─────────────────────┐ │
│  │  Reserved     │    300MB           │ │
│  │  Memory       │                   │ │
│  └───────────────┴─────────────────────┘ │
└─────────────────────────────────────────┘
```

### 内存区域详解

| 区域 | 用途 | 默认占比 |
|------|------|----------|
| **Reserved Memory** | 系统预留，固定 300MB | 300MB |
| **User Memory** | 用户代码（UDF 等）使用的内存 | 40% |
| **Execution Memory** | Shuffle 排序、Hash 表、缓存 | 动态协调 |
| **Storage Memory** | RDD 缓存、广播变量、Task 结果 | 动态协调 |

### 动态协调机制

```
当 Execution Memory 不足时：
  → 可以借用空闲的 Storage Memory

当 Storage Memory 不足时：
  → 可以借用空闲的 Execution Memory
  → 如果 Execution 正在使用，则只能溢出到磁盘
```

### 配置示例

```properties
# Executor 内存配置
spark.executor.memory 4g

# Storage Memory 占比（可与其他内存协调）
spark.memory.storageFraction 0.5

# Shuffle 溢写阈值
spark.shuffle.memoryFraction 0.2  # 已废弃（2.0+）
```

### 常见内存问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| **OOM** | 数据量超过内存 | 增加分区数、使用 spill to disk |
| **GC 频繁** | 太多小对象 | 增加 User Memory、减少缓存 |
| **缓存失效** | Execution 借用了 Storage | 增加 executor memory |

### 最佳实践

```scala
// 1. 合理设置分区数，避免 OOM
val rdd = sc.textFile("hdfs://...", 200)  // 根据数据量调整

// 2. 及时清理不需要的缓存
rdd.unpersist()  // 释放缓存

// 3. 使用 Kryo 序列化减少内存占用
sparkConf.set("spark.serializer", "org.apache.spark.serializer.KryoSerializer")
```
