## 题目一：HBase 的整体架构是怎样的？请描述 ZooKeeper、RegionServer、Master 的作用。

### 参考答案

HBase 采用 **Master-Slave** 架构设计，主要包含以下组件：

| 组件 | 作用 |
|------|------|
| **ZooKeeper** | 负责协调和选主，提供分布式锁服务。HBase 通过 ZooKeeper 实现 Master 的高可用，监控 RegionServer 的上下线，存储元数据（如 ROOT 表位置）。 |
| **Master** | 负责表的增删改查（DDL 操作）、Region 的分配与调度、垃圾文件回收、负载均衡等。不参与实际的数据读写。 |
| **RegionServer** | 负责处理数据的读写请求，一个 RegionServer 可以管理多个 Region。它包含 MemStore 和 BlockCache 等组件。 |
| **HDFS** | 底层存储，HBase 数据最终持久化到 HDFS 上。 |

**读写流程简述：**
1. 客户端先访问 ZooKeeper，获取 ROOT 表位置
2. 通过 ROOT 表获取 META 表位置
3. 从 META 表中找到目标 Region 所在的 RegionServer
4. 直接与 RegionServer 通信读写数据

**高可用机制：** Master 与 ZooKeeper 结合实现高可用。当 Active Master 故障时，ZooKeeper 选举新的 Master 接替工作。

📎 **来源：** [HBase 官方文档架构说明](https://hbase.apache.org/book.html#arch)

## 题目二：RowKey 设计应遵循哪些原则？为什么 RowKey 设计如此重要？

### 参考答案

RowKey 是 HBase 中用于唯一标识一行数据的键，其设计原则如下：

### 核心原则

1. **唯一性**：RowKey 必须唯一标识一行数据，类似数据库主键。
2. **散列性**：避免热点问题，RowKey 应均匀分布。常用方法：
   - 字符串反转（如时间戳反转）
   - 哈希（如 MD5 取前几位）
   - 加盐（Salt：在 RowKey 前加随机前缀）
3. **长度控制**：RowKey 长度应尽量短（不超过 100 字节），因为它会在内存和磁盘中存储多次。
4. **查询模式匹配**：根据业务查询习惯设计，考虑：
   - 范围查询：RowKey 有序，可做范围扫描
   - 点查询：RowKey 直接定位

### 热点问题的处理

**热点现象**：大量请求集中在少数几个 RegionServer 上，导致负载不均。

**解决方案**：
```java
// 示例：给 RowKey 加盐
String rowKey = (hash(userId) % N) + "_" + userId + "_" + timestamp;
```
或使用时间戳反转：
```java
// 时间戳反转，使新数据散列
String rowKey = userId + "_" + (Long.MAX_VALUE - timestamp);
```

📎 **来源：** [HBase RowKey 设计最佳实践](https://hbase.apache.org/book.html#rowkey.design)

## 题目三：HBase 的存储结构是什么？请解释 MemStore、HFile（StoreFile）以及 Flush 机制。

### 参考答案

### 存储结构层次

```
Table (表)
  └── Region (按 RowKey 范围切分)
       └── Store (按列族切分)
            ├── MemStore（写缓存）
            └── StoreFile/HFile（磁盘文件）
```

### 核心组件

| 组件 | 说明 |
|------|------|
| **MemStore** | 写缓存，数据先写入 MemStore，当达到阈值（默认 128MB）时，触发 Flush 落盘生成 HFile。 |
| **HFile** | HBase 的存储格式，顺序写，键值对按 RowKey 排序。包含 Data Block、Meta Block、Index Block 等。 |
| **Store** | 一个列族对应一个 Store，包含 MemStore 和多个 HFile。 |

### Flush 触发条件

满足以下任一条件时触发 Flush：
1. MemStore 大小达到 `hbase.memstore.flush.size`（默认 128MB）
2.  Region 内存超过阈值
3.  WAL 文件数量过多（`hbase.regionserver.max.logs`）
4.  定时刷新（`hbase.regionserver.optionalflushinterval`）

### Minor Compaction 和 Major Compaction

- **Minor Compaction**：合并小文件，通常保留 3~10 个 HFile
- **Major Compaction**：合并所有 HFile 为一个，删除标记的记录和过期数据（默认 7 天）

📎 **来源：** [HBase 存储结构与 Compaction](https://hbase.apache.org/book.html#compaction)

## 题目四：什么是 HBase 协处理器（Coprocessor）？它有哪两种类型？各自的使用场景是什么？

### 参考答案

协处理器是运行在 RegionServer 上的插件，可以在数据写入或读取时触发自定义逻辑。

### 两种类型

| 类型 | 说明 |
|------|------|
| **Observer** | 类似于数据库触发器，在特定事件前后执行。用于二级索引、数据验证、审计等。 |
| **Endpoint** | 类似于存储过程，在 RegionServer 上执行计算逻辑，返回结果给客户端。用于聚合、计算等。 |

### Observer 类型（常用）

| 协处理器 | 触发时机 |
|----------|----------|
| **PrePut** / **PostPut** | 数据写入前/后 |
| **PreGet** / **PostGet** | 数据读取前/后 |
| **PreDelete** / **PostDelete** | 数据删除前/后 |
| **PreOpen** | 打开 Scanner 前 |

### Endpoint 类型

用于自定义 RPC 操作，例如：
- 实现自定义聚合（Sum、Avg）
- 执行复杂的计算逻辑

### 使用示例（Observer）

```java
public class MyObserver extends BaseRegionObserver {
    @Override
    public void postPut(final ObserverContext<RegionServerServices> c,
                        final Put put, final WALEdit edit) throws IOException {
        // 在数据写入后执行逻辑，如更新索引表
    }
}
```

### 加载方式

1. **动态加载**：通过 HBase Shell 或 API 动态挂载到表上
2. **配置加载**：在 `hbase-site.xml` 中配置，全局生效

📎 **来源：** [HBase Coprocessor 官方文档](https://hbase.apache.org/book.html#cp)

## 题目五：HBase 如何实现高可用（HA）？当 RegionServer 故障时会发生什么？

### 参考答案

### 高可用机制

HBase 通过以下方式实现高可用：

1. **Master 高可用**：借助 ZooKeeper 进行选主，Active Master 故障时自动切换。
2. **Region 副本**：通过 `replication` 机制，将数据复制到多个 RegionServer（需要开启集群复制）。
3. **WAL 持久化**：数据先写 WAL，确保故障后可恢复。

### RegionServer 故障处理流程

```
1. RegionServer 故障（心跳超时，默认 30s）
2. ZooKeeper 通知 Master
3. Master 将该 RS 上的 Region 分配到其他活着的 RS
4. 重新打开 Region，恢复服务
5. 如果开启了复制，受影响的数据从其他集群同步
```

### 关键配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `zookeeper.session.timeout` | 30s | ZooKeeper 会话超时，超过则认为 RS 死亡 |
| `hbase.master.failover.delay` | 30s | Master 故障检测延迟 |
| `hbase.regionserver.restart.on.expire` | true | RS 过期是否自动重启 |

### 数据恢复

- **MemStore 数据**：未 Flush 的数据会丢失（可通过 WAL 重放恢复，但 WAL 也在故障的 RS 上）
- **HFile 数据**：持久化在 HDFS 上，不受影响
- **写操作**：已写 WAL 的操作可重放

### 最佳实践

1. 开启 `hbase.regionserver.recovery.retain` 确保 WAL 保留
2. 配置合理的 ZooKeeper 超时时间
3. 监控 RS 健康状态，提前预警

📎 **来源：** [HBase 高可用与故障恢复](https://hbase.apache.org/book.html#ops.monitoring)
