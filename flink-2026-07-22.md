## 题目 1：Flink 的整体架构是怎样的？JobManager 和 TaskManager 的作用是什么？

### 参考答案

Flink 架构采用 **Master-Slave** 模式，主要包含以下组件：

| 组件 | 职责 |
|------|------|
| **JobManager** | 整个集群的协调者，负责接收 Flink 作业（Job），协调 Job 的提交 DAG（Directed Acyclic Graph），调度 Task 到 TaskManager 执行，负责 Checkpoint 的协调。每个作业对应一个 JobManager |
| **TaskManager** | 真正执行 Task 的工作节点，以 Slot 为单位管理资源。一个 TaskManager 包含多个 Slot，Slot 数量决定该节点的并发执行能力 |
| **ResourceManager** | 负责管理 TaskManager 上的 Slot，如 YARN / Kubernetes 的资源分配 |
| **Dispatcher** | 负责接收用户提交的 Job，为每个 Job 启动一个新的 JobMaster |

**作业提交流程：**
1. 用户通过 Client（CLI 或 API）提交 JobGraph
2. Dispatcher 接收 Job 并为每个 Job 启动 JobManager
3. JobManager 向 ResourceManager 请求执行 Task 所需的 Slot
4. ResourceManager 向 TaskManager 发起指令，让其启动 Task
5. TaskManager 启动 TaskExecutor 并注册自己的 Slot 到 ResourceManager
6. JobManager 将 JobGraph 中的算子链（Operator Chain）调度到对应的 Slot 上执行

**来源：** https://ci.apache.org/projects/flink/flink-docs-release-1.18/docs/concepts/flink-architecture/

## 题目 2：Flink 的 Watermark 机制是什么？如何处理乱序数据？

### 参考答案

**Watermark（水位线）** 是 Flink 用于处理**事件时间（Event Time）**乱序的一种机制，本质上是一个时间戳，代表"所有时间戳小于该 Watermark 的数据已经全部到达"。

**核心概念：**

- **乱序问题：** 在真实场景中，由于网络延迟、系统抖动等原因，数据到达的顺序不一定与事件时间顺序一致
- **Watermark 是一条推进式的时间戳**：`W(t)` 表示所有事件时间 `< t` 的数据都已经到达（或者至少"我们认为"已经到达）
- **窗口触发时机：** 当 EventTime > window_end_time 时，窗口才会被触发计算

**Watermark 生成方式：**

```java
// 方式一：固定时间间隔生成 Watermark
WatermarkStrategy
    .forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner(SerializableTimestampAssigner);

// 方式二：周期性 Watermark（默认 200ms 一次）
WatermarkStrategy.forMonotonousTimestamps();
```

**乱序处理策略：**
- **allowedLateness()**：设置允许的延迟时间，窗口触发后仍然保留一段窗口状态，等待迟到数据
- **sideOutputTag()**：将迟到严重的数据输出到侧输出流

```java
DataStream<String> output = stream
    .keyBy(...)
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .allowedLateness(Time.minutes(2))
    .sideOutputLateData(lateOutputTag)
    .sum(1);
```

**来源：** https://ci.apache.org/projects/flink/flink-docs-release-1.18/docs/concepts/time/#event-time-and-watermarks

## 题目 3：Flink 的 Checkpoint 和 Savepoint 有什么区别？

### 参考答案

| 维度 | Checkpoint | Savepoint |
|------|-----------|-----------|
| **触发方式** | Flink 自动周期性触发（由 JobManager 协调） | 手动触发（用户通过 CLI 或 API 调用） |
| **用途** | 容错保障，作业失败自动恢复 | 计划性备份，如版本升级、集群迁移 |
| **删除机制** | 作业取消或删除时自动清理 | 除非手动删除，否则一直保留 |
| **存储位置** | 由 Flink 管理，通常在 State Backend 指定目录 | 存储在用户指定的目录（可跨集群恢复） |
| **格式** | Flink 内部格式，理论上不保证跨版本兼容 | 标准化格式，设计上尽量保证兼容性 |
| **依赖** | 需要开启 Checkpoint：`env.enableCheckpointing(interval)` | 依赖 `SavepointFormatType`（默认可兼容算子） |

**Checkpoint 核心原理：**
- 基于 **Chandy-Lamport 分布式快照算法**
- JobManager 定期向所有 Source 算子发送 Barrier（屏障）
- Barrier 随数据流向下游广播，各算子收到所有输入的 Barrier 后做快照
- 快照写入 State Backend（HDFS / RocksDB 等）

```java
env.enableCheckpointing(60_000); // 每 60 秒一次
env.getCheckpointConfig().setCheckpointStorage("hdfs:///checkpoint/");
```

**来源：** https://ci.apache.org/projects/flink/flink-docs-release-1.18/docs/ops/state/checkpoints/

## 题目 4：Flink 的状态管理机制是什么？状态后端（State Backend）有哪些类型？

### 参考答案

**Flink 中的状态分为两大类：**

| 类型 | 说明 | 示例 |
|------|------|------|
| **Keyed State** | 按 Key 分组，只能在 KeyedStream 上使用，作用域为当前 Key | ValueState, ListState, MapState, ReducingState, AggregatingState |
| **Operator State** | 作用域为整个算子实例，所有 Key 共享 | ListState, UnionListState, BroadcastState |

**常用 Keyed State 特性：**
```java
ValueStateDescriptor<T> descriptor = new ValueStateDescriptor<>(
    "myValue",  // 状态名称
    TypeInformation.of(new TypeHint<T>() {})
);
ValueState<T> valueState = getRuntimeContext().getState(descriptor);
```

**状态后端（State Backend）类型：**

| State Backend | 存储介质 | 适用场景 | 缺点 |
|--------------|---------|---------|------|
| **HashMapStateBackend** | JVM Heap（内存） | 小状态、低延迟本地开发/测试 | 状态过大会 OOM，不支持大状态 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘 + 内存） | 大状态、增量 Checkpoint、超过内存限制的状态 | 序列化开销，磁盘 I/O 延迟 |

```java
// 配置 EmbeddedRocksDBStateBackend（生产常用）
env.setStateBackend(new EmbeddedRocksDBStateBackend());
env.getCheckpointConfig().setCheckpointStorage("hdfs:///checkpoint/");
```

**状态过期与清理：**
- 可通过 `.ttl(Time.seconds(10))` 设置状态 TTL
- 结合 `StateTtlConfig` 控制清理策略（懒清理/全量清理）

```java
StateTtlConfig ttlConfig = StateTtlConfig
    .newBuilder(Time.seconds(10))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
    .build();
```

**来源：** https://ci.apache.org/projects/flink/flink-docs-release-1.18/docs/concepts/stateful-stream-processing/

## 题目 5：Flink 的窗口函数有哪几种？滚动窗口、滑动窗口、会话窗口的区别是什么？

### 参考答案

Flink 支持两大类窗口：**Keyed Windows** 和 **Non-Keyed Windows**。按功能划分，窗口类型如下：

### 1. 滚动窗口（Tumbling Window）
- 窗口大小固定，**不重叠**，首尾相接
- 适用于固定时间周期统计，如"每分钟的 PV"

```java
.stream.keyBy(...)
  .window(TumblingEventTimeWindows.of(Time.minutes(1)))
  .sum(1);
```

### 2. 滑动窗口（Sliding Window）
- 窗口大小固定，但**可以重叠**，由窗口大小和滑动步长共同决定
- 适用于"平滑统计"场景，如"每 5 分钟计算一次过去 10 分钟的平均值"

```java
.stream.keyBy(...)
  .window(SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(5)))
  .sum(1);
```

### 3. 会话窗口（Session Window）
- 以**活跃间隔**来界定窗口，数据到达后若超过一定时间没有新数据，则窗口关闭
- 适用于用户行为分析，如"用户 30 秒内无操作则结束会话"

```java
.stream.keyBy(...)
  .window(SessionWindow.withGap(Time.seconds(30)))
  .sum(1);
```

### 4. 全局窗口（Global Window）
- 所有相同 Key 的数据归入同一个窗口，需配合**触发器**使用
- 适用于自定义业务逻辑

### 窗口函数分类

| 函数类型 | 说明 | 典型场景 |
|---------|------|---------|
| **增量聚合函数**（ReduceFunction, AggregateFunction） | 每条数据到来即更新状态，输出单条结果 | 求和、计数、平均值 |
| **全量窗口函数**（ProcessWindowFunction） | 窗口触发时才进行全量计算，可访问窗口元数据 | 复杂计算、TopN、排序 |

```java
// 增量 + 全量组合使用（最佳实践）
stream
    .keyBy(...)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .aggregate(new MyAggregateFunction(), new MyProcessWindowFunction())
```

**来源：** https://ci.apache.org/projects/flink/flink-docs-release-1.18/docs/dev/datastream_api/operators/windows/
