## 题目1：Java基础 - HashMap的扩容机制，负载因子为什么是0.75？

【答案】

### HashMap底层数据结构

- **JDK 1.7**：数组 + 链表（Entry数组），当发生哈希冲突时，相同bucket的多个Entry以链表形式存储（头插法）
- **JDK 1.8**：数组 + 链表 + 红黑树。当链表长度超过8时自动转换为红黑树（桶长度<64时优先扩容）；当红黑树节点数<=6时退化为链表

### 扩容机制

1. **触发条件**：size（实际元素数量）>= threshold（扩容阈值）时触发扩容
2. **扩容计算**：`newCap = oldCap × 2`，`newThreshold = oldCap × 2 × loadFactor`
3. **扩容过程**：新建一个2倍大小的数组，遍历旧数组每个bucket，将所有Entry重新计算index后放入新数组
4. **JDK 1.8优化**：使用`resize()`方法，扩容时无需重新计算哈希，只需判断原index位的bit是0还是1，bit为0则新index=旧index，bit为1则新index=旧index+旧容量。大幅提升扩容效率

### 负载因子为什么是0.75？

这是一个时间和空间的trade-off：

1. **空间与时间的平衡**：
   - 负载因子越小（如0.5），哈希冲突越少，查询效率更高，但空间浪费严重（需要2倍的空间）
   - 负载因子越大（如0.9），空间利用率高，但哈希冲突增多，链表/红黑树变长，查询性能下降

2. **泊松分布考虑**：根据泊松分布，链表长度超过8的概率已经非常小（约0.00000006），0.75是一个经验最优值

3. **树化阈值8的选择**：链表长度超过8时转换为红黑树，配合0.75的负载因子，使得在内存使用和查询性能之间达到最优平衡

4. **避免极端情况**：0.75在时间和空间上都接近最优，即使在hash设计得不够均匀时也有较好的表现

## 题目2：Java进阶 - ReentrantLock与synchronized的区别是什么？

【答案】

### 基本特性对比

| 特性 | ReentrantLock | synchronized |
|------|--------------|--------------|
| 锁类型 | 可重入锁（独占） | 可重入锁（独占） |
| 等待方式 | `lock()`会阻塞，可 tryLock 非阻塞 | 阻塞直到获取锁 |
| 公平性 | 支持公平锁/非公平锁 | 非公平锁 |
| 锁粒度 | 需要手动释放（`unlock()`） | 自动释放 |
| 功能丰富度 | 支持多条件变量、轮询、超时 | 功能单一 |
| 中断响应 | 支持 `lockInterruptibly()` 可中断等待 | 不可中断 |

### ReentrantLock核心原理

1. **基于AQS（AbstractQueuedSynchronizer）**实现，通过CAS操作维护状态和同步队列
2. **可重入原理**：内部通过计数器记录重入次数，每次`lock()`计数+1，`unlock()`计数-1
3. **公平锁**：多线程获取锁时，按进入队列的顺序依次获取（FIFO）
4. **非公平锁**：新线程直接尝试CAS获取锁，可能"插队"抢先

### 场景选择建议

- **优先使用 synchronized**：简单场景、JVM自动管理、不需要超时/可中断
- **使用 ReentrantLock**：需要限时获取锁、可中断等待、公平锁、多条件等待

### 经典示例：synchronized + wait/notify 与 ReentrantLock + Condition

```java
// synchronized方式
synchronized (lock) {
    while (condition不满足) {
        lock.wait();
    }
    // 处理逻辑
    lock.notifyAll();
}

// ReentrantLock方式
lock.lock();
try {
    while (condition不满足) {
        condition.await(); // 精确等待
    }
    // 处理逻辑
    condition.signalAll();
} finally {
    lock.unlock();
}
```

ReentrantLock的Condition支持一个锁创建多个条件变量，而synchronized只有一个隐式条件

## 题目3：Spring Boot - @ComponentScan和@SpringBootApplication的关系

【答案】

### @SpringBootApplication的组成

`@SpringBootApplication`是一个组合注解，包含三个核心注解：

```java
@SpringBootConfiguration   // 本质是@Configuration，标记为配置类
@EnableAutoConfiguration   // 开启自动配置
@ComponentScan             // 组件扫描
```

### @ComponentScan的作用

1. **默认扫描规则**：默认扫描`@SpringBootApplication`所在包及其子包下的所有标注了`@Component/@Service/@Repository/@Controller/@Configuration`的类
2. **自定义扫描**：可以通过`basePackages`或`basePackageClasses`指定扫描路径

```java
@SpringBootApplication
@ComponentScan(basePackages = {"com.example.demo", "com.example.other"})
public class Application { ... }
```

### @EnableAutoConfiguration的工作原理

1. **核心机制**：利用Spring Factories机制，在`spring-boot-autoconfigure.jar`的`META-INF/spring.factories`中定义了大量自动配置类
2. **加载流程**：
   - Spring Boot应用启动时，`AutoConfigurationImportSelector`读取`spring.factories`
   - 过滤掉不符合条件（无对应class、无对应bean等）的自动配置类
   - 通过`@Conditional`系列注解（如`@ConditionalOnClass`、`@ConditionalOnBean`）控制生效条件
3. **常见自动配置**：DataSource自动配置、WebMvc自动配置、Redis自动配置等

### 为什么建议将Application主类放在根包

- `@ComponentScan`默认扫描主类所在包及其子包
- 如果主类放在`com.example.app`，则会自动扫描`com.example.app`下所有组件
- 放在其他位置需要手动指定扫描路径

## 题目4：Spring Cloud - Nacos的服务注册与发现原理

【答案】

### 服务注册流程

1. **服务启动**：Nacos客户端启动时，将自身信息（IP、端口、服务名、集群名等）封装为`Instance`注册到Nacos Server
2. **心跳机制**：客户端默认每5秒发送一次心跳（` heartbeat`），携带服务实例信息
3. **服务续约**：Nacos Server收到心跳后，若超过15秒未收到，则标记为不健康；超过30秒则移除实例
4. **元数据更新**：可以通过`NacosServiceInstance`动态更新实例元数据（如权重、标签）

### 服务发现原理

1. **主动拉取**：服务消费者启动时，通过`NacosClient`从Nacos Server拉取服务提供者列表并缓存在本地
2. **定时更新**：本地缓存默认每10秒更新一次（可通过`refresh`相关配置调整）
3. **订阅变更**：通过Nacos Server的`订阅（subscribe）`机制，当服务列表变更时Server主动推送变更给Client（基于HTTP长轮询或gRPC）

### 集群选举机制

- Nacos默认采用**Raft算法**进行Leader选举
- 如果是临时实例，Nacos会采用类似Eureka的心跳机制
- 通过`distro`协议实现数据同步，适用于多机房场景

### Nacos与Eureka对比

| 特性 | Nacos | Eureka |
|------|-------|--------|
| 一致性 | CP（Raft）/AP（distro）可选 | AP |
| 健康检查 | TCP/HTTP/MySQL/Agent | 心跳 |
| 复杂度 | 功能更多，学习成本稍高 | 简单 |
| 配置管理 | 支持配置管理 | 不支持 |
| 发展趋势 | 活跃度高 | 已停止维护 |

## 题目5：Spring Cloud - Sentinel的滑动窗口算法实现限流原理

【答案】

### 滑动窗口算法概述

滑动窗口算法是Sentinel实现限流的核心，它通过将时间线切分为多个小窗口，动态计算最近N个窗口的请求总量来判断是否限流。

### 工作原理

1. **窗口划分**：将时间轴划分为固定大小的窗口（如1秒划分为2个500ms窗口）
2. **窗口数组**：使用数组（循环数组）存储每个窗口的统计信息
3. **时间窗口滑动**：当前时间窗口过期后，自动滑动到下一个窗口
4. **请求统计**：`curSlot.windowIn introduction()`方法将请求放入当前窗口并计数

### 核心数据结构

```java
// 滑动窗口核心组件
private final WindowWrap<T>[] windowArray; // 窗口数组
private final long windowLengthInMs;        // 单窗口长度
private final long intervalInMs;            // 滑动窗口总长度
private final long sampleCount;             // 采样窗口数量
```

### 限流计算过程

1. **计算当前窗口**：根据当前时间戳计算属于哪个窗口索引
2. **窗口重置**：如果当前窗口的开始时间早于记录的开始时间（窗口已过期），重置该窗口
3. **统计求和**：遍历滑动窗口内的所有窗口，累加请求数
4. **判断阈值**：累加值 >= 阈值 → 限流；累加值 < 阈值 → 通过

### Sentinel其他限流策略

1. **直接拒绝**（`RuleConstant.CONTROL_BEHAVIOR_DEFAULT`）：请求超过阈值直接拒绝
2. **冷启动**（`CONTROL_BEHAVIOR_WARM_UP`）：预热期逐渐提升阈值
3. **排队等待**（`CONTROL_BEHAVIOR_RATE_LIMITER`）：FIFO队列，超时等待
4. **冷启动+排队**（`CONTROL_BEHAVIOR_WARM_UP_RATE_LIMITER`）：结合预热和队列

### 与Leaky Bucket/Token Bucket对比

| 特性 | 滑动窗口 | Leaky Bucket | Token Bucket |
|------|---------|--------------|-------------|
| 实现复杂度 | 中 | 复杂 | 复杂 |
| 精确度 | 较高 | 平滑 | 平滑 |
| 突发处理 | 一般 | 差 | 好 |

## 题目6：Kafka - 高吞吐量原因分析

【答案】

### Kafka高吞吐量的核心技术

### 1. 顺序写磁盘

- **原理**：Kafka写入数据时追加到文件末尾（append-only），利用顺序I/O的高性能
- **机械磁盘顺序写速度**可达~600MB/s，远高于随机写（~0.1MB/s）
- **Page Cache**：利用操作系统Page Cache，写入内存即返回，由OS异步刷盘

### 2. 零拷贝技术

- **传统方式**：磁盘→内核缓冲区→用户缓冲区→Socket缓冲区→网卡，需要4次拷贝、4次上下文切换
- **Kafka零拷贝**：使用`sendfile()`系统调用，磁盘→内核缓冲区→Socket缓冲区→网卡，只需3次拷贝
- Linux的`transferTo()`实现，数据无需经过用户空间，直接从Page Cache发送到网络

### 3. 分区（Partition）并行

- Kafka Topic支持多个分区，不同分区可分布在不同Broker
- 消费者可并行消费不同分区，实现水平扩展
- 分区内消息处理顺序由Leader分区保证

### 4. 批量处理（Batch）

- **消息批量发送**：Producer将多条消息缓存后批量发送，减少网络往返
- **消息批量写入**：Broker将多条消息批量写入磁盘
- **消息批量拉取**：Consumer可配置每次拉取消息的数量和大小上限
- **压缩批处理**：支持BZIP2、Snappy、GZIP、LZ4、ZSTD等多种压缩算法

### 5. 高效数据结构

- **稀疏索引**：Kafka使用稀疏的索引文件快速定位消息
- **页缓存**：依赖OS的页缓存而非JVM堆内存，减少GC压力
- **日志分段**：通过分段（segment）设计，支持旧数据快速删除和独立读写

### 性能对比（单Broker）

| 操作 | 吞吐量 |
|------|--------|
| 顺序写入 | ~600MB/s |
| 读取（缓存命中） | ~1GB/s |
| 端到端延迟 | ~2ms |

## 题目7：RabbitMQ - 延迟队列的多种实现方式

【答案】

### 延迟队列应用场景

- 订单超时取消（下单后30分钟未支付自动关闭）
- 定时任务调度
- 消息重试（失败后延迟N秒重试）
- 定期提醒（如生日提醒）

### 实现方式一：TTL+DLX（Dead Letter Exchange）

**原理**：利用消息或队列的TTL（Time To Live）特性，配合死信交换机实现延迟

1. **设置消息TTL**：
```java
rabbitTemplate.convertAndSend("exchange", "routingKey", message, m -> {
    m.setExpiration("30000"); // 30秒
    return m;
});
```

2. **设置队列TTL**（整队列延迟）：
```java
@Bean
public Queue delayQueue() {
    return QueueBuilder.durable("delay.queue")
            .ttl(30000)  // 30秒
            .build();
}
```

3. **死信交换机配置**：消息TTL过期后进入死信交换机，路由到实际消费队列

**缺点**：消息按过期时间重新排序，可能产生**队首消息阻塞**问题

### 实现方式二：延迟插件（rabbitmq-delayed-message-exchange）

- **原理**：插件内部使用Mnesia数据库记录延迟投递事件，按超时时间排序
- **配置**：声明`x-delayed-message`类型交换机
```java
@Bean
public CustomExchange delayExchange() {
    Map<String, Object> args = new HashMap<>();
    args.put("x-delayed-type", "direct");
    return new CustomExchange("delayed.exchange", "x-delayed-message", true, false, args);
}
```

- **优点**：支持精确延迟，不存在队首阻塞
- **缺点**：需要安装插件

### 实现方式三：RabbitMQ Shovel插件

- 将消息从一个Broker转发到另一个Broker，源队列设置TTL实现延迟

### 常见问题

1. **消息顺序性**：延迟队列不保证消息顺序，需要业务方自行处理
2. **延迟精度**：TTL精度为毫秒，但实际延迟取决于消息在队列中的位置
3. **集群问题**：DLX方式在集群中可能无法正常工作，需使用延迟插件

## 题目9：算法 - 搜索旋转排序数组（LeetCode 33）

【答案】

### 题目描述

假设一个排序数组按升序排列，在某个节点被旋转（如`[0,1,2,4,5,6,7]`变成`[4,5,6,7,0,1,2]`）。给定目标值，找到它的索引，不存在返回-1。要求时间复杂度O(log n)。

### 解题思路

使用二分查找变体，关键是根据mid划分哪一段是有序的：

1. 如果`nums[mid] == target`，找到答案
2. **左半部分有序**（`nums[left] <= nums[mid]`）：
   - target在`[nums[left], nums[mid])`之间 → 搜索左半部分
   - 否则 → 搜索右半部分
3. **右半部分有序**（`nums[mid] <= nums[right]`）：
   - target在`[nums[mid], nums[right]]`之间 → 搜索右半部分
   - 否则 → 搜索左半部分

### 代码实现（Java）

```java
class Solution {
    public int search(int[] nums, int target) {
        int left = 0, right = nums.length - 1;
        
        while (left <= right) {
            int mid = left + (right - left) / 2;
            
            if (nums[mid] == target) {
                return mid;
            }
            
            // 左半部分有序
            if (nums[left] <= nums[mid]) {
                // target在左半部分
                if (nums[left] <= target && target < nums[mid]) {
                    right = mid - 1;
                } else {
                    left = mid + 1;
                }
            } else {
                // 右半部分有序
                // target在右半部分
                if (nums[mid] < target && target <= nums[right]) {
                    left = mid + 1;
                } else {
                    right = mid - 1;
                }
            }
        }
        
        return -1;
    }
}
```

### 图解过程

以`nums = [4,5,6,7,0,1,2]`, `target = 0`为例：

```
第1轮：left=0, right=6, mid=3
      nums[3]=7 > nums[0]=4，左半部分有序
      target=0不在[4,7)之间 → 搜索右半部分
      left=4

第2轮：left=4, right=6, mid=5
      nums[5]=1 > nums[4]=0，左半部分有序
      target=0在[0,1)之间 → 搜索左半部分
      right=4

第3轮：left=4, right=4, mid=4
      nums[4]=0 == target → return 4
```

### 复杂度分析

- **时间复杂度**：O(log n)，每次将搜索区间减半
- **空间复杂度**：O(1)，仅用常数额外空间

## 题目10：算法 - 全排列（LeetCode 46）

【答案】

### 题目描述

给定一个不含重复数字的数组，返回其所有可能的全排列。

### 解题思路：回溯算法（Backtracking）

全排列是经典的回溯问题：
1. **选择**：从剩余未选择的元素中选一个
2. **递归**：处理剩余元素的全排列
3. **撤销**：回退选择，继续尝试其他选项

### 代码实现（Java）

```java
class Solution {
    public List<List<Integer>> permute(int[] nums) {
        List<List<Integer>> result = new ArrayList<>();
        List<Integer> path = new ArrayList<>();
        boolean[] used = new boolean[nums.length];
        
        backtrack(nums, path, used, result);
        return result;
    }
    
    private void backtrack(int[] nums, List<Integer> path, 
                          boolean[] used, List<List<Integer>> result) {
        // 终止条件：path长度等于nums长度
        if (path.size() == nums.length) {
            result.add(new ArrayList<>(path));
            return;
        }
        
        for (int i = 0; i < nums.length; i++) {
            // 跳过已使用的元素
            if (used[i]) {
                continue;
            }
            
            // 做选择
            used[i] = true;
            path.add(nums[i]);
            
            // 递归
            backtrack(nums, path, used, result);
            
            // 撤销选择（回溯）
            path.remove(path.size() - 1);
            used[i] = false;
        }
    }
}
```

if (i > 0 && nums[i] == nums[i-1] && !used[i-1]) {
