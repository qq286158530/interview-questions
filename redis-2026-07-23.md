## 题目一：Redis 有哪些数据类型？底层分别是如何实现的？

### 参考答案

Redis 提供了丰富的数据类型，常见的有 **5 种基本数据类型**和**4 种扩展数据类型**。

**5 种基本数据类型：**

| 类型 | 应用场景 | 底层数据结构 |
|------|---------|-------------|
| **String（字符串）** | 缓存对象、常规计数、分布式锁、共享 session | SDS（简单动态字符串） |
| **Hash（哈希）** | 缓存对象、购物车 | 压缩列表 / 哈希表 |
| **List（列表）** | 消息队列（需自行实现全局唯一ID）、链表场景 | quicklist（Redis 3.2+） |
| **Set（集合）** | 聚合计算（并集/交集/差集），如点赞、共同关注 | 哈希表 / 整数集合 |
| **Zset（有序集合）** | 排行榜、电话姓名排序 | 压缩列表 / 跳表 |

**4 种扩展数据类型：**

| 类型 | 版本 | 应用场景 |
|------|------|---------|
| **BitMap（位图）** | 2.2+ | 二值状态统计（签到、判断登陆、连续签到总数） |
| **HyperLogLog** | 2.8+ | 海量数据基数统计（百万级 UV 计数） |
| **GEO（地理信息）** | 3.2+ | 存储地理位置（滴滴叫车） |
| **Stream（流）** | 5.0+ | 消息队列，支持消费组形式消费 |

**String 底层 SDS 的优势：**
- O(1) 复杂度获取字符串长度（C 字符串为 O(n)）
- 二进制安全（不依赖 `\0` 结尾，可存储图片/音频等二进制数据）
- 不会发生缓冲区溢出（自动扩容）
- 节省内存（设计了 5 种头部长度类型：sdshdr5/8/16/32/64）

**压缩列表的连锁更新问题：**
压缩列表节点的 `prevlen` 属性会根据前一个节点长度使用 1 字节或 5 字节存储。当插入较大节点时，可能触发连锁的空间扩展（每个节点的 `prevlen` 都需扩容），造成性能下降。Redis 3.2 引入 quicklist 替代，Redis 7.0 用 listpack 完全替代压缩列表。

> **来源：** [图解 Redis - 数据结构篇](https://xiaolincoding.com/redis/data_struct/data_struct.html)

## 题目二：Redis 持久化机制有哪些？RDB 和 AOF 各自的优缺点是什么？

### 参考答案

Redis 共有 **三种持久化方式**：

### 1. RDB 快照（Redis Database）

将某一时刻的内存数据以**二进制**方式写入磁盘。Redis 提供了两个命令：
- `save`：在主线程执行，会**阻塞**主线程
- `bgsave`：创建子进程生成 RDB 文件，**不阻塞**主线程

RDB 快照采用**写时复制（COW）**技术：fork 出子进程后，父子进程共享物理内存；主线程执行写操作时，被修改的数据会复制出一份副本给子进程写入 RDB，主线程继续处理新请求。

**配置自动触发（Redis.conf）：**
```
save 900 1      # 900秒内至少1次修改
save 300 10     # 300秒内至少10次修改
save 60 10000   # 60秒内至少10000次修改
```

**优点：** 恢复数据速度快（全量二进制，直接加载进内存）
**缺点：** 快照频率难把握——频率太低丢失数据多，频率太高影响性能

### 2. AOF 日志（Append Only File）

每执行一条写操作命令，就将该命令以**追加**方式写入 AOF 文件。Redis 重启时读取 AOF 文件的命令逐一执行来恢复数据。

**AOF 写回策略（appendfsync）：**

| 策略 | 行为 | 安全性 | 性能 |
|------|------|--------|------|
| **Always** | 每次写命令执行完同步写回硬盘 | 最高（不丢数据） | 最低 |
| **Everysec**（默认） | 每秒将内核缓冲区写回硬盘 | 较快（最多丢1秒数据） | 较好 |
| **No** | 交由操作系统决定何时写回 | 最差 | 最好 |

**AOF 重写机制：** 随着命令增多，AOF 文件过大导致恢复慢。AOF 重写由后台子进程 `bgrewriteaof` 完成，读取当前数据库中所有键值对，每类键只记录最新一条命令，大大压缩文件体积。重写期间主线程将新命令同时写入「AOF 缓冲区」和「AOF 重写缓冲区」，确保数据一致性。

**优点：** 丢失数据少
**缺点：** 恢复速度慢（需逐一执行命令），AOF 文件通常比 RDB 大

### 3. 混合持久化（Redis 4.0+）

结合 RDB 和 AOF 的优点：
- AOF 重写时，先以 **RDB 格式**写入全量数据
- 后续增量命令以 **AOF 格式**追加

加载时先加载 RDB 部分（快速），再加载后续 AOF 增量（数据完整）。

**优点：** 启动快 + 数据丢失少
**缺点：** AOF 文件可读性差，不兼容 Redis 4.0 以前版本

> **来源：** [图解 Redis - 持久化篇](https://xiaolincoding.com/redis/storage/aof.html)

## 题目三：Redis 是单线程吗？为什么单线程的 Redis 还能保持高性能？

### 参考答案

**Redis 单线程指的是**：「接收客户端请求 → 解析请求 → 进行数据读写等操作 → 发送数据给客户端」这一完整流程由**主线程**完成。

但 Redis **并不是完全单线程**，Redis 启动时会启动**后台线程（BIO）**：
- **Redis 2.6**：2 个后台线程——关闭文件、AOF 刷盘
- **Redis 4.0+**：新增 lazy free 线程，异步释放内存（大 key 删除用 `unlink` 代替 `del`）
- **Redis 6.0+**：3 个 I/O 线程分担网络 I/O 压力（但命令执行仍是单线程）

### Redis 单线程仍然高效的原因

1. **大部分操作在内存中完成**：Redis 瓶颈是内存和网络带宽，而非 CPU，单线程足够
2. **高效的数据结构**：SDS、哈希表、跳表等数据结构使操作复杂度低
3. **避免锁竞争**：单线程不存在多线程竞争，无需加锁解锁，无死锁问题
4. **I/O 多路复用**：使用 epoll/select/kqueue 机制，一个线程监听处理大量 Socket 请求，一个线程同时处理多个 I/O 流

### Redis 6.0 多线程改进

主要改进的是**网络 I/O** 部分（接收请求和发送响应），而**命令执行仍然是单线程**，因此不改变 Redis 的线程安全特性。

```
Redis 6.0 线程数配置建议：
- 4 核 CPU → io-threads 设置为 2 或 3
- 8 核 CPU → io-threads 设置为 6
```

## 题目四：什么是缓存雪崩、击穿、穿透？各有什么解决方案？

### 参考答案

缓存异常会产生三个经典问题，核心都是数据库压力骤增。

### 1. 缓存雪崩

**定义：** 大量缓存数据在同一时间过期，或者 Redis 故障宕机，导致大量请求直接访问数据库，数据库被冲垮。

**两种诱因及解决方案：**

| 诱因 | 解决方案 |
|------|---------|
| 大量数据同时过期 | ① 过期时间加随机数，分散过期时间 ② 互斥锁（只允许一个请求构建缓存）③ 后台异步更新缓存 |
| Redis 故障宕机 | ① 服务熔断/请求限流 ② 构建主从集群保证高可用 |

### 2. 缓存击穿

**定义：** 热点数据（频繁访问的数据）在缓存中过期，此时大量并发请求访问该热点数据，直接击穿到数据库。

**解决方案：**
- 互斥锁（与雪崩相同）
- 热点数据不设置过期时间，由后台异步更新

### 3. 缓存穿透

**定义：** 数据既不在缓存中，也不在数据库中（如业务误删、黑客攻击），请求直接打到数据库。

**解决方案：**

| 方案 | 说明 |
|------|------|
| 非法请求限制 | 在 API 入口校验请求参数合理性 |
| 缓存空值/默认值 | 缓存一个空值或默认值，快速返回 |
| 布隆过滤器 | 数据写入时在布隆过滤器标记，查询时先检查布隆过滤器 |

**布隆过滤器原理：** 由 N 个哈希函数 + 位图数组组成。数据写入时计算 N 个哈希值，将位图数组对应位置设为 1。查询时若有任何对应位为 0，则数据一定不存在。存在误判（哈希冲突），但不存在的数据一定不会被误判。

> **来源：** [图解 Redis - 缓存雪崩/击穿/穿透](https://xiaolincoding.com/redis/cluster/cache_problem.html)

## 题目五：Redis 如何实现高可用？主从复制、哨兵、集群模式分别解决了什么问题？

### 参考答案

Redis 高可用方案从基础到高级分为三个层次。

### 1. 主从复制（一主多从）

**原理：** 主节点处理写操作，从节点只读。主节点将写操作异步同步给从节点，实现**读写分离**。

**特点：**
- 主从命令复制是**异步**的，主节点不会等到从节点执行完才返回
- 无法保证强一致性（主从数据可能短暂不一致）
- 主节点宕机需要手动恢复

### 2. 哨兵模式（Sentinel）

**解决的问题：** 主从模式下主节点故障需要人工手动切换，哨兵实现了**自动化故障转移**。

**核心功能：**
- **监控**：哨兵周期性检查主从节点是否存活
- **自动故障转移**：主节点宕机后，哨兵从从节点中选举出新主节点，其他从节点改跟新主节点
- **通知**：将故障转移结果通知客户端（通过发布订阅）

**工作流程：**
1. 主观下线：单个哨兵认为主节点不可用
2. 客观下线：多个哨兵投票，超过 quorum 数量则认为主节点确实宕机
3. 选举领头的哨兵节点，由它负责故障转移
4. 从从节点中选举出新主节点（优先级 + 同步偏移量 + RunID）

### 3. 切片集群（Cluster 模式）

**解决的问题：** 主从复制和哨兵模式都是一主多从，存储容量受单机内存限制。Cluster 通过**数据分片**实现分布式存储。

**核心机制：**
- **哈希槽（Hash Slot）**：Redis Cluster 将整个 keyspace 划分为 **16384 个槽**，每个节点负责一部分槽的读写
- **数据迁移**：支持节点间槽的迁移，对客户端透明
- **故障转移**：与哨兵类似，集群内节点互相监控，自动进行故障转移

**集群 vs 哨兵：**
| 特性 | 哨兵 | 集群 |
|------|------|------|
| 数据存储 | 主从复制，单机容量即总容量 | 分片存储，总容量 = 单机容量 × 节点数 |
| 写操作 | 主节点写入 | 只有主节点可写（每个槽有对应主节点） |
| 复杂度 | 较低 | 较高（需处理数据迁移和跨槽操作） |

> **来源：** [图解 Redis - 主从复制/哨兵/集群篇](https://xiaolincoding.com/redis/cluster/master_slave_replication.html)
