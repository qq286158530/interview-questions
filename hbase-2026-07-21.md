## 题目一：HBase 的整体架构是怎样的？请详细说明 RegionServer、Region、HMaster 的作用及数据写入流程。

### 参考答案

**架构概览**

HBase 采用典型的 Master-Slave 架构，主要包含三个核心组件：

| 组件 | 说明 |
|------|------|
| **HMaster** | 负责表的管理（创建/删除/修改）、Region 的分配与调度、负载均衡、故障恢复。元数据存储在 ZooKeeper 中。 |
| **RegionServer** | 负责处理数据的读写请求，一个 RegionServer 可以管理多个 Region，是真正处理请求的工作节点。 |
| **ZooKeeper** | 存储元数据（ROOT/META 表位置）、选举 Master、监控 RegionServer 上下线。 |

**核心概念**

- **Region**：按 RowKey 范围划分的数据分片，每个 Region 默认约 1GB，承载一定范围的 rowkey 数据。
- **Store**：每个 Region 按列族（Column Family）划分 Store，每个 Store 含一个 MemStore 和多个 HFile。
- **MemStore**：写缓存，先将数据写入内存，满了之后 flush 成 HFile。
- **HFile**：HBase 的底层存储文件格式，基于 LSM-Tree（Log-Structured Merge Tree），数据有序。

**数据写入流程（源码级）**

```
1. 客户端请求 ZK，获取 META 表所在的 RegionServer 地址
2. 客户端访问 META RegionServer，查询目标 rowkey 所在的 RegionServer
3. 客户端直接向目标 RegionServer 发起 Put 请求
4. RegionServer 先将数据写入 WAL（Write-Ahead Log）—— 保障宕机可恢复
5. 数据写入 MemStore，同时记录写入 MemStore 的序列号（sequenceId）
6. 当 MemStore 达到阈值（约 128MB）后，触发 flush，生成新的 HFile
7. HFile 异步进行 Compaction（合并），删除标记过的数据，优化读取性能
```

**特点总结**

- **顺序写**：数据写入 HBase 实际上是顺序写 WAL + 写 MemStore，磁盘 IO 效率极高。
- **LSM-Tree**：HFile 多层合并，牺牲少量读性能换取极致写性能，适合大数据顺序写入场景。
- **强一致性**：HBase 是 CP 系统（CAP 理论），保证数据强一致性。

## 题目二：HBase 的 RowKey 设计原则有哪些？如何设计 RowKey 来避免热点问题？

### 参考答案

**RowKey 的重要性**

RowKey 是 HBase 数据的唯一主键，决定了数据的存储位置（哪个 Region、哪台机器）。好的 RowKey 设计能实现：
- 数据均匀分布，避免热点
- 利用 HBase 的字典序特性做范围查询
- 减少 IO，提升读写性能

**设计原则**

### 1. 长度原则
- RowKey 长度应尽量**短**（建议 16 字节以内，最长 64KB）
- 理由：RowKey 会冗余存储在每个 HFile 索引中，影响内存和磁盘占用

### 2. 散列原则
- RowKey 必须保证**足够随机**，避免数据集中在少数 Region（热点）
- 方法：
  - 哈希后作为 RowKey 前缀：`md5(rowkey) + original_key`
  - 盐（Salting）：对 RowKey 追加随机前缀

### 3. 唯一原则
- RowKey 必须全局唯一，否则会覆盖同 rowkey 的历史版本

### 4. 有序原则
- 利用字典序排列，可以做范围查询（如时间戳前缀的 RowKey 可以按时间扫描）

**热点场景与解决方案**

| 热点场景 | 原因 | 解决方案 |
|----------|------|----------|
| 时间序列数据 | 同一时间戳作为前缀导致数据集中 | 时间戳倒置（如 `Long.MAX_VALUE - timestamp`）、加盐哈希 |
| 用户ID/设备ID 作为前缀 | 热门用户数据集中 | 哈希打散或加盐 |
| 固定字符串前缀 | 所有数据集中在少数 Region | 哈希取模或 MD5 打散 |

**实战案例**

假设需要存储 IoT 设备数据，设计 RowKey：

```
方案1（推荐）：设备ID反转 + 时间戳反转
RowKey = (设备ID反转的MD5前4位) + (Long.MAX_VALUE - timestamp)

方案2：盐值 + 设备ID + 时间戳
RowKey = (设备ID % N 的随机前缀) + 设备ID + timestamp

方案3：业务字段拼接
RowKey = 业务前缀_日期_用户ID_Hash(设备ID)
```

**判断是否热点**

```bash
# 通过 HBase UI 或 Shell 查看 Region 分布
hbase> split 'tableName'   # 手动拆分热点 Region
hbase> merge_region 'encoded_region_name1', 'encoded_region_name2'  # 合并空 Region
```

## 题目三：HBase 的 MemStore 和 HFile 是如何工作的？为什么需要先写 MemStore 再刷盘？

### 参考答案

**MemStore 的作用**

MemStore 是 RegionServer 内存中的写缓存，核心作用：

1. **提供实时读取**：写入 MemStore 的数据立即可被读取（Read-your-writes 一致性）
2. **批量刷盘优化**：先将数据积累在内存，批量刷成 HFile，减少随机磁盘 IO
3. **数据排序**：MemStore 内部按 RowKey + Column Family + Column 有序排列，flush 后生成有序的 HFile

**MemStore 的刷盘（Flush）条件**

触发 flush 的条件（满足任一即触发）：
```
1. MemStore 大小达到阈值（hbase.hregion.memstore.flush.size = 128MB）
2. Region 下所有 MemStore 总大小超过 java heap 的 40%（可配置）
3. WAL 文件数量超过阈值（hbase.regionserver.max.logs）
4. 手动执行 flush 命令
5. 定期 flush（周期约 1 小时，防止数据长时间在内存）
```

**HFile 的底层结构**

HFile 是 HBase 的持久化存储格式，基于 **LSM-Tree（Log-Structured Merge Tree）**：

```
HFile 结构：
┌─────────────────────────────────┐
│  Data Block（数据块，默认 64KB）   │ ← 实际 KV 数据存储
│  Block Index（块索引）            │ ← 每个 Block 的 key 索引，二级稀疏索引
│  Bloom Filter（布隆过滤器）         │ ← 快速判断 key 是否存在
│  File Info（文件元信息）           │ ← LastKey、KeyCount 等
│  Trailer（尾部信息）               │ ← 偏移量指针
└─────────────────────────────────┘
```

**刷盘流程（从 MemStore 到 HFile）**

```
1. MemStore 达到阈值，RegionServer 发起 flush 请求
2. 将 MemStore 中的 KV 数据按列族生成新的 HFile 文件
3. 同时更新 MemStore 序列号（sequenceId）到 HFile Trailer
4. HFile 写入磁盘后，WAL 中对应的数据可以被安全丢弃
5. 后台定期进行 Minor Compaction（合并小 HFile）和 Major Compaction（全量合并）
```

**Compaction 机制**

| 类型 | 说明 |
|------|------|
| **Minor Compaction** | 合并少量小 HFile，删除标记的已删除数据，不清理过期版本 |
| **Major Compaction** | 合并一个 Store 下所有 HFile，删除所有过期数据和 tombstone 标记 |

**为什么先写 MemStore 而不是直接写 HFile？**

1. **写性能**：MemStore 是内存写，速度远快于随机磁盘写
2. **批量效应**：内存批量积累后顺序写 HFile，磁盘 IO 效率最大化
3. **LSM-Tree 设计**：牺牲读性能（需合并多 HFile）换取极致写性能
4. **WAL 保障**：数据已写入 WAL 确保持久化，MemStore 丢失可从 WAL 恢复

**常见调优参数**

```xml
<property>
  <name>hbase.hregion.memstore.flush.size</name>
  <value>134217728</value>  <!-- 128MB -->
</property>
<property>
  <name>hbase.regionserver.global.memstore.upperLimit</name>
  <value>0.4</value>  <!-- heap 的 40% -->
</property>
```

## 题目四：HBase 的协处理器（Coprocessor）是什么？有哪两类？各自的使用场景是什么？

### 参考答案

**协处理器概述**

HBase 协处理器是一种**运行在 RegionServer 端**的扩展机制，允许用户将自定义代码注入到 HBase 的读写路径中，在数据存储的**最近位置**执行计算，避免大量数据传输。

**两类协处理器**

### 1. Observer（观察者）—— 类似触发器

**原理**：在特定事件前后插入自定义逻辑，类似数据库的触发器。

| 类型 | 触发时机 | 常见用途 |
|------|----------|----------|
| **PreGetObserver** | 读取数据之前 | 权限校验、列级过滤 |
| **PostGetObserver** | 读取数据之后 | 修改返回结果、记录审计日志 |
| **PrePutObserver** | 写入数据之前 | 数据格式校验、写入前处理 |
| **PostPutObserver** | 写入数据之后 | 写入从表（二级索引）、更新统计 |
| **PreDeleteObserver** | 删除数据之前 | 级联删除关联数据 |
| **PostDeleteObserver** | 删除数据之后 | 清理关联数据 |
| **PreOpenObserver** | Region 打开前 | 初始化资源 |
| **PostOpenObserver** | Region 打开后 | 初始化后置处理 |
| **RegionObserver** | WAL 写入相关 | WAL 日志拦截 |

### 2. Endpoint（端点）—— 类似存储过程

**原理**：将自定义计算逻辑下沉到 RegionServer，在数据所在节点并行执行，结果返回客户端后聚合。

**典型场景**：
- 在 HBase 端执行聚合计算（COUNT、SUM、AVG），避免全量数据传输
- 自定义 scan 逻辑，如自定义过滤器
- 实现"存储过程"，封装复杂业务逻辑

**使用示例（Java）**

```java
// Observer 示例：数据写入后自动创建二级索引
public class IndexCoprocessor extends RegionObserver {
    @Override
    public void postPut(ObserverContext<RegionCoprocessorEnvironment> c,
                        Put put, WALEdit edit, Durability durability) {
        // 获取写入的 rowkey 和值
        byte[] rowkey = put.getRow();
        // 构造索引 Put，写入索引表
        Put indexPut = new Put("idx_" + new String(rowkey));
        // ... 填充索引数据 ...
        Table indexTable = c.getEnvironment().getTable(TableName.valueOf("index_table"));
        indexTable.put(indexPut);
    }
}
```

**加载方式**

```bash
# 方式1：通过 HBase Shell 加载（表级别）
hbase> disable 'myTable'
hbase> alter 'myTable', METHOD => 'table_att', 'COPROCESSOR' => 'hdfs:///hdfs/path/Coprocessor.jar|com.example.IndexCoprocessor|1001|arg1=xxx'
hbase> enable 'myTable'

# 方式2：通过 HTableDescriptor 配置（全局）
hTableDescriptor.setValue("COPROCESSOR_CLASSES", "...");
```

**注意事项**

- Observer 代码中的异常会导致 HBase 操作失败，需要做好异常处理
- Endpoint 调用的 JAR 包需放到 HBase 的 classpath 中
- 全局协处理器影响所有表，需谨慎使用
- Coprocessor 权限建议配合 HBase Security（ACL）使用，防止恶意代码

## 题目五：HBase 如何保证数据一致性和高可用？ WAL 和 Replication 机制的作用是什么？

### 参考答案

**一致性保证**

HBase 遵循 **CAP 理论** 中的 **CP（一致性+分区容错性）**，在网络分区时优先保证一致性：

| 级别 | 说明 |
|------|------|
| **强一致性** | 默认配置下，所有读写操作都保证数据一致性 |
| **写一致性** | 可配置 `Put.setDurability(Durability.ASYNC_WAL)` 牺牲持久化换性能 |
| **读一致性** | 通过 MVCC（多版本并发控制）保证，读取时只能看到已 flush 的数据 |

**MVCC 机制**

HBase 使用 MVCC 实现读写互不阻塞：
- 写入时分配全局递增的序列号（sequenceId）
- 读取时只能看到已提交（sequenceId <= 当前读事务）的数据
- 写入 MemStore 时记录 sequenceId，flush 和 compaciton 时按序列号过滤

**高可用（HA）保障**

### 1. WAL（Write-Ahead Log）机制

**核心思想**：先写日志再写内存，宕机后通过日志恢复数据。

```
WAL 写入流程：
1. 数据先写入 WAL buffer（顺序写磁盘）
2. WAL 持久化成功后，才写入 MemStore
3. 若 RegionServer 宕机，WAL 未刷盘的数据可通过日志重放恢复
4. 每个 Region 有一个独立的 WAL 文件（HLog）
```

**WAL 配置选项**：

```java
Put put = new Put(rowkey);
// SYNC_WALS：同步刷盘（默认，最安全）
// ASYNC_WALS：异步刷盘（高性能，可能丢数据）
// SKIP_WAL：不写 WAL（最快，但宕机丢失数据）
put.setDurability(Durability.SYNC_WALS);
```

### 2. HBase Replication（主从复制）

**作用**：跨集群/跨数据中心的数据同步，实现容灾和高可用。

```
主集群（Master） →  从集群（Slave）
    ↓                     ↓
  RegionServer         RegionServer
    ↓                     ↓
   WAL                  异步拉取并回放
```

**复制类型**：

| 类型 | 说明 |
|------|------|
| **同步复制（Symmetric Replication）** | 主从双向同步，适合跨机房容灾 |
| **异步复制（Async Replication）** | 主到从异步复制，有延迟但性能好 |

**开启 Replication 示例**：

```bash
# 添加 peer
hbase> add_peer '1', "zk1,zk2,zk3:/hbase"
# 启用表复制
hbase> disable 'sourceTable'
hbase> alter 'sourceTable', {NAME => 'cf', REPLICATION_SCOPE => '1'}
hbase> enable 'sourceTable'
```

### 3. RegionServer 故障恢复

```
RegionServer 宕机后的恢复流程：
1. ZooKeeper 检测到心跳超时（默认 3 分钟）
2. HMaster 将该 RegionServer 上的所有 Region 标记为 offline
3. HMaster 重新分配这些 Region 到其他活着的 RegionServer
4. 被分配的 RegionServer 先从 WAL 恢复未持久化的数据
5. Region 上线，开始接收请求
```

**容灾最佳实践**

1. **同城双活**：同机房部署两个 HBase 集群，通过 Replication 双向同步
2. **异地容灾**：跨地域部署，从集群开启异步复制，主集群故障时切换
3. **定期快照**：对重要业务表做 snapshot，保留历史版本
