## 题目1：Java基础 | try-catch-finally 执行顺序

**【答案】**

try-catch-finally 的执行顺序是 Java 中经典的考点，以下是详细规则：

### 1. 基本执行顺序
- **try 块**：首先执行，可能抛出异常的代码放在其中
- **catch 块**：当 try 块中发生异常时，按顺序匹配每个 catch 子句的异常类型，匹配成功则执行该 catch 块
- **finally 块**：无论是否发生异常、是否捕获异常，finally 块中的代码**总是会执行**

### 2. catch 匹配规则
- 按 catch 子句声明的异常类型**从浅到深**匹配（子类在前，父类在后）
- 一旦匹配成功，后续 catch 块不再执行
- 如果没有匹配到任何 catch 子句，异常会向上抛出，finally 仍会执行

### 3. finally 的特殊场景
- **return 在 try 或 catch 中**：finally 仍会执行（return 值被保存，等 finally 执行完后才返回）
- **System.exit()**：直接终止 JVM，finally 不会执行
- **守护线程被 kill**：finally 可能不执行
- **死循环**：如果 finally 中有无限循环，前面的 return 不会生效

### 4. 示例分析

```java
public class Test {
    public static void main(String[] args) {
        System.out.println(test());
    }
    
    public static int test() {
        try {
            return 1;
        } catch (Exception e) {
            return 2;
        } finally {
            System.out.println("finally");
        }
    }
}
// 输出：
// finally
// 1
```

```java
public static int test2() {
    try {
        int a = 1 / 0; // 抛出 ArithmeticException
        return 1;
    } catch (NullPointerException e) {
        return 2;
    } catch (Exception e) {
        return 3;
    } finally {
        System.out.println("finally");
    }
}
// 输出：
// finally
// 3
```

### 5. JDK7+ try-with-resources
- 资源在 try 语句中声明，自动调用 `close()` 方法
- 即使 try 块提前 return，**在 close() 抛异常时，原异常会被抑制，close() 异常会被addSuppressed()**

## 题目2：Java进阶 | synchronized 锁升级过程

**【答案】**

JDK6 引入了"锁升级"（Lock Escalation）机制来优化锁性能，避免一开始就直接使用重量级锁。synchronized 的锁升级过程是不可逆的，只能从低到高升级，不能降级。

### 锁的四种状态（无锁 → 偏向锁 → 轻量级锁 → 重量级锁）

#### 1. 无锁（No Lock）
- 对象没有被加锁，mark word 的 bit 字段表示对象无锁状态

#### 2. 偏向锁（Biased Lock）
- **适用场景**：一个线程反复进入同步块（无竞争）
- **原理**：将 mark word 指向线程 ID，以后该线程进入同步块只需检查 mark word 是否指向自己，无需再加锁
- **开启方式**：JDK5 默认关闭，JDK6 默认开启，可通过 `-XX:+UseBiasedLocking` 开启
- **撤销**：当有其他线程尝试获取偏向锁时，偏向锁会被撤销，升级为轻量级锁

#### 3. 轻量级锁（Lightweight Lock）
- **适用场景**：多个线程交替进入同步块（轻度竞争），但不适用两个线程同时竞争
- **原理**：
  - 线程在执行同步块前，在自己的栈帧中创建"锁记录"（Lock Record），将 mark word 复制到锁记录
  - CAS 操作尝试将对象头的 mark word 替换为指向锁记录的指针
  - 成功则获得轻量级锁，失败则自旋等待（自适应自旋）
- **自旋**：线程不阻塞，而是空转等待，通过忙循环重试 CAS
- **升级条件**：自旋次数超过阈值（默认 10 次），或等待线程数超过 CPU 核心数一半，升级为重量级锁

#### 4. 重量级锁（Heavyweight Lock）
- **原理**：依赖操作系统的 Mutex Lock，需要用户态切换到内核态，开销很大
- **状态**：线程被阻塞，等待操作系统调度
- **特点**：不再自旋，线程直接进入阻塞队列

### 锁升级流程图

```
无锁 → (偏向锁开启) → 偏向锁 → (其他线程竞争) → 轻量级锁 → (自旋超过阈值) → 重量级锁
```

### 逃逸分析结合
- 如果分析出对象不会逃逸出方法或线程，JIT 编译器可能会消除 synchronized 锁（锁消除）
- 还可以进行锁粗化：将多个连续的加锁操作合并为一次

## 题目3：Spring Boot | @Configuration 和 @Bean 的区别与底层原理

**【答案】**

### 1. 基本区别

| 特性 | @Configuration | @Bean |
|------|--------------|------|
| 作用对象 | 类 | 方法 |
| 配合注解 | 标注在配置类上 | 标注在 @Configuration 类的方法上 |
|  bean 名称 | 默认类名首字母小写 | 默认方法名 |
| 作用 | 声明式定义 bean | 编程式定义 bean |

### 2. @Configuration 的核心作用：防止配置类方法重复执行

```java
@Configuration
public class AppConfig {
    @Bean
    public UserService userService() {
        return new UserService();
    }
    
    @Bean
    public OrderService orderService() {
        return new OrderService(userService()); // 调用 userService() 实际返回同一个实例
    }
}
```

如果 @Configuration 标注的类没有 proxyBeanMethods=true 或者没有标注 @Configuration（只是 @Component），则 userService() 调用会创建一个新实例而非获取容器中的单例 bean。

### 3. 底层原理：CGLIB 动态代理

Spring 对标注 @Configuration 的类会创建**子类代理**：
- Spring 通过 CGLIB 生成 AppConfig 的子类，重写所有 @Bean 方法
- 代理方法会先检查容器中是否已有该 bean
- 如果有，直接返回容器中的 bean；如果没有，才调用父类方法创建并注册到容器

### 4. @Configuration vs @Component 对比

```java
// @Component 的配置类
@Component
public class AppConfig {
    @Bean
    public UserService userService() {
        return new UserService(); // 每次调用都创建新对象！
    }
    
    @Bean
    public OrderService orderService() {
        return new OrderService(userService()); // 会创建新的 UserService 实例
    }
}

// @Configuration 的配置类
@Configuration
public class AppConfig {
    @Bean
    public UserService userService() {
        return new UserService(); // 只执行一次，容器单例
    }
}
```

### 5. proxyBeanMethods 参数

- `proxyBeanMethods = true`（默认）：Full 模式，启用代理，调用 @Bean 方法返回单例
- `proxyBeanMethods = false"：Lite 模式，不创建代理，调用 @Bean 方法每次返回新对象（用于提升性能，减少不必要的代理开销）

## 题目4：Spring Cloud | Sentinel 和 Hystrix 熔断器对比

**【答案】**

Sentinel 和 Hystrix 都是流量控制和熔断降级的组件，但设计理念和实现细节有显著区别。

### 1. 核心设计理念

| 特性 | Sentinel | Hystrix |
|------|----------|--------|
| 设计理念 | 流量控制 + 熔断降级 + 系统自适应保护 | 熔断降级为主，延迟容忍和容错 |
| 作者 | 阿里中间件团队 | Netflix（已停止维护） |
| 文档状态 | 活跃维护 | 停止维护 |

### 2. 熔断策略对比

#### Sentinel 熔断策略（三种）

**① 慢调用比例（SLOW_REQUEST_RATIO）**
- 设置最大响应时间阈值 RT
- 如果一个请求的响应时间超过 RT 计为慢调用
- 当慢调用比例 ≥ 阈值时，熔断器打开，后续请求被拒绝
- 熔断恢复：经过熔断时长后，变为半开状态，放行一个请求测试

**② 异常比例（ERROR_ratio）**
- 当异常比例 ≥ 阈值时，熔断器打开
- 单位统计时长内请求量 ≥ 最小请求数才生效

**③ 异常数（ERROR_COUNT）**
- 当异常数 ≥ 阈值时，熔断器打开

#### Hystrix 熔断策略
- 基于异常比例熔断
- 熔断器打开后，经过一段时间（Sleep Window）变为半开状态
- 半开状态放行部分请求，如果成功则关闭熔断器，否则继续打开

### 3. 限流算法对比

| 算法 | Sentinel | Hystrix |
|------|----------|--------|
| 限流算法 | 滑动窗口计数器、令牌桶、漏桶、直接拒绝、热点参数限流 | 只支持线程池限流 |
| 线程隔离 | 信号量隔离（不创建线程池） | 线程池隔离 / 信号量隔离 |

### 4. Sentinel 的独特优势

- **支持热点参数限流**：可对入参的特定值进行精准限流
- **系统自适应限流**：根据 CPU、LOAD、入口 QPS 等自动计算限流阈值
- **基于响应时间的熔断**：结合慢调用比例，比纯异常比例更合理
- **Dashboard 控制台**：实时查看规则生效情况，支持推送规则到 Nacos 等配置中心
- **注解埋点**：提供 @SentinelResource 注解，支持资源定义和异常处理分离

### 5. 总结

Sentinel 是 Hystrix 的继任者，功能更全面，性能更好，且仍在持续维护。Sentinel 强调**流量控制**，而 Hystrix 强调**线程池隔离**。在 Spring Cloud Alibaba 体系中，推荐使用 Sentinel 替代 Hystrix。

📖 来源：https://developer.aliyun.com/article/712545

## 题目5：Spring Cloud | Nacos 配置中心原理

**【答案】**

Nacos 是阿里开源的配置中心和服务发现组件，核心实现基于 **发布-订阅** 机制和 **长轮询**。

### 1. 配置管理核心概念

- **命名空间（Namespace）**：用于隔离不同环境（dev/test/prod）
- **配置组（Group）**：用于隔离不同业务线或模块
- **配置ID（Data ID）**：具体某份配置文件的标识

### 2. 配置读取原理

#### 客户端启动时
1. 客户端连接 Nacos Server，建立 HTTP 长连接
2. 从服务端获取配置信息（GET /v1/cs/configs）
3. 获取成功后将配置内容缓存在本地文件（容灾）

#### 客户端监听配置变更
Nacos 使用**长轮询 + 通知**机制感知配置变更：

```
客户端 → 长轮询请求 → Nacos Server
         ↓ (无变化，等待maxTime)
         ← 返回 304 (无变化通知)
         
服务端配置变更 → 通知对应客户端 → 客户端重新拉取配置 → 触发刷新回调
```

### 3. 长轮询（Long Polling）机制详解

Nacos Server 收到配置变更请求后：
1. 不立即返回，先将请求挂起
2. 遍历该客户端关联的所有配置，检查是否有变更
3. 发生变更或超时（默认 30s）才返回
4. 客户端收到返回后，**立即**发起下一次长轮询请求

### 4. 配置变更推送原理（Server 主动 push）

```
┌─────────────┐         ┌──────────────────┐         ┌─────────────┐
│   Client    │ ←──────→ │  Nacos Server    │ ←──────→ │   Client    │
│  (订阅方)   │  订阅    │  (配置中心)       │   发布    │  (另一实例) │
└─────────────┘         └──────────────────┘         └─────────────┘
     ↑                            ↑
     │ (长轮询/配置变更通知)      │ (配置变更事件)
     └────────────────────────────┘
```

Nacos 使用 JDK 的 `ScheduledThreadPoolExecutor` 管理长轮询任务，服务端配置变更时，通过内部事件机制通知所有持有该配置长轮询的客户端。

### 5. 客户端容灾

当 Nacos Server 宕机时，客户端会：
1. 读取本地缓存文件恢复配置（防止配置丢失）
2. 继续重试连接服务端
3. 连接恢复后立即同步最新配置

### 6. 配置变更监听示例（Spring Boot 注解方式）

```java
@RefreshScope
@RestController
public class ConfigController {
    @Value("${custom.property:default}")
    private String customProperty;
    
    // 当 Nacos 配置变更后，会自动刷新 customProperty 的值
    // 同时触发 Bean 的重新创建（@RefreshScope 作用域）
}
```

## 题目6：Kafka | 分区分配策略（Partition Assignment Strategy）

**【答案】**

Kafka 通过分区分配策略决定哪个消费者负责哪个分区，直接影响消费者的负载均衡。

### 1. 消费者位移管理

- 消费者使用 Offset 记录消费进度
- 存储在 `__consumer_offsets` 主题（Kafka 内部主题）
- 提交位移方式：`手动提交` 或 `自动提交`（默认 5s 间隔）

### 2. 三种分区分配策略

#### 策略一：Range（范围策略）
- **针对每个主题**单独分配，不跨主题
- 计算方式：`n = 分区数 % 消费者数`，前 n 个消费者多分配一个分区
- **问题**：如果同时消费多个主题，可能造成负载不均

```
示例：主题A有10个分区，3个消费者
10 % 3 = 1
- 消费者0：分区0-3 (4个)
- 消费者1：分区4-7 (4个)
- 消费者2：分区8-9 (2个)
```

#### 策略二：RoundRobin（轮询策略）
- 将所有主题的分区混合在一起，轮询分配
- 优点：负载更均衡，适合多主题消费场景
- 缺点：可能导致同一分区的消息顺序跨主题混乱

#### 策略三：StickyAssignor（粘性策略）- **Kafka 0.11+**
- 在 RoundRobin 基础上增加**粘性**特性
- 重平衡（Rebalance）时，尽可能保持原有的分区分配不变
- 优点：减少 Rebalance 时的重复消费和消息乱序
- 重平衡触发场景：
  - 消费者加入或离开
  - 分区数量变更
  - 订阅主题变更

### 3. Rebalance（再平衡）机制

Rebalance 是消费者组内分区所有权变化的过程：
1. **JoinGroup**：所有消费者加入组
2. **SyncGroup**：Leader 消费者制定分配方案并同步给其他消费者
3. **Heartbeat**：正常工作时通过心跳维持组关系
4. **LeaveGroup**：消费者主动离开或被移除

### 4. 分区分配优先级

- 可通过 `partition.assignment.strategy` 配置指定多个策略（逗号分隔）
- 按配置顺序优先级依次尝试，直到成功

## 题目7：RabbitMQ | 消息确认机制（Confirm / Ack）

**【答案】**

RabbitMQ 提供完善的消息确认机制确保消息可靠投递，包括**发布确认**和**消费确认**两个维度。

### 1. 消费端确认（手动 Ack）

#### 确认模式
- **autoAck = true**（自动确认）：消息投递给消费者后立即删除
- **autoAck = false**（手动确认）：等待消费者调用 `basicAck()` 才删除消息

#### Ack 确认方式
- `basicAck(deliveryTag, false)`：确认消息，已正确处理
- `basicNack(deliveryTag, false, true)`：拒绝消息，requeue=true 表示重新入队
- `basicReject(deliveryTag, false)`：单条拒绝，等价于 `basicNack` 不带 requeue 参数

#### 消息确认流程
```
生产者 → 交换机 → 队列 → 消费者
                        ↓
                 等待 basicAck()
                        ↓
              消息从队列删除
```

### 2. 发布端确认（Publisher Confirm）

- 启用：`channel.confirmSelect()`
- 确认机制：
  - **同步 confirm**：每发送一条消息，调用 `waitForConfirms()` 等待确认
  - **异步 confirm**：通过 `addConfirmListener()` 注册回调，批量接收确认结果

#### 确认结果
- `ack = true`：Broker 成功接收
- `ack = false` + `nackReason`：失败，通常是因为磁盘已满或队列达上限

### 3. 事务模式（Transaction）- 不推荐
- 通过 `txSelect()` 开启事务
- 每条消息都需要 Broker 确认，性能极差（比普通模式慢 100 倍）
- **已被 Publisher Confirm 替代**

### 4. 消息可靠性保障方案

```
                    ┌──────────────┐
                    │  本地消息表   │ ← 定期重试投递失败的消息
                    └──────┬───────┘
                           ↓
┌──────────┐      ┌────────────┐      ┌─────────┐      ┌──────────┐
│ Producer │ ───→ │  RabbitMQ   │ ───→ │  Queue  │ ───→ │ Consumer │
└──────────┘      │ (持久化配置) │      │(持久化)  │      │ (手动Ack) │
                  └────────────┘      └─────────┘      └──────────┘
```

保障要点：
1. **生产者开启 Publisher Confirm**：确认消息成功到达 Broker
2. **队列和消息开启持久化**：`durable=true`，重启不丢失
3. **消费者手动 Ack**：确保消息处理完成后才确认
4. **本地消息表兜底**：投递失败的消息入库，定时任务重试

📖 来源：https://www.rabbitmq.com/documentation.html

## 题目8：Spring | Spring 循环依赖解决方案

**【答案】**

循环依赖（Cycle Dependency）是 Spring 中经典的问题：A 依赖 B，B 依赖 A，形成死循环。

### 1. Spring 三级缓存解决循环依赖

Spring 使用三级缓存（Three-Level Cache）来解决构造器注入以外的循环依赖：

#### 三级缓存结构
```
singletonObjects        ── 一级缓存：成熟 Bean（完全初始化）
earlySingletonObjects  ── 二级缓存：提前曝光的 Bean（已实例化，未完成属性填充）
singletonFactories      ── 三级缓存：Bean 工厂（用于生成提前曝光的引用）
```

#### Bean 创建流程

```
创建 A：
1. 调用 getSingleton("A") → 一二三级缓存都查不到 → 开始创建 A
2. 添加 A 的工厂到三级缓存 `singletonFactories.put("A", factory)`
3. 实例化 A（new A()）
4. 属性填充 B：调用 getSingleton("B")
   → 查不到 B → 开始创建 B
   → 添加 B 的工厂到三级缓存
   → 实例化 B
   → 属性填充 A：调用 getSingleton("A")
      → 从三级缓存拿到 A 的工厂，生成早期引用
      → 放入二级缓存 `earlySingletonObjects.put("A", aEarly)`
      → 删除三级缓存 `singletonFactories.remove("A")`
   → B 属性填充完成 → B 初始化完成 → 放入一级缓存
5. A 属性填充 B（从一级缓存拿到 B）→ A 初始化完成
6. 一级缓存放入 A
```

### 2. 为什么需要二级缓存（earlySingletonObjects）

- 避免通过工厂每次生成新的早期引用
- 保证同一个 Bean 的早期引用在整个创建过程中唯一

### 3. 构造器注入循环依赖 – 无法解决

```java
@Component
public class A {
    private final B b;
    public A(B b) { this.b = b; } // 构造器注入
}

@Component
public class B {
    private final A a;
    public B(A a) { this.a = a; } // 构造器注入
}
```
Spring 无法解决构造器循环依赖，会抛出 `BeanCurrentlyInCreationException`。

### 4. setter 属性注入循环依赖 – 可以解决

通过三级缓存，setter 注入（属性注入）的循环依赖可以被解决。

### 5. prototype 作用域的循环依赖 – 无法解决

prototype 作用域每次 getBean 都创建新实例，不缓存，循环依赖无法被解决。

### 6. 实际工作中的解决方案

1. **使用 @Lazy 延迟加载**：打破启动时的初始化顺序
2. **使用构造器 + @Autowired(required=false)**：允许依赖可选
3. **重新设计**：通过中间服务或事件机制解耦
4. **使用 setter 注入替代构造器注入**

## 题目9：AI Agent | RAG (Retrieval Augmented Generation) 检索增强生成原理

**【答案】**

RAG 是当前 AI Agent 中最核心的技术之一，将外部知识库检索与 LLM 生成结合，解决大模型知识过时/幻觉问题。

### 1. RAG 核心工作流程

```
用户问题 → 检索系统 → 相关文档 → Prompt 组装 → LLM 生成 → 最终答案
                ↑
           向量数据库
           (文档 Embedding)
```

### 2. 检索阶段（Retrieval）

#### 向量化（Embedding）
- 将文档切分为 Chunk（块）
- 使用 Embedding 模型（如 text2vec、BGE）将每个 Chunk 转为向量
- 存入向量数据库（Milvus、Pinecone、Chroma、Faiss 等）

#### 相似度搜索
- 用户问题同样通过 Embedding 模型转为向量
- 在向量数据库中做 **ANN（Approximate Nearest Neighbor）** 搜索
- 计算余弦相似度 / 点积，找出 Top-K 个最相关 Chunk

#### 检索优化策略
- **Query 重写**：将用户问题改写为更适合检索的形式
- **Hybrid Search**：结合关键词检索（BM25）和向量检索
- **Rerank**：初筛后用 Rerank 模型（如 BGE-Reranker）重新排序

### 3. 生成阶段（Generation）

将检索到的 Chunk 作为上下文，连同用户问题一起组装成 Prompt：

```
[系统提示] 你是一个助手，基于以下参考文档回答问题：
---
{检索到的文档1}
{检索到的文档2}
---
用户问题：{用户原始问题}
```

LLM 基于上下文 + 问题生成答案，有效减少幻觉。

### 4. RAG 的局限性

- **上下文长度限制**：LLM 上下文窗口有限（4K-128K）
- **检索质量依赖**：检索到的文档不相关，生成质量就差
- ** Embedding 模型选择**：不同语言场景需选择合适的 Embedding 模型
- **实时性问题**：新知识需要重新索引才能被检索到

### 5. RAG 与 Fine-tuning 对比

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 知识更新 | 实时更新（改文档） | 需要重新训练 |
| 成本 | 低（检索 + 调用 LLM） | 高（GPU 训练） |
| 可解释性 | 高（可追溯来源） | 低（知识压缩在参数中） |
| 幻觉问题 | 大幅减少 | 减少但仍有 |
| 适用场景 | 知识库问答、实时性要求高 | 特定风格/领域微调 |

## 题目10：算法 | 二叉树的最大深度（LeetCode 104）

**【答案】**

### 题目描述
给定一个二叉树 root，返回其最大深度。二叉树的最大深度是从根节点到最远叶子节点的最长路径上的节点数。

### 解法一：深度优先搜索（DFS）- 递归

```java
public int maxDepth(TreeNode root) {
    if (root == null) {
        return 0;
    }
    int leftDepth = maxDepth(root.left);
    int rightDepth = maxDepth(root.right);
    return Math.max(leftDepth, rightDepth) + 1;
}
```

**复杂度分析**
- 时间复杂度：O(n)，每个节点访问一次
- 空间复杂度：O(h)，h 为树的高度（递归栈深度，最坏 O(n)，平衡树 O(log n)）

### 解法二：广度优先搜索（BFS）- 层序遍历

```java
public int maxDepth(TreeNode root) {
    if (root == null) {
        return 0;
    }
    Queue<TreeNode> queue = new LinkedList<>();
    queue.offer(root);
    int depth = 0;
    
    while (!queue.isEmpty()) {
        int size = queue.size();
        // 遍历当前层的所有节点
        for (int i = 0; i < size; i++) {
            TreeNode node = queue.poll();
            if (node.left != null) queue.offer(node.left);
            if (node.right != null) queue.offer(node.right);
        }
        depth++;
    }
    return depth;
}
```

**复杂度分析**
- 时间复杂度：O(n)
- 空间复杂度：O(m)，m 为最大层的节点数（最坏 O(n)）

### 解法三：迭代（先序遍历 + 栈）

```java
public int maxDepth(TreeNode root) {
    if (root == null) return 0;
    
    int maxDepth = 0;
    Stack<Pair<TreeNode, Integer>> stack = new Stack<>();
    stack.push(new Pair<>(root, 1));
    
    while (!stack.isEmpty()) {
        Pair<TreeNode, Integer> pair = stack.pop();
        TreeNode node = pair.getKey();
        int depth = pair.getValue();
        maxDepth = Math.max(maxDepth, depth);
        if (node.left != null) stack.push(new Pair<>(node.left, depth + 1));
        if (node.right != null) stack.push(new Pair<>(node.right, depth + 1));
    }
    return maxDepth;
}
```

### 核心思路

树的最大深度 = 1 + max(左子树最大深度, 右子树最大深度)

### 扩展：LeetCode 559（最大深度 - N 叉树）

```java
public int maxDepth(Node root) {
    if (root == null) return 0;
    if (root.children.isEmpty()) return 1;
    
    int maxChildDepth = 0;
    for (Node child : root.children) {
        maxChildDepth = Math.max(maxChildDepth, maxDepth(child));
    }
    return maxChildDepth + 1;
}
```
