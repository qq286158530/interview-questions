## 题目一：HBase 的整体架构是怎样的？请描述各组件的作用。

### 参考答案

HBase 是一个基于 Google BigTable 思想设计的分布式列式存储系统，其整体架构如下：

**1. ZooKeeper**
- 负责 HBase Master 的选举与协调
- 存储元数据信息（如 .META. 表的位置）
- 提供分布式锁机制，保证集群状态一致性

**2. HMaster**
- 是 HBase 的主节点，负责：
  - 管理 Region 的分配与调度
  - 处理建表、删表、修改表结构等 DDL 操作
  - 监控 RegionServer 的健康状态
  - 在 RegionServer 故障时进行 Region 迁移
- HMaster 可以部署多个实现高可用，但同一时刻只有一个主 Master 生效

**3. HRegionServer**
- 是 HBase 的工作节点，负责：
  - 管理用户数据的读写请求
  - 合并 MemStore、刷写 HFile 等操作
  - 向 HMaster 汇报自己的状态
- 每个 HRegionServer 包含多个 HRegion，每个 HRegion 对应一段连续的行键区间

**4. HRegion**
- 是 HBase 中数据存储和调度的基本单位
- 每个 HRegion 存储一段行键区间 `[startKey, endKey)` 的数据
- 当某个 HRegion 过大（默认超过 10GB）时，会自动触发分裂（Split）

**5. Store**
- 每个 HRegion 包含多个 Store，每个 Store 对应一张表的一个列族（Column Family）
- 每个 Store 由一个 MemStore 和多个 HFile 组成

**6. HFile**
- 是 HBase 数据在 HDFS 上的物理存储格式
- 采用 LSM-Tree（Log-Structured Merge-Tree）结构，写入先写 MemStore，满了后刷写成 HFile
- HFile 内部采用 B+Tree 索引，支持高效顺序扫描

**7. .META. 表**
- 记录所有 Region 和 RegionServer 的映射关系
- 类似 Redis 的 slot 概念，是 HBase 定位数据的关键

## 题目二：HBase 的 RowKey 设计有哪些原则？如何避免热点（Hotspotting）问题？

### 参考答案

**RowKey 设计原则**

**1. 长度原则**
- RowKey 长度不宜过长，建议控制在 10~100 字节之间
- 过长的 RowKey 会占用大量 MemStore 内存，影响存储和查询效率

**2. 唯一性原则**
- RowKey 必须唯一标识一行数据，类似主键
- 业务主键组合后作为 RowKey（如 `userId_orderId`）

**3. 散列（Hash）原则**
- 避免让 RowKey 有某种固定顺序（如时间戳递增），导致数据集中在少数几个 Region 上
- 可以对原始 RowKey 做 MD5 哈希或盐（Salt）处理，使数据均匀分布到所有 Region

**4. 业务相关性原则**
- 将经常被查询的字段放在 RowKey 前面，充分利用 HBase 字典序排序的特性进行范围查询

**如何避免热点（Hotspotting）**

| 策略 | 说明 | 示例 |
|------|------|------|
| 加盐（Salting） | 在 RowKey 头部拼接随机前缀（0~N 的数字）| `salting_userid` |
| 哈希（Hashing） | 对 RowKey 取 MD5/SHA-1 的前几位作为前缀 | `md5(id)[:8]_userid` |
| 反转（Reversing） | 将有规律的字段反转（如手机号、邮箱）| 反转手机号 |
| 时间戳反转 | 查询"最近数据"时，将时间戳用 MAX - timestamp | `MAX_TS - timestamp_userid` |
| 随机分配 | 业务允许时使用 UUID 作为 RowKey | `UUID` |

**热点场景示例**
- ❌ 错误：`20260621_userId` — 每天的数据集中在同一个 Region
- ✅ 正确：`md5(userId)[:4]_{date}_{userId}` — 数据打散

## 题目三：简述 HBase 的数据写入流程，并解释 MemStore 和 HFile 的作用。

### 参考答案

**数据写入流程（Put 操作）**

```
客户端 → ZooKeeper → 获取 .META. 表位置 → 定位目标 RegionServer → 写入
```

**具体步骤：**

1. **定位 Region**：客户端先请求 ZooKeeper 获取 .META. 表位置，查询目标 RowKey 属于哪个 RegionServer
2. **写入 WAL**：数据先写入 Write-Ahead Log（WAL），即 HLog，存放在 RegionServer 本地磁盘。WAL 用于在 RegionServer 故障时恢复数据
3. **写入 MemStore**：数据写入成功后，放入对应 Store 的 MemStore 内存缓冲区（按 RowKey 排序）
4. **响应客户端**：写入 MemStore 后即返回客户端成功（此时数据还未持久化到磁盘）
5. **后台刷写**：当 MemStore 达到阈值（默认 128MB）时，触发 Flush 刷写成 HFile
6. **HFile 合并**：随着 HFile 增多，后台会进行 Minor Compaction（小合并）和 Major Compaction（大合并），合并后生成更大的 HFile

**MemStore 的作用**
- 内存缓冲区，写入会先进入 MemStore，按 RowKey 排序（类似有序集合）
- 支持单行数据多次写入的合并（Mutations）
- 刷写后生成 HFile，是数据持久化的中间状态

**HFile 的作用**
- HBase 数据在 HDFS 上的物理存储格式
- 基于 LSM-Tree 结构：追加写（Append-Only），通过后台合并提升读写性能
- 包含数据块、索引块、布隆过滤器（Bloom Filter）等，可以快速定位数据

## 题目四：HBase 的 LSM-Tree 结构是什么？它相比 B-Tree 有什么优势？

### 参考答案

**LSM-Tree（Log-Structured Merge-Tree）概述**

LSM-Tree 是一种针对写多读少场景优化的数据结构，最早由 O'Neil 等人提出，被 HBase、RocksDB、Cassandra 等系统广泛采用。

**LSM-Tree 的三层结构**

```
MemTable（内存）→ Immutable MemTable → SSTable/HFile（磁盘）
     ↑ 刷写                ↑ 合并              ↑ 合并
   写入                  满了切换             定期合并
```

1. **MemTable**：内存中的有序数据结构（跳表/SkipList），接收新写入
2. **Immutable MemTable**：当 MemTable 写满后，切换为只读的 Immutable MemTable，同时创建新的 MemTable 接收写入
3. **SSTable/HFile**：Immutable MemTable 刷写到磁盘形成 HFile，多个 HFile 定期合并成更大的 HFile

**LSM-Tree 的合并策略**
- **Minor Compaction**：合并少量小的 HFile，生成一个稍大的 HFile
- **Major Compaction**：合并所有 HFile，生成一个完整的 HFile，并删除过期/删除的数据

**LSM-Tree vs B-Tree 对比**

| 特性 | LSM-Tree | B-Tree |
|------|----------|--------|
| 写放大 | 低（顺序写）| 高（随机写 + 页面分裂）|
| 读放大 | 高（可能需要检查多层）| 低（一次定位）|
| 空间放大 | 较高（有重叠数据）| 低 |
| 写性能 | 极高（顺序写）| 一般（随机写）|
| 适用场景 | 写多读少（如日志、消息队列）| 读写均衡 |
| 典型系统 | HBase、RocksDB、Cassandra | InnoDB、Oracle |

**HBase 中 LSM-Tree 的体现**
- 写入：先写 WAL（日志）再写 MemStore，完全是顺序追加写，性能极高
- 读取：需要先查 MemStore，再从新到旧逐层查 HFile
- 布隆过滤器：HFile 中包含 BloomFilter，可以快速判断某个 RowKey 是否不存在，减少无效 IO

## 题目五：HBase 协处理器（Coprocessor）是什么？有哪两类？各自的使用场景是什么？

### 参考答案

**协处理器概述**

HBase 协处理器是一种允许用户在 RegionServer 上运行自定义代码的机制，类似于数据库的触发器（Trigger）和存储过程（Stored Procedure）。它让数据处理逻辑尽可能靠近数据本身，减少网络传输，提升性能。

**两类协处理器**

**1. Observer（观察者）—— 类似触发器**
- 挂载在特定的钩子（Hook）上，当这些事件发生时自动触发执行
- **应用场景**：
  - `PreGet` / `PostGet`：在数据读取前后执行，可用于实现列级权限控制、审计日志
  - `PrePut` / `PostPut`：在数据写入前后执行，可用于数据验证、实时计算、触发通知
  - `PreDelete` / `PostDelete`：在删除前后执行，可用于删除关联数据
  - `PreOpen` / `PostOpen`：Region 打开时执行，可用于初始化或加载数据
  - `PreSplit` / `PostSplit`：Region 分裂前后执行，可用于数据再分配逻辑

**2. Endpoint（端点）—— 类似存储过程**
- 允许在 RegionServer 上执行服务端计算，用户自定义一个 RPC 服务端方法
- 客户端通过 RPC 调用，在每个相关 Region 上执行，然后将结果返回客户端聚合
- **应用场景**：
  - 服务端聚合计算：如在 RegionServer 上直接执行 `SUM`、`COUNT`、`AVG` 等聚合操作，减少数据传输
  - 复杂过滤：将复杂的过滤逻辑下推到数据节点，减少网络 IO
  - 跨行事务：在多个 Region 上执行原子操作（需要结合 WAL 保证）

**协处理器的加载方式**

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| 加载到表 | 通过 HBase Shell 或 API 为单张表加载 | 开发测试 |
| 加载到集群 | 在 `hbase-site.xml` 中配置，作用于所有表 | 生产环境全局逻辑 |

**注意事项**
- 协处理器代码需打成 JAR 包，并放到 HDFS 或 RegionServer 本地
- 错误的协处理器可能导致 RegionServer 崩溃，需谨慎测试后再上线
- Observer 是同步执行的，会阻塞主请求；Endpoint 是异步 RPC 调用
