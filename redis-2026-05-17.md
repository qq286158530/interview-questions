## 题目一：Redis 有哪几种数据类型？它们的典型使用场景是什么？

### 答案

Redis 支持 5 种基础数据类型，外加 3 种特殊类型：

| 数据类型 | 底层实现 | 典型使用场景 |
|---------|---------|------------|
| **String** | SDS（简单动态字符串） | 缓存、计数器、分布式锁、限流 |
| **Hash** | 压缩列表 / 哈希表 | 存储对象、购物车、用户画像 |
| **List** | 快速链表 / 压缩列表 | 消息队列、关注列表、最新消息流 |
| **Set** | 哈希表 / 整数集合 | 标签系统、好友关系、去重 |
| **Sorted Set** | 跳跃表 + 哈希表 | 排行榜、延时队列、有序通知 |
| **Bitmaps** | 位图 | 用户签到、在线统计 |
| **HyperLogLog** | 基数估算 | UV 统计 |
| **Geospatial** | GeoHash | 附近的人、LBS 服务 |

**示例代码：**

```python

GET page:view:20250517

# Hash: 用户信息

HGETALL user:10001

# List: 消息队列
LPUSH queue:tasks '{"job":"send_email","id":1}'
BRPOP queue:tasks 0   # 阻塞弹出

# Sorted Set: 排行榜
ZADD leaderboard 1500 "玩家A" 1200 "玩家B" 1800 "玩家C"
ZREVRANGE leaderboard 0 9 WITHSCORES   # 获取 Top 10

# Set: 标签

SISMEMBER tags:article:101 "Redis"   # 是否包含某标签
```

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 答案

Redis 有两种持久化方式，可以同时启用，也可以单独使用。

### RDB（Redis Database）

- **原理**：定时将内存中的**全量数据**快照（point-in-time snapshot）写入二进制文件 `dump.rdb`
- **触发方式**：
  - `SAVE`（同步阻塞，阻塞所有客户端）
  - `BGSAVE`（fork 子进程异步执行，推荐）
  - 自动触发：`save 900 1 / save 300 10 / save 60 10000`（满足任一条件即触发）
  - `FLUSHALL` + `AOF` 关闭时不会生成 RDB
- **优点**：恢复速度快，适合冷备份
- **缺点**：可能丢失最后一次快照之后的数据

### AOF（Append Only File）

- **原理**：每执行一条写命令，就追加到 `appendonly.aof` 文件末尾
- **同步策略**（`appendfsync`）：
  - `always`：每条命令都 fsync，最安全但最慢
  - `everysec`（默认）：每秒执行一次 fsync，折中方案
  - `no`：由操作系统决定何时同步，最快但可能丢失较多数据
- **优点**：数据安全性更高
- **缺点**：AOF 文件通常比 RDB 大，恢复较慢
- **重写机制**：`BGREWRITEAOF` 异步重写 AOF 文件，移除过期/重复命令，压缩文件体积

### 对比总结

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件体积 | 小 | 大 |
| 恢复速度 | 快 | 慢 |
| 数据完整性 | 可能丢失数据 | 可配置（everysec 最多丢1秒） |
| fork 压力 | 大（全量快照） | 小（增量追加） |
| 适用场景 | 冷备、恢复快 | 要求高数据可靠性 |

### 生产环境推荐配置

```conf
# RDB
save 900 1
save 300 10
save 60 10000

# AOF
appendonly yes
appendfsync everysec
aof-rewrite-incremental-fsync yes   # 边写边刷盘，防止阻塞
auto-aof-rewrite-percentage 100      # AOF 文件超过上次重写后大小的 100% 时触发重写
auto-aof-rewrite-min-size 64mb      # 最小 64MB 才触发重写
```

## 题目三：Redis 的过期 key 删除策略有哪些？什么是缓存穿透、缓存击穿、缓存雪崩？如何应对？

### 答案

### 过期 key 删除策略

Redis 同时使用**惰性删除**和**定期删除**两种策略：

1. **惰性删除（Lazy Expiration）**
   - 每当访问一个 key 时，Redis 检查是否过期，过期则删除
   - 优点：对 CPU 友好，不影响正常请求
   - 缺点：过期 key 过多又长时间不访问，会占用内存

2. **定期删除（Active Expiration）**
   - Redis 每隔一段时间（默认每 100ms）随机抽取一批设置了过期时间的 key，检查并删除过期的
   - `hz` 参数控制每秒执行多少次过期扫描（默认 10，可设 1~100）

> Redis 不使用定时器（timer）删除，因为大量 key 同时过期会阻塞主线程。

### 三大缓存问题

#### 1. 缓存穿透
**定义**：查询一个**根本不存在**的数据，缓存没有，数据库也没有，请求直接打到数据库。

**危害**：大量无效请求击穿数据库。

**解决方案：**
- **布隆过滤器（BloomFilter）**：在缓存层前加一层过滤器，不存在的 key 直接返回
- **缓存空值**：将 `null` 结果也缓存起来（设置较短 TTL）
- **参数校验**：接口层做基础合法性校验

```python
# 布隆过滤器方案示例
def get_user(uid):
    if not bloom_filter.might_contain(uid):      # 布隆过滤器判断
        return None                               # 直接返回，不查缓存和DB
    cache_val = redis.get(f"user:{uid}")
    if cache_val:
        return json.loads(cache_val)
    db_val = db.query("SELECT * FROM users WHERE uid=%s", uid)
    if db_val:
        redis.setex(f"user:{uid}", 3600, json.dumps(db_val))
    return db_val
```

#### 2. 缓存击穿
**定义**：某个**热点 key 突然过期**，瞬间大量并发请求同时击穿缓存，直接打爆数据库。

**解决方案：**
- **互斥锁（Mutex）**：只允许一个请求去加载数据，其他等待
- **永不过期 + 异步更新**：key 不设置 TTL，后台异步更新
- **逻辑过期**：存储时带逻辑过期时间，命中后异步更新

```python
# 互斥锁方案
def get_user(uid):
    cache_val = redis.get(f"user:{uid}")
    if cache_val:
        return json.loads(cache_val)
    # 尝试获取锁
    lock = redis.set("lock:user:" + uid, "1", nx=True, ex=5)
    if lock:
        db_val = db.query(...)
        redis.setex("user:" + uid, 3600, json.dumps(db_val))
        redis.delete("lock:user:" + uid)
        return db_val
    else:
        time.sleep(0.1)
        return get_user(uid)   # 等待后重试
```

#### 3. 缓存雪崩
**定义**：大量 key **在同一时间集中过期**，导致大量请求同时穿透到数据库。

**解决方案：**
- **过期时间加随机值**：`TTL = baseTTL + random(0, 600)`
- **多级缓存**：本地缓存 + Redis + 数据库
- **Redis 高可用**：主从 + 哨兵 / 集群
- **服务熔丝 / 限流**：Guava / Sentinel / 计数器限流

```python
# 过期时间加随机值
ttl = base_ttl + random.randint(0, 600)  # 基础 TTL + 0~10分钟随机
redis.setex(f"user:{uid}", ttl, json.dumps(data))
```

### 总结对比

| 问题 | 原因 | 核心解决思路 |
|------|------|------------|
| 缓存穿透 | 查询不存在的数据 | 布隆过滤器 / 缓存空值 |
| 缓存击穿 | 热点 key 过期 | 互斥锁 / 永不过期 / 逻辑过期 |
| 缓存雪崩 | 大量 key 同时过期 | 随机 TTL / 多级缓存 / 高可用 |

## 题目四：Redis 主从复制原理是什么？说说你对哨兵（Sentinel）和集群（Cluster）的理解。

### 答案

### 主从复制原理

Redis 主从复制采用 **PSYNC**（Partial Sync）机制，支持全量同步和部分同步：

#### 全量同步流程
1. 从节点保存主节点 IP + Port，向主节点发送 `PSYNC ? -1`
2. 主节点执行 `BGSAVE` 生成 RDB 快照，同时记录后续所有写命令到 **replication buffer**
3. RDB 生成后发送给从节点
4. 从节点清空本地数据，加载 RDB
5. 主节点将 replication buffer 中的命令发给从节点（命令传播）
6. 从节点执行这些命令，与主节点保持同步

#### 部分同步流程（适用于网络闪断等场景）
- 主节点维护一个 **replication buffer**（环形缓冲区）持续记录命令
- 从节点断线重连后，发送 `PSYNC <runid> <offset>`
- 若 runid 匹配且 offset 在缓冲区范围内，主节点只需发送缺失的增量命令

#### 关键配置

```conf
replicaof 192.168.1.100 6379    # 从节点配置主节点
replica-read-only yes            # 从节点只读
```

### 哨兵（Sentinel）

**作用**：监控主从节点健康、自动故障转移、通知客户端新主节点地址。

**工作原理：**
1. **监控**：Sentinel 每秒向主从节点发送 PING，超过 `down-after-milliseconds` 判定为主观下线
2. **投票**：多个 Sentinel 投票，超过 quorum 数量认定客观下线
3. **故障转移**：
   - 在从节点中选举一个作为新主节点
   - 其他从节点改为复 制新主节点
   - 通知客户端更新主节点地址
4. **配置更新**：Sentinel 自身也是集群，有 Leader 负责执行故障转移

```conf
sentinel monitor mymaster 192.168.1.100 6379 2   # 监控 mymaster，quorum=2
sentinel down-after-milliseconds mymaster 30000
sentinel parallel-syncs mymaster 1               # 每次同时同步的从节点数
```

**适用场景**：中小型业务，主从模式，需要自动故障转移。

### 集群（Cluster）

**作用**：数据分片（slot），支持横向扩展，多主多从，高可用。

**工作原理：**
- **16384 个 slot**：每个 key 通过 CRC16(key) % 16384 映射到某个 slot
- **每个主节点负责一部分 slot**：如 3 主 3 从，每主约 5460 个 slot
- **节点间通信**：使用 Gossip 协议传播集群状态
- **故障转移**：同 Sentinel 机制，PFAIL -> FAIL -> 选举新主

```
┌─────────┐        ┌─────────┐        ┌─────────┐
│ Master A│────────│ Master B│────────│ Master C│
│(slot 0~5460) │    │(slot 5461~10922)│ (slot 10923~16383)│
│  ↓复制  │        │  ↓复制   │       │  ↓复制   │
│ Replica │        │ Replica  │      │ Replica  │
└─────────┘        └─────────┘        └─────────┘
```

**客户端路由**：客户端可直连任意节点，内部通过 `MOVED` 重定向到正确节点（智能客户端会缓存 slot 映射）。

**写入限制**：默认情况下，**只支持单 key 事务**（因为 key 可能在不同节点），多 key 操作需使用 `Hash Tag` 将相关 key 映射到同一 slot：

```python
# Hash Tag 确保 user:1001 和 user:1001:profile 在同一 slot
# {user:1001}:name 和 {user:1001}:profile 都会路由到同一节点
```

| 特性 | 主从 + Sentinel | Cluster |
|------|----------------|---------|
| 数据分片 | 无（所有数据一份） | 有（16384 slot） |
| 写能力 | 单一主节点 | 多主节点 |
| 扩容 | 纵向（内存更大） | 横向（加节点） |
| 复杂度 | 低 | 高 |
| 适用规模 | < 100G | > 100G 或需高并发写入 |

## 题目五：什么是 Redis 内存淘汰策略？当内存满时会发生什么？如何排查内存泄漏？

### 答案

### 内存淘汰策略（Eviction Policy）

当 Redis 内存使用达到 `maxmemory` 上限时，Redis 根据配置的淘汰策略决定**清除哪些 key**。

**淘汰策略共 8 种：**

| 策略 | 含义 |
|------|------|
| `noeviction` | 不淘汰，返回错误（默认） |
| `volatile-lru` | 从设置了过期时间的 key 中，使用 LRU 算法淘汰 |
| `allkeys-lru` | 从所有 key 中，使用 LRU 算法淘汰 |
| `volatile-lfu` | 从设置了过期时间的 key 中，使用 LFU 算法淘汰 |
| `allkeys-lfu` | 从所有 key 中，使用 LFU 算法淘汰 |
| `volatile-random` | 从设置了过期时间的 key 中随机淘汰 |
| `allkeys-random` | 从所有 key 中随机淘汰 |
| `volatile-ttl` | 从设置了过期时间的 key 中，淘汰 TTL 最短的 |

```conf
maxmemory 2gb
maxmemory-policy allkeys-lru
```

**LRU vs LFU 区别：**
- **LRU（Least Recently Used）**：最近最少使用，淘汰最久未访问的
- **LFU（Least Frequently Used）**：最不经常使用，淘汰访问频率最低的（更精准，热点更稳定）

### 内存满时会发生什么？

1. 若 `maxmemory-policy` 为 `noeviction`：所有写操作（`SET`、`INCR`、`HSET`、`LPUSH` 等）返回 `OOM` 错误
2. 若设置了淘汰策略：Redis 按策略删除某些 key 后，再执行写操作
3. `DEL` 操作即使内存满也会执行
4. 读操作（`GET`）不受影响

```bash
# 查看当前内存使用
redis-cli INFO memory | grep used_memory_human

# 查看淘汰统计
redis-cli INFO stats | grep evicted_keys
```

### 常见内存泄漏原因

| 原因 | 描述 | 排查方法 |
|------|------|---------|
| **大 Key** | 单个 key 存储了几十 MB 甚至几 GB 数据 | `redis-cli --bigkeys` 扫描 |
| **热 Key 过多** | 大量 key 同时过期导致雪崩 | 分析 keyspace 分布 |
| **客户端泄漏** | 客户端没有关闭连接，连接数持续增长 | `redis-cli CLIENT LIST` |
| **AOF 重写失败** | AOF 重写失败导致临时文件不删除 | 检查磁盘空间和 AOF 状态 |
| **Lua 脚本泄漏** | 脚本中的全局变量未清理 | 审查 Lua 脚本 |
| **Monitor 长时间运行** | `MONITOR` 命令消耗大量内存 | 避免在生产环境使用 |

### 大 Key 排查实战

```bash
# 扫描大 key（scan 方式，对生产影响小）
redis-cli --bigkeys

# 分析 string 类型 key 大小
redis-cli DEBUG OBJECT ENCODING "bigkey"
redis-cli MEMORY USAGE "bigkey"   # 精确字节数

# 找出内存占用最高的 key
redis-cli --memkeys | head -20

# 离线分析 RDB 文件
redis-cli --rdb dump.rdb --bytes 102400000 | grep "biggest"
```

### 内存优化建议

- **对大对象使用分桶**：将大 Hash/Set 拆成多个小 key
- **设置合理的过期时间**：避免大量 key 集中过期
- **使用合适的编码**：`ziplist`、`intset` 比 raw 更省内存
- **开启 LRU 淘汰策略**：确保内存满时淘汰冷数据
