## 题目一：Redis 和 Memcached 有什么区别？为什么 Redis 比 Memcached 更适合做缓存？

### 参考答案

**Redis 与 Memcached 的共同点：**
- 都是基于内存的数据库，一般都用来当做缓存使用
- 都有过期策略
- 两者的性能都非常高

**Redis 与 Memcached 的区别：**

| 特性 | Redis | Memcached |
|------|-------|-----------|
| 数据类型 | 支持 String、Hash、List、Set、ZSet 等丰富数据类型 | 只支持简单的 key-value 数据类型 |
| 持久化 | 支持数据持久化，可以将内存中的数据保持在磁盘中 | 没有持久化功能，数据全部存在内存中 |
| 集群模式 | 原生支持集群模式（主从复制、哨兵、切片集群） | 没有原生集群模式，需要客户端实现分片 |
| 高级特性 | 支持发布订阅模型、Lua 脚本、事务等 | 不支持 |

**为什么 Redis 更适合做缓存？**

1. **高性能**：Redis 的 QPS（每秒处理请求数）可达 10W+，是 MySQL 的 10 倍，而 Memcached 虽也很快，但数据类型单一限制了其应用场景

2. **高并发**：直接访问 Redis 能够承受的请求远大于直接访问 MySQL，可以把数据库中的部分数据转移到缓存中，分担数据库压力

3. **丰富的数据结构**：Redis 提供的多种数据类型可以更优雅地解决业务问题，如用 Set 实现点赞、共同关注等功能

4. **数据安全**：Redis 支持 RDB、AOF、混合持久化，宕机后可快速恢复数据

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？该如何选择？

### 参考答案

**Redis 的三种持久化方式：**

1. **RDB（Redis Database）**：将某一时刻的内存数据，以二进制的方式写入磁盘
2. **AOF（Append Only File）**：每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里
3. **混合持久化**：Redis 4.0 新增，集成了 RDB 和 AOF 的优点

**RDB 与 AOF 的区别：**

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件格式 | 二进制 | 文本命令 |
| 恢复速度 | 快（直接读入内存） | 慢（需要执行所有命令） |
| 数据完整性 | 可能丢失较多数据 | 可配置丢失少量数据 |
| 文件大小 | 较小（紧凑） | 较大（包含所有命令） |
| 性能影响 | fork子进程，不阻塞主线程 | 始终有一定影响 |

**AOF 的三种写回策略：**

- `always`：每次写操作执行完后同步将 AOF 日志写回硬盘，可靠性最高，性能最差
- `everysec`：每隔一秒将缓冲区内容写回硬盘，默认策略，平衡方案
- `no`：由操作系统控制写回时机，性能最好，但数据安全性最低

**RDB 的触发方式：**
- 手动执行 `save`（阻塞）或 `bgsave`（非阻塞）命令
- 配置自动触发：`900秒内至少1次修改` 或 `300秒内至少10次修改` 或 `60秒内至少10000次修改`

**混合持久化原理：**
- AOF 重写时，先将以 RDB 格式写入文件前半部分
- 然后将增量命令以 AOF 格式写入后半部分
- 重启恢复时先加载 RDB 内容，再加载增量 AOF 内容

**选择建议：**
- 如果需要快速启动和快速恢复，优先选择 RDB
- 如果对数据安全性要求高，选择 AOF 或混合持久化
- 生产环境推荐混合持久化，兼顾启动速度和安全性

## 题目三：Redis 是单线程吗？为什么单线程的 Redis 还能保持高性能？

### 参考答案

**Redis 确实是单线程的吗？**

Redis 单线程指的是「接收客户端请求 → 解析请求 → 进行数据读写等操作 → 发送数据给客户端」这个过程是由一个线程（主线程）来完成的。

但是，Redis 程序并不是完全单线程的，Redis 在启动时会启动后台线程（BIO）：

- **Redis 2.6 版本**：启动 2 个后台线程，处理关闭文件、AOF 刷盘任务
- **Redis 4.0 版本之后**：新增 lazyfree 线程，异步释放内存

**后台线程任务：**

| 线程 | 任务 |
|------|------|
| BIO_CLOSE_FILE | 关闭文件任务队列 |
| BIO_AOF_FSYNC | AOF 刷盘任务队列 |
| BIO_LAZY_FREE | 释放内存任务队列 |

**为什么单线程的 Redis 还能保持高性能？**

1. **内存操作**：Redis 的大部分操作都在内存中完成，并且采用了高效的数据结构（SDS、跳表、压缩列表等），瓶颈可能是内存或网络带宽，而非 CPU

2. **避免了多线程开销**：单线程模型避免了多线程切换带来的时间开销，也不会有死锁问题

3. **I/O 多路复用**：采用 epoll/select/kqueue 等机制，允许一个线程处理多个 Socket 连接，当有请求到达时才处理，没有请求时则阻塞

**Redis 6.0 之后的变化：**
- 引入多线程处理网络 I/O（但命令执行仍是单线程）
- 主要用于提高网络 I/O 的并行度
- 官方表示性能提升至少一倍以上

## 题目四：Redis 的五种基本数据类型是什么？它们的底层数据结构分别是什么？

### 参考答案

**Redis 五种基本数据类型：**

| 数据类型 | 应用场景 | 底层数据结构 |
|----------|----------|--------------|
| String（字符串） | 缓存对象、常规计数、分布式锁、共享 session | SDS（简单动态字符串） |
| Hash（哈希） | 缓存对象、购物车 | 压缩列表 / 哈希表 |
| List（列表） | 消息队列、列表数据存储 | quicklist（3.2+版本） |
| Set（集合） | 点赞、共同关注、抽奖活动 | 哈希表 / 整数集合 |
| Zset（有序集合） | 排行榜、电话姓名排序 | 跳表 / 压缩列表 |

**各底层数据结构说明：**

**SDS（Simple Dynamic String）：**
- 获取字符串长度复杂度 O(1)
- 支持二进制数据（不仅能存文本，还能存图片、音频等）
- 拼接字符串安全，不会缓冲区溢出（自动扩容）

**压缩列表（Ziplist）：**
- 内存紧凑，节省空间
- 元素个数小于 512 且每个元素小于 64 字节时使用
- Redis 7.0 已废弃，改用 listpack

**哈希表（Hashtable）：**
- 时间复杂度 O(1)
- 元素不满足压缩列表条件时使用

**quicklist：**
- Redis 3.2 版本引入
- 双向链表 + 压缩列表的组合
- 兼顾插入/删除效率和存储空间

**跳表（Skip List）：**
- 平均查找时间复杂度 O(log N)
- 相比平衡树实现更简单
- Zset 不满足压缩列表条件时使用

**整数集合（IntSet）：**
- 只用于 Set 类型
- 集合元素都是整数且个数小于 512 时使用

## 题目五：Redis 如何实现服务高可用？主从复制和哨兵模式有什么区别？

### 参考答案

**Redis 高可用架构演进：**

1. **主从复制（Master-Slave）**：一主多从，读写分离
2. **哨兵模式（Sentinel）**：主从复制 + 自动故障转移
3. **切片集群（Cluster）**：数据分片，横向扩展

**主从复制原理：**

- 主服务器进行读写操作，将写操作同步给从服务器
- 从服务器只读，接受主服务器同步过来的写操作命令
- 命令复制是异步进行的（主服务器不等从服务器执行完就返回结果）

**主从复制的缺点：**
- 无法自动故障转移
- 手动恢复耗时
- 无法实现强一致性保证

**哨兵模式的作用：**

1. **监控**：哨兵进程监控主从服务器是否正常运行
2. **自动故障转移**：当主服务器宕机时，自动选举从服务器为新的主服务器
3. **通知**：将故障转移的结果通知给客户端

**哨兵核心概念：**

- **主观下线**：单个哨兵认为主服务器不可用
- **客观下线**：多个哨兵投票认为主服务器不可用
- **选举领导者**：哨兵们投票选出一个领导者来执行故障转移
- **故障转移**：选举条件最好的从服务器为新主服务器，其他从服务器改为复制新主服务器

**主从复制 vs 哨兵模式：**

| 特性 | 主从复制 | 哨兵模式 |
|------|----------|----------|
| 故障检测 | 无 | 有（主观/客观下线） |
| 故障自动转移 | 无 | 有 |
| 配置复杂度 | 简单 | 较复杂（至少3个哨兵） |
| 客户端感知 | 需要自行处理 | 哨兵通知客户端 |
| 适用场景 | 数据备份、读写分离 | 高可用要求 |

**切片集群（Redis Cluster）：**
- Redis 3.0 引入
- 通过哈希槽（16384个槽）将数据分布到多个节点
- 每个节点负责一部分哈希槽
- 支持自动故障转移和路由查询
