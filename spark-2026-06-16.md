## 题目 1：Spark 的架构是怎样的？Driver 与 Executor 之间如何协作？

### 参考答案

Spark 采用 **Master-Slave 架构**，核心组件包括：

| 组件 | 角色 |
|------|------|
| **Driver** | 客户端入口，负责构建 DAG、切割 Stage、调度 Task 到 Executor |
| **Executor** | 每个 Worker 节点上的进程，负责执行 Task 并汇报结果 |
| **Cluster Manager** | 资源管理器（Standalone / YARN / K8s），负责分配资源 |

**协作流程：**

1. 用户提交 Spark 作业，Driver 端构建 **DAG**（有向无环图）
2. DAG 经过**窄依赖/宽依赖**划分，被切分为多个 **Stage**
3. 每个 Stage 包含多个 **Task**，以 **TaskSet** 为单位发送到 Executor
4. Executor 接收 Task，在线程池中执行，完成后向 Driver 回报结果
5. Driver 汇总所有结果后，作业结束

**核心机制：**
- **DAG Scheduler**：将 DAG 切分为 Stage，构建 TaskSet
- **Task Scheduler**：将 Task 调度到集群 Executor
- **SchedulerBackend**：管理 Executor 连接，维护集群拓扑

> 📎 来源：[Apache Spark 官方文档 - Cluster Mode Overview](https://spark.apache.org/docs/latest/cluster-overview.html)

## 题目 2：RDD 的血缘机制（Lineage）是什么？如何实现容错？

### 参考答案

**血缘机制（Lineage）** 是 RDD 的核心特性之一，用于追踪 RDD 之间的依赖关系。

**原理：**

每个 RDD 都知道自己是从哪个父 RDD 经过什么 Transformation 产生的。例如：

```
rdd2 = rdd1.map(_ * 2).filter(_ > 10)
        └── Lineage: rdd2 <- filter <- map <- rdd1
```

**容错方式：**

| 情况 | 容错机制 |
|------|----------|
| **窄依赖**（map/filter） | 父 RDD 某分区丢失，只需重新计算该分区，效率高 |
| **宽依赖**（groupByKey/reduceByKey） | 父 RDD 某分区丢失，需要重新计算所有上游分区，可能触发**检查点（Checkpoint）** |

**Checkpoint vs Cache：**

- `cache()`：存在内存/磁盘，血缘链不断，用于反复访问
- `checkpoint()`：截断血缘链，结果写入磁盘，用于长链路容错

> 📎 来源：[Apache Spark 官方文档 - RDD Programming Guide](https://spark.apache.org/docs/latest/rdd-programming-guide.html#rdds)

## 题目 3：DataFrame 和 DataSet 的区别是什么？RDD、DataFrame、DataSet 如何选型？

### 参考答案

**三者的关系：**

```
RDD[Person]  (无schema，类型安全)
    ↓  encode
DataFrame = DataSet[Row]  (有schema，无类型安全)
    ↓  encode
DataSet[T]  (有schema，类型安全，Scala/Java)
```

**关键区别：**

| 特性 | DataFrame | DataSet |
|------|-----------|---------|
| **Schema** | 有（通过 Spark SQL 自动推断） | 有（编译时确定） |
| **类型安全** | ❌ 无（运行时才发现错误） | ✅ 有（编译时检查） |
| **语言支持** | 一切语言 | 仅 Scala/Java |
| **操作方式** | DSL / SQL | DSL / Lambda |

**选型建议：**

| 场景 | 推荐 |
|------|------|
| Python 用户，灵活分析 | DataFrame |
| Scala/Java，结构化数据，类型安全优先 | DataSet |
| 非结构化数据，低级操作 | RDD |
| 性能优先，数据处理管道 | DataFrame / DataSet（通过 Catalyst 优化） |

## 题目 4：Spark Shuffle 的原理是什么？为什么 Shuffle 是性能瓶颈？

### 参考答案

**Shuffle 的触发条件：**

当一个 RDD 操作的输出需要按照某种规则重新分区时（如 `groupByKey`、`reduceByKey`、`join`、`distinct`），就会触发 Shuffle。

**执行过程（以 ReduceByKey 为例）：**

```
Stage 1                              Stage 0
┌─────────┐                          ┌─────────┐
│ Map Task│ ── Shuffle Write ──→     │ Reduce   │
│ (分区1) │    ↓ 写入本地磁盘         │ Task    │
└─────────┘    (按照key hash分区)     └─────────┘
┌─────────┐                          ┌─────────┐
│ Map Task│ ── Shuffle Write ──→     │ Reduce   │
│ (分区2) │    ↓                     │ Task    │
└─────────┘                          └─────────┘
```

**Shuffle Write 阶段：**
- 每个 Map Task 将数据按 key 的 hash 值写入本地磁盘，生成 N 个文件（N = Reducer 个数）
- 涉及排序（Sort-Based Shuffle）和序列化，内存压力大

**Shuffle Read 阶段：**
- 每个 Reducer 通过网络抓取所有 Map 节点上属于它的数据分片
- 大量网络 I/O 和磁盘 I/O

**成为瓶颈的原因：**

1. **大量磁盘 I/O**：数据必须落盘，无法纯内存操作
2. **网络 I/O**：跨节点数据传输
3. **内存压力**：Shuffle Read 时数据可能需要在 JVM 堆外分配
4. **磁盘空间**：中间文件可能撑爆磁盘
5. **Task 重试**：Shuffle 数据丢失会导致整个 Stage 重算

**优化建议：**
- 提前在 Map 端做 **预聚合（Pre-aggregate）**：使用 `reduceByKey` 而非 `groupByKey`
- 增大 `spark.shuffle.file.buffer`（默认 32KB）
- 使用 **Sort-Based Shuffle**（Spark 2.x+ 默认）替代 Hash Shuffle
- 合理设置 `spark.sql.shuffle.partitions`（默认 200）

> 📎 来源：[Apache Spark 官方文档 - Shuffle Behavior](https://spark.apache.org/docs/latest/configuration.html#shuffle-behavior)

## 题目 5：Spark 的内存管理模型是怎样的？Executor 内存如何划分？

### 参考答案

Spark 采用**统一内存管理（Unified Memory）**模型（Spark 2.x+），将内存分为两大区域：

**内存区域划分：**

```
┌──────────────────────────────────────────────┐
│               Executor Memory                │
│  ┌──────────────────┐  ┌─────────────────┐  │
│  │   Execution      │  │    Storage      │  │
│  │   Memory         │  │    Memory       │  │
│  │  (Shuffle/排序)  │  │  (Cache/广播)   │  │
│  └──────────────────┘  └─────────────────┘  │
│  ┌──────────────────────────────────────┐   │
│  │        User Memory (RDD残留/用户代码) │   │
│  └──────────────────────────────────────┘   │
└──────────────────────────────────────────────┘
```

**各区域说明：**

| 区域 | 用途 | 配置参数 |
|------|------|----------|
| **Storage Memory** | 缓存 RDD、广播变量 | `spark.memory.storageFraction`（默认 0.5） |
| **Execution Memory** | Shuffle 排序、Hash Join、聚合缓冲区 | 动态借用 Storage |
| **User Memory** | 用户数据结构、元数据 | `1 - spark.memory.fraction`（默认 0.6） |

**动态协商机制：**

- Execution 内存不足时，可借用空闲的 Storage 内存（可驱逐）
- Storage 内存不足时，**不可**借用 Execution 内存（Shuffle 数据不能被驱逐）

**配置建议：**

```bash
# 典型配置示例
spark.executor.memory = 4g
spark.memory.fraction = 0.6       # 用于 Execution + Storage 的比例
spark.memory.storageFraction = 0.5  # Storage 占统一内存的比例
spark.driver.memory = 2g
```

> 📎 来源：[Apache Spark 官方文档 - Memory Management](https://spark.apache.org/docs/latest/configuration.html#memory-management)

## 题目 6：广播变量（Broadcast Variable）是什么？它解决了什么问题？

### 参考答案

**什么是广播变量：**

广播变量是只读的、分布式共享的变量。Driver 端将一份数据**打包一次**，发送给每个 Executor，Executor 将其缓存后供该节点上所有 Task 共享访问，无需每个 Task 各自拷贝一份。

**解决的问题：**

假设有 1000 个 Partition 的 `join` 操作，每个 Task 都拷贝一份小表（10MB），：
- **不使用广播**：1000 × 10MB = **10GB** 网络传输
- **使用广播**：1 × 10MB = **10MB** 网络传输

**使用方式：**

```python
# Python
broadcast_var = spark.sparkContext.broadcast(large_dict)

# 在 Task 中使用 .value 获取
result = data.filter(lambda x: x in broadcast_var.value)
```

```scala
// Scala
val broadcastVar = spark.sparkContext.broadcast(Map("a" -> 1, "b" -> 2))
broadcastVar.value
```

**注意事项：**

| 要点 | 说明 |
|------|------|
| 适用场景 | 小表/小数据集的 lookup，大变量跨 Task 共享 |
| **不要广播** | 大数据（>20MB 建议评估），频繁变化的变量 |
| 过期机制 | 广播变量在 Driver 端更新后需要重新广播 |
| 存储位置 | 默认存储在 Executor 的 Storage Memory 中 |

**原理简述：**
Spark 使用 **BitTorrent-like** 的广播策略，大 Executor 会分发数据给小 Executor，减少 Driver 压力。
