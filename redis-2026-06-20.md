## 题目一：Redis 持久化机制有哪些？各有何优缺点？

**参考答案：**

Redis 提供了三种持久化机制，用于将内存数据保存到磁盘，以防重启后数据丢失：

### 1. RDB（Redis Database Snapshot）

**原理：** 在指定时间间隔内，将 Redis 在某一时刻的内存数据快照（dump.rdb）写入磁盘。触发方式包括 `save`（同步阻塞）或 `bgsave`（fork 子进程异步执行）。

**优点：**
- 恢复速度快，二进制文件直接读入内存即可恢复
- 文件紧凑，适合备份和灾难恢复
- fork 子进程执行，不阻塞主线程

**缺点：**
- 全量快照，频率不好把控——太频繁影响性能，太稀疏会丢失较多数据
- 存在数据窗口：上次快照到故障之间的数据会丢失

---

### 2. AOF（Append Only File）

**原理：** 每执行一条写操作命令，就将该命令以追加方式写入一个日志文件。Redis 重启时，通过重放日志文件中的命令来恢复数据。

**三种写回策略（appendfsync）：**

| 策略 | 说明 | 安全性 | 性能 |
|------|------|--------|------|
| Always | 每个写命令执行完后同步写回磁盘 | 最安全，数据零丢失 | 最差 |
| Everysec | 每秒异步将缓冲区内容写回磁盘 | 最多丢1秒数据 | 较好 |
| No | 由操作系统决定何时写回 | 依赖系统，可能丢较多数据 | 最好 |

**优点：**
- 数据丢失风险低（可配置）
- AOF 重写机制（bgrewriteaof）可压缩文件，避免体积无限增长

**缺点：**
- 文件体积通常比 RDB 大
- 恢复速度比 RDB 慢（需逐条执行命令）

---

### 3. 混合持久化（Redis 4.0+）

**原理：** 开启混合持久化后，AOF 重写时，先将以 RDB 格式写入 AOF 文件前半部分，再将重写期间的增量命令以 AOF 格式追加到后半部分。

**优点：** 兼具 RDB 快速恢复和 AOF 少丢失的优点

**缺点：** AOF 可读性差，且不兼容 4.0 之前版本

## 题目二：Redis 的过期键删除策略有哪些？分别适用于什么场景？

**参考答案：**

Redis 有三种过期键删除策略：

### 1. 定时删除（TTL 定时器）

为每个设置了过期时间的 key 创建一个定时器，当过期时间到达时立即删除该 key。

**优点：** 对内存最友好，过期键第一时间被删除，不会占用内存  
**缺点：** 对 CPU 不友好，当有大量 key 同时过期时，定时器会占用大量 CPU资源，影响 Redis 响应性能

---

### 2. 惰性删除（Lazy Expiry）

放任过期键不管，当访问某个 key 时才检查其是否过期——如果过期则删除，否则返回该键。

**优点：** 对 CPU 友好，只有在真正访问时才会检查删除，不需要额外资源  
**缺点：** 对内存不友好，如果大量过期 key 从不被访问，它们会一直占用内存（内存泄漏风险）

---

### 3. 定期删除（Periodic Deletion）

每隔一段时间（默认 100ms），Redis 从设置了过期时间的 key 集合中随机选取一部分进行检查，删除其中已过期的 key。

**实现细节：**
- 每轮检查从所有带过期时间的 key 中随机抽取 `hz`（默认10）个
- 通过 `active-expire-effort` 参数可调整每次删除的数量上限
- 平衡了 CPU 和内存：对 CPU 不会造成瞬间压力，对内存也不会无限堆积

---

### 实际应用：Redis 采用组合策略

Redis 同时使用了**惰性删除 + 定期删除**两种策略：
- **惰性删除**在客户端访问 key 时触发，兜底处理漏网之鱼
- **定期删除**作为主动兜底，防止大量过期 key 堆积内存

> 注意：Redis 的 `lazyfree` 机制（大 key 异步删除）也是一种延迟删除手段，避免 del 命令阻塞主线程。

## 题目三：Redis 主从复制原理是什么？主节点宕机后如何保证高可用？

**参考答案：**

### 主从复制原理

Redis 主从复制采用 **一主多从** 架构，主节点负责写操作，从节点负责读操作（读写分离）。复制流程分为两个阶段：

#### 阶段一：全量复制（Full Sync）

1. 从节点发送 `PSYNC ? -1`（或 `PSYNC <runid> <offset>`）给主节点，请求同步
2. 主节点执行 `bgsave`，在后台生成 RDB 快照，同时将新来的写命令缓存到 **复制缓冲区（replication buffer）**
3. RDB 快照生成完毕后，主节点发送给从节点
4. 从节点清空本地数据，加载收到的 RDB 快照

#### 阶段二：增量复制（Command Propagation）

主节点将**复制缓冲区**中积累的增量命令发送给从节点（通过 `PSYNC` 的 offset 机制），此后主节点持续将写命令异步发送给从节点。

> 增量复制的核心是 **replication offset**（复制偏移量）和 **replication backlog buffer**（复制积压缓冲区），类似 MySQL 的 binlog 位置机制。

---

### 主节点宕机后如何保证高可用？

#### 方案一：哨兵模式（Sentinel）

哨兵是一个独立进程，用于监控主从节点的健康状态：

1. **监控：** 哨兵周期性向主从节点发送 PING，心跳检测存活
2. **自动故障转移：** 当主节点不可达时，哨兵自动从多个从节点中选举一个新主节点，其他从节点改为从属新主节点
3. **通知：** 故障转移后，哨兵通过发布订阅机制通知客户端新的主节点地址

> 哨兵集群本身也是多节点部署（至少 3 个），通过 Raft 共识算法保证决策正确性。

#### 方案二：Redis Cluster（切片集群）

Redis Cluster 是 Redis 3.0 引入的分布式方案：
- 数据被分为 16384 个 slot，通过 CRC16(key) % 16384 分配
- 每个主节点负责一部分 slot，并拥有自己的从节点（主从复制）
- 当某个主节点宕机，其从节点自动接管，整个过程对客户端透明

## 题目四：什么是 Redis 缓存穿透、缓存击穿、缓存雪崩？如何解决？

**参考答案：**

### 1. 缓存穿透

**问题描述：** 请求查询一个**既不存在于缓存、也不存在于数据库**的数据（如恶意用户用不存在的 ID 大量请求），导致请求直接打到数据库，数据库压力剧增。

**解决方案：**
- **布隆过滤器（Bloom Filter）：** 在缓存层前加一层布隆过滤器，所有查询先经过布隆过滤器过滤——如果布隆过滤器判断 key 不存在，直接返回，避免查库
- **缓存空值（Null Value）：** 即使查询结果为空，也将 `null` 写入缓存，并设置较短的 TTL（如 30 秒），防止同一 key 重复穿透
- **参数合法性校验：** 接口层对 ID 格式等做基础校验，拦截非法请求

---

### 2. 缓存击穿

**问题描述：** 某**热点 key 突然过期**，而此时有大量并发请求同时访问该 key，导致这些请求全部穿透到数据库（数据库被打爆）。

**解决方案：**
- **互斥锁（Mutex Lock）：** 缓存失效时，只允许一个请求去查库并重建缓存，其他请求等待锁释放后从缓存获取（Redis SETNX 实现分布式锁）
- **逻辑过期（不设置 TTL）：** 热点 key 不设置过期时间，而是在 value 中包含逻辑过期时间字段，后台线程异步更新缓存——所有请求都能读到旧数据，但不会击穿数据库
- **热点数据永不过期：** 定期由后台任务主动刷新热点 key，而不是依赖 TTL 自动失效

---

### 3. 缓存雪崩

**问题描述：** **大量缓存 key 在同一时间集中过期**，或者 Redis 宕机，导致大量请求直接打到数据库。

**解决方案：**
- **过期时间随机化：** 为缓存过期时间加上随机偏移量（`TTL + random(0, 3600)`），避免集中过期
- **多级缓存：** 构建本地缓存（Caffeine/Guava Cache）+ Redis 多级缓存架构，即使 Redis 失效，本地缓存也能兜底
- **Redis 高可用架构：** 使用 Redis Sentinel（哨兵）或 Redis Cluster 保证 Redis 自身的高可用
- **服务降级/熔断：** 当数据库压力过大时，通过 Hystrix/Sentinel 等熔断器对查询进行服务降级，返回兜底数据或友好提示
- **提前预热：** 系统启动时主动将热点数据加载到缓存，避免启动后瞬间大量请求击穿

## 题目五：Redis 的内存淘汰策略有哪些？什么是 LRU 和 LFU？

**参考答案：**

### 为什么要内存淘汰？

当 Redis 实际使用的内存超过 `maxmemory` 配置时，Redis 需要主动删除一些 key 以释放内存空间，这就需要内存淘汰策略（Eviction Policy）。

### 八种内存淘汰策略

| 策略 | 说明 |
|------|------|
| **noeviction** | 不淘汰任何 key，满后拒绝写入（默认策略） |
| **volatile-lru** | 在设置了过期时间的 key 中，使用 LRU 算法淘汰 |
| **allkeys-lru** | 在所有 key 中，使用 LRU 算法淘汰 |
| **volatile-lfu** | 在设置了过期时间的 key 中，使用 LFU 算法淘汰 |
| **allkeys-lfu** | 在所有 key 中，使用 LFU 算法淘汰 |
| **volatile-random** | 在设置了过期时间的 key 中，随机淘汰 |
| **allkeys-random** | 在所有 key 中，随机淘汰 |
| **volatile-ttl** | 在设置了过期时间的 key 中，淘汰 TTL 最短的（即最快过期的）|

---

### LRU（Least Recently Used）— 最近最少使用

**核心思想：** 如果一个数据最近被访问，那么将来被访问的概率也更高。LRU 淘汰最近最久未被访问的 key。

**Redis 中的 LRU 实现：**
- Redis 使用采样方式近似实现 LRU（不是严格 LRU），通过 `maxmemory-samples` 配置采样数量（默认 5）
- 采样方式比严格 LRU 节省大量内存，且效果接近严格 LRU

---

### LFU（Least Frequently Used）— 最不经常使用

**核心思想：** 如果一个数据在最近被访问的次数少，那么将来被访问的频率也更低。LFU 淘汰访问频率最低的 key。

**Redis 中的 LFU 实现（Redis 4.0+）：**
- 每个 key 维护一个 **访问计数器（counter）** 和 **访问时间（last decrement time）**
- 访问计数器采用对数增长机制：每次访问 counter += 1；但 counter 不会无限增长，会随时间衰减
- 通过 `lfu-decay-time`（默认 1 分钟）控制衰减速度——多久不访问 counter 减半
- 相比 LRU，LFU 更适合区分"热点"和"温点"数据，避免一次性热点将历史热点挤出

---

### 如何选择策略？

- **热点数据优先保留** → allkeys-lru 或 volatile-lru
- **访问频率区分明显** → allkeys-lfu 或 volatile-lfu
- **所有 key 一视同仁** → allkeys-random
- **希望 key 自然过期** → volatile-ttl
- **数据不允许丢失** → noeviction（但需确保内存充足）

> 实际生产中，`allkeys-lru` 是最常用的策略，尤其在缓存场景下。
