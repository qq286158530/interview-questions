## 题目一：HBase 的整体架构是怎样的？包含哪些核心组件？

### 参考答案

HBase 是一个基于 Google BigTable 论文设计的分布式列族数据库（Column-Family NoSQL），整体架构如下：

**1. 核心组件**

| 组件 | 角色 | 说明 |
|------|------|------|
| **ZooKeeper** | 协调服务 | 存储 ROOT/META 表位置、选主（Master Election）、心跳检测 |
| **HMaster** | 主节点 | 负责表结构变更、Region 分裂/合并/迁移、负载均衡。元数据管理，不参与实际数据读写 |
| **HRegionServer** | 数据节点 | 负责管理 Region（数据分片），处理读写请求。是实际数据操作的工作节点 |
| **HRegion** | 数据分片 | 一张表按 RowKey 范围横向切分后的数据子集，每个 Region 负责一段 RowKey 区间 |
| **Store** | 列族存储 | 每个 Region 按列族（Column Family）分成多个 Store，每个 Store 含一个 MemStore + 多个 HFile |
| **HFile** | 数据文件 | HBase 在 HDFS 上的实际存储格式（基于 SSTable），含索引、BloomFilter、数据块等 |
| **MemStore** | 写缓存 | 数据写入时先写入 MemStore，满了之后 Flush 成 HFile（ Minor Compaction / Major Compaction）|
| **WAL（Write-Ahead Log）** | 预写日志 | 数据写入 MemStore 前先写 WAL，保证宕机后数据可恢复 |

**2. 读写流程**

```
# 写数据流程
Client → ZooKeeper → 获取 META 表位置 → 定位 RegionServer → 写 WAL → 写 MemStore → 返回 ACK

# 读数据流程
Client → ZooKeeper → 获取 META 表位置 → 定位 RegionServer → 先查 MemStore（未命中）→ 查 BlockCache → 查 HFile → 合并结果返回
```

**3. 一级索引与二级索引**

- **一级索引**：RowKey 本身就是主索引，按 RowKey 字典序排序，支持快速点查和范围扫描
- **二级索引**：HBase 本身不支持，需借助协处理器（Coprocessor）或外部组件（Phoenix、ElasticSearch）实现

## 题目二：HBase 的 RowKey 如何设计？有哪些原则和技巧？

### 参考答案

RowKey 是 HBase 唯一的主键，直接决定数据分布和查询效率。设计原则如下：

**1. 核心设计原则**

| 原则 | 说明 | 反例 |
|------|------|------|
| **长度适中** | 建议 16~100 字节，越短越好 | 过长的 RowKey 浪费内存与存储 |
| **唯一性** | 确保唯一标识一行数据 | 重复导致数据覆盖 |
| **散列性** | 避免热点（HotSpot），数据均匀分布到各 RegionServer | 所有数据集中在一个 Region |
| **可排序** | 按字典序排列，支持范围扫描 | 设计不考虑排序场景 |

**2. 常见 RowKey 设计技巧**

```
① 盐值（Salting）加随机前缀，分散热点
   RowKey = MD5(row).substring(0,1) + original_key
   缺点：无法范围扫描

② 哈希（Hashing）反转
   将时间戳反转：original_key + reversed_timestamp
   适合写多读少、且按时间范围查询的场景

③ 组合列（Composite Key）
   RowKey = userId + orderId + timestamp
   适合多维度查询，例如：查某用户的所有订单

④ 分桶（Bucket）
   预分区 + 固定前缀，保证数据均匀分布
   例如：region = hash(key) % num_regions
```

**3. 热点问题解决方案**

```sql
-- 示例：避免时间戳热点（将时间戳放在 RowKey 前面）
-- 错误写法（连续时间戳造成热点）
RowKey = timestamp + userId

-- 正确写法（反转时间戳 或 加盐）
RowKey = userId + reversed_timestamp
RowKey = (timestamp % N) + userId  -- N 为预分区数量
```

## 题目三：HBase 的存储结构是怎样的？MemStore 和 HFile 的关系是什么？

### 参考答案

**1. HBase 存储结构分层**

```
MemStore（写缓存）
    ↓ Flush（触发条件见下）
HFile（SSTable，在 HDFS 上）
    ↓ Compaction（合并）
更大的 HFile
```

**2. MemStore 详解**

- 数据写入时先写入 MemStore（内存结构，按 RowKey/Column 有序）
- 每个 Store 对应一个 MemStore
- Flush 触发条件（满足任一即触发）：
  - MemStore 大小达到 `hbase.hregion.memstore.flush.size`（默认 128MB）
  - Region 中所有 MemStore 总大小超过 `hbase.regionserver.global.memstore.size`（默认堆内存 40%）
  - WAL 文件数量超过 `hbase.regionserver.max.logs`
  - 定期 Flush（`hbase.regionserver.flush.interval`，默认 1 小时）
- Flush 时按 Column Family 生成 HFile

**3. HFile 结构（SSTable 格式）**

```
HFile 结构：
┌─────────────────────────────┐
│  Data Block（数据块）        │  ← 存储 KeyValue 对，可配置大小（默认 64KB）
├─────────────────────────────┤
│  Meta Index（块索引）        │  ← 每个 Data Block 的起始 RowKey 索引
├─────────────────────────────┤
│  BloomFilter（布隆过滤器）   │  ← 快速判断某 RowKey/Column 是否存在，减少磁盘 IO
├─────────────────────────────┤
│  File Info（文件信息）       │  ← TimeRange、KV 数量、Min/Max RowKey 等
├─────────────────────────────┤
│  Trailer（尾部信息）         │  ← 指向各区域的偏移量
└─────────────────────────────┘
```

**4. Compaction 机制**

| 类型 | 说明 |
|------|------|
| **Minor Compaction** | 将多个小 HFile 合并成少量大 HFile，不删除过期/删除数据 |
| **Major Compaction** | 将一个 Store 下所有 HFile 合并成一个，**删除所有过期和被标记删除的数据**（TTL / Version）|

**5. BlockCache 读缓存**

- LRU 缓存，读取时将 HFile 数据块缓存于此
- 读请求顺序：MemStore → BlockCache → HFile
- 命中 BlockCache 可极大提升读取性能

## 题目四：HBase 的协处理器（Coprocessor）是什么？有哪些使用场景？

### 参考答案

**1. 什么是 Coprocessor**

Coprocessor 是 HBase 提供的扩展机制，允许用户将自定义逻辑**嵌入到 HBase Server 端**执行，无需修改 HBase 核心代码。

**2. 两种协处理器类型**

| 类型 | 说明 | 典型场景 |
|------|------|----------|
| **Observer**（观察者）| 类似数据库触发器，在特定钩子点插入自定义逻辑 | 数据校验、副本同步、审计日志、二级索引 |
| **Endpoint**（端点）| 类似存储过程，可在 Server 端执行自定义计算并返回结果 | 聚合计算（求和/平均）、复杂过滤逻辑 |

**3. Observer 常用钩子**

```
Observer 钩子列表：
├── prePut / postPut           → 写数据前/后触发
├── preDelete / postDelete     → 删除数据前/后触发
├── preGet / postGet           → 读数据前/后触发
├── preOpen / postOpen         → Region 打开前/后触发
├── preFlush / postFlush       → MemStore Flush 前/后触发
└── preCompact / postCompact   → Compaction 前/后触发
```

**4. 二级索引实现示例（使用 Coprocessor）**

```
场景：为 user_id 列建立二级索引

实现思路：
1. 写数据时，prePut 拦截，提取 user_id 字段
2. 在索引表（如 idx_user_id）中写入：RowKey = user_id + original_rowkey，Value = 原表 RowKey
3. 查询时，先从索引表查得原表 RowKey，再定位原表数据

┌─────────────┐      prePut       ┌─────────────┐
│  主表 data  │ ←───────────────→  │ Coprocessor │
│  RowKey=100 │   拦截写操作        │  构建索引    │
└─────────────┘                    └──────┬──────┘
                                          │
                                    写入索引表
                                          ↓
                                   ┌─────────────┐
                                   │ 索引表 idx  │
                                   │ RowKey=uid_100│
                                   └─────────────┘
```

**5. Endpoint 示例**

```java
// 定义协议接口
public interface SumEndpoint extends CoprocessorProtocol {
    long sum(byte[] columnFamily, byte[] qualifier) throws IOException;
}

// Server 端实现：直接对 MemStore + HFile 中数据求和，不必将全表数据拉回 Client
// Client 调用：scan.setCoprocessor() 注册 Endpoint，批量计算
```

**6. 注意事项**

- Coprocessor 运行在 RegionServer 上，代码 Bug 可能影响集群稳定性
- 推荐使用**加载方式**：`hbase.coprocessor.region.classes`（按表加载）或 `hbase.coprocessor.user.groups`（按权限加载）
- 高并发场景注意并发控制，避免死锁

## 题目五：HBase 如何保证数据一致性和高可用？

### 参考答案

**1. 数据一致性模型**

HBase 提供**强一致性**读写：

| 操作 | 一致性保证 |
|------|-----------|
| 写数据 | 写入 WAL + MemStore 后返回 ACK，保证持久化 |
| 读数据 | 读取时只访问拥有该 Region 最新版本的 HRegionServer，无脏读 |
| 原子性 | 单行数据（同一 RowKey 不同 Column）保证原子性 |
| 跨行事务 | **不支持**，跨行跨表需要依赖外部组件（Tephra、TransactionServer）|

**2. WAL（预写日志）保证持久性**

```
写入流程：
1. Client 发起写请求
2. 数据先写入 WAL（类似 MySQL redo log）
3. WAL 同步到 HDFS 多副本（默认 3 副本）后返回
4. 数据写入 MemStore

→ 即使 RegionServer 宕机，重启后从 WAL 恢复 MemStore 未 Flush 的数据
```

**3. 高可用（HA）机制**

| 机制 | 说明 |
|------|------|
| **HMaster HA** | 多个 Master 通过 ZooKeeper 抢锁，只有一个 active，其他 standby。宕机后 ZooKeeper 自动选主 |
| **RegionServer HA** | ZooKeeper 心跳检测（超时 `zookeeper.session.timeout`），宕机后 Master 将其上的 Region 迁移到其他节点 |
| **HDFS 副本** | HFile 数据在 HDFS 层自动保持 3 副本，任意 DataNode 宕机不影响数据完整性 |
| **WAL 多副本** | WAL 同步写入 HDFS，RegionServer 宕机后数据可从 HDFS 恢复 |

**4. 容灾恢复流程**

```
RegionServer 宕机
       ↓
ZooKeeper 心跳超时（默认 90s），标记该节点死亡
       ↓
HMaster 抢占到锁，开始恢复流程
       ↓
Master 将该节点上所有 Region 分配到其他存活节点
       ↓
其他节点从 WAL 重放（Replay）未 Flush 的数据
       ↓
恢复完成，对外继续服务
```

**5. 备份与恢复方案**

```bash
# 快照备份（Snapshot）
hbase shell> snapshot 'table_name', 'snapshot_name'
hbase shell> export_snapshot '-snapshot snapshot_name' -copy-to hdfs:///backup/

# 快照恢复
hbase shell> restore_snapshot 'snapshot_name'

# 复制（Replication）— 跨集群同步
hbase shell> add_peer '1', 'CLUSTER_KEY => "node1,node2:/hbase"'
```
