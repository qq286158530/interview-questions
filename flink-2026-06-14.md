## 一、Flink 端到端精确一次（End-to-End Exactly-Once）语义是如何实现的？

Flink 自身通过 Checkpoint 实现了内部状态的 Exactly-Once，但真正的端到端精确一次需要 Source 和 Sink 共同配合。

### 端到端 Exactly-Once 的三要素

| 环节 | 保障机制 | 说明 |
|------|---------|------|
| **Source** | 可重放 + Checkpoint 保存偏移量 | Kafka 支持 offset 重放，从 Checkpoint 恢复时从头消费 |
| **Flink 内部** | Checkpoint（Chandy-Lamport） | 算子状态快照，故障恢复重放 |
| **Sink** | 两阶段提交（2PC） | Checkpoint 完成时才真正提交，避免数据重复或丢失 |

### 两阶段提交（Two-Phase Commit）详解

Flink Sink（尤其是写入外部存储的 Sink）使用两阶段提交协议：

```
阶段1（预提交）：
  Checkpoint 完成 → Sink 发起预提交（写入prepare/commit标记）
  ↓
阶段2（提交/回滚）：
  JobManager 确认所有算子 Checkpoint 成功 → Sink 真正提交数据
  若任意算子 Checkpoint 失败 → 所有 Sink 回滚
```

**Kafka Sink EXACTLY_ONCE 配置：**
```java
KafkaSink<String> kafkaSink = KafkaSink.<String>builder()
    .setBootstrapServers("localhost:9092")
    .setRecordSerializer(new SimpleStringSerializer("output-topic"))
    .setDeliveryGuarantee(DeliveryGuarantee.EXACTLY_ONCE)
    .setTransactionalIdPrefix("flink-")
    .build();

stream.sinkTo(kafkaSink);

// 环境配置
env.enableCheckpointing(30_000, CheckpointingMode.EXACTLY_ONCE);
env.getCheckpointConfig().setExternalizedCheckpointCleanup(
    ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION);
```

**支持端到端 Exactly-Once 的 Sink：**
- Kafka Sink（`DeliveryGuarantee.EXACTLY_ONCE`）
- Hive Sink（ACID 写入）
- File Sink（Prestore + 原子性Rename）
- JDBC Sink（事务写入）

**常见陷阱：**
- Sink 不支持事务（如普通 MySQL JDBC）→ 只能 At-Least-Once
- `setExternalizedCheckpointCleanup(DELETE_ON_CANCELLATION)` 会丢失 Checkpoint → 生产环境建议用 `RETAIN_ON_CANCELLATION`
- 多个 Sink 共享同一事务 ID 会导致冲突

## 二、Flink Watermark 策略有哪些？如何处理迟到数据？

Watermark 是 Flink 处理乱序事件的核心机制，本质是一个时间戳 T，表示"时间 ≤ T 的事件都已到达"。

### 常用 Watermark 策略

#### 1. forBoundedOutOfOrderness（乱序容忍策略，最常用）
```java
WatermarkStrategy
    .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner((event, timestamp) -> event.getTimestamp())
```
- 含义：最多容忍 10 秒乱序
- 窗口在 Watermark ≥ window_end_time 时触发
- 适用场景：网络延迟可控的在线数据源

#### 2. forMonotonousTimestamps（单调递增策略）
```java
WatermarkStrategy
    .<Event>forMonotonousTimestamps()
    .withTimestampAssigner((event, timestamp) -> event.getTimestamp())
```
- 含义：时间戳严格单调递增，不容忍乱序
- 适用场景：严格有序的数据源（如本地日志）

#### 3. noWatermarks（无 Watermark）
```java
WatermarkStrategy.noWatermarks()
```
- 用于 Processing Time 模式或不需要事件时间的场景

### 迟到数据的三种处理方式

| 方式 | 机制 | 配置 |
|------|------|------|
| **allowedLateness** | 窗口触发后，额外等待一段时间 | `.allowedLateness(Time.seconds(5))` |
| **sideOutputLateData** | 将迟到数据侧输出到旁路流 | `.sideOutputLateData(lateOutputTag)` |
| **直接丢弃** | 窗口触发后抛弃迟到数据（默认） | 无需配置 |

### 完整示例：Watermark + 窗口 + 迟到处理
```java
OutputTag<Event> lateTag = new OutputTag<Event>("late-data"){};

DataStream<Event> stream = env
    .assignTimestampsAndWatermarks(
        WatermarkStrategy
            .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(10))
            .withTimestampAssigner((e, ts) -> e.getTimestamp())
    )
    .keyBy(e -> e.getKey())
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .allowedLateness(Time.seconds(5))          // 窗口触发后额外等5秒
    .sideOutputLateData(lateTag)              // 迟到超过5秒的数据进入旁路
    .sum("count");

// 获取迟到数据
DataStream<Event> lateStream = stream.getSideOutput(lateTag);
```

### Watermark 传播机制

Watermark 沿算子链向下游传播，规则如下：
- 算子收到所有输入流的 Watermark 后，才推进自己的 Watermark
- 多输入算子（如 union、co-flatmap）：取所有输入 Watermark 的最小值
- 有状态算子：必须等 Watermark 越过窗口结束时间才触发窗口

## 三、Flink Checkpoint 和 Savepoint 有什么区别？如何选择？

### 核心区别

| 维度 | Checkpoint | Savepoint |
|------|-----------|-----------|
| **触发方式** | Flink 自动周期性触发 | 手动触发（用户命令） |
| **用途** | 故障恢复（automatic recovery） | 计划性停机维护、版本发布 |
| **存储位置** | JobManager 内存（快速）/ 外部存储 | 外部存储（用户指定） |
| **清理策略** | Job 取消/超时自动删除 | 需手动删除 |
| **格式** | Flink 内部优化格式 | 兼容旧版本，兼容性更好 |
| **状态大小** | 通常较小 | 通常较大（全量快照） |

### Checkpoint 配置
```java
env.enableCheckpointing(60_000);  // 每60秒
env.getCheckpointConfig().setCheckpointIdOfLatestCheckpointRetained更加;

// 高级配置
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(500);  // 两次checkpoint最小间隔
env.getCheckpointConfig().setMaxConcurrentCheckpoints(1);       // 最多1个并发
env.getCheckpointConfig().setCheckpointTimeout(600_000);       // 超时10分钟
env.getCheckpointConfig().setTolerableCheckpointFailureNumber(3); // 允许3次失败
```

### Savepoint 操作
```bash
# 创建 Savepoint（停止 Job 时自动保存）
flink stop --savepointPath /path/to/savepoint <jobId>

# 从 Savepoint 恢复
flink run -d -s /path/to/savepoint/_metadata <jar>

# 手动触发 Savepoint
flink savepoint <jobId> /path/to/savepoint

# 删除 Savepoint
flink savepoint -d <savepointPath>
```

### 状态存储后端选择

| Backend | 存储介质 | 适用场景 |
|---------|---------|---------|
| **HashMapStateBackend** | JVM Heap | 小状态（<10MB）、低延迟需求 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘） | 大状态、巨大 Key 基数、需要 TTL |
| **MetricsReportingRocksDBStateBackend** | RocksDB + 上报 | 生产大状态 + 监控 |

```java
// 推荐生产配置：RocksDB + 增量 Checkpoint
EmbeddedRocksDBStateBackend rocksDB = new EmbeddedRocksDBStateBackend(true);
env.setStateBackend(rocksDB);
env.getCheckpointConfig().setCheckpointStorage("hdfs://namenode:8020/flink-cp");

// 增量 Checkpoint（只保存新增的 SST 文件）
env.getCheckpointConfig().enableIncrementalCheckpointing();
```

## 四、Flink 状态后端（State Backend）如何选型？RocksDB 为何适合大状态场景？

### 三种 State Backend 对比

| Backend | 存储介质 | 容量上限 | 读写性能 | TTL 支持 |
|---------|---------|---------|---------|---------|
| **HashMapStateBackend** | JVM Heap | JVM 堆内存 | 最高（内存读写） | 支持 |
| **EmbeddedRocksDBStateBackend** | 本地 RocksDB | 磁盘容量 | 中等（序列化开销） | 支持 |
| **JdbcStateBackend** | 外部数据库 | 数据库容量 | 低（网络 IO） | 支持 |

### 为什么大状态要用 RocksDB？

```
内存Backend:  状态全在内存 → 100万Key → OOM
RocksDB Backend:  状态在磁盘 → 增量读取 → 不怕大状态
```

**RocksDB 工作原理：**
- 每个 Key-Value 对写入时先写内存表（MemTable），达到阈值后 flush 到磁盘（SST File）
- 读取时先查 MemTable，查不到再查 SST File（布隆过滤器加速）
- 写入磁盘的 SST File 是有序的，适合顺序扫描

**RocksDB 增量 Checkpoint 优势：**
- 全量 Checkpoint：每次快照都要复制全部状态文件
- 增量 Checkpoint：只复制新增的 SST 文件，大幅减少网络和存储开销

```java
// 生产环境推荐配置
EmbeddedRocksDBStateBackend rocksDB = new EmbeddedRocksDBStateBackend(true);
env.setStateBackend(rocksDB);

// 开启增量 Checkpoint
env.getCheckpointConfig().enableIncrementalCheckpointing();

// 设置 State TTL（自动清理过期状态）
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.days(7))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)
    .cleanupInRocksdbCompactFilter(1000)  // RocksDB compaction 时清理
    .build();
```

### 状态过期（TTL）清理策略

| 策略 | 说明 |
|------|------|
| **OnCreate** | 只有在创建时检查过期 |
| **OnReadAndWrite** | 读取和写入时都检查过期 |
| **RocksDB Compaction Filter** | RocksDB compaction 时异步清理（推荐大状态使用） |

### 常见问题

**Q：RocksDB 写入慢怎么办？**
- 调大 RocksDB write buffer：`state.backend.rocksdb.memory.write-buffer-size`
- 使用 SSD 存储本地磁盘
- 开启增量 Checkpoint 减少写入压力

**Q：Key 数量巨大（亿级）选什么 Backend？**
- 必须选 RocksDBStateBackend
- HashMapStateBackend 会 OOM

## 五、Flink 窗口函数有哪些？如何处理 Late Element？

### 窗口函数分类

```
┌─────────────────────────────────────────────┐
│                  窗口函数                     │
├──────────────────┬──────────────────────────┤
│   Time Windows   │     Count Windows        │
│  (时间驱动)       │     (数量驱动)           │
├──────────────────┼──────────────────────────┤
│ Tumbling         │ countWindow(5)           │
│ Sliding          │ countWindow(5, 2)        │
│ Session          │ sessionWindow(gap)       │
│ Global           │ global()                 │
└──────────────────┴──────────────────────────┘
```

### 1. 滚动窗口（Tumbling Window）
- 大小固定，不重叠
- 每条数据属于且仅属于一个窗口
```java
// 每10秒一个窗口
stream.keyBy(...)
    .window(TumblingEventTimeWindows.of(Time.seconds(10)))
    .sum("count");
```

### 2. 滑动窗口（Sliding Window）
- 大小固定，可重叠
- 参数：窗口大小 + 滑动步长
```java
// 窗口大小10秒，滑动步长5秒（每5秒输出一次近10秒结果）
stream.keyBy(...)
    .window(SlidingEventTimeWindows.of(Time.seconds(10), Time.seconds(5)))
    .sum("count");
```

### 3. 会话窗口（Session Window）
- 由活跃间隙（gap）分隔
- 动态大小，取决于用户实际行为
```java
// 间隔10秒无数据则关闭窗口
stream.keyBy(...)
    .window(SessionWindow.withGap(Time.seconds(10)))
    .sum("count");
```

### 4. 全局窗口（Global Window）
- 所有数据进入同一个窗口
- 需要配合 Trigger 使用
```java
stream.keyBy(...)
    .window(GlobalWindows.create())
    .trigger(CountTrigger.of(100))  // 每100条触发一次
    .sum("count");
```

### 窗口函数类型

| 函数类型 | 说明 | 示例 |
|---------|------|------|
| **ReduceFunction** | 增量聚合，更高效 | `.reduce((a, b) -> new Tuple2(a.f0, a.f1 + b.f1))` |
| **AggregateFunction** | 更灵活的聚合，支持输出不同类型 | `.aggregate(new AvgAggregate())` |
| **FoldFunction** | 老旧 API，不推荐 | - |
| **WindowFunction / ProcessWindowFunction** | 全量窗口数据处理，可访问上下文 | `.process(new MyProcessFunction())` |

### Late Element（迟到数据）完整处理

```java
OutputTag<ServerEvent> lateTag = new OutputTag<ServerEvent>("late"){};

SingleOutputStreamOperator<UrlViewCount> result = stream
    .assignTimestampsAndWatermarks(
        WatermarkStrategy
            .<ServerEvent>forBoundedOutOfOrderness(Duration.ofSeconds(30))
            .withTimestampAssigner((e, ts) -> e.getTimestamp())
    )
    .keyBy(e -> e.getUrl())
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .allowedLateness(Time.minutes(1))       // 窗口触发后多等1分钟
    .sideOutputLateData(lateTag)             // 超过1分钟的迟到数据走旁路
    .aggregate(new UrlCountAggregate(), new UrlCountProcess());

// 获取迟到数据
DataStream<ServerEvent> lateStream = result.getSideOutput(lateTag);
lateStream.addSink(new SinkToKafka());  // 迟到数据单独处理（如补录到Kafka）
```

### Late Element 处理流程图

```
Event(Time=9:58) ─┐
Event(Time=9:59) ─┼─→ window[9:55-10:00] 触发（Watermark >= 10:00）
Event(Time=10:01) ─┤   allowedLateness=1min → 窗口等迟到数据到 10:01
                  │
[Late] Event(Time=9:56) ──→ allowedLateness 范围内 ─→ 触发增量计算
[Late] Event(Time=9:52) ──→ allowedLateness 之外 ─→ 侧输出到旁路流
```
