## 题目一：Redis Cluster 是如何实现数据分片的？Hash Slot 的原理是什么？

### 参考答案

**Redis Cluster 是 Redis官方提供的分布式集群方案，采用去中心化架构。**

### 核心机制：Hash Slot（哈希槽）

Redis Cluster 将整个键值空间划分为 **16384（2^14）个哈希槽**，每个槽由集群中的一个主节点负责存储。

**为什么是 16384 个槽？**
- 节点间通过心跳交换消息，消息中包含节点已分配的槽位信息（2KB足以表示16384 bit 的 bitmap）
- 槽位越少，心跳包越小，节点数量多时网络开销可控
- 16384 是经过权衡的取值，既保证了足够的分片粒度，又控制了元数据开销

### 数据定位流程**

```
key → CRC16(key) % 16384 → hash slot → 查询槽位对应的节点
```

1. 客户端对 key 使用 CRC16 算法计算哈希值
2. 对16384 取模，得到对应的 hash slot 编号
3. 向任意一个集群节点发送请求（节点会返回 MOVED 重定向，指引客户端到正确节点）
4. 客户端直接连接正确节点，后续请求缓存槽位信息

### 故障转移与高可用

- 每个主节点可以有一个或多个从节点（replica）
- 主节点故障时，从节点自动升主（需要 Sentinel 配合或 Cluster 内置投票机制）
- 故障转移：集群内超过半数节点认为某主节点不可达时，触发选举

### 跨槽查询的限制

Redis Cluster 不支持跨多个槽的批量操作（如 `MGET` 跨槽、`SUNION` 跨槽），因为 key 可能分布在不同节点上。解决方案：
- 将相关数据放在同一个槽（同 key 或 key 的 hash tag）
- 使用 Hash Tag：`{user:1}:name` 和 `{user:1}:age` 会被路由到同一槽

## 题目二：Redis Pub/Sub（发布/订阅）的原理是什么？与 Redis Stream 有何区别？

### 参考答案

### Pub/Sub 模式

Redis 内置的发布/订阅模式，是一种**广播式消息传递机制**：

**角色：**
- **Publisher（发布者）**：向指定频道（channel）发送消息
- **Subscriber（订阅者）**：订阅一个或多个频道，等待消息

**核心命令：**
- `PUBLISH <channel> <message>`：发布消息
- `SUBSCRIBE <channel1> <channel2> ...`：订阅频道
- `PSUBSCRIBE <pattern>`：按模式订阅（如 `news:*`）

**原理：**
- 订阅者通过 `SUBSCRIBE` 命令将自己注册到指定频道的链表中
- `PUBLISH` 时，Redis 遍历频道订阅者链表，将消息发送给所有订阅者
- 消息**不持久化**——若订阅者不在线，消息直接丢弃

**使用场景：**
- 实时通知系统（如聊天室消息、新闻推送）
- 跨系统事件广播

### Redis Stream（5.0+）——有持久化的消息流

Stream 是 Redis 5.0 引入的新数据结构，类比 Kafka 的消费模式，核心区别在于：

| 特性 | Pub/Sub | Stream |
|------|---------|--------|
| 持久化 | ❌ 不持久，消息内存中即发即弃 | ✅消息持久化到 RDB/AOF |
| 消费确认 | ❌ 无 ACK | ✅ 支持 XACK确认消费 |
| 消费者组 | ❌ 不支持 | ✅ 支持 XGROUP，多消费者负载均衡 |
| 历史消息 | ❌只能消费在线订阅者的新消息 | ✅ 可用 XRANGE 回溯历史 |
| 丢消息风险 | 高（订阅者离线即丢失） | 低（持久化 + ACK） |

**Stream 核心命令：**
- `XADD`：添加消息（自动生成唯一 ID）
- `XREAD [COUNT n] [BLOCK ms]`：读取新消息
- `XGROUP CREATE`：创建消费者组
- `XACK`：确认已处理消息

**何时选 Pub/Sub vs Stream：**
- 只需要实时广播、容忍消息丢失 → Pub/Sub
- 需要消息持久化、消费确认、历史回溯 → Stream

## 题目三：Redis 如何实现分布式锁？Redisson 的原理是什么？有什么缺陷？

### 参考答案

### 分布式锁的核心要求

1. **互斥性**：同一时刻只能有一个客户端持有锁
2. **防死锁**：锁最终必须能释放（超时机制）
3. **防抢锁**：持锁客户端崩溃时，其他客户端能自动解锁
4. **可重入性**（可选）：同一客户端可多次获取同一把锁

### 使用 Redis 实现分布式锁

**基础方案（SETNX）：**
```lua
-- 加锁
SET lock_key unique_value NX PX 30000
-- NX：key不存在才设置（互斥）
-- PX 30000：过期时间30秒

-- 解锁（Lua脚本，保证原子性）
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

**为什么要用 Lua 脚本解锁？**
- 判断 value 和删除 key 不是原子操作，中间可能出现误删其他客户端已获取的锁
- Lua 脚本在 Redis 中原子执行，避免并发问题

### Redisson 分布式锁原理

Redisson 是最流行的 Redis 客户端，其分布式锁基于 Lua 脚本 + 看门狗（Watchdog）机制：

1. **加锁**：使用 `SETNX` + 自动续期
2. **看门狗**：锁有过期时间，Redisson 内部启动一个后台线程，若锁仍被持有，则每隔 `lockWatchdogTimeout / 3` 自动续期（重新设置过期时间）
3. **可重入**：使用 Hash 结构存储 `{ownerThreadId: count}`，同一线程重入时累加计数

###分布式锁的常见缺陷

1. **Redis 单点问题**：单机 Redis 做锁有单点故障风险，解决方案：使用 RedLock（多台 Redis 实例，多数节点加锁成功才算成功）
2. **时钟漂移**：多台机器时钟不一致可能导致锁提前释放，Redlock 对时钟要求严格
3. **主从复制延迟**：主节点加锁成功后异步复制给从节点，此时主节点故障可能导致锁丢失（Redlock针对性解决）
4. **锁粒度过大**：锁住整个数据范围导致并发性能差，解决方案：细粒度锁或分段锁

### Redlock 算法

为解决单机 Redis 的高可用问题，Redlock 方案：
- 在 N（N≥5 建议）个独立 Redis 实例上分别加锁
- 客户端向所有实例加锁，若超过半数成功，且总耗时 < 锁过期时间，则认为加锁成功
- 缺点：实现复杂，且仍无法完全避免时钟漂移问题

## 题目四：Redis Pipeline（管道）和 Lua 脚本有什么区别？各自适用场景是什么？

### 参考答案

### Redis Pipeline（管道）

**原理：** 将多个 Redis 命令打包一次性发送给服务器，服务器按序返回所有响应，减少网络往返次数（RTT）。

**使用示例（以 Jedis 为例）：**
```java
Pipeline pipe = jedis.pipelined();
pipe.set("key1", "value1");
pipe.get("key1");
pipe.incr("counter");
List<Object> results = pipe.exec(); // 一次性获取所有响应
```

**特点：**
- 客户端侧批量命令，**服务端仍逐一执行**每个命令
- 不保证原子性（各命令独立执行）
- 显著提升批量读写性能（网络开销从 N 次 RTT 降为 1 次）

**适用场景：**
- 批量写入/读取大量独立 key（如数据初始化、批量更新）
- 对原子性无要求的场景

### Lua 脚本

**原理：** 将一段 Lua 脚本整体发送给 Redis 执行，Redis 保证脚本中**所有命令原子执行**（中途不会执行其他客户端命令）。

**使用示例：**
```lua
-- 实现 INCRBY + 边界检查的原子操作
local current = redis.call('GET', KEYS[1])
if current and tonumber(current) + tonumber(ARGV[1]) > 100 then
    return redis.call('SET', KEYS[1], 100)
else
    return redis.call('INCRBY', KEYS[1], ARGV[1])
end
```

**特点：**
- 原子性保障：脚本执行期间不会被其他命令插入
- 服务端执行，减少网络往返
- 支持脚本缓存（`SCRIPT LOAD` + `EVALSHA`，避免每次传输脚本）
- 脚本执行时间不宜过长（Redis 官方建议< 1ms，避免阻塞）

**适用场景：**
- 需要原子性的复合操作（如检查+更新、判断+设置）
- 自定义业务逻辑需要在 Redis 端执行（如库存扣减、乐观锁）
- 减少网络往返的同时需要原子性保证

### 核心对比

| 特性 | Pipeline | Lua 脚本 |
|------|----------|----------|
| 原子性 | ❌ 无保证 | ✅ 原子执行 |
| 网络开销 | 1 次 RTT（批量命令） | 1 次 RTT（单个脚本调用） |
| 失败处理 | 各命令独立，可部分成功 | 全部回滚（原子性） |
| 适用场景 | 批量独立读写 | 复合原子操作 |
| 执行位置 | 客户端批量，服务端逐条执行 | 服务端整体原子执行 |

**实战建议：** Pipeline 用于"读多个 key"或"写多个 key"且不需要原子性的场景；Lua 脚本用于"读+写+判断"等需要原子性保障的复合操作。

## 题目五：Redis 如何实现缓存与数据库的双写一致性？有哪些常见问题？

### 参考答案

### 缓存读写模式

#### 模式一：Cache Aside（旁路缓存，最常用）

**读操作：**
```
读取数据 → Cache Miss → 查询数据库 → 写入缓存 → 返回数据
                      ↓
              Cache Hit → 直接读取缓存 → 返回数据
```

**写操作：**
```
更新数据库 → 删除缓存（而非更新）→ 返回
```

**为什么删除而非更新？**
-避免并发场景下缓存与数据库数据不一致（先更新数据库 vs 后删除缓存的执行顺序问题）
- 删除操作代价更低，且下次 Cache Miss 时自然恢复一致

#### 模式二：Read Through（读穿透）

缓存作为数据库的代理，应用程序只与缓存交互：
- Cache Miss 时，缓存自动查询数据库并写入缓存层
- 对应用透明（类似 MySQL Proxy）

#### 模式三：Write Through（写穿透）

- 更新数据库的同时同步更新缓存
- 缺点：写入放慢，且可能写了缓存却数据库写入失败

#### 模式四：Write Behind（写回）

- 更新缓存后立即返回，缓存异步批量写回数据库
- 性能最高，但风险最大（缓存故障可能丢失数据）

### 双写一致性的常见问题与解决方案

**问题一：并发读写导致的不一致**
- 场景：线程A写数据库→线程B读数据库→线程A删缓存→线程B将旧数据写入缓存
- 解决：延迟双删（删除缓存→更新数据库→延迟Nms再删一次缓存）

**问题二：缓存删除失败**
- 场景：更新数据库成功，但删除缓存失败
- 解决：引入消息队列（如 RabbitMQ/RocketMQ）重试删除操作（最终一致性）

**问题三：主从复制延迟**
- 场景：主库更新后，从库尚未同步完成，读请求打到从库拿到旧数据
- 解决：
  - 强制读主库（但增加主库压力）
  - 延迟感知：业务层加延迟去重
  - 使用分布式数据库中间件（如 Canal监听 binlog同步缓存）

### 一致性级别

| 级别 | 说明 | 适用场景 |
|------|------|---------|
| **强一致性** | 缓存与数据库完全同步 | 金融、交易等核心数据 |
| **最终一致性** | 允许短暂不一致，最终自动恢复 | 大多数互联网业务（社交、新闻） |
| **弱一致性** | 不保证最终一致 | 允许数据差异的业务（如 UV 统计） |

### 实战建议

- **大多数互联网业务**：采用 Cache Aside + 延迟双删 + MQ 重试，实现最终一致性
- **核心交易数据**：不推荐 Redis 做缓存，直接读写主从数据库
- **热点数据**：使用 Redis 缓存 + 消息队列异步更新数据库，同时做好缓存击穿防护（互斥锁 / 永不过期 + 异步重建）
