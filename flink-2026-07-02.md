## 题目一：Flink 的架构是什么样的？请详细说明 JobManager 和 TaskManager 的作用

### 参考答案

Flink 采用主从架构，由 **JobManager** 和 **TaskManager** 组成：

### JobManager（主节点）
- **JobMaster**：负责调度作业，将JobGraph转换为ExecutionGraph，协调作业执行
- **ResourceManager**：管理TaskManager的slot资源，进行资源分配
- **Dispatcher**：接收用户提交的作业，为每个作业启动一个新的JobMaster，同时提供REST接口

### TaskManager（工作节点）
- **Slot**：TaskManager提供多个Slot，每个Slot可以运行一个Task（即一个算子的并行实例）
- **执行引擎**：实际执行数据流的算子，处理数据

### 工作流程
1. 用户代码 → JobGraph → 提交给Dispatcher
2. Dispatcher启动JobMaster，JobMaster向ResourceManager申请slot
3. ResourceManager向TaskManager发起slot请求，TaskManager分配slot
4. JobMaster将ExecutionGraph的task分配到对应slot执行

## 题目二：什么是 Watermark？Flink 如何处理乱序数据？

### 参考答案

### Watermark（水位线）的概念
Watermark 是用于处理**事件时间（Event Time）**乱序的一种机制，本质上是一个**时间戳**，表示"所有时间戳小于该watermark的数据都已经到达"。

### 核心思想
```
Watermark(t) 表示：所有 event_time < t 的数据都已经到达
```

### 乱序处理流程
1. **Watermark 机制**：当收到 Watermark w 时，窗口会关闭所有 event_time < w 的数据
2. **maxOutOfOrderness**：设置允许的最大乱序程度（如3秒），Watermark = current_max_time - maxOutOfOrderness
3. **窗口延迟关闭**：使用 `allowedLateness()` 设置额外等待时间，窗口会在watermark触发后仍保留一定时间

### 示例代码
```java
WatermarkStrategy<Event> wms = WatermarkStrategy
    .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(3))
    .withTimestampAssigner((event, timestamp) -> event.getEventTime());

DataStream<Event> stream = env.fromSource(
    source, wms, "MySource"
);
```

### 三种触发时机
- **watermark >= 窗口结束时间**：触发窗口计算
- **allowedLateness 延迟期**：迟到数据仍可触发窗口重计算
- **窗口 purge**：延迟期过后，窗口状态被清除

## 题目三：Flink Checkpoint 和 Savepoint 的区别是什么？

### 参考答案

### Checkpoint（检查点）
- **用途**：Flink故障恢复的快照机制，由Flink自动管理
- **触发方式**：由JobManager按固定间隔自动触发，或手动触发
- **存储位置**：通常存储在HDFS、S3等分布式存储
- **特点**：
  - 需要source算子支持（如Kafka可以.seekToOffset"回退"）
  - 是增量快照，采用Chandy-Lamport算法
  - 删除策略：按配置保留最近N个

### Savepoint（保存点）
- **用途**：手动触发的有状态的作业快照，用于运维操作
- **触发方式**：由用户显式触发 `bin/flink savepoint <jobId> [targetDir]`
- **存储位置**：用户指定目录
- **特点**：
  - 不会自动删除，需手动管理
  - 与作业解耦，可用于不同作业恢复（如修改后的作业）
  - 是全量快照

### 核心区别

| 维度 | Checkpoint | Savepoint |
|------|-----------|-----------|
| 触发者 | Flink自动 | 用户手动 |
| 用途 | 故障恢复 | 运维（停止、升级、分叉） |
| 删除 | 自动删除旧版本 | 手动删除 |
| 全量/增量 | 增量 | 全量 |

- https://ci.apache.org/projects/flink/flink-docs-release-1.17/docs/ops/state/savepoints/

## 题目四：Flink 窗口函数有哪些类型？Session Window 如何实现？

### 参考答案

### 窗口函数分类

#### 1. 时间窗口（Time Windows）
- **滚动窗口（Tumbling Window）**：固定大小、不重叠，如 `TumblingEventTimeWindows.of(Time.seconds(10))`
- **滑动窗口（Sliding Window）**：固定大小、可重叠，如 `SlidingEventTimeWindows.of(Time.seconds(10), Time.seconds(5))`
- **会话窗口（Session Window）**：基于活动间隙的动态窗口

#### 2. 计数窗口（Count Windows）
- **滚动计数窗口**：固定条数，如 `countWindow(100)`
- **滑动计数窗口**：如 `countWindow(100, 50)`

### Session Window 实现

```java
DataStream<Tuple2<String, Long>> stream = ...;

// 基于事件时间的会话窗口，间隙为5秒
DataStream<String> sessionWindowed = stream
    .keyBy(0)
    .window(EventTimeSessionWindows.withGap(Time.seconds(5)))
    .sum(1);

// 动态间隙会话窗口
DataStream<String> dynamicSessionWindowed = stream
    .keyBy(0)
    .window(EventTimeSessionWindows.withDynamicGap(new SessionWindowTimeGapExtractor<Tuple2<String, Long>>() {
        @Override
        public double extract(Tuple2<String, Long> element) {
            // 根据数据动态计算间隙，如根据用户行为类型
            return element.f1 * 1000; // 毫秒
        }
    }))
    .sum(1);
```

### Session Window 特点
- 无固定开始/结束时间，由数据本身触发
- 当两条数据的event time差值超过gap时，触发窗口关闭
- 支持 `allowedLateness()` 延迟处理

- https://ci.apache.org/projects/flink/flink-docs-release-1.17/docs/dev/datastream/operators/process_function/

## 题目五：Flink 状态管理机制是什么？Keyed State 和 Operator State 有何区别？

### 参考答案

### 状态管理概述
Flink 是有状态流处理引擎，状态（State）是算子实例在处理数据时需要记住的信息，如聚合结果、缓存数据等。

### 状态类型

#### 1. Keyed State（键控状态）
- **作用域**：仅在 KeyedStream 上使用，按 key 分区
- **生命周期**：与 key 绑定，key 消失时状态也消失
- **存储结构**：
  - `ValueState<T>`：单个值
  - `ListState<T>`：列表
  - `MapState<K, V>`：键值对
  - `ReducingState<T>` 和 `AggregatingState<IN, OUT>`：聚合

```java
public class CountWindowAverage extends RichFlatMapFunction<Tuple2<String, Long>, Tuple2<String, Double>> {
    private ValueState<Count> state; // 键控状态

    @Override
    public void open(Configuration parameters) {
        state = getRuntimeContext().getState(
            new ValueStateDescriptor<>("myCount", Count.class)
        );
    }

    @Override
    public void flatMap(Tuple2<String, Long> input, Collector<Tuple2<String, Double>> out) {
        Count current = state.value();
        if (current == null) {
            current = new Count();
        }
        current.count++;
        current.sum += input.f1;
        state.update(current);
        out.collect(new Tuple2<>(input.f0, current.sum / current.count));
    }
}
```

#### 2. Operator State（算子状态）
- **作用域**：算子实例级别，所有并行实例共享
- **生命周期**：与算子实例绑定，重启后可重新分配
- **类型**：
  - **ListState**：算子状态分为列表，并行度改变时均匀分配
  - **UnionListState**：广播状态，重启时所有并行实例获得完整状态
  - **BroadcastState**：广播状态，用于动态流更新

```java
public class SourceFunctionexample implements SourceFunction<String>, CheckpointedFunction {
    private ListState<String> checkpointedCount; // 算子状态

    @Override
    public void initializeState(FunctionInitializationContext context) throws Exception {
        checkpointedCount = context.getOperatorStateStore()
            .getListState(new ListStateDescriptor<>("count", String.class));
    }

    @Override
    public void snapshotState(FunctionSnapshotContext context) throws Exception {
        checkpointedCount.clear();
        checkpointedCount.add("current count state");
    }
}
```

### 核心区别

| 维度 | Keyed State | Operator State |
|------|------------|---------------|
| 作用域 | 按 key 分区 | 算子实例级别 |
| 支持算子 | 仅 KeyedStream | 所有算子 |
| 并行度变化 | key 自动重新分区 | 需要状态分发策略 |
| 存储结构 | Value/List/Map/Aggregating | List/UnionList/Broadcast |

### 状态后端
- **HashMapStateBackend**：状态存储在TaskManager JVM内存，性能高，适合大状态
- **EmbeddedRocksDBStateBackend**：状态存储在RocksDB，适合超大状态，支持增量checkpoint
