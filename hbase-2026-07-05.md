## 题目一：HBase 的整体架构是怎样的？请说明 HMaster、HRegionServer、ZooKeeper 的职责

**参考答案：**

HBase 采用典型的 Master-Slave 架构，主要由三类组件构成：

### 1. HMaster（主节点）
- 负责表的 DDL 操作（创建、删除、修改表结构）
- 负责监控所有 HRegionServer 的健康状态
- 负责 Region 的分配与负载均衡
- 协调 Region 的分裂（Split）和迁移（Move）
- 管理元数据表的路由信息（`-ROOT-` 和 `META` 表）

### 2. HRegionServer（数据节点）
- 负责处理客户端的读写请求，是真正数据存储和读取的地方
- 每个 HRegionServer 管理多个 Region，每个 Region 对应一张表的一个数据分片
- 负责 MemStore 的写入、HFile 的合并与刷写
- 与 HDFS DataNode 协同工作，利用数据本地性提升读取性能
- 向 ZooKeeper 定期发送心跳，维持存活状态

### 3. ZooKeeper（协调服务）
- 维护集群的整体状态信息（哪些 HMaster 存活、哪些 RegionServer 可用）
- 存储 `META` 表的路由位置信息
- 实现分布式锁机制，用于 Master 选举和 RegionServer 故障检测
- HBase 通过 Watch 机制监听 ZooKeeper 节点变化，实现快速故障转移

**架构图简述：**
```
客户端 → ZooKeeper（获取 META 路由）→ HRegionServer（读写数据）
                    ↑
              HMaster（管理元数据）
```

## 题目二：HBase 的 RowKey 设计有哪些原则？为什么 RowKey 设计至关重要？

**参考答案：**

### RowKey 设计核心原则

#### 1. 长度原则
- RowKey 最大支持 64KB，通常建议控制在 **16~100 字节**以内
- 越短的 RowKey 在 HBase 的内部存储和传输中效率越高
- 避免使用过长的字符串作为 RowKey

#### 2. 散列原则
- RowKey 必须具有**高散列性**，避免数据集中在少数 Region 上（热点问题）
- 常见做法：对 RowKey 取 MD5 或 Hash 打散，或在 RowKey 头部加入随机前缀
- 热点问题会导致某个 RegionServer 负载过重，而其他节点空闲

#### 3. 唯一性原则
- RowKey 是 HBase 中数据的唯一标识，必须保证全局唯一

#### 4. 有序性原则
- HBase 中的数据按 RowKey 的字典顺序排列存储
- 可以利用这一特性，将**时间相关的数据**按时间倒序设计（如 `userId + reverseTimestamp`），实现"最新数据优先读取"

### 常见 RowKey 设计模式

| 场景 | RowKey 设计方案 |
|------|----------------|
| 用户行为轨迹 | `userId + timestamp` |
| 时序数据（最新优先）| `deviceId + (Long.MAX_VALUE - timestamp)` |
| 多表关联查询 | `joinKey + tableId` |
| 防止热点 | `MD5(userId).substring(0,4) + userId + timestamp` |

### 为什么至关重要？
- 直接影响 **数据分布均匀性**（热点问题）
- 直接影响 **读写性能**（顺序 Scan vs 随机 Read）
- 决定 **查询模式**（Get 高效 vs Scan 需谨慎）

## 题目三：HBase 的 MemStore 和 HFile 是如何工作的？为什么需要 Compaction？

**参考答案：**

### 数据写入流程（MemStore → HFile）

1. **客户端写入**：数据先写入 WAL（Write-Ahead Log），保证数据持久性
2. **写入 MemStore**：数据首先放入 RegionServer 的内存区域 MemStore 中（按 RowKey 排序的 SKV 结构）
3. **刷写（Flush）**：当 MemStore 达到阈值（默认 128MB）时，刷写为 HFile 文件落盘
4. **HFile 存储**：每个 Column Family 的数据对应独立的 HFile 文件

### HFile 的结构

HFile 是 HBase 的最终存储格式，结构如下：

| 结构层 | 说明 |
|--------|------|
| Data Block（数据块）| 存储 KeyValue 键值对，默认 64KB，可配置 |
| Meta Block（元数据块）| 存储布隆过滤器（BloomFilter）信息 |
| File Info | 文件元信息（最后修改时间、KV 数量等）|
| Index Block（索引块）| 两级索引：Root Index + Intermediate Index |
| Trailer | 固定偏移量信息，指向各数据块起始位置 |

### 为什么需要 Compaction？

随着时间推移，一个 Column Family 会产生大量小 HFile，导致：
- **读取放大**：一次查询需要合并多个 HFile，IO 开销大
- **文件描述符耗尽**：小文件过多

HBase 通过 **Compaction（压缩合并）** 解决此问题：

#### Minor Compaction（小合并）
- 选取部分较小的 HFile 进行合并
- 由后台线程自动触发，通常每小时检查一次
- 不会删除标记为删除的数据

#### Major Compaction（大合并）
- 将一个 Store 下的所有 HFile 合并为一个
- **会真正删除**标记为删除的单元格（墓碑标记）
- 默认每周自动执行一次
- 合并期间 IO 较高，建议在业务低峰期手动触发

## 题目四：HBase 的协处理器（Coprocessor）是什么？有哪两种类型？分别用于什么场景？

**参考答案：**

### 什么是协处理器？

协处理器是 HBase 提供的**扩展机制**，允许用户将自定义代码**部署到 RegionServer 节点上**，在数据写入和读取的过程中自动执行，从而在服务端完成计算，避免大量数据传输。

### 类型一：Observer Coprocessor（观察者协处理器）

**原理：** 类似于数据库触发器，在特定事件发生时被回调触发。

| 类型 | 触发时机 |
|------|----------|
| `RegionObserver` | Get/Put/Delete/Scan 等数据操作前后 |
| `WALObserver` | WAL 写入前后 |
| `MasterObserver` | DDL 操作（创建表、删除列族）前后 |

**典型应用场景：**
- **二级索引**：当数据写入时，自动将索引数据写入另一张索引表
- **数据验证**：写入前校验数据格式、权限
- **审计日志**：记录所有数据变更操作
- **自动填充**：写入时自动补充时间戳、版本号等字段

### 类型二：Endpoint Coprocessor（端点协处理器）

**原理：** 类似于存储过程，允许在 RegionServer 上执行自定义的聚合计算。

**典型应用场景：**
- 在服务端直接执行 `SUM`、`AVG`、`COUNT` 等聚合操作，减少网络传输
- 实现跨 Region 的复杂 Join（将 Join 下推到数据节点）
- 自定义压缩算法、加密逻辑

### 使用限制与注意事项
- 协处理器代码随 RegionServer 加载，bug 可能导致 RegionServer 崩溃
- Observer 协处理器不应执行耗时过长的操作
- 需在 HBase 配置文件中注册，或通过 HBase Shell 动态加载

## 题目五：HBase 和 HDFS、RDBMS 的核心区别是什么？什么场景下应该选择 HBase？

**参考答案：**

### HBase vs HDFS

| 对比维度 | HBase | HDFS |
|----------|-------|------|
| 设计目标 | 随机实时读写 | 批处理大规模文件存储 |
| 数据模型 | 列族数据库（NoSQL）| 文件系统 |
| 延迟 | 低延迟（毫秒级）| 高延迟（秒级）|
| 读取方式 | 支持随机读写 Get/Scan | 顺序读写为主 |
| 一致性 | 强一致性（行级）| 最终一致性 |
| 适用场景 | 在线业务系统 | 离线数据分析 |

### HBase vs RDBMS（关系型数据库）

| 对比维度 | HBase | RDBMS |
|----------|-------|-------|
| 数据模型 | 无 schema（Schema-less）| 强 schema（固定表结构）|
| 事务支持 | 仅行级事务，无跨行跨表事务 | 完整 ACID 事务 |
| 查询能力 | 仅支持主键和 Scan，不支持 JOIN | 丰富的 SQL 查询、JOIN、聚合 |
| 扩展性 | 水平扩展（PB 级）| 垂直扩展为主 |
| 索引 | 无二级索引（需依赖协处理器实现）| 支持主键索引、二级索引 |
| 数据类型 | 字节数组（byte[]）| 丰富的数据类型 |

### 什么场景选择 HBase？

✅ **适合使用 HBase 的场景：**
- **海量数据存储**：数据量达到 TB~PB 级别，RDBMS 无法承载
- **高并发随机读写**：如用户行为日志、消息流水、传感器数据
- **写多读少或读写均衡**：写入性能极佳
- **需要强一致性**：HBase 保证行级强一致性
- **半结构化/非结构化数据**：无需固定表结构
- **与 Hadoop 生态集成**：与 HDFS、MapReduce、Spark 深度集成

❌ **不适合使用 HBase 的场景：**
- 需要复杂 SQL 查询、多表 JOIN
- 需要二级索引、多条件组合查询
- 数据量较小（GB 以下），传统 RDBMS 足够
- 需要完整事务支持（跨行跨表）

**典型使用案例：**
- Facebook 用来存储消息基础设施
- Twitter 存储推文和用户时间线
- 淘宝用于交易历史记录存储
- 京东用于商品详情和订单数据
