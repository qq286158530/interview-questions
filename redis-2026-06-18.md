## 题目一：Redis 有哪些数据类型？应用场景是什么？

### 答案

Redis 支持五种基本数据类型，每种都有其独特的应用场景：

**1. String（字符串）**
- 最基本的数据类型，可存储任何字符串（包括 JSON、数字等）
- 最大容量 512MB
- 应用场景：缓存、计数器、分布式锁、Session 存储

**2. Hash（哈希）**
- 是一个键值对集合，特别适合存储对象
- 应用场景：存储用户信息、商品详情、配置信息

**3. List（列表）**
- 按插入顺序排序的字符串链表，可在头部或尾部操作
- 应用场景：消息队列（利用 LPUSH + BRPOP）、最新消息列表、关注列表

**4. Set（集合）**
- 无序且不重复的字符串集合，支持交集、并集、差集运算
- 应用场景：标签系统、好友关系、去重场景抽奖

**5. Sorted Set（有序集合，ZSet）**
- 每个元素关联一个分数，按分数排序，支持范围查询
- 应用场景：排行榜、延时队列、优先级队列

**6. Bitmap（位图）**
- 针对 String 类型的位操作，适合大数据量布尔统计
- 应用场景：用户签到、在线状态统计、布隆过滤器基础

**7. HyperLogLog**
- 用于基数统计，内存占用极低（固定 12KB）
- 应用场景：UV 统计、独立 IP 统计

## 题目二：Redis 的持久化机制是什么？各有何优缺点？

### 答案

Redis 提供了两种持久化机制：

**RDB（Redis Database）**

原理：定时将内存中的数据快照写入二进制文件 `dump.rdb`。

优点：
- 文件紧凑，恢复速度快（比 AOF 快）
- 适合大规模数据备份和灾难恢复
- 性能影响小（fork 子进程完成，不阻塞主进程）

缺点：
- 可能丢失最后一次快照后的数据
- fork 进程时可能因数据量大而导致短暂卡顿

配置示例：
```conf
save 900 1   # 900秒内至少1个key变化则触发
save 300 10  # 300秒内至少10个key变化则触发
```

**AOF（Append Only File）**

原理：记录所有写命令到日志文件，重启时重放命令恢复数据。

优点：
- 数据安全性更高（可配置同步策略）
- 写入append，不存在数据损坏风险
- rewrite 机制可压缩文件

缺点：
- 文件通常比 RDB 大
- 恢复速度慢于 RDB
- 高并发下可能有一定性能影响

同步策略：
```conf
appendonly yes
appendfsync always     # 每个写命令都同步（最安全但最慢）
appendfsync everysec   # 每秒同步（默认，推荐）
appendfsync no         # 依赖系统（最快但可能丢数据）
```

**生产环境推荐策略**：
同时开启 RDB 和 AOF，主从复制保证数据安全。

## 题目三：Redis 的过期键删除策略有哪几种？

### 答案

Redis 有三种过期键删除策略：

**1. 定时删除**
- 设置键的同时，创建一个定时器（Timer），键过期时立即删除
- 优点：对内存友好，确保过期键立即被删除
- 缺点：对 CPU 不友好，在 CPU 紧张时会响性能

**2. 惰性删除（Lazy Expire）**
- 每次访问键时检查是否过期，过期则删除
- 优点：对 CPU 友好，只在必须访问时才检查
- 缺点：对内存不友好，过期键可能长期占用内存

**3. 定期删除**
- 每隔一段时间，抽查一部分键，删除其中过期的键
- 是定时删除和惰性删除的折中方案
- 通过 `hz` 参数控制检查频率

**Redis 采用的是惰性删除 + 定期删除的组合策略**

**内存淘汰机制（当内存不足时）**

当内存达到上限，Redis 采用以下策略淘汰键：

| 策略 | 说明 |
|------|------|
| `volatile-lru` | 从已设置过期时间的键中，淘汰最近最少使用的 |
| `allkeys-lru` | 从所有键中淘汰最近最少使用的 |
| `volatile-ttl` | 从已设置过期时间的键中，淘汰即将过期的 |
| `volatile-random` | 从已设置过期时间的键中随机淘汰 |
| `allkeys-random` | 从所有键中随机淘汰 |
| `no-eviction` | 不淘汰，返回错误（默认） |
| `volatile-lfu` | 从已设置过期时间的键中淘汰使用频率最少的 |
| `allkeys-lfu` | 从所有键中淘汰使用频率最少的 |

## 题目四：Redis 主从复制原理是什么？如何解决数据一致性？

### 答案

**主从复制架构**

```
Master ──────> Slave1
  │                │
  └──────> Slave2 ──┘
```

**复制原理分为三个阶段**：

**1. 建立连接阶段**
- 从服务器向主服务器发送 `PSYNC` 命令
- 主服务器返回 `FULLRESYNC` 和运行 ID，开始全量同步

**2. 全量同步**
- 主服务器 fork 子进程生成 RDB 文件
- 发送给从服务器，从服务器加载到内存
- 主服务器将这期间的写命令缓存起来

**3. 增量同步**
- 全量同步后，进入命令传播阶段
- 主服务器将写命令发送给从服务器
- 从服务器执行命令保持数据一致

**关键配置**：
```conf
replicaof masterip masterport  # 从节点配置
masterauth <password>          # 主节点密码认证
replica-read-only yes          # 从节点只读
```

**数据一致性问题及解决方案**

主从复制存在延迟，可能导致数据不一致：

**解决方案一：强制读主库**
- 对一致性要求高的读请求直接访问主库

**解决方案二：RedLock（分布式锁）**
- 写入后短暂延迟再读取
- 使用分布式锁保证读写顺序

**解决方案三：使用 Canal 订阅 binlog**
- 业务直接读取主库 binlog
- 保证强一致性，但架构复杂

**解决方案四：Redis Cluster**
- 客户端哈希到对应节点
- 写入和读取路由到同一节点

**脑裂问题（Split-Brain）**
当主节点网络分区，从节点升级为主节点后，原主节点恢复可能导致数据冲突。

解决方案：
```conf
min-replicas-to-write 2  # 写入前至少2个从节点确认
min-replicas-max-lag 10 # 从节点延迟不超过10秒
```

## 题目五：Redis Cluster 的工作原理？如何实现分布式？

### 答案

**Redis Cluster 架构**

Redis Cluster 是 Redis 官方提供的分布式解决方案：

- **数据分片**：16384 个哈希槽（Hash Slot）
- **节点通信**：Gossip 协议（节点间定期交换信息）
- **故障检测**：PFAIL → FAIL 的主观/客观故障检测
- **自动故障转移**：从节点自动升级为主节点

**数据分布算法**

```
slot = CRC16(key) % 16384
```

使用 CRC16 校验算法计算 key 的哈希槽，确保数据均匀分布。

**节点角色**

| 角色 | 说明 |
|------|------|
| Master | 主节点，负责读写操作 |
| Replica | 从节点，复制主节点数据 |
| Slot | 槽，共 16384 个 |

**请求路由**

客户端可以连接任意节点，节点计算 key 所属槽：
- 如果槽属于当前节点，直接处理
- 如果槽属于其他节点，返回 MOVED 错误，告知客户端重定向

```
GET name
-MOVED 5798 127.0.0.1:6380  # 返回目标节点地址
```

**故障转移流程**

1. 集群中半数以上主节点标记某个主节点为 PFAIL（可能的故障）
2. 集群广播 FAIL 消息
3. 从节点向集群申请成为新的主节点（投票机制）
4. 获得多数投票的从节点升级为主节点
5. 新主节点接管故障节点的哈希槽

**集群配置示例**

```bash
# 节点1
redis-server --port 6379 --cluster-enabled yes --cluster-config-file nodes-6379.conf

# 节点2
redis-server --port 6380 --cluster-enabled yes --cluster-config-file nodes-6380.conf

# 初始化集群（至少3个主节点）
redis-cli --cluster create 127.0.0.1:6379 127.0.0.1:6380 127.0.0.1:6381 \
  127.0.0.1:6382 127.0.0.1:6383 127.0.0.1:6384 --cluster-replicas 1
```

**与 Codis/代理方案的对比**

| 特性 | Redis Cluster | Codis | 代理方案 |
|------|--------------|-------|---------|
| 客户端支持 | 需要集群支持 | 任意客户端 | 任意客户端 |
| 架构复杂度 | 中等 | 较高 | 较高 |
| 迁移方式 | 官方自动迁移 | zookeeper 协调 | 代理转发 |
| 可扩展性 | 一般 | 好 | 好 |
| 运维成本 | 中等 | 高 | 高 |
