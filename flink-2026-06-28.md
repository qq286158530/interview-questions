## 题目 1：Flink 的架构是怎样的？为什么 Flink 能做到 exactly-once 语义？

### 参考答案

Flink 采用 **Master-Slave 架构**，核心组件包括：

- **JobManager**：协调整个作业的执行，负责调度任务、管理 Checkpoint、协调算子等
- **TaskManager**：执行具体的 Task，每个 TaskManager 是一个 JVM 进程，负责执行算子子任务
- **ResourceManager**：管理 TaskManager 的 slot 和资源分配
- **Dispatcher**：接收用户提交的作业，启动 JobManager

Flink 通过 **Chandy-Lamport 分布式快照算法** 实现 exactly-once 语义：

1. **Barrier 对齐机制**：Checkpoint 过程中，数据流中会注入 Barrier，Barrier 将数据分割成不同的快照。算子收到所有输入分区的 Barrier 后才做快照，保证各分区数据一致。
2. **两阶段提交（Two-Phase Commit）**：Flink 的 Sink 通过两阶段提交协议与 Checkpoint 配合：算子完成快照时先"预提交"，Checkpoint 完成时再"正式提交"。如果作业失败，已预提交的数据会被回滚，保证输出只有一次。
3. **状态后端**：Flink 将状态存储在状态后端（如 RocksDB），与 Checkpoint 机制协同，保证故障恢复后状态精确恢复。

> **来源参考**：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/flink-architecture/

## 题目 2：什么是 Watermark？Watermark 的作用是什么？如何处理乱序数据？

### 参考答案

**Watermark（水位线）** 是 Flink 中用于处理**事件时间（Event Time）**乱序问题的机制，本质是一个时间戳。

### 作用

在事件时间语义下，数据到达的顺序可能和事件发生顺序不一致（网络延迟、分布式系统等）。Watermark 用于告诉算子：**在这个时间点之前的数据都已经到达了**，算子可以安全地触发窗口计算。

### 核心机制

- **Watermark 是一条数据**，携带时间戳 T，表示"所有时间戳 ≤ T 的数据都已到达"。
- 窗口会在以下两个条件同时满足时触发：
  1. 收到 Watermark 且 Watermark 时间 ≥ 窗口结束时间
  2. 窗口内已有数据

### 乱序处理策略

| 策略 | 说明 |
|------|------|
| **时间戳分配 + Watermark 生成** | 通常使用 `assignTimestampsAndWatermarks` 设置 Watermark 生成策略，常用 `BoundedOutOfOrdernessTimestampExtractor` 允许一定程度的乱序 |
| **allowedLateness** | 设置允许的延迟时间，窗口触发后仍然保留状态一段时间，迟到数据仍能触发窗口计算 |
| **侧输出流（Side Output）** | 超时未处理的数据可以通过侧输出流单独收集，避免丢失 |

### 示例代码

```java
WatermarkStrategy<MyEvent> strategy = WatermarkStrategy
    .<MyEvent>forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner((event, timestamp) -> event.getTimestamp());

DataStream<MyEvent> stream = env.fromSource(
    source, strategy, "MySource"
);
```

## 题目 3：Flink 的 Checkpoint 和 Savepoint 有什么区别？

### 参考答案

| 维度 | Checkpoint | Savepoint |
|------|------------|-----------|
| **触发方式** | Flink 自动周期性触发（由 JobManager 协调） | 手动触发（通过 CLI 命令） |
| **用途** | 作业故障恢复，保障 exactly-once | 作业计划性停机维护、版本升级、A/B 测试 |
| **存储位置** | 由 `state.backend` 配置决定，默认由 JobManager 管理 | 用户指定目录（手动指定） |
| **是否可删除** | 作业取消后默认删除 | 需手动删除 |
| **兼容性** | 与作业拓扑强关联，拓扑变化后可能失效 | 记录了完整的执行拓扑，支持从 Savepoint 恢复 |
| **性能影响** | 对作业性能有一定影响（异步快照可降低影响） | 暂停作业进行，需要暂停数据处理 |

### 核心原理

两者都基于 **Chandy-Lamport 分布式快照算法**，通过在数据流中插入 Barrier 实现状态一致性快照。区别在于生命周期管理和使用场景。

> **来源参考**：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/checkpoints/

## 题目 4：Flink 窗口函数有哪些类型？它们有什么区别？

### 参考答案

Flink 支持两大类窗口：**滚动窗口（Tumbling Window）** 和 **滑动窗口（Sliding Window）**，以及 **会话窗口（Session Window）** 和 **全局窗口（Global Window）**。

### 1. 滚动窗口（Tumbling Window）

- **特点**：窗口大小固定，不重叠
- **适用场景**：按固定时间或固定数量统计，如"每分钟UV"、"每100条计算一次"
- **示例**：每 5 分钟统计一次销售额

```java
DataStream<Tuple2<String, Long>> counts = stream
    .keyBy(x -> x.f0)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .sum(1);
```

### 2. 滑动窗口（Sliding Window）

- **特点**：窗口大小固定，但可以重叠（通过 slide 参数控制）
- **参数**：`size`（窗口大小）+ `slide`（滑动步长）
- **适用场景**：需要平滑统计、计算移动平均等
- **示例**：窗口大小 10 分钟，滑动步长 5 分钟

```java
.window(SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(5)))
```

### 3. 会话窗口（Session Window）

- **特点**：基于活动间隙（gap）动态创建窗口，数据间隔超过 gap 则开启新窗口
- **适用场景**：用户行为分析，按会话统计
- **示例**：用户操作间隔超过 30 秒视为不同会话

```java
.window(EventTimeSessionWindows.withGap(Time.minutes(1)))
```

### 4. 全局窗口（Global Window）

- **特点**：所有数据分配到同一个窗口，需要配合自定义触发器使用
- **适用场景**：需要自定义触发逻辑的场景

### 核心区别总结

| 窗口类型 | 是否重叠 | 大小是否固定 | 典型用途 |
|----------|----------|--------------|----------|
| 滚动窗口 | 否 | 是 | 固定周期统计 |
| 滑动窗口 | 是 | 是 | 移动平均、趋势分析 |
| 会话窗口 | 否 | 否 | 用户会话分析 |
| 全局窗口 | — | — | 自定义聚合 |

> **来源参考**：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream/operators/windows/

## 题目 5：Flink 的状态后端（State Backend）有哪几种？各自适用什么场景？

### 参考答案

Flink 有三种状态后端：

### 1. HashMapStateBackend

- **存储方式**：将状态存储在 TaskManager 的 JVM 堆内存中（Java 对象）
- **特点**：读写性能最高，适合状态较小的场景
- **适用场景**：
  - 状态较小（如聚合统计维度低）
  - 对延迟极其敏感
  - 机器内存充足
- **限制**：受 JVM 堆内存大小限制，不适合超大状态

```java
env.setStateBackend(new HashMapStateBackend());
```

### 2. EmbeddedRocksDBStateBackend

- **存储方式**：将状态存储在 RocksDB（嵌入式KV数据库）中，RocksDB 数据存储在本地磁盘
- **特点**：支持增量 Checkpoint，状态可以超过内存大小
- **适用场景**：
  - 大状态场景（GB级别状态）
  - 状态需要持久化
  - 需要增量 Checkpoint 减少快照开销
- **限制**：读写性能比 HashMapStateBackend 低（需序列化/反序列化）

```java
env.setStateBackend(new EmbeddedRocksDBStateBackend());
```

### 3.（旧版）FsStateBackend

- **存储方式**：将状态存储在 TaskManager 内存中，但定期将快照持久化到 HDFS 等文件系统
- **特点**：适合超大门状态的快照存储，但状态读写在内存
- **现状**：Flink 1.13+ 已废弃，功能已被 HashMapStateBackend + 外部 Checkpoint 取代

### 选型建议

| 场景 | 推荐后端 |
|------|----------|
| 小状态、低延迟 | HashMapStateBackend |
| 大状态、需要增量 Checkpoint | EmbeddedRocksDBStateBackend |
| 状态非常大但不想用 RocksDB | HashMapStateBackend + 外部文件系统 Checkpoint |

> **来源参考**：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/state_backends/
