## 题目一：Redis 数据类型有哪些？底层是如何实现的？

### 答案

**Redis 数据类型分为五大基础类型和四种扩展类型：**

| 类型 | 特点 | 应用场景 |
|------|------|----------|
| String | 字符串 | 缓存对象、计数器、分布式锁、Session |
| Hash | 哈希表 | 缓存对象、购物车 |
| List | 列表 | 消息队列（需注意：无消费组支持）、文章列表 |
| Set | 集合 | 点赞、共同关注、抽奖 |
| Zset | 有序集合 | 排行榜、热搜排名 |

**扩展类型：**

- **BitMap**：签到、用户在线状态（2.2+）
- **HyperLogLog**：UV统计（2.8+）
- **GEO**：地理位置（3.2+）
- **Stream**：消息队列（5.0+）

**底层数据结构：**

| 数据类型 | 底层实现 |
|----------|----------|
| String | SDS（简单动态字符串） |
| List | quicklist（3.2+ 替代了双向链表+压缩列表） |
| Hash | 压缩列表 / 哈希表 |
| Set | 整数集合 / 哈希表 |
| Zset | 压缩列表 / 跳表 |

## 题目二：Redis 是单线程吗？为什么单线程还这么快？

### 答案

**Redis 是单线程的吗？**

Redis 单线程指的是「接收客户端请求→解析请求→执行命令→返回数据」这个过程由主线程完成。但 Redis 并不是完全单线程：

- **Redis 2.6** 启动 2 个后台线程：`bio_close_file`（关闭文件）、`bio_aof_fsync`（AOF 刷盘）
- **Redis 4.0+** 新增 `bio_lazy_free` 线程，用于异步释放内存

**为什么单线程还这么快？**

1. **内存操作**：大部分操作在内存完成，采用高效数据结构，瓶颈是内存/网络而非 CPU
2. **无竞争开销**：避免了多线程切换和锁竞争，不会有死锁问题
3. **I/O 多路复用**：使用 epoll 机制，一个线程处理多个 Socket 连接

**Redis 6.0 引入多线程**：仅针对网络 I/O（发送响应），命令执行仍是单线程，官方提示性能提升至少 1 倍。

## 题目三：Redis 有哪几种持久化方式？各自的优缺点是什么？

### 答案

**Redis 共有三种持久化方式：**

### 1. AOF 日志（Append Only File）

**原理**：每执行一条写命令，就将该命令追加到 AOF 文件末尾。

**写回策略**：

| 策略 | 含义 | 优点 | 缺点 |
|------|------|------|------|
| Always | 同步写回 | 数据不丢失 | 占用 CPU，影响性能 |
| Everysec | 每秒写回 | 性能适中 | 最多丢 1 秒数据 |
| No | 操作系统决定 | 无开销 | 数据丢失不可控 |

**AOF 重写**：后台 `bgrewriteaof` 子进程完成，读取数据库所有键值对，用一条命令记录，避免重复写入。

### 2. RDB 快照（Redis Database）

**原理**：将某一时刻的内存数据以二进制格式写入磁盘。

**命令**：

- `save`：主线程执行，会阻塞
- `bgsave`：子进程执行，不阻塞

**优势**：恢复速度快（直接读入内存），适合备份。

### 3. 混合持久化（Redis 4.0+）

**原理**：AOF 重写时，先以 RDB 格式写入全量数据，再记录增量 AOF 命令。

**优点**：启动快 + 数据丢失少  
**缺点**：AOF 可读性差，不兼容 4.0 之前版本

| 方式 | 恢复速度 | 数据安全性 | 文件大小 |
|------|----------|------------|----------|
| RDB | 快 | 差（全量快照） | 小 |
| AOF | 慢 | 好（可配置） | 大 |
| 混合 | 快 | 好 | 中 |

## 题目四：Redis 如何实现高可用？哨兵模式和 Cluster 切片集群的区别？

### 答案

### 高可用方案对比

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| 主从复制 | 一主多从，读写分离 | 数据备份 |
| 哨兵模式 | 主从复制 + 自动故障转移 | 自动化运维 |
| Cluster | 数据分片（16384 槽） | 大数据量、高并发 |

### 主从复制

- 主节点可读写，从节点只读
- 异步同步，**无法保证强一致性**
- 主节点宕机需手动恢复

### 哨兵模式

- 监控主从节点状态
- 自动故障转移，选举新主节点
- 客户端自动感知新主节点

### Cluster 切片集群

- **16384 个哈希槽**，通过 CRC16 取模分配
- 每个节点存储部分数据
- 支持自动故障转移和数据迁移

### 脑裂问题

**原因**：主节点失联但仍接收写请求，哨兵选举出新主节点后，旧主节点降级为从时会清空缓冲区导致数据丢失。

**解决方案**：配置 `min-slaves-to-write` 和 `min-slaves-max-lag`，确保从库数量和复制延迟达标才允许主库接收写请求。

## 题目五：Redis 的过期删除策略和内存淘汰策略有什么区别？

### 答案

### 一、过期删除策略

Redis 采用「惰性删除 + 定期删除」组合策略：

**惰性删除**：访问 key 时才检查是否过期
- ✅ CPU 友好，开销小
- ❌ 过期 key 长期占用内存

**定期删除**：每隔一段时间随机检查 20 个 key
- 删除比例 > 25% 继续检查，否则停止
- 循环执行，时间上限 25ms
- ✅ CPU 和内存平衡
- ❌ 难以精准控制频率

### 二、内存淘汰策略（内存满时）

| 策略 | 含义 |
|------|------|
| `noeviction` | 不淘汰，返回错误（默认） |
| `volatile-lru` | 从已设置过期时间的 key 中淘汰最近最少使用 |
| `allkeys-lru` | 所有 key 中淘汰最近最少使用 |
| `volatile-ttl` | 从已设置过期时间的 key 中淘汰最早过期 |
| `volatile-random` | 从已设置过期时间的 key 中随机淘汰 |
| `allkeys-random` | 所有 key 中随机淘汰 |
| `volatile-lfu` | 从已设置过期时间的 key 中淘汰最少使用（4.0+） |
| `allkeys-lfu` | 所有 key 中淘汰最少使用（4.0+） |

### 三、持久化中的过期键处理

**RDB**：
- 生成时：过期键不写入
- 加载时：主服务器会过滤过期键

**AOF**：
- 写入时：过期键保留，删除时追加 DEL 命令
- 重写时：过期键不写入
