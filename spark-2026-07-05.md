## 题目一：Spark 架构核心原理

### 问题
请描述 Apache Spark 的整体架构设计，以及 Driver、Executor、Master、Worker 各自的作用和协作关系。

### 答案

**Spark 架构概述：**

Spark 采用 **Master-Slave** 架构，核心组件包括：

| 组件 | 角色 | 职责 |
|------|------|------|
| **Driver** | 主控节点 | 解析用户代码，构建 DAG 图，将作业拆分为多个 Stage，按序调度 Task 到 Executor 执行 |
| **Executor** | 工作节点 | 真正执行 Task，将数据存入内存或磁盘；每个应用独占 Executor 进程 |
| **Master** | 资源管理主节点 | 负责集群资源管理（Worker 注册、资源分配）；Cluster 模式下参与调度 |
| **Worker** | 资源管理从节点 | 向 Master 注册本机资源，启动和管理 Executor 进程 |

**协作流程：**
1. 用户提交 Spark Application，Driver 启动并创建 `SparkContext`
2. Driver 向 Cluster Manager 申请资源（Master/YARN/K8s）
3. Master 通知 Worker 启动 Executor
4. Executor 向 Driver 注册，反向建立连接
5. Driver 根据 Shuffle 划分 Stage，构建 DAG，调度 Task 到 Executor
6. Executor 执行 Task，结果返回 Driver

**两种部署模式：**
- **Client 模式**：Driver 在提交机器上运行，适合交互式调试
- **Cluster 模式**：Driver 由 Master 在集群内部启动，适合生产环境

## 题目二：RDD 深度理解

### 问题
什么是 RDD？RDD 有哪些核心特性？为什么 Spark 选择 RDD 作为核心数据结构？

### 答案

**RDD（Resilient Distributed Dataset）** 是 Spark 最底层的数据抽象，表示不可变、可分区、可以并行计算的数据集合。

**五大核心特性：**

1. **` partitions ` — 数据分区列表**
   RDD 的数据被切分为多个分区，每个分区是并行计算的最小单位。

2. **` computeFn(partiton) ` — 计算函数**
   每个分区都有计算函数，延迟执行（Lazy Evaluation）。

3. **` dependencies ` — 依赖关系**
   RDD 通过依赖关系构建 Lineage（血统链），支持容错。当某个分区数据丢失时，可从父 RDD 重新计算恢复。

4. **` partitioner ` — 分区器（可选）**
   对于键值对 RDD，可指定 Hash 分区器或 Range 分区器，决定 Shuffle 时的分区策略。

5. **` preferredLocations ` — 优先位置（可选）**
   记录每个分区最佳计算位置（数据本地性），实现数据移动最小化。

**为什么选择 RDD：**
- **简单性**：不可变、纯函数式，易于推理和优化
- **高效容错**：通过 Lineage 而非数据副本实现容错，开销低
- **数据本地性**：调度时考虑数据位置，减少网络传输
- **可组合性**：转换算子（map/filter/groupBy）天然支持 DAG 优化

> 💡 注意：RDD 适合底层控制和迭代计算；DataFrame/Dataset 适合结构化数据的声明式查询优化（Catalyst + Tungsten）。

## 题目三：DataFrame vs DataSet 区别

### 问题
Spark 中 DataFrame 和 DataSet 有何区别？各自适用场景是什么？

### 答案

**核心区别：**

| 特性 | DataFrame | DataSet |
|------|-----------|---------|
| **数据类型** | `Row`（弱类型，类似数据库表） | 强类型 JVM 对象（Case Class） |
| **语言支持** | 多种语言（Python/SQL/R） | 仅 Scala/Java |
| **编译时检查** | 无（运行时才发现错误） | 有（编译期类型安全） |
| **序列化** | Tungsten 外部序列化 | Encoder 序列化 |
| **操作方式** | DSL / SQL | Lambda 表达式 |

**统一性：**
在 Scala 中，`DataFrame = DataSet[Row]`，二者共享同一套执行引擎。

**DataFrame 优势：**
- 简洁的 SQL 风格操作（`select`, `filter`, `groupBy`）
- 丰富的内置函数和 UDF 支持
- 跨语言通用，生态完善

**DataSet 优势：**
- 编译时类型检查，更安全
- 支持面向对象风格的 Lambda 操作
- 复杂业务逻辑的代码可读性更好

**选择建议：**
- **结构化/半结构化数据**，多用 SQL/DSL → 选 **DataFrame**
- **业务逻辑复杂**，需要强类型保障 → 选 **DataSet**
- Python/R 用户 → 只能选 **DataFrame**

```scala
// DataFrame 示例
df.filter($"age" > 25).groupBy("department").count()

// DataSet 示例
case class Person(name: String, age: Int)
val ds: Dataset[Person] = spark.createDataSet(...)
ds.filter(_.age > 25).groupBy(_.department).count()
```

## 题目四：Shuffle 机制与优化

### 问题
什么是 Shuffle？为什么 Shuffle 是 Spark 性能瓶颈？如何优化 Shuffle？

### 答案

**什么是 Shuffle：**

Shuffle 是 Spark 在 **宽依赖**（Wide Dependency）时必须经历的数据重分区过程。当一个算子需要访问其他分区的数据时（如 `groupByKey`、`join`、`reduceByKey`），所有数据必须按 Key 重新分区并跨节点传输。

**Shuffle 执行流程：**

```
Map端                    Shuffle Write               Reduce端
┌─────────┐              ┌───────────┐               ┌─────────┐
│ ShuffleMapTask │  ──→  │ 磁盘文件   │  ──→  │ ResultTask │
│ (写分区数据)    │       │ (桶文件)    │       │ (读取数据)  │
└─────────┘              └───────────┘               └─────────┘
```

**为什么是性能瓶颈：**
1. **磁盘 I/O**：Shuffle 数据先写磁盘再读磁盘
2. **网络 I/O**：跨节点数据传输
3. **内存压力**：Map 端缓存、中间结果、聚合缓存共同挤压内存
4. **分区数影响**：过多小分区增加调度和文件开销

**核心优化策略：**

| 优化项 | 说明 |
|--------|------|
| **减少 Shuffle 次数** | 避免不必要的 `repartition`/`coalesce`，用 map-side 预聚合（如 `reduceByKey` > `groupByKey`） |
| **合理设置分区数** | 每个分区建议 128~256MB 数据；避免过多小分区 |
| **使用广播变量** | 小表 join 时广播，避免 Shuffle |
| **调整 Shuffle 文件数量** | `spark.sql.shuffle.partitions`（默认200），按数据量调整 |
| **开启 Shuffle 压缩** | `spark.shuffle.compress=true`，使用 Kryo 序列化 |
| **升级到 Sort-based Shuffle** | Spark 2.x+ 默认 Sort-based Shuffle，减少文件数 |

```scala
// 优化示例：小表广播
val result = bigTable.join(broadcast(smallTable), "key")

// 调整 Shuffle 分区
spark.conf.set("spark.sql.shuffle.partitions", "100")
```

## 题目五：广播变量与内存管理

### 问题
什么是广播变量？Spark 的内存管理模型是怎样的？如何调优 Spark 内存？

### 答案

### Part 1：广播变量

**广播变量原理：**
将**只读共享数据**（如维度表、配置表）高效地发送到每个 Executor，所有 Task 共享一份内存副本，避免每个 Task 独立存储造成巨大开销。

```
不使用广播：每个 Task 都复制一份数据  →  N 个 Task = N 份数据
使用广播：每个 Executor 持有一份数据  →  N 个 Task = 1 份/Executor
```

**使用方式：**
```scala
val broadcastVar = spark.sparkContext.broadcast(MapData)
val result = data.filter(row => broadcastVar.value.contains(row.id))
```

**注意事项：**
- 只适合小表（< 20MB），过大会导致 OOM
- 广播大表前先 `collect` 到 Driver 端再广播
- Spark SQL 中 `broadcast()` hint 可自动触发广播

---

### Part 2：Spark 内存管理模型

Spark 采用 **统一内存管理**（Unified Memory Management），堆内存被划分为两大区域：

```
┌──────────────────────────────────────────────────┐
│                    Heap Memory                     │
├─────────────────────┬────────────────────────────┤
│    Execution Memory  │     Storage Memory         │
│  (Shuffle/Hash/Agg) │   (RDD Cache/Broadcast)    │
│                     │                             │
│  ── 动态共享 ──       │   ── 动态共享 ──            │
│  不足时可借用对方      │   不足时也可借用对方         │
├─────────────────────┴────────────────────────────┤
│              User Memory (预留)                    │
│  (用户数据结构、元数据、内部元数据)                   │
├──────────────────────────────────────────────────┤
│              Reserved Memory (300MB)               │
└──────────────────────────────────────────────────┘
```

**核心参数：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `spark.memory.fraction` | 0.6 | 堆内存中用于 Execution + Storage 的比例 |
| `spark.memory.storageFraction` | 0.5 | Storage 区域占用的固定比例上限 |
| `spark.executor.memory` | — | 每个 Executor 的堆内存大小 |

**动态占用规则：**
- Storage 空间充裕时，Execution 可完全使用 `1 - fraction` 的全部空间
- Storage 空间不足时，可驱逐（evict）己方缓存数据
- **Execution 内存被 Storage 占用时，无法驱逐**，可能触发 OOM
