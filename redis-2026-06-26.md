## 题目一：Redis 和 Memcached 有什么区别？为什么选择 Redis 作为 MySQL 的缓存？

### 答案

**Redis 与 Memcached 的共同点：**
- 都是基于内存的数据库，一般都用来当作缓存使用
- 都有过期策略
- 两者的性能都非常高

**Redis 与 Memcached 的区别：**

| 特性 | Redis | Memcached |
|------|-------|-----------|
| 数据类型 | 支持丰富数据类型（String、Hash、List、Set、ZSet） | 只支持简单的 key-value 类型 |
| 持久化 | 支持数据持久化，重启后可恢复 | 无持久化功能，数据全在内存 |
| 集群模式 | 原生支持集群模式 | 无原生集群，需客户端实现分片 |
| 高级特性 | 支持发布订阅、Lua 脚本、事务等 | 不支持 |

**为什么选择 Redis 作为 MySQL 的缓存？**

1. **高性能**：用户第一次访问 MySQL 数据较慢（从硬盘读取），将数据缓存在 Redis 中后可直接从内存读取，速度极快。

2. **高并发**：单台 Redis 的 QPS（每秒处理请求数）是 MySQL 的 10 倍。Redis 单机 QPS 能轻松破 10w，而 MySQL 单机 QPS 很难破 1w。把部分数据转移到缓存后，用户请求可直接由 Redis 处理，减轻数据库压力。

## 题目二：Redis 的过期删除策略是什么？惰性删除和定期删除有什么区别？

### 答案

Redis 对过期键采用了「惰性删除 + 定期删除」两种策略配合使用。

**惰性删除**：不主动删除过期键，每次从数据库访问 key 时才检测是否过期，若过期则删除。

- **优点**：每次访问时才检查，对 CPU 时间最友好，系统资源消耗少
- **缺点**：如果过期 key 一直没有被访问，则会一直占用内存，造成空间浪费

**定期删除**：每隔一段时间随机从数据库中取出一定数量的 key 进行检查，删除其中的过期 key。

- 流程：从过期字典中随机抽取 20 个 key → 检查并删除已过期的 key → 如果过期 key 超过 5 个（25%），继续重复抽取；否则停止
- 为了防止循环过度导致卡死，Redis 限制了每次定期删除的执行时间不超过 25ms
- **优点**：能够删除一部分过期数据，减少内存浪费；对 CPU 时间相对友好
- **缺点**：执行频率难以把控——太频繁耗 CPU，太稀疏则过期 key 堆积

**两者配合使用**，在合理使用 CPU 时间和避免内存浪费之间取得平衡。

## 题目三：Redis 持久化机制有哪些？RDB 和 AOF 各自有什么优缺点？混合持久化是什么？

### 答案

Redis 共有三种持久化方式：**RDB 快照**、**AOF 日志**、**混合持久化**。

**RDB（Redis Database）快照**：将某一时刻的内存数据以二进制格式写入磁盘。

- 触发方式：`save`（主线程执行，会阻塞）、`bgsave`（子进程执行，不阻塞）
- 也可通过配置文件设置自动触发（900秒内至少1次修改 / 300秒内至少10次 / 60秒内至少10000次）
- 恢复数据速度快，因为是直接读取二进制文件
- 缺点：快照频率不好把握——太频繁影响性能，太低则数据丢失多

**AOF（Append Only File）日志**：每执行一条写操作命令，就以追加方式写入日志。

- 写回策略有三种：
  - `Always`：每次写命令执行完同步写回硬盘（最安全但最慢）
  - `Everysec`：每秒将缓冲区写回硬盘（默认，推荐使用）
  - `No`：由操作系统控制何时写回（最快但最不安全）
- AOF 文件过大会触发**重写机制**（bgrewriteaof），压缩文件体积
- 缺点：数据恢复速度比 RDB 慢

**混合持久化**（Redis 4.0+）：结合 RDB 和 AOF 的优点。

- AOF 重写时，先以 RDB 格式写入全量数据，再以 AOF 格式写入增量命令
- 加载时先快速加载 RDB 部分，再加载增量 AOF 部分
- 优点：启动快 + 丢失数据少；缺点：可读性差，兼容性低（4.0之前版本不可用）

## 题目四：Redis 主从复制原理是什么？哨兵模式解决了什么问题？集群脑裂是什么，如何避免？

### 答案

**主从复制**：采用一主多从模式，主服务器进行读写操作，从服务器只读并接收主服务器同步的写操作命令。

- 命令传播阶段：主服务器收到写命令后异步发送给从服务器（不等待从服务器执行完成）
- 因此主从数据无法做到强一致性，存在一定不一致是正常现象
- 首次同步采用全量复制；后续采用增量复制

**哨兵模式**：监控主从服务器状态，在主服务器故障时自动进行故障转移，选举新的主节点。

- 解决了主从架构中"故障需手动恢复"的问题
- 哨兵节点本身也需集群部署以保证高可用

**集群脑裂**：指在主从架构中，主节点网络突然中断，但与客户端网络正常。哨兵误判主节点宕机，选举出新主节点，此时存在两个主节点都在接收写入。当网络恢复后，旧主节点降为从节点并执行全量同步，导致之前客户端写入的数据全部丢失。

**解决方案**：配置以下两个参数：

```conf
min-slaves-to-write 1    # 主节点必须至少有1个从节点连接
min-slaves-max-lag 12    # 主从复制延迟不能超过12秒
```

这样，当主节点与从节点通信超时或从节点数量不足时，主节点会拒绝接收客户端写请求，避免在脑裂期间产生数据丢失。

## 题目五：Redis 单线程模型是什么？为什么单线程的 Redis 还能这么快？Redis 6.0 之后引入了多线程是为什么？

### 答案

**Redis 单线程指的是**：「接收客户端请求 → 解析请求 → 数据读写等操作 → 发送响应」这一过程由主线程完成。但 Redis 实际上并非完全单线程，启动时会创建后台线程（BIO）处理关闭文件、AOF 刷盘、lazyfree 释放内存等耗时任务。

**Redis 单线程为什么还这么快？**

1. **内存操作 + 高效数据结构**：Redis 大部分操作在内存中完成，且底层使用了 SDS（动态字符串）、跳表、压缩列表等高效数据结构，性能瓶颈在内存/网络而非 CPU
2. **无锁竞争**：单线程避免了多线程切换开销和死锁问题
3. **I/O 多路复用**：采用 epoll 机制，一个线程处理多个 Socket 请求，内核同时监听多个 fd，有请求到达才处理，实现了高性能

**Redis 6.0 之后引入多线程**：主要针对**网络 I/O** 进行优化。

- Redis 6.0 前网络 I/O 和命令执行都是单线程；6.0 后用多个 I/O 线程分担网络读写压力
- **命令执行仍是单线程**，多线程仅用于网络层面
- 官方测试：多线程 I/O 使性能提升至少一倍以上
- 默认 I/O 线程只处理写响应（write client socket），读请求需设置 `io-threads-do-reads yes`
- 建议：4 核 CPU 设置 2-3 个 I/O 线程，8 核设置 6 个

> **来源**：[图解 Redis · 线程模型](https://xiaolincoding.com/redis/base/redis_interview.html)
