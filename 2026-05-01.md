## 题目1：String 为什么设计成不可变的？它带来了哪些好处？

### 答案

**不可变性的实现方式：**
String 类被 `final` 修饰，不能被继承；内部存储字符的数组 `value` 被 `private final` 修饰； String 类没有提供任何公开的 setter 方法来修改底层字符数组，所有"修改"操作都返回新对象。

**为什么要设计成不可变的（5个核心原因）：**

1. **字符串常量池优化**：JVM 中维护了一块字符串常量池，相同字面量的 String 对象可以共享。不可变性保证了池中对象的安全性和复用效率——如果 String 可变，两个相同字符串引用指向同一对象时，一个修改会影响另一个，导致常量池逻辑被破坏。

2. **HashCode 缓存与 HashMap 性能**：String 在 `hashCode()` 计算时会缓存哈希值（`private int hash`），因为 String 不可变，hash 值始终固定，可以反复使用。这是 HashMap 等哈希结构性能优化的重要前提。频繁修改 String 会导致哈希失效。

3. **线程安全**：不可变对象天然线程安全，不需要任何同步措施。多线程环境下可以直接共享 String 对象而无需担心数据竞争。

4. **安全性**：Java 的类加载机制、文件路径、网络连接等场景都大量使用 String 作为参数。如果 String 可变，可能导致认证信息被恶意修改（如数据库密码、文件路径等），造成严重的安全漏洞。

5. **面向对象设计的纯粹性**：String 在 Java 中被设计为"值对象"，类似基本数据类型，强调值相等性而非引用身份。不可变性使得 `String.intern()`、散列存储、参数传递等行为都可预测。

**补充——不可变性的代价：**
每次"修改"（concat、replace 等）都会创建新对象，旧对象成为垃圾。如果频繁修改会产生大量临时对象，增加 GC 压力。因此对大量字符串拼接的场景，建议使用 `StringBuilder`（非线程安全）或 `StringBuffer`（线程安全）。

## 题目2：volatile 和 synchronized 的核心区别是什么？各自解决了什么问题？

### 答案

| 对比维度 | volatile | synchronized |
|---------|---------|-------------|
| **作用** | 保证变量可见性 | 保证原子性+可见性+有序性 |
| **原理** | 写操作后强制刷新 CPU 缓存到主存；读操作前强制从主存取 | 通过 Monitor 锁实现，线程互斥进入临界区 |
| **对原子性的影响** | 不保证！`i++` 这种操作 volatile 无法保护 | 保证！整个同步块代码原子执行 |
| **对有序性的影响** | 禁止指令重排序（内存屏障） | 保证有序（临界区内代码块外禁止重排序） |
| **性能** | 轻量，开销小 | 重量级，涉及内核态切换 |
| **可否修饰** | 变量（字段） | 方法、代码块 |

**volatile 的核心作用（两步）：**
1. **写屏障（Write Barrier）**：对 volatile 变量的写操作，会先执行完毕并将结果刷回主存，通知所有 CPU 核心的缓存失效。
2. **读屏障（Read Barrier）**：读操作强制从主存读取最新值，禁止使用缓存中的旧值。

**典型使用场景：**
```java
// volatile 适用于：状态标志位、哨兵变量
volatile boolean ready = false;
// 不适用于：count++ 这种复合操作
```

**synchronized 的底层原理（JDK 1.6 之后优化）：**
锁升级过程：**无锁 → 偏向锁（第一次CAS加锁）→ 轻量级锁（自旋）→ 重量级锁（Monitor）**。非竞争场景下几乎无性能损耗。

**两者配合使用（最常见场景）：**
```java
private volatile boolean flag = false;
// volatile 保证可见性
public synchronized void setFlag() {
    // synchronized 保证原子性
    this.flag = true;
}
```

## 题目3：Spring @Import 注解的作用和使用场景

### 答案

**@Import 的核心作用：**
跳过传统 `@ComponentScan`，直接向 Spring IoC 容器注册 Bean 定义。支持四种子用法：

**1. 导入普通配置类**
```java
@Import(MyConfig.class)
public class AppConfig { }
```
将 `MyConfig` 中所有 `@Bean` 方法定义的 Bean 直接注入 IoC 容器，不扫描该类包路径。

**2. 导入实现了 ImportSelector 接口的类**
```java
@Import(MyImportSelector.class)
public class AppConfig { }

public class MyImportSelector implements ImportSelector {
    @Override
    public String[] selectImports(AnnotationMetadata importingClassMetadata) {
        // 动态返回要注册的Bean全限定类名数组
        return new String[]{"com.example.FooBean", "com.example.BarBean"};
    }
}
```
常用于 Spring Boot 自动装配：`spring.factories` 文件中的 `EnableAutoConfiguration` 就是通过 ImportSelector 机制实现的。

**3. 导入实现了 ImportBeanDefinitionRegistrar 接口的类**
```java
@Import(MyRegistrar.class)
public class AppConfig { }

public class MyRegistrar implements ImportBeanDefinitionRegistrar {
    @Override
    public void registerBeanDefinitions(AnnotationMetadata metadata,
                                        BeanDefinitionRegistry registry) {
        // 手动注册Bean定义，可指定Bean名称、作用域等
        RootBeanDefinition def = new RootBeanDefinition(FooBean.class);
        registry.registerBeanDefinition("foo", def);
    }
}
```

**4. 导入普通 @Configuration 类（配合 @ComponentScan）**
```java
@Import({DatabaseConfig.class, RedisConfig.class})
public class MainConfig { }
```

**@Import vs @ComponentScan 对比：**
- `@ComponentScan`：扫描指定包路径，满足条件的类都被加载（需要类有 `@Component/@Service/@Repository/@Controller` 等元注解）
- `@Import`：精确导入，直接将目标类的 `@Bean` 定义注入容器，不依赖包扫描

**典型应用场景：**
- **Spring Boot 自动装配**：`@EnableAutoConfiguration` 本质上就是通过 `@Import(AutoConfigurationImportSelector)` 批量导入配置类
- **MyBatis 集成**：`@MapperScan` 内部使用了 `ImportBeanDefinitionRegistrar` 机制
- **条件装配**：配合 `@Conditional` 实现环境感知的精确注册

## 题目4：Sentinel 滑动窗口限流算法原理是什么？与令牌桶的区别？

### 答案

**Sentinel 滑动窗口算法核心原理：**

Sentinel 使用**滑动窗口算法**统计请求量和响应时间，将时间轴切分为多个小窗口（Bucket），每个窗口记录通过请求数、拒绝请求数、总耗时等指标。

**结构示意：**
```
时间轴 → [窗口1(0-1s)] [窗口2(1-2s)] [窗口3(2-3s)] [窗口4(3-4s)] ...
         ↑                                    ↑
       滑动窗口起点（当前时间 - 窗口大小）      当前窗口
```

**限流判断流程：**
1. 每次请求到达时，计算当前时间戳属于哪个窗口 bucket
2. 向前滑动，统计所有活跃 bucket 的请求总数
3. `通过的请求数 = sum(各窗口请求数)`
4. 如果 `通过的请求数 >= 阈值`，拒绝请求；否则通过并更新计数器

**关键参数：**
- `sampleCount`：窗口内 bucket 数量（默认 2）
- `intervalInMs`：窗口总时长（默认 1 秒），即 1 秒内最大 QPS

**Sentinel 的两种统计维度：**
- **QPS 限流**：滑动窗口计数
- **平均响应时间限流**：滑动窗口内所有 RT 求平均

**与令牌桶（Token Bucket）的核心区别：**

| 特性 | Sentinel 滑动窗口 | 令牌桶 |
|------|-----------------|--------|
| **允许突发流量** | 依赖窗口内统计，允许窗口边界突发 | 允许以 maxQps 持续产生令牌，支持一定突发 |
| **精度** | 高（精确到滑动窗口） | 依赖时间精度 |
| **实现复杂度** | 中等，需维护多个 bucket | 简单（单桶计数） |
| **内存占用** | 较高（多个 bucket） | 极低 |
| **适用场景** | 精确流量整形、服务降级 | 固定速率限流、突发场景 |

**Sentinel 滑动窗口优势：**
解决了**窗口边界突发问题**（ Leap Array 算法）。老式计数器限流在窗口切换瞬间可能涌入 2 倍 QPS，滑动窗口将统计平滑分散到多个 bucket，避免了这个问题。

## 题目5：Feign 远程调用核心原理是什么？

### 答案

**Feign 是一个声明式 HTTP 客户端**，只需定义一个接口并添加注解，即可完成远程 HTTP 调用，无需手写 HTTP 客户端代码。

**核心原理分三步：**

**Step 1：编译期注解处理 + 动态代理生成**

1. 项目启动时，Feign 扫描所有标记了 `@FeignClient` 的接口
2. 通过 `Feign.Builder` 为每个接口生成 **JDK 动态代理对象**（基于 `InvocationHandler`）
3. 代理类实现了目标接口，调用方法时触发 `invoke()` 方法

**Step 2：请求构造（RequestTemplate）**

`InvocationHandler.invoke()` 接收到方法调用后：
1. 读取方法上的 `@RequestMapping` 等注解，提取 URL、HTTP 方法（GET/POST）、请求参数
2. 通过 `RequestTemplate.Builder` 构建 `RequestTemplate` 对象
3. 参数处理：@Param → URL 查询参数；@RequestBody → 请求体；@RequestHeader → 请求头

**Step 3：HTTP 执行 + 响应处理**

1. `RequestTemplate` 被转换为 `Request` 对象（封装 URL、Headers、Body）
2. 通过 `Client`（可插入 OkHttp / Apache HttpClient / Feign 默认 Client）发送 HTTP 请求
3. 响应回来后，通过 `Decoder` 将 `Response` 反序列化为方法返回类型
4. 异常处理：HTTP 4xx/5xx 通过 `ErrorDecoder` 转换为自定义业务异常

**关键扩展点（可定制）：**
- `Encoder`/`Decoder`：请求编解码（JSON、Protobuf）
- `ErrorDecoder`：自定义错误处理
- `Client`：替换底层 HTTP 客户端（支持连接池、超时配置）
- `Logger`：请求日志增强

**与 Ribbon 的关系：**
Spring Cloud 整合 Feign 时，Feign 自动集成了 Ribbon，实现了**客户端负载均衡**。`@FeignClient(name = "service-id")` 中的 `name` 指向 Ribbon 注册的服务名，Ribbon 从 Eureka/Nacos 获取服务实例列表并完成负载均衡。

**典型注解：**
```java
@FeignClient(name = "user-service", path = "/api/users")
public interface UserClient {
    @GetMapping("/{id}")
    User getUser(@PathVariable Long id);
    
    @PostMapping
    Result saveUser(@RequestBody User user);
}
```

## 题目6：Kafka 日志段（Log Segment）机制与索引原理

### 答案

**日志段是 Kafka 存储的核心概念：**

每个 Topic 的每个 Partition 在磁盘上对应一个目录，分割为多个 **Log Segment 文件**（`.log`、`.index`、`.timeindex` 三件套）。

**文件结构：**
```
partion-0/
  00000000000000000000.log      # 数据文件（消息集合）
  00000000000000000000.index    # 偏移量索引（物理位置映射）
  00000000000000000000.timeindex # 时间戳索引
  00000000000000000100.log      # 第二个 segment
  00000000000000000100.index
  00000000000000000100.timeindex
  ...
```

**Log Segment 核心配置：**
- `log.segment.bytes`：单个 .log 文件最大字节数（默认 1GB）
- `log.roll.ms / log.roll.hours`：滚动时间（到达时间阈值也触发新建 segment）
- `log.retention.ms`：消息保留时间（超过则整个 segment 删除）

**索引文件的作用：**

Kafka 使用**稀疏索引**（Sparse Index），并非每条消息建立索引，而是每隔一定消息数（如 4096 条）建立一个索引项：

**偏移量索引（.index）：**
- 结构：每条索引项 = `相对偏移量（4字节） + 物理位置（4字节）`
- `相对偏移量 = 绝对偏移量 - segment.baseOffset`，使用相对值减少索引文件大小
- 查找时，通过二分查找定位到目标偏移量所在的物理位置，再顺序扫描 .log 文件

**时间戳索引（.timeindex）：**
- 结构：每条索引项 = `时间戳（8字节） + 物理位置（4字节）`
- 支持按时间戳快速定位消息（用于日志保留按时间删除场景）

**为什么用稀疏索引而非稠密索引：**
- Kafka 写入量极大（每秒 GB 级别），稠密索引（每条消息都建立映射）会产生巨大的索引开销
- 稀疏索引在查找时只需扫描少量 .log 内容（二分定位 + 顺序扫描），兼顾查找效率和存储效率
- 查找复杂度：O(log n) 定位 segment + O(k) 在 segment 内顺序扫描

**写入流程：**
1. 消息追加到当前活跃 segment 的 .log 文件尾部（顺序写入，零拷贝）
2. 每写入 `index.interval.bytes` 条消息，同步追加一条索引记录到 .index 和 .timeindex

## 题目7：RabbitMQ 延迟消息的几种实现方式及原理

### 答案

**方式一：死信队列（Dead Letter Queue）模式**

利用 TTL（Time-To-Live）+ 死信交换机（DLX）实现。

**原理：**
1. 生产者将消息发到普通队列 `normal-queue`，设置 `x-message-ttl`
2. 消息过期后自动进入 `DLX（Dead Letter Exchange）`
3. DLX 绑定死信队列 `delay-queue`，消费者监听此队列即可延迟消费

```
生产者 → normal-queue（TTL=5s） → [过期] → DLX → delay-queue → 消费者
```

**缺点：** 不同 TTL 需要不同队列，TTL 相同时消息按 FIFO 处理，不支持更精确的延迟调度。

**方式二：RabbitMQ 延迟插件（rabbitmq_delayed_message_exchange）**

官方插件，基于 `x-delayed-message` 交换机类型。

**原理：**
1. 消息携带 `x-delay` 头（毫秒级延迟数值）
2. 消息被投递到延迟交换机，不是直接进入队列
3. 交换机内部维护一个定时器，到期后将消息路由到目标队列

```
生产者 → 延迟交换机（x-delay=5000）→ 目标队列 → 消费者
```

**优势：** 支持任意精确延迟、消息顺序保持、单个交换机多队列

**方式三：定时轮询 + 顺序表（延迟任务表）**

不适合对延迟精度要求高的场景。

**原理：**
1. 生产者将消息入库（延迟任务表，包含目标执行时间）
2. 消费者端定时任务（如每 500ms）轮询表，取出到达执行时间的消息
3. 消费消息并从表中删除

```
MySQL: delay_tasks(id, execute_time, payload, status)
消费者: 每500ms SELECT * FROM delay_tasks WHERE execute_time <= NOW()
```

**三种方式对比：**

| 方式 | 精度 | 消息顺序 | 资源消耗 | 适用场景 |
|------|------|---------|---------|---------|
| 死信队列 | 分钟级 | FIFO（TTL相同时） | 低 | 简单延迟、批量任务 |
| 延迟插件 | 毫秒级 | 保持 | 中 | 生产级延迟消息 |
| 轮询数据库 | 秒级 | 不保证 | 高（频繁DB轮询） | 低精度、非核心场景 |

**生产环境推荐：** 使用 `rabbitmq_delayed_message_exchange` 插件，是目前最成熟、精度最高的方案。

## 题目8：LangChain 核心组件与 Agent 开发流程

### 答案

**LangChain 是一个用于构建 LLM 应用的开源框架**，核心价值在于将 LLM 与外部工具、数据源、记忆模块解耦，以组件化方式组合。

**核心组件（6大模块）：**

**1. Model I/O（模型输入输出）**
- `PromptTemplate`：模板化提示词，动态注入变量
- `ChatModel / LLM`：统一封装多种大模型 API
- `OutputParser`：解析 LLM 输出（JSON、列表、结构化对象）

**2. Retrieval（检索增强）**
- `DocumentLoader`：多格式文档加载（PDF、TXT、Markdown）
- `TextSplitter`：文档分块（按字符数/句子/递归字符分割）
- `Embeddings`：文本向量化（OpenAI Embedding / HuggingFace）
- `VectorStore`：向量数据库集成（Chroma、Pinecone、FAISS、Milvus）
- `Retriever`：检索器接口，支持 BM25、相似度搜索

**3. Chains（链）**
- `LLMChain`：最基本的链：Prompt → LLM → Output
- `RetrievalQAChain`：RAG 标准实现链
- `SequentialChain`：顺序执行多个链
- `ConversationChain`：对话上下文链

**4. Memory（记忆）**
- `ConversationBufferMemory`：保留完整对话历史
- `ConversationSummaryMemory`：对历史摘要压缩，节省 token
- `VectorStoreRetrievedMemory`：向量存储的对话记忆

**5. Agents（智能体）**
- `Agent`：核心抽象，执行 Tools，决定行动
- `Tool`：外部工具（搜索、数据库、API调用、代码执行）
- ` Toolkit`：多个 Tool 的打包集合
- `AgentExecutor`：Agent 的执行引擎，运行 agent + tool 循环

**6. Callbacks**
- 钩子系统：记录 LLM 调用、日志、流式输出等事件

**Agent 典型开发流程：**

```python
from langchain.agents import load_tools, initialize_agent, AgentType
from langchain.llms import OpenAI

# 1. 初始化大模型
llm = OpenAI(temperature=0)

# 2. 加载工具（Tools）
tools = load_tools(["serpapi", "llm-math"], llm=llm)

# 3. 创建 Agent
agent = initialize_agent(
    tools, 
    llm, 
    agent_type=AgentType.ZERO_SHOT_REACT_DESCRIPTION,
    verbose=True
)

# 4. 执行
result = agent.run("2024年诺贝尔物理学家是谁？")
```

**Zero-shot ReAct 模式（最常用）：**
```
Question → Think（推理） → Action（执行Tool） → Observation（观察结果） → Final Answer
```
Agent 循环执行：先推理（LLM 思考下一步行动），再调用 Tool，获取结果后继续推理，直到给出最终答案。

**与 Spring AI / Dify 的区别：**
- LangChain：Python 生态，最灵活，社区最活跃
- Spring AI Java 版：Java 开发者友好，功能尚在追赶 LangChain
- Dify：可视化、低代码，适合产品化 AI 应用

## 题目9：二叉树的最近公共祖先（Lowest Common Ancestor）

### 答案

**题目：** 给定一棵二叉树和两个节点 p 和 q，找到它们的最近公共祖先。

**解法一：后序遍历 + 递归（最优解，O(n) 时间）**

**核心思路：**
- 最近公共祖先（LCA）的定义：`root` 本身可以是 LCA
- 如果 p 或 q 恰好是 root，则 root 就是 LCA
- 对左右子树递归查找：
  - 左右子树都找到了 p 和 q → 当前 root 是 LCA
  - 只有一边找到 → LCA 在那一边
  - 两边都没找到 → 当前子树没有 p 或 q

```python
class Solution:
    def lowestCommonAncestor(self, root: TreeNode, p: TreeNode, q: TreeNode) -> TreeNode:
        # 递归终止条件
        if not root or root == p or root == q:
            return root
        
        # 在左右子树中分别查找 p 和 q
        left = self.lowestCommonAncestor(root.left, p, q)
        right = self.lowestCommonAncestor(root.right, p, q)
        
        # p 和 q 都不在左子树中，结果在右子树
        if not left:
            return right
        # p 和 q 都不在右子树中，结果在左子树
        if not right:
            return left
        
        # 左右子树各找到一个，LCA 就是当前 root
        return root
```

**复杂度分析：**
- 时间：O(n)，每个节点最多访问一次
- 空间：O(n)（递归栈深度，最坏退化成链表）

**解法二：父指针 + 哈希集合**

如果每个节点有 `parent` 指针：
1. 从 p 出发，将所有祖先节点加入集合
2. 从 q 出发向上遍历，第一个出现在集合中的就是 LCA

```python
def lca_with_parent(p, q):
    ancestors = set()
    while p:
        ancestors.add(p)
        p = p.parent
    
    while q:
        if q in ancestors:
            return q
        q = q.parent
```

**变形题：二叉搜索树（BST）的 LCA**
BST 有序性可以利用，直接比较 root 与 p、q 的大小关系：
```python
if root.val > p.val and root.val > q.val:
    return lca(root.left, p, q)
elif root.val < p.val and root.val < q.val:
    return lca(root.right, p, q)
else:
    return root
```

## 题目10：股票最大利润问题（无限次交易 vs 最多 k 笔交易）

### 答案

**问题描述：** 给定一只股票的每日价格prices，可以无限次买卖（买入前必须先卖出才能再买），求最大利润。

**解法一：贪心（最优，O(n)）**

**核心思想：** 只要相邻两天价格上涨，就前一天买、第二天卖。

```python
def maxProfit(prices):
    profit = 0
    for i in range(1, len(prices)):
        if prices[i] > prices[i-1]:
            profit += prices[i] - prices[i-1]
    return profit
```

**为什么贪心是正确的？**
对于任意上涨段 `[buy, ..., sell]`，将其拆分为每天的"买-卖"等价于一次性买卖：
- 整体利润 = Σ(sell_i - buy_i)
- 分解利润 = Σ(prices[i] - prices[i-1])（仅统计上涨日）
- 两者相等，且分解操作能捕捉所有利润，不遗漏。

**解法二：动态规划（通用框架，适合 k 笔交易限制）**

当限制最多交易 k 笔时（k=1 即 Best Time to Buy and Sell Stock II），使用 DP：

```python
def maxProfit_k_transactions(k, prices):
    if not prices or k == 0:
        return 0
    
    n = len(prices)
    # 最多交易 floor(k/2) 笔实际买卖（一次买卖=2笔交易）
    k = min(k, n // 2)
    
    # DP[i][j] = 第 i 天最多 j 笔交易的最大利润
    # 空间优化：只保留前一天的数据
    buy = [-prices[0]] + [0] * k  # 买入了j股时的最大收益（负利润）
    sell = [0] * (k + 1)          # 卖出了j股时的最大收益
    
    for i in range(1, n):
        # 买入状态更新（从 sell[j] 转移而来）
        buy[0] = max(buy[0], -prices[i])  # 第一次买入
        for j in range(1, k + 1):
            buy[j] = max(buy[j], sell[j] - prices[i])
        # 卖出状态更新
        for j in range(1, k + 1):
            sell[j] = max(sell[j], buy[j-1] + prices[i])
    
    return sell[k]
```

**三种情况对比：**

| 场景 | 解法 | 复杂度 |
|------|------|--------|
| 无限次交易 | 贪心 | O(n) 时间，O(1) 空间 |
| 最多 k=2 笔交易 | DP（空间优化版） | O(n) 时间，O(k) 空间 |
| 冷却期冷冻（变种） | DP + 状态机 | O(n) 时间，O(1) 空间 |

**变种：含冷冻期**（卖出的下一天不能买）
状态机：`hold[i]` = 第 i 天持仓的最大利润，`sell[i]` = 第 i 天已卖出的最大利润，`cooldown[i]` = 冷冻期
```python
def maxProfit_with_cooldown(prices):
    sell = 0
    prev_sell = 0
    buy = -prices[0]
    for p in prices[1:]:
        prev_buy = buy
        buy = max(buy, prev_sell - p)
        prev_sell = sell
        sell = max(sell, prev_buy + p)
    return sell
```
