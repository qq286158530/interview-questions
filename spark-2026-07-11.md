## 题目一：Spark 任务调度流程是怎么样的？

### 参考答案

Spark 任务调度分为几个层次，从高层到底层依次是：

1. **Application（应用层）**：用户提交一个 Spark 应用程序，Driver 进程启动后执行 `sparkContext` 初始化，构建 DAG（有向无环图）。

2. **DAGScheduler（逻辑调度层）**：将 DAG 拆分成多个 Stage。每个 Stage 包含一批可以在一台节点上完成的任务（基于 RDD 依赖是否为 Shuffle 划分边界）。Shuffle 依赖会触发 Stage 切分。

3. **TaskScheduler（物理调度层）**：将每个 Stage 转换为一批 Task（ShuffleMapTask 或 ResultTask），以 TaskSet 的形式提交到集群。

4. **Executor 接收任务**：Task 被发送到具体的 Executor 进程，Executor 调用 `taskRunner.run()` 执行任务。

5. **结果回收**：ResultTask 执行完毕后结果返回 Driver，DAGScheduler 收到结果后继续调度下一个 Stage。

**核心流程图**：

```
Application
    ↓
sparkContext (创建 Driver)
    ↓
DAGScheduler (切分 Stage，构建 DAG)
    ↓
TaskScheduler (提交 TaskSet 到集群)
    ↓
Executor (接收并执行 Task)
    ↓
结果返回 Driver
```

## 题目二：RDD 的五大特性是什么？

### 参考答案

RDD（Resilient Distributed Dataset）是 Spark 最核心的数据抽象，具备以下五大特性：

1. **` partitions` — 一组分区列表**
   RDD 的数据被划分为多个分区，每个分区是数据集的一个逻辑片段。分区的数量决定了对齐 Task 的并行度。

2. **`compute(transformation)` — 一个计算函数**
   每个 RDD 都知道如何通过父 RDD 计算得到自己，即 `compute` 方法。它接收父 RDD 的分区，返回对应的新分区。

3. **`dependencies` — RDD 之间的依赖关系**
   RDD 通过 `dependencies` 声明与父 RDD 的依赖关系，分为窄依赖（窄依赖）和宽依赖（Shuffle 依赖）。窄依赖允许 pipeline 执行，宽依赖需要等父分区全部完成。

4. **`partitioner` — 可选的分区器**
   对于 Key-Value 类型的 RDD，可以指定分区器（HashPartitioner 或 RangePartitioner），这决定了 Shuffle 时数据如何划分到不同的 Reduce 任务。

5. **`preferredLocations` — 每个分区的最优位置**
   RDD 可以记录每个分区最好优先放到哪些节点上，实现数据本地性优化，减少网络传输。

## 题目三：Spark Shuffle 机制是什么？有几种 Shuffle Manager？

### 参考答案

**Shuffle（洗牌）** 是 Spark 中将数据重新分布的过程，发生在宽依赖（ShuffleDependency）处。当一个 RDD 需要按 Key 进行聚合、排序或重新分区时，所有相同 Key 的数据必须被拉到同一个节点上，这个数据重分发过程即为 Shuffle。

### Spark Shuffle 发展历程与 Shuffle Manager

| Shuffle Manager | 版本 | 特点 |
|---|---|---|
| **Hash Shuffle Manager** | 1.x | 每个上游 Task 为每个下游 Task 生成一个文件，文件数 = 上游 Task 数 × 下游 Task 数，文件过多，GC 压力大 |
| **Sort Shuffle Manager** | 1.5+（ Tungsten-sort） | 引入排序和文件合并机制，每个上游 Task 只生成少量文件（2个：数据文件+索引文件），大幅减少文件数 |
| **Unsafe Shuffle（钨丝计划）** | 1.5+ | 基于 Tungsten 直接操作二进制内存，减少序列化开销，要求数据能完全放入内存且无自定义分区器 |
| **Sort Shuffle Manager（统一）** | 2.0+ | 统一使用 Sort Shuffle，Unsafe Shuffle 作为特殊场景的优化选项 |

### Sort Shuffle 工作流程

1. **Map 端**：先将数据写入内存缓冲区（默认 32KB），缓冲区满时溢写到磁盘（spill）；
2. **排序**：根据下游分区编号对每个分区的数据在内存/磁盘中排序；
3. **归并**：所有溢写文件归并成一个排序文件，同时生成索引文件（记录每个分区的起始位置）；
4. **Reduce 端**：根据索引文件去拉取自己对应分区的数据。

## 题目四：Spark 内存管理机制是怎样的？堆内与堆外内存如何划分？

### 参考答案

Spark 有一套统一的内存管理系统，基于 JVM 堆内内存和堆外内存（off-heap）进行管理。

### 堆内内存（Heap Memory）

Executor 的 JVM 堆内存，默认占 `(Java MaxMemory - 300MB) × 0.6`，划分为以下区域：

| 区域 | 占比 | 用途 |
|---|---|---|
| **Execution（执行内存）** | 约 30% | Shuffle 排序、Hash 聚合、JOIN 等计算过程的临时内存 |
| **Storage（存储内存）** | 约 30% | RDD Cache、Broadcast 变量、广播表等持久化数据 |
| **User Memory（约 40%）** | — | 用户代码（如 UDF 创建的对象）、Spark 内部元数据 |

Execution 和 Storage 之间可以互相借用空间：当 Execution 内存不足时可以借用 Storage，Storage 空间不足时也可以借用 Execution（但 Storage 只能驱逐自身缓存的块，不能驱逐 Execution 的内存）。

### 堆外内存（Off-Heap Memory）

通过 `spark.memory.offHeap.enabled=true` 开启，使用 `sun.misc.Unsafe` 直接操作堆外内存，主要用于：
- 减少 GC 开销
- 适合需要大量内存且避免序列化开销的场景（如 Tungsten Shuffle）

### 内存配置参数

```scala
//  Execution + Storage 总内存占 JVM 堆的比例（默认 0.6）
spark.memory.fraction = 0.6

// Storage 占比（默认 0.5，即 Execution 和 Storage 各占一半）
spark.memory.storageFraction = 0.5

// 堆外内存大小（默认关闭）
spark.memory.offHeap.enabled = true
spark.memory.offHeap.size = 2GB
```

## 题目五：广播变量（Broadcast Variable）的作用和原理是什么？

### 参考答案

### 什么是广播变量

广播变量是 Spark 用来高效分发只读数据给所有 Task 的机制。当多个 Task 需要使用同一份小数据集时（如查询维表、配置参数、共享字典等），如果不使用广播，每个 Task 都会在各自节点上独立拉取一份数据，造成大量冗余网络传输。使用广播后，每个 Executor 只需要在本地保存一份数据，所有在该 Executor 上运行的 Task 共享这份数据。

### 广播变量的使用

```scala
// 定义广播变量
val broadcastVar = sc.broadcast(Array(1, 2, 3, 4, 5))

// 在 Task 中使用（自动在 Executor 端本地读取，无需网络传输）
val result = broadcastVar.value.map(_ * 2)
```

### 广播原理（Spark 2.x+ 基于 TorrentBroadcast）

Spark 2.x 起默认使用 TorrentBroadcast（类似于 BitTorrent 的 P2P 机制），工作原理如下：

1. **Driver 端**：将数据切分为多个块（chunk，默认 4MB），存储到 Executor 的 BlockManager 中。

2. **块分发**：当某个 Executor 需要某个块时：
   - 首先尝试从本地 BlockManager 读取（已缓存则直接返回）；
   - 若未缓存，则从其他持有该块的 Executor 处并行下载；
   - 下载后同时将块存入本地 BlockManager，供其他 Task 使用。

3. **去中心化**：避免了 Driver 成为单点瓶颈，多个 Executor 可以相互分担带宽压力。

### 注意事项

- 广播变量是**只读**的，更新不会传播；
- 数据量较小时（< 20KB）广播收益不明显，Spark 会自动优化；
- 广播的数据会存储在 Storage 内存中，需确保 `spark.memory.fraction` 足够；
- 使用完应调用 `broadcastVar.unpersist()` 释放缓存。
