## 题目一：HBase 的架构是怎样的？请说明 Master、RegionServer、ZooKeeper 的作用

### 参考答案

HBase 是一个分布式的、面向列族的 NoSQL 数据库，构建在 Hadoop HDFS 之上。其核心架构如下：

**1. Master（主节点）**
- 负责监控所有 RegionServer 实例
- 处理表结构变更（创建、删除、修改列族等）
- 分配 regions 到各个 RegionServer
- 实现集群的负载均衡
- 负责垃圾文件回收和 schema 变更

**2. RegionServer（Region 服务器）**
- 负责服务和管理多个 regions
- 直接处理客户端的读写请求
- 数据的 flush 和 compaction 操作
- 一个 RegionServer 可以服务 1000 个左右的 regions

**3. ZooKeeper（动物园管理员）**
- 维护 HBase 集群的配置信息
- 提供分布式同步服务
- 负责 Master 和 RegionServer 之间的通信
- 维护 RegionServer 的在线状态列表
- 通过心跳检测实现故障检测与恢复

**数据存储流程：**
1. 客户端写入 → WAL（Write Ahead Log）→ MemStore
2. MemStore 刷写 → HFile（持久化到 HDFS）
3. 读取时先查 MemStore，未命中则查 BlockCache → HFile

## 题目二：HBase 中 RowKey 设计的原则有哪些？如何避免热点问题？

### 参考答案

**RowKey 设计原则：**

1. **长度原则**
   - RowKey 是一个字节数组，建议控制在 16~100 字节之间
   - 越短越好，减少存储开销

2. **散列原则**
   - RowKey 应该均匀分布，避免数据倾斜
   - 可以使用 MD5、SHA1 等哈希算法对长字符串进行哈希

3. **唯一性原则**
   - RowKey 必须唯一标识一行数据

4. **业务相关性**
   - 将查询最频繁的字段放在 RowKey 前缀

**避免热点（Hotspotting）的策略：**

1. **加盐（Salting）**
   - 在 RowKey 前面添加随机前缀，使数据分散到不同 region
   - 缺点：无法支持范围查询

2. **哈希（Hashing）**
   - 对 RowKey 进行哈希运算，如 `MD5(rowKey).substring(0, 10)`
   - 保证数据均匀分布

3. **反转（Reversing）**
   - 反转固定长度的字段，如手机号反转
   - 适用于高频查询字段尾部变化小的场景

4. **时间戳反转**
   - 查询最新数据时使用 `Long.MAX_VALUE - timestamp` 作为 RowKey

## 题目三：HBase 中 MemStore 和 HFile 是什么？它们之间的关系是什么？

### 参考答案

**MemStore（内存存储）**
- 位于 RegionServer 的内存中，是一个写缓冲区
- 当数据写入 HBase 时，首先写入 WAL，然后写入 MemStore
- 按 RowKey、Column Family、Column Qualifier、Timestamp 有序排列
- 当 MemStore 达到阈值（默认 128MB）时，触发 flush 操作

**HFile（HBase File Format）**
- HBase 存储在 HDFS 上的文件格式
- 基于 SSTable（BigTable 的存储格式）设计
- 存储实际的 KeyValue 数据
- 包含索引文件、Bloom Filter 等元数据

**关系流程：**

```
写入流程：
客户端 → WAL → MemStore → (flush) → HFile

读取流程：
客户端 → MemStore → BlockCache → HFile → 返回
```

**刷写（Flush）触发条件：**
1. MemStore 大小超过 `hbase.hregion.memstore.flush.size`（默认 128MB）
2. RegionServer 的 MemStore 总大小超过堆内存的 40%
3. WAL 日志数量超过 `hbase.regionserver.max.logs`

**HFile 的主要结构：**
- Data Block：存储实际的 KeyValue 数据
- Meta Block：存储 Bloom Filter 信息
- File Info：文件元信息
- Index Block：数据索引

## 题目四：什么是 HBase 的 Compaction（压缩）？Minor 和 Major Compaction 的区别是什么？

### 参考答案

**Compaction（压缩）定义**
Compaction 是 HBase 中将多个小的 HFile 合并成更少更大的 HFile 的过程，以减少文件数量、提升读取性能并清理已删除的数据。

**Minor Compaction（小型压缩）**
- 合并相邻的小 HFile（通常 3~10 个）为单个 HFile
- 不会删除已标记为删除的数据
- 由 RegionServer 自动触发，代价较低
- 触发条件：`hbase.hstore.compaction.min`（默认 3 个文件）

**Major Compaction（大型压缩）**
- 将一个 Column Family 的所有 HFile 合并为一个
- 会清理所有已删除的 cells（TTL 过期、版本过期的数据）
- 删除标记被丢弃，磁盘空间释放
- 触发方式：手动触发或自动定期执行（默认 7 天）
- **注意**：Major Compaction 会产生大量 IO，建议在业务低峰期执行

**两者的核心区别：**

| 特性 | Minor Compaction | Major Compaction |
|------|------------------|------------------|
| 合并文件数 | 少量（3~10个） | 所有文件 |
| 删除处理 | 保留删除标记 | 清理删除数据 |
| 触发方式 | 自动 | 自动+手动 |
| IO 消耗 | 较小 | 较大 |
| 时机 | 频繁 | 定期（周级）|

## 题目五：HBase 的协处理器（Coprocessor）是什么？有哪几种类型？各自的使用场景是什么？

### 参考答案

**协处理器（Coprocessor）概念**
协处理器是运行在 RegionServer 上的自定义代码，允许用户在数据写入和读取时执行自定义逻辑，无需修改 HBase 核心代码。

**两种协处理器类型：**

**1. Observer Coprocessor（观察者协处理器）**
类似于数据库触发器，在特定事件发生时执行回调。

- `PrePut`：写入数据之前执行
- `PostPut`：写入数据之后执行
- `PreGet`：读取数据之前执行
- `PostGet`：读取数据之后执行
- `PreDelete`：删除数据之前执行
- `PostDelete`：删除数据之后执行

**使用场景：**
- 实现二级索引（写入主表时同步更新索引表）
- 实时数据验证与过滤
- 审计日志记录
- 自动计算汇总数据

**2. Endpoint Coprocessor（端点协处理器）**
类似于数据库存储过程，允许在 RegionServer 上执行自定义的分布式计算。

**使用场景：**
- 聚合计算（SUM、AVG、COUNT）
- 自定义过滤器
- 跨行跨表的复杂业务逻辑

**加载方式：**
1. **动态加载**：通过 HBase Shell 动态加载到表或列族
2. **静态加载**：配置 `hbase-site.xml`，随 HBase 启动加载

**使用限制：**
- 协处理器代码错误可能导致 RegionServer 崩溃
- 无法进行远程调试
- 需要谨慎处理异常和资源占用
