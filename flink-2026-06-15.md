## 题目一：Flink 的 JobManager 和 TaskManager 架构是怎样的？它们各自负责什么职责？

### 参考答案

**1. 核心组件架构**

Flink 运行时采用 **Master-Slave 架构**：

| 组件 | 角色 | 职责 |
|------|------|------|
| **JobManager** | Master（主节点） | 协调整个作业的执行：调度任务、管理 Checkpoint、接收 TaskManager 心跳 |
| **TaskManager** | Slave（工作节点） | 实际执行算子，存储状态，从 JobManager 接收指令并汇报进度 |

**2. JobManager 内部结构**

```
JobManager
├── Dispatcher          # 接收 REST 请求，启动 JobMaster
├── ResourceManager     # 管理 Slot资源，与 YARN/K8s/Mesos 交互
├── JobMaster           # 负责单个 Job 的调度和执行协调
└── MetricRegistry     # 指标收集
```

- **Dispatcher**：提供 Web UI、REST API，管理多个 JobMaster（每个 Job 一个）
- **ResourceManager**：类似 YARN 的 RM，管理 TaskManager 的 Slot 资源
- **JobMaster**：核心组件，与 TM 通信、发送算子、执行 Checkpoint

**3. TaskManager 架构**

```
TaskManager
├── TaskSlot × N       # 资源隔离槽位，每个 Slot 可运行一个或多个子任务
├── Task ← 运行任务     # 实际执行算子逻辑的单元
├── Network Manager    # 负责 Task 间数据交换（Shuffle）
├── Memory Manager     # 管理堆内外内存
└── IOManager         # 磁盘 IO 管理
```

**4. Slot 与 Task 调度**

- **Slot 概念**：TM 的最小资源单元，每个 Slot 有固定内存和一定比例的 CPU
- **Slot 数量** = `TM数量 × 每TM Slot数`，决定并发上限
- **Task 放置策略**：Flink 默认用 **Eager 调度**（一次性分配所有 Slot），复杂 DAG 场景下可能用**流式调度**

```java
// 配置每 TM 的 Slot 数（通常 = CPU 核数）
taskmanager.numberOfTaskSlots: 4

// 简单 wordcount 示例：Source 并发 2，Map 并发 2，Sink 并发 1
env.socketTextStream(...).setParallelism(2)
  .flatMap(...).setParallelism(2)
  .print().setParallelism(1);
```

**5. 常见面试追问**

> **Q：JobManager 挂了怎么办？**
> - Standalone 模式下：JobManager 单点，是严重故障
> - 高可用模式（HA）：通过 Zookeeper 或 K8s 做 Leader 选举，备用 JM 接管

> **Q：Slot 和 Task 并发数的关系？**
> - Slot 是资源容器的概念，Task 是执行单元
> - 算子并行度 ≤ 总 Slot 数才能运行
> - `parallelism = max(所有算子并行度)` 是最低 Slot 要求

## 题目二：Flink Table API / SQL 和 DataStream API 有什么区别？如何选择？

### 参考答案

**1. 核心区别**

| 维度 | Table API / SQL | DataStream API |
|------|-----------------|----------------|
| **编程风格** | 声明式（类 SQL） | 过程式（函数式） |
| **类型系统** | 强类型 + 弱类型混合 | 纯强类型（TypeInformation） |
| **表达能力** | 受限于 SQL 语义 | 任意逻辑均可表达 |
| **优化空间** | 编译期优化（Catalyst）、运行时优化 | 依赖开发者经验 |
| **调试难度** | 黑盒程度高 | 白盒，可单步调试 |
| **生态兼容性** | 可直接对接 Hive、HBase、Iceberg | 需自定义 Connector |

**2. Table API 示例**

```java
Table result = tableEnv.from("Orders")
    .groupBy($("city"), $("hour"))
    .select(
        $("city"),
        $("hour"),
        $("amount").sum().as("total_amount"),
        $("id").count().as("order_count")
    );
```

**3. DataStream API 示例**

```java
DataStream<Order> orders = env.addSource(new KafkaSource<>("orders"));
orders
    .keyBy(Order::getCity)
    .window(TumblingEventTimeWindows.of(Time.hours(1)))
    .aggregate(new OrderAggregate())
    .addSink(newJdbcSink());
```

**4. 动态表（Dynamic Tables）— 核心概念**

Flink SQL 的底层是**动态表**模型：

```
        ┌─────────────────────────────────────┐
        │         Dynamic Table (Changelog)   │
        └─────────────────────────────────────┘
                   ▲ 持续追加 / 更新
        ┌─────────┴──────────┐
        │    Continuous     │
        │    Query          │
        └─────────┬──────────┘
                  ▼
        ┌─────────────────────────────────────┐
        │  DataStream / 外部 Sink (Kafka/DB)  │
        └─────────────────────────────────────┘
```

- **追加（Append）查询**：只输出新增行
- **更新（Retract）查询**：使用 Retract 机制，老结果发撤回消息，新结果发更新消息

**5. 选择原则**

```
业务场景：
  ├── 简单 ETL / 聚合 / 报表  → Table API / SQL（开发效率高）
  ├── 复杂事件处理 / 自定义状态 → DataStream API
  └── 已有 Java/Scala 代码需集成 → DataStream API

数据规模：
  ├── 大规模批+流混合  → Table API + Blink 优化器
  └── 纯流处理，状态复杂 → DataStream API + RocksDB
```

**6. 混合使用（推荐）**

```java
// Table API 写业务逻辑，DataStream 写复杂自定义算子
Table table = tableEnv.from("Orders");
// ... Table 操作 ...

// 转换为 DataStream 做高级处理
DataStream<Row> ds = tableEnv.toDataStream(table, Row.class);
ds.filter(row -> complexFilter(row)).print();

// 再转回 Table
Table finalTable = tableEnv.fromDataStream(ds);
tableEnv.executeSql("INSERT INTO sink SELECT * FROM finalTable");
```

## 题目三：Flink 中的时间语义（Event Time、Processing Time、Ingestion Time）有什么区别？如何处理时间歪斜？

### 参考答案

**1. 三种时间语义对比**

| 时间语义 | 定义 | 优点 | 缺点 |
|---------|------|------|------|
| **Event Time** | 事件产生时间（数据自带时间戳） | 结果精确，可重放 | 依赖 Watermark，延迟高 |
| **Processing Time** | 处理机处理时的系统时间 | 无延迟，零开销 | 结果不精确，不可重放 |
| **Ingestion Time** | 数据进入 Flink Source 的时间 | 介于两者之间 | 仍依赖单点时钟 |

```java
// 设置时间语义
env.setStreamTimeCharacteristic(TimeCharacteristic.EventTime);

// 事件时间窗口（必须指定 Watermark）
env.getStreamTimeCharacteristic(TimeCharacteristic.EventTime);
DataStream<Tuple2<String, Long>> watermarked = stream
    .assignTimestampsAndWatermarks(
        WatermarkStrategy
            .<Tuple2<String, Long>>forBoundedOutOfOrderness(Duration.ofSeconds(10))
            .withTimestampAssigner((e, ts) -> e.f1) // 从数据中提取事件时间
    );
```

**2. 时间歪斜（Time Skew）产生原因**

```
正常顺序：[A@10:00][B@10:01][C@10:02] → Watermark@10:02 触发窗口
乱序情况：[A@10:00][C@10:02][B@10:01] → Watermark@10:01 触发，B 被漏掉
```

常见原因：
- 网络延迟不确定
- 数据来自不同数据源（多终端、多地域）
- 分布式日志采集的天然乱序

**3. 应对策略**

**策略一：合理设置 Watermark 延迟**
```java
// 有界乱序容忍度（最大延迟）
WatermarkStrategy.forBoundedOutOfOrderness(Duration.ofSeconds(15))
```

**策略二：使用空闲检测（Idleness）**
```java
WatermarkStrategy
    .forBoundedOutOfOrderness(Duration.ofSeconds(20))
    .withIdleness(Duration.ofMinutes(1)) // 数据源空闲1分钟则标记为 idle
```
- 多数据源场景：某些源长期无数据，Flink 会等 Watermark，造成结果不输出
- 解决：标记空闲数据源，让其他源的数据先触发窗口

**策略三：Allowed Lateness 处理迟到数据**
```java
.window(TumblingEventTimeWindows.of(Time.minutes(5)))
.allowedLateness(Time.minutes(2))  // 窗口关闭后2分钟内仍可收 late event
.sideOutputLateData(lateTag)       // 超出2分钟的数据进入侧输出
```

**策略四：重新对齐（Re-align）**
```java
// 来自多个数据源的数据，统一用同一个 Watermark 策略
DataStream<Integer> source1 = env.addSource(new KafkaSource(...)).assignTimestampsAndWatermarks(...);
DataStream<Integer> source2 = env.addSource(new KafkaSource(...)).assignTimestampsAndWatermarks(...);
```

**4. 最佳实践建议**

- **事件时间优先**：生产环境务必用 Event Time，Processing Time 只用于快速验证
- **Watermark 设置**：用历史数据 P99 延迟作为参考（不是平均值）
- **监控**：关注 `currentInput1Watermark` 和 `currentOutputWatermark` 的差值，过大说明延迟配置不合理
- **日志埋点**：记录迟到数据量和侧输出数据量，动态调整 allowedLateness

- https://cloud.tencent.com/developer/article/1785231

## 题目四：Flink 的反压（Backpressure）是如何产生和处理的？生产环境如何定位和解决反压？

### 参考答案

**1. 反压产生原理**

Flink 的反压是**逐级反馈**的分布式背压机制：

```
Kafka(Source) → Map → Filter → Sink(Kafka)
   ↑            ↑      ↑       ↓
   └────────────┴──────┴───────┘
   反压信号沿数据流反方向传播
```

- **正常情况**：下游处理速度 ≥ 上游发送速度，数据流畅通
- **反压情况**：下游处理慢，缓冲区满 → 向上游发送 **阻塞信号** → 上游停止发送数据

**2. Checkpoint Barrier 对反压的影响**

```
反压时：
Source → [===Barrier===] → (下游处理慢，Barrier 排队等待)
                    ↑
            Barrier 无法及时传递
            → Checkpoint 延长
            → Checkpoint 超时 → Job 重启
```

这就是为什么反压严重时 Job 会周期性重启。

**3. 生产环境定位反压步骤**

**Step 1：通过 Web UI 查看 Task 级别的背压状态**

Flink Web UI 每个算子节点显示：
- `OK`（绿色）：正常
- `LOW / HIGH`（黄色/红色）：存在反压

**Step 2：定位最慢的算子（瓶颈）**

```
反压链：Source(OK) → Map(OK) → Filter(HIGH) → Sink(HIGH)
                              ↑
                        瓶颈在这里！
```

**Step 3：使用 Flink Metrics 精确定位**

| Metric | 含义 |
|--------|------|
| `numRecordsOutPerSecond` | 输出速率低说明下游是瓶颈 |
| `numRecordsInPerSecond` | 输入速率低说明上游是瓶颈 |
| `currentInputGateWatermarks` | 输入端水位线延迟 |
| `backPressureLevel` | 反压等级（OK/LOW/HIGH） |

**4. 常见原因及解决方案**

| 原因 | 表现 | 解决方案 |
|------|------|---------|
| **下游 Sink 写入慢**（DB/HTTP） | Sink 节点 HIGH | 批量写入、增加并发、异步 IO |
| **数据倾斜（KeyBy 不均）** | 个别 SubTask HIGH | 重新设计 Key、加盐、反序列化检查 |
| **状态过大（RocksDB 吞吐不足）** | 所有节点 HIGH | 减小状态体积、增加 TM 内存、使用 SSD |
| **业务逻辑复杂**（大对象序列化） | 所有节点同步 HIGH | 简化对象、使用 POJO 而非 Tuple |
| **网络 Shuffle 瓶颈** | 个别节点 HIGH | 增加并发度、检查网络拓扑 |

**5. 数据倾斜定位代码示例**

```java
// 在 map 算子中埋点，输出每个 key 的处理时间
.map(new ProcessFunction<Tuple2<String, Long>, String>() {
    @Override
    public void processElement(Tuple2<String, Long> value, Context ctx, Collector<String> out) {
        long start = System.currentTimeMillis();
        // ... 业务逻辑 ...
        long cost = System.currentTimeMillis() - start;
        if (cost > 100) {
            System.err.println("SLOW_KEY: " + value.f0 + " cost=" + cost + "ms");
        }
    }
});
```

**6. Flink 1.13+ 自动化反压调优**

```java
// 开启自动反压调优（Flink 1.13+）
env.getConfig().setAutoWatermarkInterval(200);

// 开启 TaskManager 的反压监控
taskmanager.metrics.latency.interval: 1000
```

## 题目五：Flink 与 Kafka 如何实现精确一次（Exactly-Once）集成？Kafka Connector 的事务机制是怎样的？

### 参考答案

**1. Kafka Connector 精确一次的必要条件**

Flink + Kafka 实现 Exactly-Once 需要同时满足：

```
✅ Kafka Source：开启 Checkpoint，使用 Committer 提交偏移量
✅ Kafka Sink：开启事务 + 两阶段提交
✅ Kafka Broker：transaction.timeout.ms 足够大（> Checkpoint 间隔）
✅ Consumer Group：隔离性配置合理
```

**2. Kafka Source 精确一次原理**

```java
KafkaSource<String> source = KafkaSource.<String>builder()
    .setBootstrapServers("localhost:9092")
    .setTopics("input-topic")
    .setGroupId("my-consumer-group")
    .setStartingOffsets(OffsetsInitializer.committedOffsets())
    .setValueOnlyDeserializationSchema(new SimpleStringSchema())
    .build();

DataStream<String> stream = env.fromSource(
    source,
    WatermarkStrategy.forBoundedOutOfOrderness(Duration.ofSeconds(5)),
    "Kafka Source"
);

// 必须开启 Checkpoint！
env.enableCheckpointing(60_000);
```

- **偏移量提交**：偏移量作为 Checkpoint 的一部分存储，Checkpoint 完成才认为消费完成
- **故障恢复**：从最近一次成功的 Checkpoint 恢复，重新消费未提交偏移量的消息
- **幂等消费**：Kafka 的 Offsets + 分区唯一性保证消息不被重复处理

**3. Kafka Sink 两阶段提交（重点）**

```java
KafkaSink<String> sink = KafkaSink.<String>builder()
    .setBootstrapServers("localhost:9092")
    .setRecordSerializer(new KafkaRecordSerializationSchema.Builder()
        .setTopic("output-topic")
        .setValueSerializationSchema(new SimpleStringSchema())
        .build())
    .setDeliveryGuarantee(DeliveryGuarantees.EXACTLY_ONCE)
    .setTransactionalIdPrefix("flink-")  // 事务ID前缀
    .setProperty("transaction.timeout.ms", "900000") // 15分钟
    .build();
```

**两阶段提交流程：**

```
┌──────────────────────────────────────────────────────────┐
│ Phase 1: preCommit                                      │
│   • Flink Checkpoint Barrier 到达 Sink                 │
│   • 当前批次的数据写入 Kafka Producer 缓冲区            │
│   • 等待同批次所有分区数据准备完毕                       │
│   • flush() 刷盘（未正式提交）                          │
└──────────────────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────┐
│ Phase 2: commit                                          │
│   • JobManager 确认 Checkpoint 完成                     │
│   • 触发 Kafka Producer.commitTransaction()             │
│   • 数据从缓冲区写入 Kafka Leader Partition             │
└──────────────────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────┐
│ Failure Recovery                                         │
│   • 若 preCommit 后 commit 前崩溃                       │
│   • 新 Producer 启动后用相同 transactionalId             │
│   • Kafka 自动回滚未 commit 的事务                      │
│   • Flink 从 Checkpoint 重放数据                        │
└──────────────────────────────────────────────────────────┘
```

**4. 事务超时配置（关键！）**

```properties
# Kafka Broker 端配置（server.properties）
transaction.timeout.ms=900000  # 15分钟，Flink Checkpoint 间隔必须小于此值

# 若 Checkpoint 间隔 > transaction.timeout：
# → 事务超时，Kafka 自动 abort
# → 数据丢失，Exactly-Once 退化为 At-Least-Once
```

**5. Exactly-Once vs At-Least-Once 配置对比**

| 模式 | 配置 | 特点 |
|------|------|------|
| **Exactly-Once** | `DeliveryGuarantee.EXACTLY_ONCE` + Checkpoint | 精确一次，但有延迟（2PC 开销） |
| **At-Least-Once** | `DeliveryGuarantee.AT_LEAST_ONCE` + Checkpoint | 可能有重复，不丢数据 |
| **None** | 关闭 Checkpoint | 最快，但可能丢数据/重复数据 |

**6. 多 Sink 场景的事务协调**

当作业有多个 Sink 时（Kafka + DB 双写）：

```
JobManager（协调器）
    ├── Kafka Sink (preCommit → commit)
    └── Jdbc Sink  (preCommit → commit)
    
全部成功后：
    → Checkpoint 完成
    → 全部 commit
    
任一失败：
    → 全部 rollback
    → 从 Checkpoint 重放
```
