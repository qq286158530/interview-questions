## 题目一：Spark 与 Hadoop MapReduce 的核心区别是什么？为什么 Spark 通常比 MapReduce 快很多？

### 参考答案

**架构层面的核心差异：**

| 对比项 | Hadoop MapReduce | Apache Spark |
|--------|------------------|--------------|
| **计算模型** | Map + Reduce 两阶段，作业结束后结束 | 基于RDD的DAG有向无环图，支持多阶段迭代 |
| **数据存储** | 中间结果写入 HDFS 磁盘 | 中间结果优先存放在内存中 |
| **数据处理** | 基于磁盘，延迟高 | 基于内存，延迟低 |
| **Task启动** | 以进程方式维护，启动慢 | 以线程方式维护，启动快 |
| **表达能力** | API较底层(map/reduce)，算法适应性差 | API较高层，方便使用 |

**Spark 更快的关键机制：**

1. **内存计算（In-Memory Computing）**：Spark 将中间数据存储在内存中，避免了 MapReduce 每次 Map 和 Reduce 之间写入磁盘的 I/O 开销。在100 倍以上内存速度比磁盘快的情况下，Spark 可以比 MapReduce 快 10~100 倍。

2. **DAG 执行引擎**：Spark 在执行作业前会构建所有转换的 DAG，通过合并操作（如多个 map 操作）、减少写操作数量来确定最佳执行计划。

3. **惰性求值（Lazy Evaluation）**：转换操作（Transformation）延迟执行，只有遇到行动操作（Action）时才真正触发计算，这使得 Spark 能够对整个 DAG 进行优化。

4. **任务启动开销小**：Spark Task 以线程方式维护，相比 MapReduce 的进程方式，任务启动更快。

**典型回答示例：**

> "Apache Spark 是一个强大的开源分布式处理引擎，以其速度和可扩展性著称。它之所以被广泛用于数据处理，是因为能够比传统的 MapReduce 更快地处理海量数据集，这得益于其内存处理能力。MapReduce 在每个 map 和 reduce 步骤后将中间数据写入磁盘，而 Spark 可以将中间数据存储在内存中，从而显著减少 I/O 开销。同时 Spark 支持批处理、实时流处理（Spark Streaming）、机器学习（MLlib）等多种工作负载，是一个多功能的统一平台。"

## 题目二：什么是 RDD（弹性分布式数据集）？它的核心特性有哪些？RDD 的宽依赖和窄依赖有什么区别？

### 参考答案

**RDD 定义：**

RDD（Resilient Distributed Dataset，弹性分布式数据集）是 Spark 中最基本的数据抽象，代表一个**不可变、可分区、里面的元素可并行计算**的集合。

**RDD 的五大核心特性：**

1. **分区列表（Partitions）**：RDD 的数据被划分为多个分区，分布在集群的不同节点上，实现并行处理。

2. **分区函数（Partitioner）**：决定数据如何分区（如 HashPartitioner、RangePartitioner）。

3. **最佳位置（Preferred Locations）**：优化数据局部性，让计算在数据所在节点上执行。

4. **计算函数（Compute Function）**：对分区执行的具体计算逻辑。

5. **依赖关系（Dependencies）**：记录当前 RDD 与父 RDD 之间的依赖关系，用于容错恢复。

**RDD 的两类算子：**

| 类型 | 特点 | 示例 |
|------|------|------|
| **Transformation（转换算子）** | 惰性执行，只记录血统，不触发计算 | map, filter, flatMap, groupByKey, reduceByKey |
| **Action（行动算子）** | 触发真正的计算，返回结果给 Driver | collect, count, reduce, saveAsTextFile |

**宽依赖 vs 窄依赖：**

```
窄依赖（Narrow Dependency）：
  父 RDD 的一个分区只会被子 RDD 的一个分区依赖（一对一或，多对一）
  例如：map, filter, union, mapPartitions

宽依赖（Wide Dependency / Shuffle Dependency）：
  父 RDD 的一个分区会被子 RDD 的多个分区依赖（一对多）
  例如：groupByKey, reduceByKey, join, sortByKey
```

**为什么设计宽窄依赖？**

- **窄依赖**：多个分区可以**并行计算**；如果某个分区数据丢失，只需重新计算对应分区的数据即可（支持流水线计算）。
- **宽依赖**：由于涉及 Shuffle，必须**等到上一阶段计算完成**才能计算下一阶段。Spark 以宽依赖为边界划分 Stage。

**Stage 划分算法（回溯算法）：**
Spark 从后往前回溯，遇到宽依赖就断开创建一个新的 Stage，遇到窄依赖就把当前 RDD 加入当前 Stage。

**典型回答示例：**

> "RDD 是 Spark 的核心数据抽象，是分布在集群节点上的不可变、分区的数据元素集合。'弹性'意味着如果某个分区丢失，Spark 可以利用 RDD 的 Lineage（记录创建它的转换序列）来重建该分区。RDD 的所有转换都是惰性求值的，只有遇到 Action 时才会真正运行。宽窄依赖的设计使得 Spark 能够以宽依赖为边界划分 Stage，实现高效的流水线计算和容错恢复。"

## 题目三：Spark 的 Shuffle 机制有哪两种实现？它们的工作原理和优缺点是什么？

### 参考答案

Spark Shuffle 分为两种实现：**Hash Shuffle** 和 **Sort Shuffle**。Spark 2.0 之后，Hash Shuffle 已不再使用，默认使用 Sort Shuffle。

**一、Hash Shuffle（已废弃）**

**工作原理：**
- 每个 Mapper Task 为每个 Reducer Task 生成一个单独的磁盘文件（产生 M*R 个文件，M=Mapper数，R=Reducer数）。
- 数据先写入内存缓冲，缓冲满后溢写到磁盘文件。

**问题：**
- 产生大量文件（未经优化时为 M*R 个），伴随大量随机磁盘 I/O 和内存开销。
- 优化后（`spark.shuffle.consolidateFiles=true`）可减少到 E*C/*T*R 个文件。

**二、Sort Shuffle（当前默认）**

**普通运行机制：**
1. 数据先写入内存数据结构（Map 或 Array）
2. 达到临界阈值时溢写到磁盘，同时进行排序
3. 最后合并所有临时文件为一个数据文件 + 一个索引文件
4. 最终生成 2*M 个文件（M 个数据文件 + M 个索引文件）

**bypass 运行机制（触发条件：Reducer 数量 < `spark.shuffle.sort.bypassMergeThreshold`，默认200）：**
- 每个 task 为每个下游 task 创建临时磁盘文件（类似 Hash Shuffle）
- 最后合并成一个磁盘文件 + 索引文件
- 特点：**不进行排序**，性能更好

**Tungsten Sort Shuffle（借助 Tungsten 项目优化）：**
- 对排序算法进行改进，优化了排序速度
- Spark 引擎自动判断是否启用

**两种 Shuffle 对比：**

| 特性 | Hash Shuffle | Sort Shuffle |
|------|-------------|--------------|
| 产生文件数（未优化） | M*R（不可控） | 2*M（可控） |
| 是否排序 | 否 | 是 |
| 磁盘 I/O | 高 | 低 |
| 内存开销 | 高 | 低 |
| 当前状态 | Spark 2.0+ 已废弃 | 默认 |

**典型回答示例：**

> "Spark Shuffle 是连接 Map 阶段和 Reduce 阶段的桥梁，涉及磁盘读写和网络 I/O，是性能瓶颈所在。Hash Shuffle 会产生大量文件（与 Reducer 数量直接相关），严重制约了 Spark 的扩展能力。Sort Shuffle 通过将数据写入一个统一的文件加索引文件，将文件数从 M*R 减少到 2*M 大幅提升了性能。Sort Shuffle 还提供了 bypass 机制，当 Reducer 数量较少时不进行排序直接写文件，兼顾了性能。"

## 题目四：Spark 的内存管理机制是怎样的？如何避免 OOM（内存溢出）问题？

### 参考答案

**Spark 内存结构（堆内内存）：**

Spark 1.6 之后采用统一内存管理，内存分为两大区域：

```
┌────────────────────────────────────────────┐
│           Spark 内存（Heap）                │
├─────────────────────┬──────────────────────┤
│   Execution Memory   │    Storage Memory    │
│   （执行内存）        │    （存储内存）        │
│   - Shuffle         │    -缓存 RDD        │
│   - 排序             │    - 广播变量         │
│   - 聚合             │    - 用户数据         │
├─────────────────────┼──────────────────────┤
│     Reserved Memory  │ User Memory │
│   （预留内存 300MB）  │    （用户内存）        │
└─────────────────────┴──────────────────────┘
```

**核心参数：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `spark.memory.fraction` | 0.6 | Execution + Storage 占堆内存的比例 |
| `spark.memory.storageFraction` | 0.5 | Storage Memory 占上述比例中的比例 |
| `spark.shuffle.file.buffer` | 32KB | Shuffle 写磁盘的缓冲区大小 |
| `spark.reducer.maxSizeInFlight` | 48MB | Reduce 端每次拉取数据的缓冲区大小 |

**动态占用机制：**

- 当 Execution 内存不足时，可以借用空闲的 Storage Memory
- 当 Storage 内存不足时，可以借用 Execution Memory
- 但 Storage Memory 不能从 Execution 借回（当 Execution 需要时会被驱逐）

**避免 OOM 的常见方法：**

1. **降低 reduce 端缓冲区大小**：`spark.reducer.maxSizeInFlight` 从 48MB 降到 12MB，减少单次拉取数据量
2. **增加 reduce 端重试次数**：`spark.shuffle.io.maxRetries` 增加重试次数，处理 GC 导致的拉取失败
3. **使用 Kryo 序列化**：比 Java 序列化节省空间（`spark.serializer=org.apache.spark.serializer.KryoSerializer`）
4. **使用广播变量**：减少网络传输的数据量（小表广播）
5. **增加分区数**：增加并行度，减少每个 task 处理的数据量
6. **使用 `cache`/`persist` 合理选择存储级别**：如 `MEMORY_AND_DISK` 避免全部存内存

**典型回答示例：**

> "Spark 动态地管理内存，将其划分为执行和存储区域。执行内存用于 shuffle、join 和聚合等任务，而存储内存用于缓存 RDD 和广播变量。当内存不足时，Spark 会溢出数据到磁盘，但会尽量将频繁访问的数据保留在内存中以加快访问速度。避免 OOM 的关键在于合理设置 `spark.memory.fraction` 参数、使用合适的序列化方式（推荐 Kryo）、以及对大表 join 使用广播变量来减少网络传输。如果出现 reduce 端 OOM，可以尝试减小 `spark.reducer.maxSizeInFlight` 参数。"

## 题目五：什么是广播变量（Broadcast Variables）？它的工作原理是什么？使用广播变量有哪些注意事项？

### 参考答案

**广播变量的定义：**

广播变量是**只读的、会被分发到集群中每个节点**的变量。在每个 Executor 上只保存一份副本，该 Executor 上的所有 Task共享这份数据，而不是每个 Task 都获取一份副本。

**工作原理：**

1. **Driver 端创建**：在 Driver端通过 `sc.broadcast(数据)` 创建广播变量
2. **序列化传输**：广播变量通过 BitTorrent 或类似机制分发到各节点
3. **本地缓存**：每个 Executor 的 BlockManager 缓存一份广播数据
4. **Task 访问**：Task 在本地直接读取 BlockManager 中的广播数据，无需跨网络

**使用示例（Scala）：**

```scala
// 定义广播变量
val broadcastData: Broadcast[collection.Map[Int, String]] = sc.broadcast(fruitMap)

// 在算子中使用
val result = dataIds.map(id => broadcastData.value(id))
```

**广播变量的作用（解决什么问题）：**

当多个 Task 需要共用一份较大的数据时（如维度表、小表 join），如果不使用广播变量：
- 每个 Task 都会通过网络传输这份数据 → **网络开销巨大**
- 每个 Task 的内存中都有一份副本 → **内存浪费严重**

使用广播变量后：
- 每个 Executor 只有一份副本
- 该 Executor 上的所有 Task 共享使用
- 大幅减少网络传输和内存占用

**使用广播变量的注意事项：**

| 注意事项 | 说明 |
|---------|------|
| **不能广播 RDD** | RDD 不存储实际数据，只存储血统信息，无法广播 |
| **只读不可修改** | 广播变量是只读的，如需修改可将数据存到外部数据源（MySQL/Redis） |
| **大小需适中** | 如果两个 RDD 数据量都比较大，将其中一个做成广播变量可能造成 Executor 内存溢出 |
| **适用场景** | 小表关联（维度表）、需要复用的配置数据、 lookup 表等 |
| **手动释放** | 使用完可调用 `broadcastData.unpersist()` 释放 Executor 上的缓存 |

**典型回答示例：**

> "广播变量是 Spark 中用于高效分发大型只读数据集的优化技术。通过 `broadcast()` 方法将大型数据集发送到集群中每个 Executor 节点并本地缓存，而不是与每个 Task 一起发送。这样可以显著减少网络流量并提高性能，特别是在需要许多 Task 访问大型数据集的场景（如小表 join）中。需要注意的是，广播变量只能广播可以放入内存的数据，且只能读不能修改。另外，不能广播 RDD，因为 RDD 不存储实际数据。"
