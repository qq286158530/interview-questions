## 基础概念（1-5）

### 1. 为什么需要消息队列？它解决了什么问题？

**参考答案：**
- **异步处理**：将非核心流程从主流程中解耦，提升系统吞吐量（如订单完成后发短信/邮件可异步）
- **削峰填谷**：在大流量冲击时缓冲，避免数据库被打死（如秒杀场景）
- **系统解耦**：生产者和消费者无需同时在线，降低模块间依赖（如日志收集）
- **可靠传输**：保证消息至少被消费一次，支持消息持久化和重试

---

### 2. 消息队列有哪些常见的模型？

**参考答案：**
- **队列模型（Point-to-Point）**：每条消息只能被一个消费者消费，消费后从队列中删除。典型实现：RabbitMQ（basic.queue）
- **发布/订阅模型（Pub/Sub）**：一条消息被所有订阅者消费。典型实现：Kafka topic、Redis Pub/Sub、RocketMQ
- **注意**：RabbitMQ 通过 Exchange + Binding 路由也可实现发布订阅；Kafka 的 Consumer Group 机制在 group 内是竞争消费，对外是广播

---

### 3. 什么是 RabbitMQ 的 Exchange 类型？各自有什么区别？

**参考答案：**
| 类型 | 路由规则 | 典型场景 |
|------|---------|---------|
| **direct** | 精确匹配 routing key | 点对点精确路由 |
| **fanout** | 广播到所有绑定队列 | 多人通知、群发 |
| **topic** | 按通配符匹配（`*` `#`） | 灵活的主题路由 |
| **headers** | 按消息头属性匹配 | 复杂条件路由（少用） |

---

### 4. Kafka 的 Topic 和 Partition 是什么关系？

**参考答案：**
- **Topic** 是消息的逻辑分类，每个 Topic 可划分多个 **Partition**
- **Partition** 是物理存储单位，每条消息在 Partition 内有一个递增的 **offset**
- Partition 允许消息并行写入，是 Kafka 高吞吐的基础
- Partition 可以设置副本（replication）保证高可用
- 消费者以 Partition 为单位分配，一个 Partition 只能被一个 Consumer 实例消费（同一 Consumer Group 内）

---

### 5. RocketMQ 的消息存储结构是怎样的？

**参考答案：**
RocketMQ 采用**顺序写 + 随机读**的存储架构：

- **CommitLog**：所有消息顺序追加写入，是真正的存储文件，每个文件默认 1GB
- **ConsumeQueue**：消息消费队列，按 Topic 和 Queue 组织，存储消息在 CommitLog 的物理位置（offset、size、tag hash），是逻辑索引
- **IndexFile**：基于消息 key 的哈希索引，支持根据 key 快速查找消息
- 消息先写 CommitLog，再异步分发到 ConsumeQueue，实现读写分离

## 进阶原理（6-10）

### 6. Kafka 是怎么保证消息不丢失的？

**参考答案：**
从三个环节保证：

**生产者端：**
- `acks=all`（或 `acks=-1`）：leader + 所有 ISR 副本都写入成功才确认
- 开启 `retries` 重试 + `enable.idempotence=true` 防止重复
- 使用 Producer 回调监听发送失败

**Broker 端：**
- `replication.factor ≥ 3`（多副本）
- `min.insync.replicas ≥ 2`（最小同步副本数）
- 关闭 `unclean.leader.election`，防止数据丢失

**消费者端：**
- 手动提交 offset（`手动提交 + 业务处理成功`后再提交）
- 或利用事务机制，将 offset 提交和业务处理放在同一事务

---

### 7. 什么是 Kafka 的 ISR 机制？

**参考答案：**
ISR（In-Sync Replicas）是当前与 Leader 保持同步的副本集合。

**判定规则：** Replica 滞后于 Leader 超过 `replica.lag.time.max.ms`（默认 30s），或滞后消息数超过 `replica.lag.max.messages`，则被踢出 ISR

**作用：**
- 只有 ISR 内的副本才有资格被选为新 Leader，保证数据不丢失
- `min.insync.replicas` 配合 ISR 控制写入一致性级别
- 如果 ISR 为空，生产者写入会受阻（`NotEnoughReplicasException`）

---

### 8. RabbitMQ 的消息确认机制（Confirm 和 ACK）有什么区别？

**参考答案：**
- **Publisher Confirm**：生产者 → Broker，确认 Broker 是否成功接收并持久化
  - 开启：`publisher-confirms=true`
  - Broker 用 `basic.ack`（成功）或 `basic.nack`（失败）回调
  - 配合 `correlationData` 可追踪每条消息状态

- **Consumer ACK**：消费者 → Broker，确认消息已处理完成
  - 自动 ACK：消费者收到消息即自动确认（适用于高吞吐量且允许丢消息的场景）
  - 手动 ACK：业务处理完成后手动 `basic.ack`
  - 拒绝：`basic.nack` 或 `basic.reject`，可设置 `requeue=true` 重试或 `requeue=false` 丢弃

**建议**：生产环境用手动 ACK + 手动 Confirm，防止消息丢失

---

### 9. Kafka 的零拷贝（Zero-Copy）原理是什么？

**参考答案：**
Kafka 利用 Linux 的 `sendfile` 系统调用实现零拷贝：

**传统 IO（4次拷贝）：**
1. 磁盘 → 内核缓冲区（read）
2. 内核缓冲区 → 用户空间（copy）
3. 用户空间 → Socket 缓冲区（copy）
4. Socket 缓冲区 → 网卡（send）

**零拷贝（2次拷贝）：**
1. 磁盘 → 内核缓冲区（DMA 拷贝）
2. 内核缓冲区 → 网卡（sendfile，直接从内核缓冲区 DMA 传输）

**具体实现：**
```java
// Kafka 使用 NIO 的 FileChannel.transferTo()
FileChannel.transferTo(position, count, socketChannel);
```
减少上下文切换和 CPU 拷贝次数，大幅提升网络传输效率。

### 10. 如何保证消息不被重复消费？（幂等性）

**参考答案：**
消息重复的根本原因：网络抖动导致生产者重试，或消费者未及时提交 offset。

**通用解决方案：**
- **业务唯一 ID + 去重表**：消费前查库/Redis 确认是否已处理
- **数据库唯一约束**：利用唯一键防止重复插入
- **分布式锁**：用 Redis SETNX / Zookeeper 做幂等锁

**Kafka 特有方案：**
- 开启 `enable.idempotence=true`（ Exactly Once 语义）
- 结合事务：将消息处理和 offset 提交绑定为原子操作

**RabbitMQ 特有方案：**
- 使用 `basic.nack` + `requeue=false` + 死信队列（DLX）人工处理
- 或消费端做防重表检查

## 高可用与性能（11-15）

### 11. Kafka 的 Leader 选举机制是怎样的？

**参考答案：**
- **Controller 选举**：第一个启动的 Broker 通过 ZK（KRaft 模式通过自身）成为 Controller，负责分区 leader 选举
- **分区 Leader 选举**：
  - 优先从 ISR 列表中选
  - 如果 ISR 为空，`unclean.leader.election=true` 则从所有副本中选（可能丢数据）
  - 遵循"ISR 中 priority 最高的 replica 成为新 leader"规则
- **Controller 触发选举**：Broker 宕机时，Controller 通知所有 Broker 更新 LeaderAndIsr

---

### 12. RabbitMQ 的集群类型有哪些？各自适用场景？

**参考答案：**
| 类型 | 说明 | 适用场景 |
|------|------|---------|
| **普通集群** | 队列仅在主节点存储，节点间共享元数据 | 负载均衡，无高可用 |
| **镜像集群** | 队列内容自动同步到所有节点 | 高可用，数据不丢失 |
| **仲裁队列（Quorum）** | 基于 Raft 协议实现的高可用队列 | K8s 环境，优于镜像 |
| **联邦（Federation）** | 跨地域集群互联 | 多机房、异地部署 |

**推荐**：生产环境用仲裁队列（Quorum）或镜像集群，关闭 `lazy queue` 防止内存溢出

---

### 13. Kafka 如何保证消息的顺序性？

**参考答案：**
- **全局有序**：一个 Topic 只建 1 个 Partition（单消费者串行消费）
- **分区有序**：相同 key 的消息路由到同一 Partition，Partition 内有序
- **业务有序**：用业务字段（如 user_id）做 partition key，保证同一用户消息有序
- **消费者有序**：单 Consumer 多线程时需要用内存队列排序，或使用 `BlockingQueue` 顺序消费
- **注意**：Kafka Consumer 的 `max.poll.records` 过大可能导致处理乱序，需合理配置

---

### 14. 消息队列的过期时间（TTL）和死信队列（DLX）有什么用？

**参考答案：**
**TTL（Time-To-Live）：**
- 设置消息或队列的存活时间，超时自动删除
- RabbitMQ：队列设置 `x-message-ttl`，或单消息设置 `expiration` 字段
- 适用场景：临时任务、超时提醒

**死信队列（DLX）：**
- 消息被拒绝（`nack` 且不重队）、超时、队列满时，进入 DLX
- 用于收集失败消息，做人工补偿或重试
- RabbitMQ：`x-dead-letter-exchange` + `x-dead-letter-routing-key`
- Kafka：利用 `__consumer_offsets` 配合重试主题实现

### 15. RocketMQ 的事务消息是如何实现的？

**参考答案：**
RocketMQ 使用**半消息（Half Message）**机制：

1. **发送阶段一**：生产者发送 Half Message，Broker 预提交（不投递给消费者）
2. **本地事务**：生产者执行本地业务，提交或回滚
3. **确认阶段**：生产者提交 commit（或 rollback），Broker 投递（或删除）消息
4. **回查机制**：Broker 超过阈值未收到确认，触发 `checkLocalTransaction` 回调，检查本地事务状态

```java
// RocketMQ 事务消息示例
TransactionListener transactionListener = new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        // 本地业务逻辑
        return LocalTransactionState.COMMIT_MESSAGE;
    }
    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 定时回查，确认事务状态
        return LocalTransactionState.COMMIT_MESSAGE;
    }
};
```

## 实战与综合（16-20）

### 16. 如何排查 Kafka 消费延迟过高的问题？

**参考答案：**
排查步骤：
1. **监控 Lag**：看消费者 lag 是否持续增长（Kafka Manager / JMX）
2. **检查消费者吞吐量**：是否 `max.poll.records` 过大或处理过慢
3. **看 Broker 健康**：ISR 副本是否正常，网络/磁盘是否瓶颈
4. **消费者 GC**：频繁 GC 导致消费暂停
5. **检查分区数**：分区数过少导致消费者并行度不够
6. **反压检查**：下游数据库或服务是否成为瓶颈

**常用命令：**
```bash
kafka-consumer-groups.sh --bootstrap-server :9092 --group <group> --describe
# 关注 CURRENT-OFFSET、LOG-END-OFFSET、LAG
```

### 17. RabbitMQ 队列积压（消息堆积）怎么处理？

**参考答案：**
**紧急处理：**
1. **扩容消费者**：增加消费者实例，提升消费速度
2. **惰性队列**：将积压消息一次性转储到磁盘（`x-queue-mode: lazy`）
3. **重置消费位点**：临时跳过积压消息（`rabbitmqctl purge_queue`）

**长期方案：**
1. **监控告警**：提前发现积压趋势
2. **消费者优化**：提高单条消息处理效率，异步化
3. **消费者幂等**：防止重复处理
4. **限流**：生产者端做流量控制，避免继续积压
5. **分区/分片**：设计足够多的队列/分区支撑并发

### 18. Kafka 和 RabbitMQ 各自的优缺点和选型建议？

**参考答案：**
| 对比维度 | Kafka | RabbitMQ |
|---------|-------|---------|
| **吞吐量** | 极高（百万级/s） | 高（万级/s） |
| **消息持久化** | 顺序写，极快 | 磁盘存储，较慢 |
| **延迟** | 毫秒级 | 微秒级（更低） |
| **功能** | 丰富的 Consumer Group、分区、LSO | 灵活的 Exchange 路由、TTL、DLX |
| **事务** | 支持 Exactly Once | 支持（需要手动实现） |
| **运维复杂度** | 高（分区/副本/ISR 概念多） | 中等 |
| **适用场景** | 日志、大数据、流处理 | 业务消息、任务队列、复杂路由 |

**选型建议：**
- **大数据/日志收集/流处理** → Kafka
- **电商/金融/业务消息/复杂路由** → RabbitMQ / RocketMQ
- **需要事务消息** → RocketMQ
- **高吞吐低延迟日志系统** → Kafka

### 19. 如何设计一个可靠的 MQ 系统？关键点有哪些？

**参考答案：**
**核心设计点：**

1. **持久化**：消息必须落盘，Broker 宕机不丢失
2. **多副本**：至少 `replication.factor=3`，`min.insync.replicas=2`
3. **生产者确认**：Confirm 机制确保消息已落盘
4. **消费者幂等**：防重复消费
5. **死信队列**：处理失败消息不丢失
6. **监控告警**：消费 Lag、队列堆积、节点健康
7. **限流熔断**：保护下游系统，避免雪崩
8. **版本兼容**：消息格式做好 Schema，避免消费端升级后不兼容

**架构建议：**
- 生产者 → MQ → 消费者，MQ 自身做集群部署
- 消息分类：重要消息走高可用队列，普通日志走普通队列
- 定期清理过期消息，防止磁盘撑满

### 20. MQ 的消息积压了怎么处理？

**参考答案：**
**整体思路：先止血（快速恢复），再排查（根治问题）**

**止血（30分钟内）：**
1. **看监控**：确认积压量级、影响范围
2. **扩容消费端**：水平扩展消费者实例
3. **开启多消费者并行**：提高消费并行度
4. **临时降级**：非核心消费先暂停，把资源给核心链路
5. **消息转存**：RabbitMQ 可以把积压消息一次性转存到新的队列/磁盘

**排查（长期解决）：**
1. 消费者性能瓶颈：代码是否有阻塞（同步调用）、数据库慢查询
2. 下游服务不可用：连接池耗尽、接口超时
3. 消费者数量不足：Topic/Queue 分区数不够
4. 消息体过大：单条消息处理耗时过长
5. 网络问题：消费者和 Broker 之间网络抖动

**预防：**
- 监控告警提前发现（Lag 指标）
- 定期压测确定消费能力上限
- 生产端做好限流，避免突发流量冲垮消费端
