## 题目一：Spark 架构与核心组件

**问题：** 请描述 Apache Spark 的整体架构，以及 Driver、Executor、Master、Worker 的作用与协作关系。

**答案：**

Spark 采用 **主从架构**，核心组件包括：

| 组件 | 角色 | 说明 |
|------|------|------|
| **Driver** | 主控节点 | 负责执行用户代码，构建 DAG 图，将任务拆分成 Task，分发给 Executor |
| **Executor** | 工作节点 | 负责执行分发的 Task，向 Driver 汇报结果；每个 Executor 独立运行在 Worker 节点上 |
| **Master** | 资源管理主节点 | 负责集群资源管理（YARN 模式下由 ResourceManager 替代） |
| **Worker** | 工作节点守护进程 | 负责管理本节点上的资源，启动 Executor 进程 |

**协作流程：**

```
用户提交 Spark App
       ↓
Driver 向 Master 注册并申请资源
       ↓
Master 让 Worker 启动 Executor
       ↓
Executor 向 Driver 注册
       ↓
Driver 将 Task 发送给 Executor 执行
       ↓
Executor 执行完毕将结果返回 Driver
```

> 📎 参考来源：Apache Spark 官方文档 - [cluster mode overview](https://spark.apache.org/docs/latest/cluster-overview.html)

## 题目二：RDD 理解与特性

**问题：** 什么是 RDD？RDD 有哪些核心特性？为什么 Spark 选择 RDD 作为核心抽象？

**答案：**

**RDD（Resilient Distributed Dataset）** 是 Spark 最底层的不可变、分布式数据集抽象。

**五大核心特性：**

| 特性 | 说明 |
|------|------|
| **Partitioned** | 数据集被划分为多个分区，分区是并行计算的基本单位 |
| **Immutable** | 数据一旦创建就不可以修改，任何 transformation 都产生新的 RDD |
| **Resilient** | 支持基于 Lineage（血统）容错，某个分区失败可从父 RDD 重新计算 |
| **Lazy Evaluation** | transformation 是惰性执行的，只有遇到 action 才会触发真正的计算 |
| **分布式** | RDD 的分区分布在集群的多个节点上，支持并行处理 |

**为什么选择 RDD：**
- **高效的容错机制**：传统 MapReduce 需要将数据写磁盘做 Checkpoint，RDD 通过 Lineage 可以在失败时只重算丢失的分区
- **灵活的 API**：通过 transformation 组合可以实现复杂的数据处理逻辑
- **统一性**：既能做批处理，也能通过 Spark Streaming 做流处理

> 📎 参考来源：Matei Zaharia et al., "[Resilient Distributed Datasets: A Fault-Tolerant Abstraction for In-Memory Cluster Computing](https://www.usenix.org/system/files/conference/nsdi12/nsdi12-final138.pdf)" (USENIX NSDI 2012)

## 题目三：Spark Shuffle 机制

**问题：** 什么是 Shuffle？为什么 Spark 需要 Shuffle？Shuffle 过程中有哪些优化？

**答案：**

**Shuffle** 是 Spark 中将数据按照 Key 进行重新分布的过程，类似于 MapReduce 的 Shuffle 阶段。当执行 `groupByKey`、`reduceByKey`、`join` 等需要跨分区聚合或关联的操作时，就会触发 Shuffle。

**为什么需要 Shuffle：**
- 某些 operation（如 `reduceByKey`）需要将相同 Key 的数据拉到同一个分区才能聚合
- `join` 操作需要将相同 Key 的两表数据拉到同一分区

**Shuffle 执行流程（Spark 2.x+）：**

```
Shuffle Write (Map 端)
  ↓ Executor 将数据写入本地磁盘，生成多个 shuffle block files
  ↓ 按照 reduceId 合并为单独的文件（避免小文件问题）
Shuffle Read (Reduce 端)
  ↓ Executor 通过 HTTP/Netty 从 Map 端拉取数据
  ↓ 按 Key 聚合后交给下游 Task 处理
```

**Shuffle 优化技术：**

| 优化 | 说明 |
|------|------|
| **Shuffle Consolidate** | 多个 Map 任务共用一个输出文件，减少文件数量 |
| **Sort-based Shuffle** | Spark 1.2+ 默认，Map 端排序写入，减少随机写 |
| **Tungsten Sort** | Spark 1.5+ 引入，使用堆外内存和 binary 序列化，提升性能 |
| **Bypass Merge** | 分区数少时直接合并文件，绕过 sorter |
| **Push-based Shuffle** | Spark 3.2+ 引入，Map 端主动推送数据到 Reduce 端，减少网络传输 |

> 📎 参考来源：Apache Spark 官方文档 - [Shuffle Behavior](https://spark.apache.org/docs/latest/rdd-programming-guide.html#shuffle-operations)

## 题目四：广播变量（Broadcast Variables）

**问题：** 什么是广播变量？广播变量适用于什么场景？使用广播变量有什么注意事项？

**答案：**

**广播变量**是 Spark 提供的只读共享变量，会自动被分发到各个节点，且在每个节点上只保存一份副本，避免每个 Task 都重复传输一份数据。

**适用场景：**
- **小表广播**：将维度表/配置表广播到所有节点，用于 `broadcast join`
- **大文件共享**：如查找表、IP 库等在多个 Task 间共享的数据
- **配置参数**：所有节点需要读取的常量配置

**使用方式：**

```scala
// 广播一个只读变量
val broadcastVar = sc.broadcast(Array(1, 2, 3))

// 在 Task 中读取（不用再传参）
rdd.map(x => broadcastVar.value.sum + x)
```

**注意事项：**

| 注意点 | 说明 |
|--------|------|
| **大小控制** | 广播的变量不宜过大（建议 < 20MB），过大反而占内存 |
| **只读不可修改** | 广播变量是只读的，修改不会生效也不会报错 |
| **释放时机** | 长时间任务使用完后要调用 `.unpersist()` 释放内存 |
| **序列化** | 广播变量需要支持 Java 序列化，自定义类需实现 Serializable |
| **Driver 设置** | 广播机制由 Driver 统一管理，Driver 挂了广播变量失效 |

> 📎 参考来源：Apache Spark 官方文档 - [Broadcast Variables](https://spark.apache.org/docs/latest/rdd-programming-guide.html#broadcast-variables)

## 题目五：Spark 内存管理

**问题：** Spark 的内存管理模型是怎样的？Storage Memory、Execution Memory、User Memory 分别用来存储什么？

**答案：**

Spark 1.6+ 采用 **统一内存管理（Unified Memory Management）**，将堆内内存划分为多个区域。

**内存区域划分：**

| 内存区域 | 占比（默认） | 用途 |
|----------|------------|------|
| **System Memory** | 300MB（保留） | JVM 运行时预留 |
| **Storage Memory** | 约 50% | 缓存 RDD 缓存、广播变量、临时数据结构 |
| **Execution Memory** | 约 50% | Shuffle 排序、Hash 表、Task 运行时内存 |
| **User Memory** | 约 40% | 用户代码数据结构、元数据等 |
| **Reserved Memory** | 300MB | 内部对象使用 |

> 注：上表中 Storage + Execution 合计约 60%（spark.memory.fraction=0.6），User Memory 占比约 40%（spark.memory.storageFraction=0.5 控制可互相借用的边界）

**Storage Memory 与 Execution Memory 可以互相借用：**
- Execution 内存不足时，可以借用 Storage 空闲内存（但 Storage 需要时可驱逐）
- Storage 内存不足时，可以借用 Execution 空闲内存（但不能驱逐正在使用的 Execution 内存）

**Spark 2.x 堆外内存：**
- 通过 `spark.memory.offHeap.enabled=true` 开启堆外内存
- `spark.memory.offHeap.size` 设置堆外大小
- 主要用于 Shuffle 排序的堆外缓冲区，减少 GC 开销

**内存溢出的常见原因：**
1. 数据量远大于可用内存
2. `groupByKey` 导致数据倾斜或 OOM
3. `collect` 拉取大量数据到 Driver
4. 广播变量过大

> 📎 参考来源：Apache Spark 官方文档 - [Memory Management](https://spark.apache.org/docs/latest/tuning.html#memory-management)
