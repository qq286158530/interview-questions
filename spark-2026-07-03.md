## 题目一：Spark 任务执行流程 & 架构原理

**请描述 Spark Application 的完整执行流程，从提交 jar 包到 Task 执行完毕，涉及到哪些核心组件？它们之间如何协作？**

### 参考答案

Spark 采用 **主从架构**，核心组件如下：

1. **Driver（驱动器）**
   - 运行用户代码（SparkContext），负责 DAG 构造、Stage 划分、Task 生成与调度。
   - 所在节点：提交任务的客户端机器或 YARN Cluster 模式的 AM 节点。

2. **Cluster Manager（集群管理器）**
   - 负责集群资源管理（YARN / Standalone / Mesos / Kubernetes）。
   - 在 YARN 模式下，表现为 ResourceManager / NodeManager。

3. **Executor（执行器）**
   - 分布在各个 Worker 节点上，每个 Executor 是一个 JVM 进程，负责执行分发的 Task。
   - Executor 启动后向 Driver 反向注册，并管理所在节点的 Task 执行和数据缓存（BlockManager）。

**完整执行流程：**

```
1. 用户通过 spark-submit 提交 Application
2. Driver 创建 SparkContext，向 Cluster Manager 申请资源
3. Cluster Manager 在 Worker 节点上启动 Executor（JVM 进程）
4. Executor 向 Driver 反向注册
5. Driver 根据 Shuffle 是否跨 partition 将 DAG 划分为多个 Stage
   （宽依赖触发 Shuffle，窄依赖在同一个 Stage 内流水线执行）
6. 每个 Stage 包含若干 Task（一个 Task 对应一个 Partition）
7. Task 被包装成 TaskSet，通过 Cluster Manager 分发到各 Executor
8. Executor 执行 Task，计算结果写入本地或 Shuffle 输出
9. 结果汇总回 Driver，Driver 输出最终结果
```

**关键设计：**
- **DAG 调度**：根据 RDD 依赖关系（有向无环图）将 Job 划分为 Stage，窄依赖"Pipelining"，宽依赖"Shuffle"。
- **TaskSet 并行度**：每个 Stage 的 Task 数量 = 该 Stage 最后一个 RDD 的 Partition 数量。
- **数据本地性**：调度器尽量将 Task 发送到数据所在节点，减少网络传输（PROCESS_LOCAL > NODE_LOCAL > RACK_LOCAL）。

## 题目二：RDD 理解与五大特性

**RDD（Resilient Distributed Dataset）的五大特性是什么？如何理解 RDD 的血统（Lineage）机制？它如何实现容错？**

### 参考答案

RDD 是 Spark 最核心的数据抽象，表示**一个弹性的、分布式的数据集**，具备以下五大特性：

1. **分区列表（Partitions）**
   - RDD 数据被划分为多个 Partition，每个 Partition 可并行处理。
   - 可通过 `partitions()` 方法获取分区列表。

2. **计算函数（Compute Function）**
   - 每个 RDD 知道如何在一个分区上执行计算（`compute(spit: Partition, context: TaskContext)`）。
   - 依赖父 RDD 的分区信息。

3. **依赖关系（Dependencies）**
   - RDD 记录与父 RDD 的依赖关系，分为：
     - **窄依赖（Narrow Dependency）**：一个父分区最多被子集一个子分区使用（如 map、filter）。支持流水线执行，一个分区失败只影响一条计算链。
     - **宽依赖（Wide Dependency）**：一个父分区被多个子分区使用（如 groupByKey、join）。触发 Shuffle，是 Stage 划分的依据。

4. **分区器（Partitioner）**（仅针对 Key-Value RDD）
   - 可选，标识 RDD 是否按 Hash 或 Range 进行分区。
   - `groupByKey`、`reduceByKey` 等操作会保留或创建新的 Partitioner，影响 Shuffle 行为。

5. **数据位置偏好（Preferred Locations）**
   - RDD 记录每个分区最佳计算位置的列表（数据本地性），调度器据此将 Task 分配到数据所在节点。

**血统机制（Lineage）与容错：**

```
RDD Lineage 是一个 DAG，记录了从源头 RDD 到当前 RDD 的完整转换链。
当某个分区数据丢失时，Spark 根据 Lineage 从父 RDD 重新计算该分区进行恢复，
无需复制整个数据集——这是 RDD "弹性"的核心。
```

- 窄依赖的 Lineage 支持高效恢复（类似流水线重跑）。
- 宽依赖（如 ShuffleMapStage 的输出）会触发 Checkpoint，将数据持久化到磁盘，进一步缩短恢复路径。

## 题目三：DataFrame 与 DataSet 的区别 & 进化关系

**Spark 中 DataFrame 和 DataSet 有什么区别？它们与 RDD 相比有哪些优势？为什么说 DataFrame 是一种"懒执行"的 API？**

### 参考答案

### 1. 三者关系

```
RDD[Row] (无类型，编译时弱检查)
    ↓ 自动包装
Dataset[Row] == DataFrame (静态类型语言如 Scala 中，DataFrame 是 Dataset[Row] 的别名)
    ↓ 泛型约束
Dataset[T] (强类型，编译时类型安全)
```

- **DataFrame**：即 `Dataset[Row]`，Row 是一种通用无类型记录，支持 Spark SQL。
- **DataSet**：强类型的 `Dataset[T]`，T 可以是 case class 或 Java Bean，编译时类型安全。

### 2. DataFrame 相对 RDD 的优势

| 特性 | RDD | DataFrame |
|------|-----|-----------|
| **类型安全** | 弱（运行时才发现错误） | 强（编译时检查） |
| **执行优化** | 无（用户控制执行逻辑） | 有（Catalyst 优化器） |
| **内存布局** | JVM 对象堆（Java 对象） | Tungsten 二进制格式（堆外内存） |
| **GC 开销** | 高（频繁对象创建） | 低（紧凑二进制） |
| **SQL 支持** | 无 | 完整支持 |
| **API 友好度** | 低（需手动处理 Shuffle） | 高（声明式） |

### 3. Catalyst 优化器的工作原理（懒执行）

DataFrame/Dataset 的所有操作（如 `select`、`filter`、`groupBy`）只是**构建逻辑执行计划**，不会立即执行。只有遇到**Action 操作**（如 `collect`、`count`、`show`）时：

```
1. DataFrame DSL  →  Logical Plan（未解析的逻辑计划）
2. Analyzer 解析  →  Analyzed Logical Plan
3. Catalyst 优化  →  Optimized Logical Plan（如 Predicate Pushdown、Column Pruning）
4. 物理计划生成   →  Physical Plans（多个可选计划）
5. 成本模型选择   →  最佳 Physical Plan
6. 编译为 RDD    →  生成 RDD lineage 并执行
```

这一机制使得 Spark 能在执行前做全局优化（如过滤下推、列剪裁），而 RDD 用户手动编写的代码无法享受此类优化。

## 题目四：Shuffle 原理与优化策略

**什么是 Shuffle？为什么 Shuffle 是 Spark 性能瓶颈？请列举至少 5 种 Shuffle 优化手段，并说明原理。**

### 参考答案

### 什么是 Shuffle

Shuffle 是**将数据按 Key 重新分区并跨节点传输**的过程。当 RDD 依赖图中出现宽依赖（如 `groupByKey`、`join`、`reduceByKey`）时，Spark 必须将具有相同 Key 的数据汇聚到同一个节点，这一跨网络传输的过程即为 Shuffle。

```
Stage 1 (Map)  →  Shuffle Write (写本地磁盘文件)  →  Shuffle Read (从其他节点拉取)  →  Stage 2 (Reduce)
```

### 为什么是性能瓶颈

1. **磁盘 I/O**：Shuffle 数据先写入本地磁盘，网络传输后再读取。
2. **网络 I/O**：数据跨节点传输，是最耗时的部分。
3. **内存压力**：Shuffle 聚合需要将数据放在内存中，大数据量易导致 OOM 或频繁 Spill 到磁盘。
4. **启动开销**：每个 Task 都要经历文件创建、序列化、网络传输、反序列化等步骤。
5. **相关性破坏**：破坏数据本地性，无法利用"计算向数据移动"原则。

### 五大优化手段

**① 设置合理的 Shuffle Partition 数量**
- 默认 200（Spark 2.x），过大或过小都有问题。
- 公式参考：`numPartitions = max(200, CPU cores * 4)`；或根据数据量调整（每分区 200KB~200MB 为宜）。
- 调参：`spark.sql.shuffle.partitions`（SQL场景）或 `spark.default.parallelism`（RDD场景）。

**② 使用 Map-Side 聚合（Combiner）**
- `reduceByKey`、`aggregateByKey` 会在 Map 端先做本地聚合（Combiner），再 Shuffle，显著减少网络传输量。
- 相比 `groupByKey`（无聚合），可减少数倍数据量。

**③ 广播变量（Broadcast Join）替代 Shuffle Join**
- 当小表可以放入内存时，将小表广播到所有 Executor 节点，在本地完成 Join，避免 Shuffle。
- 调参：`spark.sql.autoBroadcastJoinThreshold`（默认 10MB）。
- 适用场景：大表 join 小表。

**④ 开启 Sort Shuffle（而不是 Hash Shuffle）**
- Spark 2.x 后默认使用 Sort Shuffle（`spark.shuffle.manager = sort`）。
- Sort Shuffle 将数据写入有序文件，Reduce 阶段只需归并排序即可，支持大数据量。
- 可选开启 `spark.shuffle.sort.bypassMergeThreshold` 进一步优化小数据量场景。

**⑤ 合理使用 Kryo 序列化**
- Shuffle 过程涉及大量对象序列化，Kryo 比 Java Serialization 快 10 倍以上。
- 配置：`spark.serializer = org.apache.spark.serializer.KryoSerializer`
- 注册自定义类：`registerKryoClasses(Array(classOf[MyClass]))`

**其他优化点：**
- 开启 `spark.shuffle.compress = true`（默认开启，压缩 Shuffle 输出）。
- 适当增大 `spark.shuffle.service.index.cache.entries`（缓存更多 Shuffle 文件元数据）。
- 避免低选择性的 `filter` 后立即 `groupBy`，利用 Predicate Pushdown 将 filter 下推到数据源。

## 题目五：广播变量原理与内存管理模型

**什么是广播变量（Broadcast Variable）？Spark 的内存管理模型是怎样的？堆内内存与堆外内存有什么区别？如何通过调参避免 OOM？**

### 参考答案

### 广播变量

**定义**：广播变量是将**只读数据**高效分发到所有 Executor 节点的机制，避免每个 Task 重复传输同一份数据。

**原理**：
1. Driver 端将数据序列化为二进制并分成多个小块（block），存储在 Driver 本地文件系统和 BlockManager（内存+磁盘混合存储）。
2. 各 Executor 的 BlockManager 作为"服务器"，响应其他 Executor 的块请求。
3. 广播变量以 `TorrentBroadcast` 方式工作：每个 Executor 拉取块后同时成为该块的"供源"，减轻 Driver 压力。

**使用示例**：
```scala
// Driver 端
val broadcastVar = sc.broadcast(Array(1, 2, 3, 4, 5))

// Executor 端（每个 Task 可访问）
val result = broadcastVar.value.map(_ * 2)
```

**典型应用**：大维度表 join 时将小表广播；配置表、字典表共享。

---

### Spark 内存管理模型

Spark 使用**统一内存管理（Unified Memory Management）**（Spark 1.6+），将堆内/堆外内存统一划分为：

```
┌─────────────────────────────────────┐
│          Execution Memory           │ ← Shuffle、排序、Hash Join 等算子的临时内存
├─────────────────────────────────────┤
│           Storage Memory            │ ← RDD Cache、Broadcast、Task 输出缓存
├─────────────────────────────────────┤
│           User Memory               │ ← 用户数据结构、自定义类、元数据等
├─────────────────────────────────────┤
│           Reserved Memory           │ ← 系统预留（300MB）
└─────────────────────────────────────┘
```

**核心规则**：
- **Execution 与 Storage 之间可以互相借用空闲内存**：Execution 空间不足时可 spill 到磁盘，但可申请 Storage 空闲区；Storage 不足时可驱逐缓存块。
- **当一方内存不足且另一方也无空闲时，双方都不能抢占对方已使用部分**，只能 Spill 到磁盘。

**堆内内存（On-Heap）**：
- 由 JVM 管理的 Java 堆空间。
- 受 GC 影响，大内存易导致长 GC 停顿。
- 参数：`spark.memory.fraction`（默认 0.6，即 60% 用于 Execution + Storage），`spark.memory.storageFraction`（默认 0.5，即 Storage 占其中一半）。

**堆外内存（Off-Heap）**：
- 使用 JDK `ByteBuffer` API 直接分配操作系统物理内存（通过 `spark.memory.offHeap.enabled = true`）。
- 优点：避免 GC 停顿，适合大内存场景（ Tungsten 序列化数据直接存在堆外）。
- 参数：`spark.memory.offHeap.size`（堆外内存大小，需设置）。

---

### 避免 OOM 的调参建议

| 参数 | 默认值 | 建议 | 说明 |
|------|--------|------|------|
| `spark.memory.fraction` | 0.6 | 数据量大时降至 0.5 | 降低内存区占比，增大用户代码空间 |
| `spark.memory.storageFraction` | 0.5 | Shuffle 密集型降至 0.4 | 减少 Storage 预留，增大 Execution |
| `spark.driver.memory` | 1g | 设为 4g~8g（大数据集） | Driver 持有元数据，大数据量需增加 |
| `spark.executor.memory` | 1g | 设为 4g~16g（大数据集） | 每个 Executor 的内存总量 |
| `spark.memory.offHeap.enabled` | false | 大数据量设为 true | 开启堆外内存，减少 GC |
| `spark.memory.offHeap.size` | 0 | 设为 executor.memory * 0.5 | 堆外内存大小 |
| `spark.sql.autoBroadcastJoinThreshold` | 10MB | 小表 100MB 以内可广播 | 广播 Join 避免 Shuffle |
| `spark.shuffle.service.enabled` | false | 生产环境设为 true | External Shuffle Service，减少 Executor 间依赖 |

**实战技巧**：当出现 `OOM: GC overhead limit exceeded` 时，首先增大 `spark.executor.memory` 或开启堆外内存；其次检查是否存在大 Key 的 `groupByKey`，考虑改用 `reduceByKey` 或 `aggregateByKey` 做 Map 端聚合。
