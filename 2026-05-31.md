## 题目1：Java基础 | HashMap的底层结构与JDK1.8后的优化

**【答案】**

HashMap是Java中最常用的键值对存储容器，其底层结构在JDK1.8后发生了重大变化。

### 底层数据结构演变

**JDK1.7及之前：数组+链表（Entry数组）**
- 使用Entry数组存放数据，每个bucket中如果存在多个Entry，则以链表形式连接
- 头插法插入新元素（可能造成死循环问题）

**JDK1.8及之后：数组+链表+红黑树**
- 当单个bucket中链表长度超过8且总Entry数超过64时，链表转为红黑树
- 长度小于6时，红黑树退化为链表
- 尾插法插入新元素，避免了死循环问题
- 引入红黑树后将时间复杂度从O(n)降低到O(logn)

### 核心参数
- `DEFAULT_INITIAL_CAPACITY = 16` — 默认初始容量
- `loadFactor = 0.75` — 负载因子，Entry数超过 capacity×loadFactor 时触发扩容
- 扩容机制：容量翻倍（2倍），重新计算每个元素的bucket位置

### put方法流程
1. 计算key的hash值（扰动函数处理减少碰撞）
2. 计算bucket下标
3. 如果该bucket为空，直接插入
4. 如果不为空，遍历链表/红黑树：
   - 如果key已存在，覆盖value
   - 如果不存在，插入新节点
5. 如果size超过threshold，触发resize扩容

### 线程不安全问题
HashMap是非线程安全的。并发场景下推荐使用：
- `ConcurrentHashMap` — 分段锁（JDK7）/ CAS+ synchronized（JDK8）
- `Collections.synchronizedMap(new HashMap<>())` — 全局锁
- `Hashtable` — 全局锁（不推荐，性能差）

### 常考面试点
- 为什么HashMap容量是2的幂次？— 位运算代替取模，提高计算效率
- hash扰动函数：key.hashCode() ^ (h >>> 16)，让高低位都参与运算
- 红黑树vs链表：为什么选8作为转换阈值？

## 题目2：Java进阶 | synchronized与ReentrantLock的核心区别

**【答案】**

### 基本概念

**synchronized**
- Java关键字，由JVM实现，自动获取/释放锁
- 修饰实例方法、静态方法、代码块
- 底层依赖monitorenter/monitorexit字节码指令

**ReentrantLock**
- java.util.concurrent.locks包下的显式锁
- 需要手动调用lock()获取锁，unlock()释放锁（必须在finally中释放）
- 继承自AbstractQueuedSynchronizer(AQS)

### 核心区别

| 特性 | synchronized | ReentrantLock |
|------|-------------|--------------|
| 锁获取方式 | 自动获取/释放 | 手动调用lock()/unlock() |
| 公平性 | 非公平锁 | 可配置公平/非公平 |
| 等待可中断 | 不可中断 | 可通过lockInterruptibly()中断等待 |
| 绑定多个条件 | 不支持 | 通过newCondition()支持多个Condition |
| 尝试获取锁 | 不支持 | tryLock()尝试获取 |
| 底层实现 | JVM内置，字节码 | AQS框架+CAS+volatile |

### ReentrantLock的独占模式核心原理

ReentrantLock内部维护了一个`state`状态变量：
- `state=0`表示无锁
- `state>0`表示已有线程持有锁，数值表示重入次数
- 获取锁时通过CAS将state从0改为1（可重入通过累加实现）
- 等待队列使用CLH队列（FIFO双向链表）

### synchronized的优化（锁升级）

JDK1.6后引入"偏向锁→轻量级锁→重量级锁"的升级机制：
- **偏向锁**：第一次CAS将线程ID写入对象头，同一线程再次访问时无需同步
- **轻量级锁**：对象头中创建Lock Record，通过CAS抢锁，自旋等待
- **重量级锁**：膨胀后使用monitor对象，涉及内核态调度，不自旋

### 使用场景建议
- 简单同步需求优先用synchronized（更简洁安全）
- 需要可中断锁、公平锁、多条件Condition时用ReentrantLock
- 禁止在锁未释放时抛异常，必须用try-finally保护unlock()

## 题目3：Spring Boot | 自动配置原理与@Conditional注解体系

**【答案】**

### 自动配置核心原理

Spring Boot自动配置的灵魂是`@EnableAutoConfiguration`注解（通过`@SpringBootApplication`间接包含）。

**加载流程：**
1. Spring Boot应用启动时，会扫描`META-INF/spring.factories`文件
2. 读取`EnableAutoConfiguration`下所有的自动配置类全限定名
3. 通过`@Conditional`系列注解进行条件匹配
4. 只加载满足条件的配置类，实现按需配置

### @Conditional注解家族

`@Conditional`是Spring 4.0引入的核心注解，其作用是：**只有当指定条件成立时，当前的配置类或Bean才会生效**。

#### 常见实现：

| 注解 | 条件 |
|------|------|
| `@ConditionalOnClass` | classpath中存在指定类时生效 |
| `@ConditionalOnMissingClass` | classpath中不存在指定类时生效 |
| `@ConditionalOnBean` | 容器中存在指定Bean时生效 |
| `@ConditionalOnMissingBean` | 容器中不存在指定Bean时生效 |
| `@ConditionalOnProperty` | 配置文件中存在指定属性时生效 |
| `@ConditionalOnResource` | 存在指定资源文件时生效 |
| `@ConditionalOnWebApplication` | 是Web应用时生效 |
| `@ConditionalOnExpression` | SpEL表达式为true时生效 |

### 实战：自定义自动配置类

```java
@Configuration
@ConditionalOnClass(DataSource.class)           // 只有DataSource类在classpath时才加载
@ConditionalOnMissingBean(DataSource.class)    // 只有容器中没有自定义DataSource Bean时才加载
@ConditionalOnProperty(                         // 只有配置文件中 enabled=true 时才加载
    prefix = "app.datasource",
    name = "enabled",
    havingValue = "true",
    matchIfMissing = true
)
@EnableConfigurationProperties(DataSourceProperties.class) // 启用属性配置类
public class MyAutoConfiguration {
    // ...
}
```

### 自动配置优先级

- 用户自定义配置 > 自动配置（通过@Bean定义的Bean优先）
- `@EnableAutoConfiguration(exclude = {...})` 可排除特定自动配置
- `spring.autoconfigure.exclude` 配置项也可排除

### 底层原理：SpringFactoriesLoader

Spring Boot使用`SpringFactoriesLoader`机制加载`META-INF/spring.factories`：
```properties
# spring-boot-autoconfigure/META-INF/spring.factories
EnableAutoConfiguration=\
  org.springframework.boot.autoconfigure.AutoConfiguration.imports
```
JDK13+后使用`AutoConfiguration.imports`文件替代`spring.factories`。

## 题目4：Spring Cloud | Gateway网关核心原理与过滤器链

**【答案】**

### Gateway定位

Spring Cloud Gateway是Spring官方推出的第二代网关，替代了Zuul（非响应式）和Zuul2。它基于**WebFlux**（响应式编程模型），使用Netty作为运行时，支持高并发。

### 核心概念

- **Route（路由）**：网关的基本构建单位，包含`id`、`uri`、`predicates`、`filters`
- **Predicate（谓词）**：路由匹配条件，如请求路径、请求时间、Header等
- **Filter（过滤器）**：请求/响应拦截和处理，支持修改请求、修改响应、熔断等

### 工作流程

```
请求 → RoutePredicateHandlerMapping（路由匹配）
      ↓
  DispatcherHandler（请求分发）
      ↓
  FilteringWebHandler（构建过滤器链）
      ↓
  顺序执行 Pre Filter（如日志、认证）
      ↓
  代理服务（Proxy Filter）
      ↓
  顺序执行 Post Filter（如修改响应）
      ↓
  响应
```

### 动态路由配置

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service        # lb://表示负载均衡
          predicates:
            - Path=/api/user/**
            - After=2026-01-01T00:00:00Z
          filters:
            - StripPrefix=1            # 去掉第一层路径
            - name: RequestRateLimiter  # 限流过滤器
              args:
                redis-rate-limiter.replenishRate: 10
                redis-rate-limiter.burstCapacity: 20
```

### 过滤器执行顺序

- 所有过滤器都实现`Ordered`接口，通过`getOrder()`返回值排序
- 数字越小优先级越高
- Pre Filter（正序）→ 代理 → Post Filter（倒序）

### 内置过滤器

**请求修改类：**
- `AddRequestHeader` / `RemoveRequestHeader` — 添加/移除请求头
- `AddRequestParameter` — 添加请求参数
- `StripPrefix` — 去掉URL前缀路径

**响应修改类：**
- `AddResponseHeader` / `RemoveResponseHeader` — 添加/移除响应头

**全局过滤器：**
- `GatewayFilter` — 局部过滤器，针对特定路由
- `GlobalFilter` — 全局过滤器，对所有路由生效（如认证、IP黑白名单）

### 与Feign的区别
- Gateway：入口网关，处理**外部请求**，对所有下游服务统一入口
- Feign：服务间调用，处理**内部服务间**的HTTP通信

## 题目5：Spring Cloud | Seata分布式事务AT模式原理

**【答案】**

### 分布式事务问题背景

在微服务架构中，一个业务操作可能涉及多个服务，每个服务有独立的数据库。普通的事务（本地事务）只能管理单个服务的数据，无法保证跨服务的原子性。

**CAP定理：** 在分布式系统中，无法同时满足一致性（Consistency）、可用性（Availability）、分区容错（Partition tolerance）。

**BASE理论：** 最终一致性，允许分布式事务在各节点执行过程中出现短暂不一致，最终达到一致状态。

### Seata三种事务模式

| 模式 | 隔离级别 | 性能 | 侵入性 |
|------|---------|------|-------|
| AT | 读写隔离（全局锁） | 最高（自动补偿） | 低（SQL自动回滚） |
| TCC | 业务干预 | 高 | 高（需写三个接口） |
| SAGA | 无隔离 | 较高 | 高（需编排） |
| XA | 强一致 | 最低 | 高（需数据库支持） |

### AT模式核心原理

**AT模式 = 两阶段提交 + 自动补偿**

**一阶段（Branch Branch）：**
1. 解析SQL，得到SQL类型（INSERT/UPDATE/DELETE）
2. 查询执行前快照（before image）
3. 执行SQL
4. 查询执行后快照（after image）
5. 生成回滚日志（undo log），与业务SQL在同一个本地事务中提交

**二阶段（异步补偿）：**
- **提交**：异步删除undo log
- **回滚**：根据undo log自动生成补偿SQL，恢复到before image

### 全局事务协调

```
TM（事务管理器）: 定义全局事务边界，开启/提交/回滚
TC（事务协调器）: 独立进程，维护全局事务状态，管理分支事务注册
TM: 发起分支事务注册 → TC: 分配XID → 各Branch: 执行SQL并注册分支
```

### 关键特性

- **全局锁**：Seata通过TC维护全局锁，保证读写隔离
- **全局唯一XID**：贯穿整个分布式事务链路
- **写隔离**：主分支提交前，需等待所有子分支执行完成（或回滚）
- **读未提交**：AT模式默认隔离级别为读未提交全局最新数据

### 使用示例

```yaml
# Seata客户端配置
seata:
  enabled: true
  tx-service-group: my-service-group
  service:
    vgroup-mapping:
      my-service-group: default
    groupl-list: 127.0.0.1:8091
```

```java
@GlobalTransactional(rollbackFor = Exception.class)  // 开启全局事务
public void createOrder(OrderDTO orderDTO) {
    orderService.create(orderDTO);      // 本地事务
    inventoryService.deductStock(...); // 分布式事务
    paymentService.pay(...);          // 分布式事务
}
```

## 题目6：消息队列 | Kafka高可用架构与数据丢失问题

**【答案】**

### Kafka高可用架构

Kafka通过**多副本机制**实现高可用，核心架构如下：

**关键概念：**
- **Topic（主题）**：消息的逻辑分类
- **Partition（分区）**：每个Topic可划分为多个分区，是并行消费的基本单位
- **Replica（副本）**：分区的副本，分Leader和Follower
- **ISR（In-Sync Replicas）**：与Leader保持同步的副本集合

### 副本同步机制

```
Producer → Broker1(Leader) ← Broker2(Follower)
          ↓ 同步写入
        Follower 拉取数据并写入本地日志
```

- Leader负责所有读写请求
- Follower被动从Leader拉取数据（Pull模式）
- 只有ISR中的Follower才有资格被选为新Leader

**acks配置（数据可靠性级别）：**
- `acks=0`：Producer不等待响应，极易丢失数据
- `acks=1`：Leader写入成功即返回，存在Leader故障丢失数据风险
- `acks=all（-1）`：ISR中所有副本写入成功才返回，最高可靠性

### 数据丢失场景与解决方案

**场景1：Producer发送失败未重试**
- 解决：配置`retries > 0`，配合`enable.idempotence=true`（幂等发送）
- 幂等Producer：同一个消息发送多次只写入一次

**场景2：Leader切换时数据未同步**
- 解决：使用`acks=all` + `min.insync.replicas >= 2`
- 确保ISR中至少2个副本都写入成功

**场景3：消费offset提交过早**
- 解决：手动提交offset（`commitSync`/`commitAsync`）在业务处理完成后
- 或关闭自动提交，`enable.auto.commit=false`

**场景4：刷盘策略问题**
- 解决：`log.flush.interval.messages=1`（每条消息强制刷盘，性能差）
- 通常依赖操作系统page cache + 磁盘顺序写的特性

### 最佳实践配置

```properties
# Producer
acks=all
retries=Integer.MAX_VALUE
enable.idempotence=true
max.in.flight.requests.per.connection=5

# Broker
min.insync.replicas=2
replication.factor=3
unclean.leader.election.enable=false   # 禁止从不同步的副本中选Leader
```

### 与RabbitMQ可靠性对比
- RabbitMQ：消息确认机制（ACK）+ 手动签收 + 持久化交换机/队列/消息
- Kafka：高吞吐量场景下的可靠性配置更复杂，需权衡性能

## 题目7：消息队列 | RabbitMQ消息确认机制与可靠性保障

**【答案】**

### RabbitMQ消息可靠性保障体系

RabbitMQ从消息发送、存储、消费三个环节保障消息不丢失。

### 1. 消息发送确认（Publisher Confirm）

**机制：** 生产者将消息写入RabbitMQ后，Broker返回确认（ACK）或拒绝（NACK）。

**配置：**
```java
// 开启 publisher confirms
channel.confirmSelect();

// 异步监听确认
channel.addConfirmListener((deliveryTag, multiple) -> {
    // ACK：消息成功写入
}, (deliveryTag, multiple) -> {
    // NACK：消息写入失败
});

// 或同步等待
boolean flag = channel.waitForConfirms();
```

### 2. 消息持久化（Persistence）

必须同时满足以下三个条件，消息才不会在RabbitMQ宕机时丢失：

```java
// 1. 交换机持久化
channel.exchangeDeclare("my-exchange", BuiltinExchangeType.DIRECT, true);

// 2. 队列持久化
channel.queueDeclare("my-queue", true, false, false, null);

// 3. 消息持久化
byte[] body = "hello".getBytes();
AMQP.BasicProperties props = new AMQP.BasicProperties.Builder()
    .deliveryMode(2)  // 2=持久化
    .build();
channel.basicPublish("my-exchange", "my-routing-key", props, body);
```

### 3. 消费确认（Consumer Ack）

**消息应答模式：**
- **Auto Ack（自动确认）**：Consumer收到消息即自动发送ACK，消息被视为已消费
- **Manual Ack（手动确认）**：必须调用`basicAck()`才确认消费完成

```java
// 手动确认示例
channel.basicConsume("my-queue", false, new DefaultConsumer(channel) {
    @Override
    public void handleDelivery(String consumerTag, Envelope envelope,
                              AMQP.BasicProperties properties, byte[] body) {
        try {
            // 业务处理
            processMessage(body);
            // 手动确认，multiple=false表示只确认这条
            channel.basicAck(envelope.getDeliveryTag(), false);
        } catch (Exception e) {
            // 失败时拒绝消息并重新入队（可设置重试上限避免死循环）
            channel.basicNack(envelope.getDeliveryTag(), false, true);
        }
    }
});
```

### 4. 死信队列（Dead Letter Queue）

当消息处理失败（被Nack、拒绝且不重入队、或超过最大重试次数）时，消息进入死信队列：

```java
// 声明死信交换机和队列
Map<String, Object> args = new HashMap<>();
args.put("x-dead-letter-exchange", "dlx.exchange");
args.put("x-dead-letter-routing-key", "dlx.routing.key");
channel.queueDeclare("my-queue", true, false, false, args);
```

### 5. 事务模式（Transaction）

```java
// 开启事务（性能极差，不推荐用于生产）
channel.txSelect();
channel.basicPublish("exchange", "routing-key", MessageProperties.PERSISTENT_TEXT_PLAIN, body);
channel.txCommit(); // 提交失败则全部回滚
```

### 可靠消息最佳实践

1. **持久化配置三件套**：交换机+队列+消息全部持久化
2. **手动ACK**：务必在业务处理完成后才ACK
3. **幂等消费**：消费端做幂等处理（如数据库唯一索引），防止重复消费
4. **publisher confirms**：生产端开启确认，失败重发
5. **DLX处理**：无法正常消费的消息进入DLQ而不是丢弃

## 题目8：AI Agent | RAG检索增强生成核心原理与主流方案

**【答案】**

### 什么是RAG？

RAG（Retrieval-Augmented Generation，检索增强生成） = 检索技术 + LLM提示。

核心思想：**让LLM在回答问题之前，先从外部知识库中检索相关内容，再基于检索内容生成答案。**

解决了大模型的三大硬伤：
1. **知识截止** — 预训练后新发生的事件/政策模型不知道
2. **幻觉问题** — 模型会编造事实，RAG提供明确参考文本降低幻觉
3. **私有知识** — 企业内部文档、产品文档无法直接被模型访问

### RAG工作流程

**离线索引阶段：**
1. 文档切分（Chunking）：将长文档分成小段落，兼顾语义完整性和召回粒度
2. 向量化（Embedding）：用Embedding模型将文本片段映射为高维向量
3. 存储：向量存入向量数据库（如Milvus、Chroma、Qdrant）

**在线检索阶段：**
1. 用户提问 → 将问题也向量化
2. 相似度检索：在向量库中搜索Top-K最相关文档片段
3. 上下文增强：将检索片段+原始问题组织成Prompt
4. LLM生成：基于增强后的Prompt生成答案

### 关键技术：Embedding与向量相似度

**Embedding**：将文本映射为768/1024/1536/3072维的稠密向量，让语义相近的文本在向量空间中距离更近。

**相似度度量方式：**
- **余弦相似度（Cosine）**：衡量方向相似性，取值[-1,1]
- **欧氏距离（L2）**：衡量绝对距离
- **点积（Dot Product）**：计算效率高

### 主流RAG方案（从Naive到Agentic）

| 方案 | 特点 | 适用场景 |
|------|------|---------|
| **Naive RAG** | 最基础的"切块→搜索→生成"流程 | 入门级知识问答 |
| **Multi-Query RAG** | LLM改写用户问题为多种表述，扩大检索召回 | 用户表述口语化、术语差异大 |
| **HyDE** | 先让LLM生成假答案，用假答案的向量引导检索 | 用户问题很短、与文档语义空间差异大 |
| **Contextual RAG** | 检索前对每个chunk做语义增强/摘要 | 减少噪声、提升召回精度 |
| **Agentic RAG** | AI Agent自主决策检索策略，支持多步推理迭代 | 复杂问答、多跳推理场景 |

### RAG vs 长上下文 vs 微调

| 维度 | RAG | 长上下文 | 微调 |
|------|-----|--------|------|
| 知识更新 | 动态更新（更新向量库即可） | 需重新输入上下文 | 需重新训练 |
| 成本 | 低（按次检索计费） | 高（token按长度收费） | 高（训练成本高） |
| "Lost in the Middle"问题 | 不存在（精确召回） | 存在（中间注意力下降） | 不存在 |
| 幻觉控制 | 通过引用溯源降低 | 无 | 通过风格学习降低 |

## 题目9：算法 | 二分查找变体——旋转排序数组搜索

**【答案】**

### 题目描述

假设一个排序数组在某个位置被旋转（如[0,1,2,4,5,6,7]变成[4,5,6,7,0,1,2]），给定一个目标值，在数组中搜索目标位置，如果不存在返回-1。要求时间复杂度O(log n)。

### 思路分析

旋转排序数组中，一半总是有序的。可以利用这个特性，每次缩小搜索范围时，总有一半是有序区间，可以判断目标是否在有序区间内。

### Java实现

```java
public int search(int[] nums, int target) {
    if (nums == null || nums.length == 0) return -1;
    
    int left = 0, right = nums.length - 1;
    
    while (left <= right) {
        int mid = left + (right - left) / 2;
        
        if (nums[mid] == target) {
            return mid;
        }
        
        // 判断哪一半是有序的
        if (nums[left] <= nums[mid]) {
            // 左半边是有序的
            if (target >= nums[left] && target < nums[mid]) {
                // 目标在左半边
                right = mid - 1;
            } else {
                // 目标在右半边
                left = mid + 1;
            }
        } else {
            // 右半边是有序的
            if (target > nums[mid] && target <= nums[right]) {
                // 目标在右半边
                left = mid + 1;
            } else {
                // 目标在左半边
                right = mid - 1;
            }
        }
    }
    
    return -1;
}
```

### 代码图解

```
例：nums = [4,5,6,7,0,1,2]，target = 0

第1轮：left=0, right=6, mid=3
       nums[mid]=7 > nums[left]=4，左半边有序
       目标0不在[4,5,6,7]之间 → left=4

第2轮：left=4, right=6, mid=5
       nums[mid]=1 > nums[left]=0？不对，左半边无序，右半边有序
       目标0在[0,1,2]之间 → left=4

第3轮：left=4, right=4, mid=4
       nums[mid]=0 == target → 返回4
```

### 关键点

1. **如何判断有序区间**：`nums[left] <= nums[mid]`说明左半边有序
2. **如何判断目标位置**：在有序区间内则收缩对应边界，否则收缩到另一侧
3. **边界条件**：用`left <= right`循环条件，处理单个元素情况

### 相关变体

- 寻找旋转数组的最小值（无重复）
- 寻找旋转数组的最小值II（允许重复，需要处理边界）
- 在旋转排序数组中找最大值

## 题目10：算法 | 滑动窗口算法——找到字符串中所有变位词

**【答案】**

### 题目描述

给定字符串s和p，找出s中所有p的变位词（字母异位词）所在的起始索引。返回所有满足条件的起始索引列表。

**示例：** s="cbaebabacd", p="abc" → 返回 [0,6]（因为"cba"和"bac"都是"abc"的变位词）

### 思路分析

**滑动窗口 + 字符计数数组**

变位词的本质是字母组成相同、排列不同。可以用一个长度为26的计数数组记录窗口内各字符的数量，与p的字符计数对比。

### Java实现

```java
public List<Integer> findAnagrams(String s, String p) {
    List<Integer> result = new ArrayList<>();
    if (s == null || p == null || s.length() < p.length()) {
        return result;
    }
    
    int[] pCount = new int[26];  // p的字符计数
    int[] window = new int[26];  // 滑动窗口的字符计数
    
    // 初始化p的计数和第一个窗口的计数
    for (int i = 0; i < p.length(); i++) {
        pCount[p.charAt(i) - 'a']++;
        window[s.charAt(i) - 'a']++;
    }
    
    // 比较第一个窗口
    if (matches(window, pCount)) {
        result.add(0);
    }
    
    // 滑动窗口
    for (int i = p.length(); i < s.length(); i++) {
        // 移入新字符
        window[s.charAt(i) - 'a']++;
        // 移出旧字符
        window[s.charAt(i - p.length()) - 'a']--;
        
        // 判断是否匹配
        if (matches(window, pCount)) {
            result.add(i - p.length() + 1);
        }
    }
    
    return result;
}

private boolean matches(int[] a, int[] b) {
    for (int i = 0; i < 26; i++) {
        if (a[i] != b[i]) return false;
    }
    return true;
}
```

### 代码图解

```
s="cbaebabacd", p="abc"

初始计数：pCount = [1,1,1,0,...]（a:1,b:1,c:1）
         window("cba") = [1,1,1,0,...] → 匹配！→ 添加0

滑动窗口 i=3（窗口"bae"）：
         移入'e'，移出'c'
         window("bae") = [0,1,1,0,...] → 不匹配

滑动窗口 i=4（窗口"abe"）：
         window("abe") = [1,1,1,0,...] → 匹配！→ 添加1

...
继续滑动，最终在i=6时窗口"bac"匹配 → 添加6
最终结果：[0, 6]
```

### 复杂度分析

- **时间复杂度：** O(n)，n为s的长度，每个字符最多被访问两次（一次移入一次移出）
- **空间复杂度：** O(1)，字符计数数组大小固定为26，与输入规模无关

### 滑动窗口通用模板

// 1. 定义窗口
int left = 0, right = 0;
// 2. 扩大窗口
while (right < s.length()) {
    add(s.charAt(right));
    right++;
    // 3. 收缩窗口（当满足条件时）
    while (condition) {
        remove(s.charAt(left));
        left++;
    }
}
