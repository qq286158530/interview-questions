## 题目一：Redis 有哪些数据类型？各自的使用场景是什么？

### 参考答案

Redis 提供了丰富的数据类型，常见的有五种：**String（字符串）、Hash（哈希）、List（列表）、Set（集合）、Zset（有序集合）**。

随着版本更新，又支持了四种扩展类型：**BitMap、HyperLogLog、GEO、Stream**。

**五种基础数据类型的应用场景：**

| 类型 | 典型应用场景 |
|------|-------------|
| String | 缓存对象、常规计数、分布式锁、共享 session 信息 |
| List | 消息队列（需注意无法以消费组形式消费）、文章列表、任务队列 |
| Hash | 缓存对象、购物车、用户画像 |
| Set | 聚合计算（并集/交集/差集）：点赞、共同关注、抽奖 |
| Zset | 排序场景：排行榜、电话姓名排序 |

**四种扩展数据类型的使用场景：**

- **BitMap（2.2+）**：二值状态统计，如签到、判断用户登录状态、连续签到总数
- **HyperLogLog（2.8+）**：海量数据基数统计，如百万级网页 UV 计数
- **GEO（3.2+）**：存储地理位置信息，如滴滴叫车
- **Stream（5.0+）**：消息队列，相比 List 支持消费组形式消费、自动生成全局唯一消息 ID

**底层实现简述：**

- String → SDS（简单动态字符串），支持二进制安全、O(1) 长度获取
- List → Redis 3.2 后统一为 quicklist（双向链表 + 压缩列表）
- Hash → 压缩列表（元素少时）或哈希表
- Set → 整数集合（纯整数+元素少时）或哈希表
- Zset → 压缩列表（元素少时）或跳表（Skip List）+ 哈希表

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 参考答案

Redis 共有三种持久化方式：**RDB 快照、AOF 日志、混合持久化**。

### RDB（Redis Database）快照

RDB 快照是记录某一时刻的内存数据，以二进制方式写入磁盘。

- **save 命令**：在主线程执行，会阻塞所有命令（生产环境避免使用）
- **bgsave 命令**：fork 子进程来生成 RDB 文件，主线程不受影响
- Redis 默认配置满足以下任一条件自动触发 bgsave：
  - 900 秒内至少 1 次修改
  - 300 秒内至少 10 次修改
  - 60 秒内至少 10000 次修改

**核心原理——写时复制（COW）：**
fork 时父子进程共享物理内存；主线程执行写操作时，才会复制出独立副本写入 RDB 文件。

**优点**：恢复速度快（全量二进制，加载即可使用）
**缺点**：快照频率不好把握，频率太低会丢失较多数据

### AOF（Append Only File）日志

每执行一条写操作命令，就将该命令追加写入 AOF 文件。Redis 重启时，逐一执行 AOF 中的命令来恢复数据。

**写回策略（appendfsync）：**

| 策略 | 行为 | 安全性 | 性能 |
|------|------|--------|------|
| Always | 每条命令执行完即同步写盘 | 最安全 | 最差 |
| Everysec（默认） | 每秒一次同步 | 较好，可容忍 1 秒数据丢失 | 较好 |
| No | 由操作系统决定何时刷盘 | 不安全 | 最好 |

**AOF 重写机制**：
当 AOF 文件过大时，后台子进程 `bgrewriteaof` 会读取数据库中所有键值对，用一条命令记录到新的 AOF 文件，从而压缩文件体积。重写期间主进程的新命令同时写入 "AOF 缓冲区" 和 "AOF 重写缓冲区"，重写完成后合并到新文件。

**优点**：数据丢失少
**缺点**：恢复速度慢（需要逐一执行命令），文件体积可能较大

### 混合持久化（Redis 4.0+）

开启混合持久化后，AOF 重写时先将以 RDB 格式写入文件前半部分，再将增量命令以 AOF 格式写入后半部分。

- **优点**：结合了 RDB 快速恢复和 AOF 减少数据丢失的优点
- **缺点**：AOF 文件可读性差，不兼容 Redis 4.0 之前版本

### 如何选择？

- **仅缓存场景**：可不开持久化
- **能容忍少量数据丢失**：RDB 即可
- **对数据安全性要求高**：AOF + everysec，或直接开启混合持久化
- **最佳实践**：RDB + AOF 混合使用，兼顾恢复速度和安全性

## 题目三：Redis 是单线程的吗？为什么单线程的 Redis 还能这么快？

### 参考答案

**Redis 单线程指的是什么？**

Redis "单线程"是指 **接收客户端请求 → 解析请求 → 执行数据读写 → 返回响应** 这一完整流程由主线程（main thread）完成。

但 Redis 程序本身并非纯单线程，启动时会启动多个后台 BIO 线程：
- **bio_close_file**：异步关闭文件
- **bio_aof_fsync**：异步 AOF 刷盘
- **bio_lazy_free（4.0+）**：异步释放内存（如 unlink、flushdb async）

这些后台线程负责耗时任务，避免阻塞主线程。

**Redis 6.0 之后引入了 I/O 多线程**：
用于分担网络 I/O 的压力（主要是发送响应数据），但**命令执行仍是单线程**。

### Redis 单线程为什么快？

1. **瓶颈不在 CPU**：Redis 数据在内存中操作，采用高效数据结构（如 SDS、跳表），瓶颈往往是内存或网络带宽，而非 CPU
2. **无竞争开销**：单线程避免了多线程间的上下文切换和锁竞争，不存在死锁问题
3. **I/O 多路复用**：基于 epoll/select/kqueue机制，一个线程处理多个 Socket 请求，效率极高

### 单线程的实现原理（事件循环）

```
初始化 → 创建 epoll 实例、监听 socket
    ↓
事件循环：
  ① 处理发送队列（write 系统调用）
  ② epoll_wait 等待事件
  ③ 连接事件 → accept → 注册读事件
  ④ 读事件 → read → 解析命令 → 执行 → 写入发送队列
  ⑤ 写事件 → write 发送响应
```

## 题目四：Redis 的过期删除策略和内存淘汰策略有什么区别？如何选择？

### 参考答案

这是两个不同层面的机制，需要区分清楚。

### 过期删除策略（主动清理过期 key）

Redis 采用 **「惰性删除 + 定期删除」** 配合使用：

**1. 惰性删除（passive way）**
- 每次访问 key 时，才检查是否过期，过期则删除
- **优点**：对 CPU 友好，几乎不消耗额外资源
- **缺点**：对内存不友好，过期 key 长期占用内存

**2. 定期删除（active way）**
- 每隔一段时间，随机抽取 20 个带过期时间的 key
- 删除已过期的 key，若过期比例 > 25% 则继续抽取
- 循环时间上限默认 25ms，避免阻塞主线程

### 内存淘汰策略（内存满时如何处理）

当 Redis 达到 `maxmemory` 上限时，触发内存淘汰，共八种策略：

**不淘汰（1 种）：**
- `noeviction`：拒绝所有写请求，返回错误（Redis 3.0+ 默认）

**在带过期时间的 key 中淘汰（4 种）：**

| 策略 | 行为 |
|------|------|
| volatile-ttl | 优先淘汰最早过期的 key |
| volatile-random | 随机淘汰任意过期 key |
| volatile-lru | 淘汰最久未使用的（LRU） |
| volatile-lfu | 淘汰使用频率最低的（LFU，4.0+） |

**在所有 key 中淘汰（3 种）：**

| 策略 | 行为 |
|------|------|
| allkeys-random | 随机淘汰任意 key |
| allkeys-lru | 淘汰全局最久未使用的 |
| allkeys-lfu | 淘汰全局使用频率最低的（4.0+） |

### 如何选择？

- **热点数据想保留** → volatile-lru / allkeys-lru
- **访问频率差异明显** → volatile-lfu / allkeys-lfu
- **所有 key 重要性相同** → allkeys-random / allkeys-lru
- **不想淘汰任何数据** → noeviction
- **有明确的冷数据** → volatile-ttl

## 题目五：什么是缓存雪崩、缓存击穿、缓存穿透？如何应对？

### 参考答案

这三个问题是 Redis 缓存开发中最常见的高频面试题。

### 缓存雪崩

**定义**：大量缓存数据在同一时间过期，导致大量请求直接打到数据库，数据库压力骤增甚至宕机。

**解决方案：**
- **失效时间随机化**：在原有过期时间基础上 + 随机值（1~10 分钟），避免集体失效
- **缓存不过期**：通过后台服务异步更新缓存数据
- **多级缓存**：建立多级缓存架构（本地缓存 + Redis），降低对 Redis 的直接压力

### 缓存击穿

**定义**：某个热点数据过期，此时大量请求并发访问该数据，穿透到数据库。

（缓存雪崩是大量数据，缓存击穿是某一个热点数据——可认为是雪崩的子集）

**解决方案：**
- **互斥锁**：使用 `setNX` 保证同一时间只有一个线程去 DB 查询并更新缓存
- **热点数据永不过期**：由后台异步更新，或临近过期前主动刷新

### 缓存穿透

**定义**：查询的数据既不在缓存中，也不在数据库中（如恶意请求不存在的数据），导致所有请求都打到数据库。

**常见原因**：
- 业务误操作删除了缓存和数据库
- 黑客故意大量请求不存在的数据

**解决方案：**

1. **参数校验**：API 入口处拦截非法请求（如 id < 0）
2. **缓存空值/默认值**：数据库查不到的数据也在 Redis 中缓存一个空值或默认值
3. **布隆过滤器（Bloom Filter）**：写入数据库时同步在布隆过滤器标记，请求进来时先查布隆过滤器，快速判断数据是否存在，避免无效数据库查询

### 三者对比总结

| 问题 | 原因 | 核心思路 |
|------|------|---------|
| 缓存雪崩 | 大量 key 同时过期 | 随机过期时间、多级缓存 |
| 缓存击穿 | 热点 key 过期 | 互斥锁、永不过期 |
| 缓存穿透 | 数据根本不存在 | 布隆过滤器、缓存空值 |
