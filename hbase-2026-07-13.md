## 题目一：HBase 的整体架构是怎样的？请详细说明各个组件的作用。

### 参考答案

HBase 是一个基于 HDFS 的分布式列式存储数据库，其架构采用 Master-Slave 模式，主要包含以下组件：

**1. HMaster（主节点）**
- 负责表的创建、删除、修改等 DDL 操作
- 管理 Region 的分配和负载均衡
- 监控 RegionServer 的健康状态
- 当 RegionServer 故障时，进行故障恢复和 Region 迁移
- 处理元数据变更请求

**2. RegionServer（从节点/数据节点）**
- 负责处理数据的读写请求
- 管理 MemStore 和 HFile
- 执行 Compaction 操作
- 一个 RegionServer 可以管理多个 Region（通常 100-1000 个）
- 直接与 HDFS DataNode 交互存储数据

**3. ZooKeeper**
- 负责 HBase 集群的协调服务
- 维护集群拓扑信息和 RegionServer 的健康状态
- 实现 Master 的高可用选举
- 存储首次寻址（first lookup）信息
- 检测 RegionServer 的上下线

**4. Region（区域）**
- 是 HBase 表的水平分片单位
- 按 RowKey 的字典序进行划分
- 每个 Region 包含多个列族的数据
- 默认大小约 10-20GB，可自动分裂

**5. WAL（Write Ahead Log）**
- 预写日志，用于数据写入 MemStore 前的持久化
- 保证数据可靠性，宕机后可从 WAL 恢复

**6. MemStore**
- 写缓存，位于 RegionServer 内存中
- 数据先写入 MemStore，达到阈值后 flush 成 HFile
- 按列族组织，每个列族有独立的 MemStore

**7. HFile**
- HBase 的实际存储文件格式
- 基于 HDFS 的有序键值对存储
- 包含数据块、索引块、Bloom Filter 等

**数据写入流程：**
1. 客户端请求 ZooKeeper 获取 meta 表位置
2. 连接 RegionServer，请求 Region 位置
3. 数据写入 WAL，然后写入 MemStore
4. MemStore 达到阈值后 flush 成 HFile
5. HFile 在 HDFS 上进行 Compaction 合并

**数据读取流程：**
1. 通过 ZooKeeper 和 meta 表定位目标 Region
2. 先查 BlockCache（读缓存），未命中则查 MemStore
3. 若仍未命中，扫描 HFile 并利用 Bloom Filter 加速
4. 合并各数据源返回最终结果

## 题目二：HBase 的 RowKey 如何设计？有哪些最佳实践和禁忌？

### 参考答案

RowKey 是 HBase 中最重要的设计要素，直接影响数据分布、读写性能和负载均衡。

### RowKey 设计原则

**1. 唯一性原则**
RowKey 必须唯一标识每一行数据，通常采用业务主键或复合键。

**2. 散列性原则（避免热点）**
热点问题是 HBase 最常见的性能杀手。设计时应避免：
- ❌ 顺序递增的 RowKey（如时间戳 20260713001, 20260713002...）
- ❌ 相同前缀的字符串
- ✅ 使用 MD5、SHA1 哈希打散
- ✅ 盐值（Salt）前缀：随机数 + 原始 key
- ✅ 哈希反转：reverse(timestamp)

**3. 长度适中**
- RowKey 长度建议控制在 16-100 字节
- 越短越好，减少内存和存储开销
- 避免使用过长或无意义的字段

**4. 业务相关性**
RowKey 设计应贴近业务查询模式：
- 如果经常按用户ID查询：RowKey = userId
- 如果经常按时间范围查询：RowKey = 时间戳 + 业务ID
- 组合查询场景：RowKey = 字段1_字段2_字段3

### 常见 RowKey 设计模式

**模式一：用户中心型**
```
RowKey = userId + timestamp + actionType
适用于：用户操作记录、聊天记录、订单查询
```

**模式二：盐值打散型**
```
RowKey = (hash(userId) % N) + userId + timestamp
适用于：高并发写入场景，避免 Region 热点
```

**模式三：维度组合型**
```
RowKey = areaId + shopId + orderId
适用于：多条件组合查询
```

**模式四：倒排索引型**
```
RowKey = tagValue + entityId
适用于：按标签查询实体的场景
```

### RowKey 设计禁忌

| 禁忌 | 原因 | 解决方案 |
|------|------|----------|
| 使用自增 ID | 导致数据写入集中在同一 Region | 哈希打散/盐值前缀 |
| RowKey 过长 | 增加存储开销，影响性能 | 缩短或压缩 |
| 相同前缀过多 | 造成 Region 热点 | 哈希处理 |
| 二进制非打印字符 | 影响调试和可读性 | Base64 编码 |

### 实战建议

1. **写入热点**：使用随机散列或预分区策略
2. **读取热点**：使用缓存 + 热点数据复制
3. **复合 RowKey**：将查询条件放在前面，支持范围扫描
4. **版本控制**：一个 RowKey 可存储多个版本（时间戳）

## 题目三：HBase 的存储结构是怎样的？请详细说明 MemStore 和 HFile 的工作原理。

### 参考答案

HBase 采用 LSM-Tree（Log-Structured Merge-Tree）存储结构，结合内存和磁盘实现高效的随机写和顺序读。

### HBase 存储层次

```
MemStore (内存) → HFile (磁盘) → HDFS
       ↓
    Compaction
       ↓
   合并 HFile
```

### MemStore（内存组件）

**作用：** 写缓存，暂存新写入的数据

**特点：**
- 位于 RegionServer 堆内存中
- 每个列族对应一个 MemStore 实例
- 数据按 RowKey 和 Column 有序排列（内存中排序）
- 达到阈值（默认 128MB）后 flush 成 HFile

**Flush 触发条件（满足任一即可）：**
1. MemStore 大小超过 `hbase.hregion.memstore.flush.size`（默认 128MB）
2. RegionServer 堆内存使用率超过阈值
3. WAL 文件数量超过限制
4. 时间戳超过 `hbase.regionserver.optionalcacheflushinterval`（默认 1 小时）

**Flush 流程：**
1. 将 MemStore 数据写出为 HFile
2. 新 HFile 目录结构：`/hbase/data/default/表名/regionID/列族名/`
3. 更新 StoreInfo，标记新的 HFile
4. 可选：删除对应的 WAL 日志

### HFile（磁盘组件）

**作用：** HBase 的实际存储文件格式，基于 Hadoop 的有序键值存储。

**HFile 结构：**

```
┌─────────────────────────────────┐
│         File Info              │  文件元信息
├─────────────────────────────────┤
│        Meta Index              │  块索引（定位数据块）
├─────────────────────────────────┤
│         Data Index              │  数据块索引
├─────────────────────────────────┤
│         Data Blocks             │  数据块（存储 KeyValue）
│    ┌─────┬─────┬─────┬─────┐   │
│    │ K/V │ K/V │ K/V │ K/V │   │
│    └─────┴─────┴─────┴─────┘   │
├─────────────────────────────────┤
│        Bloom Filter            │  布隆过滤器
├─────────────────────────────────┤
│        Trailer                 │  尾部信息（偏移量等）
└─────────────────────────────────┘
```

**KeyValue 结构：**
```
[KeyLength (4B)] [ValueLength (4B)] [RowKey] [ColumnFamily] [Qualifier] [Timestamp] [Type] [Value]
```

**HFile 核心特性：**
1. **数据块大小**：默认 64KB，可按列族配置
2. **Bloom Filter**：快速判断 Key 是否存在于文件中
3. **块索引**：通过起始 RowKey 定位数据块
4. **时间范围索引（TSI）**：快速过滤时间范围外的版本

### Compaction（合并压缩）

**Minor Compaction（小合并）：**
- 合并少量 HFile（默认 3-10 个）
- 删除过期数据和 tombstone 标记
- 不删除已删除的数据

**Major Compaction（大合并）：**
- 合并一个列族下的所有 HFile
- 删除所有 tombstone 标记（真正删除数据）
- 合并后只保留最新版本
- **注意**：大合并会产生大量 IO，建议在业务低峰期执行

**合并触发条件：**
- HFile 数量超过阈值（默认 3）
- HFile 大小超过配置
- 时间触发

### 数据读取流程

```
1. 先查 BlockCache（读缓存）
2. 未命中 → 查 MemStore
3. 若仍未命中 → 扫描 HFile
4. 利用 Bloom Filter 跳过不包含 key 的 HFile
5. 合并所有来源的 Value（按时间戳排序）
```

### 性能调优参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| hbase.hregion.memstore.flush.size | 128MB | MemStore flush 阈值 |
| hfile.block.cache.size | 0.4 | BlockCache 占比 |
| hbase.hstore.compaction.min | 3 | Minor Compaction HFile 数 |
| hbase.hstore.compaction.max | 10 | Minor Compaction 最大 HFile 数 |

## 题目四：HBase 的协处理器（Coprocessor）是什么？有哪几种类型？各自的使用场景是什么？

### 参考答案

协处理器是 HBase 提供的扩展机制，允许用户在 RegionServer 上运行自定义代码，实现对数据的预处理和后处理。

### 协处理器起源

协处理器的概念最初来源于 Google BigTable 的实现，后在 HBase 中得到实现和增强。

### 协处理器类型

#### 1. Observer（观察者）

**原理：** 类似于数据库触发器，在特定事件前后自动执行回调方法。

**三种 Observer：**

| 类型 | 触发时机 | 应用场景 |
|------|----------|----------|
| RegionObserver | Get/Put/Delete/Scan 等数据操作前后 | 数据验证、访问控制、列值修改 |
| MasterObserver | CreateTable/DeleteTable/ModifyTable 等元数据操作 | 表结构审计、权限控制 |
| WALObserver | WAL 写入前后 | WAL 复制、审计 |

**核心方法：**
```java
// RegionObserver 示例
public class MyObserver extends BaseRegionObserver {
    @Override
    public void preGet(ObserverContext<RegionCoprocessorEnvironment> e, 
                       Get get, List<Cell> result) throws IOException {
        // 在 GET 之前执行的逻辑
        System.out.println("PreGet: " + get.getRow());
    }
    
    @Override
    public void postGet(ObserverContext<RegionCoprocessorEnvironment> e, 
                        Get get, List<Cell> result) {
        // 在 GET 之后执行的逻辑
    }
}
```

**典型应用场景：**
- **二级索引**：当数据写入时，自动更新索引表
- **数据验证**：写入前校验数据格式和业务规则
- **列值自动计算**：如实时计算 SUM、COUNT 等聚合值
- **访问控制**：基于用户权限过滤列或行
- **审计日志**：记录所有数据访问操作

#### 2. Endpoint（端点）

**原理：** 类似于存储过程，允许在 RegionServer 上执行分布式计算。

**工作方式：**
1. 用户定义一个继承自 CoprocessorProtocol 的接口
2. 在 RegionServer 上实现该接口的方法
3. 客户端通过 HTable.coprocessorExec() 调用
4. 框架在每个相关 Region 上执行并汇总结果

**示例场景：**
```java
// 定义协议接口
public interface AggregateProtocol extends CoprocessorProtocol {
    long sum(byte[] columnName) throws IOException;
    long count() throws IOException;
}

// 服务端实现
public class AggregateImpl implements AggregateProtocol {
    public long sum(byte[] columnName) {
        // 在每个 Region 上执行求和逻辑
    }
}
```

**典型应用场景：**
- **聚合计算**：求和、计数、平均值、最大最小值
- **批量数据处理**：批量转换、清洗数据
- **自定义过滤**：复杂条件的数据筛选

### 协处理器加载方式

**1. 通过 HBase 配置（系统级）：**
```xml
<property>
    <name>hbase.coprocessor.user.region.classes</name>
    <value>com.example.MyObserver|com.example.MyEndpoint</value>
</property>
```

**2. 通过 Shell（表级）：**
```hbase
disable 'myTable'
alter 'myTable', METHOD => 'table_att', 
    'COPROCESSOR' => 'hdfs://path/to/coprocessor.jar|com.example.MyObserver|1001'
enable 'myTable'
```

### 协处理器注意事项

| 事项 | 说明 |
|------|------|
| 线程安全 | Observer 多线程并发调用，需保证线程安全 |
| 异常处理 | 避免在 Observer 中抛出异常，会导致操作失败 |
| 性能影响 | 复杂逻辑会影响读写性能，建议异步处理 |
| 版本兼容 | 不同 HBase 版本 API 可能有变化 |
| 资源竞争 | 避免在 Observer 中执行耗时操作 |

### 实战案例：构建二级索引

使用协处理器实现订单表到客户表的二级索引：

```java
// 订单写入时自动更新客户订单列表
public class OrderIndexObserver extends BaseRegionObserver {
    @Override
    public void postPut(ObserverContext<RegionCoprocessorEnvironment> e, 
                        Put put, WALEntry walEntry) {
        // 提取订单中的 customerId
        // 向客户索引表写入索引记录
    }
}
```

## 题目五：HBase 如何实现高可用（HA）？RegionServer 故障时如何进行故障恢复？

### 参考答案

HBase 的高可用机制涉及多个层面，确保在单点故障时集群仍能正常运行。

### HBase 高可用架构

```
┌─────────────────────────────────────────────────────┐
│                    Zookeeper                        │
│         (集群协调 + Master 选举 + 元数据)            │
└─────────────────────────────────────────────────────┘
         ↑                    ↑                    ↑
    ┌────────┐           ┌────────┐           ┌────────┐
    │Master-1│           │Master-2│           │Master-N│
    │(Active)│  ←选举→   │(Standby)           │(Standby)│
    └────────┘           └────────┘           └────────┘
         ↑                    ↑                    ↑
    ┌────────┐           ┌────────┐           ┌────────┐
    │ RS-1   │           │ RS-2   │           │ RS-N   │
    │Region1 │           │Region2 │           │Region3 │
    │Region4 │           │Region5 │           │Region6 │
    └────────┘           └────────┘           └────────┘
```

### 1. Master 高可用

**实现方式：**
- ZooKeeper 临时节点选举
- 同一时刻只有一个 Active Master
- 其他 Master 作为 Standby 监控状态
- Active Master 故障后，ZooKeeper 自动触发选举

**故障转移过程：**
1. ZooKeeper 检测到 Active Master 心跳超时
2. 剩余 Master 竞争创建临时节点
3. 胜出者成为新的 Active Master
4. 新 Master 重新分配 Region，修复元数据

### 2. RegionServer 高可用

**核心机制：HLog（WAL）故障恢复**

**故障检测：**
- ZooKeeper 定期接收 RegionServer 心跳
- 超时未响应则判定为故障
- Master 收到故障通知，开始恢复流程

**恢复流程：**

```
1. Master 检测到 RegionServer 宕机
       ↓
2. 标记该 RS 上的所有 Region 为 OFFLINE
       ↓
3. 将宕机 RS 上的 WAL 文件分割（Log splitting）
       ↓
4. 根据 WAL 中的数据，按 Region 分组
       ↓
5. 将恢复的日志数据回放（Replay）到对应 Region
       ↓
6. 恢复完成后，Region 重新上线
```

**WAL 恢复细节：**
- WAL 文件存储在 HDFS `/hbase/WALs/` 目录
- 每个 RegionServer 有独立的 WAL 目录
- 日志分割由 Master 的 LogSplitting 线程完成
- 分割后的日志以 `oldWALFileName + RegionName` 命名
- RegionServer 恢复时读取对应日志回放数据

### 3. Region 复制（Region Replication）

**作用：** 一个 Region 的多个副本分布在不同的 RegionServer 上。

**配置：**
```xml
<property>
    <name>hbase.region.replication.replica.count</name>
    <value>3</value>
</property>
```

**副本类型：**
- **Primary Replica**：处理所有写操作
- **Secondary Replicas**：处理读操作，提供高可用读取

**故障转移：**
- Primary 故障时，Secondary 自动升级
- 通过 ZooKeeper 协调完成切换
- 切换时间通常在秒级

### 4. 数据高可用（底层 HDFS）

HBase 数据存储在 HDFS 上，天然具备以下特性：
- **数据块副本**：默认 3 副本存储
- **机架感知**：副本分布在不同机架
- **自动恢复**：DataNode 故障时自动复制数据

### 5. ZooKeeper 高可用

- ZooKeeper 本身是集群模式（奇数节点）
- 通常部署 3 或 5 个节点
- 容忍 N/2 节点故障

### 高可用配置建议

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| hbase.master.lease.task.delay | 300000 | Master 租约超时 |
| zookeeper.session.timeout | 180000 | ZK 会话超时 |
| hbase.regionserver.failfast.enabled | true | 故障快速检测 |
| hbase.region.replica.replica.count | 3 | Region 副本数 |
| dfs.replication | 3 | HDFS 数据副本数 |

### 常见故障场景及处理

| 场景 | 处理方式 |
|------|----------|
| RegionServer 宕机 | WAL 回放 + Region 迁移 |
| Master 宕机 | ZooKeeper 选举新 Master |
| ZooKeeper 故障 | 确保奇数节点，继续服务 |
| HDFS 故障 | HDFS 自动恢复副本 |
| 网络分区 | ZooKeeper 脑裂保护 |
