## 题目一：RDD 为什么是惰性求值？Lazy Evaluation 机制详解

### 📖 问题描述

Spark 中 RDD 的转换（Transformation）操作是惰性求值的，为什么要这样设计？这样做有什么好处？请结合源码或执行流程说明。

---

### ✅ 答案

**1. 什么是惰性求值（Lazy Evaluation）？**

RDD 的Transformation操作（如 `map`、`filter`、`flatMap`）不会立即执行，只是记录下逻辑转换图（Lineage），只有遇到 Action 操作（如 `collect`、`count`、`saveAsTextFile`）时才会真正触发计算。

**2. 惰性求值的工作原理**

```
RDD Lineage（血统链）示例：

rdd1 = sc.textFile("hdfs://...")          // 记录：HadoopRDD
rdd2 = rdd1.map(_.toUpperCase)              // 记录：MapPartitionsRDD
rdd3 = rdd2.filter(_.contains("ERROR"))     // 记录：MapPartitionsRDD
rdd4 = rdd3.collect()                       // ★ Action 触发计算
```

DAGScheduler 将整条 Transformation 链合成一个 Stage 拓扑图，从末尾的 Action 向前回溯，遇到 Shuffle 就划分新的 Stage。

**3. 惰性求值的好处**

| 好处 | 说明 |
|------|------|
| **减少中间结果落地** | 无需为每个中间转换步骤都执行一次计算和存储 |
| **统一优化机会** | Spark 在提交 Job 前可以对整个 Transformation 链进行优化（如 pipeline 化、去掉冗余步骤） |
| **减少磁盘 I/O** | 只在必要时才真正执行，避免不必要的计算 |
| **天然支持容错** | RDD 通过 Lineage 记录血统，任何节点失败只需从上游重新计算对应 Partition，不需要完整的 Checkpoint |

**4. 与传统 eagerness 的对比**

传统 eager 语义：每一步都立即执行 → 产生大量中间临时文件 → 无法做全局优化。

Spark lazy 语义：构建有向无环图（DAG）→ Action 时一次性优化执行 → 效率更高。

## 题目二：Spark Shuffle 机制原理 —— Hash Shuffle vs Sort Shuffle

### 📖 问题描述

Spark 中的 Shuffle 是性能瓶颈之一。请详细解释 Shuffle 的执行过程，以及 Hash Shuffle 和 Sort Shuffle 的区别演进。

---

### ✅ 答案

**1. Shuffle 的本质**

Shuffle 是将数据按照 Key 进行重新分区的过程，通常发生在 `groupByKey`、`join`、`reduceByKey` 等需要跨节点重分布数据的算子中。本质是 **Map 端写磁盘 + 网络跨节点传输 + Reduce 端读磁盘**。

**2. Hash Shuffle（Spark 1.x 旧方案）**

```
[Map Task 0] ──key A──→ File 0 (A) ──┐
[Map Task 1] ──key B──→ File 0 (B) ──┼──→ [Reduce Task 0] (接收A/B)
[Map Task 2] ──key A──→ File 1 (A) ──┤
                                      └──→ [Reduce Task 1] (接收C)
[Map Task 0] ──key C──→ File 1 (C) ──┘
```

**问题：**
- 每个 Mapper 为每个 Reducer 创建一个文件，`M × R` 个文件数 → 引发大量小文件随机写
- 文件系统压力巨大（大量 open/close 操作）
- 内存占用高，容易触发 OOM

**优化版本（Consolidation）：**
多个 Map Task 共享同一个输出文件的不同的 Bucket，减少文件数：`文件数 = C × R`（C = CPU Core 数或指定 partition 数）

**3. Sort Shuffle（Spark 1.1+ 引入，1.2+ 成为默认）**

```
[Map Task]
  ├─ 按 partitionId 排序写入 → 单个文件（按 partition 分段）
  └─ 附带索引文件 (.index) 记录每段偏移量

[Reduce Task]
  └─ 根据 index 文件直接 seek 到对应段读取
```

**Sort Shuffle 的核心优势：**
- 每个 Map Task 只输出 **2 个文件**（数据文件 + 索引文件），彻底消除小文件问题
- 数据内部按 partitionId 排序，Reduce 端读取有序，效率更高
- 支持 **Bypass 模式**：当 Shuffle Read Partition 数 ≤ 200 时，绕过 Sort 直接输出（类似 Hash 快速路径）

**4. 关键参数**

```scala
// 控制是否启用 Sort Shuffle（1.2+ 默认 true）
spark.shuffle.manager = sort

// Sort Shuffle Bypass 阈值
spark.shuffle.sort.bypassMergeThreshold = 200

// 是否开启 shuffle file consolidation（针对 Hash Shuffle）
spark.shuffle.consolidateFiles = true  // 在不使用 Sort Shuffle 时可开启
```

## 题目三：Spark 内存管理 —— Unified Memory（统一内存管理）机制

### 📖 问题描述

Spark 1.6+ 引入了 Unified Memory（统一内存管理）模型，相比旧的 Static Memory（静态内存管理）有哪些改进？请详细说明内存区域的划分、动态抢占机制，以及实际调参时的注意事项。

---

### ✅ 答案

**1. 两种内存模型对比**

| 特性 | Static Memory（1.5 及之前） | Unified Memory（1.6+） |
|------|------|------|
| 划分方式 | 固定比例，Storage 与 Execution 严格隔离 | Storage 与 Execution 可动态共享 |
| 内存利用率 | 低，资源浪费 | 高，按需弹性分配 |
| 配置复杂度 | 高，需要手动平衡 | 中等，有默认值即可 |

**2. Unified Memory 区域划分**

```
总堆内存 = spark.memory.fraction × (Java Heap - 300MB)
  ├─ Storage Memory（用户：RDD Cache + Broadcast）
  │    └─ 可借用空间：Execution 空闲部分
  └─ Execution Memory（用户：Shuffle 排序/哈希/join 缓冲）
       └─ 可借用空间：Storage 空闲部分
```

**关键公式：**
```
Storage 可用 = 初始 Storage 区域 + max(0, Execution 已用 - 初始 Execution 区域)
Execution 可用 = 初始 Execution 区域 + max(0, Storage 已用 - 初始 Storage 区域)
```

**3. 动态抢占机制（Memory Eviction）**

当 Execution 内存不足，需要向 Storage 借空间时：
- 如果 Storage 有空闲 → 直接借用
- 如果 Storage 空间紧张 → **强行驱逐 Storage 中低优先级的缓存块**（按 LRU 规则）

反过来，Storage 需回收时：
- 如果 Execution 有空闲 → 直接回收
- 如果 Execution 空间紧张 → **等待 Execution 释放**（不会强行驱逐，因为 Execution 任务正在运行）

**4. off-heap 内存**

```
spark.memory.offHeap.enabled = true
spark.memory.offHeap.size = 4GB
```

OFF-heap 跳过 GC，但需要手动管理，有 JNI 开销。

**5. 常见 OOM 原因与调参建议**

| 问题 | 原因 | 调参方案 |
|------|------|------|
| Shuffle OOM | Execution 内存不足，Sort 排序溢出 | 调大 `spark.shuffle.memoryFraction`，或开启 `spark.shuffle.compress` |
| Cache 后任务 OOM | Storage 占满，Task 申请不到 Execution 内存 | 减少 `spark.memory.storageFraction`（默认 0.5）或降低 cache 比例 |
| GC 频繁 | 对象短生命周期过多 | 调低 `spark.memory.fraction`（默认 0.6），或增大 `spark.executor.memory` |

**示例：**
```scala
val conf = new SparkConf()
  .set("spark.memory.fraction", "0.6")
  .set("spark.memory.storageFraction", "0.5")
  .set("spark.shuffle.memoryFraction", "0.2")
```

## 题目四：广播变量（Broadcast Variables）原理与最佳实践

### 📖 问题描述

广播变量在 Spark 中用于高效分发只读大变量给所有 Executor。请说明：
1. 不使用广播变量时数据如何分发？
2. 广播变量的实现原理是什么？
3. 什么场景适合使用广播变量？有哪些坑？

---

### ✅ 答案

**1. 不使用广播变量时的分发方式**

假设在 Driver 端有一个 `Map[String, String]` 类型的配置表（共 10MB），在 Map 算子中使用：

```scala
val configMap: Map[String, String] = loadConfig()  // 10MB
rdd.map(row => lookup(row, configMap))  // 每条记录都传 10MB
```

**问题：**
- 每个 Task 都在网络中传输这个 10MB 数据
- 1000 个 Task = 传输 10GB 数据！
- Driver 成为瓶颈，拖慢整个 Job

**2. 广播变量的实现原理**

```
Broadcast 流程：
  ① Driver 端调用 sc.broadcast(value)，创建 TorrentBroadcast 对象
  ② 将数据切分 block（默认 4MB一块），写入 Executor 本地磁盘 /tmp/
  ③ 每个 Executor 启动 BroadcastBlockFetcher 线程，BitTorrent 风格并行拉取
     （Executor 0 有 Block 0,1 → 共享给 Executor 1,2 拉取）
  ④ 存储在 Executor 的 BlockManager 中（StorageLevel = MEMORY_AND_DISK）
  ⑤ Task 执行时直接从本地 BlockManager 读取，不走网络
```

**Broadcast 数据流图：**
```
Driver                        Executor 0                    Executor 1
  │                               │                              │
  │  broadcast(v)                 │                              │
  │ ───────写入──────────→        │                              │
  │  BlockManager[block0..2]      │                              │
  │                               │ ←── BitTorrent Fetch ───→    │
  │                               │  (p2p style, 互相共享 blocks)  │
  │                               │                              │
  Task → 本地读取 (0网络开销)       │ ←── 本地读取 ──────────────→ Task
```

**3. 广播变量的使用方式**

```scala
// 1. 创建广播变量
val broadcastConfig = sc.broadcast(configMap)

// 2. 在 Task 中读取（只在 Executor 本地读取一次）
rdd.map { row =>
  val cfg = broadcastConfig.value   // 注意：这是只读快照，不要修改！
  doLookup(row, cfg)
}

// 3. 广播变量不再使用时销毁（可选）
broadcastConfig.unpersist()
```

**4. 适用场景**

| 场景 | 效果 |
|------|------|
| 大型配置表（如字典表、城市码表） | 网络传输 O(Tasks×Size) → O(Size)，节省数百倍 |
| 小型关联表（< 20MB）的 join | 替代 Shuffle Join，直接 Broadcast，零 Shuffle |
| 静态枚举映射 | 直接广播，避免每次 UDF 调用重新创建 |

**5. 常见坑**

| 坑 | 说明 | 解决方案 |
|----|------|------|
| 广播大对象导致 OOM | `spark.broadcast.blockSize`（默认 4MB）过小产生过多小 block，或单 block 超 `spark.rpc.message.maxSize` | 分段广播、分箱策略 |
| 修改广播变量的 value | 广播变量是只读的，修改不会同步 | 不要修改，必要时重新 broadcast |
| 广播时机不对 | 在窄依赖中间广播，仍然会触发全量 Shuffle | 在 Shuffle 之前广播 |
| 使用 `++` 或 `+=` 更新 | 每次都产生新对象，旧对象 GC 压力大 | 预聚合后广播 |

## 题目五：DataFrame vs DataSet vs RDD 深度对比与选型

### 📖 问题描述

Spark 提供了三种核心数据集抽象：RDD、DataFrame、DataSet。请从数据结构、API 使用、运行效率、类型安全四个维度进行深度对比，并说明各自适用的业务场景。

---

### ✅ 答案

**1. 三者定义与核心特征**

| 维度 | RDD | DataFrame | DataSet |
|------|-----|-----------|---------|
| **数据结构** | 无 schema，每行是 JVM 对象 | 有 schema，每行是 `Row` 对象 | 有 schema，每行是 JVM 对象（强类型） |
| **API 风格** | 函数式（map/filter/reduce） | SQL-like（select/where/groupBy） | 面向对象（dataset.filter(_.age > 18)） |
| **编译时检查** | 无（运行时才发现错误） | 只校验 SQL 语法 | 编译时即检查类型安全 |
| **运行效率** | 低（对象序列化/反序列化开销大） | 高（ Tungsten + SQL Catalyst 优化） | 高（同 DataFrame 优化层） |
| **GC 压力** | 高（JVM 对象多） | 低（ Tungsten 堆外内存管理） | 中等 |
| **Catalyst 优化** | ❌ 不参与 | ✅ 完全参与 | ✅ 完全参与 |

**2. 运行效率对比（原因解析）**

```
RDD 执行路径：
  map/filter → JVM对象 → 序列化 → 网络传输 → 反序列化 → JVM对象 → 下游算子

DataFrame 执行路径：
  select/filter → Logical Plan → Catalyst Optimizer → Physical Plan
                 → Project → Filter → Exchange → ...
                 → Tungsten 二进制堆外内存，无 JVM 对象 GC
```

**Catalyst 优化示例（DataFrame）：**
```scala
// 原始代码（不做任何优化）
df.filter($"age" > 18).select($"name", $"age" * 12 as "age_month")

// Catalyst 自动优化为：
// ① Predicate Pushdown：filter 先执行，减少数据量
// ② Column Pruning：只读取 name 和 age 列
// ③ 常量折叠：age * 12 在编译时求值
```

RDD 则无任何优化机会，忠实执行每一步。

**3. DataFrame vs DataSet 关键区别**

```scala
// DataSet（强类型）—— 需要 case class
case class Person(name: String, age: Int)
val ds: Dataset[Person] = spark.createDataset(people)

// 编译时检查 ✓
ds.filter(_.age > 18)  // 类型安全，编译期报错

// DataFrame（弱类型）—— 只有 schema，无类型
val df: DataFrame = people.toDF("name", "age")
df.filter($"age" > 18)  // 运行时才知道 $"age" 是否存在

// Spark 2.0+：DataFrame = Dataset[Row]（内部统一）
```

**4. 选型决策树**

```
数据源是什么？
  ├─ 非结构化/自定义格式（如日志解析）→ RDD（需要自定义解析）
  ├─ 结构化数据（Parquet/JSON/数据库）→ DataFrame/DataSet
  │
是否需要编译时类型安全？
  ├─ Scala/Java 工程，需要强类型保障 → DataSet
  ├─ Python 用户 / 临时分析 / SQL 为主 → DataFrame
  │
是否需要细粒度控制（如自定义 Partitioner）？
  ├─ 是 → RDD
  ├─ 否 → DataFrame/DataSet（享受 Catalyst 优化）
```

**5. 融合使用示例**

```scala
// 最佳实践：RDD 做原始数据解析，DataFrame 做分析
val rawRDD = sc.textFile("hdfs://logs/app.log").filter(_.nonEmpty)

// 解析成 DataFrame，享受 SQL 能力
val df = rawRDD
  .map(parseLine)
  .toDF("timestamp", "level", "message")
  .filter($"level" === "ERROR")
  .groupBy($"message")
  .count()
  .orderBy($"count".desc)
```
