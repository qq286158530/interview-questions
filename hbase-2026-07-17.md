## 题目一：HBase 的数据结构（KeyValue 模型）是怎样的？为什么设计成这样？

### 参考答案

**KeyValue 模型结构**

HBase 中每一行数据由一个或多个 KeyValue 对组成，每个 KeyValue 的结构如下：

```
KeyValue = RowKey Length(2B) + RowKey + Column Family Length(1B) + Column Family
         + Column Qualifier + Timestamp(8B) + KeyType + Value
```

**各字段含义：**

| 字段 | 说明 |
|------|------|
| RowKey | 行键，唯一标识一行 |
| Column Family | 列族，HBase 中组织数据的基本单位 |
| Column Qualifier | 列修饰符，列族下的具体列 |
| Timestamp | 时间戳，默认64位整数，用于版本控制 |
| KeyType | 操作类型（Put/Delete/DeleteColumn 等） |
| Value | 列的实际值 |

**设计原因：**

1. **LSM-Tree 架构适配**：KeyValue 顺序追加写入 MemStore，天然支持顺序写
2. **支持多版本**：通过 Timestamp 实现数据版本管理
3. **高效读取**：通过 RowKey + CF + CQ + Timestamp 组合可以快速定位
4. **稀疏存储**：null 列不存储，节省空间
5. **支持并发写入**：无锁设计，写入性能高

**读取特性：**
- 读取某列时，会遍历该列的所有版本（按 Timestamp 倒序）
- 默认返回最新版本，可通过指定版本数获取历史版本
- KeyType 为 Delete 的记录会标记该列被删除

> **来源**：[HBase 官方文档 - KeyValue](https://hbase.apache.org/book.html#keyvalue)

## 题目二：HBase 的 WAL（Write Ahead Log）机制是什么？它在数据写入中起到什么作用？

### 参考答案

**WAL（预写日志）机制**

WAL 是 HBase 用于实现数据持久化和故障恢复的核心机制。当数据写入时，遵循以下流程：

```
客户端写入 → WAL（追加到日志文件） → MemStore（内存）→ 返回成功
```

**WAL 的核心作用：**

1. **故障恢复**：当 RegionServer 崩溃时，通过重放 WAL 恢复未刷写到磁盘的数据
2. **数据持久化保证**：写入 WAL 成功才算真正落盘，避免数据丢失
3. **分布式一致性**：配合 ZooKeeper 实现分布式协调

**WAL 机制细节：**

- **写入方式**：Append-only 顺序写入，性能高
- **文件滚动**：当 WAL 文件过大（默认 HDFS 块大小的 0.95 倍）时创建新文件
- **可靠性级别**：
  - `ASYNC_WAL`：异步写入，性能最高，存在丢失风险
  - `SYNC_WAL`：同步写入，Master 和 RegionServer 之间同步
  - `FSYNC_WAL`：强制刷盘，最安全，性能最低
  - `SKIP_WAL`：跳过 WAL，性能最高，但不保证恢复

**数据恢复流程：**

1. RegionServer 崩溃后，Master 将其管理的 regions 分配给其他 RegionServer
2. 新 RegionServer 重放 WAL 中未刷写的数据到 MemStore
3. 恢复完成后继续对外提供服务

**性能优化：**
- WAL 本身的高性能依赖 HDFS 的追加写特性
- 可通过 `WALCellHook` 实现写入加速（如批量压缩）

> **来源**：[HBase 官方文档 - Write Ahead Log (WAL)](https://hbase.apache.org/book.html#wal)

## 题目三：HBase 的 Region 是什么？它是如何分裂（Split）的？

### 参考答案

**Region 的概念**

Region 是 HBase 中数据分布式存储和负载均衡的基本单位：

- 一个表可以划分为多个 Region
- 每个 Region 存储一定范围的 RowKey 数据 `[startKey, endKey)`
- 每个 Region 约 1~2GB（可配置 `hbase.hregion.max.filesize`）
- 一个 Region 只属于一个 RegionServer

**Region 分裂（Split）机制**

```
Region(过大) → Split → 两个新 Region → 分配给不同 RegionServer
```

**分裂触发条件：**

1. **自动分裂**（默认）：
   - 当 Region 下的 StoreFile 大小超过阈值时触发
   - 阈值 = `hbase.hregion.memstore.flush.size` × `hbase.hregion.split.multiplier`
   - 默认 256MB × 2 = 512MB 触发分裂

2. **手动分裂**：
   - 通过 HBase Shell 或 API 手动执行 `split 'tableName', 'splitKey'`

**分裂流程：**

1. RegionServer 在本地做分裂操作，生成两个女儿 Region
2. 将女儿 Region 的元信息写入 ZooKeeper（`/hbase/region-in-transition`）
3. Master 监听到变化，将新 Region 分配给其他 RegionServer
4. 更新 .META. 表，记录新 Region 的起止 RowKey

**注意事项：**

- 分裂发生在 RegionServer 本地，HDFS 上文件通过短路读取复制
- 分裂后，原 Region 下线，女儿 Region 上线
- **热点问题**：热点 RowKey 会导致同 region 持续写入，分裂频繁

**分裂策略（可自定义）：**

| 策略 | 说明 |
|------|------|
| IncreasingToUpperBoundRegionSplitPolicy | 默认，按需增长阈值 |
| ConstantSizeRegionSplitPolicy | 固定大小分裂 |
| KeyPrefixRegionSplitPolicy | 按前缀保证 RowKey 前 N 位相同的在同一 Region |
| DelimitedKeyPrefixRegionSplitPolicy | 按分隔符保证前缀一致 |

> **来源**：[HBase 官方文档 - Region Splits](https://hbase.apache.org/book.html#ops.region.splits)

## 题目四：HBase 的存储结构中，BlockCache 是什么？它是如何提升读取性能的？

### 参考答案

**BlockCache（块缓存）机制**

BlockCache 是 RegionServer 级别的读缓存，位于堆内存中，用于缓存热点 HFile 数据块。

**Block（数据块）概念：**
- HFile 中的最小读取单元，默认 64KB（可配置 `hbase.blocksize`）
- 每个 Block 包含多个 KeyValue 对

**BlockCache 结构（LRU 缓存）：**

```
BlockCache
├── L1: SingleQueue（所有 Block 按访问顺序）
├── L2: CompoundBloomFilter（布隆过滤器，快速判断 Block 是否包含目标 Key）
└── 内存淘汰：超过上限时，按 LRU 策略淘汰最久未访问的 Block
```

**读取流程：**

```
客户端读取请求
    ↓
检查 MemStore（内存最新数据）
    ↓ 命中则返回
检查 BlockCache（HFile 缓存）
    ↓ 命中则返回，并加入 LRU
读取 HFile（磁盘）
    ↓
返回数据
```

**BlockCache 类型：**

| 类型 | 说明 |
|------|------|
| LruBlockCache | 默认，JVM 堆内缓存 |
| BucketCache | 堆外缓存（off-heap），适合大缓存场景 |
| CombinedBlockCache | L1（堆内元数据）+ L2（堆外数据） |

**性能优化配置：**

```xml
<property>
  <name>hfile.block.cache.size</name>
  <value>0.25</value>  <!-- 默认占堆内存 25% -->
</property>
```

**性能影响因素：**
- 缓存命中率越高，读取延迟越低
- 顺序扫描会污染缓存（scan 不加 cache 会禁用缓存）
- 大量随机点查场景建议增大 BlockCache

> **来源**：[HBase 官方文档 - BlockCache](https://hbase.apache.org/book.html#blockcache)

## 题目五：HBase 的读写流程是怎样的？请详细描述从客户端到数据返回的完整链路。

### 参考答案

**写数据流程（Put）**

```
1. 客户端
   └─ 通过 ZooKeeper 获取 .META. 表位置
   └─ 查询 .META. 确定目标 RowKey 所在的 Region 和 RegionServer

2. 客户端 → RegionServer
   └─ 将数据写入 WAL（Write Ahead Log）
   └─ WAL 追加成功后，数据写入 MemStore
   └─ 返回客户端写入成功

3. 后台异步操作
   └─ MemStore 达到阈值 → flush 成 HFile
   └─ HFile 达到阈值 → 触发 Compaction
   └─ 删除的 Cell 超过版本 → 清理过期数据
```

**详细步骤：**

```
[客户端]
    │
    ├─ 1. 获取 ZooKeeper 连接
    ├─ 2. 从 ZooKeeper 获取 -ROOT- 表位置（老版本）
    ├─ 3. 查询 .META. 表，获取目标 Region 的位置
    ├─ 4. 缓存 .META. 信息到客户端（避免重复查询）
    │
[RegionServer]
    ├─ 5. 写入 WAL（append 到 HDFS）
    ├─ 6. WAL 写成功后，写入 MemStore（按序）
    └─ 7. 返回客户端 ACK
```

**读数据流程（Get/Scan）**

```
1. 客户端定位 Region（同写流程 1-4）

2. RegionServer 读取链
   ├─ 并发检查三处数据源：
   │   ├─ MemStore（最新写入，未刷盘）
   │   ├─ BlockCache（热点 HFile 数据）
   │   └─ HFile（磁盘历史数据）
   │
   └─ 合并三处数据，返回最新版本给客户端

3. 版本合并规则
   ├─ 返回 Timestamp 最大的有效数据
   ├─ 如果 KeyType = Delete，视为不存在
   └─ 多版本合并后返回单版本结果
```

**Scan 流程与 Get 的区别：**

| 特性 | Get | Scan |
|------|-----|------|
| 数据范围 | 单行或多行指定 RowKey | 范围扫描 |
| 缓存 | BlockCache | 默认不加缓存（scan 加 cache 则用） |
| 性能 | 快（点查） | 较慢（范围扫描） |
| 并发 | 单线程 | 可并发 scan（Scan 缓存） |

**批量读取优化：**

- `setCaching(int caching)`：每次 RPC 获取的 Row 数量
- `setCacheBlocks(boolean cacheBlocks)`：是否使用 BlockCache
- `setBatch(int batch)`：每行返回的列数

**常见面试追问：**

- **Q：WAL 写入失败怎么办？**
  A：写入失败直接抛异常，数据不会写入 MemStore，保证数据不丢但不持久化。

- **Q：MemStore 写入成功但 RegionServer 崩溃了怎么办？**
  A：通过 WAL 重放恢复，MemStore 数据会丢失但不会丢数据。

- **Q：如何保证读写强一致性？**
  A：HBase 默认提供强一致性保证，读写都落在同一个 RegionServer。

> **来源**：[HBase 官方文档 - Data Model](https://hbase.apache.org/book.html#datamodel)
