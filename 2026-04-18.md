## 题目1：JVM 垃圾回收算法有哪些？分别有什么优缺点？

【答案】

JVM 垃圾回收算法主要有以下几种：

### 1. 标记-清除算法（Mark-Sweep）
**原理**：首先标记出所有需要回收的对象，然后统一回收所有被标记的对象。

**优点**：
- 实现简单，不需要移动对象

**缺点**：
- 效率问题：标记和清除两个过程的效率都不高
- 空间问题：标记清除后会产生大量不连续的内存碎片，碎片太多可能导致在分配大对象时无法找到足够的连续内存而不得不触发另一次垃圾收集

### 2. 复制算法（Copying）
**原理**：将可用内存按容量划分为大小相等的两块，每次只使用其中一块。当这一块的内存用完了，就将还存活着的对象复制到另一块上面，然后再把已使用过的内存空间一次清理掉。

**优点**：
- 不会产生内存碎片
- 分配效率高（只需移动堆顶指针）

**缺点**：
- 可用内存缩小为原来的一半，代价较高
- 如果大多数对象存活率很高，则复制次数会很多，效率会下降

### 3. 标记-整理算法（Mark-Compact）
**原理**：标记过程与"标记-清除"算法一样，但后续步骤不是直接对可回收对象进行清理，而是让所有存活的对象都向一端移动，然后直接清理掉端边界以外的内存。

**优点**：
- 不会产生内存碎片
- 利用率高（所有内存都能使用）

**缺点**：
- 移动对象需要更新引用地址，有额外的开销
- 整理过程效率相对较低

### 4. 分代收集算法（Generational Collection）
**原理**：根据对象存活周期的不同将内存划分为几块，一般把 Java 堆分为新生代和老年代。新生代每次垃圾收集都有大量对象死亡，少量存活，采用复制算法；老年代对象存活率高、没有额外空间担保，采用标记-清除或标记-整理算法。

**优点**：结合多种算法优点，针对不同年代使用最合适的算法，是目前主流 JVM 采用的算法。

## 题目2：synchronized 和 volatile 有什么区别？

【答案】

synchronized 和 volatile 是 Java 并发编程中两个重要的关键字，它们既有区别又有联系：

### 1. 作用范围不同
- **volatile** 只能修饰变量
- **synchronized** 可以修饰方法、代码块、静态方法

### 2. 原子性保证
- **volatile** 不保证原子性，只能保证可见性和有序性
- **synchronized** 保证原子性、可见性和有序性

### 3. 性能差异
- **volatile** 是轻量级同步机制，不会阻塞线程，性能更好
- **synchronized** 是重量级锁，会导致线程阻塞，开销较大

### 4. 底层原理
- **volatile** 通过插入内存屏障禁止指令重排，实现可见性
- **synchronized** 通过 monitorenter 和 monitorexit 指令实现，通过对象监视器锁实现同步

### 5. 使用场景
- **volatile** 适用于：状态标志变量、双重检查锁定（Double Check Lock）中的可见性保证
- **synchronized** 适用于：需要保证原子性操作的场景，如计数器、订单扣减等

### 一句话总结
volatile 是线程同步的轻量级实现，只能修饰变量；synchronized 可以修饰方法和代码块。volatile 能保证可见性但不保证原子性，synchronized 两者都能保证。

## 题目3：Spring Cloud Gateway 和 Netflix Zuul 有什么区别？

【答案】

### 1. 架构层面
- **Zuul 1.x**：基于阻塞 I/O，每个请求对应一个线程，无法支持高并发
- **Gateway**：基于 Spring WebFlux（Reactor 编程），完全异步非阻塞，支持高并发

### 2. 线程模型
- **Zuul**：同步阻塞模型，使用线程池处理请求
- **Gateway**：异步非阻塞模型，使用事件循环处理请求，资源消耗更少

### 3. 过滤器的实现方式
- **Zuul**：使用 Servlet 过滤器的扩展点，通过 FilterType 指定过滤器类型
- **Gateway**：使用 Spring 5.0 WebFlux 的 HandlerFilterPipeline，提供了更灵活的过滤器链

### 4. 动态配置
- **Zuul**：需要结合 Spring Cloud Config 或 Nacos 等配置中心实现动态路由
- **Gateway**：原生支持 YAML 配置，可结合配置中心实现动态路由更新

### 5. 性能
- **Gateway** 在吞吐量上有明显优势，响应延迟更低
- **Zuul** 在高并发场景下容易成为瓶颈

### 6. 维护状态
- **Zuul** 1.x 不支持连接复用
- **Gateway** 支持连接池和 HTTP/2

### 总结
Spring Cloud Gateway 是 Netflix Zuul 的替代方案，采用异步非阻塞架构，性能更优，是未来微服务网关的主流选择。

## 题目4：Spring Boot 中 @Bean 和 @Component 有什么区别？什么时候该用哪个？

【答案】

### @Component 注解
- 用于**自动扫描**方式管理 Bean
- 类上标注 @Component，Spring 会自动扫描并注册为 Bean
- 适用于自己编写的类

### @Bean 注解
- 用于**显式声明**方式管理 Bean
- 在 @Configuration 类的方法上使用，返回对象会注册为 Bean
- 适用于引入第三方库类、配置类、静态工厂创建的对象

### 主要区别

| 区别 | @Component | @Bean |
|------|-------------|-------|
| 管理方式 | 自动扫描 | 显式声明 |
| 适用场景 | 自己编写的类 | 第三方类、静态工厂 |
| 命名方式 | 默认类名首字母小写，可自定义 | 方法名即为 Bean 名称 |
| 注入方式 | 属性注入、构造器注入 | 方法参数注入 |
| 优先级 | 较低 | 较高 |

### @Bean 的特殊用法

```java
@Configuration
public class MyConfig {
    // 指定 Bean 名称
    @Bean(name = "customName")
    public SomeService someService() {
        return new SomeServiceImpl();
    }
    
    // 初始化和销毁回调
    @Bean(initMethod = "init", destroyMethod = "cleanup")
    public DataSource dataSource() {
        return new DataSource();
    }
}
```

### 选择建议
- **自己写的类** → 用 @Component（及其衍生的 @Service、@Repository、@Controller）
- **引入第三方组件** → 用 @Bean 手动注册

## 题目5：RabbitMQ 消息确认机制（Confirm 和 Return）是什么？

【答案】

RabbitMQ 提供了两种消息确认机制来确保消息可靠投递：

### 1. 生产者确认（Publisher Confirm）

**概念**：消息从生产者到达 RabbitMQ Broker 后，Broker 会返回确认信息。

**三种确认模式**：
- **同步确认**：每条消息等待确认，性能差但可靠
- **异步批量确认**：开启批量确认，提高性能
- ** publisher confirm 机制**：通过.channel.waitForConfirms() 等待 Broker 确认

**配置示例**：
```java
// 开启确认模式
channel.confirmSelect();
// 发送消息
channel.basicPublish(exchange, routingKey, MessageProperties.PERSISTENT_TEXT_PLAIN, message.getBytes());
// 等待确认
channel.waitForConfirmsOrDie();
```

### 2. 消费者确认（Consumer Acknowledge）

**概念**：消费者收到消息后，需要发送确认回执。

**三种确认模式**：
- **自动确认（autoAck=true）**：消息一发出立即确认，可能丢失消息
- **手动确认（autoAck=false）**：消费者必须调用 basicAck() 确认消息
- **否定确认（basicNack/basicReject）**：可以拒绝消息，决定是否重新入队

**代码示例**：
```java
// 手动确认
channel.basicConsume(queueName, false, new DefaultConsumer(channel) {
    @Override
    public void handleDelivery(String consumerTag, Envelope envelope,
                               AMQP.BasicProperties properties, byte[] body) {
        try {
            // 处理消息
            process(body);
            // 手动确认
            channel.basicAck(envelope.getDeliveryTag(), false);
        } catch (Exception e) {
            // 拒绝消息，requeue=false 表示不重新入队
            channel.basicNack(envelope.getDeliveryTag(), false, false);
        }
    }
});
```

### 3. Return Listener（Return 监听器）

**概念**：当消息无法路由到指定队列时（如路由键不匹配），触发 Return Listener。

**使用场景**：
- 消息无法路由时的处理
- 记录日志、补偿处理

```java
channel.addReturnListener(new ReturnListener() {
    public void handleReturn(int replyCode, String replyText,
                            String exchange, String routingKey,
                            AMQP.BasicProperties properties, byte[] body) {
        // 消息无法路由时的处理逻辑
    }
});
```

## 题目6：Kafka 消费者组（Consumer Group）是什么？如何实现负载均衡？

【答案】

### Consumer Group 概念

**消费者组**：由多个消费者实例组成，共同消费一个或多个 Topic 的消息。组内消费者共享相同的 group.id。

### 核心特性

1. **消息独占**：一条消息只会被消费者组内的一个消费者消费
2. **并行消费**：多个消费者可以同时消费不同分区的消息
3. **动态扩容**：可以动态添加或减少消费者，Kafka 自动重新分配分区

### 分区分配策略

Kafka 提供了三种分区分配策略：

1. **Range（范围策略）**：按 Topic 对消费者进行分配
   - 每个消费者分配连续的分区范围
   - 可能导致分配不均

2. **RoundRobin（轮询策略）**：将所有 Topic 的分区组成轮询队列
   - 分配更均匀
   - 消费多个 Topic 时效果更好

3. **StickyAssignor（粘性分配策略）**：在保证均衡的前提下，尽量保持原有的分配关系
   - 减少分区重新分配的开销
   - Kafka 0.11+ 支持

### 负载均衡原理

```
Topic: orders
Partitions: P0, P1, P2, P3

Consumer Group A:
  - Consumer1: P0, P1
  - Consumer2: P2, P3

Consumer Group B (独立消费):
  - Consumer3: P0, P1, P2, P3
```

### Rebalance（再均衡）

当以下情况发生时，触发 Rebalance：
- 消费者加入或离开组
- 消费者心跳超时
- 分区数量变化
- 订阅的 Topic 发生变化

**再均衡过程**：
1. Coordinator 通知所有消费者停止消费
2. 所有消费者重新加入组
3. Coordinator 发送新的分区分配方案
4. 消费者开始按照新方案消费

### 注意事项

- 消费者数量不建议超过分区数（多余的消费者会闲置）
- 避免频繁的 Rebalance（影响吞吐量）
- 合理设置心跳参数：session.timeout.ms、heartbeat.interval.ms

## 题目7：什么是 AI Agent？它的核心架构是什么？

【答案】

### AI Agent 定义

AI Agent（人工智能智能体）是一种能够感知环境、进行自主决策并执行动作的智能软件系统。它以大语言模型（LLM）为"大脑"，代表用户自动化完成复杂任务。

### 核心公式

**Agent = LLM + Planning（规划）+ Memory（记忆）+ Tools（工具）**

### 核心架构

#### 1. 推理与规划（Reasoning / Planning）
- 依赖 LLM 分析当前任务状态
- 拆解目标，生成思考路径
- 决定下一步行动
- 使用 Chain-of-Thought (CoT) 提示技术逐步推理

#### 2. 记忆（Memory）
- **短期记忆**：上下文历史，用于保持对话连续性
- **长期记忆**：外部知识库检索（向量数据库或知识图谱）
- 防止模型遗忘历史信息，并从过去经验中学习

#### 3. 执行与工具（Acting / Tools）
- 执行具体操作：查询信息、调用外部工具
- 工具扩展了 LLM 的能力
- 包括：Function Call、MCP、Shell 命令、代码执行、API 调用

#### 4. 观察（Observation）
- 接收工具执行的反馈
- 将反馈纳入上下文用于下一轮推理
- 形成闭环反馈机制

### Agent Loop（智能体循环）

Agent 的运行本质是一个 while 循环：

1. **初始化**：加载 System Prompt、可用工具列表、用户请求
2. **循环迭代**：LLM 推理 → 工具调用 → 上下文更新
3. **终止条件**：LLM 判断任务完成
4. **安全兜底**：设置最大迭代轮次（如 10-20 轮）防止死循环

### Agent vs 传统编程 vs Workflow

| 维度 | 传统编程 | Workflow | Agent |
|------|----------|----------|-------|
| 决策主体 | 程序员 | 产品/开发 | AI |
| 灵活性 | 低 | 中 | 高 |
| 技能要求 | 高 | 中 | 低 |
| 修改成本 | 数天至数周 | 数小时至数天 | 数分钟 |

## 题目8：【算法】给定一个字符串，请你找出其中不含有重复字符的最长子串的长度。

【答案】

### 方法一：滑动窗口（最优解）

**思路**：使用双指针维护一个不重复字符的窗口，右指针不断右移，当遇到重复字符时，左指针收缩窗口。

```java
public int lengthOfLongestSubstring(String s) {
    // 记录字符最后一次出现的索引
    Map<Character, Integer> map = new HashMap<>();
    int left = 0;
    int maxLen = 0;
    
    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        // 如果字符已存在，调整左指针
        // 取 max 是为了处理 abba 这种后面字符覆盖前面索引的情况
        if (map.containsKey(c)) {
            left = Math.max(left, map.get(c) + 1);
        }
        // 更新字符最后出现位置
        map.put(c, right);
        // 更新最大长度
        maxLen = Math.max(maxLen, right - left + 1);
    }
    
    return maxLen;
}
```

**时间复杂度**：O(n)，每个字符最多访问两次
**空间复杂度**：O(min(m, n))，m 为字符集大小

### 方法二：优化的滑动窗口（数组实现）

```java
public int lengthOfLongestSubstring(String s) {
    int[] index = new int[128];  // 记录字符最后出现的位置
    Arrays.fill(index, -1);
    
    int left = 0;
    int maxLen = 0;
    
    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        // 左指针移动到不重复位置
        left = Math.max(left, index[c] + 1);
        index[c] = right;
        maxLen = Math.max(maxLen, right - left + 1);
    }
    
    return maxLen;
}
```

### 示例

```
输入: "abcabcbb"
输出: 3
解释: 无重复字符的最长子串是 "abc"，长度为 3

输入: "bbbbb"
输出: 1
解释: 无重复字符的最长子串是 "b"，长度为 1

输入: "pwwkew"
输出: 3
解释: 无重复字符的最长子串是 "wke"，长度为 3
```

### 关键点
1. 使用 Map 或数组记录字符最后出现的位置
2. 左指针需要取 max，避免左指针回退
3. 每次更新最大长度

## 题目9：【算法】请实现一个 LRU（最近最少使用）缓存机制。

【答案】

### LRU 缓存概述

LRU（Least Recently Used）是一种常用的缓存淘汰策略，当缓存满时，淘汰最长时间未被访问的数据。

### 实现方案：HashMap + 双向链表

```java
public class LRUCache {
    // 容量
    private int capacity;
    // HashMap 用于 O(1) 查找
    private Map<Integer, Node> map;
    // 虚拟头尾节点
    private Node head, tail;
    
    public LRUCache(int capacity) {
        this.capacity = capacity;
        this.map = new HashMap<>();
        head = new Node(0, 0);
        tail = new Node(0, 0);
        head.next = tail;
        tail.prev = head;
    }
    
    // 获取值
    public int get(int key) {
        if (!map.containsKey(key)) return -1;
        Node node = map.get(key);
        // 移动到链表头部（最近使用）
        moveToHead(node);
        return node.value;
    }
    
    // 放入值
    public void put(int key, int value) {
        if (map.containsKey(key)) {
            Node node = map.get(key);
            node.value = value;
            moveToHead(node);
        } else {
            Node newNode = new Node(key, value);
            map.put(key, newNode);
            addToHead(newNode);
            if (map.size() > capacity) {
                // 淘汰尾部节点（最久未使用）
                Node removed = removeTail();
                map.remove(removed.key);
            }
        }
    }
    
    // 添加到头部
    private void addToHead(Node node) {
        node.prev = head;
        node.next = head.next;
        head.next.prev = node;
        head.next = node;
    }
    
    // 移动到头部
    private void moveToHead(Node node) {
        removeNode(node);
        addToHead(node);
    }
    
    // 移除节点
    private void removeNode(Node node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }
    
    // 移除尾部节点
    private Node removeTail() {
        Node node = tail.prev;
        removeNode(node);
        return node;
    }
    
    // 双向链表节点
    class Node {
        int key, value;
        Node prev, next;
        Node(int k, int v) { key = k; value = v; }
    }
}
```

### 使用 LinkedHashMap 简化实现

```java
public class LRUCache extends LinkedHashMap<Integer, Integer> {
    private int capacity;
    
    public LRUCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }
    
    public int get(int key) {
        return super.getOrDefault(key, -1);
    }
    
    public void put(int key, int value) {
        super.put(key, value);
    }
    
    @Override
    protected boolean removeEldestEntry(Map.Entry<Integer, Integer> eldest) {
        return size() > capacity;
    }
}
```

### 复杂度分析

| 操作 | 时间复杂度 | 空间复杂度 |
|------|-----------|-----------|
| get | O(1) | O(capacity) |
| put | O(1) | O(capacity) |

### 应用场景
- 浏览器缓存
- CDN 缓存
- 数据库连接池
- Redis 缓存淘汰策略

## 题目10：Spring Cloud Alibaba Nacos 的服务注册与发现机制是什么？

【答案】

### Nacos 概述

Nacos 是 Spring Cloud Alibaba 的核心组件，提供服务注册发现和配置管理功能。

### 服务注册机制

#### 1. 服务注册过程

```
1. 服务启动 → 读取配置（nacos.server-addr, namespace, group 等）
2. 向 Nacos Server 发送 REST 请求注册服务
3. Server 保存服务信息到注册表
4. Server 向客户端发送成功响应
```

**注册 API**：
```
POST /nacos/v1/ns/instance
{
  "serviceName": "order-service",
  "ip": "192.168.1.100",
  "port": 8080,
  "clusterName": "DEFAULT"
}
```

#### 2. 服务心跳机制

- 服务提供者每隔 5 秒发送心跳（续约）
- 如果 15 秒未收到心跳，标记为不健康
- 如果 30 秒未收到心跳，删除该实例

#### 3. 服务分级存储模型

```
Namespace（命名空间）→ Group（分组）→ Cluster（集群）→ Service → Instance
```

- **Namespace**：用于环境隔离（dev、test、prod）
- **Group**：用于逻辑分组
- **Cluster**：用于地域隔离（同城多机房）
- **Service**：微服务名称
- **Instance**：服务实例

### 服务发现机制

#### 1. 主动拉取

```
客户端 → 定时任务（每 30 秒）→ Nacos Server → 获取服务列表 → 本地缓存
```

#### 2. 推送更新（订阅）

```
客户端订阅服务 → Nacos Server 保存订阅关系 → 服务变更 → 推送更新 → 客户端更新本地缓存
```

#### 3. 负载均衡

Nacos 与 Spring Cloud LoadBalancer 集成，支持多种负载均衡策略：

- **权重模式**：实例配置权重，权重越高分配流量越多
- **标签模式**：根据实例标签进行路由
- **可用敏感度**：优先选择健康的实例

### 集群部署

Nacos 支持两种一致性协议：

1. **AP 模式**（默认）：适用于注册发现场景
   - 允许短暂数据不一致
   - 分区可用性优先

2. **CP 模式**：适用于配置管理场景
   - 保证强一致性
   - Raft 协议选主

### 与 Eureka 的区别

| 特性 | Nacos | Eureka |
|------|-------|--------|
| 一致性 | AP/CP 可切换 | AP |
| 配置管理 | 原生支持 | 需结合 Config |
| 多环境 | Namespace 隔离 | 无 |
| 权重路由 | 支持 | 不支持 |
| 健康检查 | TCP/HTTP/MYSQL | 心跳 |
