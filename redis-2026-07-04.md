## 题目一：Redis 有哪几种数据类型？分别适用于什么场景？

### 参考答案

Redis 支持五种基础数据类型，补充还有几种特殊数据类型：

| 数据类型 | 底层实现 | 常用命令 | 典型应用场景 |
|---------|---------|---------|------------|
| **String** | SDS（简单动态字符串） | `SET/GET/MGET/MSET/INCR/DECR` | 缓存简单值、计数器、分布式锁（SETNX）、限流 |
| **Hash** | 压缩列表或 Hash 表 | `HSET/HGET/HGETALL/HEXISTS` | 存储对象，如用户信息、商品详情（比 String 省空间） |
| **List** | 压缩列表或 QuickList | `LPUSH/RPOP/LRANGE/LLEN` | 消息队列（配合 BRPOP）、最新列表、任务队列 |
| **Set** | IntSet 或 Hash 表 | `SADD/SMEMBERS/SISMEMBER/SDIFF/SINTER` | 标签系统、好友关系、去重集合、抽奖 |
| **Sorted Set（ZSet）** | 压缩列表或 SkipList + Hash | `ZADD/ZRANGE/ZRANK/ZSCORE` | 排行榜、延迟队列、有序通知 |
| **Bitmap** | String 类型按位操作 | `SETBIT/GETBIT/BITOP/BITCOUNT` | 签到统计、用户在线状态、布隆过滤器辅助 |
| **HyperLogLog** | String 类型 | `PFADD/PFCOUNT/PFMERGE` | 独立访客（UV）统计，误差约 0.81% |
| **Geospatial** | ZSet（GeoHash） | `GEOADD/GEOPOS/GEODIST/GEORADIUS` | 附近的人、附近商家、LBS 应用 |

### 补充知识点

- Redis 7.0 引入了 **多线程 I/O**（默认关闭），但核心命令执行仍是单线程，保证原子性
- String 类型最大 512MB，但一般建议不超过 100KB，避免大 key 问题
- ZSet 的 Score 为双精度浮点数，可用于权重排序

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 参考答案

Redis 有两种持久化方式，可以同时启用，也可以单独使用。

### RDB（Redis Database）

- **原理**：定时将内存中的**全量数据快照**保存为 `dump.rdb` 文件
- **触发方式**：
  - 手动：`SAVE`（阻塞）或 `BGSAVE`（后台异步）
  - 自动：`save m n`（m 秒内 n 次写操作触发）
- **优点**：恢复速度快（全量恢复），fork 子进程不影响主进程处理请求
- **缺点**：可能丢失最近一次快照之后的数据

### AOF（Append Only File）

- **原理**：每个写命令追加到 `appendonly.aof` 文件末尾
- **刷盘策略**（`appendfsync`）：
  - `always`：每次写都刷盘，最安全但最慢
  - `everysec`（默认）：每秒刷盘一次，最多丢 1 秒数据
  - `no`：由 OS 决定刷盘时机，最快但最不安全
- **Rewrite 机制**：AOF 文件过大会自动压缩（`BGREWRITEAOF`），去除冗余命令
- **优点**：数据安全性更高（可配置），AOF 日志是追加操作，易于恢复
- **缺点**：文件体积通常比 RDB 大，恢复速度慢

### 对比与选择建议

| 维度 | RDB | AOF |
|-----|-----|-----|
| 文件体积 | 小（紧凑快照） | 大（持续追加） |
| 恢复速度 | 快 | 慢 |
| 数据完整性 | 可能丢数据 | 可配置（everysec 丢 1 秒） |
| 资源消耗 | fork 瞬时内存翻倍风险 | 持续磁盘 I/O |
| 启动优先级 | 先加载 RDB | AOF 优先（AOF 更完整时） |

**生产环境推荐策略**：
```
# 同时开启，优先用 AOF 恢复
appendonly yes
appendfsync everysec
rdbcompression yes
save 900 1 save 300 10 save 60 10000
```

## 题目三：什么是缓存穿透、缓存击穿、缓存雪崩？如何解决？

### 参考答案

这是 Redis 缓存最经典的三类问题，属于高级/架构层面必备知识。

### 1. 缓存穿透

**定义**：查询一个**不存在的数据**，缓存没有，数据库也没有，每次都打到数据库。

**危害**：恶意请求或爬虫频繁查询不存在的数据，击垮数据库。

**解决方案**：
- **布隆过滤器（BloomFilter）**：将所有存在的数据哈希到一个 bitmap，请求先过布隆过滤器，拦截不存在数据的查询
- **缓存空值**：即使查询结果为空，也缓存一个空值（如 `"NULL"`，设置较短 TTL）
- **参数校验**：对 ID 格式合法性做基础校验

### 2. 缓存击穿

**定义**：某个**热点 key 过期**的瞬间，大量并发请求都发现缓存不存在，一起打到数据库。

**危害**：数据库瞬时压力过大，甚至宕机。

**解决方案**：
- **互斥锁（分布式锁）**：只允许一个请求去查数据库，其他等待或从缓存拿旧数据
  ```lua
  -- Redis 实现互斥锁
  if redis.call('SETNX', lock_key, 1) == 1 then
      redis.call('EXPIRE', lock_key, timeout)
      -- 查数据库、写缓存
      redis.call('DEL', lock_key)
  end
  ```
- **逻辑过期**：不给 key 设 TTL，设一个 `expire_time` 字段。查到时判断是否逻辑过期，是则异步重建缓存，返回旧数据（不阻塞）
- **永不过期**：热点数据不设过期时间，由后台异步更新

### 3. 缓存雪崩

**定义**：大量 key **同时过期**，或 Redis 宕机，导致大量请求全部打到数据库。

**解决方案**：
- **过期时间随机化**：在 TTL 基础上加随机数，如 `TTL = base + rand(0, 3600)`
- **多级缓存**：本地缓存（Caffeine/Guava）+ Redis + 数据库
- **Redis 高可用**：主从 + 哨兵，或 Redis Cluster，避免全盘崩溃
- **服务熔级/限流**：Guava RateLimiter、Sentinel 等，防止数据库过载
- **预热**：系统启动前提前加载热点数据到缓存

### 总结对比

| 问题 | 原因 | 核心对策 |
|-----|------|---------|
| 穿透 | 查询不存在的数据 | 布隆过滤器 + 空值缓存 |
| 击穿 | 热点 key 过期瞬间并发 | 互斥锁 / 逻辑过期 |
| 雪崩 | 大量 key 同时过期 / Redis 宕机 | 过期随机 + 多级缓存 + 高可用 |

## 题目四：Redis 主从复制原理是什么？如何实现读写分离？

### 参考答案

### 主从复制架构

Redis 支持一主多从架构，主节点负责写，从节点负责读，实现读写分离。

### 复制原理（PSYNC2 协议）

1. **全量同步**（Full Resync）：
   - 从节点发送 `PSYNC ? -1`（首次同步，请求主节点 run_id 和偏移量）
   - 主节点执行 `BGSAVE` 生成 RDB 快照，同时用 **复制积压缓冲区（replication buffer）** 记录期间所有写命令
   - 主节点将 RDB 文件发送给从节点，从节点清空本地数据，加载 RDB
   - 主节点将复制积压缓冲区中的命令发给从节点，完成同步

2. **增量同步（命令传播）**：
   - 主节点每次写命令后，同时发送给所有从节点（`replication stream`）
   - 从节点根据收到的命令更新本地数据
   - 依赖 **主从 offset** + **复制积压缓冲区** 实现断线重连后的增量同步

### 读写分离配置

```
# 从节点配置（只读模式，默认开启）
replica-read-only yes
```

### 注意事项

- **主从延迟**：从节点复制是异步的，存在延迟，延迟敏感的读操作仍在主节点
- **复制风暴**：一个主节点挂载大量从节点时，全量同步会对主节点造成压力
- **主从切换（Sentinel）**：主节点故障时，Sentinel 自动选举新主，从节点重新指向新主
- **Redis Cluster 模式**：slot 分片，不同节点承担不同分片的读写，提供了另一种扩展方案

## 题目五：Redis 的过期 key 删除策略是什么？如何排查和处理大 key？

### 参考答案

### 过期 key 删除策略

Redis 采用**惰性删除 + 定期删除**相结合的方式：

1. **惰性删除（Lazy Expire）**：
   - 访问 key 时才检查是否过期，过期则删除
   - 优点：对 CPU 友好，不用额外开销
   - 缺点：过期 key 长期不被访问会一直占用内存

2. **定期删除（Active Expire）**：
   - 每 100ms 随机抽取一批设置了过期时间的 key，检查并删除过期的
   - 限制每次删除时间不超过 25ms（`hz` 参数控制频率，默认 10）
   - 平衡了内存和 CPU

3. **内存淘汰策略（当内存达到上限 `maxmemory` 时）**：
   - `noeviction`：不淘汰，返回错误（默认）
   - `volatile-lru` / `allkeys-lru`：LRU 算法淘汰（已设置 TTL / 全局）
   - `volatile-lfu` / `allkeys-lfu`：LFU 算法淘汰
   - `volatile-random` / `allkeys-random`：随机淘汰
   - `volatile-ttl`：淘汰 TTL 最短的 key

### 大 Key 排查与处理

**什么是大 key**：
- String 类型 value > 10MB
- List/Set/ZSet/HASH 元素过多（> 10000 个元素）

**排查方法**：
```bash
# 1. bigkeys 命令（扫描大 key）
redis-cli --bigkeys

# 2. memory usage 手动检查
redis-cli --scan | xargs -I{} redis-cli memory usage {}

# 3. Redis 6.2+ JSON 格式输出
redis-cli --bigkeys --format json

# 4. 在线分析（生产慎用 SCAN）
redis-cli --scan --pattern '*' | head -100 | xargs -I{} redis-cli type {}
```

**处理方案**：
1. **拆分**：将大 hash 按字段拆分为多个小 hash（`user:1000:base`、`user:1000:ext`）
2. **压缩**：List 改用 String 存储序列化后的紧凑格式（如 JSON）
3. **异步删除**：`UNLINK`（非阻塞删除）替代 `DEL`
4. **延迟队列**：将大 key 的清理操作放入后台队列，逐步处理
5. **避免生产**：写入前做大小评估，业务层面限制单 key 大小
