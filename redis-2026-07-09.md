## 题目一：Redis 有哪些数据类型？底层是怎么实现的？

### 参考答案

Redis 提供了丰富的数据类型，常见的有五种：**String（字符串）、Hash（哈希）、List（列表）、Set（集合）、Zset（有序集合）**。

随着 Redis 版本的更新，后面又支持了四种数据类型：**BitMap、HyperLogLog、GEO、Stream**。

### 五种数据类型的应用场景

| 数据类型 | 应用场景 |
|---------|---------|
| String | 缓存对象、常规计数、分布式锁、共享 session 信息 |
| List | 消息队列（需自行实现全局唯一 ID，不支持消费组） |
| Hash | 缓存对象、购物车 |
| Set | 聚合计算（并集、交集、差集），如点赞、共同关注、抽奖活动 |
| Zset | 排序场景，如排行榜、电话和姓名排序 |

### 底层数据结构实现

| 数据类型 | 底层数据结构 |
|---------|------------|
| String | SDS（简单动态字符串），编码有 int、raw、embstr |
| List | quicklist（Redis 3.2+），之前是双向链表或压缩列表 |
| Hash | 压缩列表或哈希表，Redis 7.0 起废弃压缩列表改为 listpack |
| Set | 哈希表或整数集合 |
| Zset | 压缩列表或跳表，Redis 7.0 起废弃压缩列表改为 listpack |

**SDS 相比 C 字符串的优势：**
- 支持二进制数据（以 len 判断结束而非 `\0`）
- 获取字符串长度复杂度 O(1)
- 拼接字符串不会缓冲区溢出（自动扩容）

## 题目二：Redis 是单线程的吗？为什么单线程还这么快？

### 参考答案

**Redis 单线程指的是"接收客户端请求→解析请求→进行数据读写等操作→发送数据给客户端"这个过程由一个线程（主线程）来完成。**

但 Redis 程序并非完全单线程，Redis 在启动时会启动后台线程（BIO）：
- Redis 2.6：2 个后台线程（关闭文件、AOF 刷盘）
- Redis 4.0+：新增 lazy free 线程，异步释放内存

**Redis 单线程依然很快的原因：**

1. **内存操作 + 高效数据结构**：Redis 的大部分操作都在内存中完成，且采用了高效的数据结构，瓶颈是内存或网络带宽而非 CPU
2. **无多线程竞争开销**：避免了多线程切换带来的时间和性能开销，也不会产生死锁问题
3. **I/O 多路复用**：采用 epoll 机制，一个线程处理多个 Socket 请求

**Redis 6.0 引入了多线程 I/O**：针对网络 I/O 采用多线程处理，但命令执行仍然单线程。官方表示性能提升至少一倍以上。

## 题目三：Redis 如何实现数据持久化？AOF 和 RDB 有什么区别？

### 参考答案

Redis 有三种数据持久化方式：**AOF 日志、RDB 快照、混合持久化**。

### AOF（Append Only File）

每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里。Redis 重启时读取 AOF 文件的命令逐一执行来恢复数据。

**写回策略（appendfsync）：**

| 策略 | 说明 | 优缺点 |
|-----|------|-------|
| Always | 每次写命令执行完后同步写回硬盘 | 数据不丢失，但 I/O 开销大 |
| Everysec | 先把命令写入内核缓冲区，每秒刷盘一次 | 折中方案，默认配置 |
| No | 由操作系统决定何时刷盘 | 性能最好，但数据丢失风险大 |

**AOF 重写机制**：当 AOF 文件过大时，后台子进程 `bgrewriteaof` 会读取数据库中的所有键值对，用一条命令记录到新的 AOF 文件，实现压缩。

### RDB（Redis DataBase）

将某一时刻的内存数据以二进制方式写入磁盘。Redis 提供了 `save`（主线程执行，会阻塞）和 `bgsave`（子进程执行，不阻塞）两个命令生成 RDB 文件。

**RDB 快照期间数据可修改**：利用写时复制（COW）技术，fork 出子进程后父子进程共享物理内存，主线程执行写操作时复制副本给子进程写入 RDB。

### 混合持久化（Redis 4.0+）

集成了 AOF 和 RDB 的优点。AOF 重写时，先将以 RDB 方式写入 AOF 文件前半部分，再将增量命令以 AOF 格式写入后半部分。重启加载时先加载 RDB 部分加速启动，再加载增量 AOF 部分减少数据丢失。

### AOF vs RDB 对比

| 特性 | AOF | RDB |
|-----|-----|-----|
| 文件格式 | 操作命令日志 | 二进制快照 |
| 数据恢复速度 | 慢（需执行所有命令） | 快（直接读入内存） |
| 数据完整性 | 取决于写回策略 | 取决于快照频率 |
| 文件大小 | 较大 | 较小 |

## 题目四：什么是缓存雪崩、击穿、穿透？如何解决？

### 参考答案

这三个问题是面试中非常常见的高频面试题。

### 缓存雪崩

**原因**：大量缓存数据在同一时间过期，或者 Redis 故障宕机，导致大量请求直接访问数据库，数据库压力骤增甚至宕机。

**解决方案：**

- **均匀设置过期时间**：给数据的过期时间加上随机数，避免同时过期
- **互斥锁**：保证同一时间只有一个请求构建缓存
- **后台更新缓存**：缓存"永久有效"，由后台线程定时更新；或缓存失效后通过消息队列通知后台更新
- **构建 Redis 高可用集群**：主从架构，故障时自动切换
- **服务熔断/请求限流**：缓存故障时暂停或限制访问数据库

### 缓存击穿

**原因**：某个热点数据过期，此时大量请求访问该热点数据，直接打垮数据库。

**解决方案：**

- **互斥锁**：与雪崩类似
- **热点数据不设置过期时间**：由后台异步更新，或在过期前提前更新

### 缓存穿透

**原因**：数据既不在缓存中也不在数据库中（比如业务误删或恶意攻击），导致大量请求打到数据库。

**解决方案：**

- **非法请求限制**：API 入口校验请求参数
- **缓存空值或默认值**：查询不存在的数据时，在缓存中设置空值/默认值
- **布隆过滤器**：写入数据库时在布隆过滤器做标记，查询前先检查布隆过滤器

**布隆过滤器原理**：由 N 个哈希函数和位图数组组成，数据写入时将 N 个哈希值对应位置设为 1。查询时所有对应位置都为 1 才认为存在（存在假阳性，但不存在一定是准确的）。

## 题目五：Redis 主从复制是如何工作的？第一次同步和增量复制有什么区别？

### 参考答案

### 主从复制概述

Redis 主从复制模式保证多台服务器的数据一致性，采用**读写分离**方式：主服务器可读写，从服务器只读。主服务器写入后自动同步给从服务器。

### 第一次同步（全量复制）

主从服务器间第一次同步分为三个阶段：

1. **建立链接、协商同步**：从服务器发送 `psync ? -1`，主服务器返回 `FULLRESYNC runID offset`
2. **主服务器同步数据给从服务器**：主服务器执行 `bgsave` 生成 RDB 文件，发送给从服务器；从服务器清空本地数据，载入 RDB
3. **主服务器发送新写操作命令**：主服务器将在生成 RDB 和传输 RDB 期间的写操作命令写入 `replication buffer`，同步给从服务器执行

### 命令传播（长连接）

第一次同步后，双方维护 TCP 长连接，主服务器通过长连接将写操作命令传播给从服务器，保证数据一致。

### 分摊主服务器压力

可以从服务器再关联自己的从服务器（形成树形结构），分担主服务器生成 RDB 和传输的压力。

### 增量复制（Redis 2.8+）

网络断开恢复后，从服务器发送 `psync offset`（非 -1），主服务器判断从服务器要读取的数据是否还在 `repl_backlog_buffer`（环形缓冲区）中：
- **数据还在**：发送 `CONTUE` 响应，增量同步
- **数据已被覆盖**：全量同步

`repl_backlog_buffer` 大小建议公式：`second × write_size_per_second × 2`

### 主从复制三个阶段对比

| 阶段 | 触发时机 | 特点 |
|-----|---------|------|
| 全量复制 | 第一次同步、数据被覆盖 | 主服务器生成并传输 RDB，耗时长 |
| 命令传播 | 日常运行 | 基于长连接，异步同步写命令 |
| 增量复制 | 网络断开恢复 | 只同步断开期间的写命令，效率高 |
