# Flink 面试题（2026-06-09）

## 题目一：Flink 的架构是怎样的？请解释 JobManager 和 TaskManager 的作用。

### 答案

Flink 采用主从架构，主要包含以下组件：

**JobManager（主节点）**
- 负责整个作业的调度和执行
- 协调 Checkpoint 和故障恢复
- 将 JobGraph 转换为 ExecutionGraph
- 管理 TaskManager 的注册和资源分配
- 包含三个核心组件：
  - **Dispatcher**：接收作业提交，启动 JobMaster
  - **ResourceManager**：管理插槽（Slot）资源
  - **JobMaster**：负责单个作业的执行

**TaskManager（从节点）**
- 执行具体的计算任务
- 管理工作线程和算子链
- 通过 Slot 来管理资源（每个 Slot 可执行一个 Task）
- 定期向 JobManager 汇报状态

**作业执行流程**：
1. 客户端提交 JobGraph
2. Dispatcher 启动 JobMaster
3. JobMaster 向 ResourceManager 请求资源
4. ResourceManager 让 TaskManager 提供 Slot
5. JobMaster 将 ExecutionGraph 分发到 TaskManager 执行

> 来源：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/flink-architecture/

## 题目二：什么是 Watermark？Flink 如何处理乱序事件？

### 答案

**Watermark（水位线）** 是 Flink 用于处理乱序事件的核心机制。

**概念**：
- Watermark 是一个时间戳，表示"所有时间戳小于该 Watermark 的数据都已到达"
- 用来触发窗口计算和处理延迟数据

**乱序处理原理**：
```
Watermark W 表示：事件时间 < W 的数据不会再出现
窗口 [start, end) 只有在 Watermark W >= end 时才触发
```

**示例**：
```java
WatermarkStrategy
    .<Tuple2<String, Long>>forBoundedOutOfOrderness(Duration.ofSeconds(5))
    .withTimestampAssigner((event, timestamp) -> event.f1);
```
上述代码允许最多 5 秒的乱序。

**关键点**：
- **Event Time 模式**：使用 Watermark 处理乱序
- **Processing Time 模式**：不考虑乱序，直接按系统时间处理
- Watermark 迟到策略：`allowedLateness()` 设置额外等待时间
- 迟到数据默认被丢弃，可通过 `sideOutputLateData()` 收集

> 来源：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream/event-time/

## 题目三：Flink 的 Checkpoint 和 Savepoint 有什么区别？

### 答案

| 特性 | Checkpoint | Savepoint |
|------|-----------|-----------|
| **触发方式** | Flink 自动周期性触发 | 手动触发（命令） |
| **用途** | 故障恢复 | 计划性维护、版本更新 |
| **保留策略** | Job 取消后删除 | 一直保留，除非手动删除 |
| **存储位置** | JobManager 内存（可配置） | 外部存储（如 HDFS） |
| **兼容性** | 与 Job 版本绑定 | 跨 Job 版本（需代码兼容） |

**Checkpoint 原理**：
1. **Barrier 对齐**：Checkpoint 屏障从 Source 向下游发送
2. **状态快照**：算子将状态写入快照
3. **异步完成**：增量写入，避免阻塞

**配置示例**：
```java
env.enableCheckpointing(10000);  // 每10秒执行一次
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(5000);
env.getCheckpointConfig().setCheckpointTimeout(60000);
env.getCheckpointConfig().setMaxConcurrentCheckpoints(1);
```

**Savepoint 用法**：
```bash
# 触发 Savepoint
bin/flink savepoint -j <jobId> <savepointPath>

# 从 Savepoint 恢复
bin/flink run -d -s <savepointPath> <jar>
```

> 来源：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/checkpoints/

## 题目四：Flink 的状态管理机制是怎样的？什么是 Keyed State 和 Operator State？

### 答案

Flink 提供两种状态管理机制：

**Keyed State（键控状态）**
- 绑定到 Key，只能在 KeyedStream 上使用
- 作用域基于当前处理的 Key
- 支持的数据结构：
  - `ValueState<T>`：单一值
  - `ListState<T>`：列表
  - `MapState<K, V>`：键值对
  - `ReducingState<T>`：聚合状态
  - `AggregatingState<I, O>`：聚合

```java
public class CountWindowAverage extends KeyedProcessFunction<String, Integer, Double> {
    private ValueState<Integer> sum;
    private ValueState<Integer> count;

    @Override
    public void open(Configuration parameters) {
        sum = getRuntimeContext().getState(new ValueStateDescriptor<>("sum", Integer.class));
        count = getRuntimeContext().getState(new ValueStateDescriptor<>("count", Integer.class));
    }

    @Override
    public void processElement(Integer value, Context ctx, Collector<Double> out) throws Exception {
        sum.update(sum.value() + value);
        count.update(count.value() + 1);
        out.collect((double) sum.value() / count.value());
    }
}
```

**Operator State（算子状态）**
- 绑定到算子实例，所有 Key 共享
- 支持两种重新分配策略：
  - **UnionListState**：所有算子获取完整列表
  - **BroadcastState**：广播状态

**状态后端选择**：
- `HashMapStateBackend`：内存存储，性能高
- `EmbeddedRocksDBStateBackend`：RocksDB 存储，支持大状态

> 来源：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream/fstate/

## 题目五：Flink 窗口函数有哪些类型？请解释 Tumbling Window 和 Session Window 的区别。

### 答案

**窗口类型分类**：

### 1. 时间窗口

**Tumbling Window（滚动窗口）**
- 无重叠、固定大小
- 示例：每 5 分钟统计一次
```java
.window(TumblingProcessingTimeWindows.of(Time.minutes(5)))
```

**Sliding Window（滑动窗口）**
- 可重叠，有窗口大小和滑动步长
- 示例：窗口大小 10 分钟，滑动 5 分钟
```java
.window(SlidingProcessingTimeWindows.of(Time.minutes(10), Time.minutes(5)))
```

**Session Window（会话窗口）**
- 动态窗口，由事件间隔触发
- 空闲 gap 决定会话超时
```java
.window(SessionWindows.withGap(Time.minutes(10)))
```

### 2. Count Window（计数窗口）
```java
.countWindow(100)           // 滚动计数窗口，每100条触发
.countWindow(100, 50)       // 滑动计数窗口
```

### 窗口函数

| 函数类型 | 说明 |
|---------|------|
| **reduce()** | 增量聚合，输入输出类型相同 |
| **aggregate()** | 通用聚合，输入输出类型可不同 |
| **apply()** | 全量窗口聚合，处理整个窗口数据 |
| **process()** | 底层 API，可访问上下文信息 |

**完整示例**：
```java
DataStream<Tuple2<String, Long>> input = ...

input
    .keyBy(v -> v.f0)
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .reduce((a, b) -> new Tuple2<>(a.f0, a.f1 + b.f1))
    .print();
```
