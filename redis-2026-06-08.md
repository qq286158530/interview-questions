## 题目一：Redis 支持哪些数据类型？各自的使用场景是什么？

### 答案

Redis 是键值对存储，但它并不只是简单的字符串 key-value，还支持多种丰富的数据结构。理解这些数据类型是掌握 Redis 的基础。

| 数据类型 | 说明 | 典型使用场景 |
|---|---|---|
| **String（字符串）** | 最基础类型，value 最大 512MB | 缓存简单值、session、计数器、分布式锁（SETNX） |
| **List（列表）** | 有序字符串列表，支持两端插入删除 | 消息队列、任务队列、最新N条记录、粉丝列表 |
| **Set（集合）** | 无序且不重复的字符串集合 | 标签系统、好友关系、去重、UV统计（配合 SINTER） |
| **Sorted Set（有序集合）** | 每个元素带分数，按分数排序 | 排行榜（ZREVRANGE）、延迟队列、实时排名 |
| **Hash（哈希）** | 键值对组成的 field-value映射 | 存储对象（如用户信息）、购物车、表结构数据 |
| **Bitmap（位图）** | 按位操作，适合做标识数组 | 用户签到、DAU统计、权限控制 |
| **HyperLogLog** | 概率数据结构，估算集合基数 | UV统计（海量数据去重计数） |
| **Streams（流）** | 类似日志的消息流，支持消费组 | 实时事件流、聊天系统、事件溯源 |

**示例：**

```bash
# String
SET user:1001:name "Alice"
GET user:1001:name

# Hash - 存储用户对象
HSET user:1001 age25 city "Shanghai"
HGETALL user:1001

# Sorted Set - 排行榜
ZADD leaderboard 1500 "Alice" 1200 "Bob" 1800 "Carol"
ZREVRANGE leaderboard 0 9 WITHSCORES
```

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 答案

Redis 虽以内存存储为主，但提供了两种持久化机制来防止数据丢失：

### RDB（Redis Database）

- **原理**：在指定时间间隔内，对内存数据生成一个快照（dump.rdb）保存到磁盘
- **触发方式**：
  - 定时自动触发（`save` 配置，如 `save 900 1`）
  - `BGSAVE` 手动异步执行
  - Redis 关闭时（通过 `shutdown` 命令）
- **优点**：恢复速度快，文件紧凑（全量数据），适合灾备
- **缺点**：可能丢失最近一次快照之后的数据

### AOF（Append Only File）

- **原理**：将每一次写操作追加到日志文件末尾，Redis 重启时重放这些操作恢复数据
- **配置**（`appendonly yes`），三种同步策略：
  - `appendfsync always`：每条写命令都同步，最安全但最慢
  - `appendfsync everysec`（**默认**）：每秒同步一次，折中方案
  - `appendfsync no`：由操作系统决定何时同步，最快但最不安全
- **优点**：数据安全性更高，支持日志重写（rewrite）压缩文件体积
- **缺点**：文件较大，恢复速度慢于 RDB

### 最佳实践：同时开启 RDB + AOF

```bash
# 推荐配置（redis.conf）
save 900 1
save 300 10
save 60 10000

appendonly yes
appendfsync everysec
```

> Redis 启动时，若 AOF 和 RDB 同时存在，优先加载 AOF（因为 AOF 更实时）。若 AOF 损坏，可先用 RDB 恢复再用 AOF 重放。

## 题目三：Redis 的缓存淘汰策略（Eviction Policy）有哪些？如何避免缓存击穿和缓存雪崩？

### 答案

当 Redis 内存达到 `maxmemory` 限制时，会根据配置的淘汰策略删除键。

### 淘汰策略分类

| 策略 | 说明 |
|---|---|
| `noeviction` | 不淘汰，返回错误（写操作会失败） |
| `allkeys-lru` | 所有键中删除最近最少使用（LRU）的键 **⭐最常用** |
| `volatile-lru` | 已设置 TTL 的键中删除 LRU 的键 |
| `allkeys-random` | 所有键中随机删除 |
| `volatile-random` | 已设置 TTL 的键中随机删除 |
| `volatile-ttl` | 已设置 TTL 的键中删除剩余 TTL 最短的键 |
| `allkeys-lfu`（Redis 4.0+） | 所有键中删除访问频率最低的 |
| `volatile-lfu`（Redis 4.0+） | 已设置 TTL 的键中删除访问频率最低的 |

### 缓存击穿（Cache Breakdown）

**问题**：热点 key 过期瞬间，大量请求穿透到数据库。

**解决方案：**
- 使用互斥锁（SETNX + EXPIRE），只允许一个请求回源
- 热点数据永不过期 + 异步更新
- 提前使用 `EXPIRE` 主动续期

### 缓存雪崩（Cache Avalanche）

**问题**：大量 key 同时过期，或 Redis 宕机，导致数据库瞬间压力暴增。

**解决方案：**
- 随机化 TTL：为 key 的过期时间加上随机偏移量（如 `TTL * random(0.5, 1.5)`）
-搭建 Redis 高可用集群（Sentinel / Cluster）
- 流量限流：数据库加保护熔断机制
- 多级缓存：本地缓存 + Redis 缓存兜底

### 缓存穿透（Cache Penetration）

**问题**：请求查询不存在的数据，每次都穿透到数据库。

**解决方案：**
- 布隆过滤器（BloomFilter）判断 key 是否存在
- 对空结果也做缓存（短期 TTL）
- 参数合法性校验

## 题目四：Redis Sentinel（哨兵）和 Redis Cluster 的区别是什么？如何选型？

### 答案

两者都是为了提高 Redis 的可用性，但解决的问题和使用场景不同。

### Redis Sentinel（哨兵）

- **定位**：高可用解决方案，适用于**主从复制**架构
- **核心功能**：
  -监控（Monitoring）：持续监控主从节点的健康状态
  - 自动故障转移（Automatic Failover）：主节点宕机时，自动将从节点提升为新主节点
  - 通知（Notification）：故障转移后通知客户端新的主节点地址
  - 配置提供（Configuration Provider）：客户端可通过哨兵获取当前主节点地址
- **架构**：需要至少 3 个哨兵节点（奇数，避免脑裂）
- **适用场景**：数据量不大，但需要高可用自动故障转移的业务

### Redis Cluster（集群）

- **定位**：分布式数据存储解决方案，支持**数据分片（Sharding）**
- **核心特性**：
  - 自动分片：16384 个 hash slot，自动分配到不同节点
  - 去中心化：无中心节点，客户端直连任意节点即可访问全部数据
  - 高可用：每个主节点可配置多个从节点（maste + N replica），自动故障转移
  - 扩展性：可水平扩容，增加节点分担数据和请求压力
- **限制**：
  - 不支持跨 slot 的事务（MULTI/EXEC 只能在同 slot 内）
  - 不支持跨 slot 的 join 操作
- **适用场景**：数据量大（超过单机内存上限）、高并发、需要水平扩展的业务

### 对比总结

| 对比项 | Sentinel | Cluster |
|---|---|---|
| 数据分片 | ❌ 不支持 | ✅ 支持（16384 slot） |
| 自动故障转移 | ✅ 支持 | ✅ 支持 |
| 水平扩容 | ❌ 不支持 | ✅ 支持 |
| 架构复杂度 | 较低 | 较高 |
| 适用数据量 | 小（单机可承载） | 大（需分片） |
| 客户端兼容性 | 好（单节点访问） | 需集群模式客户端 |

### 如何选型

- **数据量 < 10GB，高可用为主** → Sentinel（主从 + 哨兵）
- **数据量 > 10GB，或需要水平扩展** → Redis Cluster
- **既要高可用又要扩展** → Redis Cluster（每个分片配从节点）

## 题目五：Redis 和 Memcached 的区别是什么？为什么 Redis 更受欢迎？

### 答案

两者都是流行的内存缓存方案，但在设计理念、功能和适用场景上有显著差异。

### 核心区别

| 对比维度 | Redis | Memcached |
|---|---|---|
| **数据结构** | 丰富（String/Hash/List/Set/ZSet/Stream等） | 仅支持简单的 key-value 字符串 |
| **持久化** | 支持（RDB + AOF） | ❌ 不支持，纯内存 |
| **主从复制** | ✅ 支持完整的主从复制 | ✅ 支持，但功能有限 |
| **内存管理** | 自己实现的内存管理，支持 LRU/LFU 等 | 使用 slab allocator，简单但有碎片问题 |
| **线程模型** | 单线程 + I/O 多路复用（6.0后支持多线程 I/O） | 多线程（libevent） |
| **支持的语言** | 所有主流语言 | 所有主流语言 |
| **数据分区** | Cluster 原生支持 | 需客户端实现一致性哈希 |
| **Lua脚本** | ✅ 支持 | ❌ 不支持 |
| **Pub/Sub** | ✅ 支持 | ❌ 不支持 |
| **存储容量** | 单 value 最大 512MB | 单 value 最大 1MB |
| **内存效率** | 对于复杂数据结构更高效 | 对于简单 kv 更节省内存 |

### 为什么 Redis 更受欢迎？

1. **数据类型丰富**：一个 Redis 能顶 Memcached +消息队列 + 排行榜 + 实时统计等多个组件
2. **持久化能力**：Redis 可以将数据落盘，服务重启后不丢数据
3. **生态完善**：官方提供 Redis Cluster、Redis Sentinel、Redis Streams 等完整生态
4. **Lua 脚本**：支持原子性脚本操作，可在服务端执行复杂逻辑
5. **Pub/Sub 和 Streams**：内置消息队列能力，减少系统依赖

### 典型选型建议

- **纯缓存，场景简单**（如 session 缓存、简单页面缓存）→ Memcached 足够
- **需要复杂数据结构、持久化、消息队列、排行榜** → 必须选 Redis
- **生产环境推荐**：Redis（一站式解决方案，维护成本更低）
