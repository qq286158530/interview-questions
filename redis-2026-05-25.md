## 题目一：Redis 有哪些数据类型？分别适用于哪些场景？

### 参考答案

Redis 支持五种基础数据类型：

| 数据类型 | 底层实现 | 典型场景 |
|---------|---------|---------|
| **String** | SDS（简单动态字符串） | 缓存、计数器、分布式锁、session共享 |
| **Hash** | 压缩列表 or 字典 | 对象存储、购物车、商品信息 |
| **List** | 压缩列表 or 双向链表 | 消息队列、粉丝列表、文章评论 |
| **Set** | 整数集合 or 字典 | 标签系统、好友关系、抽奖、去重 |
| **Zset** | 压缩列表 or 跳表+字典 | 排行榜、延时队列、有序消息 |

此外 Redis 6.x+ 还支持 **Bitmaps**（位图）、**HyperLogLog**（基数统计）、**Geospatial**（地理位置）等扩展类型。

### 扩展知识

- String 类型最大可存储 **512MB** 数据
- Zset 的 score 用于排序，可用于实现延迟队列（时间戳作为 score）
- Hash 在 field 数量少时使用压缩列表以节省内存

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 参考答案

Redis 提供了两种持久化方式：

#### RDB（Redis Database）

- **原理**：在指定时间点生成数据集的快照（snapshot），fork 子进程完成
- **触发方式**：
  - `bgsave` 异步执行
  - 配置 `save m n`（m 秒内 n 次修改则触发）
  - `flushall`（空 RDB，慎用）
  - 主从复制时自动触发
- **优点**：恢复速度快，适合备份和灾难恢复
- **缺点**：最后一次 bgsave 之后的数据会丢失

#### AOF（Append Only File）

- **原理**：每个写命令追加到文件末尾
- **同步策略**（`appendfsync`）：
  - `always`：每条命令都 fsync，性能差但最安全
  - `everysec`（默认）：每秒同步一次
  - `no`：由操作系统决定何时同步
- **优点**：数据安全性更高，支持重写（rewrite）压缩文件
- **缺点**：文件通常比 RDB 大，恢复速度慢

### 生产环境建议

| 场景 | 推荐策略 |
|-----|---------|
| 仅缓存，数据可丢失 | 关闭持久化 或 RDB |
| 数据重要，允许一定丢失 | RDB + AOF 同时开启 |
| 追求高性能缓存 | AOF everysec + RDB |

> ⚠️ **注意**：Redis 4.0 之后支持混合持久化（`aof-use-rdb-preamble yes`），在 AOF 重写时使用 RDB 格式写入开头，兼顾两者优点。

## 题目三：Redis 的过期 key 删除策略是什么？

### 参考答案

Redis 采用 **惰性删除 + 定期删除** 组合策略：

#### 1. 惰性删除（Lazy Expire）

- **原理**：每次访问 key 时检查是否过期，过期则删除
- **优点**：节省 CPU 资源，只处理必须访问的 key
- **缺点**：过期 key 过多时占用内存

#### 2. 定期删除（Active Expire）

- **原理**：每隔一段时间（默认 100ms），随机抽取设置了过期时间的 key 检查删除
- **实现**：`activeExpireCycle()` 函数，分批、渐进式清理
- **优点**：缓解内存压力
- **缺点**：仍可能有漏网之鱼

### 内存淘汰策略（当内存满时）

| 策略 | 说明 |
|-----|------|
| `noeviction` | 拒绝写入（默认） |
| `volatile-lru` | 从过期 key 中 LRU 淘汰 |
| `allkeys-lru` | 从所有 key 中 LRU 淘汰 |
| `volatile-ttl` | 淘汰最近要过期的 key |
| `volatile-random` | 随机淘汰过期 key |
| `allkeys-random` | 随机淘汰所有 key |
| `volatile-lfu` | 从过期 key 中 LFU 淘汰 |
| `allkeys-lfu` | 从所有 key 中 LFU 淘汰 |

> 💡 **实际建议**：生产环境推荐 `volatile-lru` 或 `allkeys-lru`，并合理设置 `maxmemory`。

## 题目四：Redis 主从复制原理是什么？哨兵（Sentinel）机制了解吗？

### 参考答案

#### 主从复制流程

1. **建立连接**：从库发送 `PSYNC <runid> <offset>` 到主库
2. **全量同步**：主库 fork 子进程生成 RDB 快照，发送给从库
3. **增量同步**：同步期间主库的写命令记录在 repl buffer，从库加载完成后补充
4. **长连心跳**：从库每秒发送 `REPLCONF ACK <offset>` 保持连接

#### 主从复制风暴

大量从库同时复制会导致主库 fork + 传输压力，可以：
- 树形拓扑（分层主从）
- 调大 `repl-backlog-size`
- 使用 `replica-serve-stale-data` 适当降级

---

#### Sentinel 哨兵机制

哨兵是 Redis 的高可用方案，核心功能：

| 功能 | 说明 |
|-----|------|
| **监控** | 周期性检查主从库是否存活 |
| **自动故障转移** | 主库宕机时选举新主库 |
| **通知** | 故障转移后更新配置并通知客户端 |
| **配置提供者** | 客户端可通过哨兵获取当前主库地址 |

哨兵集群通过 **Raft 共识算法** 选举 leader，避免脑裂。

> ⚠️ **注意**：哨兵本身也有单点问题，生产环境至少部署 3 个节点。

## 题目五：如何实现一个分布式锁？Redis 分布式锁有哪些注意事项？

### 参考答案

#### 基础实现

```lua
-- 获取锁（SET NX EX 原子操作）
if redis.call('SET', KEYS[1], ARGV[1], 'NX', 'EX', ARGV[2]) then
    return 1
else
    return 0
end

-- 释放锁（Lua 脚本保证原子性）
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
```

#### Java 示例（Redisson 风格）

```java
String uuid = UUID.randomUUID().toString();
Boolean success = redisTemplate.opsForValue()
    .setIfAbsent("lock:order:" + orderId, uuid, 30, TimeUnit.SECONDS);
if (success) {
    try {
        // 业务逻辑
    } finally {
        // 释放锁（Lua 脚本）
        String script = "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                        "return redis.call('del', KEYS[1]) else return 0 end";
        redisTemplate.execute(new DefaultRedisScript<>(script), 
            Collections.singletonList("lock:order:" + orderId), uuid);
    }
}
```

### 分布式锁注意事项

| 问题 | 解决方案 |
|-----|---------|
| **锁过期，业务未完成** | 适当延长锁时间，或使用看门狗（watchdog）自动续期 |
| **误删他人锁** | 用唯一 value（如 UUID），释放前校验 |
| **主从切换丢锁** | 使用 RedLock（多 Redis 实例多数投票）|
| **可重入性** | value 存储线程信息 + 计数器 |
| **锁粒度** | 尽量小，避免锁住无关数据 |

### RedLock 算法

- 向 N 个独立 Redis 实例获取锁
- 过半成功才算获取成功
- 失败时向所有实例释放锁
- 适用于对可靠性要求极高的场景
