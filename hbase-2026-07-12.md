## 题目一：HBase 的数据存储结构是怎样的？请详细说明 HFile 的内部结构。

### 参考答案

HBase 数据最终以 **HFile** 的格式存储在 HDFS 上，每个 HFile 对应一个 Column Family 的部分数据。

#### HFile 内部结构（由多层索引组成）

```
┌─────────────────────────────────────────┐
│  HFile (Data File)                       │
├─────────────────────────────────────────┤
│  Data Block (数据块)                     │
│    - 每个 Block 默认 64KB/128KB          │
│    - 实际KV数据存储的位置                 │
│    - Scan 操作以 Block 为单位遍历         │
├─────────────────────────────────────────┤
│  Meta Block (元数据块)                   │
│    - 存储 BloomFilter 数据               │
│    - 用于快速判断某 RowKey 是否存在       │
├─────────────────────────────────────────┤
│  File Info (文件信息)                    │
│    - 文件级别元数据                       │
│    - 包含 Max SequenceId 等              │
├─────────────────────────────────────────┤
│  Index Block (索引块)                    │
│    - 两级索引：Data Block 索引 + Meta 索引│
│    - 每个 Data Block 有一条索引条目        │
│    - 索引按 Block 最后一条 Key 排序       │
├─────────────────────────────────────────┤
│  Trailer                                │
│    - 定长结构，指向各数据块起始位置         │
│    - 读取文件时最后读取                   │
└─────────────────────────────────────────┘
```

#### 关键概念

**1. Data Block**
- HBase 读写的基本单位，默认大小 64KB（可配置 `hfile.block.blocksize`）。
- Block 越小，索引越大，区间扫描越快；Block 越大，顺序扫描吞吐越高。

**2. BloomFilter**
- 存储在 Meta Block 中，用于快速判断某 RowKey + Column 是否存在于当前 HFile。
- 有效避免无效的磁盘 IO，是 HBase 读优化的关键机制。

**3. KeyValue 结构**
- HFile 中每条数据以 KeyValue 形式存储，格式为：
  ```
  KeyLength(4B) + ValueLength(4B) + RowKey + ColumnFamily + Column Qualifier + Timestamp + Type + Value
  ```

**4. SequenceId**
- 每个 HFile 的 Trailer 中记录最大 SequenceId，表示该文件写入的最大序列号，用于崩溃恢复时重放 WAL。

## 题目二：ZooKeeper 在 HBase 集群中扮演什么角色？为什么 HBase 离不开 ZooKeeper？

### 参考答案

ZooKeeper 是 HBase 集群的**分布式协调服务**，在整个 HBase 架构中起到了元数据管理、故障检测和分布式锁的作用。

#### ZooKeeper 的核心作用

**1. 元数据存储与路由**
- HBase 将 `-ROOT-` 表和 `.META.` 表的位置信息注册到 ZooKeeper 中。
- 客户端首先连接 ZooKeeper 获取 `META` 表位置，再定位到具体的 RegionServer，无需通过 HMaster 获取元数据（避免 HMaster 成为瓶颈）。

**2. HMaster 高可用选举**
- HMaster 在 ZooKeeper 中创建临时节点，通过选举机制确保同一时刻只有一个 Active HMaster。
- 当 Active HMaster 宕机时，Standby HMaster 感知到节点消失，重新竞争创建临时节点成为新的 Active。

**3. RegionServer 健康检测**
- 每个 RegionServer 启动时在 ZooKeeper 中注册临时节点，并定期发送心跳。
- HMaster 监听 ZooKeeper 的子节点列表，一旦检测到某 RegionServer 节点消失（即宕机），立即触发 Region 迁移。

**4. 分布式锁与协调**
- 表的 schema 修改（如 alter table）需要先在 ZooKeeper 上获取分布式锁。
- 控制并发操作，避免多 HMaster 同时修改表结构导致数据不一致。

#### ZooKeeper 节点结构示例
```
/hbase
  /master          # HMaster 地址
  /region-server   # 各 RegionServer 临时节点
  /meta-region-server  # .META. 表所在的 RegionServer
  /table           # 表的锁节点
```

#### 为什么 HBase 离不开 ZooKeeper？
- **去中心化**：客户端不依赖 HMaster 获取元数据，HMaster 宕机不影响已连接客户端的读写。
- **故障自愈**：RegionServer 宕机能在秒级被检测到并自动迁移 Region。
- **强一致性保证**：ZooKeeper 的 ZAB 协议保证分布式锁的原子性。

## 题目三：HBase 中 Region 是什么时候分裂的？分裂流程是怎样的？Split Policy 有哪些？

### 参考答案

#### Region 分裂时机

当一个 Region 达到以下条件时会触发自动分裂：
- **大小阈值**：Region 中某个 Store 的 HFile 大小超过 `hbase.hregion.max.filesize`（默认 10GB）。
- 分裂由 RegionServer发起，而非 HMaster 控制。

#### 分裂流程

```
1. RegionServer 决定分裂某个 Region
2. 在 ZooKeeper 创建临时节点 /hbase/region-in-transition/{region-name}，标记 SPLITTING 状态
3. 关闭该 Region（停止读写服务），强制刷写 MemStore 到 HFile
4. 在 HDFS 上创建两个子 Region 的目录，将原 Region HFile 链接（Reference）到子 Region
5. 在 .META. 表中更新元数据：标记原 Region 为 SPLIT，两个新 Region 上线
6. 删除 ZooKeeper 临时节点
7. HMaster 感知到新 Region 上线，进行负载均衡调度
```

#### Reference 文件机制
- 分裂瞬间，原 HFile 并不物理复制，而是创建两个 **Reference 文件**指向原 HFile 的不同偏移范围。
- 后续 Compaction 时会真正合并/分裂数据。

#### Split Policy（分裂策略）

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| `IncreasingToUpperBoundRegionSplitPolicy`（默认） | 分裂阈值随 Region 数量动态调整，从小到大逐步放宽 | 通用场景，避免小表过早分裂 |
| `ConstantSizeRegionSplitPolicy` | 分裂阈值固定为 `hbase.hregion.max.filesize` | 大表场景 |
| `KeyPrefixRegionSplitPolicy` | 按 RowKey 前缀分组，保证同一前缀的数据在同一 Region | 有明确热点前缀的场景 |
| `DelimitedKeyPrefixRegionSplitPolicy` | 按指定分隔符前的 Key 前缀分裂 | 需要自定义前缀分割的场景 |

## 题目四：HBase 的 Compaction 机制是什么？Minor Compaction 和 Major Compaction 有何区别？

### 参考答案

#### Compaction 概述

Compaction 是 HBase 后台将多个小 HFile 合并成大 HFile 的过程，用于减少文件数量、提升读性能，并清理已标记删除的数据。

#### 两种 Compaction 类型

**1. Minor Compaction（小合并）**
- 由 RegionServer 后台线程自动触发（每 `hbase.server.compactor.thread.run.delay` 毫秒检查一次）。
- 将**少数几个**小 HFile（通常 < 10 个）合并成一个较大的 HFile。
- **不会**删除标记为删除的数据（Tombstone Marker）和过期版本。
- 合并后的 HFile 数量减少，但总量基本不变。

**2. Major Compaction（大合并）**
- 默认**每天**自动执行一次（可通过 `hbase.hregion.majorcompaction` 配置）。
- 将一个 Store 下**所有** HFile 合并成一个 HFile。
- **会删除**所有 Tombstone Marker 和过期版本数据，真正释放磁盘空间。
- 合并过程会产生大量 IO 和网络流量（读 HDFS + 写 HDFS），通常安排在业务低峰期。

#### 对比表

| 特性 | Minor Compaction | Major Compaction |
|------|-----------------|-----------------|
| 触发频率 | 较频繁（分钟~小时级） | 较低（通常每天一次） |
| 合并文件数 | 少量（< 10 个） | 全部 HFile |
| 删除数据清理 | ❌ 不清理 | ✅ 清理 |
| IO 压力 | 较小 | 较大 |
| 读优化效果 | 有限 | 显著 |

#### Compaction 参数调优

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `hbase.hstore.compaction.min` | 3 | Minor Compaction 最少 HFile 数 |
| `hbase.hstore.compaction.max` | 10 | Minor Compaction 最多 HFile 数 |
| `hbase.hregion.majorcompaction` | 86400000ms (1天) | Major Compaction 间隔 |

## 题目五：如何对 HBase 进行读写性能优化？列举常见的优化策略。

### 参考答案

HBase 读写性能优化的核心思路是：**减少 IO、避免热点、提升缓存命中率**。以下是常见的优化策略：

#### 读性能优化

**1. 合理设计 RowKey（减少跨 Region 扫描）**
- 避免全表扫描，RowKey 设计应支持**点查*和**范围查*。
- 热点 RowKey 会导致单 Region 压力过大，读延迟升高。

**2. 启用并合理配置 BlockCache**
- 读请求会先查 MemStore（写缓存），再查 BlockCache（读缓存），最后才读 HFile。
- `hfile.block.cache.size` 默认占总堆内存 40%，热点读场景可适当调大。

**3. 合理使用 BloomFilter**
- 开启 BloomFilter（`BloomFilterType.ROW` 或 `ROW.COLUMN`），快速判断 Key 是否存在，避免无效磁盘 IO。
- 代价是增加少量内存和存储开销。

**4. 避免读取大量版本**
- HBase 默认保留 3 个版本，`MAX_VERSIONS` 设置过大会增加 IO。
- 按需配置列族的版本数：`ALTER 'table', NAME => 'cf', VERSIONS => 1`。

**5. 使用 Covering Index（协处理器优化）**
- 通过 Endpoint 协处理器在服务器端完成聚合计算，减少数据传输量。

#### 写性能优化

**1. 批量写入（Bulk Load）**
- 亿级数据导入使用 **HBase BulkLoad**（MapReduce 生成 HFile 直接加载），绕过 MemStore 写入路径，吞吐率提升 10 倍以上。

**2. 关闭 WAL（高吞吐场景）**
- 对于可丢失的数据（原始日志等），可关闭 WAL（`setWriteToWAL(false)`）提升写入速度 3~5 倍。
- 风险：机器宕机会丢失未持久化数据。

**3. 合理配置 MemStore**
- `hbase.hregion.memstore.flush.size` 默认 128MB，过小会导致频繁刷写，过大会导致内存压力。
- 监控 `memstore.flushedCellsCount` 和 `memstore.compactedCellsCount`，找到平衡点。

**4. 避免热点 RowKey**
- 对递增 RowKey（时间戳等）加盐（Salting）或 Hash 打散，使写入分布到多个 RegionServer。

**5. 调整 Compaction 策略**
- 写入高峰期可临时调高 Compaction 并发度：`hbase.hstore.compaction.max`。
- Major Compaction 安排在低峰期，避免 IO 竞争。

#### 常见配置参数速查

| 参数 | 默认值 | 优化建议 |
|------|--------|----------|
| `hfile.block.blocksize` | 65536 | 顺序扫描多可调大；随机点查多可调小 |
| `hfile.block.cache.size` | 0.4 | 读多写少可调高；写多读少可调低 |
| `hbase.hregion.memstore.flush.size` | 134217728 | 内存充裕时适当调大 |
| `hbase.regionserver.handler.count` | 30 | 并发读多时调大 |
