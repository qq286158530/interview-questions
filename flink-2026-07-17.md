## 题目一：Flink 运行架构核心组件及职责

### 📋 题目
请描述 Apache Flink 的运行架构，Flink 集群中有哪些核心角色？各自承担什么职责？

### ✅ 参考答案

Flink 采用 **Master-Slave 架构**，主要包含以下核心组件：

#### 1. JobManager（Master 节点）
- **职责**：负责整个集群的调度和任务协调
  - 接收用户提交的 Flink 作业（Job）
  - 协调 Job 的执行，包括任务分配、资源调度
  - 监控 TaskManager 的心跳和状态
  - 管理 Checkpoint 和故障恢复
  - 包含三个核心组件：
    - **Dispatcher**：接收作业提交，启动 JobManager
    - **ResourceManager**：管理 TaskManager 的 slot 资源
    - **JobMaster**：负责单个 Job 的执行调度

#### 2. TaskManager（Worker 节点）
- **职责**：执行具体的计算任务
  - 启动后向 ResourceManager 注册自己的 slot 资源
  - 接收 JobManager 分配的 task 并执行
  - 负责数据的 shuffle、broadcast 等操作
  - 与其他 TaskManager 交换数据
  - 存储状态数据（RocksDB 或 Heap）

#### 3. Slot（资源单元）
- TaskManager 的最小资源单元
- 每个 TaskManager 有多个 slot（通常 4-8 个）
- 每个 slot 可以运行一个或多个 subtask
- Slot 数量决定 TaskManager 的并行处理能力

#### 4. 作业提交流程
```
Client → Dispatcher → JobManager → ResourceManager(申请slot) → TaskManager(注册slot) → JobMaster(分发task)
```

## 题目二：Watermark 机制与事件时间处理

### 📋 题目
什么是 Watermark（水位线）？它在 Flink 中起到什么作用？如何处理乱序和迟到数据？

### ✅ 参考答案

#### Watermark 定义
Watermark 是 Flink 用于处理**事件时间（Event Time）**的机制，本质是一个时间戳，用于标记「所有 event_time ≤ watermark 的事件都已到达」。

#### 核心作用

**1. 标识事件时间进度**
- 没有 Watermark，事件时间窗口不知道何时触发
- Watermark 推动事件时间窗口的触发和计算

**2. 解决乱序问题**
- 允许一定程度的乱序或迟到事件被正确处理

#### Watermark 生成策略

**① 有界无序（Boundless Out-of-Order）**
```java
WatermarkStrategy
    .forBoundedOutOfOrderness(Duration.ofSeconds(30))
```
- 取当前最大事件时间 - 固定延迟时间
- 最常用的策略

**② 周期性（Periodic）**
- 定期生成 watermark（如每 N 条数据或每 M 毫秒）

**③ 标点式（Punctuated）**
- 根据特定事件触发 watermark 生成

#### 迟到数据处理策略

**① Allowed Lateness（允许延迟）**
```java
.window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .allowedLateness(Time.minutes(2))
```
- 窗口触发后，仍等待迟到数据 N 分钟
- 迟到数据会触发窗口增量计算

**② Side Output（侧输出）**
```java
.Output<Tag, String>(tag)
```
- 将超过 allowed lateness 的数据路由到侧输出流
- 用于收集「真正迟到」的数据单独处理

**③ 窗口延迟触发**
- 只有 watermark 到达窗口结束时间 + allowed lateness 后，窗口才真正删除

#### 面试加分点
- Watermark 是单流标记，跨流 join 时需要考虑多流 watermark 对齐问题
- 实际生产中，watermark 延迟设置需要根据数据特点调优（太长增加延迟，太短可能丢数据）

## 题目三：Checkpoint 与状态恢复机制

### 📋 题目
Flink 的 Checkpoint 机制是什么？它和 Savepoint 有什么区别？如何配置生产级的 Checkpoint 参数？

### ✅ 参考答案

#### Checkpoint 机制
Flink 通过 **Chandy-Lamport 分布式快照算法**实现 Checkpoint，定期将所有 Task 的状态快照到远程存储（分布式文件系统），实现故障恢复。

#### Checkpoint vs Savepoint

| 特性 | Checkpoint | Savepoint |
|------|------------|------------|
| 触发方式 | 自动（由 Flink 自动定时触发） | 手动（用户命令触发） |
| 用途 | 故障自动恢复 | 计划性维护、版本升级 |
| 清理策略 | Job 停止时自动删除 | 需要手动删除 |
| 兼容性 | 可能因代码变更不兼容 | 具备更好的向前兼容性 |
| 存储位置 | 状态后端配置的目录 | 用户指定目录 |

#### Checkpoint 配置参数（生产环境推荐）

```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

// 启用 Checkpoint，间隔 1 分钟
env.enableCheckpointing(60 * 1000);

// 精确一次语义
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);

// Checkpoint 超时时间
env.getCheckpointConfig().setCheckpointTimeout(10 * 60 * 1000);

// 最小间隔（避免频繁 Checkpoint）
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30 * 1000);

// 最多同时执行 1 个 Checkpoint
env.getCheckpointConfig().setMaxConcurrentCheckpoints(1);

// 开启增量 Checkpoint（RocksDB 推荐）
env.getCheckpointConfig().setPreferCheckpointForRecovery(true);
```

#### 状态后端选择

| 状态后端 | 适用场景 | 状态大小 |
|----------|----------|----------|
| **MemoryStateBackend** | 本地开发调试 | 小状态 |
| **FsStateBackend** | 小状态、短窗口 | 中等状态（≤TM内存） |
| **RocksDBStateBackend** | 大状态、长窗口、大键值 | TB 级别 |

#### 增量 Checkpoint（RocksDB）
- 只存储上次 Checkpoint 以来的增量变更
- 大幅减少 Checkpoint 数据量和时间
- 适合超大状态场景（计算 DAU 等）

## 题目四：窗口函数详解

### 📋 题目
Flink 支持哪些窗口类型？各自的适用场景是什么？请实现一个会话窗口（Session Window）统计用户行为。

### ✅ 参考答案

#### 窗口类型总览

```
┌─────────────────────────────────────────────────────────────┐
│                        Window Types                          │
├─────────────┬─────────────┬─────────────┬───────────────────┤
│  Time Windows│ Count Windows│ Session Windows│ Custom Windows │
├─────────────┼─────────────┼─────────────┼───────────────────┤
│ Tumbling    │ Tumbling    │ EventTime   │ Global           │
│ Sliding     │ Sliding     │ Processing  │ Windows          │
│ Session     │             │ Session     │                  │
└─────────────┴─────────────┴─────────────┴───────────────────┘
```

#### 1. Tumbling Window（滚动窗口）
- **特点**：窗口长度固定，不重叠
- **适用**：固定周期的聚合统计
```java
// 每 5 分钟统计一次
timeWindow(TumblingEventTimeWindows.of(Time.minutes(5)))
```

#### 2. Sliding Window（滑动窗口）
- **特点**：窗口长度固定，可重叠
- **适用**：需要「平滑」聚合或重叠分析
```java
// 窗口长 10 分钟，滑动步长 5 分钟
timeWindow(SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(5)))
```

#### 3. Session Window（会话窗口）
- **特点**：动态窗口，由不活跃间隔触发关闭
- **适用**：用户行为分析、会话拆分
```java
// 不活跃间隔 30 分钟则关闭窗口
window(EventTimeSessionWindows.withGap(Time.minutes(30)))
```

#### 会话窗口实现示例

```java
public class SessionWindowExample {
    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = 
            StreamExecutionEnvironment.getExecutionEnvironment();
        
        DataStream<Event> events = env.addSource(new KafkaSource<>());
        
        // 按用户 ID 分组，使用会话窗口
        events.keyBy(e -> e.getUserId())
            .window(EventTimeSessionWindows.withGap(Time.minutes(30)))
            .process(new ProcessWindowFunction<Event, SessionResult, String, TimeWindow>() {
                @Override
                public void process(String userId, 
                                   Context context, 
                                   Iterable<Event> elements,
                                   Collector<SessionResult> out) {
                    // 统计该会话内的事件数和行为列表
                    List<Event> eventList = new ArrayList<>();
                    for (Event e : elements) {
                        eventList.add(e);
                    }
                    
                    long startTime = eventList.get(0).getEventTime();
                    long endTime = eventList.get(eventList.size()-1).getEventTime();
                    long duration = endTime - startTime;
                    
                    out.collect(new SessionResult(
                        userId, 
                        eventList.size(), 
                        duration,
                        startTime,
                        endTime
                    ));
                }
            })
            .print();
        
        env.execute();
    }
}
```

#### 窗口函数类型

| 类型 | 特点 | 适用场景 |
|------|------|----------|
| **ProcessWindowFunction** | 全量数据，Context 丰富 | 需要遍历所有数据、访问窗口信息 |
| **ReduceFunction** | 增量聚合，性能高 | 简单聚合（求和、计数） |
| **AggregateFunction** | 增量聚合，接口更灵活 | 复杂聚合逻辑 |

## 题目五：Flink 状态管理核心面试题

### 📋 题目
Flink 的状态分为哪几类？如何选择合适的状态后端？请说明 keyed-state 和 operator-state 的区别和适用场景。

### ✅ 参考答案

#### 状态分类

```
┌─────────────────────────────────────────────────────────────┐
│                        Flink State                            │
├─────────────────────────────┬───────────────────────────────┤
│           Keyed State         │         Operator State        │
├─────────────────────────────┼───────────────────────────────┤
│ ValueState<T>               │ ListState<T>                   │
│ MapState<K, V>              │ UnionListState<T>              │
│ ListState<T>                │ BroadcastState<K, V>          │
│ ReducingState<T>            │                                 │
│ AggregatingState<IN, OUT>  │                                 │
└─────────────────────────────┴───────────────────────────────┘
```

#### Keyed State vs Operator State

| 维度 | Keyed State | Operator State |
|------|-------------|----------------|
| **作用域** | Key 绑定，按 key 分区 | 算子级别，所有 key 共享 |
| **使用位置** | keyBy 之后的算子 | 任何算子（含 source） |
| **并行度变更** | 按 Key-Group 自动重分配 | 可配置 rescale 策略 |
| **创建方式** | `getRuntimeContext().getState()` | 实现 `CheckpointedFunction` |

#### 状态后端对比

| 状态后端 | 存储介质 | 读写性能 | 适用场景 |
|----------|----------|----------|----------|
| **MemoryStateBackend** | JVM 堆内存 | 最快 | 开发调试 |
| **FsStateBackend** | TaskManager 内存 + 远程文件系统 | 快 | 小状态、短窗口 |
| **RocksDBStateBackend** | 本地 RocksDB + 远程文件系统 | 较慢（序列化） | 大状态、长窗口、超大 key |

#### 生产环境选择建议

```java
// 小状态（< 1GB）选 FsStateBackend
env.setStateBackend(new FsStateBackend("hdfs:///checkpoints"));

// 大状态（> 10GB）选 RocksDBStateBackend
env.setStateBackend(new RocksDBStateBackend("hdfs:///checkpoints"));

// 开启增量 Checkpoint（仅 RocksDB 支持）
((RocksDBStateBackend) env.getStateBackend())
    .enableIncrementalCheckpointing(true);
```

#### 状态 TTL 配置

```java
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.days(7))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)  // 读写时更新
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)  // 不返回过期数据
    .cleanupInRocksdbCompactFilter(1000)  // RocksDB 增量清理
    .build();

ValueStateDescriptor<String> descriptor = 
    new ValueStateDescriptor<>("myState", String.class);
descriptor.enableTimeToLive(ttlConfig);
```

#### 面试高频追问

**Q：为什么 ValueState 不建议存大 Map？**
- A：RocksDB 存储时，ValueState 的 value 会整体序列化/反序列化
- 大 Map 会导致严重的序列化开销，影响性能
- 正确做法：使用 `MapState` 替代

**Q：Checkpoint 恢复时状态如何分配到新并行度？**
- A：
  - Keyed State：按 Key-Group 自动映射（Key 不变，Group 重新分配）
  - Operator State（ListState）：均匀划分到各 subtask
  - Operator State（UnionListState）：每 subtask 收到完整全量
