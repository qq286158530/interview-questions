## 题目一：Redis 和 Memcached 有什么区别？为什么 Redis 选择作为 MySQL 的缓存？

### 答案

**Redis 与 Memcached 的共同点：**
- 都是基于内存的数据库，一般都用来当做缓存使用
- 都有过期策略
- 两者的性能都非常高

**Redis 与 Memcached 的区别：**

| 特性 | Redis | Memcached |
|------|-------|-----------|
| 支持的数据类型 | 丰富（String、Hash、List、Set、ZSet） | 仅支持 key-value |
| 数据持久化 | 支持 RDB 和 AOF 两种持久化方式 | 无持久化功能 |
| 集群模式 | 原生支持集群模式 | 需依赖客户端实现分片 |
| 高级特性 | 支持发布订阅、Lua脚本、事务等 | 不支持 |

**为什么用 Redis 作为 MySQL 的缓存？**

主要是因为 Redis 具备「高性能」和「高并发」两种特性：

1. **高性能**：用户第一次访问 MySQL 中的数据较慢（从硬盘读取），将数据缓存到 Redis 后，下一次访问直接从内存获取，操作速度极快。

2. **高并发**：单台设备的 Redis 的 QPS（每秒处理请求数）是 MySQL 的 10 倍。Redis 单机 QPS 能轻松破 10w，而 MySQL 单机很难破 1w。因此可以把部分数据库请求转移到缓存层，减轻数据库压力。

## 题目二：Redis 的持久化机制有哪些？RDB 和 AOF 有什么区别？

### 答案

Redis 共有三种数据持久化方式：

### 1. AOF 日志（Append Only File）

每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里。Redis 重启时，会读取该文件记录的命令，逐一执行来恢复数据。

**AOF 写回策略：**
- **Always**：每次执行完命令后同步将 AOF 日志写回硬盘（最安全，但性能最差）
- **Everysec**：每秒将命令写入 AOF 文件的内核缓冲区，然后由操作系统决定何时写入硬盘（默认策略，兼顾性能和安全）
- **No**：由操作系统控制写回时机（性能最好，但数据安全性最低）

**AOF 重写机制**：当 AOF 文件过大时，Redis 会启用后台子进程（bgrewriteaof）重写 AOF 文件，压缩文件体积。

### 2. RDB 快照（Redis Database）

将某一时刻的内存数据，以二进制的方式写入磁盘。Redis 提供了两个命令：
- `save`：在主线程生成 RDB 文件，会阻塞主线程
- `bgsave`：创建子进程来生成 RDB 文件，不阻塞主线程

**RDB 快照配置触发条件：**
```
save 900 1      # 900秒内至少1次修改
save 300 10     # 300秒内至少10次修改
save 60 10000   # 60秒内至少10000次修改
```

### 3. 混合持久化（Redis 4.0+）

结合了 RDB 和 AOF 的优点。AOF 重写时，先将以 RDB 格式写入文件前半部分，后半部分记录增量 AOF 命令。

**RDB 和 AOF 对比：**

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件体积 | 小（二进制） | 大（文本命令） |
| 恢复速度 | 快 | 慢 |
| 数据完整性 | 可能丢失较多数据 | 可配置丢失少 |
| 性能影响 | bgsave不影响主线程 | Always模式影响大 |

## 题目三：Redis 过期删除策略和内存淘汰策略有什么区别？

### 答案

这是两个完全不同的概念：

### 过期删除策略（针对已过期的 key）

Redis 会扫描所有带过期时间的 key，每秒进行多次采样删除：

1. **惰性删除**：key 被访问时检查是否过期，过期则删除。优点是对 CPU 友好，缺点是过期 key 可能一直未被访问。

2. **定期删除**：每隔一段时间，Redis 会扫描带有过期时间的 key，随机抽取一部分检查，删除其中过期的 key。

Redis 默认每秒进行 10 次采样删除，配置参数 `hz` 可调整频率。

### 内存淘汰策略（针对内存不足时的 key）

当 Redis 内存使用达到 `maxmemory` 限制时，会触发内存淘汰策略：

**8 种淘汰策略：**

| 策略 | 说明 |
|------|------|
| `volatile-lru` | 从已设置过期时间的 key 中，淘汰最近最少使用的 |
| `volatile-ttl` | 从已设置过期时间的 key 中，淘汰最早过期的 |
| `volatile-random` | 从已设置过期时间的 key 中，随机淘汰 |
| `allkeys-lru` | 从所有 key 中，淘汰最近最少使用的（最常用） |
| `allkeys-random` | 从所有 key 中，随机淘汰 |
| `no-eviction` | 不淘汰，返回错误（默认） |
| `volatile-lfu` | 从已设置过期时间的 key 中，淘汰使用频率最低的 |
| `allkeys-lfu` | 从所有 key 中，淘汰使用频率最低的 |

**推荐配置**：对于缓存场景，通常使用 `allkeys-lru` 或 `volatile-lru`。

## 题目四：什么是缓存雪崩、击穿、穿透？如何解决？

### 答案

### 1. 缓存雪崩

**问题描述**：大量缓存数据在同一时间过期，或者 Redis 缓存层突然宕机，导致大量请求直接打到数据库，造成数据库压力骤增甚至崩溃。

**解决方案**：
- 使用 Redis 高可用架构（主从复制、哨兵模式、集群）
- 设置不同的过期时间，避免大量 key 同时过期
- 使用双缓存策略，备份一份缓存
- 对数据库访问进行限流保护

### 2. 缓存击穿

**问题描述**：某个热点 key 突然过期，导致大量请求直接打到数据库。

**解决方案**：
- 对热点 key 设置永不过期
- 使用互斥锁，只允许一个线程去查询数据库并更新缓存
- 使用逻辑过期方式（返回旧数据，后台异步更新）

### 3. 缓存穿透

**问题描述**：请求的数据既不在缓存中，也不在数据库中（比如恶意攻击），导致每次请求都打到数据库。

**解决方案**：
- 对请求参数进行合法性校验
- 使用布隆过滤器（BloomFilter）判断数据是否存在
- 对于不存在的数据，在缓存中设置空值或默认值
- 使用 Redis 的 `BITCOUNT` 或专门的黑名单机制

## 题目五：Redis 主从复制、哨兵模式、集群模式有什么区别？

### 答案

### 1. 主从复制（Master-Slave）

**原理**：主服务器进行读写操作，从服务器只读并同步主服务器的数据。

- 主服务器收到写命令后，异步发送给从服务器
- 主从服务器之间的数据复制是异步的，无法做到强一致性
- 采用「读写分离」模式，主服务器处理写操作，从服务器处理读操作

**优点**：实现 Redis 的读写分离，提高并发能力
**缺点**：主服务器宕机后，需要手动恢复，无法自动故障转移

### 2. 哨兵模式（Sentinel）

**原理**：在主从复制基础上，增加哨兵进程监控主从服务器的运行状态。

**哨兵的功能**：
- 监控：定期检查主从服务器是否正常运行
- 自动故障转移：当主服务器宕机时，哨兵自动选举从服务器为新的主服务器
- 通知：向管理员发送故障通知
- 配置管理：提供新的主服务器地址给客户端

**优点**：实现了高可用，无需手动干预故障恢复
**缺点**：只有一个主服务器，存储能力受单机限制

### 3. 集群模式（Cluster）

**原理**：Redis 3.0 引入的分布式方案，使用数据分片（16384 个槽位）将数据分布在多个节点上。

**特点**：
- 每个节点保存一部分数据，理论上可以水平扩展
- 每个节点都可以作为主节点，负责一部分槽位
- 支持自动故障转移
- 客户端可以连接任意节点，自动路由到正确的节点

**优点**：
- 数据分布式存储，解决单机存储容量限制
- 支持高并发写入
- 自动故障转移，高可用

**三种模式对比**：

| 特性 | 主从复制 | 哨兵模式 | 集群模式 |
|------|---------|---------|---------|
| 数据冗余 | 有 | 有 | 有 |
| 自动故障转移 | 无 | 有 | 有 |
| 横向扩展 | 无 | 无 | 有 |
| 写能力 | 单机 | 单机 | 多机 |
| 存储能力 | 单机 | 单机 | 分片 |
