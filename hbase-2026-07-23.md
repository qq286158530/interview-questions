## 题目一：HBase 的整体架构是怎样的？请描述 RegionServer、Region、HMaster 的作用及数据读写流程。

### 答案

HBase 采用 **Master-Slave 架构**，核心组件如下：

| 组件 | 作用 |
|------|------|
| **HMaster** | 负责表的 CRUD 操作（创建、删除、修改表结构），以及 RegionServer 的负载均衡、故障恢复、Schema 变更。不参与实际数据读写 |
| **RegionServer** | 负责管理 Region，处理客户端的读写请求。每个 RegionServer 包含多个 Region 和一个 WAL（Write-Ahead Log） |
| **Region** | 表的数据分片，按 RowKey 范围划分，每个 Region 负责一段行键区间 |
| **ZooKeeper** | 负责选举 Master、存储元数据位置（如 .META. 表所在的 RegionServer 地址），实现服务发现与协调 |

### 写数据流程

1. 客户端连接 ZooKeeper，获取 `.META.` 表所在的 RegionServer 地址
2. 从 `.META.` 表中查询目标 RowKey 所在的 RegionServer
3. 将数据写入 **WAL**（Write-Ahead Log，保证宕机可恢复）
4. 将数据写入 **MemStore**（内存缓冲区，达到阈值后 flush 成 HFile）
5. 返回写入成功

### 读数据流程

1. 同样先查 ZooKeeper → `.META.` 表获取目标 RegionServer
2. 在 BlockCache（读缓存）中查找
3. 若未命中，查询 MemStore
4. 若仍未命中，通过 BloomFilter 定位 HFile，读取数据并写入 BlockCache 返回

## 题目二：HBase 的存储结构是怎样的？请解释 MemStore、HFile、Store、StoreFile 之间的关系。

### 答案

HBase 数据从内存到磁盘的存储层次如下：

```
RegionServer
 └── Region（按 RowKey 范围分片）
      └── Store（每个列族对应一个 Store）
           ├── MemStore（写缓存，LSM-Tree 结构）
           │     ↓ flush（达到阈值时）
           └── StoreFile（HFile on HDFS）
                └── HFile（实际的存储文件格式）
```

### 各层详解

**MemStore**
- 写缓存，LSM-Tree（Log-Structured Merge-Tree）的核心思想
- 数据先写入 MemStore，按 RowKey 排序，积累到阈值（默认 128MB）后 flush 成 HFile
- 每个列族有独立的 MemStore

**HFile**
- HBase 在 HDFS 上的实际存储格式，源于 BigTable 的 SSTable
- 结构：Data Block → Meta Block → File Info → Data Index → Meta Index → Trailer
- 包含 BloomFilter（快速判断 key 是否存在）和 Data Index（加速 Block 定位）
- 每个 HFile 有对应的 Major Compaction（合并所有 HFile 去重）或 Minor Compaction（合并部分小 HFile）

**LSM-Tree 思想**
- 写操作顺序追加 WAL + 写入 MemStore，顺序写性能高
- 读操作需要合并 MemStore 和所有 HFile（最新数据在 MemStore 中）
- 通过 Compaction 定期合并小文件，减少读放大

## 题目三：HBase 的 RowKey 设计有哪些原则？如何避免热点问题（Hotspotting）？

### 答案

### RowKey 设计原则

1. **长度原则**：RowKey 长度不宜过长（建议 10~100 字节），过长会占用 HBase 的两个存储空间（MemStore 和 HFile）
2. **唯一性**：RowKey 必须唯一标识一行数据
3. **散列性**：RowKey 的前缀应尽量分散，避免数据集中到少数 RegionServer 上

### 避免热点的策略

| 策略 | 说明 |
|------|------|
| **加盐（Salting）** | 在 RowKey 前拼接随机前缀（如 0~9），让写请求分散到不同 Region，但会降低读效率 |
| **哈希（Hashing）** | 对 RowKey 做 MD5 或其他哈希，取哈希值的前缀作为 RowKey 的一部分，保证分布均匀 |
| **反转（Reversing）** | 将时间戳等固定前缀反转，让时间跨度大的数据分散到不同 Region |
| **时间戳反转** | 若查询总是针对最新数据，可将 `Long.MAX_VALUE - timestamp` 作为 RowKey 后缀，保证新数据在前面 |

### 典型场景示例

```
# 热点：用户ID作为前缀，时间戳在后 → 同一用户的数据总在一个 Region
hot_rowkey = "user123_20260723180000"

# 改进：哈希打散
good_rowkey = "hash(user123)_20260723180000"

# 更优：加盐
better_rowkey = "salt_user123_20260723180000"  # salt = userId.hashCode() % 10
```

## 题目四：HBase 的读写性能如何优化？请从读缓存、写缓存、Compaction 三个方面说明。

### 答案

### 读性能优化

**1. BlockCache（读缓存）**
- LRU（Least Recently Used）策略，默认占 RegionServer 堆内存的 40%
- 热点数据放在 BlockCache 中可显著降低读延迟
- 优化方式：合理设计 RowKey 使数据访问具有局部性，避免全表扫描污染缓存

**2. BloomFilter**
- 每个 HFile 内置 BloomFilter，用于快速判断某个 RowKey 是否存在该文件中
- 避免不存在的 RowKey 触发全表扫描，大幅减少 IO

**3. 读路径优化**
- 避免 `scan.setFilter()` 使用正则过滤，尽量在客户端过滤
- 使用 `scan.setCaching(int)` 批量取数据，减少 RPC 次数（但过大会占内存）

### 写性能优化

**1. WAL 优化**
- 写入时先写 WAL 保证持久化，可通过 `setDurability(Durability.SKIP_WAL)` 跳过 WAL提升吞吐量（但丢失风险增加，适合批量导入）
- 使用 `BulkLoad` 绕过 WAL，直接生成 HFile 写入 HDFS，适合离线批量导入

**2. MemStore 配置**
- `hbase.hregion.memstore.flush.size` 控制 flush 阈值（默认 128MB）
- `hbase.hregion.memstore.mslab.enabled` 开启 MSLAB 减少内存碎片

**3. 预分区（Pre-splitting）**
- 建表时指定 `SPLITS` 或 `SPLITS_FILE`，提前将表切分为多个 Region，避免自动 split 带来的性能抖动

### Compaction 优化

| 类型 | 说明 |
|------|------|
| **Minor Compaction** | 合并部分小 HFile，不删除过期数据 |
| **Major Compaction** | 合并所有 HFile，删除过期版本（TTL）和被删除的数据，默认 7 天触发一次 |
| **优化建议** | 生产环境建议关闭自动 Major Compaction，改为手动在业务低峰期触发，避免 IO 抖动影响在线服务 |

## 题目五：HBase 协处理器（Coprocessor）是什么？Observer 和 Endpoint 有什么区别？

### 答案

HBase 协处理器是扩展 HBase 功能的机制，允许用户在 RegionServer 上运行自定义代码，实现业务逻辑的下推。

### 两种协处理器类型

#### 1. Observer（观察者，类似触发器）
- 类似于数据库的触发器，在特定事件前后自动执行
- 三个核心钩子：
  - **PrePut / PostPut**：在 Put 写入前/后执行，可用于数据验证、权限检查、自动化关联写入
  - **PreGet / PostGet**：在 Get 查询前/后执行，可用于动态列过滤、访问控制
  - **PreDelete / PostDelete**：在删除前/后执行，可用于级联删除关联数据

```java
// 示例：使用 Observer 实现写入自动记录审计日志
public class AuditObserver extends BaseRegionObserver {
    @Override
    public void postPut(ObserverContext<RegionCoprocessorEnvironment> e,
                        Put put, WALEdit edit) throws IOException {
        // 记录谁在什么时间写了什么数据
        System.out.println("Audit: Put on row " + Bytes.toString(put.getRow()));
    }
}
```

**使用场景**：数据验证、访问控制、审计日志、增量复制、关联表更新

#### 2. Endpoint（端点，类似存储过程）
- 类似于数据库的存储过程，由客户端主动调用，在 RegionServer 上执行计算并将结果返回
- 必须实现 `CoprocessorProtocol` 接口
- 适合将计算移动到数据所在节点，减少数据传输（类似 MapReduce 的计算向数据靠拢）

```java
// 示例：自定义 Endpoint 实现跨列聚合
public class MyAggregator implements CoprocessorProtocol {
    // 在 RegionServer 上执行聚合计算
    public long sumColumn(byte[] cf, byte[] qual) { ... }
}
```

**使用场景**：聚合查询（如求和、计数、最大值）、复杂条件统计、跨行跨列计算

### Observer vs Endpoint 对比

| 维度 | Observer | Endpoint |
|------|----------|----------|
| 触发方式 | 自动（事件驱动） | 客户端主动调用 |
| 用途 | 数据验证、过滤、关联更新 | 分布式计算、聚合查询 |
| 执行位置 | RegionServer | RegionServer |
| 结果返回 | 不返回，副作用 | 返回计算结果给客户端 |

### 配置方式

```xml
<!-- 在 hbase-site.xml 中配置 -->
<property>
  <name>hbase.coprocessor.region.classes</name>
  <value>com.example.AuditObserver,com.example.MyEndpoint</value>
</property>
```

或通过 Shell 建表时指定：
```bash
create 't1', 'cf1', {METHOD => 'table_att', 'COPROCESSOR' => 'hdfs:///path/to/coprocessor.jar|com.example.AuditObserver|1001'}
```
