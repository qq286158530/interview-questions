## 题目一：Spark 架构核心——Driver 与 Executor 是如何协同工作的？

### 参考答案

Spark 采用 **Master-Slave 架构**，核心组件包括 Driver、Executor、Cluster Manager 和 Worker。

**协作流程：**

1. **提交阶段**：用户通过 `spark-submit` 向 Cluster Manager（如 Standalone、YARN、K8s）提交 Application，Cluster Manager 分配资源并启动 Driver 进程。
2. **初始化阶段**：Driver 执行用户代码，创建 `SparkContext`，向 Cluster Manager 申请 Executor 资源。
3. **任务分发**：Driver 将应用程序代码和依赖发送给 Executor，同时将 Job 拆分为多个 Stage，Stage 再拆分为 Task，以 **TaskSet** 为单位分发到各个 Executor 的线程池中并行执行。
4. **结果回收**：Executor 将 Task 执行结果通过 Block Manager 存储或通过网络传回 Driver，Driver 汇总后触发后续 Stage 或结束 Job。
5. **资源释放**：Application 结束后，Cluster Manager 回收所有 Executor 资源。

**关键点：**
- **Driver 是整个 Spark 应用的"大脑"**，负责 DAG 切分、任务调度、结果汇总。
- **Executor 是"工人"**，每个 Executor 拥有独立的 JVM 进程和内存槽位（core），负责执行具体 Task。
- **通信层面**：Driver 与 Executor 之间通过 `CoarseGrainedExecutorBackend` 进行 RPC 通信；Executor 之间的 Shuffle 数据传输依赖 `BlockStoreShuffleClient`。

## 题目二：RDD 的五大特性是什么？为什么 RDD 是粗粒度的？

### 参考答案

**RDD（R eliable Distributed Dataset）五大特性：**

| 序号 | 特性 | 说明 |
|------|------|------|
| ① | **分区列表（Partitions）** | 每个 RDD 包含多个分区，分区是并行计算的基本单位 |
| ② | **计算函数（Compute Function）** | 每个分区上都有 compute(val: T) => Iterator[T] 函数，用于执行具体计算 |
| ③ | **依赖关系（Dependencies）** | RDD 之间存在 NarrowDependency（窄依赖，一对一/范围）或 ShuffleDependency（宽依赖，一对多） |
| ④ | **分区器（Partitioner）** | 可选，对于 Key-Value RDD，指定哈希/范围分区策略 |
| ⑤ | **分区位置列表（Preferred Locations）** | 可选，记录每个分区的"最优位置"，实现数据本地性调度 |

**为什么是粗粒度？**

RDD 的转换操作（如 `map`、`filter`、`flatMap`）只记录**血统（Lineage）**和转换规则，**不会立即执行**。直到遇见 Action 操作（如 `collect`、`count`、`saveAsTextFile`）时，才从头到尾一次性触发计算。这种"延迟计算（Lazy Evaluation）"使得 Spark 能够：
- **优化执行计划**：Action 触发前，Spark 会进行Pipeline 优化、算子下推等。
- **节省资源**：避免中间结果立即落盘或物化。
- **容错高效**：通过 Lineage 重算丢失的分区，而非复制全量数据。

## 题目三：DataFrame 和 DataSet 的区别是什么？DataFrame = DataSet[Row] ？

### 参考答案

**核心区别：**

| 维度 | DataFrame | DataSet |
|------|-----------|---------|
| **数据类型** | 纯 Row 记录（类似关系型表） | 强类型 JVM 对象（Case Class） |
| **编译时检查** | 无（运行时才发现类型错误） | 有（编译期即检查） |
| **语法糖** | DSL 风格，`df.filter("age > 30")` | Lambda + 对象操作，`ds.filter(_.age > 30)` |
| **序列化** | 依赖 Tungsten / UnsafeRow（字节码） | 使用 Encoder 自动序列化 |
| **操作灵活性** | 支持 SQL、DSL | 支持 Lambda + SQL |
| **向后兼容** | — | `DataSet[Row]` 即为 DataFrame |

**是的，DataFrame = DataSet[Row]**，在 Scala 中：
```scala
type DataFrame = DataSet[Row]  // 语法别名
```
DataFrame 是弱类型的 Row 集合，适合处理结构化/半结构化数据；DataSet 是强类型的，适合需要编译期类型安全的业务逻辑。

**Spark 2.x 统一了 API**：`spark.read.json` 返回 DataFrame，`spark.read.parquet` 返回 DataFrame，内部都基于 Spark SQL 的 Catalyst 优化器和 Tungsten 内存管理。

## 题目四：Shuffle 机制原理是什么？为什么 Shuffle 是 Spark 性能瓶颈？

### 参考答案

**Shuffle 本质**：将具有相同 Key 的数据重新分区到同一个 Partition（通常在不同节点），以便后续聚合（Aggregation）、连接（Join）或分组（GroupBy）操作能够正确执行。

**执行流程（以 WordCount 为例）：**

```
Stage 0 (Map Stage):
  各节点 Executor 执行 map()，输出 (word, 1)
       ↓
  Shuffle Write: 每个 Mapper 将数据写入本地磁盘，按 Reduce Partition 编号分文件
       ↓
  Spark 在 Shuffle 过程中对同一批次的输出文件进行**排序（Sort-based Shuffle）**
       ↓
Stage 1 (Reduce Stage):
  Reduce 节点通过 HTTP/Fetch 请求拉取（Shuffle Read）属于自己分区的数据块
       ↓
  对拉取的数据进行排序/聚合，输出结果
```

**为什么是性能瓶颈？**

1. **磁盘 I/O**：Shuffle 数据先写磁盘再读磁盘，涉及大量序列化/反序列化（Tungsten）。
2. **网络 I/O**：跨节点数据传输，同一 Key 的数据可能在不同机器，需要多次网络拉取。
3. **内存压力**：Shuffle Read 时需要将数据放入内存的 hash map 或 sorter，大数据量易触发 OOM。
4. **Hadoop Shuffle vs Spark Shuffle**：Spark ShuffleManager 可插拔，默认 SortShuffleManager 比 HashShuffle 节省文件描述符，但仍无法避免磁盘 I/O。
5. **Reduce 端数据倾斜**：某 Key 数据量过大导致个别任务执行时间远超其他（Data Skew），可使用 `salting` 或 `广播小表` 优化。

## 题目五：Spark 广播变量（Broadcast）是什么？如何用它优化 join 性能？广播的内存模型是怎样的？

### 参考答案

**广播变量原理：**

广播变量是只读的、缓存在每个 Executor 内存中的共享变量。Driver 端将数据打包后一次性发送给各 Executor（而非每个 Task 单独发送），Executor 收到后存在 `BlockManager` 中供该节点上所有 Task 共用。

```scala
// 定义广播变量
val broadcastMap = sc.broadcast(smallTableAsMap)

// 在算子中使用
val result = largeRDD.map(row => {
  val lookupTable = broadcastMap.value  // 读取广播数据
  lookupTable.getOrElse(row.key, "default")
})
```

**join 性能优化：**

| 方式 | 通信量 | 适用场景 |
|------|--------|----------|
| 普通 `large.join(small)` | 每条 large 数据都发送一次 key 到 reduce 端 | 默认，无优化 |
| `broadcast(small)` | small 只发一次到各 Executor，large 数据不动 | 小表 build 大表 probe（维表关联） |
| `map-side join` | 无 Shuffle，数据在各自节点完成关联 | 两表都在内存且一张很小 |

**内存模型（Spark 2.x+ Tungsten）：**

```
Executor 堆内存结构:
┌─────────────────────────────────────────────┐
│  Unified Memory (堆内)                       │
│  ┌──────────────────┬───────────────────┐    │
│  │  Execution Memory │  Storage Memory    │    │
│  │  (Shuffle/Sort等) │  (Cache/广播变量)   │    │
│  │  动态借用，反之亦然                    │    │
│  └──────────────────┴───────────────────┘    │
└─────────────────────────────────────────────┘
```

广播变量存在 **Storage Memory** 区域，使用 `BroadcastManager` + `TorrentBroadcast`（分块传输）实现。
- `spark.broadcast.blockSize`：控制每个广播块大小（默认 4MB）。
- `spark.sql.autoBroadcastJoinThreshold`：默认 10MB，决定 Spark SQL 何时自动广播小表。

**常见问题：**
- 广播数据超过 `spark.sql.autoBroadcastJoinThreshold` 或 Executor 内存不足时，Spark 会回退到 Sort-Merge Join。
- 广播变量不可修改，需更新时重新 `broadcast()`。
