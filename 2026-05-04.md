## 题目1：Java 深拷贝与浅拷贝的区别？如何实现深拷贝？

**【答案】**

### 基本概念

- **浅拷贝（Shallow Copy）**：只复制对象本身的字段值，对于引用类型字段，复制的是引用地址，而非引用对象本身。因此原对象和拷贝对象会共享内部的引用对象。
- **深拷贝（Deep Copy）**：不仅复制对象本身，还递归复制所有引用类型的成员变量。复制后的对象与原对象完全独立，互不影响。

### 示例说明

```java
// 浅拷贝示例
class Person implements Cloneable {
    String name;
    Address address; // 引用类型

    @Override
    protected Person clone() throws CloneNotSupportedException {
        return (Person) super.clone(); // 浅拷贝
    }
}

// 深拷贝方式1：重写clone方法
@Override
protected Person clone() throws CloneNotSupportedException {
    Person p = (Person) super.clone();
    p.address = this.address.clone(); // 递归拷贝引用对象
    return p;
}

// 深拷贝方式2：序列化（推荐）
public DeepCopyable deepCopy() {
    try {
        ByteArrayOutputStream bos = new ByteArrayOutputStream();
        ObjectOutputStream oos = new ObjectOutputStream(bos);
        oos.writeObject(this);
        ByteArrayInputStream bis = new ByteArrayInputStream(bos.toByteArray());
        ObjectInputStream ois = new ObjectInputStream(bis);
        return (DeepCopyable) ois.readObject();
    } catch (Exception e) {
        throw new RuntimeException(e);
    }
}

// 深拷贝方式3：JSON序列化
public <T> T deepCopyViaJson(T obj, Class<T> clazz) {
    return new ObjectMapper().convertValue(obj, clazz);
}
```

### 区别对比

| 特性 | 浅拷贝 | 深拷贝 |
|------|--------|--------|
| 引用类型字段 | 共享引用对象 | 独立复制对象 |
| 拷贝层次 | 只拷贝一层 | 递归拷贝所有层级 |
| 对象独立性 | 与原对象共享内部引用 | 完全独立 |
| 实现复杂度 | 简单（Object.clone()） | 复杂（需递归或序列化） |
| 性能 | 高效 | 较差（序列化有开销） |

### 面试追问

- **Object 的 clone() 为什么需要实现 Cloneable 接口？**  
  `Cloneable` 是一个标记接口，不实现它而调用 `clone()` 会抛 `CloneNotSupportedException`。它的存在本身就是一种"我允许被克隆"的选择加入（opt-in）机制。

- **所有对象都能用序列化做深拷贝吗？**  
  不行。需要所有成员变量的类也实现了 `Serializable`，且包含的引用类型都能序列化。

## 题目2：Java 中的 wait()、notify()、notifyAll() 原理是什么？为什么必须在同步块中调用？

**【答案】**

### 核心原理

这三个方法是 `Object` 类的方法，是 Java 内置的**管程（Monitor）** 机制，用于线程间协调。

**关键机制：**
- 每个 Java 对象都关联一个**监视器锁（Monitor）** 和一个**等待队列（Wating Queue）**。
- 当线程执行 `object.wait()` 时，线程会释放该对象的监视器锁，并加入该对象的等待队列。
- `notify()` 从等待队列中随机唤醒一个线程（`notifyAll()` 唤醒所有），被唤醒的线程需要重新竞争对象的监视器锁，获得后才能继续执行。

### 代码示例

```java
synchronized (lock) {
    while (condition不满足) {
        lock.wait(); // 释放锁，进入等待队列
    }
    // 执行任务
    lock.notifyAll(); // 唤醒等待的线程
}
```

### 为什么必须在同步块中调用？

1. **防止死锁与不可预期行为**：如果 `wait()` 和 `notify()` 不在同步块中，无法保证 `wait()` 和 `notify()` 之间操作的原子性，可能出现"线程错过了信号"（Missed Signal）问题。

2. **与监视器语义绑定**：这些方法的语义就是基于对象监视器工作的，调用时必须持有该对象的锁，才能对等待队列进行安全的入队/出队操作。

3. **API 设计要求**：`IllegalMonitorStateException` 如果不在同步块中调用会直接抛出。

### 与 Lock 的 Condition 对比

```java
// ReentrantLock + Condition 方式
ReentrantLock lock = new ReentrantLock();
Condition condition = lock.newCondition();

lock.lock();
try {
    while (!condition) {
        condition.await(); // 等待
    }
    // 业务逻辑
    condition.signalAll(); // 唤醒
} finally {
    lock.unlock();
}
```

`Condition.await()/signal()` 的优势在于一个锁可以创建多个条件队列，比 `wait()/notify()` 更灵活，适合多线程精确协作场景。

## 题目3：Spring Bean 的生命周期是怎样的？

**【答案】**

### 整体流程

Spring Bean 生命周期从容器启动开始，到 Bean 销毁结束，主要分为以下阶段：

### 1. 实例化（Instantiation）
- Spring 依据配置（XML、注解、JavaConfig）实例化 Bean 对象。
- 通过 `BeanPostProcessor.postProcessBeforeInstantiation()` 可以在实例化前进行拦截。

### 2. 属性填充（Populate）
- Spring 完成依赖注入（`@Autowired`、`@Resource`、`setter` 注入等）。
- 引用其他 Bean 或简单值被注入到当前 Bean 的属性中。

### 3. 初始化（Initialization）

**初始化回调顺序：**
1. `BeanNameAware.setBeanName()` — 注入 Bean 名称
2. `BeanFactoryAware.setBeanFactory()` — 注入 BeanFactory
3. `ApplicationContextAware.setApplicationContext()` — 注入 ApplicationContext
4. `BeanPostProcessor.postProcessBeforeInitialization()` — 初始化前处理
5. `@PostConstruct` 标注的方法（初始化前）
6. `InitializingBean.afterPropertiesSet()` — 初始化回调
7. 自定义 `init-method`（XML 或 `@Bean(initMethod="...")`）
8. `BeanPostProcessor.postProcessAfterInitialization()` — 初始化后处理

### 4. 销毁（Destruction）

**销毁回调顺序：**
1. `@PreDestroy` 标注的方法
2. `DisposableBean.destroy()`
3. 自定义 `destroy-method`

### 代码示例

```java
@Component
public class UserService implements InitializingBean, DisposableBean {

    @PostConstruct
    public void init() {
        System.out.println("@PostConstruct 初始化");
    }

    @Override
    public void afterPropertiesSet() throws Exception {
        System.out.println("InitializingBean 初始化");
    }

    public void customInit() {
        System.out.println("custom init-method");
    }

    @PreDestroy
    public void preDestroy() {
        System.out.println("@PreDestroy 销毁");
    }

    @Override
    public void destroy() throws Exception {
        System.out.println("DisposableBean 销毁");
    }
}
```

### 面试追问

- **BeanPostProcessor 和 BeanFactoryPostProcessor 区别是什么？**  
  前者作用于每个 Bean 的创建过程（实例化/初始化前后），后者作用于 Spring 容器本身（在 Bean 定义加载完成后、Bean 实例化之前）。

- **Bean 作用域（Scope）会影响生命周期吗？**  
  `prototype` 作用域下，容器只负责创建和属性填充，不负责销毁。容器关闭时不会调用销毁方法。

## 题目4：Spring Cloud Feign 远程调用的原理是什么？

**【答案】**

### Feign 概述

Feign 是 Netflix 开发的声明式 HTTP 客户端，通过接口+注解的方式发起 HTTP 调用，让 HTTP 远程调用像本地方法调用一样简单。

### 核心原理

#### 1. 启动时的处理

- `@EnableFeignClients` 启用 Feign 扫描。
- `@FeignClient` 注解的接口被 `FeignClientFactoryBean` 捕获。
- Spring 为每个接口创建一个 **JDK 动态代理对象**（`InvocationHandler` 为 `ReflectiveFeign.FeignInvocationHandler`）。

#### 2. 代理处理流程

当调用 Feign 接口方法时：

```
客户端调用接口方法
    ↓
JDK 动态代理拦截（FeignInvocationHandler）
    ↓
根据 @RequestLine 等注解解析请求信息
    ↓
从 Hystrix 或 Resilience4j 获取命令执行
    ↓
使用 HttpURLConnection / OkHttp / Ribbon 等发送 HTTP 请求
    ↓
编码器（Encoder）序列化参数为 JSON/表单
    ↓
发送请求到目标服务
    ↓
解码器（Decoder）反序列化响应
    ↓
返回结果给调用方
```

#### 3. 关键组件

| 组件 | 作用 |
|------|------|
| `Contract` | 解析 @RequestLine、@Headers 等注解，生成 `MethodHandler` |
| `MethodHandler` | 封装请求模板和执行逻辑 |
| `Encoder / Decoder` | 请求体的序列化和响应的反序列化 |
| `Client` | 实际发起 HTTP 请求的组件（可替换 OkHttp、HttpClient） |
| `RequestInterceptor` | 请求拦截器，可统一添加 Header 等 |
| `LoadBalancerInterceptor` | 集成 Ribbon，实现客户端负载均衡 |

#### 4. 与 Ribbon 的集成

```java
// Feign 自动集成 Ribbon，通过拦截 URL 占位符实现负载均衡
@FeignClient(name = "user-service")
public interface UserClient {
    @GetMapping("/user/{id}")
    User getUser(@PathVariable("id") Long id);
}
```

Ribbon 从注册中心（如 Eureka/Nacos）获取 `user-service` 的实例列表，通过负载均衡策略选择目标机器。

#### 5. 常见面试追问

- **Feign 和 OpenFeign 的区别？**  
  OpenFeign 是 Feign 的社区升级版，支持 Spring MVC 注解（如 `@GetMapping`），并在 Spring Cloud 2.x 后取代了 Feign。

- **Feign 请求超时如何配置？**  
  ```yaml
  feign:
    client:
      config:
        default:
          connectTimeout: 5000
          readTimeout: 5000
  ```

## 题目5：Spring Cloud Gateway 是如何实现动态路由与过滤器链的？

**【答案】**

### 核心概念

Spring Cloud Gateway 基于 **Spring WebFlux**（响应式编程）和 **Project Reactor**，使用 **Netty** 作为服务器实现非阻塞 I/O。

### 路由（Route）工作机制

路由是 Gateway 的核心概念，由三个部分组成：
- **Predicate（断言）**：判断请求是否匹配（路径、时间、Header 等）
- **Filter（过滤器）**：对请求/响应进行预处理和后处理
- **URI**：目标服务地址

### 动态路由实现

#### 1. 路由定义

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service  # lb:// 表示从注册中心获取
          predicates:
            - Path=/api/user/**
          filters:
            - StripBar=1  # 去掉第一层路径
```

#### 2. 动态更新（无需重启）

```java
// 通过 RouteDefinitionWriter 动态添加路由
RouteDefinition definition = new RouteDefinition();
definition.setId("dynamic-route");
definition.setUri(URI.create("http://example.com"));
// 设置 predicates 和 filters...
routeDefinitionWriter.save(Mono.just(definition)).subscribe();
```

还可以监听 Nacos/Apollo 等配置中心的变更事件，自动刷新路由。

### 过滤器链（Filter Chain）原理

#### 过滤类型

| 类型 | 说明 | 执行顺序 |
|------|------|----------|
| **PRE** | 请求转发前执行（如鉴权、限流） | 数字越小越靠前 |
| **POST** | 响应返回前执行（如添加 Header、日志） | 数字越小越靠后 |

#### 内置过滤器

```yaml
filters:
  - StripBar=1                    # 路径去除前缀
  - AddRequestHeader=X-Request-Foo # 添加请求头
  - AddResponseHeader=X-Response-Bar
  - Retry=3                       # 重试3次
  - RequestRateLimiter=10,20      # 限流（每秒10请求，突发20）
```

#### 自定义过滤器

```java
@Component
public class AuthFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (token == null) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        return chain.filter(exchange);
    }
}
```

### 请求处理流程

```
请求 → Netty → Gateway Handler Mapping（匹配路由）
                        ↓
                 Gateway Web Handler（执行过滤器链 PRE）
                        ↓
                 Proxy Exchange（转发到下游服务）
                        ↓
                 Gateway Web Handler（执行过滤器链 POST）
                        ↓
                 响应 → Netty → 客户端
```

### 与 Zuul 1.x 的区别

| 特性 | Zuul 1.x | Gateway |
|------|----------|---------|
| 线程模型 | 同步阻塞（Servlet） | 响应式非阻塞（Netty） |
| 长连接 | 不支持 | 支持 |
| 性能 | 较低 | 高 |
| 路由匹配 | 基于 Servlet 路径映射 | 基于 Predicate 断言 |

## 题目6：Kafka 如何保证消息不丢失？

**【答案】**

Kafka 通过三重保障机制，从生产者、Broker、消费者三个环节全面保证消息不丢失。

### 1. 生产者端：不丢消息

#### acks 配置

```java
// acks=0：发送即成功，不等待确认（最快，丢风险大）
// acks=1：等待 leader 副本写入成功确认
// acks=all/-1：等待 ISR（in-sync replicas）全部写入确认
producer.send(record, new Callback() {
    @Override
    public void onCompletion(RecordMetadata metadata, Exception exception) {
        if (exception != null) {
            // 发送失败，触发重试或记录告警
        }
    }
});
```

#### 重试机制 + 幂等性

```yaml
# 生产者配置
enable.idempotence=true          # 开启幂等性
retries=3                        # 重试次数
max.in.flight.requests.per.connection=5  # 防止乱序
```

开启 `enable.idempotence=true` 后，Kafka 会为每个 Producer 分配 PID（Producer ID），并为每条消息分配序列号，从而实现 **Exactly Once** 的生产语义。

### 2. Broker 端：持久化存储

#### 副本机制（Replication）

- 每个 Partition 有多个副本，分散在不同的 Broker 上。
- 只有 **ISR（In-Sync Replicas）** 中的副本才被视为"已同步"。
- ISR 定义：`replica.lag.max.messages=4000`（跟随者最大落后消息数）和 `replica.lag.time.max.ms=10000`（最大落后时间）。

#### 持久化配置

```yaml
# Broker 配置
min.insync.replicas=2           # ISR 中最少副本数
unclean.leader.election.enable=false  # 禁止从落后太多的副本中选举 leader
```

#### 日志落盘

Kafka 使用 **顺序写** + **操作系统页缓存**，默认配置下消息先写入页缓存（内存），由 OS 异步刷盘。追求极致可靠时可配置：

```yaml
log.flush.interval.messages=1   # 每条消息都刷盘（性能很差，不推荐）
```

### 3. 消费者端：不丢消息

#### 手动提交 Offset

```java
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        // 处理消息
        process(record);
    }
    // 手动提交 offset，确保处理完再提交
    consumer.commitSync();
}
```

**最佳实践**：处理完消息后再提交 offset。如果处理成功但提交失败（下次重启会重复消费），业务侧需要做幂等处理（如唯一键/去重表）。

### 总结

| 环节 | 保证措施 |
|------|----------|
| 生产者 | acks=all + 幂等性 + 重试 |
| Broker | 多副本 + ISR + 持久化 |
| 消费者 | 手动提交 offset + 业务幂等 |

## 题目7：RabbitMQ 惰性队列（Lazy Queue）是什么？适用于哪些场景？

**【答案】**

### 什么是惰性队列

惰性队列（Lazy Queue）是 RabbitMQ 3.6+ 引入的特性。普通队列将消息存储在内存中以保证高性能；而惰性队列**将消息尽可能存储在磁盘上**，只在消费者需要时才从磁盘加载到内存。

### 工作原理

```
普通队列：
  生产消息 → 内存队列（高性能） → 消费者拉取

惰性队列：
  生产消息 → 磁盘文件（mmap 映射） → 消费者拉取 → 内存缓存
```

惰性队列将消息写入磁盘文件，通过 OS 的 `mmap`（内存映射）机制映射到虚拟地址空间。当消费者需要消费时，数据从磁盘加载到内存。

### 配置方式

```java
// 声明时设置
Map<String, Object> args = new HashMap<>();
args.put("x-queue-mode", "lazy");
channel.queueDeclare("my-lazy-queue", true, false, false, args);
```

```yaml
# Spring Boot 配置
spring:
  rabbitmq:
    queue:
      my-queue:
        mode: lazy
```

### 适用场景

| 场景 | 说明 |
|------|------|
| **消息堆积场景** | 生产速度 > 消费速度，堆积大量消息时避免 OOM |
| **偶发性消费** | 消息积压几天后才被消费的批处理场景 |
| **内存敏感** | 内存有限的 Broker，需要保护内存不被消息占满 |
| **大消息** | 单条消息较大（如几十 MB），不宜大量堆积在内存 |

### 优缺点

| 特性 | 惰性队列 | 普通队列 |
|------|----------|----------|
| 内存占用 | 低 | 高 |
| 吞吐量 | 较低（磁盘 I/O） | 高 |
| 消息堆积能力 | 极强（磁盘空间内理论上无限） | 受内存限制 |
| 消费延迟 | 略高（需从磁盘加载） | 低（内存直接消费） |

### 注意事项

- 惰性队列第一次被消费者消费时，如果消息在磁盘上，加载可能带来瞬时 I/O 压力。
- 磁盘空间耗尽可能导致 RabbitMQ 宕机，需要监控磁盘使用情况。
- 与 `x-max-length`（队列最大长度）配合使用时，先入队的老消息会被丢弃而非从磁盘加载。

### 面试追问

- **惰性队列和普通队列可以互相转换吗？**  
  不能，需要删除重建。当前 RabbitMQ 不支持动态修改队列的 `x-queue-mode`。

## 题目8：RAG（检索增强生成）的工作流程和核心组件是什么？

**【答案】**

### RAG 概述

RAG = Retrieval-Augmented Generation（检索增强生成），是一种结合外部知识库检索与大模型生成能力的技术架构，用于解决大模型"幻觉"和"知识过时"问题。

### 核心工作流程

```
用户问题
   ↓
1. Embedding（向量化）
   ↓
2. 检索（Retrieval）→ 从知识库中找出 Top-K 相关文档
   ↓
3. Context 组装（将检索结果注入 Prompt）
   ↓
4. LLM 生成（Generation）→ 基于上下文生成答案
   ↓
最终回答
```

### 核心组件详解

#### 1. Document Loader（文档加载器）

负责从多种数据源加载原始文档（PDF、Word、网页、数据库等），提取文本内容。

#### 2. Text Splitter / Chunking（文本分块）

将长文档切分成适合检索的小块（Chunk）。切分策略直接影响检索质量：

```python
# 常用策略：按固定长度重叠切分
text = "这是一个很长的文档..."
chunks = text_splitter.split_text(text, chunk_size=500, overlap=50)
```

切分粒度太粗：噪声多；太细：上下文不完整。

#### 3. Embedding Model（向量化模型）

将文本转换为高维向量（Embedding），用于语义相似度计算。常用模型：
- OpenAI `text-embedding-ada-002`
- 国产：`BGE`、`M3E`、`JinaAI`
- 开源：`sentence-transformers`

#### 4. Vector Store（向量数据库）

存储 Embedding 向量并支持最近邻检索。常用选型：

| 数据库 | 特点 |
|--------|------|
| **Pinecone** | 云原生托管，按使用付费 |
| **Milvus** | 开源，支持海量向量 |
| **Chroma** | 轻量级，适合本地快速验证 |
| **FAISS** | Facebook 开源，GPU 加速 |
| **Weaviate** | 支持混合检索（向量+关键词） |

#### 5. Retrieval（检索器）

从向量数据库中检索与用户问题最相关的 Top-K 文档块。常用检索策略：

- **相似度检索**：基于余弦相似度或欧氏距离
- **混合检索**：向量检索 + BM25 关键词检索融合
- **重排序（Reranker）**：用更精确的模型（如 `bge-reranker`）对初筛结果重新排序

#### 7. LLM（大语言模型）

from langchain_community.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter

from langchain_community.vectorstores import Chroma

## 题目9：LeetCode 42 — 接雨水（Hard）

**【答案】**

### 题目描述

给定 `n` 个非负整数表示每个宽度为 1 的柱子的高度图，计算按此排列的柱子下雨后能接多少雨水。

### 解法一：双指针（最优解）- O(n) 时间

```python
def trap(height: list[int]) -> int:
    if not height:
        return 0

    left, right = 0, len(height) - 1
    left_max, right_max = 0, 0
    water = 0

    while left < right:
        if height[left] < height[right]:
            # 优先处理矮的那一侧
            if height[left] >= left_max:
                left_max = height[left]
            else:
                water += left_max - height[left]
            left += 1
        else:
            if height[right] >= right_max:
                right_max = height[right]
            else:
                water += right_max - height[right]
            right -= 1

    return water
```

**核心思想**：每个位置能接的雨水 = `min(左侧最高, 右侧最高) - 当前高度`。用双指针从两端向中间收缩，左右指针只处理各自方向的"矮边"，保证每次都是最优子结构。

### 解法二：单调栈 - O(n) 时间

```python
def trap(height: list[int]) -> int:
    stack = []  # 存索引
    water = 0

    for i in range(len(height)):
        while stack and height[i] > height[stack[-1]]:
            bottom_idx = stack.pop()
            if not stack:
                break
            width = i - stack[-1] - 1
            bounded_height = min(height[i], height[stack[-1]]) - height[bottom_idx]
            water += width * bounded_height
        stack.append(i)

    return water
```

**核心思想**：维护一个递减单调栈。当遇到比栈顶更高的柱子时，形成低洼（凹陷），计算能接的雨水量。

### 解法三：暴力 + 预计算 - O(n) 时间和空间

```python
def trap(height: list[int]) -> int:
    n = len(height)
    left_max = [0] * n
    right_max = [0] * n

    # 预处理左侧最高
    for i in range(1, n):
        left_max[i] = max(left_max[i-1], height[i-1])

    # 预处理右侧最高
    for i in range(n-2, -1, -1):
        right_max[i] = max(right_max[i+1], height[i+1])

    # 计算雨水
    water = 0
    for i in range(n):
        min_height = min(left_max[i], right_max[i])
        if min_height > height[i]:
            water += min_height - height[i]

    return water
```

### 复杂度对比

| 解法 | 时间复杂度 | 空间复杂度 |
|------|-----------|-----------|
| 双指针 | O(n) | O(1) |
| 单调栈 | O(n) | O(n) |
| 暴力+预计算 | O(n) | O(n) |

### 测试用例

```python
assert trap([0,1,0,2,1,0,1,3,2,1,2,1]) == 6
assert trap([4,2,0,3,2,5]) == 9
assert trap([]) == 0
```

## 题目10：LeetCode 146 — LRU 缓存（Design）

**【答案】**

### 题目描述

设计并实现一个 LRU（最近最少使用）缓存数据结构，支持 `get` 和 `put` 操作，且两个操作的时间复杂度都必须是 **O(1)**。

### 数据结构选型

- **哈希表（HashMap）**：O(1) 查找 key
- **双向链表（Doubly Linked List）**：O(1) 维护访问顺序

两者结合：HashMap 存 key → 链表节点，链表按访问时间从新到旧排列。

### 完整实现

```python
class LRUCache:

    class Node:
        def __init__(self, key=0, val=0):
            self.key = key
            self.val = val
            self.prev = None
            self.next = None

    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = {}  # key -> Node

        # 哨兵节点（简化边界处理）
        self.head = self.Node()  # 最近使用端（伪头部）
        self.tail = self.Node()  # 最久未使用端（伪尾部）
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node: 'LRUCache.Node'):
        """从链表中移除节点"""
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_head(self, node: 'LRUCache.Node'):
        """插入到最近使用端"""
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        node = self.cache[key]
        # 访问后移到最近端
        self._remove(node)
        self._add_to_head(node)
        return node.val

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            # 更新已有节点
            node = self.cache[key]
            node.val = value
            self._remove(node)
            self._add_to_head(node)
        else:
            # 新建节点
            node = self.Node(key, value)
            self.cache[key] = node
            self._add_to_head(node)

            # 超出容量，淘汰最久未使用的
            if len(self.cache) > self.capacity:
                lru = self.tail.prev  # 最久未使用
                self._remove(lru)
                del self.cache[lru.key]
```

### 关键设计点

1. **哨兵节点**：head 和 tail 作为哨兵，避免空指针判断，代码更简洁。
2. **先删后加**：更新已有节点时，先从原位置移除，再插入到头部。
3. **容量超限时删除 tail.prev**：tail.prev 即为最久未使用的节点。

### 测试用例

```python
cache = LRUCache(2)
cache.put(1, 1)
cache.put(2, 2)
assert cache.get(1) == 1      # 返回 1，key=1 变为最近使用
cache.put(3, 3)                # 容量满，淘汰 key=2（最久未用）
assert cache.get(2) == -1      # key=2 已被淘汰
cache.put(4, 4)                # 容量满，淘汰 key=1（之前是最近使用的，现在最久未用）
assert cache.get(1) == -1      # key=1 已被淘汰
assert cache.get(3) == 3      # 返回 3
assert cache.get(4) == 4      # 返回 4
```

### Java 实现（面试常考）

```java
class LRUCache {
    private class Node {
        int key, val;
        Node prev, next;
        Node(int k, int v) { key = k; val = v; }
    }

    private int capacity;
    private Map<Integer, Node> map = new HashMap<>();
    private Node head = new Node(0, 0);
    private Node tail = new Node(0, 0);

    public LRUCache(int capacity) {
        this.capacity = capacity;
        head.next = tail;
        tail.prev = head;
    }

    public int get(int key) {
        if (!map.containsKey(key)) return -1;
        Node node = map.get(key);
        remove(node);
        addToHead(node);
        return node.val;
    }

    public void put(int key, int value) {
        if (map.containsKey(key)) {
            remove(map.get(key));
        }
        Node node = new Node(key, value);
        addToHead(node);
        map.put(key, node);
        if (map.size() > capacity) {
            Node lru = tail.prev;
            remove(lru);
            map.remove(lru.key);
        }
    }

    private void remove(Node n) {
        n.prev.next = n.next;
        n.next.prev = n.prev;
    }

    private void addToHead(Node n) {
        n.next = head.next;
        n.prev = head;
        head.next.prev = n;
        head.next = n;
    }
}
```
