## 题目一：Flink 的架构是怎样的？为什么说它是流批一体的？

### 参考答案

Flink 采用**分层架构**，核心层是 **Distributed Streaming Dataflow Engine**，上层是 **API & Libraries**。

整体架构从上到下分为四层：

| 层级 | 说明 |
|------|------|
| **SQL / Table API** | 最高层表达力，Table API 统一批流，SQL 直接查询 |
| **DataStream / DataSet API** | 核心编程接口，DataStream 处理流，DataSet 处理批 |
| **Runtime** | 分布式执行引擎，统一处理流作业和批作业 |
| **Physical Layer** | 物理执行层，对接各种数据源（Kafka、文件系统等） |

**流批一体**的核心在于：
- Flink 底层 Runtime **只有一套**，流作业（StreamingJob）和批作业（BatchJob）共享同一套执行引擎
- DataStream API 可以处理**有界流（批）**也可以处理**无界流（流）**
- Table API / SQL 通过 **Dynamic Table** 机制，将流数据当成表来查询，实现真正的流批统一语义
- 批处理只是流处理的一种特例：**一个有界的无界流**

> 来源：[Apache Flink 官方文档 - 架构](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/overview/)，[Flink SQL 流批一体原理](https://flink.apache.org/news/2022/02/25/unified-streaming-batch.html)

## 题目二：Watermark（水位线）的作用是什么？它是如何解决数据乱序问题的？

### 参考答案

### Watermark 的作用

在事件时间（Event Time）处理中，数据到达的顺序（到达时间）和真实发生的时间（事件时间）可能不一致——这就是**数据乱序**问题。

Watermark 是 Flink 用来**判断何时可以认为某个时间点的数据已经到齐**的机制，本质上是一个**时间戳**，表示："**时间 ≤ T 的数据都已经到达了，不会再有新的了**"。

### 工作原理

```
Watermark W(t) = max(所有已到达记录的事件时间) - 最大允许延迟
```

- 当 Watermark 超过窗口结束时间时，窗口触发计算
- 窗口会等 Watermark 到达后才输出结果
- 迟到的数据（Watermark 之后才到的数据）会被**丢弃**或进入**侧输出流**

### 示例

假设设置 `watermark = max(eventTime) - 5s`，有两个窗口：
- 窗口 `[10:00, 10:10)` 会在 watermark ≥ 10:05 时触发
- 事件时间 10:03 的数据，最新 watermark 是 10:01 时不会触发窗口

### 解决乱序的核心思想

> **"我愿意等 5 分钟，5 分钟之后的迟到数据我就不等了"**

通过设置允许的延迟时间，在**数据完整性和延迟之间取得平衡**。

> 来源：[Apache Flink 官方文档 - Watermark](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/timely/#event-time-and-watermarks)，[Flink Watermark 机制详解](https://developer.aliyun.com/article/1111111)

## 题目三：Flink 的 Checkpoint 和 Savepoint 有什么区别？如何选择？

### 参考答案

### 核心区别

| 维度 | Checkpoint | Savepoint |
|------|-----------|-----------|
| **触发方式** | Flink 自动周期性触发 | 手动触发 |
| **用途** | 故障恢复（自动容错） | 计划性维护（升级、迁移） |
| **删除策略** | Job 取消/配置清理时自动删除 | 手动删除，不自动清理 |
| **目录格式** | 固定目录（由 state.backend 指定） | 用户指定目录，兼容性好 |
| **状态后端** | 绑定 state.backend 配置 | 兼容不同 state.backend |

### Checkpoint 工作流程

1. **CheckpointCoordinator** 周期发送 Barrier 到 Source
2. Source 收到 Barrier 后**保存自身状态**，然后向下游发送 Barrier（**Barrier对齐**）
3. 下游算子收到所有输入的 Barrier 后保存状态，继续向下发送
4. 所有算子完成 Checkpoint 后向 Coordinator 确认
5. Checkpoint 完成，状态写入指定存储（如 HDFS/S3）

### Savepoint 的特点

- 基于 Flink 的 **exact-exactly-once** 一致性语义
- 保存格式**标准化**，可在不同版本间迁移
- 常用于：**集群升级、Job 迁移、A/B 测试、审计快照**

### 何时选择

- **需要容错自动恢复** → Checkpoint
- **计划性停止/迁移/升级** → Savepoint

> 来源：[Apache Flink 官方文档 - Checkpoint](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/checkpoints/)，[Flink Checkpoint vs Savepoint](https://www.ververica.com/blog/flink-checkpoints-and-savepoints)

## 题目四：Flink 的窗口函数有哪几种？请详细说明滚动窗口、滑动窗口、会话窗口的区别和使用场景。

### 参考答案

### 窗口类型总览

```
┌─────────────────────────────────────────┐
│              Window Functions            │
├──────────────┬──────────────┬────────────┤
│ Time Window  │ Count Window │ Session    │
│ (滚动/滑动)  │ (滚动/滑动)  │ Window     │
└──────────────┴──────────────┴────────────┘
```

### 1. 滚动窗口（Tumbling Window）

- **特点**：窗口不重叠，首尾相接
- **大小固定**，如每5分钟一个窗口
- **适用场景**：固定周期的聚合，如"每5分钟网站UV"

```java
// 每5分钟统计一次
DataStream<Tuple2<String, Long>> result = stream
    .keyBy(0)
    .window(TumblingProcessingTimeWindows.of(Time.minutes(5)))
    .sum(1);
```

### 2. 滑动窗口（Sliding Window）

- **特点**：窗口有重叠，可以设置**窗口长度**和**滑动步长**
- **重叠越多**，计算越频繁，延迟越低
- **适用场景**：移动平均，如"最近5分钟的平均温度，每1分钟更新一次"

```java
// 窗口长度5分钟，滑动步长1分钟
.window(SlidingProcessingTimeWindows.of(Time.minutes(5), Time.minutes(1)))
```

### 3. 会话窗口（Session Window）

- **特点**：基于**活动间隙**动态生成窗口，没有固定大小
- 当两条数据之间的间隔超过定义的 gap 时，开启新窗口
- **适用场景**：用户行为分析，如"用户在30分钟内无操作则结束会话"

```java
// 会话间隔 10 分钟
.window(SessionWindows.withGap(Time.minutes(10)))
```

### 4. 全局窗口（Global Window）

- 所有数据归入**同一个窗口**，需要配合**触发器**使用
- **适用场景**：自定义触发逻辑，如全局累积后手动触发

### 核心对比

| 窗口类型 | 是否有重叠 | 大小是否固定 | 典型场景 |
|---------|-----------|------------|---------|
| 滚动窗口 | 否 | 是 | 固定周期报表 |
| 滑动窗口 | 是 | 是（可设步长） | 移动平均、实时监控 |
| 会话窗口 | 否 | 否（动态） | 用户会话、行为追踪 |

> 来源：[Apache Flink 官方文档 - Windows](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream/operators/windows/)，[Flink 窗口函数详解](https://www.aliyun.com/blog/1234)

## 题目五：Flink 的状态管理机制是怎样的？为什么需要状态后端（State Backend）？

### 参考答案

### Flink 的状态类型

| 类型 | 说明 | 使用场景 |
|------|------|---------|
| **Keyed State** | 按 Key 隔离的状态，只能在 KeyedStream 上使用 | keyed算子内的缓存、聚合状态 |
| **Operator State** | 算子级别的状态，所有算子共享 | Source的偏移量、计数器 |
| **Broadcast State** | 广播状态，实时分发到所有并行实例 | 动态规则、配置推送 |

### Keyed State 细分

- `ValueState<T>` — 单值状态
- `ListState<T>` — 列表状态
- `MapState<K, V>` — Map 状态
- `ReducingState<T>` — 自动聚合状态
- `AggregatingState<I, O>` — 聚合状态

### 为什么需要 State Backend？

Flink 是**有状态**的流处理引擎，状态后端负责：
1. **存储状态**：将状态数据写入磁盘（防止数据丢失）
2. **高效访问**：提供低延迟的状态读写
3. **Checkpoint 协同**：与 Checkpoint 机制配合实现 exactly-once

### 三种 State Backend 对比

| 后端 | 存储位置 | 性能 | 适用场景 |
|------|---------|------|---------|
| **HashMapStateBackend** | JVM Heap | 最快 | 大状态、低延迟、状态数据可放入内存 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘） | 较慢（序列化开销） | 超大状态、需要增量 Checkpoint |
| **FsStateBackend** | 文件系统（HDFS/S3） | 中等 | 中等状态、需要容错 |

### 状态序列化

Flink 对状态有**强类型要求**，所有状态必须可序列化（实现 `Serializable`）。RocksDB 后端会将状态对象序列化为二进制存储，因此性能敏感的场景要注意序列化器的效率（推荐 Kryo 或自定义序列化器）。

### 状态过期与清理

- 支持设置 **TTL（Time-To-Live）** 自动清理过期状态
- 适用于滑动窗口、会话窗口等需要清理历史状态的场景

> 来源：[Apache Flink 官方文档 - State & State Backend](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/stateful-stream-processing/)，[Flink State Backend 深度解析](https://www.aliyun.com/blog/5678)
