### 题目1：Flink 的整体架构是怎样的？JobManager 和 TaskManager 分别承担什么职责？

【答案】

**Flink 整体架构**：

Flink 采用 Master-Slave 架构，主要由 Client、JobManager 和 TaskManager 三类组件构成。

**Client（客户端）**：
- 不属于运行时，负责将用户程序转换成 JobGraph
- 将 JobGraph 提交给 JobManager 后可以断开连接（detached模式），也可以保持连接接收进度报告（attached模式）

**JobManager（Master节点）**：
- 负责协调分布式应用的执行，包括任务调度、资源管理、Checkpoint协调、故障恢复等
- 包含三大核心组件：
  - **ResourceManager**：管理 TaskManager 上的 slot 资源，负责资源分配（对应 YARN/K8s/Standalone 等多种部署模式）
  - **Dispatcher**：提供 REST 接口提交 Flink 应用，为每个作业启动一个 JobMaster，同时运行 WebUI
  - **JobMaster**：负责管理单个 JobGraph 的执行，多个作业可以同时运行，各自有独立的 JobMaster

**TaskManager（Worker节点）**：
- 执行具体的 tasks，缓存和交换数据流
- 最小的资源调度单位是 **task slot**，表示 TaskManager 固定比例的资源（内存，不隔离 CPU）
- 多个 subtask 可以共享同一个 slot，甚至同一个 JVM 进程

**任务链（Operator Chain）**：
- Flink 将多个 operator subtask 链化成单个 task，由一个线程执行
- 好处：减少线程间交接开销、增加吞吐、降低延迟

### 题目2：什么是 Watermark（水印）？它在 Flink 中是如何工作的？

【答案】

**Watermark 的定义**：
Watermark（水印）是 Flink 用于处理事件时间（Event Time）乱序数据的一种机制，本质上是一个时间戳，表示"所有时间戳小于该水印的数据都已经到达"。

**为什么需要 Watermark**：
在真实流数据场景中，数据到达的顺序往往和事件发生的顺序不一致（乱序）。如果用 Processing Time（处理时间）则结果不确定；如果用 Event Time，需要等到所有数据都到达才能计算，但数据无限且可能永远不到齐。Watermark 给出了一种"相对确定性"的保证。

**Watermark 的工作原理**：

1. **单调递增性**：Watermark 时间戳单调递增，不会倒退
2. **触发窗口计算**：当 Watermark 时间戳 ≥ 窗口结束时间时，窗口触发计算并输出
3. **乱序容忍**：Watermark 允许一定的乱序，乱序程度由 Watermark 的延迟策略决定

**Watermark 的两种生成方式**：

```java
// 方式一：固定延迟水印
WatermarkStrategy
  .<Tuple2<String, Long>>forBoundedOutOfOrderness(Duration.ofSeconds(10))
  .withTimestampAssigner((event, timestamp) -> event.f1);

// 方式二：周期性水印（通过 PunctuatedWatermarkGenerator）
```

**多流 Watermark 对齐**：
- 当一个算子有多个输入流时，取所有流中最小 Watermark 作为当前算子的 Watermark
- 避免部分流数据被错误地"判定为迟到"

**乱序数据处理**：
- 设置 `allowedLateness`：窗口触发后，仍保留一定时间，迟到数据不会重复计算
- 设置 `sideOutputLateData`：将迟到数据输出到侧输出流，避免丢失

### 题目3：Flink 的 Checkpoint 机制是如何实现的？它和 Savepoint 有什么区别？

【答案】

**Checkpoint（检查点）机制**：

Flink 通过 **Chandy-Lamport 分布式快照算法**的变体（轻量级异步快照）实现 Exactly-Once 语义。

**核心概念 - Barrier（屏障）**：
- Barrier 是 Flink 快照的核心元素，被注入到数据流中，与数据记录一起流动
- Barrier 将数据流中的记录分为两类：属于当前快照的记录 vs 属于下一个快照的记录
- 每个 Barrier 携带快照 ID，不会打断数据流（多个快照可以同时进行）

**Barrier 对齐流程（Aligned Checkpoint）**：
1. Source 注入 Barrier 到数据流，记录当前快照的偏移量
2. 当算子收到**所有输入**的 Barrier n 时，才触发快照
3. 算子快照状态，发送 Barrier n 到下游
4. Sink 收到所有输入的 Barrier n 后，向 JobManager 确认快照完成

**非对齐 Checkpoint（Unaligned Checkpoint）**：
- 自 Flink 1.11 起支持
- 当某条路径数据严重积压导致 Barrier 对齐耗时过长时，非对齐 Checkpoint 将 in-flight 数据纳入状态快照
- 大幅减少背压场景下的快照延迟，但会增加状态存储 I/O

**状态后端（State Backend）**：
- **HashMapStateBackend**：状态存储在 TaskManager 的 JVM 堆内存中，高速但受限于内存大小
- **EmbeddedRocksDBStateBackend**：状态存储在 RocksDB（LSM树），支持超大状态，TTL 清理

**Checkpoint 和 Savepoint 的区别**：

| 特性 | Checkpoint | Savepoint |
|------|-----------|-----------|
| 触发方式 | Flink 自动周期执行 | 手动触发 |
| 清理策略 | 新 checkpoint 完成后自动清理旧 checkpoint | 需要手动删除 |
| 用途 | 故障恢复 | 应用程序升级、集群迁移、A/B测试 |
| 兼容性 | 与 Job 绑定，Job 重启后状态丢失 | 标准化格式，不依赖 Job，可跨版本迁移 |
| 状态大小 | 通常较小（增量快照） | 通常较大（全量快照） |

### 题目4：Flink 支持哪几种窗口？窗口是如何触发计算的？

【答案】

**窗口的分类**：

**1. 按时间划分**：
- **滚动窗口（Tumbling Window）**：固定大小、不重叠，例如每5分钟一个窗口
  ```java
  TumblingEventTimeWindows.of(Time.minutes(5))
  ```
- **滑动窗口（Sliding Window）**：固定大小、可重叠，由窗口大小和滑动步长参数控制
  ```java
  SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(5))
  ```
- **会话窗口（Session Window）**：根据活动间隙动态创建，窗口边界由数据本身决定
  ```java
  EventTimeSessionWindows.withGap(Time.minutes(10))
  ```

**2. 按数据量划分**：
- **计数窗口（Count Window）**：达到指定数据量时触发
  ```java
  countWindow(100)          // 滚动计数窗口，每100条触发
  countWindow(100, 50)      // 滑动计数窗口，每50条滑动一次
  ```

**窗口触发计算的条件**：

1. **时间窗口**：Watermark 到达窗口结束时间，触发窗口计算
2. **计数窗口**：窗口内数据条数达到设定值时触发
3. **会话窗口**：在指定的 gap 时间内没有新数据到达时触发

**窗口的两大核心API**：

```java
// 增量聚合：每条数据来都更新状态，最后输出（如 sum、min）
.reduce((v1, v2) -> new Value(v1.get() + v2.get()))

// 全量聚合：收集窗口内所有数据，最后一次性计算（如 sorted）
.process(new ProcessWindowFunction<...>() {
    public void process(Context ctx, Iterable<T> values, Collector<O> out) {
        // 窗口所有数据在 values 中
    }
})
```

**迟到数据处理**：
- `allowedLateness(Time.minutes(5))`：窗口触发后仍保留5分钟，迟到数据不重复输出
- `sideOutputLateData(outputTag)`：超过 allowedLateness 的数据从侧输出流输出

### 题目5：Flink 的状态管理是如何工作的？Keyed State 和 Operator State 有哪些区别？

【答案】

**Flink 中的状态（State）**：

状态是 Flink 处理流数据时remember 跨事件信息的能力。例如：窗口聚合存储的中间结果、模式匹配记住已发生的事件序列、PID 训练保存的模型参数等。

**Flink 状态的两大类型**：

**1. Keyed State（键控状态）**：
- 只能在 KeyedStream（即 keyBy 之后）上使用
- 状态按 Key 分区，每个 Key 独立维护自己的状态
- 本质上是嵌入式的 Key/Value 存储，访问仅限当前事件的 Key
- 支持的状态类型：
  - `ValueState<T>`：单值状态
  - `ListState<T>`：列表状态
  - `MapState<K, V>`：映射状态
  - `ReducingState<T>`：自动聚合状态
  - `AggregatingState<IN, OUT>`：自定义聚合状态

**2. Operator State（算子状态）**：
- 作用域是整个算子实例，不依赖 Key
- 所有并行实例共享状态（ListState / UnionListState）或各自独立（BroadcastState）
- **BroadcastState** 是一种特殊的 Operator State，用于将数据广播到所有并行实例，常用于动态规则更新

```java
// Keyed State 示例
ValueState<Double> sumState = getRuntimeContext().getState(
    new ValueStateDescriptor<>("sum", Double.class)
);

// Operator State 示例（ListState）
ListState<String> checkpointState = getRuntimeContext().getListState(
    new ListStateDescriptor<>("checkpoint", String.class)
);
```

**状态一生周期管理（TTL）**：
```java
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.hours(1))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)
    .build();
```

**状态重分布（Rescale）**：
- Keyed State 按 Key Group 自动重新分配（Key Group 是状态重分布的原子单位，数量等于最大并行度）
- Operator State（ListUnion）由用户代码在恢复时自己处理重新分配逻辑

**状态存储后端**：

| 后端 | 存储介质 | 适用场景 |
|------|---------|---------|
| HashMapStateBackend | JVM Heap | 小状态、低延迟、开发和测试 |
| EmbeddedRocksDBStateBackend | RocksDB（磁盘） | 大状态、生产环境、支持增量 Checkpoint |
