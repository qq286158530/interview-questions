## 题目一：HBase 的整体架构是怎样的？RegionServer 和 HMaster 分别负责什么？

### 参考答案

HBase 采用 **Master-Slave（主从）架构**，整体由以下组件构成：

**核心组件：**

| 组件 | 职责 |
|------|------|
| **HMaster** | 负责表的管理工作——DDL 操作（建表、删表）、Region 的分配与调度、RegionServer 的负载均衡、垃圾文件回收，以及处理 schema 修改请求。HMaster 不参与实际的数据读写。 |
| **RegionServer** | 承担数据的读写请求（Get/Put/Scan/Delete），管理本节点上的 Region，包括 MemStore 刷写、HFile 合并与分裂等操作。是直接服务客户端的一层。 |
| **ZooKeeper** | 负责选举 active HMaster、存储元数据入口（`-ROOT-` → `.META.`）位置、检测 RegionServer 存活状态。 |
| **HDFS** | HBase 最终将数据落盘到 HDFS，利用其副本机制保证数据可靠性。 |

**Region 分布逻辑：**
- 一张表按 RowKey 区间横向切分为多个 **Region**，每个 Region 分散到不同的 RegionServer 上。
- `-ROOT-` 表（现已优化，可能直接由 ZooKeeper 记录）指向 `.META.` 表，`.META.` 表记录所有 Region 的位置信息，客户端通过三级查询定位数据：
  `ZooKeeper → -ROOT- → .META. → Region`

**一句话总结：** HMaster 是"指挥官"，负责管理；RegionServer 是"士兵"，负责干活；ZooKeeper 是"联络员"，负责协调和选主。

> 📎 来源：[HBase 官方文档 - Architecture](https://hbase.apache.org/book.html#arch.overview)

## 题目二：HBase 的 RowKey 设计应遵循哪些原则？为什么？

### 参考答案

RowKey 是 HBase 设计的核心，直接决定读写性能。设计原则如下：

**1. 长度原则：越短越好**
- RowKey 存储在每个 KeyValue 中，过长会占用大量存储空间。
- 建议控制在 **16~24字节**以内（可用 MD5/Hash 压缩）。

**2. 散列原则：避免热点**
- 热点（HotSpot）是指大量请求集中到少数 RegionServer 上，造成性能瓶颈。
- ❌ 避免使用单调递增/递减的 RowKey（如时间戳开头 `20260608001`），写入会永远落在最后一个 Region。
- ✅ 使用 Hash 打散（如 `MD5(rowkey).substring(0, 5) + original_key`）或雪花算法。

**3. 唯一性原则**
- RowKey 是 HBase 中行的唯一标识，类似主键，必须保证唯一。

**4. 业务贴合原则**
- 将查询条件中最常用的字段放在 RowKey 前缀位置，支持 **Scan 范围查询**。
- 例如：查询某用户订单 → RowKey 设计为 `userId + orderId`

**5. 组合原则**
- 多个查询维度可拼接成复合 RowKey，但要确保高位字段散列均匀。

**实战技巧：**
```
# 热门场景示例：用户帖子列表
RowKey = reversed_timestamp + userId
# reversed_timestamp = Long.MAX_VALUE - timestamp
# 好处：新帖子排在前面，且时间戳反转后自然散列
```

> 📎 来源：[HBase 官方文档 - RowKey Design](https://hbase.apache.org/book.html#rowkey.design)

## 题目三：描述 HBase 写数据的流程，以及 MemStore 刷写到 HFile 的机制。

### 参考答案

**HBase 写流程（Put）：**

```
客户端 → ZooKeeper → .META. → RegionServer
       ↓
   RegionServer 接收 Put 请求
       ↓
   写入 WAL（Write-Ahead Log）— 先日志再内存，保证持久性
       ↓
   写入 MemStore（内存中，按 RowKey 排序）
       ↓
   返回 ACK 给客户端
```

**关键点：**
- **WAL（Write-Ahead Log）**：先顺序写入 WAL，再修改 MemStore。RegionServer 崩溃时可通过 WAL 恢复未持久化的数据。
-写入 MemStore 是**内存操作**，性能极高（百万级 QPS）。

**MemStore 刷写（Flush to HFile）触发条件：**

| 触发条件 |阈值 |
|----------|------|
| MemStore 大小 | 单个 MemStore ≥ `hbase.hregion.memstore.flush.size`（默认 128MB） |
| 整个 RegionServer 内存 | RegionServer 所有 MemStore 总和 ≥ `hbase.regionserver.global.memstore.size`（默认 heap 的 40%） |
| WAL 日志数量 | WAL 日志数量 ≥ `hbase.regionserver.max.logs`（默认 32） |
| 定时触发 | 定期检查线程，每隔一段时间强制刷写 |

**刷写为 HFile：**
- MemStore 达到阈值时，将内存中已排序的数据 **顺序写入 HDFS**，生成一个新的 HFile。
- HFile 一旦写入就不会再修改（Immutable），后续通过 **HFile合并（Compaction）** 合并小文件。

**HFile 结构简述：**
```
HFile = Header + Data Block + Meta Block + Index Block + Trailer
- Data Block：存储实际的 KeyValue 数据（可配置大小，默认 64KB）
- Index Block：记录每个 Data Block 的起始 key，加速随机读
```

> 📎 来源：[HBase 官方文档 - Writing Data](https://hbase.apache.org/book.html#write)
> 📎 来源：[HBase 官方文档 - MemStore](https://hbase.apache.org/book.html#memstore)

## 题目四：HBase 协处理器（Coprocessor）是什么？它有哪两种类型？各自的使用场景是什么？

### 参考答案

**什么是协处理器？**
协处理器是 HBase 提供的扩展机制，允许用户将自定义逻辑 **运行在 RegionServer 端**，直接在被访问的数据节点上执行代码，减少网络传输。

**两种类型：**

**1. Observer（观察者）—类似于触发器**
- 挂在特定的钩子（Hook）上，当这些事件发生时自动触发执行。
- 核心方法：
  - `preGet / postGet`：读操作前后
  - `prePut / postPut`：写操作前后
  - `preDelete / postDelete`：删除操作前后
  - `preOpen / postOpen`：Region 打开时
  - `preSplit / postSplit`：Region 分裂前后
  - `preCompact / postCompact`：Compaction 前后

- **使用场景**：
  - 实现 **二级索引**（如在 postPut 中同步更新索引表）
  - 实现 **数据验证**（prePut 中检查数据合法性）
  - 实现 **审计日志**（记录谁在什么时间修改了什么数据）
  - 实现 **列级别权限控制**（preGet 中检查用户权限）

**2. Endpoint（端点）— 类似于存储过程**
- 允许在 RegionServer 上执行自定义的 **RPC 调用处理器**，支持客户端请求 RegionServer 执行任意自定义方法。
- 类似于 RDBMS 中的存储过程，但更灵活——可以调用多个 Endpoint汇总结果。

- **使用场景**：
  - 在服务端直接对数据进行 **聚合计算**（如 COUNT、SUM、AVG），减少数据传输
  - 实现跨 Region 的复杂查询逻辑
  - 在 Region层面执行自定义的数据转换

**配置方式：**
```xml
<!-- 表级别加载协处理器 -->
<property>
  <name>coprocessor$Observer</name>
  <value>com.example.MyObserver|path/to.jar|priority</value>
</property>
```
或通过 HBase Shell：
```bash
disable 'myTable'
alter 'myTable', METHOD => 'table_att', 'coprocessor' => 'path/to.jar|className|priority'
enable 'myTable'
```

> 📎 来源：[HBase 官方文档 - Coprocessors](https://hbase.apache.org/book.html#cp)

## 题目五：HBase 读数据的流程是怎样的？为什么比写数据慢？有哪些优化手段？

### 参考答案

**HBase 读流程（Get/Scan）：**

```
客户端 → ZooKeeper → .META. → RegionServer
       ↓
   RegionServer 接收 Get 请求
       ↓
   ① 先查 MemStore（内存中最近写入但未刷写的数据）
       ↓
   ② 再查 BlockCache（读缓存，热点数据）
       ↓
  ③ 最后查 HFile（磁盘文件，需遍历磁盘）
       ↓
   合并所有结果返回客户端
```

**为什么读比写慢？**
1. **多层查找**：需要依次查询 MemStore → BlockCache → HFile，数据可能分布在多个层级。
2. **磁盘 IO**：HFile 在 HDFS 上，读取涉及磁盘 IO（而写只需要写 MemStore 内存）。
3. **HFile 较多**：大量 HFile 时需要合并遍历多个文件（可通过 Compaction 优化）。

**优化手段：**

| 优化方向 | 具体措施 |
|----------|----------|
| **BlockCache** | 合理配置 `hfile.block.cache.size`（默认 heap 的 40%），将热点数据常驻读缓存 |
| **HFile 合并** | 定期 Compaction 将小 HFile 合并成大文件，减少需扫描的文件数 |
| **BloomFilter** | 开启 BloomFilter（默认 `row` 级别），快速判断某个 RowKey 是否存在于 HFile 中，避免无效 IO |
| **列族设计** | 将冷数据与热数据分离到不同列族，避免读取时扫描无关数据 |
| **RowKey 设计** | 良好的 RowKey 设计减少 Scan 范围，减少需要遍历的数据量 |
| **批量读** | 使用 `getList()` 或 `scan` 的 batch 功能减少请求次数 |
| **短路读（Short-Circuit Read）** | 配置 `hbase.rs.shortcircuit.read` 让读取本地副本时绕过 RPC 直接读 HDFS 块 |

**Compaction 机制补充：**
- **Minor Compaction**：只合并部分小 HFile，删除过期（TTL）和已删除的数据
- **Major Compaction**：合并一个 Region 下的所有 HFile，生成一个完整的大文件（建议在业务低峰期执行，避免 IO 风暴）

> 📎 来源：[HBase 官方文档 - BlockCache](https://hbase.apache.org/book.html#blockcache)
