# HBase 面试题（2026-07-04）

> 本文精选 5 道高频 HBase 面试题，涵盖架构、RowKey 设计、存储结构、MemStore/HFile、协处理器等核心知识点。

---

## 题目一：HBase 的整体架构是怎样的？包含哪些核心组件？

### 参考答案

HBase 是一个面向列族的分布式 NoSQL 数据库，构建在 HDFS 之上，采用**主从架构**。其核心组件包括：

| 组件 | 职责 |
|------|------|
| **HMaster** | 负责表的管理（创建、删除 schema）、Region 的分配与负载均衡、故障检测与恢复。不参与实际数据读写。 |
| **RegionServer** | 负责管理 Region 的实际数据读写请求。每个 RegionServer 可管理多个 Region（通常 10~1000 个）。 |
| **ZooKeeper** | 维护集群元数据、选举主 HMaster、检测 RegionServer 存活状态，提供分布式协调服务。 |
| **Region** | HBase 表的水平分片单元，每个 Region 包含[startKey, endKey) 区间内的数据。 |
| **HDFS** | 底层分布式文件系统，负责数据块的存储和副本管理。 |

**架构特点：**
- HMaster 通常部署 1 主多备，通过 ZooKeeper 选举主节点。
- 客户端读写数据直接与 RegionServer 通信，不经过 HMaster（HMaster 仅负责元数据管理）。
- ZooKeeper 中存储了 `-ROOT-` 表和 `.META.` 表的位置信息，客户端通过三级定位定位到具体的 RegionServer。

## 题目二：HBase 中 RowKey 设计的原则有哪些？如何避免热点问题？

### 参考答案

RowKey 是 HBase 中数据访问的核心，设计好坏直接影响读写性能和集群稳定性。

### RowKey 设计原则

1. **长度原则**：RowKey 推荐越短越好（不超过 16 字节），因为：
   - MemStore 和 HFile 中都会存储 RowKey
   - HBase 底层 key-value 会在磁盘上存储多次，KeyLength 越大，存储膨胀越严重

2. **唯一性原则**：RowKey 必须唯一标识一行数据，类似关系型数据库的主键。

3. **散列原则（避免热点）**：相同前缀的 RowKey 会导致数据集中在同一个 RegionServer 上，造成热点。解决方案：
   - **Hash 打散**：对 RowKey 做 MD5 或其他哈希，将原始 key 散列到不同的节点
   - **盐值（Salting）**：在 RowKey 头部拼接随机前缀
   - **反转（Reverse）**：将长尾的固定前缀反转，如手机号 `13800138000` 反转为 `00008310083`

### 常见设计模式

| 场景 | RowKey 设计示例 |
|------|----------------|
| 用户订单查询 | `userId + reverse(timestamp)` |
| 时间序列数据 | `salt(hash(timestamp)) + timestamp` |
| 多条件查询 | `联合字段拼接`（如 `userId + orderId`）|

## 题目三：简述 HBase 的存储结构（MemStore 与 HFile），以及刷写和合并的机制。

### 参考答案

HBase 的存储分为**内存存储（MemStore）** 和**磁盘存储（HFile）** 两层。

### MemStore（写缓存）

- 当数据写入 HBase 时，先写入 WAL（Write-Ahead Log）预写日志，然后写入 MemStore。
- MemStore 是一个 **LSM-Tree（Log-Structured Merge-Tree）** 的内存缓冲结构，按 RowKey 排序。
- 当 MemStore 达到阈值（默认 `hbase.hregion.memstore.flush.size=128MB`）时，触发**刷写（Flush）**，将数据顺序写入 HDFS 生成 HFile。

### HFile（磁盘存储）

- HFile 是 HBase 的磁盘文件格式，基于 Google BigTable 的 SSTable。
- HFile 结构包含：
  - **Data Block**：存储 key-value 数据对（默认 64KB）
  - **Meta Block**：存储布隆过滤器信息
  - **File Info**：文件元信息（最后修改时间、MR 引用等）
  - **Bloom Filter**：快速判断某 RowKey 是否可能存在于该文件中，避免无效 IO
  - **Index Block**：记录每个 Data Block 的起始 RowKey，支持二分查找

### Compaction（合并）机制

| 类型 | 说明 |
|------|------|
| **Minor Compaction** | 将多个小的 HFile 合并为一个较大的 HFile，不处理所有数据 |
| **Major Compaction** | 将一个 Region 的所有 HFile 合并为一个，**删除标记的单元格在此过程中被物理删除**，并合并时间戳保留最新 N 个版本 |

**刷写触发条件（满足任一即触发）：**
- MemStore 大小达到阈值
- WAL 文件数量超过 `hbase.regionserver.max.logs`
- Region 的所有列族触发刷写

## 题目四：HBase 的数据写入流程是什么？为什么需要 WAL？

### 参考答案

### 写入流程

1. **客户端请求**：客户端通过 ZooKeeper 获取 `-ROOT-` → `.META.` 定位到目标 RegionServer。
2. **写 WAL**：先将数据写入 WAL（预写日志），保证数据持久性。WAL 记录了所有尚未成功刷写到 HFile 的操作。
3. **写 MemStore**：数据写入 MemStore（按 RowKey 排序）。
4. **返回客户端**：写入成功立即返回客户端（此时数据已在内存中，但未落盘）。
5. **异步刷写**：后台线程定期将 MemStore 刷写为 HFile。

### 为什么需要 WAL？

WAL（Write-Ahead Logging）是 HBase **容错机制的核心**：

- **防止数据丢失**：MemStore 数据在内存中，如果节点突然崩溃，未持久化的数据会丢失。WAL 记录了所有未刷写的操作，RegionServer 重启后可以**重放 WAL** 恢复数据。
- **保证一致性**：写入 WAL 后才算成功，崩溃恢复时不会产生脏数据。
- **批量写入优化**：写入 MemStore 是内存操作，极快；WAL 顺序写入磁盘，可靠且批量。

> **注意**：如果业务对写入性能要求极高且可容忍少量数据丢失，可通过 `put.setWriteToWAL(false)` 跳过 WAL，但这会失去故障恢复保障。

## 题目五：什么是 HBase 协处理器（Coprocessor）？它有哪两类？各自的使用场景是什么？

### 参考答案

### 概念

协处理器是 HBase 0.92 引入的**自定义计算逻辑扩展机制**，允许用户在 **RegionServer 上直接运行用户代码**，无需在客户端做二次过滤或聚合，从而极大减少网络传输数据量。

### 两类协处理器

| 类型 | 加载方式 | 说明 |
|------|---------|------|
| **Observer（观察者）** | 系统自动加载，类似数据库触发器 | 被动监听 HBase 生命周期中的钩子事件（HBase 会在指定操作前后调用） |
| **Endpoint（端点）** | 动态调用，类似存储过程 | 主动调用，类似于 RPC，允许用户在 RegionServer 上执行自定义的分布式计算 |

### Observer 类型

| Observer 类型 | 触发时机 | 典型应用场景 |
|-------------|---------|-------------|
| **PreGetObserver** | GET 请求之前 | 行级安全校验（列权限控制）、动态列过滤 |
| **PostGetObserver** | GET 请求之后 | 敏感数据脱敏（返回前对字段加密） |
| **PrePutObserver / PostPutObserver** | PUT 写入前后 | 写入前数据校验、写入后触发异步索引构建 |
| **PreDeleteObserver / PostDeleteObserver** | DELETE 前后 | 删除时同步清理关联数据 |
| **RegionObserver** | 更通用的 Observer | 实现更复杂的表级操作拦截 |

### Endpoint 类型

Endpoint 的典型使用场景：
- **自定义聚合**：在 RegionServer 上并行计算 COUNT、SUM、AVG 等聚合，然后汇总到客户端（如实现自定义的二级索引聚合）。
- **动态列计算**：在服务端直接对列进行计算，只返回结果而非全量数据。
- 使用时需定义 Protobuf 接口，通过 `CoprocessorService` 暴露。

### 配置方式

协处理器可通过三种方式加载：
1. **HBase Shell 配置**：`alter 'table', METHOD => 'table_att', 'coprocessor' => '...'`
2. **hbase-site.xml 全局配置**：对所有表生效
3. **Java API**：创建表时通过 `HTableDescriptor` 添加
