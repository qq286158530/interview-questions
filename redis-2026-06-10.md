## 题目1：Redis数据结构 | 五种数据类型底层实现与版本演进

**题目：** Redis有哪些数据类型？它们的底层数据结构是什么？在Redis 7.0版本中底层数据结构有哪些变化？

**【答案】**

### Redis五种核心数据类型及底层实现

**1. String（字符串）**
- 底层实现：SDS（Simple Dynamic String，简单动态字符串）
- SDS vs C字符串：
  - SDS使用`len`属性记录长度，获取字符串长度复杂度O(1)，C字符串为O(n)
  - SDS是二进制安全的，可以存储图片、音频等二进制数据（不以空字符判断结束）
  - SDS API是安全的，拼接字符串不会造成缓冲区溢出（自动扩容）
- 应用场景：缓存对象、常规计数、分布式锁、共享session信息

**2. List（列表）**
- Redis 3.2之前：双向链表 或 压缩列表（ziplist）
  - 元素个数 < 512 且 每个元素 < 64字节 → 使用压缩列表
  - 否则 → 使用双向链表
- Redis 3.2之后：统一使用quicklist（双向链表 + 压缩列表的组合）
- 应用场景：消息队列（需自行实现全局唯一ID，不支持消费组形式）

**3. Hash（哈希）**
- 底层实现：压缩列表 或 哈希表（hashtable）
  - 元素个数 < 512 且 所有值 < 64字节 → 使用压缩列表
  - 否则 → 使用哈希表
- Redis 7.0起：压缩列表（ziplist）已被废弃，改用listpack替代
- 应用场景：缓存对象、购物车

**4. Set（集合）**
- 底层实现：整数集合（intset） 或 哈希表
  - 元素都是整数 且 元素个数 < 512 → 使用整数集合
  - 否则 → 使用哈希表
- 应用场景：聚合计算（并集、交集、差集），如点赞、共同关注、抽奖活动

**5. Zset（有序集合）**
- 底层实现：压缩列表 或 跳表（skiplist）+ 哈希表
  - 元素个数 < 128 且 每个元素 < 64字节 → 使用压缩列表
  - 否则 → 使用跳表（支持O(log N)范围查询）+ 哈希表（支持O(1)查找）
- Redis 7.0起：压缩列表被废弃，改用listpack
- 应用场景：排行榜、电话和姓名排序

### Redis 7.0重大变化：压缩列表（ziplist）被废弃

Redis 7.0中，压缩列表数据结构被标记为废弃，交由**listpack**（紧凑列表）实现。listpack是压缩列表的改进版，核心区别是：

- 避免了连锁更新问题：压缩列表中，插入/删除元素可能触发后续元素entry长度字段的连锁更新（最坏O(n²)），listpack通过记录总长度避免此问题
- listpack设计更紧凑，遍历效率更高

## 题目2：Redis持久化 | AOF日志与RDB快照的对比与混合持久化

**题目：** Redis有哪几种持久化方式？AOF日志和RDB快照有什么区别？什么是混合持久化？它在什么场景下最有价值？

**【答案】**

### 三种持久化方式

**1. AOF日志（Append Only File）**
- 原理：每执行一条写操作命令，就把该命令以追加的方式写入文件
- Redis重启时，读取AOF文件中的命令逐一执行来恢复数据
- Redis先执行写操作命令，再记录日志（好处：不会阻塞当前命令；风险：宕机可能丢失数据）
- 三种写回策略（通过`appendfsync`配置）：
  - `always`：每次执行完同步写回硬盘（最安全，数据零丢失，但性能最低）
  - `everysec`（默认）：每秒同步（性能适中，最多丢失1秒数据）
  - `no`：由操作系统决定何时写回（性能最好，但数据丢失不可控）

**2. RDB快照**
- 原理：记录某一时刻的内存数据，以二进制写入磁盘
- 与AOF记录命令不同，RDB记录的是实际数据，恢复速度更快
- 两种生成方式：
  - `save`：主线程执行，会阻塞（不推荐）
  - `bgsave`：子进程生成，避免阻塞（Redis默认方式）
- 自动触发条件（`save`配置）：
  - 900秒内至少1次修改
  - 300秒内至少10次修改
  - 60秒内至少10000次修改

**3. 混合持久化（Redis 4.0+）**
- 原理：AOF重写时，先以RDB格式写入AOF文件前半部分，后将增量命令以AOF格式写入后半部分
- 恢复时：先加载RDB部分（快速），再加载AOF增量部分（减少数据丢失）
- 优点：启动快 + 丢失数据少
- 缺点：AOF可读性差；与Redis 4.0之前版本不兼容

### AOF重写机制

随着写命令增多，AOF文件会越来越大。Redis通过**AOF重写机制**压缩文件：

- 原理：读取当前数据库中所有键值对，用一条命令记录到新的AOF文件
- 例如：执行了`set name xiaolin`和`set name xiaolincoding`，重写后只保留`set name xiaolincoding`
- 由后台子进程`bgrewriteaof`完成，避免阻塞主进程
- 使用**写时复制（COW）**技术：子进程与父进程共享物理内存，子进程只读，父进程修改数据时复制副本

## 题目3：Redis缓存 | 缓存穿透、缓存击穿、缓存雪崩的原因与解决方案

**题目：** 在使用Redis作为缓存时，常遇到缓存穿透、缓存击穿、缓存雪崩三种问题。请分析它们产生的原因，并给出至少两种可行的解决方案。

**【答案】**

### 缓存穿透

**原因：** 查询一个不存在的数据，由于缓存和数据库中都没有，每次请求都穿透到数据库。

**危害：** 大量不存在的数据请求直接打到数据库，可能导致数据库宕机。

**解决方案：**

1. **布隆过滤器（Bloom Filter）**
   - 在访问缓存之前，先用布隆过滤器判断数据是否存在
   - 布隆过滤器存储已存在的所有数据key，不存在则直接返回
   - 存在误判（可能把不存在的判断为存在，但不会把存在的判断为不存在）

2. **返回空值（缓存空对象）**
   - 当数据库查询不到时，在缓存中存入一个空值（设置较短的TTL，如60秒）
   - 后续相同key的请求直接从缓存返回空值，不再查数据库

**缓存击穿**

**原因：** 某个热点数据过期时，恰好有大量并发请求同时访问这个热点数据，导致所有请求都打到数据库。

**解决方案：**

1. **互斥锁（Mutex）**
   - 只允许一个线程去加载数据，其他线程等待
   - 获取锁成功 → 查数据库写缓存 → 释放锁；获取锁失败 → 短暂等待后重试从缓存获取

2. **热点数据永不过期**
   - 不设置TTL，但通过异步线程在后台更新缓存
   - 或设置逻辑过期时间，发现过期时用互斥锁只允许一个线程去重建缓存

**缓存雪崩**

**原因：** 大量缓存数据在同一时间过期（集体失效），导致大量请求直接打到数据库。

**解决方案：**

1. **过期时间随机化**
   - 给缓存过期时间加上随机偏移量（如 `baseTTL + random(0, baseTTL/5)`），避免同一时间集体失效

2. **多级缓存**
   - 设置多级缓存（如L1本地缓存 + L2 Redis缓存），本地缓存可用不同过期时间
   - 即使Redis缓存失效，本地缓存仍可提供服务

3. **服务降级/熔断**
   - 当数据库压力过大时，对非核心业务进行降级，返回默认值或友好提示
   - 保护数据库不被压垮

## 题目4：Redis线程模型 | 为什么单线程的Redis性能这么高？

**题目：** Redis是单线程的吗？为什么单线程的Redis还能达到每秒10万QPS的高性能？Redis 6.0引入多线程的目的是什么？

**【答案】**

### Redis单线程指的是什么

Redis单线程指的是**接收客户端请求 → 解析请求 → 进行数据读写等操作 → 发送数据给客户端**这个过程由一个线程（主线程）来完成。

但Redis程序并非完全单线程，Redis启动时会启动后台线程（BIO）：

- **Redis 2.6**：2个后台线程 → 处理关闭文件、AOF刷盘
- **Redis 4.0+**：新增lazyfree线程 → 异步释放Redis内存（如执行`unlink key`/`flushdb async`时异步删除，避免阻塞主线程）

### Redis单线程为什么快

**1. 内存操作，性能瓶颈不在CPU**
- Redis大部分操作在内存中完成，采用高效的数据结构（SDS、跳表等）
- 性能瓶颈可能是内存大小或网络带宽，而非CPU，因此单线程不是问题

**2. 非阻塞I/O多路复用**
- Redis采用`select/epoll`机制，实现一个线程处理多个IO流
- 内核同时监听多个Socket的连接请求和数据请求，有请求到达时交给Redis线程处理

**3. 避免锁竞争和线程切换开销**
- 单线程无需考虑多线程竞争，无需加锁解锁
- 无线程切换开销，不会因死锁导致性能损耗

**4. 高效的数据结构**
- SDS：O(1)获取长度，二进制安全
- 跳表：O(log N)范围查询，O(1)查找元素
- 压缩列表/listpack：紧凑存储，内存友好

### Redis 6.0引入多线程

Redis 6.0之前网络I/O和命令执行都是单线程。随着网络硬件性能提升，**网络I/O成为性能瓶颈**，因此Redis 6.0对网络I/O采用多线程处理。

**关键点：命令执行仍然是单线程的**，多线程仅用于网络I/O（read client socket/write client socket）。

- 默认`io-threads-do-reads=no`，需要手动开启`io-threads-do-reads=yes`
- 官方建议：4核CPU设2-3个线程，8核CPU设6个线程，线程数必须小于机器核数
- Redis 6.0启动时默认额外创建6个线程：bio_close_file、bio_aof_fsync、bio_lazy_free（各1个） + io_thd_1/2/3（3个I/O线程）

## 题目5：Redis集群 | 主从复制、哨兵模式与切片集群的区别与选型

**题目：** Redis如何实现高可用？主从复制、哨兵模式、切片集群三种方案有什么区别？各自适用什么场景？

**【答案】**

### 主从复制（Replication）

**原理：** 一主多从模式，主服务器负责读写操作，从服务器只读并同步主服务器数据。

- 命令传播阶段：主服务器收到写命令后异步发送给从服务器（不保证强一致性）
- 主从服务器之间采用**读写分离**：主服务器处理写操作，从服务器处理读操作

**缺点：**
- 主服务器宕机时，需要手动将从服务器切换为主服务器（无法自动故障转移）
- 数据不一致：命令传播是异步的，主从数据时时刻刻可能不一致

**适用场景：** 数据量不大、允许一定延迟的场景（如读多写少的业务）。

### 哨兵模式（Sentinel）

**原理：** 哨兵是一个独立的进程，用于监控主从服务器的运行状态，实现自动故障转移。

**核心功能：**
1. **监控**：哨兵会定期检测主从服务器是否运行正常
2. **自动故障转移**：当主服务器宕机时，哨兵自动将从服务器升级为主服务器，并通知其他从服务器重新配置
3. **通知**：故障转移后，通过发布/订阅机制通知应用层

**工作流程：**
- 多个哨兵组成哨兵集群，通过Raft协议选举领导者
- 主观下线：单个哨兵认为主服务器不可用
- 客观下线：多数哨兵都认为主服务器不可用，触发故障转移
- 由哨兵领导者执行故障转移（选择从服务器中数据最新的作为新主服务器）

**适用场景：** 解决了主从复制的自动故障转移问题，适合对高可用有要求的场景。

### 切片集群（Cluster）

**原理：** Redis 3.0引入的数据分片方案，将数据分布在多个Redis实例上，每个实例存储一部分数据。

**核心概念：**
- **哈希槽（Hash Slot）**：Redis Cluster将整个数据库分为16384个槽，每个key通过CRC16算法计算槽位，分配到对应节点
- **节点间通信**：Gossip协议，节点间互相心跳检测，自动发现和迁移槽位

**特点：**
- 每个节点都是主从复制（保证高可用）
- 客户端直接连接任意节点，通过MOVED/ASK重定向到正确节点
- 支持自动数据迁移

**适用场景：** 数据量巨大（超过单台机器内存上限），需要水平扩展的场景。

### 三者对比总结

| | 主从复制 | 哨兵模式 | 切片集群 |
|---|---|---|---|
| 高可用 | 无自动故障转移 | 自动故障转移 | 自动故障转移 + 水平扩展 |
| 数据规模 | 单机容量 | 单机容量 | 多机分布式存储 |
| 写并发 | 单主 | 单主 | 多主（每个分片一个主） |
| 复杂度 | 低 | 中 | 高 |
| 适用场景 | 读多写少，数据量小 | 中小型高可用场景 | 大数据量高并发场景 |
