## 题目一：Flink 的架构原理是什么？

### 参考答案

Flink 是**流处理优先**的新一代大数据处理框架，核心架构分为**分分层**：

**1. 架构分层**

```
┌─────────────────────────────────────────────┐
│                 Flink API                    │
│  DataStream API（流处理）/ DataSet API（批处理）│
├─────────────────────────────────────────────┤
│              Flink Core Engine               │
│     StreamOperator / Function / JobGraph     │
├─────────────────────────────────────────────┤
│              Resource Manager                │
│   Standalone / YARN / Kubernetes / Mesos     │
├─────────────────────────────────────────────┤
│              Distributed Runtime             │
│    TaskManager / Slot / JobManager           │
└─────────────────────────────────────────────┘
```

**2. 核心组件**

| 组件 | 职责 |
|------|------|
| **JobManager** | 协调作业执行：调度 Task、协调 Checkpoint、协调故障恢复 |
| **TaskManager** | 执行具体的 Task，一个 TM 包含多个 **Slot** |
| **Slot** | 资源分配单元，定义了该 TaskManager 能处理的子任务数量 |

**3. 作业提交流程**

```
用户代码 → 生成 StreamGraph → 优化为 JobGraph → 提交到 JM → JM 调度到 TM → TM 启动后 JM 协调执行
```

**4. Task 执行模型**

- 一个 StreamGraph 的每个节点（Operator）可以设置**并行度（Parallelism）**
- 每个并行实例称为 **SubTask**，运行在 **Slot** 中
- 同一作业的多个 SubTask 通过 **Stream Shuffle** 传输数据

**5. 与 Spark 的核心区别**

- Spark 最早设计为**批处理优先**，流处理是后来添加的（RDD + 微批）
- Flink 从一开始就是**原生流处理**，批处理是流处理的特例（有限流）
- Flink 支持**精确一次（Exactly-Once）**语义，Spark Structured Streaming 也支持但实现不同

> 📎 来源：https://ververica.cn/flink-training-faq/flink-architecture/

## 题目二：Flink 的 Watermark（水位线）机制是什么？如何处理乱序数据？

### 参考答案

**1. 什么是 Watermark**

Watermark 是 Flink 用于处理**事件时间（Event Time）**乱序问题的机制，本质是一个**时间戳**，表示"**所有时间戳小于该 Watermark 的数据都已经到达**"。

```
Watermark T 含义：不会再有时间戳 < T 的事件到来
```

**2. 为什么需要 Watermark**

在现实场景中：
- 网络延迟
- 数据重试
- 分布式采集顺序不确定

导致数据到达顺序与事件产生顺序不一致（乱序）。

**3. Watermark 生成方式**

```java
// 方式1：周期性生成（Periodic Watermark）
WatermarkStrategy
    .forBoundedOutOfOrderness(Duration.ofSeconds(10))  // 允许10秒乱序
    .withTimestampAssigner((event, timestamp) -> event.getTimestamp());

// 方式2：标点式生成（Punctuated Watermark）
// 在特定业务事件时触发生成 Watermark
```

**4. Watermark 传播机制**

Watermark 在算子间**向下游传播**，遵循以下规则：

```
上游向下游广播 Watermark
每个算子维护多输入流的最小 Watermark（取最小值）
当 Watermark >= 窗口结束时间时，触发窗口计算
```

**5. 窗口计算触发时机**

```java
// 窗口何时被触发？
// 条件：Watermark >= 窗口结束时间 + allowedLateness
watermark >= window_end_time + allowed_lateness
```

**6. 乱序处理完整示例**

```java
stream.assignTimestampsAndWatermarks(
    WatermarkStrategy
        .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(10))
        .withTimestampAssigner((event, ts) -> event.getTimestamp())
)
.keyBy(e -> e.getKey())
.window(TumblingEventTimeWindows.of(Time.minutes(5)))
.allowedLateness(Time.minutes(2))  // 允许额外迟到2分钟
.reduce((a, b) -> a.merge(b));
```

**7. 窗口的四种状态**

| 状态 | 说明 |
|------|------|
| **未触发** | Watermark 未到达，窗口等待中 |
| **触发并输出** | Watermark 到达，窗口计算并输出结果 |
| **触发但保留** | allowedLateness 时间段内，迟到数据仍可触发窗口 |
| **清理删除** | 超过 allowedLateness，窗口状态被清除 |

> 📎 来源：https://nightlies.apache.org/flink/flink-docs-release-1.18/zh/docs/concepts/time-state/

## 题目三：Flink 的 Checkpoint（检查点）和 Savepoint 有什么区别？

### 参考答案

**1. Checkpoint 的核心概念**

Checkpoint 是 Flink **故障恢复**的基石，基于 **Chandy-Lamport 分布式快照算法**，实现**精确一次（Exactly-Once）**语义。

**2. Checkpoint vs Savepoint 对比**

| 特性 | Checkpoint | Savepoint |
|------|------------|-----------|
| **触发方式** | Flink 自动周期性触发 | 手动触发（用户命令） |
| **用途** | 故障自动恢复 | 计划性备份/升级 |
| **清理策略** | 作业删除时自动清理 | 需要手动删除 |
| **依赖** | CheckpointCoordinator | JobGraph 快照 |
| **兼容性** | 跟随作业版本 | 手动创建的快照通常向前兼容 |
| **存储位置** | 临时目录（可配置） | 用户指定目录 |
| **数据一致性** | Exactly-Once | Exactly-Once |

**3. Checkpoint 执行流程（Chandy-Lamport）**

```
JobManager 的 CheckpointCoordinator:
  1. 向所有 Source 算子发送 Checkpoint 触发消息（barrier）
  2. Source 算子快照自己的状态（偏移量等），写入状态后端
  3. Source 向下游广播 barrier
  4. 中间算子收到 barrier 后：
     - 阻塞输入（对齐已到达的 barrier）
     - 快照自己的状态
     - 向下游发送 barrier
  5. Sink 算子收到 barrier 后：
     - 快照状态
     - 向 CheckpointCoordinator 确认完成
  6. 所有算子都确认后，Checkpoint 完成
```

**4. Barrier 对齐机制（Barrier Alignment）**

```java
// 多输入算子（如 union、join）的 Barrier 对齐过程：
// 输入1: 收到 barrier-1，暂时缓存，继续接收后续数据
// 输入2: 收到 barrier-1，等待
// 输入1: 又收到数据 event-5，缓存
// 输入2: 收到 barrier-1，开始快照
// 输入1: 收到 barrier-1，对齐完成，开始快照
// 两个 barrier 对齐后，才向下游发送
```

**5. 状态后端（State Backend）**

| 状态后端 | 适用场景 | 存储介质 |
|----------|----------|----------|
| **HashMapStateBackend** | 小状态、低延迟 | JVM Heap |
| **EmbeddedRocksDBStateBackend** | 大状态、增量 Checkpoint | RocksDB（本地磁盘） |
| **FileSystemStateBackend** | 大状态、JobManager HA | HDFS/S3 |

**6. Checkpoint 配置示例**

```java
StreamExecutionEnvironment env = ...;
env.enableCheckpointing(60_000);  // 每60秒做一次 Checkpoint
env.getCheckpointConfig().setCheckpointStorage("hdfs:///checkpoint-dir");
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30_000);
env.getCheckpointConfig().setCheckpointTimeout(600_000);
env.getCheckpointConfig().setTolerableCheckpointFailureNumber(3);
```

> 📎 来源：https://nightlies.apache.org/flink/flink-docs-release-1.18/zh/docs/ops/state/checkpoints/

## 题目四：Flink 的状态管理机制是什么？如何实现故障恢复？

### 参考答案

**1. 状态（State）的分类**

Flink 有两种基本状态：

| 状态类型 | 说明 | 适用场景 |
|----------|------|----------|
| **Keyed State** | 基于 Key 的状态，每个 Key 独立 | keyBy 之后使用 |
| **Operator State** | 算子级别状态，所有实例共享 | Source（如 Kafka offset） |

**2. Keyed State 的具体类型**

```java
// 1. ValueState<T> - 单值状态
ValueState<Long> countState = getRuntimeContext().getState(
    new ValueStateDescriptor<>("count", Long.class));

// 2. ListState<T> - 列表状态
ListState<String> listState = getRuntimeContext().getListState(
    new ListStateDescriptor<>("list", String.class));

// 3. MapState<K, V> - 映射状态
MapState<String, Integer> mapState = getRuntimeContext().getMapState(
    new MapStateDescriptor<>("map", String.class, Integer.class));

// 4. ReducingState<T> - 聚合状态
// 5. AggregatingState<IN, OUT> - 聚合状态（可改变类型）
```

**3. 状态一致性保证**

Flink 的状态通过 **Checkpoint** 实现**精确一次**一致性：

```
正常执行：处理数据 → 更新状态
故障恢复：读取 Checkpoint → 恢复到快照状态 → 重新处理数据
```

**4. 故障恢复流程**

```
1. 作业运行中，CheckpointCoordinator 周期性触发快照
2. 每个算子将状态快照写入持久化存储（CheckpointStorage）
3. 故障发生时，JobManager 检测到 Task 失败
4. JobManager 从最近的 Checkpoint 恢复：
   - 重启失败的 Task
   - 从状态快照恢复算子状态
   - 从记录的 Kafka offset 重新消费（需配合 Kafka 的事务）
5. 重新处理从快照点之后的数据
```

**5. 状态过期与清理**

```java
// 状态 TTL（Time-To-Live）
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.days(7))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)  // 每次读写都更新
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)  // 过期数据不可见
    .build();

ValueStateDescriptor<T> descriptor = new ValueStateDescriptor<>(
    "state", TypeInformation.of(String.class));
descriptor.enableTimeToLive(ttlConfig);
```

**6. 增量 Checkpoint（Incremental Checkpoint）**

```java
// 配置增量 Checkpoint
env.getCheckpointConfig().setCheckpointStorage("hdfs:///checkpoint");
env.enableIncrementalCheckpoints();

// RocksDB 增量 Checkpoint：只保存变更的 SST 文件，而非全量状态
// 显著减少 Checkpoint 数据量和恢复时间
```

## 题目五：Flink 的窗口函数有哪些？滚动窗口、滑动窗口、会话窗口的区别？

### 参考答案

**1. Flink 窗口函数分类概览**

```
┌──────────────────────────────────────────────────┐
│                    Window                         │
├──────────────────────────────────────────────────┤
│         Time Window          │   Count Window     │
│  ┌─────────┬─────────┐       │                    │
│  │滚动窗口 │滑动窗口 │会话窗口│   全局窗口         │
│  └─────────┴─────────┴───────┘                    │
└──────────────────────────────────────────────────┘
```

**2. 滚动窗口（Tumbling Window）**

- 窗口大小固定，**不重叠**
- 每个元素属于且仅属于一个窗口

```java
// 滚动事件时间窗口：每5分钟一个窗口
stream.keyBy(e -> e.getKey())
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .sum("amount");

// 滚动计数窗口：每100条数据一个窗口
stream.keyBy(e -> e.getKey())
    .window(TumblingCountWindow.of(100))
    .sum("amount");
```

**3. 滑动窗口（Sliding Window）**

- 窗口大小固定，**可重叠**
- 由**窗口大小**和**滑动步长**两个参数决定

```java
// 滑动事件时间窗口：窗口大小10分钟，滑动步长5分钟
stream.keyBy(e -> e.getKey())
    .window(SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(5)))
    .sum("amount");

// 滑动计数窗口：窗口大小100条，滑动50条
stream.keyBy(e -> e.getKey())
    .window(SlidingCountWindow.of(100, 50))
    .sum("amount");

// 示意图：
// 窗口大小10分钟，滑动5分钟：
// |--w1--| |--w2--| |--w3--| |--w4--|
//   |--w2--| |--w3--| |--w4--| |--w5--|
```

**4. 会话窗口（Session Window）**

- 根据**会话间隔**动态创建窗口
- 当两条数据之间的间隔超过指定的 gap 时，开启新窗口

```java
// 会话窗口：会话间隔超过3分钟则开启新窗口
stream.keyBy(e -> e.getKey())
    .window(SessionWindow.withGap(Time.minutes(3)))
    .sum("amount");

// 动态间隙会话窗口（可动态调整）
stream.keyBy(e -> e.getKey())
    .window(SessionWindow.withDynamicGap(e -> e.getGapMillis()))
    .sum("amount");
```

**5. 全局窗口（Global Window）**

- 所有相同 Key 的数据被分配到**同一个窗口**
- 需要配合**触发器**使用，否则永远不会触发

```java
stream.keyBy(e -> e.getKey())
    .window(GlobalWindows.create())
    .trigger(CountTrigger.of(100))  // 每100条触发一次
    .sum("amount");
```

**6. 三种窗口对比**

| 特性 | 滚动窗口 | 滑动窗口 | 会话窗口 |
|------|----------|----------|----------|
| **窗口大小** | 固定 | 固定 | 动态 |
| **重叠性** | 不重叠 | 可重叠 | 不固定 |
| **数据分配** | 一对一 | 一对多 | 动态 |
| **适用场景** | 固定周期统计 | 平滑统计 | 不规则事件流 |
| **内存占用** | 低 | 较高 | 动态 |

**7. 增量聚合 vs 全量聚合**

```java
// 增量聚合（ReduceFunction / AggregateFunction）：边处理边聚合
stream.keyBy(e -> e.getKey())
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .reduce((value1, value2) -> {
        // 增量合并，状态只保存聚合结果
        value1.setAmount(value1.getAmount() + value2.getAmount());
        return value1;
    });

// 全量聚合（ProcessWindowFunction）：先缓存再计算
stream.keyBy(e -> e.getKey())
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .process(new ProcessWindowFunction<...>() {
        @Override
        public void process(Context context, Iterable<T> elements, 
                           Collector<O> out) {
            // 遍历窗口内所有元素
            for (T element : elements) { ... }
        }
    });
```
