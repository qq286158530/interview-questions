## 题目一：HBase 的数据写入流程是怎样的？为什么说 HBase 写入性能高？

### 参考答案

### HBase 数据写入完整流程

```
客户端                    RegionServer                  HDFS
  │                           │                          │
  │─── Put Request ──────────>│                          │
  │                           │                           │
  │                           │ 1. 写 WAL（Write-Ahead Log）
  │                           │─── Append WAL ──────────>│
  │                           │<─── Ack ────────────────│
  │                           │                           │
  │                           │ 2. 写 MemStore（内存）
  │                           │    （按 RowKey+Column 排序）
  │                           │                           │
  │<─── Ack ──────────────────│                          │
```

### 各步骤详解

**① WAL 写入**
- 数据首先写入 WAL（Write-Ahead Log），默认存储在 HDFS 上
- 目的：若 MemStore 宕机丢失，可通过 WAL 回放恢复数据
- WAL 采用 **顺序写入**，性能很高（HDFS 对顺序写做了优化）

**② MemStore 写入**
- 数据写入 MemStore（SkipList 结构，按 RowKey 字典序保证有序）
- 写入即返回客户端，因此 **HBase 写入是内存写 + 异步刷盘**，延迟极低
- 每个 MemStore 达到阈值（默认 128MB）后触发 Flush 生成 HFile

### 为什么 HBase 写入性能高？

| 优化点 | 说明 |
|--------|------|
| **顺序写 WAL** | WAL 写入 HDFS 是顺序追加，无需随机 IO |
| **内存写入** | 数据先写 MemStore，内存写入速度远快于随机磁盘写入 |
| **写放大控制** | Minor Compaction 减少不必要的 HFile 合并 |
| **无锁写** | HBase 写入不涉及行锁（乐观锁机制），并发写入吞吐量大 |
| **Region 分布** | 数据分散到多个 RegionServer 并行写入，充分利用集群能力 |
| **HDFS 优化** | HDFS 默认 3 副本写入可流水线传输，客户端只需等一个副本写完即可返回 |

### 写入性能瓶颈

- 热点 RowKey 导致写入集中到单一 RegionServer → **设计良好的 RowKey 是性能关键**
- WAL 写入阻塞（sync 等待）→ 可通过 `hbase.wal.provider` 选择不同 WAL 实现
- MemStore 刷写阻塞写入 → 监控 `flush queue size`，避免刷写风暴

📎 **来源：** [HBase 官方文档 - Writing Data](https://hbase.apache.org/book.html#writing)

## 题目二：HBase 如何保证数据强一致性？它与 Cassandra 的最终一致性有何本质区别？

### 参考答案

### HBase 的强一致性保证

HBase 是 **写后读一致（Write-After-Read-Consistency）** 的典型实现：

```
写操作：
Client ──> RegionServer ──> MemStore (flush) ──> HFile
                 │
                 └──> WAL（同步写，保证宕机可恢复）

读操作：
Client ──> RegionServer ──> MemStore（优先） + HFile（兜底）
```

**强一致性实现机制：**
1. **MemStore 刷写原子性**：Flush 操作是原子的，要么全部成功要么全部失败
2. **Region 级别 MVCC**：HBase 使用多版本并发控制（MVCC），读取时只能看到已提交（committed）的版本
3. ** WAL 保证**：数据写入 WAL 后才算成功，宕机可回放
4. **读写互斥**：同一行的写入通过 `row lock` 互斥，避免脏读

### HBase vs Cassandra 一致性模型对比

| 维度 | HBase | Cassandra |
|------|-------|-----------|
| **一致性模型** | 强一致性（写操作后所有副本一致） | 最终一致性（可配置一致性级别） |
| **副本写入** | 先写主副本，再异步复制到备副本 | 全部副本并行写入（QUORUM/NODESYNC） |
| **副本数** | `replication=3`，3个副本全部写入才算成功 | `replication=3`，可配置写入 N 个副本返回 |
| **读取路径** | 先读主副本，可能触发副本数据修复 | 读取最新副本（Read Repair），后台同步 |
| **CAP 定理** | 偏向 CP（一致性 + 分区容忍） | 偏向 AP（可用性 + 分区容忍） |
| **适用场景** | 金融、订单、库存等强一致性场景 | 日志、消息、社交Feed等高可用场景 |

### 本质区别

> **HBase**：数据先写入主副本所在的 RegionServer，通过 HDFS 副本机制保证 3 副本一致，读取无需协调，因此是 **同步复制 + 强一致**。
>
> **Cassandra**：数据同时写入多个节点（CL=QUORUM 时写入 N/2+1 节点即返回），读取时由客户端协调合并，是 **异步复制 + 最终一致**。

**一句话总结**：HBase 的强一致性来源于 HDFS 的副本同步写入机制（DataNode 直接保证）；Cassandra 则通过可配置一致性级别，在一致性与可用性之间灵活取舍。

📎 **来源：** [HBase 官方文档 - Consistency](https://hbase.apache.org/book.html#consistency)

## 题目三：HBase 的 Region 是什么？它是如何进行分裂（Split）的？分裂过程中服务会中断吗？

### 参考答案

### Region 的概念

Region 是 HBase 表 **水平分片** 的基本单元：

```
Table (按 RowKey 切分)
  ├── Region 1: [a    ~ gou]     → RegionServer-1
  ├── Region 2: [gou  ~ mo]     → RegionServer-2
  ├── Region 3: [mo   ~ ta]     → RegionServer-3
  └── Region 4: [ta   ~ zzz]    → RegionServer-4
```

- 每个 Region 默认存储一个列族的数据
- 初期一个表只有一个 Region，随着数据增长自动分裂
- Region 大小阈值由 `hbase.hregion.max.filesize` 控制（默认 10GB）

### Region 分裂（Split）机制

**触发条件：** 当一个 Region 下的所有 HFile 大小之和超过阈值时

**分裂流程：**

```
Step 1: RegionServer 创建 split.txt（记录分裂点），向 HMaster 报告
Step 2: HMaster 协调，在原 RegionServer 上执行分裂
Step 3: 关闭当前 Region（停止写入，强制刷写 MemStore）
Step 4: 将原 Region 数据文件分为两个子 Region
         ├── Region-A: [startKey ~ splitKey]
         └── Region-B: [splitKey ~ endKey]
Step 5: 在对应 RegionServer 上打开两个子 Region，恢复服务
Step 6: HMaster 更新 META 表，记录新 Region 位置
```

### 分裂过程中服务会中断吗？

**对于写入/更新：是的，会短暂不可用。**

| 阶段 | 影响 |
|------|------|
| **Region 下线** | 该 Region 不可写入，约 **几秒** |
| **分裂操作** | HDFS 文件复制（实际是引用，不复制数据） |
| **Region 上线** | 新 Region 上线，恢复服务 |

**对于读取：受影响但一般可容忍**

- 客户端如果有缓存的 Region 地址，仍会请求旧的 RegionServer
- 正常情况下 Master 会快速更新 META，客户端会重新路由
- 整体影响时间通常 **< 30 秒**

### 预分区（Pre-Splitting）优化

生产环境通常 **在创建表时就预分区**，避免自动分裂带来的性能抖动：

```bash
# 创建表时指定 4 个 Region 切分点
create 't_user', 'cf', SPLITS => ['a', 'g', 'm', 't']
```

**预分区的好处：**
- 数据一开始均匀分布，避免初期热点
- 减少自动分裂带来的服务抖动
- 可以结合业务 RowKey 特点设计最优分区

📎 **来源：** [HBase 官方文档 - Region Splits](https://hbase.apache.org/book.html#region.split)

## 题目四：HBase 的 Filter（过滤器）在扫描场景下有哪些常用实现？使用 Filter 有什么性能注意事项？

### 参考答案

### 常用 HBase Filter

HBase 提供了丰富的过滤器，在服务端（RegionServer）执行，减少网络传输数据量：

| 过滤器 | 类名 | 用途 |
|--------|------|------|
| **行键过滤器** | RowFilter | 按正则/比较器匹配 RowKey |
| **列族过滤器** | FamilyFilter | 过滤列族 |
| **列名过滤器** | QualifierFilter | 按列名（Qualifier）过滤 |
| **值过滤器** | ValueFilter | 按 Value 内容过滤 |
| **时间戳过滤器** | TimestampsFilter | 按时间戳范围过滤 |
| **单列值过滤器** | SingleColumnValueFilter | **最常用**，根据某列的值决定是否返回整行 |
| **列前缀过滤器** | ColumnPrefixFilter | 按列名前缀过滤 |
| **分页过滤器** | PageFilter | 按行数分页 |
| **首次键过滤器** | FirstKeyOnlyFilter | 只返回每行第一个 KeyValue（用于高效计数） |
| **全键匹配** | BinaryPrefixComparator / BinaryComparator | 按二进制比较 |

### 单列值过滤器（最常用示例）

```java
// 场景：查询年龄（age列）大于 25 的用户
SingleColumnValueFilter filter = new SingleColumnValueFilter(
    Bytes.toBytes("cf"),           // 列族
    Bytes.toBytes("age"),           // 列名
    CompareOperator.GREATER,        // 比较操作
    Bytes.toBytes("25")             // 比较值
);
filter.setFilterIfMissing(true);   // 如果该列不存在则不返回该行

Scan scan = new Scan();
scan.setFilter(filter);
```

### 性能注意事项

> ⚠️ **Filter 虽然在服务端执行，但以下问题会严重影响性能：**

**1. Filter 无法利用 BlockIndex 加速**
- HBase 的 BlockIndex 是基于 RowKey 建立的
- 非 RowKey 过滤（如 `ValueFilter`）必须扫描所有数据块
- **后果**：全表扫描，数据量大时 IO 开销极大

**2. 组合 Filter 的短路问题**
```java
// 使用 FilterList 组合多个过滤器
FilterList list = new FilterList(FilterList.Operator.MUST_PASS_ALL);
// MUST_PASS_ALL: 全部满足才返回（AND）
// MUST_PASS_ONE: 满足任一即可（OR）
```
- **建议**：将过滤效果最强的过滤器放前面（如 RowKey 过滤先于列值过滤）

**3. setBatch 和 setCaching 的区别**

| 参数 | 说明 |
|------|------|
| `setBatch(int)` | 每行返回的 KeyValue 数量上限（适合列数很多的情况） |
| `setCaching(int)` | 每次 RPC 从 RegionServer 批量取的行数（默认 100） |
| `setCacheBlocks(boolean)` | 是否缓存结果（扫描时一般设为 false） |

**4. 最佳实践**

- **尽量在 RowKey 层面过滤**：将过滤条件设计进 RowKey，利用范围扫描替代 Filter
- **先粗过滤后细过滤**：先通过 RowKey/列族过滤，再使用列值过滤器
- **使用 PageFilter 做翻页**：不要用 `scan.setStartRow()` 手动计算偏移（偏移量越大越慢）
- **扫描时关闭 BlockCache**：`scan.setCacheBlocks(false)`，避免污染热点数据块的缓存

📎 **来源：** [HBase 官方文档 - Filters](https://hbase.apache.org/book.html#filter)

## 题目五：HBase 与 HDFS 的关系是什么？为什么 HBase 依赖 HDFS？它能脱离 HDFS 独立运行吗？

### 参考答案

### HBase 与 HDFS 的关系

HBase 是 **构建在 HDFS 之上的分布式列式存储数据库**，两者是 **层叠关系**：

```
┌─────────────────────────────────────┐
│            Application              │
│               HBase API              │
├─────────────────────────────────────┤
│  HBase（分布式数据库）               │
│  ┌──────────┐  ┌──────────────────┐ │
│  │ HMaster  │  │  RegionServer    │ │
│  └──────────┘  └──────────────────┘ │
│       │                │             │
│       └────────┬───────┘             │
│                ▼                     │
│         HDFS Client                   │
├─────────────────────────────────────┤
│         HDFS（分布式文件系统）        │
│   ┌─────────┐  ┌─────────┐  ┌─────────┐
│   │DataNode │  │DataNode │  │DataNode │
│   │ (副本1) │  │ (副本2) │  │ (副本3) │
│   └─────────┘  └─────────┘  └─────────┘
└─────────────────────────────────────┘
```

### 为什么 HBase 必须依赖 HDFS？

| HDFS 提供的特性 | HBase 如何利用 |
|----------------|----------------|
| **高可靠性副本** | HBase 数据默认 3 副本，任意 DataNode 宕机不丢数据 |
| **数据本地性** | RegionServer 优先读取本地 DataNode 的数据，减少网络 IO |
| **顺序写入优化** | HBase 的 MemStore flush 后生成 HFile，HDFS 顺序写入性能极高 |
| ** Namenode 元数据管理** | Region 信息、元数据全部存在 HDFS 上 |
| **容错机制** | DataNode 宕机后 HDFS 自动副本重建，HBase 自动恢复 |

### HBase 能脱离 HDFS 独立运行吗？

**理论上可以，但生产环境不建议。**

HBase 的存储层是 **可插拔的（Pluggable）**，通过 `hbase.storage.schema` 可以切换：

| 存储后端 | 说明 |
|----------|------|
| **HDFS（默认）** | 生产环境标准选择 |
| **Apache Kudu** | 适合同时需要随机读写和顺序扫描的场景 |
| **Apache Ivy / 本地文件系统** | 仅用于测试和开发 |
| **AWS S3** | 云原生方案（通过 S3 存储 HFile，但一致性挑战较大） |

**生产环境选择 HDFS 的原因：**
1. HDFS 的副本机制与 HBase 的高可用天然契合
2. HDFS 的 DataNode 崩溃恢复由 HDFS 自动处理，HBase 无需关心
3. HDFS 的 Block 复制策略（机架感知）与 HBase Region 分布策略协同
4. HDFS 社区成熟，与 Spark、Hive、MapReduce 等生态无缝集成

### 一句话总结

> HBase = **HDFS 上的内存数据库** + **有序稀疏映射表**
>
> HDFS 负责底层数据的可靠存储和副本管理，HBase 负责上层的随机读写和结构化查询，两者结合才实现了"高性能 + 高可靠"的分布式存储。

📎 **来源：** [HBase 官方文档 - HDFS](https://hbase.apache.org/book.html#hdfs)
