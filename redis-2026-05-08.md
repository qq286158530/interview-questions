## 题目一：Redis 如何实现分布式锁？Redisson 的实现原理是什么？有哪些注意事项？

### 答案

### 1. 基础实现：SETNX + EXPIRE

Redis 实现分布式锁最经典的写法是使用 `SET key value NX PX ttl` 命令：

```java
// 加锁
String lockKey = "product:lock:10086";
String uuid = UUID.randomUUID().toString();
Boolean result = redisTemplate.opsForValue()
    .setIfAbsent(lockKey, uuid, 30, TimeUnit.SECONDS);

// 释放锁（必须判断 value 才能释放自己的锁）
if (result) {
    try {
        // 业务逻辑
    } finally {
        if (uuid.equals(redisTemplate.opsForValue().get(lockKey))) {
            redisTemplate.delete(lockKey);
        }
    }
}
```

**为什么用 `SET ... NX PX` 而不是先 `SETNX` 再 `EXPIRE`？**

因为 `SETNX` 和 `EXPIRE` 是两条命令，不是原子的。如果进程在 `SETNX` 成功后、`EXPIRE` 执行前崩溃，锁就永远不会过期，形成**死锁**。

### 2. Redisson 分布式锁原理

Redisson 是 Redis 史上最完善的分布式锁实现，内部基于 Lua 脚本保证原子性，核心原理：

#### 加锁（`tryLock`）：
```java
// Redisson 内部执行 Lua 脚本
// KEYS[1] = 锁名称，ARGV[1] = 过期时间(ms)，ARGV[2] = 唯一标识（UUID+ThreadId）
if (redis.call('exists', KEYS[1]) == 0) then
    redis.call('hset', KEYS[1], ARGV[2], '1');
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return 1;
end;
```

- 使用 **Hash 结构** 存储锁，field 是唯一标识，value 是计数器
- 支持**可重入锁**（同一线程可多次加锁）
- **看门狗（WatchDog）机制**：如果未指定过期时间，每 10 秒自动续期（`expire` → `pexpire`），防止业务未执行完锁就过期
- 底层使用 **Netty** 实现，支持公平锁、非公平锁、读写锁等多种模式

#### 释放锁：
```java
// Lua 脚本：只有 value 匹配才删除，保证只释放自己的锁
if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then
    return redis.call('hdel', KEYS[1], ARGV[2]);
else
    return 0;
end;
```

### 3. 分布式锁注意事项

| 注意点 | 说明 |
|-------|------|
| **锁必须设置 TTL** | 防止进程崩溃后锁无法释放 |
| **value 必须唯一** | 用 UUID/ThreadId，防止误删他人锁 |
| **释放锁必须原子** | 用 Lua 脚本或 `if GET == DEL`（后者不推荐，仍有并发问题） |
| **集群环境下** | 使用 RedLock 算法（多数节点加锁成功才算成功），单节点不可靠 |
| **锁粒度尽量小** | 锁范围越小并发度越高 |
| **幂等性设计** | 配合幂等接口，即使锁失效也能防止重复操作 |

### 4. Redisson vs 手写实现

| 对比项 | 手写 SETNX | Redisson |
|-------|-----------|---------|
| 原子性 | 依赖 Lua 脚本才有 | 原生支持 |
| 可重入 | 不支持 | 支持 |
| 锁续期 | 不支持 | 看门狗自动续期 |
| 公平/非公平 | 需自行实现 | 多种模式内置 |
| 集群支持 | 无 | RedLock 支持 |
| 复杂度 | 简单但易踩坑 | 功能完善 |

## 题目二：Redis Cluster 的数据分片原理是什么？槽迁移过程中如何保证服务可用性？

### 答案

### 1. 数据分片原理：槽（Slot）

Redis Cluster 没有使用一致性哈希，而是将整个 key space 划分为 **16384 个槽（Slot）**，每个 master 节点负责一部分槽：

```
槽编号 = CRC16(key) % 16384
```

**为什么用 16384？**
- 16384 = 2^14，便于高效管理和传输
- 节点间心跳包携带所有槽信息，16384 个槽信息约 2KB，通信开销合理
- 节点数量上限约 1000 个时，16384 的分片更均衡

### 2. 集群结构

- 每个 master 可以有 **0~N 个 replica（从节点）**
- 集群内所有节点两两互联（`Gossip` 协议）
- 节点故障检测和自动故障转移由 master 的从节点完成

```
[Master A] ---- 槽 0-5460 ---- [Replica A1]
[Master B] ---- 槽 5461-10922 ---- [Replica B1]
[Master C] ---- 槽 10923-16383 ---- [Replica C1]
```

### 3. 槽迁移过程

槽迁移是 Redis Cluster 运维中最核心的操作，例如将槽 5000 从节点 A 迁移到节点 B：

#### 迁移步骤：

1. **目标节点 importing**：在目标节点 B 执行 `CLUSTER SETSLOT 5000 IMPORTING <sourceNodeId>`，B 开始记录迁入信息
2. **源节点 migrating**：在源节点 A 执行 `CLUSTER SETSLOT 5000 MIGRATING <targetNodeId>`，A 标记该槽正在迁出
3. **迁移 keys**：遍历槽 5000 的所有 key，逐个 `MIGRATE targetIP targetPort key 0 timeout COPY FLUSH` 迁移
4. **广播新分配**：所有节点执行 `CLUSTER SETSLOT 5000 NODE <targetNodeId>`，同步槽归属信息

#### MOVED 重定向：

迁移过程中，如果客户端请求 key 所在的槽正在迁移：
- 如果 key 还在源节点 → 返回 `MOVED <slot> <targetIP:port>`
- 如果 key 已迁移到目标节点 → 返回 `MOVED <slot> <targetIP:port>`

**ASK 重定向**（迁移中间态）：
```
ASK 5000 192.168.1.100:6379
```
表示"这个槽正在迁移，先去目标节点执行，但不要更新客户端的槽缓存"。与 MOVED 的区别：ASK 不会更新客户端本地路由表。

### 4. 迁移期间服务可用性保证

| 策略 | 说明 |
|-----|------|
| **COPY 模式** | `MIGRATE ... COPY`，源节点保留 key 直至全部迁移完成，期间仍可提供服务 |
| **缓存客户端路由** | 使用智能客户端（如 JedisCluster），本地缓存槽→节点映射，遇到 ASK/MOVED 自动重试 |
| **批量迁移** | 使用 `CLUSTER SETSLOT <slot> MIGRATING` 分批迁移，而非一次性迁移整个槽 |
| **超时设置** | `MIGRATE` 命令设置合理 timeout，防止单个 key 阻塞 |

### 5. 故障检测与转移

- 集群内节点每 1 秒发送 PING，其他节点 PONG 响应
- 超过 `cluster-node-timeout`（默认 15s）未响应，标记为**疑似下线（PFAIL）**
- 超过半数 master 节点认为某节点 PFAIL，则标记为**客观下线（FAIL）**
- 从节点收到 FAIL 消息后，向集群广播 `failover` 请求，征集选票
- 获得多数票的从节点成为新的 master，旧 master 重新上线后成为其从节点

## 题目三：Redis 大 key 问题如何排查和解决？热 key 问题又该如何处理？

### 答案

### 1. 什么是大 key 和热 key

| 类型 | 定义 | 危害 |
|-----|------|------|
| **大 key** | 单个 key 的 value 体积过大（如 String > 10MB，Collection > 1万元素） | RDB/AOF 重写阻塞、网络带宽占用、集群迁移慢、内存不均 |
| **热 key** | QPS 极高的 key（如热点商品、明星热搜） | 单节点 CPU/网卡瓶颈、请求倾斜 |

### 2. 大 key 排查方法

#### 方法一：`redis-cli --bigkeys`
```bash
redis-cli --bigkeys
# 扫描并输出每种类型最大的 key
```

#### 方法二：`MEMORY USAGE` 命令
```bash
# 查看单个 key 占用内存
MEMORY USAGE user:info:10086
# 返回字节数
```

#### 方法三：`SCAN` + `DEBUG OBJECT`
```bash
redis-cli --scan | head -10000 | xargs -I {} redis-cli DEBUG OBJECT {} 2>/dev/null | grep serializedlength
```

#### 方法四：RDB 分析工具 `redis-rdb-tools`
```python
# 生成内存报告
rdb --command memory dump.rdb
# 找出最大的 key
rdb --bigkeys dump.rdb
```

### 3. 大 key 解决方案

| 方案 | 适用场景 | 实现方式 |
|-----|---------|---------|
| **拆分存储** | 大 Hash/List/Set | 按 ID 分桶，如 `hash:{id}:{bucket}` |
| **压缩存储** | 文本类大 String | `SETRANGE` 或压缩后存储（gzip） |
| **改为粒度更小的 key** | 业务允许重构 | 将一个大 key 拆成多个小 key |
| **lazy delete** | 必须删除大 key | `UNLINK`（非阻塞删除）而非 `DEL` |
| **定期清理** | 可接受延迟删除 | 分批次 `SCAN` + `UNLINK` |

**Hash 拆分的例子：**
```java
// 原设计：一个 Hash 存所有用户资料，用户量大了就成大 key
HSET user:profile:all user:001 '{"name":"张三",...}'

// 优化：按用户 ID 末位分桶
HSET user:profile:0 user:001 '{"name":"张三",...}'
HSET user:profile:1 user:002 '{"name":"李四",...}'
// 每个桶最多 1000 用户
```

### 4. 热 key 排查方法

```bash
# 实时统计 hot key（Redis 4.0+）
redis-cli --hotkeys

# 通过 Redis 监控（如 Prometheus + redis_exporter）观察 QPS
# 或通过 TCP DUMP 分析请求模式
```

### 5. 热 key 解决方案

| 方案 | 说明 | 优点 | 缺点 |
|-----|------|-----|------|
| **本地缓存** | 在应用服务器内存缓存热 key（如 Caffeine/Guava Cache） | 最快，绕过网络 | 数据一致性难保证 |
| **热点 key 打散** | 在 key 后缀加随机数，将请求分散到多个节点 | 实现简单 | 数据冗余，一致性更差 |
| **多副本读写分离** | 读从节点，分担主节点压力（注意延迟） | 简单有效 | 读取可能不是最新 |
| **Redis Cluster 多副本** | 热点 key 的 master 有多个 replica | 负载均衡 | 运维复杂度增加 |
| **热点探测 + 预热** | 用 `MONITOR` 或 Proxy 层统计热 key，提前加载到缓存 | 主动防御 | 需要额外的探测系统 |

**本地缓存示例：**
```java
// 使用 Spring Cache + Caffeine 本地缓存
@Cacheable(value = "hotProducts", key = "#productId, T(com.example.util.CacheUtil).localCache())
public Product getHotProduct(Long productId) {
    return productService.findById(productId);
}
```

## 题目四：Redis 和 Memcached 有何区别？该如何选择？

### 答案

### 1. 核心区别对比

| 维度 | Redis | Memcached |
|-----|-------|-----------|
| **数据结构** | String/Hash/List/Set/Zset/Bitmap/HyperLogLog/Geo/Stream | 仅 String（二进制安全） |
| **持久化** | 支持 RDB + AOF | 不支持，纯内存 |
| **主从复制** | 支持，支持读写分离 | 不支持 |
| **集群** | Redis Cluster / Codis / 主从 | 客户端分片（hash mod）|
| **事务** | 支持 MULTI/EXEC/WATCH（Lua 脚本原子性更强） | 不支持 |
| **过期策略** | 惰性删除 + 定期删除 | 惰性删除（LRU）|
| **内存管理** | 自有内存管理（jemalloc），内存分配灵活 | slab allocator，固定 chunk 大小 |
| **并发模型** | 单进程单线程（6.0 后支持多线程 IO）| 多线程（Slab Rotations）|
| **分布式** | 内置 Cluster | 需第三方实现 |
| **数据量** | 支持 > 100GB | 数据量过大会导致 slab 碎片 |
| **PUB/SUB** | 支持 | 不支持 |
| **Lua 脚本** | 支持复杂原子操作 | 不支持 |

### 2. 关键场景分析

#### 为什么 Memcached 用 slab allocator 而 Redis 用 jemalloc？

**Memcached 的 slab allocator：**
- 将内存按固定大小（slab class，如 64B/128B/256B...）划分 chunk
- 相同大小 chunk 组成 slab，避免内存碎片
- 缺点：内存浪费严重（存 100B 数据需要 128B chunk），且无法动态调整

**Redis 的 jemalloc：**
- 按需分配内存，小块优先复用
- 适合多样化的 value 大小
- 内存利用率更高

### 3. 如何选择

**选 Redis 的场景：**
- 需要多种数据结构（List 队列、Set 去重、Zset 排行榜、Hash 对象）
- 需要持久化或主从复制
- 需要集群高可用
- 需要发布/订阅、Stream 消息队列
- 需要 Lua 脚本实现原子操作
- 数据量 > 10GB，需要内存高效利用

**选 Memcached 的场景：**
- 纯缓存，只存 String（Session、页面缓存）
- 追求极端 QPS（无事务/持久化开销）
- 多线程并发写入（Memcached 多线程，Redis 单线程需要开多个实例）
- 简单 kv 存储，不需要复杂数据结构

### 4. 实际项目经验

**典型错误：把 Memcached 当数据库用**

Memcached 不支持持久化，一旦重启数据全丢。有人用它存用户会话数据，服务重启后大量用户被迫重新登录。

**Redis 替代 Memcached 方案：**
```java
// 原 Memcached 用法
memcached.get("session:" + userId);
memcached.set("session:" + userId, userSession, expireSeconds);

// 改为 Redis（直接替换，TTL 替代过期）
redis.setex("session:" + userId, expireSeconds, userSession);
redis.get("session:" + userId);
```

## 题目五：Redis 管道（Pipeline）和事务（MULTI/EXEC）有什么区别？Lua 脚本有什么优势？

### 答案

### 1. 管道（Pipeline）

**解决的问题：** 减少网络往返次数（RTT）。

Redis 执行一条命令的流程：
```
客户端发送命令（请求）→ Redis 执行 → 返回结果（响应）
```
如果需要执行 100 条命令，客户端和 Redis 之间有 100 次网络往返（RTT）。

Pipeline 将多个命令打包，一次性发送给 Redis，Redis 按顺序执行并打包返回：

```java
// Jedis 示例
Pipeline pipeline = jedis.pipelined();
for (int i = 0; i < 100; i++) {
    pipeline.incr("counter");
    pipeline.get("counter");
}
List<Object> results = pipeline.exec();
pipeline.close();
```

**特点：**
- 客户端侧批量打包，Redis 侧顺序执行
- **不保证原子性**，中间某条失败不会影响其他命令
- 适合批量读写的非事务场景

### 2. 事务（MULTI/EXEC）

```java
// Jedis 事务示例
Transaction tx = jedis.multi();
tx.set("name", "Alice");
tx.incr("age");
tx.exec();  // 原子性执行
```

**Redis 事务特性：**
- `MULTI` 开启事务，后续命令进入队列
- `EXEC` 一次性执行队列中所有命令
- **部分支持回滚**：如果 EXEC 前命令本身有语法错误，整个事务失败；如果 EXEC 后某条执行失败，其他命令已执行（Redis 不支持自动回滚）
- `WATCH` 支持乐观锁，监视 key 变化后事务取消

**Redis 事务为什么不支持真正的回滚？**
Redis 作者的观点：代码错误应该尽早暴露，事务回滚会掩盖错误。真正需要原子性的场景用 Lua 脚本。

### 3. Pipeline vs 事务 vs Lua

| 特性 | Pipeline | 事务（MULTI/EXEC） | Lua 脚本 |
|-----|----------|-------------------|---------|
| 原子性 | ❌ 无 | ✅ 基本保证 | ✅ 原子性最强 |
| 回滚 | ❌ 不支持 | ⚠️ 不自动回滚 | ✅ 可自定义回滚逻辑 |
| 监视（Watch） | ❌ 不支持 | ✅ 支持 | ❌ 不支持 |
| 减少 RTT | ✅ 显著减少 | ✅ 显著减少 | ✅ 显著减少 |
| 执行失败处理 | 需业务代码处理 | 不自动回滚 | 脚本内可控制 |
| 灵活性 | 低 | 中 | 高 |
| 适用场景 | 批量读写，非原子 | 批量写，希望原子 | 复杂原子逻辑 |

### 4. Lua 脚本优势

Redis 在执行 Lua 脚本时具有原子性保证，脚本执行期间不会执行其他命令：

```lua
-- Lua 脚本：分布式锁实现
local key = KEYS[1]
local uuid = ARGV[1]
local ttl = ARGV[2]

if redis.call('exists', key) == 0 then
    redis.call('hset', key, uuid, '1')
    redis.call('pexpire', key, ttl)
    return 1
elseif redis.call('hexists', key, uuid) == 1 then
    redis.call('hincrby', key, uuid, 1)
    redis.call('pexpire', key, ttl)
    return 1
else
    return 0
end
```

**Lua 脚本优势：**
1. **原子性**：脚本执行期间不会被其他命令插入
2. **可编程**：支持条件判断、循环、函数，实现复杂原子逻辑
3. **减少网络开销**：一个脚本可以执行多条 Redis 命令
4. **复用**：脚本缓存在 Redis 中，通过 `EVALSHA` 多次调用
5. **自定义逻辑**：可根据返回值/条件决定下一步操作

**EVAL vs EVALSHA：**
- `EVAL script numKeys keys[] args[]`：直接执行脚本
- `EVALSHA sha1 numKeys keys[] args[]`：执行缓存的脚本（SHA1 校验），避免每次传脚本内容

```bash
# 脚本缓存
redis-cli SCRIPT LOAD "$(cat lock.lua)"
# 返回 sha1 值，如 "a5c3c5b3b..."

# 后续调用
redis-cli EVALSHA a5c3c5b3b... 1 mylock uuid 30000
```

### 5. 生产环境建议

- **批量数据入库**：Pipeline（高性能）
- **库存扣减、账户转账**：Lua 脚本（强原子性）
- **批量读取缓存**：Pipeline
- **需要 WATCH 乐观锁**：事务（MULTI/EXEC）
- **跨 Redis 实例操作**：Lua 脚本（不支持，只能单实例）
