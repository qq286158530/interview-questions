## 题目 1：HBase 的架构由哪些核心组件组成？各自的作用是什么？

### 参考答案

HBase 采用经典的 **Master/Slave 架构**，核心组件如下：

| 组件 | 角色 | 主要职责 |
|------|------|----------|
| **HMaster** | 协调者（馆长） | 负责 Table 和 Region 的管理：建表/删表、Region 分布调整、负载均衡、RegionServer 宕机后迁移 Region |
| **HRegionServer** | 数据节点（书架管理员） | 维护 Region，处理读写请求；负责 Region 的 split（拆分）和 compact（合并） |
| **ZooKeeper** | 协调服务（导航系统） | 保障 HMaster 高可用（选举）、实时监控 RegionServer 上下线、存储 -ROOT- 表和 META 表的寻址入口 |
| **HDFS** | 底层存储（仓库） | HBase 所有数据文件（HFile、WAL）最终都存储在 HDFS 上，利用其副本机制保证高可靠 |

**读写数据路径简述：**
- 写：Client → ZooKeeper → 找到 META 表位置 → 访问 RegionServer → 先写 WAL → 再写 MemStore（满时刷写成 HFile）
- 读：Client → ZooKeeper → 找到 META → 定位 RegionServer → 依次从 MemStore、BlockCache、HFile 中查找（类似 LSM-Tree 读路径）

## 题目 2：说说 HBase 的 RowKey 设计原则。

### 参考答案

RowKey 是 HBase 中数据的唯一标识，设计好坏直接影响读写性能和负载均衡。核心原则如下：

### 1. 散列原则（避免热点）
- 如果 RowKey 按时间戳或顺序递增（如 `timestamp_xxx`），会导致所有写入集中到同一个 RegionServer，产生**热点问题**。
- 解决：将 RowKey 高位设为散列字段（如 MD5 前缀、随机数），低位放时间字段，使数据均衡分布到各个 Region。

### 2. 长度原则
- RowKey 长度建议 **10~100 字节**，不超过 16 字节，最大 64KB。
- 过长的 RowKey 会占用 MemStore 空间（每行数据都以 RowKey 为索引），减少实际数据的存储量。

### 3. 唯一原则
- RowKey 必须全局唯一，重复的 RowKey 后写入的数据会**覆盖**先前的数据（以最新时间戳版本为准）。

### 4. 业务关联原则
- 将查询时最常用的条件放在 RowKey 高位，提高查询效率。
- 常见技巧：**加盐**（前缀随机数）、**哈希**（确定性的 Hash 值）、**反转**（如手机号反转避免固定前缀热点）。

## 题目 3：MemStore 和 HFile 的作用是什么？数据写入的完整流程是怎样的？

### 参考答案

### MemStore（写缓冲区）
- MemStore 是 **HBase 的写缓冲区**，每个列族对应一个 MemStore。
- 数据写入时先写入 WAL（Write-Ahead Log）保证持久性，然后写入 MemStore。
- 当 MemStore 满（默认约 128MB），会**刷写（flush）**到磁盘，生成一个新的 HFile。

### HFile（HBase 底层存储格式）
- HFile 是 HBase 在 HDFS 上的**实际存储格式**，每个 HFile 属于一个列族，一个列族可以有多个 HFile。
- HFile 采用 **LSM-Tree（Log-Structured Merge-Tree）** 结构，对磁盘顺序写入做了优化，写入速度很快。
- HFile 内部有索引块，支持高效地根据 RowKey 查找数据。

### 完整写入流程
1. **写 WAL**：Client 将数据写入 RegionServer 的 WAL（预写日志），防止服务器崩溃数据丢失。
2. **写 MemStore**：数据同时写入 MemStore（内存），此时对客户端可见。
3. **响应客户端**：MemStore 和 WAL 都写入成功后，返回写成功。
4. **触发 Flush**：当 MemStore 达到阈值，刷写成 HFile 文件。
5. **触发 Compaction**：随着 HFile 增多，后台会进行 minor/major compaction，合并小文件。

## 题目 4：HBase 的 Compaction（合并）机制有哪两种？有什么区别？

### 参考答案

HBase 通过 Compaction 将多个小 HFile 合并成大 HFile，减少磁盘文件数量，提升查询性能，同时清理过期数据和删除标记（tombstone）。

### Minor Compaction（小合并）
- **触发条件**：StoreFile 数量达到 `hbase.hstore.compactionThreshold`（默认 3）。
- **行为**：将**多个小文件**合并成一个较大的 HFile，只做有限的清理（版本合并、删除标记清理）。
- **特点**：执行快、影响小，通常在后台静默进行。

### Major Compaction（大合并）
- **触发条件**：默认每周一次，或手动触发。
- **行为**：将一个 Region 下**所有 HFile 合并成一个**大文件。
- **特点**：
  - 清理该 Region 所有过期数据（根据 TTL）和被删除的 Cell（tombstone 标记）。
  - 合并后删除被清理的文件，释放磁盘空间。
  - 对磁盘 I/O 影响较大，建议在业务低峰期执行。

### 两者核心区别

| | Minor Compaction | Major Compaction |
|---|---|---|
| 合并范围 | 部分 StoreFile | 某 Region 下所有 StoreFile |
| 触发频率 | 较频繁 | 较稀疏（默认每周） |
| 清理 tombstone | 部分清理 | 完全清理 |
| 性能影响 | 较小 | 较大 |
| 配置参数 | `hbase.hstore.compactionThreshold` | `hbase.hstore.compaction.max`、TTL 相关 |

## 题目 5：HBase 的协处理器（Coprocessor）是什么？有哪些类型？各自的使用场景？

### 参考答案

协处理器是 HBase 提供的**扩展机制**，允许用户将自定义逻辑运行在 RegionServer 上，直接操作数据而不需要额外计算框架。

### 两类协处理器

#### 1. Observer（观察者）—— 类似触发器
- **原理**：在特定事件前后自动触发回调，类似于数据库的触发器。
- 子类型及触发时机：
  - **PreGetObserver**：在 GET 请求处理之前调用，可用于实现**列级权限控制**、**数据过滤**。
  - **PostGetObserver**：在 GET 返回结果之后调用，可用于**审计日志**、**结果脱敏**。
  - **PrePutObserver / PostPutObserver**：在数据写入前/后调用，可用于**数据校验**、**自动填充字段**、**写前检查**。
  - **PreDeleteObserver / PostDeleteObserver**：删除前后调用。
  - **RegionObserver**：更细粒度地监控 Region 生命周期事件。

- **典型场景**：
  - 实现**二级索引**（写入主表时同步更新索引表）。
  - **权限控制**（特定用户只能读取特定列）。
  - **审计日志**（记录每次读写操作）。

#### 2. Endpoint（端点）—— 类似存储过程
- **原理**：将自定义的计算逻辑**下发到数据所在节点**执行（MapReduce 风格），减少数据传输。
- 类似关系型数据库的**存储过程**，但更强大——支持分布式并行执行。
- **典型场景**：
  - 在服务端直接执行**聚合计算**（COUNT、SUM、AVG），减少网络开销。
  - 实现**自定义过滤器**，在服务端完成复杂条件筛选。
  - **动态列族**：根据条件动态创建列。

### 加载方式
```xml
<!-- hbase-site.xml -->
<property>
  <name>hbase.coprocessor.user.region.classes</name>
  <value>com.example.MyObserver,com.example.MyEndpoint</value>
</property>
```
或在建表时通过 `coprocessor` 参数指定。
