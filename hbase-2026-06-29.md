## 题目一：HBase 的读写流程是什么？

### 参考答案

**写流程：**

1. **Client 访问 ZooKeeper**：首先Client访问ZooKeeper，获取 `-ROOT-` 表所在的位置信息（Meta表在哪个HRegionServer上）。
2. **访问 Meta 表**：Client 从 ZooKeeper 获取的地址访问 Meta 表，获取待写入数据对应的 Region 和 HRegionServer 信息。
3. **写入请求**：Client 向目标 HRegionServer 发起写入请求。
4. **写入 WAL**：先将数据写入 Write-Ahead Log（WAL），防止数据丢失。
5. **写入 MemStore**：将数据写入 MemStore（内存缓存），并保证数据有序。
6. **写入成功**：WAL 和 MemStore 均写入成功后，返回写入成功响应。
7. **MemStore 刷写**：当 MemStore 达到阈值（默认128MB），触发 flush 操作，将数据刷写到磁盘生成 HFile。
8. **Compaction 合并**：当 StoreFile 数量过多时，触发 Compact 合并操作。
9. **Region 分裂**：当 Region 过大时，触发 Split 操作将 Region 一分为二。

**读流程：**

1. **访问 ZooKeeper**：Client 先访问 ZooKeeper，获取 Meta 表所在的 HRegionServer。
2. **读取 Meta 表**：Client 访问 Meta 表，获取目标数据所在的 Region 和 HRegionServer。
3. **查询数据**：Client 向目标 HRegionServer 发起读请求，先查 MemStore（内存），再查 StoreFile（磁盘）。
4. **返回结果**：HRegionServer 将查询结果返回给 Client。

> 📌 **面试亮点**：读写流程是HBase最基础的八股文，建议配合架构图理解 ZooKeeper、Meta表、RegionServer 在其中的协调作用。

## 题目二：HBase RowKey 设计原则有哪些？如何避免热点问题？

### 参考答案

HBase 的 RowKey 设计需要遵循以下三大原则：

**1. 长度原则**
- RowKey 长度建议控制在 10~100 字节，越短越好，不超过 16 字节
- 原因：HFile 中 KeyValue 是按 Key-Value 存储的，RowKey 过长会占用大量存储空间；同时 MemStore 缓存效率降低，影响检索性能

**2. 散列原则**
- 如果 RowKey 按时间戳顺序递增，不要将时间放在高位，否则所有新数据都写入同一个 Region，造成热点
- 建议将 RowKey 高位作为散列字段（由程序循环生成），低位放时间字段，使数据均衡分布到各个 RegionServer

**3. 唯一原则**
- RowKey 必须在设计上保证唯一性，重复的 RowKey 后写入的数据会覆盖前面的数据

### 热点问题解决方案

| 方法 | 原理 | 优缺点 |
|------|------|--------|
| **加盐（Salting）** | 在 RowKey 前面添加随机前缀，使数据分散到不同 Region | 写均匀，但读无法预测 |
| **哈希（Hashing）** | 对 RowKey 做哈希，客户端可重构完整 RowKey 进行精确读取 | 写均匀，读可预测 |
| **反转（Reversing）** | 反转固定长度或数字格式的 RowKey（如手机号反转） | 避免固定前缀热点，牺牲有序性 |
| **预分区** | 建表时预先创建多个 Region，避免动态分裂热点 | 从源头解决热点问题 |

> 📌 **面试亮点**：热点问题是 HBase 实际生产中的高频问题，除了掌握理论，最好了解过实际项目中 RowKey 的设计经验。

## 题目三：HBase 的 MemStore 和 HFile 是什么？Flush 流程是怎样的？

### 参考答案

**MemStore：**
- 是写缓存，位于内存中
- 数据写入时先写入 WAL，然后写入 MemStore，保证数据有序（RowKey 有序）
- 当 MemStore 达到阈值（默认128MB）时，触发 flush 操作将数据刷写到磁盘

**HFile（Hadoop File）：**
- 是 HBase 的底层存储格式，采用 LSM-Tree（Log-Structured Merge-Tree）结构
- 每个 MemStore 刷写后生成一个 HFile
- HFile 包含多级索引（Bloom Filter、Block Index）以加速读取

**Flush 流程（三个阶段）：**

1. **prepare 阶段**：对 MemStore 做快照（Snapshot），阻塞写请求，创建新的 CellSkipListSet 接收后续写入
2. **flush 阶段**：将 prepare 阶段的快照数据持久化到 `.tmp` 临时目录，生成 HFile
3. **commit 阶段**：将临时文件移动到 ColumnFamily 目录，注册 StoreFile，完成数据持久化

**Compaction 类型：**

| 类型 | 说明 |
|------|------|
| **Minor Compaction** | 部分 HFile 合并，不清理过期/删除数据 |
| **Major Compaction** | 合并一个 Region 下的所有 HFile，清理过期数据和删除标记 |

> 📌 **面试亮点**：MemStore 和 HFile 是理解 HBase 写入性能的关键，结合 Flush 流程可以回答"为什么 HBase 写入快"这类延伸问题。

## 题目四：HBase 协处理器（Coprocessor）是什么？分为哪两类？各自用途？

### 参考答案

HBase 协处理器类似于数据库的触发器，允许在特定操作前后执行自定义代码逻辑，于 HBase 0.92 版本引入。

### 两类协处理器

**1. Observer（观察者协处理器）**
- 类似于触发器，监控和拦截数据操作事件
- 在 `put`、`get`、`scan` 等操作前后执行钩子函数
- 常见应用场景：
  - **权限管理**：在 preGet / prePut 中拦截未授权的访问
  - **二级索引**：数据变更时自动更新索引表
  - **日志记录**：在操作前后记录审计日志
  - **数据验证**：写入前校验数据合法性
- 实现接口：`RegionObserver`、`MasterObserver`、`WALObserver`

**2. Endpoint（端点协处理器）**
- 类似于关系型数据库的存储过程，允许在 RegionServer 上直接对数据进行计算
- 可以自定义 RPC 协议，在服务端执行聚合、计算等操作，减少网络传输
- 常见应用场景：
  - **聚合计算**：在服务端执行 COUNT、SUM、AVG 等聚合操作
  - **复杂查询**：在服务端完成复杂的条件过滤和计算
  - **数据转换**：在服务端对数据进行格式转换

### 加载方式

| 方式 | 说明 |
|------|------|
| **静态加载** | 配置在 `hbase-site.xml`，作用于所有表，需重启 HBase |
| **动态加载** | 通过 HBase Shell 或 API 加载到指定表，无需重启 |

> 📌 **面试亮点**：协处理器是 HBase 的高级特性，常用于解决"二级索引"、"聚合计算"等痛点问题，面试中能说出实际应用场景会加分。

## 题目五：HRegionServer 宕机后如何恢复？ZooKeeper 在其中起什么作用？

### 参考答案

**恢复流程：**

1. **故障检测**：ZooKeeper 实时监控 HRegionServer 的心跳（Heartbeat）。正常情况下 HRegionServer 每隔 3 秒向 ZooKeeper 发送心跳，一旦宕机，心跳停止，超过 `SessionTimeout`（默认 90 秒）ZooKeeper 判定该节点宕机。
2. **通知 HMaster**：ZooKeeper 将 HRegionServer 下线信息通知给 HMaster，触发失效备援（Failover）。
3. **Region 迁移**：HMaster 将宕机的 HRegionServer 所负责的所有 Region 迁移到其他活着的 HRegionServer 上。
4. **WAL 重播恢复数据**：HRegionServer 宕机前，MemStore 中还未持久化到磁盘的数据通过 WAL 重播来恢复：
   - WAL 文件保存在 `/hbase/WAL/{RegionServer路径}/` 目录下
   - 宕机后，系统读取该目录下的 WAL 文件，按 Region 切分成 `recover.edits` 临时文件
   - 当 Region 被分配到新的 HRegionServer 时，若存在 `recover.edits`，则重播恢复数据

### ZooKeeper 的核心作用

| 作用 | 说明 |
|------|------|
| **选主保证** | ZooKeeper 的选举机制保证任何时候集群中只有一个 HMaster 在运行 |
| **健康监控** | 实时监控 RegionServer 的上下线，将异常信息通知给 Master |
| **元数据存储** | 存储 HBase 的 Schema 和所有 Region 的寻址入口（`-ROOT-`、`.META.` 表位置） |

> 📌 **面试亮点**：这个问题综合考察了对 HBase 高可用机制的理解，ZooKeeper 是整个 HBase 集群的"神经中枢"，要能说清楚它和 HMaster、HRegionServer 的协作关系。
