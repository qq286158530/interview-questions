## 题目 1：Redis 有哪几种数据结构？分别用在什么场景？

### 答案

Redis 支持 5 种基础数据结构：

| 数据结构 | 底层实现 | 常用场景 |
|---------|---------|---------|
| **String** | SDS（简单动态字符串） | 缓存、计数器、分布式锁、Session |
| **List** | QuickList（双向链表+压缩列表） | 消息队列、最新N条记录、简单任务队列 |
| **Hash** | 压缩列表 or HT（哈希表） | 存储对象（如用户信息、商品详情） |
| **Set** | 整数数组 or HT | 标签系统、去重、好友关系、共同关注 |
| **Zset** | 压缩列表 or SkipList+HT | 排行榜、延时队列、有序消息 |

**实战口诀：**
- String 是万金油，几乎什么都能存，但不适合存大对象（>10KB考虑压缩或分片）。
- List 做队列时注意阻塞风险，可用 `BRPOP` 实现阻塞消费。
- Hash 适合字段不固定的对象，比 String 序列化后存更省空间。
- Set 的 `SINTER`（交集）常用于标签筛选、共同好友等功能。
- Zset 的 `ZRANGEBYSCORE` 是排行榜核心，评分为时间戳时即为延时队列。

## 题目 2：Redis 的持久化机制是什么？RDB 和 AOF 各自有什么优缺点？

### 答案

Redis 有两种持久化方式：**RDB** 和 **AOF**，可以同时开启。

### RDB（Redis Database）

- **原理**：定时将内存数据集快照写入二进制文件（dump.rdb）。
- **触发方式**：
  - 手动：`SAVE`（阻塞）或 `BGSAVE`（后台 fork 子进程）
  - 自动：按 `save 900 1 / save 300 10 / save 60 10000` 条件触发
- **优点**：恢复快（启动时直接加载dump文件），fork子进程不影响主进程响应。
- **缺点**：可能丢失最近一次快照后的数据（最多丢一个周期）。

### AOF（Append Only File）

- **原理**：每次写操作追加到日志文件（appendonly.aof），支持三种刷盘策略：
  - `always`：每条命令都 `fsync`，最安全但最慢
  - `everysec`（默认）：每秒 `fsync` 一次
  - `no`：由系统决定刷盘时机，最快但最不安全
- **重写（rewrite）**：AOF文件过大会后台执行 `BGREWRITEAOF`，压缩命令。
- **优点**：数据安全性更高（everysec模式最多丢1秒数据）。
- **缺点**：文件比 RDB 大，恢复速度慢。

### 生产建议

```
# 同时开启，取长补短
appendonly yes
appendfsync everysec
save 900 1  # RDB 作为兜底
```

> ⚠️ 若部署在容器或云环境，注意 `dir` 配置路径可写，否则 `BGSAVE` 会失败。

## 题目 3：Redis 的过期键删除策略有哪些？什么是内存淘汰机制？

### 答案

### 过期键删除策略

| 策略 | 说明 | 适用场景 |
|-----|------|---------|
| **定时删除** | 定时器扫描并删除到期键 | 内存友好，但 CPU 负载高 |
| **惰性删除** | 访问键时检查是否过期，过期则删除 | CPU 友好，但浪费内存 |
| **定期删除** | 每隔一段时间抽检，删除过期键 | 平衡方案（Redis 采用） |

Redis 采用 **惰性删除 + 定期删除** 混合策略。

### 内存淘汰机制（maxmemory-policy）

当 Redis 内存达到 `maxmemory`，会按以下策略淘汰数据：

| 策略 | 行为 |
|-----|------|
| `volatile-lru` | 从已设置过期时间的键中，淘汰最近最少使用的 |
| `allkeys-lru` | 从所有键中淘汰最近最少使用的 |
| `volatile-random` | 从已设置过期时间的键中随机淘汰 |
| `allkeys-random` | 从所有键中随机淘汰 |
| `volatile-ttl` | 淘汰最早过期的 |
| `noeviction` | 不淘汰，返回错误（默认） |
| `volatile-lfu` / `allkeys-lfu` | LFU 版本（Redis 4.0+） |

> **实战建议**：线上推荐 `allkeys-lru`，适合作为缓存场景；若既做缓存又做持久化，用 `volatile-lru` 并给需要持久化的键不设过期时间。

## 题目 4：Redis 哨兵（Sentinel）集群是怎么工作的？如何实现故障自动切换？

### 答案

### 架构组成

- **Sentinel 节点**：至少 3 台（奇数），监控主从节点健康、选主、通知客户端。
- **Master 节点**：负责读写。
- **Slave 节点**：从 Master 同步数据，提供读服务。

### 工作流程

1. **监控**：Sentinel 每 10 秒向主从节点发送 `PING`，超时（`down-after-milliseconds`）则判定主观下线（SDown）。
2. **选主**：Sentinel 集群中认为主节点客观下线（ODown）后，entinel 之间发起投票，获得多数票的 Sentinel 成为 Leader 负责选主。
3. **选主规则**：根据从节点的优先级（`replica-priority`）、复制偏移量、runid 字典序等综合评分。
4. **切换**：Leader Sentinel 将旧主节点的某个从节点升为新主节点，并让其他从节点指向新主（`replicaof`）。
5. **通知**：通过 `+switch-master` 事件通知客户端更新路由。

### 配置示例

```conf
# sentinel.conf
sentinel monitor mymaster 127.0.0.1 6379 2   # 2 表示 2 个 Sentinel 同意即 ODown
sentinel down-after-milliseconds mymaster 30000
sentinel failover-timeout mymaster 180000
```

### 注意事项

- Sentinel 不负责缓存预热，故障切换后应用需有重连机制。
- 建议客户端使用支持 Sentinel 的客户端库（如 Jedis 的 `JedisSentinelPool`），不要手动写死 IP。

## 题目 5：什么是缓存击穿、缓存穿透、缓存雪崩？如何应对？

### 答案

这三个问题是高并发系统中最常见的 Redis 相关风险。

### 缓存穿透

**原因**：查询不存在的数据，请求直接打到数据库，数据库也无此数据可缓存。

**危害**：大量恶意或异常请求击垮数据库。

**解决方案：**
- **布隆过滤器（BloomFilter）**：在缓存层前加一层，将所有合法 key 存入过滤器，请求进来先过滤。
- **空对象缓存**：数据库也查不到的结果，也缓存一个空值（带短 TTL，如 60 秒）。

### 缓存击穿

**原因**：某个热点 key 过期瞬间，大量并发请求发现缓存不存在，全部涌入数据库。

**解决方案：**
- **互斥锁（Mutex）**：只允许一个线程去查数据库并写缓存，其他线程等待。
  ```
  SETNX key lock  # 获取锁
  # 查库写缓存
  DEL key lock    # 释放锁
  ```
- **永不过期**：热点数据不设 TTL，靠逻辑过期（附带字段标记过期时间）配合后台异步刷新。
- **熔断降级**：超过阈值直接返回默认值或友好提示。

### 缓存雪崩

**原因**：大量 key 同时过期，或 Redis 集群宕机，导致请求全部涌向数据库。

**解决方案：**
- **过期时间随机化**：`TTL = baseTTL + random()`
- **多级缓存**：Nginx 本地缓存 + Redis 分布式缓存 + MySQL。
- **高可用架构**：Redis Sentinel 或 Cluster，避免单点故障。
- **熔断限流**：Guava RateLimiter 或 Sentinel，保护数据库。
- **预热**：系统启动或大促前，主动加载热点数据到 Redis。
