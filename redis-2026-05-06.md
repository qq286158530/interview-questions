## 题目一：Redis 有哪几种数据类型？它们的典型应用场景是什么？

### 答案

Redis 支持五种基础数据类型：

| 数据类型 | 底层实现 | 典型应用场景 |
|----------|----------|--------------|
| **String** | SDS（简单动态字符串） | 缓存、计数器、分布式锁（SETNX）、限流（incr） |
| **List** | QuickList（压缩列表+双向链表） | 消息队列（lpush/brpop）、排行榜、关注列表 |
| **Hash** | HashTable + 压缩列表（小数据量时） | 对象存储（用户信息）、购物车、商品详情 |
| **Set** | IntSet（整数集合）+ HashTable | 标签系统、去重、好友关系、抽奖（SRANDMEMBER） |
| **Sorted Set（ZSet）** | SkipList + HashTable | 排行榜、有序集合、延迟队列（score=时间戳） |

**扩展类型（Module/Commands）：**
- **Bitmap**：签到统计、用户活跃天数、布隆过滤器替代
- **HyperLogLog**：UV 统计（独立访客数量）
- **Geospatial**：附近的人、LBS 服务
- **Stream**：消息队列（Redis 版 Kafka）

**面试加分点：** 能说出每种类型的命令组合使用技巧，如用 `ZUNIONSTORE` 做交集排行榜对比、用 `BITOP` 做日活统计等。

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 答案

Redis 有两种持久化机制：

#### RDB（Redis Database）

- **原理**：在指定时间间隔内，将内存中的数据快照（Snapshot）写入二进制文件 `dump.rdb`。
- **触发方式**：
  - 手动：`BGSAVE`（后台异步）、`SAVE`（同步，阻塞）
  - 自动：`save m n`（m 秒内 ≥ n 次修改则触发）
- **优点**：文件紧凑、恢复速度快（fork 子进程完成，适合备份/灾难恢复）
- **缺点**：最后一次快照之后的数据会丢失，fork 时会复制页表，有短暂停顿

#### AOF（Append Only File）

- **原理**：将每一次写操作以追加方式记录到文件（Redis 协议格式），重启时重放命令恢复数据。
- **策略（appendfsync）**：
  - `always`：每条命令都 fsync，性能差但最安全
  - `everysec`（默认）：每秒 fsync 一次，异步完成
  - `no`：由 OS 决定何时 fsync
- **优点**：数据安全性更高，支持重写（`BGREWRITEAOF`）压缩文件
- **缺点**：文件体积大、恢复速度慢

#### 混合持久化（Redis 4.0+）

- 开启 `aof-use-rdb-preamble yes` 后，AOF 重写时先用 RDB 格式写入，再追加增量 AOF 命令，兼顾恢复速度和完整性。

#### 如何选择？

- **纯缓存场景**：可关闭持久化
- **对数据安全要求不高的缓存**：RDB 足够
- **对数据安全要求高的场景**：AOF + 定期 RDB 备份
- **综合推荐**：主从复制 + AOF（everysec）+ 定期 RDB 快照

## 题目三：Redis 的过期 key 清理策略是什么？请描述惰性删除和定期删除的区别。

### 答案

Redis 处理过期 key 有两种策略，组合使用：

#### 1. 惰性删除（Lazy Expiration）

- **原理**：每次访问 key 时，先检查是否过期，若过期则删除并返回空。
- **优点**：对 CPU 友好，只在必须访问时才删除，不用额外资源。
- **缺点**：如果 key 过期后一直不被访问，就会占用内存（内存泄漏风险）。
- **实现**：`db.c/expireIfNeeded()`，在执行任何 key 命令前调用。

#### 2. 定期删除（Active Expiration）

- **原理**：Redis 每隔一段时间（默认 100ms）随机抽取设置了过期时间的 key，检查是否过期，过期则删除。
- **实现**：`redis.c/activeExpireCycle()`，分为：
  - **快速阶段**：扫描 16 个库，每个库最多检查 20 个 key，超时立即退出
  - **慢速阶段**：全量扫描，超过时间限制则停止，下次继续
- **优点**：减少内存占用，防止大量过期 key 堆积
- **缺点**：对 CPU 有一定开销

#### 内存淘汰策略（Memory Eviction）

当内存达到 `maxmemory` 上限时，Redis 还会触发**内存淘汰策略**（不是删除过期 key，而是淘汰任意 key）：

| 策略 | 行为 |
|------|------|
| `noeviction` | 不淘汰，返回错误（默认） |
| `volatile-lru` | 在设置了过期时间的 key 中淘汰最近最少使用的 |
| `allkeys-lru` | 所有 key 中淘汰 LRU |
| `volatile-lfu` | 在设置了过期时间的 key 中淘汰最不常用的 |
| `allkeys-lfu` | 所有 key 中淘汰 LFU |
| `volatile-random` | 随机淘汰设置了过期时间的 key |
| `allkeys-random` | 随机淘汰所有 key |
| `volatile-ttl` | 淘汰 TTL 最短的 key |

**面试加分点：** 可以提到 `SCAN` 命令避免阻塞，以及 `INFO stats` 中的 `evicted_keys` 指标监控。

## 题目四：Redis 主从复制原理是什么？如何处理哨兵模式下的故障转移？

### 答案

#### 主从复制原理

Redis 主从复制采用 **psync（Partial Sync）** 机制，支持全量同步和部分同步：

**全量同步流程：**
1. 从库发送 `PSYNC ? -1` 向主库请求同步
2. 主库判断是全量同步，创建 RDB 快照并用缓冲区记录后续写命令（replication buffer）
3. 主库发送 RDB 文件给从库
4. 从库清空本地数据，执行 LOAD RDB
5. 主库将缓冲区中的写命令发送给从库（命令传播）
6. 从库执行这些命令，与主库保持同步

**部分同步（Redis 2.8+）：**
- 主库为每个从库维护一个 **replication offset**（复制偏移量）
- 从库也维护自己的 **replication offset**
- 如果从库断开重连后 offset 还在主库缓冲区（repl-backlog-size）内，则只补发增量命令
- 否则退化为全量同步

**主从复制配置：**
```bash
replicaof <masterip> <masterport>   # 指定主库
replica-read-only yes                # 从库只读（默认）
```

#### 哨兵模式（Sentinel）与故障转移

Sentinel（哨兵）是 Redis 的高可用方案，核心功能：**监控主库是否宕机、自动故障转移、通知客户端**。

**故障转移流程：**
1. **主观下线（Sdown）**：一个哨兵认为主库心跳无响应（`down-after-milliseconds` 内未响应）
2. **客观下线（Odown）**：多个哨兵投票达成共识（`quorum` 配置）
3. **选举领衔哨兵**：所有在线哨兵通过 Raft 算法选出一个 Leader（最先发投票请求的哨兵可能当选）
4. **选择新主库**：Leader 哨兵从所有从库中选择：
   - 排除主观下线的从库
   - 优先级高的（`replica-priority` 配置值越大越优先）
   - 复制偏移量最大的（数据最新）
   - Run ID 字典序最小的
5. **切换**：Leader 向新主库执行 `SLAVEOF NO ONE`，向其他从库发送 `SLAVEOF 新主库` 命令
6. **通知**：哨兵更新配置，通知客户端新的主库地址

**配置要点：**
```bash
sentinel monitor mymaster <master-ip> <master-port> <quorum>
sentinel down-after-milliseconds mymaster 30000
sentinel parallel-syncs mymaster 1   # 故障转移时每次只能有1个从库同步新主库
```

## 题目五：什么是缓存击穿、缓存穿透、缓存雪崩？如何应对？

### 答案

这是 Redis 面试中最常考的"三穿"问题，都是缓存场景下的典型风险：

#### 1. 缓存穿透（Cache Miss）

- **定义**：查询一个**根本不存在的数据**（既不在缓存，也不在数据库），大量请求绕过缓存直接打到数据库。
- **危害**：数据库压力剧增，严重时引发数据库宕机。
- **常见场景**：恶意请求（id=-1 或不存在的主键）、业务 bug

**解决方案：**
- **布隆过滤器（BloomFilter）**：在缓存层前加一个 Bitmap，所有合法 key 都加入过滤器，查询时先判断是否可能存在
- **缓存空值**：对查询为空的 key 也在缓存中存一个空值（`NULL`），并设置较短 TTL（如 60s）
- **参数校验**：对不合理的主键（如负数 ID）直接拦截

#### 2. 缓存击穿（Hot Key / Cache Breakdown）

- **定义**：某个**热点 key** 突然过期，大量并发请求同时击穿缓存，打到数据库。
- **危害**：瞬间数据库压力巨大，可能导致数据库宕机

**解决方案：**
- **互斥锁（SETNX）**：第一个请求加锁，从数据库加载数据并写回缓存，其他请求等待锁释放后从缓存获取
  ```bash
  SET lock_key unique_value NX EX 10  # 加锁
  GET data_key                         # 查缓存，无则查库
  DEL lock_key                         # 释放锁
  ```
- **热点 key 永不过期**：用逻辑过期字段代替 TTL，异步更新
- **预热**：系统启动或低峰期主动将热点数据加载到缓存

#### 3. 缓存雪崩（Cache Avalanche）

- **定义**：大量 key **同时失效**，或缓存服务本身宕机，导致大量请求同时打到数据库。
- **危害**：数据库瞬时压力过大，引发级联故障

**解决方案：**
- **过期时间随机化**：给每个 key 的 TTL 加一个随机偏差（`TTL * (0.5~1.5)` 随机）
- **多级缓存**：本地缓存（Guava/Caffeine）+ Redis + 数据库，多层兜底
- **高可用架构**：Redis Cluster/Sentinel 避免单点故障
- **熔断/限流**：数据库访问加限流保护（Redis + Lua 脚本限流，或 Sentinel Hystrix）
- **构建独立的 Redis 缓存集群**：与业务数据库隔离

#### 对比总结

| 场景 | 原因 | 核心解决思路 |
|------|------|-------------|
| 缓存穿透 | 查询不存在的数据 | 布隆过滤器 + 缓存空值 |
| 缓存击穿 | 热点 key 过期 | 互斥锁 + 永不过期 |
| 缓存雪崩 | 大量 key 同时失效 | TTL 随机化 + 多级缓存 + 高可用 |
