## 题目一：Flink 的架构是怎样的？为什么 Flink 能做到低延迟高吞吐？

### 参考答案

Flink 采用**流式计算架构**，核心是一个分布式流式数据处理引擎，其架构可以分为以下几层：

### 1. 架构分层

| 层级 | 说明 |
|------|------|
| **API Layer** | DataStream API（无界流）、DataSet API（有界批） |
| **Runtime Layer** | 统一的分布式流处理引擎 |
| **Physical Layer** | JobManager + TaskManager 部署 |

### 2. 核心组件

- **JobManager**：协调分布式执行，负责调度任务、协调 Checkpoint、协调故障恢复等
- **TaskManager**：执行具体的 Task，每个 TaskManager 是一个 JVM 进程，启动多个 TaskSlot
- **TaskSlot**：资源分配单元，每个 TaskSlot 有固定内存，可执行一个或多个算子实例

### 3. 低延迟高吞吐的原因

1. **流式原生架构**：Flink 是天生的流计算引擎，数据一条一条处理，不需要微批处理
2. **基于事件的处理**：采用 **Pipeline 方式**，数据在算子间流水线传输，无需等批
3. **内存管理**：Flink 在 JVM 上实现了自定义内存管理（堆外内存），避免 Java 序列化开销
4. **增量检查点**：使用 **Chandy-Lamport 变体**实现分布式快照，轻量级
5. **异步 I/O**：访问外部数据时支持异步非阻塞，提高资源利用率

## 题目二：Flink 的 Watermark 机制是什么？如何处理乱序数据？

### 参考答案

### 1. 什么是 Watermark（水位线）

Watermark 是 Flink 用于处理**事件时间（Event Time）**乱序的一种机制，本质是一个**时间戳**，表示"所有时间戳小于该 Watermark 的数据都已经到达"。

```
Watermark(t) 表示：所有 event_time < t 的数据都已到达（或者至少我们假设已到达）
```

### 2. 解决乱序的核心思想

- 窗口在 **Watermark 时间 >= 窗口结束时间** 时触发计算
- 对于晚到的数据（时间戳 < Watermark），默认被**丢弃**
- 可以通过 **allowedLateness()** 设置允许延迟，迟到数据会触发增量计算

### 3. 代码示例

```java
WatermarkStrategy<ClickEvent> wms = WatermarkStrategy
    .<ClickEvent>forBoundedOutOfOrderness(Duration.ofSeconds(10)) // 最大乱序10秒
    .withTimestampAssigner((event, timestamp) -> event.getEventTime())
    .withIdleness(Duration.ofMinutes(1)); // 空闲源处理

DataStream<ClickEvent> stream = env.fromSource(
    kafkaSource, wms, "Kafka Source"
);
```

### 4. 常用 Watermark 策略

| 策略 | 说明 |
|------|------|
| `forMonotonousTimestamps` | 单调递增，适用于严格有序数据 |
| `forBoundedOutOfOrderness` | 允许一定乱序，最常用 |
| `forGenerator` | 自定义 Watermark 生成逻辑 |

### 5. 窗口触发时机

```
窗口 [10:00 - 10:10) 的触发条件：
- Watermark >= 10:10  （不迟到）
- 或 窗口内有数据到达
- 触发后，Watermark 继续推进，窗口不关闭（等待 allowedLateness 内的迟到数据）
```

## 题目三：Flink Checkpoint 和 Savepoint 的区别是什么？Flink 如何保证 Exactly-Once 语义？

### 参考答案

### 1. Checkpoint vs Savepoint

| 特性 | Checkpoint | Savepoint |
|------|------------|-----------|
| **触发方式** | Flink 自动周期性触发 | 手动触发（用户命令） |
| **用途** | 故障恢复 | 计划性停止/升级/重启 |
| **存储位置** | JobManager 内存（可配置） | 外部存储（HDFS/S3） |
| **保留策略** | 任务失败/取消时删除 | 一直保留直到手动删除 |
| **状态后端** | 可嵌入 JobManager | 必须外部存储 |

### 2. Checkpoint 原理——Chandy-Lamport 分布式快照

Flink 采用 **Chandy-Lamport 变体算法**，核心步骤：

```
1. JobManager 定期向所有 Source 算子发送 Barrier（检查点屏障）
2. Source 收到 Barrier 后，保存自身状态，向下游广播 Barrier
3. 中间算子收到所有输入的 Barrier 后，保存状态，将 Barrier 向下游传递
4. Sink 算子收到所有输入 Barrier 后，向 JobManager 确认 Checkpoint 完成
```

**关键概念**：
- **Barrier 对齐**：算子必须等所有输入的 Barrier 都到达才做快照（保证一致性）
- **Barrier 不对齐**（exactly-once 可选）：收到即处理，性能更好但可能轻微重复

### 3. Exactly-Once 语义保证

Exactly-Once 需要端到端保证，涉及三个环节：

| 环节 | 保证方式 |
|------|---------|
| **Source** | 可回放（Replay），Kafka 通过 Offset 重放 |
| **Flink 内部** | Checkpoint 分布式快照 |
| **Sink** | 幂等写入 / 事务提交（两阶段提交） |

**两阶段提交 Sink**（如 Kafka Sink）：
```
1. 预提交：数据写入但未提交
2. Checkpoint 完成时，调用 commit()
3. 失败时回滚，已写数据不暴露
```

## 题目四：Flink 的状态管理机制是怎样的？状态后端（State Backend）有哪些，各有什么区别？

### 参考答案

### 1. 状态分类

| 类型 | 说明 | 示例 |
|------|------|------|
| **Keyed State** | 基于 Key 的状态，只能在 KeyedStream 上使用 | ValueState、MapState、ListState |
| **Operator State** | 算子级别状态，所有算子实例共享 | ListState、UnionListState、BroadcastState |

### 2. Keyed State 种类

```java
// ValueState: 单值状态
ValueState<Long> countState = getRuntimeContext().getState(
    new ValueStateDescriptor<>("count", Long.class)
);

// MapState: KV 映射状态
MapState<String, Integer> mapState = getRuntimeContext().getMapState(
    new MapStateDescriptor<>("map", String.class, Integer.class)
);

// ListState: 列表状态
ListState<Event> eventList = getRuntimeContext().getListState(
    new ListStateDescriptor<>("events", Event.class)
);
```

### 3. 状态后端（State Backend）

| 状态后端 | 存储介质 | 适用场景 |
|---------|---------|---------|
| **HashMapStateBackend** | JVM Heap | 小状态、低延迟；测试/开发环境 |
| **EmbeddedRocksDBStateBackend** | RocksDB（磁盘） | 大状态、增量 Checkpoint；生产环境 |

### 4. 核心区别

```
HashMapStateBackend:
  - 状态存储在 JVM 堆内存
  - 快速读写，但受限于内存大小
  - 状态大小 GB 级别

EmbeddedRocksDBStateBackend:
  - 状态存储在 RocksDB（LSM 树结构）
  - 支持增量 Checkpoint
  - 状态可达到 TB 级别
  - 读写有序列化/反序列化开销
```

### 5. 配置示例

```java
// 代码中设置
env.setStateBackend(new EmbeddedRocksDBStateBackend());

// 或 flink-conf.yaml
state.backend: rocksdb
state.checkpoints.dir: hdfs:///flink/checkpoints
```

## 题目五：Flink 窗口函数有哪些？滚动窗口、滑动窗口、会话窗口的区别是什么？

### 参考答案

### 1. 窗口类型总览

| 窗口类型 | 说明 | 关键参数 |
|---------|------|---------|
| **滚动窗口（Tumbling）** | 固定大小，不重叠 | size |
| **滑动窗口（Sliding）** | 固定大小，可重叠 | size, slide |
| **会话窗口（Session）** | 间隔 gap 内无数据则关闭窗口 | gap |
| **全局窗口（Global）** | 所有数据进入同一窗口 | 需要自定义触发器 |

### 2. 滚动窗口 vs 滑动窗口

```
滚动窗口 size=10s:
|  0-10s  |  10-20s  |  20-30s  |  30-40s  |
  窗口1      窗口2      窗口3      窗口4

滑动窗口 size=10s, slide=5s:
|  0-10s  |  5-15s  |  10-20s  |  15-25s  |  20-30s  |
  窗口1      窗口2      窗口3      窗口4      窗口5
  数据被多个窗口重复统计
```

**滚动窗口适用场景**：每个数据只属于一个窗口，如每日UV统计
**滑动窗口适用场景**：需要平滑统计，如10分钟移动平均

### 3. 会话窗口（Session Window）

```java
DataStream<Tuple2<String, Long>> keyedStream = stream
    .keyBy(0)
    .window(SessionWindows.withGap(Time.seconds(30)))  // 30秒无数据则关窗
    .sum(1);
```

会话窗口根据**活跃间隔**动态创建和关闭，适用于用户行为分析（如用户30秒无操作则认为会话结束）。

### 4. 窗口函数分类

| 函数类型 | 说明 | 例子 |
|---------|------|------|
| **增量聚合函数** | 每条数据触发计算，结果是一个值 | reduce()、aggregate() |
| **全量聚合函数** | 窗口关闭时一次性计算 | process() |

```java
// 增量聚合：来一条处理一条，性能好
keyedStream
    .window(TumblingProcessingTimeWindows.of(Time.seconds(10)))
    .reduce((v1, v2) -> v1 + v2);

// 全量聚合：等窗口数据齐了再处理，功能强
keyedStream
    .window(TumblingProcessingTimeWindows.of(Time.seconds(10)))
    .process(new ProcessWindowFunction<...>());
```
