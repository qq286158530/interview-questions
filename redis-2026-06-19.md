## 1. Redis 为什么这么快？

**参考答案：**

Redis 之所以性能极高，主要有以下 4 个原因：

**① 纯内存操作**
Redis 的所有数据都存储在内存中，内存的读写速度是纳秒级别，而传统数据库读写磁盘是毫秒级别，两者相差数个数量级。

**② 高效的 I/O 模型**
Redis 基于单线程事件循环 + I/O 多路复用技术，单个线程可以同时处理多个网络连接上的 I/O 事件，避免了多线程的上下文切换和锁竞争问题。

**③ 优化的内部数据结构**
Redis 内部实现了高度优化的数据结构（如 SDS、跳表、压缩列表、字典等），会根据数据大小动态选择最合适的编码方式，在性能和空间效率之间取得最佳平衡。

**④ 简洁高效的通信协议**
Redis 使用自己设计的 RESP（REdis Serialization Protocol）协议，实现简单、解析速度快且二进制安全，客户端与服务端通信的序列化/反序列化开销很小。

> Redis 单线程是指执行命令的主线程，网络 IO 读写在 Redis 6.0 后引入了多线程。Redis 的性能瓶颈主要在内存和网络，而非 CPU。

## 2. Redis 的持久化机制有哪些？RDB 和 AOF 有什么区别？

**参考答案：**

Redis 支持两种持久化方式：**RDB（快照）** 和 **AOF（命令日志）**。

### RDB 持久化

RDB 在指定时间间隔内将内存中的数据快照写入磁盘，生成二进制格式的 `.rdb` 文件。可以通过 `save`（阻塞）或 `bgsave`（后台 fork 子进程执行）命令触发，也可配置自动触发。

### AOF 持久化

AOF 通过记录每个写操作命令来实现持久化，默认关闭。Redis 会将写命令追加到 AOF 缓冲区，再根据同步策略写入磁盘。

AOF 有三种刷盘策略：
| 策略 | 说明 | 性能 | 数据安全性 |
|------|------|------|------------|
| `always` | 每次写命令立即 fsync | 最低 | 最高 |
| `everysec`（默认）| 每秒 fsync 一次 | 中等 | 最多丢 1 秒数据 |
| `no` | 由操作系统决定 | 最高 | 取决于系统 |

### RDB vs AOF 对比

| 对比项 | RDB | AOF |
|--------|-----|-----|
| 数据完整性 | 可能丢失几分钟数据 | 最多丢 1 秒数据 |
| 恢复速度 | 快（二进制直接加载）| 慢（逐条 replay 命令）|
| 文件大小 | 小（压缩后）| 大（命令追加）|
| 性能影响 | 低（fork 后后台保存）| 较高（每次写都记录）|

### 混合持久化

Redis 4.0 引入混合持久化，开启 `aof-use-rdb-preamble yes` 后，AOF 重写时会先将数据以 RDB 格式写入文件开头，再追加重写期间的命令。这样兼顾了快速恢复和实时性。

## 3. 什么是缓存击穿、穿透、雪崩？如何解决？

**参考答案：**

这是 Redis 缓存最经典的问题，面试中几乎必问。

### 缓存击穿

**问题**：某个热点数据缓存过期时，大量请求穿透缓存直接打到数据库。

**解决方案**：
- **互斥锁**：缓存失效时，第一个线程获取锁重建缓存，其他线程等待或重试。
- **永不过期策略**：缓存不设置过期时间，而是在值中维护逻辑过期时间，发现逻辑过期时异步更新缓存。

### 缓存穿透

**问题**：查询不存在的数据，缓存没有命中，请求直接落库。如果大量恶意请求不存在的数据，会压垮数据库。

**解决方案**：
- **布隆过滤器**：将所有存在的数据哈希到布隆过滤器，查询前先检查，不存在的直接返回。
- **缓存空值**：对于不存在的数据也写入缓存，并设置较短的过期时间。
- **参数校验**：接口层做好参数校验，拦截明显不合理请求。

### 缓存雪崩

**问题**：大量缓存同时失效或缓存服务宕机，导致大量请求涌向数据库。

**解决方案**：
- **过期时间随机化**：给缓存过期时间加上随机值，避免同时过期。
- **高可用缓存集群**：使用 Redis Cluster 构建多节点集群，支持自动故障转移。
- **多级缓存**：引入本地缓存（如 Caffeine）作为二级缓存。
- **限流降级**：使用 Sentinel 或 RateLimiter 进行限流和熔断。

## 4. Redis 主从复制的原理是什么？

**参考答案：**

Redis 主从复制通过异步复制实现，master 负责写操作，slave 自动同步数据并处理读请求，实现读写分离和数据备份。

### 复制过程

分为三个阶段：**建立连接 → 数据同步 → 命令传播**

**① 全量同步**（首次连接或断线重连时）
- slave 发送 `psync ? -1` 请求同步
- master 创建 RDB 文件，同时将新写入的命令缓存到复制缓冲区
- master 将 RDB 文件发送给 slave
- slave 清空旧数据并加载 RDB
- master 将缓冲区中的命令发送给 slave

**② 增量同步**（断线重连后）
- slave 发送 `psync {runId} {offset}` 带上主节点 ID 和复制偏移量
- 如果 runId 匹配且偏移量在复制积压缓冲区内，master 发送增量数据
- 否则触发全量同步

### 关键机制

- **复制积压缓冲区**：master 维护的固定长度 FIFO 队列（默认 1MB），存储最近的写命令，用于增量复制
- **复制偏移量**：主从各自维护偏移量，衡量同步进度
- **主节点 ID**：每个 master 有唯一 ID，标识数据版本

### 主从拓扑结构

常见三种：**一主一从**、**一主多从**、**树状主从**（跨地域部署降低主节点负载）

## 5. Redis 的内存淘汰策略有哪些？

**参考答案：**

当 Redis 内存使用达到 `maxmemory` 阈值时，会根据配置的淘汰策略删除数据。

Redis 8 种内存淘汰策略：

| 策略 | 说明 |
|------|------|
| `noeviction` | 不淘汰，返回错误（默认）|
| `volatile-lru` | 从已设置过期时间的 key 中淘汰最近最少使用的 |
| `volatile-lfu` | 从已设置过期时间的 key 中淘汰最不经常使用的 |
| `volatile-ttl` | 从已设置过期时间的 key 中淘汰即将过期的 |
| `volatile-random` | 从已设置过期时间的 key 中随机淘汰 |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用的 |
| `allkeys-lfu` | 从所有 key 中淘汰最不经常使用的 |
| `allkeys-random` | 从所有 key 中随机淘汰 |

### 选择建议

- **纯缓存场景**：推荐 `allkeys-lru`，自动保留热点数据
- **混合存储（重要数据不设过期时间）**：推荐 `volatile-lru`
- **需要保护长期热门数据**：推荐 `allkeys-lfu`
- **会话管理（TTL 精确过期）**：推荐 `volatile-ttl`

### LRU vs LFU 的区别

- **LRU（Least Recently Used）**：基于时间维度，淘汰最近最少访问的键，适合有明显时间局部性的场景
- **LFU（Least Frequently Used）**：基于次数维度，淘汰访问频率最低的键，适合长期访问模式明显的场景
