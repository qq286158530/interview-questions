## 题目一：Flink 的架构是怎样的？JobManager 和 TaskManager 的职责是什么？

### 参考答案

Flink 采用**主从架构**，主要由以下组件构成：

**1. JobManager（主节点）**
- **JobMaster**：负责调度 TaskManager 执行具体的 Stream Graph，每个 Job 独享一个 JobMaster
- **ResourceManager**：管理 Slot（资源单位），与 YARN/Kubernetes/Mesos 等资源管理框架协作
- **Dispatcher**：提供 REST 接口，接收用户提交的 Job，为每个 Job 启动一个新的 JobMaster，同时启动 Web UI

**2. TaskManager（工作节点）**
- 负责执行具体的 Task（计算逻辑）
- 以 **Slot** 为单位管理资源，每个 TaskManager 拥有一到多个 Slot
- 运行时与 JobManager 保持心跳，汇报自身状态

**3. Stream Graph / JobGraph**
- 用户编写的 DataStream API 代码首先被翻译为 Stream Graph
- Flink 优化器将 Stream Graph 转换为 JobGraph，再提交给 JobManager 执行

**职责分工总结**：
- JobManager 负责协调和调度，是"大脑"
- TaskManager 负责实际的数据处理，是"执行者"
- 二者通过 Actor System 或 RPC 通信

> 📎 来源：[Apache Flink 官方文档 - Concepts](https://nightlies.apache.org/flink/flink-docs-stable/docs/concepts/overview/)

## 题目二：什么是 Watermark？它是如何解决乱序事件的？

### 参考答案

**Watermark（水位线）是 Flink 用于处理**事件时间（Event Time）**乱序问题的核心机制。

### 核心概念

1. **乱序问题**：在分布式环境中，由于网络延迟等原因，后产生的事件可能先到达，导致基于事件时间的窗口无法正确触发。

2. **Watermark 的本质**：一个**时间戳**，表示"所有时间戳 ≤ T 的数据都已经到达"。它告诉 Flink：**不会再有时间戳小于当前 Watermark 的事件了**。

3. **窗口触发条件**：当 `Watermark >= 窗口结束时间` 时，窗口才会被触发计算。

### 代码示例

```java
WatermarkStrategy<Event> strategy = WatermarkStrategy
    .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(10))  // 允许10秒乱序
    .withTimestampAssigner((event, timestamp) -> event.getTimestamp());

DataStream<Event> stream = env.fromSource(
    kafkaSource,
    strategy,
    "Kafka Source"
);
```

### 两种 Watermark 策略

| 策略 | 说明 |
|------|------|
| **Fixed Watermark** | 固定延迟，简单但不适应动态乱序 |
| **Bounded Out-of-Orderness** | 设置最大乱序时间，更实用 |

### 注意事项

- Watermark 过小：可能丢失迟到数据
- Watermark 过大：延迟较高但数据完整性好
- 可以配合 **侧输出（Side Output）** 机制收集迟到数据

> 📎 来源：[Apache Flink 官方文档 - Event Time & Watermarks](https://nightlies.apache.org/flink/flink-docs-stable/docs/dev/datastream/overview/)

## 题目三：Flink 的 Checkpoint 和 Savepoint 有什么区别？

### 参考答案

| 维度 | Checkpoint | Savepoint |
|------|------------|-----------|
| **触发方式** | Flink 自动周期性执行 | 用户手动触发 |
| **用途** | 故障恢复 | 计划性维护（升级、扩缩容） |
| **优化方向** | 快速恢复 | 兼容性、可移植性 |
| **是否增量** | 支持增量 | 始终为全量完整快照 |
| **删除机制** | Job 取消时自动删除 | 除非手动删除，否则一直保留 |

### Checkpoint 详解

- 是 Flink **故障恢复**的核心机制
- 基于 **Chandy-Lamport 算法**的变体——**异步屏障快照（Asynchronous Barrier Snapshotting）**
- 流程：
  1. Checkpoint Coordinator 向所有 Source 发送 checkpoint barrier
  2. Source 快照自身状态（偏移量），向下游发送 barrier
  3. 下游算子收到 barrier 后执行对齐（Barrier Alignment），快照自己的状态
  4. 状态写入配置的存储后端（如 HDFS/S3）

### Savepoint 详解

- 用于**手动备份**，Job 取消后不自动删除
- 支持 Job 暂停/恢复、集群迁移、代码升级等场景
- 格式：`savepoint-xxxx-yyyy`，可跨 Flink 版本恢复

### 状态后端对比

| 状态后端 | 存储介质 | 适用场景 |
|----------|----------|----------|
| **HashMapStateBackend** | JVM Heap | 小状态、低延迟；测试环境 |
| **EmbeddedRocksDBStateBackend** | 本地磁盘 + 内存 | 大状态、生产环境；支持增量 checkpoint |

> 📎 来源：[Apache Flink 官方文档 - Fault Tolerance](https://nightlies.apache.org/flink/flink-docs-stable/docs/learn-flink/fault_tolerance/)

## 题目四：Flink 窗口有哪些类型？滚动窗口、滑动窗口、会话窗口的区别是什么？

### 参考答案

Flink 支持两大类窗口：**Count Window（计数窗口）** 和 **Time Window（时间窗口）**，时间窗口又分为滚动、滑动、会话三种。

### 1. 滚动窗口（Tumbling Window）

- **特点**：窗口大小固定，**不重叠**
- 适用于：按固定时间段聚合，如"每小时的销售额"
- 示例：`timeWindow(Time.minutes(5))`

```
|--window1--|--window2--|--window3--|
  0-5min      5-10min    10-15min
```

### 2. 滑动窗口（Sliding Window）

- **特点**：窗口大小固定，**可重叠**
- 由两个参数决定：窗口大小 + 滑动步长
- 适用于：需要平滑聚合或"重叠分析"的场景，如"过去5分钟的平均值，每1分钟更新一次"
- 示例：`timeWindow(Time.minutes(5), Time.minutes(1))`

```
|----window1----|
     |----window2----|
          |----window3----|
```

### 3. 会话窗口（Session Window）

- **特点**：由**事件之间的间隔**动态决定窗口边界
- 适用于：用户行为分析，如"用户空闲超过30秒则认为会话结束"
- 示例：`window(ProcessingTimeSessionWindows.withGap(Time.seconds(30)))`

### 4. 全局窗口（Global Window）

- 所有相同 key 的数据进入同一个窗口
- 需要配合**自定义触发器**使用

### 代码示例

```java
// 滚动窗口：每5秒一个窗口
stream.keyBy(t -> t.userId)
      .window(TumblingProcessingTimeWindows.of(Time.seconds(5)))
      .sum("price");

// 滑动窗口：窗口大小10秒，滑动步长5秒
stream.keyBy(t -> t.userId)
      .window(SlidingProcessingTimeWindows.of(Time.seconds(10), Time.seconds(5)))
      .sum("price");

// 会话窗口：间隔30秒视为新会话
stream.keyBy(t -> t.userId)
      .window(ProcessingTimeSessionWindows.withGap(Time.seconds(30)))
      .sum("price");
```

> 📎 来源：[Apache Flink 官方文档 - Windows](https://nightlies.apache.org/flink/flink-docs-stable/docs/dev/datastream/operators/windows/)

## 题目五：Flink 的状态管理机制是怎样的？如何选择合适的状态后端？

### 参考答案

### 状态管理概述

Flink 是**有状态的流处理引擎**，状态分为两类：

1. **Keyed State（按键分区状态）**：与 Key 绑定的状态，只能在 `keyBy` 之后使用
   - ValueState、TListState、TMapState、TSetState、AggregatingState 等

2. **Operator State（算子状态）**：与算子实例绑定的状态，所有算子都可以使用
   - ListState、UnionListState、BroadcastState 等

### 状态后端选择

| 特性 | HashMapStateBackend | EmbeddedRocksDBStateBackend |
|------|--------------------|----------------------------|
| **存储介质** | JVM Heap | 本地 RocksDB（磁盘）+ 内存 |
| **容量** | 受 JVM 堆大小限制 | 可存储TB级数据 |
| **速度** | 快（内存操作） | 较慢（涉及序列化/SSTable） |
| **增量 Checkpoint** | 不支持 | 支持 |
| **生产推荐** | ❌ 小状态/测试 | ✅ 大状态生产环境 |

### 生产环境建议

```java
// 生产环境推荐 RocksDB + HDFS/S3
Configuration config = new Configuration();
config.setString("state.backend", "rocksdb");
config.setString("state.checkpoints.dir", "hdfs:///checkpoints/");
config.setString("execution.checkpointing.mode", "EXACTLY_ONCE");
```

### 状态过期策略

- **TTL（Time-To-Live）**：通过 `StateTtlConfig` 设置状态过期时间
- 自动清理过期状态，避免状态无限膨胀

```java
StateTtlConfig ttlConfig = StateTtlConfig
    .newBuilder(Time.days(1))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)
    .build();
```

> 📎 来源：[Apache Flink 官方文档 - State & Fault Tolerance](https://nightlies.apache.org/flink/flink-docs-stable/docs/learn-flink/fault_tolerance/)
