## 📌 第 1 题：Spark 与 Hadoop 的核心区别是什么？为什么 Spark 比 Hadoop 快？

### 题目
Spark 与 Hadoop 的核心区别是什么？为什么 Spark 比 Hadoop 快？

### 答案

| 对比维度 | Hadoop MapReduce | Spark |
|---|---|---|
| **数据处理模型** | 基于磁盘，批处理优先 | 基于内存，计算优先 |
| **中间结果存储** | 写入 HDFS，每次 Shuffle 落盘 | 内存中完成，数据不动则不落盘 |
| **迭代计算** | 每次迭代都读写磁盘，I/O 开销巨大 | 数据缓存在内存中，迭代效率高 |
| **任务启动开销** | 每个 Task 独立启动 JVM | Task 在 Executor 线程池中复用线程，启动开销小 |
| **DAG 优化** | 无 DAG，每个 Job 独立分解为 Map/Reduce | 构建 DAG，统一优化执行计划 |
| **数据共享** | Job 间无共享，需借助外部存储 | 通过广播变量、缓存实现高效共享 |

**Spark 更快的根本原因：**

1. **内存计算**：Spark 将中间结果保存在内存中，避免了 MapReduce 每次 Shuffle 强制落盘的开销。对于迭代式机器学习、图计算等场景，性能提升可达 10～100 倍。
2. **DAG 优化**：Spark 根据宽依赖构建 DAG，将一个 Job 内多个算子合并为一个 Stage，减少任务调度开销。
3. **线程池复用**：Executor 启动后，Task 以线程形式运行在进程内，避免了 Hadoop 每个 Task 启动独立 JVM 的昂贵开销。
4. **丰富的执行优化**：包括 mapSideCombine、pipeline、广播变量等优化手段。

## 📌 第 2 题：RDD 的特性是什么？为什么 RDD 是不可变的？

### 题目
RDD 的特性是什么？为什么 RDD 是不可变的？RDD 的容错机制是如何实现的？

### 答案

**RDD（Resilient Distributed Dataset）的五大核心特性：**

1. **分区列表（Partitions）**：RDD 数据被划分为多个分区，每个分区是并行的基本单位，默认由 HDFS 分块决定。
2. **分区计算函数（Partitioner）**：指定如何对键值对 RDD 进行分区（如 HashPartitioner、RangePartitioner）。
3. **依赖关系（Dependencies）**：记录当前 RDD 与父 RDD 的依赖关系，分两种：
   - **窄依赖（Narrow Dependency）**：父 RDD 每个分区最多被子 RDD 一个分区使用（如 map、filter）。
   - **宽依赖（Wide Dependency）**：父 RDD 每个分区被子 RDD 多个分区使用（如 groupByKey、join），触发 Shuffle。
4. **计算函数（Compute）**：对每个分区执行计算的函数。
5. **优先位置（Preferred Locations）**：调度任务时优先将计算安排在数据所在的节点（数据本地性）。

**为什么 RDD 是不可变的？**

RDD 设计为只读（Immutable），每次对数据进行转换操作（Transformation），返回的是一个全新的 RDD，原 RDD 保持不变。这带来了两个核心好处：

- **简化容错**：由于只记录 RDD 之间的依赖关系（Lineage），当某个分区丢失时，只需根据依赖链重新计算父 RDD 的对应分区，无需检查点（Checkpoint）数据副本，极大降低了容错成本。
- **避免副作用**：在分布式并发环境下，修改共享状态会导致竞态条件，不可变设计天然线程安全。

**容错机制（Lineage）：**

Spark 不通过复制数据副本来实现容错，而是通过 **Lineage（血统）** 记录 RDD 的转换图。当某个 Partition 丢失时，Spark 根据 Lineage 从父 RDD 重新计算该 Partition。对于宽依赖场景，由于父分区可能被多个子分区引用，Spark 会优先将中间结果缓存或 Checkpoint 到磁盘，以降低重算代价。

## 📌 第 3 题：Spark Shuffle 的过程是什么？为什么 Shuffle 是性能瓶颈？

### 题目
Spark Shuffle 的过程是什么？为什么 Shuffle 是性能瓶颈？如何优化 Shuffle 操作？

### 答案

**Shuffle 的过程：**

Spark Shuffle 发生在**宽依赖**算子（如 `groupByKey`、`reduceByKey`、`join`、`sort` 等）执行时，需要将分布在不同节点不同 Partition 中的数据按 Key 进行重新分组和聚合。

具体过程分为 **Shuffle Write** 和 **Shuffle Read** 两个阶段：

```
Shuffle Write (Map 阶段):
  - 每个 Mapper 将输出数据按目标 Partition 写入本地磁盘
  - 写入前按 Key 进行排序（sort-based Shuffle）
  - 每个 Partition 生成一个单独的文件

Shuffle Read (Reduce 阶段):
  - 每个 Reducer 主动去所有 Mapper 节点拉取属于自己的 Partition 数据
  - 拉取后进行归并排序（merge sort）
  - 执行用户定义的 Reduce 函数
```

**Shuffle 为什么是性能瓶颈？**

1. **磁盘 I/O 开销**：Shuffle Write 将数据写入磁盘，Shuffle Read 从磁盘读取数据，涉及大量磁盘读写。
2. **网络 I/O 开销**：数据需要在不同节点间传输，跨网络拉取数据是分布式系统中代价最高的操作之一。
3. **序列化开销**：数据需要在网络传输前进行序列化，到达后进行反序列化，增加了 CPU 负担。
4. **磁盘小文件问题**：每个 Mapper 为每个 Partition 生成一个文件，大量小文件会给文件系统造成压力。
5. **内存压力**：Shuffle Read 阶段需要在内存中缓冲数据，容易触发 OOM。

**Shuffle 优化策略：**

| 优化手段 | 说明 |
|---|---|
| **减少 Shuffle 数据量** | 在 map 端提前做 combine（`mapSideCombine=true`），如 `reduceByKey` 优于 `groupByKey` |
| **合理设置分区数** | 分区数不宜过少（并行度低）也不宜过多（小文件多），一般为 core 数的 2～3 倍 |
| **使用广播变量** | 将小表广播到每个节点，避免 Shuffle（如 `broadcast.join`） |
| **开启排序合并** | Spark 2.x 默认为 sort-based Shuffle，比旧的 hash-based 更稳定 |
| **调整缓冲大小** | `spark.shuffle.file.buffer`（默认 32KB）调大可减少磁盘 write 次数 |
| **开启 Shuffle 压缩** | `spark.shuffle.compress=true`，减少网络传输数据量 |
| **避免不必要的数据排序** | `sortByKey` 如无需全局排序，可考虑其他聚合方式 |

## 📌 第 4 题：Spark 内存管理机制是怎样的？堆内与堆外内存如何划分？

### 题目
Spark 内存管理机制是怎样的？Executor 的堆内内存（On-heap）和堆外内存（Off-heap）如何划分？静态内存管理和统一内存管理有什么区别？

### 答案

**Spark Executor 内存结构（统一内存管理，Spark 2.x+）：**

Spark 2.x 引入统一内存管理，将 Executor 内存划分为以下几块：

| 区域 | 说明 | 占比（约） |
|---|---|---|
| **User Memory** | 用户代码使用的内存（非 Spark 内部管理） | ~37% |
| **Spark Memory** | Spark 内部管理的内存池 | ~63% |
| └─ **Storage Memory** | 缓存 RDD、广播变量、Broadcast 块等 | 可动态借用 |
| └─ **Execution Memory** | Shuffle 排序、Hash 聚合、缓存写入等 | 可动态借用 |
| **Reserved Memory** | 预留 300MB，供系统使用 | 固定 300MB |

**堆内 vs 堆外内存：**

- **On-heap Memory（堆内）**：由 JVM 统一管理，通过 GC 回收。`spark.executor.memory` 控制总大小。容易产生内存碎片和 GC 停顿。
- **Off-heap Memory（堆外）**：绕过 JVM 直接向操作系统申请内存（通过 `spark.memory.offHeap.enabled=true`），避免 GC 开销，适合需要极低延迟的场景。`spark.memory.offHeap.size` 控制大小。

**静态内存管理 vs 统一内存管理（Spark 1.6+）：**

| | 静态内存管理（Spark 1.5 及之前） | 统一内存管理（Spark 1.6+） |
|---|---|---|
| **划分方式** | 运行时固定划分，互不借用 | Storage 与 Execution 内存可动态借用 |
| **灵活性** | 低，各区域独立 | 高，内存池共享 |
| **调优难度** | 高，需手动配置各区域大小 | 较低，但仍需合理配置 |
| **问题** | Storage 空闲但 Execution 满时无法借用 | 容易出现两者都满时的争夺问题 |

**常见内存调优参数：**

```bash
spark.executor.memory = 4g          # Executor 总堆内内存
spark.memory.fraction = 0.6        # Spark 内存占总内存的比例
spark.memory.storageFraction = 0.5 # Storage 占 Spark 内存池的比例
spark.memory.offHeap.enabled = true # 开启堆外内存
spark.memory.offHeap.size = 2g      # 堆外内存大小
```

## 📌 第 5 题：广播变量（Broadcast Variable）和累加器（Accumulator）的作用是什么？有什么区别？

### 题目
广播变量（Broadcast Variable）和累加器（Accumulator）的作用是什么？有什么区别？使用广播变量有什么注意事项？

### 答案

**广播变量（Broadcast Variable）：**

广播变量是将一份**只读数据**高效地分发到集群中每个节点的方式，避免每个 Task 都携带一份数据副本，从而极大减少网络传输和内存占用。

```scala
// Scala 示例
val broadcastVar = sc.broadcast(Array(1, 2, 3, 4, 5))

// 在 Executor 中使用（每个节点只有一份副本）
val result = data.filter(x => broadcastVar.value.contains(x))
```

**累加器（Accumulator）：**

累加器是**只写**的共享变量，用于在分布式环境下安全聚合数据（如计数、求和）。Driver 可以读取最终值，但只能由 Executor 写入。

```scala
// Scala 示例
val accum = sc.longAccumulator("My Accumulator")
data.foreach(x => if (x > 10) accum.add(1))
println(accum.value)  // Driver 读取
```

**广播变量 vs 累加器对比：**

| 特性 | 广播变量 | 累加器 |
|---|---|---|
| **读写权限** | 只读（Driver 写，Executor 读） | 只写（Executor 写，Driver 读） |
| **用途** | 共享只读数据，减少网络传输 | 分布式聚合（计数、求和等） |
| **Task 副本** | 每个节点一份，多 Task 共享 | 每个 Task 独立副本，最后合并 |
| **容错** | 无需容错（只读数据，丢失可重放） | 仅一次语义（at-least-once） |
| **更新次数** | 无更新 | 只支持累加操作（add） |

**广播变量的注意事项：**

1. **大小控制**：广播变量适合小规模数据（一般 < 20MB），数据过大会导致广播超时或 OOM。
2. **广播阈值**：通过 `spark.sql.autoBroadcastJoinThreshold`（默认 10MB）控制 DataFrame join 时自动广播的阈值。
3. **广播失效场景**：当 `spark.sql.autoBroadcastJoinThreshold` 设为 `-1` 时禁用自动广播，需手动调用 `broadcast()`。
4. **序列化问题**：广播变量必须可序列化（Java/Kryo），且序列化后的数据会在节点间传输。
5. **广播策略选择**：Spark 默认使用 HTTP broadcast，对于大集群可切换至 Torrent broadcast（类似 BitTorrent 机制）。
