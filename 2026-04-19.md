## 题目1：Java基础 - ThreadLocal的原理、内存泄漏原因及解决方案

### 【答案】

**ThreadLocal原理**

ThreadLocal是Java提供的线程本地存储机制，核心思想是为每个线程创建独立的变量副本，实现线程隔离。

**底层数据结构：**
- Thread类中有一个`ThreadLocal.ThreadLocalMap`类型的成员变量`threadLocals`
- ThreadLocalMap是ThreadLocal的内部类，采用类似HashMap的Entry数组实现
- Entry继承自WeakReference<ThreadLocal<?>>，key是ThreadLocal对象的弱引用
- 每个线程首次访问ThreadLocal的`get()/set()`方法时，会初始化自己的ThreadLocalMap

**内存泄漏原因：**

1. **Entry的弱引用设计**：ThreadLocalMap的Entry key是WeakReference<ThreadLocal<?>>，当ThreadLocal对象没有强引用时会被GC回收，导致key为null
2. **但value是强引用**： Entry的value是强引用，指向真正存储的数据
3. **只有key为null的垃圾回收**： 当ThreadLocal对象被回收后，key变为null，但value仍然被Entry引用，且线程存活期间不会被回收
4. **线程池场景**： 如果线程来自线程池且会被复用，ThreadLocalMap中的脏数据会持续存在，导致内存泄漏

**解决方案：**

1. **手动清理**： 在使用完ThreadLocal后，调用`remove()`方法删除Entry
2. **try-finally模式**：
```java
ThreadLocal<Object> tl = new ThreadLocal<>();
try {
    tl.set(value);
    // 业务逻辑
} finally {
    tl.remove();
}
```
3. **静态ThreadLocal**： 将ThreadLocal声明为静态变量，减少被GC的可能性，但仍需手动remove
4. **使用InheritableThreadLocal**： 可继承的ThreadLocal，但同样存在内存泄漏风险

**最佳实践**：
- 务必在使用完毕后调用remove()
- 避免将大对象放入ThreadLocal
- 线程池中使用ThreadLocal要格外小心

## 题目2：Java进阶 - JVM类加载机制与双亲委派模型

### 【答案】

**JVM类加载机制**

类加载过程分为5个阶段：
1. **加载(Loading)**： 通过类的全限定名获取类的二进制字节流，将字节流转化为方法区的运行时数据结构，在堆中生成Class对象
2. **验证(Verification)**： 确保Class文件的字节流符合虚拟机要求，包括文件格式验证、元数据验证、字节码验证、符号引用验证
3. **准备(Preparation)**： 为类变量分配内存并设置初始值（零值），final static常量会被直接赋值为指定值
4. **解析(Resolution)**： 将常量池中的符号引用替换为直接引用
5. **初始化(Initialization)**： 执行类构造器<clinit>()方法，包括静态变量赋值和静态代码块

**双亲委派模型**

**工作流程：**
当类加载器收到加载请求时，首先委派给父类加载器处理，父类加载器再委派其父类，直到BootstrapClassLoader。只有父加载器无法完成时，才由当前加载器尝试加载。

**三层类加载器：**
- **Bootstrap ClassLoader**： C++实现，负责加载JAVA_HOME/lib目录下的核心类库
- **Extension ClassLoader**： Java实现，负责加载JAVA_HOME/lib/ext目录下的扩展类库
- **Application ClassLoader**： Java实现，负责加载用户类路径(classpath)上的类库

**双亲委派的优势：**
1. **安全性**： 防止核心API被篡改，如用户自定义java.lang.String不会被加载
2. **避免重复加载**： 父类已加载的类不会被子类重复加载
3. **保证类加载的层次性**： 这是JDK团队的经验总结，确保类的层次结构

**破坏双亲委派的情况：**
1. **JNDI服务**： 需要独立加载SPI实现类，通过ThreadContextClassLoader实现
2. **OSGI**： 采用网状结构的类加载器，不遵循双亲委派
3. **Spring等框架**： 使用ThreadContextClassLoader实现热部署等功能

**线程上下文类加载器(TCCL)：**
- 每个线程都有一个TCCL，默认是ApplicationClassLoader
- 可以通过`Thread.currentThread().setContextClassLoader()`设置
- JDK6之后，ServiceLoader使用TCCL来加载SPI实现

## 题目3：Spring Boot - Spring Boot启动流程及监听器机制

### 【答案】

**Spring Boot启动流程**

1. **创建SpringApplication对象**
   - 判断应用类型(Servlet/Reactive/None)
   - 初始化initializers(List<ApplicationContextInitializer>)
   - 初始化listeners(List<ApplicationListener>)
   - 推断main方法所在类

2. **执行run()方法**
   - 创建StopWatch用于计时
   - 获取并启动监听器
   - 准备环境变量(Environment)
   - 创建ApplicationContext
   - 准备Context上下文
   - 刷新容器(refreshContext)
   - 调用runners

**关键阶段详解：**

**prepareContext阶段：**
```java
// 1. 绑定environment到context
ConfigurableEnvironment environment = prepareEnvironment(listeners, bootstrapContext);
// 2. 设置context环境
context.setEnvironment(environment);
// 3. 执行initializer
applyInitializers(context);
// 4. 加载bean定义
load(org.springframework.boot.SpringApplicationBannerZone.class, sources.toArray(new Object[0]));
```

**refreshContext阶段：**
- 底层调用AbstractApplicationContext.refresh()
- 完成BeanFactory的创建和刷新
- 实例化所有非懒加载的单例Bean
- 触发Spring Boot特有的自动配置

**SpringApplicationRunListener机制：**

监听器接口定义：
```java
public interface SpringApplicationRunListener {
    void starting();  // 应用刚开始
    void environmentPrepared(ConfigurableEnvironment environment);  // 环境准备完成
    void contextPrepared(ConfigurableApplicationContext context);  // Context准备完成
    void contextLoaded(ConfigurableApplicationContext context);  // Context加载完成
    void started(ConfigurableApplicationContext context);  // 启动完成
    void running(ConfigurableApplicationContext context);  // 应用正在运行
}
```

**EventPublishingRunListener：**
- Spring Boot默认使用的监听器
- 将SpringApplicationEvent转换为SpringApplicationRunListener事件
- 通过Multicaster广播给所有ApplicationListener

**启动监听执行顺序：**
1. ApplicationStartingEvent - 应用启动开始
2. ApplicationEnvironmentPreparedEvent - 环境准备完成
3. ApplicationContextInitializedEvent - Context初始化完成
4. ApplicationPreparedEvent - Context准备完成
5. ContextRefreshedEvent - Context刷新完成
6. ApplicationStartedEvent - 应用启动完成
7. ApplicationReadyEvent - 应用就绪

## 题目4：Spring Cloud - Feign远程调用原理

### 【答案】

**Feign概述**

Feign是Netflix开发的声明式HTTP客户端，通过编写简单的接口和注解，即可完成HTTP请求的发送。

**核心原理**

**1. 启动时扫描和注册**

- @EnableFeignClients注解触发@Import(FeignClientsRegistrar.class)
- FeignClientsRegistrar实现ImportBeanDefinitionRegistrar接口
- 扫描所有@FeignClient注解的接口，为每个接口创建BeanDefinition
- BeanDefinition的beanClass指向FeignClientFactoryBean（工厂Bean）

**2. 动态代理创建**

当注入@FeignClient标注的接口时：
- 调用FeignClientFactoryBean的getObject()方法
- 通过ReflectiveFeign.newInstance()创建JDK动态代理
- 为接口的每个方法创建MethodHandler
- MethodHandler使用HardCodedTarget获取服务名和方法信息

**3. 请求发送流程**

```
1. 调用接口方法
   ↓
2. SynchronousMethodHandler.invoke()
   ↓
3. Feign.Builder.build() 创建的client执行请求
   ↓
4. LoadBalancerFeignClient.client() 获取实际地址
   ↓
5. 通过Ribbon获取服务实例列表和负载均衡
   ↓
6. 构造真实HTTP请求（Encode/Decode）
   ↓
7. 返回Response
```

**关键组件：**

**Contract（契约层）：**
- 解析@FeignClient接口上的注解
- 解析方法上的@RequestMapping等注解
- 将注解转换为MethodHandler可识别的元数据

**Client（HTTP客户端）：**
- Client.Default：默认的HTTP客户端
- OkHttpClient：基于OKHttp3
- ApacheHttpClient：基于Apache HttpClient
- LoadBalancerFeignClient：集成Ribbon负载均衡

**Encoder/Decoder：**
- Encoder将请求参数编码为HTTP请求体
- Decoder将响应体解码为Java对象

**RequestInterceptor：**
- 请求拦截器，可添加公共参数（如header）
- 每个FeignClient可配置多个拦截器

**@FeignClient配置属性：**
```java
@FeignClient(name = "service-name", 
    url = "http://localhost:8080",
    fallback = xxxFallback.class,
    configuration = FeignConfig.class,
    primary = true)
```

## 题目5：Spring - 事务传播行为与隔离级别

### 【答案】

**Spring事务传播行为（Propagation）**

事务传播行为定义了当一个事务方法被另一个事务方法调用时，事务如何传播。

**7种传播行为：**

1. **REQUIRED（默认）**： 如果当前存在事务，加入该事务；否则创建新事务
2. **REQUIRES_NEW**： 总是创建新事务，挂起当前事务
3. **SUPPORTS**： 如果当前存在事务，加入该事务；否则以非事务执行
4. **NOT_SUPPORTED**： 以非事务执行，挂起当前事务
5. **MANDATORY**： 必须在事务中执行，否则抛异常
6. **NEVER**： 必须不在事务中执行，否则抛异常
7. **NESTED**： 如果当前存在事务，在嵌套事务中执行；否则创建新事务

**NESTED vs REQUIRES_NEW：**
- NESTED： 使用savepoint机制，与原事务是同一个数据库连接
- REQUIRES_NEW： 开启全新事务，与原事务完全独立

**Spring隔离级别（Isolation）**

隔离级别定义了一个事务可能受其他并发事务影响的程度。

**4种隔离级别：**

1. **DEFAULT**： 使用数据库的默认隔离级别
2. **READ_UNCOMMITTED（读未提交）**： 可读取未提交的数据，最不安全，会产生脏读
3. **READ_COMMITTED（读已提交）**： 只能读取已提交数据，防止脏读，但可能产生不可重复读
4. **REPEATABLE_READ（可重复读）**： 同一事务内多次读取数据一致，防止脏读和不可重复读，但可能产生幻读
5. **SERIALIZABLE（串行化）**： 完全串行执行，完全隔离，但性能最差

**常见问题：**
- **脏读**： 读取到其他事务未提交的数据
- **不可重复读**： 同一事务内，两次读取同一数据结果不同（其他事务提交了修改）
- **幻读**： 同一事务内，两次查询结果集数量不同（其他事务提交了插入/删除）

**@Transactional配置：**
```java
@Transactional(
    propagation = Propagation.REQUIRED,
    isolation = Isolation.READ_COMMITTED,
    readOnly = false,
    timeout = -1,
    rollbackFor = {Exception.class},
    noRollbackFor = {BusinessException.class}
)
```

**注意事项：**
- 尽量将@Transactional放在Service层而非Controller层
- 同一个类内部方法调用不会触发代理，导致事务失效
- private方法上的@Transactional也会失效（Spring AOP不代理private方法）

## 题目6：消息队列 - Kafka分区分配策略

### 【答案】

**Kafka分区分配策略**

分区分配策略决定了消费者组中各个消费者实例如何分配Topic的分区。

**三种分配策略：**

**1. RangeAssignor（范围分配）**

- 默认策略
- 按Topic逐个分配
- 计算公式：`n = 分区数 % 消费者数量`，前n个消费者多分配1个分区

**示例：**
```
Topic A有10个分区，消费者C1、C2、C3
分配结果：
- C1: P0, P1, P2, P3
- C2: P4, P5, P6
- C3: P7, P8, P9
```

**问题：**
- 可能导致分配不均
- 如果多个Topic，可能导致部分消费者负载过重

**2. RoundRobinAssignor（轮询分配）**

- 将所有Topic的分区和消费者排序后轮询分配
- 保证更均衡的分配

**示例：**
```
Topic A: P0, P1, P2
Topic B: P0, P1
消费者：C1, C2

分配结果：
- C1: A-P0, A-P2, B-P1
- C2: A-P1, B-P0
```

**3. StickyAssignor（粘性分配）**

- Kafka 0.11+引入
- 目标：分区分配尽可能均匀，同时在重新分配时尽可能保留原有分配

**特点：**
- 初始分配与RoundRobin类似
- 重新分配时，尽量少移动分区
- 最大化消费者和分区的对应关系稳定性

**自定义分配策略：**

实现PartitionAssignor接口：
```java
public interface PartitionAssignor {
    String name();
    Subscription subscription(Cluster cluster);
    Map<String, List<TopicPartition>> assign(Cluster cluster, 
        Map<String, Subscription> subscriptions);
}
```

**消费者组再平衡触发条件：**
- 消费者组成员变化（加入/离开）
- 消费者订阅的Topic变化
- 分区数量变化
- 订阅的正则表达式匹配到新Topic

**再平衡过程：**
1. ConsumerGroup状态变为PreparingRebalance
2. 所有消费者停止拉取数据并提交offset
3. ConsumerGroup协调者选举新leader
4. 新leader制定分配方案并同步给所有成员
5. 所有成员执行新方案

## 题目7：消息队列 - RocketMQ事务消息原理

### 【答案】

**RocketMQ事务消息机制**

RocketMQ事务消息用于解决分布式事务问题，确保本地事务和消息发送的原子性。

**两种消息类型：**

1. **普通消息**： 发送即遗忘，无法保证最终一致性
2. **事务消息**： 支持二阶段提交，保证本地事务和消息发送的原子性

**事务消息流程：**

**第一阶段（Half消息）：**
1. 生产者发送Half（预处理）消息到RocketMQ
2. 此时消费者无法看到该消息
3. RocketMQ存储预处理消息，返回half put success

**第二阶段（执行本地事务）：**
1. 生产者收到half put success后，执行本地事务
2. 根据本地事务结果，发送COMMIT或ROLLBACK
3. 如果是COMMIT，Half消息变为可消费
4. 如果是ROLLBACK，消息被删除

**事务消息状态：**
- **UNKNOWN**： 生产者不知道事务结果，会执行回查
- **COMMIT_MESSAGE**： 提交，消息可被消费
- **ROLLBACK_MESSAGE**： 回滚，消息被删除

**回查机制：**

当生产者在特定期限内未返回结果时，RocketMQ会回查本地事务状态：
```
1. 发送half消息成功
2. 执行本地事务
3. 本地事务失败，未发送COMMIT/ROLLBACK
4. RocketMQ等待超时（默认60秒）
5. 向生产者回查本地事务状态
6. 生产者查询本地事务结果
7. 根据结果发送COMMIT或ROLLBACK
```

**关键配置参数：**

```java
// TransactionMQProducer配置
producer.setTransactionCheckInterval(1000);  // 回查间隔
producer.setTransactionCheckTimeout(30);  // 事务超时时间
```

**事务消息使用示例：**

```java
// 1. 创建事务监听器
TransactionListener transactionListener = new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        // 执行本地事务
        boolean success = orderService.createOrder(msg);
        return success ? LocalTransactionState.COMMIT_MESSAGE 
                       : LocalTransactionState.ROLLBACK_MESSAGE;
    }
    
    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 回查本地事务状态
        String orderId = msg.getKeys();
        return orderService.isOrderSuccess(orderId) 
            ? LocalTransactionState.COMMIT_MESSAGE
            : LocalTransactionState.ROLLBACK_MESSAGE;
    }
};

// 2. 创建事务生产者
TransactionMQProducer producer = new TransactionMQProducer("producer_group");
producer.setTransactionListener(transactionListener);
```

**事务消息限制：**
- 不支持批量消息
- 不支持延迟消息
- 单事务流程耗时上限为15秒（可配置）

**与普通消息的区别：**
- 预处理阶段增加网络开销
- 需要实现TransactionListener
- 事务失败时有回查机制保证最终一致

## 题目8：AI Agent - 向量数据库与Embedding在RAG中的应用

### 【答案】

**RAG核心组件**

RAG（检索增强生成）结合了检索系统和LLM，核心依赖向量数据库和Embedding技术。

**Embedding（嵌入）技术**

**什么是Embedding：**
- 将文本、图像等高维数据转换为低维稠密向量
- 语义相近的内容在向量空间中距离更近
- 使"语义检索"成为可能

**常用Embedding模型：**
- OpenAI的text-embedding-ada-002、text-embedding-3-small/large
- 开源的BGE、M3E、Sentence-BERT等
- 向量维度通常为256/512/768/1024/1536/3072

**Embedding流程：**
```
原始文本 → 分词(Tokenization) → 模型编码 → 256/768维向量
```

**向量数据库**

**核心功能：**
- 存储Embedding向量
- 高效相似度检索（ANN近似最近邻）
- 支持海量向量快速查询

**主流向量数据库：**
| 名称 | 特点 |
|------|------|
| Milvus | 开源、功能丰富、性能强 |
| Pinecone | 云原生、托管服务 |
| Weaviate | 支持混合搜索 |
| Chroma | 轻量级、适合本地 |
| Qdrant | Rust实现、高性能 |

**相似度度量方式：**
- **余弦相似度(Cosine)**： 衡量向量夹角，最常用
- **欧氏距离(Euclidean)**： 直线距离，适合图像
- **点积(Dot Product)**： 计算速度快

**RAG检索流程**

```
1. 文档处理
   原始文档 → 分块(Chunking) → Embedding → 存入向量数据库

2. 检索阶段
   用户Query → Embedding → 向量数据库相似度搜索 → Top-K相关文档

3. 生成阶段
   用户Query + 检索文档 → LLM → 生成答案
```

**分块策略（Chunking）：**

| 策略 | 特点 | 适用场景 |
|------|------|----------|
| 固定大小分块 | 简单可控，可能截断语义 | 通用场景 |
| 递归分块 | 按层级分割，保证语义完整 | 文档结构复杂 |
| 按段落分块 | 保留完整段落语义 | 文章、新闻 |
| Semantic分块 | 按语义相似度聚类 | 高度结构化文档 |

**分块参数考虑：**
- chunk_size： 通常100-512个token
- overlap： 重叠token数，保证上下文连续

**Advanced RAG技术：**

**1. 混合检索：**
```python
# 结合关键词和向量检索
results = vector_db.search(query_vector, top_k=5)  # 向量检索
keyword_results = bm25.search(query_text)  # 关键词检索
final_results = merge(results, keyword_results)  # 融合
```

**2. 重排序(Reranker)：**
- 使用交叉编码器对初筛结果重新排序
- 如Cohere Rerank、BGE-Reranker
- 牺牲速度换取更高精度

**3. Query改写：**
- HyDE： 用假设的理想答案辅助检索
- Query Decomposition： 将复杂问题分解

**评估指标：**
- **召回率(Recall)**： 相关文档是否被检索到
- **精确率(Precision)**： 检索结果是否都相关
- **MRR@K**： 前K个结果中第一个相关文档的排名倒数均值
- **NDCG@K**： 综合考虑相关性和排名位置

## 题目9：算法 - 合并区间

### 【答案】

**题目描述**

给出一个区间的集合，请合并所有重叠的区间。

**示例：**
```
输入: intervals = [[1,3],[2,6],[8,10],[15,18]]
输出: [[1,6],[8,10],[15,18]]
解释: 区间 [1,3] 和 [2,6] 重叠, 合并为 [1,6]。

输入: intervals = [[1,4],[4,5]]
输出: [[1,5]]
解释: 区间 [1,4] 和 [4,5] 可被视为重叠区间。
```

**解题思路**

**核心思想：**
先按区间的起始位置排序，然后遍历合并。

**步骤：**
1. 按起点升序排序
2. 初始化结果数组，将第一个区间加入
3. 遍历剩余区间：
   - 如果当前区间起点 <= 结果最后一个区间的终点，说明重叠，合并（取较大终点）
   - 否则，不重叠，直接加入结果

**Java实现：**

```java
public int[][] merge(int[][] intervals) {
    if (intervals == null || intervals.length <= 1) {
        return intervals;
    }
    
    // 按起点升序排序
    Arrays.sort(intervals, (a, b) -> a[0] - b[0]);
    
    List<int[]> result = new ArrayList<>();
    int[] current = intervals[0];
    result.add(current);
    
    for (int i = 1; i < intervals.length; i++) {
        if (intervals[i][0] <= current[1]) {
            // 重叠，合并区间（取终点最大值）
            current[1] = Math.max(current[1], intervals[i][1]);
        } else {
            // 不重叠，直接加入
            current = intervals[i];
            result.add(current);
        }
    }
    
    return result.toArray(new int[result.size()][]);
}
```

**复杂度分析：**
- **时间复杂度**：O(n log n)，排序O(n log n)，遍历O(n)
- **空间复杂度**：O(log n)（排序栈空间）或O(n)（结果存储）

**变种问题：**

**1. 插入区间：**
```java
public int[][] insert(int[][] intervals, int[] newInterval) {
    List<int[]> result = new ArrayList<>();
    int i = 0;
    
    // 添加所有在新区间之前的区间
    while (i < intervals.length && intervals[i][1] < newInterval[0]) {
        result.add(intervals[i++]);
    }
    
    // 合并所有与新区间重叠的区间
    while (i < intervals.length && intervals[i][0] <= newInterval[1]) {
        newInterval[0] = Math.min(newInterval[0], intervals[i][0]);
        newInterval[1] = Math.max(newInterval[1], intervals[i][1]);
        i++;
    }
    result.add(newInterval);
    
    // 添加剩余区间
    while (i < intervals.length) {
        result.add(intervals[i++]);
    }
    
    return result.toArray(new int[result.size()][]);
}
```

**2. 无重叠区间（贪心）：**
- 移除最少区间使区间互不重叠
- 按终点排序，选择终点最早的区间保留

## 题目10：算法 - 环形链表检测

### 【答案】

**题目描述**

给定一个链表，判断链表中是否有环。

**进阶：** 你能否不使用额外空间（O(1)空间复杂度）解决此题？

**示例：**
```
输入：head = [3,2,0,-4], pos = 1（尾节点连接到索引1的节点）
输出：true
解释：链表中有一个环，其后尾部连接到第二个节点。
```

**解题方法**

**方法一：哈希表（Set）**
- 时间复杂度：O(n)
- 空间复杂度：O(n)

```java
public boolean hasCycle(ListNode head) {
    Set<ListNode> seen = new HashSet<>();
    ListNode current = head;
    
    while (current != null) {
        if (seen.contains(current)) {
            return true;
        }
        seen.add(current);
        current = current.next;
    }
    return false;
}
```

**方法二：快慢指针（推荐）**
- 时间复杂度：O(n)
- 空间复杂度：O(1)

```java
public boolean hasCycle(ListNode head) {
    if (head == null || head.next == null) {
        return false;
    }
    
    ListNode slow = head;      // 慢指针，一次走一步
    ListNode fast = head.next;  // 快指针，一次走两步
    
    while (slow != fast) {
        if (fast == null || fast.next == null) {
            return false;  // 到达末尾，无环
        }
        slow = slow.next;
        fast = fast.next.next;
    }
    return true;  // 快慢指针相遇，有环
}
```

**为什么快慢指针一定能检测出环？**

设：
- 链表环长为C
- 入环前长度为L
- 快指针速度是慢指针的2倍

当慢指针刚进入环时，快指针已在环内距离入口L的位置。

在环内，相对距离为C-L，快慢指针每回合距离减少1，经过C-L回合后相遇。

数学证明：慢指针在环内走过的距离 <= L + C，快指针走过的距离 <= 2(L+C)，在n轮内必相遇。

**扩展：返回环的起始节点**

```java
public ListNode detectCycle(ListNode head) {
    if (head == null) return null;
    
    ListNode slow = head;
    ListNode fast = head;
    
    // 第一步：判断是否有环
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) {
            // 第二步：找环起点
            slow = head;
            while (slow != fast) {
                slow = slow.next;
                fast = fast.next;
            }
            return slow;
        }
    }
    return null;
}
```

**原理：**
- 第一相遇时，慢指针走了L + a圈
- 第一相遇时，快指针走了L + a + b圈（b为入环前距离）
- 2(L + a) = L + a + b → L = b
- 即：从相遇点继续走到入环点，与从head走到入环点的距离相同
