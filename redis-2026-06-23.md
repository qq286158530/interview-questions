## 题目一：Redis 是单线程吗？为什么单线程还能保持高性能？

### 参考答案

**Redis 是单线程的吗？**

Redis 单线程指的是「接收客户端请求 → 解析请求 → 进行数据读写等操作 → 发送数据给客户端」这个核心流程是由一个线程（主线程）来完成的，这也是我们常说 Redis 是单线程的原因。

但 Redis 程序本身并不是完全单线程的，Redis 在启动时会启动后台线程（BIO）：

- **Redis 2.6 版本**：会启动 2 个后台线程，分别处理关闭文件、AOF 刷盘这两个任务；
- **Redis 4.0 版本之后**：新增了一个 lazyfree 线程，用来异步释放 Redis 内存。例如执行 `unlink key` / `flushdb async` / `flushall async` 等命令，会把这些删除操作交给后台线程来执行，好处是不会阻塞主线程。

**为什么单线程还能保持高性能？**

1. **瓶颈不在 CPU**：Redis 的大部分操作都在内存中完成，并且采用了高效的数据结构（SDS、跳表、压缩列表等），因此 Redis 的瓶颈是内存和网络带宽，而非 CPU。既然 CPU 不是瓶颈，采用单线程就没有问题。

2. **避免竞争开销**：Redis 采用单线程模型，避免了多线程之间的竞争、省去了多线程切换带来的时间和性能开销，也不会产生死锁问题。

3. **I/O 多路复用**：Redis 采用了 I/O 多路复用机制（`select`/`epoll`）处理大量的客户端 Socket 请求。在单线程情况下，内核同时监听多个 Socket 上的连接请求或数据请求，一旦有请求到达，就交给 Redis 线程处理，实现了一个线程处理多个 IO 流的效果。

4. **Redis 6.0 多线程优化**：Redis 6.0 对于网络 I/O 采用多线程处理（但命令执行仍然是单线程），对性能提升至少一倍以上。默认 I/O 多线程只针对发送响应数据（write client socket），如果需要开启多线程处理读请求，需要将 `io-threads-do-reads` 配置项设为 yes。

## 题目二：Redis 有哪几种持久化方式？RDB 和 AOF 各自的优缺点是什么？

### 参考答案

**Redis 的三种持久化方式：**

**1. RDB 快照（Redis Database）**

将某一时刻的内存数据，以二进制的方式写入磁盘。Redis 提供了两个命令来生成 RDB 文件：
- `save`命令：在主线程执行，会阻塞主线程；
- `bgsave`命令：创建子进程生成 RDB 文件，避免阻塞主线程。

Redis 可以通过配置文件实现自动触发 bgsave：
```
save 900 1   # 900秒内至少1次修改
save 300 10  # 300秒内至少10次修改
save 60 10000 # 60秒内至少10000次修改
```

**2. AOF 日志（Append Only File）**

每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里。Redis 重启时，通过逐一执行 AOF 文件中的命令来恢复数据。

AOF 提供了三种写回策略（`appendfsync`）：
- `Always`：每次写操作执行完后同步将 AOF 日志写回硬盘，可靠性最高但性能最差；
- `Everysec`（默认）：每秒将缓冲区内容写回硬盘，推荐配置；
- `No`：由操作系统控制写回时机，性能最好但数据丢失风险最大。

**3. 混合持久化（Redis 4.0+）**

结合了 RDB 和 AOF 的优点。AOF 重写时，先将以 RDB 方式写入文件前半部分，再将增量命令以 AOF 格式写入后半部分。重启加载时，先快速加载 RDB 部分，再加载 AOF 增量数据。

**RDB 优缺点：**
- ✅ 优点：恢复速度快（二进制文件，直接读入内存）；
- ❌ 缺点：无法做到实时持久化，fork 子进程时有短暂阻塞；

**AOF 优缺点：**
- ✅ 优点：数据丢失少，可配置多种策略；
- ❌ 缺点：文件体积较大，恢复速度慢（需要逐条执行命令）；

**RDB 在快照时数据能修改吗？**
可以。执行 `bgsave` 时通过 **写时复制（Copy-On-Write, COW）** 技术，主线程的写操作会复制一份副本给子进程写入 RDB，主线程继续修改原数据。

## 题目三：如何解决 Redis 缓存雪崩、缓存击穿和缓存穿透？

### 参考答案

**1. 缓存雪崩**

大量缓存数据在同一时间过期（失效），此时大量请求都无法在 Redis 中处理，全部涌向数据库，导致数据库压力骤增甚至宕机。

**解决方案：**
- **过期时间随机化**：在原有失效时间基础上增加一个随机值（如 1~10 分钟），降低缓存集体失效的概率；
- **设置缓存不过期**：通过后台服务更新缓存数据，避免因缓存失效引发的雪崩；
- **双缓存策略**：设置主缓存和备用缓存，延长数据可用性。

**2. 缓存击穿**

某个热点数据过期了，此时大量请求访问该热点数据，直接冲垮数据库。与缓存雪崩类似，可认为是雪崩的一个子集。

**解决方案：**
- **互斥锁**：使用 `setNX` 保证同一时间只有一个业务线程请求缓存，未获取锁的请求等待或返回默认值；
- **热点数据永不过期**：由后台异步更新缓存，或在热点数据过期前提前更新；
- **逻辑过期**：不设置实际过期时间，存储逻辑过期时间字段，读取时判断是否过期，过期则异步重建缓存。

**3. 缓存穿透**

请求的数据既不在缓存中，也不在数据库中。大量恶意请求访问不存在的数据，数据库压力骤增。

**原因**：业务误操作（缓存和数据库都被误删）或恶意攻击（故意大量访问不存在的数据）。

**解决方案：**
- **非法请求限制**：在 API 入口处判断请求参数是否合理，过滤恶意请求；
- **设置空值/默认值**：缓存一个空值或默认值，阻挡后续穿透请求；
- **布隆过滤器（Bloom Filter）**：在写入数据库时使用布隆过滤器做标记，查询时先通过布隆过滤器快速判断数据是否存在，无需查询数据库即可过滤掉不存在的请求。

## 题目四：Redis 主从复制、哨兵模式和 Cluster 集群的区别是什么？如何选型？

### 参考答案

**1. 主从复制（Master-Slave）**

- 原理：主节点进行读写操作，从节点只读并同步主节点的写操作命令（异步复制）；
- 特点：一主多从，读写分离，主节点故障需要手动恢复；
- 数据一致性：无法保证强一致性，因为主节点不会等从节点执行完命令再返回客户端；
- 适用场景：数据量不大、对一致性要求不高的读多写少场景。

**2. 哨兵模式（Sentinel）**

- 在主从复制基础上增加了监控、自动故障转移功能；
- 哨兵（Sentinel）是一个独立进程，监控主从节点的存活状态；
- 当主节点故障时，哨兵自动从从节点中选举出一个新的主节点，实现自动故障转移；
- 客户端通过哨兵获取当前的主节点地址，实现透明切换；
- 适用场景：对可用性要求较高、需要自动故障转移的生产环境。

**3. Redis Cluster（切片集群）**

- 当缓存数据量巨大到单台服务器无法缓存时使用；
- 采用 **哈希槽（Hash Slot）** 机制：共有 16384 个哈希槽，每个 key 通过 CRC16 算法计算后对 16384 取模，决定归属哪个槽，进而决定归属哪个节点；
- 支持自动分片：数据分布在不同的 Redis 节点上，每个节点负责一部分哈希槽；
- 去中心化：每个节点都保存完整的数据分布信息，客户端可连接任意节点；
- 适用场景：数据量大、需要水平扩展、高并发读写的场景。

**选型建议：**
| 场景 | 推荐方案 |
|------|---------|
| 单机部署，简单缓存 | 无需集群 |
| 读多写少，需要主从备份 | 主从复制 |
| 高可用，生产环境 | 哨兵模式 |
| 数据量大，需要水平扩展 | Redis Cluster |

## 题目五：Redis 的过期删除策略和内存淘汰策略有什么区别？

### 参考答案

**一、过期删除策略（删除已过期的 key）**

Redis 对 key 设置了过期时间后，会将 key 带上过期时间存储到**过期字典（expires dict）** 中。Redis 采用「惰性删除 + 定期删除」两种策略配合使用：

**1. 惰性删除（Lazy Expiration）**
- 每次访问 key 时，才检测是否过期，过期则删除；
- ✅ 优点：对 CPU 友好，只占用很少的系统资源；
- ❌ 缺点：如果过期 key 一直不被访问，内存就不会释放，对内存不友好。

**2. 定期删除（Periodic Expiration）**
- 每隔一段时间，随机抽取一批 key 检查，删除其中已过期的 key；
- 流程：从过期字典随机抽取 20 个 key；若已过期 key 超过 5 个（25%），继续抽取；循环执行，时间上限 25ms；
- ✅ 优点：定期删除过期 key，释放内存；
- ❌ 缺点：难以权衡执行频率，配置不当会影响性能或浪费内存。

**二、内存淘汰策略（内存满时的处理方式）**

当 Redis 运行内存达到 `maxmemory` 阈值时，触发内存淘汰机制。八种策略分为两类：

**不淘汰（1种）：**
- `noeviction`：内存满时返回错误，不淘汰任何数据（Redis 3.0+ 默认策略）。

**在设置了过期时间的数据中淘汰（4种）：**
- `volatile-random`：随机淘汰设置了过期时间的任意 key；
- `volatile-ttl`：优先淘汰更早过期的 key；
- `volatile-lru`：淘汰所有设置了过期时间的 key 中最久未使用的（Redis 3.0 之前默认）；
- `volatile-lfu`（Redis 4.0+）：淘汰所有设置了过期时间的 key 中最少使用的。

**在所有数据中淘汰（3种）：**
- `allkeys-random`：随机淘汰任意 key；
- `allkeys-lru`：淘汰整个键值中最久未使用的 key；
- `allkeys-lfu`（Redis 4.0+）：淘汰整个键值中最少使用的 key。

**LRU vs LFU 的区别：**
- **LRU（Least Recently Used）**：淘汰最近最少使用的数据，Redis 通过在对象头中添加 `lru` 字段记录访问时间戳，采用随机采样 5 个 key 淘汰最久未使用的；
- **LFU（Least Frequently Used）**：淘汰访问频率最低的数据，Redis 的 `lru` 字段高 16 位存 `ldt`（最后访问时间），低 8 位存 `logc`（访问计数器），解决了 LRU 无法处理"缓存污染"的问题（偶尔访问一次的数据长期占用缓存）。
