## 题目一：Flink 的架构是怎样的？请描述 Flink 的核心组件及其作用

### 参考答案

Flink 采用**流式计算模型**，其核心架构分为以下几层：

**1. 分布式执行层（Distributed Execution）**
- **JobManager**：负责整个作业的调度，包括任务分配、Checkpoint 协调、故障恢复等。Flink 1.7+ 以后使用 **Dispatcher** + **ResourceManager** + **JobMaster** 的细粒度架构。
- **TaskManager**：负责实际执行算子任务，每个 TaskManager 是一个 JVM 进程，持有若干个 **Task Slot**。
- **Task Slot**：TaskManager 的资源抽象，每个 Slot 有固定的内存（Heap）配额，CPU 由 TaskManager 统一管理。Slot 之间共享 TCP 连接和 Heartbeat。

**2. 流式 API 层**
- **DataStream API**：用于无界数据流处理，支持窗口、状态、时间语义。
- **DataSet API**：用于有界批处理（在 1.15 后已deprecated，Flink SQL 成为主力）。

**3. Table API / SQL 层**
- 统一的关系 API，支持 Flink SQL，可直接跑在 DataStream 或 DataSet 之上。

**4. 运行时（Runtime）**
- 基于 **StreamGraph** → **JobGraph** → **ExecutionGraph** 的三层转换：
  - StreamGraph：由用户代码直接生成，保留源码级算子结构。
  - JobGraph：做过 operator chaining（算子链合并），减少数据交换。
  - ExecutionGraph：并行化后的物理执行图，记录每个 subtask 的部署位置。

**5. 状态后端（State Backend）**
- **HashMapStateBackend**：状态存储在 TaskManager JVM Heap，适合小状态。
- **EmbeddedRocksDBStateBackend**：状态溢出到 RocksDB（本地磁盘），支持超大状态增量 Checkpoint。

## 题目二：Flink 的 Watermark 机制是什么？它是如何解决乱序和迟到数据的？

### 参考答案

**Watermark（水位线）是 Flink 实现**事件时间（Event Time）**处理的核心机制，用于在乱序事件流中判定何时可以关闭一个窗口。

### 为什么需要 Watermark？

在真实场景中，数据因网络延迟、分布式系统等原因，事件到达的顺序往往和事件发生的时间（Event Time）不一致。如果只依赖**处理时间（Processing Time）**，结果将不确定且不可重现。

### Watermark 的定义

Watermark 是一个**时间戳 T**，语义如下：

> **当 Watermark = T 时，Flink 认为所有 Event Time < T 的数据都已到达（或已无法到达），可以触发窗口计算了。**

Watermark 插入位置在 Source 算子或通过 `assignTimestampsAndWatermarks()` 指定。

### 两种 Watermark 策略

```java
// 固定延迟 Watermark（适合离线测试）
WatermarkStrategy
  .forBoundedOutOfOrderness(Duration.ofSeconds(10))
  .withTimestampAssigner((event, ts) -> event.getTimestamp());

// 单调递增 Watermark（适合严格有序的数据源，如 Kafka 单分区）
WatermarkStrategy.forMonotonousTimestamps()
  .withTimestampAssigner(...);
```

### 处理迟到数据（Late Data）

- **默认策略**：Watermark 到达窗口_end_time 后，窗口立即触发，迟到数据被丢弃。
- **allowedLateness()**：设置一个额外等待时间，允许窗口在触发后继续保留状态，迟到数据仍可触发窗口计算：
  ```java
  .allowedLateness(Time.minutes(5))
  ```
- **sideOutputLateData()**：将迟到数据侧输出到额外流：
  ```java
  .sideOutputLateData(outputTag)
  ```

### 乱序与 Watermark 的关系

Watermark 解决了"**乱序多少**"的问题；`allowedLateness` 解决了"**还能等多久**"的问题。两者配合使用，既能保证计算结果的准确性，又能控制状态膨胀。

## 题目三：Flink Checkpoint 与 Savepoint 的区别是什么？Flink 如何保证精确一次（Exactly-Once）语义？

### 参考答案

### Checkpoint

- **触发方式**：由 Flink 自动周期性触发（由 JobManager 的 CheckpointCoordinator 协调）。
- **目的**：为 Flink 作业提供**故障恢复**能力，状态存储在指定的 State Backend（内存/RocksDB）。
- **存储路径**：通常写入分布式文件系统（如 HDFS、S3、NFS）。
- **清理策略**：默认保留最近一次成功完成的 Checkpoint，旧的可配置清理。
- **作业兼容**：同一个作业的不同版本 Checkpoint **可以复用**（前提是算子拓扑不变）。

### Savepoint

- **触发方式**：由用户**手动命令**触发（`bin/flink savepoint`）。
- **目的**：用于**计划内维护**（如升级、扩缩容、迁移作业）。
- **存储路径**：用户指定目录，不会被自动清理。
- **作业兼容**：需要作业图完全兼容（算子 ID 必须保持一致，否则无法恢复）。
- **恢复命令**：`flink run -s savepoint_path`。

### 对比表

| 特性 | Checkpoint | Savepoint |
|------|-----------|-----------|
| 触发方 | Flink 系统自动 | 用户手动 |
| 作业停止后是否保留 | 否（可配置） | 是 |
| 用途 | 故障恢复 | 计划维护/迁移 |
| 算子 ID 兼容性 | 自动兼容 | 必须手动指定 |

### Exactly-Once 语义保证

Flink 通过 **Chandy-Lamport 分布式 Checkpoint 协议**实现端到端的 Exactly-Once，核心三要素：

1. **Source 端的 Barrier 对齐**：Kafka Source 会在 checkpoint 期间暂停消费，确保 offset 记录与 state 的一致性。
2. **算子状态快照**：每个算子在收到所有输入的 Barrier 后，对本地状态做快照。
3. **Sink 端的幂等/事务写入**：
   - **幂等 Sink**（如 MySQL INSERT ON DUPLICATE KEY UPDATE）：天然提供 Exactly-Once。
   - **事务 Sink**（如 Flink Kafka Sink 的 `EXACTLY_ONCE` 模式）：两阶段提交，Barrier 与数据一起提交。

```java
// Kafka Sink Exactly-Once 配置示例
KafkaSink.builder()
  .setBootstrapServers("localhost:9092")
  .setRecordSerializer(KafkaRecordSerializationSchema.builder()
      .setTopic("output-topic")
      .setValueSerializationSchema(new SimpleStringSchema())
      .build())
  .setDeliveryGuarantee(DeliveryGuarantee.EXACTLY_ONCE)
  .setTransactionalIdPrefix("flink-")
  .build();
```

## 题目四：Flink 窗口函数有哪几种类型？滚动窗口、滑动窗口、会话窗口的区别是什么？

### 参考答案

### 窗口类型总览

Flink 支持两大类窗口：

**1. 键控窗口（Keyed Windows）**
- 按某个 Key 分组后，各 Key 独立维护窗口。
- `DataStream.keyBy(...).window(...)`

**2. 非键控窗口（Non-Keyed Windows）**
- 全局窗口，所有数据共享一个窗口。
- `DataStream.windowAll(...)`
- 并行度被限制为 1（不适合大数据量场景）。

---

### 滚动窗口（Tumbling Window）

- **特点**：窗口大小固定，**不重叠**，首尾相接。
- **适用场景**：需要周期性统计，如"每 5 分钟统计一次 PV"。
- **示例**：
  ```java
  // 每5分钟一个窗口，互不重叠
  .window(TumblingProcessingTimeWindows.of(Time.minutes(5)))
  ```
- **示意图**：`[0-5) [5-10) [10-15) ...`

---

### 滑动窗口（Sliding Window）

- **特点**：窗口大小固定，但可以**重叠**。由两个参数决定：窗口长度（size）和滑动步长（slide）。
- **适用场景**：需要平滑统计或兼顾历史数据，如"最近 10 分钟的平均温度，每 1 分钟更新一次"。
- **当 slide < size 时**：窗口重叠，数据会被多个窗口重复统计。
- **当 slide = size 时**：退化为滚动窗口。
- **示例**：
  ```java
  // 窗口大小10分钟，滑动步长1分钟
  .window(SlidingProcessingTimeWindows.of(Time.minutes(10), Time.minutes(1)))
  ```
- **示意图**：`[0-10) [1-11) [2-12) ...`

---

### 会话窗口（Session Window）

- **特点**：根据**活跃间隙**动态创建和关闭窗口。没有固定开始结束时间，数据之间的间隔超过 gap 就开启新窗口。
- **适用场景**：用户行为分析，如"用户操作间隔超过 30 秒视为一次新会话"。
- **两种 Gap 策略**：
  - **ProcessingTime Session**：基于处理时间，简单但不确定。
  - **EventTime Session**：基于事件时间，需要 Watermark 支持。
- **示例**：
  ```java
  // 会话间隔超过5秒则开始新窗口
  .window(ProcessingTimeSessionWindows.withGap(Time.seconds(5)))
  
  // 动态 Gap（基于数据本身）
  .window(EventTimeSessionWindows.withDynamicGap(new SessionWindowTimeGapSelector<String, TimeDuration>() {
      @Override public TimeDuration select(String s) {
          return TimeDuration.ofSeconds(s.length()); // 动态gap
      }
  }))
  ```

---

### 全局窗口（Global Window）

- 所有数据落入同一个窗口，需要配合**触发器（Trigger）**才能手动触发计算。
- 很少单独使用，适合"无限数据积累后手动触发"场景。

---

### 窗口函数（Window Functions）

窗口内的数据如何计算：

| 类型 | 说明 | 示例 |
|------|------|------|
| **ReduceFunction** | 增量聚合，内存效率高 | `reduce((v1,v2) -> v1+v2)` |
| **AggregateFunction** | 通用增量聚合，可输出不同类型 | `aggregate(new MyAggFunc())` |
| **FoldFunction** | 老旧 API，不推荐 | - |
| **ProcessWindowFunction** | 全量窗口数据，灵活但内存开销大 | `process(new MyProcessFunc())` |
| **增量+全量混合** | 先 Reduce/Aggregate，再 ProcessWindowFunction | 推荐写法 |

**推荐写法（增量+全量混合）**：
```java
.aggregate(
  new MyAggregateFunction(),  // 增量聚合，状态小
  new MyProcessWindowFunction() // 全量上下文，可访问窗口信息
)
```

## 题目五：Flink 的状态管理机制是什么？如何避免状态过大导致的 OOM？

### 参考答案

### Flink 状态分类

**1. 按数据结构分**

| 类型 | 说明 | 示例 |
|------|------|------|
| **ValueState\<T\>** | 单值状态 | `state.update(newValue)` |
| **ListState\<T\>** | 列表状态 | `state.add(item)`, `state.update(list)` |
| **MapState\<K, V\>** | 键值映射 | `state.put(k, v)`, `state.get(k)` |
| **ReducingState\<T\>** | 列表自动聚合 | 同 ListState，但自动调用 ReduceFunction |
| **AggregatingState\<IN, OUT\>** | 列表聚合为单值 | 自动调用 AggregateFunction |
| **FoldingState\<T, R\>** | 旧 API，不推荐 | - |

**2. 按作用域分**
- **Keyed State**：只能在 KeyedStream 上使用，状态按 Key 分区，各 Key 独立维护。
- **Operator State**（Non-Keyed）：整个算子实例共享一个状态（如 Kafka Source 的分区 offset），支持 **ListState** 和 **UnionListState**。

---

### 状态后端（State Backend）

| Backend | 存储位置 | 容量 | 读写性能 |
|---------|---------|------|---------|
| **HashMapStateBackend** | TaskManager JVM Heap | 受 Heap 大小限制 | 最快 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘 + Heap索引） | 可很大（磁盘决定） | 较慢（序列化开销） |

```java
// 配置 State Backend
// 方式1：全局配置（flink-conf.yaml）
state.backend: rocksdb
state.checkpoints.dir: hdfs://namenode:8020/flink-checkpoints

// 方式2：代码配置
env.setStateBackend(new EmbeddedRocksDBStateBackend());
env.getCheckpointConfig().setCheckpointStorage("hdfs://namenode:8020/flink-checkpoints");
```

---

### 如何避免状态过大导致 OOM

**1. 合理选择 State Backend**
- 小状态（MB 级别）→ HashMapStateBackend
- 大状态（GB 级别）→ EmbeddedRocksDBStateBackend

**2. 状态过期清理（TTL）**
```java
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.days(7))
  .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite) // 读写都刷新TTL
  .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)
  .build();
ValueStateDescriptor<String> desc = new ValueStateDescriptor<>("my-state", String.class);
desc.enableTimeToLive(ttlConfig);
```

**3. 增量 Checkpoint（RocksDB）**
```yaml
# flink-conf.yaml
state.backend.incremental: true
```
只上传本次 Checkpoint 与前一次的 diff，而非全量状态，大幅减少网络传输和存储。

**4. 算子链合并（Operator Chaining）**
减少数据在算子间的流动，减少中间状态。

**5. 合理设计 Key 的粒度**
- Key 越少，状态越少（如按 user_id 分组 vs 按 device_id 分组）。
- 避免 high cardinality key（如随机 UUID），会导致状态爆炸。

**6. 使用 UnionListState 而非 ListState**
当算子扩缩容时，UnionListState 会将完整状态广播给所有新实例，再由各实例自行过滤自己需要的部分，避免状态重复复制。

**7. 定期监控 Checkpoint 大小**
```java
// 通过 Metric 监控
CheckpointMetrics.getTotalStateSize()
```
超过阈值时及时报警或优化。
