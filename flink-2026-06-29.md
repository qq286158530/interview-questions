## 题目 1：Flink 的架构是怎样的？为什么 Flink 能做到 exactly-once 语义？

### 参考答案

Flink 采用 **Master-Slave 架构**，核心组件包括：

- **JobManager**：协调整个作业的执行，负责调度任务、管理 Checkpoint、协调算子等
- **TaskManager**：执行具体的 Task，每个 TaskManager 是一个 JVM 进程，负责执行算子子任务
- **ResourceManager**：管理 TaskManager 的 slot 和资源分配
- **Dispatcher**：接收用户提交的作业，启动 JobManager

Flink 通过 **Chandy-Lamport 分布式快照算法** 实现 exactly-once 语义：

1. **Barrier 对齐机制**：Checkpoint 过程中，数据流中会注入 Barrier，Barrier 将数据分割成不同的快照。算子收到所有输入分区的 Barrier 后才做快照，保证各分区数据一致。
2. **两阶段提交（Two-Phase Commit）**：Flink 的 Sink 通过两阶段提交协议与 Checkpoint 配合：算子完成快照时先"预提交"，Checkpoint 完成时再"正式提交"。如果作业失败，已预提交的数据会被回滚，保证输出只有一次。
3. **状态后端**：Flink 将状态存储在状态后端（如 RocksDB），与 Checkpoint 机制协同，保证故障恢复后状态精确恢复。

> **来源参考**：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/flink-architecture/

## 题目 2：什么是 Watermark？Watermark 的作用是什么？如何处理乱序数据？

### 参考答案

**Watermark（水位线）** 是 Flink 中用于处理**事件时间（Event Time）**乱序问题的机制，本质是一个时间戳。

### 作用

在事件时间语义下，数据到达的顺序可能和事件发生顺序不一致（网络延迟、分布式系统等）。Watermark 用于告诉算子：**在这个时间点之前的数据都已经到达了**，算子可以安全地触发窗口计算。

### 核心机制

- **Watermark 是一条数据**，携带时间戳 T，表示"所有时间戳 ≤ T 的数据都已到达"。
- 窗口会在以下两个条件同时满足时触发：
  1. 收到 Watermark 且 Watermark 时间 ≥ 窗口结束时间
  2. 窗口内已有数据

### 乱序处理策略

| 策略 | 说明 |
|------|------|
| **时间戳分配 + Watermark 生成** | 通常使用 `assignTimestampsAndWatermarks` 设置 Watermark 生成策略，常用 `BoundedOutOfOrdernessTimestampExtractor` 允许一定程度的乱序 |
| **allowedLateness** | 设置允许的延迟时间，窗口触发后仍然保留状态一段时间，迟到数据仍能触发窗口计算 |
| **侧输出（Side Output）** | 将严重迟到或被丢弃的数据输出到侧输出流，供后续处理 |

### 注意事项

- Watermark 生成方式影响数据完整性：设置过短可能导致数据丢失，过长则增加延迟
- 多分区输入时，算子取所有分区 Watermark 的**最小值**作为当前 Watermark

> **来源参考**：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/timestamps_watermarks/

## 题目 3：Flink Checkpoint 的工作原理是什么？如何配置 Checkpoint？

### 参考答案

**Checkpoint** 是 Flink 实现故障恢复的核心机制，基于 **Chandy-Lamport 分布式快照算法**，将状态快照保存到持久化存储。

### 工作原理

1. **JobManager 触发 Checkpoint**：定时向所有算子发送 Barrier（检查点屏障）
2. **Barrier 流动**：Barrier 沿数据流向下游算子传递，将数据切分成多个快照
3. **状态快照**：算子收到所有输入的 Barrier 后，将自身状态写入状态后端（可配置为 RocksDB、HashMap 等）
4. **Checkpoint 完成**：JobManager 确认所有算子快照完成后，Checkpoint 才算成功

### 核心配置参数

```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

// 启用 Checkpoint，间隔 10 秒
env.enableCheckpointing(10000);

// Checkpoint 存储模式：EXACTLY_ONCE（默认）
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);

// Checkpoint 超时时间
env.getCheckpointConfig().setCheckpointTimeout(60000);

// 最小间隔：两次 Checkpoint 至少间隔 1 秒
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(1000);

// 允许的最大并发 Checkpoint 数
env.getCheckpointConfig().setMaxConcurrentCheckpoints(1);

// 开启非对齐 Checkpoint（减少 Barrier 对齐等待）
env.getCheckpointConfig().enableUnalignedCheckpoints();
```

### 状态后端选择

| 状态后端 | 适用场景 |
|---------|---------|
| **HashMapStateBackend** | 状态较小、低延迟场景，数据存储在 JVM 堆内存 |
| **EmbeddedRocksDBStateBackend** | 大状态、增量 Checkpoint 场景，数据存储在 RocksDB，支持增量快照 |

> **来源参考**：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/checkpoints/

## 题目 4：Flink 的窗口函数有哪些？滚动窗口、滑动窗口、会话窗口的区别是什么？

### 参考答案

Flink 支持两种类型的窗口：**Keyed Windows** 和 **Non-Keyed Windows**，窗口函数分为三大类：

### 窗口类型

| 窗口类型 | 说明 | 适用场景 |
|---------|------|---------|
| **滚动窗口（Tumbling Window）** | 大小固定、不重叠，例如每 5 分钟一个窗口 | 固定周期的统计，如每分钟 PV |
| **滑动窗口（Sliding Window）** | 大小固定、可重叠，由"窗口大小"和"滑动步长"共同决定 | 需要平滑统计，如每 5 分钟计算最近 10 分钟的均值 |
| **会话窗口（Session Window）** | 基于活动间隙动态创建，窗口在 inactivity 超时后关闭 | 用户行为分析，如用户超过 30 秒无操作则结束会话 |

### 代码示例

```java
// 滚动窗口：每 5 分钟统计一次
input.keyBy(x -> x.key)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .sum(1);

// 滑动窗口：窗口大小 10 分钟，滑动步长 5 分钟
input.keyBy(x -> x.key)
    .window(SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(5)))
    .sum(1);

// 会话窗口：会话间隔 30 秒
input.keyBy(x -> x.key)
    .window(EventTimeSessionWindows.withGap(Time.seconds(30)))
    .sum(1);
```

### 全窗口函数（ProcessWindowFunction）

与增量聚合函数（`ReduceFunction`、`AggregateFunction`）不同，`ProcessWindowFunction` 可以访问窗口的完整元数据，适合需要复杂窗口逻辑的场景：

```java
input.keyBy(x -> x.key)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .process(new ProcessWindowFunction<Tuple2<String, Long>, String, String, TimeWindow>() {
        @Override
        public void process(Context context, Iterable<Tuple2<String, Long>> values,
                           Collector<String> out) {
            long count = 0;
            for (Tuple2<String, Long> val : values) {
                count++;
            }
            out.collect("窗口内共有 " + count + " 条数据");
        }
    });
```

> **来源参考**：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream/operators/windows/

## 题目 5：Flink 状态后端有哪些？如何选择合适的状态后端？

### 参考答案

Flink 提供三种状态后端（State Backend），用于存储算子状态和快照。

### 三种状态后端

| 状态后端 | 存储位置 | 适用场景 | 特点 |
|---------|---------|---------|------|
| **HashMapStateBackend** | JVM 堆内存 | 状态较小、要求低延迟 | 性能最高，但受限于内存，且故障恢复需要全量拉取快照 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘 + 内存） | 大状态、增量 Checkpoint | 支持增量快照，状态可超过内存限制，但读写有一定延迟 |
| **EnvironmentDefault** | 取决于 flink-conf.yaml 配置 | 一般生产场景 | 灵活配置，默认同 HashMapStateBackend |

### 代码中指定状态后端

```java
// 使用 HashMapStateBackend
env.setStateBackend(new HashMapStateBackend());

// 使用 EmbeddedRocksDBStateBackend（支持增量 Checkpoint）
env.setStateBackend(new EmbeddedRocksDBStateBackend());
```

### 生产环境选型建议

1. **小状态、低延迟**（如实时 ETL）→ `HashMapStateBackend`
2. **大状态、超大窗口**（如明细计算）→ `EmbeddedRocksDBStateBackend`
3. **开启增量 Checkpoint** 的场景 → 必须使用 `EmbeddedRocksDBStateBackend`

### 状态过期与清理

- 通过 `state TTL` 限制状态生命周期，防止状态无限膨胀：
```java
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.days(7))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
    .build();
```

> **来源参考**：https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/state_backends/
