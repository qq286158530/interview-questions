## 题目一：HBase 的整体架构是怎样的？RegionServer 和 HMaster 的职责是什么？

### 参考答案

HBase 采用经典的 **Master-Slave 架构**，核心组件如下：

| 组件 | 说明 |
|------|------|
| **HMaster** | 负责表的管理工作——创建表、删除表、`ALTER` 表结构等 DDL 操作；同时管理 RegionServer 的负载均衡、故障转移、Region 切分以及元数据管理。HMaster 不参与实际的数据读写路径，挂了不影响数据写入。|
| **RegionServer** | 数据读写的实际执行者。每个 RegionServer 负责管理一组 Region（数据分片），包含 MemStore + HFile 的写入路径，并向 Zookeeper 汇报心跳。|
| **Zookeeper** | 存储 `.META.` 表位置、所有 RegionServer 的状态、元数据入口（root region location）。HMaster 通过 Zookeeper 实现选主和故障检测。|
| **HDFS** | HBase 的底层存储引擎，HFile 最终持久化到 HDFS 上，利用其副本机制保证数据高可用。|

**写入流程简述**：客户端 → Zookeeper 获取 `.META.` 位置 → 从 `.META.` 获取目标 Region位置 → 直接请求目标 RegionServer（写入 MemStore，满足阈值后刷盘为 HFile）。

**读取流程简述**：客户端 → Zookeeper → `.META.` → RegionServer，先查 MemStore（未命中）→遍历 HFile（借助 BlockCache）→ 合并返回结果。

> **来源**：https://hbase.apache.org/book.html#architecture

## 题目二：HBase 的 RowKey 设计有哪些原则？为什么 RowKey 设计至关重要？

### 参考答案

### RowKey 设计四大原则

1. **长度原则**：越短越好，推荐不超过 16 字节（最长 64KB）。RowKey 会在每次写入时复制存储，过长的 RowKey 会浪费 HDFS 副本和 MemStore 空间。
2. **散列原则**：避免热点（Hot Spot）。不要使用时间戳或连续递增字段作为 RowKey 的前缀，否则所有写入会集中到同一个 RegionServer，导致热点。解决方案：加盐（Salt）前缀、哈希前缀或反转时间戳。
3. **唯一性原则**：RowKey 即为主键，同一表的同一 RowKey 数据只存在一行（类似 MySQL 的唯一索引），如果业务上无唯一性需求则无需关注。
4. **查询匹配原则**：HBase 按 RowKey 字典序排序，**前缀匹配**是 HBase 最高效的查询方式。设计时要考虑业务未来会用哪些字段做查询，将最常用的查询字段放在 RowKey 前缀。

### 常见设计模式

| 场景 | 设计方案 | 示例 |
|------|---------|------|
| 用户订单查询 | 用户 ID 作为 RowKey 前缀 | `userId_orderId` |
| 避免热点 | 加盐/哈希 | `MD5(userId).substring(0,4)_userId` |
| 时间连续写入 | 哈希+时间反转 | `userId_reverseTimestamp` |
| 多条件组合查询 | 拼接多个字段 | `regionId_storeId_productId` |

**为什么至关重要？**
- HBase 没有二级索引，查询能力完全依赖 RowKey 的前缀匹配
- 热点问题会直接导致部分节点过载而其他节点空闲，严重影响集群稳定性和吞吐量
- 一旦表数据量上来后修改 RowKey 代价极大（需重写全表数据）

> **来源**：https://hbase.apache.org/book.html#rowkey.design

## 题目三：简述 HBase 的存储结构——MemStore 和 HFile 的作用是什么？它们之间如何协作？

### 参考答案

### 核心存储组件

**MemStore（写缓冲）**
- 写入 HBase 的数据首先进入 MemStore（内存缓冲区），按 RowKey 字典序排序（预排序）
- 每个 Store（列族）对应一个 MemStore，大小由 `hbase.hregion.memstore.flush.size` 控制（默认 128MB）
- 作用：**减少随机 I/O**：将随机写入先收集到有序内存结构，再批量顺序写入 HFile，大幅提升写性能
- 当 MemStore 满了或达到阈值时，触发 **flush** 刷盘操作，生成新的 HFile

**HFile（HBase 的存储文件）**
- HFile 是 HBase 在 HDFS 上的最终存储格式，基于 **Hadoop SequenceFile + B+Tree变体**（Google BigTable 的 SSTable）
- HFile 结构（从旧到新版本 v1~v3，核心不变）：

```
HFile
├── Data Block（数据块）：存储 KeyValue 对，按 key 排序，支持二分查找
├── Meta Block（可选）：存储元信息，如布隆过滤器数据
├── BlockIndex（块索引）：记录每个 Data Block 的起始 key 和偏移量
└── Trailer：元信息区，记录版本、块数量、偏移量等
```

- 每个 HFile 还内嵌 **布隆过滤器（BloomFilter）**，用于快速判断一个 RowKey 是否不存在于该文件中（避免无效 I/O）

### MemStore ↔ HFile 协作流程

```
写入请求 → WAL（Write-Ahead Log）→ MemStore（内存排序）
                              ↓ flush触发条件满足
                          生成 HFile（HDFS顺序写）
                              ↓
                     多个 HFile 通过 Minor Compaction 合并
                              ↓
                     Major Compaction 重写全量 HFile
                              ↓
                     删除过期/标记删除的数据
```

**两种 Compaction 策略**：
- **Minor Compaction**：将多个小 HFile 合并成少量大 HFile，不删除过期数据
- **Major Compaction**：将一个 Store 下所有 HFile 合并成一个，清理所有过期/删除数据（默认 7 天执行一次）

## 题目四：HBase 如何保证数据一致性？WAL（预写日志）的作用是什么？

### 参考答案

### HBase 的数据一致性保证

HBase 提供 **强一致性**保证——写入成功后，所有读请求都会读到最新写入的值。这是因为：
- 数据写入 MemStore 后，**必须先写入 WAL**，才能返回客户端成功
- RegionServer 故障重启时，通过重放 WAL 中未刷盘的 MemStore 数据来恢复内存中的数据

### WAL（Write-Ahead Log）机制

```
写入流程：
1. 客户端发起 Put 请求
2. RegionServer 先将操作写入 WAL（HDFS 上的顺序日志文件）
3. WAL 写入成功后，数据才写入 MemStore
4. 返回客户端 "写入成功"
```

**WAL 的核心作用**：防止数据丢失。如果 RegionServer 在数据未刷盘前崩溃，重启后从 WAL 重放未持久化的操作，保证数据不丢失。

**WAL 相关调优参数**：
| 参数 | 默认值 | 说明 |
|------|--------|------|
| `hbase.wal.dir` | HDFS 根目录 | WAL 存储路径 |
| `hbase.regionserver.flush.perWal` | 每写 N 条 flush 一次 | 批量 flush 策略 |
| `hbase.wal.replication.enabled` | true | 是否开启 WAL 复制（跨集群容灾）|

**一致性级别**（可配置）：
- `WRITE_ONE`：默认，只写 primary replica
- `ASYNC_WAL`：异步写 WAL，性能更高但可能丢数据
- `SYNC_WAL`：同步写 WAL，强制 fsync（高可靠低性能）

> **来源**：https://hbase.apache.org/book.html#wal

## 题目五：HBase 协处理器（Coprocessor）是什么？Observer 和 Endpoint 有什么区别？

### 参考答案

### 协处理器的本质

HBase 协处理器是 **运行在 RegionServer 端的扩展机制**，允许用户将自定义逻辑注入到 HBase 的读写路径中，类似数据库的触发器/存储过程，而无需修改 HBase 源码。

### 两种协处理器类型

####1. Observer（观察者）—— 类似触发器

在指定事件前后自动执行自定义代码，用于**拦截**和**修改**已有操作。

**可用 Hook节点**：
| Hook | 触发时机 |
|------|---------|
| `preGet` / `postGet` | GET 查询前后 |
| `prePut` / `postPut` | PUT写入前后 |
| `preDelete` / `postDelete` | DELETE 删除前后 |
| `preOpen` / `postOpen` | Region 打开时 |
| `preFlush` / `postFlush` | MemStore flush 前/后 |
| `preCompact` / `postCompact` | Compaction 前/后 |

**典型应用场景**：
- 实现二级索引（写入主表时自动同步更新索引表）
- 权限控制（列级/行级访问控制）
- 自动数据分层（冷热数据分离）
- 数据验证（写入前校验数据格式）

#### 2. Endpoint（端点）—— 类似存储过程

Endpoint 是**可被客户端调用的 RPC 服务**，运行在 RegionServer 上，支持对 Region 内的数据进行**聚合计算**。

**典型应用场景**：
- 在服务端执行 `SUM`、`AVG`、`COUNT` 等聚合操作（减少网络传输）
- 复杂业务逻辑的 Server-side 计算
- 自定义过滤器

**调用方式**（通过 `CoprocessorRpcChannel`）：
```java
// 客户端调用 Endpoint 示例
Scan scan = new Scan();
AggregationClient aggregationClient = new AggregationClient(conf);
long sum = aggregationClient.sum(TableName.valueOf("myTable"),
    new LongColumnInterpreter(), scan);
```

### 对比总结

| 维度 | Observer | Endpoint |
|------|---------|---------|
| 工作模式 | 事件驱动（自动触发） | RPC 调用（按需触发）|
| 返回值 | 可修改操作结果 | 返回计算结果给客户端 |
| 典型用途 | 触发器、索引同步 | 聚合计算、Server-side 逻辑 |
| 部署方式 | 继承 `BaseRegionObserver` | 实现 `CoprocessorProtocol` |

> **来源**：https://hbase.apache.org/book.html#cp
