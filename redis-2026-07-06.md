## 题目一：Redis 有哪些数据结构？它们的使用场景是什么？

### 答案

Redis 提供了 5 种基础数据结构，另外还有几种高级数据结构：

**5 种基础数据类型：**

| 数据结构 |底层实现| 典型使用场景 |
|---------|--------|------------|
| **String** | SDS（简单动态字符串） | 缓存、计数器、分布式锁、Session 共享 |
| **Hash** | 压缩列表 / 哈希表 | 对象存储（如用户信息）、购物车 |
| **List** | 快速链表 / 压缩列表 | 消息队列（lpush + brpop）、最新消息列表、关注列表 |
| **Set** | 哈希表 / 整数集合 | 去重、标签系统、好友关系、共同关注 |
| **Zset（Sorted Set）** | 压缩列表 + 跳表 | 排行榜、有序消息队列、延时队列 |

**高级数据结构：**

- **Bitmap**：位图，适合签到、活跃用户统计（`SETBIT` / `GETBIT`）
- **HyperLogLog**：基数统计，如 UV 统计，内存极小（`PFADD` / `PFCOUNT`）
- **Geospatial**：地理位置，`GEOADD` / `GEORADIUS`，适合附近的人功能
- **Stream**：流式数据结构，完整的消息队列，支持消费组（`XADD` / `XREADGROUP`）

**使用建议：**
- String 存储 JSON 时建议先压缩（`gzip`），避免内存浪费
- Hash 字段不宜过多（建议 < 100），否则退化为哈希表，内存开销大
- Zset 做排行榜时记得用 `ZREMRANGEBYRANK` 定期清理历史数据

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 答案

Redis 有两种持久化机制：

**RDB（Redis Database）— 快照**
- 触发方式：`BGSAVE`（后台异步）、`SAVE`（同步阻塞）或自动触发（`save` 配置）
- 原理：fork 出一个子进程，遍历内存数据写入 RDB 文件
- 优点：恢复速度快（一个大文件），适合全量备份
- 缺点：可能丢失最后一次快照后的数据（最多丢失 `save` 间隔内的数据）

**AOF（Append Only File）— 操作日志**
- 触发方式：每个写命令追加到 AOF 缓冲区，再刷盘（`appendfsync` 配置）
- `appendfsync` 三种策略：
  - `always`：每条命令都 fsync，零数据丢失，但慢
  - `everysec`（默认）：每秒一次，性能与数据安全折中
  - `no`：依赖 OS 刷盘，最快但可能丢失秒级数据
- 重写机制：`BGREWRITEAOF` 压缩 AOF 文件，去掉过期/已合并的命令
- 优点：数据安全性更高（`everysec` 最多丢 1 秒）
- 缺点：AOF 文件比 RDB 大，恢复慢

**对比总结：**

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件大小 | 小（紧凑快照） | 大（持续追加） |
| 恢复速度 | 快 | 慢 |
| 数据安全性 | 可能丢数据 | 丢数据少 |
| 性能影响 | fork 阻塞 | 持续 IO |
| 场景 | 备份、冷备 | 缓存、金融 |

**生产环境推荐策略：**
- 主从架构中，从节点开启 RDB（`save ""` 关闭，配置 `replica-serve-stale-data`）
- 开启 AOF + `everysec`，同时定期做 RDB 快照备份
- Redis 4.0+ 支持混合持久化（`aof-use-rdb-preamble yes`），AOF 重写时先写 RDB 再追加增量

## 题目三：Redis 的过期 key 删除策略是什么？

### 答案

Redis 采用**惰性删除 + 定期删除**相结合的方式：

**1. 惰性删除（Lazy Free）**
- 每次访问 key 时检查是否过期，过期则删除
- 优点：对 CPU 友好（只处理被访问的 key）
- 缺点：过期 key 长期不被访问会一直占用内存（内存泄漏风险）

**2. 定期删除**
- 每 100ms 随机抽取一批设置了过期时间的 key 检查删除
- 流程：从每个 DB 的过期字典中随机取 `active-expire-effort`（默认 1）% 的 key，删除其中过期的
- 优点：主动释放内存，控制内存上限
- 缺点：对 CPU 略有压力

**3. 内存淘汰策略（当内存达到 maxmemory 上限）**

| 策略 | 说明 |
|------|------|
| `noeviction` | 不淘汰，返回错误（默认） |
| `volatile-lru` | 从已设置过期时间的 key 中淘汰 LRU |
| `allkeys-lru` | 从所有 key 中淘汰 LRU（最常用） |
| `volatile-lfu` | 从已设置过期时间的 key 中淘汰 LFU |
| `allkeys-lfu` | 从所有 key 中淘汰 LFU |
| `volatile-random` | 随机淘汰已设置过期时间的 key |
| `allkeys-random` | 随机淘汰所有 key |
| `volatile-ttl` | 淘汰 TTL 最短的 key |

**实践建议：**
- 热点数据用 `allkeys-lru`，防止内存打满
- 用 `EXPIRE` 而非存储空值来标记"不存在"（避免大量 key 堆积）
- 字符串类型建议用 `SETEX` 一次性设置值和过期时间

## 题目四：Redis 主从复制原理是什么？如何处理故障转移？

### 答案

**主从复制流程：**

1. **建立连接**：从节点保存主节点地址，发送 `PING`
2. **权限验证**：如果主节点开启了 `requirepass`，从节点需发送 `AUTH`
3. **全量同步**：
   - 从节点发送 `PSYNC ? -1`（或 `PSYNC <runid> <offset>`）
   - 主节点执行 `BGSAVE`，生成 RDB 快照，同时将新写命令写入缓冲区
   - 主节点发送 RDB 文件给从节点
   - 从节点清空本地数据，加载 RDB
4. **增量同步**（命令传播）：
   - 主节点将后续写命令异步发送给从节点（`replication feed`）
   - 从节点执行这些命令，保持与主节点数据一致
5. **心跳检测**：
   - 从节点每 `repl-ping-replica-period`（默认 1s）发送心跳
   - 主节点检测到从节点超时则认为其宕机

**故障转移（Sentinel 哨兵模式）：**

| 步骤 | 说明 |
|------|------|
| 1. 主观下线 | 哨兵连续收到对主节点的 `down-after-milliseconds` 超时反馈 |
| 2. 客观下线 | 多个哨兵投票，超过 `quorum` 阈值认为主节点客观下线 |
| 3. 选举 leader | 哨兵们通过 Raft 算法选出一个 leader 负责故障转移 |
| 4. 选新主 | leader 从健康从节点中按优先级/同步进度/RunID 选新主 |
| 5. 通知 | 更新配置，发布 `+switch-master` 事件，客户端订阅重新连接 |
| 6. 原主恢复 | 原主作为从节点加入（若配置了 `down-after-milliseconds` 内的恢复） |

**常见问题处理：**
- **脑裂**（Split-Brain）：配置 `min-replicas-to-write` + `min-replicas-max-lag`，主节点写操作需等待足够从节点确认
- **复制风暴**：大量从节点同时全量同步会导致主节点 fork + 网络压力，建议用树形复制或适当减少从节点数量

## 题目五：Redis 缓存穿透、缓存击穿、缓存雪崩是什么？如何解决？

### 答案

这是面试中最常考的高并发场景题。

**① 缓存穿透（Cache Penetration）**

- **问题**：查询一个数据库和缓存都不存在的 key，每次都打到 DB
- **原因**：恶意攻击（大量不存在 key 的请求）或业务 bug
- **解决方案：**
  - **布隆过滤器（BloomFilter）**：将所有存在的 key 存入布隆过滤器，查询前先判断
  - **缓存空值**：即使 key 不存在，也缓存一个空值（`null`）或特殊标记，过期时间短一些（如 30s）
  - **参数校验**：对请求参数做基础校验，拦截明显恶意的 ID 格式

**② 缓存击穿（Cache Breakout）**

- **问题**：某个热点 key 过期瞬间，大量并发请求同时击穿到 DB
- **解决方案：**
  - **互斥锁（Mutex）**：`SETNX` 加锁，只允许一个请求回源，其他等待（Redisson 分布式锁）
  - **逻辑过期**：不给 key 设置实际过期时间，而是在 value 中存储"逻辑过期时间"，业务层判断并异步回源
  - **永不过期**：对热点 key 不设置过期时间，由后台任务主动更新

**③ 缓存雪崩（Cache Avalanche）**

- **问题**：大量 key 同时过期，或 Redis 宕机，导致大量请求打到 DB
- **解决方案：**
  - **过期时间随机化**：设置过期时间时加上随机值（`TTL = base + random(0, 3600)`）
  - **多级缓存**：本地缓存（Guava/Caffeine）+ Redis + DB，多层兜底
  - **Redis 高可用**：主从 + 哨兵 / Cluster，避免全量击穿
  - **服务熔级降级**：Hystrix / Sentinel 对 DB 访问限流，超过阈值返回兜底数据
  - **提前预热**：系统启动时主动加载热点数据到缓存

**总结对比：**

| 场景 | 本质 | 核心解决方案 |
|------|------|-------------|
| 穿透 | key 不存在 | 布隆过滤器 + 缓存空值 |
| 击穿 | 热点 key 过期 | 互斥锁 / 逻辑过期 |
| 雪崩 | 大量 key 同时过期 | 过期时间随机化 + 多级缓存 + 高可用 |
