## 1. HBase 的整体架构是怎样的？请描述 RegionServer 和 HMaster 的主要职责。

### 答案

HBase 采用典型的 Master-Slave 架构，主要由 **ZooKeeper**、**HMaster** 和 **RegionServer** 三类组件构成。

#### 核心组件

| 组件 | 职责 |
|------|------|
| **ZooKeeper** | 协调服务，负责 Master 选举、 元数据存储、 故障检测 |
| **HMaster** | 负责表的 CRUD操作、Region 分配、负载均衡、Schema 变更 |
| **RegionServer** | 负责数据读写、MemStore & HFile 管理、Compaction、Minor/Major GC |

#### 写入流程

1. 客户端请求 ZooKeeper，获取 `.META.` 表所在 RegionServer；
2. 客户端连接目标 RegionServer，数据先写入 **MemStore**（内存）；
3. MemStore 写满后 Flush 成 **HFile**（有序的 SSTable）；
4. 数据写入 WAL（预写日志），保证宕机可恢复。

#### 读取流程

1. 先查 MemStore（未持久化的写），再查 BlockCache（热点数据缓存）；
2. 未命中则扫描 HFile，利用 **BloomFilter** 快速定位文件；
3. 结果合并排序返回客户端。

#### 高可用

- HMaster 可以部署多个，通过 ZooKeeper 选举主节点；
- RegionServer 宕机后，其上的 Region 会被其他节点接管，**近实时恢复**。

> 📎 来源：[HBase 官方文档 - Architecture](https://hbase.apache.org/book.html#architecture)

## 2. HBase 中 RowKey 设计的原则有哪些？如何避免热点现象？

### 答案

RowKey 是 HBase 数据的"主键"，设计好坏直接影响读写性能和集群均衡。

### 四大设计原则

#### ① 长度原则
- RowKey 最大支持 **64KB**，建议越短越好（不超过 16 字节）；
- 原因：MemStore 排序、HFile 存储、网络传输都会放大数据量。

#### ② 散列原则
- 避免单调递增或递减的 RowKey（如时间戳开头），否则所有请求打到同一 RegionServer 导致**热点**。
- 常用散列方案：
  ```java
  // 方案1：盐值（Salt）
  // 在 RowKey 前加随机前缀：0_xxx, 1_xxx, 2_xxx ...
  
  // 方案2：Hash 散列
  // MD5(rowKey).substring(0, 5) + originalRowKey
  
  // 方案3：反转（Reversal）
  // 手机号反转 123456789 -> 987654321
  ```

#### ③ 唯一原则
- RowKey 必须全局唯一，重复会覆盖（put 操作语义）。

#### ④ 查询友好原则
- 根据查询场景决定 RowKey 字段顺序：
  ```java
  // 场景：查某用户的订单
  // 正确设计：userId_orderId  （前缀匹配）
  // 错误设计：orderId_userId  （无法利用字典序）
  ```

### 热点解决方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| 盐值前缀 | 简单，打散效果好 | 无法范围查询，增加复杂度 |
| Hash 散列 | 散列均匀 | 无法范围查询，Key 不可读 |
| 反转 | 简单，打散单调递减 Key | 可读性差，仅适合局部反转 |
| 时间戳在低位 | 保持时间范围查询能力 | 无法单独对抗热点 |

> 📎 来源：[HBase RowKey 设计指南 - CSDN](https://blog.csdn.net/gangliepi/article/details/106902644)

## 3. HBase 的存储结构是怎样的？请详细说明 MemStore 和 HFile 的工作原理。

### 答案

HBase 采用 **LSM-Tree**（Log-Structured Merge-Tree）存储结构，兼顾写性能和查询性能。

### 核心存储组件

```
MemStore (堆内内存)
    ↓ Flush
StoreFile (HFile)
    ↓ Compaction
更大的 StoreFile
```

#### MemStore（写缓存）

- 作用：接收客户端写入，先写入内存中的 MemStore，按 **RowKey + ColumnFamily + Column + Timestamp** 有序排列；
- Flush 触发条件（满足任一即触发）：
  - 单个 MemStore 大小超过 `hbase.hregion.memstore.flush.size`（默认 128MB）；
  - RegionServer 内存超过 `hbase.regionserver.global.memstore.size.lower.limit`；
  - WAL 日志超阈值，阻塞写入强制 Flush。
- 特点：**写放大**严重（数据写两次：WAL + MemStore），但保证高可靠。

#### HFile（SSTable）

- HBase 在 HDFS 上的实际存储格式，基于 **HFile v3**；
- 结构（自上而下）：
  ```
  [Data Block]      ← 数据块（KeyValue 对），可配置大小（默认 64KB）
  [Meta Block]     ← 元数据块（BloomFilter 等）
  [File Info]      ← 文件元信息（MaxSeqId、AvgKeyLen 等）
  [Trailer]        ← 尾部指针，定位各块位置
  ```
- 特点：**只读**，一旦写入不可更改，只能通过 Compaction 合并重写。

#### Compaction（合并压缩）

| 类型 | 说明 |
|------|------|
| **Minor Compaction** | 将多个小 HFile 合并成少量大 HFile，删除过期的 Cell（TTL） |
| **Major Compaction** | 将一个 ColumnFamily 下所有 HFile 合并成一个，删除所有过期/删除标记的数据 |

### LSM-Tree 读写流程图

```
写入:  Client → WAL → MemStore → (Flush) → HFile
                                    ↓
读取:  Client ← BlockCache ← MemStore ← HFile (扫描多个)
```

## 4. HBase 的 BloomFilter 是什么？它如何提升查询性能的？

### 答案

### 什么是 BloomFilter

BloomFilter 是由 **Burton Howard Bloom** 于 1970 年提出的概率数据结构，用于判断"**一个元素是否一定不存在或可能存在**"。

### HBase 中的应用

- HBase 在每个 HFile 的 **Meta Block** 中存储 BloomFilter 元数据；
- 当读取某个 RowKey 时，先用 BloomFilter 判断该 RowKey **一定不在** 哪个 HFile 中，跳过这些文件；
- 若 BloomFilter 返回"可能存在"，则必须打开该 HFile 进行扫描。

### 工作原理

```
原始数据:  ["apple", "banana", "cherry"]
       ↓
      哈希函数:  h1(key) → bit位 2
                 h2(key) → bit位 5
                 h3(key) → bit位 9
       ↓
Bit Array:  [0, 0, 1, 0, 0, 1, 0, 0, 0, 1, ...]
                              ↑  ↑        ↑
                            h1   h2       h3

查询 "apple": h1,h2,h3 对应位全为1 → "可能存在"
查询 "grape": h1或h2或h3对应位有0 → "一定不存在"  ← 快速跳过
```

### HBase BloomFilter 类型

| 类型 | 说明 |
|------|------|
| `ROW` | 按 RowKey 计算 BloomFilter，适合行级查询 |
| `ROWCOL` | 按 RowKey + Column 组合计算，适合列级查询（更精确但空间更大） |

### 性能收益

- 假设 HFile 数量为 N，跳过比例可达 **(N-1)/N**；
- 在大规模集群中（数十万 HFile），效果显著，可将随机读延迟降低 **50%-80%**；
- 代价：每个 HFile 增加约 **1%** 的存储空间。

### 注意事项

- BloomFilter 存在**假阳性**（可能误判为存在，但实际不存在），但**不会有假阴性**；
- 适用于读多写少、数据量大的场景。

> 📎 来源：[HBase BloomFilter 原理与实践 - InfoQ](https://www.infoq.cn/article/2021/03/hbase-bloomfilter-principle)

## 5. HBase 协处理器（Coprocessor）是什么？Observer 和 Endpoint 有什么区别？

### 答案

协处理器是 HBase 提供的**扩展机制**，允许用户将业务逻辑下沉到 RegionServer 节点上执行，无需频繁移动数据。

### 两种协处理器类型

#### ① Observer（观察者）—— 类似数据库触发器

类似于 RDBMS 的触发器，在特定事件前后自动执行回调。

| 类型 | 触发时机 | 典型场景 |
|------|---------|---------|
| **RegionObserver** | Get/Put/Delete/Scan 等操作前后 | 数据访问控制、列级权限、动态过滤 |
| **WALObserver** | WAL 写入前后 | WAL 审计 |
| **MasterObserver** | 表创建/删除/修改前后 | 审计日志、自动化 Schema 校验 |

**示例：实现列级权限控制**
```java
public class AuthCoprocessor extends RegionObserver {
    @Override
    public void preGet(final ObserverContext<RegionCoprocessorEnvironment> e,
                       final Get get, final List<Cell> result) throws IOException {
        // 检查用户权限，若无权限抛出 AccessDeniedException
        if (!hasPermission(get.getRow(), get.getColumnFamily())) {
            throw new AccessDeniedException("No permission to access this data");
        }
    }
}
```

#### ② Endpoint（端点）—— 类似数据库存储过程

自定义 RPC 协议，将计算逻辑分布到所有 RegionServer 并行执行，结果在客户端汇总。

**示例：自定义聚合（求最大值）**
```java
// 1. 定义协议接口
public interface MaxValueProtocol extends CoprocessorProtocol {
    long getMaxValue(byte[] columnFamily, byte[] qualifier) throws IOException;
}

// 2. 实现 Endpoint
public class MaxValueEndpoint implements RegionObserver, MaxValueProtocol {
    @Override
    public long getMaxValue(byte[] cf, byte[] cq) {
        // 在 RegionServer 内部扫描本地数据，返回最大值
        return scanLocalData(cf, cq);
    }
}

// 3. 客户端调用（自动汇聚结果）
String rowKey = ...;
AggregationClient ac = new AggregationClient(conf);
Long max = ac.max(TableName.valueOf("myTable"), 
                  new MaxValueColumnInterpreter(), scan);
```

### Observer vs Endpoint 对比

| 维度 | Observer | Endpoint |
|------|----------|----------|
| 执行位置 | 插入在读写流程中 | 独立 RPC 调用 |
| 触发方式 | 自动（事件驱动） | 客户端手动发起 |
| 数据流向 | 数据向代码靠拢 | 代码向数据靠拢 |
| 返回结果 | 通过修饰原操作 | 客户端汇总 |
| 典型用途 | 权限、过滤、审计 | 聚合、计算下推 |

### 注意事项

- 协处理器运行在 RegionServer 进程中，**bug 可能导致 RegionServer 崩溃**；
- 建议先在测试环境充分验证后再上线生产；
- HBase 2.x 对协处理器 API 有较大变更，需注意版本兼容性。

> 📎 来源：[HBase 协处理器官方文档](https://hbase.apache.org/book.html#cp)
