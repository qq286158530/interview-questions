## 题目 1：HBase 的数据模型是怎么组织的？列族（Column Family）有哪些特点？为什么需要列族？

### 参考答案

### HBase 数据模型层次结构

| 层次 | 说明 |
|------|------|
| **Namespace（命名空间）** | 相当于数据库的 Database，用于多租户隔离或业务划分 |
| **Table（表）** | 由一个或多个 Column Family 组成，需预先定义 Schema |
| **Row（行）** | 每行数据由 RowKey 唯一标识，行是 HBase 读写的基本单位 |
| **Column Family（列族）** | 多个列的逻辑分组，需在建表时定义，影响数据的物理存储结构 |
| **Column（列）** | 形如 `CF:Qualifier`，不需要在建表时定义，可动态添加 |
| **Cell（单元格）** | 由 RowKey + Column Family + Qualifier + Timestamp 唯一确定，存储 Value |
| **Version（版本）** | 每个 Cell 默认保留 3 个版本，通过 Timestamp 区分，可配置 |

### 列族的核心特点

1. **物理存储隔离**：同一个列族的所有的 KeyValue 在 HDFS 上存储在一起，不同列族数据存储在不同文件，有利于**按列族独立访问和压缩**。
2. **Schema 预定义**：列族需要在建表时声明，但列（Qualifier）可以随时动态添加。
3. **独立调优**：每个列族可以设置独立的**存储策略（TTL、压缩算法、数据块大小、缓存策略）**。
4. **数量控制**：建议一个表的列族**不超过 2~3 个**，列族过多会导致文件碎片化、Flush 和 Compaction 复杂度上升。

### 为什么需要列族？

- **I/O 优化**：查询时只读取需要的列族，减少不必要的数据读取。
- **生命周期管理**：不同类型的数据可以设置不同的 TTL 或压缩方式。
- **HDFS 友好**：列族数据对应独立的 HFile，便于配合 HDFS 副本机制做数据管理。

## 题目 2：描述 HBase 的一次完整写数据流程（写入路径）。

### 参考答案

HBase 采用 **LSM-Tree（Log-Structured Merge-Tree）** 结构，写入性能极高，完整流程如下：

### 写流程步骤分解

```
Client 请求
    │
    ▼
1. 【寻址】ZooKeeper → -ROOT- 表 → META 表 → 获取目标 Region 位置
    │
    ▼
2. 【定位】将 RegionServer 地址缓存到 Client 本地（减少ZK请求）
    │
    ▼
3. 【写 WAL】向 RegionServer 写入 WAL（Write-Ahead Log）
   - WAL 顺序写入 HDFS，保证服务器崩溃后可恢复
    │
    ▼
4. 【写 MemStore】数据写入 MemStore（内存）
   - 同时更新 MemStore 和 WAL 才算写入成功
    │
    ▼
5. 【返回客户端】写入 MemStore 成功后，立即返回 "写入成功"
    │
    ▼
6. 【后台异步 Flush】MemStore 达到阈值（默认 128MB）→ 刷写成 HFile
    │
    ▼
7. 【后台 Compaction】HFile 数量过多 → 触发 Minor/Major Compaction 合并
```

### 关键点说明

- **WAL 作用**：防止 MemStore 数据未刷写时 RegionServer 宕机导致的数据丢失。重启后通过回放 WAL 恢复数据。
- **MemStore 刷写时机**：
  - 达到 `hbase.hregion.memstore.flush.size`（默认 128MB）
  - MemStore 总大小超过 `heap * hbase.regionserver.global.memstore.size`（默认 40% Heap）
  - 写入阻塞时间超过 `hbase.regionserver.global.memstore.size.lower.limit`
- **写入是分布式的**：多个 RegionServer 可并行写入，热点 RowKey 会导致写入集中在某一台机器。

> **来源：** [CSDN《2025大数据面试：HBase高频30题》](https://blog.csdn.net/2501_91473495/article/details/153786841)

## 题目 3：HBase 的读数据流程是怎样的？为什么说读性能比写性能差？

### 参考答案

### 读流程步骤分解

```
Client 请求
    │
    ▼
1. 【寻址】同写流程，通过 ZooKeeper → META 表定位 RegionServer
    │
    ▼
2. 【定位 Region】在 META 表中查到 RowKey 所在的 Region 信息
    │
    ▼
3. 【MemStore 检查】先查 MemStore（内存）——最新写入的数据还在 MemStore 中
    │
    ▼
4. 【BlockCache 检查】再查 BlockCache（读缓存）——热点数据已被缓存
    │
    ▼
5. 【HFile 检查】最后到 HFile 中查找（利用 HFile 索引 + BloomFilter 定位）
    │
    ▼
6. 【合并返回】将各数据源的结果按 Timestamp 合并，返回最新版本给 Client
```

### 读性能为何相对较差

1. **多数据源合并**：HBase 读数据需要同时扫描 MemStore、BlockCache 和 HFile，再按时间戳合并结果，不像写入只需操作内存。
2. **LSM-Tree 的读放大（Read Amplification）**：最新数据在 MemStore，较老数据在多层 HFile 中，最坏情况需扫描多个 HFile。
3. **磁盘 I/O**：数据不在缓存时，需要从 HDFS 读取 HFile，涉及网络和磁盘 I/O。
4. **BloomFilter 优化**：HBase 在 HFile 中使用 BloomFilter 快速判断某 RowKey 是否在该文件中，减少无效 I/O。

### 优化手段

- **读缓存（BlockCache）**：将热点 HFile 数据块缓存在 RegionServer 堆内存中。
- **布隆过滤器（BloomFilter）**：快速定位目标 RowKey 所在的 HFile，避免全量扫描。
- **短路读（Short-Circuit Read）**：绕过网络直接从本地 HDFS 读取数据。
- **Bulk Load**：大批量导入时跳过 WAL 和 MemStore，直接生成 HFile 写入 HDFS，绕过读放大。

## 题目 4：HBase 的 RowKey 如何设计才能避免热点（Hotspotting）？有哪些常见技巧？

### 参考答案

热点问题是指**大量读写请求集中在某一个或少数几个 RegionServer 上**，导致集群负载不均、部分节点压力过大。

### 热点产生的原因

- RowKey 按字典顺序组织，若 RowKey 高位是时间戳或有序递增的数值，则所有新数据都会写入同一个 Region。
- 顺序写入导致数据全部落在同一个 Region 的末尾。

### 解决热点的核心思路

> **核心原则：将数据打散，让写入均匀分布到所有 Region。**

### 常见设计技巧

#### 1. 加盐（Salting）
- 在 RowKey 高位添加**随机前缀**（0~N 的数字），使数据分散到不同 Region。
- 优点：均匀分布；缺点：无法范围查询，加密后 RowKey 长度增加。
```
原始 RowKey:  20260715001_xxx
盐后 RowKey:  3_20260715001_xxx
```

#### 2. 哈希（Hashing）
- 对 RowKey 取 Hash 值，取其前几位作为前缀，保证**确定性的分散**。
- 优点：分布均匀、可复现；缺点：相同前缀的数据可能不在同一批次内。
```
RowKey: MD5(rowKey).substring(0,4) + originalRowKey
```

#### 3. 反转（Reversing）
- 将固定前缀部分的字符串反转，使原本有规律的前缀变得无序。
- 适用于手机号、身份证号等有规律前缀的字段。
```
原始: 13800138000_order
反转: 00083001831_order
```

#### 4. 随机数（Randomizing）
- 在 RowKey 末尾添加随机数，打散数据。
- 适用于**写多读少**场景，读性能会下降。

#### 5. 时间戳反转 + 业务字段组合
- 将时间戳放在 RowKey 低位，高位放业务标识，使时间上有序的同时保证业务字段散列。
```
业务ID（散列）+ 时间戳（递增，低位）
例如: userId_hash + Long.MAX_VALUE - timestamp
```

### 综合设计建议

- **高频查询字段放在 RowKey 高位**，兼顾读性能。
- **结合加盐和业务字段**，在高并发写入场景下效果最佳。
- 避免使用单调递增的数字作为 RowKey 高位。

## 题目 5：HBase 如何实现高可用（HA）？Master 宕机后会发生什么？

### 参考答案

### HBase 高可用保障机制

#### 1. ZooKeeper 协调与 Master election
- ZooKeeper 负责 **HMaster 选举**：通过临时节点和乐观锁机制，保证**只有一个 HMaster 处于 Active 状态**。
- 其他 HMaster 处于 Standby，随时准备接替。

#### 2. RegionServer 心跳与故障恢复
- RegionServer 定期向 ZooKeeper 发送心跳（默认 3 秒一次）。
- 若 ZooKeeper 在 `zookeeper.session.timeout`（默认 90 秒）内未收到心跳，则判定该 RegionServer 宕机。
- **宕机后**，HMaster 自动将故障 RegionServer 上的 Region 重新分配（Assignment）到其他存活节点。
- 宕机节点的 **WAL** 会被其他 RegionServer 分割回放（Split + Replay），恢复未刷写到 HFile 的数据。

#### 3. 数据副本（HDFS 副本机制）
- HBase 所有数据存储在 HDFS 上，HDFS 默认 3 副本策略保证**数据层面的高可用**。
- 即使某个 DataNode 宕机，HBase 仍能从其他副本读取数据。

#### 4. META 表和 ZooKeeper 元数据保障
- `-ROOT-` 表和 `META` 表的 Region 位置信息存储在 ZooKeeper 中。
- 即使 HMaster 宕机，客户端仍能通过 ZooKeeper 定位数据，读写不受影响。

### Master 宕机后会发生什么？

| 场景 | 后果 |
|------|------|
| **读写请求** | **不受影响**（客户端直接访问 RegionServer，不依赖 HMaster） |
| **建表/删表** | **暂停**，需等新 Master 选举完成后恢复 |
| **Region 迁移** | **暂停**，故障 Region 暂时不可用 |
| **负载均衡** | **暂停**，新 Master 上任后自动恢复 |
| **故障恢复时长** | 通常在 30s~120s 内完成选举和数据恢复 |

### 补充：读写分离与二级索引的高可用

- **读请求**：可走 Zookeeper 定位 META → RegionServer，不依赖 Master。
- **写请求**：同理，依赖 WAL 和 MemStore，不走 Master。
- **二级索引（如 Phoenix 索引）**：索引表和主表同样受 HBase HA 机制保护，Region 故障时索引也会跟随迁移。
