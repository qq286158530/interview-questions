## 题目一：HBase 的架构是怎样的？核心组件有哪些？

### 参考答案

HBase 是一个基于 HDFS 的列族（Column-Family）NoSQL 数据库，采用 **Master-Slave 架构**，核心组件包括：

| 组件 | 角色 | 说明 |
|------|------|------|
| **HMaster** | Master 节点 | 负责表的元数据管理、Region 分配、负载均衡、故障恢复 |
| **HRegionServer** | Slave 节点 | 负责数据读写、MemStore/StoreFile 管理、与 HDFS 交互 |
| **ZooKeeper** | 协调服务 | 保存元数据（.META. 表位置）、Master 心跳、分布式锁 |
| **WAL（Write-Ahead Log）** | 预写日志 | 数据写入 MemStore 前先写入 WAL，用于故障恢复 |
| **BlockCache** | 读缓存 | LRU 缓存热点数据块 |
| **MemStore** | 写缓存 | 数据按列族组织，先写入内存（有序），定期刷写成 HFile |
| **HFile** | 数据文件 | HBase 在 HDFS 上的实际存储格式，支持有序查找 |

**写入流程简述：**
```
Client → ZooKeeper → 找到 .META. 表 → 找到对应 RegionServer
→ 写入 WAL → 写入 MemStore → 返回 ACK
```

**刷盘/合并机制：**
- MemStore 达到阈值（默认 128MB）× 数量（默认 4）× 时间（默认 1h）→ 刷写成 HFile
- Minor Compaction：合并少量 HFile
- Major Compaction：合并一个列族下所有 HFile，删除标记数据

**分裂（Split）机制：**
当一个 Region 大小超过阈值（默认 10GB）时，HRegionServer 会将其分裂为两个子 Region，HMaster 感知后更新元数据。

**数据写入不依赖 Master：**
客户端通过 ZooKeeper 获取元数据后，直接与 HRegionServer 通信，Master 只负责管理，不参与数据读写。

## 题目二：HBase 的 RowKey 设计原则有哪些？如何优化？

### 参考答案

RowKey 是 HBase 中数据检索的唯一主键，设计质量直接影响读写性能。

### 核心原则

1. **长度适中**
   - 建议 16~100 字节，建议不超过 200 字节
   - RowKey 会作为 HFile 的一部分存储，过长浪费磁盘与缓存

2. **唯一性**
   - RowKey 必须唯一对应一行数据，类似关系型数据库的主键

3. **散列性（避免热点）**
   - 避免使用单调递增/递减的 RowKey（如时间戳 1,2,3...），否则所有写入集中在一个 RegionServer 上，造成热点
   - **方法：加盐（Salt）**：在 RowKey 前加随机前缀（如 MD5 的前几位）
   - **方法：哈希（Hash）**：对原始 Key 做 MD5/SHA 取 hash 值打散
   - **方法：反转（Reverse）**：如手机号反转 `8613812345678 → 7654321881368`

4. **查询模式匹配**
   - HBase 按 RowKey 字典序存储，**前缀查询高效**
   - 如果按时间范围查询，`[startRow, stopRow)` 设计要靠前
   - 示例：监控场景可用 `deviceId + reverse(timestamp)` 使时间均匀分布

5. **尽量短**
   - 同一列族下的所有 Key-Value 对共享同一个 RowKey，过长的 RowKey 造成存储膨胀

### 常见设计模式

| 场景 | RowKey 设计 | 说明 |
|------|------------|------|
| 用户行为序列 | `userId + actionType + timestamp` | 支持按用户+行为类型范围查询 |
| 消息系统 | `receiverId + timestamp` | 支持查询某用户收到的消息 |
| 时序数据 | `metricId + reverse(timestamp)` | 打散时间戳避免热点 |
| 多租户 | `tenantId + entityId` | 天然隔离不同租户数据 |

## 题目三：HBase 的存储结构是什么？HFile 和 MemStore 的关系？

### 参考答案

### HBase 存储层次

```
Table (表)
└── Column Family (列族)
    └── Region (分区)
        └── Store (每个列族对应一个 Store)
            ├── MemStore（写缓存，内存有序）
            └── StoreFile（HFile，磁盘有序）
                └── HFile（具体数据文件）
```

### HFile 内部结构

HFile 是 HBase 在 HDFS 上的实际存储格式，结构如下：

```
┌─────────────────────────────────────────┐
│  Data Block Index（数据块索引）           │  ← 定位某个 RowKey 在哪个 Block
├─────────────────────────────────────────┤
│  File Info（文件信息：KEYVALUE键值对数量） │
├─────────────────────────────────────────┤
│  Bloom Filter（布隆过滤器）               │  ← 快速判断某个 Key 是否存在
├─────────────────────────────────────────┤
│  Data Block（数据块，默认 64KB/128KB）    │  ← 实际 KV 数据存储
│  ┌──────────────────────────────────┐   │
│  │ KeyValue: RowKey/CF/Column/Value  │   │  按列族分开存储
│  │ KeyValue: RowKey/CF/Column/Value  │   │  同一 RowKey 的不同列可分散
│  └──────────────────────────────────┘   │
├─────────────────────────────────────────┤
│  Meta Block（时间戳、版本数等元信息）      │
├─────────────────────────────────────────┤
│  File Traile（文件尾，标识数据块偏移量）   │
└─────────────────────────────────────────┘
```

### MemStore 与 HFile 的关系

| 维度 | MemStore | HFile |
|------|----------|-------|
| 存储位置 | 内存 | HDFS 磁盘 |
| 数据结构 | 有序跳表（SortedMap） | 有序数据块 |
| 写入 | 先写 MemStore，内存排序 | 由 MemStore 刷盘生成 |
| 作用 | 写缓冲、合并写 IO | 持久化存储、读优化 |
| 刷盘条件 | 达到 `hbase.hregion.memstore.flush.size`（默认128M）或其他触发条件 | 由 MemStore 刷盘生成 |
| 读取 | 先查 MemStore，再查 HFile | 通过 BlockCache 缓存热点块 |

### 读写流程

**写入流程：**
1. 数据先写入 WAL（预写日志）
2. 数据写入 MemStore（按 RowKey 排序）
3. 当 MemStore 达到阈值，刷写成 HFile（Minor Compaction 或直接刷盘）
4. HFile 数量过多时触发 Compaction 合并

**读取流程：**
1. 先查 BlockCache（读缓存）
2. 未命中则查 MemStore
3. 再按顺序扫描所有 HFile（结合 Bloom Filter 加速）
4. 合并各数据源结果返回

**删除机制：**
HBase 删除数据时并不立即删除，而是**先打标记（Delete Marker）**，在 Major Compaction 时才真正删除。

## 题目四：什么是 HBase 协处理器（Coprocessor）？有哪些类型？使用场景？

### 参考答案

### 基本概念

协处理器是 HBase 提供的一种**扩展机制**，允许用户在 RegionServer 上运行自定义代码，拦截 HBase 的读写操作，就地处理数据，而无需把数据全部返回给客户端再处理。

### 两种类型

#### 1. Observer Coprocessor（观察者，类比触发器）

类似关系型数据库的触发器（Trigger），在特定事件前后执行钩子逻辑。

**可用钩子：**

| 钩子 | 时机 |
|------|------|
| `preOpen` | Region 打开前 |
| `postOpen` | Region 打开后 |
| `preGet` / `postGet` | GET 请求前后 |
| `prePut` / `postPut` | PUT 请求前后 |
| `preDelete` / `postDelete` | DELETE 请求前后 |
| `preBulkLoadHFile` | BulkLoad 批量加载前 |
| `preCompact` / `postCompact` | Compaction 前后 |

**使用场景：**
- **二级索引**：写入数据时自动同步写入索引表
- **数据校验**：写入前校验字段合法性
- **审计日志**：记录所有写操作到日志表
- **权限控制**：细粒度行级/列级权限控制

#### 2. Endpoint Coprocessor（端点，类比存储过程）

类似关系型数据库的存储过程，允许在 RegionServer 上执行服务端计算，客户端可以调用并获取聚合结果。

**使用场景：**
- `AggregationCoprocessor`：服务端求 SUM/AVG/MIN/MAX 等聚合
- 自定义服务端计算逻辑，如跨行跨列联合查询

**调用方式（示例）：**
```java
// 通过 HTable 调用 Endpoint
AggregationClient client = new AggregationClient(conf);
long sum = client.sum(tablename, new LongSumQualifier());
```

### 配置方式

**方式一：表级配置（通过 HBase Shell）**
```ruby
create 'mytable', 'cf', 'coprocessor'=>'1|org.apache.hadoop.hbase.coprocessor.AggregateImplementation|1073741824|arg1=1,arg2=2'
```

**方式二：全局配置（hbase-site.xml）**
```xml
<property>
  <name>hbase.coprocessor.user.region.classes</name>
  <value>com.example.SecondaryIndex,com.example.AuditLogger</value>
</property>
```

### 注意事项

- 协处理器运行在 RegionServer 上，**错误代码可能拖垮整个 RegionServer**
- 需要做好异常捕获和超时控制
- Observer 之间可以组成链（Chain），按优先级顺序执行
- Endpoint 需要客户端主动调用，不像 Observer 自动触发

## 题目五：HBase 如何保证数据一致性和高可用？与关系型数据库有何区别？

### 参考答案

### 数据一致性

HBase 的一致性模型是 **最终一致性（Eventual Consistency）**，但支持配置为强一致：

| 一致性级别 | 实现方式 |
|-----------|---------|
| **最终一致（默认）** | 写入 WAL + MemStore，异步刷盘，副本间异步复制 |
| **强一致性** | 配置 `hbase.wal.provider=filesystem`，开启 WAL 同步写 |
| **单行原子性** | HBase 保证单行 PUT/DELETE 的原子性，不存在部分写入 |

> 注意：HBase **不支持跨行跨列的事务（ACID）**，这是与关系型数据库的本质区别之一。

### 高可用机制

1. **Region 自动恢复**
   - 当某个 HRegionServer 宕机，HMaster 会将该节点上所有 Region 自动分配到其他节点（通常 5~10 分钟内完成）
   - 依赖 ZooKeeper 感知节点心跳超时（默认 3× zookeeper.session.timeout）

2. **WAL 故障恢复**
   - 写入数据先写 WAL，宕机重启后通过 WAL 重放（Replay）恢复 MemStore 中未刷盘的数据

3. **HDFS 副本机制**
   - HBase 数据存储在 HDFS 上，默认 3 副本，NameNode 自动保证副本数
   - 任何副本所在节点宕机，HDFS 自动复制补齐副本数

4. **META 表与 ZooKeeper**
   - 元数据（表在哪个 Region、Region 在哪个 RS 上）存储在 ZooKeeper 和 `.META.` 表中
   - 即使 Master 宕机，已写入的数据仍可正常读写，只有元数据变更（如建表、分裂）受影响

### 与关系型数据库（RDBMS）对比

| 维度 | HBase | MySQL/PostgreSQL |
|------|-------|-----------------|
| **CAP 定理** | CP（A 可配置但牺牲性能） | CA（通常） |
| **事务支持** | 单行原子性，无跨行事务 | ACID 完整事务 |
| **一致性** | 最终一致（可配置强一致） | 强一致 |
| **扩展性** | PB 级水平扩展 | 垂直扩展为主 |
| **查询能力** | 主键/前缀查询，多条件查询需借助二级索引或 Phoenix | 丰富 SQL，任意条件联合查询 |
| **数据结构** | 列族 NoSQL，Schema 灵活 | 固定 Schema |
| **JOIN 支持** | 不支持，需自行实现 | 原生支持 |
| **典型场景** | 写入密集型、时序数据、海量存储 | 事务密集型、复杂查询 |

### 高可用配置建议

- 生产环境 ZooKeeper 节点数 ≥ 3，奇数个
- HDFS 副本数 = 3
- WAL 同步写入（`hbase.wal.sync.method = fsync`）适用于对数据丢失零容忍的场景
- 合理设置 Split 和 Major Compaction 时间窗口，避免在业务高峰期触发
