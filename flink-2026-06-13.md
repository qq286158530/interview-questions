## 题目一：Flink 的背压（Back Pressure）机制是如何工作的？如何定位和解决背压问题？

### 参考答案

**背压（Back Pressure）** 是流处理系统中常见的问题，当下游算子的处理速度跟不上上游数据的发送速度时，就会产生背压，导致数据在上游缓冲区中堆积。

**Flink 背压的工作原理：**

Flink 采用 **TCP 协议 + 缓冲池** 实现背压传播：

1. **Credit-Based Flow Control（基于信用的流量控制）**
   - 下游算子消费数据时，会给上游发送 "credit"（信用），表示自己有多大的处理能力
   - 上游只有收到 credit 后才会发送数据，避免缓冲区溢出

2. **BufferPool 机制**
   - 每个 TaskManager 为每个 Channel 维护一定数量的缓冲区
   - 当下游处理慢 → 本地缓冲区满 → 通知上游减慢发送速度
   - 背压沿着数据流方向向上游传播

3. **反压传播链**
   ```
   Sink（慢） ← Buffer 满 ← Operator ← Buffer 满 ← Source
   ```

**定位背压问题的方法：**

1. **Flink Web UI 查看**
   - TaskManager → Task → BackPressure 标签页
   - 显示各 Subtask 的背压状态（OK / HIGH / LOW）
   - 背压比例 > 0.1 说明存在明显背压

2. **使用 `flink` 命令行工具**
   ```bash
   flink list # 查看运行中的 Job
   flink backpressure [jobId] # 查看 Job 的背压统计
   ```

3. **Metrics 指标监控**
   - `checkpoint.input_rate` / `checkpoint.output_rate`
   - `queueLength`（队列长度）
   - `busyTimeMsPerSecond`（忙碌时间占比）

**解决背压的常用手段：**

| 策略 | 说明 |
|------|------|
| 增加并行度 | 加快下游处理能力 |
| 增加 TaskManager Slot | 分配更多资源 |
| 优化算子逻辑 | 减少单条数据处理耗时 |
| 启用 RocksDB 状态后端 | 减少状态读写瓶颈 |
| 调整 Buffer 超时 | `env.setBufferTimeout` |
| 业务层面削峰 | 上游加缓冲或限流 |

📎 **来源：** [Apache Flink 官方文档 - 背压机制](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/monitoring/back_pressure/)

## 题目二：Flink SQL 与 Table API 的核心概念是什么？如何实现统一的流批处理？

### 参考答案

**Flink SQL 和 Table API** 是 Flink 提供的高级抽象层，提供了关系型编程接口，支持 SQL 查询和关系型变换。

**核心概念：**

1. **Dynamic Table（动态表）**
   - Flink 将流（Stream）看作不断变化的表（Dynamic Table）
   - 流上的数据等价于表中不断 INSERT 的行
   - 查询结果也是不断变化的 Dynamic Table

2. **流与表的等价转换**
   ```
   Stream → Table（流转表，DataStream 转为 Table）
   Table → Stream（表转流，将查询结果写回流）
   ```

3. **连续查询（Continuous Query）**
   - 动态表的 SQL 查询是"连续执行"的
   - 新数据到来时自动触发增量计算
   - 等价于持续运行的 Flink DataStream 作业

**统一流批处理的原理：**

Flink 通过 **Table API & SQL** 实现真正的 unified processing：

```java
// 同一套代码，既可以处理流也可以处理批
TableEnvironment tEnv = TableEnvironment.create(EnvironmentSettings.inStreamingMode());
// 或批处理模式
TableEnvironment tEnv = TableEnvironment.create(EnvironmentSettings.inBatchMode());

// 注册表（可以直接从 DataStream 或 DataSet 注册）
tEnv.createTemporaryView("Orders", ds, Schema.);

// SQL 查询
Table result = tEnv.sqlQuery("SELECT product, COUNT(*), SUM(amount) FROM Orders GROUP BY product");

// 写回流或批目标
result.execute().insertInto("OutputTable");
```

**支持的 Connector：**

- Kafka、Pulsar（流式输入/输出）
- MySQL、PostgreSQL（维表 join）
- FileSystem（批式读写）
- Elasticsearch、Druid（OLAP 查询）

**Table SQL 与 DataStream 的互转：**

```java
// DataStream → Table
Table table = tEnv.fromDataStream(stream, $("id"), $("name"), $("ts"));

// Table → DataStream（撤回模式）
DataStream<Row> stream = tEnv.toDataStream(table).returns(Types.STRING);
```

📎 **来源：** [Apache Flink 官方文档 - Table API & SQL](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/table/overview/)

## 题目三：Flink 与 Kafka 如何集成？如何实现端到端的 Exactly-Once？

### 参考答案

**Flink 与 Kafka 的集成方式：**

Flink 提供了 **Kafka Connector**，支持从 Kafka 读取数据（Source）和向 Kafka 写入数据（Sink）。

**1. Kafka Source（消费 Kafka）**

```java
KafkaSource<String> kafkaSource = KafkaSource.<String>builder()
    .setBootstrapServers("localhost:9092")
    .setGroupId("my-consumer-group")
    .setTopics("input-topic")
    .setValueOnlyDeserializer(new SimpleStringDeserializer())
    .setStartingOffsets(OffsetsInitializer.latest())
    .build();

DataStream<String> stream = env.fromSource(
    kafkaSource,
    WatermarkStrategy.forBoundedOutOfOrderness(Duration.ofSeconds(30)),
    "KafkaSource"
);
```

**2. Kafka Sink（写入 Kafka）**

```java
KafkaSink<String> kafkaSink = KafkaSink.<String>builder()
    .setBootstrapServers("localhost:9092")
    .setRecordSerializer(new SimpleStringSerializer("output-topic"))
    .setDeliveryGuarantee(DeliveryGuarantee.EXACTLY_ONCE)
    .setTransactionalIdPrefix("flink-")
    .build();

stream.sinkTo(kafkaSink);
```

**3. 端到端 Exactly-Once 实现要点：**

- **Source 端**：使用 `read_committed` 隔离级别，Flink Checkpoint 时保存 Kafka offset
- **内部处理**：Flink Checkpoint 保证算子状态 Exactly-Once
- **Sink 端**：Kafka Sink 的 `DeliveryGuarantee.EXACTLY_ONCE` 使用两阶段提交

**Exactly-Once 配置清单：**

```java
// 环境级别开启 Checkpoint
env.enableCheckpointing(30000, CheckpointingMode.EXACTLY_ONCE);
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(5000);
env.getCheckpointConfig().setCheckpointTimeout(60000);

// Kafka Source Exactly-Once
.setProperty("isolation.level", "read_committed")

// Kafka Sink Exactly-Once（开启两阶段提交）
.setDeliveryGuarantee(DeliveryGuarantee.EXACTLY_ONCE)
```

**Kafka 版本兼容说明：**

| Kafka 版本 | 推荐 Flink Kafka Connector 版本 |
|-----------|-------------------------------|
| Kafka 2.4+ | flink-connector-kafka 1.13+ |
| Kafka 3.x | flink-connector-kafka 1.15+ |

📎 **来源：** [Apache Flink Kafka Connector 官方文档](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/connectors/datastream/kafka/)

## 题目四：Flink 有哪几种时间语义？各自的适用场景是什么？

### 参考答案

Flink 支持三种时间语义，是流处理中区分不同业务场景的核心概念。

**三种时间语义：**

| 时间语义 | 说明 | 数据来源 | 适用场景 |
|---------|------|---------|---------|
| **Processing Time** | 处理时间，数据被处理时的机器时间 | `System.currentTimeMillis()` | 实时监控、对延迟要求高、不关心数据乱序 |
| **Event Time** | 事件时间，数据自带的时间戳字段 | 数据中的 timestamp 字段 | 需要精确统计、数据源存在乱序、需要历史数据分析 |
| **Ingestion Time** | 摄入时间，数据进入 Flink Source 的时间 | Source 算子接收时的时间 | 介于以上两者之间，不需要 watermark，但仍有乱序风险 |

**Event Time 详解：**

```java
// 设置时间语义
env.setStreamTimeCharacteristic(TimeCharacteristic.EventTime);

// 提取时间戳 + Watermark
DataStream<Event> stream = env
    .assignTimestampsAndWatermarks(
        WatermarkStrategy
            .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(10))
            .withTimestampAssigner((event, timestamp) -> event.getTimestamp())
    );
```

**为什么需要 Watermark（针对 Event Time）：**

- 数据自带 timestamp，但到达顺序可能乱序
- Flink 需要 Watermark 判断"窗口的数据是否到齐"
- `forBoundedOutOfOrderness(Duration.ofSeconds(10))` 表示最多容忍 10 秒乱序

**三种语义的对比：**

```
Event Time:     数据自带时间戳 → 需要 Watermark → 结果精确但延迟较高
                [1:00] [1:01] [0:59] [1:02] → 按时间戳排序 → [0:59][1:00][1:01][1:02]

Ingestion Time: 进入 Source 时打时间戳 → 不需要 Watermark → 延迟中等
                [1:00:03] 收到 → [1:00:05] 收到 → 按摄入顺序处理

Processing Time: 处理时取系统时间 → 无需特殊处理 → 延迟最低但不精确
                1:00:03 到达 → 1:00:05 处理 → 直接处理
```

**最佳实践建议：**

- **实时监控/告警**：Processing Time（延迟优先）
- **精确统计/报表**：Event Time + Watermark（准确性优先）
- **金融交易/订单**：Event Time（数据完整性优先）
- **跨时区数据处理**：Event Time（不受服务器时区影响）

📎 **来源：** [Apache Flink 官方文档 - Time and Watermarks](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/time-and-watermarks/)

## 题目五：Flink CDC（Change Data Capture）是什么？如何实现数据库变更的实时流式捕获？

### 参考答案

**CDC（Change Data Capture，变更数据捕获）** 是一种捕获数据库变更的技术，Flink CDC 可以将 MySQL、PostgreSQL、MongoDB 等数据库的增量数据以流的方式实时接入 Flink。

**Flink CDC 的工作原理：**

Flink CDC 基于 **Debezium**（开源的分布式平台，用于捕获数据库变更），将数据库的 binlog（MySQL）或 WAL（PostgreSQL）解析为流式数据。

**核心流程：**
```
MySQL → Binlog → Debezium → Flink CDC Connector → Flink DataStream
```

**快速使用示例：**

```java
// 添加 Maven 依赖
// flinkcdc 3.0+ 基于 Flink 1.18+

MySqlSource<String> mySqlSource = MySqlSource.<String>builder()
    .hostname("localhost")
    .port(3306)
    .databaseList("mydb")       // 监控的数据库
    .tableList("mydb.orders")  // 监控的表
    .username("flink")
    .password("flink")
    .deserializer(new JsonDeserializer())  // JSON 格式输出
    .startupOptions(StartupOptions.initial())  // 首次全量 + 增量
    .build();

DataStream<String> cdcStream = env.fromSource(
    mySqlSource,
    WatermarkStrategy.noWatermarks(),
    "MySQL CDC Source"
);
```

**CDC 数据格式示例：**

```json
{
  "before": {"id": 1, "amount": 100},
  "after":  {"id": 1, "amount": 150},
  "op": "u",      // 操作类型：c=INSERT, u=UPDATE, d=DELETE, r=READ（全量快照）
  "ts_ms": 1677123456789
}
```

**Flink CDC 的优势：**

| 特性 | 说明 |
|------|------|
| 全量 + 增量一体化 | 首次启动自动全量同步，后续增量监听 binlog |
| 实时性 | 毫秒级延迟 |
| 无侵入 | 不改业务代码，不影响源库 |
| Exactly-Once | 通过 Checkpoint 保证 |
| Schema Evolution | 支持 DDL 变更同步 |

**全量快照 + 增量集成（Debezium Pipeline）：**

```java
// 从某个时间点开始增量（自动先全量再增量）
.startupOptions(StartupOptions.specificOffset("mysql-bin.000001", 4))

// 仅做全量同步（只读历史数据）
.startupOptions(StartupOptions.initial())
```

**常见 CDC 场景：**

1. **数据湖入湖**：MySQL → Kafka → Flink CDC → Iceberg/Hudi
2. **实时数仓**：业务库实时同步到数仓
3. **多源 join**：将多表 CDC 数据实时关联

📎 **来源：** [Flink CDC 官方文档](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/connectors/datastream/cdc/overview/)
