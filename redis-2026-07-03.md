## 题目一：Redis 数据类型有哪些？底层是如何实现的？

### 答案

Redis 提供了丰富的数据类型，常见的有 **5 种基本数据类型**和**4 种扩展数据类型**。

### 五大基本数据类型及应用场景

| 类型 | 应用场景 |
|------|----------|
| **String（字符串）** | 缓存对象、常规计数、分布式锁、共享 session 信息等 |
| **List（列表）** | 消息队列（需自行实现全局唯一 ID，不支持消费组形式消费） |
| **Hash（哈希）** | 缓存对象、购物车等 |
| **Set（集合）** | 聚合计算（并集、交集、差集），如点赞、共同关注、抽奖活动等 |
| **Zset（有序集合）** | 排序场景，如排行榜、电话和姓名排序等 |

### 四大扩展数据类型

| 类型 | 版本 | 应用场景 |
|------|------|----------|
| **BitMap（位图）** | 2.2+ | 二值状态统计，如签到、判断用户登录状态、连续签到用户总数等 |
| **HyperLogLog** | 2.8+ | 海量数据基数统计，如百万级网页 UV 计数等 |
| **GEO（地理信息）** | 3.2+ | 存储地理位置信息，如滴滴叫车等 |
| **Stream（流）** | 5.0+ | 消息队列，支持自动生成全局唯一消息 ID，支持消费组形式消费数据 |

### 底层数据结构实现

Redis 对象的底层数据结构随着版本更新有所变化，核心数据结构如下：

- **SDS（简单动态字符串）**：String 类型的底层实现，相比 C 字符串具备 O(1) 获取长度、二进制安全、防止缓冲区溢出、节省内存等优势。
- **双向链表（LinkedList）**：早期 List 类型的底层实现之一，Redis 3.2 后被 quicklist 替代。
- **压缩列表（ziplist）**：List/Hash/Zset 在元素较少时的底层实现，内存紧凑，但存在连锁更新问题，Redis 7.0 已废弃，由 listpack 替代。
- **哈希表（hashtable）**：Hash/Set 类型的底层实现之一，采用链式哈希解决冲突，支持渐进式 rehash。
- **跳表（skiplist）**：Zset 的底层实现之一，支持平均 O(logN) 查找，同时维护元素权重和成员。
- **整数集合（intset）**：Set 类型在元素全为整数且数量较少时的底层实现，支持类型升级以节省内存。
- **quicklist**：Redis 3.2 后 List 的底层实现，结合了双向链表和压缩列表的优点。
- **listpack**：Redis 5.0 引入，替代压缩列表，解决连锁更新问题。

> **参考来源**：https://xiaolincoding.com/redis/base/redis_interview.html

## 题目二：Redis 持久化方式有哪几种？各有何优缺点？

### 答案

Redis 共有三种数据持久化方式：**AOF 日志、RDB 快照、混合持久化**。

### 1. AOF 日志（Append Only File）

**原理**：每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里，Redis 重启时读取该文件并逐一执行命令来恢复数据。

**写回策略**（通过 `appendfsync` 配置）：

| 策略 | 含义 | 优点 | 缺点 |
|------|------|------|------|
| **Always** | 每次写命令执行完后同步写回硬盘 | 数据不丢失 | IO 开销大，性能差 |
| **Everysec**（默认） | 每次写命令执行完后先写入内核缓冲区，每秒刷盘一次 | 性能适中 | 最多丢失 1 秒数据 |
| **No** | 由操作系统决定何时刷盘 | 性能最好 | 数据丢失不可控 |

**AOF 重写机制**：当 AOF 文件过大时，后台子进程 `bgrewriteaof` 读取数据库中的键值对，用一条命令记录到新的 AOF 文件，从而压缩文件体积。

**优点**：数据丢失少（取决于写回策略）
**缺点**：文件体积可能很大，恢复速度慢于 RDB

### 2. RDB 快照（Redis Database）

**原理**：将某一时刻的内存数据以二进制格式写入磁盘，是数据的全量快照。

**触发方式**：
- `save` 命令：在主线程执行，会阻塞
- `bgsave` 命令：创建子进程执行，不阻塞
- 配置文件自动触发：满足以下任一条件自动执行 `bgsave`
  - 900 秒内至少 1 次修改
  - 300 秒内至少 10 次修改
  - 60 秒内至少 10000 次修改

**Copy-On-Write（COW）机制**：执行 `bgsave` 时，子进程与父进程共享物理内存，主线程执行写操作时会复制副本，子进程将副本写入 RDB 文件。

**优点**：恢复速度快，全量备份适合冷备
**缺点**：快照频率难以把控，频率太低丢失数据多，频率太高影响性能

### 3. 混合持久化（Redis 4.0+）

**原理**：在 AOF 重写过程中，先将以 RDB 格式写入 AOF 文件前半部分，再将增量命令以 AOF 格式写入后半部分。

**优点**：结合了 RDB 快速启动和 AOF 减少数据丢失的优点
**缺点**：AOF 文件可读性差，兼容性差（4.0 之前版本无法使用）

### 持久化对比总结

| 特性 | RDB | AOF | 混合持久化 |
|------|-----|-----|-----------|
| 启动速度 | 快 | 慢 | 快 |
| 数据完整性 | 取决于快照频率 | 可配置 | 好 |
| 文件体积 | 小 | 大 | 中 |
| 对性能影响 | 较大小 | 较小 | 较小 |

## 题目三：什么是缓存雪崩、击穿、穿透？如何解决？

### 答案

缓存异常会引发三个经典问题：**缓存雪崩、缓存击穿、缓存穿透**。

### 1. 缓存雪崩

**定义**：大量缓存数据在同一时间过期，或者 Redis 故障宕机，导致大量请求直接访问数据库，数据库压力骤增甚至崩溃。

**原因**：
- 大量数据同时过期
- Redis 故障宕机

**解决方案**：
- **大量数据同时过期**：均匀设置过期时间（+随机数）；互斥锁保证单线程构建缓存；后台异步更新缓存。
- **Redis 故障宕机**：服务熔断或请求限流机制；构建 Redis 主从高可靠集群。

### 2. 缓存击穿

**定义**：某个热点数据过期后，大量请求同时访问该热点数据，直接冲垮数据库。

**解决方案**：
- 互斥锁：保证同一时间只有一个请求构建缓存
- 热点数据不设置过期时间，由后台异步更新或提前续期

### 3. 缓存穿透

**定义**：数据既不在缓存中也不在数据库中（如业务误删或恶意攻击），导致大量请求直接访问数据库。

**原因**：
- 业务误操作，缓存和数据库数据都被删除
- 黑客恶意攻击访问不存在的数据

**解决方案**：
- 非法请求限制：API 入口校验请求参数合理性
- 缓存空值或默认值
- **布隆过滤器（推荐）**：在写入数据库时标记到布隆过滤器，查询时先查布隆过滤器判断数据是否存在。布隆过滤器说数据不存在则一定不存在，说存在则可能误判（但概率很低）。

### 对比总结

| 问题 | 原因 | 核心策略 |
|------|------|----------|
| 缓存雪崩 | 大量数据同时过期 / Redis 宕机 | 过期时间均匀化 + 高可用集群 + 限流熔断 |
| 缓存击穿 | 热点数据过期 | 互斥锁 + 后台异步更新 |
| 缓存穿透 | 数据既不在缓存也不在数据库 | 布隆过滤器 + 缓存空值 + 请求校验 |

## 题目四：Redis 的过期删除策略和内存淘汰策略有什么区别？

### 答案

**过期删除策略**和**内存淘汰策略**虽然都是删除操作，但触发条件和目的完全不同。

### 过期删除策略

**目的**：删除已过期的 key（这些 key 仍然占用内存）

**Redis 采用「惰性删除 + 定期删除」组合策略**：

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **惰性删除** | 每次访问 key 时检查是否过期，过期则删除 | CPU 友好，系统资源开销小 | 内存不友好，过期 key 长期占用内存 |
| **定期删除** | 每秒随机抽查 20 个 key，若过期比例 >25% 则继续抽查 | 平衡 CPU 和内存 | 实现复杂，可能仍有漏网之鱼 |

惰性删除由 `expireIfNeeded` 函数实现，定期删除由 `activeExpireCycle` 函数实现（每轮随机抽 20 个 key，默认每秒执行 10 次）。

### 内存淘汰策略

**目的**：当 Redis 运行内存超过 `maxmemory` 阈值时，淘汰数据以释放空间。

**8 种内存淘汰策略**：

| 类别 | 策略 | 淘汰范围 | 说明 |
|------|------|----------|------|
| 不淘汰 | `noeviction`（默认） | — | 内存超限禁止写入，只读/删可正常执行 |
| 过期 key | `volatile-random` | 设有过期时间的 key | 随机淘汰 |
| 过期 key | `volatile-ttl` | 设有过期时间的 key | 优先淘汰最早过期的 |
| 过期 key | `volatile-lru` | 设有过期时间的 key | 最近最少使用 |
| 过期 key | `volatile-lfu` | 设有过期时间的 key | 最近最不常用（Redis 4.0+） |
| 所有 key | `allkeys-random` | 所有 key | 随机淘汰 |
| 所有 key | `allkeys-lru` | 所有 key | 最近最少使用 |
| 所有 key | `allkeys-lfu` | 所有 key | 最近最不常用（Redis 4.0+） |

### LRU vs LFU

- **LRU（Least Recently Used）**：淘汰最近最少使用的数据。Redis 采用近似 LRU 算法（随机采样 5 个，淘汰最久未使用的），避免维护完整链表的性能开销。但无法解决缓存污染问题（如批量读取只读一次的数据）。
- **LFU（Least Frequently Used）**：淘汰访问频率最低的数据。Redis 4.0 引入，通过 `logc` 字段记录访问频次，支持随时间衰减，比 LRU 更合理地应对偶发性访问。

## 题目五：Redis 主从复制、哨兵、Cluster 集群有什么区别？如何保证高可用？

### 答案

Redis 有三种常见的高可用部署模式：**主从复制、哨兵模式、Cluster 集群**。

### 1. 主从复制

**原理**：一主多从架构，主节点负责读写操作，从节点只读并同步主节点的写操作命令。采用读写分离策略，主从命令复制是**异步**的。

**优点**：读写分离提高并发能力，数据冗余备份
**缺点**：主节点故障时需要手动恢复，无法自动切换

**数据一致性**：主从复制是异步的，无法保证强一致性，数据不一致难以避免。

### 2. 哨兵模式（Sentinel）

**原理**：哨兵是一个独立的进程，用于监控主从节点的健康状态。当主节点故障时，哨兵自动完成故障转移——从从节点中选举出新主节点，其他从节点切换到新主节点。

**核心功能**：
- 监控：定期检查主从节点是否存活
- 自动故障转移：主节点故障时自动选举新主节点
- 通知：向客户端发送故障转移事件

**优点**：解决了主从复制无法自动切换的问题，实现了高可用
**缺点**：每个 Redis 节点存储完整数据，数据量受单机内存限制

### 3. Cluster 集群模式（切片集群）

**原理**：将数据分布在多个节点上，每个节点存储一部分数据。Redis Cluster 采用 **16384 个哈希槽（Hash Slot）** 进行数据分区。

**映射关系**：
1. 通过 `CRC16(key) % 16384` 计算 key 对应的哈希槽
2. 将 16384 个槽分配到不同节点（可手动指定或自动平均分配）

**优点**：
- 数据分布在多个节点，突破单机内存限制
- 每个节点可独立读写，提高整体吞吐量
- 支持自动故障转移

**缺点**：不支持跨槽命令（需手动计算路由），数据迁移有复杂度

### 集群脑裂问题

**现象**：主节点网络故障但与客户端连接正常，哨兵误判主节点宕机并选举新主节点，形成两个主节点同时存在。恢复后旧主节点降级为从节点并清空数据同步新主节点，造成数据丢失。

**解决方案**：配置 `min-slaves-to-write` 和 `min-slaves-max-lag`，要求主节点必须有一定数量的从节点且复制延迟在阈值内，否则禁止写入，从而避免脑裂期间客户端写入旧主节点。

### 高可用方案对比

| 方案 | 数据分片 | 自动故障转移 | 适用场景 |
|------|----------|--------------|----------|
| 主从复制 | 否（一主多从） | 否 | 数据量小，只需读写分离 |
| 哨兵模式 | 否 | 是 | 数据量小，需要高可用 |
| Cluster | 是（16384 槽） | 是 | 数据量大，需要分片和高可用 |
