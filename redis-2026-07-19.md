## 题目一：Redis 有哪些数据类型？底层是如何实现的？

### 参考答案

Redis 提供了丰富的数据类型，常见的有五种：**String（字符串）、Hash（哈希）、List（列表）、Set（集合）、Zset（有序集合）**。

随着版本更新，又支持了四种数据类型：**BitMap（2.2）、HyperLogLog（2.8）、GEO（3.2）、Stream（5.0）**。

### 五种数据类型的底层实现

| 数据类型 | 底层数据结构（元素少时） | 底层数据结构（元素多时） |
|----------|------------------------|------------------------|
| String   | SDS（简单动态字符串）  | SDS                    |
| List     | quicklist              | quicklist              |
| Hash     | 压缩列表（ziplist）/ listpack | 哈希表（hashtable） |
| Set      | 整数集合（intset）     | 哈希表（hashtable）    |
| Zset     | 压缩列表（ziplist）/ listpack | 跳表（skiplist） + 哈希表 |

**关键点：**

- **SDS**：相比 C 字符串，SDS 记录了 `len`（长度），获取长度 O(1)；二进制安全；自动扩容避免缓冲区溢出。
- **List**：Redis 3.2 后统一使用 **quicklist**（双向链表 + 压缩列表的组合），替代了之前的双向链表和压缩列表。
- **Hash**：元素少时用压缩列表节省内存，元素多时切换为哈希表以保证查询效率。
- **Zset**：需要同时保证有序和快速查询，使用**跳表 + 哈希表**的组合，哈希表用于按值查找，跳表用于按排名range查询。
- **Redis 7.0** 起，压缩列表已被 **listpack** 替代，彻底解决了连锁更新的性能问题。

## 题目二：Redis 是单线程吗？为什么单线程还这么快？

### 参考答案

**Redis 单线程指的是"接收客户端请求→解析请求→读写操作→发送响应"这个过程由主线程完成。** 但 Redis 启动时会启动后台线程（BIO）：

- **Redis 2.6**：2 个后台线程——关闭文件（AOF 刷盘）、内存释放
- **Redis 4.0+**：新增 `lazyfree` 线程，异步删除大 key（`unlink`、 `flushdb async`）
- **Redis 6.0+**：引入了 **I/O 多线程**，仅用于网络 I/O 读写（`write client socket`），命令执行仍是单线程

**Redis 单线程为什么快？**

1. **内存操作**：大部分操作在内存中完成，采用高效数据结构（SDS、跳表等），瓶颈是内存/带宽而非 CPU
2. **无锁竞争**：单线程避免多线程切换开销和死锁问题
3. **I/O 多路复用**：使用 `epoll` 机制，一个线程处理多个 Socket 连接，主线程只负责"等事件来"和"处理事件"

**Redis 6.0 后引入多线程**：主要是为了提升网络 I/O 并行度（接收/发送数据），官方实测性能提升至少一倍。

```c
// Redis 6.0 I/O 多线程配置
io-threads 4   // 默认4个线程（3个I/O线程 + 1个主线程处理命令）
io-threads-do-reads yes  // 开启多线程读
```

## 题目三：Redis 如何实现数据持久化？AOF 和 RDB 有什么区别？

### 参考答案

Redis 有三种持久化方式：**AOF 日志、RDB 快照、混合持久化**。

### AOF（Append Only File）

- 每执行一条写命令，就将该命令追加到 `aof_buf` 缓冲区
- 通过 `write()` 写入内核缓冲区，由内核决定何时刷盘
- **三种写回策略**：

| 策略       | 安全性 | 性能 |
|-----------|--------|------|
| Always    | 最高（同步刷盘） | 最低 |
| Everysec  | 中等（每秒刷盘） | 较高 |
| No        | 最低（由OS控制） | 最高 |

- **AOF 重写**：后台子进程 `bgrewriteaof` 读取当前数据库所有键值对，用一条命令记录，压缩文件体积
- 重写期间，主进程把新命令同时写入 **AOF 缓冲区**和 **AOF 重写缓冲区**，解决数据一致性问题

### RDB（Redis Database）

- 记录某一时刻的内存数据，以二进制方式写入磁盘
- `save`（主线程，阻塞）和 `bgsave`（子进程，非阻塞）两种生成方式
- `bgsave` 利用 **Copy-On-Write（写时复制）**：子进程共享父进程内存页表，只有在父进程修改数据时才复制副本
- 可配置自动触发：`900s内≥1次修改` 或 `300s内≥10次修改` 或 `60s内≥10000次修改`

### 混合持久化（Redis 4.0+）

- AOF 重写时，先以 RDB 格式写入前半部分，再以 AOF 格式追加增量命令
- 兼具 RDB 快速恢复和 AOF 少数据丢失的优点

### 对比总结

| 特性       | AOF        | RDB         |
|-----------|------------|-------------|
| 文件格式   | 命令日志    | 二进制快照   |
| 恢复速度   | 慢（全量命令回放） | 快（直接读内存） |
| 数据完整性 | 较好（可配置） | 较差（快照间隔期数据丢失） |
| 性能影响   | 较低（异步） | 较高（fork子进程） |

## 题目四：什么是缓存雪崩、击穿、穿透？如何解决？

### 参考答案

这三个问题是缓存异常中最常考察的难题，都是大量请求落在数据库上造成压力。

### 缓存雪崩

**原因**：大量缓存数据**同时过期**或 **Redis 故障宕机**，导致大量请求直接击穿到数据库。

**解决方案**：
- **大量数据同时过期**：给过期时间加随机值（均匀分散）；互斥锁（保证只有1个请求构建缓存）；后台异步更新缓存（数据"永久有效"）
- **Redis 宕机**：服务熔断/请求限流；主从架构保证高可用

### 缓存击穿

**原因**：某个**热点数据过期**，此时大量并发请求刚好访问到这个 key，全部打到数据库。

**解决方案**：与雪崩类似——互斥锁；热点数据不设置过期时间，后台异步更新。

### 缓存穿透

**原因**：请求的数据既不在缓存也不在数据库（数据本身不存在），如恶意攻击或业务误删。

**解决方案**：
1. **参数校验**：API 入口拦截非法请求
2. **缓存空值/默认值**：即使数据库无数据也缓存空结果
3. **布隆过滤器**：写入数据库时在布隆过滤器标记，查询时先过布隆过滤器（布隆说过不存在→一定不存在；说过存在→可能误判）

### 三者对比

| 问题   | 原因                    | 核心差异           |
|--------|------------------------|------------------|
| 雪崩   | 大量缓存同时过期/宕机   | 数据**有**，只是缓存没了 |
| 击穿   | 热点 key 过期           | 数据**有**，只是1个key没了 |
| 穿透   | 数据本就不存在           | 数据**本身没有**     |

### 预防措施汇总

- 缓存数据过期时间 = `固定时间 + 随机偏移量`
- 服务熔断 + 请求限流
- 主从 + 哨兵保证 Redis 高可用
- 热点 key 不设过期时间，后台异步更新
- 布隆过滤器拦截不存在数据的请求

## 题目五：Redis 主从复制原理是什么？增量复制如何实现？

### 参考答案

主从复制是 Redis **高可用**的基础，采用**读写分离**模式：主服务器处理写操作，从服务器只读并同步主服务器的写命令。

### 第一次同步（全量复制）

分三个阶段：

1. **建立连接、协商同步**：从服务器发送 `psync ? -1`，主服务器返回 `FULLRESYNC runID offset`
2. **主服务器生成并发送 RDB**：执行 `bgsave` 生成 RDB 文件，发送给从服务器；从服务器清空本地数据并载入 RDB
3. **发送增量命令**：主服务器将步骤2期间收到的写命令（写入 `replication buffer`）发送给从服务器执行

### 命令传播（长连接）

第一次同步后，双方维护 TCP 长连接，主服务器通过该连接将后续所有写命令传播给从服务器，保证数据一致。

### 分摊主服务器压力（链式复制）

通过"经理"从服务器架构，让从服务器拥有自己的从服务器，主服务器只需生成一次 RDB 即可同步给多个从服务器。

### 增量复制（Redis 2.8+）

网络断开恢复后，不需要全量重同步，只同步断开期间的增量命令：

- 依赖 **repl_backlog_buffer**（环形缓冲区）：主服务器在命令传播时同时写入；记录最近执行的写命令
- 依赖 **replication offset**：主从各自记录复制进度偏移量
- 主服务器根据 `master_repl_offset - slave_repl_offset` 判断数据是否还在环形缓冲区中：在则增量同步；不在则全量同步

**重要配置**：`repl_backlog_size` 应设置足够大（公式：`second × write_size_per_second`），避免环形缓冲区被新数据覆盖导致频繁全量同步。

### 主从数据不一致的应对

- 监控 `INFO replication` 中的 `master_repl_offset` 和 `slave_repl_offset` 差值
- 差值超阈值则跳过该从节点
- 配置 `min-slaves-max-lag` 限制数据同步延迟

### 脑裂问题

**脑裂**：哨兵误判主节点故障，选举出新主节点，集群出现两个主节点。原主节点恢复后降为从节点时会清空本地数据（执行全量同步），导致之前写入的数据丢失。

**解决方案**：
```conf
min-slaves-to-write 1      # 主节点至少1个从节点
min-slaves-max-lag 10      # 同步延迟不能超过10秒
```
主节点检查从节点数量和延迟，不满足则拒绝写请求，防止假故障期间写入的数据在切换后丢失。
