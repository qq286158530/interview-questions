## 题目一：Redis 数据类型有哪些？底层是如何实现的？

### 参考答案

Redis 提供了丰富的数据类型，常见的有 **五种数据类型**：String（字符串）、Hash（哈希）、List（列表）、Set（集合）、Zset（有序集合）。

随着 Redis 版本的更新，后面又支持了四种数据类型：BitMap（2.2 版新增）、HyperLogLog（2.8 版新增）、GEO（3.2 版新增）、Stream（5.0 版新增）。

### 五种数据类型的应用场景

| 类型 | 应用场景 |
|------|----------|
| **String** | 缓存对象、常规计数、分布式锁、共享 session 信息等 |
| **Hash** | 缓存对象、购物车等 |
| **List** | 消息队列（需注意不支持消费组）、文章列表、关注列表等 |
| **Set** | 聚合计算（并集、交集、差集），如点赞、共同关注、抽奖活动等 |
| **Zset** | 排序场景，如排行榜、电话和姓名排序等 |

### 底层数据结构实现

Redis 五种数据类型的底层数据结构并非固定一种，而是根据数据量大小在多种结构之间切换：

| 数据类型 | 底层实现（数据量小） | 底层实现（数据量大） | 备注 |
|----------|----------------------|----------------------|------|
| **String** | SDS（简单动态字符串） | SDS | SDS 解决了 C 字符串的三大缺陷：O(1) 获取长度、二进制安全、防止缓冲区溢出 |
| **List** | quicklist | quicklist | Redis 3.2 后统一使用 quicklist，替代了双向链表 + 压缩列表 |
| **Hash** | listpack（或旧版压缩列表） | 哈希表 | Redis 7.0 压缩列表废弃，替换为 listpack |
| **Set** | 整数集合（intset） | 哈希表 | 元素都是整数且数量 < 512 时用整数集合 |
| **Zset** | listpack（或旧版压缩列表） | 跳表（SkipList） | 元素 < 128 个且每个 < 64 字节时用压缩列表 |

### SDS 的优势

Redis 的 String 类型底层采用 SDS，相比 C 语言字符串有以下优势：

- **O(1) 复杂度获取长度**：SDS 有 `len` 属性记录长度，无需遍历
- **二进制安全**：`buf[]` 存储二进制数据，不依赖 `\0` 结尾，可保存图片、音频等
- **防止缓冲区溢出**：SDS 的 `alloc - len` 可预判空间，空间不足时自动扩容
- **节省内存**：根据字符串长短设计 5 种头部类型（sdshdr5/8/16/32/64）

## 题目二：Redis 是单线程吗？为什么单线程还这么快？

### 参考答案

**Redis 单线程指的是「接收客户端请求 → 解析请求 → 进行数据读写等操作 → 发送数据给客户端」这个过程是由一个线程（主线程）来完成的。**

但 Redis 程序并不是单线程的，Redis 在启动时会启动后台线程（BIO）：

- **Redis 2.6 版本**：启动 2 个后台线程，处理关闭文件、AOF 刷盘任务
- **Redis 4.0 版本之后**：新增 lazy free 线程，异步释放内存（如 `unlink key`、`flushdb async`）

### 三个后台任务队列

- `BIO_CLOSE_FILE`：处理关闭文件任务
- `BIO_AOF_FSYNC`：AOF 刷盘任务（everysec 策略）
- `BIO_LAZY_FREE`：异步释放内存任务

### Redis 6.0 引入多线程 I/O

Redis 6.0 之前网络 I/O 和命令执行都是单线程。6.0 之后，为提高网络 I/O 并行度，采用了 **多个 I/O 线程** 处理网络请求（默认额外创建 3 个 I/O 线程），但**命令执行仍然是单线程**。

### 单线程为什么还这么快？

1. **内存操作**：大部分操作在内存中完成，采用高效数据结构（SDS、跳表等），瓶颈是内存或网络带宽而非 CPU
2. **无锁竞争**：单线程避免了多线程切换开销和死锁问题
3. **I/O 多路复用**：使用 epoll 机制，一个线程处理多个 Socket 请求，监听连接和数据请求，有请求到达才处理

## 题目三：Redis 有哪几种持久化方式？各有何优缺点？

### 参考答案

Redis 共有三种数据持久化方式：**AOF 日志、RDB 快照、混合持久化**。

### 1. AOF 日志（Append Only File）

**原理**：每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里。Redis 重启时读取 AOF 文件的命令逐一执行来恢复数据。

**写回策略**（`appendfsync` 配置项）：

| 策略 | 含义 | 优点 | 缺点 |
|------|------|------|------|
| **Always** | 每次写命令执行完后同步写回硬盘 | 数据不丢失 | 频繁 IO，性能差 |
| **Everysec**（默认） | 每秒将内核缓冲区写入硬盘 | 性能与安全性平衡 | 最多丢失 1 秒数据 |
| **No** | 由操作系统控制写回时机 | 性能最好 | 数据依赖操作系统，可能丢失较多 |

**AOF 重写机制**：当 AOF 文件过大时，后台子进程 `bgrewriteaof` 会读取数据库中所有键值对，用一条命令记录到新 AOF 文件，实现压缩。重写期间主进程的新命令同时写入「AOF 缓冲区」和「AOF 重写缓冲区」，最终合并到新 AOF 文件。

**优点**：数据丢失少  
**缺点**：恢复速度慢（需逐条执行命令）；AOF 文件较大时重写开销大

### 2. RDB 快照

**原理**：将某一时刻的内存数据以二进制方式写入磁盘，记录的是实际数据而非命令。

**快照生成方式**：
- `save` 命令：在主线程执行，会阻塞
- `bgsave` 命令：创建子进程生成 RDB，避免阻塞（默认方式）

**触发配置**（`save` 选项）：
```
900 1   # 900秒内至少1次修改
300 10  # 300秒内至少10次修改
60 10000 # 60秒内至少10000次修改
```

**COW（Copy-On-Write）机制**：bgsave 创建子进程时与父进程共享内存，只读。当主线程写数据时，触发写时复制，该副本数据被写入 RDB，主线程继续修改原数据。

**优点**：恢复速度快（全量数据直接读入内存）  
**缺点**：快照频率难以把握，频率太低丢失数据多，频率太高影响性能

### 3. 混合持久化（Redis 4.0+）

**原理**：在 AOF 重写过程中，先将以 RDB 方式写入 AOF 文件前半部分，再将重写缓冲区中的增量命令以 AOF 格式写入后半部分。

**优点**：结合了 RDB 快速恢复和 AOF 减少数据丢失的优点  
**缺点**：AOF 可读性差；兼容性差（4.0 之前版本无法使用）

## 题目四：什么是缓存雪崩、击穿、穿透？如何解决？

### 参考答案

缓存异常是 Redis 面试中最常考察的高频问题，简称「缓存三穿透」：

### 1. 缓存雪崩（Cache Avalanche）

**含义**：大量缓存 key 在同一时间过期失效，或缓存服务宕机，导致大量请求直接打到数据库。

**解决方案**：
- **过期时间随机化**：给 key 的过期时间加上随机值，避免同时过期
- **互斥锁 / 分布式锁**：缓存未命中时，只允许一个线程查库并写缓存
- **多级缓存**：本地缓存 + Redis 缓存 + 数据库，多层兜底
- **服务降级 / 熔断**：数据库压力过大时，返回默认值或友好提示

### 2. 缓存击穿（Cache Breakdown）

**含义**：某个热点 key 过期失效的瞬间，大量并发请求同时穿透到数据库。

**解决方案**：
- **互斥锁**：缓存未命中时加锁，只允许一个线程查库
- **热点数据永不过期**：不设置 TTL，而是通过逻辑过期（存一个过期时间戳，访问时判断是否过期）
- **提前预热**：系统启动或低峰期将热点数据加载到缓存

### 3. 缓存穿透（Cache Penetration）

**含义**：查询一个数据库和缓存中都不存在的数据（如恶意请求用不存在的 ID），请求每次都穿透到数据库。

**解决方案**：
- **参数校验**：对请求参数做基础校验，过滤非法参数
- **缓存空值 / 短TTL**：将查询结果为空的数据也缓存一个空值（TTL 短一些）
- **布隆过滤器（Bloom Filter）**：将所有存在的数据 hash 到 bitmap，不存在的数据被布隆过滤器拦截
- **双向缓存**：缓存层 + 数据库层双重记录

### 总结对比

| 类型 | 核心问题 | 关键特征 | 主流解决方案 |
|------|----------|----------|-------------|
| **雪崩** | 大量 key 同时过期 | 集中失效 | 随机 TTL、互斥锁、多级缓存 |
| **击穿** | 热点 key 过期的瞬间 | 单一热点 | 互斥锁、永不过期、预热 |
| **穿透** | 查询不存在的数据 | 数据本身不存在 | 参数校验、空值缓存、布隆过滤器 |

## 题目五：Redis 主从复制、哨兵模式、集群模式分别是如何工作的？

### 参考答案

### 1. 主从复制（Master-Slave Replication）

**原理**：一主多从模式，主服务器进行读写操作，从服务器只读。主服务器收到写命令后异步同步给从服务器。

**工作流程**：
- 从服务器启动后向主服务器发送 `SYNC` 命令
- 主服务器收到命令后执行 bgsave 生成 RDB 快照，发送给从服务器
- 从服务器加载 RDB 文件完成全量同步
- 之后主服务器每执行一条写命令就发送给从服务器（命令传播阶段）

**特点**：
- 异步复制，主从数据可能不一致
- 无法做到强一致性保证
- 是哨兵和集群的基础

### 2. 哨兵模式（Sentinel）

**解决的问题**：主从模式下主服务器宕机后需要手动恢复，哨兵实现了自动故障转移。

**核心功能**：
- **监控**：Sentinel 不断检查主从服务器是否正常运行
- **选主通知**：当主服务器宕机时，Sentinel 从多个从服务器中选举出一个升级为新的主服务器
- **故障转移**：通知所有从服务器连接新的主服务器

**工作原理**：
- Sentinel 通过心跳机制检测主服务器是否存活（默认 10 秒一次）
- 主观下线后，Sentinel 向其他 Sentinel 询问是否也认为主服务器下线
- 超过半数 Sentinel 认为下线后，触发客观下线，开始故障转移
- Sentinel 使用 Raft 算法选举领头的 Sentinel，由它完成故障转移

### 3. 集群模式（Cluster）

**解决的问题**：数据量非常大时（海量数据、高并发），单机或主从模式无法满足需求，需要数据分片存储。

**数据分片原理**：
- Redis Cluster 将整个数据库分为 **16384 个槽位（slot）**
- 每个节点负责一部分槽位（如 3 节点集群，每个节点约 5460 个槽位）
- 对 key 进行 `CRC16(key) % 16384` 计算，决定 key 落在哪个槽位

**集群特点**：
- **去中心化**：每个节点保存完整路由信息，客户端可连接任意节点
- **高可用**：每个主节点可配置多个从节点（主从切换），实现故障自动转移
- **数据迁移**：支持在节点间迁移槽位和数据
- **并发能力**：多个节点并行处理请求，扛住更高并发

**与哨兵的区别**：

| 对比项 | 哨兵模式 | 集群模式 |
|--------|----------|----------|
| 数据存储 | 主从复制（一主多从） | 分片集群（数据分片） |
| 槽位数 | 无 | 16384 个 slot |
| 客户端连接 | 连接单一主节点 | 可连接任意节点 |
| 故障转移 | 哨兵监控 + 选举 | 集群内节点投票 + 槽位迁移 |
| 适用场景 | 数据量适中、高可用 | 海量数据、高并发 |
