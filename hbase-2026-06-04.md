## 题目一：HBase 的架构是怎样的？各组件的作用是什么？

### 参考答案

HBase 采用 **Master-Slave（主从）架构**，核心组件包括：

| 组件 | 作用 |
|------|------|
| **ZooKeeper** | 协调服务，负责选举Master、监控RegionServer上下线、存储元数据（如.META.表位置） |
| **Master** | 负责管理所有RegionServer，如表结构的DDL操作、Region的分配、负载均衡、Schema变更 |
| **RegionServer** | 负责数据的读写，是真正存储数据和处理请求的服务节点 |
| **Region** | 表的数据按RowKey切分后的子表，是HBase的负载均衡和数据分片基本单位 |
| **HDFS** | 底层存储，HBase将数据最终存储在HDFS上，利用其副本机制保证数据可靠性 |
| **WAL（Write-Ahead Log）** | 写操作先写入WAL，用于故障恢复，防止数据丢失 |
| **BlockCache** | 读缓存，将热数据缓存在内存中，提升读性能 |
| **MemStore** | 写缓存，数据写入先到MemStore，满了之后Flush成HFile |
| **HFile** | HBase的实际存储文件，存储在HDFS上 |

**读写流程简述：**
- **写流程**：先写WAL → 写入MemStore（内存） → 响应客户端
- **读流程**：先查MemStore → 查BlockCache → 读HFile → 合并返回

## 题目二：HBase 的 RowKey 设计需要遵循哪些原则？有哪些常见优化技巧？

### 参考答案

RowKey 是 HBase 查询的核心，设计质量直接影响读写性能，需遵循以下原则：

### 设计原则

1. **唯一性原则**：RowKey 必须唯一标识一行数据，是数据的 主键
2. **长度原则**：RowKey 长度建议控制在 **3~100字节**之间，越短越好（Bytes.compareTo() 逐字节比较）
3. **散列原则**：避免数据热点（Hotspotting），使数据和负载均匀分布到多个RegionServer上

### 常见优化技巧

| 技巧 | 说明 |
|------|------|
| **加盐（Salting）** | 在RowKey前添加随机前缀（如0~9），使数据分散到不同Region，但会失去有序性 |
| **哈希（Hash）** | 对RowKey取MD5或数字哈希，将数据打散，但查询时需要知道原RowKey |
| **反转（Reversing）** | 反转RowKey某段（如手机号后4位），使热点数据分散到不同Region |
| **时间戳反转** | 如 `Long.MAX_VALUE - timestamp`，让新数据排在前面，常用于时序场景 |
| **尽量不要超过1MB** | 过长的RowKey会增加存储成本和读写延迟 |

### 应用场景示例

- **用户订单查询**：RowKey = `userId + reversed_timestamp`，按用户ID查询本用户订单
- **时间序列数据**：RowKey = `deviceId + (Long.MAX_VALUE - timestamp)`

## 题目三：简述 HBase 中 MemStore 和 HFile 的作用，以及 Flush 的触发条件。

### 参考答案

### MemStore（写缓存）

- 数据写入时，先写入 **WAL（预写日志）**，然后写入 **MemStore**
- MemStore 是内存中的有序数据结构（类似跳表），按 RowKey 排序
- 当 MemStore 写满（达到阈值）时，触发 **Flush**，将数据顺序写入磁盘生成 HFile

### HFile（存储文件）

- HFile 是 HBase 的实际存储文件格式，存储在 HDFS 上
- 由多个 **Data Block、Index Block、Meta Block、Bloom Filter** 等组成
- **Data Block**：存储 KeyValue 键值对，是读取的最小单元，默认64KB
- **Bloom Filter**：快速判断一个Key是否存在于HFile中，减少无效IO
- **Index Block**：索引块，通过 RowKey 快速定位 Data Block

### Flush 触发条件

| 触发条件 | 阈值 |
|------|------|
| MemStore 大小超过 `hbase.hregion.memstore.flush.size` | 默认 **128MB** |
| 所有 MemStore 大小超过 `hbase.regionserver.global.memstore.size`（RegionServer级别） | 默认约占堆内存40% |
| WAL 日志数量超过 `hbase.regionserver.max.logs` | 默认 **32个** |
| 定时触发 | 定期检查，兜底 Flush |

### 重要特性

- **合并（Compaction）**：HFile 过多时，HBase 会自动进行 **Minor Compaction**（合并部分小HFile）和 **Major Compaction**（合并所有HFile并删除标记删除的数据）
- **先写内存后刷盘**：保证写性能优异，数据最终由 Compaction 合并到 HFile

## 题目四：HBase 的数据版本和生命周期是如何管理的？

### 参考答案

### 数据版本管理

HBase 支持 **多版本（Multi-Version）**，每个单元格（Cell）可以存储多个版本的数据，通过时间戳（Timestamp）区分：

- **默认每个列族只保留3个版本**，可通过 `hbase.column.max.version` 配置
- 读取时默认返回 **最新版本**（最新时间戳），也可以指定时间戳读取历史版本
- 删除操作有两种：
  - **删除列（Delete）**：标记要删除的列，下一次 Major Compaction 时真正删除
  - **删除列的特定版本（DeleteColumn）**：删除指定时间戳的版本

### 生命周期（TTL）

- 可以设置列族的 **TTL（Time-To-Live）**，数据超过指定秒数后自动被删除
- TTL 在 **Major Compaction** 时生效，HBase 会清理过期数据
- 公式：`存活时间 = 列族TTL秒数 - (数据写入时间戳 / 1000)`

### 存储结构中的体现

- 每个 Cell 的结构为：`RowKey + Column Family + Column Qualifier + Timestamp → Value`
- 相同 RowKey + 列，在不同时刻写入，Timestamp 不同，形成多版本

## 题目五：HBase 协处理器（Coprocessor）是什么？有哪两类，各自的用途是什么？

### 参考答案

### 什么是协处理器

协处理器是 HBase 提供的 **扩展机制**，允许用户将自定义逻辑 **部署在 RegionServer 上运行**，从而在数据存储层面直接处理数据，无需在客户端做额外过滤。

### 两类协处理器

#### 1. Observer（观察者）—— 类似于触发器

类似于数据库的 **触发器（Trigger）**，在特定事件前后自动执行回调逻辑：

| 类型 | 触发时机 | 典型用途 |
|------|----------|----------|
| **PrePut** | 数据写入 MemStore **之前** | 数据校验、过滤、权限控制 |
| **PostPut** | 数据写入 MemStore **之后** | 写入后记录操作日志、触发二级索引更新 |
| **PreGet** | 读取数据**之前** | 列级权限检查、数据脱敏 |
| **PostGet** | 读取数据**之后** | 结果处理、过滤敏感字段 |
| **PreDelete** | 删除操作**之前** | 删除前审计 |
| **PostDelete** | 删除操作**之后** | 删除后清理关联数据 |
| **RegionObserver** | 综合型，覆盖表级操作 | 综合权限、数据处理 |

#### 2. Endpoint（端点）—— 类似于存储过程

类似于数据库的 **存储过程**，可以在 **服务端任意扫描数据并聚合计算**，将计算逻辑推到数据所在节点：

- **典型应用**：在服务端计算列平均值、求和、统计行数，客户端只需调用接口获取结果
- 常与 **Observer** 配合使用，实现复杂的分布式数据处理（如二级索引构建）

### 应用场景示例

- **二级索引**：通过 Observer 监听 Put 操作，在写入主表数据时同步更新索引表
- **聚合计算**：在 RegionServer 端计算统计值（如 sum、avg），减少网络传输
- **权限控制**：在 PreGet/PrePut 中检查用户是否有权限访问/修改数据
- **数据验证**：在 PrePut 中检查写入数据的格式和合法性

> 📖 参考：https://xiaolincoding.com/hbase/hbase_coprocessor.html
