## 题目一：Spark Driver 和 Executor 的作用是什么？请描述 Spark 作业的完整执行流程。

### 参考答案

**Driver（驱动程序）**
- 运行在集群 Master 节点（或 client 模式下的客户端机器）
- 负责创建 SparkContext，构建 DAG 有向无环图
- 将 DAG 划分为多个 Stage，并将 Task 分发到 Executor
- 收集 Executor 的执行结果

**Executor（执行器）**
- 运行在集群 Worker 节点上，每个节点可以启动多个 Executor
- 负责执行 Driver 分发过来的 Task
- 将计算结果返回给 Driver
- 提供 In-Memory 存储，支持 RDD Cache 和广播变量

**Spark 作业执行流程：**
1. 用户提交 Spark 应用程序，创建 SparkContext
2. SparkContext 向 Cluster Manager 申请资源，启动 Executor
3. Driver 将代码中的 RDD transformations 构造成 DAG
4. DAG Scheduler 将 DAG 划分为多个 Stage（以 Shuffle 为边界）
5. Task Scheduler 将每个 Stage 的 Task（以 Partition 为单位）发送到 Executor
6. Executor 执行 Task，计算结果写回内存或磁盘
7. 执行完成后，Driver 通知 Cluster Manager 释放资源

## 题目二：RDD 的血统（Lineage）机制是如何实现容错的？它与 Checkpoint 有何区别？

### 参考答案

**Lineage（血统）机制**
Spark RDD 采用不可变（Immutable）设计，当某个 RDD 的分区数据丢失时，不需要像 Hadoop 那样做数据副本复制。Spark 会通过血统关系，从源头重新计算丢失的分区。

每个 RDD 都记录了它的依赖关系（Lineage），包含：
- 父 RDD 引用
- 转换操作（transformation）

示例：
```scala
val lines = sc.textFile("hdfs://...")       // RDD lineage 起点
val errors = lines.filter(_.contains("ERROR"))  // filter transformation
val count = errors.count()                    // action 触发执行
```
当 errors 的某个 partition 丢失时，Spark 会根据 lineage 找到 lines 对应的 partition，重新执行 filter 操作来恢复数据。

**Checkpoint（检查点）**
- 将 RDD 的中间结果持久化到可靠存储（HDFS），切断 lineage 链
- 适用于长 lineage 链（迭代计算场景），避免重复计算导致的时间成本过高
- Checkpoint 会触发一个新的 job 来完成数据落地

**两者的核心区别：**

| 特性 | Lineage | Checkpoint |
|------|---------|-----------|
| 存储位置 | 内存 | HDFS 等可靠存储 |
| 重建速度 | 快（内存重算） | 快（直接读取） |
| 适用场景 | 短 lineage 链 | 长 lineage 链或迭代计算 |
| 数据安全性 | 节点故障可能丢失 | 可靠存储，容灾能力强 |

## 题目三：Spark Shuffle 的工作原理是什么？为什么 Shuffle 会影响性能？有哪些优化策略？

### 参考答案

**Shuffle 的工作原理**

Shuffle 是 Spark 中将数据重新分布的过程，发生在需要跨 Partition 重新聚合数据的操作（如 `groupByKey`、`reduceByKey`、`join`、`repartition` 等）时。

以 `reduceByKey` 为例，执行流程如下：
1. **Map 阶段**：每个 Executor 对自己本地的数据按照 Key 进行局部聚合
2. **写入本地磁盘**：Map 端将相同 Key 的数据写入本地临时文件（根据分区器决定写入哪个文件）
3. **Fetch 阶段**：每个 Reduce Task 去各个 Map 节点的磁盘上拉取属于自己分区的数据（涉及网络 IO）
4. **Reduce 阶段**：对拉取到的数据按照 Key 进行合并、排序，最终得到结果

**Shuffle 影响性能的原因**
- **磁盘 IO**：Map 端结果需要写入磁盘，Reduce 端再从磁盘读取
- **网络 IO**：数据需要跨节点传输，数据量大时网络开销显著
- **序列化/反序列化**：跨节点传输需要将数据序列化，反序列化后再处理
- **磁盘空间压力**：大量中间文件会占用磁盘空间
- **GC 压力**：大量临时对象增加 GC 负担

**常见优化策略**
1. **使用 Map端的预聚合**：用 `reduceByKey` 代替 `groupByKey`，在 Map 端先做局部聚合，减少数据量
2. **合理设置分区数**：分区数不宜过少（并行度低）也不宜过多（小任务过多），一般设为 executor cores 的 2-3 倍
3. **使用广播变量**：小表 JOIN 时，将小表广播到每个 Executor，避免 Shuffle
4. **启用 Shuffle 优化参数**：
   - `spark.sql.shuffle.partitions`（默认 200）：调整 Shuffle 分区数
   - `spark.shuffle.service.enabled`：开启 External Shuffle Service，减轻 Executor 压力
5. **使用排序合并（SortMergeJoin）替代广播广播**：在大表 JOIN 时，通过合理分区和排序减少 Shuffle 数据量

## 题目四：Spark 内存管理模型是怎样的？Executor 的内存分为哪几部分？静态内存管理和统一内存管理有什么区别？

### 参考答案

**Spark Executor 内存模型（统一内存管理，Spark 1.6+）**

Executor 的总内存 = `spark.executor.memory` 配置值，内部分为两大区域：

**1. 堆内内存（On-Heap Memory）**
- **Execution 内存**：用于 Shuffle 排序、Hash Join 构造、缓存 RDD 数据（Storage）
- **Storage 内存**：用于缓存 RDD/Broadcast/Result 数据
- **User Memory**：用于用户代码（UDF、Spark 内部元数据等），默认占比约 30%
- **Reserved Memory**：保留内存约 300MB，用于系统预留

**2. 堆外内存（Off-Heap Memory）**
- 通过 `spark.memory.offHeap.enabled=true` 开启
- 使用 Unsafe API 直接操作堆外内存，减少 GC 开销
- 主要用于 Shuffle 和缓存

**Execution 与 Storage 之间的动态共享（统一内存管理）**
- 默认情况下，Execution 和 Storage 内存可以相互借用（默认各占 50%）
- Storage 内存不足时可以借用 Execution 内存，反之亦然
- 通过 `spark.memory.fraction`（默认 0.6）和 `spark.memory.storageFraction`（默认 0.5）控制

**静态内存管理（Spark 1.5 及之前）**
- Execution 和 Storage 内存严格分离，互不借用
- 内存分配是固定的，无法动态调整
- 容易造成资源浪费（一方内存紧张，另一方却有空闲）

**统一内存管理的优势**
- 动态共享提高了内存利用率
- 迭代计算场景下，Execution 和 Storage 可以灵活切换
- 通过 `spark.memory.storageFraction` 可以调整基础比例

## 题目五：广播变量（Broadcast Variable）的原理是什么？它解决了什么问题？使用时应注意什么？

### 参考答案

**广播变量的原理**

广播变量是为了解决同一个 Executor 上多个 Task 重复读取同一份大数据的效率问题。

**没有广播变量时：**
```
Task 1: 需要读取变量 tableA (100MB)  → 每个 Task 都需要序列化传输
Task 2: 需要读取变量 tableA (100MB)  → 每个 Task 都需要序列化传输
Task 3: 需要读取变量 tableA (100MB)  → 每个 Task 都需要序列化传输
```
每个 Task 在启动时，Driver 都会将变量副本发送到 Executor，如果同一个 Executor 上有 100 个 Task，那么同一份 100MB 的数据会被传输 100 次，造成巨大的网络和内存开销。

**使用广播变量后：**
```
Task 1: 需要读取变量 tableA (100MB)  → 广播变量
Task 2: 复用 Executor 缓存的 tableA  → 无需重复传输
Task 3: 复用 Executor 缓存的 tableA  → 无需重复传输
```
Driver 将变量只发送一次到每个 Executor，Executor 将其缓存在内存中，所有在该 Executor 上运行的 Task 共享同一份数据。

**广播变量解决的问题**
1. **减少网络传输**：大变量只传输一次，而非每个 Task 传输一次
2. **降低内存消耗**：同一 Executor 上的 Task 共享广播数据
3. **加速 JOIN**：小表广播后变成 Map JOIN，避免 Shuffle

**使用注意事项**

1. **变量大小**：广播变量适合中等大小的数据（几十 MB 到几百 MB）。太大（如几个 GB）会导致 Executor 内存溢出。

2. **只读特性**：广播变量是只读的，一旦创建就不能修改。

3. **序列化方式**：Spark 默认使用 Java 序列化，效率较低。可以通过自定义序列化器（如 Kryo）提升性能：
   ```scala
   val conf = new SparkConf().set("spark.serializer", "org.apache.spark.serializer.KryoSerializer")
   ```

4. **广播时机**：确保广播变量在使用前已经创建好：
   ```scala
   val broadcastVar = sc.broadcast(Array(1, 2, 3))
   val result = data.map(x => broadcastVar.value.mkString)
   ```

5. **清理时机**：广播变量默认不会自动从 Executor 内存中清除，需要手动 `broadcastVar.unpersist()` 来释放。
