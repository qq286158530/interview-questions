## 题目一：Flink 的架构是怎样的？请描述 Flink 的核心组件及各自职责

### 参考答案

Flink 采用 **流式计算模型**，一切皆为流，其架构核心组件如下：

| 组件 | 职责 |
|------|------|
| **JobManager** | 作业的主节点，负责调度 Task、协调 Checkpoint、处理异常。将 JobGraph 转换为 ExecutionGraph 并分发给 TaskManager |
| **TaskManager** | 作业的工作节点，每个 TM 是一个 JVM 进程，启动时向 RM 注册。有多个 TaskSlot，每个 slot 可运行一个 Task |
| **ResourceManager** | 管理 TaskManager资源（slot），负责 slot 的注册、分配与回收。对接 YARN/K8s 等资源调度器 |
| **Slot** | TM 的资源抽象，默认每 slot 共享该 TaskManager 的网络和内存资源 |
| **Dispatcher** | 为每个提交的用户作业启动一个 JobMaster，同时提供 REST 接口 |

**作业执行流程：**
```
StreamGraph → JobGraph → ExecutionGraph →物理执行图
```
Flink 程序经 StreamGraphBuilder 生成 StreamGraph，JobGraph 由 Client 生成后提交给 JM，JM 申请资源并调度 ExecutionGraph，TM接收后启动对应的 Task 执行。

>📚 来源：[Apache Flink 官方文档 -架构](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/flink-architecture/)

## 题目二：Flink 的 Watermark 机制是什么？如何解决乱序数据问题？

### 参考答案

**Watermark（水位线）** 是 Flink 用于处理**事件时间（Event Time）** 乱序问题的核心机制。

### 本质
Watermark 是一个时间戳 `T`，表示"所有时间戳 ≤ T 的数据已经全部到达"。当窗口收到 watermark ≥ 窗口_end_time 时，窗口触发计算并销毁。

### 解决乱序的核心思想
- 设置 `watermark = max(eventTime) - maxOutOfOrderness`，即"当前已观察到的最大事件时间减去允许的最大乱序延迟"
- 迟于 watermark 的数据默认被**丢弃**，可通过 `AllowedLateness` 延迟窗口销毁时间
- 使用 `SideOutputTag` 将迟到数据侧输出到额外流，保留数据而非丢弃

### 关键 API 示例
```java
WatermarkStrategy<Event> strategy = WatermarkStrategy
    .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner((event, ts) -> event.getEventTime())
    .withIdleness(Duration.ofMinutes(1));
env.fromSource(source, strategy, "Source");
```

### 多并行度下的 Watermark
每个分区独立产生 watermark，Flink 取所有分区 watermark 的**最小值**作为全局 watermark，保证所有分区数据就绪才触发窗口。

### 常见问题
| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 窗口不触发 | watermark 未超过窗口边界 | 检查 watermark 生成逻辑，降低乱序延迟 |
| 数据被误丢弃 | watermark 推进太快 | 调大 `maxOutOfOrderness` 或设置 `AllowedLateness` |
| 多分区 watermark 停滞 | 某分区长时间无数据 | 配置 `withIdleness` 标记空闲分区 |

> 📚 来源：[Apache Flink 官方文档 - Event Time & Watermarks](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/time-concepts/)

## 题目三：Flink 的 Checkpoint 和 Savepoint 有什么区别？Chandy-Lamport 算法在 Flink 中如何实现？

### 参考答案

### Checkpoint vs Savepoint 对比

| 特性 | Checkpoint | Savepoint |
|------|-----------|-----------|
| **触发方式** | Flink 自动周期性触发 | 用户手动触发（程序不运行也可） |
| **用途** | 故障恢复（exactly-once 语义保障） | 计划性停机维护、版本回滚 |
| **存储位置** | 由 Flink 管理（StateBackend） | 手动指定目录 |
| **保留策略** | 新 checkpoint 生成后自动清理旧数据 | 需手动删除 |
| **格式兼容性** | 与作业版本绑定 | 向后兼容，支持作业升级 |
| **状态大小** | 通常较小（增量存储） | 完整快照 |

### Chandy-Lamport分布式快照算法在 Flink 中的实现

Flink 采用 **Chandy-Lamport 变体算法**（称为 "Distributed Snapshots via Checkpointing"），核心思想：

1. **Barrier（屏障）对齐**：JM 向所有输入流注入 `CheckpointBarrier`，数据流中 barrier 携带 checkpoint ID
2. **Barrier 对齐机制**：当并行 source 收到 barrier 时，**暂停数据发送**直到所有输入路径的 barrier 到达，才向下游广播数据
3. **状态快照**：每个算子完成本地状态快照后，向 JM 确认，完成后 barrier 继续前进
4. **Changelog**：Flink 1.18+ 支持增量 Checkpoint，通过 State Changelog 将大状态分层存储到高速内存和低成本存储

```java
// 开启 Checkpoint 配置
env.enableCheckpointing(60000); // 每 60 秒
env.getCheckpointConfig().setCheckpointStorage("hdfs:///flink/checkpoints");
```

### Exactly-Once 保证的条件
- 算子必须**所有输入 barrier 对齐后**才快照
- Source 算子必须**可重置**（Kafka 可通过 offset 重放）
- Sink 算子必须**事务提交**（如2PC）

> 📚 来源：[Apache Flink 官方文档 - Checkpointing](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/checkpoints/)

## 题目四：Flink 的窗口函数有哪几种？请说明 Session Window 和 Global Window 的区别及使用场景

### 参考答案

### 窗口分类

| 窗口类型 | 子类型 | 触发机制 |特点 |
|---------|--------|---------|------|
| **Time Window** | Tumbling（滚动）、Sliding（滑动） | 按时间驱动 | 固定大小，不重叠/可重叠 |
| **Count Window** | Tumbling、Sliding | 按数据条数驱动 | 不推荐（数据延迟不确定） |
| **Session Window** | 动态 Gap | 按数据间隔驱动 | 适合用户行为分析 |
| **Global Window** | 无自然触发 | 需自定义触发器 | 全局汇总或自定义触发 |

### Session Window vs Global Window 深度对比

#### Session Window
```java
DataStream<ClickEvent> sessionStream = stream
    .keyBy(e -> e.getUserId())
    .window(EventTimeSessionWindow.withGap(Time.seconds(10)));
```
- **触发条件**：两条数据的时间戳之差超过指定的 session gap
- **动态性**：窗口大小不固定，取决于数据分布
- **使用场景**：用户行为分析（点击流、页面访问），区分不同用户会话
- **局限**：当数据稀疏时，窗口可能长时间不触发

#### Global Window
```java
DataStream<Tuple2<String, Integer>> global = stream
    .keyBy(e -> e.getKey())
    .window(GlobalWindow.create())
    .trigger(CountTrigger.of(100))  // 必须自定义触发器
    .reduce((a, b) -> a);
```
- **永不自动触发**：除非绑定自定义触发器（CountTrigger、TimeTrigger 等）
- **使用场景**：
  - 全量数据累积后一次性输出（如每收到 100 条数据触发一次）
  - 自定义触发逻辑（满足特定业务条件）
  - 全局状态聚合

### 关键区别总结

| 维度 | Session Window | Global Window |
|------|--------------|--------------|
| 窗口大小 | 动态，基于 gap | 无界，需自定义触发 |
| 自动触发 | 是 | 否 |
| 数据分组 | 按 key + session间隔 | 按 key，全量累积 |
| 典型场景 | 用户会话识别 | 自定义批量处理 |

> 📚 来源：[Apache Flink 官方文档 - Windows](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream_api/operators/windows/)

## 题目五：Flink 的状态管理机制是怎样的？为什么需要 state backend？Flink 支持哪些状态后端？

### 参考答案

### 什么是 Flink 状态

在流处理中，**状态（State）** 是算子在处理过程中需要记住的信息，如聚合值、缓存数据等。Flink 的状态分为：

- **Keyed State**：作用于 KeyedStream，按 key 隔离（`keyBy` 后的状态）
- **Operator State**：作用于整个算子实例，所有并行实例共享（如 Kafka Connector 的 offset）

### 状态后端（State Backend）架构

Flink 将状态存储在 TM 的内存或外部存储中，**state backend** 决定：
1. **状态存储位置**：内存 / RocksDB / 其他
2. **状态序列化方式**：Kryo / Avro / 自定义
3. **Checkpoint 机制**：状态如何持久化到外部存储

### 三种 State Backend 对比

| Backend | 存储介质 | 状态大小 | 读写性能 | 适用场景 |
|---------|---------|---------|---------|---------|
| **HashMapStateBackend** | TM JVM Heap | 受限于内存 | 最高（纯内存） | 小状态、低延迟业务 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘+内存） | 支持 TB 级 | 较低（磁盘+序列化） | 大状态、增量 checkpoint |
| **JdbcStateBackend** | JDBC兼容数据库 | 取决于数据库 | 最低 | 状态需持久化到关系型数据库 |

### RocksDB 状态后端详解
```java
EmbeddedRocksDBStateBackend rocksDB = new EmbeddedRocksDBStateBackend(true); // true = 增量 checkpoint
env.setStateBackend(rocksDB);
```
- **LSM 树结构**：写入性能高，适合大批量状态写入
- **增量 Checkpoint**：只写入变更的 SST 文件，减少 Checkpoint 数据量
- **内存+磁盘**：热数据在 MemTable（内存），冷数据在 SST 文件（磁盘）

### 状态过期与清理
```java
StateTtlConfig ttlConfig = StateTtlConfig
    .newBuilder(Time.hours(24))
    .setUpdateType(StateTtlConfig.UpdateType.OnReadAndWrite)
    .cleanupFullSnapshot()
    .build();

ValueStateDescriptor<String> descriptor = new ValueStateDescriptor<>("cache", String.class);
descriptor.enableTimeToLive(ttlConfig);
```
- `UpdateType`：OnCreateAndWrite（仅创建/写入时）/ OnReadAndWrite（读也刷新）
- `cleanupFullSnapshot`：Checkpoint 时清理过期数据，避免状态膨胀

### 最佳实践
1. **优先使用 RocksDB**：状态超过 10GB 或需要增量 Checkpoint 时
2. **避免大对象作为状态**：状态越大序列化开销越大，Checkpoin 时间越长
3. **及时清理状态**：设置合理的 TTL，避免状态持续膨胀
4. **状态分区**：keyBy 后的 KeyedState 自动按 key 分区，避免单 key 热点

> 📚 来源：[Apache Flink 官方文档 - State Backend](https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/state_backends/)
