### 题目1：Spark RDD 的血缘机制（Lineage）是什么？如何实现容错？

**【答案】**

RDD（Resilient Distributed Dataset）通过**血缘关系（Lineage）**实现容错。每个 RDD 都记录了它的依赖关系和转换操作（Transformation），当某个分区数据丢失时，只需根据血缘重新计算该分区即可，而不需要整个数据集重新计算。

**Lineage 的核心机制：**

1. **DAG 构建**：Spark 根据 Transformation 的依赖关系构建有向无环图（DAG）
2. **窄依赖 vs 宽依赖**：
   - **窄依赖**（Narrow Dependency）：父 RDD 的每个分区最多被一个子分区使用（如 map、filter、union）。可以流水线执行，容错时只需重新计算对应分区
   - **宽依赖**（Wide Dependency）：父 RDD 的一个分区可能被多个子分区使用（如 groupByKey、join、sortByKey）。会触发 Shuffle，需要等父分区全部就绪

3. **容错过程**：
   - 当某个 Partition 丢失时，Spark 从源头（如 HDFS）重新读取数据
   - 沿 Lineage 向上回溯，只重新计算丢失 Partition 相关的依赖链
   - 窄依赖的 Transformation 可以高效局部重算

4. **checkpoint 机制**：对于长 lineage 链或复杂 DAG，可设置 checkpoint 将 RDD 物化到可靠存储（如 HDFS），切断 lineage 链，加速容错

**为什么不用复制（Replication）？**  
Lineage 比复制更高效：复制需要双份存储且同步复杂，Lineage 只在出错时才重算。

📖 来源：https://spark.apache.org/docs/latest/rdd-programming-guide.html

### 题目2：Spark Shuffle 的原理是什么？为什么 Shuffle 是性能瓶颈？

**【答案】**

**Shuffle 是 Spark 中将数据重新分区、跨节点传输的过程**，通常发生在需要全局聚合或数据重分组的操作（如 groupByKey、join、reduceByKey、sortByKey 等）时。

**Shuffle 执行流程：**

1. **Shuffle Write（Map 阶段）**：
   - Map Task 将结果写入本地磁盘
   - 按照 Key 进行分区（默认 HashPartitioner），相同分区号的数据写入同一个文件
   - 每个分区对应一个文件段（segment）

2. **Shuffle Read（Reduce 阶段）**：
   - Reduce Task 根据分区编号，去各个 Executor 拉取属于它的数据
   - 边拉取边排序（如果是需要排序的算子，如 sortByKey）
   - 拉取完成后进行聚合、归约

**为什么是性能瓶颈？**

1. **磁盘 I/O**：Shuffle 数据先落盘再网络传输，比内存操作慢数个数量级
2. **网络 I/O**：数据跨节点传输，带宽是稀缺资源
3. **序列化开销**：数据需要序列化（Kryo/Java）和反序列化
4. **磁盘空间压力**：大量中间文件可能撑爆磁盘
5. **GC 压力**：大量临时对象导致频繁 GC

**优化策略：**

- **合理设置分区数**：分区过少导致数据倾斜，过多增加调度开销（一般 200 左右）
- **使用 mapPartitions 替代 map**：减少函数调用次数
- **减少 Shuffle 数据量**：在 Shuffle 前进行 filter/Map 端聚合
- **使用广播变量**：小表广播避免 Shuffle
- **开启 Sort-Based Shuffle**（Spark 1.2+ 默认）：替代 Hash-Based Shuffle，减少文件数

📖 来源：https://spark.apache.org/docs/latest/rdd-programming-guide.html#shuffle-operations

### 题目3：Spark Broadcast 广播变量的原理和使用场景是什么？

**【答案】**

**Broadcast 广播变量**是将一个只读变量缓存在每个 Executor 上，供所有 Task 共享，避免每个 Task 都持有一份副本，从而节省大量的序列化/反序列化和网络传输开销。

**原理：**

1. **广播过程**：
   - Driver 端将变量序列化（默认使用 Kryo）
   - 将序列化后的数据块存储到 BlockManager（内存+磁盘）
   - 每个 Executor 只拉取一份到本地，其他 Task 共享这份数据

2. **读取机制**：
   - Task 通过 BoltManager 获取广播变量的引用
   - 首次访问时从 Driver 或其他 Executor 拉取
   - 缓存在 Executor 内存中，同一 Executor 上的 Task 无需重复拉取

**使用场景：**

1. **大表 join 小表**：将小表广播到每个 Executor，用 map-side join 替代 Shuffle join，避免 Shuffle
2. **配置参数/字典表**：维表数据广播到各节点
3. **跨 Stage 共享数据**：某些算子需要在多个 Task 间共享只读数据
4. **避免重复大对象传输**：如机器学习中的特征向量、模型参数

**使用示例：**

```scala
// 定义广播变量
val broadcastVar = sc.broadcast(Array(1, 2, 3, 4, 5))

// 在算子中使用
rdd.map(x => x * broadcastVar.value.sum)
```

**注意事项：**

- 广播变量只读，修改无效
- 适合小到中等大小的数据（一般 < 20MB），过大会撑爆 Executor 内存
- 大数据可用 Tungsten API 和分区广播优化

📖 来源：https://spark.apache.org/docs/latest/rdd-programming-guide.html#broadcast-variables

### 题目4：Spark 内存管理模型是怎样的？堆内内存和堆外内存有什么区别？

**【答案】**

Spark 采用**统一内存管理**（Spark 2.0+），将内存分为**堆内内存（On-heap）和堆外内存（Off-heap）**两大区域，统一管理 Execution（计算）和 Storage（存储）内存。

**内存区域划分：**

1. **System Memory**：预留的系统内存（约 300MB）
2. **Storage Memory**：
   - 缓存 RDD / DataFrame 数据（persist/cache）
   - 广播变量、临时存储 shuffle 中间数据（unroll 时使用）
3. **Execution Memory**：
   - Shuffle 排序、聚合、Hash Join 的临时内存
   - 算子执行时的中间对象
4. **Other Memory**：用户代码使用的内存（如 UDF）

**Execution vs Storage 内存动态管理：**

- 两者共享统一内存池，可以相互借用
- Storage 占用 Execution 空间时，若 Execution 需要可强制驱逐
- Execution 占用 Storage 空间时，Storage 可落盘（ Storage Level 设置为 MEMORY_AND_DISK）

**堆内 vs 堆外内存：**

| 特性 | 堆内内存（On-heap） | 堆外内存（Off-heap） |
|------|-------------------|---------------------|
| 管理方式 | JVM GC 管理 | 手动管理（unsafe 操作） |
| 大小配置 | `spark.memory.fraction`（默认 0.6） | `spark.memory.offheap.size` |
| 序列化 | Java 对象（需序列化） | 序列化后的原始字节 |
| GC 开销 | 有 GC 压力 | 无 GC，但管理复杂 |
| 适用场景 | 通用场景 | 大规模数据、减少 GC 干扰 |

**内存管理配置调优：**

- `spark.memory.fraction=0.6`：JVM 堆中用于 Spark 内存的比例
- `spark.memory.storageFraction=0.5`：Storage 内存占比（固定，不动态调整）
- `spark.executor.memory`：Executor 堆内存大小
- `spark.memory.offheap.enabled=true`：开启堆外内存

📖 来源：https://spark.apache.org/docs/latest/configuration.html#memory-management

### 题目5：DataFrame 和 DataSet 的区别是什么？RDD 和 DataFrame 之间的转换如何实现？

**【答案】**

**DataFrame：**

- 类似于传统数据库的二维表结构，有列名和类型（`Dataset[Row]`）
- 每一行是 `Row` 对象，可以理解为一个通用的临时容器
- 支持 SQL 查询、列式存储、 Catalyst 优化器
- 优点：简洁易用，优化强大；缺点：编译时无类型安全

**DataSet：**

- 强类型的数据集合（`Dataset[T]`），T 可以是 case class 或 Java Bean
- 结合了 RDD（强类型）和 DataFrame（优化）的优点
- Encoder 自动序列化，支持 Tungsten 高效序列化
- 编译时类型安全，运行时才检查类型（Java 需要反射， Scala 编译器会检查）

**三者对比：**

| 特性 | RDD | DataFrame | DataSet |
|------|-----|-----------|---------|
| 数据结构 | 分布式对象集合 | 分布式 Row 集合 | 分布式 T 类型集合 |
| 类型安全 | 编译时检查 | 运行时检查 | 编译时检查 |
| 操作方式 | 函数式算子 | SQL / DSL | 算子 + SQL |
| 优化 | 无（手动优化） | Catalyst 优化器 | Catalyst 优化器 |
| 序列化 | Java/Kryo | Row 格式 + Tungsten | Encoders |
| 优点 | 灵活可控 | 优化好、易用 | 类型安全+优化 |
| 缺点 | 繁琐、优化差 | 无编译时检查 | API 相对复杂 |

**RDD ↔ DataFrame 转换：**

```scala
// RDD -> DataFrame（需要手动指定 schema）
val rdd: RDD[Person] = ...
val df = rdd.toDF("name", "age")

// DataFrame -> RDD
val rdd2 = df.rdd

// RDD -> DataSet
val ds = rdd.toDS()

// DataSet -> RDD
val rdd3 = ds.rdd

// DataFrame -> DataSet（需要 encoder）
import spark.implicits._
case class Person(name: String, age: Int)
val ds2 = df.as[Person]
```

**使用建议：**
- 结构化数据 → DataFrame / DataSet（享受 Catalyst 优化）
- 非结构化/半结构化数据 → RDD
- 需要类型安全 → DataSet（ Scala）
- ETL、SQL 查询 → DataFrame
- 推荐优先使用 DataFrame 和 DataSet，除非需要底层 API

📖 来源：https://spark.apache.org/docs/latest/sql-programming-guide.html
