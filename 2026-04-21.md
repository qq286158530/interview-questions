## 题目1：JDK8接口新特性 — 接口可以定义默认方法和静态方法
### 【答案】
**JDK8之前接口的局限性：**
- 接口中只能声明抽象方法，不能有具体实现
- 当需要给接口新增方法时，所有实现类都必须修改，导致大规模代码改动

**JDK8接口的两大新特性：**

**1. 默认方法（Default Methods）**
```java
public interface Animal {
    // 抽象方法
    void eat();
    
    // 默认方法 - 可以有具体实现
    default void sleep() {
        System.out.println("Animal is sleeping");
    }
    
    // 接口内静态方法
    static void description() {
        System.out.println("Animal interface");
    }
}
```
- 默认方法用 `default` 关键字修饰
- 实现类可以继承/重写默认方法，也可以不重写直接使用
- 解决接口升级问题：新增加方法不需要所有实现类都改
- **多继承冲突解决规则**：若一个类同时实现两个接口，且两个接口有同名的默认方法，必须显式覆盖 resolve

**2. 静态方法**
- 用 `static` 关键字修饰，属于接口本身
- 可通过 `InterfaceName.method()` 直接调用
- 用途：提供工具方法，与接口强绑定

**实际应用场景：**
- `List.sort()` — `java.util.List` 的默认方法
- `Iterator.forEachRemaining()` — `java.util.Iterator` 的默认方法

## 题目2：CompletableFuture — JDK8异步编程利器
### 【答案】
**为什么需要CompletableFuture？**
- `Future` 接口只能获取结果，不能组合、链式调用
- `CompletableFuture` 弥补了 Future 的不足，支持回调式编程

**核心用法：**

**1. 创建异步任务**
```java
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
    // 异步执行的任务
    return "result";
});
```

**2. 异步回调 - thenApply/thenAccept**
```java
future.thenApply(result -> result.toUpperCase())
      .thenAccept(System.out::println);
```

**3. 组合多个Future**
```java
CompletableFuture.allOf(f1, f2, f3).join(); // 等待所有完成
CompletableFuture.anyOf(f1, f2).join();     // 任意一个完成
```

**4. 异常处理**
```java
future.exceptionally(ex -> {
    System.out.println("Error: " + ex.getMessage());
    return "default";
});
```

**5. 常用API一览**
| 方法 | 作用 |
|------|------|
| `supplyAsync` | 创建带返回值的异步任务 |
| `runAsync` | 创建无返回值的异步任务 |
| `thenApply` | 任务完成后转换结果 |
| `thenAccept` | 消费结果，无返回值 |
| `thenCompose` | 扁平化链式异步调用 |
| `allOf/anyOf` | 组合多个CompletableFuture |
| `exceptionally` | 异常处理 |

**与线程池结合使用：**
```java
ExecutorService pool = Executors.newFixedThreadPool(4);
CompletableFuture.supplyAsync(() -> work(), pool);
```

## 题目3：Spring Cloud LoadBalancer — 客户端负载均衡
### 【答案】
**什么是客户端负载均衡？**
- 传统Nginx是服务端负载均衡，请求先到Nginx再分发
- Spring Cloud LoadBalancer 是客户端负载均衡 Ribbon 的替代品
- 客户端直接知道所有服务实例地址，自己选择调用哪个

**工作原理：**
1. 服务消费者从注册中心（Nacos/Eureka）获取服务实例列表
2. LoadBalancer 根据配置策略（轮询/随机/响应加权）选择实例
3. 直接发起请求到目标实例

**Spring Cloud 2020版本之后的替代方案：**
- Spring Cloud LoadBalancer 替换了已停维护的 Ribbon
- 默认使用 `RoundRobinLoadBalancer`（轮询策略）
- 可自定义 `@LoadBalancerClient` 配置

**代码示例：**
```java
@Configuration
public class CustomLoadBalancerConfig {
    @Bean
    public ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
            Environment env, LoadBalancerClientFactory factory) {
        String name = env.getProperty(LoadBalancerClientFactory.PROPERTY_NAME);
        return new RandomLoadBalancer(
            factory.getLazyProvider(name, ServiceInstanceListSupplier.class), name);
    }
}
```

**与Feign集成：**
```java
@FeignClient(name = "user-service", loadBalancer = "roundRobin")
public interface UserClient { ... }
```

## 题目4：Spring Cloud Gateway — 动态路由与过滤器链
### 【答案】
**Gateway vs Zuul：**
- Zuul 1.x：基于Servlet，同步阻塞，性能较差
- Zuul 2.x：支持异步但迟迟未发布
- Gateway：基于Spring WebFlux，响应式异步，性能优秀

**核心概念：**
- **Route**：路由id、目标url、谓词数组、过滤器数组
- **Predicate**：匹配请求条件（路径、时间、Header等）
- **Filter**：请求前后拦截，可修改request/response

**工作流程：**
```
请求 → Predicate匹配 → Filter(前置) → 代理服务 → Filter(后置) → 响应
```

**代码配置示例：**
```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service  # lb=loadbalance
          predicates:
            - Path=/user/**
            - After=2024-01-01T00:00:00Z
          filters:
            - StripPrefix=1       # 去掉第一层路径
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 10
                redis-rate-limiter.burstCapacity: 20
```

**自定义全局过滤器：**
```java
@Component
public class AuthFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (token == null) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        return chain.filter(exchange);
    }
}
```

## 题目5：Kafka实现高吞吐量的核心技术 — 零拷贝与顺序写
### 【答案】
**Kafka高吞吐量的6大核心技术：**

**1. 顺序写磁盘**
- 追加写模式，避免随机寻址
- 顺序写速度可达 600MB/s，远超随机写
- Page Cache 利用操作系统缓存

**2. 零拷贝（Zero-Copy）**
传统数据发送流程（4次拷贝）：
```
磁盘 → 内核缓冲区 → 用户空间 → Socket缓冲区 → 网卡
```
Kafka零拷贝（2次拷贝）：
```
磁盘 → 内核缓冲区 → （sendfile系统调用） → 网卡
```
关键系统调用：`FileChannel.transferTo()`

**3. 批量压缩**
- 消息支持批量压缩（GZIP、Snappy、LZ4、ZSTD）
- 减少网络传输量，批量压缩效率高

**4. 分区（Partition）并行**
- 主题可分多个分区，不同分区可并行处理
- 消费者组内分区分配给不同消费者实现并行消费

**5. 高效序列化**
- 使用自定义高效序列化框架（Kafka自己的Binary Protocol）
- 比JSON/XML高效很多

**6. 无锁设计**
- 消息追加到文件是单线程顺序写
- 规避了锁竞争，最大化吞吐

## 题目6：RabbitMQ优先级队列 — 实现插队与VIP服务
### 【答案】
**优先级队列应用场景：**
- 紧急工单优先处理
- VIP客户订单优先处理
- 限时抢购的订单优先级

**开启优先级队列：**
```java
Map<String, Object> args = new HashMap<>();
args.put("x-max-priority", 10); // 最大优先级10
Queue queue = new Queue("priority-queue", true, false, false, args);
```

**发送优先级消息：**
```java
MessagePostProcessor messagePostProcessor = message -> {
    message.getMessageProperties().setPriority(5); // 优先级0-10
    return message;
};
rabbitTemplate.convertAndSend("priority-queue", "urgent order", messagePostProcessor);
```

**消费者配置：**
```java
@Bean
public SimpleRabbitListenerContainerFactory rabbitListenerContainerFactory(
        ConnectionFactory connectionFactory) {
    SimpleRabbitListenerContainerFactory factory = new SimpleRabbitListenerContainerFactory();
    factory.setConnectionFactory(connectionFactory);
    factory.setDefaultRequeueRejected(false); // 拒绝后不重队
    return factory;
}
```

**注意事项：**
- 优先级队列最大优先级有限制（默认255）
- 队列积压时优先级才生效，空队列无意义
- 过多优先级会消耗更多内存

**与延迟队列结合：**
- 可实现"等待时间长的任务自动降级"
- 超时未处理的低优先级任务降为最低级

## 题目7：AI Agent — Function Calling / Tool Use 核心原理
### 【答案】
**什么是Function Calling（函数调用）？**
- 让大语言模型学会调用外部工具/API扩展能力
- 模型输出结构化JSON，描述要调用的函数和参数
- 而不是像传统LLM一样直接输出文字

**工作流程：**
```
用户问题 → LLM判断需要调用工具 → 输出函数名+参数JSON → 执行工具 → 返回结果 → LLM整合回答
```

**代码示例（OpenAI风格）：**
```python
# 定义可调用函数
functions = [
    {
        "name": "get_weather",
        "description": "查询城市天气",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "城市名称"}
            }
        }
    }
]

# 模型输出
response = openai.ChatCompletion.create(
    messages=[{"role": "user", "content": "北京天气怎么样？"}],
    functions=functions,
    function_call="auto"
)
# response.choices[0].message.function_call
# {"name": "get_weather", "arguments": '{"city": "北京"}'}
```

**LangChain中的Tool Use：**
```python
from langchain.agents import load_tools, initialize_agent, AgentType

tools = load_tools(["serpapi", "python_repl"])
agent = initialize_agent(tools, llm, agent=AgentType.ZERO_SHOT_REACT_DESCRIPTION)
agent.run("查询北京今天的天气")  # Agent自动决定调用工具
```

**Function Calling的价值：**
- 解决LLM不知道实时信息的问题（联网搜索、查数据库）
- 让LLM执行实际操作（发邮件、下单、查询）
- 弥补幻觉，函数执行结果是确定的事实

## 题目8：Trie树（前缀树） — 高效字符串检索与前缀匹配
### 【答案】
**什么是Trie树？**
也叫前缀树，是一种树形数据结构，用于高效存储和检索字符串集合。

**核心思想：**
- 每个节点代表一个字符
- 从根到某节点的路径就是一个字符串
- 节点可存储额外信息（如频率、是否为终止节点）

**结构示意：**
```
        root
       /
      'a'
     /
    'p'
   /
  'p'
 / \
'l' 't'   （app, apt, be, best...）
```

**Java实现：**
```java
class TrieNode {
    boolean isEnd;
    TrieNode[] children = new TrieNode[26]; // 只支持小写字母
    int pass; // 经过此节点的字符串数量（可做前缀计数）
}

class Trie {
    private TrieNode root = new TrieNode();
    
    public void insert(String word) {
        TrieNode node = root;
        for (char c : word.toCharArray()) {
            int idx = c - 'a';
            if (node.children[idx] == null) {
                node.children[idx] = new TrieNode();
            }
            node = node.children[idx];
            node.pass++;
        }
        node.isEnd = true;
    }
    
    public boolean search(String word) {
        TrieNode node = findNode(word);
        return node != null && node.isEnd;
    }
    
    public List<String> startsWith(String prefix) {
        // 找到前缀节点后DFS遍历所有子树
        TrieNode node = findNode(prefix);
        List<String> result = new ArrayList<>();
        dfs(node, prefix, result);
        return result;
    }
}
```

**时间复杂度：**
- 插入：O(m)，m为字符串长度
- 查找：O(m)
- 前缀匹配：O(m)

**应用场景：**
- 搜索提示/自动补全
- IP路由前缀匹配
- 字符串检索、去重、前缀计数

## 题目9：滑动窗口最大值 — 单调队列的经典应用
### 【答案】
**问题描述：**
给定数组 `nums` 和窗口大小 `k`，求每个滑动窗口的最大值。

**暴力解法：** O(nk)，每个窗口遍历k个元素 → 数据量大会超时

**单调递减队列优化：O(n)：**
核心思想：维护一个存储数组下标的单调递减队列
- 队首永远是当前窗口最大值的下标
- 新元素入队时，弹出所有比它小的元素（它们永远不可能成为最大值）

```java
public int[] maxSlidingWindow(int[] nums, int k) {
    if (nums == null || k <= 0) return new int[0];
    
    int[] result = new int[nums.length - k + 1];
    LinkedList<Integer> queue = new LinkedList<>(); // 存下标
    int idx = 0;
    
    for (int i = 0; i < nums.length; i++) {
        // 移除超出窗口范围的元素
        while (!queue.isEmpty() && queue.peek() < i - k + 1) {
            queue.poll();
        }
        
        // 移除比当前元素小的元素（它们永远不会是最大值）
        while (!queue.isEmpty() && nums[queue.peekLast()] < nums[i]) {
            queue.pollLast();
        }
        
        // 当前元素入队
        queue.add(i);
        
        // 窗口形成后记录最大值
        if (i >= k - 1) {
            result[idx++] = nums[queue.peek()];
        }
    }
    return result;
}
```

**关键点分析：**
- `LinkedList` 充当单调递减队列
- `queue.peek()` = 当前窗口最大值的下标
- 每个元素最多入队出队各一次 → O(n)时间复杂度

**应用场景：**
- 股票价格窗口最大收益
- 实时排行榜
- 流量控制中的最大并发数

## 题目10：Spring Boot配置文件加载顺序与优先级
### 【答案】
**配置文件类型：**
- `application.properties`
- `application.yaml` / `application.yml`
- `application-{profile}.properties`（Profile特定配置）

**Spring Boot配置文件加载顺序（从低到高）：**
1. `classpath:/application.properties`
2. `classpath:/application.yml`
3. `file:./config/application.properties`（当前目录/config/）
4. `file:./application.properties`（当前目录）

**YAML vs Properties 优先级：**
- 同一位置，`properties` 优先于 `yaml`
- 但 `file:./config/` 目录下的配置 > `file:./` 目录下的配置

**多Profile激活方式：**
```yaml
# application.yml
spring:
  profiles:
    active: dev,redis  # 激活dev和redis
---
# application-dev.yml
server:
  port: 8080
---
# application-prod.yml
server:
  port: 80
```

**命令行覆盖优先级：**
```bash
java -jar app.jar --server.port=8888
# 命令行参数 > 所有配置文件
```

**加载顺序总览（高优先级覆盖低优先级）：**
```
命令行参数 > file:./config/ > file:./ > classpath:./ > 
classpath:/config/ > classpath:/（最高到最低）
```

**Spring Boot 3.x变化：**
- 支持 `application.yaml` 但仍兼容 `application.properties`
- 配置绑定可以使用 `@ConfigurationProperties` 绑定到POJO

**@PropertySource 注解：**
- 默认只加载 `*.properties`，不支持直接加载 `*.yml`
- 如需加载YAML，可用 `YamlPropertiesFactoryBean`
