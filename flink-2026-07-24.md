## 题目一：Flink 的整体架构是怎样的？JobManager 和 TaskManager 的作用是什么？

### 参考答案

Flink 采用 **Master-Slave 架构**，由 JobManager（主节点）和 TaskManager（从节点）组成。

**核心组件：**

| 组件 | 作用 |
|------|------|
| **JobManager** | 负责作业调度、Checkpoint 协调、作业图（JobGraph）生成，是整个系统的"大脑" |
| **TaskManager** | 负责执行具体的算子任务，持有 slot 资源，是实际的"工人" |
| **ResourceManager** | 管理 slot 资源，与 YARN/K8s 等资源管理框架协作 |
| **Dispatcher** | 接收用户提交的作业，为每个作业启动一个新的 JobMaster |

**作业执行流程：**

1. 用户编写的 `DataStream API` 或 `DataSet API` 程序首先被编译为 **JobGraph**
2. JobManager 接收 JobGraph，将其分为多个 **ExecutionJobVertex**，并申请 slot 资源
3. TaskManager 提供 slot，JobManager 将算子子任务（subtask）分配到各 slot 中执行
4. 数据在 subtask 之间通过 **Stream Record** 进行传递

**Slot 与并行度：**
- 每个 TaskManager 有一个或多个 **Slot**，每个 Slot 可以执行一个 subtask
- Slot 的数量决定了 TaskManager 的并发执行能力
- `parallelism.default` 控制全局默认并行度

> **来源说明：** 本题参考 Apache Flink 官方文档 https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/flink-architecture/

## 题目二：什么是 Watermark（水位线）？它是如何解决乱序事件问题的？

### 参考答案

**Watermark（水位线）** 是 Flink 实现**事件时间（Event Time）**处理的核心机制，用于处理乱序事件和决定何时触发窗口计算。

### 基本概念

Watermark 本质上是一个**时间戳**，表示"所有时间戳小于该水位线的数据都已到达"。Flink 会根据 Watermark 来推进事件时间的进度。

### 解决乱序的原理

在现实场景中，数据由于网络延迟、分布式传输等原因，事件到达的顺序往往与事件发生的时间不一致（即**乱序**）。Watermark 的处理策略：

- **Watermark = 事件时间 - 允许延迟时间（T）**
- Flink 等待直到 `Watermark >= 窗口结束时间` 才触发窗口计算
- 允许延迟时间 T 内的乱序数据仍然可以被处理

### 代码示例

```java
// 使用 WatermarkStrategy 指定Watermark生成策略
WatermarkStrategy<MyEvent> wms = WatermarkStrategy
    .<MyEvent>forBoundedOutOfOrderness(Duration.ofSeconds(10))  // 允许10秒乱序
    .withTimestampAssigner((event, timestamp) -> event.getEventTime());

DataStream<MyEvent> stream = env.fromSource(
    kafkaSource, wms, "Kafka Source"
);
```

### 多并行度下的 Watermark

- 每个分区（partition）独立产生 Watermark
- **窗口的 Watermark = 所有分区 Watermark 的最小值**（因为必须等最慢的分区）
- 这确保了所有相关数据都已到达后才触发计算

### 副作用：Late Data（迟到数据）

- 窗口关闭后，仍有数据到达称为 **Late Element**
- 默认情况下迟到数据被丢弃
- 可通过 `sideOutputLateData()` 将迟到数据输出到侧输出流

> **来源说明：** 本题参考 Apache Flink 官方文档 https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/concepts/time-/ 和 Flink Forward 会议演讲资料。

## 题目三：Flink 的 Checkpoint 机制是如何工作的？为什么需要 Checkpoint？

### 参考答案

**Checkpoint** 是 Flink 实现**容错**的核心机制，通过定期保存分布式快照，确保作业失败后能从最近快照恢复，保证 **Exactly-Once** 或 **At-Least-Once** 的处理语义。

### Checkpoint 执行流程

1. **JobManager 触发 Checkpoint**：定期向所有 Source 算子发送 `CheckpointTrigger` 消息
2. **Barrier（屏障）对齐**：
   - Source 算子向下游发送 **Checkpoint Barrier**（简称 barrier），将数据流分为快照前和快照后
   - 下游算子收到所有输入通道的 barrier 后才执行快照（**Barrier Alignment**）
3. **状态快照**：各算子将自身状态（Keyed State / Operator State）异步写入状态后端
4. **完成确认**：JobManager 收到所有算子的确认后，Checkpoint 完成

### 关键配置参数

```java
StreamExecutionEnvironment env = ...;
env.enableCheckpointing(60_000);              // 每60秒执行一次Checkpoint
env.getCheckpointConfig().setCheckpointTimeout(600_000);  // Checkpoint超时10分钟
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30_000); // 两个Checkpoint最小间隔
env.getCheckpointConfig().setMaxConcurrentCheckpoints(1);     // 最多1个并发Checkpoint
```

### 状态后端选择

| 状态后端 | 适用场景 |
|---------|---------|
| **HashMapStateBackend** | 小状态、低延迟测试环境，数据存储在 JVM 堆内 |
| **EmbeddedRocksDBStateBackend** | 大状态、生产环境，数据存储在 RocksDB， 支持增量Checkpoint |

### Exactly-Once vs At-Least-Once

- **Exactly-Once**：需要 Barrier 对齐，容错性好但可能增加处理延迟
- **At-Least-Once**：跳过 Barrier 对齐，延迟低但可能产生重复数据

> **来源说明：** 本题参考 Apache Flink 官方文档 https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/ops/state/checkpoints/ 和 Flink 社区经典博客。

## 题目四：Flink 支持哪几种窗口？Session Window（会话窗口）如何实现？

### 参考答案

Flink 支持丰富的窗口类型，主要分为以下几类：

### 按时间划分

| 窗口类型 | 说明 |
|---------|------|
| **TumblingEventTimeWindow** | 滚动窗口，固定大小，不重叠。如 `window(TumblingEventTimeWindows.of(Time.seconds(5)))` |
| **SlidingEventTimeWindow** | 滑动窗口，大小固定，可重叠。如 `window(SlidingEventTimeWindows.of(Time.seconds(10), Time.seconds(5)))` |
| **SessionWindow** | 会话窗口，根据数据之间的间隔动态创建窗口 |
| **GlobalWindow** | 全局窗口，所有相同 key 的数据划入同一窗口，需配合 `Trigger` 使用 |

### 按计数划分

- **CountWindow**：基于元素数量的窗口，如每 100 条数据触发一次计算

### Session Window 的实现

Session Window 根据**会话间隔**（Gap）来划分窗口。当两条数据之间的时间间隔超过设定的 Gap 时，开启新的窗口。

```java
DataStream<String> stream = ...;

// 使用 EventTime 的 Session Window，Gap 为 10 秒
DataStream<String> sessionWindowed = stream
    .keyBy(r -> r.getKey())
    .window(EventTimeSessionWindows.withGap(Time.seconds(10)));
```

### 动态 Gap（可配置会话超时）

```java
// 使用 SessionWindowTimeGapExtractor 动态计算每个 key 的 Gap
DataStream<String> stream = ...;
DataStream<String> sessionWindowed = stream
    .keyBy(r -> r.getKey())
    .window(EventTimeSessionWindows.withGap(
        new SessionWindowTimeGapExtractor<String>() {
            @Override
            public double extract(String element, long previousWindowEnd) {
                // 根据元素内容动态决定 Gap，例如活跃用户 Gap 大
                return element.isActive() ? 30 : 10;
            }
        }
    ));
```

### 会话窗口的迟到数据处理

默认情况下，迟到数据会触发窗口重新计算（如果使用 `TimeCharacteristic.EventTime` 且开启了 `allowedLateness`）：

```java
// 允许5秒的迟到数据
stream
    .keyBy(r -> r.getKey())
    .window(EventTimeSessionWindows.withGap(Time.seconds(10)))
    .allowedLateness(Time.seconds(5));  // 窗口关闭后5秒内的数据仍可触发窗口更新
```

> **来源说明：** 本题参考 Apache Flink 官方文档 https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/dev/datastream_api/windows/ 和 Flink 社区实践案例。

## 题目五：Flink 的状态管理机制是怎样的？Keyed State 和 Operator State 有何区别？

### 参考答案

Flink 的**状态管理**是其区别于普通流处理框架的核心能力之一。Flink 作业在处理过程中，可以将需要持久化的数据存储为**状态（State）**，并在 Checkpoint 时写入持久化存储，实现容错。

### 状态分类

#### 1. Keyed State（键控状态）

- **必须**在 `keyBy` 之后的算子中使用
- 状态按 Key 进行分区，每个 Key 维护独立的状态
- Flink 自动保证线程安全（同一 Key 的数据总是由同一 subtask 处理）

**支持的类型：**

| 类型 | 说明 |
|------|------|
| `ValueState<T>` | 单值状态，存储单个值 |
| `ListState<T>` | 列表状态 |
| `MapState<K, V>` | 映射状态，键值对 |
| `ReducingState<T>` | 自动聚合（Reduce）的列表状态 |
| `AggregatingState<I, O>` | 聚合状态，输入输出类型可不同 |

**使用示例：**

```java
public class CountWindowAverage extends RichFlatMapFunction<Tuple2<String, Long>, Tuple2<String, Double>> {
    private ValueState<CountAndSum> state;  // 自定义的 POJO

    @Override
    public void open(Configuration parameters) {
        state = getRuntimeContext().getState(
            new ValueStateDescriptor<>("countAndSum", TypeInformation.of(CountAndSum.class))
        );
    }

    @Override
    public void flatMap(Tuple2<String, Long> input, Collector<Tuple2<String, Double>> out) throws Exception {
        CountAndSum current = state.value();
        if (current == null) {
            current = new CountAndSum();
        }
        current.count++;
        current.sum += input.f1;
        state.update(current);
        out.collect(Tuple2.of(input.f0, current.sum / (double) current.count));
    }
}
```

#### 2. Operator State（算子状态）

- 不依赖 `keyBy`，整个算子共享同一份状态
- 常用于 Source 算子（如 Kafka Connector 的消费偏移量）
- 支持 **ListState**（列表状态）、**UnionListState**（故障恢复时分发给所有 subtask）、**BroadcastState**（广播状态）

**使用示例（ListState）：**

```java
public class SourceWithState implements SourceFunction<String> {
    private transient ListState<Long> offsetState;

    @Override
    public void initializeState(StateInitializationContext context) throws Exception {
        offsetState = context.getOperatorStateStore().getListState(
            new ListStateDescriptor<>("offset", Long.class)
        );
    }

    @Override
    public void run(SourceContext<String> ctx) throws Exception {
        for (Long offset : offsetState.get()) {
            // 从上次保存的 offset 恢复，继续消费
        }
    }
}
```

### 状态后端对比

| 状态后端 | 存储介质 | 容量 | 性能 |
|---------|---------|------|------|
| **HashMapStateBackend** | JVM Heap | 受限于内存/JVM堆大小 | 最高 |
| **EmbeddedRocksDBStateBackend** | RocksDB（本地磁盘） | 可扩展到 TB 级 | 稍低（但支持增量Checkpoint） |

### 选择建议

- **小状态、低延迟**（如简单的聚合、过滤）：`HashMapStateBackend`
- **大状态、生产环境**（如 Join、窗口状态）：`EmbeddedRocksDBStateBackend`
