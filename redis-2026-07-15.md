## 题目一：Redis 有哪几种数据类型？分别适用于哪些场景？

### 参考答案

Redis 支持 5 种基础数据类型：

| 数据类型 | 底层实现 | 常用命令 | 典型场景 |
|----------|----------|----------|----------|
| **String** | SDS（简单动态字符串） | `GET/SET/MGET/MSET/INCR/DECR` | 缓存、计数器、分布式锁、Session |
| **Hash** | 压缩列表 / 哈希表 | `HGET/HSET/HGETALL/HMGETHMSET` | 存储对象（如用户信息、商品详情） |
| **List** | 压缩列表 / 双向链表 | `LPUSH/RPUSH/LPOP/RPOP/LRANGE` | 消息队列、排行榜、评论列表 |
| **Set** | 整数集合 / 哈希表 | `SADD/SREM/SMEMBERS/SINTER` | 标签系统、去重、关注/粉丝列表 |
| **Zset** | 压缩列表 / 跳表 | `ZADD/ZRANGE/ZREVRANK/ZSCORE` | 有序排行榜、延迟队列 |

此外还有 3 种高级数据类型：
- **Bitmap**：位图，适合签到、活跃用户统计
- **HyperLogLog**：基数统计，用于 UV 统计
- **Geospatial**：地理位置，用于附近的人功能

## 题目二：Redis 的持久化机制是什么？RDB 和 AOF 有什么区别？如何选择？

### 参考答案

Redis 提供了两种持久化机制：

#### RDB（Redis Database）
- **原理**：定时将内存中的所有数据快照（point-in-time snapshot）写入 `dump.rdb` 文件
- **触发方式**：`SAVE`（同步阻塞）或 `BGSAVE`（后台异步）；也可配置 `save` 规则自动触发
- **优点**：恢复速度快，适合大规模数据备份
- **缺点**：可能丢失最近一次快照之后的数据

#### AOF（Append Only File）
- **原理**：记录所有写操作命令，以追加方式写入 `appendonly.aof` 文件
- **同步策略**：`always`（每个命令同步）、`everysec`（每秒同步，默认）、`no`（由 OS 决定）
- **重写机制**：`BGREWRITEAOF` 压缩 AOF 文件，避免文件过大
- **优点**：数据安全性更高，丢失最多 1 秒数据
- **缺点**：文件体积通常比 RDB 大，恢复速度较慢

#### 生产环境如何选择？

| 场景 | 推荐策略 |
|------|----------|
| 数据安全性要求极高 | AOF + everysec |
| 可以容忍少量数据丢失 | RDB |
| 最佳实践 | RDB + AOF 混合持久化（Redis 4.0+）|

> Redis 4.0 开始支持混合持久化：`RDB + AOF`，在 AOF 重写时先写 RDB 快照，再追加增量 AOF。

## 题目三：Redis 如何实现分布式锁？有哪些注意事项？

### 参考答案

#### 基本实现（SET NX + EX 组合）
```bash
SET lock_key unique_value NX EX 30
```
- `NX`：仅当 key 不存在时设置成功
- `EX 30`：设置 30 秒过期时间，防止死锁

#### 释放锁（Lua 脚本保证原子性）
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

#### 注意事项

1. **防误删**：释放锁时必须验证 value，只有持有锁的客户端才能释放（用唯一 ID 或 UUID）
2. **过期时间**：要预估业务执行时间，设置足够长的 TTL，避免锁自动释放后业务还没执行完
3. **可重入**：同一条客户端可以多次获取锁，需要用计数器实现
4. **主从一致性**：单节点 Redis 的锁在主从切换时可能丢失，推荐使用 **RedLock** 算法（多节点多数派）
5. **Redis 版本**：2.6.12 之前需要用 `SETNX + EXPIRE` 分步操作，存在竞态风险

#### 成熟方案
生产环境推荐使用 Redisson 客户端，它封装了完整的分布式锁实现（可重入锁、公平锁、读写锁等）。

## 题目四：Redis 过期 key 的删除策略有哪些？

### 参考答案

Redis 有三种过期 key 删除策略：

#### 1. 定时删除（主动）
- 为每个设置了过期时间的 key 创建一个定时器，到期立即删除
- **优点**：节约内存，对 CPU 不友好
- **缺点**：CPU 压力大时会影响 Redis 性能

#### 2. 惰性删除（被动）
- 每次访问 key 时检查是否过期，过期则删除
- **优点**：对 CPU 友好，只在必要时检查
- **缺点**：过期的 key 长期占用内存，造成内存泄漏

#### 3. 定期删除（折中）
- 每隔一段时间（如 100ms）随机抽取一批设置了过期时间的 key 进行检查，删除其中过期的 key
- 通过 `hz` 参数控制检查频率
- **优点**：平衡了 CPU 和内存
- **缺点**：可能存在部分漏网之鱼

#### 内存淘汰机制

当内存达到 `maxmemory` 上限时，Redis 还会通过 **6 种淘汰策略** 腾出空间：

| 策略 | 行为 |
|------|------|
| `volatile-lru` | 从已设置过期时间的 key 中淘汰最近最少使用 |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用 |
| `volatile-lfu` | 从已设置过期时间的 key 中淘汰最不经常使用 |
| `allkeys-lfu` | 从所有 key 中淘汰最不经常使用 |
| `volatile-random` | 从已设置过期时间的 key 中随机淘汰 |
| `allkeys-random` | 从所有 key 中随机淘汰 |
| `no-eviction` | 不淘汰，返回错误（默认）|

## 题目五：Redis 主从复制原理是什么？如何解决数据不一致问题？

### 参考答案

#### 主从复制流程

```
1. 从节点执行 REPLICAOF / SLAVEOF host port（或配置 slaveof）
2. 从节点保存主节点地址，建立 Socket 连接
3. 从节点发送 PING 检验主节点连通性
4. 主节点执行 BGSAVE，生成 RDB 快照，同时将后续命令写入 buffer
5. 主节点发送 RDB 文件给从节点
6. 从节点清空本地数据，加载 RDB 文件
7. 主节点将 buffer 中的增量命令同步给从节点（命令传播）
8. 从节点每秒发送 OFFSET 给主节点，主节点持续推送缺失的数据
```

#### 全量同步（SYNC）与增量同步（PSYNC）

- **SYNC**：首次复制时，从节点发送 `SYNC`，主节点生成 RDB 并传输
- **PSYNC**：支持断点续传，从节点记录 `run_id` 和 `offset`，重连后发送 `PSYNC run_id offset`
- **Redis 2.8+** 支持 PSYNC2，优化了 run_id 变化时的兜底策略

#### 数据不一致的常见原因及解决方案

| 原因 | 解决方案 |
|------|----------|
| 主从网络延迟 | 使用集群版（Redis Cluster）|
| 从节点异步复制 | 开启 `min-replicas-to-write` 强制写确认 |
| 主节点宕机 | 使用 Sentinel 哨兵实现自动故障转移 |
| 脑裂问题（网络分区）| 配置 `min-slaves-to-write` 和 `min-slaves-max-lag` |

#### 哨兵机制（Sentinel）

Sentinel 负责监控主从节点健康状态，自动完成：
- 主节点故障检测与自动切换
- 通知应用方（新主节点地址）
- 配置变更同步
